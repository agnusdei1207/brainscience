# 522. 인피니밴드 RDMA (InfiniBand RDMA)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인피니밴드(InfiniBand)는 서버 간 초고속 데이터 전송을 위해 설계된 스위치 패브릭 네트워크이며, RDMA(Remote Direct Memory Access)는 CPU와 OS 커널의 개입 없이 원격 서버의 메모리를 직접 읽고 쓰는 핵심 기술이다.
> 2. **가치**: 기존 TCP/IP의 복잡한 네트워크 스택 오버헤드를 제거하여 수 마이크로초(μs) 대의 초저지연과 수백 Gbps의 대역폭을 동시에 달성, 슈퍼컴퓨터(HPC) 및 AI 클러스터 성능을 극대화한다.
> 3. **융합**: 고성능 컴퓨팅 아키텍처, PCIe 스위치 패브릭, NVMe-oF 스토리지 시스템과 완벽하게 융합되어 현대 대규모 분산 처리 시스템의 혈관 역할을 수행한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
인피니밴드 RDMA는 두 대 이상의 컴퓨터 노드가 네트워크를 통해 데이터를 주고받을 때, 송신 측 장치(HCA, Host Channel Adapter)가 수신 측 장치의 메모리 주소에 직접 데이터를 밀어 넣거나(Write) 가져오는(Read) 메커니즘이다. 이 과정에서 호스트 CPU의 컨텍스트 스위칭, 데이터 복사, TCP/IP 헤더 처리 과정이 완전히 생략된다.

### 💡 비유
인피니밴드 RDMA는 **"회사 간의 직통 우편 튜브"**와 같다.
과거(TCP/IP)에는 서류를 보낼 때 비서(CPU)가 봉투에 넣고, 우체국(커널 스택)에 접수하고, 배달부(네트워크)가 전달하면 상대방 비서가 다시 봉투를 뜯어 책상에 올려놔야 했다. 하지만 RDMA 직통 튜브를 설치하면, 내 책상(메모리)에서 서류를 넣는 즉시 상대방 책상 서랍(원격 메모리)에 꽂힌다. 비서들은 이 과정에 전혀 개입하지 않아 자신의 원래 업무(연산)에 집중할 수 있다.

### 등장 배경 및 발전 과정

#### 1. TCP/IP 스택의 치명적 한계점
클러스터 컴퓨팅이 발전하면서 노드 간 통신량이 폭증했다. 그러나 TCP/IP는 범용 네트워크(인터넷)의 신뢰성을 위해 설계되었기에, 패킷이 도착할 때마다 인터럽트 발생, 커널 공간에서 유저 공간으로의 메모리 복사(Copy), 체크섬 계산 등으로 CPU 자원의 30~50%를 소모해버리는 **I/O 병목 현상**이 발생했다.

#### 2. 무손실(Lossless) 하드웨어 네트워크의 등장
1999년, 컴팩, HP, IBM 등이 주도하여 시스템 내부의 PCI 버스 철학을 외부 네트워크로 확장하려는 시도가 인피니밴드(InfiniBand)로 결실을 맺었다. IB는 처음부터 패킷 손실이 없는(Lossless) 완벽한 하드웨어 흐름 제어를 전제로 설계되었으며, RDMA 기능이 프로토콜 자체에 내장되어 HPC(High Performance Computing)의 절대적 표준이 되었다.

### 심층 동작 원리 (RDMA Write 시나리오)

#### ① Memory Registration (준비 단계)
통신을 시작하기 전, 애플리케이션은 OS 커널에 요청하여 자신이 사용할 버퍼(RAM)를 Paging out(스왑)되지 않도록 메모리에 고정(Pinning)한다. 그리고 이 공간의 주소와 접근 권한 키(R_Key, L_Key)를 HCA 하드웨어에 등록한다.

#### ② Verbs API 호출 (커널 우회)
송신 측 애플리케이션이 `ibv_post_send` (RDMA Write) 함수를 호출하여 Send Queue(SQ)에 작업 요청(WQE, Work Queue Element)을 넣는다. 이때 원격 서버의 대상 메모리 주소와 R_Key를 함께 명시한다. 이 과정은 커널 진입(System Call) 없이 유저 스페이스에서 HCA 칩셋으로 직접 전달된다.

#### ③ 하드웨어 비동기 전송
송신 측 HCA의 DMA 엔진이 호스트 메모리에서 데이터를 퍼 올려 IB 패킷으로 만든 후 패브릭으로 쏜다. 패킷을 받은 수신 측 HCA는 패킷에 적힌 원격 메모리 주소와 키를 확인하고, **수신 측 CPU에 아무런 인터럽트를 걸지 않은 채** 수신 측 메모리 버퍼에 데이터를 직접 써넣는다.

#### ④ 완료 통지 (Completion)
데이터 전송이 끝나면 송신 측 HCA는 Completion Queue(CQ)에 완료 상태를 기록하고, 애플리케이션은 이를 폴링(Polling)하여 성공 여부를 확인한다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 수천 대의 GPU를 동원하여 초거대 AI(LLM) 파운데이션 모델을 학습하는 데이터센터를 설계 중이다. All-Reduce 통신 시 네트워크 대역폭 병목으로 인해 GPU가 연산을 멈추고 데이터를 기다리는 현상(GPU Starvation)이 발생할 우려가 큼.
**판단 및 아키텍처 전략**:
1. **InfiniBand 네트워크 전면 도입**: 노드 간 통신에 400Gbps NDR(Next Data Rate) InfiniBand 스위치와 ConnectX HCA 카드를 채택하여 무손실 하드웨어 네트워크를 구축한다.
2. **NVIDIA NCCL + GPUDirect RDMA 연동**: 통신 라이브러리(NCCL)가 호스트 CPU 메모리를 거치지 않고, 노드 A의 GPU VRAM에서 노드 B의 GPU VRAM으로 직접 데이터를 쏘아 보내는 **GPUDirect RDMA** 아키텍처를 구현한다.
3. **결과**: 수천 대의 노드가 얽힌 대규모 병렬 학습 환경에서도 통신 지연이 10마이크로초 이내로 통제되며, GPU의 연산 효율성(Utilization)을 90% 이상으로 유지할 수 있다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **메모리 고정(Pinning) 남용**: RDMA를 쓰려면 메모리 영역을 물리적으로 고정(Pinning)해야 하는데, 애플리케이션이 무분별하게 너무 넓은 영역을 고정해 버리면 OS가 스왑(Swap)을 하지 못해 심각한 시스템 메모리 부족(OOM) 사태가 발생한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **CPU Offloading**: 일반 이더넷 TCP/IP 사용 시 40Gbps에서 CPU 코어 하나가 100% 점유되나, IB RDMA 사용 시 400Gbps를 전송하면서도 CPU 점유율은 **0~1%**에 불과함.
- **초저지연**: 노드 간 Ping-pong 지연 시간이 50~100μs(TCP)에서 **1.5μs(IB RDMA)** 수준으로 30배 이상 극적으로 향상됨.

### 미래 전망 및 진화 방향
클라우드 벤더(AWS, Azure) 등은 비싼 IB 전용망 대신 기존 이더넷 인프라 위에서 돌아가는 **RoCEv2**로 빠르게 전환하고 있으나, 최상위 랭크 슈퍼컴퓨터와 거대 AI 전용 클러스터에서는 여전히 순수 InfiniBand가 절대적인 성능 우위를 지키고 있다. 앞으로는 DPU(Data Processing Unit) 내부에 RDMA 엔진이 통합되어 복잡한 라우팅과 혼잡 제어까지 하드웨어 칩이 스스로 처리하는 지능형 패브릭으로 진화할 것이다.
