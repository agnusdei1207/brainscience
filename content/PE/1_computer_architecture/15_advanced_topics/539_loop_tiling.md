# 539. 루프 타일링 (Loop Tiling / Loop Blocking)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 거대한 배열이나 행렬을 순회하는 반복문(Loop)을 처리할 때, 데이터를 한 번에 끝까지 스캔하지 않고 CPU 캐시(Cache) 크기에 딱 맞는 작은 '타일(블록)' 단위로 잘라서 연산하도록 반복문의 구조를 변환하는 소프트웨어 최적화 기법이다.
> 2. **가치**: 캐시에 한 번 올라온 데이터가 밀려나기(Eviction) 전에 최대한 여러 번 재사용(Data Reuse)하게 만들어 캐시 적중률(Hit Rate)을 극대화하고, 치명적인 메인 메모리 접근(Memory Wall) 병목을 수십 배 이상 완화한다.
> 3. **융합**: 공간적/시간적 지역성(Locality) 이론을 극대화한 알고리즘이며, 오토 벡터라이제이션(SIMD), 딥러닝 텐서(Tensor) 코어 연산, 그리고 GPU 프로그래밍(CUDA Shared Memory)의 핵심 성능 튜닝 기법으로 융합되어 사용된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
루프 타일링(Loop Tiling, 또는 루프 블로킹 Loop Blocking)은 중첩된 루프(Nested Loop)를 변형하여 데이터 세트를 여러 개의 작은 청크(Chunk, 타일)로 나누는 컴파일러 또는 수동 코드 최적화 기술이다. 타일의 크기는 CPU의 L1 또는 L2 캐시에 쏙 들어갈 수 있는 크기(Cache Size)로 계산되어 설정된다.

### 💡 비유
루프 타일링은 **"거대한 도서관에서의 논문 작성 방식"**과 같다.
- **일반 루프 (비효율)**: 1만 권의 책에서 'A단어'를 모두 찾기 위해 책장(메모리)에서 책 1권을 책상(캐시)으로 가져와 A단어를 찾고 제자리에 돌려놓는다. 그다음 'B단어'를 찾기 위해 아까 봤던 1만 권을 또 처음부터 꺼내온다. 책상을 수만 번 왔다 갔다 하느라 쓰러진다.
- **루프 타일링 (효율)**: 책상(캐시 크기)에 딱 들어가는 **100권(타일)**만 가져온다. 이 100권 안에서 A단어, B단어, C단어를 모두 찾아 메모를 끝낸다(재사용 극대화). 다 끝난 100권만 버리고, 다음 100권을 가져온다. 책상과 책장을 오가는 횟수가 기하급수적으로 줄어든다.

### 등장 배경 및 발전 과정

#### 1. 메모리 장벽 (Memory Wall)의 도래
CPU의 연산 속도(클럭)는 매년 50%씩 빨라졌지만, 메인 메모리(DRAM)의 접근 속도는 10%씩만 빨라졌다. 거대한 행렬 곱셈($N \times N$)을 할 때, 배열이 CPU 캐시보다 크면 이전에 캐싱해둔 데이터가 새 데이터에 밀려나버려(Capacity Miss), 나중에 다시 필요할 때 무조건 메인 메모리에서 수백 사이클을 기다려 다시 읽어와야 하는 비극이 반복되었다.

#### 2. 캐시를 의식한 알고리즘(Cache-oblivious / Cache-aware)의 발전
프로그래머들은 데이터를 메모리에서 한 번 퍼올렸을 때 "뼈를 깎아서라도 캐시 안에서 모든 뽕을 뽑고(Reuse) 버리자"는 철학을 세웠다. 루프의 순서를 뒤집어 데이터를 타일 형태로 조각내는 방식이 고안되었고, 이는 과학 연산(BLAS 라이브러리)과 딥러닝 행렬 곱(MatMul)의 절대적 표준이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (표)

| 요소명 | 상세 역할 | 작동 매커니즘 / 효과 | 비유 |
|--------|----------|----------------------|------|
| **Original Loop** | 변환 전 원본 중첩 루프 | 거대한 행/열을 끝까지 순회하여 심각한 캐시 미스 유발 | 통째로 훑어보기 |
| **Tile Size (B)** | 쪼개는 블록의 크기 | CPU 캐시 사이즈에 맞춰 데이터가 밖으로 밀려나지 않는 최적의 블록(Tile) 크기 결정 (예: 64x64) | 책상 크기에 맞춘 책 더미 |
| **Outer Loop (제어 루프)**| 타일 간의 이동 제어 | 전체 행렬을 거대한 타일 단위로 건너뛰며 스텝(Step) 이동 (`i += B`) | 구역(Zone) 이동 |
| **Inner Loop (연산 루프)**| 타일 내부의 촘촘한 연산 | 선택된 특정 타일(B x B) 내부에서만 데이터를 반복 재사용하며 실제 곱셈/덧셈 수행 | 구역 내부 샅샅이 뒤지기 |

### 정교한 구조 다이어그램 (행렬 곱셈 최적화 비교)

```c
================================================================================
[ Matrix Multiplication: Standard vs Tiled Algorithm ]
================================================================================

[ 1. Standard Matrix Multiply (O(N^3) 캐시 파괴자) ]
  for (int i = 0; i < N; i++) {
      for (int j = 0; j < N; j++) {
          float sum = 0;
          for (int k = 0; k < N; k++) {
              sum += A[i][k] * B[k][j];  // B행렬은 세로(Column)로 읽으므로 극악의 Cache Miss 발생!
          }
          C[i][j] = sum;
      }
  }
  * 문제점: N이 10,000이면 B행렬의 캐시 라인이 다음 루프가 오기 전에 전부 밀려남(Evicted).

[ 2. Tiled Matrix Multiply (Cache Friendly) ]
  // 블록 크기 B (예: L1 캐시에 들어갈 수 있는 64)
  for (int i = 0; i < N; i += B) {              // (Outer) 
      for (int j = 0; j < N; j += B) {          // (Outer) 타일의 좌상단 좌표 결정
          for (int k = 0; k < N; k += B) {      // (Outer)
              // ----------------------------------------------------
              // 아래부터는 [B x B] 크기의 아주 작은 타일 내부 연산
              for (int ii = i; ii < min(i + B, N); ii++) {
                  for (int jj = j; jj < min(j + B, N); jj++) {
                      float sum = 0;
                      for (int kk = k; kk < min(k + B, N); kk++) {
                          sum += A[ii][kk] * B[kk][jj]; 
                      }
                      C[ii][jj] += sum;
                  }
              }
              // ----------------------------------------------------
          }
      }
  }
  * 마법의 효과: 타일 BxB(예: 64x64)는 L1 캐시에 완전히 들어간다!
    타일 내부 연산이 끝날 때까지 데이터가 캐시 밖으로 밀려나지 않아 메모리 속도가 수십 배 향상됨.
```

### 심층 동작 원리

#### ① 캐시 용량 미스 (Capacity Miss)의 제거
기존 루프에서는 A행렬 1개 행과 B행렬 전체가 필요했다. B행렬 용량이 1GB라면 32KB L1 캐시에 절대 들어갈 수 없어 루프가 돌 때마다 메인 메모리 I/O가 터졌다. 타일링은 A행렬의 일부(예: 64x64)와 B행렬의 일부(64x64)를 L1 캐시에 딱 맞게 복사(Load)해 두고, 그 안에서 발생할 수 있는 모든 $64^3$번의 덧셈/곱셈을 다 끝내버린다. **메모리에 다녀오는 횟수를 수학적으로 $O(N^3)$에서 $O(N^3 / B)$로 줄여버린다.**

#### ② 공간적/시간적 지역성 (Locality)의 동시 확보
- **공간적 지역성(Spatial Locality)**: 배열을 한 번 읽을 때 캐시 라인(64 Byte)을 통째로 가져오는데, 타일 안에서 그 라인을 버리지 않고 꽉 채워 쓴다.
- **시간적 지역성(Temporal Locality)**: 방금 읽은 값을 1마이크로초 뒤에 다시 곱셈에 써먹는다.

## Ⅲ. 융합 비교 및 다각도 분석

### 루프 최적화 기법 3대장 트레이드오프

| 최적화 기법 | 핵심 동작 | 목적 | 부작용 (Trade-off) |
|-------------|-----------|------|--------------------|
| **Loop Unrolling (루프 언롤링)** | 루프 안의 명령어를 N배로 복사해 길게 나열 | 분기문 예측 실패(Branch) 오버헤드 감소 및 ILP 극대화 | 코드가 비대해져 I-Cache 미스 발생 확률 증가 (Code Bloat) |
| **Loop Tiling (루프 타일링)** | 3중 루프를 6중 루프로 쪼개어 블록(Tile) 단위 연산 | 데이터 재사용성(Reuse) 극대화로 메모리 I/O 지연(Memory Wall) 해결 | 코드가 극도로 복잡해지며, CPU 캐시 사이즈(B)를 정확히 알아야 최적화됨 |
| **Loop Interchange (루프 순서 변경)**| `for(i) for(j)`를 `for(j) for(i)`로 뒤집음 | C언어의 Row-major 메모리 배치에 맞게 순차 접근(Sequential Access) 유도 | 데이터 구조에 따라 의존성이 꼬여 결과값이 틀려질 수 있음 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 자체 개발한 AI 비전 카메라의 객체 인식 알고리즘(CNN 컨볼루션 계층)이 에지(Edge) 기기의 ARM CPU에서 1장당 2초가 걸려 실시간 처리가 불가능하다. 프로파일러(`perf`)로 확인하니 L2 캐시 미스율(Cache Miss Rate)이 80%에 육박하며, CPU 코어가 메모리를 기다리느라 멈춰(Stall) 있다.
**판단 및 턴어라운드 전략 (Loop Tiling & Vectorization)**:
1. **타일 크기(B) 산정**: ARM 칩의 L1 데이터 캐시는 32KB이다. Float(4Byte) 배열 3개가 캐시에 동시 체류해야 하므로, $3 \times B^2 \times 4 \text{Bytes} \le 32 \text{KB}$ 공식을 통해 최적의 타일 크기 B를 약 40~50으로 계산한다.
2. **타일링 적용**: 4중첩 컨볼루션 루프를 6중첩의 타일 루프로 코드를 전면 재작성(리팩토링)한다. 
3. **루프 언롤링 + SIMD 융합**: 타일 내부(Inner Loop)의 연산은 강제로 `#pragma GCC unroll`과 ARM NEON(SIMD) 인트린직을 먹여 CPU가 비순차 실행을 하도록 강제한다.
4. **결과**: 캐시 미스율이 80%에서 5% 미만으로 곤두박질치고, 1장당 처리 속도가 2초에서 **0.1초(실시간 FPS 달성)**로 20배 이상 기적적으로 수직 상승한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **추상화된 현대 언어에서의 남용**: Python(NumPy 없이 순수 파이썬)이나 Java 같은 가상머신(VM) 기반 언어에서 직접 6중첩 루프를 짜서 타일링을 시도하는 것은 최악의 헛수고다. 객체 헤더와 가비지 컬렉터(GC) 때문에 메모리 주소가 연속되지 않아 타일링의 효과가 완전히 파괴된다. 루프 타일링은 메모리 주소가 철저히 통제되는 C/C++, Rust, CUDA 환경에서만 유효한 극한의 마이크로 최적화 기법이다. (AI 코드는 그냥 이미 타일링이 완벽하게 짜여진 BLAS, cuBLAS 라이브러리를 호출하는 게 정답이다).

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **메모리 대역폭(Memory Bandwidth) 절약**: O(N)번 발생하던 외부 메모리 Fetch를 캐시 안에서 흡수하여, 이론적으로 캐시 크기 $\sqrt{M}$ 에 비례하여 메모리 요청량을 획기적으로 줄임.
- **성능 가속 (Speedup)**: 행렬 곱, 영상 필터 등 2D/3D 배열 연산에서 단순 스칼라 루프 대비 **최소 5배에서 50배 이상의 성능 점프**.

### 미래 전망 및 진화 방향
과거에는 프로그래머가 CPU 캐시 크기를 계산해 수동으로 타일 코드를 짰지만, 현대의 LLVM/GCC 기반 컴파일러는 `Polly` 플러그인 등 다면체 최적화(Polyhedral Optimization) 수학 모델을 통해 개발자 몰래 백그라운드에서 자동으로 코드를 타일링해 버린다. 또한 NVIDIA GPU의 최신 Hopper 아키텍처는 하드웨어 내부에 **TMA (Tensor Memory Accelerator)**를 박아 넣어, 개발자가 루프를 짤 필요도 없이 하드웨어 칩이 스스로 글로벌 메모리에서 비동기적으로 타일을 퍼 와서 텐서 코어에 쑤셔 넣는 극단적인 자동화 영역으로 진화하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **루프 타일링이 뭔가요?**: 도서관(메모리)에서 책 10,000권의 단어를 조사할 때, 1권씩 책상에 가져왔다 갖다 놓기를 만 번 반복하지 않고, 내 책상(캐시) 크기에 딱 맞게 책 50권(타일)씩 묶어서 가져오는 지혜로운 방법이에요.
2. **왜 50권씩 묶나요?**: 책상이 좁아서 51권을 올리면 책이 바닥으로 떨어져 버리기 때문이에요. 내 책상 크기(CPU 캐시 사이즈)를 정확히 계산해서 한 번에 올릴 수 있는 최대량을 가져오는 거죠.
3. **가장 좋은 점은요?**: 무거운 책을 들고 도서관 복도를 1만 번 왔다 갔다 하느라 다리가 부러지는(Memory Wall) 현상을 막고, 자리에 편히 앉아서 조사만 빨리 끝낼 수 있답니다!
