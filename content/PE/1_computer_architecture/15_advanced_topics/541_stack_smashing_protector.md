# 541. 스택 스매싱 프로텍터 (SSP, Stack Smashing Protector)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 해커가 버퍼 오버플로우 기법으로 메모리의 반환 주소(Return Address)를 조작하는 것을 막기 위해, 컴파일러가 지역 변수와 반환 주소 사이에 무작위 난수(Canary)를 끼워 넣고 함수 종료 시 이 값의 변조 여부를 검사하는 소프트웨어 기반 방어 기법이다.
> 2. **가치**: 코드의 메모리 취약점(C/C++의 고질적 문제)을 개발자가 미처 다 잡지 못하더라도, 컴파일러 옵션(예: GCC의 `-fstack-protector`) 하나만으로 시스템 해킹의 90% 이상을 차지하는 스택 스매싱 공격을 انتہائی 저렴한 비용으로 1차 방어해 낸다.
> 3. **융합**: ASLR(메모리 배치 무작위화), NX/DEP(데이터 실행 방지) 기술과 함께 현대 운영체제의 필수적인 '3대 메모리 방어선'으로 융합되어 활동하며, 최근 하드웨어 방어(Intel CET)의 도입을 촉발한 선구자적 기술이다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
스택 스매싱 프로텍터(Stack Smashing Protector, SSP), 흔히 **스택 카나리(Stack Canary)**라고 불리는 이 기술은 함수 호출 시 생성되는 스택 프레임(Stack Frame)에서 버퍼(지역 변수)와 중요한 제어 데이터(SFP, RET) 사이에 검증용 마커(Canary)를 삽입하는 기술이다. 함수가 종료(Return)되기 직전 이 카나리 값이 원본과 달라졌는지 확인하여, 만약 변경되었다면 즉시 프로세스를 강제 종료(Abort)시킨다.

### 💡 비유
스택 카나리는 **"탄광 속의 카나리아 새"**와 같다.
옛날 광부들은 일산화탄소 가스를 감지하기 위해 가스에 민감한 아주 작은 새(카나리아)를 새장에 넣어 탄광 안쪽에 매달아 두었다(스택 카나리 삽입). 만약 독가스(버퍼 오버플로우 공격)가 광부(반환 주소)에게 도달하려면, 반드시 그 앞에 있는 카나리아를 먼저 거쳐야 한다. 광부는 퇴근하기 전 카나리아를 쳐다보고(함수 종료 전 검사), 새가 죽어있으면(카나리 값 변조) 독가스가 퍼졌음을 직감하고 즉시 대피(프로세스 종료)한다.

### 등장 배경 및 발전 과정

#### 1. 1990년대 알레프 원(Aleph One)의 Phrack 매거진 충격
1996년, 해커 Aleph One은 "Smashing The Stack For Fun And Profit"이라는 전설적인 문서를 통해, C언어의 `strcpy`, `gets` 같은 함수가 버퍼(배열)의 경계를 검사하지 않는 약점을 이용해 메모리의 반환 주소를 덮어쓰고 루트(Root) 권한을 탈취하는 기법을 세상에 폭로했다. 이후 전 세계의 서버가 스택 스매싱에 의해 추풍낙엽처럼 털리기 시작했다.

#### 2. StackGuard와 ProPolice의 등장
버그 없는 코드를 짜는 것은 불가능했으므로, 해커의 오버플로우가 발생하더라도 반환 주소까지 닿기 전에 감지하는 아이디어가 고안되었다. 1997년 StackGuard가 제안되었고, IBM이 이를 개선한 **ProPolice**를 GCC 컴파일러에 기증하면서 현대 리눅스/윈도우(GS 옵션)의 기본 스택 보호 기술로 자리 잡았다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (표)

| 요소명 | 상세 역할 | 작동 원리 / 생성 메커니즘 | 방어 역할 |
|--------|----------|---------------------------|-----------|
| **Local Buffer** | 데이터 저장 공간 | C언어에서 선언한 지역 배열 변수 (해커의 공격 진입점) | 침입 통로 |
| **Canary Value** | 무결성 검증 값 | 프로그램 실행 시 스레드 로컬 스토리지(TLS) 또는 글로벌 변수에 저장된 무작위 난수 | 지뢰 / 센서 |
| **Prolog (프롤로그)**| 검증 값 심기 | 함수 진입 시 스택을 할당한 후, 레지스터(TLS)에서 카나리 값을 읽어 버퍼 바로 밑에 PUSH함 | 카나리아 새장 걸기 |
| **Epilog (에필로그)**| 검증 및 폭파 | 함수 종료 전, 스택에 있는 카나리 값을 꺼내어 TLS의 원본과 XOR 비교. 다르면 `__stack_chk_fail` 호출 | 새 생존 확인 |
| **Return Address**| 실행 제어권 위치 | 함수가 끝난 후 돌아갈 메모리 주소 (해커의 최종 타겟) | 광부 (보호 대상) |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ Stack Smashing Protector (Canary) Memory Layout ]
================================================================================

 [ 1. 정상적인 함수 호출 시 메모리 (Stack Frame) ]
  고주소 (High Address)
  ┌─────────────────────────┐
  │ 함수의 매개변수 (Args)   │
  ├─────────────────────────┤
  │ Return Address (RET)    │ <-- 해커가 조작하고 싶어 하는 핵심 타겟
  ├─────────────────────────┤
  │ Saved Frame Ptr (SFP)   │
  ├─────────────────────────┤
  │ ⭐️ Canary (난수: 0x8F) ⭐️│ <-- 프롤로그에서 여기에 지뢰(카나리)를 매설함
  ├─────────────────────────┤
  │ Local Buffer [char 8]   │ <-- 프로그램 변수 구역
  │ [ ][ ][ ][ ][ ][ ][ ][ ]│
  └─────────────────────────┘
  저주소 (Low Address - 스택이 자라는 방향)

 [ 2. 버퍼 오버플로우 해킹 공격 발생! ]
  해커가 `strcpy`를 통해 8바이트 배열에 16바이트 악성 데이터를 밀어 넣음.
  (스택은 저주소에서 고주소 방향으로 값을 덮어씀)
  
  [A][A][A][A][A][A][A][A] [B][B][B][B] [C][C][C][C] 
   (Local Buffer 덮어씀)     (Canary 파괴!)   (RET 주소 변조 성공?)

 [ 3. 함수 에필로그 검증 (방어 성공) ]
  함수가 끝나기 직전, CPU는 스택에 있는 Canary 위치를 읽어봄.
  "원래 0x8F여야 하는데, 지금 [B][B][B][B]로 바뀌어 있네?"
  -> 해킹 시도 감지! Return 주소([C][C][C][C])로 점프하지 않고 프로세스를 
     즉시 강제 종료(SIGABRT) 시켜버림.
```

### 심층 동작 원리

#### ① 난수 카나리 (Random Canary)의 생성
카나리 값이 `0x1234`로 고정되어 있다면 해커는 공격 데이터 중간에 `0x1234`를 교묘하게 섞어서 오버플로우를 시도할 것이다(우회 공격). 따라서 현대 OS는 프로세스가 기동할 때 커널에서 생성된 강력한 랜덤 난수를 스레드 전용 메모리(TLS, `fs:0x28` 등)에 저장한다. 공격자는 실행할 때마다 바뀌는 이 카나리 값을 절대 예측할 수 없다.

#### ② 널 카나리 (Terminator Canary) 기법
C언어의 문자열 함수(`strcpy`, `puts` 등)는 널 문자(`0x00`)를 만나면 문자열의 끝으로 인식하고 복사를 중단한다. 이를 역이용하여, 랜덤 카나리 값의 첫 번째 바이트를 무조건 `0x00`으로 설정하는 기법이다. 해커가 악성 데이터를 덮어쓰려 해도, 카나리의 첫 바이트가 문자열의 끝인 줄 알고 함수가 스스로 복사를 멈춰버리기 때문에 카나리를 뚫고 나갈 수 없게 된다.

#### ③ 변수 재배치 (Variable Reordering)
배열 버퍼가 오버플로우 되면 배열보다 '위에' 선언된 다른 지역 변수(예: 권한 체크를 하는 `int is_admin`)가 덮어씌워질 수 있다. ProPolice(SSP) 컴파일러는 코드에 적힌 순서를 무시하고, 메모리 배치를 강제로 조작하여 **버퍼(배열) 변수들을 무조건 스택의 가장 밑바닥(저주소)으로 끄집어 내린다**. 따라서 배열이 넘치더라도 다른 일반 변수들을 건드리지 못하고 오직 카나리만 들이받게 만든다.

## Ⅲ. 융합 비교 및 다각도 분석

### 3대 메모리 취약점 방어 기법 비교

| 방어 기술 | 방어 대상 및 원리 | 취약점 (우회 방법) | 비유 |
|-----------|-------------------|--------------------|------|
| **SSP (Canary)** | 스택 버퍼 오버플로우 시 RET 변조 차단 | 메모리 유출(Memory Leak) 버그를 통해 카나리 값을 미리 읽어낸 후 똑같이 덮어씀 | 출입문 도어락 |
| **ASLR / PIE** | 실행 시마다 힙, 스택, 라이브러리 주소를 무작위 배치 | 주소가 랜덤이므로 해커가 점프(JOP/ROP)할 목적지 주소를 계산할 수 없음 | 매일 바뀌는 비밀기지 |
| **NX Bit (DEP)** | 스택이나 힙 영역의 메모리에 '실행 불가' 속성을 부여 | 스택에 심어둔 쉘코드가 실행되지 않음. 해커는 기존 코드를 짜깁기하는 ROP 기법으로 이를 우회함 | 총기 소지 금지 구역 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대국민 웹 서비스에 사용되는 고성능 이미지 리사이징 C++ 라이브러리를 빌드해야 한다. 초당 수천 장의 이미지를 처리해야 하므로 성능이 생명이다. 보안팀은 모든 빌드에 `-fstack-protector-all` 옵션을 강제하려 하고, 개발팀은 함수를 호출할 때마다 카나리를 심고 검사하는 CPU 오버헤드 때문에 성능이 30% 하락한다며 반대하고 있다.
**판단 및 타협 전략 (최적의 컴파일러 옵션 선택)**:
1. **`-fstack-protector-all` 기각**: 이 옵션은 내부 변수가 아예 없는 단순한 수학 함수에조차 무조건 카나리를 삽입하므로 엄청난 성능 쓰레기(Overhead)를 양산한다.
2. **`-fstack-protector-strong` 채택**: 현대 컴파일러의 훌륭한 절충안이다. 로컬 변수에 배열(Array)이 있거나, 스택 변수의 주소 포인터(`&var`)를 다른 곳으로 넘기는 등 **실제로 오버플로우 위험이 있는 함수에만 선택적으로 카나리를 삽입**한다.
3. **결과**: 성능 오버헤드는 전체 애플리케이션 기준 1~2% 내외로 극소화하면서도, 실질적인 스택 스매싱 위험 구역은 100% 방어하는 최적의 보안-성능 밸런스를 달성한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **정보 유출(Information Leak) 취약점 방치**: 카나리는 무작위 난수지만, 프로그램 안에 `printf` 포맷 스트링 버그(Format String Bug)나 범위를 넘어선 메모리 읽기(Out-of-bounds Read) 버그가 존재한다면 헛수고가 된다. 해커가 이 버그를 이용해 카나리 값을 슬쩍 읽어낸(Leak) 뒤, 오버플로우 공격 페이로드의 정해진 위치에 읽어낸 카나리 값을 그대로 끼워 넣으면 방어 로직은 완벽히 무력화된다. 카나리는 만능 방패가 아니라 '눈 가리고 아웅'하는 지연 전술일 뿐이다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **취약점 공격 무력화**: 스크립트 키디(Script Kiddie)들이 흔하게 사용하는 고전적인 스택 오버플로우 쉘코드 익스플로잇의 **99% 이상을 컴파일 단계에서 차단**.
- **성능 가성비**: 하드웨어 교체 없이 순수 컴파일러 옵션 추가만으로 극강의 방어 체계를 구축 (오버헤드 1% 미만).

### 미래 전망 및 진화 방향
소프트웨어적인 스택 카나리는 수십 년간 훌륭하게 작동해 왔지만, 앞서 언급한 '메모리 릭을 통한 우회'라는 치명적 약점을 극복할 수 없었다. 이 때문에 최신 컴퓨팅 아키텍처는 소프트웨어적인 꼼수(Canary)를 버리고, 하드웨어 칩셋 자체가 아예 해커가 접근할 수 없는 딴 세상(격리된 메모리)에 반환 주소를 복사해두는 **Intel CET (Shadow Stack)** 나 **ARM PAC** 같은 순수 하드웨어 방어 기술(HW-CFI)로 진화하며 소프트웨어 카나리를 영광스러운 은퇴의 길로 보내고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **스택 스매싱이 뭔가요?**: 나쁜 도둑이 일기장(메모리)에 억지로 긴 글을 쓰다가, 뒷장에 적혀있던 '집으로 가는 약도(반환 주소)'까지 지워버리고 '도둑의 소굴로 가는 약도'로 바꿔치기하는 나쁜 짓이에요.
2. **어떻게 방어하나요?**: 컴파일러 요정이 일기장 중간에 도둑이 절대 모르는 '비밀 암호 도장(카나리)'을 쾅 찍어둬요.
3. **왜 도장이 필요한가요?**: 도둑이 글을 길게 써서 약도를 바꾸려면 무조건 중간에 있는 이 도장까지 덮어써야 하거든요. 나중에 집에 갈 때 도장이 망가진 걸 발견하면 즉시 112를 부르고 멈춰버린답니다!
