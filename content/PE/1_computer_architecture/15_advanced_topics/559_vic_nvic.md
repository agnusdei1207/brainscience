# 559. 벡터형 인터럽트 컨트롤러 (VIC, NVIC) - 초고속 인터럽트 분기 처리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 하드웨어 인터럽트 발생 시 CPU가 "누가 불렀는지" 확인하기 위해 장비들을 일일이 뒤지는 비효율을 없애고, 각 인터럽트 번호에 해당하는 실행 주소(Vector)를 하드웨어가 즉시 찾아내어 CPU를 해당 지점으로 바로 점프시키는 **지능형 인터럽트 관리 장치**이다.
> 2. **가치**: 인터럽트 처리에 소요되는 시간(Latency)을 극한으로 줄여 실시간 제어의 정밀도를 높이며, 우선순위 기반의 **인터럽트 중첩(Nesting)** 기능을 통해 중요한 작업을 0순위로 처리하게 돕는다.
> 3. **융합**: ARM Cortex-M 계열의 **NVIC(Nested Vectored Interrupt Controller)**는 하드웨어 레벨의 레지스터 스택킹 기능과 융합되어, 임베디드 시스템에서 소프트웨어 개입 없는 초고속 문맥 교환을 실현한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
인터럽트 컨트롤러는 외부 장치(키보드, 타이머, 센서 등)의 요청을 CPU에 전달하는 중간 관리자다.
- **VIC (Vectored Interrupt Controller)**: 인터럽트 소스별로 고유한 서비스 루틴(ISR) 주소를 표(Vector Table)로 관리하고, 신호가 오면 해당 주소를 CPU에 직접 전달한다.
- **NVIC (Nested Vectored Interrupt Controller)**: VIC의 기능에 '중첩(Nesting)' 기능을 더해, 현재 처리 중인 낮은 순위 인터럽트를 가로채고 더 중요한 인터럽트를 먼저 처리하게 해주는 고성능 컨트롤러다.

### 💡 비유
인터럽트 컨트롤러는 **"호텔의 지능형 교환기"**와 같다.
- **과거 (Non-vectored)**: 손님이 벨을 누르면 프론트 직원이 복도를 돌며 "누가 눌렀어요?"라고 물어봐야 한다. (Polling 방식, 매우 느림).
- **VIC (Vectored)**: 벨이 울리는 순간 교환기 화면에 "301호 손님"이라고 뜨고, 직원은 즉시 301호로 뛰어간다. (빠름).
- **NVIC (Nested)**: 301호 청소를 하다가도 1층 로비에 "불이야!" 벨이 울리면, 직원은 청소를 즉시 중단하고 불부터 끄러 간다. 불을 다 끄면 아까 하던 301호 청소를 마저 한다. (우선순위 관리).

### 등장 배경 및 발전 과정

#### 1. 대규모 주변 장치의 등장
컴퓨터에 연결되는 장치가 많아지면서 "누가 신호를 보냈는가?"를 찾는 시간 자체가 시스템 병목이 되었다. 

#### 2. 실시간성(Real-time)의 요구
자율주행이나 산업용 로봇은 1마이크로초의 지연도 용납하지 않는다. 하드웨어 레벨에서 ISR 주소를 즉시 로딩해주는 벡터형 방식이 MCU 아키텍처의 표준이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### VIC vs NVIC 핵심 기능 비교 (표)

| 구분 | VIC (Vectored) | NVIC (Nested Vectored) |
|------|-----------------|------------------------|
| **핵심 기술** | Vector Table 기반 분기 | **인터럽트 중첩(Nesting) + 하드웨어 스택킹** |
| **분기 방식** | 하드웨어가 ISR 주소 공급 | 하드웨어가 ISR 주소 공급 |
| **우선순위** | 고정 또는 단순 우선순위 | **정교한 선점(Preemption) 지원** |
| **컨텍스트 저장**| 소프트웨어가 수동 저장 | **하드웨어가 자동으로 스택에 저장** |
| **주요 적용** | 구형 ARM7 / ARM9 | **현대 ARM Cortex-M 전 시리즈** |

### 정교한 구조 다이어그램 (NVIC 동작 매커니즘)

```ascii
================================================================================
[ NVIC: Nested Interruption & Hardware Stacking ]
================================================================================

 [ External Interrupts ]
  - Timer  (High Priority) ──┐
  - UART   (Low Priority)  ──┼──▶ [ ⚡️ NVIC Logic ] ──▶ [ CPU Core ]
  - Sensor (Medium)        ──┘             │
                                           ▼
 ┌────────────────────────────────────────────────────────────────────────────┐
 │  Vector Table (In Flash/RAM)                                               │
 │  - Index 0: Reset Handler Address                                          │
 │  - Index 15: SysTick Handler Address                                       │
 │  - Index 32: ⭐️ UART_ISR_ADDR (Current Executing) ⭐️                        │
 └─────────────────────────────────────────┬──────────────────────────────────┘
                                           │ (Low Priority ISR 도중 고순위 발생!)
                                           ▼
 ┌────────────────────────────────────────────────────────────────────────────┐
 │  🛡️ Hardware Action (Auto Stacking)                                        │
 │  1. 현재 R0-R3, R12, LR, PC, xPSR 레지스터를 스택에 광속으로 저장 (H/W)       │
 │  2. Vector Table 에서 고순위 ISR 주소 로딩                                   │
 │  3. CPU PC(Program Counter) 업데이트                                       │
 └────────────────────────────────────────────────────────────────────────────┘
```

### 심층 동작 원리

#### ① 벡터 테이블 (Vector Table)
메모리의 특정 구역에 각 인터럽트 번호와 그에 대응하는 실행 함수의 시작 주소를 순서대로 적어둔 표다. NVIC는 인터럽트 신호가 들어오면 이 표를 '하드웨어적'으로 참조하여 0순위로 CPU에 꽂아준다.

#### ② 테일 체이닝 (Tail-Chaining)
연속으로 인터럽트가 발생했을 때, 첫 번째 인터럽트가 끝나고 레지스터를 복원했다가 다시 저장하는 낭비를 하지 않는다. 하드웨어가 "어차피 또 인터럽트 할 거네?"라고 판단하여 복원/저장 과정을 생략하고 바로 다음 ISR로 점프한다. 지연 시간을 획기적으로 줄이는 비결이다.

#### ③ 후착 우선 처리 (Late Arrival)
저우선순위 인터럽트가 발생하여 레지스터를 막 스택에 저장하고 있는데, 더 높은 우선순위 인터럽트가 도착하면? 하드웨어는 현재 진행 중인 저장 작업을 그대로 활용하여 즉시 높은 순위 인터럽트 주소로 갈아탄다.

## Ⅲ. 융합 비교 및 다각도 분석

### 인터럽트 처리 방식의 진화

| 방식 | 동작 원리 | 성능 | 단점 |
|------|-----------|------|------|
| **Polling** | CPU가 수시로 장비들에 전화 걸어 확인 | 최악 | CPU가 일은 안 하고 전화만 돌림 |
| **Non-Vectored**| 장비가 벨을 누르면 CPU가 누구인지 물어봄 | 보통 | 확인하는 데 시간 소요 (지터 발생) |
| **Vectored (VIC)**| 벨소리에 발신 번호가 찍혀서 바로 응답 | 우수 | 긴급한 상황(중첩) 처리가 곤란 |
| **Nested (NVIC)**| 발신 번호 확인 + 긴급 상황 가로채기 | **최상** | 하드웨어 설계가 복잡해짐 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 산업용 인버터 제어 보드(Cortex-M4 기반). 모터 제어를 위한 고속 PWM 인터럽트(100kHz)와 일반 데이터 통신을 위한 UART 인터럽트가 공존한다. 가끔 통신 로그를 찍는 동안 모터가 튀는 현상(지연)이 발생한다.
**판단 및 NVIC 튜닝 전략**:
1. **Priority Grouping 설정**: NVIC의 `Priority Grouping` 레지스터를 설정하여 선점(Preemption) 우선순위와 하위 우선순위(Sub-priority)를 명확히 나눈다.
2. **선점 우선순위 부여**: 모터 제어 인터럽트에는 가장 높은 선점 순위(0)를, UART에는 낮은 순위(15)를 부여한다.
3. **결과**: UART ISR이 실행 중이더라도 모터 제어 신호가 오는 즉시 하드웨어가 UART를 중단시키고 모터를 제어한다. NVIC의 **Auto-stacking** 기능 덕분에 소프트웨어 부하 없이 $1\mu s$ 이내에 반응하여 모터 정밀도를 완벽히 확보한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **모든 인터럽트의 우선순위 동일화**: NVIC를 쓰면서도 귀찮다는 이유로 모든 우선순위를 기본값(0)으로 두면 안 된다. 이는 NVIC의 최대 무기인 '중첩' 기능을 포기하는 것이다. 시스템의 **'Safety-critical'** 순위에 따라 반드시 계층적으로 번호를 매겨야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **인터럽트 지연 시간(Latency)**: 수백 사이클에서 **12사이클 이내**로 단축 (Cortex-M 기준).
- **시스템 결정성(Determinism)**: 소프트웨어 조건문에 의존하지 않아 지연 시간이 항상 일정하게 유지됨.

### 미래 전망 및 진화 방향
최신의 NVIC는 **보안(TrustZone)** 기술과 결합하고 있다. 보안 영역에서 발생한 인터럽트와 비보안 영역의 인터럽트를 하드웨어적으로 분리하여, 해커가 인터럽트를 가로채 시스템을 장악하는 것을 원천 차단한다. 또한, 멀티코어 환경을 위해 **GIC (Generic Interrupt Controller)**로 확장되며 수만 개의 분산 인터럽트를 조율하는 거대 신경망의 중추로 발전하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **인터럽트 컨트롤러가 뭔가요?**: 학교 교무실에서 선생님(CPU)에게 걸려오는 수많은 전화(인터럽트)를 관리해 주는 '똑똑한 비서'예요.
2. **왜 벡터(Vectored) 방식인가요?**: 전화벨이 울릴 때 누군지 물어보지 않아도, 비서가 미리 "이건 양호실 전화니까 바로 받으세요!"라며 정답 페이지를 펴주기 때문이에요.
3. **가장 좋은 점은요?**: 비서는 아주 힘이 세서, 선생님이 심부름꾼 전화를 받고 있더라도 교장 선생님 전화가 오면 즉시 "잠시만요!" 하고 끊은 뒤 교장 선생님 전화부터 연결해 준답니다!
