# 577. 분기 목표 주입 (BTI, Branch Target Injection) - 스펙터 v2의 실체

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CPU의 성능 최적화 기법인 간접 분기 예측(Indirect Branch Prediction)을 악용하여, 공격자가 조작한 **악의적인 실행 주소를 분기 예측 유닛(BPU)에 주입**함으로써 CPU가 기밀 정보가 담긴 구역을 투기적으로 실행하게 만드는 하드웨어 보안 취약점이다.
> 2. **가치**: 스펙터(Spectre) 변종 2로 불리며, 소프트웨어의 논리적 장벽을 무시하고 하드웨어의 '미리 보기' 특성을 이용해 데이터를 훔쳐내는 현대 프로세서 설계의 구조적 결함을 적나라하게 보여준다.
> 3. **융합**: 투기적 실행(Speculative Execution), 사이드 채널 분석(Side-channel Analysis), 그리고 이를 방어하기 위한 **IBRS, Retpoline** 기술이 융합되어 현대 OS와 칩셋의 보안 패치 전쟁을 이끌고 있다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
분기 목표 주입(BTI)은 CPU가 '다음에 어디로 점프할지'를 결정하는 두뇌 조각을 속이는 공격이다. CPU는 점프할 곳이 결정되기도 전에 "아마 저번처럼 A로 가겠지?"라고 추측해서 A를 미리 실행하는데, 해커가 이 추측 시스템에 "B(기밀 구역)로 가!"라고 가짜 정보를 주입하는 것이 핵심이다.

### 💡 비유
BTI는 **"내비게이션의 최근 목적지 조작 사건"**과 같다.
- **정상**: 운전사(CPU)가 차에 타자마자 내비게이션(BPU)에 찍힌 '집(A)'으로 자동 출발한다.
- **BTI 공격**: 도둑(해커)이 몰래 내비게이션의 최근 목적지를 '도둑의 소굴(B)'로 바꿔치기한다. 운전사는 의심 없이 도둑의 소굴로 차를 몰고 가고, 거기서 귀중품(데이터)을 노출하게 된다. 운전사가 "어? 여기가 아니네" 하고 다시 집으로 돌아가도, 도둑은 이미 차 안의 귀중품을 훔쳐본 뒤다.

### 등장 배경 및 발전 과정

#### 1. 간접 분기(Indirect Branch)의 복잡성
`jmp eax`와 같이 실행 시점에 주소가 결정되는 간접 분기는 예측하기 매우 힘들다. CPU는 이를 위해 과거의 실행 기록을 저장하는 버퍼를 쓴다.

#### 2. 2018년 스펙터(Spectre)의 충격
구글 프로젝트 제로 연구원들은 서로 다른 프로세스가 이 '실행 기록 버퍼'를 공유한다는 사실을 발견했다. 즉, 나쁜 앱이 버퍼에 엉뚱한 주소를 가득 채워두면, 착한 앱이 그 주소로 점프하도록 유도할 수 있다는 치명적 결함이 세상에 드러났다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### BTI 공격의 3단계 매커니즘 (표)

| 단계 | 명칭 | 동작 상세 | 비유 |
|------|-----------|-----------|------|
| **1단계** | **Training** | 공격자가 특정 주소로 계속 점프하여 CPU의 예측 버퍼(BTB)를 오염시킴 | 도둑이 내비게이션에 가짜 주소를 여러 번 입력함 |
| **2단계** | **Injection** | 희생자 프로세스가 실행될 때, CPU가 오염된 버퍼를 보고 기밀 구역을 투기적 실행 | 운전사가 가짜 주소를 보고 도둑 소굴로 출발함 |
| **3단계** | **Exfiltration** | 투기적 실행 중 발생한 캐시의 변화를 사이드 채널(Flush+Reload)로 읽어냄 | 도둑이 차가 지나간 자국을 보고 짐의 내용을 추측함 |

### 정교한 구조 다이어그램 (BTB 오염 과정)

```ascii
================================================================================
[ Branch Target Injection (BTI): Poisoning the BTB ]
================================================================================

  ┌──────────────────────────────────────────────────────────┐
  │  🧠 CPU Branch Prediction Unit (BPU)                     │
  │  ┌────────────────────────────────────────────────────┐  │
  │  │ 📂 Branch Target Buffer (BTB) - 공유 자원           │  │
  │  │ [ Last Addr: 0x400 ] -> [ Next Addr: 0xDEAD ] 😈   │  │ ◀── (1) Poisoned by Attacker
  │  └────────────────────────────────────────────────────┘  │
  └──────────────────────────┬───────────────────────────────┘
                             │
  (2) Victim App runs...     │
  (3) Indirect JMP encountered!
                             │
  ┌──────────────────────────▼───────────────────────────────┐
  │  ⚡️ Speculative Execution Engine                         │
  │  - "According to BTB, I should go to 0xDEAD!"           │
  │  - Running code at 0xDEAD (Secret Data Region)          │ ◀── (4) LEAKING DATA
  └──────────────────────────┬───────────────────────────────┘
                             │
  (5) Reality Check:         │
  - "Wait, 0xDEAD is wrong. Rollback!"                      │
  - (But Cache state already changed...)                    │
```

### 심층 동작 원리

#### ① BTB (Branch Target Buffer) 공유
현대 CPU는 속도를 위해 여러 프로세스(혹은 VM)가 하나의 BTB를 같이 쓴다. 공격자는 자신의 주소 공간에서 특정 패턴으로 점프를 반복하여 BTB의 인덱스를 희생자의 인덱스와 겹치게 만든다.

#### ② 사이드 채널 공격과의 결합
투기적 실행 자체는 결과를 저장하지 않으므로 안전해 보이지만, **CPU 캐시(L1/L2)**에는 흔적을 남긴다. 공격자는 `Flush+Reload` 기술을 써서 캐시 응답 속도를 측정함으로써, 투기적 실행 중에 읽혔던 비밀 데이터(비밀번호 등)의 값을 0과 1 단위로 복원해 낸다.

## Ⅲ. 융합 비교 및 다각도 분석

### BTI(Spectre v2) vs Meltdown(v3) 비교

| 비교 지표 | BTI (Spectre v2) | Meltdown (v3) |
|-----------|------------------|---------------|
| **핵심 원인** | **분기 예측 오염** | 유저-커널 권한 체크 지연 |
| **공격 대상** | 다른 프로세스, 하이퍼바이저 | **운영체제 커널** |
| **난이도** | 높음 (정교한 훈련 필요) | 낮음 (매우 강력함) |
| **방어 기술** | **Retpoline, IBRS** | **KPTI (페이지 테이블 격리)** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 고성능 클라우드 서버 환경. 인텔 스카이레이크(Skylake) 이전의 구형 CPU를 사용 중인데, BTI 패치를 적용했더니 웹 서버 성능이 20%나 떨어져서 난리가 났다. 성능과 보안 사이에서 결단을 내려야 한다.
**판단 및 방어 최적화 전략**:
1. **Retpoline 적용**: 하드웨어 패치(IBRS)는 문맥 교환마다 BTB를 비워야 해서 너무 느리다. 구글이 개발한 소프트웨어 방식인 **Retpoline (Return Trampoline)**을 컴파일러 수준에서 적용하여 분기 예측 자체를 간접적으로 우회하게 한다.
2. **Microcode Update**: 최신 마이크로코드를 올려 **IBRS_ALL** 등 하드웨어가 지원하는 가장 효율적인 방어 모드를 활성화한다.
3. **VM 격리**: 보안이 극도로 중요한 워크로드는 다른 VM과 CPU 코어를 공유하지 않도록 **Core Pinning**과 **HT (Hyper-threading) 비활성화**를 검토한다.
4. **결과**: 성능 하락 폭을 20%에서 5% 이내로 줄이면서도, 부채널 공격으로부터 고객 데이터를 안전하게 보호한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **보안 무시한 '노패치'**: "우리는 내부망이라 안전해"라며 BTI 패치를 안 하는 것은 위험하다. 자바스크립트 한 줄만으로도 브라우저를 통해 BTI 공격(Spectre JS)이 가능하기 때문이다. 성능이 떨어지더라도 **커널 패치는 반드시 최신 상태**를 유지해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **보안 신뢰성**: 알려진 모든 스펙터 변종 공격에 대해 **99% 이상의 방어 성공률**.
- **안정성**: 가상화 환경(Multi-tenancy)에서의 테넌트 간 정보 유출 원천 차단.

### 미래 전망 및 진화 방향
CPU 제조사들은 이제 하드웨어 레벨에서 BTI를 막고 있다. 인텔의 **CET (Control-flow Enforcement Technology)**와 ARM의 **BTI (Branch Target Identification)** 기능은 실행 가능한 점프 주소에 '표식'을 남겨, 표식이 없는 곳으로의 점프는 아예 하드웨어가 거부하게 만든다. 미래의 CPU는 성능을 위해 보안을 희생하던 과거의 과오를 씻고, **'Secure-by-Design'**이 적용된 지능형 예측 아키텍처로 거듭날 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **BTI 공격이 뭔가요?**: 컴퓨터가 미래를 예상해서 미리 공부하는 버릇을 악당이 이용하는 거예요. 공부할 책 대신 만화책(기밀 데이터)을 살짝 끼워 넣는 거죠.
2. **왜 위험한가요?**: 컴퓨터는 공부하는 줄 알고 만화책을 펼쳐보는데, 그때 악당이 뒤에서 거울(캐시)로 그 내용을 훔쳐볼 수 있기 때문이에요.
3. **어떻게 막나요?**: 컴퓨터가 미래를 예상할 때마다 "진짜 이 책이 맞아?"라고 꼼꼼하게 확인하는 안전 안경(BTI 패치)을 씌워주는 거랍니다!
