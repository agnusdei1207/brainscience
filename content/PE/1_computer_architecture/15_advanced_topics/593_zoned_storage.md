# 593. 존 스토리지 (Zoned Storage / ZNS) - 데이터 배치의 질서 혁명

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기존의 무작위 쓰기가 가능한 블록 디바이스 추상화에서 벗어나, SSD나 HDD의 주소 공간을 여러 개의 독립된 **존(Zone)으로 나누고 각 존 내부에서는 반드시 순차적으로만 데이터를 쓰도록 강제**하는 차세대 스토리지 아키텍처이다.
> 2. **가치**: SSD 내부의 복잡한 가비지 컬렉션(GC) 부하를 제거하여 쓰기 증폭(WAF)을 1에 가깝게 낮춤으로써 수명을 비약적으로 늘리고, 대규모 클러스터에서 지연 시간의 일관성(Tail Latency 제거)을 확보한다.
> 3. **융합**: NVMe의 **ZNS (Zoned Namespaces)** 규격, SMR(기와식 자기 기록) HDD 기술, 그리고 호스트 레벨의 파일 시스템(F2FS 등)과 융합되어 하이퍼스케일 클라우드 인프라의 표준 저장 기술로 자리 잡았다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
존 스토리지(Zoned Storage)는 저장 장치의 주소 공간을 'Zone'이라는 거대한 덩어리로 관리하는 방식이다. 각 존은 오직 한 방향으로만 데이터를 쌓을 수 있으며(Sequential Write), 데이터를 지울 때는 존 전체를 한꺼번에 비워야 한다.

### 💡 비유
존 스토리지는 **"노트장에 연필로 글을 쓰는 규칙"**과 같다.
- **일반 스토리지 (무작위 쓰기)**: 노트장 아무 페이지나 펼쳐서 중간에 글을 끼워 넣고, 지우개로 한 글자만 지우고 다시 쓸 수 있다. (편하지만 종이가 금방 너덜너덜해짐 - SSD 수명 단축).
- **존 스토리지**: 노트장의 각 장(Zone)은 반드시 첫 줄부터 순서대로만 써야 한다. 중간을 지울 수 없다. 내용을 고치려면 그 장 전체를 찢어버리고(Reset) 새 장에 처음부터 다시 써야 한다. (불편해 보이지만 종이가 깨끗하게 유지되고 관리가 완벽함).

### 등장 배경 및 발전 과정

#### 1. 낸드 플래시의 물리적 특성
낸드는 읽기/쓰기는 페이지(Page) 단위지만, 삭제는 블록(Block) 단위로만 가능하다. 이 불일치 때문에 SSD는 내부적으로 데이터를 이리저리 옮기는 가비지 컬렉션(GC)을 수행하며 수명과 성능을 깎아 먹는다.

#### 2. 하이퍼스케일러의 요구
구글이나 아마존 같은 기업들은 수백만 개의 SSD를 쓴다. SSD마다 제각각 GC를 하느라 성능이 널뛰는 것을 참을 수 없게 되었고, 아예 "우리가 순서대로 쓸 테니 SSD는 복잡한 짓(FTL) 하지 마라"라고 선언하며 ZNS 규격이 탄생했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 존 스토리지의 주요 상태 및 제어 (표)

각 존은 하드웨어적으로 다음 상태를 유지하며 관리된다.

| 존 상태 (State) | 상세 설명 | 비유 |
|-----------------|-----------|------|
| **Empty** | 데이터가 하나도 없는 깨끗한 상태 | 빈 일기장 |
| **Open** | 데이터를 쓰고 있는 도중인 상태 (Write Pointer 활성) | 지금 글을 적고 있는 페이지 |
| **Full** | 존의 용량이 꽉 차서 더 이상 쓸 수 없는 상태 | 마지막 줄까지 꽉 찬 페이지 |
| **Reset** | 존의 데이터를 모두 지우고 다시 Empty로 만드는 명령 | 일기장을 찢고 새로 시작하기 |

### 정교한 구조 다이어그램 (ZNS vs Conventional SSD)

```ascii
================================================================================
[ Storage Architecture Evolution: From Random to Zoned ]
================================================================================

 [ 1. Conventional SSD (Black-box) ]
  Host LBA: [ 0 ][ 1 ][ 2 ][ 3 ][ 4 ]
               │ (Complex FTL Mapping)
               ▼
  NAND Flash: [ 2 ][ X ][ 0 ][ 1 ][ 4 ] (Fragmented & GC Required)

 [ 2. Zoned Storage (ZNS SSD) ]
  Host Logic: "Sequential Only!"
               │
  ┌────────────▼────────────┐  ┌────────────▼────────────┐
  │  ⭐️ Zone 0 (Active)     │  │  ⭐️ Zone 1 (Empty)      │
  │  [ Data 0 ][ Data 1 ][  ]│  │  [        Empty       ]│
  │             ▲            │  │  ▲                     │
  └─────────────┼────────────┘  └──┼─────────────────────┘
          Write Pointer          Write Pointer (at start)

 * 핵심 원리: 하드웨어가 관리하던 복잡한 주소 변환(L2P)을 없애고, 
   호스트가 데이터의 물리적 순서를 100% 보장함.
```

### 심층 동작 원리

#### ① 쓰기 포인터 (Write Pointer)
각 존에는 데이터를 쓸 위치를 가리키는 '쓰기 포인터'가 있다. 데이터가 써질 때마다 포인터는 옆으로 한 칸씩만 이동한다. 이전 위치로 돌아가서 덮어쓰는 것은 하드웨어 레벨에서 거부된다.

#### ② 가비지 컬렉션의 제거 (No Garbage Collection)
일반 SSD는 데이터를 덮어쓸 때 기존 데이터를 다른 곳으로 옮겨야 하므로 GC가 발생한다. 존 스토리지는 덮어쓰기 자체가 불가능하므로 SSD 내부에서 데이터를 옮길 일이 없다. 이로 인해 '쓰기 증폭(WAF)'이 이론적 최솟값인 1에 수렴하게 된다.

#### ③ 오버 프로비저닝 (Over-provisioning) 절감
일반 SSD는 GC를 위해 실제 용량의 7~20%를 예비 공간으로 숨겨둔다. 존 스토리지는 GC가 필요 없으므로 이 예비 공간을 모두 사용자에게 개방할 수 있어, 동일 하드웨어에서 가용 용량이 20%가량 늘어난다.

## Ⅲ. 융합 비교 및 다각도 분석

### 일반 SSD vs ZNS SSD 성능 비교

| 비교 지표 | Conventional SSD | **Zoned Storage (ZNS)** |
|-----------|-------------------|--------------------------|
| **지연 시간 (Latency)**| 가변적 (GC 시 폭증) | **일정함 (Deterministic)** |
| **수명 (Lifespan)** | 보통 | **최상 (WAF 1.0 달성)** |
| **가용 용량** | 보통 (예비 공간 제외) | **높음 (예비 공간 불필요)** |
| **구현 난이도** | 낮음 (기존 앱 호환) | **높음 (전용 파일시스템 필요)** |
| **비유** | 자유로운 낙서장 | **정갈한 원고지** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대규모 로그 수집 플랫폼을 운영 중이다. 초당 수십만 개의 로그가 쏟아지는데, 특정 시간마다 SSD의 응답 속도가 100ms를 넘어가면서 로그 유실이 발생한다.
**판단 및 존 스토리지 도입 전략**:
1. **ZNS SSD 도입**: 기존 NVMe SSD를 ZNS 지원 모델로 교체한다.
2. **Log-structured FS 연동**: 리눅스의 **F2FS**나 **RocksDB (ZNS backend)**를 사용하여, 애플리케이션이 데이터를 존 단위로 순차적으로 쏘게 만든다.
3. **Zone Mapping**: 로그 데이터의 '중요도'나 '만료 시간'에 따라 존을 나눈다. (예: 1시간 뒤 지울 로그는 Zone A, 1년 보관할 로그는 Zone B).
4. **결과**: SSD 내부 GC가 사라져 Tail Latency가 10ms 이내로 고정되며, 로그 유실 Zero를 달성하고 SSD 교체 주기를 2배 이상 연장한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **기존 앱의 단순 이식**: 일반적인 관계형 DB(MySQL 등)를 아무 설정 없이 존 스토리지에 올리면 작동하지 않는다. DB 엔진이 중간에 데이터를 수정하려 하기 때문이다. 반드시 **ZNS 전용 엔진**이나 **추상화 라이브러리(libzbc)**를 거쳐야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **TCO (총소유비용) 절감**: SSD 수명 연장 및 가용 용량 증가로 **인프라 비용 30% 이상 절감**.
- **서비스 가용성**: 99.99% 지연 시간 지표 안정화로 고품질 실시간 서비스 제공 가능.

### 미래 전망 및 진화 방향
존 스토리지는 이제 **'컴퓨테이셔널 스토리지'**와 결합하고 있다. 존 내부의 데이터를 CPU로 가져오지 않고 SSD 안에서 직접 검색하거나 압축하는 방식이다. 또한, 차세대 고속 인터페이스인 **CXL (Compute Express Link)** 환경에서 수만 개의 SSD를 하나의 거대 자원으로 관리하는 **'Fabrics-based Zoned Storage'**가 클라우드 데이터센터의 미래가 될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **존 스토리지가 뭔가요?**: 장난감 상자(SSD)를 여러 칸으로 나누고, 장난감을 넣을 때 무조건 밑바닥부터 차곡차곡 쌓아야 하는 규칙이에요.
2. **왜 그렇게 하나요?**: 장난감을 아무렇게나 던져넣으면 나중에 정리하느라 시간이 오래 걸리고 상자도 망가지지만, 순서대로 넣으면 정리할 필요가 없어서 훨씬 빠르고 상자도 오래 쓰기 때문이에요.
3. **가장 좋은 점은요?**: 내가 장난감을 꺼내고 싶을 때 상자가 "청소 중이야, 기다려!"라고 방해하는 일이 없어서 언제든 바로 놀 수 있답니다!
