# 107. 패리티 비트 (Parity Bit)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 컴퓨터가 네트워크 선이나 메모리 버스로 S/W 데이터(예: 7비트 ASCII)를 쏠 때, 중간에 벼락이나 노이즈를 맞아 비트가 0에서 1로 뒤집히는 S/W 파단 현상을 감시하기 위해, **맨 끝에 1가닥의 감시용 비트(1비트)를 덧붙여서 보내는 가장 단순한 오류 탐지 기법**입니다.
> 2. **가치**: 복잡한 수학 연산 회로가 필요 없이, 아주 싸구려 **XOR 게이트(배타적 논리합)** 몇 개만 납땜해 두면 0클럭에 가까운 지연(Latency)으로 전송 무결성 록(Lock)을 검증할 수 있어 옛날 모뎀 통신이나 현재의 저속 시리얼 통신(UART/RS-232)에서 절대적으로 군림합니다.
> 3. **한계 (파국 지옥)**: 단 1개의 비트가 깨진 것은 귀신같이 잡아내지만(Error Detection), 그게 몇 번째 비트인지 알 수 없어 **스스로 고칠 수 없고(No Correction)**, 만약 억세게 운이 나빠 **2개의 비트가 동시에 깨져버리면 에러가 없다고 S/W 통과 빔을 쏴버리는 치명적인 장님 버그** 구조를 가집니다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 통신 채널에서 데이터의 무결성을 확인하기 위해 덧붙이는 검사 비트. 데이터 비트 묶음(보통 1바이트) 안에 있는 '1'의 총개수를 짝수나 홀수로 맞추기 위해 끼워 넣는 깍두기 비트입니다.
- **💡 비유**: 박스에 사과 7개를 담아 택배를 보낼 때, 도둑이 중간에 사과를 훔쳐갈까 봐 **"내가 보낸 박스의 사과 개수는 무조건 짝수 개야!"** 라고 약속하는 겁니다. 만약 7개를 넣었으면 빈자리(패리티 공간)에 사과 1개를 더 채워 8개(짝수)로 맞춰 보냅니다. 받는 사람이 뜯어봤는데 사과가 7개(홀수)면 "아! 배달 사고다!" 라며 반송(재요청) 때리는 S/W 통달 시스템입니다.
- **등장 배경 및 발전 과정**:
  1. 노이즈 지옥: 초기 천공 카드(Punched Card)나 아날로그 구리선 통신 시절, 먼지 하나만 끼거나 정전기가 살짝 튀어도 0이 1로 읽혀 데이터가 끔찍하게 꼬였습니다.
  2. 아키텍트의 최소 방어막: "데이터가 망가졌는지 최소한 눈치라도 채게 만들자! 비트 하나만 더 붙여서 S/W 체킹 망을 짜!" $\rightarrow$ 패리티 비트의 성배 탄생.
  3. 진화와 패배: 속도가 빨라지면서 한 번에 2비트 이상이 동시에 깨지는 버스트 에러(Burst Error)가 빈발하자, 이 원시적인 패리티 비트는 한계를 맞고 CRC(순환 중복 검사)나 ECC(오류 정정 코드)라는 괴물 아키텍처에 자리를 물려주며 하위 호환 망으로 밀려납니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

하드웨어 XOR 회로가 패리티 비트를 굽고 렌더링하는 S/W 물리 파이프라인.

#### 1. 짝수 패리티 (Even Parity) vs 홀수 패리티 (Odd Parity)
패리티는 양쪽 컴퓨터가 사전에 합의한 록(Lock) 규칙에 따라 나뉩니다.
- **짝수 패리티 (Even Parity)**: 데이터 안의 '1'의 개수 + 패리티 비트의 '1'까지 합친 총 '1'의 개수를 무조건 **짝수(Even)** 로 맞춤.
- **홀수 패리티 (Odd Parity)**: 합친 총 '1'의 개수를 무조건 **홀수(Odd)** 로 맞춤.

*예제 시뮬레이션 (짝수 패리티 기준)*:
- 전송할 데이터가 `1 0 1 1 0 0 1` (1이 4개임. 이미 짝수)
  $\rightarrow$ 패리티 비트는 굳이 1을 보탤 필요 없이 **`0`** 을 달아줍니다. (송신 비트 덩어리: `1 0 1 1 0 0 1` + `0`)
- 전송할 데이터가 `1 0 1 1 0 0 0` (1이 3개임. 홀수라 에러 남!)
  $\rightarrow$ 짝수로 강제 렌더링하기 위해 패리티 비트에 **`1`** 을 쑤셔 넣습니다. (송신 비트 덩어리: `1 0 1 1 0 0 0` + `1`)

#### 2. 하드웨어 게이트의 통달망 (XOR Tree)
"이 비트들 안의 1의 개수가 짝수냐 홀수냐"를 더하기 연산기(Adder)로 세는 것은 S/W CPU 낭비입니다. H/W 엔지니어는 이를 **XOR 게이트 수풀(Tree)** 하나로 압살시켜 버립니다.
- XOR 게이트는 2개의 입력이 다르면 1, 같으면 0을 뱉습니다. 여러 비트를 계속 XOR로 엮어나가면, 궁극적으로 **"1의 개수가 홀수 개이면 1, 짝수 개이면 0"** 이 튀어나오는 기적의 회로 S/W가 0.1 나노초 만에 렌더링 됩니다.

#### 3. 장님 버그 데드락 (2-Bit 에러의 비극)
- 전송 중 `1011(홀수 패리티)` 을 보냈습니다. (1이 3개)
- 번개가 쳐서 선 1개가 깨져 `1111` 로 들어왔습니다. 수신 측 XOR 회로는 "어라? 1이 4개(짝수)네? 홀수여야 하는데! 버려!" 라며 완벽히 차단 록(Lock)을 겁니다.
- **파단 늪**: 만약 벼락을 세게 맞아 **비트 2개가 동시에 뒤집혀서** `1110` 이 되어 들어오면? "어? 1이 3개(홀수)네? 이상 없음! 통과 빔!" 이라며 **치명적인 오염 데이터를 정상 데이터로 S/W 통과시켜버리는 최악의 보안 결함**을 발생시킵니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 에러 탐지/정정 아키텍처 폼팩터 결착
| 시스템 규격 | **패리티 비트 (Parity Bit)** | **체크섬 (Checksum)** | **CRC (순환 중복 검사)** | **해밍 코드 / ECC** |
|---|---|---|---|---|
| **S/W H/W 연산 부하** | 극단적으로 낮음 (XOR 게이트 몇 개) | 낮음 (단순 덧셈 캐리 버림) | 보통 (다항식 나눗셈 H/W 엔진 필요) | 높음 (다중 패리티 교차 행렬 검증) |
| **탐지율 S/W 파이프라인** | 단일 비트 100%, 2비트는 **탐지 불가 지옥** | 패킷의 덩어리 검증 (순서 꼬임 등 일부 못 잡음) | 연속된 버스트 에러(Burst Error) **99.99% 압살 탐지** | 모든 단일 비트 탐지 및 **스스로 고침 보증** |
| **에러 정정(Correction) 여부** | **불가능** (오류 나면 그냥 재전송 요청) | **불가능** (TCP/IP 재요청 록) | **불가능** (재전송 ARQ 록) | **S/W 자가 치유(복구) 100% 가능 압살!** |
| **주요 활약 생태계** | RS-232 직렬 포트, 구형 메모리 칩 | TCP/IP IP 헤더 | 이더넷(Ethernet) 랜카드 MAC S/W | 엔터프라이즈 서버 ECC RAM |

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 시스템 엔지니어의 IoT 펌웨어 세팅 록 (Baud Rate & Parity)
- **시나리오**: 공장 자동화 HMI 화면과 센서 컨트롤러 간에 RS-232 시리얼 통신 선을 깔았습니다. 
- **아키텍트 튜닝 빔**: S/W 포트 세팅 값에 `9600, 8, N, 1` 이라고 적힌 것을 많이 봅니다. 여기서 **`N`** 이 바로 `None Parity` (패리티 안 씀)의 약자입니다. 만약 공장에 전기 모터 노이즈가 너무 심해 데이터가 자꾸 깨진다면, 아키텍트는 펌웨어를 뜯어고쳐 세팅을 **`9600, 8, E, 1` (Even Parity 사용 강제 록)** 이나 **`O` (Odd Parity)** 로 용접해야 합니다. 통신 대역폭은 1비트 더 손해 보지만(Overhead), 화면에 쓰레기 센서 값이 출력되어 모터가 폭주하는 시스템 파국은 미연에 컷(Cut) 할 수 있습니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

- **결론**: 패리티 비트는 한 번에 비트 2개가 깨지는 현상 앞에서는 무기력하게 무너지는 허술한 보안관입니다. 하지만 컴퓨터 아키텍처의 여명기에 덧셈기(ALU)를 태우지 않고 오직 XOR 게이트의 대칭성 S/W만으로 물리적 채널의 무결성을 사수한 가장 우아하고 경제적인 S/W 폼팩터입니다. 이 1차원 패리티 비트 사상을 수십 개로 교차시켜 짜 올린 것이 바로 현대 서버 RAM을 수호하는 해밍 코드(ECC)의 본체 S/W 이기도 합니다.
