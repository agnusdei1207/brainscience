# 27. 메모리 맵 파일 (Memory Mapped File / mmap)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 디스크에 저장된 파일을 일반적인 `Read/Write` 시스템 콜을 거치지 않고, **프로세스의 가상 주소 공간(Virtual Address Space)의 일부로 직접 연결(Mapping)하여 메모리 읽기/쓰기 명령으로 파일을 다루는 기술**입니다.
> 2. **가치 (I/O의 광속화)**: 유저 모드와 커널 모드 사이의 **데이터 복사(Context Switch)를 최소화**하며, 운영체제의 페이지 캐시(Page Cache)를 직접 이용함으로써 파일 입출력 성능을 극한으로 압살합니다.
> 3. **융합**: 여러 프로세스가 동일한 파일을 메모리에 맵핑하여 데이터를 주고받는 **공유 메모리(Shared Memory)** 기술의 근간이며, 대규모 데이터베이스와 고성능 로그 시스템의 물리적 저장 엔진으로 융합되어 사용됩니다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 파일을 메모리의 배열(Array)처럼 취급하는 방식입니다. `mmap()` 이라는 시스템 호출을 통해 구현됩니다.
- **💡 비유**: **'원격 제어 로봇 팔'** 과 같습니다. 
  - **일반 I/O**: 창고(디스크)에 있는 물건을 택배(버퍼 복사)로 받아서 집에서 쓰는 방식. 
  - **메모리 맵**: 우리 집 벽에 구멍을 뚫어 창고 안이 바로 보이게 하고, 집 안에서 손만 뻗어(메모리 주소 접근) 창고 물건을 바로 만지는 방식입니다.
- **등장 배경**:
  1. 시스템 콜의 무거움: 대용량 파일을 1바이트씩 `read()` 로 읽는 것은 너무 느리고 CPU 소모가 컸습니다. 
  2. 가상 메모리의 유연성: "어차피 가상 주소는 가짜인데, RAM 말고 디스크 주소를 연결하면 안 돼?"라는 발상의 전환이 mmap을 낳았습니다.

---

### II. 아키텍처 및 핵심 원리 (Deep Dive)

메모리 맵 파일이 어떻게 OS의 페이징 메커니즘을 훔쳐 쓰는지 해부합니다.

#### 1. 요구 페이징과의 결합
- `mmap` 을 호출한다고 파일이 즉시 램으로 다 올라오지 않습니다. 
- CPU가 맵핑된 주소를 처음 읽는 순간 **페이지 폴트(Page Fault)** 가 발생합니다. 
- 운영체제는 이때서야 디스크의 파일 내용을 해당 페이지 프레임에 로드합니다. (Lazy Loading ROI).

#### 2. 복사 없는 전송 (Zero-copy)
- 일반 I/O: [디스크 $\rightarrow$ 커널 버퍼] $\rightarrow$ [유저 버퍼]. (복사 2번).
- **mmap**: [디스크 $\rightarrow$ 유저 주소와 연결된 커널 페이지 캐시]. (복사 1번).
- **결과**: 메모리 대역폭 낭비를 절반으로 줄여 대용량 처리를 압살합니다.

#### 3. 동기화 (Sync)
- 메모리에서 수정한 내용은 운영체제가 나중에(또는 명시적으로 `msync` 호출 시) 디스크에 Write-back 합니다.

---

### III. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 일반 파일 I/O vs 메모리 맵 I/O 결착망
| 항목 | **일반 I/O (Read/Write)** | **메모리 맵 I/O (mmap)** |
|---|---|---|
| **접근 방식** | 시스템 콜 함수 호출 | **포인터 및 배열 접근** |
| **복사 횟수** | 2회 이상 | **1회 (Zero-copy)** |
| **메모리 사용** | 버퍼 공간 필요 | **가상 주소 공간 활용** |
| **적합한 상황** | 작은 파일, 일회성 읽기 | **거대 파일, 빈번한 랜덤 접근** |
| **비유** | 심부름 시키기 | **직접 가서 만지기** |

---

### IV. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 시스템 아키텍트의 '대용량 DB' 엔진 설계
- **시나리오**: 테라바이트 급의 인덱스 파일을 가진 검색 엔진을 만듭니다. 검색 요청이 초당 수만 건 들어옵니다. 
- **아키텍트 분석**: "매번 `read()` 로 인덱스를 뒤지면 커널 모드 전환 오버헤드 때문에 서버가 타버린다. 전체 인덱스를 **mmap** 으로 맵핑해야 한다."
- **전략적 조치 (Efficient Indexing)**: 
  - 인덱스 파일 전체를 가상 주소 공간에 맵핑합니다. 
  - 운영체제의 **페이지 교체 알고리즘(LRU 등)** 이 자주 쓰이는 인덱스 조각만 자동으로 램에 유지하게 유도합니다. 
  - 이를 통해 별도의 캐시 로직을 짜지 않고도 OS의 지능을 빌려 파일 검색 성능을 100배 부스트 시키는 극한의 최적화를 달성합니다.

---

### V. 기대효과 및 결론 (Future & Standard)

- **결론**: 메모리 맵 파일은 "저장 장치와 기억 장치의 경계를 허문" 기술입니다. 가상 메모리 아키텍처가 줄 수 있는 가장 강력한 선물 중 하나입니다. 앞으로의 아키텍처는 램만큼 빠른 **비휘발성 메모리(NVMe, NVDIMM)** 기술과 결합하여, 모든 파일을 애초에 메모리처럼 다루는 **'단일 수준 저장소(Single-level Store)'** 시대로 진화하며 I/O 병목이라는 단어 자체를 역사 속으로 밀어낼 것입니다.
