# 7. 페이지 테이블 (Page Table)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 각 프로세스마다 독립적으로 존재하며, **가상 메모리의 '페이지(Page)'가 실제 RAM의 어느 '프레임(Frame)'에 위치하는지**를 1:1로 기록해둔 주소 변환용 데이터베이스입니다.
> 2. **가치 (연결의 고리)**: 이 테이블 덕분에 CPU는 연속적인 가상 공간을 보며 작업하고, 하드웨어는 파편화된 물리 메모리를 조립하여 실행하는 **'불연속 할당'** 의 기적을 실현합니다.
> 3. **융합**: 하드웨어인 **MMU** 에 의해 참조되며, 운영체제(OS) 커널에 의해 관리되는 **하드웨어-소프트웨어의 대표적인 협업 접점**입니다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 가상 주소를 물리 주소로 변환하기 위한 지도입니다. 메모리의 특정 영역에 상주하며, 프로세스가 생성될 때 함께 생성됩니다.
- **💡 비유**: **'호텔 투숙객 명부'** 와 같습니다. 
  - 가상 주소(손님 이름)가 들어오면, 명부(페이지 테이블)를 확인하여 "301호(물리 주소)"라고 안내해 주는 것과 같습니다. 
  - 손님은 자기 방 번호만 알면 되지만, 지배인(MMU)은 이 명부를 보고 실제 방으로 안내합니다.
- **등장 배경**:
  1. 페이징의 필수 조건: 데이터를 조각냈으니, 어디에 뒀는지 기억할 장부가 필요했습니다. 
  2. 동적 매핑: 프로그램 실행 중에 메모리 위치가 바뀌어도 테이블만 고치면 되므로 유연성이 비약적으로 상승했습니다.

---

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

페이지 테이블의 내부 구조와 주소 번역 과정을 해부합니다.

#### 1. 페이지 테이블 엔트리 (PTE, Page Table Entry)
테이블의 각 행(Row)은 다음과 같은 정보의 집합입니다.
- **Frame Number**: 실제 물리 메모리의 블록 번호. (가장 핵심).
- **Present/Absent Bit (V)**: 현재 이 페이지가 RAM에 있는지(1), 디스크에 있는지(0) 표시.
- **Read/Write Bit**: 읽기 전용인지 수정 가능한지 권한 표시.
- **Reference Bit**: 최근에 사용되었는지 표시. (페이지 교체 시 활용).
- **Dirty Bit**: 내용이 수정되었는지 표시. (디스크 기록 여부 결정).

#### 2. 주소 변환 파이프라인
1. 가상 주소에서 **페이지 번호(p)** 를 인덱스로 추출합니다.
2. 페이지 테이블의 **p번째 엔트리**를 읽습니다.
3. 거기 적힌 **프레임 번호(f)** 를 가져옵니다.
4. **결과**: $[f | \text{offset}]$ 이라는 진짜 주소가 완성됩니다.

#### 3. 저장 위치와 병목
- 페이지 테이블은 용량이 커서 **주기억장치(RAM)** 에 저장됩니다. 
- **비극**: 데이터 하나를 읽으려고 (1) 테이블 읽기, (2) 실제 데이터 읽기 등 **메모리를 두 번 방문**해야 합니다. 
- **해결**: 이 병목을 해결하기 위해 MMU 내부에 **TLB** 라는 초고속 캐시를 둡니다.

---

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 페이지 테이블 관리 방식 ROI
| 방식 | 특징 | 장점 | 단점 |
|---|---|---|---|
| **단일 레벨** | 통째로 하나로 관리 | 단순함, 빠른 검색 | **메모리 낭비 심각** (안 쓰는 영역도 생성) |
| **다단계 (Multi-level)** | 테이블을 쪼개서 계층화 | **메모리 절약 극대화** | 검색 속도가 느려짐 (여러 번 참조) |
| **역 페이지 (Inverted)** | 물리 메모리 기준 관리 | 전체 시스템 메모리 절약 | 특정 프로세스 주소 찾기 힘듦 |

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 시스템 아키텍트의 '페이지 테이블 오버헤드' 관리
- **시나리오**: 64비트 시스템에서 16GB RAM을 사용하는데, 페이지 테이블 자체가 너무 많은 메모리를 잡아먹습니다. 
- **아키텍트 분석**: "64비트 주소 공간을 단일 레벨로 관리하면 테이블 크기만 수십 PB가 된다. 배보다 배꼽이 더 크다."
- **전략적 조치 (Sparse Mapping)**: 
  - **다단계 페이지 테이블(Multi-level Page Table)** 을 도입합니다. 
  - 실제 사용 중인 주소 영역에 대해서만 하위 테이블을 생성하는 '희소 맵핑' 전략을 취합니다. 
  - 이를 통해 64비트의 광활한 가상 세계를 유지하면서도, 실제 테이블 용량은 수 MB 수준으로 압살하는 경제적 ROI를 달성합니다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

- **결론**: 페이지 테이블은 컴퓨터가 부리는 **'논리적 마술의 대본'** 입니다. 이 대본이 있기에 하드웨어는 파편화된 현실 위에서 소프트웨어의 연속된 꿈을 실현할 수 있습니다. 앞으로의 아키텍처는 단순히 주소를 연결하는 것을 넘어, 데이터의 보안 속성까지 실시간으로 매핑하는 **'인텔리전트 메타-테이블'** 구조로 진화하여 메모리 관리의 새로운 패러다임을 열 것입니다.
