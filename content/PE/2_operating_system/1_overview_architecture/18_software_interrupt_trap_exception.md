# 소프트웨어 인터럽트 / 트랩 (Trap) / 예외 (Exception)

## Ⅰ. 개요

### 1. 소프트웨어 인터럽트 및 동기적 인터럽트의 정의
소프트웨어 인터럽트(Software Interrupt)는 외부 하드웨어 장치가 아닌, **CPU 내부에서 명령어를 실행하는 과정에서 발생하는 인터럽트**를 총칭합니다. 이는 현재 실행 중인 명령어의 흐름이나 결과에 의해 발생하므로 프로그램의 실행과 동기화되어 일어나는 **동기적(Synchronous) 특성**을 가집니다.

### 2. 주요 분류: 트랩, 예외, 폴트, 어보트
소프트웨어 인터럽트는 그 발생 목적과 복구 가능성에 따라 크게 다음과 같이 분류됩니다.
* **Trap (트랩)**: 프로그램이 의도적으로 커널의 서비스를 요청하기 위해 발생시키는 소프트웨어 인터럽트 (예: System Call).
* **Exception (예외)**: 프로그램 실행 중 의도치 않은 오류나 예외 상황에 의해 발생하는 인터럽트. 내부적으로 Fault와 Abort로 나뉩니다.

---

## Ⅱ. 아키텍처 및 핵심 원리

### 1. 트랩 및 예외 처리 아키텍처 다이어그램

아래 다이어그램은 유저 모드에서 실행 중인 프로세스가 소프트웨어 인터럽트(트랩)를 통해 커널 모드로 진입하고, 서비스를 제공받거나 예외를 처리하는 흐름을 보여줍니다.

```text
       [ User Space ]                         [ Kernel Space ]
       
+-------------------------+              +-------------------------+
| User Process            |              | OS Kernel               |
|                         |              |                         |
|  instruction 1          |              |  System Call Handler    |
|  instruction 2          |   Trap       |  +-------------------+  |
|  int 0x80 (System Call) |------------->|  | 1. Save Context   |  |
|                         |              |  | 2. Check Syscall# |  |
|                         |              |  | 3. Execute Routine|  |
|                         |              |  +-------------------+  |
|  instruction 3 (Resume) |<-------------|          |              |
|                         |   IRET/Sysret|          v              |
+-------------------------+              |  Exception Handler      |
                                         |  +-------------------+  |
+-------------------------+   Exception  |  | - Page Fault      |  |
| User Process (Error)    |   (Divide    |  | - Divide by zero  |  |
|  a = 10 / 0;            |-- by zero) ->|  | - Invalid Opcode  |  |
|  // Crash or Signal     |<-------------|  +-------------------+  |
+-------------------------+   SIGFPE     +-------------------------+
```

### 2. 상세 메커니즘
1. **발생원(Trigger)**:
   * **명령어 기반 (Trap)**: `INT`, `SYSCALL`, `SYSENTER` 등의 명령어를 명시적으로 호출.
   * **오류 기반 (Exception)**: ALU의 연산 결과 오류(0으로 나누기 등) 또는 MMU의 메모리 접근 위반(Page Fault 등).
2. **모드 전환(Mode Switch)**: 유저 모드(Ring 3)에서 커널 모드(Ring 0)로 CPU 권한(Privilege level)이 상승합니다.
3. **처리 및 복구**:
   * **Trap**: 커널이 요청된 서비스(I/O, 프로세스 제어 등)를 수행하고 다음 명령어로 정상 복귀.
   * **Fault**: 문제를 해결(예: 디스크에서 메모리로 페이지 적재)한 후, **오류를 발생시킨 명령어부터 재실행**.
   * **Abort**: 복구할 수 없는 치명적인 하드웨어/시스템 오류로 프로세스를 강제 종료(Kill).

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 트랩(Trap) vs 폴트(Fault) vs 어보트(Abort)

| 구분 | 발생 원인 | 의도성 | 복귀 시점(Return Address) | 주요 사례 |
| :--- | :--- | :--- | :--- | :--- |
| **Trap** | 명시적 소프트웨어 명령어 | 의도적 (Intentional) | 트랩을 발생시킨 **다음 명령어** | 시스템 콜 (System Call), 디버깅 중단점 (Breakpoint) |
| **Fault** | 명령어 실행 중 복구 가능한 오류 | 비의도적 (Unintentional) | 폴트를 발생시킨 **해당 명령어 (재실행)** | 페이지 폴트 (Page Fault), 세그멘테이션 폴트 |
| **Abort** | 하드웨어 고장, 복구 불가능 오류 | 비의도적, 치명적 | 복귀하지 않음 (프로세스 종료) | 머신 체크 익셉션, 메모리 패리티 에러 |

### 2. 하드웨어 인터럽트와의 비교
하드웨어 인터럽트는 CPU 클럭 및 명령어 사이클과 무관하게(Asynchronous) 외부 핀(Pin)을 통해 인가되는 반면, 소프트웨어 인터럽트/예외는 철저하게 명령어 파이프라인 내부에서 발생하며(Synchronous) 명령어의 연산 결과에 종속된다는 핵심적인 차이가 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 시스템 콜(System Call)을 통한 시스템 보호
응용 프로그램이 하드웨어나 중요 파일 시스템에 직접 접근하는 것을 막기 위해 최신 운영체제는 이중 모드(Dual-mode)를 채택합니다. 응용 프로그램은 반드시 '트랩(Trap)'을 통해 커널에 서비스를 요청해야 하며, 커널은 인자의 유효성과 접근 권한을 철저히 검증한 후 명령을 대행합니다. 이는 악의적인 프로세스로부터 시스템 전체를 보호하는 핵심 보안 메커니즘입니다.

### 2. 요구 페이징(Demand Paging)과 페이지 폴트(Page Fault)
실무의 가상 메모리 관리에서 페이지 폴트는 오류라기보다는 정상적인 메커니즘의 일부입니다. CPU가 유효하지 않은 페이지에 접근 시 '폴트' 예외가 발생하고, 커널의 Page Fault Handler가 스왑 영역에서 메모리로 데이터를 가져온 후, 예외가 발생했던 명령어부터 재실행(Restart)시킵니다. 시스템 엔지니어는 이 Page Fault 발생 비율을 모니터링하여 시스템의 메모리 병목 현상(Thrashing)을 진단해야 합니다.

## Ⅴ. 기대효과 및 결론

소프트웨어 인터럽트(트랩 및 예외)는 단순한 오류 처리 메커니즘을 넘어, **운영체제와 사용자 응용 프로그램 간의 안전하고 규격화된 통신 창구(Interface)** 역할을 합니다. 트랩을 통한 시스템 콜은 커널의 자원 보호와 다중 프로그래밍의 안정성을 보장하며, 페이지 폴트와 같은 예외 처리는 가상 메모리라는 환상을 사용자에게 끊김 없이 제공합니다. 결론적으로, 이 동기적 인터럽트 체계는 현대 운영체제의 견고함(Robustness)과 안정성을 지탱하는 가장 중요한 아키텍처적 기반이라고 평가할 수 있습니다.
