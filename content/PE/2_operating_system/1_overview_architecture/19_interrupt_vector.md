# 인터럽트 벡터 (Interrupt Vector)

## Ⅰ. 개요

### 1. 인터럽트 벡터의 정의
인터럽트 벡터(Interrupt Vector)는 시스템에 다양한 인터럽트가 발생했을 때, 운영체제가 해당 인터럽트를 처리할 수 있는 **인터럽트 서비스 루틴(ISR)의 메모리 시작 주소를 찾기 위한 고유한 식별 번호(Index) 또는 주소 포인터**입니다. 

### 2. 도입 배경 및 목적
초기 컴퓨터 시스템에서는 인터럽트가 발생하면 하나의 공통 진입점으로 이동한 후, 폴링(Polling)을 통해 어떤 장치가 인터럽트를 발생시켰는지 식별하는 소프트웨어적인 분기 방식을 사용했습니다. 이는 장치가 많아질수록 지연 시간이 길어지는 치명적인 단점이 있었습니다. 이를 극복하기 위해, 하드웨어가 직접 고유 번호(벡터)를 제공하고 이 번호를 테이블의 인덱스로 사용하여 해당 ISR로 즉시 점프할 수 있게 하는 하드웨어/소프트웨어 하이브리드 인덱싱 체계인 인터럽트 벡터가 도입되었습니다.

## Ⅱ. 아키텍처 및 핵심 원리

### 1. 인터럽트 벡터 테이블(IVT) 및 IDT 구조 다이어그램

x86 아키텍처를 기준으로, 인터럽트 벡터 번호를 받아 인터럽트 디스크립터 테이블(IDT)을 참조하고 실제 ISR 코드로 분기하는 구조를 나타냅니다.

```text
 [ Hardware/CPU ]                             [ Main Memory (Kernel Space) ]
 
                                              Interrupt Descriptor Table (IDT)
  1. Interrupt Occurs                        +--------------------------------+
     (e.g., Vector #14 - Page Fault)         | Entry 0: Divide Error          |
              |                              |          [ Segment Selector ]  |
              v                              |          [ Offset Address   ]  |
  +-------------------------+                +--------------------------------+
  | CPU IDTR Register       |-- Base Addr -->| ...                            |
  | (Points to IDT Base)    |                +--------------------------------+
  +-------------------------+     +--------->| Entry 14: Page Fault ISR Addr  |
              |                   |          |           Target: 0xC0105000   |
  2. Compute Memory Address       |          +--------------------------------+
     Base + (Vector * 8 bytes) ---+          | ...                            |
                                             +--------------------------------+
                                                             |
                                                             | 3. Jump to ISR
                                                             v
                                              +--------------------------------+
                                              | Page Fault Service Routine     |
                                              | (Target Address: 0xC0105000)   |
                                              |   - Save registers             |
                                              |   - Handle Page Fault          |
                                              |   - IRET                       |
                                              +--------------------------------+
```

### 2. 동작 원리 논리 전개
1. **벡터 번호 획득**: 하드웨어 인터럽트의 경우 PIC/APIC가 데이터 버스를 통해 CPU에 0~255 사이의 벡터 번호를 제공합니다. 소프트웨어 예외나 시스템 콜의 경우 CPU 내부 연산기나 명령어(`int 0x80`)에서 자체적으로 번호를 생성합니다.
2. **테이블 오프셋 계산**: CPU는 내부 레지스터(IDTR)에 저장된 IDT(Interrupt Descriptor Table)의 베이스 주소에 `(벡터 번호 × 엔트리 크기)`를 더하여 메모리상의 정확한 디스크립터 위치를 계산합니다.
3. **권한 검증 및 분기**: 디스크립터 내에 저장된 세그먼트 셀렉터와 오프셋 주소를 읽어냅니다. 이때 유저 모드에서 커널 모드로의 진입 권한(DPL 검사)을 확인한 후, 이상이 없으면 해당 메모리 주소(ISR)로 Program Counter(EIP/RIP)를 변경하여 점프합니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 벡터 방식과 논-벡터(Non-Vectored) 방식의 비교

| 비교 항목 | Vectored Interrupt (인터럽트 벡터) | Non-Vectored Interrupt (폴링 체인) |
| :--- | :--- | :--- |
| **ISR 진입 방법** | 하드웨어가 제공한 인덱스로 즉시 분기 (O(1)) | 단일 진입점 진입 후 장치 상태를 순차적 검사 (O(n)) |
| **처리 속도 및 지연** | 매우 빠름 (지연 시간 최소화) | 장치 수에 비례하여 처리 지연 시간 증가 |
| **메모리 요구사항** | IVT/IDT를 유지하기 위한 추가 메모리 필요 | 최소한의 메모리만 필요 |
| **하드웨어 지원** | 고급 인터럽트 컨트롤러 필수 | 단순한 인터럽트 라인 하나로 구현 가능 |

### 2. x86의 IVT (Real Mode) vs IDT (Protected Mode)
* **IVT (Interrupt Vector Table)**: 16비트 리얼 모드에서 사용되며, 무조건 물리 메모리의 0x0000 위치에 고정 존재. 보안/권한 검사 기능이 없습니다.
* **IDT (Interrupt Descriptor Table)**: 32/64비트 보호 모드에서 사용되며, 메모리 상 어느 곳에든 존재 가능(IDTR로 지정). 각 엔트리(디스크립터)에 권한 레벨(Privilege Level)이 포함되어 유저 프로세스의 임의 커널 접근을 차단합니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 보안 측면의 인터럽트 후킹(Interrupt Hooking)
실무와 정보 보안 영역에서 인터럽트 벡터 테이블(IDT)은 시스템 흐름을 장악하기 위한 주요 타겟이 됩니다. 악성 코드(Rootkit 등)는 IDT의 특정 엔트리(예: 시스템 콜 벡터)를 가로채어 자신들의 악성 루틴 주소로 변조(IDT Hooking)함으로써, 시스템 콜의 파라미터를 조작하거나 결과를 은폐합니다. 최신 커널은 이를 방어하기 위해 커널 메모리를 읽기 전용으로 보호하고 패치 가드(PatchGuard)와 같은 무결성 검증 메커니즘을 적용합니다.

### 2. IRQ 라인과 벡터의 동적 매핑(Dynamic Mapping)
과거 ISA 버스 시절에는 IRQ와 인터럽트 벡터가 하드코딩(Static)되어 충돌이 빈번했습니다. 현대의 PCI/PCIe 기반 시스템과 APIC 환경에서는 운영체제가 부팅 시 장치를 탐색(Enumeration)하고 IRQ와 인터럽트 벡터 번호를 동적으로 할당(Dynamic Allocation)하여 리소스 충돌을 근본적으로 해결합니다.

## Ⅴ. 기대효과 및 결론

인터럽트 벡터는 복잡하고 방대한 컴퓨터 시스템의 다양한 이벤트들을 **규격화된 배열(테이블) 형태로 매핑하여 O(1)의 시간 복잡도로 신속하게 처리**할 수 있게 하는 핵심 자료구조이자 아키텍처적 약속입니다. 이를 통해 운영체제는 수백 가지의 하드웨어 장치와 수많은 소프트웨어 예외 상황을 일관성 있고 빠르게 라우팅할 수 있습니다. 시스템 엔지니어는 커널 덤프 및 디버깅 시 이 벡터 번호(예: Page Fault = 14, General Protection Fault = 13)를 통해 시스템 장애의 근본 원인을 직관적으로 추적하고 해결할 수 있어야 합니다.
