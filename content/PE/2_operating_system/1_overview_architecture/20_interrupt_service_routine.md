# 인터럽트 서비스 루틴 (ISR, Interrupt Service Routine)

## Ⅰ. 개요

### 1. 인터럽트 서비스 루틴(ISR)의 정의
인터럽트 서비스 루틴(ISR)은 인터럽트 핸들러(Interrupt Handler)라고도 불리며, 특정한 하드웨어 인터럽트나 소프트웨어 예외가 발생했을 때 **해당 이벤트를 실질적으로 처리하기 위해 운영체제 커널 내에 정의된 특수한 함수이자 코드 블록**입니다. 

### 2. ISR의 핵심 요구사항
* **신속성 (Low Latency)**: ISR 실행 중에는 다른 인터럽트가 마스킹(차단)되는 경우가 많으므로, 전체 시스템 응답성 저하를 막기 위해 가능한 한 빠르고 간결하게 실행되어야 합니다.
* **비블로킹 (Non-blocking)**: ISR 내부에서는 스레드를 블록(Sleep, Wait)시키는 시스템 콜(예: 세마포어 대기, 긴 디스크 I/O)을 절대로 호출해서는 안 됩니다.

## Ⅱ. 아키텍처 및 핵심 원리

### 1. ISR 실행 컨텍스트 스위칭 다이어그램

인터럽트 발생부터 ISR 완료 후 복귀까지의 하드웨어-소프트웨어 간 상호작용과 문맥 보존(Context Saving) 흐름입니다.

```text
  [ User Process ]                       [ Kernel Space (ISR Context) ]
  
  Executing Instruction N
            |
  (Interrupt Signal Arrives) 
            |
            v
  +----------------------+      H/W      +--------------------------------+
  | Hardware checks      |-------------->| Push EFLAGS, CS, EIP to Stack  |
  | Privilege & Vector   |               | Change to Kernel Stack         |
  +----------------------+               +--------------------------------+
                                                         |
                                                         v
                                         +--------------------------------+
                                         | ISR Entry (Assembly wrapper)   |
                                         |  - Push General Registers      |
                                         +--------------------------------+
                                                         |
                                                         v
                                         +--------------------------------+
                                         | Actual C Function ISR          |
                                         |  - Acknowledge Device (ACK)    |
                                         |  - Read/Write Data Port        |
                                         |  - Schedule Bottom-Half        |
                                         +--------------------------------+
                                                         |
                                                         v
                                         +--------------------------------+
                                         | ISR Exit                       |
                                         |  - Pop General Registers       |
                                         |  - IRET (Interrupt Return)     |
                                         +--------------------------------+
            ^                                            |
            |                                            |
  Executing Instruction N+1  <---------------------------+
  (Process Resumes)
```

### 2. 분할 처리 아키텍처: Top-half와 Bottom-half
ISR을 작고 빠르게 유지하기 위해, 현대 운영체제(Linux 등)는 인터럽트 처리를 두 단계로 나눕니다.
* **Top-half (전반부 처리)**: 인터럽트 발생 즉시 실행되는 실제 ISR 부분. 가장 시급한 작업(하드웨어 레지스터 읽기/초기화, 데이터 복사, 디바이스 ACK 응답)만을 수행합니다. 이 동안은 다른 인터럽트가 비활성화됩니다.
* **Bottom-half (후반부 처리)**: 시급하지 않은 나머지 작업(복잡한 데이터 연산, 네트워크 프로토콜 스택 전달 등)을 지연시켜 처리합니다. Tasklet, Workqueue, SoftIRQ 등의 기법이 사용되며, 이때는 인터럽트가 활성화된 상태로 수행되어 시스템 동시성을 높입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 일반 함수 호출(Function Call) vs ISR(Interrupt Service Routine) 호출

| 비교 항목 | 일반 함수 호출 | 인터럽트 서비스 루틴 (ISR) |
| :--- | :--- | :--- |
| **호출 주체** | 소프트웨어 프로세스 내의 코드 | 하드웨어 이벤트 또는 비동기적 예외 |
| **컨텍스트 교환** | 최소한의 레지스터(Caller-saved) 스택 저장 | 완전한 프로세서 상태(PC, Status Reg, 모든 레지스터) 저장 및 커널 스택 전환 |
| **실행 예측성** | 언제 호출될지 코드 흐름을 통해 명확히 예측 가능 | 언제 호출될지 예측 불가능 (Asynchronous) |
| **실행 제약 사항** | 블로킹 가능, 복잡한 로직 허용 | 블로킹 절대 금지, 최대한 짧고 빠른 로직 요구 |

### 2. 인터럽트 중첩 (Nested Interrupts)
ISR 실행 도중 우선순위가 더 높은 인터럽트가 발생할 경우, 현재 ISR의 상태를 스택에 다시 저장하고 새로운 ISR을 서비스하는 **중첩 인터럽트**가 허용될 수 있습니다. 이를 지원하기 위해서는 커널 스택 오버플로우 방지 메커니즘과 세밀한 인터럽트 마스킹 정책이 필수적입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. RTOS(Real-Time OS)에서의 ISR 설계 지침
항공기 제어, 자율주행, 의료 기기 등에 사용되는 RTOS에서는 인터럽트 반응 시간(Interrupt Latency)이 생명과 직결됩니다. 기술사는 이러한 시스템을 설계할 때, ISR 내부 코드를 O(1) 시간 복잡도로 설계하고, 캐시 미스를 최소화하며, 모든 무거운 처리는 데몬 스레드(Bottom-half)로 오프로드(Offload) 하도록 강제하는 아키텍처를 수립해야 합니다.

### 2. 디바이스 드라이버 최적화
리눅스 환경에서 네트워크 카드(NIC) 드라이버를 개발할 때, 패킷 하나당 ISR을 호출하면 기가비트 이상의 트래픽에서 시스템이 멈추는 현상이 발생합니다. 이를 해결하기 위해 첫 패킷 수신 시에만 ISR을 호출하고, 이후에는 커널이 폴링 기반으로 패킷을 수거해가는 **NAPI(New API) 하이브리드 메커니즘**을 적용하는 것이 실무적 표준입니다.

## Ⅴ. 기대효과 및 결론

인터럽트 서비스 루틴(ISR)은 운영체제가 외부 세계(하드웨어 장치)와 소통하는 가장 최전방의 창구입니다. ISR의 설계 품질은 곧 운영체제의 안정성, 응답성, 전체 처리량을 좌우합니다. 
Top-half와 Bottom-half의 개념을 통해 실시간 반응성과 동시성이라는 두 마리 토끼를 잡을 수 있었으며, 최신 운영체제 아키텍처에서는 ISR의 부하를 코어별로 분산시키고 다이나믹하게 로드 밸런싱하는 방향으로 기술이 진화하고 있습니다. ISR의 정확한 이해는 안정적이고 고성능의 시스템 소프트웨어와 디바이스 드라이버를 구현하기 위한 가장 필수적인 기반 지식입니다.
