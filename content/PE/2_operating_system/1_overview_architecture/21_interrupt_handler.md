# 인터럽트 핸들러 (Interrupt Handler)

## Ⅰ. 개요

### 1. 인터럽트 핸들러의 정의와 ISR과의 관계
인터럽트 핸들러(Interrupt Handler)는 좁은 의미로는 인터럽트 서비스 루틴(ISR)과 동일하게 인터럽트 발생 시 실행되는 코드를 지칭합니다. 그러나 운영체제 및 디바이스 드라이버 아키텍처 관점에서 넓은 의미의 **인터럽트 핸들러**는 특정 하드웨어 장치를 관리하는 드라이버가 커널의 인터럽트 서브시스템에 **등록(Registration)하고 관리하는 콜백(Callback) 함수이자 논리적인 관리 모듈**을 의미합니다.

### 2. 인터럽트 핸들러의 역할
인터럽트 핸들러는 단순한 코드 실행을 넘어 다음과 같은 종합적인 역할을 수행합니다.
* 장치의 인터럽트 상태 레지스터 확인 및 초기화(Acknowledge).
* 장치 버퍼와 메인 메모리 간의 데이터 송수신(I/O 처리).
* 블록된 프로세스(I/O를 대기 중인 사용자 프로그램) 깨우기(Wake-up).
* 동일한 인터럽트 라인(IRQ)을 공유하는 장치들 간의 식별 및 충돌 관리.

## Ⅱ. 아키텍처 및 핵심 원리

### 1. 인터럽트 핸들러 등록 및 호출 다이어그램 (공유 IRQ 환경)

PCI 기반 시스템 등에서는 IRQ 라인이 제한적이므로, 여러 디바이스가 하나의 IRQ를 공유(Shared IRQ)하게 됩니다. 이 환경에서 커널이 올바른 핸들러를 찾아 호출하는 아키텍처입니다.

```text
 [ Hardware IRQ Lines ]                [ OS Kernel Interrupt Subsystem ]

                      +--------------------------------------------------------+
                      | Interrupt Descriptor Table (IDT) / Vector Table        |
   IRQ 11 (Shared)    |  Vector 0x2B ---> Common IRQ Dispatcher                |
 -------------------->|                        |                               |
                      +------------------------|-------------------------------+
                                               v
                      +--------------------------------------------------------+
                      | IRQ 11 Action List (Linked List of Handlers)           |
                      |                                                        |
                      |  [ Handler Node 1 ]       [ Handler Node 2 ]           |
                      |   - dev_id: USB_Ctrl  ---> - dev_id: NIC_Eth0  ---> NULL
                      |   - handler_func: funcA    - handler_func: funcB       |
                      +--------------------------------------------------------+
                               |                         |
                               v                         v
                       +---------------+         +---------------+
                       | USB Handler   |         | NIC Handler   |
                       | (funcA)       |         | (funcB)       |
                       | 1. Is it mine?|-- No -->| 1. Is it mine?|-- Yes!
                       |    (Check HW) |         |    (Check HW) |
                       | 2. Handle...  |         | 2. Handle &   |
                       | 3. Return     |         |    Return     |
                       |    IRQ_NONE   |         |    IRQ_HANDLED|
                       +---------------+         +---------------+
```

### 2. 핵심 동작 원리
1. **드라이버 적재 및 등록**: 디바이스 드라이버가 로드될 때, 커널 API(예: 리눅스의 `request_irq()`)를 호출하여 자신의 인터럽트 핸들러 함수, IRQ 번호, 그리고 장치 고유 식별자(dev_id)를 커널에 등록합니다.
2. **인터럽트 발생 및 디스패치**: 하드웨어 인터럽트가 발생하면 커널의 공통 디스패처가 실행되고, 해당 IRQ 번호에 연결된(Linked List) 등록된 핸들러들을 순차적으로 호출합니다.
3. **핸들러 내 장치 식별**: 각 핸들러는 즉시 해당 하드웨어 장치의 상태 레지스터를 읽어, 이 인터럽트가 자신이 담당하는 장치에서 발생한 것인지 확인합니다.
4. **결과 반환**: 자신이 발생시킨 인터럽트라면 처리를 수행하고 `IRQ_HANDLED`를 반환하며, 아니라면 즉시 빠져나와 `IRQ_NONE`을 반환하여 다음 핸들러가 검사할 수 있도록 합니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 전용 핸들러 (Dedicated) vs 공유 핸들러 (Shared)

| 비교 항목 | 전용(Dedicated) 인터럽트 핸들러 | 공유(Shared) 인터럽트 핸들러 |
| :--- | :--- | :--- |
| **IRQ 매핑** | 1개 장치당 1개의 고유 IRQ 라인 사용 | 여러 장치가 1개의 IRQ 라인 공유 |
| **장치 판별 로직** | 필요 없음 (IRQ 발생 = 해당 장치 발생 보장) | 필수 (호출 시 장치 레지스터를 읽어 본인 것인지 확인해야 함) |
| **확장성** | 하드웨어 IRQ 핀 수의 제한으로 확장성 낮음 | IRQ 라인 부족 문제를 해결, 높은 확장성 |
| **성능 오버헤드** | 없음 (즉시 처리) | 체인 내의 다른 핸들러를 순차적으로 호출하며 검사하는 오버헤드 존재 |

### 2. 폴링 기반 드라이버 설계와의 비교
핸들러 기반(Interrupt-driven) 구조는 CPU 자원을 절약하지만 빈번한 문맥 교환 비용이 발생합니다. 반면 폴링은 즉각적인 응답이 보장되나 지속적인 CPU 점유가 단점입니다. 따라서 최신 네트워크 핸들러는 트래픽이 적을 때는 인터럽트 모드로 작동하고, 폭주 시에는 내부적으로 폴링 루프 모드로 전환하는 하이브리드 핸들러 아키텍처를 채택합니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. MSI/MSI-X (Message Signaled Interrupts) 패러다임 전환
기술사로서 최신 하드웨어 아키텍처를 진단할 때, 레거시 핀 기반 공유 IRQ의 병목과 관리 복잡성을 해결하는 방안으로 PCIe 버스의 **MSI/MSI-X** 도입을 적극 고려해야 합니다. MSI는 별도의 IRQ 핀이 아니라 메모리 쓰기(Memory Write) 메시지를 통해 CPU에 인터럽트를 직접 전달합니다. 이를 통해 수백 수천 개의 인터럽트 벡터를 독립적으로 할당하여 **공유 없는 완전한 전용 인터럽트 핸들러 매핑**이 가능해져 코어별 로드 밸런싱 효율이 비약적으로 상승합니다.

### 2. 드라이버 개발 시 핸들러 안정성 확보 전략
커널 공간에서 구동되는 핸들러의 버그(메모리 누수, 무한 루프, 부적절한 락킹)는 전체 운영체제의 패닉(Kernel Panic, BSOD)을 초래합니다. 시스템 통합 및 개발 실무에서는 핸들러 내에서 스핀락(Spinlock) 사용 시 데드락을 방지하기 위한 IRQ 비활성화 매크로(`spin_lock_irqsave`) 사용 여부를 철저히 코드 리뷰하고 검증해야 합니다.

## Ⅴ. 기대효과 및 결론

인터럽트 핸들러 메커니즘은 운영체제가 한정된 인터럽트 자원을 체계적이고 유연하게 관리할 수 있도록 돕는 소프트웨어 아키텍처의 정수입니다. 동적인 핸들러 등록 체계와 공유 메커니즘은 Plug-and-Play와 핫스왑(Hot-swap) 기능을 가능하게 하여 사용자 편의성을 극대화했습니다. 향후 초연결 시대의 방대한 IoT 디바이스와 고속 NVMe/100G NIC 환경에서, 병목 없는 데이터 처리를 위해서는 인터럽트 핸들러의 최적화 수준이 시스템 전체의 성능을 결정짓는 핵심 지표가 될 것입니다.
