# 다중 프로그래밍 (Multiprogramming)

## 정의

**다중 프로그래밍(Multiprogramming)**은 **메모리에 여러 프로그램을 동시에 적재**하여, 하나의 프로그램이 I/O 대기 상태가 되면 다른 프로그램에게 CPU를 할당함으로써 **CPU 활용도를 극대화**하는 기법이다.

```
┌─────────────────────────────────────────────────────────────────┐
│           단일 프로그래밍 vs 다중 프로그래밍                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   단일 프로그래밍 (Uniprogramming):                              │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │                                                         │ │
│   │   시간 ──────────────────────────────────────────────▶ │ │
│   │                                                         │ │
│   │   프로그램A: [CPU][I/O대기....][CPU][I/O대기....]       │ │
│   │               ↑           ↑           ↑                │ │
│   │            CPU가 idle   CPU가 idle   CPU가 idle         │ │
│   │                                                         │ │
│   │   CPU 활용도: ~30% (I/O 많은 프로그램의 경우)            │ │
│   │                                                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│   다중 프로그래밍 (Multiprogramming):                           │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │                                                         │ │
│   │   시간 ──────────────────────────────────────────────▶ │ │
│   │                                                         │ │
│   │   프로그램A: [CPU][I/O..][    CPU    ][I/O..]           │ │
│   │   프로그램B:      [CPU][I/O..][  CPU  ]                 │ │
│   │   프로그램C:           [CPU][I/O..]                     │ │
│   │                     ↑                                   │ │
│   │               A가 I/O 대기 중 B가 CPU 사용               │ │
│   │                                                         │ │
│   │   CPU 활용도: ~80% 이상 달성 가능                        │ │
│   │                                                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 핵심 개념

### 1. 다중 프로그래밍의 동작 원리

```
┌─────────────────────────────────────────────────────────────────┐
│             다중 프로그래밍 동작 메커니즘                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │                                                         │ │
│   │   [메모리]                                               │ │
│   │   ┌───────────────────────────────────────────────┐    │ │
│   │   │  프로그램 A  │  프로그램 B  │  프로그램 C     │    │ │
│   │   └───────────────────────────────────────────────┘    │ │
│   │                          │                              │ │
│   │                          ▼                              │ │
│   │   ┌───────────────────────────────────────────────┐    │ │
│   │   │             준비 큐 (Ready Queue)              │    │ │
│   │   │          [B] → [C] → [A] → ...                │    │ │
│   │   └───────────────────────────────────────────────┘    │ │
│   │                          │                              │ │
│   │                          ▼                              │ │
│   │   ┌───────────────────────────────────────────────┐    │ │
│   │   │               CPU 스케줄러                     │    │ │
│   │   │         "다음 실행할 프로세스 선택"            │    │ │
│   │   └───────────────────────────────────────────────┘    │ │
│   │                          │                              │ │
│   │                          ▼                              │ │
│   │   ┌───────────────────────────────────────────────┐    │ │
│   │   │              CPU (실행 중)                     │    │ │
│   │   └───────────────────────────────────────────────┘    │ │
│   │                                                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│   핵심: I/O 요청 시 즉시 문맥 교환하여 CPU 낭비 방지            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2. 다중 프로그래밍도 (Degree of Multiprogramming)

```
┌─────────────────────────────────────────────────────────────────┐
│              다중 프로그래밍도와 성능 관계                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   다중 프로그래밍도 = 메모리에 동시 적재된 프로세스 수            │
│                                                                 │
│   CPU 활용도 그래프:                                             │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │  CPU                                                  ▲ │ │
│   │  활 100% ┤                                        ╱    │ │
│   │  용      ├─────────────────────────────────────╱───────│ │
│   │  도      ├───────────────────────────────╱────────────│ │
│   │   50%    ├───────────────────────╱───────────────────│ │
│   │          ├─────────────────╱────────────────────────│ │
│   │    0%    ├──────────╱───────────────────────────────│ │
│   │          └─────┼─────┼─────┼─────┼─────┼─────┼─────▶ │ │
│   │                1     2     3     4     5     6       N │ │
│   │                      다중 프로그래밍도                   │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │  프로그래밍도 │  장점                  │  단점           │ │
│   ├───────────────┼────────────────────────┼────────────────┤ │
│   │  낮음 (1-2)   │  문맥교환 오버헤드 적음 │  CPU 활용도 낮음│ │
│   │  적절 (3-5)   │  균형 잡힌 성능         │  -              │ │
│   │  높음 (10+)   │  CPU 활용도 극대화      │  스래싱 위험    │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3. 문맥 교환 (Context Switch)

```
┌─────────────────────────────────────────────────────────────────┐
│                 문맥 교환 과정                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   I/O 요청 발생 시:                                              │
│                                                                 │
│   ① 실행 중인 프로세스 A가 I/O 요청                              │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │   Process A: Running → Waiting (I/O 대기 상태)          │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│   ② 프로세스 A의 문맥 저장                                       │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │   - PCB(Process Control Block)에 저장:                  │ │
│   │   • 레지스터 값 (PC, SP, General Registers)             │ │
│   │   • 메모리 관련 정보                                     │ │
│   │   • 열린 파일 목록                                       │ │
│   │   • 스케줄링 정보                                        │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│   ③ 스케줄러가 다음 프로세스 B 선택                              │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │   CPU Scheduler: Ready Queue → Process B 선택           │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│   ④ 프로세스 B의 문맥 복원 후 실행                               │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │   - PCB에서 B의 문맥 로드                                │ │
│   │   - CPU가 B의 코드 실행 시작                             │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│   문맥 교환 오버헤드: 순수 overhead (유용한 작업 안 함)          │
│   → 최소화가 중요                                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4. 장점과 단점

```
┌─────────────────────────────────────────────────────────────────┐
│           다중 프로그래밍 장단점                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   장점:                                                         │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │ ① CPU 활용도 극대화                                      │ │
│   │    - I/O 대기 시간을 다른 프로세스 실행에 활용            │ │
│   │    - 유휴 시간(Idle Time) 최소화                          │ │
│   │                                                         │ │
│   │ ② 시스템 처리량(Throughput) 증가                         │ │
│   │    - 단위 시간당 완료 작업 수 증가                        │ │
│   │                                                         │ │
│   │ ③ 자원 효율성                                            │ │
│   │    - 메모리, I/O 장치 등을 여러 프로세스가 공유            │ │
│   │                                                         │ │
│   │ ④ 응답성 향상 (시분할과 결합 시)                          │ │
│   │    - 대화형 시스템에서 빠른 응답                          │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│   단점:                                                         │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │ ① 복잡성 증가                                            │ │
│   │    - 스케줄링 알고리즘 필요                               │ │
│   │    - 메모리 관리 복잡                                     │ │
│   │                                                         │ │
│   │ ② 오버헤드 발생                                          │ │
│   │    - 문맥 교환 비용                                       │ │
│   │    - 메모리 접근 경쟁                                     │ │
│   │                                                         │ │
│   │ ③ 동기화 문제                                            │ │
│   │    - 공유 자원 접근 시 경쟁 조건(Race Condition)          │ │
│   │    - 교착 상태(Deadlock) 가능성                          │ │
│   │                                                         │ │
│   │ ④ 메모리 요구 증가                                       │ │
│   │    - 여러 프로그램 동시 적재 필요                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 기술사 시험 포인트

### 1. 핵심 정의

> **다중 프로그래밍**: 메모리에 여러 프로그램 적재, I/O 대기 시 CPU 스위칭
> **다중 프로그래밍도**: 메모리에 동시 적재된 프로세스 수

### 2. 빈출 키워드

- **CPU 활용도**: I/O 대기 시간 활용으로 향상
- **문맥 교환**: 프로세스 전환 비용
- **준비 큐**: 실행 대기 프로세스 목록
- **스래싱**: 과도한 다중 프로그래밍으로 성능 저하

### 3. 예상 문제

**Q1. 다중 프로그래밍 시스템에서 CPU 활용도가 향상되는 원리를 설명하시오.**

**Q2. 다중 프로그래밍도를 무제한 높이는 것의 문제점을 기술하시오.**

**Q3. 다중 프로그래밍과 시분할 시스템의 차이점을 설명하시오.**
