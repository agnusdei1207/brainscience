# 시분할 시스템 (Time-sharing System)

## 정의

**시분할 시스템(Time-sharing System)**은 CPU 시간을 작은 단위(Time Slice/Time Quantum)로 분할하여 여러 사용자에게 순차적으로 할당함으로써, 각 사용자가 마치 자신만을 위해 컴퓨터를 독점하고 있는 것처럼 느끼게 하는 운영체제 기법이다. 다중 프로그래밍의 확장된 형태로, **응답 시간 최소화**와 **인터랙티브(interactive) 환경 제공**이 핵심 목표이다.

```
+------------------------------------------------------------------+
|                    시분할 시스템 개념도                             |
+------------------------------------------------------------------+

    사용자 A        사용자 B        사용자 C        사용자 D
        |              |              |              |
        v              v              v              v
    +--------+     +--------+     +--------+     +--------+
    | 터미널  |     | 터미널  |     | 터미널  |     | 터미널  |
    +---+----+     +---+----+     +---+----+     +---+----+
        |              |              |              |
        +--------------+--------------+--------------+
                              |
                              v
                    +------------------+
                    |   CPU 스케줄러    |
                    |   (Time Slicing) |
                    +--------+---------+
                             |
                    +--------v---------+
                    |   중앙처리장치     |
                    |      (CPU)       |
                    +--------+---------+
                             |
                    +--------v---------+
                    |   공유 메모리     |
                    |   공유 파일 시스템 |
                    +------------------+

    시간 ->
    |----|----|----|----|----|----|----|----|----|----|
    [ A ][ B ][ C ][ D ][ A ][ B ][ C ][ D ][ A ][ B ]...
       <- Time Quantum (예: 10-100ms) ->

    각 사용자 관점: 연속적인 실행 (빠른 응답)
    실제 CPU 관점: 순차적 분할 실행
```

## 핵심 개념

### 1. 시분할 시스템의 핵심 원리

```
+================================================================+
|                   시분할 시스템 작동 원리                         |
+================================================================+
|                                                                |
|  시간 축 (Time Axis)                                           |
|  |------|------|------|------|------|------|------|           |
|  |  Q1  |  Q2  |  Q3  |  Q4  |  Q5  |  Q6  |  Q7  | ...       |
|  |------|------|------|------|------|------|------|           |
|  | P_A  | P_B  | P_C  | P_A  | P_B  | P_C  | P_A  |           |
|                                                                |
|  Q = Time Quantum (시간 할당량)                                |
|  P = Process (프로세스)                                        |
|                                                                |
|  핵심 메커니즘:                                                |
|  1. 각 프로세스에 고정된 시간 할당량 배분                        |
|  2. 시간 할당량 소진 시 강제 CPU 회수 (선점)                    |
|  3. 다음 프로세스에게 CPU 할당 (문맥 교환)                      |
|  4. 준비 큐(Ready Queue)를 순회하며 반복                        |
|                                                                |
+================================================================+
```

#### Time Quantum (시간 할당량) 결정 요소

```
+-------------------+-------------------------------------------+
|     요소           |                 영향                       |
+-------------------+-------------------------------------------+
| 하드웨어 속도      | 빠른 CPU -> 작은 Quantum 가능              |
|                   | 느린 CPU -> 큰 Quantum 필요                |
+-------------------+-------------------------------------------+
| 사용자 수          | 많은 사용자 -> 작은 Quantum                |
|                   | 적은 사용자 -> 큰 Quantum                  |
+-------------------+-------------------------------------------+
| 워크로드 유형      | 대화형 -> 작은 Quantum (빠른 응답)         |
|                   | 일괄처리 -> 큰 Quantum (처리량)             |
+-------------------+-------------------------------------------+
| 문맥 교환 오버헤드 | 높은 오버헤드 -> 큰 Quantum                |
|                   | 낮은 오버헤드 -> 작은 Quantum              |
+-------------------+-------------------------------------------+
| 메모리 크기        | 작은 메모리 -> 스와핑 증가 -> 큰 Quantum   |
+-------------------+-------------------------------------------+

    Quantum 크기 결정 공식 (경험적):
    Quantum >= 10 x 문맥 교환 시간

    예시:
    - 문맥 교환 시간: 10us
    - 권장 Quantum: 100us 이상
    - 실제 시스템: 10ms ~ 100ms (사용자 체감 고려)
```

### 2. 다중 프로그래밍 vs 시분할 시스템

```
+================================================================+
|              다중 프로그래밍 vs 시분할 시스템 비교                 |
+================================================================+

[다중 프로그래밍 (Multiprogramming)]

    목표: CPU 활용도 극대화 (Throughput 중심)

    시간 ->
    |-------I/O-------|       |-------I/O-------|
    |      P1 대기     |  P2   |      P1 대기     |
    CPU: [    P1      ][ P2  ][       P1        ]
             ^
             P1이 I/O 요청 -> P2로 전환 (이벤트 기반)

    특징: 프로세스가 I/O 등으로 대기할 때만 CPU 양보
          (비선점형 스케줄링 가능)


[시분할 시스템 (Time-sharing System)]

    목표: 응답 시간 최소화 (Response Time 중심)

    시간 ->
    |--Q--|--Q--|--Q--|--Q--|--Q--|--Q--|--Q--|
    [ P1 ][ P2 ][ P3 ][ P1 ][ P2 ][ P3 ][ P1 ]
       ^                              ^
       |____ 시간 할당량 강제 회수 ____|
            (선점형 스케줄링 필수)

    특징: 정해진 시간 경과 시 강제로 CPU 회수
          (선점형 스케줄링 필수)
```

#### 상세 비교표

```
+-------------------+-------------------------------------------+
|     구분          |    다중 프로그래밍    |    시분할 시스템    |
+-------------------+-------------------------------------------+
| 핵심 목표          | CPU 활용도 극대화     | 응답 시간 최소화   |
|                   | Throughput 향상       | Interactivity     |
+-------------------+-------------------------------------------+
| 스케줄링 방식      | 비선점형 가능         | 선점형 필수       |
+-------------------+-------------------------------------------+
| CPU 전환 시점      | I/O 대기, 종료 시      | 시간 할당량 만료   |
+-------------------+-------------------------------------------+
| 사용자 인터페이스  | 일괄 처리 (Batch)     | 대화형 (Interactive)|
+-------------------+-------------------------------------------+
| 응답 시간          | 수 초 ~ 수 분          | 수 ms ~ 수 초     |
+-------------------+-------------------------------------------+
| 프로세스 수        | 제한적 (메모리 크기)   | 스와핑으로 확장   |
+-------------------+-------------------------------------------+
| 문맥 교환 빈도     | 낮음                  | 높음              |
+-------------------+-------------------------------------------+
| 오버헤드          | 낮음                  | 높음              |
+-------------------+-------------------------------------------+
| 적용 분야          | 일괄 처리 시스템       | 온라인 시스템     |
|                   | 백엔드 서버           | 워크스테이션      |
+-------------------+-------------------------------------------+
```

### 3. 인터랙티브 시스템 (Interactive System)

**정의**: 사용자와 시스템 간에 실시간으로 상호작용이 가능한 컴퓨팅 환경으로, 사용자 입력에 대해 즉각적인 피드백을 제공하는 시스템이다.

```
+================================================================+
|               인터랙티브 시스템의 사용자 경험                      |
+================================================================+

[사용자 관점 - 시분할 시스템 없이]
    사용자: "명령어 입력"
    시스템: ... (대기) ... (다른 작업 처리) ...
    시스템: "결과 출력" (30초 후)
    사용자 체감: 느림, 답답함, 생산성 저하


[사용자 관점 - 시분할 시스템 적용]
    사용자: "명령어 입력"
    시스템: "결과 출력" (100ms 후)
    사용자 체감: 빠름, 반응성 좋음, 생산성 향상


[인터랙티브 세션 예시]
    $ ls -la                    <- 사용자 입력
    total 48
    drwxr-xr-x  12 user  ...
    -rw-r--r--   1 user  ...
                            <- 50ms 내 응답
    $ grep "error" log.txt      <- 사용자 입력
    2024-01-15 error: ...
    2024-01-16 error: ...
                            <- 80ms 내 응답
    $ vim config.txt            <- 사용자 입력
    (에디터 화면 즉시 표시)     <- 30ms 내 응답
```

#### 인터랙티브 시스템 응답 시간 기준

```
+-------------------+-------------------+-------------------+
|     작업 유형      |   목표 응답 시간   |    사용자 체감     |
+-------------------+-------------------+-------------------+
| 키보드 입력 echo   | < 10ms            | 즉각적             |
| 단순 명령어 실행   | < 100ms           | 매우 빠름          |
| 화면 갱신         | < 100ms           | 부드러움           |
| 파일 열기         | < 500ms           | 빠름               |
| 검색/필터링       | < 1s              | 수용 가능          |
| 복잡한 연산       | < 2s              | 약간 지연          |
+-------------------+-------------------+-------------------+

    인터랙티브 시스템 설계 원칙:
    1. 0.1초 이내: 즉각적 반응 (사용자가 대기 없다고 느낌)
    2. 1초 이내: 자연스러운 반응 (대화 흐름 유지)
    3. 10초 이내: 사용자 주의 유지 가능
    10초 초과: 진행 표시 및 취소 옵션 필요
```

## 상세 설명

### 1. 라운드 로빈 스케줄링 (Round Robin Scheduling)

시분할 시스템에서 가장 널리 사용되는 CPU 스케줄링 알고리즘이다.

```
+================================================================+
|                 라운드 로빈 스케줄링 동작                          |
+================================================================+

    준비 큐 (Ready Queue): [ P1 ] -> [ P2 ] -> [ P3 ] -> [ P4 ]
                                     ^                    |
                                     |____________________|

    시간 ->
    |--Q--|--Q--|--Q--|--Q--|--Q--|--Q--|--Q--|--Q--|
    [ P1 ][ P2 ][ P3 ][ P4 ][ P1 ][ P2 ][ P3 ][ P4 ]...

    Time Quantum: Q (예: 10ms)

    동작 과정:
    1. P1이 Q 시간 동안 실행
    2. 타이머 인터럽트 발생
    3. P1을 준비 큐의 끝으로 이동
    4. P2에게 CPU 할당
    5. Q 시간 후 P2를 큐 끝으로 이동
    6. P3, P4, P1, ... 순으로 반복
```

#### 라운드 로빈 성능 분석

```
    성능 지표:

    1. 반환 시간 (Turnaround Time)
       T_turnaround = T_completion - T_arrival

    2. 대기 시간 (Waiting Time)
       T_waiting = T_turnaround - T_burst

    3. 응답 시간 (Response Time)
       T_response = T_first_response - T_arrival


    예시: 4개 프로세스, Quantum = 2ms

    프로세스  | 버스트 시간
    ---------|------------
      P1     |    8ms
      P2     |    4ms
      P3     |    2ms
      P4     |    6ms

    실행 순서:
    P1(2) -> P2(2) -> P3(2) -> P4(2) -> P1(2) -> P2(0) -> P4(2) -> P1(2) -> P4(2) -> P1(2)

    시간 ->
    | 0-2 | 2-4 | 4-6 | 6-8 | 8-10|10-12|12-14|14-16|16-18|18-20|
    [ P1 ][ P2 ][ P3 ][ P4 ][ P1 ][ P2 ][ P4 ][ P1 ][ P4 ][ P1 ]

    완료 시점: P1=20, P2=12, P3=6, P4=18
    반환 시간: P1=20, P2=12, P3=6, P4=18
    평균 반환 시간 = (20+12+6+18)/4 = 14ms
```

#### Quantum 크기에 따른 성능 변화

```
    Quantum이 너무 작은 경우:
    +------------------------------------------+
    | Q = 1ms (과도하게 작음)                   |
    |                                          |
    | [P1][P2][P3][P1][P2][P3][P1][P2][P3]...   |
    |                                          |
    | 문제점:                                   |
    | - 문맥 교환 오버헤드 급증                  |
    | - CPU 사이클의 상당 부분이 문맥 교환에 소모|
    | - 실제 작업 수행 시간 감소                 |
    +------------------------------------------+

    Quantum이 너무 큰 경우:
    +------------------------------------------+
    | Q = 1000ms (과도하게 큼)                  |
    |                                          |
    | [   P1 전체 실행 (8ms)   ] [  P2  ] ...   |
    |                                          |
    | 문제점:                                   |
    | - FCFS(First-Come-First-Served)와 동일    |
    | - 응답 시간 악화                          |
    | - 시분할 시스템의 장점 상실                |
    +------------------------------------------+

    적절한 Quantum:
    +------------------------------------------+
    | Q = 10-100ms (균형 잡힌 선택)             |
    |                                          |
    | - 문맥 교환 오버헤드 < CPU 시간의 1%      |
    | - 사용자 체감 응답 시간 양호              |
    | - 공정한 CPU 시간 배분                    |
    +------------------------------------------+
```

### 2. 선점형 vs 비선점형 스케줄링

```
+================================================================+
|              선점형 vs 비선점형 스케줄링 비교                      |
+================================================================+

[비선점형 스케줄링 (Non-preemptive)]

    실행 중인 프로세스가 자발적으로 CPU를 양보할 때까지 계속 실행

    시간 ->
    |-------|       |-------|       |-------|
    [  P1   ] 대기  [  P2   ] 종료  [  P3   ]
       I/O 요청
       (자발적)

    특징:
    - 문맥 교환 오버헤드 낮음
    - 응답 시간 예측 어려움
    - 다중 프로그래밍에 적합
    - 예: FCFS, SJF (Non-preemptive)


[선점형 스케줄링 (Preemptive)]

    운영체제가 실행 중인 프로세스로부터 강제로 CPU 회수

    시간 ->
    |--Q--|--Q--|--Q--|--Q--|--Q--|
    [ P1 ][ P2 ][ P3 ][ P1 ][ P2 ]
       ^                ^
       |__ 타이머 인터럽트로 강제 회수 __|

    특징:
    - 문맥 교환 오버헤드 높음
    - 응답 시간 보장 가능
    - 시분할 시스템에 필수
    - 예: Round Robin, SRTF, Priority (Preemptive)
```

#### 선점형 스케줄링의 구현 메커니즘

```
+------------------------------------------------------------------+
|                  선점형 스케줄링 구현: 타이머 인터럽트               |
+------------------------------------------------------------------+

    하드웨어 타이머 (Timer Chip, e.g., 8254, HPET)
         |
         | 주기적 인터럽트 발생 (예: 1ms마다)
         v
    +------------------+
    |  타이머 인터럽트  |
    |  핸들러          |
    +--------+---------+
             |
             v
    +------------------+
    | 1. 현재 프로세스 |
    |    실행 시간 증가 |
    +--------+---------+
             |
             v
    +------------------+
    | 2. Quantum 만료? |
    |    (time >= Q)   |
    +--------+---------+
             |
        +----+----+
        |         |
       Yes        No
        |         |
        v         v
    +-------+  +-------+
    | 3.    |  | 계속  |
    | 스케줄|  | 실행  |
    | 러 호출|  +-------+
    +---+---+
        |
        v
    +------------------+
    | 4. 문맥 교환     |
    |    (Context      |
    |     Switch)      |
    +------------------+
```

### 3. 시분할 시스템의 메모리 관리

#### 스와핑 (Swapping) 기법

```
+================================================================+
|                 시분할 시스템의 스와핑                            |
+================================================================+

    문제: 메모리 크기 < 동시 사용자 수 x 프로세스 크기

    해결: 스와핑 (Swapping)
    - 실행 중인 프로세스만 메모리에 유지
    - 대기 중인 프로세스는 디스크로 스왑 아웃


    메인 메모리                    디스크 (Swap Area)
    +------------------+          +------------------+
    |      커널        |          |                  |
    +------------------+          |   Swapped-out    |
    |    Process A     |  <--->   |   Processes      |
    |    (실행 중)      |  swap   |                  |
    +------------------+          |   [Process B]    |
    |    Process C     |          |   [Process D]    |
    |    (실행 중)      |          |   [Process E]    |
    +------------------+          |                  |
    |    여유 공간      |          |                  |
    +------------------+          +------------------+


    스와핑 동작 시나리오:

    1. P1의 Quantum 만료
       - P1을 스왑 아웃 (메모리 -> 디스크)
       - P2를 스왑 인 (디스크 -> 메모리)
       - P2 실행 시작

    2. 메모리 부족 시
       - 가장 오랫동안 비활성 프로세스 선택
       - 해당 프로세스 스왑 아웃
       - 새 프로세스를 위한 공간 확보
```

#### 스와핑 오버헤드 분석

```
    스와핑 비용:
    +------------------------------------------+
    | 스왑 아웃 시간 = 프로세스 크기 / 디스크 쓰기 속도 |
    | 스왑 인 시간 = 프로세스 크기 / 디스크 읽기 속도  |
    +------------------------------------------+

    예시:
    - 프로세스 크기: 100MB
    - 디스크 전송 속도: 100MB/s
    - 스왑 아웃 시간: 1초
    - 스왑 인 시간: 1초
    - 총 스와핑 시간: 2초

    최적화 기법:
    1. 작은 Time Quantum 사용 (빈번한 스와핑 방지)
    2. 우선순위 기반 스와핑 (낮은 우선순위 프로세스 먼저)
    3. 페이지드 스와핑 (전체가 아닌 일부만 스왑)
    4. SSD 사용 (HDD 대비 10x 빠른 스와핑)
```

#### Linux CFS (Completely Fair Scheduler)

```
+------------------------------------------------------------------+
|             Linux CFS: 현대적 시분할 스케줄러                       |
+------------------------------------------------------------------+

    핵심 개념: "완전한 공정성"
    - 모든 프로세스에게 공평한 CPU 시간 배분
    - 가상 런타임(vruntime) 기반 스케줄링

    Red-Black Tree 구조:
                    +-------+
                    | min   | <- 다음 실행할 프로세스
                    | vruntime
                    +-------+
                        |
            +-----------+-----------+
            |                       |
       +----v----+             +----v----+
       | Task A  |             | Task B  |
       | vruntime|             | vruntime|
       | = 100   |             | = 150   |
       +---------+             +---------+
            |                       |
       +----+----+             +----+----+
       |         |             |         |
    +--v--+   +--v--+      +--v--+   +--v--+
    |TaskC|   |TaskD|      |TaskE|   |TaskF|
    | 200 |   | 250 |      | 180 |   | 300 |
    +-----+   +-----+      +-----+   +-----+

    스케줄링 원리:
    1. 항상 최소 vruntime을 가진 프로세스 선택
    2. 실행 시간 동안 vruntime 증가
    3. 우선순위가 높을수록 vruntime 증가율 낮음
    4. 공정한 CPU 시간 배분 보장
```

## 실무 적용

### 1. 서버 환경에서의 시분할 시스템 튜닝

```
    Linux 서버 시분할 파라미터 조정:

    1. CPU 스케줄러 클래스 선택
    # echo fair > /sys/kernel/debug/sched_features

    2. 타임 슬라이스 조정 (CFS)
    # sysctl -w kernel.sched_min_granularity_ns=10000000
    # sysctl -w kernel.sched_wakeup_granularity_ns=15000000

    3. 실시간 프로세스 우선순위
    # chrt -f -p 50 <pid>   # FIFO 실시간 스케줄링
    # chrt -r -p 50 <pid>   # RR 실시간 스케줄링

    4. CPU 선호도 (Affinity) 설정
    # taskset -c 0,1,2,3 <command>

    5. 사용자별 CPU 시간 제한
    # ulimit -t 3600  # 최대 1시간 CPU 시간
```

### 2. 클라우드 환경에서의 시분할 자원 관리

```
+------------------------------------------------------------------+
|             클라우드 가상머신 시분할 자원 할당                       |
+------------------------------------------------------------------+

    [물리 서버]
    +--------------------------------------------------+
    |  하이퍼바이저 (Hypervisor)                         |
    |  +------------------------------------------+    |
    |  |            CPU 스케줄러                    |    |
    |  |  - vCPU 할당                              |    |
    |  |  - Time Sharing across VMs                |    |
    |  +------------------------------------------+    |
    |                      |                           |
    |       +--------------+--------------+            |
    |       |              |              |            |
    |  +----v----+    +----v----+    +----v----+      |
    |  |  VM 1   |    |  VM 2   |    |  VM 3   |      |
    |  | 2 vCPU  |    | 4 vCPU  |    | 2 vCPU  |      |
    |  | 4GB RAM |    | 8GB RAM |    | 4GB RAM |      |
    |  +---------+    +---------+    +---------+      |
    +--------------------------------------------------+

    vCPU 스케줄링 방식:
    1. Share-based: 비율 기반 CPU 시간 배분
    2. Reserve: 최소 보장 CPU 시간
    3. Limit: 최대 허용 CPU 시간

    예시 (VMware):
    - VM1: shares=1000, reserve=500MHz, limit=2000MHz
    - VM2: shares=2000, reserve=1000MHz, limit=4000MHz
    - VM3: shares=1000, reserve=500MHz, limit=2000MHz
```

### 3. 실시간 시스템에서의 시분할 제약

```
    실시간 시스템 (Real-Time System) 요구사항:
    - Hard Real-Time: 데드라인 절대 보장 (예: 항공우주, 의료기기)
    - Soft Real-Time: 데드라인 통계적 보장 (예: 비디오 스트리밍)

    시분할 시스템의 한계:
    - 일반적인 라운드 로빈은 실시간 보장 불가
    - 선점 지연 (Preemption Latency) 존재
    - 문맥 교환 오버헤드로 인한 지터 (Jitter)

    실시간 보장을 위한 수정:
    1. 우선순위 기반 선점형 스케줄링 (Priority Preemption)
    2. Priority Inheritance Protocol (우선순위 역전 방지)
    3. CPU 리저베이션 (CPU Reservation)
    4. 할당된 Time Budget 보장

    Linux PREEMPT_RT 패치:
    - 커널 선점성 향상
    - 인터럽트 핸들러를 스레드로 변환
    - 실시간 뮤텍스 (Priority Inheritance)
    - 지연 시간 마이크로초 단위로 단축
```

## 기술사 시험 포인트

### 1. 핵심 암기 사항

```
+------------------------------------------------------------------+
|              시분할 시스템 핵심 정리                                 |
+------------------------------------------------------------------+

1. 정의: CPU 시간을 Time Quantum 단위로 분할하여 다수 사용자에게 할당

2. 핵심 목표:
   - 응답 시간 최소화 (Minimize Response Time)
   - 인터랙티브 환경 제공 (Interactive Environment)

3. 필수 구성요소:
   - 선점형 스케줄링 (Preemptive Scheduling)
   - 타이머 인터럽트 (Timer Interrupt)
   - 문맥 교환 (Context Switch)
   - 스와핑 (Swapping)

4. 대표 스케줄링 알고리즘:
   - 라운드 로빈 (Round Robin)
   - 다단계 피드백 큐 (Multilevel Feedback Queue)

5. 다중 프로그래밍과의 차이:
   - 다중 프로그래밍: CPU 활용도 극대화 (Throughput 중심)
   - 시분할 시스템: 응답 시간 최소화 (Response Time 중심)
```

### 2. 빈출 문제 유형

```
1. 시분할 시스템의 특징 설명
   Q: 시분할 시스템의 핵심 목표와 다중 프로그래밍과의 차이점 설명
   A:
   - 핵심 목표: 응답 시간 최소화, 인터랙티브 환경 제공
   - 다중 프로그래밍: CPU 활용도 극대화, 일괄 처리 중심
   - 시분할: 선점형 스케줄링 필수, Time Quantum 기반 전환

2. 라운드 로빈 스케줄링 계산
   Q: 프로세스 집합에 대한 평균 반환 시간, 대기 시간 계산
   A: Time Quantum 기반 실행 순서 도출 후 각 지표 계산

3. Time Quantum 크기 결정 요인
   Q: Time Quantum이 너무 작거나 큰 경우의 문제점
   A:
   - 너무 작음: 문맥 교환 오버헤드 급증, CPU 효율 저하
   - 너무 큼: FCFS와 동일, 응답 시간 악화

4. 선점형 vs 비선점형 스케줄링 비교
   Q: 각 방식의 특징과 적용 분야
   A: 선점형(시분할, 실시간), 비선점형(일괄처리)

5. 인터랙티브 시스템 응답 시간 기준
   Q: 사용자 체감 기준 응답 시간 분류
   A: 0.1초(즉각적), 1초(자연스러움), 10초(주의 유지 한계)
```
