# 818. NVGRE (Network Virtualization using Generic Routing Encapsulation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: NVGRE는 가상 머신의 Layer 2 프레임을 Layer 3 패킷(IP) 안에 GRE(Generic Routing Encapsulation) 프로토콜을 사용하여 캡슐화하는 클라우드 네트워크 가상화 기술이다.
> 2. **가치**: 기존 VLAN의 4,096개 논리망 한계를 1,600만 개 이상(24-bit VSID)으로 확장하여 대규모 멀티 테넌트(Multi-tenant) 클라우드 데이터센터 인프라를 가능하게 한다.
> 3. **융합**: 마이크로소프트(MS)가 주도하여 Hyper-V 환경에 깊이 통합되어 있으며, VXLAN과 경쟁 구도를 가지나 개념적(Overlay Network) 원리는 동일하다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
NVGRE(Network Virtualization using Generic Routing Encapsulation)는 이더넷 MAC 프레임(L2)을 IP 패킷(L3) 내부의 페이로드로 포장(Encapsulation)하여 전송함으로써, 물리적 네트워크 인프라(Underlay) 위에 수많은 독립적인 가상 네트워크(Overlay)를 생성하는 네트워크 가상화 표준 규격(RFC 7637)이다.

### 💡 비유
NVGRE는 **"택배 상자 안에 편지봉투 통째로 집어넣기"**와 같다.
편지봉투(L2 프레임)에는 가상 머신(VM)들끼리의 주소(MAC)만 적혀 있다. 이 편지봉투만으로는 서울에서 부산(다른 라우터 너머)으로 갈 수 없다. 그래서 편지봉투를 커다란 택배 상자(IP 패킷 + GRE 헤더)에 넣고, 상자 겉면에는 실제 물류센터(물리 서버 호스트)의 IP 주소를 적어 배달한다. 도착하면 상자를 뜯고(디캡슐레이션) 안에 있는 편지봉투를 가상 머신에 전달한다.

### 등장 배경 및 발전 과정

#### 1. 기존 기술(VLAN)의 치명적 한계점
- **ID 개수 한계**: 전통적인 IEEE 802.1Q VLAN은 ID 필드가 12비트이므로 최대 4,096개의 논리 네트워크만 만들 수 있다. 수만 명의 고객(테넌트)을 수용해야 하는 클라우드에서는 턱없이 부족했다.
- **L2 네트워크의 경계**: VLAN은 라우터를 넘어갈 수 없는 L2 기술이다. 따라서 데이터센터의 물리적 스위치(Spine-Leaf) 구조와 클라우드의 가상 네트워크 확장이 충돌하는 구조적 병목이 발생했다.

#### 2. 패러다임 변화
이 한계를 극복하기 위해 오버레이 네트워크(Overlay Network) 기술이 등장했다. Cisco/VMware는 UDP 기반의 **VXLAN**을 내놓았고, Microsoft/Intel/Dell은 기존에 이미 라우팅 환경에서 검증된 터널링 프로토콜인 GRE를 활용한 **NVGRE**를 IETF에 제안하며 시장이 양분(이후 VXLAN이 우세해짐)되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (표)

| 요소명 | 상세 역할 | 내부 동작 메커니즘 | 관련 프로토콜 | 비유 |
|--------|----------|-------------------|---------------|------|
| **NVE (Network Virtualization Edge)** | 캡슐화 및 디캡슐레이션 수행 | 가상 스위치(vSwitch)에서 L2 프레임을 가로채어 NVGRE 터널로 포장 | Hyper-V vSwitch | 택배 포장/해체 센터 |
| **VSID (Virtual Subnet ID)** | 가상 네트워크 식별 (24-bit) | 테넌트를 구분하여 서로 통신하지 못하도록 완벽히 격리 | GRE Key Field | 고객 전용 고유 코드 |
| **Outer IP Header** | 물리적 네트워크 라우팅 | 언더레이 망의 출발지/목적지 Host 물리 IP 할당 | IPv4 / IPv6 | 택배 상자 겉면 주소 |
| **Inner MAC Frame** | 가상 머신 간의 실제 통신 | VM이 생성한 원본 이더넷 프레임 | Ethernet II | 편지봉투 (내용물) |

### 심층 동작 원리 (Step-by-Step)

#### ① 패킷 생성 및 캡슐화 (Encapsulation)
- 가상 머신(VM 1)이 동일한 서브넷에 있는 다른 호스트의 가상 머신(VM 3)으로 이더넷 프레임을 보낸다.
- 하이퍼바이저의 가상 스위치(NVE)가 이 프레임을 가로챈다.
- NVE는 매핑 테이블을 확인하여 목적지 VM 3이 Host B(10.0.0.2)에 있음을 확인하고, 해당 테넌트의 고유 ID인 VSID를 할당하여 GRE 헤더를 덧붙인다.
- 그 겉면에 Host A에서 Host B로 향하는 IP 헤더와 물리 MAC 헤더를 씌워 전송한다.

#### ② L3 언더레이 전송
- 물리적 라우터와 스위치(Spine-Leaf)들은 안에 들어있는 VM의 MAC 주소나 IP 주소는 전혀 볼 수 없다.
- 오직 겉면에 적힌 Outer IP(Host A -> Host B)만 보고 패킷을 라우팅한다. 이 덕분에 브로드캐스트 스톰(Broadcast Storm)이 방지되고 L2 네트워크를 무한정 넓게 쓸 수 있다.

#### ③ 디캡슐레이션 및 전달 (Decapsulation)
- Host B의 NVE가 패킷을 수신하면, Outer 헤더와 GRE 헤더를 제거(뜯어냄)한다.
- GRE 헤더에 있던 VSID를 확인하여 이 패킷이 어떤 테넌트의 망인지 확인한다.
- 분리된 원본 L2 프레임을 가상 스위치를 통해 목적지인 VM 3에게 전달한다.

## Ⅲ. 융합 비교 및 다각도 분석

### NVGRE vs VXLAN 심층 비교

| 비교 지표 | NVGRE | VXLAN |
|-----------|-------|-------|
| **표준화** | RFC 7637 (Microsoft, Intel 주도) | RFC 7348 (VMware, Cisco 주도) |
| **전송 프로토콜**| IP 프로토콜 47 (GRE) 기반 | UDP 기반 (포트 4789) |
| **네트워크 식별**| 24-bit VSID | 24-bit VNI |
| **ECMP 로드밸런싱**| 지원 어려움 (Outer IP 헤더만으로 해시 계산 시 트래픽 쏠림) | 매우 우수 (UDP Source Port를 해시값으로 변형하여 분산) |
| **오버헤드** | 헤더 크기가 상대적으로 작음 | UDP 헤더 포함으로 약간 더 큼 |
| **시장 점유율**| Microsoft Azure, System Center 제한적 | 클라우드 산업 표준(De facto)으로 압승 |

**과목 융합 (OS + 네트워크 구조)**: 
NVGRE는 TCP/UDP 계층 없이 순수 IP 위에 GRE 프로토콜(Protocol 47)로 얹혀 전송된다. 물리 네트워크의 스위치들은 4계층 포트 번호를 기반으로 ECMP(Equal-Cost Multi-Path) 해시 로드밸런싱을 수행하는데, NVGRE는 포트 번호가 없기 때문에 데이터센터 내부에서 병렬 회선(Spine 스위치)으로 트래픽을 골고루 분산시키기가 매우 까다롭다는 치명적 구조적 단점을 지녔다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 레거시 Microsoft Hyper-V 기반의 프라이빗 클라우드를 운영 중인 기업이 네트워크 격리 한계(VLAN 4K 초과)에 도달하여 오버레이 네트워크를 도입하려 함.
**판단**:
1. **아키텍처 결정**: 인프라가 100% Windows Server / System Center(VMM) 기반이라면 MS에 최적화된 **NVGRE**를 도입하는 것이 통합 관리 측면에서 유리함.
2. **이종 환경 통합 시**: 만약 VMware(ESXi)나 오픈소스(K8s, OpenStack) 환경과 하이브리드 클라우드를 구성해야 한다면, 벤더 종속적인 NVGRE 대신 산업 표준으로 자리 잡은 **VXLAN** 기반의 컨트롤 플레인(BGP EVPN)으로 전면 마이그레이션을 결단해야 함.
3. **하드웨어 오프로드 검토**: NVGRE 캡슐화는 호스트 CPU 자원을 많이 소모하므로, NVGRE Task Offload 기능을 지원하는 물리적 네트워크 인터페이스 카드(SmartNIC)의 도입이 필수적.

### 주의사항 및 안티패턴 (Anti-patterns)
- **MTU 불일치 문제 (Fragmentation)**: NVGRE는 원본 패킷에 헤더를 추가하므로 패킷 크기가 커진다. 언더레이 스위치의 점보 프레임(Jumbo Frame, MTU 9000 등)을 설정하지 않으면 파편화(Fragmentation)가 발생하여 네트워크 성능이 극심하게 저하된다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **확장성**: 4,000개에서 **1,600만 개 이상**의 격리된 가상 네트워크 생성 지원.
- **민첩성**: 물리 스위치(VLAN) 설정 변경 없이 호스트(하이퍼바이저) 수준의 소프트웨어 설정만으로 수 초 내에 네트워크 토폴로지 구축 및 마이그레이션(Live Migration) 보장.

### 미래 전망 및 결론
NVGRE는 클라우드 초기 네트워크 가상화를 이끈 훌륭한 아키텍처지만, 로드밸런싱(ECMP)의 한계와 산업 생태계의 패권 경쟁에서 VXLAN에 밀려 현재는 MS 진영 내부 솔루션으로 축소되었다. 그러나 클라우드 네이티브 환경에서 **오버레이 터널링(Overlay Tunneling)과 언더레이(Underlay) 물리망 분리**라는 SDN(Software Defined Networking)의 핵심 철학을 실현한 주요 기술로서 아키텍처적 가치가 매우 크다.

## 👶 어린이를 위한 3줄 비유 설명
1. **NVGRE가 뭔가요?**: 서로 다른 동네에 사는 컴퓨터끼리 비밀 편지를 주고받을 때, 진짜 주소를 숨기기 위해 커다란 택배 상자로 편지봉투를 통째로 포장하는 기술이에요.
2. **왜 그렇게 하나요?**: 옛날 방식(VLAN)은 구분할 수 있는 동네 이름표가 4,000개밖에 없었는데, 이 택배 상자 방식(NVGRE)을 쓰면 1,600만 개의 동네를 만들 수 있어서 클라우드처럼 큰 곳에서 아주 유용해요.
3. **어떻게 작동하나요?**: 보낼 때 포장(캡슐화)해서 고속도로(물리 네트워크)를 달리고, 도착지에 오면 경비원(가상 스위치)이 상자를 뜯고 진짜 편지를 안전하게 전달해 준답니다!
