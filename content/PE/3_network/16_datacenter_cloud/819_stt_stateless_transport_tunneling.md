# 819. STT (Stateless Transport Tunneling)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: STT는 가상 머신의 L2 프레임을 가짜 TCP 헤더를 씌워 IP 패킷 안에 캡슐화하는 오버레이 네트워크(Overlay Network) 터널링 프로토콜이다.
> 2. **가치**: TCP 헤더를 사용함으로써, 현대 물리적 네트워크 인터페이스 카드(NIC)에 이미 널리 내장된 TSO(TCP Segmentation Offload) 및 LRO(Large Receive Offload) 하드웨어 가속 기능을 그대로 활용하여 호스트 CPU의 패킷 처리 부하를 극적으로 줄인다.
> 3. **융합**: VMware의 NSX 구조(Nicira 인수 기반)에서 적극 활용되었으나, 연결 상태(State)를 추적하지 않는 '가짜 TCP'라는 특성 탓에 방화벽 등 중간 장비와의 호환성 문제가 불거져 현재는 VXLAN 계열에 주도권을 내어준 기술이다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
STT(Stateless Transport Tunneling)는 클라우드 데이터센터에서 네트워크 가상화를 구현하기 위해 이더넷 프레임을 TCP/IP 터널 안으로 캡슐화하는 기술이다. 이름의 'Stateless'가 의미하듯, 겉보기에는 TCP 패킷처럼 생겼지만 실제로는 TCP의 3-Way Handshake나 연결 상태(State) 추적, 재전송 매커니즘을 전혀 수행하지 않는 가상 데이터그램 프로토콜이다.

### 💡 비유
STT는 **"우편 배달부의 꼼수"**와 같다.
택배 회사(물리 NIC)는 '책' 모양의 상자(TCP 패킷)만 보면 자동으로 기계(TSO)가 알아서 빠르고 예쁘게 여러 개로 나눠서 포장해 주는 최첨단 기계를 갖고 있다. 그래서 STT는 전혀 상관없는 잡동사니(VM의 L2 프레임)를 보내면서, 겉포장만 책(가짜 TCP 헤더)처럼 꾸며서 기계에 넣는다. 기계는 그것이 진짜 책인 줄 알고 엄청난 속도로 분할 처리해 주는 원리다.

### 등장 배경 및 발전 과정

#### 1. VXLAN 및 NVGRE의 성능적 한계
초기 오버레이 기술인 VXLAN(UDP 기반)과 NVGRE(GRE 기반)는 소프트웨어 가상 스위치(vSwitch)에서 패킷을 캡슐화할 때 호스트 CPU 자원을 엄청나게 소모했다. 당시 대부분의 물리 NIC 하드웨어는 TCP 트래픽만 가속(Offload)해 줄 뿐, UDP나 GRE로 캡슐화된 패킷은 가속해 주지 않았기 때문에 10Gbps 망에서도 가상 네트워크 성능이 1~2Gbps 수준으로 곤두박질치는 현상이 발생했다.

#### 2. Nicira의 창의적 해결책 (STT의 등장)
SDN 선구자인 Nicira(이후 VMware에 인수됨)는 소프트웨어 레벨에서 하드웨어 가속을 끌어내기 위한 해킹에 가까운 아이디어를 냈다. "기존 NIC가 TCP만 가속한다면, 캡슐화된 패킷 겉면에 가짜 TCP 헤더를 붙이자!" 이것이 STT의 탄생 배경이며, 이 아이디어 덕분에 물리적 장비 교체 없이도 가상 네트워크 성능을 와이어 스피드(Line-rate) 가까이 끌어올릴 수 있었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (표)

| 요소명 | 상세 역할 | 내부 동작 메커니즘 | 관련 기술 | 비유 |
|--------|----------|-------------------|-----------|------|
| **Fake TCP Header** | 하드웨어 가속기 기만 | TCP 포맷을 모방하여 NIC의 TSO/LRO 엔진이 패킷을 분할/병합하도록 유도 | TCP Header | 책 표지 위장 |
| **STT Header** | 가상 네트워크 식별 및 메타데이터 | 64비트 Context ID(VNI 역할)를 포함하여 패킷 조립 시 참조됨 | Overlay ID | 택배 내용물 태그 |
| **TSO (TCP Segmentation Offload)** | 송신 가속 | vSwitch가 보낸 64KB의 거대한 패킷을 물리 NIC가 MTU(1500B) 크기로 쪼갬 | NIC Hardware | 자동 절단기 |
| **LRO (Large Receive Offload)** | 수신 가속 | 물리 NIC가 잘게 쪼개져 들어온 패킷들을 64KB 크기로 병합하여 vSwitch로 올림 | NIC Hardware | 자동 조립기 |

### 심층 동작 원리 (Stateless의 비밀)

#### ① TSO/LRO 트리거링
VM이 매우 큰 데이터를 보내면, 하이퍼바이저의 vSwitch(OVS 등)는 이를 자르지 않고 거대한 덩어리(예: 64KB) 그대로 STT 헤더와 TCP 헤더를 씌운다. 이때 TCP 시퀀스 번호(Sequence Number)는 STT 헤더와 데이터의 오프셋을 계산하여 정교하게 조작된다. 물리 NIC는 이를 일반적인 TCP 세션으로 착각하고 고속 칩셋을 사용하여 패킷을 잘게 쪼갠 후(TSO), 각 조각마다 복제된 TCP 헤더를 붙여 네트워크로 방출한다.

#### ② 왜 "Stateless"인가?
일반적인 TCP 통신은 SYN-ACK 핸드셰이크를 거치고 연결 상태(State)를 커널의 소켓 테이블에 유지해야 한다. 하지만 STT는 커널 TCP 스택을 완전히 우회하여 vSwitch에서 곧바로 조작된 TCP 헤더를 씌워 전송한다. 당연히 재전송이나 혼잡 제어도 수행하지 않는다. (패킷 손실 처리는 캡슐화 내부에 있는 원본 VM 패킷의 프로토콜에 맡긴다).

## Ⅲ. 융합 비교 및 다각도 분석

### STT vs VXLAN 심층 비교

| 비교 지표 | STT (Stateless Transport Tunneling) | VXLAN (Virtual eXtensible LAN) |
|-----------|-------------------------------------|--------------------------------|
| **기반 프로토콜** | TCP (Stateless 속임수) | UDP |
| **등장 목적** | 기존 하드웨어(NIC)의 TSO/LRO 기능을 소프트웨어 수정한 것만으로 재활용 | 터널링 표준화 및 ECMP 로드밸런싱 극대화 |
| **하드웨어 의존도**| 레거시 하드웨어에서 엄청난 성능 향상 | 초기엔 CPU 부하가 컸으나, 이후 NIC 벤더들이 VXLAN Offload를 기본 지원함 |
| **방화벽 호환성** | **매우 나쁨** (중간 방화벽이 TCP 핸드셰이크 없는 패킷을 비정상 트래픽으로 간주하여 Drop 시킴) | 우수 (일반적인 UDP 포트 4789로 인식) |
| **ECMP 분산** | TCP Source Port 해싱으로 우수함 | UDP Source Port 해싱으로 우수함 |
| **현재 위상** | 사장됨 (VXLAN 패권 확립) | 클라우드 인프라의 글로벌 표준 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 과거 VMware NSX를 초기 버전으로 구축한 데이터센터에서, 다른 존(Zone)으로 트래픽이 넘어갈 때 중간에 위치한 보안 장비(IPS/차세대 방화벽)에서 원인 모를 패킷 드랍이 대량 발생함.
**판단 및 원인 규명**:
1. **문제의 원인**: NSX의 기본 오버레이 프로토콜로 설정된 **STT** 트래픽을 중간 방화벽이 검사하는 과정에서 발생. 방화벽은 엄격한 상태 기반 감시(Stateful Inspection)를 하므로, TCP SYN 과정 없이 갑자기 나타난 STT의 가짜 TCP 패킷을 **"세션 하이재킹 또는 비정상 스푸핑"** 공격으로 오인하여 차단(Drop)해 버린 것임.
2. **해결 전략**: 
   - 임시 조치: 방화벽 정책에서 해당 트래픽의 TCP 상태 검사를 예외 처리(State-bypass)함.
   - 근본 조치: NSX 컨트롤러 설정을 변경하여 터널링 프로토콜을 **VXLAN 또는 GENEVE**로 마이그레이션함. 물리 NIC 펌웨어 업데이트를 통해 VXLAN Offload를 활성화하여 STT 없이도 성능을 보장함.

### 주의사항 및 안티패턴 (Anti-patterns)
- **미들박스(Middlebox)의 덫**: 프로토콜을 "해킹"하여 다른 용도로 쓰는 방식(TCP를 Stateless로 쓰기)은 반드시 네트워크 중간에 있는 지능형 장비(NAT, Firewall, IDS)와의 충돌을 유발한다. 이는 네트워크 아키텍처 설계 시 가장 지양해야 할 프로토콜 오용(Misuse)의 대표적 사례다.

## Ⅴ. 기대효과 및 결론

### 기술적 기여도
STT 자체는 시장에서 도태되었으나, SDN(Software Defined Networking) 초기 시절에 **"소프트웨어의 영리함으로 하드웨어의 한계를 어떻게 극복할 수 있는가"**를 보여준 기념비적인 프로토콜이다. STT 덕분에 vSwitch 기반의 오버레이 네트워크가 성능 문제로 실패하지 않고 시장에 안착할 수 있는 징검다리 역할을 톡톡히 해냈다.

### 미래 전망 및 진화 방향
현재 클라우드 데이터센터는 STT의 꼼수를 더 이상 사용하지 않는다. Intel, Mellanox(NVIDIA), Broadcom 등의 스마트 NIC(SmartNIC, DPU)가 하드웨어 레벨에서 VXLAN과 최신 확장 프로토콜인 **GENEVE (Generic Network Virtualization Encapsulation)** 오프로딩을 완벽하게 지원하게 되었기 때문이다. 이제 네트워크 가상화의 패러다임은 프로토콜 해킹을 넘어 DPU 가속을 통한 순수 하드웨어 처리(Hardware Offload) 시대로 진입했다.

## 👶 어린이를 위한 3줄 비유 설명
1. **STT가 뭔가요?**: 가짜 상자 껍데기를 씌우는 마술이에요. 우체국 기계가 '네모난 상자(TCP)'만 보면 엄청 빠르게 배달해 주니까, 장난감(가상머신 데이터)을 네모난 상자처럼 꾸며서 몰래 보내는 거예요.
2. **왜 그렇게 했나요?**: 기계를 새로 사려면 돈이 너무 많이 드니까, 똑똑한 엔지니어들이 옛날 기계를 속여서 최신 기계처럼 빠르게 쓰려고 머리를 쓴 거랍니다.
3. **왜 지금은 안 쓰나요?**: 중간에 깐깐한 경찰관(방화벽)이 "어? 이 상자는 겉은 네모난데 진짜 상자가 아니잖아!" 하고 버려버리는 문제가 생겨서, 지금은 아예 진짜 좋은 기계를 새로 사서(VXLAN/DPU) 쓴답니다!
