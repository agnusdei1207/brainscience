# 822. 컨테이너 네트워킹 인터페이스 (CNI, Container Network Interface)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 쿠버네티스(Kubernetes)나 컨테이너 런타임이 컨테이너(Pod)를 생성하거나 삭제할 때, 네트워크 인터페이스를 연결하고 IP 주소를 할당/제거하기 위해 호출하는 일관된 API 표준 규격이다.
> 2. **가치**: 컨테이너 오케스트레이션 플랫폼(K8s)과 실제 네트워크 구현체(Calico, Cilium 등)를 완벽히 분리(Decoupling)하여, 사용자가 자신의 환경에 맞는 최적의 네트워크 솔루션을 레고 블록처럼 갈아 끼울 수 있는 플러그인 생태계를 창조했다.
> 3. **융합**: 가상 이더넷(veth), 오버레이 터널링(VXLAN), BGP 라우팅, 그리고 최신 eBPF 커널 기술과 융합되어 클라우드 네이티브 네트워크의 코어 엔진 역할을 수행한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
CNI(Container Network Interface)는 CNCF(Cloud Native Computing Foundation) 산하의 프로젝트로, 리눅스 컨테이너 환경에서 네트워크를 설정하는 데 필요한 실행 파일(Binary)의 인터페이스 형식과 설정 파일 포맷(JSON)만을 정의한 명세서(Specification)이다. CNI 자체는 네트워크를 만들지 않으며, 이 명세를 준수하여 개발된 **CNI 플러그인**들이 실제 네트워크 생성 작업을 대행한다.

### 💡 비유
CNI는 **"건축의 범용 수도 배관 규격"**과 같다.
쿠버네티스(아파트 관리실)가 새 집(컨테이너)을 지을 때, 집에 물(네트워크)이 나와야 한다. 관리실은 직접 배관 공사를 하지 않는다. 대신 "우리 아파트는 15mm 파이프 규격(CNI 표준)을 쓴다"고 공지하고 배관공(CNI 플러그인)을 부른다. Calico 배관공은 튼튼한 쇠파이프(라우팅)로, Flannel 배관공은 부드러운 고무호스(오버레이)로 연결을 해준다. 관리실은 누가 어떤 방식으로 연결하든 '물만 나오면' 만족한다.

### 등장 배경 및 발전 과정

#### 1. 도커(Docker) 네트워크의 폐쇄성 (CNM의 한계)
초기 도커는 자체 네트워크 모델인 CNM(Container Network Model)을 사용했다. 이는 도커 엔진에 너무 꽉 결합되어 있어서 다양한 네트워크 벤더들이 자신들의 솔루션을 적용하기가 매우 까다로웠다.

#### 2. K8s와 CNI의 승리
CoreOS가 주도하여 만든 CNI는 극도로 단순한 철학("인터페이스를 추가하고 뺀다" 단 두 가지 명령)을 가졌다. 쿠버네티스가 도커의 CNM 대신 CNI를 공식 표준으로 채택하면서, 수십 개의 네트워크 벤더(Cisco, VMware, AWS 등)가 앞다투어 CNI 플러그인을 개발하게 되었고, 현재 클라우드 네이티브 생태계의 절대적 표준이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (표)

| 요소명 | 상세 역할 | 내부 동작 메커니즘 | 관련 주체 | 비유 |
|--------|----------|-------------------|-----------|------|
| **Container Runtime** | 컨테이너 생명주기 관리 및 CNI 호출 | 컨테이너의 네트워크 네임스페이스(NetNS)를 생성 후 CNI 바이너리 실행 | Kubelet, containerd | 아파트 관리실 |
| **CNI Configuration** | 네트워크 속성 정의 | 사용할 플러그인 이름, 서브넷(IP 대역), 라우팅 정보를 담은 JSON 파일 | `/etc/cni/net.d/` | 공사 설계도 |
| **CNI Plugin (Main)** | 실제 네트워크 인터페이스 구성 | 가상 이더넷(veth) 브릿지 생성 및 컨테이너 NetNS에 꽂기 | 브릿지(bridge), ipvlan | 배관 설치공 |
| **IPAM Plugin** | IP 주소 할당 및 관리 | 중복되지 않게 IP 주소(IPAM)를 발급하고 해제(DHCP 또는 Host-local) | host-local, dhcp | 동호수 부여팀 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ Kubernetes Pod Creation & CNI Execution Flow ]
================================================================================

 1. Kubelet (Worker Node Agent)
    │ "API 서버로부터 Pod 배포 명령 수신"
    ▼
 2. Container Runtime (containerd / CRI-O)
    │ "Pause 컨테이너 실행 및 Network Namespace (NetNS) 생성"
    ▼
 3. CNI Call (실행 파일 호출)
    │  $ CNI_COMMAND=ADD CNI_NETNS=/var/run/netns/pod_ns CNI_IFNAME=eth0 \
    │    /opt/cni/bin/calico < /etc/cni/net.d/10-calico.conflist
    ▼
 ┌────────────────────────────────────────────────────────────────────────┐
 │                      CNI Plugin (e.g., Calico / Flannel)               │
 │                                                                        │
 │  a. [ IPAM Plugin 호출 ]                                               │
 │     "이 Pod가 쓸 IP 하나 줘!" -> IPAM 반환: "10.244.1.5"               │
 │                                                                        │
 │  b. [ Interface & Bridge Setup ]                                       │
 │     - Node의 물리/가상 스위치(cni0)에 veth 짝꿍 하나 연결(veth1234)    │
 │     - Pod NetNS 안에 나머지 veth 짝꿍 넣고 이름 'eth0'으로 변경        │
 │     - eth0에 IP(10.244.1.5) 할당 및 라우팅 테이블(Default Gateway) 설정│
 │                                                                        │
 └────────────────────────────────────────────────────────────────────────┘
    │ 반환: { "ip4": { "ip": "10.244.1.5" }, "interfaces": [ ... ] }
    ▼
 4. Container Runtime
    │ "네트워크 셋업 완료! 이제 App 컨테이너(Nginx) 실행!"
```

### 심층 동작 원리 (ADD와 DEL의 미학)

CNI 명세는 매우 단순하다. 기본적으로 4가지 명령만 명세되어 있다: `ADD`, `DEL`, `CHECK`, `VERSION`.
- **ADD**: 컨테이너가 생성될 때 런타임은 환경변수에 식별자(Container ID)와 네트워크 네임스페이스 경로를 넣고 CNI 플러그인을 호출한다. 플러그인은 가상 케이블(veth)을 연결하고, IP를 부여한 뒤 결과를 JSON으로 리턴한다.
- **DEL**: 파드가 삭제될 때 런타임은 `DEL` 명령으로 플러그인을 부른다. 플러그인은 할당했던 IP를 반납(IPAM)하고, veth 인터페이스를 깔끔하게 지워 리소스 누수를 막는다.
- *단순함의 힘*: 이 바이너리 호출 방식 덕분에, CNI 개발자는 데몬(Daemon)을 띄우고 통신할 필요 없이 일회성 스크립트나 프로그램만 짜면 완벽히 K8s와 연동된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 주요 CNI 플러그인 구현체 비교 매트릭스

| CNI 플러그인 | 네트워크 구성 방식 (Dataplane) | 라우팅 제어 (Control plane) | 특징 및 장단점 | 주요 사용처 |
|--------------|--------------------------------|-----------------------------|----------------|-------------|
| **Flannel** | 오버레이 터널링 (VXLAN) | etcd (단순 키-밸류 저장) | 설정이 가장 쉽고 가벼움. 단, Network Policy(보안) 미지원 | 소규모 클러스터, K8s 입문용 |
| **Calico** | 언더레이 (순수 L3 라우팅) | BGP (Border Gateway Protocol)| 성능 우수. BGP를 써서 스위치와 직접 연동 가능. 강력한 보안 통제 | 대규모 상용 클러스터의 표준 |
| **Cilium** | 커널 바이패스 (eBPF) | eBPF Map | 최강의 성능(iptables 미사용). 네트워크 흐름의 엄청난 가시성 제공 | 초대형 고성능 클러스터 |
| **AWS VPC CNI**| 퍼블릭 클라우드 종속 (VPC 라우팅)| AWS ENI (Elastic Network I/F)| 오버레이가 없어 패킷 크기 손실 없음. VPC IP 고갈 위험 존재 | EKS (Amazon K8s) 기본값 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 온프레미스(On-premise) 환경에서 1,000대 규모의 노드를 가진 K8s 클러스터를 구축하려고 한다. 노드당 50개의 파드가 떠서 총 50,000개의 파드가 통신해야 한다.
**판단 및 전략적 CNI 선택**:
1. **Flannel 기각**: VXLAN 오버레이 패킷 포장 오버헤드가 크고, 부서 간 통신을 차단하는 Network Policy 기능이 없어 보안 감사를 통과할 수 없다.
2. **Calico 도입**: **L3 라우팅(BGP) 모드**로 Calico를 설정한다. 파드 패킷이 오버레이 터널링(캡슐화) 없이 노드의 라우팅 테이블을 통해 평문으로 물리 스위치로 전달되므로(Underlay 모드) 네트워크 성능 손실이 0%에 가깝다. 데이터센터의 Top-of-Rack 스위치와 Calico 노드가 직접 BGP 세션을 맺어(BGP Peering) 파드의 IP를 전사 망에 광고(Advertise)하게 만든다.
3. **Cilium 고려 (미래 대비)**: 만약 iptables의 규칙 수가 10만 개를 넘어가면서 노드의 CPU를 소모하기 시작한다면, iptables를 우회하는 **eBPF 기반의 Cilium CNI**로 전환을 검토한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **IP 대역(CIDR) 충돌**: CNI 설정 시 부여하는 파드용 IP 대역(Pod CIDR)이 회사의 기존 사내망(VPN) 대역이나 노드의 물리 IP 대역과 겹치면 패킷 라우팅 블랙홀에 빠진다. 구축 전 IP 설계(IPAM) 플래닝이 가장 중요하다.
- **여러 CNI 혼용 금지**: K8s 클러스터 내에서 여러 CNI를 섞어 쓰는 것(Multus 등 특수 목적 제외)은 극심한 라우팅 테이블 충돌을 야기한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **이식성(Portability)**: CNI라는 단일 인터페이스 덕분에, AWS(VPC CNI)에서 개발한 매니페스트(YAML)를 온프레미스(Calico)로 가져와도 단 한 줄의 코드 수정 없이 네트워크가 동작함.
- **생태계 확장**: 네트워크 장비 벤더들이 독자적인 SDN 컨트롤러를 쉽게 CNI 규격으로 맞춰 K8s 클러스터에 납품할 수 있어 인프라 비용 경쟁력 확보.

### 미래 전망 및 진화 방향
CNI 생태계는 과거 "연결성" 중심에서 "보안과 가시성(Observability)" 중심으로 진화하고 있다. **Cilium(eBPF 기반)**의 대부상에서 알 수 있듯, 향후 CNI는 단순히 IP를 할당하는 플러그인을 넘어 커널 깊숙한 곳에서 패킷을 직접 통제하고, 서비스 메시(Service Mesh)의 역할 일부(L7 라우팅 및 암호화)까지 흡수하는 클라우드 인프라의 핵심 OS처럼 동작할 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **CNI가 뭔가요?**: 아파트(서버)에 새로운 집(컨테이너)이 지어질 때마다 자동으로 수도관(네트워크)을 연결해주고 집 호수(IP 주소)를 붙여주는 똑똑한 자동 배관공 시스템이에요.
2. **왜 좋나요?**: 아파트 관리소장이 직접 파이프를 고르지 않아도 돼요. 튼튼한 쇠파이프를 쓰는 A배관공을 부르든, 고무호스를 쓰는 B배관공을 부르든 관리소장은 똑같은 명령("연결해!") 하나만 쓰면 되거든요.
3. **어떻게 일하나요?**: 집이 다 지어지면 1초 만에 파이프를 꽂아주고, 집이 부서져서 없어지면 잊지 않고 물을 잠그고 호스를 딱 정리해 준답니다!
