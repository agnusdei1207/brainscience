# 825. Cilium (실리움) - eBPF 기반 고성능 K8s CNI

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 쿠버네티스 네트워킹을 위해 전통적인 리눅스 `iptables` 규칙이나 Netfilter를 완전히 배제하고, 리눅스 커널 안에서 동적으로 코드를 실행하는 혁명적인 **eBPF(Extended Berkeley Packet Filter)** 기술을 기반으로 구축된 CNI 플러그인이다.
> 2. **가치**: 패킷이 유저 스페이스나 복잡한 커널 스택을 거치지 않고 소켓(Socket) 레벨에서 즉시 라우팅되므로, 기존 CNI 대비 레이턴시(지연)를 획기적으로 낮추고 대규모 클러스터(수만 개의 파드)에서도 성능 저하 없는 완벽한 스케일링을 보장한다.
> 3. **융합**: 단순한 네트워크 연결을 넘어, L7(HTTP, gRPC) 보안 필터링, 허블(Hubble)을 통한 네트워크 토폴로지 시각화, 그리고 `kube-proxy` 완전 대체까지 아우르는 클라우드 네이티브 네트워크 인프라의 최종 진화 형태이다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
Cilium은 컨테이너 워크로드(Kubernetes, Docker) 간의 네트워크 연결, 로드밸런싱, 투명한 보안(Security), 그리고 가시성(Observability)을 제공하는 오픈소스 소프트웨어다. 가장 큰 특징은 리눅스 커널의 슈퍼파워라 불리는 **eBPF**를 데이터 플레인(Data Plane)으로 채택하여, 커널 패닉이나 재부팅 없이도 커널의 네트워크 처리 방식을 실시간으로 프로그래밍한다는 점이다.

### 💡 비유
Cilium은 **"톨게이트를 없애고 자동차에 하이패스 칩을 심은 것"**과 같다.
기존 방식(`iptables` 기반 CNI)은 고속도로 톨게이트에 수만 장의 수배자 명단(규칙)을 붙여놓고, 차가 지나갈 때마다 경찰이 명단을 1번부터 1만 번까지 일일이 대조하는 방식이라 엄청난 정체가 발생했다. 
반면 Cilium(eBPF)은 자동차가 차고(소켓)에서 출발하려 할 때, 엔진 내부에서 즉시 규칙을 검사하고 목적지로 가는 완벽한 지름길(Direct Routing)을 알려주어 톨게이트 자체를 통과하지 않게 만든다.

### 등장 배경 및 발전 과정

#### 1. iptables의 붕괴 (O(N)의 저주)
쿠버네티스 생태계를 지탱하던 Kube-Proxy와 기존 CNI들(Flannel, Calico 초기)은 방화벽과 로드밸런싱을 위해 리눅스의 `iptables`를 사용했다. 하지만 `iptables`는 애초에 수만 개의 룰이 1초마다 바뀌는 컨테이너 환경을 위해 설계된 것이 아니었다. 파드가 5,000개가 넘어가면 룰 갱신에만 수 초가 걸리고, 규칙이 선형적으로 검색되어(O(N)) 트래픽이 폭주하면 노드 CPU가 100%를 치는 끔찍한 병목이 발생했다.

#### 2. 커널의 혁명, eBPF의 도입
2014년 리눅스 커널 3.15부터 `eBPF`가 도입되면서, 사용자(개발자)가 샌드박스화된 안전한 코드를 커널 스페이스에 밀어 넣어 실행할 수 있게 되었다. Isovalent(Cilium 창립 회사)는 이 eBPF를 K8s 네트워크에 최초로 접목하여, iptables를 완전히 들어내고 커널에서 패킷을 직접 제어하는 Cilium 프로젝트를 발표하며 클라우드 네이티브 네트워킹의 패러다임을 바꿨다.

|--------|----------|-------------------|-----------|------|
| **Cilium Agent** | 노드별 관리 데몬 | K8s API 서버와 통신하며 파드/서비스 변경 감지, eBPF 프로그램을 컴파일하여 커널에 주입 | DaemonSet | 현장 지휘관 |
| **eBPF Programs**| 초고속 데이터 플레인 | 커널의 네트워크 훅(Hook: TC, cgroup/skb)에 걸려 패킷 라우팅 및 보안 규칙 즉시 실행 | C언어로 작성, JIT 컴파일 | 지능형 센서 |
| **eBPF Maps** | 커널-유저 상태 공유 | IP/MAC 주소, 로드밸런싱 룰, 보안 정책 상태를 저장하는 커널 내부의 고속 해시 테이블 | Key-Value Store | 실시간 블랙리스트 |
| **Hubble** | 네트워크 가시성 도구 | eBPF가 수집한 L3/L4/L7 메타데이터를 UI 및 메트릭으로 시각화 | Prometheus, UI | CCTV 관제 센터 |
| **Kube-proxy Replacement**| K8s 서비스 라우팅 대행 | eBPF를 사용하여 iptables 기반의 느린 kube-proxy를 완벽히 끄고 동일 기능 수행 | Socket-based LB | 낡은 우체국 철거 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ 1. Legacy (iptables) vs 2. Cilium (eBPF) Packet Flow ]
================================================================================

[ 1. Legacy (Calico/Flannel + kube-proxy) - 엄청나게 긴 커널 스택 ]
  Pod A (App) -> [ Socket ] -> [ TCP/IP Stack ]
                                      ▼
             [ Netfilter / iptables (수만 개 룰 순차 검사!) ]  <-- 병목 지점
                                      ▼
                         [ qdisc (Traffic Control) ] -> [ eth0 (NIC) ]

[ 2. Cilium (eBPF) - 커널 스택 우회 및 다이렉트 처리 ]
  Pod A (App) -> [ Socket ] 
                    │
           < cgroup/skb Hook > (소켓 계층에서 바로 eBPF 프로그램 실행)
             - eBPF Map 조회: "목적지 IP(Pod B) 허용됨. 로드밸런싱 대상 아님"
                    │
                    ▼ (TCP/IP 스택과 iptables를 완전히 점프(Bypass)해버림!)
                    │
           < TC (Traffic Control) Hook >
             - 목적지 MAC 주소 바로 재작성
                    │
               [ eth0 (NIC) ]  --------> (물리 네트워크 전송)
```

### 심층 동작 원리

#### ① Kube-proxy 완전 대체 (eBPF Host-Routing)
기존 K8s에서는 A 파드가 `Service B`를 호출할 때, `iptables`가 패킷의 목적지 IP(VIP)를 NAT(Network Address Translation)하여 실제 B 파드 IP로 바꿨다. Cilium은 **Kube-proxy를 노드에서 아예 지워버린다**. 대신 eBPF가 소켓 계층(Socket Layer)에 훅(Hook)을 걸어두어, 애플리케이션이 `connect()` 함수를 부르는 순간 커널 eBPF Map을 조회해 VIP를 실제 IP로 즉시 변환(Socket-based LB)해 버린다. 네트워크 스택을 타기도 전에 주소가 변환되므로 압도적으로 빠르다.

#### ② L7 보안 및 API 인지 정책
Cilium은 IP/Port(L3/L4) 수준의 방화벽을 넘어 HTTP URL, 메소드, gRPC 헤더 기반의 보안 정책을 적용할 수 있다. 
- 예: `Frontend` 파드는 `Backend` 파드의 `/public` API만 `GET`으로 호출할 수 있고, `/admin` 경로나 `POST` 요청은 eBPF에서 즉각 Drop 시켜버린다. (사이드카 프록시인 Envoy와 eBPF가 협력하여 처리)

#### ③ 허블(Hubble)을 통한 심층 가시성
네트워크 장애가 났을 때 `tcpdump`를 뜰 필요가 없다. eBPF는 패킷이 통과할 때마다 1% 미만의 극미한 오버헤드로 메타데이터(어떤 파드가, 어떤 URL을 호출했고, 응답이 몇 밀리초 걸렸으며, 왜 Drop 되었는지)를 추출하여 Hubble에 전달한다. 운영자는 웹 UI를 통해 클러스터 전체의 데이터 흐름도를 직관적으로 시각화하여 볼 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### 차세대 CNI 빅매치: Cilium vs Calico

| 비교 지표 | Cilium (Isovalent) | Calico (Tigera) |
|-----------|--------------------|-----------------|
| **데이터 플레인 (Core)**| **eBPF First** (iptables 배제) | 기본 iptables (최근 eBPF 모드 추가 지원) |
| **Kube-proxy 의존성** | 완전 대체 가능 (Socket LB) | 유지 (또는 eBPF 모드에서 대체) |
| **보안 가시성** | **Hubble** (L7 레벨 상세 흐름 시각화) | Calico Enterprise (유료 버전에서 지원) |
| **라우팅 철학** | eBPF 맵을 통한 최단 경로 포워딩 | BGP 라우팅 프로토콜의 성숙함과 인프라 연동 |
| **최적 사용처** | 고성능 마이크로서비스, 완벽한 모니터링 요구 환경 | 엔터프라이즈 레거시 망(BGP)과의 완벽한 융합 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 글로벌 핀테크 서비스 K8s 클러스터. 노드가 500대, 파드가 15,000개가 뜨면서 노드의 `iptables` 룰이 10만 줄을 넘어갔다. 새로운 파드가 뜰 때마다 룰이 갱신되며 Kube-proxy 프로세스가 CPU를 80% 이상 소모하고, 신규 API 호출 시 네트워크 타임아웃이 간헐적으로 발생함.
**판단 및 전략**:
1. **마이그레이션 결정**: 네트워크 아키텍처의 한계(O(N) 탐색)에 봉착했으므로, O(1) 해시 테이블 탐색을 지원하는 **Cilium으로의 전면 CNI 교체**를 결정한다.
2. **Kube-proxy Free 모드 전환**: 클러스터에서 Kube-proxy 데몬셋을 삭제하고, Cilium의 eBPF 기반 로드밸런싱 기능을 활성화한다.
3. **Hubble 도입 (보안/운영)**: 허블을 켜서 금융 트랜잭션의 패킷 흐름(gRPC 호출 지연 시간, DNS 분석)을 실시간으로 프로메테우스(Prometheus)로 수집하여 SOC(Security Operations Center) 팀에 전달한다.
4. **결과**: 노드의 네트워크 처리 CPU 점유율이 80%에서 5% 미만으로 곤두박질치고, 네트워크 지연(Latency)의 99퍼센타일(P99) 값이 40% 이상 개선됨.

### 주의사항 및 안티패턴 (Anti-patterns)
- **최신 커널 의존성**: eBPF는 리눅스 커널과 강하게 결합되어 있다. Cilium의 모든 강력한 기능을 쓰려면 적어도 **리눅스 커널 4.19 이상(권장 5.4 이상)**이 필수적이다. 금융권 등 보수적인 기업에서 사용하는 구형 RHEL 7(커널 3.10) 같은 서버에서는 호환성 문제로 도입이 원천 차단될 수 있다. 커널 업그레이드가 최우선 선행 과제다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **처리량(Throughput) 및 지연율(Latency)**: iptables 기반 대비 네트워크 성능이 베어메탈(물리망) 수준의 99%까지 도달하며 지연율 수십 마이크로초 감소.
- **룰 스케일링 무한대**: 서비스나 파드가 10만 개로 늘어나더라도, eBPF 해시 테이블 맵핑 속도는 O(1)로 유지되어 성능 저하가 절대 발생하지 않음 (Zero-Cost Scaling).

### 미래 전망 및 진화 방향
Cilium은 CNI를 넘어 이스티오(Istio) 같은 서비스 메시(Service Mesh) 진영을 위협하고 있다. 기존 서비스 메시는 각 파드마다 무거운 사이드카 프록시(Envoy)를 띄워야 했으나, **Cilium Service Mesh**는 사이드카 없이 커널 레벨(eBPF)에서 곧바로 L7 라우팅, mTLS 암호화, 트래픽 추적을 수행하는 **"사이드카 없는 서비스 메시 (Sidecarless Service Mesh)"** 시대를 선도하며 클라우드 네이티브 인프라의 최종 보스로 등극하고 있다.

1. **Cilium(eBPF)이란?**: 고속도로 톨게이트(iptables)에서 수만 장의 명단을 하나하나 대조하며 검사하던 경찰 아저씨들을 싹 없애고, 자동차(데이터) 엔진 안에 스마트 내비게이션 칩을 심은 거예요.
2. **왜 혁명인가요?**: 차가 출발하자마자 내비게이션이 1초 만에 "너는 통과!", "너는 불합격!"을 알려주고 톨게이트도 없는 전용 지름길로 안내하니까 차가 절대 막히지 않아요.
