# 826. Kube-Proxy 쿠버네티스 서비스 트래픽 부하 분산 (iptables / IPVS 모드)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 쿠버네티스(Kubernetes) 환경에서 고정된 단일 가상 IP(Service VIP)로 들어오는 트래픽을 백엔드의 여러 파드(Pod)들로 로드밸런싱하기 위해, 노드의 커널 네트워크 규칙을 조작하는 데몬이다.
> 2. **가치**: `iptables` 또는 `IPVS`라는 리눅스 커널의 검증된 네트워크 필터링 기술을 활용하여, 별도의 물리적 L4 스위치 장비 없이도 클러스터 내의 수천 개 서비스에 대한 완벽한 분산 라우팅을 투명하게(Transparent) 소프트웨어로 구현해 낸다.
> 3. **융합**: `iptables` 모드의 선형적 규칙 스캔(O(N))에 따른 성능 저하를 극복하기 위해 `IPVS` 모드(O(1) 해시 테이블)가 도입되었으며, 최근에는 이마저도 eBPF(Cilium)로 완전히 대체되는 기술 진화의 한가운데 서 있다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
`kube-proxy`는 K8s 클러스터의 모든 워커 노드(Worker Node)에 하나씩 실행되는 네트워크 프록시 데몬(DaemonSet)이다. K8s API 서버를 감시하다가 새로운 `Service`(가상 IP)나 `Endpoint`(파드 IP)가 생성/삭제되면, 노드의 리눅스 커널 방화벽(`iptables` 또는 `IPVS`) 규칙을 즉시 업데이트하여 트래픽이 올바른 파드로 찾아가도록 길을 터준다.

### 💡 비유
`kube-proxy`는 **"전국 우체국의 자동 경로 재탐색 내비게이션"**과 같다.
대표 콜센터 번호(Service VIP: 1588-0000)로 전화를 걸면, 내비게이션(kube-proxy가 심어둔 iptables)이 즉각 개입하여 "현재 서울에 상담원(Pod) A, B, C가 대기 중이니 확률적으로 돌려(Load Balancing) 상담원 B의 진짜 직통 번호(Pod IP)로 연결해라!"라고 목적지를 스리슬쩍 바꿔치기(DNAT)하는 역할을 한다.

### 등장 배경 및 발전 과정

#### 1. 컨테이너 IP의 맹점
파드(Pod)는 생성과 소멸을 밥 먹듯이 하며, 그때마다 IP가 변경된다. 클라이언트가 특정 파드의 IP를 기억하고 통신하는 것은 불가능하다. 이를 해결하기 위해 K8s는 절대로 변하지 않는 가상의 IP 주소인 **Service VIP**를 만들었다.

#### 2. Kube-proxy 모드의 진화
- **Userspace 모드 (초기)**: `kube-proxy` 프로그램 자체가 패킷을 받아서 파드로 넘겼다(프록시). 유저 스페이스와 커널 스페이스 간의 잦은 스위칭으로 성능이 최악이었다.
- **iptables 모드 (표준)**: 프록시 역할을 그만두고, 리눅스 커널의 방화벽 규칙(`iptables`)만 싹 세팅해 놓고 빠지는 방식으로 진화하여 성능이 비약적으로 올랐다. (현재의 기본값)
- **IPVS 모드 (대규모 환경)**: 파드가 1만 개가 넘어가자 iptables의 규칙 탐색이 너무 느려지는 병목이 생겼고, 로드밸런싱 전용 커널 모듈인 `IPVS`를 도입해 O(1)의 속도를 확보했다.

  │  - 규칙 2: 50% 확률로 KUBE-SEP-BBB (Pod B-2)로 보내라 (DNAT)│
  │  - 규칙 3: 나머지 100%는 KUBE-SEP-CCC (Pod B-3)로 (DNAT)    │
  │                                                            │
  │    (만약 Pod B-1이 당첨되었다면, 목적지 IP를 변환시킴)     │
  │    [ Src: 192.168.1.5 | Dst 변환!: 192.168.2.55 (Pod B-1) ]│

                          │

 2. 라우팅 테이블(CNI 담당)에 의해 변환된 목적지(Pod B-1)로 패킷이 전송됨.
 * kube-proxy는 이 수많은 복잡한 규칙들을 백그라운드에서 열심히 타이핑하여 커널에 심어놓을 뿐,
   실제 패킷은 kube-proxy 프로세스를 거치지 않고 "커널 내부(Netfilter)"에서 빛의 속도로 변환됨.

### 심층 동작 원리

#### ① iptables 모드의 확률적 부하 분산 (Random Load Balancing)
`iptables`는 애초에 방화벽용이라 똑똑한 라우팅 알고리즘이 없다. 그래서 K8s는 `iptables`의 `statistic` 모듈을 이용한 **꼼수(확률 기반 NAT)**를 쓴다. 파드가 3개면, 첫 번째 파드에 걸릴 확률 33%(1/3), 두 번째 파드에 걸릴 확률 50%(1/2), 세 번째에 걸릴 확률 100%(1/1)라는 규칙을 순차적으로 적어 넣어 거의 균등하게 트래픽이 나뉘도록 조작(DNAT)한다.

#### ② iptables의 O(N) 병목 한계
서비스 1개당 10개의 파드가 붙어있고 이런 서비스가 1,000개 있다면, iptables 안에는 수만 줄의 규칙이 생성된다. 클라이언트의 패킷이 도착하면, 리눅스 커널은 1번 줄부터 10,000번 줄까지 위에서 아래로 순서대로 읽으면서(`O(N)` 선형 검색) 일치하는 규칙을 찾는다. 당연히 뒷번호에 있는 서비스로 갈수록 속도가 느려지고 CPU가 폭발한다. 

#### ③ IPVS 모드의 도입 (O(1) 속도 혁명)
IPVS(IP Virtual Server)는 애초에 L4 로드밸런싱을 위해 만들어진 커널 모듈이다. 규칙을 일렬로 세우지 않고 **해시 테이블(Hash Table)** 구조를 사용하므로, 서비스가 10만 개가 있어도 목적지 검색 속도가 O(1)로 항상 일정하다. 또한 라운드 로빈(RR), 최소 연결(Least Connection) 등 훨씬 정교한 부하 분산 알고리즘을 지원한다. 대규모 클러스터(보통 1,000 파드 이상)에서는 IPVS 모드 셋팅이 필수적이다.

## Ⅲ. 융합 비교 및 다각도 분석

### Kube-Proxy 운영 모드 3종 심층 비교

| 비교 지표 | Userspace 모드 (구형) | iptables 모드 (표준) | IPVS 모드 (대규모용) |
|-----------|-----------------------|----------------------|----------------------|
| **패킷 처리 위치**| Userspace (kube-proxy 프로세스) | Kernel Space (Netfilter) | Kernel Space (IPVS 해시맵) |
| **속도 및 성능**| 매우 느림 (컨텍스트 스위칭 지옥) | 빠름 | **가장 빠름 (O(1) 라우팅)** |
| **부하 분산 방식**| Round Robin | Random (확률 기반) | **RR, LC, Destination Hashing 등 다양함** |
| **스케일링 한계** | 수백 개 파드 | 약 1,000~5,000개 서비스 한계 | 수만 개 서비스 거뜬히 지원 |
| **재시도 (Retry)**| 실패 시 다른 파드로 재시도 가능 | **불가능 (파드 응답 없으면 타임아웃 떨어짐)** | 실패 시 다른 파드로 재시도 불가능 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 급성장하는 SaaS 기업의 K8s 클러스터. 노드 100대에 수천 개의 마이크로서비스가 배포되어 있다. 최근 노드들의 CPU `sys(커널)` 사용률이 비정상적으로 높고, Service VIP를 호출할 때 핑(Ping) 지연 시간이 100ms 이상 튀는 현상이 간헐적으로 발생함.
**원인 진단**: `kube-proxy`가 기본값인 **iptables 모드**로 동작 중. 서비스 추가/삭제 배포가 일어날 때마다 kube-proxy가 수만 줄의 iptables 룰 전체를 통째로 지우고 다시 덮어쓰는 작업(Lock 발생)을 하느라 일시적인 네트워크 멈춤(Stall)과 병목이 발생함.
**판단 및 튜닝 전략**:
1. **커널 모듈 로드**: 모든 워커 노드에 `ip_vs`, `ip_vs_rr`, `ip_vs_wrr`, `ip_vs_sh` 커널 모듈이 로드되도록 프로비저닝 스크립트(Ansible 등)를 수정한다.
2. **IPVS 모드 전환**: Kube-proxy의 ConfigMap 설정에서 `mode: "ipvs"`로 변경 후 데몬셋을 롤링 재시작한다.
3. **결과**: `iptables` 규칙이 파드 수와 무관하게 수십 줄로 확 줄어들며, 수만 개의 분산 룰은 IPVS의 메모리(해시) 상에서 초고속으로 처리되어 노드 CPU 안정화 및 레이턴시 튀는 현상 완벽 해결.

### 주의사항 및 안티패턴 (Anti-patterns)
- **과도한 트래픽 핑퐁 (SNAT의 늪)**: 외부 트래픽이 NodePort 등을 통해 클러스터로 들어올 때, iptables는 목적지 파드가 다른 노드에 있으면 트래픽을 그 노드로 꺾어버린다. 이때 응답이 꼬이지 않게 출발지 IP를 노드 IP로 덮어쓰는(SNAT) 작업이 일어나, **애플리케이션은 클라이언트의 진짜 원본 IP를 알 수 없게 되는 치명적 문제**가 생긴다. 이를 막으려면 서비스 설정에 `externalTrafficPolicy: Local`을 적용해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **가용성 보장**: 파드가 죽으면 API 서버가 이를 인지하여 수 초 내에 Endpoints에서 제거하고, kube-proxy가 즉시 커널 룰을 고쳐 죽은 파드로 트래픽이 가지 않게 막음.
- **성능 (IPVS 전환 시)**: 규칙 갱신 시간이 수 초(iptables)에서 수 밀리초(IPVS)로 단축되며, 서비스 개수 증가에 따른 지연 시간 증가율 0% 보장.

### 미래 전망 및 진화 방향
클라우드 네이티브 네트워킹의 궁극적 목표는 아이러니하게도 **"kube-proxy의 완벽한 멸종"**이다. `iptables`든 `IPVS`든 K8s를 위해 태어난 기술이 아니기 때문에 태생적 한계가 명확하다. 앞서 다룬 **Cilium (eBPF 기반)** CNI는 `kube-proxy` 프로세스 자체를 아예 지워버리고(Kube-proxy Free), 소켓 계층에서 훨씬 더 빠르고 효율적으로 VIP 라우팅을 가로채어 처리하는 시대로 기술 생태계를 리드하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **kube-proxy란?**: 아파트 전체(노드)에 들어오는 전화(트래픽)를 어느 방(파드)으로 연결할지 정해주는 '전화 교환수 로봇'이에요.
2. **iptables 모드는 뭔가요?**: 로봇이 수첩에 1번부터 1만 번까지 규칙을 적어두고, 전화가 올 때마다 1번부터 끝까지 손가락으로 짚어가며(O(N)) 읽는 방식이라 전화가 많아지면 너무 느려져요.
3. **IPVS 모드는 뭔가요?**: 수첩 대신 최신식 '초고속 이름 검색 컴퓨터(해시)'를 도입해서, 사람이 1만 명이든 10만 명이든 단 0.1초 만에 연결할 방을 찾아주는 훌륭한 업그레이드 방식이랍니다!
