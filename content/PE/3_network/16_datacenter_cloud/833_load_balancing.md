# 833. 로드 밸런싱 (Load Balancing - L4/L7 방식 및 분산망 적용 구조)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인터넷 트래픽(클라이언트 요청)이 하나의 서버로 몰려 시스템이 마비되는 것을 막기 위해, 중간에 스위치(Proxy) 장비를 두어 여러 대의 백엔드 서버로 트래픽을 지능적으로 분산시켜 주는 네트워크 핵심 아키텍처다.
> 2. **가치**: IP와 Port만 보고 단순 무식하게 분배하는 고속의 L4 로드밸런싱과, HTTP 헤더나 쿠키 등 페이로드를 뜯어보고 정교하게 분배하는 지능형 L7 로드밸런싱을 상황에 맞게 융합하여 서비스 가용성과 무중단 확장을 보장한다.
> 3. **융합**: 클라우드 네이티브 환경의 Kube-proxy, Ingress Controller, Service Mesh(Envoy)와 완벽히 융합되어 과거 하드웨어(L4 스위치) 중심에서 소프트웨어 정의 네트워크(SDN) 기반 분산 아키텍처로 진화했다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
로드 밸런싱(Load Balancing)은 네트워크나 애플리케이션 트래픽을 여러 대의 서버(서버 팜/클러스터)에 고르게 분배하는 기술이다. 이를 수행하는 장비나 소프트웨어를 로드 밸런서(Load Balancer)라 부르며, 사용자에게는 단일 진입점(Virtual IP)을 제공하면서도 뒷단에서는 수십 대의 서버가 일하게 만들어 시스템의 단일 장애점(SPOF)을 제거한다.

### 💡 비유
로드 밸런싱은 **"놀이공원 정문의 줄 세우기 알바생"**과 같다.
입구에 10개의 매표소(서버)가 있다. 알바생(로드 밸런서)이 없다면 사람들은 눈치 게임을 하다가 1번 매표소에만 100명이 줄을 서고, 2번 매표소는 놀고 있는 비효율이 발생한다.
- **L4 알바생**: 그냥 무조건 "1번 가세요, 2번 가세요, 3번 가세요(라운드 로빈)" 하고 기계적으로 사람들을 나눠 보낸다. 일처리는 엄청나게 빠르다.
- **L7 알바생**: 손님의 티켓을 확인(패킷 분석)한 뒤, "오, VIP 손님이시네요? 저쪽 고급 매표소로 가세요"라거나 "연간 회원이시군요? 5번 매표소로 가세요"라고 내용에 맞춰 아주 똑똑하게 안내한다. 대신 한 명 한 명 표를 확인해야 하니 조금 느리다.

### 등장 배경 및 발전 과정

#### 1. 스케일 업(Scale-up)의 한계와 DNS 라운드 로빈의 문제
웹 초기, 트래픽이 늘면 서버의 CPU를 좋은 것으로 바꾸는 스케일 업에 의존했다. 이후 서버를 늘리는 스케일 아웃(Scale-out)이 등장했지만, 사용자를 여러 서버로 보내기 위해 DNS 라운드 로빈(하나의 도메인에 여러 IP 맵핑)을 썼더니 특정 서버가 죽어도 DNS 캐시 때문에 죽은 서버로 계속 접속하는 치명적 한계가 발생했다.

#### 2. L4/L7 스위치의 등장과 클라우드 진화
이러한 문제를 해결하기 위해 백엔드 서버들의 상태(Health Check)를 1초마다 감시하고 살아있는 서버로만 트래픽을 꺾어주는 전용 하드웨어(L4 스위치, F5 등)가 등장했다. 이후 MSA 시대가 열리며 하드웨어를 버리고 Nginx, HAProxy, Envoy 같은 소프트웨어 로드 밸런서(ALB, NLB)가 대세를 이루게 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### L4 vs L7 로드 밸런서 심층 비교 (표)

| 비교 지표 | L4 (Transport Layer) 로드 밸런서 | L7 (Application Layer) 로드 밸런서 |
|-----------|----------------------------------|------------------------------------|
| **분석 기준** | IP 주소, MAC 주소, TCP/UDP Port | HTTP 헤더, URL 경로(Path), 쿠키, 페이로드 |
| **속도 및 성능**| **매우 빠름 (패킷 겉면만 보고 던짐)** | 상대적으로 느림 (패킷을 열어서 파싱해야 함) |
| **보안(TLS) 처리**| 암호화된 채로 통과 (Pass-through) | LB 단에서 암호화를 풀고(TLS Termination) 검사 후 재암호화 가능 |
| **라우팅 지능** | 단순 기계적 분배 (동일 포트는 동일 처리) | 고도화된 분배 (`/video`는 미디어 서버로, `/login`은 인증 서버로) |
| **클라우드 예시**| AWS NLB (Network Load Balancer), Kube-proxy | AWS ALB (Application Load Balancer), Ingress, Istio |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ 1. L4 Load Balancing Architecture (NAT 방식) ]
================================================================================

  [ Client ]  (Src IP: 1.1.1.1, Dst IP: 200.0.0.1)  <- 200.0.0.1은 LB의 VIP
      │
      ▼
  ┌──────────────────────────────────────────────┐
  │              L4 Load Balancer                │
  │ - 패킷 내용 안 봄. TCP 80 포트 확인.         │
  │ - 라운드 로빈 알고리즘 적용 -> Server B 당첨 │
  │ - [DNAT 변환]: Dst IP를 10.0.0.2로 조작      │
  └──────────────────┬───────────────────────────┘
                     │ (Src: 1.1.1.1, Dst: 10.0.0.2)
       ┌─────────────┼─────────────┐
       ▼             ▼             ▼
   [ Server A ]  [ Server B ]  [ Server C ]
   (10.0.0.1)    (10.0.0.2)    (10.0.0.3)

================================================================================
[ 2. L7 Load Balancing Architecture (Reverse Proxy 방식) ]
================================================================================

  [ Client ]  GET /api/pay (Header: Cookie=VIP)
      │
      ▼
  ┌──────────────────────────────────────────────┐
  │              L7 Load Balancer                │
  │ 1. TCP 세션 맺고 HTTP 까보기 (TLS 복호화)    │
  │ 2. Path가 "/api/pay" 이고 VIP 쿠키 발견!     │
  │ 3. "VIP 전용 결제 서버 풀"로 타겟 그룹 변경  │
  └──────────────────┬───────────────────────────┘
                     │ (LB가 새로운 TCP 세션을 맺어 전송)
       ┌─────────────┼─────────────┐
       ▼             ▼             ▼
  [ Web Svc ]  [ Pay Svc(일반) ] [ Pay Svc(VIP) ]
```

### 심층 동작 원리 (핵심 알고리즘)

분배를 "어떻게" 할 것인가를 결정하는 스케줄링 알고리즘이다.
1. **Round Robin (라운드 로빈)**: 들어오는 순서대로 A, B, C 서버에 공평하게 1개씩 순차 할당한다. (서버 스펙이 모두 같을 때 유리)
2. **Least Connection (최소 연결)**: 현재 연결되어 있는 세션(활성 커넥션) 수가 가장 적은 서버로 트래픽을 보낸다. (작업 처리 시간이 들쭉날쭉한 경우 최고)
3. **Source Hashing (해시 알고리즘)**: 클라이언트의 출발지 IP를 해싱(`Hash(IP) % N`)하여 특정 서버에 고정시킨다. (쇼핑몰 장바구니 등 **세션 유지(Sticky Session)**가 필수적인 환경에서 사용)
4. **Weighted (가중치 방식)**: 서버 스펙이 다를 때, 최신 서버에 가중치 3을 주고 구형 서버에 1을 주어 3:1 비율로 분배한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 로드 밸런서 구현 위치(Topology)에 따른 분류

| 분류 | 특징 | 장점 | 단점 | 대표 기술 |
|------|------|------|------|-----------|
| **하드웨어 (Appliance)**| L4 스위치 전용 장비를 랙 최상단에 물리적으로 설치 | 극강의 안정성과 ASIC 기반 초고속 패킷 처리 | 매우 비싸고 클라우드 동적 확장에 부적합 | F5 BIG-IP, Citrix |
| **소프트웨어 (Reverse Proxy)**| 일반 x86 서버에 소프트웨어를 띄워 프록시로 사용 | 저렴하고 클라우드 오토스케일링과 완벽 연동됨 | 서버 CPU를 파싱에 소모함 | Nginx, HAProxy |
| **클라이언트 사이드 LB**| 클라이언트 코드 내부에 분산 알고리즘을 내장 | 중간 병목(SPOF)이 아예 없는 궁극의 분산 | 여러 언어 지원 및 유지보수 지옥 (Fat Client) | Ribbon, gRPC LB |
| **서비스 메시 (사이드카)**| 각 파드 옆에 붙은 프록시(Envoy)가 분산 처리 수행 | 중앙 집중 통제 + 완벽한 탈중앙화 라우팅 | 복잡도 증가, 메모리 소비 | Istio |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대형 게임사의 MMO 서버 런칭일. 수십만 명의 유저가 동시에 접속(TCP 소켓 연결)을 시도하며 트래픽 폭주가 예상된다. 패킷 내용(L7)을 뜯어볼 필요는 없으며, 무조건 가장 빠르고 가벼운 속도로 100대의 게임 서버에 연결만 분산시켜 주면 된다.
**판단 및 아키텍처 전략**:
1. **L7(ALB/Nginx) 절대 금지**: 모든 패킷의 L7 페이로드를 까보고 프록시 커넥션을 두 번 맺는 L7 LB를 쓰면, 로드 밸런서 자체가 CPU 과부하로 터져버리는 병목(SPOF)이 된다.
2. **L4 (NLB / IPVS) 적극 채택**: TCP/UDP 포트만 보고 패킷의 목적지 IP만 스와이프(DNAT) 해버리는 **L4 네트워크 로드밸런서(Network Load Balancer)**를 전면에 배치한다.
3. **알고리즘 최적화**: 게임 접속 시간이 유저마다 천차만별이므로 단순 라운드 로빈은 특정 서버에 부하를 누적시킨다. **Least Connection(최소 연결)** 알고리즘을 적용하여 한가한 서버로 유저를 밀어 넣는다.
4. **결과**: 수십만 건의 TCP 연결을 LB의 병목 없이 100대의 백엔드에 완벽하게 흩뿌려 런칭 장애를 방어함.

### 주의사항 및 안티패턴 (Anti-patterns)
- **세션 불일치 (Session Mismatch)**: L4 로드밸런서를 거쳐 웹 서버 A에 로그인한 유저가, 다음 클릭 시 로드밸런서에 의해 웹 서버 B로 던져져 "로그인 풀림" 현상을 겪는 고전적 안티패턴이다.
  - **해결책 1**: 로드밸런서 단에서 **Sticky Session(세션 결합)**을 켜서 쿠키를 박아 특정 사용자는 무조건 A로만 가게 만든다. (단점: 서버 A가 터지면 유저 세션도 날아감, 핫스팟 발생 위험)
  - **해결책 2 (Best Practice)**: 로드밸런서는 세션을 무시하고(Stateless), 백엔드 서버들이 유저 세션 정보를 무조건 **외부의 Redis(인메모리 DB)에 중앙 저장**하도록 아키텍처를 진화시킨다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **가용성 99.99% 달성**: 헬스 체크(Health Check)를 통해 10대의 서버 중 3대가 뻗어도, 로드밸런서가 즉시 죽은 서버를 제외하고 7대로만 트래픽을 보내므로 사용자(Client)는 시스템 장애를 전혀 눈치채지 못함.
- **무중단 확장성 (Scale-out)**: 트래픽이 2배가 되면 백엔드 서버를 10대에서 20대로 늘린 뒤 로드밸런서 설정만 갱신하면 1초 만에 트래픽이 분산됨 (다운타임 제로).

### 미래 전망 및 진화 방향
클라우드 네이티브에서 로드 밸런싱의 패러다임은 외부(Ingress)와 내부(East-West)로 나뉘어 진화 중이다. 외부는 클라우드 벤더의 완전 관리형 서버리스 로드밸런서(AWS ALB, Gateway API)가 장악했고, 내부망 통신은 무거운 중앙 LB 장비를 거치지 않고 파드 옆에 기생하는 **사이드카 프록시(Envoy) 기반의 서비스 메시**나, 아예 커널을 해킹하는 **eBPF(Cilium) 기반 소켓 로드밸런싱**으로 대체되며 "지연 시간 제로"를 향해 맹렬히 진화하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **로드 밸런싱이란?**: 마트에 사람이 너무 많아서 계산대(서버) 10개를 열어두고, 안내원(로드 밸런서)이 "1번 계산대로 가세요! 다음 분은 2번으로 가세요!" 하고 예쁘게 나누어주는 거예요.
2. **L4 방식은 뭔가요?**: 안내원이 손님이 뭐 샀는지는 안 보고 묻지도 따지지도 않고 기계적으로 빈 계산대로 엄청 빨리 휙휙 보내는 거예요.
3. **L7 방식은 뭔가요?**: 안내원이 손님 장바구니를 쓰윽 살펴보고 "어? 아이스크림 사셨네요? 녹기 전에 5번 급행 계산대로 가세요!" 하고 똑똑하게 맞춤형 안내를 해주는 거랍니다!
