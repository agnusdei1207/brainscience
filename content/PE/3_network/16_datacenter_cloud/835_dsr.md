# 835. DSR (Direct Server Return) 비대칭 트래픽 라우팅

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: L4 로드밸런서 환경에서 클라이언트의 요청(Inbound)은 로드밸런서를 거쳐 백엔드 서버로 들어가지만, 서버가 내보내는 무거운 응답 패킷(Outbound)은 로드밸런서를 거치지 않고 **서버에서 클라이언트로 직접 쏘아버리는(Direct Return)** 비대칭 네트워크 라우팅 기법이다.
> 2. **가치**: 동영상 스트리밍, 대용량 파일 다운로드 환경에서 로드밸런서가 응답 트래픽에 짓눌려 터져버리는 병목(Bottleneck) 현상을 원천적으로 제거하여 인프라 비용을 극적으로 아껴준다.
> 3. **융합**: 루프백(Loopback) 인터페이스 조작, MAC 주소 기반의 라우팅, 그리고 클라우드 네이티브 네트워크의 고도화된 트래픽 분산 최적화(LVS/IPVS) 기술과 결합되어 대규모 데이터센터의 한계를 돌파한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
일반적인 로드밸런싱(NAT 방식)은 클라이언트의 요청이 LB(Load Balancer)를 거쳐 서버로 가고, 서버의 응답도 반드시 LB를 거쳐 클라이언트로 돌아가는 '대칭(Symmetric)' 구조다. 
**DSR (Direct Server Return)**은 요청 패킷(작은 크기)만 LB가 분배해 주고, 응답 패킷(큰 크기)은 백엔드 서버가 로드밸런서를 우회하여 클라이언트의 IP로 곧장 전송하는 '비대칭(Asymmetric)' 네트워크 터널링 기술이다.

### 💡 비유
DSR은 **"대형 마트의 전화 주문 배달 시스템"**과 같다.
- **일반 방식 (NAT)**: 손님이 마트 콜센터(LB)에 전화해 10kg짜리 쌀을 주문한다. 창고 직원(서버)이 쌀을 콜센터로 가져오면, 콜센터 직원이 그걸 받아서 다시 손님 집으로 배달한다. 콜센터 직원은 쏟아지는 무거운 쌀가마니를 나르느라 허리가 부러진다(LB 병목).
- **DSR 방식**: 손님이 콜센터(LB)에 전화(요청 1KB)하면, 콜센터는 창고 직원(서버)에게 "손님 집 주소"만 무전으로 틱 넘겨준다. 창고 직원은 쌀 10kg(응답 1GB)을 트럭에 싣고 콜센터를 거치지 않고 직접 손님 집으로 배달(Direct Return)해 버린다. 콜센터는 전화만 받으면 되므로 절대로 과부하가 걸리지 않는다.

### 등장 배경 및 발전 과정

#### 1. 인바운드와 아웃바운드의 비대칭성
웹 생태계가 텍스트에서 이미지, 동영상으로 넘어가면서 요청(Inbound) 트래픽은 몇십 바이트에 불과하지만, 응답(Outbound) 트래픽은 수 메가바이트~기가바이트로 **1:100 이상의 비대칭성**을 띠게 되었다. 모든 응답이 비싼 L4 하드웨어 스위치를 통과하자, L4 장비의 네트워크 대역폭(Bandwidth)이 순식간에 고갈되었다.

#### 2. 로드밸런서 우회(Bypass) 전략의 탄생
이 병목을 풀기 위해 "응답은 그냥 서버가 알아서 클라이언트에게 보내면 안 될까?"라는 발상이 나왔다. 하지만 클라이언트 입장에서는 `1.1.1.1(LB)`에 요청을 냈는데 `2.2.2.2(서버)`에서 답이 오면 해킹(Spoofing)으로 간주하고 패킷을 버린다. 이를 속이기 위해 서버가 자신이 `1.1.1.1`인 것처럼 위장하여 패킷을 쏘는 운영체제 레벨의 트릭(Loopback VIP 할당)이 DSR의 핵심으로 자리 잡았다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### NAT 모드 vs DSR 모드 패킷 흐름 비교 (표)

| 비교 구간 | NAT (Network Address Translation) 모드 | DSR (Direct Server Return) 모드 |
|-----------|----------------------------------------|---------------------------------|
| **Client -> LB (요청)** | Dst IP가 LB의 VIP로 옴 | Dst IP가 LB의 VIP로 옴 |
| **LB -> Server (전달)** | LB가 Dst IP를 Server의 사설 IP로 변환(DNAT) | IP 변환 없이 **MAC 주소만 Server의 MAC으로 변경(MAC Routing)**하여 던짐 |
| **Server -> LB (응답)** | Src IP를 자신의 사설 IP로 셋팅하여 LB로 전송 | **없음. (LB를 거치지 않음)** |
| **Server -> Client (직접 응답)** | (불가능) | **Src IP를 LB의 VIP로 속여서 클라이언트에게 직접 전송** |

### 심층 동작 원리 (어떻게 속이는가?)

#### ① L2 MAC 라우팅 (LB의 조작)
LB는 클라이언트가 보낸 패킷의 목적지 IP(VIP: 200.0.0.1)를 서버 IP(10.0.0.10)로 바꾸지 않는다. 대신 데이터 링크 계층(L2)으로 내려가서, 패킷의 겉포장인 목적지 MAC 주소만 서버 A의 MAC 주소로 바꾼다. 이 때문에 **LB와 백엔드 서버는 반드시 같은 L2 스위치(동일 서브넷/VLAN)에 물려 있어야만 한다** (DSR의 가장 큰 물리적 제약).

#### ② 루프백 VIP 설정과 ARP 무시 (서버의 조작)
서버 A가 패킷을 받았는데, 패킷의 도착지 IP가 200.0.0.1이다. 자기 물리 IP(10.0.0.10)와 다르면 커널이 패킷을 버린다(Drop). 이를 막기 위해 서버 A의 가상 랜카드(Loopback, `lo:0`)에 LB와 똑같은 VIP(200.0.0.1)를 세팅해 둔다.
*치명적 문제*: 서버가 VIP를 달고 있으면, 외부에서 "200.0.0.1 누구야?"라는 ARP 요청이 왔을 때 서버 A가 "나야!" 하고 대답해버려 LB와 IP 충돌이 난다. 따라서 서버 A 커널에서 **반드시 `ARP Ignore(응답 무시)` 설정**을 켜서, 자기는 조용히 웅크린 채 들어오는 패킷만 받도록 만들어야 한다.

#### ③ 클라이언트를 향한 직접 응답
처리가 끝난 서버 A는 클라이언트(8.8.8.8)에게 응답을 보낸다. 이때 보내는 출발지 IP는 방금 패킷을 받았던 루프백 주소인 VIP(200.0.0.1)가 된다. 클라이언트는 LB(200.0.0.1)에게 물어봤고 200.0.0.1로부터 답을 받았으므로 TCP 세션이 완벽하게 성립된다. LB는 이 과정에서 완전히 배제된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 트래픽 분산 모드 3대장 비교 (NAT vs DSR vs Tunneling)

| 비교 지표 | NAT 모드 (Network Address Translation) | DSR 모드 (Direct Server Return) | Tunneling 모드 (IPIP / GRE) |
|-----------|----------------------------------------|---------------------------------|-----------------------------|
| **응답 경로** | 서버 -> LB -> 클라이언트 | **서버 -> 클라이언트 (직통)** | 서버 -> 클라이언트 (직통) |
| **망 구성 제약**| 없음 (라우터 넘어 통신 가능) | **LB와 서버가 동일 L2 서브넷에 있어야 함** | 서로 다른 네트워크(L3)에 있어도 됨 |
| **LB 부하량** | 매우 높음 (병목 발생 쉬움) | **매우 낮음 (요청 패킷만 처리)** | 매우 낮음 |
| **서버 설정** | 불필요 | Loopback 인터페이스 조작 및 ARP 제어 필수 (복잡함) | Tunneling 디바이스 설정 필수 |
| **적합한 서비스**| 일반 웹사이트, API 서버 | **비디오 스트리밍(CDN), 대용량 파일 다운로드** | 다중 데이터센터 간 트래픽 분산 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대형 OTT(예: 넷플릭스, 유튜브) 서비스의 캐시 에지(Edge) 서버 팜 구축. 사용자가 영화를 클릭하면(요청: 1KB), 서버는 4K 동영상(응답: 5GB)을 쏜다. 10Gbps 성능의 고가 L4 스위치(NAT 모드)를 샀지만, 동접자가 2,000명만 넘어가도 L4 스위치의 아웃바운드 대역폭이 10Gbps를 꽉 채워 터져버림.
**판단 및 아키텍처 전환 전략**:
1. **L4 장비 증설 기각**: L4 스위치를 10대 사려면 수억 원의 인프라 비용이 증발한다.
2. **DSR 아키텍처 전면 도입**: L4 스위치의 모드를 `Direct Routing (DSR)`으로 바꾼다.
3. **서버 커널 튜닝**: 백엔드로 쓰이는 100대의 캐시 리눅스 서버에 접속하여 `ifconfig lo:0 VIP up`으로 루프백 주소를 달고, `sysctl`을 건드려 `arp_ignore=1`, `arp_announce=2`를 세팅하여 충돌을 막는다.
4. **결과**: 영화 데이터(5GB)는 백엔드 서버의 10Gbps/25Gbps 랜카드를 통해 L4 스위치를 거치지 않고 외부 인터넷 망(ISP)으로 다이렉트로 빠져나간다. L4 스위치는 오직 인바운드(1KB 요청)만 처리하므로 1대의 스위치로 수백만 명의 트래픽을 거뜬히 방어하게 된다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **클라우드 환경에서의 좌절**: AWS, Azure, GCP 같은 퍼블릭 클라우드에서는 가상 머신(VM)이 할당받지 않은 IP(VIP)를 목적지/출발지로 달고 패킷을 보내면, 클라우드 인프라의 안티-스푸핑(Anti-Spoofing) 방화벽이 이를 해킹으로 간주하고 즉각 차단(Drop)해 버린다. 즉, **퍼블릭 클라우드에서는 전통적인 L2 기반 DSR 구현이 사실상 불가능**하다. 클라우드에서는 eBPF나 IPIP 터널링을 응용한 변형된 방식의 DSR을 찾아야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **네트워크 대역폭 100배 확충**: 로드밸런서의 아웃바운드 대역폭 소모를 0으로 만들어, 인프라 비용을 동결한 채 10배~100배의 멀티미디어 트래픽 아웃풋(Throughput) 달성.
- **레이턴시(지연 시간) 단축**: 패킷이 로드밸런서를 한 번 덜 거치게 되므로 미세한 네트워크 홉(Hop) 감소 및 지연 시간 향상 효과.

### 미래 전망 및 진화 방향
전통적인 L2 DSR은 물리 네트워크(L2 스위치)에 종속된다는 치명적 약점 때문에, 마이크로서비스가 수많은 물리 장비를 넘나드는 쿠버네티스 환경에서는 쓰기 어려웠다. 하지만 최근 Cilium과 같은 **eBPF 기반 클라우드 네이티브 네트워크** 기술은 커널 소켓 레벨에서 패킷을 직접 통제하여 클라우드 환경의 서브넷 제약을 극복한 **오버레이 기반 DSR (또는 eBPF DSR)**을 구현해 냈다. DSR 철학은 현대 CNI(컨테이너 네트워크 인터페이스)의 초고속 데이터 전송 최적화의 심장으로 화려하게 부활하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **DSR이 뭔가요?**: 피자집 전화받는 직원(로드밸런서)이 주문만 딱 받고, 배달 오토바이(응답)는 가게 뒷문으로 나가서 손님 집으로 직행하게 만드는 아주 빠른 배달 시스템이에요.
2. **왜 굳이 그렇게 하나요?**: 옛날엔 오토바이가 배달 나갈 때 무조건 좁은 앞문(전화받는 직원)을 뚫고 나가야 해서 배달이 다 밀리고 직원이 깔려 쓰러졌거든요.
3. **가장 좋은 점은 뭐예요?**: 앞문 직원은 전화기만 가볍게 잡고 있으면 되니까, 1명이 수만 명의 주문을 받아도 절대 고장 나지 않아서 피자집이 엄청나게 돈을 잘 벌 수 있답니다!
