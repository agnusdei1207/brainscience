# 836. VPC (Virtual Private Cloud) - 퍼블릭 클라우드 내 가상 사설 통신망

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 누구나 접속할 수 있는 거대한 공용 퍼블릭 클라우드(AWS, Azure, GCP) 환경 내에서, 고객이 자신만의 IP 대역, 서브넷, 라우팅 테이블, 방화벽을 독자적으로 정의하여 격리된 **"논리적인 나만의 프라이빗 데이터센터"**를 구축하는 SDN(소프트웨어 정의 네트워크) 기술이다.
> 2. **가치**: 다른 고객의 인스턴스와 네트워크 트래픽이 섞이지 않는 완벽한 논리적 격리(Isolation)를 통해 온프레미스(On-premise)와 동일한 수준의 최고 보안 등급을 제공하며, 퍼블릭 클라우드 도입의 최대 진입 장벽이었던 보안 우려를 종식시켰다.
> 3. **융합**: 서브넷 분할(Subnetting), NAT 게이트웨이, 라우팅 테이블, 보안 그룹(Security Group) 등 전통적 네트워크 기술과 융합되며, VPN이나 Direct Connect를 통해 사내 온프레미스 망과 연결되는 하이브리드 클라우드의 기초 뼈대가 된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
VPC(Virtual Private Cloud)는 클라우드 제공자가 거대한 물리적 네트워크 자원을 논리적으로 쪼개어, 각 고객에게 독립된 가상 사설망을 제공하는 서비스다. 고객은 VPC 내부에서 사설 IP 주소 범위(CIDR)를 마음대로 선택하고, 퍼블릭 서브넷(인터넷과 통신)과 프라이빗 서브넷(인터넷과 단절)을 직접 설계하여 서버 리소스들을 안전하게 배치할 수 있다.

### 💡 비유
VPC는 **"거대한 공유 오피스 건물(퍼블릭 클라우드) 안에 차려진 우리 회사 전용 잠금식 사무실"**과 같다.
- **클래식 클라우드 (VPC 이전)**: 공유 오피스 로비에 책상을 펴고 일하는 것. 전 세계 누구나 내 책상(서버) 앞을 지나가며 모니터를 훔쳐보거나(해킹 위험), 내 IP 대역을 남이 써버릴 수 있다.
- **VPC 도입 후**: 튼튼한 유리벽과 지문 인식기(격리)가 달린 사무실을 배정받는다. 사무실 안의 자리 배치(서브넷 설계)는 내 마음대로다. 외부 손님용 응접실(퍼블릭 서브넷)과 사장님 전용 비밀 방(프라이빗 서브넷)을 나눌 수 있고, 다른 회사는 절대 우리 사무실 문을 열고 들어올 수 없다.

### 등장 배경 및 발전 과정

#### 1. EC2-Classic의 보안 한계 (초창기 클라우드)
AWS 초창기(EC2-Classic)에는 모든 고객의 가상 서버(인스턴스)가 거대한 하나의 공유 네트워크(Flat Network) 위에 무작위로 섞여서 생성되었다. 사설 IP는 AWS가 알아서 배정했고, 고객은 네트워크 통제권이 없었다. 기업들은 "우리 고객의 결제 DB가 경쟁사 서버와 같은 망에 떠다니는 걸 믿을 수 없다"며 클라우드 도입을 주저했다.

#### 2. 진정한 IaaS의 완성 (VPC의 탄생)
2009년 AWS가 VPC를 최초로 발표하며 시장의 판도를 바꿨다. 하드웨어를 가상화하는 것(EC2)을 넘어, **네트워크(스위치, 라우터)까지 100% 소프트웨어로 가상화(SDN)**하여 고객에게 쥐여주었다. 고객은 온프레미스에서 쓰던 `10.0.0.0/16` 대역의 IP를 클라우드에 똑같이 구성할 수 있게 되었고, 이로써 기업들의 대규모 클라우드 마이그레이션이 시작되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 주요 구성 요소 (표)

| 구성 요소 | 상세 역할 | 핵심 동작 메커니즘 | 물리 네트워크 대응 | 비유 |
|-----------|----------|--------------------|--------------------|------|
| **VPC (CIDR Block)** | 가장 큰 논리적 격리망 | 고객이 지정한 IP 범위(예: `10.0.0.0/16`). 이 안에서만 통신이 일어남 | 데이터센터 (IDC) 건물 | 사무실 전체 영역 |
| **Subnet (서브넷)** | VPC 내부의 분할 구역 | VPC의 큰 IP 대역을 더 잘게 쪼갠 덩어리. 각 가용 영역(AZ)에 묶임 | L2 VLAN / 스위치 | 부서별 파티션 |
| **Route Table** | 트래픽 이정표 | "10.0.1.0으로 가는 패킷은 A로, 인터넷(0.0.0.0/0)으로 가는 건 B로 가라"는 룰 | L3 라우터 | 안내 데스크 |
| **Internet Gateway (IGW)**| 인터넷 연결 관문 | VPC 내부의 사설 IP와 외부의 공인 IP 간의 통신을 변환(NAT)하며 열어줌 | ISP 모뎀 / 백본 라우터 | 1층 외부 출입문 |
| **NAT Gateway** | 프라이빗망 인터넷 접근 | 인터넷(외부)에서 들어오진 못하지만, 내부에서 외부로 나갈 수 있게만 해주는 단방향 게이트웨이 | NAT 장비 / 방화벽 | 창문 밖으로 물건만 던지기 |
| **Security Group (SG)**| 인스턴스 단위 방화벽 | 포트 및 IP 기반 접근 제어 (Stateful 방화벽) | 인스턴스 소프트웨어 방화벽 | 개인 책상 앞 경호원 |

### 심층 동작 원리 (SDN 오버레이)

클라우드의 VPC는 물리적인 랜선을 꼽아서 만들어지지 않는다. AWS 내부에 수백만 대의 물리 서버가 있지만, 이들은 AWS 자체의 거대한 **언더레이(Underlay)** 네트워크로 연결되어 있다. 
고객이 VPC(`10.0.0.0`)를 만들면, AWS의 **오버레이 터널링(Overlay Tunneling, GENEVE 또는 VXLAN 등)** 기술이 작동한다. 고객 A의 웹 서버와 DB 서버가 물리적으로 완전히 다른 데이터센터(AZ) 건물에 떨어져 있어도, 패킷을 암호화하고 가상의 ID(VNI)를 붙여 터널로 쏘아주기 때문에, 마치 같은 L3 라우터 밑에 있는 것처럼 완벽하게 동작한다. 고객의 라우팅 설정은 즉각(ms 단위) 글로벌 SDN 컨트롤러를 통해 모든 호스트에 반영된다.

## Ⅲ. 융합 비교 및 다각도 분석

### VPC 기반 망 분리 방식의 진화

| 방식 | 통신 흐름 | 격리 수준 | 특징 및 한계 |
|------|-----------|-----------|--------------|
| **VPC Peering (피어링)** | VPC A <--> VPC B (1:1 직통) | 완벽 격리 (인터넷 안 거침) | VPC가 수십 개가 되면 N(N-1)/2 개의 그물망 설정이 필요해 관리 지옥(Mesh Hell)에 빠짐 |
| **Transit Gateway (TGW)**| VPC A <--> TGW <--> VPC B, C, D | 중앙 집중식 라우팅 | 수백 개의 VPC와 온프레미스 망을 중앙의 십자로(Hub)에 꽂아 한 번에 라우팅하는 현대 클라우드 네트워크의 표준 |
| **PrivateLink (VPC Endpoint)**| 서비스 A -> VPC Endpoint -> 타사 서비스 | 대상 서비스의 특정 포트만 노출 | 네트워크 전체를 뚫지 않고 API 단위로만 내부망 통신을 허용하여 최고 수준의 보안 달성 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 한 스타트업이 초기 개발 편의성을 위해 기본 VPC(Default VPC)의 `Public Subnet`에 WAS, DB, 캐시 서버를 모두 몰아넣고 공인 IP를 달아놓았다. 최근 해커의 포트 스캐닝(Port Scanning)으로 인해 DB 인스턴스의 3306 포트가 털려 랜섬웨어에 감염되었다.
**판단 및 아키텍처 재설계 전략 (망 분리)**:
1. **서브넷 계층화**: Custom VPC를 새로 파고 서브넷을 엄격히 2 Tier(또는 3 Tier)로 쪼갠다.
2. **자원 재배치**: 
   - `Public Subnet`: 사용자와 직접 닿는 L7 로드밸런서(ALB)와 NAT Gateway, 배스천 호스트(Bastion Host)만 배치한다.
   - `Private Subnet`: 비즈니스 로직(WAS)과 데이터베이스(DB) 서버를 모조리 숨긴다. 이 서버들은 공인 IP(Public IP)를 영구적으로 제거한다.
3. **보안 그룹(SG) 체인**: WAS 서버의 보안 그룹은 "ALB에서 오는 80 포트 트래픽"만 허용하고, DB 서버의 보안 그룹은 "WAS 서버에서 오는 3306 트래픽"만 허용한다.
4. **결과**: 외부의 해커는 DB가 우주 어디에 존재하는지 IP조차 알아낼 수 없는(Routable하지 않은) 원천적 격리 상태가 되어 랜섬웨어의 위협이 완전히 종식된다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **IP 대역(CIDR)의 겹침 (Overlapping CIDR)**: 초기 VPC를 설계할 때 아무 생각 없이 `10.0.0.0/16`을 쓴다. 나중에 회사가 커져서 다른 회사(스타트업)를 인수합병(M&A)했는데, 그 회사도 똑같이 `10.0.0.0/16`을 쓰고 있다면 두 회사의 VPC를 피어링(연결)하는 순간 라우팅 충돌이 발생해 네트워크가 파괴된다. VPC 생성 시에는 반드시 사내 표준 IP 할당 계획(IPAM)에 따라 겹치지 않는 유니크한 사설 대역(예: `10.43.0.0/16`)을 할당받아 생성해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **보안 사고 리스크(Blast Radius) 감소**: 인터넷을 통한 직접 해킹 가능성을 아키텍처 레벨에서 99% 차단 (Public 망에서 Private 망 접근 원천 불가).
- **글로벌 인프라 확장성**: 일본 리전(Region)과 미국 리전에 흩어진 VPC를 Transit Gateway로 묶어, 전 세계 어디서든 `ping 10.1.x.x` 사설 IP로 안전하게 통신 가능한 글로벌 전용망 구축.

### 미래 전망 및 진화 방향
단순한 가상 망 분리를 넘어서, 클라우드 네이티브 시대의 VPC는 서비스 메시(Service Mesh) 및 서버리스(Serverless)와 결합하고 있다. 서버(EC2)가 없어도 AWS Lambda 같은 서버리스 함수가 VPC 안에서 기동하여 프라이빗 DB에 접근하고, 복잡한 IP 라우팅 테이블 관리는 **Cilium (eBPF)** 기반의 K8s 클러스터 라우팅과 통합되어 점차 "IP 주소를 몰라도 서비스 이름만으로 격리 통신이 되는" **ID 기반 네트워킹(Identity-based Networking)** 시대로 진화하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **VPC가 뭔가요?**: 엄청나게 큰 공용 놀이터(클라우드) 안에서 펜스를 치고 우리 가족만 놀 수 있게 만든 '비밀 아지트'예요.
2. **퍼블릭 / 프라이빗 서브넷은 뭐예요?**: 아지트 안에서도 '마당(퍼블릭)'은 배달부 아저씨나 손님이 올 수 있게 열어두고, '안방(프라이빗)'은 문을 꽉 잠가서 소중한 금고(데이터베이스)를 숨겨두는 곳이에요.
3. **왜 굳이 이렇게 하나요?**: 놀이터에 있는 나쁜 도둑들이 우리 집 금고를 훔쳐 가려고 해도, 안방 문이 어디 있는지도 모르게 꽁꽁 숨겨서 100% 안전하게 보호하기 위해서랍니다!
