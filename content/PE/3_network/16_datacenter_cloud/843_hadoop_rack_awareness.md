# 843. 하둡(Hadoop) 랙 인식 (Rack Awareness) - 토폴로지 통신 및 데이터 복제

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 하둡 분산 파일 시스템(HDFS)에서 데이터 블록을 복제할 때, 모든 노드가 평등하다고 가정하지 않고 서버가 꽂혀 있는 물리적인 **'랙(Rack)'의 위치 정보**를 인식하여 복제본을 전략적으로 배치하는 기술이다.
> 2. **가치**: 특정 랙의 상단 스위치(ToR Switch)가 고장 나더라도 다른 랙에 복제본이 살아남게 하여 데이터 가용성을 극대화하고, 동일 랙 내 노드 간 통신을 우선시하여 네트워크 대역폭 낭비를 최소화한다.
> 3. **융합**: HDFS의 블록 배치 정책(Block Placement Policy), 맵리듀스(MapReduce)의 데이터 지역성(Data Locality) 최적화, 그리고 클러스터 토폴로지 스크립트가 융합되어 거대 빅데이터 클러스터의 신경망을 형성한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
랙 인식(Rack Awareness)은 네임노드(NameNode)가 데이터노드(DataNode)들의 물리적 배치 상태(어느 랙에 꽂혀 있는지)를 파악하고 있는 상태를 말한다. 하둡은 기본적으로 데이터를 3중 복제하는데, 이 복제본들을 서로 다른 랙에 흩어 놓음으로써 하드웨어 장애에 대비한다.

### 💡 비유
랙 인식은 **"소중한 보물을 보관하는 은행 금고 분산 전략"**과 같다.
- **랙 인식 없음**: 보물 지도 3장을 모두 'A빌딩 1층' 금고 3개에 나눠 보관한다. 만약 빌딩에 불이 나거나 정전이 되면(랙 스위치 장애), 금고 3개가 모두 있어도 지도를 꺼낼 수 없다.
- **랙 인식 있음**: 지도 1장은 'A빌딩 1층'에, 나머지 2장은 'B빌딩 3층'에 보관한다. A빌딩이 통째로 무너져도 B빌딩에 지도가 살아있으므로 보물을 찾을 수 있다. 또한, 같은 빌딩 사람들끼리 지도를 주고받는 게 옆 빌딩까지 가는 것보다 훨씬 빠르다(네트워크 효율).

### 등장 배경 및 발전 과정

#### 1. 대규모 클러스터의 물리적 한계
빅데이터 클러스터는 수천 대의 서버로 구성된다. 이 서버들은 40~80대씩 묶여 '랙(Rack)'이라는 철제 선반에 꽂히며, 각 랙 상단에는 모든 서버를 연결하는 'L2/L3 스위치'가 있다. 이 스위치가 죽으면 랙 전체 서버가 네트워크에서 고립된다.

#### 2. 데이터 가용성 vs 네트워크 비용의 딜레마
데이터를 무조건 다른 랙에만 저장하면 랙 사이를 연결하는 '백본망' 트래픽이 폭주한다. 반면 같은 랙에만 저장하면 스위치 장애 시 데이터를 잃는다. 하둡은 이 두 마리 토끼를 잡기 위해 **'2+1 복제 규칙'**이라는 천재적인 절충안을 내놓았다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### HDFS 블록 배치 정책 (3중 복제 기준) (표)

| 복제본 순서 | 배치 위치 | 목적 | 비유 |
|-------------|-----------|------|------|
| **1번째 복제** | 클라이언트가 실행 중인 로컬 노드 | 쓰기 속도 극대화 및 네트워크 부하 제로 | 내 책상 위에 두기 |
| **2번째 복제** | **다른 랙(Remote Rack)**의 임의의 노드 | 랙 전체 장애(ToR Switch 고장)에 대비한 안전장치 | 옆집 친구네 두기 |
| **3번째 복제** | **2번째와 같은 랙**의 다른 노드 | 랙 간 트래픽(Cross-rack traffic)을 줄이면서 가용성 확보 | 옆집의 다른 방에 두기 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ Hadoop Rack Awareness Topology & Data Replication ]
================================================================================

        [ 🏢 Core Switch / Backbone ]
               /            \
      ┌───────/───┐        ┌──\────────┐
      │ [ Rack 1 ] │        │ [ Rack 2 ] │
      │ ToR Switch │        │ ToR Switch │
      └─────┬──────┘        └─────┬──────┘
      ┌─────┴─────┐        ┌──────┴──────────┐
      │ Node A (1)│        │ Node D (2) ⭐️   │
      ├───────────┤        ├─────────────────┤
      │ Node B    │        │ Node E (3) ⭐️   │
      ├───────────┤        ├─────────────────┤
      │ Node C    │        │ Node F          │
      └───────────┘        └─────────────────┘

 [ Replication Flow ]
 (1) Client가 Node A에서 쓰기 시작 -> 1번 블록 Node A에 저장 (Local)
 (2) NameNode: "안전을 위해 다른 랙으로 가라!" -> 2번 블록 Node D에 저장 (Remote Rack)
 (3) NameNode: "비용 절감을 위해 같은 랙에 하나 더!" -> 3번 블록 Node E에 저장 (Same Rack as 2)

 * 효과: Rack 2 스위치가 타버려도 Rack 1(Node A)에 데이터가 살아있음!
```

### 심층 동작 원리

#### ① 토폴로지 인식 (Network Topology)
하둡은 트리 구조의 네트워크 지도를 그린다. 각 노드는 `/default-rack/node1` 과 같은 경로명을 가진다. 
관리자가 `topology.script.file.name` 설정을 통해 각 IP가 어느 랙에 속하는지 알려주면, 네임노드는 이를 기반으로 노드 간의 **'거리(Distance)'**를 계산한다.
- 같은 노드: 거리 0
- 같은 랙, 다른 노드: 거리 2
- 다른 랙: 거리 4

#### ② 데이터 지역성 (Data Locality)
맵리듀스나 스파크(Spark) 작업을 돌릴 때, 하둡은 "데이터가 있는 노드로 프로그램을 보내는" 방식을 쓴다.
1. **Node Local**: 데이터가 있는 그 컴퓨터에서 연산 (최상)
2. **Rack Local**: 데이터가 옆 컴퓨터에 있어 랙 스위치만 거쳐서 가져옴 (우수)
3. **Off Switch**: 데이터가 다른 랙에 있어 백본망을 타고 가져옴 (최악)
랙 인식이 되어 있어야만 스케줄러가 프로그램을 2번(Rack Local) 위치로 보낼 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### 랙 인식 적용 전/후 비교

| 비교 지표 | 랙 인식 미설정 (Flat) | 랙 인식 설정 (Aware) |
|-----------|------------------------|----------------------|
| **데이터 안정성** | 낮음 (한 랙이 죽으면 데이터 유실 가능) | **매우 높음 (랙 장애에도 100% 생존)** |
| **네트워크 부하** | 높음 (랜덤 배치로 백본 트래픽 증가) | **최적화 (랙 내 통신 유도로 대역폭 절약)** |
| **연산 효율** | 보통 (데이터 위치 고려 부족) | **최상 (Data Locality 기반 스케줄링)** |
| **네임노드 부하** | 낮음 | 약간 높음 (복잡한 배치 계산 필요) |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 500대 규모의 하둡 클러스터를 운영 중인데, 최근 데이터센터 전력 공사 중 랙 하나가 통째로 다운되는 사고가 발생했다. 랙 인식을 설정하지 않아 운이 없게도 3개 복제본이 모두 해당 랙에 있던 파일들이 발견되어 서비스 장애가 터졌다.
**판단 및 턴어라운드 전략**:
1. **토폴로지 스크립트 작성**: 서버 IP 대역별로 랙 번호를 매핑하는 파이썬 스크립트를 작성하여 모든 노드에 배포한다.
2. **HDFS Balancer 실행**: 랙 인식 설정을 켠 후, 기존에 랙 편향적으로 저장되어 있던 블록들을 강제로 재배치(`hdfs balancer`)한다.
3. **네트워크 대역폭 모니터링**: 랙 간 통신(L3) 부하가 줄어드는지 확인하고, 스파크 작업의 'Rack Local' 비중이 늘어나는지 체크한다.
4. **결과**: 향후 어떤 랙이 통째로 폭발(?)하더라도 데이터 유실 제로(Zero Data Loss)를 보장하며, 클러스터 전체 처리량(Throughput)이 약 15% 향상된다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **가상화 환경(VM)에서의 랙 인식**: 퍼블릭 클라우드(AWS, Azure)에서는 사용자가 물리 랙 위치를 알 수 없다. 이때 무턱대고 가짜 랙 번호를 부여하면 오히려 성능이 왜곡된다. 클라우드에서는 랙 대신 **가용 영역(Availability Zone)** 정보를 랙 인식 정보로 활용하여 데이터 센터 장애에 대비하는 것이 표준이다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **가용성 향상**: 하드웨어 가용성(SLA)을 99.9%에서 랙 인식을 통해 **99.999%**로 끌어올림.
- **백본 비용 절감**: 랙 외부로 나가는 트래픽을 **30~50% 감소**시켜 값비싼 코어 스위치 부하 경감.

### 미래 전망 및 진화 방향
클라우드 네이티브 빅데이터 환경(EMR, Dataproc)에서는 사용자가 수동으로 랙 설정을 할 필요가 없다. 클라우드 사업자가 인프라의 물리적 배치를 이미 알고 있으므로, API를 통해 자동으로 'Zone-aware' 배치를 수행한다. 나아가 최근에는 하드웨어의 전력 소모나 온도까지 고려하여 데이터를 배치하는 **'Green-aware' 데이터 배치** 기술로 진화하며 ESG 경영 시대의 인프라 효율화를 꾀하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **랙 인식이 뭔가요?**: 학교에서 모둠별로 숙제를 나눠 보관할 때, 한 모둠 친구들한테만 숙제 검사지를 다 주지 않고, 다른 반 친구한테도 한 장 맡겨두는 지혜로운 방법이에요.
2. **왜 그렇게 하나요?**: 우리 반 친구들이 단체로 독감에 걸려서 학교에 못 와도(랙 장애), 옆 반 친구한테 맡겨둔 검사지가 있으면 숙제를 제출할 수 있으니까요!
3. **가장 좋은 점은요?**: 같은 반 친구들끼리 검사지를 주고받는 게 복도를 지나 옆 반까지 가는 것보다 훨씬 빠르고 편해서 공부 효율이 쑥쑥 올라간답니다!
