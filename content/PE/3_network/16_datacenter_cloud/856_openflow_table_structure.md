# 856. OpenFlow Flow Table 구조 - 매치, 액션, 카운터 엔트리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SDN 스위치의 데이터 평면에서 패킷을 처리하는 가장 핵심적인 데이터 단위로, 패킷의 특징을 찾는 **Match Field**, 처리 방식을 정의한 **Instruction/Action**, 그리고 처리 이력을 기록하는 **Counter**로 구성된 데이터 엔트리 집합이다.
> 2. **가치**: 기존의 경직된 라우팅 테이블(L3)이나 MAC 테이블(L2)과 달리, L2~L4의 모든 헤더를 조합하여 '사용자 정의 흐름'을 생성할 수 있게 함으로써 네트워크 제어의 유연성을 극대화한다.
> 3. **융합**: 다단계 파이프라인 처리(Multi-stage Table), 우선순위(Priority) 기반 충돌 해결, 그리고 하드웨어 가속 기술(TCAM)이 융합되어 복잡한 네트워크 정책을 소프트웨어 속도로 처리한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
OpenFlow Flow Table은 스위치 내부에 존재하는 '지시서 목록'이다. 패킷이 스위치에 들어오면, 스위치는 이 테이블을 위에서부터 아래로 훑으며 패킷과 일치하는 규칙(Entry)을 찾고, 그에 따라 패킷을 배달하거나 폐기한다.

### 💡 비유
Flow Table은 **"공항 입국 심사대의 업무 매뉴얼"**과 같다.
- **Match (심사 대상)**: 여권 국적, 비자 종류, 여행 목적 등을 확인한다.
- **Action (심사 결과)**: "입국 허가(Forward)", "추가 조사실 이동(Go to Table)", "입국 거부(Drop)"를 결정한다.
- **Counter (통계)**: 오늘 입국한 미국인이 몇 명인지 수첩에 체크한다.
매뉴얼(Flow Table)만 잘 짜여 있으면 심사관(스위치)은 고민 없이 기계적으로 빠르게 일을 처리할 수 있다.

### 등장 배경 및 발전 과정

#### 1. 고정된 기능의 한계
전통적 스위치는 "IP 주소가 뭐니?"만 물어볼 수 있었다. 하지만 클라우드에서는 "이 패킷이 A 고객의 가상 망에서 온 거니?" 같은 복잡한 질문이 필요해졌고, 이를 수용하기 위해 유연한 테이블 구조가 설계되었다.

#### 2. 파이프라인의 도입 (OpenFlow 1.1+)
초기에는 테이블이 1개뿐이라 조건이 많아지면 테이블이 너무 길어졌다. 이후 '다단계 파이프라인' 구조를 도입하여, 1번 테이블에서는 VLAN만 보고, 2번 테이블에서는 IP만 보는 식으로 효율적인 처리가 가능해졌다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### Flow Entry의 6대 구성 요소 (표)

| 구성 요소 | 상세 내용 | 비유 |
|-----------|-----------|------|
| **Match Fields** | Ingress Port, MAC, IP, TCP/UDP 포트 등 패킷 헤더 정보 | 검색 조건 |
| **Priority** | 규칙 간의 우선순위. 높은 숫자가 먼저 적용됨 | 우선순위 |
| **Counters** | 처리된 패킷 수, 바이트 수, 활성화 시간 등 | 통계 장부 |
| **Instructions** | 다음 테이블로 가기(Go-To), 액션 실행(Apply-Actions) 등 | 다음 단계 안내 |
| **Timeouts** | 규칙이 테이블에 머무는 시간 (Idle / Hard) | 유효 기간 |
| **Cookie** | 컨트롤러가 관리용으로 붙인 고유 식별자 | 관리 번호 |

### 정교한 구조 다이어그램 (파이프라인 처리 과정)

```ascii
================================================================================
[ OpenFlow Flow Table Pipeline Architecture ]
================================================================================

  (Incoming Packet) 
         │
         ▼
  ┌──────────────────────┐
  │ ⭐️ Flow Table 0      │
  ├──────────────────────┤      (Match Found!)
  │ Match: VLAN 10       │ ──▶ [ Instruction: Write Metadata=1, Go to Table 1 ]
  │ Match: Any           │
  └──────────────────────┘
         │
         ▼
  ┌──────────────────────┐
  │ ⭐️ Flow Table 1      │
  ├──────────────────────┤      (Match Found!)
  │ Match: IP 8.8.8.8    │ ──▶ [ Action: Output Port 3, Dec TDL ]
  │ Match: Metadata=1    │ ──▶ [ Action: Set Queue 5, Go to Table 2 ]
  └──────────────────────┘
         │
         ▼
  ┌──────────────────────┐
  │ ⭐️ Flow Table 2      │
  ├──────────────────────┤      (Match Found!)
  │ Match: TCP Port 80   │ ──▶ [ Action: Modify Header (NAT), Output Port 1 ]
  └──────────────────────┘
         │
         ▼
  (Outgoing Packet)
```

### 심층 동작 원리

#### ① 다단계 파이프라인 (Multiple Tables)
패킷은 0번 테이블부터 시작하여 `Go-To-Table` 명령을 통해 다음 테이블로 이동할 수 있다. 
- **이유**: 만약 조건 A가 10개, 조건 B가 10개라면, 단일 테이블에서는 모든 조합인 100개의 규칙이 필요하다. 하지만 다단계 테이블을 쓰면 10+10=20개의 규칙만으로도 처리가 가능하다 (기하급수적 메모리 절약).

#### ② 메타데이터(Metadata) 전파
테이블 0에서 판별한 결과(예: "이 패킷은 우수 회원 트래픽임")를 메모리 공간인 'Metadata'에 기록하여 다음 테이블로 넘길 수 있다. 다음 테이블은 패킷 헤더뿐만 아니라 이 메타데이터를 보고 더 지능적인 판단을 내린다.

## Ⅲ. 융합 비교 및 다각도 분석

### 전통적 L2/L3 테이블 vs OpenFlow Flow Table

| 비교 지표 | MAC / Routing Table | OpenFlow Flow Table |
|-----------|----------------------|----------------------|
| **검색 필드** | 고정 (MAC or IP) | **가변 (L2 ~ L4 조합)** |
| **동작(Action)**| Forwarding only | **Forward, Drop, Header Rewrite, Meter 등** |
| **갱신 주체** | 라우팅 프로토콜 (BGP/OSPF) | **SDN 컨트롤러 (OpenFlow)** |
| **복잡도** | 낮음 | **높음 (유연한 정책 구현 가능)** |
| **메모리 활용**| 전용 칩 (CAM) | **범용/고속 칩 (TCAM)** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 사내 가상화 망에서 특정 웹 서버에 대한 디도스(DDoS) 공격이 감지되었다. 공격자는 IP를 수시로 바꾸지만, 패킷 크기가 1234바이트로 일정하고 특정 TCP 플래그를 사용한다는 특징이 있다.
**판단 및 Flow Table 설계 전략**:
1. **L4-level Matching**: `TCP Flags`와 `Packet Length`를 Match 필드로 사용하는 긴급 차단 규칙을 생성한다.
2. **Highest Priority 설정**: 기존의 일반 웹 허용 규칙보다 높은 우선순위(예: 65535)를 부여하여 테이블 최상단에 배치한다.
3. **Drop Action**: 일치하는 패킷은 즉시 폐기(Drop)하도록 지시한다.
4. **결과**: 정상적인 사용자 트래픽에는 영향을 주지 않으면서, 공격 패킷만 핀셋으로 솎아내듯 0.001초 만에 하드웨어 단에서 차단한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **우선순위(Priority) 설계 부재**: 모든 규칙의 우선순위를 똑같이 주면, 패킷이 어떤 규칙에 걸릴지 예측할 수 없다(비결정론적 동작). 반드시 **'세밀한 규칙은 높게, 일반적인 규칙은 낮게'** 계층적으로 우선순위를 부여해야 보안 구멍을 막을 수 있다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **보안 세분화**: 사용자/앱 단위의 마이크로 세그먼테이션 정책을 하드웨어 성능 저하 없이 **100% 수용**.
- **네트워크 가상화**: 물리 장비 하나에 수천 개의 독립적인 Flow Table을 구성하여 가상 네트워크 효율 **50% 향상**.

### 미래 전망 및 진화 방향
Flow Table 구조는 이제 **P4 기반의 '완전 가변 테이블'**로 진화하고 있다. OpenFlow는 정해진 필드(IP, Port 등)만 Match 할 수 있었지만, 미래의 테이블은 개발자가 "패킷의 50번째 바이트부터 4비트를 읽어라"라고 직접 정의하는 수준에 이르렀다. 이는 위성 통신이나 특수 산업용 프로토콜 등 표준화되지 않은 영역까지 소프트웨어 정의 네트워크가 장악하는 시대를 열 것이다.
