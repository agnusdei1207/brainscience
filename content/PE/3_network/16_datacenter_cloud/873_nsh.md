# 873. NSH (Network Service Header) - 패킷의 비즈니스 클래스 티켓

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서비스 체이닝(SFC) 환경에서 패킷이 어떤 가상 네트워크 기능(VNF)들을 거쳐야 하는지에 대한 **경로 정보와 메타데이터를 담은 특수 네트워크 헤더** 규격(RFC 8300)이다.
> 2. **가치**: 기존의 복잡한 VLAN/IP 기반의 경로 제어 한계를 넘어서, 패킷 본체(Payload)와 분리된 독립된 '서비스 지도'를 제공함으로써 동적이고 유연한 네트워크 서비스 배포를 가능케 한다.
> 3. **융합**: VXLAN-GPE나 GRE와 같은 터널링 프로토콜과 융합되어 전송되며, SDN 컨트롤러가 정의한 지능형 경로(Service Path)를 물리적 토폴로지 변경 없이 실시간으로 구현한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
NSH(Network Service Header)는 패킷 앞에 붙는 일종의 '수행 비서'다. 패킷이 데이터센터에 들어오면, NSH가 붙어서 "이 패킷은 먼저 1번 방화벽에 들렀다가, 2번 가속기를 거쳐서 나가세요"라고 안내한다. 각 장비를 거칠 때마다 NSH 내부의 인덱스가 줄어들며 다음 목적지를 가리킨다.

### 💡 비유
NSH는 **"놀이공원의 우선 탑승 패스(Magic Pass)"**와 같다.
- **일반 패킷**: 줄을 서서 이정표를 보고 길을 찾아야 한다. (IP 라우팅).
- **NSH가 붙은 패킷**: 패스권(NSH)에 가야 할 놀이기구 순서([자이로드롭] -> [롤러코스터] -> [회전목마])가 이미 적혀 있다. 입구에서 패스를 보여주면 직원이 바로 다음 기구로 안내해준다. 기구를 하나 탈 때마다 티켓에 도장을 찍어(Service Index 감소) 다음 기구가 무엇인지 표시한다.

### 등장 배경 및 발전 과정

#### 1. 토폴로지 종속성의 한계
예전에는 패킷을 특정 장비로 보내려면 선을 직접 연결하거나 복잡한 VLAN 설정을 해야 했다. 장비 위치가 바뀌면 네트워크 전체가 꼬였다.

#### 2. SFC(Service Function Chaining) 표준화
IETF(국제 인터넷 표준화 기구)는 "패킷 자체에 경로를 심자"는 아이디어를 냈고, 2018년 NSH를 표준으로 확정했다. 이제 장비가 어디에 있든 NSH만 있으면 논리적으로 사슬처럼 엮을 수 있게 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### NSH 헤더의 3대 구성 요소 (표)

| 요소명 | 상세 역할 | 핵심 데이터 | 비유 |
|--------|----------|-------------|------|
| **Base Header** | 버전 및 전체 길이 정보 | 버전, 프로토콜 타입 | 티켓의 위조 방지 홀로그램 |
| **Service Path Header**| **(핵심)** 경로 제어 | **SPI** (경로 ID), **SI** (순서 인덱스) | 가야 할 코스 번호와 현재 순서 |
| **Context Headers**| 부가 정보 전달 | 사용자 ID, 보안 등급, 과금 정보 | VIP 고객 등급 정보 |

### 정교한 구조 다이어그램 (NSH 기반 SFC 흐름)

```ascii
================================================================================
[ NSH-based Packet Journey: Decrementing Service Index ]
================================================================================

  [ Ingress Node ] ──▶ [ Classifier ] ── (Add NSH: SPI=100, SI=255) ──┐
                                                                      │
  ┌───────────────────────────────────────────────────────────────────┴───────┐
  │  🌐 Transport Network (Underlay)                                          │
  │                                                                           │
  │  [ SFF 1 ] ─────▶ [ SFF 2 ] ─────▶ [ SFF 3 ] ─────▶ [ Egress Node ]       │
  │      │               │               │                                    │
  └──────┼───────────────┼───────────────┼────────────────────────────────────┘
         │               │               │
 [ SF 1: Firewall ]  [ SF 2: IPS ]   [ SF 3: Logger ]
 (SI: 255 -> 254)    (SI: 254 -> 253) (SI: 253 -> 252)
 
 * 동작 원리: 
   1. SFF(Forwarder)가 NSH를 보고 SF(Function)에게 패킷 전달.
   2. SF는 일을 마친 후 SI 값을 1 감소시켜 SFF에게 돌려줌.
   3. SFF는 줄어든 SI를 보고 다음 목적지(SF 2)를 결정.
```

### 심층 동작 원리

#### ① SPI (Service Path Identifier)
이 패킷이 타야 할 '노선 번호'다. 예를 들어 SPI 100번은 [보안 특화 노선], 200번은 [미디어 가속 노선] 등으로 정의된다.

#### ② SI (Service Index)
경로 내에서의 현재 위치다. 보통 255부터 시작해서 장비를 하나 거칠 때마다 1씩 줄어든다. SI가 0이 되면 모든 서비스를 마친 것으로 간주하고 목적지로 보낸다.

#### ③ 메타데이터 전파 (Context Metadata)
NSH의 가장 강력한 기능이다. 앞단의 방화벽이 패킷을 검사해서 "이 사용자는 위험도가 높음"이라는 결과를 NSH에 적어두면, 뒷단의 로그 수집기가 이 정보를 보고 더 상세한 로그를 남길 수 있다. 장비끼리 대화할 필요 없이 패킷이 스스로 정보를 날라준다.

## Ⅲ. 융합 비교 및 다각도 분석

### NSH vs 캡슐화 기술 비교 (VXLAN / GRE)

| 비교 지표 | 일반 VXLAN / GRE | **NSH (Network Service Header)** |
|-----------|------------------|-----------------------------------|
| **주요 목적** | 터널링 (L2 over L3) | **서비스 경로 제어 (SFC)** |
| **상태 유지** | 목적지만 알면 됨 | **어디를 거쳐왔는지 기억함** |
| **메타데이터** | 없음 (고정 헤더) | **사용자 정의 데이터 운반 가능** |
| **융합 방식** | 운송 수단 | **운송 수단 안에 든 화물 명세서** |
| **비유** | 택배 트럭 | **택배 상자의 배송 추적표** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 통신사 클라우드에서 기업 고객들에게 "맞춤형 네트워크 보안 체인"을 제공하려 한다. 고객사마다 거쳐야 하는 보안 장비의 종류와 순서가 모두 다르다.
**판단 및 NSH 적용 전략**:
1. **Control Plane 구축**: SDN 컨트롤러(ONOS 등)를 통해 고객별 SPI(경로 ID)를 미리 정의한다.
2. **NSH-aware SFF 배치**: 가상 스위치인 **Open vSwitch(OVS)**를 NSH를 인식할 수 있는 버전으로 빌드하여 배포한다.
3. **Variable Metadata 활용**: 첫 번째 인증 장비에서 사용자 등급을 NSH 메타데이터에 심어, 이후 모든 보안 장비가 별도의 DB 조회 없이 정책을 즉시 적용하게 한다.
4. **결과**: 물리 배선 변경 없이 소프트웨어 클릭만으로 **수천 가지의 개인화된 보안망**을 1분 만에 개통한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **장비의 NSH 미지원**: 모든 보안 장비(VNF)가 NSH를 알아들을 수 있는 것은 아니다. NSH를 모르는 장비(Legacy SF) 앞에 패킷에서 NSH를 떼었다가 다시 붙여주는 **'SFC Proxy'**를 반드시 배치해야 한다. 이를 누락하면 패킷이 Legacy 장비에서 멈춰버리는 사고가 난다.

---

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **운영 유연성**: 서비스 경로 변경 시간 **수 주 -> 수 초** 단축.
- **네트워크 효율**: 불필요한 장비를 거치지 않고 필요한 서비스만 골라 타게 함으로써 대역폭 30% 절감.

### 미래 전망 및 진화 방향
NSH는 이제 **'인텐트 기반 네트워킹(IBN)'**과 결합하고 있다. 사람이 경로를 짜지 않아도, AI가 보안 위협을 감지하면 즉시 NSH 인덱스를 수정해 패킷을 검사실(Honey-pot)로 격리하는 **'자율 대응 SFC'**가 연구되고 있다. 또한, 6G 시대에는 위성망과 지상망 사이를 오가는 패킷의 복잡한 국가별 규제(Sovereignty)를 준수하기 위한 '법적 경로 증명서'로서 NSH의 역할이 더욱 커질 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **NSH가 뭔가요?**: 컴퓨터 세상의 편지(패킷) 겉면에 붙이는 '특별한 스탬프'예요. 
2. **뭐라고 써 있나요?**: "이 편지는 먼저 1번 방패 아저씨한테 가서 검사받고, 그다음에 2번 요정한테 가서 응원받으세요"라는 지도가 그려져 있어요.
3. **왜 좋은가요?**: 편지가 스스로 길을 알고 있어서, 복잡한 미로(네트워크) 속에서도 헤매지 않고 정확한 순서대로 심부름을 다녀올 수 있게 도와주기 때문이에요!
