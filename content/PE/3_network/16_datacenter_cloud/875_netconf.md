# 875. NETCONF (Network Configuration Protocol) - 네트워크 관리의 현대적 표준

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 네트워크 장비의 설정(Configuration)과 상태 정보를 관리하기 위해 정의된 **XML 기반의 프로토콜**(RFC 6241)로, 기존 SNMP의 한계를 극복하고 프로그래밍 가능한 네트워크 관리를 실현하는 핵심 기술이다.
> 2. **가치**: 설정의 '커밋(Commit)'과 '롤백(Rollback)' 기능을 하드웨어 레벨에서 지원하여 대규모 망 변경 시의 안정성을 보장하며, 데이터 모델링 언어인 **YANG**과 결합하여 벤더에 독립적인 일관된 관리 환경을 제공한다.
> 3. **융합**: SSH 보안 전송, 트랜잭션 관리 매커니즘, 그리고 SDN 컨트롤러의 사우스바운드(Southbound) API와 융합되어 클라우드 데이터센터의 자동화된 운영(NetDevOps)을 가능케 한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
NETCONF는 "장비에게 XML로 된 명령서를 보내는 법"이다. 사람이 일일이 CLI(명령줄)를 치는 게 아니라, 컴퓨터 프로그램이 장비에게 "이 설정을 적용하고, 성공하면 보고해!"라고 원격으로 지시하는 프로토콜이다.

### 💡 비유
NETCONF는 **"은행의 전산 뱅킹 시스템"**과 같다.
- **과거 (SNMP)**: 장부의 한 줄만 쓱 보고 가거나, 숫자를 살짝 고치는 수준. (복잡한 일은 못 함).
- **CLI (수동)**: 사람이 직접 장부를 들고 펜으로 적는 것. 실수할 위험이 크고 시간이 오래 걸림.
- **NETCONF (뱅킹 시스템)**: "A계좌에서 B계좌로 100원 이체해"라는 **트랜잭션**을 보낸다. 돈이 빠져나갔는데 입금이 안 되면 전체를 취소(Rollback)한다. 모든 과정은 전산 기록(XML)으로 남고 안전하게 전송된다.

### 등장 배경 및 발전 과정

#### 1. SNMP의 한계
SNMP는 장비 모니터링에는 좋았지만, 설정을 바꾸는 기능은 매우 취약했다. 복잡한 설정을 한 번에 바꾸기도 힘들었고 보안도 약했다.

#### 2. 네트워크 자동화의 요구 (2006년 표준화)
수만 대의 서버가 있는 클라우드에서 장비 설정을 수동으로 할 수 없게 되자, IETF는 "소프트웨어가 다루기 쉬운(XML 기반) 안정적인 설정 규격"인 NETCONF를 개발했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### NETCONF의 4계층 아키텍처 (표)

| 계층 (Layer) | 상세 내용 | 비유 |
|--------------|-----------|------|
| **Content** | 실제 설정 데이터 (**YANG 모델** 기반 XML) | 편지 내용 |
| **Operations** | 어떤 행위를 할지 결정 (`<get-config>`, `<edit-config>`) | 편지의 목적 (조회 혹은 수정) |
| **Messages** | RPC 기반의 요청/응답 구조 (`<rpc>`, `<rpc-reply>`) | 편지 봉투 (보내는 양식) |
| **Transport** | 데이터를 안전하게 실어 나름 (**SSH**, TLS) | 등기 우편 서비스 |

### 정교한 구조 다이어그램 (트랜잭션 기반 설정 흐름)

```ascii
================================================================================
[ NETCONF Operation: Safe Configuration with Candidate Store ]
================================================================================

  [ 🧠 SDN Controller / NMS ]             [ 🏗️ Network Device ]
           │                                      │
  (1) <edit-config> (XML)                         │
      "Change IP to 1.1.1.1"                      │
           └──────────────────▶ (2) Update 📂 Candidate Config
                                    (실제 반영 전 '임시 저장소'에 저장)
                                                  │
  (3) <validate>                                  │
           └──────────────────▶ (4) Check syntax & Logic ✅
                                                  │
  (5) <commit>                                    │
           └──────────────────▶ (6) Copy Candidate to 📂 Running Config
                                    (⭐️ 실제 서비스에 즉시 반영 ⭐️)
                                                  │
  (7) <rpc-reply> (OK) ◀──────────────────────────┘
```

### 심층 동작 원리

#### ① 데이터 저장소 (Datastores)의 분리
NETCONF는 장비 내에 세 가지 저장소를 둔다.
- **Running**: 현재 서비스에 돌아가고 있는 진짜 설정.
- **Candidate**: 내가 지금 고치고 있는 임시 설정. (Commit 전까지 서비스 영향 Zero).
- **Startup**: 장비가 재부팅될 때 읽어오는 설정.

#### ② 원자적 트랜잭션 (Atomicity)
여러 설정을 한꺼번에 보냈을 때, 하나라도 실패하면 전체를 예전 상태로 되돌린다. "전부 성공하거나, 전부 실패하거나"를 보장하여 네트워크가 절반만 설정되어 꼬이는 현상을 막는다.

#### ③ 잠금(Locking) 기능
내가 설정을 고치는 동안 다른 관리자나 프로그램이 접근하지 못하도록 장부 전체에 락을 걸 수 있다. (동시성 제어).

## Ⅲ. 융합 비교 및 다각도 분석

### CLI vs SNMP vs NETCONF 비교

| 비교 지표 | CLI (Standard) | SNMP | **NETCONF** |
|-----------|----------------|------|-------------|
| **형식** | 비정형 텍스트 | 바이너리 (PDU) | **정형화된 XML** |
| **자동화** | 어려움 (Parsing 필수) | 쉬움 (모니터링 위주) | **최상 (API 기반)** |
| **트랜잭션** | 없음 | 없음 | **있음 (Commit/Rollback)** |
| **보안** | 장비별 상이 | 취약 (v1/v2) | **강력 (SSH 필수)** |
| **비유** | 사람이 수다 떨기 | 무전기로 신호 보내기 | **공식 공문 주고받기** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 전국 1,000개 매장의 라우터 보안 정책을 동시에 업데이트해야 한다. 한 곳이라도 설정이 잘못되면 해당 매장은 카드 결제가 안 된다.
**판단 및 자동화 전략**:
1. **NETCONF 기반 배포**: Python의 **ncclient** 라이브러리를 사용하여 XML 형식의 정책을 생성한다.
2. **Candidate -> Validate**: 설정을 바로 적용하지 않고 `Candidate` 저장소에 넣은 뒤, `Validate` 명령어로 장비가 스스로 문법 에러를 체크하게 한다.
3. **Confirmed Commit**: `Confirm-timeout` 옵션을 걸고 커밋한다. 만약 설정 후 관리자의 최종 확인이 없으면 10분 뒤 자동으로 옛날 설정으로 돌아가게(Auto-rollback) 배수진을 친다.
4. **결과**: 인적 설정 오류로 인한 매장 장애 발생률을 **0%**로 억제하며, 단 1시간 만에 전사 보안 패치를 완료한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **YANG 모델 무시**: NETCONF는 껍데기고 실제 데이터는 **YANG**이다. 제조사마다 미세하게 다른 YANG 모델을 무시하고 공통 코드를 짜면 실행 시 에러가 난다. 반드시 **'OpenConfig'**와 같은 벤더 중립적 표준 모델을 지원하는지 확인해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **운영 안정성**: 잘못된 설정으로 인한 네트워크 장애 복구 시간 **90% 단축**.
- **배포 속도**: CLI 대비 네트워크 대규모 변경 속도 **10배 이상 향상**.

### 미래 전망 및 진화 방향
NETCONF는 이제 **RESTCONF**로 진화하며 웹 개발자들에게 더 친숙한 형태로 변하고 있다. 또한, 텔레메트리(Telemetry) 기술과 결합하여 장비가 "나 지금 아파!"라고 실시간으로 알려주는 **'Push-based Monitoring'**의 기반이 되고 있다. AI가 망 상태를 분석하고 NETCONF로 설정을 교정하는 **'Self-Driving Network'**의 필수 인터페이스로 영원히 자리 잡을 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **NETCONF가 뭔가요?**: 멀리 떨어진 로봇(라우터)에게 "이 일을 해!"라고 적은 '마법의 편지'를 보내는 방법이에요.
2. **왜 그냥 말로 안 하나요?**: 말로 하면 로봇이 잘못 알아들을 수도 있지만, 정해진 양식(XML)에 맞춰 편지를 쓰면 로봇이 절대로 실수하지 않기 때문이에요.
3. **가장 좋은 점은요?**: 로봇이 일을 하다가 실수하면, "잠깐! 아까 하던 대로 다시 돌아가!"라고 명령 한마디로 순식간에 원래대로 되돌릴 수 있답니다!
