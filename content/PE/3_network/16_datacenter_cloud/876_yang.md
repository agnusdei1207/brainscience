# 876. YANG (Yet Another Next Generation) - 네트워크 데이터의 설계도

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 네트워크 장비의 설정(Configuration)과 상태(State) 데이터를 정의하기 위해 고안된 **데이터 모델링 언어**(RFC 6020)로, NETCONF나 RESTCONF 프로토콜이 실어 나르는 데이터의 구조와 규칙을 명시한다.
> 2. **가치**: 벤더마다 제각각이었던 네트워크 데이터 형식을 표준화(Model-driven)하여, 소프트웨어가 장비의 종류와 상관없이 일관된 방식으로 자원을 관리하고 제어할 수 있는 '상호 운용성'을 제공한다.
> 3. **융합**: XML/JSON 인코딩 기술, SDN 컨트롤러의 추상화 계층(MD-SAL), 그리고 네트워크 자동화 스크립트와 융합되어 '코드로 관리되는 인프라(IaC)'의 논리적 기반이 된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
YANG은 "데이터의 모양을 그리는 문법"이다. 데이터베이스의 스키마(Schema)나 프로그래밍 언어의 인터페이스(Interface) 정의와 유사하다. "이 장비는 '인터페이스 이름'이라는 글자와 '속도'라는 숫자를 데이터로 가진다"라고 컴퓨터가 이해할 수 있게 적어두는 약속이다.

### 💡 비유
YANG은 **"가구 조립 설명서의 규격 정의서"**와 같다.
- **NETCONF**: 설명서를 배달하는 우체부.
- **YANG**: 설명서 안에 "나사는 지름 5mm여야 함", "다리는 4개여야 함"이라고 적힌 **규격** 그 자체다.
이 규격(YANG)이 똑같다면, 이케아 가구든 한샘 가구든 상관없이 똑같은 전동 드릴(SDN 컨트롤러)로 조립할 수 있게 된다.

### 등장 배경 및 발전 과정

#### 1. MIB(Management Information Base)의 한계
과거 SNMP 시절에는 MIB라는 구조를 썼지만, 계층 구조가 복잡하고 설정을 표현하기엔 너무 딱딱했다.

#### 2. 모델 기반 네트워킹 (2010년 표준화)
사람이 명령어를 외우는 시대가 가고, 소프트웨어가 데이터를 다루는 시대가 왔다. IETF는 "사람과 기계가 모두 읽기 편하면서도 엄격한 데이터 규칙을 가진 언어"인 YANG을 만들었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### YANG 모델의 주요 계층 구조 (표)

| 노드 유형 | 상세 설명 | 특징 | 비유 |
|-----------|-----------|------|------|
| **Container** | 관련 있는 데이터들을 묶는 최상위 바구니 | 실제 값을 가지지 않음 | 가구의 '서랍장' 본체 |
| **List** | 동일한 형식이 반복되는 데이터 (예: 여러 인터페이스) | 고유한 Key(이름 등) 필수 | 서랍장 속 '서랍'들 |
| **Leaf** | 실제 데이터 값을 담는 최소 단위 | 문자열, 숫자, 불리언 등 | 서랍 속 '물건' 1개 |
| **Leaf-list** | 특정 타입의 값들이 여러 개 나열된 형태 | 중복 허용 가능 | 서랍 속 똑같은 '양말'들 |

### 정교한 구조 다이어그램 (YANG -> XML 변환 과정)

```ascii
================================================================================
[ YANG Modeling: From Schema to Actual Management Data ]
================================================================================

 [ 📝 YANG Model Definition ]
  module interface-config {
    container interfaces {
      list interface {
        key "name";
        leaf name { type string; }
        leaf mtu { type uint32; default 1500; }
      }
    }
  }
           │
           ▼ (YANG을 기반으로 데이터 생성)
           │
 [ 📂 Instance Data (XML for NETCONF) ]
  <interfaces>
    <interface>
      <name>GigabitEthernet1</name>
      <mtu>9000</mtu>
    </interface>
  </interfaces>
           │
           ▼ (또는 JSON으로 변환)
           │
 [ 📂 Instance Data (JSON for RESTCONF) ]
  { "interfaces": { "interface": [ { "name": "Ge1", "mtu": 9000 } ] } }
```

### 심층 동작 원리

#### ① 강력한 제약 조건 (Constraints)
YANG은 단순히 구조만 짜는 게 아니라 로직도 넣을 수 있다. `must`나 `when` 키워드를 써서 "인터페이스가 활성화된 상태일 때만 속도를 변경할 수 있다"는 비즈니스 규칙을 하드웨어 레벨에서 강제할 수 있다.

#### ② 벤더 독립적 모델 (OpenConfig)
시스코, 주니퍼 등 제조사들은 각자 자기만의 YANG 모델을 제공한다. 하지만 구글 등 대형 사업자들은 **OpenConfig**라는 공통 YANG 모델을 만들어, 제조사에 상관없이 동일한 '이름'과 '구조'로 모든 장비를 조종하려 하고 있다.

#### ③ 증분 업데이트와 RPC 정의
YANG은 데이터 구조뿐만 아니라, 장비가 수행할 특수 동작(`rpc` - 예: `reboot`)과 장비가 보내는 알림(`notification` - 예: "온도 과부하!")의 규격까지 모두 한 파일에 정의한다.

## Ⅲ. 융합 비교 및 다각도 분석

### XML Schema (XSD) vs YANG 비교

| 비교 지표 | XML Schema (XSD) | **YANG** |
|-----------|------------------|----------|
| **주요 목적** | 일반적인 데이터 구조 정의 | **네트워크 장비 제어 특화** |
| **가독성** | 낮음 (XML 형식이 복잡함) | **높음 (C언어 스타일 가독성)** |
| **상태 구분** | 구분 없음 | **Config(설정)와 State(상태) 구분** |
| **확장성** | 보통 | **`augment`를 통한 무한 확장 지원** |
| **비유** | 일반 건축 설계도 | **네트워크 전용 인테리어 설계도** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 서로 다른 제조사의 스위치 3종이 섞인 멀티 벤더 데이터센터. 중앙 컨트롤러에서 "전체 스위치의 MTU 값을 9000으로 통일"하라는 명령을 내려야 한다.
**판단 및 모델링 전략**:
1. **Standard Model 우선**: 제조사 전용(Private) 모델 대신 **OpenConfig YANG 모델**을 지원하는지 확인한다.
2. **YANG Validation**: 배포 전, 오픈소스 도구인 **pyang**을 사용하여 생성한 XML 데이터가 YANG 규격에 맞는지 검증한다. (Parsing Error 사전 차단).
3. **Augmentation 활용**: 만약 표준 모델에 없는 제조사 고유 기능이 필요하다면 `augment` 기능을 써서 표준 모델을 훼손하지 않고 살짝 덧붙여 사용한다.
4. **결과**: 장비마다 명령어를 다르게 칠 필요 없이, 하나의 Python 코드로 3개 회사 장비를 동시에 완벽하게 제어한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **YANG 없는 NETCONF**: 데이터 모델 없이 단순히 XML만 주고받는 것은 "눈 가리고 운전하는 것"과 같다. 데이터의 유효성을 체크할 방법이 없기 때문이다. 반드시 장비로부터 **`.yang` 파일**을 확보하여 개발 도구에 로딩한 뒤 자동화를 시작해야 한다.

---

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **상호 운용성**: 제조사 간 설정 호환성 **100% 확보** (표준 모델 적용 시).
- **개발 생산성**: 데이터 구조가 정형화되어 있어 자동화 코드 개발 시간 **50% 단축**.

### 미래 전망 및 진화 방향
YANG은 이제 네트워크를 넘어 **'디지털 트윈 (Digital Twin)'**의 핵심 언어가 되고 있다. 물리적인 장비의 상태를 가상 세계에 완벽히 복제하기 위해 YANG으로 모든 상태값을 정의하는 것이다. 또한, 차세대 텔레메트리 규격인 **gNMI**와 결합하여, 수조 개의 센서 데이터를 YANG 구조에 담아 초고속으로 실어 나르는 '데이터 중심 네트워킹'의 심장 역할을 수행할 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **YANG이 뭔가요?**: 장난감 로봇(네트워크 장비)에게 명령을 내릴 때 쓰는 '공통 일기장 양식'이에요.
2. **왜 이게 필요한가요?**: 어떤 친구는 일기를 가로로 쓰고, 어떤 친구는 세로로 쓰면 선생님이 읽기 힘들겠죠? 그래서 모두가 똑같은 칸(구조)에 맞춰서 쓰기로 약속한 거예요.
3. **가장 좋은 점은요?**: 일기장 양식이 똑같으니까, 어떤 로봇이든 "3번째 칸에 적힌 숫자를 봐!"라고 한마디만 하면 모두가 똑똑하게 알아듣고 움직인답니다!
