# 879. 텔레메트리 (Streaming Telemetry) - 실시간 지능형 모니터링

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 관리 시스템(NMS)이 장비에게 매번 상태를 물어보는 구식 폴링(Polling) 방식에서 벗어나, **네트워크 장비가 스스로 자신의 상태 데이터를 실시간 스트림 형태로 전송(Push)**하는 차세대 모니터링 기술이다.
> 2. **가치**: 데이터 수집 주기를 분(Minute) 단위에서 마이크로초($\mu s$) 단위로 단축하여 찰나의 패킷 정체나 미세 버스트(Micro-burst)를 놓치지 않고 감지하며, 시스템 부하를 획기적으로 낮춘다.
> 3. **융합**: gRPC 전용 전송 프로토콜, GPB (Google Protocol Buffers) 인코딩, 그리고 시계열 데이터베이스(TSDB)와 융합되어 '자율 운영 네트워크(Self-driving Network)'의 눈 역할을 수행한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
스트리밍 텔레메트리(Streaming Telemetry)는 네트워크 장비의 '생방송 시스템'이다. 장비 내부의 CPU 사용량, 포트별 트래픽, 온도 등 모든 지표를 쉴 새 없이 중앙 수집기(Collector)로 쏴준다.

### 💡 비유
텔레메트리는 **"전통적 건강검진과 스마트워치의 차이"**와 같다.
- **과거 (SNMP Polling)**: 1년에 한 번 병원에 가서 검진을 받는 것. 검사받는 그 순간의 상태만 알 수 있고, 검사 끝나고 다음 날 아픈 것은 알 길이 없다.
- **텔레메트리 (Streaming)**: 스마트워치를 24시간 차고 있는 것. 내 심박수와 걸음수가 실시간으로 서버에 기록된다. 심장이 0.1초만 이상하게 뛰어도 즉시 스마트폰이 경고를 보낸다.

### 등장 배경 및 발전 과정

#### 1. SNMP의 한계 (Polling Overload)
수만 대의 스위치가 있는 IDC에서 5분마다 SNMP로 물어보면, 전체 네트워크 대역폭의 상당량이 "잘 있니?" "응 잘 있어"라는 대화에 낭비된다. 정작 진짜 장애가 났을 때는 그 대화조차 못 해서 감지가 늦어진다.

#### 2. 빅데이터 분석의 시대
네트워크 장애를 사후에 분석하는 게 아니라, AI가 실시간으로 학습해서 장애를 '예측'해야 하는 시대가 왔다. 이를 위해선 초고해상도의 데이터 스트림이 필요했고, 구글이 **gNMI**를 발표하며 텔레메트리 시대가 본격화되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 폴링(SNMP) vs 스트리밍(Telemetry) 비교 (표)

| 비교 지표 | SNMP Polling | **Streaming Telemetry** |
|-----------|---------------|--------------------------|
| **데이터 주체** | 수집기가 물어봄 (Pull) | **장비가 스스로 보냄 (Push)** |
| **수집 주기** | 보통 1분 ~ 5분 | **수 밀리초 ~ 수 초** |
| **오버헤드** | 요청/응답 반복으로 높음 | **연결 유지 후 데이터만 쏴서 낮음** |
| **데이터 모델** | MIB (비정형/파편화) | **YANG (정형/표준화)** |
| **정밀도** | 낮음 (중간 장애 유실) | **매우 높음 (Micro-burst 감지)** |

### 정교한 구조 다이어그램 (gNMI 스트리밍 모델)

```ascii
================================================================================
[ Streaming Telemetry Architecture: gRPC based Push ]
================================================================================

  [ 🏗️ Network Device ]                 [ 📊 Central Collector ]
           │                                      │
  (1) <Subscribe Request> ◀───────────────────────┤
      "Send me /interfaces every 1s"              │
           │                                      │
  (2) Establish ⚡️ gRPC Tunnel (HTTP/2) ──────────▶│
           │                                      │
  (3) 🚀 Continuous Stream (Push)                 │
      ┌──────────────────────┐                    │
      │ 📦 GPB Encoded Data  │ ──────────────────▶│ (4) Time-series DB
      │ { port: 1, bps: 10G }│                    │     (Prometheus / Influx)
      └──────────────────────┘                    │
           │                                      │
  (5) 🚀 Continuous Stream                        │
      ┌──────────────────────┐                    │
      │ 📦 GPB Encoded Data  │ ──────────────────▶│ (6) Real-time Dashboard
      │ { port: 1, bps: 12G }│                    │     (Grafana)
      └──────────────────────┘                    │
```

### 심층 동작 원리

#### ① gRPC와 HTTP/2 전송
텔레메트리는 구글의 **gRPC**를 주 통로로 쓴다. HTTP/2를 기반으로 하므로 하나의 연결로 수천 개의 지표를 동시에 실어 나를 수 있고, 데이터가 없을 때는 조용히 있다가 변화가 생길 때만 쏘는 지능형 전송이 가능하다.

#### ② 프로토콜 버퍼 (Protobuf) 압축
데이터를 JSON이나 XML처럼 텍스트로 보내지 않는다. 이진(Binary) 형태의 **Protocol Buffers**로 압축해서 쏜다. 텍스트 대비 용량이 1/10로 줄어들어, 네트워크 장비의 CPU에 부담을 주지 않으면서도 방대한 정보를 나를 수 있다.

#### ③ 인밴드 텔레메트리 (INT)
한 단계 더 나아간 기술이다. 패킷이 스위치를 통과할 때, 스위치가 패킷의 꼬리에 "나 지금 온도 40도야"라고 메모를 남긴다. 목적지에 도착한 패킷만 분석하면 전 세계 네트워크의 실시간 건강 지도를 그릴 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### 텔레메트리 도입 전/후 장애 감지 능력

| 상황 | 기존 모니터링 (5분 주기) | **텔레메트리 (1초 주기)** |
|------|--------------------------|---------------------------|
| **1초간의 트래픽 폭주** | 감지 불가 (평균치에 묻힘) | **즉시 감지 (Peak 수치 확인)** |
| **간헐적 링크 흔들림** | 5분 뒤에야 "어? 죽었었네?" | **0.1초 만에 알람 발생** |
| **광케이블 품질 저하** | 통신 끊길 때까지 모름 | **빛의 세기 변화 실시간 추적** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대형 증권사의 트레이딩 망. 0.001초의 지연이 수억 원의 손실로 이어진다. 최근 원인 모를 '주문 지연'이 발생했는데, 기존 SNMP 대시보드에는 모든 장비가 '초록불(정상)'로 나와서 원인을 못 찾고 있다.
**판단 및 텔레메트리 고도화 전략**:
1. **gNMI 기반 수집 체계 구축**: 모든 코어 스위치에 gNMI 설정을 켜고, 수집기로 **Telegraf**나 **Cisco Pipeline**을 배치한다.
2. **On-change Subscription**: 지표가 변할 때만 즉시 쏘는 'On-change' 모드를 활성화하여, 대역폭 점유율이 1%라도 변하면 즉시 기록하게 한다.
3. **Micro-burst 분석**: 수집된 데이터를 **Grafana**의 고해상도 대시보드에 뿌린다.
4. **결과**: 기존 5분 주기 대시보드에선 안 보이던 '0.5초간의 비정상적 트래픽 폭주(Micro-burst)' 지점을 찾아내고, 해당 스위치의 버퍼 크기를 튜닝하여 문제를 완벽히 해결한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **수집 서버의 과부하**: 장비 1,000대가 1초마다 데이터를 쏘면 수집 서버가 먼저 죽는다. 반드시 **Kafka**와 같은 메시지 큐를 중간에 두어 데이터를 완충하고, 필요한 지표만 선별적으로 스트리밍하는 전략이 필요하다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **장애 인지 시간(MTTD)**: 기존 대비 **90% 이상 단축** (분 단위 -> 초 단위).
- **인프라 가용성**: 미세한 징후 포착을 통한 선제 대응으로 다운타임 **50% 예방**.

### 미래 전망 및 진화 방향
텔레메트리는 이제 **'인텐트 기반 네트워킹(IBN)'**의 피드백 루프가 되고 있다. AI가 텔레메트리 데이터를 보고 "어? 3번 장비가 곧 죽을 것 같아"라고 예측하여 미리 트래픽을 돌려버리는 **'Self-Healing'** 인프라의 핵심 기술이다. 또한, 6G 시대에는 위성망과 지상망의 초정밀 시각 동기화 데이터를 텔레메트리로 관리하여 우주 인터넷의 안정성을 책임지게 될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **텔레메트리가 뭔가요?**: 컴퓨터 장비들이 자기가 지금 얼마나 아픈지, 배가 얼마나 부른지 중앙 컴퓨터에게 매초마다 알려주는 '실시간 건강 일기'예요.
2. **왜 좋은가요?**: 예전에는 아저씨가 가끔 와서 "너 괜찮니?"라고 물어봐야 답을 할 수 있었는데, 이제는 장비가 스스로 "나 아파요!"라고 바로 소리치기 때문에 금방 고쳐줄 수 있어요.
3. **가장 신기한 점은요?**: 아주 잠깐만 아프고 지나가는 것도 다 기록하기 때문에, 병이 깊어지기 전에 미리미리 약(패치)을 줄 수 있는 똑똑한 방법이랍니다!
