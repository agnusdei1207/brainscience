# 880. 오버레이 SDN과 언더레이 SDN - 2층 침대 같은 네트워크 구조

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 물리적인 스위치와 케이블로 이루어진 실제 전송망인 **언더레이(Underlay)**와, 그 위에서 가상화 기술을 통해 논리적으로 구축된 가상 네트워크인 **오버레이(Overlay)**를 분리하여 제어하는 SDN 설계 모델이다.
> 2. **가치**: 언더레이는 복잡한 기능 없이 '고속 전송'에만 집중하게 하고, 오버레이는 소프트웨어적으로 '복잡한 정책과 격리'를 담당하게 함으로써 대규모 클라우드 환경의 유연성과 성능을 동시에 잡는다.
> 3. **융합**: VXLAN/NVGRE 터널링 프로토콜, BGP-EVPN 제어 기술, 그리고 가상 스위치(OVS)가 융합되어 물리적 경계를 넘나드는 거대 논리 망을 완성한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
- **Underlay SDN**: 실제 눈에 보이는 스위치와 라우터를 직접 제어하는 것. 목적지까지 패킷을 가장 빠르게 보내는 '길'을 닦는 역할.
- **Overlay SDN**: 물리 망 위에 캡슐화(Tunneling)를 통해 만든 가상의 망. 사용자별로 네트워크를 쪼개고 보안 정책을 거는 '서비스' 역할.

### 💡 비유
오버레이와 언더레이는 **"고속도로와 택배 시스템"**의 관계와 같다.
- **언더레이 (고속도로)**: 아스팔트가 깔린 진짜 도로. 트럭이 쌩쌩 달릴 수 있는 물리적 인프라다. 도로가 튼튼하고 넓어야 한다.
- **오버레이 (택배 상자)**: 고속도로를 달리는 트럭 안의 상자들. 상자 겉면에는 "A동 101호 행"이라고 적혀 있다. 고속도로(언더레이)는 상자 안의 내용물(사용자 데이터)을 몰라도 겉면의 주소(터널 헤더)만 보고 목적지 도시까지 빠르게 운반해준다.

### 등장 배경 및 발전 과정

#### 1. VLAN의 확장성 한계
과거에는 물리 망 하나를 쪼개기 위해 VLAN을 썼으나, 최대 4,096개까지만 만들 수 있어 대규모 클라우드에는 부족했다.

#### 2. 네트워크 가상화의 완성
물리 장비(벤더 장비)를 일일이 고치는 대신, 서버 내부의 가상 스위치에서 터널을 뚫어 가짜 망을 만드는 것이 훨씬 빠르고 유연하다는 것을 깨닫고 오버레이 중심의 SDN이 대세가 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 오버레이 vs 언더레이 비교 분석 (표)

| 구분 | 언더레이 (Underlay) | **오버레이 (Overlay)** |
|-----------|----------------------|------------------------|
| **실체** | 물리 스위치, 라우터, 광케이블 | 가상 스위치(OVS), 터널(VXLAN) |
| **주요 프로토콜**| OSPF, IS-IS, BGP (IP Reachability) | **VXLAN, NVGRE, Geneve** |
| **핵심 역할** | 고속 패킷 포워딩, 대역폭 확보 | **테넌트 격리, 보안 정책, 가상 IP** |
| **제어 대상** | 하드웨어 (Leaf-Spine) | 소프트웨어 (Hypervisor / VM) |
| **비유** | 실제 땅과 도로 | 지도 위에 그린 가상의 경계선 |

### 정교한 구조 다이어그램 (Encapsulation 기반 오버레이)

```ascii
================================================================================
[ Overlay SDN: Tunneling over Physical Underlay ]
================================================================================

  [ 📦 Virtual Machine A ]               [ 📦 Virtual Machine B ]
  (IP: 10.0.0.1)                         (IP: 10.0.0.2)
           │                                      │
  ═════════╪══════════════════════════════════════╪═════════════════════════════
  [ 🚀 OVERLAY LAYER (Logical) ]                  │
           │                                      │
  ┌────────▼────────────────┐            ┌────────▼────────────────┐
  │  VTEP (Encapsulation)   │ ──(Tunnel)─▶│  VTEP (Decapsulation)   │
  │  [Outer IP][VXLAN][Data]│            │                         │
  └────────┬────────────────┘            └────────┬────────────────┘
           │                                      │
  ═════════╪══════════════════════════════════════╪═════════════════════════════
  [ 🛤️ UNDERLAY LAYER (Physical) ]                │
           │                                      │
  ┌────────▼──────┐      ┌──────────────┐      ┌──▼────────────┐
  │  Leaf Switch  │◀────▶│ Spine Switch │◀────▶│  Leaf Switch  │
  └───────────────┘      └──────────────┘      └───────────────┘
      (Only knows Outer IP - High Speed Forwarding)

 * 핵심 포인트: 물리 스위치(언더레이)는 내부의 가상 IP(10.0.0.1)를 몰라도 됨. 
   오직 봉투 겉면의 물리 주소(Outer IP)만 보고 배달함.
```

### 심층 동작 원리

#### ① 캡슐화 (Encapsulation)
오버레이의 마법이다. 유저 패킷(L2)을 통째로 '화물'로 취급하여 새로운 IP 패킷(L3) 안에 집어넣는다. 이를 통해 물리적으로 멀리 떨어진 서버들이 마치 하나의 스위치에 꽂힌 것처럼 통신하게 한다.

#### ② 언더레이의 단순화 (IP Fabric)
언더레이 SDN의 목표는 "안 죽고 빠르게 전송하기"다. 복잡한 ACL이나 정책을 다 빼버리고 오직 최단 경로(ECMP)로 패킷을 쏘는 데 집중한다. 주로 **Spine-Leaf** 구조를 사용하여 병목을 없앤다.

#### ③ BGP-EVPN (제어 평면의 통합)
오버레이와 언더레이를 잇는 두뇌다. 어떤 가상 머신(MAC 주소)이 어느 물리 서버(IP 주소)에 있는지에 대한 정보를 BGP 프로토콜을 통해 전 세계 스위치들에 공유한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 기술적 구현 방식의 차이

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **Host-based Overlay** | 서버의 하이퍼바이저에서 캡슐화 수행 | 물리 장비 변경 불필요, 유연성 극상 | 서버 CPU 소모 발생 |
| **Network-based Overlay**| 스위치 하드웨어에서 캡슐화 수행 | 성능 우수 (HW 가속) | 특정 제조사 장비 필요 |
| **Hybrid** | 서버와 스위치가 협력하여 처리 | 성능과 유연성의 조화 | 관리 복잡도 상승 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대규모 데이터센터에 '멀티 테넌트' 클라우드 서비스를 구축하려 한다. 수천 개의 고객사가 각자 독립된 사설망을 원한다. 
**판단 및 아키텍처 전략**:
1. **언더레이 설계**: 물리 스위치들은 **BGP 기반의 L3 패브릭**으로 구성한다. VLAN 설정을 최소화하여 장비 간 연결성을 극도로 단순화한다. (Underlay simplicity).
2. **오버레이 도입**: 가상화 플랫폼 상단에 **VXLAN 기반의 오버레이 SDN (VMware NSX 또는 OpenStack Neutron)**을 얹는다.
3. **분리 운영**: 보안 정책(Micro-segmentation)은 오버레이에서 소프트웨어로 관리하고, 대역폭 보장(Traffic Engineering)은 언더레이 SDN 컨트롤러가 담당하게 한다.
4. **결과**: 물리 배선 변경 없이 매일 수백 개의 가상 망을 생성/삭제할 수 있는 민첩성을 확보하면서도, 물리 망의 최대 전송 성능을 100% 활용한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **오버레이 오버헤드 무시**: 캡슐화를 하면 패킷 크기가 커진다(약 50Byte). 물리 망의 **MTU(최대 전송 단위)**를 9000(Jumbo Frame)으로 키워두지 않으면, 패킷이 쪼개지면서(Fragmentation) 성능이 반토막 난다. 오버레이 도입 시 언더레이의 하드웨어 설정 변경이 반드시 선행되어야 한다.

---

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **망 확장성**: VLAN 한계(4천 개)를 넘어 **1,600만 개 이상의 가상 망** 구성 가능 (VXLAN).
- **운영 효율**: 물리 네트워크 작업 시간 **90% 단축** (전부 소프트웨어로 처리).

### 미래 전망 및 진화 방향
오버레이와 언더레이의 경계는 점점 희미해지고 있다. **SmartNIC / DPU** 기술이 발전하면서, 서버 CPU가 하던 오버레이 연산을 랜카드가 하드웨어적으로 처리하게 되었다. 미래에는 5G 슬라이싱과 결합하여, 무선 구간(언더레이)의 물리적 자원 할당과 클라우드 구간(오버레이)의 논리적 격리가 하나의 오케스트레이터로 통합되는 **'풀 스택 가상화'** 시대로 진입할 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **언더레이가 뭔가요?**: 레고 블록을 조립할 때 밑에 깔아두는 '커다란 판'과 같아요. 튼튼한 판이 있어야 그 위에 성을 쌓을 수 있거든요.
2. **오버레이는요?**: 그 판 위에 내 마음대로 방을 나누고 색칠해서 만드는 '나만의 가상 도시'예요. 판은 그대로지만 도시 모양은 매일 바꿀 수 있죠.
3. **왜 나눠서 만드나요?**: 가짜 도시(오버레이)가 무너져도 밑에 깔린 진짜 판(언더레이)은 멀쩡하기 때문에, 언제든 금방 다시 도시를 세울 수 있어서 아주 편리하답니다!
