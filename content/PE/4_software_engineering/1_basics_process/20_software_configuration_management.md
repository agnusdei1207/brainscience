# 형상 관리 (SCM, Software Configuration Management)

## Ⅰ. 개요

**1. 형상 관리(SCM, Software Configuration Management)의 정의**
*   형상 관리란 소프트웨어 개발 생명주기(SDLC) 전 과정에서 생성되는 모든 산출물(소스코드, 명세서, 설계 문서, 데이터베이스 스키마, 테스트 케이스 등)의 변경 사항을 식별하고, 체계적으로 통제 및 관리하여 소프트웨어의 **무결성(Integrity), 추적성(Traceability), 가시성(Visibility)**을 보장하는 일련의 활동입니다.

**2. 등장 배경 및 필요성**
*   **잦은 요구사항 변경:** 소프트웨어 개발 중 발생하는 필연적인 요구사항 변경을 안전하게 수용하고 추적할 수 있는 기제가 필요합니다.
*   **동시 공학(Concurrent Engineering)의 충돌 방지:** 다수의 개발자가 동일한 코드를 동시에 수정할 때 발생하는 덮어쓰기 문제나 의존성 충돌을 방지해야 합니다.
*   **장애 복구 및 롤백(Roll-back):** 배포된 시스템에 치명적인 결함이 발견되었을 때, 안정성이 검증된 이전 상태(버전)로 신속하고 정확하게 복구하기 위한 기준점이 필요합니다.

## Ⅱ. 아키텍처 및 핵심 원리

### 1. 형상 관리의 4대 핵심 절차 아키텍처

형상 관리는 일반적으로 식별, 통제, 상태 보고, 감사의 4단계 프로세스로 이루어집니다.

```text
+--------------------------------------------------------------------------+
|                       형상 관리 프레임워크 (SCM Framework)               |
+--------------------------------------------------------------------------+
      |               |                  |                  |
      V               V                  V                  V
+------------+  +------------+     +------------+     +------------+
|  형상 식별 |->|  형상 통제 |---->| 상태 보고  |---->|  형상 감사 |
|  (Ident.)  |  |  (Control) |     | (Status)   |     |  (Audit)   |
+------------+  +------------+     +------------+     +------------+
      |               |                  |                  |
   [CI 선정]    [변경 요청(CR)]      [변경 이력]         [무결성 검증]
   [베이스라인]  [CCB 심의/승인]     [통계 리포팅]       [물리적/기능적 검사]
   [명명 규칙]  [체크인/체크아웃]    [추적성 매트릭스]   [베이스라인 확정]
      |               |                  |                  |
+--------------------------------------------------------------------------+
|                  형상 관리 저장소 (Repository / CMDB)                    |
+--------------------------------------------------------------------------+
```

### 2. 형상 관리의 핵심 요소 및 원리
*   **형상 항목 (CI, Configuration Item):** 형상 관리의 대상이 되는 소프트웨어 생명주기의 모든 최소 단위 산출물입니다.
*   **베이스라인 (Baseline, 기준선):** 특정 시점에 공식적으로 승인되어 확정된 형상 항목들의 집합입니다. 베이스라인이 설정된 이후의 변경은 반드시 공식적인 형상 통제 절차(CCB 승인 등)를 거쳐야 합니다. (예: 기능적 베이스라인, 분배 베이스라인, 제품 베이스라인)
*   **버전 제어 (Version Control):** 형상 항목의 진화 과정을 기록하여, 누가(Who), 언제(When), 무엇을(What), 왜(Why) 변경했는지 이력을 관리하고 관리 단위를 부여하는 메커니즘입니다.
*   **체크인(Check-in) / 체크아웃(Check-out):** 저장소로부터 최신 버전을 가져오고(Check-out), 수정된 버전을 저장소에 등록(Check-in)하는 동기화 메커니즘입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 집중형(CVCS) vs 분산형(DVCS) 형상 관리 시스템
*   **CVCS (Centralized VCS - 예: SVN, CVS):** 단일 중앙 서버에만 저장소가 존재합니다. 네트워크 연결이 필수적이며, 중앙 서버 장애 시 전체 개발 작업이 중단되는 단일 장애점(SPOF) 리스크가 있습니다.
*   **DVCS (Distributed VCS - 예: Git, Mercurial):** 개발자 각자의 로컬 PC에 전체 저장소의 복제본(Clone)을 보유합니다. 네트워크 없이도 오프라인 커밋과 브랜칭이 가능하며, 병합(Merge) 기능이 매우 강력하여 현대 개발 생태계의 표준으로 자리잡았습니다.

### 2. CI/CD(지속적 통합/배포) 관점에서의 SCM
현대의 SCM은 단순한 버전 저장소를 넘어 파이프라인의 **트리거(Trigger)** 역할을 수행합니다. Git 저장소의 특정 브랜치(예: `main` 또는 `release`)에 코드가 병합(Merge)되는 순간, Webhook을 통해 CI/CD 서버(Jenkins, GitHub Actions)가 작동하여 자동 빌드, 테스트, 배포가 이루어지는 데브옵스(DevOps)의 원천(Source of Truth)으로 기능합니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 실무 적용 시 고려사항 (브랜치 전략)
성공적인 형상 관리를 위해서는 조직 규모와 제품 특성에 맞는 명확한 브랜치 전략(Branch Strategy) 수립이 필수적입니다.
*   **Git Flow:** `master`, `develop`, `feature`, `release`, `hotfix` 등 명확한 브랜치 목적을 분리하는 전략. 정기적인 릴리스 주기를 갖는 제품에 적합합니다.
*   **GitHub Flow / Trunk-Based Development:** 단일 메인 브랜치에 짧은 주기로 지속적으로 병합하는 전략. 빠르고 민첩한 웹 서비스 배포와 지속적 배포(CD) 환경에 매우 적합합니다.

### 2. 기술사적 판단 (Expert Opinion)
최근 클라우드 네이티브 아키텍처 환경에서는 소스코드뿐만 아니라 인프라 구성 정보(IaC: Terraform 등), 컨테이너 설정(Dockerfile), 쿠버네티스 매니페스트(YAML)까지 소프트웨어 구동에 필요한 모든 환경 설정이 코드로 작성됩니다(**Everything as Code**). 따라서 기술사는 형상 관리의 범위를 전통적인 애플리케이션 소스코드에서 인프라 및 설정 파일로 확장하는 '통합 구성 관리 체계'를 구축하도록 프로젝트를 기획해야 하며, SCM의 보안 결함(예: 하드코딩된 비밀번호 커밋)을 방지하기 위한 보안 스캐닝(DevSecOps) 통제 방안을 설계에 반영해야 합니다.

## Ⅴ. 기대효과 및 결론

### 1. 기대효과
*   **프로젝트 가시성 및 통제력 강화:** 변경의 원인과 영향을 추적하여 관리 통제력을 확보하고, 이해관계자 간의 투명한 의사소통을 지원합니다.
*   **고품질 산출물 인도:** 무결성이 검증된 일관된 버전의 산출물을 배포함으로써, 운영 환경에서의 치명적인 오류나 구성 불일치로 인한 장애를 예방합니다.
*   **팀워크 및 생산성 극대화:** 충돌 없는 동시 개발 환경을 제공하고, 언제든지 이전 상태로 롤백할 수 있는 심리적 안전감을 부여하여 개발 속도를 향상시킵니다.

### 2. 결론
형상 관리(SCM)는 더 이상 소프트웨어 개발 완료 후 수행하는 부가적인 문서 작업이 아닙니다. SCM은 잦은 변경 속에서도 프로젝트가 표류하지 않도록 돕는 나침반이자, 애자일 개발과 CI/CD 파이프라인을 견고하게 떠받치는 소프트웨어 공학의 가장 중요하고 핵심적인 인프라입니다.
