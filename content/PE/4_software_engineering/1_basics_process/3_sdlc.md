# 소프트웨어 생명주기 (SDLC, Software Development Life Cycle)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어가 탄생부터 폐기까지 거치는 체계적인 단계별 프로세스로, 요구사항 분석부터 유지보수까지의 전 과정을 공학적으로 표준화한 프레임워크다.
> 2. **가치**: 무질서한 개발 활동을 예측 가능한 워크플로우로 변환하여 프로젝트 실패 위험을 최소화하고 품질, 일정, 비용의 3요소를 균형 있게 제어한다.
> 3. **융합**: 전통적 순차적 모델에서 현대의 반복적·점진적 모델(Agile, DevOps)로 진화하며 지속적 통합/배포(CI/CD)와 완전히 결합되었다.

---

## 1. 개요 및 정의 (Context & Background)

### 1.1 소프트웨어 생명주기의 정의

**소프트웨어 생명주기(SDLC, Software Development Life Cycle)**는 소프트웨어 제품이 계획 단계에서부터 최종 폐기(Retirement)까지 거치는 전체 수명 주기를 체계적으로 관리하기 위한 프레임워크다. 1960~70년대 소프트웨어 위기(Software Crisis) 당시, 예산 초과와 일정 지연, 품질 저하가 만연하자 이를 해결하기 위해 공학적 접근법으로 정립되었다.

```
    [ Software Development Life Cycle - 개념적 흐름 ]

    ┌─────────────────────────────────────────────────────────────────┐
    │                                                                 │
    │   비즈니스 요구 ──→ 소프트웨어 제품 ──→ 비즈니스 가치 창출      │
    │        ↓                 ↑                  ↓                  │
    │   ┌─────────────────────────────────────────────────┐          │
    │   │              SDLC Process Framework             │          │
    │   │  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌───┐│          │
    │   │  │요구 │→│설계 │→│구현 │→│테스트│→│배포 │→│유지││          │
    │   │  │분석 │ │     │ │     │ │     │ │     │ │보수││          │
    │   │  └─────┘ └─────┘ └─────┘ └─────┘ └─────┘ └───┘│          │
    │   └─────────────────────────────────────────────────┘          │
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘
```

### 1.2 SDLC와 소프트웨어 공학의 관계

SDLC는 **소프트웨어 공학(Software Engineering)**의 핵심 구성요소로, 다음과 같은 관계를 갖는다.

| 구분 | 소프트웨어 공학 | SDLC |
|:---:|:---|:---|
| **위상** | 전체 학문 분야 | 구체적 프로세스 프레임워크 |
| **목표** | 소프트웨어 개발의 체계화 | 개발 단계의 표준화 |
| **내용** | 이론, 방법론, 도구, 관리 | 단계별 활동, 산출물, 기법 |
| **관계** | SDLC를 포함한 상위 개념 | 소프트웨어 공학의 실천적 적용 |

### 2.2 단계별 상세 분석

#### ① 요구사항 분석 (Requirements Analysis)

```
    [ Requirements Analysis Process ]

    ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
    │  요구사항    │     │  요구사항    │     │  요구사항    │
    │    수집      │ ──→ │    분석      │ ──→ │    명세      │
    │              │     │              │     │              │
    │ - 이해관계자 │     │ - 타당성     │     │ - SRS 문서   │
    │   인터뷰     │     │   분석       │     │ - Use Case   │
    │ - 설문조사   │     │ - 우선순위   │     │ - User Story │
    │ - 문서 분석  │     │   선정       │     │ - 요구사항   │
    │ - 관찰       │     │ - 분류       │     │   추적 매트릭스│
    └──────────────┘     └──────────────┘     └──────────────┘
```

**요구사항 유형 분류**

| 유형 | 설명 | 예시 |
|:---:|:---|:---|
| **기능적 요구사항** | 시스템이 수행해야 할 기능 | "사용자는 로그인할 수 있어야 한다" |
| **비기능적 요구사항** | 품질 속성, 제약사항 | "응답 시간 3초 이내, 가용성 99.9%" |
| **사용자 요구사항** | 사용자 관점의 요구 | 직관적이고 쉬운 UI |
| **시스템 요구사항** | 기술적 관점의 요구 | 특정 DBMS, API 표준 준수 |

**요구사항 명세서(SRS) 구성요소**
- 기능적 요구사항 (Functional Requirements)
- 비기능적 요구사항 (Non-functional Requirements)
- 인터페이스 요구사항 (Interface Requirements)
- 데이터 요구사항 (Data Requirements)
- 제약사항 (Constraints)
- 품질 속성 (Quality Attributes)

**설계 원칙 (Design Principles)**

| 원칙 | 설명 | 효과 |
|:---:|:---|:---|
| **모듈화 (Modularity)** | 시스템을 독립적 모듈로 분할 | 개발 병렬성, 유지보수성 향상 |
| **응집도 (Cohesion)** | 모듈 내 요소 간 연관성 극대화 | 모듈 독립성 향상 |
| **결합도 (Coupling)** | 모듈 간 의존성 최소화 | 수정 전파 효과 감소 |
| **추상화 (Abstraction)** | 불필요한 세부사항 숨김 | 복잡성 관리 |
| **정보 은닉 (Information Hiding)** | 모듈 내부 구조 캡슐화 | 인터페이스 기반 통신 |

**구현 단계 핵심 활동**

- 지속적 통합(Continuous Integration) 구축
- 단위 테스트(Unit Testing) 작성
- 코드 리뷰(Code Review) 수행
- 정적 분석(Static Analysis) 도구 활용

**테스트 레벨 계층 구조**

| 레벨 | 대상 | 목적 | 수행자 |
|:---:|:---|:---|:---|
| **단위 테스트** | 개별 모듈/함수 | 내부 로직 검증 | 개발자 |
| **통합 테스트** | 모듈 간 인터페이스 | 모듈 간 상호작용 검증 | 개발자/테스터 |

| **인수 테스트** | 비즈니스 관점 | 사용자 요구 충족 | 사용자/고객 |

**배포 전략 비교**

| 전략 | 설명 | 장점 | 단점 |
|:---:|:---|:---|:---|
| **Big Bang** | 전체 시스템 일괄 교체 | 단순함 | 위험 높음 |
| **Phased** | 단계적 순차 배포 | 위험 분산 | 전환 기간 길다 |
| **Parallel** | 신/구 시스템 병행 운영 | 안전성 최고 | 비용 증가 |
| **Blue-Green** | 두 환경 전환 방식 | 즉시 롤백 가능 | 리소스 2배 |
| **Canary** | 일부 사용자 대상 선배포 | 위험 최소화 | 복잡한 라우팅 |

### 3.2 SDLC 모델 특성 비교 매트릭스

| 모델 | 구조 | 위험 관리 | 변경 수용성 | 문서화 | 적합 프로젝트 |

| **프로토타입** | 반복적 | 중간 | 높음 | 중간 | 요구사항 불명확 |
| **나선형** | 위험기반 | 높음 | 중간 | 높음 | 대규모, 고위험 |

| **Agile** | 반복적 | 중간 | 매우 높음 | 낮음 | 변화 빠른 비즈니스 |
| **DevOps** | 지속적 | 높음 | 매우 높음 | 자동화 | 클라우드 네이티브 |

### 4.2 조직 성숙도에 따른 SDLC 적용 전략

| CMMI 레벨 | 조직 특성 | 권장 SDLC 접근법 |
|:---:|:---|:---|
| **Initial** | 프로세스 혼재, 예측 불가 | 단순 폭포수로 기본 수립 |
| **Managed** | 프로젝트 단위 관리 | V-Model로 검증 체계 확립 |
| **Defined** | 조직 표준 프로세스 | 테일러링 가능한 표준 SDLC |
| **Quantitatively Managed** | 정량적 목표 관리 | 메트릭 기반 반복적 모델 |
| **Optimizing** | 지속적 프로세스 개선 | DevOps, CI/CD 파이프라인 |

## 5. SDLC 핵심 산출물 (Key Deliverables)

### 5.1 단계별 필수 산출물 매트릭스

| 단계 | 핵심 산출물 | 목적 | 검토 대상 |
|:---:|:---|:---|:---|
| **요구사항 분석** | SRS, Use Case Diagram, 요구사항 추적표 | 요구사항 공식화 | 이해관계자 전체 |
| **설계** | 아키텍처 문서, DB 스키마, API 명세, 클래스 다이어그램 | 구현 가이드 | 아키텍트, 개발자 |
| **구현** | 소스 코드, 단위 테스트 코드, 빌드 스크립트 | 실행 가능한 제품 | 개발팀 |
| **테스트** | 테스트 계획서, 테스트 케이스, 결함 리포트 | 품질 검증 | QA 팀 |
| **배포** | 배포 가이드, 운영 매뉴얼, 설치 스크립트 | 운영 이관 | 운영팀 |
| **유지보수** | 변경 요청서, 변경 이력, 버전 관리 기록 | 변경 관리 | 전체 |

    [ Requirements Traceability Matrix ]

    ┌─────────┬─────────┬─────────┬─────────┬─────────┬─────────┐

## 7. 기대효과 및 결론 (Future & Standard)

### 7.1 SDLC 도입의 정량적 효과

| 개선 영역 | Before SDLC | After SDLC | 개선효과 |
|:---|:---|:---|:---|
| **프로젝트 성공률** | ~30% | ~70% | 2.3배 향상 |
| **일정 준수율** | ~40% | ~75% | 1.9배 향상 |
| **요구사항 충족률** | ~50% | ~85% | 1.7배 향상 |
| **재작업 비용** | ~40% of Total | ~15% of Total | 62% 감소 |
| **결함 탐지 비용** | 운영 단계 100x | 요구/설계 1x | 90%+ 비용 절감 |

### 7.2 현대적 SDLC 진화 방향

### 7.3 결언

소프트웨어 생명주기(SDLC)는 단순한 개발 방법론이 아니라, 소프트웨어 제품의 전 수명 주기를 관리하는 **전사적 거버넌스 프레임워크**다. 기술사는 프로젝트의 특성, 조직의 성숙도, 비즈니스 환경을 종합적으로 고려하여 최적의 SDLC 모델을 선정하고, 지속적으로 개선해야 한다.

현대의 SDLC는 다음과 같이 진화하고 있다:
- **자동화**: CI/CD 파이프라인을 통한 빌드-테스트-배포 자동화
- **측정**: 메트릭 기반의 객관적 의사결정
- **협업**: 개발-운영-보안의 통합(DevSecOps)
- **AI 지원**: 요구사항 분석, 코드 생성, 테스트 자동화의 AI 활용
