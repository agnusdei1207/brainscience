# 폭포수 모델 (Waterfall Model)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 개발을 폭포가 계단식으로 떨어지듯 순차적으로 진행하는 전통적 모델로, 각 단계가 완료되어야 다음 단계로 넘어갈 수 있는 문서 중심의 엄격한 프로세스다.
> 2. **가치**: 요구사항이 명확하고 변경이 적은 프로젝트에서 높은 예측 가능성과 체계적 문서화를 통해 프로젝트 성공률을 극대화한다.
> 3. **융합**: 현대에는 순수 폭포수 대신 단계적 검증을 강화한 V-Model이나 반복적 요소를 결합한 하이브리드 모델로 진화하여 적용된다.

---

## 1. 개요 및 정의 (Context & Background)

### 1.1 폭포수 모델의 탄생과 의의

**폭포수 모델(Waterfall Model)**은 1970년 윈스턴 로이스(Winston W. Royce)가 발표한 논문 "Managing the Development of Large Software Systems"에서 최초로 체계화했다. 이름은 개발 단계가 마치 폭포수처럼 위에서 아래로 일방향으로 흐른다는 데서 유래했다.

```
    [ Waterfall Model - Classic Representation ]

    ┌─────────────────────────────────────────────────────────────────────┐
    │                                                                     │
    │       ┌──────────────────┐                                         │
    │       │   요구사항 분석   │  ← 사용자 요구 수집, 타당성 분석        │
    │       │ (Requirements)   │    SRS(Software Requirements Spec)      │
    │       └────────┬─────────┘                                         │
    │                │                                                    │
    │                │ "한 번 내려가면 다시 올라갈 수 없다"              │
    │                ▼                                                    │
    │       ┌──────────────────┐                                         │
    │       │      설계        │  ← 아키텍처, 상세 설계                   │
    │       │    (Design)      │    SDD(Software Design Document)        │
    │       └────────┬─────────┘                                         │
    │                │                                                    │
    │                ▼                                                    │
    │       ┌──────────────────┐                                         │
    │       │      구현        │  ← 코딩, 단위 테스트                     │
    │       │ (Implementation) │    Source Code, Unit Test Results       │
    │       └────────┬─────────┘                                         │
    │                │                                                    │
    │                ▼                                                    │
    │       ┌──────────────────┐                                         │
    │       │     테스트       │  ← 통합, 시스템, 인수 테스트             │
    │       │    (Testing)     │    Test Reports, Defect Logs            │
    │       └────────┬─────────┘                                         │
    │                │                                                    │
    │                ▼                                                    │
    │       ┌──────────────────┐                                         │
    │       │      배포        │  ← 설치, 사용자 교육, 운영 이관          │
    │       │   (Deployment)   │    User Manuals, Installation Guides    │
    │       └────────┬─────────┘                                         │
    │                │                                                    │
    │                ▼                                                    │
    │       ┌──────────────────┐                                         │
    │       │     유지보수     │  ← 수정, 적응, 완전화, 예방              │
    │       │  (Maintenance)   │    Change Requests, Version History     │
    │       └──────────────────┘                                         │
    │                                                                     │
    └─────────────────────────────────────────────────────────────────────┘
```

### 1.2 폭포수 모델의 핵심 원칙

| 원칙 | 설명 | 실무적 의미 |
|:---:|:---|:---|
| **순차성 (Sequentiality)** | 이전 단계 완료 후 다음 단계 진행 | 단계별 Sign-off 필수 |
| **단계별 완결성 (Phase Completion)** | 각 단계는 완전한 산출물 생성 | 문서 검토 및 승인 게이트 |
| **문서 중심 (Documentation Driven)** | 모든 활동과 결과를 문서화 | 추적 가능성, 인수인계 용이 |
| **위에서 아래로 (Top-Down)** | 상위 단계가 하위 단계를 구속 | 설계는 요구사항에 종속 |
| **피드백 제한 (Limited Feedback)** | 이전 단계로의 복귀는 예외적 | 변경관리(Change Control) 엄격 |

```
    │                     요구사항 분석 (Requirements)                │
    ├─────────────────────────────────────────────────────────────────┤

    │  │ 계약서  │ ──→ │ 인터뷰/워크샵   │ ──→ │ Requirements    │  │
    │  └─────────┘     └─────────────────┘     │ Specification)  │  │
    │                  ┌─────────────────┐     └─────────────────┘  │
    │  ┌─────────┐     │ 기존 시스템     │     ┌─────────────────┐  │
    │  │ 비즈니스│ ──→ │ 분석/요구사항   │ ──→ │ Use Case 명세서 │  │
    │  │ 목표    │     │ 분류/우선화     │     └─────────────────┘  │
    │  └─────────┘     └─────────────────┘                          │
    │                  ┌─────────────────┐     ┌─────────────────┐  │
    │                  │ 타당성 분석     │ ──→ │ 요구사항 추적표 │  │
    │                  │ (기술/경제)     │     │ (RTM)           │  │
    │                  └─────────────────┘     └─────────────────┘  │
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘
```

**요구사항 분석 단계 체크리스트**
- [ ] 모든 이해관계자 식별 및 인터뷰 완료
- [ ] 기능적/비기능적 요구사항 분류
- [ ] 요구사항 우선순위 합의 (MoSCoW 등)
- [ ] 타당성 분석 (기술적, 경제적, 운영적)
- [ ] SRS 문서 검토 및 Sign-off

```
    │                                                                 │

```

**설계 원칙 (Design Principles) 적용**

| 원칙 | 폭포수 모델에서의 적용 | 검증 방법 |
|:---:|:---|:---|
| **모듈화** | 기능별 모듈 분할, 인터페이스 정의 | 구조적 설계 검토 |
| **응집도** | 단일 기능 모듈 구성 | FTR(Formal Technical Review) |
| **결합도** | 모듈 간 인터페이스 최소화 | 영향도 분석 |
| **추상화** | 계층적 설계, 정보 은닉 | 아키텍처 검토 |
| **정보 은닉** | 캡슐화, 인터페이스 노출만 | 상세 설계 검토 |

## 3. 폭포수 모델 변형 (Variants)

### 3.1 고전적 폭포수 vs 변형 모델들

### 6.2 단계별 필수 체크리스트

|:---|:---|:---|

| **설계** | 아키텍처 검토, 인터페이스 정의, DB 설계 승인 | SDD, ERD, API 명세 |

| **테스트** | 테스트 계획 승인, 결함 수정 완료, 회귀 테스트 | 테스트 보고서, 결함 리포트 |
| **배포** | 설치 가이드 검증, 사용자 교육, 운영 이관 | 운영매뉴얼, 교육자료 |

|:---|:---|
| **요구사항 변경** | 변경관리 위원회(CCB) 운영, 범위 변경 시 재계약 |
| **통합 이슈 지연 발견** | 정기적 빌드, 지속적 통합(CI) 도구 활용 |
| **사용자 기대 불일치** | 중간 시연, 프로토타입 활용 |
| **성능 문제 후반 발견** | 설계 단계에서 성능 목표 정의, 초기 부하 테스트 |
| **일정 지연** | 버퍼(Contingency) 포함, Critical Path 관리 |

```
    │  └───────────────────────┘     └───────────────────────────┘  │
    │                                                                 │
    │  적용 분야:                     적용 분야:                     │
    │  - 안전 중요 시스템             - 엔터프라이즈 시스템          │
    │  - 규제 산업                    - 대규모 SI 프로젝트           │
    │  - 하드웨어 의존 시스템          - 공공/금융 분야               │
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘
```

### 7.2 핵심 요약

| 측면 | 핵심 포인트 |
|:---|:---|
| **정의** | 순차적, 문서 중심, 단계별 완결성을 강조하는 전통적 SDLC 모델 |
| **장점** | 예측 가능성, 체계적 문서화, 관리 용이성, 명확한 책임 소재 |
| **단점** | 변경 취약성, 사용자 참여 제한, 위험 노출 지연, 작동 SW 제공 지연 |
| **적합성** | 요구사항 명확, 변경 적음, 규제 산업, 외주 계약 |
| **진화** | V-Model, Water-Scrum-Fall 등 하이브리드 형태로 적용 |

### 7.3 기술사적 관점

기술사는 폭포수 모델을 단순히 "구식"으로 치부하지 말고, **언제, 어떤 상황에서** 폭포수 모델이 최적의 선택인지 판단할 수 있어야 한다. 특히:

1. **규제 산업 프로젝트**에서는 폭포수 모델의 문서 중심 특성이 필수적
2. **대규모 SI 사업**에서는 계약상 명확한 범위와 산출물 정의를 위해 폭포수가 선호
3. **안전 중요 시스템**에서는 V-Model 형태의 폭포수가 표준
4. **하이브리드 접근**으로 폭포수의 장점과 Agile의 장점을 결합하는 것이 최신 트렌드
