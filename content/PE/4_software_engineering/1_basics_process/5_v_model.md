# V-모델 (V-Model)

## 정의

**V-모델(V-Model)**은 폭포수 모델을 확장하여 **개발 단계와 테스트 단계를 대응**시킨 모델이다. V자 형태로 좌측은 개발(검증), 우측은 테스트(확인) 단계를 표현하며, 각 개발 단계에 대응하는 테스트 단계를 정의한다.

```
┌─────────────────────────────────────────────────────────────────┐
│           V-모델 개념도                                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│                      검증(Verification)                         │
│                         ↗                                    ↖  │
│               요구사항분석 ────────────────── 인수테스트         │
│                    ↗                          ↖                │
│              시스템설계 ──────────────────── 시스템테스트        │
│                  ↗                            ↖                │
│            상세설계 ─────────────────────── 통합테스트          │
│                ↗                              ↖                │
│          단위설계 ───────────────────────── 단위테스트          │
│              ↘                              ↙                  │
│                     코딩 (Implementation)                       │
│                                                                 │
│                      확인(Validation)                           │
│                                                                 │
│   ←─────────────────────────────────────────────────────────▶  │
│                     개발                  테스트                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 핵심 개념

### 1. 검증(Verification) vs 확인(Validation)

```
┌─────────────────────────────────────────────────────────────────┐
│           검증 vs 확인                                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   검증 (Verification):                                          │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │ 질문: "제품을 올바르게 만들고 있는가?"                   │ │
│   │                                                         │ │
│   │ • 요구사항 명세서가 정확한지 확인                        │ │
│   │ • "Are we building the product right?"                  │ │
│   │ • 개발 단계 (좌측)                                       │ │
│   │ • 리뷰, 검토, 정적 분석                                  │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│   확인 (Validation):                                            │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │ 질문: "올바른 제품을 만들고 있는가?"                     │ │
│   │                                                         │ │
│   │ • 사용자 요구를 충족하는지 확인                          │ │
│   │ • "Are we building the right product?"                  │ │
│   │ • 테스트 단계 (우측)                                      │ │
│   │ • 실제 실행, 동적 테스트                                  │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2. 단계별 대응 관계

```
┌─────────────────────────────────────────────────────────────────┐
│           V-모델 단계별 대응 관계                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   개발 단계                ↔    테스트 단계                     │
│   ──────────────────────────────────────────────────────────── │
│   요구사항 분석            ↔    인수 테스트 (Acceptance Test)   │
│   (Requirements Analysis)       • 사용자 요구 충족 여부 확인   │
│                                 • 실제 운영 환경에서 테스트     │
│                                                                 │
│   시스템 설계              ↔    시스템 테스트 (System Test)     │
│   (System Design)               • 전체 시스템 기능 테스트      │
│                                 • 비기능 요구 테스트            │
│                                                                 │
│   상세 설계                ↔    통합 테스트 (Integration Test)  │
│   (Detailed Design)             • 모듈 간 인터페이스 테스트    │
│                                 • 컴포넌트 통합 테스트          │
│                                                                 │
│   단위 설계                ↔    단위 테스트 (Unit Test)         │
│   (Module Design)               • 개별 모듈/함수 테스트        │
│                                 • 화이트박스 테스트             │
│                                                                 │
│   코딩 (Coding)                                                  │
│   • 소스 코드 작성                                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3. 테스트 계획 수립

```
┌─────────────────────────────────────────────────────────────────┐
│           V-모델에서의 테스트 계획                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   특징: 개발 단계에서 대응하는 테스트 계획을 미리 수립           │
│                                                                 │
│   요구사항 분석 단계:                                           │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │ • 인수 테스트 계획 수립                                  │ │
│   │ • 인수 기준(Acceptance Criteria) 정의                   │ │
│   │ • 사용자 시나리오 작성                                   │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│   시스템 설계 단계:                                             │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │ • 시스템 테스트 계획 수립                                │ │
│   │ • 시스템 테스트 케이스 작성                              │ │
│   │ • 성능, 보안 테스트 계획                                 │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│   상세 설계 단계:                                               │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │ • 통합 테스트 계획 수립                                  │ │
│   │ • 통합 순서 및 전략 정의                                 │ │
│   │ • 인터페이스 테스트 케이스                               │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│   단위 설계 단계:                                               │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │ • 단위 테스트 계획 수립                                  │ │
│   │ • 단위 테스트 케이스 작성                                │ │
│   │ • Mock/Stub 정의                                         │ │
│   └─────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4. V-모델 vs 폭포수 모델

```
┌─────────────────────────────────────────────────────────────────┐
│           V-모델 vs 폭포수 모델 비교                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐ │
│   │  구분      │  폭포수 모델      │  V-모델                │ │
│   ├────────────┼──────────────────┼────────────────────────┤ │
│   │ 테스트     │  구현 후 수행      │  개발 단계와 병행 계획│ │
│   │ 시점       │                  │                        │ │
│   ├────────────┼──────────────────┼────────────────────────┤ │
│   │ 결함       │  늦게 발견        │  조기 발견 가능       │ │
│   │ 발견       │  (테스트 단계)    │                        │ │
│   ├────────────┼──────────────────┼────────────────────────┤ │
│   │ 문서화     │  단계별 문서      │  테스트 계획서 추가   │ │
│   ├────────────┼──────────────────┼────────────────────────┤ │
│   │ 비용       │  결함 수정 비용   │  초기 비용 높음       │ │
│   │            │  높음             │  전체 비용 절감       │ │
│   ├────────────┼──────────────────┼────────────────────────┤ │
│   │ 품질       │  후반부 검증      │  전 단계 품질 관리    │ │
│   │ 보장       │                  │                        │ │
│   └────────────┴──────────────────┴────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 기술사 시험 포인트

### 1. 핵심 정의

> **V-모델**: 개발 단계와 테스트 단계를 대응시킨 모델
> **검증(Verification)**: "제품을 올바르게 만들고 있는가?"
> **확인(Validation)**: "올바른 제품을 만들고 있는가?"

### 2. 빈출 키워드

- **좌측(개발)**: 요구사항분석 → 시스템설계 → 상세설계 → 단위설계 → 코딩
- **우측(테스트)**: 인수테스트 ← 시스템테스트 ← 통합테스트 ← 단위테스트
- **대응 관계**: 요구사항↔인수, 시스템설계↔시스템테스트
- **조기 결함 발견**: 테스트 계획을 개발 단계에서 수립

### 3. 예상 문제

**Q1. V-모델에서 검증(Verification)과 확인(Validation)의 차이를 설명하시오.**

**Q2. V-모델의 개발 단계와 테스트 단계의 대응 관계를 기술하시오.**

**Q3. V-모델이 폭포수 모델보다 품질 관리에 유리한 이유를 설명하시오.**
