# 625. 테스트 더블 Mock과 Stub의 차이 - 가짜 객체들의 섬세한 역할 분담

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 단위 테스트 시 의존성 있는 실제 객체를 대신하여 사용하는 모든 가짜 객체를 **'테스트 더블(Test Double)'**이라 하며, 그중 **Stub(스터브)**은 미리 준비된 답변을 던지는 용도이고 **Mock(모크)**은 특정 호출이 일어났는지 검증하는 용도다.
> 2. **혁신**: Stub은 상태(State) 검증에 집중하여 "내가 원하는 값을 받았는가"를 체크하고, Mock은 행위(Behavior) 검증에 집중하여 "가짜 객체의 메서드가 몇 번 호출되었는가"를 체크한다.
> 3. **가치**: 외부 API, 데이터베이스 등 제어하기 힘든 환경을 격리하여 테스트의 속도와 신뢰성을 높이며, 객체 간의 협력(Interaction)이 올바르게 설계되었는지 검증하는 도구가 된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
테스트 더블(Test Double)은 영화 촬영의 '스턴트 더블(대역 배우)'에서 따온 용어다. 실제 배우(실제 DB/API)가 연기하기 위험하거나 비싼 상황에서 대역을 쓰는 것과 같다. 이 중 Stub과 Mock은 가장 많이 쓰이면서도 가장 많이 혼동되는 개념이다.

### 💡 비유
- **Stub (자동 응답기)**: 내가 누구든, 어떤 말을 하든 상관없이 "지금은 전화를 받을 수 없습니다"라고 정해진 멘트만 날린다. 나는 그 답변(상태)을 듣고 내 할 일을 한다.
- **Mock (비밀 첩보원)**: 내가 첩보원에게 "저 사람에게 암호를 전달해"라고 시켰다. 나중에 첩보원을 불러서 "너 진짜 저 사람한테 암호 전달했어? 몇 번 말했어? 정확히 어떤 암호를 줬어?"라고 **그의 행동(행위)**을 꼬치꼬치 캐묻는 것과 같다.

### 등장 배경: 'Isolation'의 필요성
로그인 기능을 테스트하는데, 실제 이메일 발송 서버를 연동하면 테스트할 때마다 가짜 메일이 수천 통 발송되고 테스트 속도도 느려진다. 이메일 서버는 믿고, 내가 짠 로그인 로직만 테스트하고 싶을 때 "이메일 서버 역할을 하는 가짜"를 세우는 기술이 발전하게 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 테스트 더블의 5가지 유형 (Gerard Meszaros 정의)

단순한 가짜부터 복잡한 순서로 나열하면 다음과 같다.

1. **Dummy (더미)**: 객체는 전달되지만 절대 사용되지 않는 깡통 객체. (주로 파라미터 채우기 용도).
2. **Fake (페이크)**: 실제 로직이 있지만 단순화된 형태. (예: 인메모리 H2 DB).
3. **Stub (스터브)**: 호출되면 미리 정의된 답을 주는 객체.
4. **Spy (스파이)**: Stub이면서 동시에 호출된 기록을 몰래 저장해두는 객체.
5. **Mock (모크)**: 기대하는 행위(Expectation)를 미리 설정하고 그대로 실행되는지 검증하는 객체.

### Mock vs Stub의 결정적 차이: 검증 방식 (ASCII Diagram)

```ascii
[ Unit Test: OrderService.placeOrder() ]

1. Stub 방식 (상태 검증 - State Verification)
┌──────────────┐      ┌──────────────┐      ┌────────────────┐
│ OrderService │ ───> │ StubPayment  │ ───> │ "Always True"  │ (답변)
└──────────────┘      └──────────────┘      └────────────────┘
       │                                             │
       └─────────── (검증: 주문 상태가 'Success'로 변했나?) ────────┘

2. Mock 방식 (행위 검증 - Behavior Verification)
┌──────────────┐      ┌──────────────┐      ┌────────────────┐
│ OrderService │ ───> │ MockPayment  │ ───> │ verify(call)   │ (감시)
└──────────────┘      └──────────────┘      └────────────────┘
                                                     │
       ┌─────────── (검증: 결제 메서드가 딱 1번 호출되었나?) ────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### Stub vs Mock 심층 비교표

| 비교 항목 | **Stub (스터브)** | **Mock (모크)** |
|:---:|:---|:---|
| **검증 대상** | 결과(상태) 값 | **객체의 행위 (상호작용)** |
| **작동 원리** | 미리 정해진 값을 리턴함 | 기대치를 설정하고 나중에 확인 |
| **코드 예시** | `when(obj.get()).thenReturn(10)` | `verify(obj, times(1)).save()` |
| **테스트 시점**| 사후 검증 (Post-check) | **사전 기대 (Expectation)** |
| **복잡도** | 단순함 | 복잡함 (상호작용 설계 필요) |
| **철학** | "결과만 맞으면 장땡" | **"제대로 협력했는지가 중요"** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 테스트 아키텍처 설계 전략)

**상황**: 사용자가 회원 가입을 하면 1) DB에 저장하고, 2) 축하 메일을 보내고, 3) 마케팅 팀에 알람을 쏘는 로직을 테스트해야 한다.
**판단 및 테스트 더블 믹스 전략**:
1. **Repository (Stub 사용)**: DB 저장 여부는 결과적으로 `user.getId()`가 생겼는지만 보면 된다. `userRepo.save()`가 호출되면 고정된 ID를 리턴하는 **Stub**을 쓴다. (상태 검증).
2. **MailSender (Mock 사용)**: 실제로 메일이 나가는지 눈으로 볼 수 없으므로, `mailSender.send()`가 **정확히 1번 호출되었는지**를 **Mock**으로 검증한다. (행위 검증).
3. **Notification (Spy 사용)**: 알람 내용에 어떤 문구가 담겼는지 나중에 확인하기 위해 호출 기록을 저장하는 **Spy**를 사용한다.
4. **결과**: 외부 인프라(DB, Mail Server) 없이도 로직의 완결성과 객체 간의 협력 구조를 1초 만에 완벽히 검증함.

### 안티패턴 및 고려사항
- **과도한 Mocking (Brittle Test)**: 모든 메서드 호출을 Mock으로 검증하면, 비즈니스 로직은 같은데 코드의 내부 구현 구조만 살짝 바뀌어도 테스트가 깨진다. (깨지기 쉬운 테스트). 가급적 **Stub 위주의 상태 검증**을 우선하고, 메일 발송처럼 결과 확인이 불가능한 '사이드 이펙트' 영역에만 Mock을 써야 한다.

## Ⅴ. 미래 전망 및 결론

### 결론: 도구가 아닌 '의도'의 차이
Mockito나 JMockit 같은 최신 도구들은 `mock()`이라는 함수 하나로 Stub 기능까지 다 수행한다. 그래서 기술적으로는 구분이 모호해졌다. 하지만 엔지니어는 "내가 지금 이 객체의 **답변**이 필요한가(Stub), 아니면 이 객체의 **행동**을 감시하려는가(Mock)"라는 설계 의도를 명확히 구분해야 한다.

### 미래 전망
앞으로는 AI가 소스코드를 분석하여 자동으로 최적의 테스트 더블 배치를 제안해 줄 것이다. 또한, 마이크로서비스(MSA) 환경에서 서비스 간 통신을 테스트할 때, 실제 서버를 띄우지 않고 네트워크 레벨의 Stub을 자동으로 생성해 주는 **'Consumer-Driven Contract(CDC) Testing'** 기술이 테스트 더블의 개념을 인프라 영역까지 확장할 전망이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **Stub(스터브)은 뭔가요?**: 인형극을 할 때, 주인공이 "지금 몇 시야?"라고 물어보면 옆에 있는 나무 인형이 무조건 "3시야!"라고만 대답하게 미리 정해놓은 거예요.
2. **Mock(모크)은 뭔가요?**: 선생님이 학생에게 "가서 우유 받아와!"라고 시킨 뒤, 나중에 "너 진짜 우유 받으러 한 번 갔다 왔어?"라고 행동을 확인하는 거예요.
3. **왜 가짜를 쓰나요?**: 진짜 우유를 가져오려면 멀리 가야 해서 시간이 오래 걸리니까, 공부하는 연습(테스트)을 할 때는 가짜 종이 우유로 빨리빨리 해보는 거랍니다!
