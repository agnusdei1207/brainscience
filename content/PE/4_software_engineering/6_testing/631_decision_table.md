# 631. 결정 테이블 (Decision Table) - 복잡한 논리 미로를 푸는 명쾌한 지도

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템이 처리해야 할 여러 가지 **조건(Condition)과 그에 따른 결과(Action)를 표 형태로 나열하여, 가능한 모든 논리적 조합을 빠짐없이 검증하는 블랙박스 테스트 기법**이다.
> 2. **혁신**: 단순한 나열이 아닌, **'True/False'의 이진 논리 조합**을 통해 기획서의 모순(Contradiction)이나 누락(Omission)을 시각적으로 즉시 찾아낼 수 있는 정형화된 도구를 제공한다.
> 3. **가치**: 비즈니스 로직이 매우 복잡하여 말이나 글로 설명하기 힘든 금융 상품 승인, 할인 정책, 권한 제어 등의 설계 및 테스트에서 논리적 완결성을 보장하는 핵심 수단이다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
결정 테이블은 "만약 ~라면 ~한다"라는 수많은 규칙을 바둑판 같은 표(Table)로 정리한 것이다. 복잡하게 얽힌 조건들을 한눈에 파악하게 해주며, 테스트 케이스를 설계할 때 "혹시 이 경우는 빼먹지 않았나?"라는 불안감을 해소해 준다.

### 💡 비유
- **텍스트 기획서 (줄글 요리법)**: "설탕을 넣고, 만약 소금이 있으면 소금을 넣되, 소금이 없는데 후추가 있다면 후추를 넣으세요. 단, 설탕과 소금이 둘 다 있다면..." 읽다 보면 머리가 아프고 실수하기 딱 좋다.
- **결정 테이블 (요리 체크리스트)**: 
  - [설탕 O, 소금 O, 후추 X] -> 결과: A맛
  - [설탕 O, 소금 X, 후추 O] -> 결과: B맛
  - [설탕 X, 소금 X, 후추 X] -> 결과: 요리 실패
  - 표의 칸만 채우면 모든 경우의 수가 자동으로 정리된다. 빈칸이 있으면 "아, 이 경우는 기획이 안 됐구나!"라고 바로 알 수 있다.

### 등장 배경: 'Logic Trap'의 방지
개발자와 테스터는 흔히 '해피 패스(정상 경로)'만 생각한다. 하지만 조건이 3~4개만 겹쳐도 경우의 수는 8~16개로 늘어난다. 이 중 기획자가 미처 생각지 못한 '이상한 조합'에서 대형 버그가 터진다. 이를 방지하기 위해 수학적인 논리 조합 표가 소프트웨어 공학에 도입되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 결정 테이블의 4대 구성 요소 및 구조 (ASCII Diagram)

```ascii
┌─────────────────────┬─────────┬─────────┬─────────┬─────────┐
│      구분 (Stub)     │  Rule 1 │  Rule 2 │  Rule 3 │  Rule 4 │
├─────────────────────┼─────────┼─────────┼─────────┼─────────┤
│ [ Condition Stub ]  │         │         │         │         │
│  - 조건 1 (로그인?)  │    Y    │    Y    │    N    │    N    │
│  - 조건 2 (VIP?)     │    Y    │    N    │    Y    │    N    │
├─────────────────────┼─────────┼─────────┼─────────┼─────────┤
│ [ Action Stub ]     │         │         │         │         │
│  - 혜택: 50% 할인    │    X    │         │         │         │
│  - 혜택: 10% 할인    │         │    X    │         │         │
│  - 혜택: 일반 가격   │         │         │    X    │    X    │
└─────────────────────┴─────────┴─────────┴─────────┴─────────┘

1. Condition Stub: 테스트에 영향을 주는 입력 조건들의 목록
2. Condition Entry: 각 조건의 참(Y/T)/거짓(N/F) 여부
3. Action Stub: 조건 조합에 따라 발생하는 기대 결과 목록
4. Action Entry: 특정 조건 조합에서 실행될 액션 표시(X)
```

### 테이블 작성 및 최적화 원리

#### ① 규칙 도출 (Rule Generation)
조건의 개수가 $n$개이면 가능한 규칙의 수는 $2^n$개다. (예: 조건 4개면 16개 규칙). 모든 조합을 표의 열(Column)로 만든다.

#### ② 모순 및 누락 발견
표를 채우다 보면 "이 조건 조합일 때는 결과가 정의되지 않았네?"(누락) 혹은 "이 조건일 때는 결과가 두 개가 나오네?"(모순)가 발견된다. 이를 코딩 전에 기획자와 협의하여 수정한다.

#### ③ 테이블 단순화 (Collapse)
특정 조건의 값과 상관없이 결과가 항상 같다면, 해당 조건을 '상관없음(-)'으로 표시하고 열을 합친다. 이를 통해 테스트 케이스의 개수를 줄일 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### 결정 테이블 vs 경계값 분석 vs 상태 전이 테스트

| 구분 | **결정 테이블 (Decision Table)** | 경계값 분석 (BVA) | 상태 전이 (State Transition) |
|:---:|:---|:---|:---|
| **핵심 관점** | **여러 조건의 '조합' (Logic)** | 입력값의 '범위' (Range) | 시간 흐름에 따른 '변화' (Flow) |
| **강점** | 복잡한 비즈니스 규칙 검증 | 데이터 타입 관련 버그 적발 | 이벤트 순서 및 워크플로우 검증 |
| **단점** | 조건이 많으면 표가 너무 커짐 | 조건 간 상호작용 확인 불가 | 논리 조합 확인이 어려움 |
| **비유** | **다중 스위치 회로 검사** | 온도계 눈금 검사 | 지하철 노선도 따라가기 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 대출 심사 로직 테스트 전략)

**상황**: 은행의 '비대면 신용 대출' 시스템을 테스트해야 한다. 조건은 1) 신용점수 700점 이상, 2) 연 소득 3천만 원 이상, 3) 타사 대출 5건 미만이다. 이 3가지가 모두 만족해야 '승인'이고, 하나라도 누락되면 '거절'인데 거절 사유가 각각 달라야 한다.
**판단 및 결정 테이블 설계 전략**:
1. **Stub 정의**: 조건 3개를 행으로 배치한다. (2^3 = 8개 규칙 생성).
2. **Action 정의**: [승인], [거절-점수부족], [거절-소득부족], [거절-대출과다]를 결과 행으로 둔다.
3. **규칙 단순화**: 만약 3번 조건(대출 5건 이상)에 걸리면 1, 2번 조건이 무엇이든 무조건 거절이라면, 해당 케이스들을 하나로 합쳐 테스트 효율을 높인다.
4. **결과**: 기획서에는 없었던 "신용점수도 낮고 소득도 낮은 멀티 거절 사유 발생 시 우선순위" 이슈를 발견하여 미리 수정함.

### 안티패턴 및 고려사항
- **조건의 폭발 (State Explosion)**: 조건이 10개면 규칙은 1,024개가 된다. 사람이 관리할 수 있는 수준을 넘는다. 이럴 때는 **결정 테이블을 여러 개로 쪼개거나(Hierarchical)**, 가장 중요한 조합만 뽑아내는 **페어와이즈(Pairwise)** 기법을 병행해야 한다.

## Ⅴ. 미래 전망 및 결론

### 결론: 논리적 무결성의 파수꾼
결정 테이블은 단순한 테스트 도구가 아니라 '설계 도구'다. 글자로 된 모호한 기획을 수학적 논리로 치환하는 과정에서 대부분의 설계 결함이 걸러진다.

### 미래 전망
앞으로는 **'AI 기반 결정 테이블 자동 생성'**이 일반화될 것이다. 자연어 처리(NLP) 기술이 기획서를 읽고 자동으로 조건과 결과를 추출하여 테이블을 그려주는 것이다. 또한, **'Model-Based Testing (MBT)'**과 결합하여, 결정 테이블의 로직을 기반으로 테스트 스크립트가 자동으로 코딩되어 실행되는 자동화 파이프라인이 구축될 전망이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **결정 테이블이 뭔가요?**: "만약 비가 오고, 바람이 불면, 장화를 신는다"처럼 여러 가지 상황에 따라 어떻게 행동할지 미리 적어둔 'OX 퀴즈 판'이에요.
2. **왜 만드나요?**: 비는 오는데 바람이 안 불 때, 혹은 눈이 올 때 어떻게 해야 할지 미리 정해두지 않으면 당황할 수 있기 때문이에요.
3. **뭐가 좋나요?**: 표를 하나씩 채우다 보면 우리가 깜빡 잊고 정하지 않은 상황까지 쏙쏙 찾아낼 수 있어서, 어떤 일이 생겨도 당황하지 않고 대처할 수 있답니다!
