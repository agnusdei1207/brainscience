# 632. 상태 전이 (State Transition) 다이어그램 - 시스템의 흐름을 설계하는 유한 오토마타

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 객체나 시스템이 가질 수 있는 **상태(State)와 그 상태들 간의 변화를 일으키는 이벤트(Event)를 시각적으로 표현**하여, 시간의 흐름에 따른 시스템의 동적 행위를 명세하는 블랙박스 테스트 기법이자 설계 도구다.
> 2. **혁신**: 단순히 화면을 나열하는 것을 넘어, **유효하지 않은 이벤트 발생 시의 예외 처리와 상태의 전이 경로(Route)**를 체계적으로 도출하여 비논리적인 흐름(Deadlock 등)을 사전에 방지한다.
> 3. **가치**: 임베디드 제어기, 통신 프로토콜, 게임 로직, 웹 서비스의 비즈니스 프로세스(주문-결제-배송) 등 상태 변화가 중요한 시스템의 복잡도를 효과적으로 관리하고 테스트 누락을 최소화한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
상태 전이 다이어그램은 시스템을 '하나의 상태에서 다른 상태로 옮겨가는 과정'으로 묘사한다. "A 상태일 때 B라는 버튼을 누르면 C 상태가 된다"는 규칙을 그림으로 그린 것이다. 테스팅 관점에서는 모든 상태를 방문했는지(Node Coverage)와 모든 전이 경로를 실행했는지(Edge Coverage)가 핵심이다.

### 💡 비유
- **지하철 노선도 (상태 전이)**: 각 역(상태)이 있고, 다음 역으로 가기 위해서는 열차 탑승(이벤트)이 필요하다. 강남역에서 2호선을 타면 역삼역으로 가지만, 반대 방향을 타면 교대역으로 간다.
- **테스팅 (노선 점검)**: 지하철이 안전한지 확인하려면 모든 역에 한 번씩 내려보고(Node), 모든 구간의 선로를 한 번씩 달려봐야(Edge) 한다. 만약 노선도에 없는 역으로 기차가 가버리거나, 다음 역으로 가는 길이 끊겨 있다면 그것이 바로 버그다.

### 등장 배경: 'Event-driven' 시스템의 복잡성
현대 소프트웨어는 대부분 사용자의 입력에 따라 상태가 변하는 이벤트 기반이다. 단순히 입력값만 검사해서는 "결제 도중에 뒤로 가기를 누르면 어떻게 되는가?" 같은 시나리오를 잡아낼 수 없다. 시스템의 '현재 상황(Context)'을 고려한 입체적인 검증을 위해 상태 전이 모델이 필수적이게 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 상태 전이 모델 구성 및 구조 (ASCII Diagram)

예시: 간단한 온/오프 스위치 시스템

```ascii
[ Idle (대기) ] ──( Start )──▶ [ Processing (처리 중) ]
       ▲                                │
       │           ( Error )            │
       └────────────────────────────────┘
       │                                │
       │           ( Finish )           │
       └────────────────────────────────┘

* 4대 구성 요소:
1. State (상태): 시스템이 머무르는 조건 (둥근 사각형)
2. Transition (전이): 상태 간의 이동 통로 (화살표)
3. Event (이벤트): 전이를 일으키는 외부 자극 (입력, 버튼 등)
4. Guard (가드/조건): 특정 조건이 만족될 때만 전이 허용 (e.g. [잔액 > 0])
```

### 상태 전이 테스트 기법의 핵심 지표 (Coverage)

#### ① 상태 커버리지 (State Coverage)
모든 가능한 상태를 최소 한 번 이상 방문하는 테스트다. (역마다 다 가봤나?)

#### ② 전이 커버리지 (Transition Coverage)
모든 전이(화살표)를 최소 한 번 이상 실행하는 테스트다. 상태 커버리지보다 강력하다.

#### ③ N-Switch 커버리지
- **0-Switch**: 화살표 하나씩만 확인 (A->B)
- **1-Switch**: 화살표 두 개를 이은 경로 확인 (A->B->C)
- **N-Switch**: N+1개의 전이 시퀀스를 확인. 복잡한 로직일수록 N의 숫자를 높여 깊게 테스트한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 결정 테이블 vs 상태 전이 다이어그램 비교

| 구분 | **결정 테이블** | **상태 전이 다이어그램** |
|:---:|:---|:---|
| **중심 관점** | 여러 조건의 **조합 (Logic)** | 시간과 이벤트에 따른 **변화 (Flow)** |
| **특징** | 현재 입력값이 가장 중요함 | **과거의 이력(상태)**이 현재 반응에 영향 |
| **강점** | 복잡한 할인/승인 정책 검증 | 통신 프로토콜, 워크플로우 검증 |
| **적합한 예** | 대출 승인 시스템, 보험료 계산 | 엘리베이터 제어, 회원 가입 프로세스 |
| **비유** | **다중 스위치 회로** | **미로 찾기** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 키오스크 주문 시스템 테스트 전략)

**상황**: 카페 키오스크 메뉴 주문 시스템을 테스트해야 한다. [메뉴 선택] -> [장바구니] -> [결제] -> [완료] 순서다.
**판단 및 상태 전이 설계 전략**:
1. **상태 정의**: [메뉴화면], [상세보기], [장바구니], [결제중], [주문완료] 5개 상태를 식별한다.
2. **이벤트 매핑**: '장바구니 담기', '수량 변경', '카드 삽입', '결제 취소' 등의 이벤트를 화살표로 잇는다.
3. **부정 전이(Negative Transition) 도출**: 가장 중요한 단계다. **[결제중] 상태에서 '메뉴 추가' 이벤트**가 발생했을 때, 시스템이 뻗지 않고 "결제 중에는 메뉴를 추가할 수 없습니다"라는 안내를 띄우는지(오류 처리) 확인하는 케이스를 추가한다.
4. **결과**: 정상 경로 외에 '결제 중 통신 장애 발생 시 [장바구니]로 복귀 실패'하는 치명적인 상태 비일관성 결함 3건을 발견하여 수정.

### 안티패턴 및 고려사항
- **모든 경로 테스트(All Paths)의 함정**: 전이가 촘촘한 시스템에서 모든 경로를 테스트하는 것은 불가능에 가깝다. 이때는 **리스크 기반(Risk-based)**으로, 결제나 보안 등 치명적인 전이 경로에 대해서만 N-Switch 값을 높여 집중 테스트해야 한다.

## Ⅴ. 미래 전망 및 결론

### 결론: 동적 시스템의 설계 나침반
상태 전이 다이어그램은 시스템의 '영혼'을 그리는 작업이다. 코드가 어떻게 짜여 있든, 시스템이 사용자의 부름에 어떻게 반응해야 하는지를 정의하는 가장 명확한 약속이다.

### 미래 전망
앞으로는 **'자동 상태 추론(State Inference)'** 기술이 쓰일 것이다. AI가 실제 구동되는 앱의 로그를 분석하여 거꾸로 상태 전이 다이어그램을 그려주는 것이다. (Reverse Modeling). 또한, **'디지털 트윈'** 환경에서 하드웨어 장비의 상태 전이를 실시간 모니터링하며, 가상 세계에서 수억 번의 상태 전이 테스트를 미리 수행하여 고장 확률을 예측하는 예방 정비 시스템의 핵심 기술로 자리 잡을 전망이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **상태 전이 다이어그램이 뭔가요?**: "신호등이 초록불일 때 버튼을 누르면 사람이 건너가고, 빨간불일 때는 기다린다"처럼 상황에 따라 어떻게 변하는지 보여주는 '행동 지도'예요.
2. **왜 그리나요?**: 지도가 없으면 로봇이 초록불인데도 가만히 있거나, 빨간불인데 갑자기 뛰어나가는 실수를 할 수 있기 때문이에요.
3. **뭐가 좋나요?**: 지도를 따라 하나씩 연습해 보면, 로봇이 어떤 상황에서도 당황하지 않고 안전하게 움직이는지 미리 확인할 수 있답니다!
