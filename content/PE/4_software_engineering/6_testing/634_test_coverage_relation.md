# 634. 구문, 분기, 조건 커버리지 포함 관계 - 코드의 빈틈을 메우는 테스트 강도의 위계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 화이트박스 테스트에서 소스 코드의 실행 정도를 측정하는 지표들로, **구문(Statement) < 분기(Decision) < 조건(Condition) < MC/DC** 순으로 테스트의 정밀도와 복잡도가 높아지는 상호 포함 및 보완 관계를 갖는다.
> 2. **혁신**: 단순히 "모든 줄을 읽었는가"를 넘어 **"모든 선택의 기로를 통과했는가"**와 **"복합 조건의 개별 원인을 검증했는가"**를 정량화하여, 논리적 결함이 숨어들 틈을 원천 차단한다.
> 3. **가치**: 미션 크리티컬한 시스템(항공, 자동차 등)에서 테스트의 충분성(Sufficiency)을 입증하는 객관적 근거가 되며, 낮은 수준의 커버리지가 주는 '거짓 신뢰'를 극복하게 해 준다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
코드 커버리지(Code Coverage)는 테스트 케이스가 소스 코드를 얼마나 훑었는지 재는 성적표다. 
- **구문 커버리지**: 코드의 모든 줄(Line) 실행.
- **분기 커버리지**: `if` 문의 True/False 모두 실행.
- **조건 커버리지**: `if(A and B)`에서 개별 조건 A, B의 True/False 모두 실행.

### 💡 비유
- **구문 커버리지 (동네 한 바퀴)**: 동네의 모든 골목길(코드 줄)을 한 번씩 밟아보는 것이다. 길만 밟으면 장땡이라, 갈림길에서 어디로 갈지 고민하지 않는다.
- **분기 커버리지 (갈림길 마스터)**: 모든 삼거리, 사거리(if문)에서 오른쪽으로도 가보고 왼쪽으로도 가보는 것이다. 갈림길의 모든 방향을 다 가봤으니 구문 커버리지는 자동으로 달성된다.
- **조건 커버리지 (신호등 검사)**: 갈림길에 신호등(복합 조건)이 2개 있다면, 각 신호등이 초록불일 때와 빨간불일 때를 모두 테스트하는 것이다. 신호등을 다 검사했다고 해서 반드시 갈림길의 모든 방향을 다 가보는 건 아니라는 점이 중요하다.

### 등장 배경: 'Coverage Blind Spot'
구문 커버리지만 100% 채웠다고 안심하다가, 특정 `else` 문에서 에러가 나거나 복합 조건문 내부의 논리 오류로 시스템이 터지는 사고가 잦았다. "어느 수준까지 테스트해야 안전하다고 말할 수 있는가?"에 대한 답을 찾기 위해 이들의 포함 관계와 위계가 정립되었다.

## Ⅱ. 커버리지별 상세 정의 및 포함 관계 (Deep Dive)

### 커버리지 위계 및 포함 관계 (ASCII Diagram)

```ascii
[ 고강도 테스트 ]
       ▲
       ┃  ┌──────────────────────────────────────────┐
       ┃  │ [ MC/DC (Modified Condition/Decision) ] │
       ┃  └────────────────────┬─────────────────────┘
       ┃           (강력한 포함) │
       ┃  ┌────────────────────▼─────────────────────┐
       ┃  │ [ Decision / Branch Coverage (분기) ]     │
       ┃  └────────────────────┬─────────────────────┘
       ┃           (기본 포함)   │
       ┃  ┌────────────────────▼─────────────────────┐
       ┃  │ [ Statement Coverage (구문) ]             │
       ┃  └──────────────────────────────────────────┘
[ 저강도 테스트 ]

* 주의: 조건 커버리지(Condition)는 분기 커버리지와 서로 
  완벽히 포함하지 않는 '보완적' 관계임. (둘을 합친 것이 복합 조건 커버리지)
```

### 3대 핵심 커버리지 상세 분석

#### ① 구문 커버리지 (Statement Coverage)
- **목표**: 소스 코드의 모든 문장(Statement)을 최소 한 번 실행.
- **한계**: `if` 문에서 조건이 참인 경우만 테스트해도 100%가 될 수 있어, 거짓일 때의 로직(`else`) 누락을 잡지 못함.

#### ② 분기 커버리지 (Decision/Branch Coverage)
- **목표**: 결정 포인트(Decision Point) 내의 전체 조건식이 True인 경우와 False인 경우를 모두 실행.
- **특징**: **분기 커버리지를 만족하면 구문 커버리지는 자동으로 만족함.**

#### ③ 조건 커버리지 (Condition Coverage)
- **목표**: 전체 조건식 내부의 각 개별 조건(Atomic Condition)이 True/False인 경우를 모두 실행.
- **주의**: 개별 조건을 다 만족해도 정작 전체 결과(분기)가 한 방향으로만 쏠릴 수 있음. 즉, **분기 커버리지를 보장하지 않음.**

## Ⅲ. 융합 비교 및 다각도 분석

### 커버리지 간의 관계 요약 (Table)

| 커버리지 | 모든 줄 실행? | 분기(T/F) 실행? | 개별 조건(T/F) 실행? | 강도 |
|:---:|:---:|:---:|:---:|:---:|
| **구문** | **Yes** | No | No | 낮음 |
| **분기** | **Yes** | **Yes** | No | 중간 |
| **조건** | 보통 | No | **Yes** | 중간 |
| **분기/조건**| **Yes** | **Yes** | **Yes** | 높음 |
| **MC/DC** | **Yes** | **Yes** | **Yes (독립적 영향)** | **최고** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 미션 크리티컬 코드 테스트 가이드)

**상황**: `if ( (A > 0) || (B < 0) ) { ... }` 라는 코드가 있다. 
**판단 및 테스트 설계 전략**:
1. **분기 커버리지 달성 (2개 케이스)**: 
   - [A=1, B=1] -> 결과 True
   - [A=0, B=1] -> 결과 False
   - (A와 B 개별 조건의 모든 경우를 보진 못함)
2. **조건 커버리지 달성 (2개 케이스)**:
   - [A=1, B=-1] -> A(T), B(T) / 결과 True
   - [A=0, B=1] -> A(F), B(F) / 결과 False
   - (이 케이스는 우연히 분기까지 만족했으나, 복잡한 식에서는 보장 안 됨)
3. **MC/DC 적용 (가장 권장)**: 개별 조건 A가 결과에 영향을 주는 케이스와, B가 결과에 영향을 주는 케이스를 각각 도출한다.
4. **결과**: 최소한의 케이스($n+1$개)로 분기와 조건을 모두 100% 만족하며 논리적 완결성 확보.

### 안티패턴 및 고려사항
- **구문 커버리지 100%의 맹신**: "우리 팀 커버리지 100%니까 버그 없어!"라고 말하는 것은 기술사적 소양이 부족함을 드러내는 것이다. 구문 커버리지는 **최소한의 예의**일 뿐이며, 비즈니스 로직의 안전성을 위해서는 반드시 **분기 커버리지 이상**을 기본 품질 게이트로 설정해야 한다.

## Ⅴ. 미래 전망 및 결론

### 결론: 숫자가 아닌 '논리'를 테스트하라
커버리지 지표는 테스트의 '양'을 재는 척도이지 '질'을 보장하지 않는다. 포함 관계를 정확히 이해하고, 시스템의 중요도에 따라 적절한 커버리지 목표를 설정하는 것이 비용 효율적인 품질 관리의 핵심이다.

### 미래 전망
앞으로는 **'Semantic Coverage'** 기술이 도입될 것이다. 단순히 실행 경로만 보는 게 아니라, AI가 코드의 의미(Context)를 분석하여 "이 로직은 금융 사고와 직결되니 MC/DC 수준으로 강제하라"고 제안하는 지능형 가이드다. 또한, **[회귀 테스트 커버리지 도구](./627_regression_coverage_tools.md)**와 결합하여, 코드 변경 시 영향을 받는 '최소 포함 관계'를 계산해 테스트 시간을 90% 단축하는 최적화 기술이 보편화될 전망이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **구문 커버리지가 뭔가요?**: 미로 찾기를 할 때, 미로 안의 모든 바닥을 한 번씩 다 밟아보는 거예요.
2. **분기 커버리지는요?**: 모든 갈림길에서 오른쪽으로도 가보고 왼쪽으로도 가보는 좀 더 꼼꼼한 방법이에요.
3. **왜 복잡하게 하나요?**: 바닥만 밟고 지나가면(구문), 막다른 길(else)에 괴물이 숨어있는지 모르고 지나칠 수 있기 때문에 모든 길을 다 가봐야 안전하답니다!
