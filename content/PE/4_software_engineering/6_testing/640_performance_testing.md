# 640. 성능 테스트 (Performance Testing) - 시스템의 한계와 체력을 측정하는 정밀 검진

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 특정 부하 상황에서 시스템의 응답 시간, 처리량(Throughput), 자원 활용률 등을 측정하여 **비기능 요구사항의 충족 여부를 확인하고 병목 지점(Bottleneck)을 찾아내는 테스트 활동**이다.
> 2. **혁신**: 단순히 "느리다"는 주관적 판단을 넘어, **부하(Load), 스트레스(Stress), 스파이크(Spike), 인듀어런스(Endurance)** 등 다각도의 시나리오를 통해 시스템의 파괴 시점과 회복 탄력성을 정량적으로 도출한다.
> 3. **가치**: 대규모 이벤트나 서비스 폭주 시 발생할 수 있는 시스템 다운 리스크를 사전에 방어하며, 최적의 하드웨어 사이징(Sizing)을 통해 인프라 투자 효율성을 극대화한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
성능 테스트는 소프트웨어의 '체력장'이다. 기능이 맞게 돌아가는지(기능 테스트)를 넘어, 얼마나 많은 사람이 동시에 들어와도 쌩쌩하게(성능 테스트) 돌아가는지를 검사한다. 임계치를 찾아내고, 문제가 생겼을 때 어디가 먼저 터지는지를 확인하는 과정이다.

### 💡 비유
- **기능 테스트 (요리 맛보기)**: 만든 음식이 맛있는지, 간이 맞는지 한 입 먹어보는 것이다.
- **성능 테스트 (뷔페 운영)**: 갑자기 손님 100명이 들이닥쳤을 때, 주방장이 음식을 제때 내놓을 수 있는지, 접시가 모자라지는 않는지, 알바생(CPU)이 지쳐 쓰러지지는 않는지 확인하는 것이다. 손님이 200명으로 늘어나면 식당이 난장판이 되는지(Stress), 아니면 "죄송합니다"라고 번호표를 주며 버티는지(Graceful Degradation)를 보는 과정이다.

### 등장 배경: 'The 3-Second Rule'
사용자는 웹사이트가 3초 안에 안 열리면 떠난다. 과거에는 서버 한 대의 성능이 중요했지만, 현대의 분산 아키텍처(MSA)에서는 수많은 서비스 간의 통신 지연이 전체 성능을 결정한다. "출시 후 터지면 늦는다"는 위기감이 개발 생명주기 전체에 걸친 성능 테스트를 의무화했다.

## Ⅱ. 성능 테스트의 유형 및 아키텍처 (Deep Dive)

### 성능 테스트 유형별 부하 프로파일 (ASCII Diagram)

```ascii
[ Load / Users ]
      ^
      ┃      Load Test (상시 부하)
      ┃   ━━━━━━━━━━━━━━━
      ┃
      ┃      Stress Test (임계치 돌파)
      ┃             ╱  
      ┃   ━━━━━━━━━┛
      ┃
      ┃      Spike Test (갑작스런 폭증)
      ┃      |┃|
      ┃   ━━━┛┗━━━━━━━━━━
      ┃
      ┃      Endurance Test (장기 체력)
      ┃   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━▶ [ Time ]
```

### 4대 핵심 성능 테스트 유형 상세

#### ① 부하 테스트 (Load Test)
- **목적**: 시스템의 설계 범위 내(정상 부하)에서 기대 성능(SLA)이 나오는지 확인.
- **내용**: 동시 접속자 수 증가에 따른 응답 시간과 처리량의 변화를 측정.

#### ② 스트레스 테스트 (Stress Test)
- **목적**: 시스템이 버틸 수 있는 **'임계점(Breakpoint)'**을 찾고, 붕괴 후 자가 회복이 되는지 확인.
- **내용**: 자원이 고갈될 때까지 부하를 계속 높여 비정상 종료를 유도.

#### ③ 스파이크 테스트 (Spike Test)
- **목적**: 수강 신청, 티켓팅처럼 갑자기 유저가 몰릴 때 시스템이 버티는지 확인.
- **내용**: 매우 짧은 시간에 부하를 계단식으로 급격히 높임.

#### ④ 인듀어런스 테스트 (Endurance / Soak Test)
- **목적**: 장시간 구동 시 발생하는 **메모리 누수(Memory Leak)**나 자원 단편화 확인.
- **내용**: 평균 부하를 24시간~1주일 이상 지속적으로 가함.

## Ⅲ. 융합 비교 및 다각도 분석

### 주요 성능 지표(Metrics) 및 목표 (Table)

| 지표 | 설명 | 목표 방향 | 비유 |
|:---:|:---|:---:|:---|
| **Response Time** | 요청부터 응답까지의 시간 | **Lower is better** | 대기 시간 |
| **Throughput** | 단위 시간당 처리 건수 (TPS) | **Higher is better** | 처리 속도 |
| **Resource Usage** | CPU, Memory, Disk I/O 점유율 | 70~80% 유지 권장 | 엔진 회전수 |
| **Error Rate** | 부하 발생 시 에러 패킷 비율 | **0% 근접 필수** | 불량률 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 신규 앱 런칭 전 성능 검증 전략)

**상황**: 한 달 뒤 신규 쇼핑 앱 런칭 이벤트가 예정되어 있다. 예상 최대 동시 접속자는 1만 명이다. 현재 서버 사양으로 감당 가능한지 의문이다.
**판단 및 성능 테스트 수행 전략**:
1. **목표 수립**: **SLA(Service Level Agreement)**를 정의한다. (예: 1만 명 접속 시 평균 응답 2초 이내, TPS 5,000 이상).
2. **Step-up Load**: 1,000명부터 시작해 2,000명씩 늘려가며 **부하 테스트**를 수행하여 성능 곡선(Saturation Point)을 찾는다.
3. **병목 구간 식별**: APM(Application Performance Monitoring) 툴을 연동하여, CPU는 노는데 DB에서 **Lock 경합**이 발생하는지, 혹은 네트워크 대역폭이 꽉 찼는지 확인한다.
4. **인프라 최적화**: 테스트 결과를 바탕으로 **[Scale-out](./622_scale_out_network.md)** 대수를 결정하고, 불필요한 DB 쿼리를 인덱싱하여 응답 속도를 개선한다.
5. **결과**: 이벤트 당일 시스템 다운 사고 없이 성공적인 런칭 완수.

### 안티패턴 및 고려사항
- **단순 툴(JMeter 등) 돌리기**: 툴 사용법보다 중요한 것은 **'사용자 시나리오'**다. 모두가 로그인만 하는 게 아니라, 검색-장바구니-결제 등 실제 사용 패턴 비율(Mix)을 반영하지 않은 테스트는 '가짜 안도감'만 줄 뿐이다.
- **테스트 환경과 운영 환경의 격차**: 테스트 장비는 사양이 낮은데 운영 장비는 높다고 해서 결과값을 단순히 곱하기로 계산하는 것은 위험하다. 가급적 **동일 사양의 클라우드 인스턴스**에서 테스트해야 한다.

## Ⅴ. 미래 전망 및 결론

### 결론: 품질의 시작은 기능이지만, 끝은 성능이다
기능이 안 되면 욕을 먹지만, 성능이 안 되면 회사가 망한다. 성능 테스트는 불확실한 미래의 트래픽이라는 공포를 데이터라는 확신으로 바꿔주는 인프라 엔지니어링의 정수다.

### 미래 전망
앞으로는 **'지속적 성능 테스트 (Continuous Performance)'**가 대세가 될 것이다. 배포가 일어날 때마다 CI/CD 파이프라인에서 자동으로 소규모 성능 테스트를 돌려, 성능 저하(Performance Regression)를 1분 만에 감지하는 것이다. 또한, AI가 성능 로그를 학습하여 "이대로 가면 30분 뒤에 메모리 부족으로 다운될 것"이라고 예견하는 **'지능형 장애 예측 시스템'**이 데이터센터의 표준이 될 전망이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **성능 테스트가 뭔가요?**: 놀이공원에 새로운 롤러코스터를 만들었을 때, "한꺼번에 100명이 타도 무너지지 않을까?" 하고 무거운 모래주머니를 싣고 쌩쌩 돌려보는 시험이에요.
2. **왜 하나요?**: 진짜 사람들이 많이 탔을 때 갑자기 멈추거나 망가지면 큰일 나기 때문에, 미리미리 튼튼한지 확인하는 거예요.
3. **좋은 점은?**: 얼마나 많은 친구가 타도 안전한지 정확히 알 수 있어서, 기다리는 친구들에게 "조금만 기다려, 안전하게 태워줄게!"라고 자신 있게 말할 수 있답니다!
