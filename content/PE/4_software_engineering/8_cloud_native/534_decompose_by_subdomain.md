# 534. 하위 도메인에 따른 분해 (Decompose by Subdomain - DDD 기반)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 도메인 주도 설계(DDD)의 핵심 개념인 바운디드 컨텍스트(Bounded Context)와 하위 도메인(Subdomain)을 기준으로 마이크로서비스의 물리적 경계를 설정하는 최고급 아키텍처 설계 패턴이다.
> 2. **가치**: 데이터베이스 테이블이나 부서 구조의 겉모습에 얽매이지 않고, 비즈니스 규칙과 용어(Ubiquitous Language)가 통일되는 맥락(Context)을 기준으로 서비스를 완벽하게 격리하여 진정한 '고응집 저결합'을 달성한다.
> 3. **융합**: 비즈니스 능력 기반 분해(Business Capability)를 보완하여, 시스템 내부에 깊숙이 숨겨진 복잡성을 풀어내고 데이터의 오염과 동기화 충돌을 방지하는 객체 지향 및 MSA 설계의 정수로 꼽힌다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
'하위 도메인에 따른 분해(Decompose by Subdomain)'는 시스템이 해결하려는 전체 비즈니스 문제(Domain)를 더 작고 응집력 있는 여러 개의 하위 도메인(Subdomain)으로 쪼개고, 각 하위 도메인별로 1개 이상의 마이크로서비스를 매핑하는 설계 기법이다. 이 패턴은 에릭 에반스(Eric Evans)가 창시한 **도메인 주도 설계(DDD, Domain-Driven Design)** 철학을 MSA 경계 식별 도구로 차용한 것이다.

### 💡 비유
이 방식은 **"동음이의어(同音異義語)가 섞이지 않도록 전문 사전(Context)을 따로 만드는 것"**과 같다.
병원에서 '차트(Chart)'라는 단어는 원무과에서는 '돈을 받아야 할 내역서'를 의미하고, 진료실에서는 '환자의 병 기록'을 의미한다. 이를 하나의 거대한 시스템(모놀리스)에 억지로 섞어 '통합 차트 DB'를 만들면 수많은 예외 처리로 코드가 엉망이 된다. 서브도메인 분해는 원무과용 '수납 차트 서비스'와 진료실용 '의료 차트 서비스'를 완전히 별개의 시스템(사전)으로 분리해 버리는 것이다.

### 등장 배경 및 발전 과정

#### 1. 공통 데이터 모델의 함정 (The Trap of Enterprise Data Model)
2000년대 엔터프라이즈 아키텍처는 전사적으로 통일된 단 하나의 거대한 ERD(Entity-Relationship Diagram)를 만드는 것을 이상향으로 여겼다. 예를 들어 `Customer` 테이블에 수십 개의 부서가 요구하는 속성 200개를 때려 넣었다. 이는 부서 간 결합도를 극도로 높여, 작은 컬럼 하나를 추가하려 해도 전사적 영향도 평가를 거쳐야 하는 병목을 낳았다.

#### 2. DDD와 MSA의 조우
마이크로서비스 아키텍처가 대두되면서, "도대체 서비스를 어디서 어떻게 잘라야 하는가?"라는 근본적 질문에 부딪혔다. 이에 대한 가장 학술적이고도 실증적인 해답으로 DDD의 '바운디드 컨텍스트'가 재조명받게 되었고, 현재 가장 강력한 분해 패턴으로 사용되고 있다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 서브도메인 분류와 전략 (표)

| 서브도메인 종류 | 특성 | 비즈니스 가치 | 대응 전략 (아키텍처 적용) | 예시 (이커머스) |
|-----------------|------|---------------|---------------------------|-----------------|
| **핵심 도메인 (Core Subdomain)** | 회사의 차별화된 경쟁력, 가장 복잡함 | 매우 높음 (돈을 벌어다 줌) | 최고급 개발자 투입, 내부에서 직접 자체 개발 (MSA 주축) | 추천 알고리즘, 상품 탐색 |
| **지원 도메인 (Supporting)** | 핵심은 아니지만 비즈니스 운영에 필수적 | 중간 | 핵심 도메인을 돕는 용도, 아웃소싱 고려 가능 | 재고 관리, 배송 추적 |
| **일반 도메인 (Generic)** | 모든 회사가 다 하는 흔한 기능, 차별성 없음 | 낮음 | 기성품(Off-the-shelf) 구매, SaaS 도입 권장 (예: AWS Cognito) | 회원가입/인증, 결제 연동 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ DDD Bounded Context Mapping to Microservices ]
================================================================================

   [ 전체 도메인: 거대 E-Commerce 시스템 ]
  
  (1) 공유 모델의 문제점 (모놀리스)
      [ Product Table ]
      - 가격, 옵션 (Sales 부서용)
      - 무게, 부피 (Shipping 부서용)
      - 제조사, A/S (Inventory 부서용)
      * 한 테이블에 모든 컬럼이 섞여 Lock 경합 및 수정 장애 발생!

  (2) 서브도메인 분해 (DDD 기반 Bounded Context 분리)

   ┌───────── Bounded Context ─────────┐     ┌──────── Bounded Context ───────┐
   │ [ 카탈로그(Catalog) 서브도메인 ]  │     │ [ 배송(Shipping) 서브도메인 ]  │
   │                                   │     │                                │
   │  Aggregate Root: [ 상품(Product)] │     │  Aggregate Root: [ 화물(Item) ]│
   │  - id: P_101                      │     │  - id: I_101                   │
   │  - name: "최신 스마트폰"          │     │  - name: "스마트폰 박스"       │
   │  - price: $1,000                  │     │  - weight: 0.5kg               │
   │  - category: "Electronics"        │     │  - dimensions: 10x5x2 cm       │
   │                                   │     │                                │
   └───────────────────────────────────┘     └────────────────────────────────┘
          ▲ (카탈로그 마이크로서비스)                   ▲ (배송 마이크로서비스)
          │                                             │
          └─────────── (Anti-Corruption Layer) ─────────┘
                       이벤트 큐를 통한 비동기 번역
                       "상품 P_101이 주문되었습니다" -> 배송 서비스는 이를 "화물 I_101 배송 지시"로 번역
```

### 심층 동작 원리

#### ① 유비쿼터스 언어 (Ubiquitous Language) 정립
코드, DB 테이블 명, 현업 기획자의 기획서에 나오는 단어가 100% 일치해야 한다. 만약 기획자가 "배송품"이라고 부르는데 DB에는 "Product_Delivery"라고 되어 있다면 번역 비용이 발생한다. 이 언어가 통일되게 유지되는 울타리가 바로 바운디드 컨텍스트다.

#### ② 모델의 파편화 (분할)
모든 부서가 쓸 수 있는 슈퍼 모델(God Object)을 만들지 않는다. 앞선 그림처럼 '스마트폰'이라는 현실의 객체를, 카탈로그 서비스는 '가격과 설명'을 가진 객체로 모델링하고, 배송 서비스는 오직 '무게와 크기'만 가진 객체로 분리하여 각자 다른 DB에 저장한다.

#### ③ 컨텍스트 맵 (Context Map)과 ACL
완전히 분리된 두 서비스가 통신해야 할 때는 **부패 방지 계층(ACL, Anti-Corruption Layer)**을 둔다. 결제 서비스의 모델이 변경되더라도, 주문 서비스는 ACL을 통해 번역된 데이터만 받으므로 내부 모델이 타 서비스의 변화에 의해 "오염(Corruption)"되는 것을 막아낸다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비즈니스 역량 분해 vs 서브도메인 분해 심층 차이

| 기준 | 비즈니스 역량 (Business Capability) | 서브도메인 (DDD Subdomain) |
|------|-------------------------------------|----------------------------|
| **분석 주안점** | 조직이 하는 '일(Task)' 중심 | 데이터와 규칙이 존재하는 '문맥(Context)' 중심 |
| **산출물 형태** | 동사 형태의 기능 명세 (예: 결제를 처리한다) | 명사 형태의 도메인 모델 (예: 결제 영수증 Aggregate) |
| **경계의 단단함** | 상대적으로 느슨함 (조직 개편 시 바뀔 수 있음) | 매우 단단함 (비즈니스 본질이 바뀌지 않는 한 영구적) |
| **장점** | 비즈니스 팀과의 커뮤니케이션이 매우 직관적임 | 극강의 소프트웨어 구조적 안정성과 유지보수성 |

**융합 전략**: "기능(Capability)"을 먼저 식별하여 시스템의 큰 덩어리(도메인)를 자른 후, 그 덩어리 내부를 구현할 때 DDD의 "서브도메인과 바운디드 컨텍스트"를 적용하여 클래스와 DB 구조를 정밀하게 조각하는 것이 현대 아키텍처의 베스트 프랙티스이다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대형 배달 앱 시스템에서 주문량이 폭주할 때마다, '사용자 정보 DB'에 락(Lock)이 걸려 주문, 리뷰, 쿠폰 발급 등 모든 시스템이 연쇄적으로 다운되는 장애 발생.
**판단 및 전략**:
1. **문제의 본질 파악**: "User"라는 하나의 거대한 엔티티를 모든 부서가 참조(JOIN)하고 있는 **데이터 결합도(Data Coupling)** 문제임.
2. **DDD 서브도메인 분해 적용**:
   - **인증 도메인**: User를 "Account(ID/PW)"로 모델링하여 독립 서비스화.
   - **주문 도메인**: User를 "Orderer(결제수단, 주소)"로 모델링하여 자체 DB에 저장.
   - **리뷰 도메인**: User를 "Reviewer(닉네임, 평점)"로 모델링.
3. **이벤트 기반 동기화**: 회원이 주소를 변경하면 '인증 도메인'이 메시지 큐(Kafka)에 `UserAddressChanged` 이벤트를 발행하고, '주문 도메인'이 이를 구독하여 자신의 DB를 비동기로 업데이트하도록 아키텍처를 전면 수정.
4. **결과**: 리뷰 서비스 트래픽이 폭주해도 인증이나 주문 서비스 DB에는 전혀 영향을 주지 않는 완벽한 장애 격리 달성.

### 주의사항 및 안티패턴 (Anti-patterns)
- **도메인 전문가 부재 (Technical DDD)**: 비즈니스 전문가와의 깊은 대화 없이, 개발자들끼리 테이블 구조만 보고 억지로 컨텍스트를 나누면 아무 의미 없는 분해가 된다. DDD는 철저한 비즈니스-IT 얼라인먼트(Alignment) 활동이다.
- **분산 트랜잭션 남발**: 서브도메인을 잘못 쪼개어, 하나의 비즈니스 트랜잭션이 여러 컨텍스트를 무리하게 가로지르게 되면 2PC나 무거운 Saga 패턴을 강제당하게 되어 성능이 급락한다. '같이 트랜잭션으로 묶여야 하는 것'은 하나의 바운디드 컨텍스트(애그리거트) 안에 두어야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **코드 베이스 감소**: 각 서비스는 자신이 관심 있는 속성(예: 수백 개 컬럼 중 10개)만 코딩하면 되므로 개별 서비스의 코드 복잡도(Cyclomatic Complexity)가 극적으로 낮아짐.
- **배포 리스크 제로화**: 배송 서비스의 도메인 로직을 수정하고 배포해도, 결제 서비스는 물리적/논리적으로 완벽히 격리되어 있어 회귀 버그(Regression Bug) 발생 확률이 0에 수렴함.

### 미래 전망 및 진화 방향
DDD의 서브도메인 분해는 설계 과정이 워낙 추상적이고 어려워 러닝 커브가 높았다. 그러나 최근 **이벤트 스토밍(Event Storming)**과 같은 시각적 협업 워크샵 기법이 대중화되면서 포스트-잇을 붙이며 직관적으로 경계를 찾아내는 방법이 정착되었다. 앞으로도 MSA의 성공 여부를 가르는 "경계 긋기"의 가장 확고한 학술적/실무적 기준으로 확고히 자리매김할 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **서브도메인 분해란?**: '자동차'라는 물건을 생각할 때, 카레이서에게는 '속도'가 중요하고, 세차장 아저씨에게는 '더러운 정도'가 중요하죠. 각자의 입장에 맞게 자동차를 다르게 생각하고 따로 관리하는 거예요.
2. **왜 굳이 따로 관리해요?**: 세차장 아저씨가 세차 기록을 적을 때, 카레이서의 엔진 기록부까지 몽땅 다 뒤적거리고 같이 써야 한다면 너무 무겁고 복잡해지기 때문이에요.
3. **무엇이 좋아지나요?**: 세차장은 세차장만의 얇은 수첩을, 카레이서는 카레이서만의 수첩을 따로 가지게 되니까 일하기가 훨씬 빠르고, 다른 사람의 수첩을 실수로 망가뜨릴 위험도 없어집니다!
