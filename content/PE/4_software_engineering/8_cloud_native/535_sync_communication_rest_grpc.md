# 535. 서비스 간 동기 통신 (REST API, gRPC)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로서비스 아키텍처(MSA)에서 분리된 서비스들이 요청(Request)을 보내고 즉시 응답(Response)이 올 때까지 대기(Blocking)하는 통신 방식이며, REST API(HTTP/1.1+JSON)와 gRPC(HTTP/2+Protobuf)가 대표적이다.
> 2. **가치**: 직관적인 흐름과 쉬운 디버깅을 제공하여 구현이 매우 단순하지만, 호출된 서비스의 장애나 지연이 호출한 서비스로 전파되는 강한 시간적 결합(Temporal Coupling)을 유발한다.
> 3. **융합**: 동기 통신의 약점인 연쇄 장애를 극복하기 위해 서킷 브레이커(Circuit Breaker), 타임아웃(Timeout), 재시도(Retry) 패턴과 같은 클라우드 네이티브 회복성(Resiliency) 설계가 필수적으로 융합되어야 한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
마이크로서비스 아키텍처는 거대한 모놀리식 시스템을 여러 독립된 서비스로 쪼갠다. 과거에는 코드 내부에서 단순한 함수 호출(Function Call)로 끝나던 일들이 이제는 네트워크를 통과하는 프로세스 간 통신(IPC, Inter-Process Communication)으로 변모한다. 그중 **동기 통신(Synchronous Communication)**은 클라이언트가 서버에 요청을 보낸 후 응답이 올 때까지 스레드를 블로킹(대기) 상태로 멈춰두는 방식이다.

### 💡 비유
동기 통신은 **"전화 통화"**와 같다.
내가 식당(주문 서비스)에 전화를 걸어 "짜장면 하나요"라고 말하면, 사장님이 "네, 주문 접수되었습니다"라고 대답할 때까지 수화기를 들고 기다려야 한다. 사장님이 바빠서 전화를 안 받거나(장애), 메뉴를 확인하느라 1분을 기다리게 하면(지연), 나 역시 그 시간 동안 수화기를 든 채 아무 일도 할 수 없다(블로킹 및 장애 전파).

### 등장 배경 및 발전 과정

#### 1. RPC에서 REST로의 전환
초기 시스템 통신은 CORBA나 SOAP 같은 무겁고 벤더 종속적인 RPC(Remote Procedure Call)를 사용했다. 웹 시대가 도래하며 HTTP 표준 위에 사람도 읽기 쉬운 JSON 포맷을 사용하는 **REST(Representational State Transfer)** 원칙이 대세가 되었고, MSA 전환의 1등 공신이 되었다.

#### 2. gRPC의 등장 (성능의 한계 극복)
REST API는 범용적이지만, HTTP/1.1의 무거운 헤더와 텍스트 기반 JSON 파싱 오버헤드가 발생했다. 서비스 간 내부 통신(East-West Traffic)이 수천 번씩 일어나는 거대 MSA 환경에서는 지연 시간(Latency)이 심각해졌다. 이에 구글은 HTTP/2의 멀티플렉싱과 이진(Binary) 직렬화 포맷인 Protocol Buffers를 결합한 초고속 동기 통신 프레임워크인 **gRPC**를 오픈소스로 공개하며 차세대 내부 통신 표준으로 자리 잡게 했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### REST API vs gRPC 비교 (표)

| 비교 지표 | REST API (HTTP/1.1) | gRPC (HTTP/2) |
|-----------|---------------------|---------------|
| **프로토콜** | HTTP/1.1 (텍스트 기반) | HTTP/2 (바이너리 기반, 멀티플렉싱) |
| **데이터 포맷** | JSON, XML 등 (가변적) | Protocol Buffers (엄격한 스키마, 작고 빠름) |
| **계약(Contract)**| OpenAPI(Swagger) 등으로 간접 정의 | `.proto` 파일로 강제 컴파일 (IDL) |
| **통신 패턴** | 단방향 요청-응답 (Unary) | 단방향, 양방향 스트리밍(Streaming) 모두 지원 |
| **장점** | 브라우저 친화적, 디버깅 용이, 생태계 큼 | 월등한 성능(페이로드 압축), 클라이언트 코드 자동 생성 |
| **단점** | 느린 파싱 속도, 무거운 페이로드, 스트리밍 한계 | 사람이 읽을 수 없음(바이너리), 브라우저 직접 호출 제한적 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ 1. REST API (JSON over HTTP/1.1) 동작 흐름 ]
================================================================================
 [ Order Service ]                                  [ Payment Service ]
       │                                                    │
       │ HTTP POST /payments                                │
       │ Content-Type: application/json                     │
       │ { "orderId": 123, "amount": 50.0 }                 │
       │───────────────────────────────────────────────────>│ (JSON Parsing & Logic)
 (Wait/Block)                                               │
       │ <──────────────────────────────────────────────────│
       │ HTTP 200 OK                                        │
       │ { "status": "APPROVED", "txId": "ABC99" }          │
       ▼                                                    ▼
   * 텍스트 파싱에 CPU 소모. HTTP/1.1은 연결당 하나의 요청만 처리 가능 (Head-of-line blocking).

================================================================================
[ 2. gRPC (Protobuf over HTTP/2) 동작 흐름 ]
================================================================================
 [ Order Service (gRPC Stub) ]                      [ Payment Service (gRPC Server) ]
       │                                                    │
       │ 1. Method Call: ProcessPayment(req)                │
       │ 2. Protobuf 이진 직렬화 (크기 극소화)                 │
       │ 3. HTTP/2 Stream (Header 압축, 멀티플렉싱)             │
       │───────────────────────────────────────────────────>│ (Protobuf 이진 역직렬화)
 (Wait/Block)                                               │ (빠른 로직 처리)
       │ <──────────────────────────────────────────────────│
       │ 4. HTTP/2 Stream 반환                              │
       │ 5. Protobuf 결과 객체로 변환                         │
       ▼                                                    ▼
   * .proto 파일을 양쪽이 공유하여, 네트워크 통신 코드를 짤 필요 없이 
     마치 로컬 함수(Stub)를 부르듯이 개발. 성능이 REST 대비 5~10배 우수.
```

### 심층 동작 원리: Protocol Buffers와 Stub

gRPC의 핵심은 **IDL(Interface Definition Language)**인 `proto` 파일이다.
1. 개발자가 `.proto` 파일에 데이터 구조와 서비스 인터페이스를 정의한다.
   ```protobuf
   message PaymentRequest { int32 order_id = 1; double amount = 2; }
   message PaymentResponse { string status = 1; }
   service Payment { rpc Process (PaymentRequest) returns (PaymentResponse); }
   ```
2. 프로토콜 버퍼 컴파일러(`protoc`)가 이 파일을 읽어 Java, Go, Python 등 원하는 언어의 클라이언트용 `Stub` 코드와 서버용 뼈대(Skeleton) 코드를 자동 생성해 준다.
3. 데이터 전송 시 필드 이름("orderId", "amount")을 보내는 JSON과 달리, `1`, `2` 같은 번호(Tag)와 이진 값만 압축하여 보내므로 패킷 사이즈가 극도로 작아진다.

## Ⅲ. 융합 비교 및 다각도 분석

### 동기 통신 vs 비동기 통신(메시지 큐) 선택 매트릭스

| 설계 관점 | 동기 통신 (REST, gRPC) | 비동기 통신 (Kafka, RabbitMQ) |
|-----------|------------------------|-------------------------------|
| **응답성 (Response)**| 즉각적인 결과 확인이 필요한 경우 (예: 로그인 인증 성공 여부) | 결과가 나중에 와도 되는 경우 (예: 이메일 발송, 로그 수집) |
| **결합도 (Coupling)**| **강한 시간적 결합** (수신자가 살아있어야만 통신 가능) | **느슨한 결합** (수신자가 죽어있어도 큐에 쌓임) |
| **시스템 복원력** | 한 쪽 장애가 전체 장애로 전파(Cascading Failure)될 위험 높음 | 장애가 격리됨 (수신자가 복구되면 큐에서 꺼내 처리) |
| **복잡도** | 아키텍처가 단순하고 흐름 추적이 쉬움 | 브로커 인프라 구축, 최종적 일관성(Eventual Consistency) 관리로 복잡도 급증 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대규모 MSA 환경에서 클라이언트 앱이 API Gateway를 거쳐 들어오고, 백엔드 내부에서는 수십 개의 마이크로서비스가 서로 호출하며 로직을 처리해야 함. 현재 모든 내부 통신이 HTTP/1.1 REST API로 구현되어 있어 타임아웃과 스레드 풀 고갈 현상이 빈번함.
**판단 및 전략**:
1. **외부/내부 프로토콜 이원화**: 브라우저나 모바일 앱이 API Gateway와 통신하는 '외부 구간(North-South)'은 범용성과 클라이언트 친화력을 고려하여 **REST API (JSON)**를 유지한다.
2. **내부 통신 gRPC 전환**: API Gateway 뒤에서 마이크로서비스끼리 통신하는 '내부 구간(East-West)'은 지연 시간과 직렬화 오버헤드를 줄이기 위해 **gRPC**로 전면 전환한다.
3. **회복성 패턴 적용**: 동기 통신의 치명적 단점인 장애 전파를 막기 위해, 모든 gRPC/REST 호출 구간에 **서킷 브레이커(Circuit Breaker, 예: Resilience4j)**를 장착한다. 특정 서비스 응답이 지연되면 즉시 회로를 차단(Open)하여 스레드 낭비를 막고 Fallback(기본값)을 반환하게 설계한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **긴 동기 통신 체인 (Long Sync Chains)**: A -> B -> C -> D 로 이어지는 깊은 동기 호출 트리를 만들면 안 된다. D 서비스가 1초 느려지면, C, B, A가 모두 1초씩 스레드를 물고 대기하며 전체 시스템의 스레드 풀이 고갈된다. 최대 호출 깊이를 최소화하거나 비동기 이벤트 큐로 끊어주어야 한다.
- **REST로 CRUD만 강제하기**: 마이크로서비스는 데이터를 단순히 조작하는 것이 아니라 비즈니스 행위(Behavior)를 수행한다. URL을 억지로 리소스 명사(CRUD)로만 맞추려다 보면 도메인 로직을 표현하기 어려워진다. 무리한 REST 준수보다는 실용적인 RPC 스타일의 URL을 섞어 쓰는 것이 유리할 때가 많다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **gRPC 도입 시 성능 향상**: 페이로드 크기가 JSON 대비 최대 1/10 수준으로 압축되며, 직렬화/역직렬화에 소요되는 CPU 사이클이 대폭 감소하여 내부 통신 지연 시간이 **50~80% 감소**함.
- **코드 생산성**: Swagger 설정 없이 `.proto` 파일 하나로 양 진영의 통신 인터페이스가 강제 동기화되므로, API 스펙 불일치로 인한 런타임 버그 확률 0%.

### 미래 전망 및 진화 방향
클라우드 네이티브의 서비스 간 통신은 단순한 라이브러리 추가를 넘어 인프라 영역으로 편입되고 있다. 현재는 애플리케이션 개발자가 통신 실패, 재시도, mTLS 암호화 코드를 짜지 않고, 인프라 계층의 **서비스 메시(Service Mesh, 예: Istio)**가 Envoy 프록시를 통해 REST와 gRPC 트래픽을 가로채어 이 모든 복잡성을 대신 투명하게(Transparent) 처리해 주는 구조로 완벽히 진화하였다.

## 👶 어린이를 위한 3줄 비유 설명
1. **동기 통신이란?**: 식당에서 주문하고 음식이 나올 때까지 카운터 앞에서 꼼짝 않고 서서 기다리는 방식이에요 (REST API).
2. **gRPC는 뭐가 달라요?**: 예전엔 "짜-장-면-하-나-주-세-요"라고 길게 말했는데, gRPC는 "1번!" 하고 암호로 짧게 말해서 0.1초 만에 주문을 끝내는 마법의 무전기예요.
3. **단점은 뭔가요?**: 주방장 아저씨가 요리를 늦게 하면, 나는 다른 데 가지도 못하고 계속 멍하니 서서 기다려야만 하는 치명적인 단점이 있답니다.
