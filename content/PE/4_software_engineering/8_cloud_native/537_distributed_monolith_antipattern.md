# 537. 안티패턴: 분산 모놀리스 (Distributed Monolith)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 애플리케이션을 물리적으로 여러 개의 서비스로 쪼개어 배포(MSA)했음에도 불구하고, 서비스 간의 강한 결합도(Coupling) 때문에 마치 하나의 거대한 모놀리식(Monolithic) 시스템처럼 동작하는 구조적 결함을 의미한다.
> 2. **가치(위험성)**: 마이크로서비스의 장점(독립 배포, 민첩성, 장애 격리)은 전혀 얻지 못한 채, 네트워크 지연(Latency), 분산 트랜잭션의 복잡성, 인프라 비용 폭증 등 MSA의 단점만 고스란히 떠안게 되는 최악의 아키텍처 재앙이다.
> 3. **융합**: 부적절한 도메인 분해(Subdomain), 공유 데이터베이스(Shared Database), 무분별한 동기 통신(REST/gRPC) 체인 남발 등 설계 초기 단계의 오판이 융합되어 발생한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
분산 모놀리스(Distributed Monolith)는 시스템이 물리적으로는 분산되어 있으나 논리적으로는 강하게 엮여 있는 상태를 일컫는 안티패턴(Anti-pattern)이다. 한 서비스(A)를 수정할 때 연관된 다른 서비스(B, C)를 동시에 수정하고 함께 배포해야만 에러가 나지 않는다면, 그 시스템은 100% 분산 모놀리스다.

### 💡 비유
분산 모놀리스는 **"삼인사각(三人四脚) 달리기"**와 같다.
원래 한 몸(모놀리스)으로 달리던 사람을 더 빨리 달리게 하겠다며 몸을 3명(MSA)으로 나눴다. 그런데 이 3명의 발목을 밧줄(강한 결합)로 꽁꽁 묶어 놓은 것이다. 3명이 각자 다른 방향으로 뛰려다 보니 계속 넘어지고(장애 전파), 한 명이 쉬고 싶어도(독립 배포 불가) 나머지 두 명이 멈춰야만 한다. 혼자 뛸 때보다 훨씬 더 느리고 고통스럽다.

### 등장 배경 및 발전 과정

#### 1. 맹목적인 마이크로서비스 도입
2010년대 중반, 넷플릭스와 우버의 MSA 성공 사례가 알려지자 수많은 기업이 유행처럼 MSA 전환을 시작했다. 하지만 도메인 주도 설계(DDD)나 조직 문화(콘웨이 법칙)의 혁신 없이, 기존 모놀리식 코드를 단순히 클래스나 패키지 단위로 잘라 물리적 서버만 여러 개로 찢어놓았다.

#### 2. 네트워크라는 환상
로컬 함수 호출(Function Call)로 1밀리초 내에 처리되던 로직을 무턱대고 네트워크 통신(REST API)으로 바꾸었다. "네트워크는 항상 신뢰할 수 있고, 지연 시간은 0이다"라는 분산 컴퓨팅의 가장 큰 착각에 빠진 결과, 시스템은 연쇄 타임아웃과 락(Lock) 경합으로 붕괴하기 시작했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 분산 모놀리스를 유발하는 4대 원인 (표)

| 발생 원인 | 증상 및 현상 | 기저 메커니즘 | 해결 방향 | 비유 |
|-----------|--------------|---------------|-----------|------|
| **공유 데이터베이스 (Shared DB)** | A 서비스가 B 서비스의 DB 테이블을 직접 JOIN하여 읽고 씀 | 스키마를 변경하면 모든 서비스가 동시 다운됨 | Database per Service (DB 분리) | 공용 냉장고 |
| **강한 동기 통신 (Chatty API)** | 단순 작업을 위해 A->B->C->D로 동기식(REST) 연쇄 호출이 발생함 | C 서비스가 느려지면 A 서비스 스레드가 고갈되어 전체 마비 | 비동기 메시지 큐 (Kafka, RabbitMQ) 도입 | 릴레이 전화 통화 |
| **공유 라이브러리 (Shared Lib)** | 비즈니스 로직(할인 정책 등)을 공통 `.jar`로 묶어 모든 서비스에 배포함 | 라이브러리 버전 업데이트 시 전체 서비스 동시 재배포 강제 | 복제(Duplication) 허용, 독립 로직화 | 단일 뇌 공유 |
| **잘못된 경계 (Wrong Context)** | 트랜잭션이 하나로 묶여야 하는 개념을 여러 서비스로 쪼갬 | 2PC(Two-Phase Commit) 등 무거운 분산 트랜잭션 강제 | DDD 기반 바운디드 컨텍스트 재설계 | 생명유지장치 쪼개기 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ Anti-Pattern: Distributed Monolith vs Best Practice: MSA ]
================================================================================

[ 1. Distributed Monolith (재앙의 아키텍처) ]
  ┌──────────┐       (REST)      ┌──────────┐
  │ Order    │ ────────────────> │ Customer │   <-- 1. 강한 동기 결합 (응답 지연 시 같이 죽음)
  │ Service  │ <──────────────── │ Service  │
  └────┬─────┘                   └─────┬────┘
       │                               │
       │ (직접 쿼리)      (직접 쿼리)  │
       │                │              │
       └───────────┐    │    ┌─────────┘
                   ▼    ▼    ▼
          [(RDBMS) Shared Enterprise DB]        <-- 2. 공유 DB 결합 (테이블 컬럼 변경 시 두 서비스 동시 폭발)
         +------------------------------+
         | Orders Table | Users Table   |
         | (Foreign Key 연결)           |       <-- 3. 서비스 간 FK 제약조건으로 분리 불가
         +------------------------------+
  * 결과: 릴리즈 데이에 Order팀과 Customer팀이 무조건 밤을 새우며 '동시 배포'를 해야 함.

[ 2. True Microservices (진정한 독립성) ]
  ┌──────────┐     (Event/MQ)    ┌──────────┐
  │ Order    │ ────(Message)───> │ Customer │   <-- 1. 느슨한 비동기 결합 (Kafka 등 활용)
  │ Service  │                   │ Service  │
  └────┬─────┘                   └─────┬────┘
       │                               │
       ▼                               ▼
 [Order DB (NoSQL)]           [Customer DB (RDBMS)] <-- 2. 데이터 완벽 격리 (Database per Service)
 +----------------+           +-------------------+
 | Orders (JSON)  |           | Users Table       | <-- 3. Customer팀은 Order팀 눈치 안 보고
 +----------------+           +-------------------+      마음대로 DB 스키마 수정 가능.
  * 결과: Order팀은 월요일 아침에 단독으로 코드를 배포함. Customer Svc가 죽어있어도 Order는 정상 작동함.
```

### 심층 동작 원리 (장애의 파급)

#### ① 연쇄 타임아웃 (Cascading Failure)
분산 모놀리스에서는 한 사용자의 요청을 처리하기 위해 서비스들이 거미줄처럼 동기 통신을 한다.
`Client -> API Gateway -> Service A -> Service B -> Service C`
만약 C의 DB 쿼리가 3초 지연되면, B는 3초를 기다리고, A도 3초를 기다린다. 시스템에 100명이 접속하면 A, B, C의 톰캣(Tomcat) 스레드 풀 100개가 순식간에 꽉 차버리고(Thread Exhaustion), 시스템 전체가 응답 불능 상태(503 Service Unavailable)에 빠진다.

#### ② 동시 배포의 저주 (Lock-step Deployment)
마이크로서비스의 목적은 각 팀이 자신들의 사이클에 맞춰 하루에도 수십 번 배포하는 것이다. 그러나 A 서비스와 B 서비스가 하나의 DB 테이블을 공유하거나 API 스펙이 꽉 물려 있다면, A팀은 B팀 코드가 배포될 때까지 기다려야 한다. 결국 수십 개의 서비스가 1달에 1번 일요일 새벽에 시스템을 멈추고 **빅뱅(Big-bang) 배포**를 하게 된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 아키텍처 진화의 3단계 스펙트럼

| 지표 | 1. 순수 모놀리스 (Monolith) | 2. 분산 모놀리스 (Anti-pattern) | 3. 진정한 마이크로서비스 (MSA) |
|------|---------------------------|---------------------------------|--------------------------------|
| **장애 전파** | JVM 내의 오류로 전체 뻗음 | 네트워크 통신 지연으로 전체 뻗음 | 장애가 해당 도메인 내로 격리됨 |
| **배포 주기** | 주/월 단위 정기 배포 | 주/월 단위 **고통스러운 동시 배포** | 일/시간 단위 **독립적 개별 배포** |
| **데이터 결합**| 완벽한 단일 DB (정합성 최고) | 단일 DB 또는 거미줄 분산 트랜잭션 | 서비스별 독자 DB 및 최종적 일관성 |
| **통신 비용** | 0 (로컬 함수 호출) | 극히 높음 (무한 REST 호출 오버헤드)| 최적화됨 (비동기 이벤트 스트리밍) |
| **평가** | 안정적이고 생산성 높은 초기 모델 | **최악의 설계 (차라리 모놀리스로 돌아가라)** | 고도의 민첩성과 확장성을 갖춘 최종 모델 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 한 스타트업이 "요즘 대세는 MSA"라며 백엔드를 15개의 Spring Boot 서버로 쪼갰다. 그러나 쪼개고 나니 화면 하나를 로딩하는데 API 통신이 20번 발생하여 화면이 뜨는 데 5초가 걸리고, AWS 서버 비용은 3배로 뛰었으며, 개발자들은 디버깅을 위해 15개 서버의 로그를 뒤지느라 탈진 상태다.
**판단 및 턴어라운드(Turn-around) 전략**:
1. **모놀리스로의 회귀 (Monolith First)**: 현재 시스템은 도메인 경계가 완전히 잘못 짤린 "분산 모놀리스"이다. 무리한 분산을 멈추고 코드를 다시 하나의 모놀리스 시스템으로 합치는(Merge) 극약 처방을 내린다. (Martin Fowler도 권장하는 전략).
2. **모듈러 모놀리스 (Modular Monolith) 구축**: 물리적 분리 대신, 단일 애플리케이션 내부에서 패키지/모듈 수준으로 결합도를 끊어내고 인터페이스를 통해서만 소통하도록 코드를 리팩토링한다.
3. **재분해 (Re-decomposition)**: 코드의 결합도가 완벽히 풀리고, 각 모듈이 독립적인 DB를 가지게 되었을 때 비로소 트래픽이 많은 특정 모듈(예: 결제) 하나만 떼어내어 물리적인 마이크로서비스로 배포한다(Strangler Fig Pattern).

### 주의사항 및 안티패턴 (추가)
- **과도한 DRY(Don't Repeat Yourself) 강박**: 개발자들은 코드 중복을 혐오한다. 그래서 주문, 결제, 배송 도메인이 모두 쓰는 '할인율 계산' 로직을 하나의 공통 서비스(Common Service)로 분리해 버린다. 이는 완벽한 병목(SPOF) 지점이 된다. MSA에서는 "결합(Coupling)을 만드느니 차라리 코드 중복(Duplication)을 허용하라"는 것이 철칙이다.

## Ⅴ. 기대효과 및 결론

### 진정한 MSA 전환 시 기대효과
분산 모놀리스의 늪에서 빠져나와 올바른 바운디드 컨텍스트(Bounded Context)로 서비스를 재정립하면:
- **인지 부하(Cognitive Load) 감소**: 개발자는 다른 팀의 코드를 알 필요 없이 자기 도메인의 코드만 집중하면 된다.
- **장애 격리 (Fault Tolerance)**: 특정 컨테이너가 OOM으로 죽어도, 메시지 큐와 서킷 브레이커가 방파제 역할을 하여 핵심 비즈니스는 멈추지 않는다.

### 미래 전망 및 결론
클라우드 벤더들이 제공하는 도구(EKS, 람다, 서비스 메시)가 좋아질수록 분산 모놀리스의 유혹에 빠지기 더 쉬워졌다. 하지만 아키텍처의 실패는 툴이 아니라 '사람과 설계'에서 온다. 마이크로서비스는 인프라 기술이 아니라 **"조직의 통신 구조와 비즈니스 도메인을 모델링하는 기술"**이라는 철학을 지키지 않으면, 분산 모놀리스라는 거대한 기술 부채에 짓눌려 기업의 속도가 파멸적으로 느려지게 될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **분산 모놀리스가 뭔가요?**: 무거운 짐 수레를 혼자 끌기 힘들어서 친구 3명과 나눠 끌기로 했는데, 친구들끼리 다리를 밧줄로 꽁꽁 묶어놓고 달리라고 하는 멍청한 규칙이에요.
2. **왜 그렇게 됐나요?**: 그냥 사람 수(서버 수)만 늘리면 저절로 빨리 달릴 줄 알고, 각자 어떻게 협력해서 움직일지(도메인 설계)는 전혀 고민하지 않아서 그래요.
3. **어떻게 고치나요?**: 묶인 밧줄을 다 끊어버리고, 서로 손을 흔들거나 무전기(비동기 통신)로 신호만 주고받으며 각자 자유롭게 뛰게 만들어 줘야 한답니다!
