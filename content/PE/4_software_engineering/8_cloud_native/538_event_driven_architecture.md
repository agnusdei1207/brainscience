# 538. 이벤트 기반 아키텍처 (EDA, Event-Driven Architecture)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템의 상태 변화나 주요 비즈니스 사실을 '이벤트(Event)'라는 불변의 메시지 형태로 발생시키고, 이를 필요로 하는 다른 서비스들이 비동기적으로 반응하여 동작하게 만드는 분산 아키텍처 패러다임이다.
> 2. **가치**: 서비스 간의 결합도를 극도로 낮춰(Decoupling), 특정 서비스의 장애가 시스템 전체로 번지는 것을 막고 새로운 비즈니스 기능(소비자)을 기존 시스템 수정 없이 레고 블록처럼 손쉽게 추가할 수 있는 궁극의 확장성을 제공한다.
> 3. **융합**: 마이크로서비스 아키텍처(MSA), 메시지 브로커(Kafka, RabbitMQ), 서버리스(Serverless/FaaS) 환경과 융합되어 현대 클라우드 네이티브의 실시간 데이터 파이프라인과 비동기 트랜잭션(Saga)을 지탱한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
이벤트 기반 아키텍처(Event-Driven Architecture)는 애플리케이션의 흐름을 '요청과 응답(Request-Response)' 기반의 동기적 명령(Command)이 아니라, "과거에 이미 일어난 사실(Event)"을 전파하고 이에 수동적으로 반응(React)하는 방식으로 설계하는 기술이다. 이벤트는 한 번 발생하면 변경할 수 없으며(Immutable), 발행자(Producer)는 누가 이 이벤트를 소비(Consumer)하는지 전혀 알 필요가 없다.

### 💡 비유
EDA는 **"경매장의 방송 스피커"**와 같다.
(동기 방식): 손님이 물건을 사기 위해 회계팀, 배송팀, 포장팀에 일일이 전화를 걸어 지시를 내린다. 한 팀이라도 전화를 안 받으면 손님은 멈춰서 기다려야 한다.
(이벤트 방식): 경매사가 스피커로 "A상품, 10만 원에 낙찰되었습니다!(이벤트 발생)"라고 방송 한 번만 딱 한다. 그러면 방송을 들은 회계팀은 장부에 돈을 적고, 포장팀은 상자를 가져오고, 배송팀은 트럭을 준비한다. 경매사는 누가 무슨 일을 하는지 신경 쓰지 않고 바로 다음 물건을 경매한다.

### 등장 배경 및 발전 과정

#### 1. API(명령) 기반 아키텍처의 한계
MSA 전환 초기에는 서비스끼리 REST API로 통신했다. 주문 서비스는 결제 서비스에 "결제해(Command)"라고 명령했다. 하지만 배송, 적립금, 알림톡 등 새로운 기능이 추가될 때마다 주문 서비스의 코드를 수정하여 새로운 API를 호출해야 했고, 통신 지연(Latency)이 겹쳐 거대한 **분산 모놀리스**의 늪에 빠졌다.

#### 2. 패러다임 변화 (비동기 이벤트 스트리밍)
이를 극복하기 위해 엔터프라이즈 서비스 버스(ESB)나 메시지 큐에 기반한 비동기 통신이 도입되었다. 나아가 Apache Kafka 같은 고성능 이벤트 스트리밍 플랫폼이 대중화되면서, 이벤트를 단순한 알림(Notification)으로 쓰는 것을 넘어 이벤트 그 자체를 데이터의 진실(State)로 취급하는 수준(Event Sourcing)까지 철학이 진화했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (표)

| 요소명 | 상세 역할 | 핵심 원칙 | 관련 기술 | 비유 |
|--------|----------|-----------|-----------|------|
| **Event (이벤트)** | 발생한 의미 있는 상태 변화 | 불변성(Immutable), 주로 과거 시제(OrderCreated) 명명 | JSON 포맷 메시지 | 발생한 사건 뉴스 |
| **Event Producer (발행자)** | 이벤트를 감지하고 생성하여 브로커로 전송 | 소비자가 누군지 모름 (Fire and Forget) | MSA 서비스 (ex. 주문) | 뉴스 기자 |
| **Event Broker (브로커)** | 이벤트를 수신, 분류, 저장, 라우팅 | 무손실 저장 및 높은 처리량(Throughput) | Kafka, RabbitMQ, EventBridge | 신문사 배포망 |
| **Event Consumer (소비자)**| 브로커를 구독하여 이벤트에 반응해 로직 수행 | 자신이 필요한 이벤트만 필터링하여 자율적 처리 | MSA 서비스 (ex. 배송) | 뉴스 구독자 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ Traditional Command (REST API) vs Event-Driven Architecture ]
================================================================================

[ 1. Request/Response (Command) ] : 강한 결합 (Tight Coupling)
  ┌─────────┐   (1) POST /payments   ┌─────────┐
  │         │ ─────────────────────> │ Payment │ (응답 올 때까지 대기)
  │  Order  │   (2) POST /points     └─────────┘
  │ Service │ ─────────────────────> ┌─────────┐
  │         │   (3) POST /delivery   │  Point  │ (장애 나면 Order도 실패처리 됨)
  └─────────┘ ─────────────────────> └─────────┘
                                     ┌─────────┐
    * 새로운 '알림톡' 기능 추가 시    │ Delivery│
      Order 코드 수정 후 재배포 필수 └─────────┘

[ 2. Event-Driven Architecture (Choreography) ] : 느슨한 결합 (Loose Coupling)
  ┌─────────┐   (1) Publish Event
  │  Order  │    "Order_Created"     ┌──────────────────┐
  │ Service │ ─────────────────────> │   Event Broker   │ (Kafka / RabbitMQ)
  └─────────┘     (JSON Payload)     └──────┬──────┬────┘
                                            │      │
     * Order 서비스는 메시지를 던지고 즉시 끝.   │      │
       Consumer가 누구든 알 필요 없음.        │      │
                                     (Subscribe)  (Subscribe)
                                        ▼      ▼
                                ┌─────────┐  ┌─────────┐
                                │ Payment │  │  Point  │
                                │ Service │  │ Service │
                                └─────────┘  └─────────┘
                                "결제 진행!"   "포인트 적립!"
    * '알림톡' 팀은 그냥 브로커에 Subscribe만 새로 달면 됨 (Order 코드는 0% 수정).
```

### 심층 동작 원리 (이벤트 토폴로지 모델)

EDA는 이벤트를 사용하는 목적에 따라 3가지 주요 패턴으로 나뉜다.

#### ① 이벤트 알림 (Event Notification)
단순히 "주문이 생성되었다"는 사실과 ID만 보낸다. (예: `{"event": "OrderCreated", "id": 123}`). 소비자는 이 신호를 받고 자세한 주문 내역을 알기 위해 **다시 주문 서비스의 API를 동기식으로 조회(콜백)**해야 한다. 구현은 쉽지만 역방향 트래픽을 유발한다.

#### ② 이벤트 상태 전송 (Event-Carried State Transfer)
소비자가 다시 API를 호출하지 않도록, 이벤트 안에 **변경된 데이터의 모든 상세 정보**를 담아 보낸다. (예: `{"id": 123, "item": "Laptop", "price": 1000, "user": "Alice"}`). 소비자는 이 데이터를 받아 자신의 로컬 DB에 캐싱(CQRS 뷰 갱신 등)해 두고 타 서비스 의존성 없이 자체적으로 작업한다.

#### ③ 이벤트 소싱 (Event Sourcing)
시스템의 현재 상태(DB 테이블의 최종 값)를 저장하는 것이 아니라, 상태를 변화시킨 **모든 이벤트의 이력(로그)을 순서대로 저장**한다. 은행 계좌처럼 "입금 100, 출금 50, 입금 20" 이벤트를 모두 저장해 두고, 현재 잔액(70)은 이벤트를 처음부터 재생(Replay)하여 계산하는 가장 진보되고 복잡한 패턴이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 시스템 통합 패턴의 진화 비교

| 지표 | 모놀리스 (Local Call) | 마이크로서비스 (REST/gRPC) | 이벤트 기반 (EDA) |
|------|-----------------------|----------------------------|-------------------|
| **통신 방식** | 런타임 메모리 참조 | 동기적 네트워크 호출 | 비동기 브로커 중계 |
| **결합도(Coupling)**| 극도로 높음 | 시간/공간적 결합 존재 | **완벽한 분리 (느슨한 결합)** |
| **장애 격리성** | 없음 (하나 죽으면 다 죽음) | 서킷 브레이커 필요 | 매우 우수 (브로커가 버퍼링 수행) |
| **데이터 일관성** | 강력한 일관성 (ACID) | 강한 일관성 유지 노력 | **최종적 일관성 (Eventual Consistency)** |
| **디버깅 난이도** | 매우 쉬움 (Call Stack) | 중간 (분산 트레이싱) | **매우 어려움 (흐름 추적 복잡)** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대규모 배달 플랫폼에서 "라이더 위치 변경" 트래픽이 초당 10만 건 발생한다. 이를 기반으로 고객에게 위치를 보여주고, 배달 예상 시간을 AI로 재계산하며, 라이더에게 배달료를 정산해야 한다. 동기 API로 구현하니 AI 서버의 연산 병목으로 라이더 앱 전체가 멈추는 사태 발생.
**판단 및 전략 (EDA 전면 도입)**:
1. **Fire and Forget**: 라이더 앱은 위치 데이터를 1초마다 **Kafka 브로커의 `RiderLocationUpdated` 토픽**으로 무조건 쏘고(Publish) 통신을 끊는다. 라이더 앱은 즉시 다음 작동을 수행한다.
2. **독립적 소비 (Consumer Groups)**: 
   - **고객 앱 서버**: 이벤트를 구독하여 실시간 지도 마커만 이동시킨다 (빠름).
   - **AI 연산 서버**: 이벤트를 천천히 묶어서(Batch) 예상 시간을 재계산한다 (느림).
   - **정산 서버**: 이벤트를 하루 단위로 모아 이동 거리를 계산한다.
3. **효과**: 가장 연산이 무거운 AI 서버가 지연(Lag)을 겪더라도, 라이더가 위치를 전송하는 메인 플로우나 고객이 지도를 보는 데는 1ms의 지연도 주지 않는 완벽한 장애 격리가 달성된다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **이벤트 핀볼 (Event Pinball)**: 이벤트가 A->B->C->A로 계속 튕겨 다니며 무한 루프를 도는 안티패턴이다. 중앙 통제자(Orchestrator)가 없기 때문에 비즈니스 흐름을 한눈에 파악하기 어렵다. 이를 막기 위해 반드시 **상관관계 ID(Correlation ID)**를 이벤트 헤더에 넣어 분산 트레이싱(Jaeger, Zipkin)을 구축해야 한다.
- **최종적 일관성(Eventual Consistency)의 오해**: 사용자가 결제를 눌렀는데, 포인트 적립 이벤트 처리가 지연되어 5분 뒤에 적립될 수 있다. 기획자나 사용자에게 "데이터는 즉시 일치하지 않을 수 있지만, 언젠가는(Eventual) 일치한다"는 UX 설계와 비즈니스 타협을 이끌어내야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **확장성 극대화**: 새로운 서비스(예: 통계 수집 시스템) 추가 시 기존 시스템의 코드를 0줄 수정하고, 브로커 구독 설정 1줄만 추가하여 기능 배포 시간 90% 단축.
- **피크 트래픽 방어**: 수강신청, 연말 정산 등 트래픽이 100배 폭주할 때, 브로커(Kafka)가 수백만 건의 이벤트를 큐에 쌓아두는 댐 역할을 하여 메인 DB와 서버의 다운(Downtime)을 100% 방어.

### 미래 전망 및 진화 방향
클라우드 벤더들은 AWS EventBridge, Azure Event Grid 형태의 **서버리스 이벤트 버스**를 핵심 인프라로 밀고 있다. 나아가 S3 파일 업로드, DB 행 변경(CDC) 등 클라우드에서 발생하는 모든 물리적 변화가 자동으로 이벤트로 변환되어 Lambda 함수를 트리거하는 완전한 이벤트 중심 클라우드 네이티브 아키텍처로 진화하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **이벤트 기반 아키텍처란?**: 선생님이 학생들에게 일일이 찾아가서 "너는 청소해, 너는 칠판 지워"라고 지시하는 게 아니라, 방송으로 "자, 수업 끝났습니다!"라고 딱 한 번만 외치는 방식이에요.
2. **어떻게 일하나요?**: '수업 끝'이라는 방송(이벤트)을 들으면, 청소 당번은 스스로 빗자루를 들고, 주번은 칠판을 지우고, 나머지는 가방을 싸서 집에 갑니다.
3. **왜 좋나요?**: 선생님이 한 명 한 명 지시하느라 목이 쉴 필요가 없고, 전학 온 새 친구가 청소를 맡게 되어도 선생님은 똑같이 "수업 끝!" 방송 하나만 하면 되니까 관리가 엄청 편해요!
