# 541. 클라이언트 사이드 vs 서버 사이드 디스커버리 (Service Discovery Patterns)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로서비스 환경에서 동적으로 변하는 목적지 IP를 찾아내고 **트래픽을 로드밸런싱하는 주체가 누구인가**에 따라 나뉘는 두 가지 핵심 아키텍처 패턴이다.
> 2. **가치**: 클라이언트 사이드 방식은 병목 없는 높은 성능과 분산 제어를 제공하고, 서버 사이드 방식은 클라이언트 코드의 종속성을 제거하여 다국어(Polyglot) 환경 지원과 인프라 관리를 용이하게 한다.
> 3. **융합**: 과거 넷플릭스 OSS 시절에는 클라이언트 사이드가 대세였으나, 클라우드 네이티브(K8s)와 서비스 메시(Service Mesh) 시대가 도래하며 서버(인프라) 사이드 라우팅 방식으로 패러다임이 완전히 융합/진화하였다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
서비스 디스커버리 생태계는 '서비스 레지스트리(전화번호부)'를 두고 작동한다. 이때, **클라이언트 사이드 디스커버리(Client-side Discovery)**는 요청을 보내는 서비스가 직접 레지스트리를 조회하여 대상 IP 목록을 받고, 자신이 직접 로드밸런싱 알고리즘을 돌려 호출하는 방식이다.
반면, **서버 사이드 디스커버리(Server-side Discovery)**는 클라이언트가 중앙의 라우터(로드밸런서/API Gateway)에게 무작정 요청을 보내면, 그 중간 라우터가 레지스트리를 조회하고 목적지로 분배해 주는 방식이다.

### 💡 비유
- **클라이언트 사이드**: "전화번호부를 직접 뒤져서 전화 걸기". 내가 배달 앱(레지스트리)을 열어 치킨집 지점 5곳의 번호를 직접 확인한 뒤, 내 머릿속으로 계산해서(별점 높은 순, 로드밸런싱) A 지점에 전화를 건다. 
- **서버 사이드**: "114 안내원(또는 교환수)에게 맡기기". 내가 114(로드밸런서)에 전화를 걸어 "가장 한가한 치킨집 연결해 주세요"라고 하면, 114 교환수가 알아서 장부를 보고 가장 적합한 치킨집에 전화를 연결해 준다. 나는 전화번호부(레지스트리)를 볼 필요가 전혀 없다.

### 등장 배경 및 발전 과정

#### 1. 클라이언트 사이드의 유행 (Netflix OSS)
MSA 초기, 중앙 집중식 하드웨어 로드밸런서(L4 스위치 등)는 비용이 비싸고 병목 현상(SPOF)이 발생하기 쉬웠다. 넷플릭스는 중앙의 라우터를 없애버리고, 각 서비스(클라이언트) 내부에 Ribbon(로드밸런서)과 Eureka Client(디스커버리) 라이브러리를 내장시켜 클라이언트 스스로 최적의 경로를 찾게 만드는 분산 컴퓨팅 철학을 유행시켰다.

#### 2. 다국어(Polyglot) 문제와 인프라의 반격 (Server-side의 부활)
하지만 클라이언트 사이드 방식은 치명적인 단점이 있었다. Java로 만든 서비스는 Ribbon을 쓰면 되지만, Python이나 Node.js로 만든 서비스는 그 언어에 맞는 로드밸런싱 라이브러리를 또 개발하고 유지보수해야 했다(Polyglot 지원의 한계). 이에 쿠버네티스(K8s)가 등장하여, 인프라 계층(Kube-proxy, Ingress)에서 이 라우팅 책임을 통째로 흡수해 버리는 서버 사이드 방식이 다시 클라우드의 절대 표준이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 두 아키텍처의 비교 (표)

| 비교 지표 | 클라이언트 사이드 디스커버리 (Client-side) | 서버 사이드 디스커버리 (Server-side) |
|-----------|--------------------------------------------|--------------------------------------|
| **로드밸런싱 주체**| 클라이언트 애플리케이션 내부 (Library) | 중간 라우터 / 로드밸런서 (Infrastructure) |
| **레지스트리 접근**| 클라이언트가 직접 레지스트리(Eureka 등) 질의 | 중간 라우터가 레지스트리(또는 K8s API) 질의 |
| **의존성 (Coupling)**| 특정 언어 및 프레임워크(Spring Cloud 등)에 종속됨 | 언어/프레임워크 독립적 (HTTP 요청만 할 줄 알면 됨) |
| **네트워크 홉(Hop)**| 1 홉 (클라이언트 -> 목적지 직행) | 2 홉 (클라이언트 -> 로드밸런서 -> 목적지) |
| **대표 기술 세트**| Eureka + Netflix Ribbon / Spring Cloud LoadBalancer | AWS ALB, Nginx, Kubernetes Service (Kube-proxy) |

                                                                    └──────────────┘

### 심층 동작 원리

#### ① Client-side의 스마트한 클라이언트 (Smart Client)
Spring Cloud 환경에서 클라이언트는 애플리케이션 기동 시 Eureka 서버로부터 전체 서비스 목록을 다운로드하여 로컬 메모리에 캐싱(Caching)한다. 클라이언트 코드에서 `restTemplate.getForObject("http://PAYMENT-SERVICE/...")` 처럼 논리적 이름만 호출하면, 내장된 로드밸런서 모듈이 백그라운드 스레드를 돌며 라운드 로빈(Round Robin) 방식으로 실제 IP를 치환하여 요청을 쏜다. 중앙 병목이 전혀 없다는 것이 최대 강점이다.

#### ② Server-side의 통제력 (Control Plane)
AWS Elastic Load Balancer나 K8s의 Kube-proxy가 이 역할을 한다. 클라이언트는 단순히 `http://payment-service.default.svc.cluster.local` 이라는 고정된 DNS 주소로 패킷을 던진다. 그러면 인프라에 숨어있는 로드밸런서(서버 사이드)가 이 패킷을 가로채어, 자신과 동기화되어 있는 레지스트리(etcd) 정보를 바탕으로 살아있는 목적지 IP로 포워딩(DNAT)한다. 클라이언트는 뒷단에 파드가 1개인지 100개인지 전혀 알 필요가 없다.

## Ⅲ. 융합 비교 및 다각도 분석

### 트레이드오프(Trade-off) 심층 분석

| 관점 | Client-side 선택 시 | Server-side 선택 시 |
|------|---------------------|---------------------|
| **성능 (지연 시간)**| 중앙 프록시를 거치지 않으므로 네트워크 홉(Hop)이 줄어 매우 빠름 | 중간에 로드밸런서를 한 번 거쳐야 하므로 약간의 지연(Latency) 발생 |
| **코드 침투성** | 각 서비스 코드에 디스커버리/LB 라이브러리를 추가하고 얽매여야 함 | 애플리케이션 코드는 네트워크 구조를 전혀 몰라도 됨 (의존성 제로) |
| **운영 복잡도** | 다양한 언어(Python, Go, JS)를 쓸 경우, 언어별로 클라이언트 라이브러리를 유지보수해야 하는 끔찍한 고통 수반 | 인프라 엔지니어가 LB와 K8s만 관리하면 됨 (개발팀은 비즈니스 로직에만 집중) |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 레거시 Java 기반의 1세대 MSA 플랫폼을 운영 중이다. 모든 통신은 Spring Cloud Netflix (Eureka + Ribbon)를 통한 Client-side Discovery로 돌아가고 있다. 그런데 최근 AI 분석을 위해 Python 부서가 합류했고, 프론트엔드 연동을 위해 Node.js 스택이 추가되었다. 각 언어별로 Eureka Client와 Ribbon 알고리즘을 포팅(Porting)하려니 개발 공수가 배보다 배꼽이 더 커진 상황이다.
**판단 및 아키텍처 전환 전략**:
1. **플랫폼 이전 (K8s 도입)**: 애플리케이션 레벨(Client-side)에 묶여있던 라우팅 책임을 버리고, 인프라 레벨(Server-side)로 넘기기 위해 쿠버네티스를 전격 도입한다.
2. **코드 종속성 제거 (Dumb Pipe, Smart Endpoint)**: Java 애플리케이션에서 Eureka, Ribbon, Feign Client 라이브러리를 모조리 삭제한다. 대신 쿠버네티스의 `Service` 리소스(Kube-proxy 기반의 Server-side Discovery)를 활용하여 언어에 종속되지 않는 범용 HTTP 호출로 코드를 리팩토링한다.
3. **결과**: Python이든 Node.js든 K8s Service 이름만 알면 통신이 가능해져 완벽한 다국어(Polyglot) 마이크로서비스 생태계가 안착된다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **LB의 단일 장애점화 (SPOF)**: Server-side 방식에서 물리적인 하드웨어 L4/L7 스위치를 중간 로드밸런서로 사용할 경우, 이 스위치가 고장 나면 전체 MSA 통신이 마비된다. 따라서 현대 Server-side 디스커버리는 반드시 고가용성이 확보된 **K8s의 분산 Kube-proxy**나 1:1로 파드 옆에 붙는 **사이드카 프록시(Envoy)** 형태로 구현해야 한다.

## Ⅴ. 기대효과 및 결론

### 진화의 완성: 서비스 메시 (Service Mesh)
클라이언트 사이드의 "네트워크 홉이 없는 고성능"과 서버 사이드의 "코드 독립성"이라는 두 마리 토끼를 다 잡은 것이 바로 **사이드카 패턴 기반의 서비스 메시(Service Mesh, 예: Istio)**다. 
애플리케이션(컨테이너) 옆에 투명한 프록시(Envoy)를 바짝 붙여둔다. 애플리케이션은 마치 서버 사이드처럼 무작정 요청을 밖으로 던지지만, 바로 옆에 붙어있는 사이드카 프록시가 이를 낚아채어 클라이언트 사이드 디스커버리 룰을 적용해 목적지로 직배송한다.

### 기술사적 결론
초창기 넷플릭스가 개척한 클라이언트 사이드 디스커버리는 당시 인프라 기술의 한계를 극복하기 위한 소프트웨어적 발악이었다. 하지만 K8s라는 강력한 클라우드 네이티브 OS가 등장한 현재, 비즈니스 코드가 네트워크 라우팅 라이브러리를 직접 들고 다니는 아키텍처는 기술 부채로 간주된다. **"네트워크의 복잡성은 인프라(Server-side/Mesh)로 내리고, 애플리케이션은 비즈니스 로직만 수행하라"**는 것이 현대 MSA의 확고한 지향점이다.
