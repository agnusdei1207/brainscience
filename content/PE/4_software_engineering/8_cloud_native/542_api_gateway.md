# 542. API 게이트웨이 (API Gateway)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로서비스 아키텍처(MSA)에서 외부 클라이언트(Web, Mobile)와 수십 개의 내부 백엔드 서비스들 사이에 위치하여, 모든 요청을 단일 진입점(Single Point of Entry)으로 모아 처리하는 거대한 역방향 프록시(Reverse Proxy)다.
> 2. **가치**: 클라이언트가 내부의 복잡한 파편화된 서비스 IP/URL을 몰라도 되게 추상화해주며, 라우팅, 인증/인가(JWT), 부하 분산, 속도 제한(Rate Limiting), 로깅 등 공통 횡단 관심사(Cross-cutting Concerns)를 중앙에서 일괄 제어하여 중복 개발을 방지한다.
> 3. **융합**: 서비스 디스커버리(Eureka, Consul), 인증 인프라(OAuth2), BFF(Backend For Frontend) 패턴과 강하게 융합되어 클라우드 네이티브 네트워크의 North-South(외부-내부) 트래픽을 관장하는 핵심 파수꾼 역할을 수행한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
API 게이트웨이는 수많은 마이크로서비스를 향하는 API 요청의 관문이다. 마치 객체 지향 디자인 패턴의 **퍼사드 패턴(Facade Pattern)**을 네트워크 아키텍처 레벨로 끌어올린 것과 같다. 클라이언트의 요청을 받아 적절한 내부 마이크로서비스로 라우팅(Routing)할 뿐만 아니라, 흩어져 있는 여러 서비스의 데이터를 한 번에 묶어(Aggregation) 전달하는 기능도 수행한다.

### 💡 비유
API 게이트웨이는 **"대형 종합병원의 중앙 접수 수납처"**와 같다.
(API Gateway가 없다면): 환자(클라이언트)가 직접 병원 지도를 보고 내과, 외과, 방사선과(마이크로서비스)를 하나하나 찾아다녀야 하고, 가는 곳마다 신분증(인증)을 내고 접수해야 한다.
(API Gateway가 있다면): 환자는 문 앞에 있는 '중앙 접수처'에만 가면 된다. 접수원이 한 번만 신분증을 확인(공통 인증)하고, "내과 가셨다가 엑스레이 찍으세요"라고 경로를 찾아주며(라우팅), 때로는 각 과의 검사 결과를 한 장의 종이로 모아서(Aggregation) 환자에게 건네준다.

### 등장 배경 및 발전 과정

#### 1. 클라이언트-마이크로서비스 간의 직접 통신(Direct Client-to-Microservice)의 고통
초기 MSA에서는 모바일 앱이 상품 정보를 보기 위해 Product 서비스, Review 서비스, Inventory 서비스에 각각 3번의 HTTP 요청을 날렸다. 이는 모바일 배터리 광탈과 모바일 네트워크 지연(Latency)의 주범이 되었고, 백엔드의 내부 IP와 스펙이 바뀔 때마다 앱 업데이트를 강제하는 심각한 결합도(Coupling)를 낳았다.

#### 2. 중앙 통제 레이어의 필요성
넷플릭스는 쏟아지는 디바이스(TV, 모바일, 웹)의 요청을 최전선에서 방어하고 라우팅하기 위해 **Zuul**이라는 API 게이트웨이를 만들었다. 이후 비동기 논블로킹(Non-blocking) I/O 모델 기반의 강력한 성능을 갖춘 **Spring Cloud Gateway**, **Kong**, **AWS API Gateway** 등으로 진화하며 클라우드 네이티브 아키텍처의 필수 인프라가 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소와 핵심 기능 (표)

| 핵심 기능 (요소) | 상세 역할 | 내부 동작 메커니즘 | 관련 기술 / 도구 | 비유 |
|------------------|-----------|-------------------|------------------|------|
| **Routing / LB** | 동적 경로 탐색 | 요청 URL(`/api/user`)을 보고 서비스 레지스트리(Eureka)를 조회하여 실제 인스턴스 IP로 전달 | Spring Cloud Gateway | 교환원 연결 |
| **Authentication**| 인증 및 인가 대행 | 클라이언트가 보낸 JWT 토큰의 서명 검증, 만료 여부를 검사 후 백엔드로 통과시킴 | OAuth2, JWT 필터 | 신분증 검사 |
| **Rate Limiting**| 과부하 보호 (Throttling)| 특정 클라이언트/IP가 1초에 100번 이상 호출하면 429(Too Many Requests) 에러 반환 | Token Bucket 알고리즘 | 놀이공원 입장 제한 |
| **Aggregation** | 데이터 취합 | 클라이언트의 요청 1번을 받아 내부의 3개 서비스 API를 비동기로 찔러서 결과를 조립해 반환 | GraphQL, RxJava | 구매 대행 조립 |
| **Protocol Translation**| 프로토콜 변환 | 외부에서는 HTTP/REST로 받고, 내부에서는 고속의 gRPC나 AMQP(메시지 큐)로 변환하여 전송 | gRPC-Gateway | 언어 통역사 |

### 심층 동작 원리

#### ① 비동기 논블로킹 (Asynchronous Non-blocking I/O)
수만 명의 사용자가 동시에 접속하는 관문이므로, 하나의 요청이 뒷단 서비스를 기다리느라 스레드를 점유(Block)하면 톰캣의 스레드 풀이 즉시 고갈된다. 현대 API 게이트웨이(Spring Cloud Gateway, Nginx)는 Netty 프레임워크 기반의 논블로킹 I/O를 사용하여, 단 몇 십 개의 스레드만으로 수만 개의 연결을 동시다발적으로 처리하는 극강의 효율을 낸다.

#### ② 프레디케이트와 필터 체인 (Predicates and Filters)
라우팅 규칙은 주로 두 가지 컴포넌트로 동작한다.
- **Predicate (조건)**: "만약 요청된 Path가 `/order/**` 이고 시간(Time)이 오전 9시 이후라면..." 이 패킷을 잡는다.
- **Filter (가공)**: 잡은 패킷을 뒷단으로 보내기 전에 "요청 헤더에 `X-User-Id`를 추가해라", "응답이 오면 응답 바디의 특정 텍스트를 마스킹해라" 등 트래픽을 마음대로 조작(Mutating)한다.

## Ⅲ. 융합 비교 및 다각도 분석

### API Gateway vs Ingress vs Service Mesh

클라우드 네이티브의 3대 통신 인프라로서 헷갈리기 쉬운 개념들이다.

| 구분 지표 | API Gateway | K8s Ingress Controller | Service Mesh (Istio) |
|-----------|-------------|------------------------|----------------------|
| **트래픽 방향**| **North-South (외부 -> 내부)** | North-South (클러스터 진입점) | **East-West (내부 파드 간)** |
| **주요 역할** | **비즈니스 특화 로직 (데이터 취합, JWT 인증, API 과금/버저닝)** | 인프라 특화 로직 (단순 L7 URL 기반 라우팅, SSL 복호화) | 내부 보안(mTLS), 트래픽 모니터링, 서킷 브레이커 대행 |
| **소속 계층** | 애플리케이션 아키텍처 계층 (개발팀 관리) | 쿠버네티스 인프라 계층 (인프라팀 관리) | 인프라 계층 (사이드카) |
| **적용 사례** | 클라이언트별 맞춤 API 제공 (BFF) | `example.com` 도메인 라우팅 | 내부 서비스 A가 B를 호출할 때 |

* ※ 실무 융합: 클라이언트 -> `[AWS ALB]` -> `[K8s Nginx Ingress]` -> `[Spring Cloud API Gateway (비즈니스 인증)]` -> `[Istio Mesh (내부 라우팅)]` -> `[백엔드 Pod]` 순으로 겹겹이 층을 이루는 것이 정석이다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 레거시 모놀리스를 20개의 마이크로서비스로 분리했다. 각 서비스가 독자적으로 JWT 검증 로직, CORS 헤더 설정, IP 차단 로직을 라이브러리로 가지고 있다 보니(분산 모놀리스 현상), 보안 정책이 한 번 바뀔 때마다 20개 서비스를 동시에 코드 수정하고 재배포해야 하는 고통을 겪고 있다.
**아키텍처 턴어라운드 (Gateway 패턴 도입)**:
1. **횡단 관심사 오프로딩(Offloading)**: 20개 마이크로서비스 내부에 흩어져 있던 인증(Spring Security JWT 필터) 및 CORS 관련 코드를 싹 다 지워버린다(경량화).
2. **Gateway 중앙화**: 맨 앞단에 **Kong API Gateway** 또는 **Spring Cloud Gateway** 파드를 띄운다. 모든 JWT 토큰 유효성 검증과 권한(Role) 확인은 오직 게이트웨이에서만 단 한 번 수행한다.
3. **내부 신뢰(Trust) 통신**: 게이트웨이 인증을 통과한 패킷은 헤더에 `X-User-Id: 101`이라는 평문 식별자만 달고 내부 서비스로 던져진다. 내부 서비스는 토큰을 검증하지 않고 이 헤더를 100% 신뢰하여 비즈니스 로직만 빠르게 처리한다.
4. **결과**: 인가/보안 정책 변경 시 오직 게이트웨이만 배포하면 되며, 백엔드 서비스의 코드 복잡도가 극적으로 낮아짐.

### 주의사항 및 안티패턴 (Anti-patterns)
- **비대한 게이트웨이 (Fat Gateway)**: 공통 로직을 넣으라고 했더니, 개발자들이 '특정 상품 할인율 계산', '장바구니 조합' 같은 무거운 도메인 비즈니스 로직까지 게이트웨이에 코딩하기 시작한다. 이는 게이트웨이를 또 다른 거대한 모놀리식 괴물(Monolithic Gateway)로 만들어 전체 시스템의 성능을 박살 낸다. 게이트웨이는 **'네트워크 횡단 관심사'**만 처리하는 얇은 막(Thin Layer)으로 유지해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **모바일/클라이언트 성능 극대화**: Aggregation 기능을 통해 모바일 앱의 백엔드 호출 횟수를 5회에서 1회로 줄여, 라디오 네트워크 지연(Over-the-air latency)을 80% 이상 감소.
- **보안 일관성 달성**: 단일 진입점에서 완벽한 Rate Limiting과 인증을 100% 보장하여, 클러스터 내부에 숨어있는 취약한 서비스로의 직접적인 DDoS 공격 원천 차단.

### 미래 전망 및 진화 방향
하나의 거대한 API 게이트웨이가 모든 클라이언트(웹, 모바일 앱, 외부 파트너사)를 처리하다 보니 게이트웨이 자체가 병목이 되는 현상이 발생했다. 이에 따라 모바일 전용 게이트웨이, 웹 전용 게이트웨이로 게이트웨이 자체를 쪼개는 **BFF (Backend For Frontend)** 아키텍처 패턴으로 진화하고 있다. 또한 관리형 클라우드 서비스(AWS API Gateway, Azure API Management)와 람다(Serverless)의 결합이 기본 인프라로 자리 잡으면서, 서버 구축 없이 설정만으로 무한대 확장이 가능한 시대로 접어들었다.

## 👶 어린이를 위한 3줄 비유 설명
1. **API 게이트웨이란?**: 놀이공원의 '중앙 정문 매표소'예요. 놀이공원 안에는 롤러코스터, 바이킹 등 수많은 놀이기구(서비스)가 있죠.
2. **왜 좋나요?**: 놀이기구마다 매표소를 두면 손님들이 탈 때마다 지갑을 꺼내서 표를 사야 하니까 너무 귀찮잖아요. 정문 매표소에서 한 번만 자유이용권을 검사(인증)하면, 안에서는 신나게 놀기만 하면 돼요.
3. **또 무슨 일을 하죠?**: 손님이 "가장 인기 있는 기구 3개 타는 코스 짜주세요!" 하면, 매표소 직원이 알아서 지도와 시간표를 묶어서 한 장의 종이(Aggregation)로 건네주는 똑똑한 안내원 역할도 한답니다.
