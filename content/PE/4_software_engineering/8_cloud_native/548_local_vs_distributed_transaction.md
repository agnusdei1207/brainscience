# 548. 로컬 트랜잭션 (Local Transaction) vs 분산 트랜잭션 (Distributed Transaction)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스의 무결성을 보장하는 기술로, 로컬 트랜잭션은 단일 DB 내에서 완벽한 ACID를 보장하는 전통적 방식인 반면, 분산 트랜잭션은 여러 개의 독립적인 마이크로서비스(DB)에 걸쳐 발생하는 데이터 변경을 일관성 있게 묶어내는 메커니즘이다.
> 2. **가치**: 모놀리식 시스템에서는 하나의 커넥션으로 쉽게 달성하던 트랜잭션을 마이크로서비스 아키텍처(MSA)에서는 네트워크를 통해 풀어내야 하므로, 시스템의 강한 정합성과 높은 가용성 사이의 뼈아픈 트레이드오프(CAP 정리)를 강제한다.
> 3. **융합**: 고전적인 2PC(Two-Phase Commit)의 성능 한계를 극복하기 위해, 메시지 큐(Kafka)를 활용한 비동기 이벤트 통신과 **사가 패턴(Saga Pattern)**으로 융합/진화하며 현대 클라우드 네이티브의 최종적 일관성(Eventual Consistency) 모델을 완성한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
- **로컬 트랜잭션 (Local Transaction)**: 오직 하나의 데이터베이스 인스턴스와 하나의 데이터베이스 커넥션 내에서 일어나는 트랜잭션. RDBMS 자체(예: MySQL 엔진)가 락(Lock)을 걸어 ACID(원자성, 일관성, 고립성, 지속성)를 100% 완벽하게 보장해 준다.
- **분산 트랜잭션 (Distributed Transaction)**: 두 개 이상의 서로 다른 네트워크 노드나 이기종(Heterogeneous) 데이터베이스에 걸쳐 데이터를 변경해야 할 때, 이들이 "모두 성공(Commit)하거나 모두 실패(Rollback)"하도록 논리적으로 묶어주는 거시적 트랜잭션이다.

### 💡 비유
- **로컬 트랜잭션**: "한 사람이 왼손에서 오른손으로 동전 옮기기". 내 뇌(DB 엔진)가 완벽히 통제하므로, 동전을 떨어뜨리면 즉시 다시 주워서 원상복구(Rollback) 할 수 있다. 실패 확률이 0에 가깝다.
- **분산 트랜잭션**: "한국에 있는 철수가 미국에 있는 스미스에게 동전 던져주기". 동전을 던지는 순간 전화(네트워크)가 끊길 수도 있고, 스미스가 자고 있을 수도 있다. 둘 다 동전을 정확히 주고받았는지 확인하기 위해 수많은 확인 전화(2PC)를 돌리거나, 잃어버렸을 때 보상해 줄 방법(Saga)을 미리 정해둬야 하는 엄청난 골칫거리다.

### 등장 배경 및 발전 과정

#### 1. RDBMS의 황금기 (Local Transaction)
과거 거대한 모놀리식(Monolithic) 시스템은 거대한 오라클 DB 하나를 썼다. `@Transactional` 어노테이션 하나만 붙이면 DB 엔진이 트랜잭션 격리 수준(Isolation Level)을 맞추어 알아서 롤백해 주었다. 개발자는 분산 환경의 무서움을 몰랐다.

#### 2. MSA 분해와 DB 찢기 (Distributed Transaction의 도래)
시스템이 클라우드로 넘어가며 50개의 마이크로서비스로 쪼개졌고, 각각 **독자적인 DB(Database per Service)**를 가지게 되었다. 
"결제를 하면(A 서비스 DB), 쿠폰을 깎고(B 서비스 DB), 배송을 지시(C 서비스 DB)한다."
A는 성공했는데 네트워크가 끊겨 B, C가 실패하면 고객의 돈만 빠져나가고 물건은 오지 않는 대참사(데이터 불일치)가 발생했다. 이 쪼개진 DB들을 어떻게 하나의 원자성(Atomicity)으로 묶을 것인가가 MSA 최대의 난제로 부상했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 트랜잭션 방식의 심층 비교 (표)

| 비교 속성 | 로컬 트랜잭션 (Local Transaction) | 분산 트랜잭션 (2PC 기반) | 분산 트랜잭션 (Saga/EDA 기반) |
|-----------|-----------------------------------|--------------------------|-------------------------------|
| **통제 주체** | 데이터베이스 엔진 (DBMS) | 글로벌 트랜잭션 코디네이터 (TC) | 각 마이크로서비스 (비동기 협력) |
| **일관성 수준**| 강력한 일관성 (Strong Consistency)| 강력한 일관성 (ACID 보장 시도) | **최종적 일관성 (Eventual)** |
| **블로킹 여부**| 로컬 DB 락 (짧은 블로킹) | 모든 원격 DB 락 대기 (치명적 블로킹)| **논블로킹 (No Lock)** |
| **네트워크 의존**| 없음 (내부 IPC) | 매우 높음 (중간에 끊기면 데드락) | 메시지 브로커(Kafka)가 완충 역할 |
| **성능 (TPS)** | 수만 ~ 수십만 TPS | 매우 낮음 (수십~수백 TPS) | 매우 높음 (스케일 아웃 용이) |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ The Nightmare of Microservices: Distributed Transaction ]
================================================================================

[ 1. Local Transaction (Monolithic) - 행복했던 시절 ]
  ┌──────────────────────────────────────────────┐
  │ Business Logic: placeOrder()                 │
  │  BEGIN TRAN                                  │
  │    UPDATE orders SET status = 'PAYED';       │  <-- DB 내부의 단일 Connection.
  │    UPDATE coupons SET used = true;           │      순식간에 Lock 걸고 풀림.
  │    UPDATE inventory SET stock = stock - 1;   │      에러 시 즉각 100% 원상복구.
  │  COMMIT (or ROLLBACK)                        │
  └──────────────────────────────────────────────┘
                        ▼
                [ 단일 RDBMS (Oracle/MySQL) ]

[ 2. Distributed Transaction Problem (MSA) - 지옥의 시작 ]
  ┌─────────────┐       (HTTP POST: 지연/실패 가능)       ┌─────────────┐
  │ Order Svc   │ ────────────────────────────────────> │ Coupon Svc  │
  │ (DB 1)      │                                       │ (DB 2)      │
  └─────────────┘                                       └─────────────┘
  * Order Svc: "나는 DB 1에 주문 완료 저장(Commit)했어!"
  * Coupon Svc: (쿠폰 깎으려다 DB 2 다운됨. 에러 발생)
  * 결과(데이터 불일치): 주문은 완료되었는데 쿠폰은 안 깎인 유령 데이터(Zombie) 발생.
  * 딜레마: Order Svc는 이미 DB 1에 Commit을 때렸기 때문에, 원격에 있는 DB 1을 다시 Rollback 시킬 권한이 없음.
```

### 심층 동작 원리 (어떻게 극복하려 했는가?)

#### ① 2PC (Two-Phase Commit)의 등장과 몰락
이 분산의 저주를 풀기 위해 XA 표준 기반의 2단계 커밋(2PC)이 등장했다. 
- **1단계(Prepare)**: 코디네이터가 모든 DB에게 "지금부터 락(Lock) 걸고 커밋할 준비 해!"라고 명령.
- **2단계(Commit)**: 모든 DB가 "준비 완료"라고 응답하면, 그때서야 "동시에 커밋!"이라고 명령.
*한계*: 2단계 진행 중 하나의 DB라도 응답이 늦어지면 전체 시스템의 DB에 락(Lock)이 걸린 채 무한 대기하는 최악의 병목 현상(Blocking)이 발생하여, 클라우드 환경에서는 절대 금기시되는 안티패턴이 되었다.

#### ② 보상 트랜잭션 (Compensating Transaction)
DBMS 엔진이 제공하는 자동 롤백을 포기한다. 대신, 취소(Rollback)를 수행하는 **"역방향 비즈니스 로직(코드)"**을 개발자가 직접 짜는 것이다.
"쿠폰을 깎아라(+1)"의 보상 트랜잭션은 "쿠폰을 다시 돌려줘라(-1)"를 호출하는 것이다. 앞선 서비스가 실패하면, 그동안 성공했던 과거 서비스들에게 "아까 한 거 취소(보상)해라"라는 메시지를 릴레이로 날려서 논리적인 원상복구를 수행한다.

## Ⅲ. 융합 비교 및 다각도 분석

### CAP 정리에 따른 트랜잭션 전략 트레이드오프

분산 트랜잭션의 세계는 CAP 정리(Consistency, Availability, Partition Tolerance)의 지배를 받는다. P(네트워크 단절)가 기본인 클라우드에서는 C와 A 중 하나를 선택해야 한다.

| 설계 지향점 | 시스템 선택 (CAP) | 사용하는 트랜잭션 기법 | 장점 | 단점 |
|-------------|-------------------|------------------------|------|------|
| **전통 금융/결제**| **CP (일관성 최우선)** | 2PC, XA 트랜잭션, Google Spanner(Truetime) | 데이터가 절대 틀어지지 않음 | 네트워크 지연 시 시스템 전체가 멈춰버림 (가용성 포기) |
| **대형 커머스/SNS**| **AP (가용성 최우선)** | Saga Pattern, Event Sourcing, 비동기 MQ | 시스템이 멈추지 않고 항상 응답함 | 1초 정도는 내 잔고와 실제 잔고가 불일치할 수 있음 (최종적 일관성) |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대형 이커머스의 MSA 아키텍처. 고객이 "주문 취소"를 누르면, `결제 취소(PG사 연동)`, `재고 원복(DB Update)`, `포인트 회수(DB Update)`가 동시에 이뤄져야 한다. PG사 연동은 외부 통신이라 3초 이상 걸리며 간헐적으로 타임아웃이 발생한다.
**판단 및 트랜잭션 설계 전략**:
1. **2PC (동기식 분산 트랜잭션) 기각**: 결제, 재고, 포인트 DB를 2PC로 묶으면 PG사 타임아웃 3초 동안 회사 내의 핵심 재고 DB 테이블에 락(Lock)이 걸려 다른 모든 고객의 구매가 정지된다(재앙).
2. **Saga 패턴 도입 (최종적 일관성)**: 
   - 주문 취소 요청이 오면 상태를 `취소 처리 중(PENDING)`으로만 로컬 트랜잭션 처리 후 사용자에게 "취소가 접수되었습니다"라고 빠른 응답(가용성)을 준다.
   - 백그라운드 이벤트 브로커(Kafka)를 통해 각 서비스에 비동기 취소 메시지를 날린다.
   - 만약 `결제 취소`가 잔액 부족으로 실패하면, 이미 올려놓은 `재고 원복` 서비스 쪽에 **"재고 다시 깎아(보상 트랜잭션)"**라는 이벤트를 던져 논리적인 정합성을 맞춘다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **분산 트랜잭션 남발 (Distributed Transaction Heavy)**: "어차피 Saga 패턴이 있으니까 마이크로서비스를 마음껏 잘게 쪼개도 되겠네?"라는 생각은 가장 위험하다. Saga 패턴은 구현, 테스트, 디버깅 복잡도가 로컬 트랜잭션의 10배에 달한다. 
**최고의 분산 트랜잭션 전략은 "분산 트랜잭션 자체를 만들지 않는 것"이다.** 강한 정합성이 필요한 데이터(예: 주문과 주문상세)는 억지로 쪼개지 말고 하나의 바운디드 컨텍스트(단일 마이크로서비스 + 단일 로컬 DB)에 몰아넣어 로컬 트랜잭션의 혜택을 누리는 것이 MSA 설계의 정석이다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **성능 및 확장성**: 동기식 2PC가 유발하는 DB 테이블 락(Lock) 대기를 제거하여, 마이크로서비스 본연의 수만 TPS 트래픽 처리량(Throughput)을 확보함.
- **장애 격리 (Fault Tolerance)**: 특정 데이터베이스가 장애로 뻗어도, 해당 트랜잭션을 제외한 시스템의 나머지 99%는 영향을 받지 않고 독자적인 생존 가능.

### 미래 전망 및 진화 방향
클라우드 벤더와 데이터베이스 제조사들은 소프트웨어 레벨의 사가(Saga) 패턴이 너무 복잡하다는 것을 알고 있다. 이에 대응하여, 전 세계에 흩어진 분산 DB임에도 불구하고 구글의 원자 시계(Atomic Clock)와 GPS를 이용해 시간의 오차를 없애어 마치 완벽한 1개의 로컬 DB처럼 ACID 트랜잭션을 보장해 주는 **Google Cloud Spanner**나 **CockroachDB (NewSQL)** 같은 혁명적인 아키텍처가 등장하며 데이터 정합성의 패러다임을 또 한 번 바꾸고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **로컬 트랜잭션이란?**: 내 주머니 안에서 왼손에 있는 사탕을 오른손으로 옮기는 거예요. 동전이 떨어질 것 같으면 내 마음대로 얼른 꽉 쥐어서 100% 안전하게 지킬 수 있어요.
2. **분산 트랜잭션이란?**: 비행기 타고 미국에 간 친구한테 택배로 사탕을 보내는 거예요. 가다가 배가 침몰할 수도 있고 친구가 이사 갔을 수도 있어서 사탕을 안전하게 전달하기가 너무너무 힘들어요.
3. **어떻게 해결하나요?**: 비행기 뜨기 전에 수십 번씩 전화를 걸어 확실히 확인하거나(너무 느림), 일단 보내놓고 만약 못 받았다고 연락 오면 나중에 새 사탕을 다시 사주는(사가 패턴) 똑똑한 방식을 쓴답니다.
