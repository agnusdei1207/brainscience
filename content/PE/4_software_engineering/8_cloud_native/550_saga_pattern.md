# 550. 사가 패턴 (Saga Pattern) - 로컬 트랜잭션들의 연속된 체인

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로서비스 아키텍처(MSA)에서 여러 서비스에 걸친 분산 트랜잭션을 2PC(Two-Phase Commit) 같은 동기적 락(Lock) 없이 비동기적으로 처리하기 위해, 일련의 독립적인 **로컬 트랜잭션들을 체인(Chain)처럼 연결**하여 실행하는 디자인 패턴이다.
> 2. **가치**: 글로벌 락(Global Lock)이 없으므로 데이터베이스가 병목 없이 수만 TPS의 초고속 성능(가용성)을 내며, 도중에 실패할 경우 **보상 트랜잭션(Compensating Transaction)**을 역순으로 실행하여 궁극적으로 데이터의 상태를 일치시키는 '최종적 일관성(Eventual Consistency)'을 보장한다.
> 3. **융합**: 이벤트 브로커(Kafka, RabbitMQ)를 이용한 비동기 통신과 결합하며, 중앙에서 흐름을 지휘하는 오케스트레이션(Orchestration) 방식과 각자가 알아서 이벤트를 구독하는 코레오그래피(Choreography) 방식으로 나뉘어 구현된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
사가(Saga) 패턴은 분산 환경의 비즈니스 트랜잭션을 T1, T2, T3... 와 같은 개별 마이크로서비스의 '로컬 트랜잭션 집합'으로 정의한다. 각 로컬 트랜잭션이 자신의 DB에 커밋(Commit)을 완료하면, 다음 트랜잭션을 트리거하는 이벤트나 메시지를 발행한다. 만약 T3 단계에서 비즈니스 규칙 위반(예: 잔액 부족)으로 실패하면, 시스템은 T3를 취소함과 동시에 이전에 성공했던 T2와 T1을 취소하기 위한 **역방향 보상(Compensation) 트랜잭션**인 C2, C1을 차례대로 실행하여 시스템을 원래 상태로 되돌린다.

### 💡 비유
사가 패턴은 **"해외여행 패키지 예약 과정(호텔, 비행기, 렌터카)"**과 같다.
- **정상(성공)**: 호텔 예약(성공) -> 그 다음 비행기 예약(성공) -> 렌터카 예약(성공) -> 여행 확정!
- **보상 트랜잭션(실패 시)**: 호텔 예약(성공) -> 비행기 예약(성공) -> 렌터카 예약(**매진으로 실패!**).
이때 이미 확정된 비행기와 호텔을 취소해야 한다. 시스템은 시간을 되돌릴 순 없지만, 비행기 항공사에 "예약 취소 수수료 지불하고 취소해 줘(C2)"라고 전화하고, 호텔에 "방금 예약한 거 취소해 줘(C1)"라고 전화해서 렌터카가 실패한 상황에 맞게 모든 예약을 원상 복구(환불)시킨다.

### 등장 배경 및 발전 과정

#### 1. 1987년, 장기 실행 트랜잭션(LLT)의 대안
Saga라는 용어는 1987년 프린스턴 대학의 논문에서 처음 등장했다. 당시 몇 시간에서 며칠씩 걸리는 거대한 데이터베이스 트랜잭션(Long Lived Transaction)이 DB 테이블 전체를 수일 동안 잠가버리는(Locking) 문제를 해결하기 위해, 트랜잭션을 잘게 쪼개고 실패 시 보상(Undo)을 하자는 개념으로 제안되었다.

#### 2. 마이크로서비스(MSA) 시대의 화려한 부활
2010년대 들어 MSA가 대세가 되면서, 서비스마다 DB가 물리적으로 찢어지는 상황(Database per Service)이 발생했다. 동기적 락을 거는 2PC 방식이 클라우드에서 치명적인 병목을 일으키자, 개발자들은 30년 전 논문 속에 묻혀있던 Saga 개념을 발굴하여 비동기 메시지 큐(Kafka)와 융합함으로써 클라우드 네이티브 시대의 절대적인 트랜잭션 표준으로 부활시켰다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 및 트랜잭션 흐름 (표)

| 트랜잭션 유형 | 상세 역할 | 속성 / 한계 | 비유 |
|---------------|-----------|-------------|------|
| **Pivot Transaction (기준점)**| 사가의 성공/실패를 확정 짓는 핵심 트랜잭션 | 이 트랜잭션이 성공하면 나머지(재고 차감 등)는 무조건 성공해야 함 | 비행기 티켓 확정 |
| **Local Transaction ($T_i$)**| 각 서비스 내부 DB에서 실행되는 순방향 비즈니스 로직 | 독립적 수행. 성공 시 다음 서비스로 이벤트 전달 | 호텔 방 잡기 |
| **Compensating Transaction ($C_i$)**| 앞서 성공한 $T_i$의 결과를 논리적으로 무효화하는 역방향 로직 | 에러 발생 시 $T$의 역순으로 실행됨. **절대 실패해선 안 됨** | 호텔 환불 처리 |
| **Retriable Transaction** | 실패하더라도 무한 재시도하면 결국 성공하는 트랜잭션 | 브로커가 보장. 비즈니스 로직 실패가 아닌 네트웍 오류일 때 적용 | 결제 완료 메일 발송 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ Saga Pattern: Success vs Compensation Flow ]
================================================================================

[ 1. 정상 흐름 (Happy Path) ]
  ┌────────────┐        ┌─────────────┐        ┌──────────────┐
  │ Order Svc  │ ──T1─> │ Payment Svc │ ──T2─> │ Inventory Svc│
  │ (주문 생성) │ (Event)│ (카드 결제)  │ (Event)│ (재고 차감)   │
  └────────────┘        └─────────────┘        └──────────────┘
    T1: 상태=PENDING       T2: 잔액 확인, 승인        T3: 재고 -1, (최종 주문 완료)

[ 2. 실패 및 보상 흐름 (Compensation Path) ]
  ┌────────────┐        ┌─────────────┐        ┌──────────────┐
  │ Order Svc  │ ──T1─> │ Payment Svc │ ──T2─> │ Inventory Svc│
  │ (주문 취소) │ <─C1── │ (결제 환불)  │ <─C2── │ (재고 없음!)  │
  └────────────┘ (Event)└─────────────┘ (Event)└──────────────┘
    T1: 상태=PENDING       T2: 잔액 확인, 승인        [T3: 재고 부족 실패 발생!!]
    C1: 상태=CANCELLED     C2: 카드사 승인 취소요청    (실패 이벤트를 역방향으로 쏨)
```

### 심층 동작 원리: 보상(Compensation)의 철학

RDBMS의 `ROLLBACK`은 물리적으로 데이터를 롤백(예: Insert된 행을 지움)한다. 하지만 사가 패턴의 보상은 물리적 롤백이 아니다. 이미 T1(주문 생성)과 T2(결제 승인)는 각자의 DB에 영구적으로 `COMMIT`되어 물리적으로 확정된 상태다.
따라서 C2(결제 환불)는 결제 이력을 DB에서 삭제(Delete)하는 것이 아니라, **"-10,000원(환불)이라는 새로운 데이터 행(Row)을 Insert"**하여 이전 작업의 결과를 논리적으로 상쇄(Semantic Undo)시키는 완전히 새로운 비즈니스 로직(코드)이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 2PC (동기식) vs Saga 패턴 (비동기식)

| 비교 지표 | 2PC (Two-Phase Commit) | Saga Pattern |
|-----------|------------------------|--------------|
| **일관성 보장** | 완벽한 ACID 강한 일관성 | **BASE (최종적 일관성)**. 도중에 불일치한 상태가 외부에 잠시 노출됨 |
| **장애 격리 (격리성, I)**| 최악 (타 DB 테이블 락 유지) | **최상** (로컬 트랜잭션 직후 락 해제. 다른 요청 방해 안 함) |
| **구현 난이도** | 인프라/미들웨어 지원 (코딩 쉬움) | **극도로 어려움** (개발자가 모든 실패 시나리오의 보상 코드를 짜야 함) |
| **처리량 (성능)** | 수십 TPS (병목 심함) | **수만 TPS** (무한 스케일 아웃 가능) |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대형 배달 플랫폼에서 `[결제 -> 음식점 수락 -> 라이더 배차]` 프로세스를 구축 중이다. 각 단계는 서로 다른 부서가 만든 마이크로서비스다. 만약 결제하고 음식점이 수락했는데, 폭우로 라이더 배차가 10분 동안 실패하면 고객에게 100% 환불 처리되어야 한다.
**판단 및 아키텍처 전략**:
1. **사가 패턴 2대 구현 방식 중 선택 (코레오그래피 vs 오케스트레이션)**:
   - 각 단계(음식점, 배차)가 독립적으로 길게 진행되므로 중앙에서 "다음은 네가 해"라고 명령하는 **오케스트레이션(Orchestration) 방식**이 적합하다. 
   - 주문 서비스(또는 전용 State Machine)가 사가 실행자(SEC, Saga Execution Coordinator) 역할을 맡아 전체 워크플로우 상태를 관리한다.
2. **보상 트랜잭션 설계**: 배차가 실패(Timeout)하면, 사가 실행자는 즉시 음식점 서비스에 "주문 강제 취소해(C2)" 이벤트를 쏘고, 이어서 결제 서비스에 "카드 승인 취소해(C1)" 이벤트를 쏜다.
3. **멱등성(Idempotency) 보장**: 네트워크 오류로 보상 이벤트가 음식점 서비스에 2번 전송될 수 있다. 따라서 음식점 서비스의 취소 로직(C2)은 두 번 호출되어도 한 번만 취소되도록 무조건 멱등성 방어 코드를 짜야 한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **격리성(Isolation) 부재에 의한 데이터 오염 (Lost Updates)**: 사가 패턴은 ACID 중 'I(격리성)'가 없다. 주문(T1) 직후 아직 배송(T2)이 끝나기 전에, 고객이 화면을 새로고침하면 '주문 완료' 상태를 볼 수 있다. 그런데 1초 뒤 배송(T2)이 실패해 사가가 보상(C1)을 시작하면, 고객 화면은 갑자기 '주문 취소'로 바뀐다. 고객이 이 사이에 주문 정보를 수정하려고 했다면 치명적 데이터 꼬임이 발생한다. 이를 막기 위해 상태값에 `PENDING` 같은 임시 상태를 두어 사가가 완료될 때까지 UI 접근을 막는(Semantic Lock) 설계가 필수다.

## Ⅴ. 기대효과 및 결론

### 정량적/정성적 기대효과
- **가용성과 성능의 해방**: 어떤 상황에서도 DB 간 동기적 락 대기가 없으므로, 트래픽 폭주 시 서비스가 뻗지 않고 유연하게 버퍼링(Load Leveling)되며 높은 TPS를 보장.
- **완벽한 서비스 독립성**: 쿠폰 서비스팀이 DB 스키마를 맘대로 바꿔도, 결제 서비스팀은 이벤트 메시지(JSON) 포맷만 맞으면 아무런 장애 없이 배포 가능.

### 미래 전망 및 진화 방향
사가 패턴은 개발자에게 '보상 트랜잭션 작성'이라는 엄청난 짐을 지웠다. 이 복잡도를 줄이기 위해 최근에는 **이벤트 소싱(Event Sourcing)**과 **CQRS(명령/조회 책임 분리)** 패턴이 사가와 결합하여 상태 관리를 더욱 명확히 하고 있다. 또한 AWS Step Functions, Netflix Conductor, Temporal 같은 완전 관리형(Managed) 서버리스 오케스트레이터가 등장하여, 개발자는 순수 비즈니스 및 보상 코드(Lambda)만 짜면 인프라가 사가의 상태 머신(재시도, 롤백 흐름 제어)을 알아서 완벽하게 돌려주는 환경으로 진화하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **사가(Saga) 패턴이란?**: 레고 성을 만들 때 친구 3명이 순서대로 1층, 2층, 3층을 쌓기로 약속(트랜잭션)한 거예요.
2. **실패하면 어떻게 하죠?**: 1층, 2층을 잘 쌓았는데 3층 친구가 갑자기 배가 아파서 집에 가버렸어요. 시간을 되돌릴 순 없으니까, 2층 친구가 자기 걸 부수고(보상), 1층 친구도 자기 걸 부숴서(보상) 다시 원상태인 빈 책상으로 만들어 놓는 규칙이에요.
3. **왜 이 방식이 좋나요?**: 3층 친구가 올 때까지 1, 2층 친구가 화장실도 못 가고 계속 붙잡혀서 기다릴 필요(Lock)가 없어서, 친구들 모두 스트레스 없이 아주 자유롭게 일할 수 있답니다!
