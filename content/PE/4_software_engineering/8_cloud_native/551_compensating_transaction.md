# 551. 보상 트랜잭션 (Compensating Transaction) - 롤백을 논리적으로 수행하는 역방향 연산

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로서비스(MSA)와 같은 분산 시스템에서, 이미 커밋(Commit)되어 되돌릴 수 없는 로컬 트랜잭션의 결과를 논리적으로 취소(상쇄)하기 위해 실행되는 '역방향 비즈니스 로직'이다.
> 2. **가치**: 글로벌 락(Lock)을 거는 2PC의 치명적 성능 저하를 피하고, 시스템의 가용성과 성능을 극대화(비동기 처리)하면서도 비즈니스 데이터의 '최종적 일관성(Eventual Consistency)'을 보장하는 궁극의 타협점이다.
> 3. **융합**: 분산 트랜잭션 관리 아키텍처인 **사가 패턴(Saga Pattern)**을 구성하는 가장 핵심적인 빌딩 블록이며, 메시지 큐(Kafka)를 통한 이벤트 기반 아키텍처(EDA)와 필수적으로 융합되어 동작한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
RDBMS의 전통적인 트랜잭션에서는 작업 도중 에러가 나면 DBMS 엔진이 `ROLLBACK` 명령어를 통해 물리적으로 데이터를 지우고 변경 전 상태로 되돌린다. 반면 분산 환경에서는 A 서비스의 트랜잭션이 성공하여 이미 DB에 커밋된 후, 한참 뒤에 B 서비스에서 실패가 발생한다. A 서비스의 DB 엔진은 이미 종료된 트랜잭션을 롤백할 능력이 없다.
따라서 개발자가 직접 **"이전의 작업 결과를 무효화하는 새로운 작업(보상 트랜잭션)"**을 코딩하여 실행시켜야 한다.

### 💡 비유
보상 트랜잭션은 **"계좌 이체 취소와 환불"**과 같다.
- **물리적 롤백 (RDBMS)**: 내가 ATM에서 친구에게 5만 원을 보내는 버튼을 눌렀는데, 통신 에러가 났다. ATM 화면에 "거래 실패"가 뜨고 내 돈 5만 원은 아예 빠져나가지도 않았다. (아무 일도 없던 것으로 무효화).
- **보상 트랜잭션 (MSA)**: 내가 쇼핑몰에서 5만 원을 결제했다(A 시스템 성공). 그런데 10분 뒤 창고(B 시스템)에서 재고가 없다는 사실이 확인되었다. 쇼핑몰은 내 결제 내역을 DB에서 아예 '삭제(Delete)'하는 것이 아니라, "-50,000원 환불 완료"라는 **새로운 영수증(역방향 작업)**을 하나 더 발급하여 내 통장 잔고를 원래대로 맞춰준다.

### 등장 배경 및 발전 과정

#### 1. 분산 트랜잭션(2PC)의 한계 봉착
초기 기업들은 여러 DB를 묶기 위해 2단계 커밋(2PC)을 썼지만, 클라우드의 잦은 네트워크 지연 환경에서 글로벌 락(Lock)으로 인해 전체 시스템이 멈춰버리는 끔찍한 병목을 겪었다.

#### 2. 장기 실행 트랜잭션 (Long-Lived Transaction)
여행 예약(비행기->호텔->렌터카)처럼 완료되는 데 몇 분에서 며칠이 걸리는 트랜잭션에 며칠 동안 DB 락을 걸어둘 수는 없었다. 락을 푸는 대신, "실패하면 환불해 주는 프로세스를 만들자"는 개념이 제안되었다. 이 개념이 마이크로서비스의 철학(DB 독립성 보장)과 정확히 맞아떨어지면서 MSA 분산 트랜잭션의 표준이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 트랜잭션의 3가지 속성 (사가 패턴 관점) (표)

사가 패턴 내의 로컬 트랜잭션들은 성격에 따라 3가지로 분류되며, 이 중 보상 트랜잭션이 핵심 방어막 역할을 한다.

| 트랜잭션 유형 | 역할 및 특징 | 예시 (이커머스) | 실패 시 동작 원리 |
|---------------|--------------|-----------------|-------------------|
| **Compensatable (보상 가능)** | 앞부분에 위치. 언제든 역방향 연산으로 되돌릴 수 있는 트랜잭션 | 카드 승인, 포인트 차감 | 실패 시, 개발자가 작성한 **보상 트랜잭션**이 반드시 호출됨 |
| **Pivot (피벗)** | 사가의 성공/실패를 확정 짓는 '돌아올 수 없는 강(Point of no return)' | 재고 창고 출고 지시 | 이 지점을 통과하면 사가는 무조건 성공해야 함 |
| **Retriable (재시도 가능)** | 피벗 이후에 위치. 비즈니스 로직상 실패할 수 없는 트랜잭션 | 결제 완료 메일 발송 | 이메일 서버가 죽으면, 살아날 때까지 무한 **재시도(Retry)** 함 (보상 불가) |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ Compensating Transaction Flow (Semantic Undo) ]
================================================================================

 (시간의 흐름 ──────────────────────────▶)

  [ Step 1: Order Svc ]         [ Step 2: Pay Svc ]          [ Step 3: Inventory Svc ]
   로컬 트랜잭션 T1               로컬 트랜잭션 T2              로컬 트랜잭션 T3
  ┌──────────────────┐          ┌──────────────────┐         ┌────────────────────┐
  │ INSERT Order     │ ─이벤트─> │ UPDATE Wallet    │ ─이벤트─> │ UPDATE Stock       │
  │ (Status: PENDING)│          │ (Balance: -5만)  │          │ (ERROR: 재고 없음!) │
  └────────┬─────────┘          └────────┬─────────┘         └────────┬───────────┘
           │ (Commit 완료됨)              │ (Commit 완료됨)             │ (Rollback)
           ▼                              ▼                           ▼
       [ Order DB ]                   [ Wallet DB ]               [ Stock DB ]
  
  (실패 감지! 역방향으로 보상 이벤트(C2, C1) 발행 ◀──────────────────────────)

  [ Step 4: Order Svc ]         [ Step 5: Pay Svc ]
   보상 트랜잭션 C1               보상 트랜잭션 C2
  ┌──────────────────┐          ┌──────────────────┐
  │ UPDATE Order     │ <─이벤트─ │ UPDATE Wallet    │ <─이벤트─ (T3에서 발행)
  │ (Status: CANCEL) │          │ (Balance: +5만)  │   "재고 부족 취소 사태 발생!"
  └────────┬─────────┘          └────────┬─────────┘
           │ (새로운 Commit)              │ (새로운 Commit)
           ▼                              ▼
       [ Order DB ]                   [ Wallet DB ]
       (상태가 변경됨)               (-5만, +5만 상쇄되어 0원 됨. 최종적 일관성 달성)
```

### 심층 동작 원리 (보상 트랜잭션의 3대 필수 조건)

#### ① 의미론적 복구 (Semantic Undo)
보상 트랜잭션은 `DELETE`가 아니다. T2에서 지갑 잔고를 차감하는 `UPDATE balance = balance - 50000`을 쳤다면, C2(보상)는 `UPDATE balance = balance + 50000` 이라는 새로운 비즈니스 행위(Semantic)를 커밋하는 것이다. 물리적인 과거 기록은 그대로 남아 고객 문의 시 로그(증적)로 활용된다.

#### ② 멱등성 (Idempotency)의 보장
분산 환경에서 브로커(Kafka) 장애나 네트워크 지연으로 인해 보상 이벤트가 C2 서비스로 두 번(Duplicate) 날아갈 수 있다. 5만 원 환불 이벤트가 2번 들어왔을 때 10만 원을 환불해주면 회사에 치명적 손실이 발생한다. 따라서 C2 코드는 "이 `order_id`에 대해 이미 환불 처리가 완료되었는가?"를 확인하는 멱등성 방어 코드(Idempotency Key 체크)가 무조건 구현되어야 한다.

#### ③ 절대 실패해서는 안 됨 (Must Succeed)
T2가 성공했는데 C2가 실패하면 시스템은 영원한 데이터 불일치 상태에 빠진다. (돈은 나갔는데 환불이 안 됨). 따라서 보상 트랜잭션(C2)은 비즈니스 규칙 위반(잔액 부족 등)으로 실패할 수 없게 설계되어야 하며, 일시적인 네트워크 오류로 실패할 경우에는 성공할 때까지 브로커가 무한히 재시도(Retry)하거나 에러 큐(Dead Letter Queue)에 넣어 관리자가 수동으로 처리하도록 알람을 울려야 한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 트랜잭션 롤백 매커니즘: DBMS vs MSA

| 구분 지표 | RDBMS Physical Rollback (Undo Log) | MSA Compensating Transaction (Saga) |
|-----------|------------------------------------|-------------------------------------|
| **실행 주체** | 데이터베이스 엔진 (DBMS) | 프로그래머가 작성한 애플리케이션 코드 |
| **복구 방식** | Undo 로그를 읽어 디스크의 값을 이전 상태로 복사 (덮어쓰기) | 상태를 반전시키는 새로운 트랜잭션을 실행 (Insert/Update) |
| **데이터 흔적**| 실패한 트랜잭션은 DB에 어떤 흔적도 남기지 않음 (증발) | 실패/취소/환불 이력이 모든 DB에 영구적으로 기록됨 (감사 추적 가능) |
| **격리성 (Isolation)**| 트랜잭션 도중 남들이 쓰다 만 데이터를 볼 수 없음 (Read Committed 등) | **심각한 격리성 문제 발생 (더티 리드)**. 보상이 끝나기 전 다른 사용자가 깎인 잔액을 볼 수 있음 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 사용자가 KTX 좌석 2개(A1, A2)를 예매하는 시스템. A1 좌석 예매 성공(T1), A2 좌석을 예매(T2)하려는데 0.1초 차이로 다른 사람이 A2를 채가서 실패했다. 고객은 "2자리가 연석이 아니면 안 산다"고 했으므로, 전체 트랜잭션을 롤백해야 한다.
**판단 및 보상 트랜잭션 설계**:
1. **보상 조건 트리거**: T2(A2 예매)가 실패하면, 예약 관리 시스템(Saga Orchestrator)은 T1(A1 예매)에 대해 "예매 취소(C1)" 이벤트를 발행한다.
2. **보상 로직 (C1) 실행**: 좌석 서비스는 A1을 다시 '빈자리'로 상태를 업데이트(`Status: AVAILABLE`)한다.
3. **격리성 이상(Anomaly) 방어**: 이 시스템의 가장 큰 문제(Lost Update)는 A1이 예약됐다가 취소되는 수 초의 시간 동안, 다른 사용자가 앱에서 A1이 매진된 것으로 보고 표를 포기할 수 있다는 점이다. 
이를 막기 위해 예매 시 상태를 바로 '확정(RESERVED)'으로 바꾸지 않고, **임시 상태인 '진행 중(PENDING)'**으로 둔다 (시맨틱 락). 만약 보상이 발생하면 PENDING을 지워버리고, 모두 성공하면 그때 RESERVED로 바꾼다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **보상 설계의 복잡도 지옥 (Saga Hell)**: 마이크로서비스가 5개를 넘어가면 끔찍한 일이 발생한다. (T1, T2, T3, T4, T5). T5에서 에러가 나면 C4, C3, C2, C1의 보상 로직을 모두 짜야 한다. 만약 C3에서 예상치 못한 NullPointerException이 터지면 시스템은 미아 상태가 된다.
**설계 원칙**: "분산 트랜잭션 사슬을 최대한 짧게(최대 2~3개) 유지하라." 강하게 결합되어 자주 롤백되는 기능들은 그냥 하나의 파드(Pod)와 하나의 DB로 묶어서 로컬 트랜잭션에 맡기는 것이 백배 천배 안전하다. 분산 트랜잭션은 꼭 필요한 도메인 간의 결합에서만 최소한으로 써야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **시스템 가용성 유지**: 락(Lock)을 전혀 쓰지 않으므로, 이커머스 플래시 세일(선착순 특가) 이벤트 시 DB 병목 없이 모놀리식 대비 **수십 배의 주문 트래픽(TPS)** 처리 가능.
- **감사 및 추적성(Auditability)**: 모든 에러 상황과 보상(환불) 이력이 취소 트랜잭션으로 명확히 남으므로, 금융 및 회계 감사 시 데이터 조작 의심을 원천 차단.

### 미래 전망 및 진화 방향
보상 트랜잭션의 구현을 개발자의 코딩에만 맡기면 버그로 인한 대형 사고가 잦다. 최근에는 이를 인프라나 프레임워크 레벨에서 돕는 기술이 뜨고 있다. 이벤트 자체가 진실의 원천이 되는 **Event Sourcing (이벤트 소싱)** 구조에서는 이전 이벤트를 취소하는 반대 이벤트를 발행하는 것으로 보상이 자연스럽게 완료된다. 또한 **AWS Step Functions**나 **Temporal** 같은 워크플로우 엔진이 각 상태 전이와 실패 시의 보상 로직(C1, C2) 트리거를 시각적, 시스템적으로 100% 보장해 주어 보상 트랜잭션의 안정성을 극대화하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **보상 트랜잭션이 뭔가요?**: 일기장에 글씨를 볼펜으로 잘못 썼을 때, 지우개로 지울 수 없으니까 빨간펜으로 두 줄을 쫙 긋고 "이거 취소!"라고 새로 적어 넣는 거예요.
2. **왜 지우개로 안 지우나요?**: 여러 친구들이 릴레이로 편지를 쓰고 있어서, 내 마음대로 지워버리면 다른 친구들이 헷갈려서 전체 이야기가 망가지거든요.
3. **가장 조심할 점은 뭔가요?**: "취소!"라고 적는 일은 절대로 실패하면 안 돼요. 만약 펜이 안 나와서 취소라고 못 적으면 편지 내용이 완전히 꼬여버리니까, 나오는지 꼭 두세 번 꾹꾹 눌러써야 한답니다! (멱등성 방어)
