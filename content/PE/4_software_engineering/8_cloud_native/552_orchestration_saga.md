# 552. 오케스트레이션 사가 (Orchestration Saga) - 중앙 통제기가 흐름 제어

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로서비스(MSA)의 분산 트랜잭션을 처리하는 사가(Saga) 패턴 중 하나로, 중앙에 거대한 지휘자(Orchestrator) 컴포넌트를 두고 이 지휘자가 각 마이크로서비스에게 "지금 결제해!", "실패했으니 재고 취소해!"라고 직접 명령을 내리는 중앙 통제형 아키텍처이다.
> 2. **가치**: 트랜잭션의 전체 흐름(비즈니스 워크플로우)과 현재 상태가 중앙 지휘자에 100% 투명하게 기록되므로, 시스템의 복잡도를 한 곳으로 모아 이해와 모니터링, 디버깅을 극도로 쉽게 만들어 준다.
> 3. **융합**: 보상 트랜잭션(Compensating Transaction) 개념과 결합하며, 개발자가 직접 State Machine(상태 머신)을 코딩하거나, AWS Step Functions, Temporal, Netflix Conductor와 같은 클라우드 네이티브 워크플로우 엔진과 융합되어 구현된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
오케스트레이션 사가(Orchestration Saga)는 분산 시스템에서 여러 서비스에 걸쳐 있는 비즈니스 트랜잭션을 일관되게 완료하기 위한 패턴이다. 이 패턴의 핵심은 **Saga Orchestrator(사가 오케스트레이터)**라는 중앙 통제기를 두는 것이다. 오케스트레이터는 전체 트랜잭션이 성공하려면 어떤 서비스들이 어떤 순서로 호출되어야 하는지(상태 머신)를 알고 있으며, 각 서비스에 동기/비동기적으로 명령(Command)을 보내고 그 결과(Reply)를 받아 다음 스텝을 결정한다.

### 💡 비유
이 방식은 **"오케스트라 교향악단의 지휘자"**와 같다.
(바이올린=결제 서비스, 첼로=재고 서비스, 팀파니=배송 서비스)
단원들은 전체 악보의 흐름을 알 필요가 없다. 오직 지휘자(Orchestrator)만이 전체 악보(비즈니스 로직)를 꿰뚫고 있다. 지휘자가 지휘봉을 바이올린으로 향하며 큐 사인을 주면 바이올린이 연주(결제)를 한다. 바이올린이 끝나면 지휘자는 첼로를 가리킨다. 만약 첼로의 줄이 끊어지는 사고(에러)가 나면, 지휘자는 즉시 연주를 멈추고 다른 단원들에게 "처음부터 다시 잔잔하게(보상 트랜잭션) 연주해!"라고 중앙에서 명령을 내린다.

### 등장 배경 및 발전 과정

#### 1. 코레오그래피 사가(Choreography Saga)의 복잡성 지옥
MSA 초기에는 중앙 통제를 없애고 각 서비스가 이벤트를 쏘고(Publish) 알아서 반응(Subscribe)하는 코레오그래피 방식이 유행했다. 하지만 서비스가 5~10개를 넘어가자 "도대체 내 주문이 지금 어디서 멈춰 있는 거지?"를 파악할 수 없는 '이벤트 핀볼(Event Pinball)' 현상이 터지며 디버깅이 불가능해졌다.

#### 2. 중앙 통제의 화려한 부활
결합도를 낮추는 것도 좋지만 비즈니스 흐름을 잃어버리는 것은 더 큰 재앙이었다. 개발자들은 "트랜잭션의 상태와 흐름만 관리하는 별도의 똑똑한 관리자(State Machine)를 두자"고 깨달았다. 이에 따라 캠운다(Camunda), 넷플릭스 컨덕터(Netflix Conductor) 같은 강력한 오케스트레이터 미들웨어가 탄생하며 엔터프라이즈 환경의 분산 트랜잭션 표준으로 자리 잡았다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (표)

| 요소명 | 상세 역할 | 작동 메커니즘 | 관련 도구 / 기술 | 비유 |
|--------|----------|-------------------|------------------|------|
| **Saga Orchestrator** | 중앙 통제기 (지휘자) | 전체 비즈니스 흐름(State Machine)을 메모리/DB에 쥐고, 각 단계별로 명령 메시지를 발행하고 응답을 기다림 | AWS Step Functions, Temporal, Camunda | 오케스트라 지휘자 |
| **Command (명령)** | 행위 지시 메시지 | 지휘자가 특정 서비스에게 "너 이거 해"라고 구체적인 액션을 지시 (의도적 결합) | Request Channel (Kafka Topic) | 지휘봉 큐 사인 |
| **Reply (응답)** | 결과 보고 메시지 | 명령을 받은 서비스가 일(로컬 트랜잭션)을 마치고 지휘자에게 "성공했어" 또는 "실패했어"라고 보고함 | Reply Channel | 연주 완료 사인 |
| **State Machine** | 상태 추적 장부 | 지휘자가 보관하는 전이도. (예: 결제 완료 -> 배송 지시 상태로 전이) 실패 시 롤백 경로를 찾아감 | RDBMS 로컬 테이블 | 지휘자의 악보 |
| **Participant Svc** | 개별 마이크로서비스 | 자신이 무슨 거대한 프로세스의 일부인지 모름. 그냥 들어오는 명령만 수행하고 결과를 리턴할 뿐 | Order, Pay, Inventory Svc | 연주 단원 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ Orchestration Saga: Happy Path & Compensation Path ]
================================================================================

  [ Saga Orchestrator (Order Service 내부에 존재한다고 가정) ]
   < State Machine: Pending -> Pay_Req -> Inv_Req -> (Error!) -> Pay_Cancel_Req >
          │
          │ 1. Command: "결제해라!" (to Payment Channel)
          ▼
   ┌─────────────┐
   │ Payment Svc │──(로컬 트랜잭션: 잔액 차감 성공)
   └─────────────┘
          │ 2. Reply: "결제 성공!"
          ▼
  [ Saga Orchestrator ] (상태 업데이트: Pay_Req_Done)
          │
          │ 3. Command: "재고 차감해라!" (to Inventory Channel)
          ▼
   ┌──────────────┐
   │ Inventory Svc│──(로컬 트랜잭션: 재고 부족 에러 발생!!)
   └──────────────┘
          │ 4. Reply: "실패! 재고 없음!"
          ▼
  [ Saga Orchestrator ] (상태 업데이트: Error_Detected)
          │ (오케스트레이터가 악보를 거꾸로 읽기 시작함 - 보상 트랜잭션)
          │
          │ 5. Command: "결제 취소해라!" (to Payment Compensation Channel)
          ▼
   ┌─────────────┐
   │ Payment Svc │──(로컬 트랜잭션: 환불 로직 실행)
   └─────────────┘
          │ 6. Reply: "환불 완료!"
          ▼
  [ Saga Orchestrator ] (상태 업데이트: Saga_Cancelled) => 클라이언트에게 최종 에러 반환
```

### 심층 동작 원리

#### ① 명시적 상태 머신 (Explicit State Machine)
오케스트레이터는 트랜잭션을 시작할 때 자신의 DB에 `Saga_ID: 101, Status: STARTED`라는 레코드를 기록한다. 단계가 지날 때마다 상태를 변경한다. 만약 진행 도중 오케스트레이터 파드(서버)가 정전으로 죽더라도, 재부팅되면 DB의 `Status`를 읽어 "아, 결제까지 성공하고 재고 확인하다 뻗었네" 하고 **정확히 중단된 지점부터 다시 스케줄링을 시작(Resilience)**할 수 있다.

#### ② 멍청한 참여자 (Dumb Participants)
결제 서비스(Participant)는 '배달앱 주문' 프로세스의 일부인지, '쇼핑몰 주문' 프로세스의 일부인지 알 필요가 전혀 없다. 그저 "결제해라"라는 Command 큐가 들어오면 돈을 빼고 "완료" 응답만 주면 된다. 따라서 **참여자 서비스 간의 결합도(Coupling)는 0%**가 되며, 다른 시스템에서도 결제 서비스를 재사용하기 매우 쉽다.

## Ⅲ. 융합 비교 및 다각도 분석

### 사가 패턴 양대 산맥: 오케스트레이션 vs 코레오그래피

| 비교 지표 | 오케스트레이션 사가 (Orchestration) | 코레오그래피 사가 (Choreography) |
|-----------|-------------------------------------|----------------------------------|
| **통제 방식** | 중앙 통제 (지휘자 존재) | 각자도생 (개별 서비스가 알아서 반응) |
| **통신 방식** | Command (명령) & Reply (응답) | Event Publish & Subscribe |
| **비즈니스 가시성**| **최상 (악보를 보면 한눈에 흐름 파악 가능)** | 최악 (코드를 전부 까봐야 전체 흐름을 알 수 있음) |
| **의존성 (Coupling)**| 오케스트레이터가 모든 서비스의 API(명령어)를 알아야 함 (중앙 결합도 상승) | 각 서비스는 서로를 전혀 몰라도 됨 (완벽한 디커플링) |
| **단일 장애점(SPOF)**| 오케스트레이터가 죽으면 모든 프로세스 마비 | 브로커만 살아있으면 중앙 장애점 없음 |
| **적용 권장 환경**| **트랜잭션 스텝이 4~5단계를 넘어가는 복잡한 환경** | 단계가 2~3개인 아주 단순한 워크플로우 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대형 통신사에서 "신규 휴대폰 개통" 시스템을 구축한다. [고객 신용조회 -> 단말기 재고 할당 -> 요금제 등록 -> 개통망 쏘기 -> 택배 송장 발행] 무려 5단계를 거쳐야 한다. 각 단계는 짧게는 1초, 길게는 신용조사 외부 API 지연으로 1분 이상 걸릴 수도 있다.
**판단 및 아키텍처 전략 (Orchestration 도입)**:
1. **코레오그래피 배제**: 5단계나 되는 프로세스를 코레오그래피(이벤트 구독)로 짜면, 요금제 등록이 실패했을 때 "단말기 할당 취소" 이벤트를 누가 쏴야 하는지 책임소재가 불분명해지고 시스템이 스파게티가 된다.
2. **AWS Step Functions (또는 Temporal) 적용**: 인프라 단에서 제공하는 시각적 워크플로우 엔진인 Step Functions를 오케스트레이터로 도입한다. 
3. **재시도와 보상 로직 시각화**: JSON 형태로 전체 5단계를 정의하고, "요금제 등록 실패 시 -> 단말기 할당 보상 트랜잭션 호출"이라는 롤백 다이어그램을 코드가 아닌 다이어그램(설정)으로 그린다.
4. **결과**: 개통 프로세스 도중 10만 건이 동시에 멈춰있어도, 운영자는 대시보드를 통해 "3만 건은 신용조회 중, 2만 건은 에러로 보상(환불) 처리 중"이라는 상황을 1초 만에 100% 모니터링할 수 있는 엄청난 운영 가시성(Visibility)을 획득한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **God Service (신적 서비스) 안티패턴**: 오케스트레이터는 "트랜잭션의 상태와 흐름"만 통제해야 한다. 그런데 개발자들이 귀찮다고 "할인율 계산 로직", "재고 차감 로직" 같은 **핵심 비즈니스 코드까지 오케스트레이터 안에 우겨넣기 시작**하면, 오케스트레이터는 모든 역할을 혼자 다 하는 또 하나의 '거대한 모놀리식 깡패(Monolith God)'로 전락한다. 이는 MSA의 탈중앙화 철학을 완벽히 박살 내는 짓이다. 오케스트레이터는 오직 "누구에게 명령을 보낼지(Routing/State)"만 결정해야 할 뿐, 직접 비즈니스 데이터를 연산해서는 안 된다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **운영/디버깅 비용 절감**: 분산 트랜잭션이 실패했을 때, 로그를 찾으러 5개 서버를 헤맬 필요 없이 중앙 오케스트레이터 DB(State Machine 로그) 하나만 까보면 실패 원인 파악 시간을 **90% 이상 단축**할 수 있음.
- **개발 생산성**: 참여자(Participant) 마이크로서비스를 개발하는 팀들은 전체 로직을 몰라도 "A 명령이 오면 A-1을 한다"라는 단일 책임만 구현하면 되므로 개발 속도 대폭 향상.

### 미래 전망 및 진화 방향
과거에는 개발자가 직접 자바(Java) 코드로 상태 머신과 이벤트 발행/구독 채널을 짜느라 뼈를 깎았다(예: Eventuate Tram 프레임워크). 하지만 현대 클라우드 네이티브의 방향성은 코드가 아니라 인프라를 활용하는 것이다. **Temporal, Netflix Conductor**처럼 오케스트레이션 사가에 특화된 전용 클러스터 미들웨어가 대세로 자리 잡았으며, 향후에는 코딩 1줄 없이 마우스 드래그 앤 드롭만으로 수백 개의 마이크로서비스 사가(Saga) 트랜잭션 흐름을 완성하는 **Low-Code 워크플로우 시대**가 표준이 될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **오케스트레이션 사가란?**: 심부름을 할 때, 엄마(오케스트레이터)가 거실에 앉아서 형한테는 "우유 사와", 동생한테는 "빵 사와"라고 무전기로 하나씩 콕 짚어서 명령을 내리는 방식이에요.
2. **어떻게 진행되나요?**: 형이 "우유 샀어!"라고 무전 치면, 엄마가 다이어리에 체크표시(상태 저장)를 하고 "다음은 빵이야!"라고 진행 상황을 완벽하게 관리해요.
3. **만약 빵이 다 팔렸다면?**: 동생이 "빵 없어!" 하면, 엄마가 모든 걸 취소하기 위해 형한테 "빨리 가서 우유 다시 환불해와!(보상 트랜잭션)"라고 지시해서 상황을 한 번에 깨끗하게 정리해 준답니다.
