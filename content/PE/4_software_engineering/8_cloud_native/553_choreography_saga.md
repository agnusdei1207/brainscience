# 553. 코레오그래피 사가 (Choreography Saga) - 이벤트 구독 기반의 자율적 흐름

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로서비스(MSA)의 분산 트랜잭션을 처리하는 사가(Saga) 패턴 중 하나로, 중앙 지휘자 없이 각 서비스가 독자적으로 일을 마친 후 **이벤트를 발행(Publish)**하면, 관심 있는 다른 서비스가 그 이벤트를 **구독(Subscribe)**하여 알아서 후속 작업을 이어가는 극단적 탈중앙화 아키텍처이다.
> 2. **가치**: 중앙 통제기가 없으므로 단일 장애점(SPOF)이나 병목 현상이 발생하지 않으며, 서비스 간의 결합도(Coupling)가 0%에 수렴하여 새로운 마이크로서비스를 시스템에 끼워 넣거나 빼는 확장(Scale-out)이 극도로 자유롭다.
> 3. **융합**: 이벤트 기반 아키텍처(EDA, Event-Driven Architecture)의 철학을 분산 트랜잭션 영역에 그대로 이식한 것이며, Apache Kafka나 RabbitMQ 같은 고성능 메시지 브로커와 완벽하게 융합되어 거대한 비동기 데이터 파이프라인을 형성한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
코레오그래피 사가(Choreography Saga)는 중앙에서 "이거 해, 저거 해"라고 지시하는 오케스트레이터(Orchestrator)가 없다. 참여하는 마이크로서비스들은 메시지 큐(Message Broker)를 사이에 두고, 어떤 비즈니스 로직(로컬 트랜잭션)이 성공하거나 실패했다는 사실(Event)만 허공에 쏘아 올린다. 다른 서비스들은 이 이벤트를 듣고 있다가 "어? 주문이 들어왔네? 그럼 내 역할인 재고를 깎아야지" 하고 자발적으로 다음 스텝을 밟는다.

### 💡 비유
이 방식은 **"훈련된 댄서들의 무용(Choreography) 공연"**과 같다.
오케스트라는 지휘자만 바라보고 연주하지만, 무용(발레)은 지휘자가 없다. 무대에 10명의 무용수가 있다. 음악이 시작되면(트랜잭션 시작), A 무용수가 점프(이벤트 발생)를 한다. 그걸 본 B 무용수와 C 무용수는 "아, 점프를 뛰면 다음은 우리가 돌기를 할 차례지!" 하고 연습된 규칙(비즈니스 로직)에 따라 스스로 돌기를 수행한다. 만약 A가 점프하다 넘어지면(에러 이벤트), 나머지 무용수들은 당황하지 않고 넘어졌을 때를 대비한 백업 율동(보상 트랜잭션)을 스스로 추며 무대를 자연스럽게 마무리한다.

### 등장 배경 및 발전 과정

#### 1. 중앙 결합도의 폐해 (오케스트레이션의 단점)
오케스트레이션 사가는 통제하긴 쉬웠으나, 오케스트레이터 서비스 하나가 전체 MSA의 API 스펙을 모두 알아야 했다(강한 결합). 새로운 서비스(예: '주문 시 알림톡 보내기' 부서)가 추가될 때마다 중앙 오케스트레이터의 소스 코드를 또 수정하고 다시 배포해야 하는 **'중앙 집권의 병목'**이 발생했다.

#### 2. 이벤트 기반 아키텍처(EDA)의 부상
Kafka와 같은 이벤트 스트리밍 기술이 성숙하면서, "시스템 상태 변경을 무조건 이벤트로 쏘고 보자"는 철학이 대세가 되었다. 어차피 이벤트를 쏘고 있다면 굳이 중앙 통제기 없이도 이벤트의 연쇄 작용(Domino)만으로 거대한 트랜잭션을 일관성 있게 끝낼 수 있다는 코레오그래피 패턴이 주목받게 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (표)

| 요소명 | 상세 역할 | 작동 메커니즘 | 관련 기술 / 도구 | 비유 |
|--------|----------|-------------------|------------------|------|
| **Event (이벤트)** | 상태의 변경 사실 통보 | "OrderCreated", "PaymentFailed" 처럼 과거형으로 명명되며 불변(Immutable) 상태를 띰 | JSON Payload | 무용수의 특정 동작 |
| **Message Broker** | 이벤트를 나르는 신경망 | 서비스 간에 이벤트를 유실 없이(Persistence) 안전하게 전달하는 통로 | Apache Kafka, RabbitMQ | 공연장의 무대 |
| **Publisher (발행자)**| 자기 일을 끝내고 통보 | 로컬 트랜잭션(DB 커밋)을 마친 후 브로커 특정 토픽에 이벤트를 던짐(Fire and Forget) | MSA 서비스 (Kafka Producer) | 동작을 끝낸 무용수 |
| **Subscriber (구독자)**| 이벤트를 듣고 자기 일 시작 | 특정 토픽의 이벤트를 24시간 감시(Poll)하다가, 이벤트가 떨어지면 로컬 트랜잭션을 시작함 | MSA 서비스 (Kafka Consumer) | 다음 차례 무용수 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ Choreography Saga: Event Publish & Subscribe Flow ]
================================================================================

  [ Client ] ──(주문 요청)──> [ Order Service ] ──(DB Commit)
                                     │
                                     │ 1. Publish: "OrderCreated" Event
                                     ▼
  ════════════════════════════════════════════════════════════════════════ (Event Bus / Kafka)
            │ (Subscribe)                                 │ (Subscribe)
            ▼ 2. 결제 시작!                               ▼ 3. 포인트 적립 시작!
   ┌─────────────────┐                          ┌─────────────────┐
   │ Payment Service │ ──(로컬 결제 로직 수행)    │  Point Service  │ ──(포인트 +10 적립)
   └─────────────────┘                          └─────────────────┘
            │
            │ 4. Publish: "PaymentFailed" (잔액 부족 에러 발생!!)
            ▼
  ════════════════════════════════════════════════════════════════════════ (Event Bus)
            │ (Subscribe)                                 │ (Subscribe)
            ▼ 5. 보상 로직 실행                           ▼ 6. 보상 로직 실행
   ┌─────────────────┐                          ┌─────────────────┐
   │ Order Service   │ ──(주문 '취소'로 Update)   │  Point Service  │ ──(방금 준 포인트 -10 회수)
   └─────────────────┘                          └─────────────────┘
  * 주목할 점: 아무도 "명령(Command)"을 내리지 않는다. 
    Point Service는 PaymentFailed 이벤트를 듣고 "내가 알아서" 보상 트랜잭션을 쳐버린다.
    Order Svc는 Payment나 Point Svc가 세상에 존재하는지조차 모른다 (결합도 0%).
```

### 심층 동작 원리

#### ① 데이터베이스와 이벤트 발행의 원자성 (Transactional Outbox Pattern)
코레오그래피에서 가장 위험한 순간은 "DB에는 주문을 저장했는데, Kafka 서버가 0.1초 뻗어서 이벤트를 쏘지 못한 경우"이다. 이렇게 되면 전체 사가 체인이 끊어져 영원히 결제가 되지 않는다.
이를 막기 위해 **트랜잭셔널 아웃박스(Transactional Outbox)** 패턴을 쓴다. 비즈니스 테이블(`orders`)에 데이터를 INSERT할 때, 같은 로컬 트랜잭션(같은 DB 커넥션) 안에서 이벤트를 저장할 `outbox` 테이블에도 INSERT를 쳐버린다(원자성 100% 보장). 이후 백그라운드 스레드(CDC 도구, Debezium 등)가 `outbox` 테이블을 스캔하여 Kafka로 안전하게 밀어 넣는다.

#### ② 자기 주도적 보상 (Autonomous Compensation)
에러가 났을 때 중앙에서 "너 환불해!"라고 지시하지 않는다. 각 서비스는 자신이 참여했던 트랜잭션의 실패 이벤트(`X_Failed`)를 모두 구독해 둔다. 결제 실패 이벤트가 뜨면, 배송 서비스는 그걸 듣고 "어? 내가 아까 포장해 둔 배송 건의 결제가 취소됐네? 포장 다시 풀어야지"라고 자기 주도적으로 롤백(보상 트랜잭션)을 실행한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 사가 패턴 양대 산맥: 코레오그래피 vs 오케스트레이션

| 비교 지표 | 코레오그래피 사가 (Choreography) | 오케스트레이션 사가 (Orchestration) |
|-----------|----------------------------------|-------------------------------------|
| **결합도 (Coupling)**| **최상 (완벽한 디커플링)**. 서비스 간 서로의 존재를 모름 | 낮음~중간. 오케스트레이터가 타 서비스 API를 알아야 함 |
| **단일 장애점 (SPOF)**| **없음**. 이벤트 버스만 살아있으면 시스템 전체가 굴러감 | 중앙 오케스트레이터 서버가 죽으면 올스톱 |
| **흐름 파악 (가시성)**| **최악**. "내 주문이 어디서 에러 났지?" 찾으려면 수십 개 서비스 로그를 다 뒤져야 함 | 최상. 중앙 DB에 "배송에서 멈춤"이라고 예쁘게 적혀 있음 |
| **무한 루프 위험** | 높음 (이벤트 핀볼 현상. A->B->C->A 무한 이벤트 루프 발생 가능) | 낮음 (중앙에서 통제) |
| **최적 도입 환경** | **참여 서비스가 2~3개인 단순한 파이프라인** | 참여 서비스가 4개 이상인 복잡한 비즈니스 로직 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 스타트업에서 유저가 "회원가입"을 했을 때 벌어지는 연쇄 작업 3가지(`[웰컴 이메일 발송]`, `[추천인 캐시 1천 원 지급]`, `[고객 통계 DB 업데이트]`)를 MSA로 구축하려 한다. 회원가입 속도가 가장 빨라야 하며, 나중에 마케팅팀이 "카카오톡 알림 전송" 기능을 쉽게 추가할 수 있게 해달라고 요구했다.
**판단 및 코레오그래피 사가 도입**:
1. **오케스트레이터 배제**: 가입 프로세스는 너무 단순하다. 굳이 무거운 상태 머신(오케스트레이터)을 도입할 필요가 없다.
2. **이벤트 발행**: `User Service`는 DB에 가입자 정보를 넣고, 즉시 카프카에 `UserRegistered` 이벤트를 딱 1번만 쏜 뒤 클라이언트(앱)에 "가입 완료" 응답을 내려버린다. (응답 속도 0.05초 극대화).
3. **자율적 구독**: 메일, 캐시, 통계 서비스는 각자 `UserRegistered`를 구독하다가 알아서 메일을 보내고 캐시를 지급한다. 캐시 지급이 실패(보상)해도 메일 발송은 정상적으로 진행된다 (느슨한 결합).
4. **기능 확장(Plug-in)**: 1달 뒤 마케팅팀이 "카카오톡 알림" 기능을 요구하면, 기존 소스 코드는 단 1줄도 수정할 필요가 없다. 그냥 새로운 카카오톡 파드를 띄워서 `UserRegistered`를 똑같이 구독(Subscribe)만 하게 만들면 즉시 기능이 추가된다. (극강의 확장성).

### 주의사항 및 안티패턴 (Anti-patterns)
- **추적 불가의 늪 (Blind Spot)**: 코레오그래피의 가장 큰 재앙은 "장애가 났는데 아무도 모른다"는 것이다. A 서비스가 이벤트를 쐈는데, B 서비스가 코딩 버그로 무시하고 가만히 있으면, 전체 주문 상태는 영원히 공중에 붕 뜬 '유령 상태(Zombie)'가 된다. 이를 방지하기 위해 이벤트를 쏠 때 반드시 고유한 **상관관계 ID(Correlation ID, 예: TraceID)**를 헤더에 붙여 분산 트레이싱(Jaeger, Zipkin)을 통해 Kiali 화면에서 "이벤트가 어디서 막혔는지" 시각적으로 감시하는 체계를 반드시 함께 구축해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **애자일 배포 속도**: 타 팀(타 마이크로서비스)과의 API 규격 회의 없이, 이벤트 JSON 스키마만 맞추면 즉시 배포할 수 있어 기능 출시 기간(Time-to-Market) **30% 이상 단축**.
- **트래픽 버퍼링(Load Leveling)**: 앞단(주문)에 초당 1만 건이 몰려도, 뒷단(배송, 결제)은 카프카 브로커에 이벤트를 쌓아두고 자신들의 처리 속도(초당 100건)에 맞춰 천천히 일을 빼가므로 서버 다운(Downtime) 확률 **Zero**.

### 미래 전망 및 진화 방향
코레오그래피 사가는 마이크로서비스 궁극의 이상향(완전한 독립)이지만, 트러블슈팅의 지옥이라는 양날의 검이다. 최근에는 이 두 사가(Saga) 패턴의 장점을 결합한 **하이브리드(Hybrid) 사가 접근법**이 주류가 되고 있다. 
예를 들어, 핵심적이고 복잡한 흐름(결제->재고->배송)은 **오케스트레이션**으로 꽉 묶어서 중앙 통제 및 실패 관리를 명확히 하고, 부가적인 작업(메일 발송, 로그 수집, 쿠폰 지급)은 그 오케스트레이터가 쏘아 올린 이벤트를 곁눈질로 듣고 알아서 동작하는 **코레오그래피**로 느슨하게 붙이는 영리한 아키텍처 혼용이 현대 클라우드 네이티브 설계의 정답으로 굳어지고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **코레오그래피 사가란?**: 반장(오케스트레이터)이 "청소해! 불 꺼!"라고 명령하지 않고, 시계가 '5시'를 땡 치면(이벤트 발생) 아이들이 각자 맡은 구역을 알아서 조용히 청소하고 나가는 방식이에요.
2. **누가 어떻게 알아서 하죠?**: 청소 당번은 '빗자루 소리(이벤트)'가 나면 쓰레받기를 대고, 주번은 '쓰레기통이 비워진 걸(이벤트)' 보면 불을 끄고 문을 닫는 식으로 눈치껏 손발을 맞춰요.
3. **가장 좋은 점은 뭐예요?**: 반장이 아파서 결석해도 청소는 완벽하게 돌아가고, 갑자기 전학 온 친구가 걸레질을 돕겠다고 끼어들어도 원래 하던 일의 순서를 안 바꿔도 된답니다!
