# 555. 이벤트 소싱 (Event Sourcing) - 상태 변경 이력의 영속화

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스에 애플리케이션의 "현재 상태(Current State)"를 덮어쓰기(UPDATE/DELETE)하여 저장하는 전통적 방식을 버리고, 상태를 변화시킨 모든 "이벤트(Event)들의 발생 이력"을 순서대로 추가(Append-only)하여 저장하는 아키텍처 패턴이다.
> 2. **가치**: 데이터의 수정/삭제가 원천적으로 불가능하므로 100% 완벽한 감사 추적(Audit Trail)과 시점 복구(Time Travel)가 가능하며, 복잡한 동시성 제어(Locking)를 없애 극한의 쓰기(Write) 성능을 달성한다.
> 3. **융합**: 읽기(Read) 시마다 이벤트를 처음부터 다시 계산하는 오버헤드를 막기 위해 스냅샷(Snapshot) 기법과 결합하며, 필연적으로 조회 성능을 극대화하는 **CQRS(명령과 조회 책임 분리)** 패턴과 완벽하게 한 쌍으로 융합되어 사용된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
이벤트 소싱(Event Sourcing)은 데이터 모델링에 대한 근본적인 역발상이다. 은행 계좌의 "현재 잔고가 5만 원이다"라는 최종 결과(상태)를 RDBMS에 저장하지 않는다. 대신, "10만 원 입금됨", "3만 원 출금됨", "2만 원 출금됨"이라는 과거의 사실(Event)들을 이벤트 저장소(Event Store)에 차곡차곡 기록한다. 현재 잔고(5만 원)를 알고 싶으면, 저장된 이벤트들을 처음부터 끝까지 재생(Replay)하여 메모리에서 계산해 내는 방식이다.

### 💡 비유
이벤트 소싱은 **"바둑/체스의 기보(棋譜) 기록"**과 같다.
- **기존 CRUD (상태 저장)**: 경기장에 들어가서 바둑판의 최종 돌 위치만 사진으로 찰칵 찍어둔다. 지금 누가 이기고 있는지는 알 수 있지만, "어쩌다 이렇게 됐지?"나 "10수 전으로 되돌려보자"는 불가능하다.
- **이벤트 소싱**: 바둑판의 최종 상태를 외우지 않는다. 대신 첫수부터 "흑이 K-10에 두었다", "백이 L-9에 두었다"는 기보(이벤트)를 노트에 순서대로 꼼꼼히 적는다. 기보만 있으면 언제든지 바둑판(현재 상태)을 100% 똑같이 재현할 수 있고, 원한다면 10수 전의 상황으로 정확히 되돌아갈(Time Travel) 수도 있다.

### 등장 배경 및 발전 과정

#### 1. CRUD 모델의 데이터 손실 한계
기존 시스템은 `UPDATE`와 `DELETE`를 통해 데이터를 덮어썼다. 이 과정에서 "과거의 소중한 비즈니스 맥락(Context)"이 영원히 삭제되는 심각한 데이터 손실이 발생했다. "이 고객이 장바구니에 물건을 담았다가 지운 횟수" 같은 정보는 CRUD 환경에서는 버려진다. 금융권에서는 이를 막기 위해 이력(History) 테이블을 수없이 만들었으나, 메인 테이블과 이력 테이블 간의 동기화 코딩은 끔찍한 버그를 양산했다.

#### 2. 빅데이터/스트리밍 시대의 도래
저장 매체(디스크) 가격이 똥값이 되면서 "데이터를 굳이 지울 필요가 있는가?"라는 의문이 생겼다. 또한 MSA로 분산 트랜잭션의 난이도가 치솟자, 차라리 원본 데이터를 덮어쓰지 말고 카프카(Kafka) 같은 로그 저장소에 이벤트만 때려 박는(Append) 방식이 로킹(Locking)도 없고 가장 빠르며 일관성을 맞추기 쉽다는 사실이 증명되면서 차세대 아키텍처로 급부상했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (표)

| 요소명 | 상세 역할 | 작동 매커니즘 | 관련 기술 | 비유 |
|--------|----------|---------------|-----------|------|
| **Event (이벤트)** | 시스템에 발생한 비즈니스 사실 | 불변(Immutable), 과거형 동사(`OrderCreated`, `ItemAdded`)로 명명됨 | JSON / Protobuf | 체스 기보 한 줄 |
| **Event Store** | 이벤트의 유일한 진실의 원천(SSOT) | 이벤트를 발생 시간(Sequence) 순서대로만 추가(Append-only)하는 특수 DB | Kafka, EventStoreDB | 기보 기록장 |
| **Aggregate (애그리거트)** | 도메인 로직 처리 주체 | 들어온 명령(Command)을 검증하고 새로운 이벤트를 생성(Emit)하는 클래스 | DDD (도메인 주도 설계) | 바둑 기사 (규칙 심판) |
| **Replay (재생)** | 현재 상태 복원 과정 | Event Store에서 특정 ID의 모든 이벤트를 가져와 빈 객체에 순서대로 적용시켜 현재 상태를 만듦 | Fold / Reduce 함수 | 기보 보고 바둑알 다시 놓기 |
| **Snapshot (스냅샷)** | Replay 성능 최적화 | 이벤트가 너무 많을 경우를 대비해, 100번째 이벤트 시점의 '현재 상태'를 통째로 캐싱해 둠 | Redis, 주기적 백업 | 중간 세이브 파일 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ Event Sourcing & CQRS Integrated Architecture ]
================================================================================

  [ Client ] ── (1. Command: "장바구니에 아이폰 담기") ──> [ Command API ]
                                                              │
                                                              ▼
 ┌──────────────────────────────────────────────────────────────────────┐
 │ [ Event Store (Kafka / EventStoreDB) ] - Append Only                 │
 │                                                                      │
 │  Seq 1: CartCreated    { cartId: 101, user: "Alice" }                │
 │  Seq 2: ItemAdded      { cartId: 101, item: "MacBook", qty: 1 }      │
 │  Seq 3: ItemAdded      { cartId: 101, item: "iPhone", qty: 2 }   <-- (2. 새로 추가됨)
 │  Seq 4: ItemRemoved    { cartId: 101, item: "iPhone", qty: 1 }       │
 └──────────────────────────────────────────────────────────────────────┘
    │ (3. Publish Event: 이벤트가 Query 모델로 비동기 전파됨)
    ▼
  [ Projector / Event Handler ] (이벤트를 해석해서 현재 상태로 뭉개는 역할)
    │ (계산: MacBook 1개, iPhone 2-1 = 1개)
    ▼
 ┌──────────────────────────────────────────────────────────────────────┐
 │ [ Query DB (Read Model / MongoDB / Redis) ]                          │
 │                                                                      │
 │  Cart 101 Document:                                                  │
 │  {                                                                   │
 │     "user": "Alice",                                                 │
 │     "items": [ {"name":"MacBook", "qty": 1}, {"name":"iPhone", "qty": 1} ] 
 │  }                                                                   │
 └──────────────────────────────────────────────────────────────────────┘
    ▲
  [ Client ] <── (4. Query: "내 장바구니 보여줘") ── [ Query API ]
```

### 심층 동작 원리

#### ① 상태의 투영 (Projection / Fold)
이벤트 소싱에서는 데이터를 읽을 때, 저장된 이벤트를 접어가는(Folding) 과정을 거친다.
`State = Fold(Events)`
초기값 0에서 시작하여 `+100`, `-50`, `+20` 이라는 이벤트를 모두 합산(Fold)하여 `70`이라는 현재 상태(State)를 도출한다. 

#### ② 스냅샷 (Snapshotting)
은행 계좌가 10년이 지나 이벤트가 10만 개 쌓였다고 치자. 현재 잔고를 보려고 10만 개의 이벤트를 매번 Replay 하는 것은 CPU와 메모리의 심각한 낭비다. 이를 막기 위해 1,000번째 이벤트마다 그 시점의 잔고 상태를 '스냅샷'으로 저장한다. 다음에 조회할 때는 가장 최근 스냅샷(예: 99,000번째 스냅샷)을 휙 불러오고, 그 이후에 발생한 1,000개의 이벤트만 추가로 Replay하여 성능을 극대화한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 상태 유지 방식: CRUD vs Event Sourcing

| 비교 지표 | 전통적 CRUD (State-oriented) | 이벤트 소싱 (Event-oriented) |
|-----------|------------------------------|------------------------------|
| **저장 대상** | 최종적으로 연산이 끝난 **결과값(상태)** | 상태를 변화시킨 **원인(행위/의도)** |
| **디스크 작업** | UPDATE, DELETE 시 물리적 덮어쓰기 (락킹 심함)| **무조건 끝에 추가 (Append-only)** (락 0%) |
| **감사(Audit)**| 트리거(Trigger)나 별도 이력 테이블 관리 필수 | **이벤트 저장소 자체가 100% 완벽한 감사 로그** |
| **오류 복구** | 백업 데이터가 없으면 DB가 날아가면 끝 | 잘못된 로직이 배포되어도, 로직 패치 후 이벤트를 처음부터 다시 Replay하면 완벽히 복구됨 |
| **시간 여행** | 불가능 | 특정 타임스탬프까지만 Replay하면 그 시점의 상태를 100% 재현 (Time Travel) |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 한 유통 회사에서 "사용자 행동 기반 AI 추천 시스템"을 만들려고 한다. 기존 RDBMS 장바구니 테이블에는 '최종적으로 담겨있는 상품'만 남아있어, 유저가 어떤 물건을 담았다가 취소했는지(변심) 알 수가 없어 AI 학습 데이터가 턱없이 부족하다.
**판단 및 이벤트 소싱 도입**:
1. **아키텍처 전환**: 장바구니 도메인을 CRUD에서 **이벤트 소싱**으로 재설계한다. 
2. **도메인 이벤트 도출**: `ItemAdded`, `ItemRemoved`, `CartCheckedOut`, `CartAbandoned` 등의 이벤트를 정의하고 카프카에 영구 저장한다.
3. **CQRS 결합**: 
   - 사용자가 장바구니 화면을 볼 때는 Redis 기반의 Query DB(최종 상태)를 빠르게 보여준다.
   - AI 추천 모델은 이벤트 스토어(Kafka)에 쌓인 모든 `ItemRemoved` 이벤트를 쭉 끌어다(Replay) 학습하여 "이 유저는 A상품을 담았다가 5초 만에 빼는 변심 패턴이 있다"는 엄청난 비즈니스 인사이트를 도출해 낸다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **이벤트 스키마 진화(Schema Evolution)의 지옥**: 이벤트 소싱에서 이벤트는 절대 지우거나 수정할 수 없다(불변성). 그런데 1년 뒤에 비즈니스가 바뀌어 `ItemAdded` 이벤트에 `할인율` 필드가 추가되었다면? V1 이벤트(할인율 없음)와 V2 이벤트(할인율 있음)가 이벤트 저장소에 섞여 있게 된다. Replay 시 코드에서 두 가지 버전을 모두 파싱하고 분기 처리해야 하는 '버전 관리 지옥'이 펼쳐진다. 이를 막기 위해 이벤트는 가장 작은 단위로 원시적으로 설계해야 하며, **Upcaster(과거 이벤트를 최신 포맷으로 런타임 변환하는 중간 코드)** 계층을 반드시 구현해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **쓰기 성능(Write Throughput) 극대화**: 디스크의 랜덤 액세스(Random I/O)와 락을 제거하고 오직 순차 쓰기(Sequential I/O)만 수행하므로, RDBMS 대비 **10~100배 이상의 쓰기 성능 향상**.
- **장애 및 논리적 버그 복구 능력**: 비즈니스 로직 버그로 인해 1만 명의 포인트가 잘못 계산되었더라도, 코드를 고치고 이벤트를 처음부터 다시 프로젝션(Re-projection)하기만 하면 즉각 100% 정상 데이터로 복원됨.

### 미래 전망 및 진화 방향
과거에는 직접 구현하기가 너무 어려워 "학자들의 장난감"이라 불렸으나, 최근에는 **EventStoreDB**, **Axon Framework** 같이 이벤트 소싱의 스냅샷, 리플레이, 업캐스팅을 자동으로 해주는 전용 생태계가 성숙했다.
앞으로의 시스템은 데이터를 잃어버리는 CRUD의 야만적인 시대를 지나, 기업에서 일어난 모든 역사적 사실(Event)을 영구 보존하여 언제든 새로운 관점의 뷰(Query DB)로 데이터를 재가공해 내는 불변의 데이터 아키텍처 시대로 완벽히 진입할 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **이벤트 소싱이란?**: 수학 문제를 풀 때, 연습장에 계산한 과정을 다 지우고 마지막 '정답'만 딱 적어놓는 게 아니라, 내가 계산한 모든 수식(과정)을 차곡차곡 다 써놓는 방식이에요.
2. **왜 과정을 다 적나요?**: 나중에 선생님이 "이 답이 어떻게 나온 거니?" 물어봤을 때(감사 추적), 지우개로 지워버렸으면 대답을 못 하지만, 과정이 다 적혀있으면 처음부터 다시 쓱 훑어보며 완벽하게 설명할 수 있거든요.
3. **가장 좋은 점은 뭐예요?**: 중간에 더하기를 빼기로 잘못 계산한 걸 발견해도, 계산 과정이 다 남아있으니까 그 부분부터 다시 고쳐서 풀면 1초 만에 100% 정확한 답으로 다시 복구할 수 있답니다!
