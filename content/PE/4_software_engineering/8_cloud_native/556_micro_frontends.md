# 556. 마이크로 프론트엔드 (Micro Frontends)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 백엔드의 마이크로서비스(MSA) 철학을 프론트엔드 영역으로 확장하여, 하나의 거대한 모놀리식 웹 애플리케이션을 기능별로 쪼개어 독립적인 팀이 개발, 테스트, 배포할 수 있는 작은 단위의 앱들로 분할하는 아키텍처이다.
> 2. **가치**: 코드 베이스가 비대해질수록 발생하는 빌드/배포 지연과 팀 간의 의존성 충돌을 해결하며, 서비스의 특정 영역만 다른 기술 스택(React, Vue 등)으로 점진적으로 리팩토링할 수 있는 유연성을 제공한다.
> 3. **융합**: Webpack Module Federation, Web Components, Iframe 등 런타임 통합 기술과 융합되며, 마이크로서비스 백엔드와 완벽하게 짝을 이루어 진정한 의미의 End-to-End 수직 분할(Vertical Slicing) 팀 구조를 완성한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
마이크로 프론트엔드(Micro Frontends)는 현대적인 웹 애플리케이션을 하나의 거대한 덩어리가 아닌, 여러 개의 작은 기능 단위(예: 결제, 장바구니, 상품 상세)로 쪼개어 관리하는 방식이다. 각 조각은 독립된 저장소(Repo)에서 관리되고 독립적으로 배포되지만, 사용자가 브라우저에서 볼 때는 하나의 매끄러운 단일 서비스처럼 보이도록 통합된다.

### 💡 비유
마이크로 프론트엔드는 **"레고(Lego) 블록으로 만든 거대 성"**과 같다.
- **모놀리식 프론트엔드**: 성 전체가 하나의 거대한 통짜 석상이다. 성벽의 돌 하나를 바꾸려면 성 전체를 부수고 다시 조각해야 한다. (빌드 한 세월).
- **마이크로 프론트엔드**: 성을 수천 개의 레고 블록으로 조립한다. '성문' 블록을 만드는 팀과 '지붕' 블록을 만드는 팀이 따로 있다. 성문 팀이 성문을 최신 자동문으로 교체해서 성에 꽂아 넣어도, 지붕 팀은 자기가 하던 일을 멈출 필요가 없다. 완성된 성(웹 사이트)은 사용자 눈에 완벽한 하나로 보인다.

### 등장 배경 및 발전 과정

#### 1. 프론트엔드 모놀리스의 역설
백엔드는 MSA로 잘 쪼개놨는데, 프론트엔드는 여전히 수백 명의 개발자가 하나의 거대한 SPA(Single Page Application) 프로젝트에 달라붙어 개발하는 기현상이 발생했다. 누군가 공통 라이브러리 버전을 올리면 다른 팀의 코드가 깨지고, 코드 한 줄 고쳤는데 전체 사이트를 빌드하느라 30분이 걸리는 비효율이 극에 달했다.

#### 2. ThoughtWorks의 제안과 표준화
2016년 소프트웨어 컨설팅 기업 ThoughtWorks가 이 개념을 공식화했다. 초기에는 Iframe이나 서버 사이드 인클루드(SSI) 방식이 쓰였으나, 최근에는 브라우저 기술의 발전으로 Webpack 5의 **Module Federation**이 등장하며 진정한 런타임 통합의 시대가 열렸다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (표)

| 요소명 | 상세 역할 | 구현 방식 | 비유 |
|--------|----------|-----------|------|
| **Container (Host)**| 여러 마이크로 앱을 로드하고 배치하는 껍데기 앱. 공통 레이아웃 제공 | Webpack Module Federation Host | 레고 밑판 |
| **Micro App (Remote)**| 독립적인 비즈니스 기능을 가진 작은 앱 (결제, 리뷰 등) | 독립된 SPA 프로젝트 (React, Vue) | 레고 블록 조각 |
| **Orchestrator** | 런타임에 어떤 마이크로 앱을 어디에 띄울지 결정하는 로더 | Single-spa, SystemJS | 조립 지시서 |
| **Shared Lib** | 디자인 시스템(CSS)이나 공통 유틸리티 | Webpack Shared 옵션, NPM 패키지 | 레고 표준 규격 |

### 심층 동작 원리

#### ① 런타임 통합 (Runtime Integration)
빌드 타임(Build-time) 통합은 NPM 패키지 형태로 가져오는 것인데, 이는 모놀리스와 다를 바 없다. 마이크로 프론트엔드의 핵심은 **런타임 통합**이다. 브라우저가 실행되는 도중, 컨테이너 앱이 "저기 서버에 있는 `Review.js` 좀 가져와서 여기 빈칸에 그려줘!"라고 요청하여 실시간으로 합쳐진다. 이를 통해 특정 기능만 배포해도 사용자는 새로고침 없이(또는 앱 전체 업데이트 없이) 최신 기능을 보게 된다.

#### ② 스타일 및 스코프 격리 (Isolation)
서로 다른 팀이 만든 앱이 합쳐질 때 CSS 클래스명이 겹쳐 디자인이 깨지는 문제가 발생할 수 있다. 이를 막기 위해 **CSS Modules, CSS-in-JS(Styled Components)**, 또는 브라우저 표준 기술인 **Shadow DOM**을 사용하여 각 앱의 스타일이 외부로 새어나가지 않게 완벽히 격리한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 통합 방식(Integration)별 트레이드오프

| 방식 | 기술적 특징 | 장점 | 단점 (Trade-off) |
|------|-------------|------|------------------|
| **Iframe** | 브라우저 내부에 또 다른 창 띄움 | 완벽한 격리, 가장 쉬움 | SEO 불리, 반응형 디자인 어려움, 메모리 낭비 |
| **Web Components**| 브라우저 표준 Custom Element 사용 | 프레임워크에 종속되지 않음 (Agnostic) | 구현 난이도 높음, 브라우저 호환성 체크 필요 |
| **Module Federation**| **(표준)** Webpack 5 엔진 활용 | **성능 최상, 라이브러리 공유 가능** | Webpack 5 환경 강제, 설정 복잡 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 한 거대 커머스 포털의 프론트엔드 프로젝트. 현재 React 16 버전을 쓰고 있는데, 결제팀은 React 18의 신기능을 쓰고 싶어 하고, 고객센터 팀은 가벼운 Vue.js로 전환하고 싶어 한다. 하지만 모놀리스 구조라 라이브러리 하나만 바꿔도 전체 서비스가 깨질 위험이 있어 3년째 기술 부채만 쌓이고 있다.
**판단 및 리팩토링 전략 (Micro Frontends 도입)**:
1. **컨테이너화**: 기존 앱을 감싸는 껍데기(Host)를 만든다.
2. **점진적 추출 (Strangler Fig Pattern)**: 가장 독립적인 '리뷰' 페이지부터 별도 Repo로 분리하여 Module Federation으로 연결한다. 이 조각은 React 18로 만든다.
3. **이기종 융합**: 고객센터 팀은 Vue.js로 앱을 만들어 컨테이너의 특정 경로에 마운트 한다.
4. **결과**: 전체 사이트를 한꺼번에 개편하는 'Big Bang' 방식의 위험 없이, 팀별로 원하는 시점에 원하는 기술로 부분 배포를 수행한다. 배포 사고가 나도 해당 영역만 기능을 끄거나 롤백하면 되므로 가용성이 비약적으로 상승한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **과도한 라이브러리 중복 (Payload Bloat)**: A 앱은 React를 쓰고 B 앱도 React를 쓰는데, 공통 설정을 안 해주면 브라우저가 React 라이브러리를 두 번 다운로드한다. 이는 성능 재앙을 부른다. Module Federation의 `shared` 설정을 통해 **공통 라이브러리는 딱 한 번만 로드**되도록 정교하게 튜닝해야 한다.
- **나노 프론트엔드 (Nano Frontends)**: 버튼 하나, 텍스트 하나를 마이크로 앱으로 쪼개는 것은 과유불급이다. 관리 포인트만 늘어나고 성능은 떨어진다. **'독립적인 팀이 책임질 수 있는 비즈니스 경계(Domain)'** 단위로 쪼개는 것이 정답이다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **배포 빈도(Deployment Frequency)**: 전체 앱 빌드 대기 시간이 사라져, 하루 1회 배포에서 **팀별 수십 회 배포**로 속도 향상.
- **평균 복구 시간(MTTR)**: 장애 발생 시 전체 롤백이 아닌 문제 구역만 즉시 교체하여 장애 복구 시간 **80% 단축**.

### 미래 전망 및 진화 방향
마이크로 프론트엔드는 이제 '옵션'이 아닌 대규모 조직의 '표준'이 되고 있다. 특히 Vercel이나 Netlify 같은 클라우드 플랫폼이 마이크로 프론트엔드의 배포와 오케스트레이션을 하드웨어 레벨에서 지원하기 시작했다. 앞으로는 개발자가 인프라 설정을 고민할 필요 없이, 소스 코드 라벨링만으로 전 세계 브라우저에 기능 조각들이 실시간으로 조립되어 뿌려지는 **'Serverless Frontend Mesh'** 시대로 나아갈 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **마이크로 프론트엔드가 뭔가요?**: 아주 큰 도화지에 그림을 한 명의 화가가 다 그리는 게 아니라, 여러 명의 화가가 각자 작은 종이에 '나무', '강물', '집'을 따로 그려서 나중에 도화지에 풀로 착착 붙이는 거예요.
2. **왜 이렇게 그리나요?**: 나무를 그리던 화가가 실수로 물감을 쏟아도, 강물이나 집 그림은 멀쩡하잖아요. 실수한 나무 그림만 새로 그려서 붙이면 되니까 훨씬 안전하고 빨라요.
3. **가장 좋은 점은요?**: 나무 팀은 크레파스로 그리고 싶고, 강물 팀은 물감으로 그리고 싶을 때 서로 싸울 필요 없이 각자 제일 잘하는 도구를 써서 멋진 작품을 완성할 수 있답니다!
