# 557. 모듈 페더레이션 (Module Federation) (Webpack 5)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Webpack 5에서 도입된 플러그인으로, 별도로 빌드된 웹 애플리케이션들이 런타임(Runtime)에 서로의 모듈을 마치 로컬 코드처럼 공유하고 호출할 수 있게 해주는 **마이크로 프론트엔드(Micro Frontends)의 핵심 엔진**이다.
> 2. **가치**: 기존의 NPM 패키지 공유 방식이 가졌던 '빌드 타임 의존성'과 '중복 다운로드' 문제를 해결하며, 여러 앱이 공통 라이브러리(React, Lodash 등)를 딱 한 번만 로드하여 메모리 효율을 극대화하는 지능형 모듈 공유를 실현한다.
> 3. **융합**: 컨테이너(Host)와 마이크로 앱(Remote) 간의 동적 연결, 그리고 CI/CD 파이프라인의 수평적 확장을 가능케 하여, 초대형 웹 프로젝트의 '분산 개발 및 통합 배포' 아키텍처를 완성한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
모듈 페더레이션(Module Federation)은 "연합(Federation)"이라는 뜻 그대로, 여러 개의 독립적인 Webpack 빌드 결과물들이 하나의 거대한 생태계를 이루어 서로의 자원을 공유하는 기술이다. 개발자는 자기 프로젝트에 없는 코드를 마치 자기 것인 양 `import` 해서 쓸 수 있으며, Webpack이 런타임에 원격 서버에서 해당 자바스크립트 파일을 가져와 연결해 준다.

### 💡 비유
모듈 페더레이션은 **"글로벌 스트리밍 서비스(예: 넷플릭스)의 콘텐츠 공유"**와 같다.
- **과거 (NPM 패키지 방식)**: 영화 한 편을 보려면 비디오 테이프를 사서(패키지 설치) 내 가방에 넣고 다녀야 한다. 테이프가 업데이트되면 새로 사야 하고, 가방은 무거워진다.
- **모듈 페더레이션**: 내 TV(Host 앱)에는 영화가 한 편도 없지만, 넷플릭스 서버(Remote 앱)에 접속해서 "오징어 게임 보여줘!"라고 누르면 실시간으로 영상을 가져와 보여준다. 영화관(다른 Remote 앱)에서 팝콘 판매 로직을 가져와 우리 집 거실에 띄울 수도 있다. 콘텐츠가 업데이트되면 서버만 바꾸면 되고, 내 TV는 아무것도 설치할 필요가 없다.

### 등장 배경 및 발전 과정

#### 1. NPM 공유 방식의 한계 (Build-time Hell)
공통 컴포넌트를 NPM 라이브러리로 만들어 공유하면, 컴포넌트 코드 한 줄만 고쳐도 이를 쓰는 수십 개의 마이크로 앱들을 일일이 다시 빌드하고 배포해야 했다. 배포 순서가 꼬이면 버전 충돌로 사이트가 뻗어버리는 일이 잦았다.

#### 2. Zack Jackson의 혁신과 Webpack 5 통합
Webpack 관리자 중 한 명인 잭 잭슨(Zack Jackson)은 "왜 코드를 빌드할 때 합쳐야만 하는가? 실행될 때 합치면 안 될까?"라는 질문을 던졌고, 이를 Webpack 5의 코어 기능으로 박아 넣었다. 이로써 프론트엔드에서도 백엔드 마이크로서비스와 같은 '독립적인 실시간 통합'이 가능해졌다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (표)

| 요소명 | 상세 역할 | 핵심 설정 | 비유 |
|--------|----------|-----------|------|
| **Host (Container)** | 리모트 모듈을 불러와서 실행하는 주체 앱 | `remotes: { app1: "app1@url" }` | 스마트 TV |
| **Remote (Micro App)**| 자신의 모듈이나 컴포넌트를 외부에 노출하는 앱 | `exposes: { "./Button": "./src/Btn" }` | 방송국 (콘텐츠 제공자) |
| **Shared (공유)** | 여러 앱이 공통으로 사용하는 라이브러리 (버전 중복 방지) | `shared: ["react", "react-dom"]` | 표준 전압 / 콘센트 규격 |
| **Remote Entry** | 리모트 앱이 어떤 자원을 가지고 있는지 알려주는 매니페스트 파일 | `remoteEntry.js` | 편성표 |

```ascii
      └───────────────────────┘
```

### 심층 동작 원리

#### ① 런타임 오케스트레이션 (Runtime Orchestration)
모듈 페더레이션은 빌드 결과물에 `remoteEntry.js`라는 아주 작은 파일을 생성한다. 이 파일은 해당 앱이 어떤 컴포넌트를 노출하고 있고, 어떤 라이브러리를 쓰고 있는지에 대한 메타데이터를 담고 있다. Host 앱은 실행 시점에 이 파일을 읽어, 필요한 모듈만 Lazy-loading(지연 로딩) 방식으로 땡겨온다.

#### ② 지능형 라이브러리 공유 (Shared Dependencies)
가장 강력한 기능이다. Host와 Remote가 모두 `React 18`을 사용한다고 설정(`shared`)해두면, Webpack은 런타임에 이를 감지한다. Host가 이미 React를 로드했다면, Remote 앱이 뜰 때는 자신의 React를 다운로드하지 않고 Host의 React를 재사용한다. 만약 버전이 다르면(Semantic Versioning), Webpack은 각자에게 맞는 버전을 따로 받을지 아니면 가장 높은 버전을 같이 쓸지 스스로 결정한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 코드 공유 기술의 진화 비교

| 비교 지표 | NPM Package | Monorepo (Lerna 등) | **Module Federation** |
|-----------|-------------|----------------------|-----------------------|
| **통합 시점** | 빌드 타임 (Build-time) | 빌드 타임 (Build-time) | **런타임 (Runtime)** |
| **배포 의존성**| 높음 (수정 시 재빌드 필수) | 높음 (전체 빌드 필요 가능성) | **없음 (Remote만 배포하면 끝)** |
| **공유 범위** | 소스 코드 수준 | 로컬 파일 수준 | **컴파일된 자바스크립트 모듈** |
| **중복 방지** | 어려움 (Node_modules 비대) | 가능 (Hoisting) | **완벽함 (Shared Chunk)** |
| **유연성** | 낮음 | 중간 | **극상 (이기종 앱 실시간 결합)** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 한 글로벌 기업의 백오피스 시스템. 50개의 개별 관리 도구들이 모여있다. 각 도구는 독립적인 팀이 관리한다. 최근 '공통 네비게이션 바'의 로고가 바뀌었는데, 50개 팀이 각자 자기 앱을 빌드하고 배포하느라 2주가 걸렸고, 그중 5개 팀은 배포 실패로 옛날 로고가 나오는 사고가 터졌다.
**판단 및 모듈 페더레이션 도입 전략**:
1. **Design System Remote 생성**: 네비게이션 바, 푸터, 로고 등 공통 UI만 담당하는 `Header-Remote` 프로젝트를 하나 만든다.
2. **Expose 설정**: `exposes: { "./Navbar": "./src/Nav" }` 로 설정하고 배포한다.
3. **Host 통합**: 50개의 관리 도구(Host)는 빌드 시점에 코드를 가져오지 않고, 설정에 `remotes: { header: "header@https://cdn.com/remoteEntry.js" }` 규칙만 넣는다.
4. **결과**: 로고가 또 바뀌면? `Header-Remote` 팀이 단 한 번만 배포하면 끝이다. 50개의 서비스는 새로고침만 하면 즉시 바뀐 로고를 보게 된다. **배포 지옥(Deployment Hell)의 완벽한 탈출**이다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **Shared 버전 미관리 (Version Conflict)**: Shared 설정을 무분별하게 `eager: true`로 주거나 버전을 명시하지 않으면, 특정 마이크로 앱이 깨졌을 때 전체 사이트가 하얗게 변하는(White Screen) 전염병이 발생한다. 반드시 **에러 바운더리(Error Boundary)**를 사용하여 원격 앱이 죽었을 때 서비스 전체가 죽지 않도록 방어 코드를 짜야 한다.
- **순환 참조 (Circular Dependency)**: A 앱이 B를 부르고, B 앱이 다시 A를 부르는 구조는 런타임 무한 루프를 유발한다. 의존성 그래프를 단방향(Host -> Remote)으로 유지하는 원칙이 필수다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **배포 속도**: 수정된 부분만 배포하면 되므로 전체 시스템 통합 배포 시간을 **95% 이상 단축**.
- **초기 로딩 최적화**: 사용자가 들어가지 않은 메뉴의 코드는 아예 다운로드조차 하지 않으므로 초기 번들 사이즈 **70% 이상 감소**.

### 미래 전망 및 진화 방향
모듈 페더레이션은 단순한 기능을 넘어 **Server-side Rendering (SSR)** 영역으로 확장되고 있다. Next.js와 같은 프레임워크에서도 여러 서버가 렌더링한 HTML 조각을 실시간으로 합치는 'Streaming Module Federation'이 연구되고 있다. 미래의 웹 개발은 더 이상 '내 프로젝트'라는 경계가 무의미해지고, 전 세계에 흩어진 자바스크립트 모듈들을 런타임에 직조(Weaving)하여 화면을 구성하는 **'Web Mesh'** 시대로 진화할 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **모듈 페더레이션이 뭔가요?**: 친구들끼리 장난감 상자를 하나씩 가지고 놀다가, 친구 상자에 있는 멋진 로봇 다리(코드)를 내 로봇 몸통에 런타임에 쓱 끼워서 노는 마법 같은 기술이에요.
2. **왜 편리한가요?**: 예전에는 친구 다리를 빌리려면 공장에 가서 복사본을 만들어와야 했지만, 이제는 친구가 허락만 하면 즉시 빌려 쓸 수 있고 친구가 다리를 업그레이드하면 내 로봇 다리도 자동으로 바뀐답니다.
3. **가장 좋은 점은요?**: 무거운 장난감 상자를 다 들고 다닐 필요 없이, 지금 딱 필요한 부품만 친구들한테 빌려 쓰니까 몸이 아주 가벼워지고(성능 향상) 빠르게 움직일 수 있어요!
