# 562. 도커(Docker) 이미지 계층(Layer) 최소화 기법

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 도커 이미지가 빌드될 때마다 생성되는 읽기 전용의 **Copy-on-Write(CoW) 레이어** 숫자를 줄이고, 불필요한 임시 파일을 제거하여 최종 이미지의 크기를 극한으로 압축하는 최적화 전략이다.
> 2. **가치**: 이미지 크기가 작을수록 네트워크 전송 비용(Egress)이 줄어들고, 쿠버네티스 등 오케스트레이터에서 컨테이너를 배포하는 기동 시간(Pull Time)을 획기적으로 단축하여 시스템 가용성을 높인다.
> 3. **융합**: 멀티 스테이지 빌드(Multi-stage Build), 베이스 이미지 슬리밍(Distroless/Alpine), 그리고 커맨드 체이닝(`&&`) 기술이 융합되어 보안성과 성능을 동시에 잡는 클라우드 네이티브 배포의 표준을 제시한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
도커 이미지는 `Dockerfile`의 각 명령어(`RUN`, `COPY`, `ADD`)가 실행될 때마다 하나의 레이어가 쌓이는 스택 구조다. 레이어 최소화 기법은 이 구조를 이해하고, 중간 과정에서 발생하는 찌꺼기 데이터를 최종 이미지에 남기지 않도록 설계하는 기술이다.

### 💡 비유
도커 이미지 레이어는 **"포토샵(Photoshop)의 레이어"**와 같다.
- **비효율적 이미지**: 1번 레이어에 배경을 칠하고, 2번 레이어에 지우개로 다 지운다. 우리 눈엔 빈 화면이지만, 포토샵 파일 용량은 배경 데이터만큼 무겁다.
- **최적화된 이미지**: 배경을 칠하지 않거나, 칠하자마자 그 레이어 안에서 바로 수정한다. 최종 결과물에는 오직 꼭 필요한 '선'과 '색'만 남겨 파일 용량을 가볍게 만드는 것이다.

### 등장 배경 및 발전 과정

#### 1. 대용량 이미지의 폐해
초기 개발자들은 편리함 때문에 Ubuntu 통째 이미지를 베이스로 썼다. 결과적으로 단순한 'Hello World' 앱 하나가 1GB가 넘는 사태가 벌어졌고, 배포 한 번 할 때마다 네트워크가 마비되는 고통을 겪었다.

#### 2. 배포 속도가 곧 경쟁력
마이크로서비스(MSA) 시대에는 초 단위의 오토 스케일링이 중요하다. 이미지를 다운로드(Pull) 받는 데 1분이 걸린다면 트래픽 폭주에 대응할 수 없다. 이에 따라 '날씬한 이미지'를 만들기 위한 기법들이 도커 생태계의 핵심 기술로 자리 잡았다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 레이어 생성 명령어와 최적화 대상 (표)

| 명령어 유형 | 레이어 생성 여부 | 최적화 전략 | 비유 |
|-------------|------------------|-------------|------|
| **`FROM`** | Yes (Base) | **Alpine**이나 **Distroless** 같은 최소형 베이스 이미지 선택 | 집 터 고르기 |
| **`RUN`** | **Yes** | 여러 개의 명령어를 `&&`로 묶고 마지막에 캐시 삭제 | 한 번에 장보기 |
| **`COPY / ADD`**| **Yes** | 필요한 파일만 콕 찝어 복사 (`.dockerignore` 활용) | 꼭 필요한 짐만 싸기 |
| **`CMD / ENV`** | No (Meta only) | 가독성을 위해 자유롭게 사용 | 이름표 달기 |

### 정교한 구조 다이어그램 (멀티 스테이지 빌드 원리)

```ascii
================================================================================
[ Multi-stage Build: The Ultimate Layer Minimization ]
================================================================================

 [ Stage 1: Build Stage (Heavy) ]        [ Stage 2: Production Stage (Slim) ]
 ┌───────────────────────────┐           ┌──────────────────────────────┐
 │ FROM golang:1.21-alpine   │           │ FROM scratch (or alpine)     │
 ├───────────────────────────┤           ├──────────────────────────────┤
 │ RUN apk add git build-base│           │                              │
 │ COPY . .                  │           │ ⭐️ COPY --from=build ⭐️      │
 │ RUN go build -o myapp     │──────────▶│ /app/myapp  .                │
 └───────────────────────────┘           └──────────────────────────────┘
  (용량: 800MB - 컴파일러 포함)             (용량: 10MB - 실행 파일만 존재)

 * 핵심: 1단계에서 사용한 무거운 도구(Git, GCC 등)는 다 버리고, 
   최종 이미지에는 '결과물'만 쏙 옮겨 담는다.
```

### 심층 동작 원리

#### ① 레이어 통합 (Command Chaining)
`RUN apt-get update`와 `RUN apt-get install -y vim`을 따로 쓰면 두 개의 레이어가 생긴다. 1번 레이어의 패키지 목록 캐시 데이터는 2번 레이어가 실행되어도 사라지지 않고 이미지 용량을 차지한다.
- **최적화**: `RUN apt-get update && apt-get install -y vim && rm -rf /var/lib/apt/lists/*` 처럼 한 줄로 묶으면, 캐시가 생성되자마자 삭제되므로 최종 이미지 레이어에는 캐시 데이터가 남지 않는다.

#### ② `.dockerignore`를 통한 다이어트
`.git`, `node_modules`, `temp` 폴더 등 빌드에 필요 없는 파일이 이미지에 포함되면 용량이 커질 뿐만 아니라 보안상 위험하다. `.dockerignore` 파일을 작성하여 이미지 빌드 시 아예 무시되도록 차단한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 베이스 이미지 선택지에 따른 용량 비교

| 이미지 종류 | 용량 (약) | 특징 | 추천 환경 |
|-------------|-----------|------|-----------|
| **Ubuntu / Debian** | 70MB ~ 150MB | 친숙함, 모든 도구 내장 | 로컬 개발, 디버깅용 |
| **Alpine Linux** | **5MB** | 초경량, musl libc 사용 | 일반적인 배포 표준 |
| **Distroless** | 2MB ~ 20MB | **Shell조차 없음**, 실행 런타임만 존재 | 보안이 극도로 중요한 금융/공공 |
| **Scratch** | **0B** | 아무것도 없는 빈 공간 | Go/Rust 등 정적 컴파일 언어 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: Java Spring Boot 앱을 배포하는데 도커 이미지 크기가 600MB다. 이로 인해 사내망 대역폭이 포화 상태이고, 배포 속도가 너무 느려 롤백(Rollback) 시에도 5분 이상 소요된다.
**판단 및 다이어트 전략**:
1. **멀티 스테이지 빌드 도입**: Maven/Gradle 빌드용 이미지와 실행용 JRE 이미지를 분리한다. (소스 코드와 빌드 툴 제거).
2. **JRE Slim 버전 사용**: 전체 JDK 대신 실행에 꼭 필요한 모듈만 담은 `openjdk:17-jre-slim` 또는 `Alpine` 기반을 사용한다.
3. **레이어 순서 최적화**: 자주 바뀌는 소스 코드(`COPY . .`)는 아래로, 잘 안 바뀌는 의존성 설치(`COPY pom.xml`)는 위로 배치하여 도커 빌드 캐시 효율을 극대화한다.
4. **결과**: 이미지 크기를 600MB에서 **120MB**로 축소. 배포 및 롤백 시간이 1분 이내로 단축되어 운영 안정성이 비약적으로 향상된다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **과도한 레이어 통합의 역설**: 모든 명령어를 억지로 한 줄로 합치면 가독성이 떨어지고, 도커의 가장 큰 장점인 **'빌드 캐시'**를 전혀 못 쓰게 된다. 코드 한 줄 고쳤는데 전체 라이브러리를 다시 받는 비효율이 발생한다. 따라서 **'자주 변하는 레이어'**와 **'거의 변하지 않는 레이어'**를 명확히 구분하여 적절히 쪼개는 밸런스가 중요하다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **네트워크 비용 절감**: 이미지 사이즈 80% 감소 시 데이터 전송량 및 저장소 비용 **80% 절감**.
- **보안성 향상**: 불필요한 도구(Shell, Curl, Package Manager)를 제거함으로써 해커가 침투했을 때 활용할 수 있는 수단(Attack Surface)을 원천 차단.

### 미래 전망 및 진화 방향
최근에는 도커 이미지 빌드를 넘어, 소스 코드만 던지면 알아서 최적화된 이미지를 생성해 주는 **Cloud Native Buildpacks (CNB)** 기술이 대두되고 있다. 또한, 이미지 레이어를 다운로드하면서 동시에 압축을 풀어 실행하는 **'zstd 압축'** 기법과, 필요한 레이어만 런타임에 즉시 땡겨오는 **'Lazy Pulling (stargz)'** 기술이 발전하며 이미지 크기에 대한 제약이 사라지는 시대로 진화하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **도커 레이어가 뭔가요?**: 샌드위치를 만들 때 빵 놓고, 햄 놓고, 치즈 놓는 것처럼 차곡차곡 쌓아 올린 재료 층이에요.
2. **왜 최소화해야 하나요?**: 샌드위치에 먹지도 않을 포장지나 비닐(임시 파일)을 중간에 끼워 넣으면 샌드위치가 너무 뚱뚱해지고 무거워서 친구한테 배달하기 힘들기 때문이에요.
3. **어떻게 줄이나요?**: 샌드위치를 다 만든 다음에 필요 없는 포장지는 쏙 빼고, 꼭 먹을 재료만 남겨서 아주 작고 맛있게 포장하는 비법이랍니다!
