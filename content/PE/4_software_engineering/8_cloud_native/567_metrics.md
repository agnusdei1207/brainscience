# 567. 메트릭 (Metrics) - Prometheus와 Grafana를 활용한 시계열 분석

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템의 상태를 나타내는 수치 데이터를 시간 순서대로 수집한 **시계열(Time-series) 데이터**로, 현재 시스템이 "얼마나 바쁜지", "에러가 얼마나 발생하는지"를 객관적인 숫자로 정량화하는 기술이다.
> 2. **가치**: 장애가 발생하기 전 징후를 미리 파악하고(Alerting), 과거 데이터를 기반으로 미래의 필요한 자원을 예측(Capacity Planning)함으로써 인프라 운영의 과학적 근거를 제공한다.
> 3. **융합**: 데이터를 수집/저장하는 **Prometheus**와 이를 화려한 차트로 보여주는 **Grafana**가 결합되어 현대 클라우드 네이티브 옵저버빌리티의 표준 대시보드 환경을 구성한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
메트릭(Metrics)은 개별 로그(Log)와 달리, 특정 시점의 시스템 상태를 하나의 '숫자'로 요약한 데이터다. 
- **Prometheus**: 시계열 DB(TSDB)를 내장한 오픈소스 메트릭 수집 시스템.
- **Grafana**: 다양한 소스(Prometheus, MySQL 등)의 데이터를 시각화하는 전문 대시보드 도구.

### 💡 비유
메트릭 분석 시스템은 **"자동차의 계기판과 블랙박스 통계"**와 같다.
- **메트릭 (계기판)**: 속도계, RPM, 연료 잔량. 운전자는 이 숫자들만 보고도 차가 잘 달리고 있는지, 곧 기름을 넣어야 할지(장애 예측) 알 수 있다.
- **Prometheus (블랙박스 통계)**: 매초마다 계기판 숫자를 기록하는 장치.
- **Grafana (네비게이션 화면)**: 기록된 숫자들을 보기 좋게 그래프로 그려서 "오늘 평균 시속은 60km였습니다"라고 요약해 보여주는 모니터다.

### 등장 배경 및 발전 과정

#### 1. 대규모 컨테이너 환경의 도래
서버가 10대일 때는 직접 접속해서 `top` 명령어로 CPU를 확인하면 됐다. 하지만 컨테이너가 수만 개가 되면 일일이 확인할 수 없다. 모든 컨테이너의 CPU, 메모리 상태를 한곳으로 모아줄 '중앙 집계 시스템'이 필수가 되었다.

#### 2. 구글 Borgmon의 후예, Prometheus
SoundCloud 팀이 구글의 내부 모니터링 시스템(Borgmon)을 모델로 만든 Prometheus가 CNCF의 졸업 프로젝트가 되면서, 쿠버네티스 생태계의 사실상 표준(De-facto standard) 모니터링 도구로 자리 잡았다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 메트릭의 4대 기본 유형 (Prometheus 기준) (표)

| 유형 (Type) | 상세 설명 | 특징 | 비유 |
|-------------|-----------|------|------|
| **Counter** | 0부터 시작해 계속 증가만 하는 값 | 누적 요청 수, 에러 총 횟수 | 자동차의 총 주행 거리계 |
| **Gauge** | 올랐다 내렸다 하는 현재 값 | 현재 CPU 사용률, 메모리 잔량 | 자동차의 현재 속도계 |
| **Histogram** | 관측값의 분포를 버킷(Bucket)에 담음 | 응답 속도 분포 (0~10ms가 몇 건인지) | 성적표의 점수대별 인원수 |
| **Summary** | 히스토그램과 비슷하지만 백분위(Quantile) 계산 | 정확한 상위 99% 지연 시간 계산 | 전교 1등부터 100등까지 순위 |

### 정교한 구조 다이어그램 (Prometheus Pull-based Flow)

```ascii
================================================================================
[ Prometheus & Grafana Monitoring Architecture ]
================================================================================

  [ 📦 Target: App A ]      [ 📦 Target: Node B ]      [ 📦 Target: DB C ]
  ( /metrics endpoint )     ( Node Exporter )          ( DB Exporter )
           │                          │                          │
           └──────────────┬───────────┴──────────────────────────┘
                          │ (1) ⭐️ Pull Metrics ⭐️ (HTTP)
                          ▼
  ┌──────────────────────────────────────────────────────────┐
  │  🔥 Prometheus Server                                    │
  │  - Retrieval (Scraper)                                   │
  │  - Time-series DB (Storage)                              │
  │  - PromQL (Query Engine)                                 │
  └──────────────────┬──────────────────────────┬────────────┘
                     │                          │
      (2) Evaluate   │                          │ (3) Visualize
          Rules      ▼                          ▼
  ┌────────────────────────┐      ┌────────────────────────────────┐
  │  🔔 AlertManager       │      │  🎨 Grafana Dashboard          │
  │  (Slack / Email 알람)   │      │  (Rich Graphs & Dashboards)    │
  └────────────────────────┘      └────────────────────────────────┘

 * 핵심 포인트: 프로메테우스가 직접 서버들을 방문하여 데이터를 '가져오는(Pull)' 방식. 
   서버가 수만 대라도 프로메테우스 설정만 바꾸면 되어 관리가 매우 유연함.
```

### 심층 동작 원리

#### ① Pull vs Push 매커니즘
대부분의 모니터링 도구는 서버가 중앙으로 데이터를 '쏘는(Push)' 방식이다. 하지만 Prometheus는 중앙 서버가 각 대상의 `/metrics` 주소에 접속해 데이터를 '긁어오는(Pull)' 방식을 쓴다. 
- **장점**: 모니터링 대상이 급증해도 중앙 서버가 감당할 만큼만 긁어올 수 있어 과부하를 막고, 대상 서버가 살아있는지(Health Check)를 즉시 알 수 있다.

#### ② PromQL (Prometheus Query Language)
메트릭 데이터를 요리하는 강력한 언어다. 
- 예: `rate(http_requests_total[5m])` -> 최근 5분간 초당 요청 횟수의 변화율(Rate)을 계산해 줘.
이 쿼리를 통해 단순한 숫자 뭉치를 "지금 서비스가 폭주 중이다"라는 의미 있는 정보로 바꾼다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 신규 쇼핑몰 오픈 후, 마케팅 이벤트만 하면 서버가 뻗는다. 로그를 봐도 에러는 없는데 그냥 응답이 안 온다. 원인을 찾기 위해 모니터링을 강화하려 한다.
**판단 및 메트릭 구축 전략**:
1. **Golden Signals 수집**: SRE의 핵심 지표인 **지연 시간(Latency), 트래픽(Traffic), 에러(Errors), 포화도(Saturation)**를 메트릭으로 정의한다.
2. **Node Exporter & Kube-state-metrics**: 인프라 자원(CPU/MEM)뿐만 아니라 쿠버네티스의 상태 정보를 모두 Prometheus로 모은다.
3. **Grafana 대시보드 통합**: 비즈니스 지표(주문 건수)와 인프라 지표를 한 화면에 겹쳐서 본다.
4. **결과**: "주문 건수가 초당 100건을 넘는 순간, DB 커넥션 포화도가 95%를 치며 지연 시간이 5초로 늘어난다"는 정확한 병목 지점을 1초 만에 파악하여 DB 스케일 업을 단행한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **High Cardinality의 늪**: 메트릭 레이블(Label)에 '사용자 ID'나 '세션 토큰'처럼 값이 무수히 많은 것을 넣으면 안 된다. Prometheus는 레이블 조합마다 별도의 시계열 데이터를 만드는데, 조합이 수백만 개가 되면 DB 메모리가 터져버린다. 레이블은 반드시 **'서비스명', '리전', 'HTTP 상태코드'** 같이 가짓수가 정해진 데이터에만 써야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **장애 인지 시간(MTTD)**: 사람이 발견하기 전 시스템이 먼저 알람을 줌으로써 인지 시간 **80% 이상 단축**.
- **인프라 비용 최적화**: 사용률이 낮은 서버를 메트릭으로 찾아내어 자원을 회수함으로써 **비용 20~30% 절감**.

### 미래 전망 및 진화 방향
메트릭 시스템은 이제 **'Thanos'**나 **'Cortex'** 같은 프로젝트를 통해 수조 개의 데이터를 무제한 기간 보관할 수 있는 클러스터 구조로 진화했다. 또한, 최근에는 **eBPF** 기술과 결합하여 애플리케이션 코드를 한 줄도 안 고쳐도 커널 레벨에서 메트릭을 자동으로 뽑아내는 'Zero-instrumentation' 시대로 나아가고 있으며, AI가 메트릭의 이상 패턴(Anomaly)을 감지하는 지능형 모니터링이 대세가 될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **메트릭이 뭔가요?**: 내 몸이 얼마나 튼튼한지 숫자로 재보는 '체중계'나 '키 재기 판'과 같아요. 
2. **Prometheus와 Grafana는요?**: Prometheus는 매일 아침 내 키와 몸무게를 수첩에 적어두는 '기록 대장'이고, Grafana는 그 수첩을 보고 "와! 지난달보다 키가 2cm나 컸네?"라고 멋진 그래프로 그려주는 '화가 친구'예요.
3. **왜 좋은가요?**: 배가 아프기 전에 "몸무게가 갑자기 줄었어!"라고 그래프가 알려주니까, 미리미리 비타민을 먹어서 건강을 지킬 수 있게 도와준답니다!
