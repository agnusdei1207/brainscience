# 568. 로그 (Logs) - 분산 로그 수집과 ELK 스택

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 애플리케이션 실행 중 발생하는 특정 사건(Event)을 시간 순서대로 기록한 텍스트 데이터로, 시스템의 과거 행적을 추적하여 버그를 찾거나 보안 위협을 분석하는 **사후 분석의 결정적 증거**이다.
> 2. **가치**: 마이크로서비스(MSA)와 같이 서버가 수천 대로 분산된 환경에서, 각 서버에 흩어진 로그를 실시간으로 한곳에 모으고 인덱싱하여 수십억 건의 데이터 중에서도 원하는 정보를 1초 만에 검색할 수 있게 한다.
> 3. **융합**: 수집(Logstash/Fluentd), 저장 및 검색(Elasticsearch), 시각화(Kibana) 기술이 융합된 **ELK/EFK 스택**은 현대 인프라 옵저버빌리티의 가장 강력한 무기이다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
로그(Logs)는 시스템에서 일어난 '팩트(Fact)'의 기록이다. "사용자가 로그인을 시도함", "DB 연결이 실패함" 등 의미 있는 모든 변화를 타임스탬프와 함께 기록한다. 분산 로그 수집은 이 조각들을 하나의 거대한 '디지털 장부'로 합치는 과정이다.

### 💡 비유
로그 수집 시스템은 **"CSI 과학수사대의 증거 분석실"**과 같다.
- **로그 (지문/CCTV)**: 범죄 현장(서버) 곳곳에 남겨진 작은 단서들.
- **Logstash (증거 수집팀)**: 전국의 현장을 돌며 지문을 채취하고 사진을 찍어 본부로 보낸다.
- **Elasticsearch (중앙 증거 보관소)**: 수백만 개의 지문을 특징별로 분류해서 보관한다. 형사가 "빨간 모자 쓴 사람!"이라고 하면 바로 찾아준다.
- **Kibana (브리핑 화면)**: 수집된 증거를 지도로 그리거나 통계를 내어 수사 반장님께 보고하는 대형 스크린이다.

### 등장 배경 및 발전 과정

#### 1. "로그 보러 서버 들어가기"의 종말
과거에는 서버가 한두 대라 에러가 나면 직접 SSH로 들어가서 `tail -f` 명령어로 로그를 봤다. 서버가 1,000대가 넘는 지금, 이는 불가능한 일이 되었다.

#### 2. 비정형 데이터의 홍수
로그는 형식이 제각각이다. 어떤 건 텍스트고 어떤 건 JSON이다. 이 파편화된 데이터를 의미 있는 데이터로 정제(Parsing)하고 통합 저장할 수 있는 전문 솔루션(ELK 등)이 표준으로 자리 잡았다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 분산 로그 수집의 4단계 파이프라인 (표)

| 단계 | 역할 | 주요 도구 | 비유 |
|------|------|-----------|------|
| **1. Agent (Shipper)** | 서버 내 로그 파일을 읽어 전송 | Filebeat, Fluent-bit | 현장 요원 |
| **2. Aggregator** | 로그를 파싱하고 변환/필터링 | Logstash, Fluentd | 중앙 검문소 |
| **3. Storage** | 로그를 인덱싱하고 영구 저장 | **Elasticsearch**, Loki | 증거 창고 |
| **4. Analysis** | 로그 검색 및 대시보드 시각화 | **Kibana**, Grafana | 분석 상황판 |

### 정교한 구조 다이어그램 (ELK 스택 기반)

```ascii
================================================================================
[ Centralized Logging Architecture: ELK Stack ]
================================================================================

  [ 📦 App Node 1 ]      [ 📦 App Node 2 ]      [ 📦 App Node 3 ]
  ┌─────────────┐        ┌─────────────┐        ┌─────────────┐
  │ Log File    │        │ Log File    │        │ Log File    │
  ├─────────────┤        ├─────────────┤        ├─────────────┤
  │ 🐜 Filebeat │        │ 🐜 Filebeat │        │ 🐜 Filebeat │ (경량 수집기)
  └──────┬──────┘        └──────┬──────┘        └──────┬──────┘
         │                      │                      │
         └─────────────┬────────┴──────────────────────┘
                       │ (Log Shipping)
                       ▼
  ┌──────────────────────────────────────────────────────────┐
  │  ⚙️ Logstash (The Aggregator)                            │
  │  - Filter: grok { match => ... }                         │
  │  - Mutate: remove_field [ "secret" ]                     │
  └──────────────────────────┬───────────────────────────────┘
                             │
                             ▼
  ┌──────────────────────────────────────────────────────────┐
  │  🔍 Elasticsearch (Search Engine)                        │
  │  - Inverted Indexing                                     │

  └──────────────────────────┬───────────────────────────────┘
                             │
                             ▼
  ┌──────────────────────────────────────────────────────────┐
  │  🎨 Kibana (Visualization)                               │

  │  - Dashboard (Error Rates / Traffic)                     │
  └──────────────────────────────────────────────────────────┘
```

### 심층 동작 원리

#### ① 로그 정규화 (Parsing & Normalization)
로그는 단순 텍스트(`2024-03-05 ERROR 500 ...`)인 경우가 많다. Logstash의 **Grok** 필터는 정규표현식을 써서 이 문장을 `{ "timestamp": "...", "level": "ERROR", "code": 500 }`과 같은 JSON 구조로 바꾼다. 이렇게 정형화되어야만 나중에 "500 에러만 모아서 보여줘"라는 쿼리가 가능해진다.

#### ② 인덱싱과 샤딩 (Indexing & Sharding)
Elasticsearch는 로그가 들어오는 즉시 **역색인(Inverted Index)**을 만든다. 단어 하나하나를 인덱스 카드로 만들어 두기 때문에, 수조 건의 로그 속에서도 'PF'라는 단어가 포함된 로그를 0.1초 만에 찾아낼 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### 주요 로그 시스템 비교 (ELK vs EFK vs Loki)

| 비교 지표 | ELK (Elasticsearch) | EFK (Fluentd) | **Loki (Grafana)** |
|-----------|----------------------|---------------|-------------------|
| **수집기** | Logstash (무거움) | Fluentd (표준) | Promtail (가벼움) |
| **저장 방식** | 전체 텍스트 인덱싱 | 전체 텍스트 인덱싱 | **메타데이터(Label)만 인덱싱** |
| **비용/용량** | 높음 (인덱스 용량 큼) | 높음 | **매우 낮음 (압축률 극상)** |
| **검색 속도** | 최상 (전문 검색 가능) | 최상 | 보통 (필터링 위주) |
| **비유** | 만능 백과사전 | 만능 백과사전 | **똑똑한 색인 카드** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 한 글로벌 핀테크 기업의 결제 서버에서 개인정보(카드번호, 주민번호)가 포함된 로그가 수집되어 보안 감고에 걸렸다. 이미 수십억 건의 로그가 Elasticsearch에 쌓여있어 수동으로 지우는 것은 불가능하다.
**판단 및 로그 거버넌스 전략**:
1. **Masking at Source**: Logstash 설정 파일에 **Mutate/Replace** 필터를 추가한다. 정규표현식으로 카드번호 패턴(`\d{4}-...`)을 찾아 `XXXX-XXXX-...`로 치환한 뒤 저장하게 한다.
2. **ILM (Index Lifecycle Management)**: 민감 정보가 포함된 오래된 로그는 30일이 지나면 자동으로 삭제(Delete Phase)되도록 정책을 수립한다.
3. **RBAC 강화**: Kibana 대시보드에 접근할 수 있는 권한을 직무별로 세분화하고, 특정 단어 검색 시 마스킹 처리된 결과만 보이도록 설정한다.
4. **결과**: 법적 컴플라이언스를 준수하면서도, 개발팀의 디버깅 효율을 해치지 않는 안전한 로그 시스템을 재구축한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **과도한 로그 생성 (Log Bloat)**: "일단 다 찍고 보자"는 생각으로 `DEBUG` 레벨 로그를 운영 환경에서 남기면 안 된다. 디스크는 금방 꽉 차고, 정작 중요한 `ERROR` 로그를 찾는 데 방해가 된다. 반드시 **로그 레벨 전략**을 수립하고, 필요한 정보(Context ID 등)만 포함해야 한다.
- **단일 로그 서버 구축**: 모든 로그를 서버 한 대에 모으는 것은 매우 위험하다. 로그 서버가 죽으면 시스템 전체의 가시성이 사라진다. 반드시 **분산 클러스터(Elasticsearch Cluster)**로 구성해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **장애 원인 파악 시간(MTTR)**: 서버마다 일일이 들어가던 시간 대비 **90% 이상 단축**.
- **트래픽 통계 확보**: 로그를 기반으로 한 실시간 사용자 유입 분석으로 비즈니스 인사이트 제공.

### 미래 전망 및 진화 방향
로그 시스템은 이제 **'Cold Storage'**와 **'Hot Storage'**의 분리로 진화하고 있다. 자주 찾는 최신 로그는 비싼 SSD에, 과거 로그는 저렴한 S3(오브젝트 스토리지)에 저장하면서도 쿼리는 한 번에 할 수 있는 **'Frozen Index'** 기술이 대세가 되고 있다. 또한, AI가 로그의 텍스트 패턴을 분석하여 이상 징후를 스스로 보고하는 **'Log Anomaly Detection'**이 인프라 운영의 자동화를 완성할 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **로그가 뭔가요?**: 컴퓨터가 오늘 무슨 일을 했는지 매초마다 쓰는 '비밀 일기장'이에요.
2. **로그 수집은 왜 하나요?**: 컴퓨터가 수천 대나 돼서 일기장도 수천 권인데, 선생님이 에러를 찾으려고 일기장을 하나하나 다 읽어볼 순 없잖아요. 그래서 한꺼번에 큰 도서관(ELK)으로 모으는 거예요.
3. **가장 좋은 점은요?**: 도서관 컴퓨터에 "어제 고장 난 일기 찾아줘!"라고 치면, 수천 권의 일기장 중에서 정답이 있는 페이지만 1초 만에 딱 펼쳐서 보여준답니다!
