# 569. 분산 추적 (Distributed Tracing) - 마이크로서비스의 항해 지도

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 복잡한 마이크로서비스(MSA) 환경에서 하나의 사용자 요청이 여러 서비스를 거쳐 처리되는 전체 과정을 실시간으로 추적하여, 각 단계별 소요 시간(Latency)과 호출 관계를 시각화하는 기술이다.
> 2. **가치**: 특정 요청이 왜 느린지, 어느 지점에서 에러가 발생했는지 "바늘 찾기" 같은 디버깅 작업을 수 분 내로 단축하며, 서비스 간의 의존성을 명확히 파악하여 시스템 최적화를 지원한다.
> 3. **융합**: 고유한 **Trace ID**를 기반으로 한 컨텍스트 전파(Context Propagation), 구글 Dapper의 후예인 **OpenTelemetry**, 그리고 **Jaeger/Zipkin** 시각화 도구가 융합되어 분산 시스템의 투명성을 확보한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
분산 추적은 분산 컴퓨팅 환경에서 요청의 일생을 기록하는 기법이다. 요청이 네트워크 경계를 넘을 때마다 고유한 ID를 실어 보내, 나중에 흩어진 파편(Span)들을 모아 하나의 거대한 호출 트리(Tree)로 재구성한다.

### 💡 비유
분산 추적은 **"글로벌 택배 배송 추적 시스템"**과 같다.
- **요청 (택배 물건)**: 고객이 보낸 소중한 패키지 하나.
- **Trace ID (운송장 번호)**: 물건이 처음 접수될 때 붙는 단 하나의 고유 번호.
- **Span (경유지 기록)**: '허브 터미널 도착', '상차 완료', '배달지 도착' 등 각 지점에서 찍히는 스탬프와 시간.
- **분산 추적 시스템 (추적 앱)**: 운송장 번호만 치면 이 물건이 어느 도시에서 몇 시에 머물렀고, 지금 어디서 지체되고 있는지 한눈에 보여주는 서비스다.

### 등장 배경 및 발전 과정

#### 1. 마이크로서비스의 그림자
서비스를 수백 개로 쪼개니 관리는 편해졌는데, 정작 장애가 나면 범인을 찾을 수 없게 되었다. "A가 B를 불렀고, B가 C를 불렀는데 C가 늦게 대답하네?"라는 인과관계를 밝히는 것이 불가능해진 것이다.

#### 2. 구글 Dapper의 공개와 표준화
2010년 구글이 내부 분산 추적 시스템인 **Dapper** 논문을 공개하면서 업계의 표준이 정립되었다. 이후 Twitter의 Zipkin, Uber의 Jaeger가 등장했고, 현재는 벤더 중립적인 표준인 **OpenTelemetry**로 통합되고 있다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 분산 추적의 핵심 개념 요소 (표)

| 용어 | 상세 설명 | 비유 |
|------|-----------|------|
| **Trace** | 하나의 사용자 요청이 완료되기까지의 전체 여정 | 택배의 전체 배송 경로 |
| **Span** | 서비스 내의 개별 작업 단위 (함수 호출, DB 조회 등) | 특정 터미널에서의 상/하차 작업 |
| **Trace ID** | 전체 Trace를 관통하는 고유 식별자 | 운송장 번호 |
| **Span ID** | 개별 Span을 식별하는 번호 (부모 ID 정보를 포함) | 경유지별 고유 바코드 |
| **Annotation** | Span 내부에 기록하는 이벤트나 태그 (에러 메시지 등) | "박스 훼손" 같은 특이사항 메모 |

### 정교한 구조 다이어그램 (MSA 호출 트리 시각화)

```ascii
================================================================================
[ Distributed Tracing Flow: Visualizing the Request Lifecycle ]
================================================================================

  [ User ] ──▶ [ Service A (Gateway) ] ──▶ [ Service B (Order) ] ──▶ [ Service C (DB) ]
                   (Span 1)                 (Span 2)                 (Span 3)

 [ Time Sequence View ]
 ──────────────────────────────────────────────────────────────────────────▶ Time
 | ⭐️ Service A (Total Trace: 500ms)                                       |
 └────────────────┬────────────────────────────────────────────────────────┘
                  │
                  │ ⭐️ Service B (300ms)
                  └────────────────────────┬────────────────────────┘
                                           │
                                           │ ⭐️ Service C (100ms)
                                           └────────────────────┘

 [ Data Propagation ]
 1. Service A: Create TraceID=101, SpanID=1. Send to B with TraceID=101.
 2. Service B: Receive TraceID=101. Create SpanID=2, ParentID=1. Send to C.
 3. Service C: Receive TraceID=101. Create SpanID=3, ParentID=2.
```

### 심층 동작 원리

#### ① 인스루멘테이션 (Instrumentation)
코드 내부에서 추적 데이터를 생성하는 행위다.
- **자동 (Auto)**: 라이브러리(Spring Cloud Sleuth 등)가 HTTP 요청을 가로채서 자동으로 ID를 붙이고 전송한다.
- **수동 (Manual)**: 개발자가 직접 중요한 비즈니스 로직 앞뒤에 `span.start()`, `span.finish()`를 코딩한다.

#### ② 컨텍스트 전파 (Context Propagation)
가장 어려운 부분이다. 서비스 A에서 B로 요청을 보낼 때, Trace ID 정보를 **HTTP Header (예: `X-B3-TraceId`)**나 gRPC 메타데이터에 실어 보내야 한다. 이를 통해 서비스 B는 자신이 새로운 요청을 받은 게 아니라, 기존 101번 요청의 연장선에 있음을 알게 된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 주요 분산 추적 도구 비교 (OpenSource vs SaaS)

| 비교 지표 | Jaeger (OpenSource) | Zipkin (OpenSource) | Datadog / NewRelic (SaaS) |
|-----------|----------------------|----------------------|---------------------------|
| **유래** | Uber 개발 | Twitter 개발 | 상용 클라우드 서비스 |
| **표준 준수** | **OpenTelemetry 기본** | B3 Propagation | 자체 프로토콜 + OTel |
| **강점** | **강력한 쿼리 및 시각화** | 단순하고 안정적 | **로그/메트릭과 완벽 결합** |
| **저장소** | Cassandra, Elasticsearch | Cassandra, MySQL | 자체 대용량 스토리지 |
| **비용** | 무료 (인프라 구축 필요) | 무료 | 유료 (데이터 양에 따라 비쌈) |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 쇼핑몰 결제 완료 버튼을 누르면 가끔씩 3초 이상 딜레이가 발생한다. 로그를 봐도 각 서비스는 "정상 처리됨"이라고만 찍힌다. 결제 서비스는 DB 탓을 하고, DB 팀은 네트워크 탓을 하는 핑퐁 게임이 벌어지고 있다.
**판단 및 추적 도입 전략**:
1. **OpenTelemetry Sidecar 배포**: 모든 파드(Pod)에 에이전트를 심어 수동 코드 수정 없이 전체 트레이스를 수집한다.
2. **Critical Path 분석**: Jaeger 화면에서 3초 걸린 트레이스를 찾는다.
3. **병목 발견**: 서비스 B가 서비스 C를 부르기 전, 인증 서비스(Auth)를 부르는 과정에서 2.5초가 소요됨을 발견한다. 알고 보니 인증 서버의 캐시가 만료되어 외부 API를 동기적으로 호출하고 있었다.
4. **결과**: 팀 간의 감정 소모 없이 **데이터 기반의 명확한 증거**를 제시하여 1시간 만에 해결책(캐시 주기 연장)을 수립한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **과도한 샘플링 (Sampling Rate)**: 초당 10만 건의 요청이 들어오는데 모든 트레이스를 다 저장하면, 추적 서버가 먼저 죽거나 스토리지 비용으로 회사가 망한다. 보통 **1% ~ 5%** 정도만 샘플링하여 추세만 파악하고, 에러가 난 트레이스는 무조건 수집하는 **Tail-based Sampling** 전략을 써야 한다.
- **Trace ID 단절 (Context Loss)**: 비동기 스레드나 메시지 큐(Kafka)를 거칠 때 ID 정보를 넘겨주지 않으면 트레이스가 뚝 끊긴다. 이 '미싱 링크'를 찾는 것이 분산 추적 운영의 가장 큰 고비다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **MTTD (장애 인지 시간)**: 서비스 간 호출 꼬임으로 인한 장애 인지 시간 **80% 단축**.
- **성능 최적화**: 가장 오래 걸리는 서비스(Hotspot)를 시각적으로 파악하여 하드웨어 자원 재배치 및 코드 최적화 지점 도출.

### 미래 전망 및 진화 방향
분산 추적은 이제 **'서비스 메시 (Service Mesh)'**와 결합하고 있다. Istio 같은 서비스 메시를 쓰면 개발자가 코드를 한 줄도 안 고쳐도 인프라 단에서 완벽한 트레이스를 그려준다. 또한, 최근에는 **eBPF** 기술을 활용하여 커널 레벨에서 패킷의 흐름을 추적함으로써, 오버헤드가 거의 없는 제로-에이전트 추적 시대로 진화하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **분산 추적이란?**: 미로처럼 복잡한 성(마이크로서비스)에서 보물 찾기를 할 때, 내 발자국마다 형광 페인트를 칠해서 내가 어디로 다녔는지 길을 표시하는 거예요.
2. **왜 하나요?**: 보물을 못 찾았을 때, 내가 어느 방에서 길을 잃었는지 지도를 보면 1초 만에 알 수 있기 때문이에요.
3. **가장 신기한 점은요?**: 내가 문을 열고 다른 방으로 건너가도 형광 페인트 색깔(Trace ID)이 변하지 않아서, 수백 명의 친구들 발자국 사이에서도 내 길만 쏙 골라낼 수 있답니다!
