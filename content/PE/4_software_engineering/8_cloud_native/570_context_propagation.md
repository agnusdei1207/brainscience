# 570. Trace ID와 Span ID의 전파 (Context Propagation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 시스템에서 하나의 작업 흐름을 유지하기 위해, 요청이 네트워크 경계(HTTP, gRPC 등)를 넘을 때마다 고유한 식별자(Trace/Span ID)를 메타데이터 헤더에 실어 다음 서비스로 전달하는 **상태 전이 기술**이다.
> 2. **가치**: 파편화된 개별 서비스의 로그와 실행 기록을 하나의 거대한 연대기(Timeline)로 묶어주는 '접착제' 역할을 하며, 분산 추적 시스템이 전체 호출 그래프를 정확히 재구성할 수 있게 한다.
> 3. **융합**: W3C Trace Context 표준, B3 Propagation 규격, 그리고 인프로세스(In-process) 컨텍스트 관리를 위한 ThreadLocal 기술이 융합되어 데이터 유실 없는 추적망을 형성한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
컨텍스트 전파(Context Propagation)는 요청이 A 서비스에서 B 서비스로 이동할 때, 추적에 필요한 정보를 잃어버리지 않도록 운반하는 행위다. 
- **Trace ID**: 전체 트랜잭션을 식별하는 '가문(Family)'의 이름.
- **Span ID**: 현재 서비스 내의 작업 단위를 식별하는 '개인'의 이름.
- **Parent ID**: 나를 호출한 상위 작업이 무엇인지 알려주는 정보.

### 💡 비유
컨텍스트 전파는 **"릴레이 경주의 바통 터치"**와 같다.
- **Trace ID (바통)**: 주자가 바뀌어도 바통은 하나다. 이 바통이 결승점에 들어와야 기록이 인정된다.
- **Span ID (주자 번호)**: 1번 주자가 뛰고 나면 바통을 2번 주자에게 넘긴다. 2번 주자는 "나는 1번 주자(Parent ID)한테 바통을 받았어"라고 기록한다.
- **전파 (바통 터치)**: 다음 주자에게 바통을 넘기지 않으면(Propagation 실패), 아무리 열심히 뛰어도 전체 경주 기록을 낼 수 없다.

### 등장 배경 및 발전 과정

#### 1. 프로토콜 간의 단절
HTTP, gRPC, Kafka 등 서비스가 대화하는 수단은 다양하다. 각 프로토콜은 데이터 전송에만 집중할 뿐, "이게 누구의 요청인지"에 대한 정보는 기본적으로 포함하지 않는다.

#### 2. 표준의 부재와 W3C의 등장
초기에는 Google(Dapper), Zipkin(B3) 등 솔루션마다 헤더 명칭(`X-B3-TraceId` 등)이 달라 호환이 안 됐다. 이를 해결하기 위해 W3C에서 **`traceparent`**라는 표준 헤더 형식을 정의하며 벤더 간 호환이 가능해졌다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 컨텍스트 전파의 2단계 매커니즘 (표)

| 단계 | 명칭 | 동작 상세 | 비유 |
|------|-----------|-----------|------|
| **1. Inject (주입)** | 송신측(Client) 처리 | 현재 메모리에 있는 ID를 HTTP 헤더 등에 박아넣음 | 편지 봉투에 주소 적기 |
| **2. Extract (추출)**| 수신측(Server) 처리 | 들어온 헤더에서 ID를 읽어 현재 스레드의 메모리에 복원 | 편지 봉투 열어보기 |

### 정교한 구조 다이어그램 (HTTP Header 기반 전파)

```ascii
================================================================================
[ Context Propagation Flow: Trace ID Journey ]
================================================================================

  [ 📦 Service A (Client) ]               [ 📦 Service B (Server) ]
 ┌─────────────────────────┐             ┌─────────────────────────┐
 │ (1) Generate Context    │             │ (4) Extract Context     │
 │ TraceID: 0xABCD         │             │ TraceID: 0xABCD         │
 │ SpanID: 0x1111          │             │ ParentID: 0x1111        │
 └──────────┬──────────────┘             └──────────▲──────────────┘
            │                                       │
            │ (2) 💉 INJECT into Header             │ (5) 🧬 NEW SPAN
            ▼                                       │ TraceID: 0xABCD
  ┌────────────────────────┐             ┌──────────┴──────────────┐
  │ 📨 HTTP Request        │             │ (6) Work...             │
  │ Header:                │             │ SpanID: 0x2222          │
  │ traceparent:           │────────────▶│                         │
  │ 00-0xABCD-0x1111-01    │             │                         │
  └────────────────────────┘             └─────────────────────────┘
            ▲
            │ (3) Network Transfer (Wire)
```

### 심층 동작 원리

#### ① W3C Trace Context 표준 포맷
가장 널리 쓰이는 표준인 `traceparent` 헤더의 구조는 다음과 같다.
`00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01`
- `00`: 버전 (Version)
- `4bf9...`: **Trace ID** (16바이트)
- `00f0...`: **Parent Span ID** (8바이트)
- `01`: 플래그 (Flags - 샘플링 여부 등)

#### ② 인프로세스 전파 (ThreadLocal)
네트워크를 넘기 전, 코드 내부에서도 전파가 필요하다. Java의 경우 **ThreadLocal**을 써서 현재 실행 중인 스레드에 ID를 저장한다. 함수를 호출할 때마다 파라미터로 넘기지 않아도, 로그 라이브러리가 스레드 로컬에서 ID를 쓱 꺼내어 로그에 자동으로 찍어준다.

#### ③ 비동기 처리의 함정 (Async Propagation)
스레드를 새로 생성(`new Thread()`)하거나 비동기 프레임워크를 쓰면 ThreadLocal 값이 복사되지 않아 추적이 끊긴다. 이를 위해 **Context Snapshot**을 찍어 새 스레드로 복사해 주는 특수한 래퍼(Wrapper) 클래스가 아키텍처의 필수 부품이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 주요 전파 규격(Propagation Format) 비교

| 규격명 | 주도 세력 | 헤더 형식 | 특징 |
|--------|-----------|-----------|------|
| **W3C Trace Context** | **표준 (W3C)** | `traceparent` | **현대적 표준, 가장 권장됨** |
| **B3 (Zipkin)** | Zipkin / OSS | `X-B3-TraceId`, `X-B3-SpanId` | 가장 널리 쓰이던 고전 방식 |
| **Jaeger** | Uber | `uber-trace-id` | Jaeger 전용 포맷 |
| **GCP / AWS** | 클라우드사 | `x-cloud-trace-context` | 각 클라우드 로드밸런서 연동용 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 사내 서비스가 Python(Django), Java(Spring), Go(Fiber)로 섞여 있다. 추적 시스템을 구축했는데, Python에서 Java로 넘어갈 때는 추적이 잘 되는데, Java에서 Go로 넘어갈 때 Trace ID가 유실되어 그래프가 끊긴다.
**판단 및 인터페이스 통일 전략**:
1. **표준 규격 강제**: 모든 서비스의 추적 라이브러리 설정을 **W3C Trace Context**로 통일한다. (기존 B3 헤더 사용 시 혼선 발생 가능성 차단).
2. **Propagator 설정**: Go 언어의 Fiber 미들웨어에서 `otelslog`와 함께 `W3CPropagator`를 명시적으로 등록한다.
3. **가시성 검증**: 테스트 요청을 날린 후, Envoy나 API Gateway 로그에서 `traceparent` 헤더가 정상적으로 실려 나가는지 `tcpdump`로 확인한다.
4. **결과**: 이기종 언어 간의 장벽을 허물고, 전체 시스템을 관통하는 100% 무결한 호출 트리를 완성한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **헤더 변조 및 삭제**: 회사 망의 보안 방화벽이나 프록시 서버에서 "알 수 없는 헤더"라며 `traceparent`를 지워버리는 경우가 흔하다. 인프라 팀과 협의하여 추적용 헤더는 반드시 **화이트리스트**에 등록해야 한다.
- **보안 기밀 유출 (Baggage)**: ID 외에 '사용자 이름' 같은 정보를 헤더에 실어 보내는 기능(Baggage)이 있는데, 이를 남용하면 외부망으로 민감 정보가 나갈 수 있다. Baggage는 반드시 **내부 통신용**으로만 써야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **데이터 일관성**: 서비스 간 데이터 불일치 발생 시, 동일한 Trace ID를 통해 원인 파악 속도 **5배 향상**.
- **트러블슈팅 정밀도**: 흩어진 로그 100만 건 중, 특정 요청과 관련된 로그 10건을 핀셋으로 집어내듯 정확히 추출.

### 미래 전망 및 진화 방향
컨텍스트 전파는 이제 **'로그와의 완전한 결합'**으로 나아가고 있다. 단순히 ID만 넘기는 게 아니라, 해당 요청의 우선순위나 디버깅 플래그까지 전파하여 장애 발생 시에만 자동으로 상세 로그를 남기는 **'지능형 적응형 추적'** 기술이 대두되고 있다. 또한, 5G/6G 통신 헤더 자체에 추적 ID를 포함하려는 시도가 이어지며, 네트워크 하드웨어와 소프트웨어가 하나로 연결되는 진정한 엔드-투-엔드 가시성 시대가 열릴 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **ID 전파가 뭔가요?**: 택배 아저씨가 물건을 옮길 때마다, 이전 아저씨가 적어준 '비밀 번호'를 다음 아저씨에게 꼭 전달해 주는 약속이에요.
2. **왜 전달해야 하나요?**: 번호를 잃어버리면 이 택배가 어디서 왔는지 알 수 없어서, 나중에 길을 잃어도 찾아줄 방법이 없기 때문이에요.
3. **가장 중요한 점은요?**: 비행기를 타든 배를 타든(프로토콜 변경), 이 비밀 번호만 잘 챙겨가면 지구 반대편에서도 내 택배를 안전하게 추적할 수 있답니다!
