# 571. 탄력성 (Resiliency) 및 결함 허용 (Fault Tolerance) 패턴

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템의 일부 구성 요소에 장애가 발생하더라도 서비스 전체가 중단되지 않고, 허용 가능한 수준에서 정상적으로 동작을 유지하며 장애 상황이 복구되면 스스로 자가 치유(Self-healing)하는 능력이다.
> 2. **가치**: 장애는 '피할 수 없는 상수'라는 전제하에 설계하여, 단일 서비스의 실패가 전체 시스템으로 번지는 **연쇄 장애(Cascading Failure)**를 차단하고 비즈니스 연속성(BCP)을 보장한다.
> 3. **융합**: 서킷 브레이커(Circuit Breaker), 재시도(Retry), 타임아웃(Timeout), 그리고 벌크헤드(Bulkhead) 패턴이 융합되어 분산 시스템의 강력한 '면역 체계'를 형성한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
- **Fault Tolerance (결함 허용)**: 시스템 어딘가 망가져도 사용자는 모르게 정상 결과를 내놓는 능력. (보통 하드웨어 이중화나 다중 실행을 통해 구현).
- **Resiliency (탄력성)**: 장애가 발생했을 때 빠르게 회복하고, 장애가 퍼지지 않게 유연하게 대처하는 능력. (소프트웨어 설계 패턴 중심).

### 💡 비유
탄력성 설계는 **"대형 여객선의 수밀 구역과 구명조끼"**와 같다.
- **기존 방식 (모놀리스)**: 배 전체가 하나의 껍데기다. 구멍이 하나만 나도 배 전체에 물이 차서 가라앉는다. (연쇄 장애).
- **탄력성 설계 (벌크헤드)**: 배 내부를 여러 개의 독립된 방(Bulkhead)으로 쪼갠다. 한쪽 방에 구멍이 나도 그 방만 폐쇄하면 배는 계속 떠 있을 수 있다. 또한, 엔진이 멈추면(장애) 즉시 돛(Fallback)을 올려서라도 항해를 계속한다.

### 등장 배경 및 발전 과정

#### 1. 클라우드와 MSA의 변동성
서버 1,000대를 운영하면 산술적으로 매일 서버 몇 대는 고장 난다. 장애가 '특별한 사건'이 아닌 '일상'이 된 환경에서, 장애를 막으려 애쓰는 것보다 **"장애와 함께 살아가는 법"**을 배우는 아키텍처가 필요해졌다.

#### 2. 넷플릭스(Netflix)와 카오스 엔지니어링
넷플릭스는 고의로 서버를 끄는 '카오스 몽키'를 돌리며 탄력성 패턴을 검증했고, 이를 통해 Hystrix(현재는 Resilience4j) 같은 라이브러리를 배포하며 탄력성 설계의 표준을 세웠다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 4대 핵심 탄력성 패턴 (표)

| 패턴명 | 상세 역할 | 작동 매커니즘 | 비유 |
|--------|----------|-------------------|------|
| **Timeout** | 무한 대기 방지 | 정해진 시간 내 응답 없으면 연결 강제 종료 | 약속 시간 넘으면 그냥 가기 |
| **Retry** | 일시적 장애 복구 | 네트워크 정크 등으로 실패 시 N번 더 시도 (Backoff 포함) | 전화 끊기면 다시 걸기 |
| **Circuit Breaker**| 장애 전파 차단 | 에러율이 높으면 호출을 즉시 차단하고 우회(Fallback) 처리 | 두꺼비집 (누전 차단기) |
| **Bulkhead** | 자원 격리 | 특정 서비스용 자원(Thread/Memory)을 물리적으로 할당/격리 | 여객선 격벽 |

### 정교한 구조 다이어그램 (서킷 브레이커 상태 전이)

```ascii
================================================================================
[ Circuit Breaker Pattern State Machine ]
================================================================================

      ┌────────────────────────────────────────────────────────────┐
      │                                                            │
      ▼         [ ✅ CLOSED ] (정상 상태)                           │
   Success  ◀──  모든 요청이 통과됨.                                │
      │         실패 횟수 카운트 중...                               │
      └──────────────┬─────────────────────────────────────────────┘
                     │ (실패율 > 임계치)
                     ▼
      ┌────────────────────────────────────────────────────────────┐
      │         [ ❌ OPEN ] (차단 상태)                             │
      │  모든 요청을 거절하고 Fallback(임시값) 반환.                  │
      │  백엔드 시스템이 쉴 수 있게 시간을 벌어줌.                     │
      └──────────────┬─────────────────────────────────────────────┘
                     │ (일정 시간 대기 후)
                     ▼
      ┌────────────────────────────────────────────────────────────┐
      │         [ ⚠️ HALF-OPEN ] (검증 상태)                        │
      │  딱 1~2개의 요청만 슬쩍 보내봄.                              │
      │  - 성공 시: CLOSED로 복귀 (자가 치유)                        │
      │  - 실패 시: OPEN으로 재진입                                  │
      └────────────────────────────────────────────────────────────┘
```

### 심층 동작 원리

#### ① 백오프(Exponential Backoff)와 지터(Jitter)
Retry 패턴 적용 시, 실패하자마자 즉시 재시도하면 죽어가는 서버를 확인사살하는 꼴이 된다. 1초, 2초, 4초... 처럼 대기 시간을 지수적으로 늘리고(Backoff), 수만 개의 클라이언트가 동시에 재시도하지 않도록 무작위 난수(Jitter)를 섞어 서버 부하를 평탄화한다.

#### ② 자원 격리 (Bulkhead Isolation)
서비스 A가 지연되면 서비스 A를 호출하려던 스레드 풀이 가득 차서, 정작 멀쩡한 서비스 B를 호출할 스레드까지 부족해지는 현상을 막는다. 각 서비스마다 **Thread Pool**이나 **Semaphore**를 따로 할당하여 "망해도 너 혼자 망해라"를 구현한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 장애 대응의 관점 변화

| 비교 지표 | 에러 처리 (Error Handling) | 탄력성 설계 (Resiliency) |
|-----------|----------------------------|--------------------------|
| **목표** | 에러 메시지 예쁘게 보여주기 | **시스템 전체 가동 유지** |
| **대응 영역** | 코드 내부 (try-catch) | **인프라/네트워크 경계** |
| **핵심 기법** | 예외 처리 | **격리, 차단, 우회** |
| **철학** | "에러가 안 나게 하자" | **"에러는 무조건 난다"** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대규모 트래픽이 몰리는 커머스 앱. '추천 상품' 서비스가 갑자기 느려져서 결제 버튼을 눌러도 반응이 없는 대규모 장애가 발생했다. 분석 결과, 추천 서비스 서버의 부하가 결제 서비스의 스레드 자원을 모두 점유하고 있었다.
**판단 및 탄력성 패턴 적용 전략**:
1. **서킷 브레이커 도입**: 추천 서비스 호출부에 서킷 브레이커를 건다. 응답이 1초 이상 걸리는 비중이 50%를 넘으면 즉시 차단을 시작한다.
2. **Fallback 구현**: 서킷이 열리면(Open), 추천 상품 대신 **"PF님이 좋아할 만한 기본 아이템"**이라는 미리 캐싱된 정적 데이터를 보여준다.
3. **Bulkhead 격리**: 추천 서비스 전용 스레드 풀을 전체의 10%로 제한한다.
4. **결과**: 추천 기능은 잠시 마비되어 '정적 데이터'가 나오지만, 가장 중요한 **결제 기능은 단 1초의 지연도 없이 정상 동작**하여 비즈니스 손실을 0으로 막는다. (Graceful Degradation 달성).

### 주의사항 및 안티패턴 (Anti-patterns)
- **무분별한 Retry**: DB 락이나 서버 메모리 부족으로 발생한 장애에 Retry를 10번씩 거는 것은 서버를 고문하는 행위다. 고칠 수 없는 에러(404 Not Found 등)에는 절대 재시도하지 말고, 고칠 수 있는 일시적 장애(503 Service Unavailable)에만 정교하게 적용해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **가용성(Availability)**: 부분 장애 시에도 핵심 기능 성공률 **99.99%** 유지.
- **장애 확산 시간**: 연쇄 장애 발생 확률 **95% 이상 감소**.

### 미래 전망 및 진화 방향
탄력성 설계는 이제 인프라가 알아서 해주는 **'Service Mesh'** 시대로 가고 있다. Istio나 Linkerd 같은 도구를 쓰면 개발자가 코딩을 안 해도 네트워크 단에서 서킷 브레이커와 타임아웃을 자동으로 걸어준다. 또한, 최근에는 AI가 평소 트래픽 패턴을 학습하여 장애 임계치를 동적으로 조절하는 **'Adaptive Resiliency'** 기술이 대두되고 있으며, 이는 사람이 잠든 사이에도 시스템이 스스로 방패를 들고 적을 막아내는 진정한 자율 운영 인프라의 근간이 될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **탄력성이 뭔가요?**: 장난감 자동차가 벽에 부딪혔을 때, 박살 나는 게 아니라 고무공처럼 팅 튕겨 나와서 다시 쌩쌩 달리는 능력이에요.
2. **왜 필요한가요?**: 집안의 모든 전등이 하나로 연결되어 있으면 전구 하나만 깨져도 집 전체가 깜깜해지는데, 탄력성 설계를 하면 깨진 전구만 딱 꺼지고 나머지는 계속 밝게 빛나기 때문이에요.
3. **가장 좋은 점은요?**: 조금 아프거나 실수를 해도 스스로 금방 기운을 차리고 다시 일을 할 수 있는 '슈퍼 파워'를 가진 컴퓨터를 만드는 것이랍니다!
