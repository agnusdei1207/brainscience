# 574. 벌크헤드 (Bulkhead) 패턴 - 자원 격리를 통한 장애 전파 방지

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템 내의 자원(스레드 풀, 메모리, 커넥션)을 서비스나 기능별로 독립된 구역으로 나누어 할당함으로써, 특정 영역의 장애(자원 고갈)가 시스템 전체로 번지는 것을 물리적으로 차단하는 **격리 설계 패턴**이다.
> 2. **가치**: 한 서비스의 병목이나 지연이 다른 멀쩡한 서비스의 자원을 잠식하는 것을 막아, 시스템의 일부분이 고장 나도 나머지 부분은 안전하게 가동되는 **'부분적 가용성'**을 보장한다.
> 3. **융합**: 서킷 브레이커와 결합하여 장애를 차단하고, 쿠버네티스의 리소스 제한(Limit/Request) 및 마이크로서비스의 물리적 클러스터 분리와 융합되어 거대한 방어선을 구축한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
벌크헤드(Bulkhead)는 선박의 '수밀 격벽'에서 유래했다. 배의 선체를 여러 칸으로 나누어 한곳에 구멍이 나서 물이 차더라도, 해당 칸만 폐쇄하면 배 전체가 가라앉는 것을 막는 구조를 소프트웨어에 이식한 것이다.

### 💡 비유
벌크헤드 패턴은 **"비행기 이코노미/비즈니스석의 화장실 분리"**와 같다.
- **격리 없음**: 비행기에 화장실이 딱 하나뿐인데, 이코노미 손님 한 명이 화장실을 고장 내버리면 비즈니스석 손님까지 화장실을 못 쓰게 된다. (전체 장애).
- **벌크헤드**: 구역마다 전용 화장실을 따로 둔다. 이코노미 화장실이 고장 나도 비즈니스석 손님은 자기 구역 화장실을 평화롭게 이용할 수 있다. (장애 격리).

### 등장 배경 및 발전 과정

#### 1. 스레드 고갈(Thread Starvation)의 공포
모든 사용자 요청을 하나의 커다란 스레드 풀에서 처리할 때, 특정 외부 API 호출이 10초 이상 지연되면 모든 스레드가 그 요청을 기다리느라 꽉 차버린다. 이때 들어오는 다른 모든 요청(로그인, 메인 페이지 조회 등)은 스레드를 할당받지 못해 즉시 에러가 난다.

#### 2. MSA의 복잡한 의존성
마이크로서비스 환경에서는 수많은 서비스가 서로를 부른다. 특정 서비스 하나가 느려졌을 뿐인데 앱 전체가 먹통이 되는 현상을 막기 위해, 기능별로 '자원 바구니'를 따로 챙겨주는 벌크헤드가 필수 기술이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 벌크헤드 구현의 2가지 주요 방식 (표)

| 방식 | 상세 내용 | 장점 | 단점 |
|------|-----------|------|------|
| **Thread Pool Isolation** | 서비스마다 전용 스레드 풀을 할당 (예: 결제용 10개, 조회용 20개) | 완벽한 물리적 격리, 타임아웃 제어 용이 | 컨텍스트 스위칭 오버헤드 발생 |
| **Semaphore Isolation** | 서비스별 최대 동시 요청 수 제한 (카운터 방식) | 오버헤드가 매우 적고 가벼움 | 네트워크 지연 시 스레드 점유 해제가 안 됨 |

### 정교한 구조 다이어그램 (스레드 풀 격리 모델)

```ascii
================================================================================
[ Bulkhead Pattern: Thread Pool Isolation ]
================================================================================

  [ Incoming Requests ]
           │
           ▼
  ┌──────────────────────────────────────────────────────────────────────────┐
  │  🏢 API Gateway / Application Server                                     │
  │                                                                          │
  │  ┌────────────────────────┐        ┌──────────────────────────────────┐  │
  │  │ 📦 Thread Pool A       │        │ 📦 Thread Pool B                 │  │
  │  │ (Order Service)        │        │ (Inventory Service)              │  │
  │  │ [T][T][T][X][X]        │        │ [T][T][T][T][T] ⚠️ FULL!         │  │
  │  └──────────┬─────────────┘        └──────────┬───────────────────────┘  │
  └─────────────┼─────────────────────────────────┼──────────────────────────┘
                │                                 │
                ▼                                 ▼
        ┌──────────────┐                  ┌──────────────┐
        │ Order Micro  │                  │ Inventory    │ ◀── 💥 지연 발생 중!
        │ Service      │ (정상 가동)        │ Service      │     (모든 스레드 점유)
        └──────────────┘                  └──────────────┘

 * 핵심 포인트: 인벤토리 서비스가 터져서 Pool B가 꽉 찼지만, 
   Pool A는 독립되어 있으므로 주문 서비스는 여전히 쌩쌩하게 돌아감.
```

### 심층 동작 원리

#### ① 쿼터(Quota) 기반의 자원 배분
시스템 전체 자원이 100이라면, 중요도에 따라 자원을 나눈다. "핵심 결제 로직"에는 50을, "광고 배너 로직"에는 10을 주는 식이다. 광고 서버가 아무리 느려져도 광고용 자원 10만 날릴 뿐, 나머지 90의 자원은 결제와 일반 기능을 위해 온전히 보존된다.

#### ② 물리적 인프라 격리 (Cluster Bulkhead)
소프트웨어 레벨을 넘어 아예 서버 뭉치(Cluster)를 나눈다. "VIP 전용 서버"와 "일반 사용자 서버"를 물리적으로 분리하여, 일반 사용자가 몰려 서버가 터져도 VIP 서비스는 쾌적하게 유지되게 만드는 극단적인 벌크헤드 방식이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 서킷 브레이커 vs 벌크헤드

| 비교 지표 | 서킷 브레이커 (Circuit Breaker) | 벌크헤드 (Bulkhead) |
|-----------|----------------------------------|----------------------|
| **주요 목적** | **실패한 호출**의 반복 방지 | **자원 잠식** 방지 |
| **작동 원리** | 상태 머신 (Open/Closed) | **자원 쪼개기 (Pool/Limit)** |
| **초점** | 네트워크/응답 정합성 | **시스템 가용 자원 보호** |
| **비유** | 두꺼비집 (전기 차단) | 배의 격벽 (침수 격리) |
| **관계** | 서로 보완적 (보통 함께 사용함) | 서로 보완적 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 한 동영상 플랫폼에서 "댓글 작성" 기능이 갑자기 느려졌다. 원인은 욕설 필터링 외부 API 서버의 장애였다. 그런데 댓글이 안 써지는 건 참겠는데, 멀쩡하던 "영상 시청" 기능까지 먹통이 되어버렸다. 분석 결과, 댓글 저장 스레드가 외부 API를 기다리느라 톰캣(WAS)의 전체 스레드 200개를 다 먹어버린 것이었다.
**판단 및 벌크헤드 적용 전략**:
1. **Resilience4j Bulkhead 설정**: 댓글 작성 기능을 호출하는 메소드에 벌크헤드 어노테이션을 붙인다.
2. **제한 수치 설정**: `maxConcurrentCalls`를 전체 스레드의 10%인 '20개'로 제한한다.
3. **Queue 대기 설정**: 20개가 넘으면 즉시 에러를 뱉게 하거나(Fail-fast), 아주 짧은 대기 큐를 둔다.
4. **결과**: 댓글 필터링 API가 멈춰도 댓글 스레드 20개만 묶일 뿐, 나머지 180개 스레드는 영상 시청 요청을 계속 처리한다. **댓글은 안 써지지만 영상은 잘 나오는** 탄력적 시스템이 완성된다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **너무 잘게 쪼갠 벌크헤드 (Fragmented Pools)**: 모든 메소드마다 스레드 풀을 따로 만들면, 메모리 낭비가 심해지고 CPU가 스레드 관리하느라 정작 일은 못 하는 '배보다 배꼽이 더 큰' 상황이 온다. 반드시 **비즈니스 도메인 단위**나 **위험도(외부 API 연동 여부 등)**에 따라 큼직하게 나누어야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **장애 격리율**: 단일 기능 장애 시 전체 시스템 다운 발생 확률 **99% 이상 차단**.
- **가용 자원 보존**: 피크 타임 트래픽 폭주 시에도 핵심 비즈니스 로직을 위한 '최소 자원' 상시 확보.

### 미래 전망 및 진화 방향
벌크헤드 패턴은 이제 인프라 수준의 **'사이드카(Sidecar)'** 모델로 진화하고 있다. Envoy나 Istio 같은 서비스 메시를 쓰면, 앱 코드를 한 줄도 안 고쳐도 인프라가 알아서 특정 서비스로 가는 트래픽의 동시성을 제어하고 격리한다. 또한, 향후에는 AI가 실시간으로 각 서비스의 중요도와 부하를 계산하여 벌크헤드의 크기를 유동적으로 조절하는 **'다이나믹 리소스 파티셔닝'** 기술이 클라우드 네이티브의 핵심이 될 것이다.

1. **벌크헤드가 뭔가요?**: 커다란 배의 안쪽을 여러 개의 방으로 튼튼하게 나눠놓는 거예요. 
2. **왜 나누나요?**: 배 한쪽 구석에 구멍이 나서 물이 들어와도, 그 방 문만 꽉 잠가버리면 배 전체가 가라앉지 않고 계속 둥둥 떠 있을 수 있기 때문이에요.
3. **컴퓨터에서는요?**: 게임하는 방이랑 공부하는 방을 따로 나눠서, 게임이 버벅거리고 고장 나도 공부하는 방은 아무 문제 없이 조용하고 깨끗하게 유지하는 비법이랍니다!
