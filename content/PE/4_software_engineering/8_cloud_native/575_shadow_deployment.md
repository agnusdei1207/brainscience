# 575. 섀도우 배포 (Shadow Deployment) - 실운영 데이터로 검증하는 유령 배포

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 실제 운영 환경으로 유입되는 사용자 트래픽을 복제(Mirroring)하여, 사용자에게는 영향을 주지 않는 격리된 신규 버전 서버로 전송해 **실제 데이터 기반의 안정성과 성능을 검증하는 배포 전략**이다.
> 2. **가치**: 테스트 환경에서는 미처 발견하지 못한 엣지 케이스(Edge Case)와 운영 환경의 데이터 부하를 100% 동일한 조건에서 미리 확인할 수 있어, 신규 기능 배포 시의 심리적/기술적 불안감을 해소한다.
> 3. **융합**: 트래픽 미러링 기능을 지원하는 서비스 메시(Istio), 데이터 일관성 검증을 위한 Diffing 도구, 그리고 비동기 데이터 분석 플랫폼과 융합되어 '무결점 배포'의 최종 단계를 구현한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
섀도우 배포(Shadow Deployment), 또는 트래픽 미러링(Traffic Mirroring)은 신규 버전(Shadow)을 운영 서버 옆에 몰래 띄워두고, 실제 요청을 두 곳에 동시에 쏘는 방식이다. 
- **운영 버전**: 요청을 처리하고 사용자에게 응답을 보낸다.
- **섀도우 버전**: 요청을 처리하지만, 결과값은 사용자에게 보내지 않고 로그로 남겨 운영 버전의 결과와 비교(Diff)만 한다.

### 💡 비유
섀도우 배포는 **"신입 조종사의 시뮬레이션 비행"**과 같다.
- **현실**: 베테랑 조종사(운영 버전)가 비행기 핸들을 잡고 승객(사용자)을 태워 목적지로 가고 있다.
- **섀도우**: 신입 조종사(신규 버전)는 옆자리 보조 핸들을 잡고 베테랑이 하는 동작을 똑같이 따라 한다. 신입이 핸들을 잘못 꺾어도 실제 비행기 날개는 움직이지 않으니 승객은 안전하다. 비행이 끝난 후, 베테랑과 신입의 조종 기록을 비교하여 신입이 얼마나 잘했는지 평가하는 것과 같다.

### 등장 배경 및 발전 과정

#### 1. 테스트 데이터의 한계
"내 컴퓨터에서는 잘 됐는데?"의 주된 원인은 실제 사용자가 넣는 기상천외한 데이터(오타, 특수문자, 초거대 파일 등)를 테스트 코드가 다 담지 못하기 때문이다.

#### 2. 카나리 배포의 위험성
카나리(Canary) 배포는 실제 사용자 1%에게 신규 버전을 노출한다. 만약 버그가 있다면 그 1%의 사용자는 장애를 겪게 된다. 섀도우 배포는 **"0%의 사용자 노출"**로 100%의 트래픽을 검증하려는 극한의 안전주의 요구에서 탄생했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 주요 배포 전략별 사용자 영향도 비교 (표)

| 전략 | 트래픽 비율 | 사용자 노출 | 실패 시 영향 | 비유 |
|------|-------------|-------------|--------------|------|
| **Blue-Green** | 0% or 100% | 즉시 전체 노출 | 서비스 전체 마비 | 새 집으로 이사 |
| **Canary** | 1% ~ 10% | 부분 노출 | 일부 사용자 장애 | 기미 상궁 |
| **Shadow** | **100% 복제** | **노출 없음** | **영향 Zero** | **유령 그림자** |

### 정교한 구조 다이어그램 (서비스 메시 기반 미러링)

```ascii
================================================================================
[ Shadow Deployment Architecture: Traffic Mirroring ]
================================================================================

      [ 🌐 Real Users ]
              │
              ▼ (1) Actual Traffic (POST /order)
  ┌──────────────────────────────────────────────────────────┐
  │  🛡️ Traffic Splitter / Service Mesh (Istio)              │
  │  - Copy Request Payload                                  │
  │  - Branch A: Real Path (Wait for Response)               │
  │  - Branch B: Shadow Path (Fire and Forget)               │
  └──────────────┬──────────────────────────────┬────────────┘
                 │                              │
        (2) Real 🚀                             │ (3) Mirror 👻
                 ▼                              ▼
  ┌────────────────────────┐          ┌────────────────────────┐
  │  📦 Prod Service (V1)  │          │  📦 Shadow Service (V2)│
  │  - Real DB Update      │          │  - Mock DB / Dry-run   │
  └──────────────┬─────────┘          └──────────────┬─────────┘
                 │ (4) Result A                  │ (5) Result B
                 ▼                               ▼
  ┌──────────────────────────────────────────────────────────┐
  │  ⚖️ Comparator / Analyzer                                 │
  │  - IF (Result A != Result B) ──▶ Send Alert!             │
  └──────────────────────────────────────────────────────────┘
                 │
                 ▼ (6) Response to User (Always from V1)
```

### 심층 동작 원리

#### ① 트래픽 미러링 (Mirroring)
가장 앞단의 로드밸런서(NGINX)나 서비스 메시(Envoy)에서 패킷을 복제한다. 이때 핵심은 **'비동기 처리'**다. 운영 버전의 응답 속도가 섀도우 버전 때문에 느려지면 안 되므로, 미러링된 트래픽은 '쏘고 잊기(Fire and Forget)' 방식으로 별도의 큐를 통해 신규 버전으로 전달된다.

#### ② 쓰기 요청의 처리 (Side-effects Management)
조회(GET) 요청은 섀도우 서버에 쏴도 안전하다. 하지만 결제(POST) 요청을 섀도우 서버에 쏘면 DB에 데이터가 두 번 쌓이는 대참사가 난다. 
- **해결책 1**: 섀도우 서버의 DB를 격리된 별도의 테스트 DB로 연결한다.
- **해결책 2**: 애플리케이션 레벨에서 'Shadow Mode' 플래그를 인식하여, 실제 DB 쓰기 로직만 건너뛰고 계산만 수행한다 (Dry-run).

## Ⅲ. 융합 비교 및 다각도 분석

### 섀도우 배포의 장단점 트레이드오프

| 장점 (Pros) | 단점 (Cons) |
|-------------|-------------|
| **완벽한 안전**: 사용자에게 장애를 줄 확률이 0%임 | **비용 발생**: 운영 서버와 동일한 수의 서버가 한 세트 더 필요함 (인프라 비용 2배) |
| **현실적 부하 테스트**: 봇이 만든 가짜 트래픽이 아닌 실제 유저 트래픽으로 성능 측정 | **복잡도 극상**: 쓰기 요청(Side-effect)을 안전하게 가로채는 로직 구현이 힘듦 |
| **정확한 결과 비교**: 구버전과 신규 버전의 응답 값을 1:1로 대조 가능 | **지연 분석**: 실시간 응답이 아니므로 로그 기반 사후 분석이 주를 이룸 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 금융 플랫폼의 '핵심 이자 계산 엔진'을 10년 만에 전면 개편했다. 테스트 코드는 통과했지만, 복잡한 세금 면제 조건이나 휴일 처리 등 수만 가지의 실데이터 변수가 걱정되어 차마 운영에 배포하지 못하고 있다.
**판단 및 섀도우 배포 전략**:
1. **Istio Mirroring 가동**: 운영망 결제 트래픽을 100% 복제하여 신규 엔진으로 보낸다.
2. **Shadow DB 구축**: 운영 DB를 스냅샷 찍어 만든 별도의 통계 DB에 섀도우 엔진을 연결한다.
3. **Diff Tool 가동**: 운영 엔진의 결과값과 섀도우 엔진의 결과값을 Kafka로 모아 실시간으로 비교한다.
4. **결과**: 일주일간 돌려보니 100만 건 중 3건에서 미세한 오차가 발견되었다. 알고 보니 구버전 엔진의 버그가 신규 엔진에서 고쳐지면서 발생한 차이였다. 이를 확인한 경영진은 확신을 가지고 **Blue-Green 전환**을 승인한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **외부 API 중복 호출**: 섀도우 서버가 외부 알림톡 발송 서비스나 결제 대행사(PG) API를 그대로 호출하면, 사용자에게 문자 메시지가 두 번 가거나 실제 결제가 두 번 일어나는 '대형 사고'가 터진다. 섀도우 배포 시 외부 서비스 호출부는 반드시 **Mock(가짜) 서버**로 교체해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **배포 성공률**: 대규모 아키텍처 변경 시 배포 성공률 **100% 육박**.
- **장애 인지**: 실제 운영 전 잠재적 버그 발견율 **95% 이상**.

### 미래 전망 및 진화 방향
섀도우 배포는 이제 **'지속적 검증(Continuous Verification)'** 단계로 넘어가고 있다. 단순히 배포 직전에만 쓰는 게 아니라, 항상 유령 서버를 띄워두고 운영 서버의 건강 상태를 체크하는 비교군으로 활용한다. 또한, AI가 두 버전의 결과 차이를 분석하여 "이 차이는 고의적인 개선인지, 아니면 실수인지"를 리포팅해주는 **'AI-powered Shadow Analysis'**가 데브옵스(DevOps)의 최종 진화형이 될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **섀도우 배포가 뭔가요?**: 무대에서 진짜 배우(기존 프로그램)가 연기하는 동안, 커튼 뒤에서 신입 배우(새 프로그램)가 똑같은 동작을 몰래 따라 해보는 거예요.
2. **왜 그렇게 하나요?**: 신입 배우가 무대 위에서 실수하면 관객들이 실망하니까, 관객 몰래 충분히 연습해서 완벽해졌을 때 주인공을 교체하기 위해서예요.
3. **가장 좋은 점은요?**: 신입이 엉뚱한 춤을 춰도 관객들은 절대 모르니까, 아무 걱정 없이 신입 배우를 실전처럼 훈련시킬 수 있답니다!
