# 576. 피처 플래그 (Feature Flag) - 코드 배포와 기능 노출의 분리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소스 코드를 수정하거나 다시 배포하지 않고도, 런타임에 특정 기능의 활성화 여부를 동적으로 제어할 수 있게 하는 **조건부 분기 기술**이다.
> 2. **가치**: 대규모 기능을 점진적으로 출시(Canary Release)하거나, 특정 타겟 그룹에게만 기능을 노출(A/B Test)하고, 장애 발생 시 즉시 기능을 끌 수 있는 **'킬 스위치(Kill Switch)'** 역할을 수행한다.
> 3. **융합**: 지속적 배포(CD), 가시성 확보를 위한 옵저버빌리티, 그리고 비즈니스 의사결정을 위한 데이터 분석 시스템과 융합되어 '실험 중심의 기민한 개발 문화'를 가능케 한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
피처 플래그(Feature Flag / Toggle)는 코드 내부에 `if (isFeatureEnabled)` 와 같은 조건문을 걸어두고, 이 조건의 참/거짓 값을 외부 설정 시스템(Config)에서 실시간으로 바꿔주는 기법이다. 이를 통해 "코드 배포 = 기능 출시"라는 공식을 깨뜨릴 수 있다.

### 💡 비유
피처 플래그는 **"신축 아파트의 전등 스위치와 중앙 통제실"**과 같다.
- **코드 배포**: 아파트 공사를 마치고 입주를 시작하는 것. 모든 방에 전등(신규 기능)이 설치되어 있다.
- **피처 플래그**: 입주자가 들어왔다고 해서 무조건 불을 켜는 게 아니라, 중앙 관리실에서 스위치를 올려야 불이 들어온다. 특정 동(VIP 고객)만 먼저 불을 켜줄 수도 있고, 전기 합선(장애)이 나면 즉시 해당 층의 스위치만 내릴 수도 있다.

### 등장 배경 및 발전 과정

#### 1. 거대해진 릴리즈의 위험성 (Big Bang Release)
수백 명의 개발자가 만든 기능을 한꺼번에 출시했다가 에러가 나면, 원인을 찾기도 힘들고 전체 롤백을 해야 하는 막대한 손실이 발생했다.

#### 2. 마틴 파울러와 데브옵스의 확산
마틴 파울러가 "Feature Toggle" 개념을 정리하면서, 코드 배포는 매일 하되 기능 활성화는 비즈니스팀이 결정하는 '디커플링' 전략이 구글, 넷플릭스 등 빅테크 기업의 필수 역량이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 피처 플래그의 4가지 주요 유형 (표)

| 유형 | 생명 주기 | 주체 | 용도 |
|------|-----------|------|------|
| **Release Toggles** | 단기 (수 주) | 개발자 | 미완성 코드의 안전한 배포 (트렁크 기반 개발) |
| **Experiment Toggles**| 중기 (수 개월)| 데이터팀 | **A/B 테스트**, 가설 검증 |
| **Ops Toggles** | 장기 (무기한) | 운영팀 | **킬 스위치 (Kill Switch)**, 부하 제어 |
| **Permission Toggles**| 장기 (무기한) | 비즈니스 | VIP 전용 기능, 유료 회원 전용 기능 제어 |

### 정교한 구조 다이어그램 (피처 플래그 관리 시스템)

```ascii
================================================================================
[ Feature Flag Management Architecture ]
================================================================================

  [ 🛠️ Flag Management UI ] ──▶ [ 🗄️ Central Flag Store (Redis/DB) ]
  (Admin: "Turn ON New-UI for 10%")           │
                                              │ (1) Real-time Sync
                                              ▼
  ┌────────────────────────────────────────────────────────────────────────────┐
  │  🏢 Application Microservices                                              │
  │                                                                            │
  │  ┌──────────────────┐      ┌───────────────────────────────────────────┐   │
  │  │ Local SDK / Agent│◀─────┤ if (flags.enabled("new-ui", user_id)) {   │   │
  │  │ (Flag Cache)     │      │    show_new_layout();                     │   │
  │  └──────────────────┘      │ } else {                                  │   │
  │                            │    show_old_layout();                     │   │
  │                            │ }                                         │   │
  │                            └───────────────────────────────────────────┘   │
  └──────────────┬─────────────────────────────────────────────────────────────┘
                 │
                 │ (2) Telemetry (Flag Exposure Data)
                 ▼
  [ 📊 Analytics Platform ] ──▶ "New-UI users buy 20% more coffee!" (Insight)
```

### 심층 동작 원리

#### ① 런타임 결정 로직
플래그 값은 앱이 시작될 때만 읽어오는 게 아니다. 사용자마다, 요청마다 실시간으로 계산된다. `user_id`의 해시값을 구하여 "해시값이 0~9인 사람(10%)만 참(True)"을 리턴하는 방식으로 정교한 타겟팅을 수행한다.

#### ② 킬 스위치 (Kill Switch) 매커니즘
새로 배포한 기능 때문에 DB 서버 CPU가 100%를 치기 시작하면, 엔지니어는 소스 코드 수정이나 재배포 없이 대시보드에서 딸깍하고 플래그를 **OFF**로 바꾼다. 즉시 트래픽이 구버전 로직으로 돌아가며 장애 상황이 종료된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 코드 릴리즈 vs 기능 릴리즈

| 비교 지표 | Code Deployment (배포) | Feature Release (출시) |
|-----------|------------------------|------------------------|
| **정의** | 코드를 운영 서버에 올리는 행위 | 사용자가 기능을 인지하고 쓰는 행위 |
| **담당자** | 개발자, 데브옵스 엔지니어 | **기획자, 마케팅, 비즈니스팀** |
| **위험성** | 높음 (시스템 중단 위험) | **낮음 (언제든 끄기 가능)** |
| **성공 기준** | 에러 로그 없음, 가동 유지 | **사용자 지표 향상, 비즈니스 가치** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 한 금융 앱의 '해외 송금' 로직을 대대적으로 개편했다. 테스트는 완벽하지만, 실운영 환경의 환율 API와 국가별 규제가 워낙 복잡해 한 번에 100% 전환하는 것은 너무 위험하다.
**판단 및 플래그 배포 전략**:
1. **Gradual Rollout**: 먼저 사내 직원(QA 그룹)에게만 플래그를 켠다.
2. **Expansion**: 장애가 없으면 일반 사용자의 1%, 5%, 10% 순으로 플래그 범위를 점진적으로 넓힌다.
3. **Observation**: 에러율 메트릭이 튈 경우 자동으로 플래그를 끄는 **'Auto-kill'** 스크립트를 연동한다.
4. **결과**: 배포 사고 발생 시 피해 범위를 극소수의 사용자(1%)로 제한하고, 롤백 시간 없이 즉각 복구하여 '무결점 출시'를 달성한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **부채 누적 (Flag Debt)**: 기능 출시가 끝났는데도 코드에 `if` 문을 지우지 않고 방치하면, 1년 뒤엔 이 코드가 왜 있는지 아무도 모르는 스파게티 코드가 된다. 반드시 **'플래그 제거(Clean-up)'**를 스프린트의 공식 작업으로 포함해야 한다.
- **성능 오버헤드**: 플래그 값을 알기 위해 매번 DB를 조회하면 시스템이 느려진다. 반드시 **로컬 캐시**를 활용하거나, 성능이 검증된 전문 솔루션(LaunchDarkly, Split.io 등)을 써야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **평균 복구 시간(MTTR)**: 장애 시 재배포(30분) 대신 플래그 조작으로 **1초 만에 복구**.
- **배포 빈도**: 배포의 두려움이 사라져 일일 배포 횟수 **수 배 이상 증가**.

### 미래 전망 및 진화 방향
피처 플래그는 이제 **'AI 실험 자동화'**로 진화하고 있다. 사람이 플래그 비중을 조절하는 게 아니라, 인공지능이 실시간 매출 지표를 보면서 "A 버전이 돈을 더 잘 버네"라고 판단하여 자동으로 트래픽을 몰아주는 **'Multi-armed Bandit'** 알고리즘이 적용되고 있다. 이는 개발자가 더 이상 운영에 신경 쓰지 않고 '설계'에만 집중하는 진정한 의미의 자율 주행 소프트웨어 시대를 열 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **피처 플래그가 뭔가요?**: 스마트폰 게임에 새로 나온 '마법 지팡이' 기능을 모든 친구에게 한꺼번에 주는 게 아니라, 일단 몇 명한테만 비밀 스위치를 켜서 써보게 하는 거예요.
2. **왜 그렇게 하나요?**: 마법 지팡이가 너무 강력해서 게임이 멈춰버리면 안 되니까, 미리 조금씩 테스트해보면서 안전한지 확인하는 거죠.
3. **가장 좋은 점은요?**: 지팡이가 고장 나면 앱을 다시 깔 필요 없이, 컴퓨터 대장님이 멀리서 버튼 하나만 "딸깍" 누르면 지팡이가 사라지고 예전 모습으로 금방 돌아온답니다!
