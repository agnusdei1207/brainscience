# 577. 서버 사이드 렌더링 (SSR) 컴포넌트 아키텍처 (Next.js 등)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 브라우저가 빈 HTML을 받아 자바스크립트로 화면을 그리는 대신, **서버에서 미리 데이터가 채워진 완성된 HTML을 생성하여 클라이언트로 전송**하는 렌더링 방식이다.
> 2. **가치**: 검색 엔진 최적화(SEO)를 극대화하고, 자바스크립트 실행 전에도 화면을 보여주어 **첫 페이지 로딩 속도(FCP)**를 획기적으로 개선하며 사용자 경험을 높인다.
> 3. **융합**: React/Vue 기반의 하이드레이션(Hydration) 기술, 에지 컴퓨팅(Edge Runtime), 그리고 최신 **서버 컴포넌트(RSC)** 아키텍처와 융합되어 고성능 웹 서비스의 필수 표준으로 자리 잡았다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
- **CSR (Client Side Rendering)**: 브라우저가 자바스크립트를 다운로드하고 실행한 후에야 화면이 보임. (빈 도화지 배달).
- **SSR (Server Side Rendering)**: 서버가 DB에서 데이터를 가져와 HTML을 완성해서 브라우저에 보냄. (완성된 그림 배달).

### 💡 비유
SSR은 **"완성된 요리를 배달받는 것"**과 같다.
- **CSR (밀키트)**: 식당에서 재료만 온다. 내가 직접 가스레인지를 켜고(JS 실행) 요리를 해야(렌더링) 먹을 수 있다. 요리 실력이 없거나 불이 약하면(성능 낮은 폰) 한참 뒤에야 먹는다.
- **SSR (완성된 짜장면)**: 배달 오자마자 바로 뚜껑만 열고 먹으면 된다. 서버(주방장)가 미리 다 만들어 보냈기 때문이다.

### 등장 배경 및 발전 과정

#### 1. SPA의 검색 차단 문제
React나 Vue로 만든 앱은 처음에 빈 `<div>`만 있어, 검색 로봇(Google bot 등)이 내용을 읽지 못해 검색 결과에 노출되지 않는 치명적 문제가 있었다.

#### 2. Next.js의 대중화
Vercel에서 만든 **Next.js**가 SSR 구현의 복잡함을 해결해주면서, 개발자들은 "개발은 React처럼 편하게, 배포는 검색 잘되는 SSR로" 하는 하이브리드 시대를 맞이하게 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### SSR의 동작 프로세스: 하이드레이션 (표)

서버에서 받은 HTML은 버튼을 눌러도 반응이 없다. 이를 '살아있는 앱'으로 만드는 과정이 필요하다.

| 단계 | 명칭 | 동작 내용 | 비유 |
|------|-----------|-----------|------|
| **1단계** | **Server Render** | 서버가 DB 조회 후 완성된 HTML 생성 및 전송 | 완성된 음식 배달 |
| **2단계** | **Visual Load** | 브라우저가 HTML을 즉시 화면에 그림 (눈에 보임) | 음식 뚜껑 열기 |
| **3단계** | **JS Bundle** | 브라우저가 앱 구동에 필요한 자바스크립트를 다운로드 | 젓가락 준비 |
| **4단계** | **Hydration** | **(핵심)** HTML에 자바스크립트 이벤트(클릭 등)를 연결 | 음식을 입에 넣기 |

### 정교한 구조 다이어그램 (렌더링 흐름 비교)

```ascii
================================================================================
[ SSR Process: From Server to Browser Interaction ]
================================================================================

  [ ☁️ Web Server (Node.js) ]             [ 🌐 Web Browser ]
           │                                      │
 (1) Request for /page                            │
           │                                      │
 (2) Fetch Data from DB                           │
 (3) Render React to String (HTML)                │
           │                                      │
 (4) Send HTML ──────────────────────────────────▶│ (5) 🖼️ First View! 
           │                                      │ (사용자는 이미 읽기 시작)
           │                                      │
 (6) Send JS Bundle ─────────────────────────────▶│ (7) ⚡️ Hydration
           │                                      │ (이벤트 연결 완료)
           │                                      │
           ▼                                      ▼
      [ Server Time ]                       [ Client Time ]

 * 효과: 사용자는 (5)번 시점에서 이미 화면을 보므로, (7)번까지 
   자바스크립트가 로드되는 동안의 지루한 '빈 화면'을 보지 않아도 됨.
```

### 심층 동작 원리

#### ① 데이터 선인출 (getServerSideProps)
Next.js 같은 프레임워크는 페이지를 그리기 전, 서버 단에서 먼저 실행되는 함수를 제공한다. 여기서 API를 호출하고 결과를 컴포넌트의 `props`로 넘긴다. 이 모든 과정이 서버 내부망에서 이루어지므로 클라이언트-서버 간 통신 횟수가 줄어든다.

#### ② 서버 컴포넌트 (React Server Components, RSC)
최신 트렌드다. 기존 SSR은 모든 컴포넌트의 자바스크립트를 클라이언트로 보냈지만, RSC는 **"오직 서버에서만 실행되고 결과(HTML 조각)만 보내는 컴포넌트"**를 정의할 수 있다. 클라이언트가 다운로드해야 할 자바스크립트 용량이 획기적으로 줄어든다.

## Ⅲ. 융합 비교 및 다각도 분석

### 렌더링 방식별 성능 지표 비교

| 비교 지표 | CSR (Client) | **SSR (Server)** | SSG (Static) |
|-----------|---------------|-------------------|---------------|
| **FCP (첫 화면)**| 느림 | **빠름** | **매우 빠름** |
| **SEO (검색)** | 불리 (일부 가능) | **매우 유리** | **매우 유리** |
| **서버 부하** | 낮음 | **높음 (매번 렌더링)** | 매우 낮음 |
| **데이터 신선도**| 실시간 | **실시간** | 낮음 (빌드 시점) |
| **추천 용도** | 어드민, 대시보드 | **뉴스, 커머스, SNS** | 블로그, 회사 소개 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 한 패션 이커머스 쇼핑몰. React 기반의 CSR로 개발했는데, 네이버/구글 검색에 상품이 하나도 노출되지 않고, 첫 페이지 로딩 시 흰 화면이 2초간 지속되어 이탈률이 40%에 달한다.
**판단 및 SSR 전환 전략**:
1. **Framework 전환**: 순수 React 대신 **Next.js** 프레임워크를 도입한다.
2. **Dynamic SSR**: 상품 상세 페이지와 기획전 페이지는 매번 데이터가 바뀌므로 `getServerSideProps`를 사용해 SSR을 적용한다.
3. **Streaming SSR**: 페이지 전체가 로딩될 때까지 기다리지 않고, 헤더와 상단 이미지를 먼저 보내주는 **Streaming** 기법을 적용한다.
4. **결과**: 검색 엔진 노출 상품 수가 0에서 100만 개로 늘어나고, 첫 화면 로딩 속도가 2초에서 **0.5초**로 단축되어 매출이 30% 상승한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **과도한 SSR 의존**: 모든 페이지를 SSR로 만들면 서버 CPU가 폭주하여 인프라 비용이 감당이 안 된다. 로그인이 필요한 '마이페이지'나 개인화된 '설정' 페이지는 굳이 검색될 필요가 없으므로 **CSR**로 남겨두는 전략이 경제적이다.
- **Window/Document 객체 참조**: 서버는 브라우저가 아니다. 코드 내부에 `window`나 `localStorage`를 직접 쓰면 서버에서 에러가 난다. 반드시 `useEffect` 내부에서 실행하거나 브라우저 여부를 체크해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **SEO 순위**: 핵심 키워드 검색 노출 확률 **200% 향상**.
- **전환율(Conversion)**: 페이지 로딩 속도 개선으로 결제 완료율 **15~20% 상승**.

### 미래 전망 및 진화 방향
SSR은 이제 **'증분 정적 재생성 (ISR)'** 및 **'에지 렌더링 (Edge Rendering)'**과 결합하고 있다. 전 세계 흩어진 CDN 서버(Vercel Edge, Cloudflare Workers)에서 사용자와 가장 가까운 곳에서 HTML을 조립해 보내주는 방식이다. 미래의 웹 아키텍처는 서버와 클라이언트의 경계가 완전히 허물어지고, 어느 시점에 어느 위치에서 렌더링하는 것이 가장 효율적인지 AI가 판단하여 최적화하는 **'Universal Rendering'** 시대로 진화할 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **SSR이 뭔가요?**: 유튜브 영상을 볼 때, 내가 직접 색칠 공부를 해서 화면을 만드는 게 아니라 방송국에서 미리 예쁘게 색칠된 만화 영화를 통째로 보내주는 거예요.
2. **왜 좋나요?**: 내가 색칠할 필요가 없어서 눈을 뜨자마자(페이지 접속하자마자) 바로 만화를 볼 수 있기 때문이에요.
3. **가장 좋은 점은요?**: 검색 로봇 친구들도 색칠 안 된 종이보다는 완성된 그림을 좋아해서, 우리 만화가 유명해지기 훨씬 쉽답니다!
