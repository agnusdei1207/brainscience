# 578. 정적 사이트 생성 (SSG) / 증분 정적 재생성 (ISR) 패턴

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 빌드 타임(Build-time)에 미리 모든 페이지의 HTML을 생성하여 CDN에 배포하는 **SSG(Static Site Generation)**와, 서비스 운영 중에 특정 주기로 정적 페이지를 백그라운드에서 갱신하는 **ISR(Incremental Static Regeneration)** 기술의 조합이다.
> 2. **가치**: 서버의 실시간 렌더링 부하를 0으로 만들면서도 CDN을 통한 초광속 페이지 로딩 속도를 보장하며, 데이터가 변할 때마다 전체 사이트를 다시 빌드하지 않고도 최신 상태를 유지하는 '유연성'을 제공한다.
> 3. **융합**: 글로벌 Edge 캐싱(Vercel, Cloudflare), Next.js의 데이터 페칭 전략, 그리고 정적 자산(Static Assets) 관리 기술이 융합되어 현대적 웹 배포의 완성형 아키텍처를 이룬다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
- **SSG (Static Site Generation)**: "미리 그려놓기". 빌드할 때 서버에서 데이터를 다 가져와서 수천 장의 HTML 파일을 미리 만들어두는 방식이다.
- **ISR (Incremental Static Regeneration)**: "필요할 때 다시 그리기". SSG로 만든 파일이 너무 오래됐을 때, 사용자의 요청이 들어오면 백그라운드에서 조용히 새 데이터를 가져와 HTML을 업데이트하는 방식이다.

### 💡 비유
SSG와 ISR은 **"도시락 가게의 운영 방식"**과 같다.
- **SSR (주문 즉시 조리)**: 손님이 오면 그때서야 고기를 굽는다. 맛은 신선하지만 시간이 걸린다.
- **SSG (미리 포장된 도시락)**: 아침에 도시락 100개를 미리 싸둔다. 손님은 오자마자 들고 가면 된다. (광속). 하지만 오후에 온 손님은 식은 밥을 먹어야 한다.
- **ISR (주기적 신상 교체)**: 도시락을 미리 싸두되, 1시간마다 주방장이 "이거 좀 오래됐네?" 하고 새 도시락으로 바꿔둔다. 손님은 여전히 1초 만에 도시락을 받으면서도, 비교적 따끈한(최신 데이터) 음식을 먹을 수 있다.

### 등장 배경 및 발전 과정

#### 1. 빌드 시간의 저주
SSG는 좋지만, 상품이 100만 개인 쇼핑몰은 빌드 한 번에 10시간이 걸린다. 상품 정보 하나 고칠 때마다 10시간을 기다리는 것은 불가능했다.

#### 2. Next.js의 ISR 혁신
Next.js는 "전체 빌드 대신, 페이지별로 유효기간(revalidate)을 두자"는 ISR을 제안했다. 이로써 **"정적 페이지의 속도"**와 **"동적 데이터의 신선도"**라는 두 마리 토끼를 잡는 데 성공했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### ISR의 동작 프로세스: Stale-While-Revalidate (표)

사용자가 페이지를 요청했을 때, 서버 내부에서 벌어지는 일은 다음과 같다.

| 단계 | 상황 | 서버 응답 내용 | 결과 |
|------|-----------|----------------|------|
| **1단계** | 유효기간 내 요청 | 캐시된(이미 빌드된) HTML 즉시 반환 | **최상의 속도** |
| **2단계** | 유효기간 만료 후 첫 요청 | **과거의 HTML**을 일단 보여줌 (Stale) | 지연 시간 없음 |
| **3단계** | 백그라운드 작업 | 서버가 몰래 데이터를 새로 받아 HTML 재생성 | 업데이트 수행 |
| **4단계** | 이후 요청 | **새로 생성된 HTML**을 반환 | 데이터 최신화 완료 |

### 정교한 구조 다이어그램 (SSG vs ISR 흐름)

```ascii
================================================================================
[ SSG and ISR Architecture: The Power of Cache ]
================================================================================

  [ 🛠️ Developer Build ] ──(Build Once)──▶ [ 📂 Static Storage (S3/Vercel) ]
                                                   │
                                                   ▼ (Push to CDN)
  [ 🌐 CDN Edge Nodes ] ◀────────────────── [ 🚀 Global Delivery ]
           │                                       │
           │ (1) Request (T=0)                     │ (3) Revalidate Request (T=61s)
           ▼                                       │
  ┌────────────────────────┐              ┌────────▼────────────────┐
  │  Cache Hit: HTML V1    │              │  🔥 Next.js Server      │
  │  (Zero Latency)        │              │  - Fetch New Data       │
  └────────────────────────┘              │  - Generate HTML V2     │
                                          └────────┬────────────────┘
                                                   │
                                                   ▼ (4) Update Cache
                                          ┌────────────────────────┐
                                          │  CDN: Cache = HTML V2  │
                                          └────────────────────────┘

 * 핵심: 사용자는 갱신 중에도 기다리지 않는다. V1을 보다가 
   다음 새로고침 때 자연스럽게 V2를 보게 된다.
```

### 심층 동작 원리

#### ① 빌드 타임 프리렌더링 (GetStaticPaths)
SSG를 하려면 서버에게 "어떤 페이지들을 미리 만들까?" 알려줘야 한다. 상품 ID 목록을 쫙 뽑아주면, 서버는 수천 번의 렌더링을 미리 수행하여 HTML 뭉치를 만든다.

#### ② On-demand Revalidation (최신 트렌드)
시간(60초 등)이 아니라, 데이터가 바뀐 정확한 시점에 페이지를 갱신하고 싶을 때 쓴다. CMS에서 상품 가격을 수정하고 '저장'을 누르면, 서버로 웹훅(Webhook)을 쏴서 해당 페이지의 캐시만 즉시 폭파하고 다시 그리게 만드는 **'수동 ISR'** 방식이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 렌더링 전략 결정 매트릭스

| 비즈니스 특성 | 추천 전략 | 이유 |
|---------------|-----------|------|
| **변하지 않는 정보** (회사소개, 약관) | **SSG** | 한 번 만들면 평생 쓴다. 서버 부하 0. |
| **주기적 변경 정보** (블로그, 뉴스) | **ISR** | 데이터가 가끔 바뀌지만, 1초를 다투진 않는다. |
| **개인화 정보** (장바구니, 내 정보) | **CSR / SSR** | 사람마다 내용이 달라 미리 만들 수 없다. |
| **재고 민감 정보** (선착순 티켓팅) | **SSR** | 1초라도 틀리면 큰일 난다. 무조건 실시간. |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 상품 수가 500만 개인 초대형 해외 직구 사이트. 모든 상품을 SSG로 빌드하니 빌드 시간이 20시간이 걸려 배포가 불가능하다. 그렇다고 SSR을 쓰자니 전 세계 사용자 응답 속도가 너무 느리다.
**판단 및 하이브리드 전략**:
1. **Fallback: blocking 설정**: 모든 상품을 미리 빌드하지 않는다. **상위 인기 상품 1,000개만 빌드 타임에 생성(SSG)**한다.
2. **Dynamic Generation**: 나머지 4,999,000개 상품은 사용자가 처음 클릭할 때만 서버가 실시간으로 만들어서 보여주고, 그 결과물을 캐시에 저장한다. (Lazy SSG).
3. **ISR 적용**: revalidate 주기를 1시간으로 설정하여 환율 변동 등을 반영한다.
4. **결과**: 빌드 시간은 20시간에서 **5분**으로 단축되고, 인기 상품은 0.01초 만에, 비인기 상품도 두 번째 방문객부터는 0.01초 만에 뜨는 최적의 시스템을 구축한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **너무 짧은 revalidate 기간**: 주기를 1초로 잡으면 사실상 SSR과 다를 바 없다. 서버 부하만 늘리고 캐시의 이점을 못 누린다. 최소 수 분 이상의 주기를 권장한다.
- **메모리 고갈 위험**: ISR로 생성된 페이지는 서버나 CDN 메모리를 차지한다. 수억 개의 페이지를 ISR로 만들면 저장소 비용이 폭증할 수 있으므로, 적절한 **LRU(Least Recently Used) 캐시 삭제 정책**이 필요하다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **인프라 비용**: SSR 대비 서버 CPU 사용량 **80% 이상 절감**.
- **Lighthouse 성능 점수**: TTI(Time to Interactive) 및 FCP 지표에서 만점(100점) 달성 가능.

### 미래 전망 및 진화 방향
SSG/ISR은 이제 **'컴포넌트 단위 ISR'**로 세분화되고 있다. 페이지 전체를 갱신하는 게 아니라, 페이지 내의 '날씨 정보'나 '주가 지수' 컴포넌트만 따로 유효기간을 두어 갱신하는 방식이다. 또한, 데이터베이스의 변경(CDC)을 감지하여 실시간으로 에지 서버의 정적 파일을 교체하는 **'Real-time Static Web'** 기술이 웹 아키텍처의 종착역이 될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **SSG가 뭔가요?**: 내일 소풍 가서 먹을 김밥을 오늘 밤에 미리 다 싸서 냉장고에 넣어두는 거예요. 소풍 가서 바로 꺼내 먹기만 하면 되니까 아주 편하죠!
2. **ISR은요?**: 김밥이 너무 오래되면 맛없으니까, 편의점 아저씨가 중간중간 김밥이 팔릴 때마다 새 김밥을 조용히 가져다 놓는 마법 같은 냉장고예요.
3. **가장 좋은 점은요?**: 김밥을 사려고 줄을 설 필요가 전혀 없고(속도), 항상 비교적 신선한 김밥(데이터)을 먹을 수 있다는 점이랍니다!
