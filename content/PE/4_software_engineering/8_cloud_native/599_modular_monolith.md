# 599. 모듈러 모놀리스 (Modular Monolith) 아키텍처 - MSA 전환의 가교

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템을 하나의 배포 단위(Single Deployment Unit)로 유지하되, 내부 소스 코드는 **비즈니스 도메인에 따라 엄격하게 분리된 모듈(Module)들로 구조화**하여 모듈 간 결합도를 낮춘 아키텍처이다.
> 2. **가치**: 마이크로서비스(MSA)의 복잡한 네트워크 비용과 분산 트랜잭션 오버헤드 없이도 코드의 유지보수성과 확장성을 확보하며, 향후 필요 시 특정 모듈을 마이크로서비스로 쉽게 떼어낼 수 있는 유연성을 제공한다.
> 3. **융합**: 도메인 주도 설계(DDD)의 **바운디드 컨텍스트(Bounded Context)**, 모듈 간의 명확한 인터페이스(Public API), 그리고 데이터베이스의 논리적 스키마 분리 기술이 융합되어 구현된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
모듈러 모놀리스는 "겉은 하나인데 속은 여럿"인 구조다. 전체 프로그램은 하나의 서버로 실행되지만, 코드 내부에서는 주문 모듈, 결제 모듈, 회원 모듈이 서로 남의 집 살림을 간섭하지 못하도록 단단한 벽(Interface)을 치고 산다.

### 💡 비유
모듈러 모놀리스는 **"방이 완벽하게 분리된 쉐어하우스"**와 같다.
- **전통적 모놀리스 (한 방 취침)**: 거실 한복판에 10명이 누워 자는 것. 한 명이 코를 골면 모두가 잠을 못 잔다. (코드 스파게티).
- **MSA (각자 다른 집)**: 10명이 각자 다른 아파트에 사는 것. 안전하지만 서로 대화하려면 전화하고 약속을 잡아야 한다. (네트워크 오버헤드).
- **모듈러 모놀리스**: 큰 집 한 채(하나의 앱)를 빌려 살지만, 각자 **잠금장치가 있는 독립된 방(Module)**을 쓴다. 대화는 복도에서만 하고 방 안은 절대 못 본다. 나중에 돈을 벌면 그 방 그대로 떼어내서 독립(MSA 전환)하기도 쉽다.

### 등장 배경 및 발전 과정

#### 1. "무늬만 MSA"의 실패
많은 기업이 MSA를 도입했다가 너무 복잡한 네트워크 장애와 데이터 정합성 문제로 고통받았다. "우리 규모에 굳이 서버를 쪼개야 하나?"라는 반성이 시작되었다.

#### 2. 합리적인 대안의 부상
쇼피파이(Shopify) 같은 거대 기업이 "우리는 다시 모놀리스로 돌아가되, 코드만 철저히 분리하겠다"고 선언하며 모듈러 모놀리스가 현대 아키텍처의 강력한 후보로 부상했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 모놀리스 아키텍처의 3단계 진화 (표)

| 단계 | 아키텍처 명칭 | 특징 | 비유 |
|------|---------------|------|------|
| **1단계** | **Spaghetti Monolith** | 모든 코드가 엉켜있어 수정이 무서움 | 실타래 뭉치 |
| **2단계** | **Layered Monolith** | UI-Service-DAO 계층은 나눴지만 도메인은 섞임 | 3층 샌드위치 |
| **3단계** | **Modular Monolith** | **도메인별로 수직적인 격벽을 세움** | 칸막이 도시락 |

### 정교한 구조 다이어그램 (Modular Monolith Internal)

```ascii
================================================================================
[ Modular Monolith: Logical Separation inside Single Runtime ]
================================================================================

  ┌──────────────────────────────────────────────────────────┐
  │  🚀 Shared Runtime (Single JVM / Node.js / Go Process)   │
  │                                                          │
  │  ┌──────────────┐      ┌──────────────┐      ┌──────────┐│
  │  │ 📦 Order     │      │ 📦 Payment   │      │ 📦 Stock ││
  │  │   Module     │      │   Module     │      │   Module ││
  │  └──────┬───────┘      └──────┬───────┘      └──────┬───┘│
  │         │                     │                     │    │
  │         └────────── [ 📜 API / Interface ] ─────────┘    │
  │                     (In-memory Calls only)               │
  └─────────────────────────────┬────────────────────────────┘
                                │
  ══════════════════════════════╪══════════════════════════════
                                │
  ┌─────────────────────────────▼────────────────────────────┐
  │  🗄️ Database (Shared Server, Isolated Schemas)           │
  │  ┌──────────────┐      ┌──────────────┐      ┌──────────┐│
  │  │ Schema: ORDER│      │ Schema: PAY  │      │ Schema:ST││
  │  └──────────────┘      └──────────────┘      └──────────┘│
  └──────────────────────────────────────────────────────────┘

 * 핵심: 모듈 간 대화는 오직 정의된 인터페이스로만 수행함. 
   Stock 모듈의 DB 테이블을 Order 모듈이 직접 조회하면 "반칙"임.
```

### 심층 동작 원리

#### ① 캡슐화와 공개 인터페이스 (Encapsulation)
각 모듈은 자신의 내부 구현(Internal classes)을 숨기고, 외부에서 호출할 수 있는 최소한의 입구(Public API)만 연다. 자바의 `module-info`나 패키지 접근 제어자를 활용해 물리적으로 선을 긋는다.

#### ② 인메모리 이벤트 (Internal Events)
모듈 간 통신을 위해 Kafka 같은 무거운 장비 대신, 프로세스 내부의 **'이벤트 버스'**를 쓴다. "주문 완료됨" 이벤트를 발행하면 결제 모듈이 이를 낚아채서 처리한다. 나중에 MSA로 갈 때는 이 통로만 Kafka로 갈아 끼우면 된다.

#### ③ 데이터베이스 논리적 격리
DB 서버는 하나지만, 테이블끼리 외래키(FK)를 맺지 않는다. 데이터 무결성은 앱 레벨에서 책임지며, 각 모듈이 자기 스키마만 관리하게 함으로써 데이터 차원의 결합도까지 끊어낸다.

## Ⅲ. 융합 비교 및 다각도 분석

### 모듈러 모놀리스 vs 마이크로서비스(MSA)

| 비교 지표 | Modular Monolith | **Microservices (MSA)** |
|-----------|------------------|-------------------------|
| **배포 단위** | 1개 (단일 아티팩트) | **서비스 개수만큼 (다수)** |
| **통신 비용** | 0 (함수 호출) | **높음 (REST / gRPC)** |
| **트랜잭션** | 쉬움 (Local DB) | **어려움 (Saga / 2PC)** |
| **기술 스택** | 통일됨 | **다양함 (Polyglot 가능)** |
| **운영 난이도** | 낮음 | **매우 높음 (K8s 필수)** |
| **추천 환경** | **성장 중인 스타트업** | 초거대 서비스, 복잡한 조직 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 신규 쇼핑몰 프로젝트 초기 단계. 경영진은 "나중에 크게 될 거니까 무조건 MSA로 가자"고 한다. 하지만 개발팀은 5명뿐이고, 인프라 운영 전문가도 없다.
**판단 및 아키텍처 전략**:
1. **모듈러 모놀리스 선제 도입**: 처음부터 서버를 쪼개지 말고, **Spring Boot** 하나의 프로젝트 안에서 도메인별로 멀티 모듈 프로젝트를 구성한다.
2. **Strict Domain Boundary**: `Module A`가 `Module B`의 내부 데이터를 직접 건드리는 코드를 작성하면 빌드가 실패하도록 체크스타일(Checkstyle)이나 **ArchUnit** 테스트를 건다.
3. **Database separation**: DB는 하나로 시작하되, 테이블 접두사(Prefix)나 스키마를 철저히 나눠서 JOIN 쿼리를 원천 금지한다.
4. **결과**: MSA의 관리 비용 없이도 깔끔한 코드를 유지하며 빠르게 런칭한다. 1년 뒤 트래픽이 폭주하면, 그때 가장 부하가 심한 '주문 모듈'만 톡 떼어내서 별도 서버로 배포한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **가짜 모듈성 (Modular-ish)**: 폴더 구조만 나눠놓고 서로의 클래스를 마음대로 `import` 해서 쓰면, 결국 거대한 스파게티가 된다. 반드시 **빌드 도구(Gradle/Maven)** 레벨에서 의존성을 제어해야 한다.

---

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **개발 생산성**: 분산 환경 디버깅 시간 감소로 전체 개발 효율 **30% 향상**.
- **운영 안정성**: 네트워크 지연에 의한 타임아웃 오류 **0%** (모든 통신이 로컬이므로).

### 미래 전망 및 진화 방향
모듈러 모놀리스는 이제 **'지속 가능한 아키텍처'**의 표준이 되고 있다. 무조건 쪼개는 것이 능사가 아니라, 비즈니스 성숙도에 맞춰 모놀리스와 MSA 사이를 유연하게 오가는 기술이 핵심이다. 최근에는 **'컴포저블 아키텍처 (Composable)'**와 결합하여, 모듈들을 부품처럼 갈아 끼우며 비즈니스 변화에 대응하는 '레고 블록형 기업 인프라'의 근간이 될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **모듈러 모놀리스가 뭔가요?**: 커다란 장난감 상자(서버) 하나에 모든 장난감을 다 때려 넣는 게 아니라, 상자 안에 작은 칸막이(모듈)를 만들어서 로봇은 로봇끼리, 인형은 인형끼리 따로 정리하는 거예요.
2. **왜 좋나요?**: 상자가 하나라 들고 다니기 편하고(배포 쉬움), 안이 잘 정리되어 있어서 내가 원하는 장난감을 1초 만에 찾을 수 있기 때문이에요.
3. **나중에는요?**: 나중에 로봇이 너무 많아지면, 로봇 칸만 쏙 빼서 더 큰 로봇 전용 상자(MSA)로 옮겨주기도 아주 쉽답니다!
