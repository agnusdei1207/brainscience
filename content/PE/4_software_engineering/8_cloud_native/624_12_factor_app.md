# 624. 클라우드 네이티브 12 Factor App - 현대적 소프트웨어의 12가지 생존 법칙

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Heroku의 엔지니어들이 정립한 방법론으로, 클라우드 환경에서 **확장성(Scalability)**이 뛰어나고 **이식성(Portability)**이 높으며, 지속적 배포(CD)에 최적화된 SaaS 애플리케이션을 만들기 위한 12가지 핵심 설계 원칙이다.
> 2. **혁신**: 소스 코드와 설정을 엄격히 분리하고, 서버의 상태를 유지하지 않는 **무상태(Stateless) 프로세스** 지향하며, 로컬 환경과 운영 환경의 격차를 최소화하여 '어디서든 돌아가는' 소프트웨어를 지향한다.
> 3. **가치**: 마이크로서비스 아키텍처(MSA)와 컨테이너(Docker/K8s) 환경에서 애플리케이션이 복잡해지더라도 안정적으로 운영될 수 있는 인프라적 기초 체력을 제공한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
12 Factor App은 "어떻게 하면 클라우드에서 가장 행복하게(안정적으로) 작동하는 앱을 만들 수 있을까?"에 대한 모범 답안지다. 특정 프로그래밍 언어나 프레임워크에 종속되지 않는 범용적인 아키텍처 가이드라인을 제시한다.

### 💡 비유
- **전통적 앱 (고급 맞춤 정장)**: 특정 고객(내 서버 하드웨어)의 몸에 딱 맞춰 제작된다. 다른 사람(다른 서버)이 입으려면 수선을 엄청나게 해야 한다.
- **12 Factor App (유니클로 티셔츠)**: 규격화되어 대량 생산된다. 누가 입어도 잘 맞고(이식성), 한 벌이 찢어지면(서버 장애) 바로 새 티셔츠를 꺼내 입으면 된다(확장성). 세탁법(배포 방식)도 전 세계 어디나 똑같다.

### 등장 배경: 'It works on my machine'의 저주
개발자 PC에서는 잘 돌아가는데 운영 서버에만 올리면 죽는 고질적인 문제를 해결해야 했다. 또한, 트래픽이 몰릴 때 서버 사양을 높이는(Scale-up) 것이 아니라 서버 대수를 늘리는(Scale-out) 방식이 클라우드의 대세가 되면서, 이에 최적화된 새로운 설계 표준이 필요해졌다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 12가지 원칙 심층 분석 (Table)

| 번호 | 원칙 (Factor) | 핵심 내용 | 실천 방안 |
|:---:|:---|:---|:---|
| **1** | **코드베이스 (Codebase)** | 버전 관리되는 하나의 코드, 여러 번 배포 | Git 리포지토리 하나가 하나의 앱 담당 |
| **2** | **의존성 (Dependencies)** | 전체 의존성을 명시적으로 선언 및 격리 | `package.json`, `requirements.txt` 사용 |
| **3** | **설정 (Config)** | 설정값(DB 주소 등)을 환경 변수에 저장 | 코드에 비번/주소 하드코딩 금지 |
| **4** | **백엔드 서비스 (Backing Services)**| 리소스를 연결된 자원으로 취급 | DB, 메시지 큐를 로컬이 아닌 URL로 연결 |
| **5** | **빌드, 릴리스, 실행** | 3단계를 엄격히 분리 | 빌드된 바이너리는 수정 없이 실행만 함 |
| **6** | **프로세스 (Processes)** | 앱을 하나 이상의 무상태 프로세스로 실행 | **데이터는 외부에 저장 (Stateless)** |
| **7** | **포트 바인딩 (Port binding)** | 서비스를 포트에 바인딩하여 공개 | 웹 서버를 직접 내장하여 스스로 실행 |
| **8** | **동시성 (Concurrency)** | 프로세스 모델을 통해 확장 | 서버 대수(Scale-out)를 늘려 부하 분산 |
| **9** | **폐기 가능성 (Disposability)** | 빠른 시작과 우아한 종료 | 끄고 켜는 것이 빨라야 장애 복구 용이 |
| **10**| **환경 일관성 (Dev/Prod Parity)**| 개발, 테스트, 운영 환경을 유사하게 유지 | 도커(Docker)를 통한 환경 통일 |
| **11**| **로그 (Logs)** | 로그를 이벤트 스트림으로 취급 | 파일 저장이 아닌 표준 출력(stdout)으로 쏨 |
| **12**| **Admin 프로세스** | 관리 작업을 일회성 프로세스로 실행 | DB 마이그레이션 등을 독립 스크립트로 처리 |

## Ⅲ. 융합 비교 및 다각도 분석

### 6번 원칙(Stateless)과 9번 원칙(Disposability)의 시너지

12 Factor App의 정수는 **"언제든 죽여도 상관없는 앱"**을 만드는 데 있다.

```ascii
[ User ] ----> [ Load Balancer ]
                   │
       ┌───────────┴───────────┐
       ▼ (Stateless)           ▼ (Stateless)
┌──────────────┐        ┌──────────────┐
│ App Instance │        │ App Instance │  <-- 8번: 동시성 (수평 확장)
│ (Process A)  │        │ (Process B)  │
└──────┬───────┘        └──────┬───────┘
       │                       │
       └───────────┬───────────┘
                   ▼ (4번: Backing Service)
        ┌─────────────────────┐
        │ External Storage    │  <-- 모든 상태(세션, 이미지)는
        │ (Redis, S3, RDS)    │      여기에 저장해야 함!
        └─────────────────────┘
```

- **상태가 없으므로 (Stateless)**: 인스턴스가 갑자기 죽어도 다른 인스턴스가 즉시 업무를 이어받을 수 있다.
- **폐기 가능하므로 (Disposability)**: 업데이트할 때 구형 서버를 끄고 신형 서버를 켜는 과정(Rolling Update)이 매우 매끄럽다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 클라우드 네이티브 전환 전략)

**상황**: 사내 낡은 Java 유료 웹 서버(Jeus/Weblogic) 위에서 돌아가는 게시판 앱을 AWS 쿠버네티스(EKS)로 옮기려 한다. 현재 세션 정보는 서버 메모리에 저장되고 있으며, DB 주소는 XML 파일에 하드코딩되어 있다.
**판단 및 12 Factor 기반 리팩토링 전략**:
1. **Factor 3 (Config)**: XML 파일의 DB 주소를 삭제하고, 쿠버네티스의 `ConfigMap`과 `Secret`을 통해 환경 변수로 주입받도록 코드를 수정한다.
2. **Factor 6 (Processes)**: 서버 메모리에 저장하던 세션 정보를 외부 캐시인 **Redis**로 옮긴다. 이로써 서버가 여러 대로 늘어나도 로그인이 풀리지 않게 만든다.
3. **Factor 11 (Logs)**: 서버 폴더 안에 `.log` 파일을 만들던 로직을 지우고, 단순히 `System.out.println`으로 쏘게 바꾼다. 로그 수집은 인프라(Fluentd)에 맡긴다.
4. **결과**: 서버 1대에서만 돌던 불안한 앱이, 트래픽에 따라 100대로 늘어났다 줄어들어도 완벽히 동작하는 무적의 클라우드 앱으로 탈바꿈함.

### 안티패턴 및 고려사항
- **로컬 파일 시스템 의존**: 업로드한 이미지를 서버 로컬 디스크(`/upload`)에 저장하는 행위는 12 Factor의 최대 적이다. 서버가 재시작되거나 대수가 늘어나면 그 파일은 영원히 유실된다. 반드시 **Factor 4(Backing Services)**에 따라 S3 같은 외부 저장소를 써야 한다.

## Ⅴ. 미래 전망 및 결론

### 결론: 인프라로부터의 독립 선언
12 Factor App은 개발자가 서버의 물리적인 위치나 사양을 고민하지 않게 해 준다. 이 원칙을 지킨 앱은 어떠한 클라우드(AWS, Azure, On-premise) 위에서도 즉시 가동될 수 있는 자유를 얻는다.

### 미래 전망
현재 12 Factor는 진화하여 **'Beyond the 12-Factor App (15 Factor)'**로 확장되고 있다. 여기에는 'API 우선 설계', '철저한 보안', '관측 가능성(Observability)'이 추가되었다. 앞으로는 개발자가 이 12가지를 고민하지 않아도 프레임워크와 서비스 메시(Istio)가 인프라 레벨에서 자동으로 이 규칙들을 강제하고 보장해 주는 **'플랫폼 중심의 자동화'**가 더욱 가속화될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **12 Factor App이 뭔가요?**: 컴퓨터 프로그램을 만들 때, 어디에서나 고장 나지 않고 튼튼하게 잘 돌아가게 만드는 '12가지 마법의 규칙'이에요.
2. **왜 규칙을 지키나요?**: 이 규칙을 안 지키면 내 컴퓨터에선 잘 되던 게임이 친구 컴퓨터에선 안 될 수 있거든요. 이 규칙만 지키면 전 세계 모든 컴퓨터에서 똑같이 잘 된답니다!
3. **가장 중요한 규칙은?**: 프로그램 안에 중요한 비밀번호를 적어두지 말고, 프로그램이 꺼져도 중요한 정보는 바깥 일기장(외부 저장소)에 따로 적어두는 거예요!
