# 602. 정보 은닉 (Information Hiding) - 최소 지식의 미학

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클래스나 모듈 내부의 복잡한 구현 세부 사항(데이터 구조, 알고리즘 등)을 외부로 드러내지 않고, **공개된 인터페이스(Interface)를 통해서만 상호작용**하도록 제한하는 설계 원칙이다.
> 2. **혁신**: 내부 구현이 변경되더라도 외부 사용자(Client)의 코드를 수정할 필요가 없게 함으로써 시스템의 결합도를 낮추고, 데이터의 무결성을 보호한다.
> 3. **가치**: "복잡성을 감추는 것이 소프트웨어 설계의 핵심"이라는 데이비드 파나스(David Parnas)의 철학을 실현하며, 캡슐화(Encapsulation)를 통해 유지보수성과 확장성을 비약적으로 향상시킨다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
정보 은닉은 객체지향의 4대 핵심 원칙 중 하나인 캡슐화의 목적이자 수단이다. 객체의 내부 상태를 직접 조작하지 못하게 막고, 오직 객체가 제공하는 메서드를 통해서만 상태를 변경하거나 조회할 수 있게 하는 것을 의미한다.

### 💡 비유
- **정보 은닉이 없는 경우 (Translucent)**: 자동차의 보닛을 열고 엔진의 피스톤을 손으로 직접 밀어서 가속하는 방식이다. 엔진 구조를 모르면 운전할 수 없고, 엔진이 바뀌면 운전 방법도 다시 배워야 한다.
- **정보 은닉이 잘 된 경우 (Hidden)**: '가속 페달'이라는 간단한 인터페이스만 제공된다. 운전자는 엔진이 가솔린인지 전기인지 몰라도 페달만 밟으면 된다. 엔진 내부 설계가 바뀌어도 페달 밟는 법은 변하지 않는다.

### 등장 배경: 파나스의 논문 (1972)
데이비드 파나스는 "시스템을 모듈로 나눌 때, 각 모듈이 다른 모듈에 대해 가능한 한 적게 알도록 설계해야 한다"고 주장했다. 이는 시스템의 한 부분을 수정했을 때 도미노처럼 다른 부분이 무너지는 현상을 막기 위한 필수적인 고안이었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 정보 은닉의 메커니즘 (ASCII Diagram)

```ascii
[ Client Module ]               [ Target Object (Module) ]
        │                               ┌───────────────────────┐
        │       1. Method Call          │  Public Interface     │
        ├──────────────────────────────>│  (getter/setter/cmd)  │
        │                               └──────────┬────────────┘
        │                                          │ 2. Internal Access
        │      <Forbidden Access>       ┌──────────▼────────────┐
        │xxxxxxxxxxxxxxxxxxxxxxxxxxxxxx │  Private Data / Impl  │
        │      (Direct Data Access)     │  (Algorithms, State)  │
        └───────────────────────────────┴───────────────────────┘

* 원리: 인터페이스는 "무엇을 하는가(What)"만 보여주고, 
  내부 구현은 "어떻게 하는가(How)"를 철저히 숨김.
```

### 구현을 위한 3대 핵심 기법

#### ① 접근 제어자 (Access Modifiers)
- **private**: 클래스 내부에서만 접근 가능. (정보 은닉의 기본)
- **protected**: 상속 관계까지만 허용.
- **public**: 누구나 접근 가능. 최소한으로 유지해야 함.

#### ② 캡슐화 (Encapsulation)
데이터(속성)와 그 데이터를 다루는 행위(메서드)를 하나로 묶는 행위이다. 관련 있는 것들을 모으되, 알맹이는 숨기고 껍데기(인터페이스)만 내놓는다.

#### ③ 인터페이스 기반 설계 (Programming to Interface)
구체적인 클래스 타입 대신 인터페이스 타입을 사용함으로써, 실제 내부 구현체가 무엇인지 클라이언트가 알 필요 없게 만든다.

## Ⅲ. 융합 비교 및 다각도 분석

### 정보 은닉 vs 캡슐화 (구분 필수)

많은 개발자가 혼동하지만, 둘은 명확한 인과관계가 있다.

| 항목 | 캡슐화 (Encapsulation) | **정보 은닉 (Information Hiding)** |
|:---:|:---|:---|
| **정의** | 데이터와 기능을 하나로 묶는 것 | 비밀(구현 세부사항)을 숨기는 것 |
| **관점** | 데이터 중심의 응집화 | 보안과 결합도 중심의 제어 |
| **수단** | Class, Struct, Module | Private modifier, Interface, PIMPL |
| **관계** | **수단** (묶어야 숨길 수 있음) | **목적** (숨기기 위해 묶음) |

### 정보 은닉의 수준별 영향 (Table)

| 은닉 수준 | 가시성 (Visibility) | 결합도 (Coupling) | 변경 영향도 |
|-----------|----------------------|-------------------|-------------|
| **낮음** | 모든 필드가 Public | 매우 높음 (강결합) | 코드 한 줄 수정 시 전체 빌드/수정 |
| **중간** | Getter/Setter 사용 | 중간 | 데이터 구조 변경 시 Setter 수정 필요 |
| **높음** | 추상 인터페이스만 노출 | **매우 낮음 (약결합)** | **내부 로직 전체를 바꿔도 외부 영향 Zero** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 도입 전략)

**상황**: 은행 시스템에서 '계좌(Account)' 클래스의 '잔액(balance)' 필드가 public으로 되어 있어, 개발자들이 여기저기서 `account.balance += 1000` 처럼 직접 수정하고 있다. 로그 기록도 안 남고 무결성 검증도 안 된다.
**판단 및 개선 전략**:
1. **Field Isolation**: `balance`를 `private`으로 바꾸고 외부 접근을 즉시 차단한다. (컴파일 에러 발생 유도).
2. **Behavioral Interface**: `deposit(amount)`, `withdraw(amount)` 메서드를 만든다. 
3. **Internal Logic**: 해당 메서드 내부에서만 잔액을 수정하고, 동시에 '로그 기록', '마이너스 잔액 체크' 등 필수 로직을 수행한다.
4. **결과**: 잔액 조작의 모든 통로가 일원화되어 데이터 무결성이 완벽히 보장되며, 나중에 잔액 저장 방식이 DB에서 캐시로 바뀌어도 클라이언트 코드는 영향받지 않는다.

### 안티패턴: 무분별한 Getter/Setter 생성
- 모든 private 필드에 대해 기계적으로 public getter/setter를 만드는 것은 사실상 정보 은닉을 포기하는 것과 같다. (Anemic Domain Model).
- 객체에게 "데이터를 달라고 하지 말고, 일을 시켜라(Tell, Don't Ask)" 원칙을 지켜야 진정한 정보 은닉이 완성된다.

---

## Ⅴ. 미래 전망 및 결론

### 결론: 변경의 고통을 줄이는 방패
정보 은닉은 단순히 '비밀주의'를 고수하는 것이 아니라, 소프트웨어의 **'변경 탄력성'**을 확보하기 위한 전략적 선택이다. 잘 숨겨진 비밀은 개발자로 하여금 자유롭게 내부를 고치고 실험할 수 있는 안전한 환경을 제공한다.

### 미래 전망
최근 마이크로서비스(MSA) 환경에서는 정보 은닉이 네트워크 레벨로 확장되고 있다. 각 서비스는 내부 DB 스키마를 절대 공유하지 않고 오직 API 규약(Contract)만을 통해 소통한다. 이는 정보 은닉이 단순히 '클래스'의 문제를 넘어 **'현대 분산 시스템의 아키텍처적 근간'**임을 증명한다.

## 👶 어린이를 위한 3줄 비유 설명
1. **정보 은닉이 뭔가요?**: 텔레비전 리모컨의 버튼만 보여주고, 안쪽의 복잡한 전선과 칩은 뚜껑으로 꽉 닫아놓는 거예요.
2. **왜 뚜껑을 닫나요?**: 사용자가 전선을 잘못 건드려서 고장 내는 걸 막고, 리모컨 겉모양은 그대로인데 안쪽 칩이 더 좋은 걸로 바뀌어도 사용자는 전처럼 편하게 쓰게 하려고요.
3. **어떤 점이 좋나요?**: "복잡한 건 내가 알아서 할게, 넌 버튼만 눌러!"라고 말하는 똑똑하고 친절한 기계를 만드는 방법이랍니다!
