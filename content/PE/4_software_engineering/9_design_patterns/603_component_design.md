# 603. 컴포넌트 (Component) 설계 - 독립적 배포와 결합의 미학

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템의 전체 기능 중 특정 역할을 수행하도록 독립적으로 개발, 테스트, **배포(Deployment)될 수 있는 최소 단위의 소프트웨어 모듈**을 설계하는 과정이다.
> 2. **혁신**: 내부 구현은 철저히 숨기고(정보 은닉) 표준화된 인터페이스를 통해서만 소통하게 함으로써, 마치 레고 블록처럼 시스템을 조립하고 부분적으로 교체할 수 있게 한다.
> 3. **가치**: 대규모 시스템 개발에서 팀 간 병렬 개발을 가능하게 하고, 장애의 전파를 막으며, 특정 컴포넌트만 따로 업데이트하는 '지속적 배포(CD)' 환경의 근간이 된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
컴포넌트는 '독립적인 배포 단위'라는 점이 단순한 '클래스'나 '함수'와 다르다. 자바의 JAR, 닷넷의 DLL, 도커의 컨테이너 등이 현대적 의미의 컴포넌트 형태이다. 컴포넌트 설계는 이러한 덩어리들을 어떻게 나누고(Cohesion) 어떻게 연결할지(Coupling)를 결정하는 일이다.

### 💡 비유
- **일체형 설계 (Monolithic)**: 구형 라디오처럼 모든 전선과 칩이 기판 하나에 납땜 되어 있다. 스피커가 고장 나면 라디오 전체를 버려야 한다.
- **컴포넌트 설계 (Modular)**: 현대식 PC와 같다. 그래픽 카드, 메모리, CPU가 각각 독립적인 부품(Component)으로 존재한다. 그래픽 성능을 높이고 싶으면 메인보드는 그대로 두고 그래픽 카드만 빼서 새것으로 꽂으면 된다.

### 등장 배경: 'Software Crisis'와 재사용성
소프트웨어 규모가 커지면서 모든 코드를 한 번에 빌드하고 관리하는 것이 불가능해졌다. 1990년대 CBD(Component Based Development)가 유행하며 "잘 만든 부품을 시장에서 사서 조립하자"는 개념이 등장했고, 이는 현재의 오픈소스 라이브러리 생태계와 마이크로서비스(MSA)로 계승되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 컴포넌트의 논리적 구조 (ASCII Diagram)

```ascii
[ Component Boundary ]
┌──────────────────────────────────────────┐
│  Provided Interface (Export)             │
│  (What I can do for others)              │
│          ▲                               │
│  ┌───────┴───────┐   ┌────────────────┐  │
│  │ Internal Logic│<──┤ Required Intf. │  │
│  │ (Encapsulated)│   │ (Dependencies) │  │
│  └───────────────┘   └───────┬────────┘  │
│                              ▼           │
└──────────────────────────────────────────┘
                  (Call to other components)

* 핵심 원리: 인터페이스(껍데기)만 남기고 알맹이는 철저히 고립시킴.
```

### 컴포넌트 결합의 3대 원칙 (By Robert C. Martin)

#### ① ADP (Acyclic Dependencies Principle) - 의존성 비순환 원칙
컴포넌트 의존성 그래프에 순환(Cycle)이 있어서는 안 된다. A가 B에, B가 C에, C가 다시 A에 의존하면 하나만 고쳐도 셋 다 다시 빌드해야 하는 지옥이 펼쳐진다.

#### ② SDP (Stable Dependencies Principle) - 안정된 의존성 원칙
안정적인 컴포넌트(변경이 적은 것)가 불안정적인 컴포넌트에 의존해서는 안 된다. 더 많이 의존받는 놈일수록 더 안정적이어야 한다.

#### ③ SAP (Stable Abstractions Principle) - 안정된 추상화 원칙
안정적인 컴포넌트는 추상적이어야 하고, 불안정한 컴포넌트는 구체적이어야 한다. 즉, 중심이 되는 핵심 로직은 인터페이스 위주로 구성하여 변경에 유연하게 대처해야 한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 클래스 vs 컴포넌트 vs 서비스 비교

| 구분 | 클래스 (Class) | **컴포넌트 (Component)** | 서비스 (Service/Microservice) |
|-----------|----------------|--------------------------|-------------------------------|
| **단위** | 코드 레벨 (Source) | **바이너리/배포 레벨** | 네트워크/프로세스 레벨 |
| **결합** | 컴파일 타임 결합 | **링크 타임 결합 (DLL/JAR)** | 런타임 결합 (HTTP/gRPC) |
| **재사용성** | 낮음 (소스 복사 필요) | **높음 (파일 참조)** | 매우 높음 (API 호출) |
| **독립성** | 낮음 | **중간** | 매우 높음 (독립 DB 가능) |

### 컴포넌트 설계 시 고려해야 할 응집도 (Table)

| 원칙 | 상세 설명 | 비유 |
|------|-----------|------|
| **REP (Release/Reuse Equivalency)** | 재사용 단위는 배포 단위와 같아야 함 | 설명서와 부품은 한 상자에 담겨야 함 |
| **CCP (Common Closure Principle)** | 동일한 이유로 변하는 클래스들을 묶음 | 거실 물건은 거실함에, 주방 물건은 주방함에 |
| **CRP (Common Reuse Principle)** | 같이 사용되지 않는 클래스들은 분리함 | 숟가락 세트에 굳이 망치를 끼워 팔지 말 것 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 설계 전략)

**상황**: 결제, 배송, 재고 기능이 뒤섞인 대형 쇼핑몰 시스템이 자꾸 죽는다. 한 곳만 고쳐도 전체 서비스를 3시간 동안 내려야 한다.
**판단 및 컴포넌트화 전략**:
1. **관심사 분리**: 결제 엔진과 배송 추적 로직을 별도의 프로젝트(컴포넌트)로 나눈다.
2. **인터페이스 정의**: 결제 컴포넌트는 `processPayment()`라는 간결한 API만 노출하고 내부의 복잡한 카드사 연동 로직은 숨긴다.
3. **의존성 역전 (DIP)**: 배송 컴포넌트가 특정 결제 모듈에 직접 의존하지 않게 하고, 인터페이스를 통해 연결한다.
4. **결과**: 결제 로직에 버그가 생겨도 배송 서비스는 정상 작동하며, 전체 시스템 중단 없이 결제 모듈만 조용히 업데이트(Hot Swap)할 수 있게 됨.

### 안티패턴: 너무 잘게 쪼개기 (Nanocomponents)
- 컴포넌트를 너무 작게 쪼개면(예: 클래스 하나당 컴포넌트 하나), 컴포넌트 간 연결 관리 오버헤드가 배보다 배꼽이 더 커진다. (Wiring Complexity).
- **'적정 크기'**는 팀의 크기와 배포 주기, 변경의 빈도를 고려하여 결정해야 한다.

---

## Ⅴ. 미래 전망 및 결론

### 결론: 부품화된 세상의 생존법
컴포넌트 설계는 "모든 것을 내가 다 만들지 않겠다"는 겸손함에서 시작된다. 현대의 소프트웨어 개발은 직접 코드를 짜는 시간보다 이미 만들어진 컴포넌트를 골라 연결하는 설계 능력이 더 중요해지고 있다.

### 미래 전망
최근에는 컴포넌트의 개념이 **'서버리스 함수(FaaS)'**나 **'플러그형 아키텍처'**로 진화하고 있다. 앞으로는 AI가 각 기능 단위의 결합도를 실시간으로 분석하여 최적의 컴포넌트 경계를 제안하고, 장애 발생 시 자동으로 컴포넌트를 격리/복구하는 'Self-Adaptive Component' 시대가 열릴 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **컴포넌트 설계가 뭔가요?**: 아주 큰 성을 만들 때, 한꺼번에 쌓지 않고 '창문 덩어리', '성문 덩어리', '탑 덩어리'를 따로따로 만들어서 나중에 조립하는 거예요.
2. **왜 따로 만드나요?**: 성문이 고장 났을 때 성 전체를 부수지 않고 성문만 쏙 빼서 새것으로 바꾸기 위해서예요.
3. **가장 좋은 점은?**: 여러 명의 친구가 각자 자기가 맡은 덩어리만 만들어서 나중에 합치면, 아주 큰 성도 아주 빨리 만들 수 있답니다!
