# 605. 아키텍처 패턴: 계층형 (Layered Architecture) - 질서 정연한 책임의 수직 분할

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템을 서로 독립적인 역할을 수행하는 여러 개의 **수직적 계층(Layer)으로 분리**하여, 각 계층이 바로 아래 계층에만 의존하게 만드는 가장 대중적인 아키텍처 패턴이다.
> 2. **혁신**: 관심사 분리(Separation of Concerns)를 통해 특정 계층의 변경(예: DB 교체)이 다른 계층(예: UI 로직)에 영향을 주지 않도록 격리하여 유지보수성을 극대화한다.
> 3. **가치**: 구조가 단순하고 직관적이어서 소규모부터 대규모 프로젝트까지 범용적으로 적용 가능하며, 팀 간의 업무 분담(Frontend, Backend, DBA)을 명확하게 한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
계층형 아키텍처(Layered Architecture)는 소프트웨어를 구성하는 컴포넌트들을 논리적 단위인 '계층'으로 묶어 배치하는 방식이다. 상위 계층은 하위 계층이 제공하는 서비스를 이용하지만, 하위 계층은 상위 계층의 존재를 알지 못한다.

### 💡 비유
- **패턴 없는 코드 (Spaghetti)**: 요리사 한 명이 주문받기, 재료 손질, 요리, 서빙, 계산까지 혼자 다 하는 상황이다. 손님이 몰리면 정신이 없고 메뉴 하나만 바뀌어도 전체 과정이 꼬인다.
- **계층형 아키텍처 (Restaurant)**: 홀 직원(Presentation), 주방장(Business Logic), 창고 관리자(Data)로 역할이 나뉘어 있다. 주방장이 요리법을 바꿔도 홀 직원이 주문받는 방식은 변하지 않으며, 창고에 식재료가 어떻게 쌓여있는지 홀 직원은 알 필요가 없다.

### 등장 배경: 복잡성 관리의 필요성
초창기 소프트웨어는 UI와 로직, DB 접근 코드가 한데 섞여 있었다. 기능이 추가될수록 코드는 수정하기 불가능할 정도로 복잡해졌고, 이를 해결하기 위해 "비슷한 일끼리 모으고 층을 쌓자"는 계층화 원칙이 정립되었다.

---

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 표준 4계층 아키텍처 구조 (ASCII Diagram)

```ascii
┌──────────────────────────────────────────┐
│   1. Presentation Layer (UI/UX)          │  <--- 사용자 인터페이스 (화면, API 엔드포인트)
└────────────────────┬─────────────────────┘
                     │ (Call)
┌────────────────────▼─────────────────────┐
│   2. Business Layer (Service/Domain)     │  <--- 비즈니스 규칙, 알고리즘, 정책 수행
└────────────────────┬─────────────────────┘
                     │ (Call)
┌────────────────────▼─────────────────────┐
│   3. Persistence Layer (DAO/Repository)  │  <--- 데이터 조회 및 저장 로직 (ORM 등)
└────────────────────┬─────────────────────┘
                     │ (Query)
┌────────────────────▼─────────────────────┐
│   4. Database Layer (Actual Storage)     │  <--- 실제 데이터가 저장되는 곳
└──────────────────────────────────────────┘

* 원리: 상위 계층은 하위 계층에만 의존하며, 계층을 건너뛰는 행위는 원칙적으로 금지됨.
```

### 각 계층의 역할 상세

#### ① 표현 계층 (Presentation Layer)
사용자와의 접점이다. 웹 브라우저의 요청을 받고 결과를 화면(HTML/JSON)으로 돌려준다. 사용자 입력 값의 기본 검증(Validation)을 수행하지만, 비즈니스 로직은 포함하지 않는다.

#### ② 비즈니스 계층 (Business/Service Layer)
시스템의 핵심 가치를 담고 있는 곳이다. "할인율 적용", "결제 승인" 등 실제 비즈니스 규칙이 구현된다. 표현 계층에서 받은 데이터를 가공하여 영속성 계층으로 전달한다.

#### ③ 영속성 계층 (Persistence Layer)
데이터베이스와의 통신을 전담한다. SQL 쿼리를 날리거나 객체-관계 매핑(ORM)을 통해 데이터를 자바 객체 등으로 변환한다.

#### ④ 데이터베이스 계층 (Database Layer)
데이터가 물리적으로 저장되는 공간(MySQL, Oracle 등)이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 계층형 아키텍처의 장단점 분석 (Table)

| 장점 (Advantages) | 단점 (Disadvantages) |
|-------------------|----------------------|
| **추상화 및 격리**: 하위 구현을 몰라도 상위 개발 가능 | **대리자(Proxy) 오버헤드**: 단순 조회 시에도 여러 층을 거쳐야 함 |
| **유지보수 용이**: 특정 층만 교체하거나 수정하기 쉬움 | **유연성 부족**: 계층 구조에 묶여 복잡한 워크플로우 표현이 어려움 |
| **테스트 용이**: Mock 객체를 활용해 계층별 독립 테스트 가능 | **Sinkhole Anti-pattern**: 단순히 하위로 요청을 전달만 하는 코드가 많아짐 |
| **표준화**: 개발자들이 익숙한 구조라 협업이 빠름 | **확장 한계**: 거대해지면 결국 'Big Ball of Mud'가 될 수 있음 |

### 폐쇄형 계층(Closed Layer) vs 개방형 계층(Open Layer)
- **폐쇄형 (기본)**: 바로 아래 계층만 호출 가능. 가장 안전하고 격리가 확실함.
- **개방형**: 하위의 모든 계층 호출 가능. 예를 들어, 서비스 계층을 거치지 않고 UI에서 바로 DB 조회 기능을 쓸 때 사용하나, 의존성이 복잡해져 권장되지 않음.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 설계 전략)

**상황**: 스타트업에서 빠르게 MVP(최소 기능 제품)를 만들어야 한다. 개발자는 3명뿐이고 요구사항은 수시로 바뀔 예정이다.
**판단 및 아키텍처 전략**:
1. **패턴 선정**: 복잡한 클린 아키텍처나 MSA보다는 **전통적 계층형 아키텍처**를 선택한다. 러닝 커브가 낮고 빠르게 결과물을 낼 수 있기 때문이다.
2. **DTO 분리**: 표현 계층과 비즈니스 계층 사이에 데이터 전달 객체(DTO)를 두어, 내부 도메인 객체가 외부로 노출되지 않게 정보 은닉을 강화한다.
3. **인터페이스 활용**: 서비스 계층과 Repository 계층 사이에 인터페이스를 두어, 나중에 MySQL에서 MongoDB로 바꿔도 비즈니스 로직을 건드리지 않게 설계한다.
4. **결과**: 2주 만에 첫 버전을 출시하고, 이후 투자 유치에 따른 기능 확장 시에도 코드 엉킴 없이 안정적으로 대응함.

### 안티패턴: 레이어드 스파게티 (Layered Spaghetti)
- 편의를 위해 UI 로직 안에 SQL 쿼리를 직접 짜거나, 서비스 계층에서 HTTP 세션 정보를 직접 건드리는 행위는 계층형 아키텍처를 파괴하는 치명적인 안티패턴이다.

## Ⅴ. 미래 전망 및 결론

### 결론: 클래식은 영원하다
계층형 아키텍처는 가장 오래되었지만 여전히 가장 강력한 기본기다. 헥사고날(Hexagonal)이나 클린 아키텍처 같은 최신 기법들도 결국 '계층화'라는 대원칙 위에 세워진 변형들이다.

### 미래 전망
최근에는 클라우드 네이티브 환경에 맞춰 계층의 경계가 더 얇아지고 유연해지는 추세다. 하지만 시스템의 '질서'를 유지해야 하는 한, 계층형 아키텍처는 사라지지 않고 **'마이크로 계층'**이나 **'서버리스 계층'**의 형태로 끊임없이 진화하며 소프트웨어의 뼈대 역할을 할 전망이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **계층형 아키텍처가 뭔가요?**: 햄버거처럼 빵, 고기, 채소를 층층이 쌓아서 만드는 방법이에요. 각 층은 자기가 할 일만 열심히 하면 돼요.
2. **왜 층을 나누나요?**: 채소가 싫어서 빼고 싶을 때, 고기나 빵을 다 버리지 않고 채소 층만 살짝 바꾸기 위해서예요.
3. **뭐가 좋나요?**: 요리사들이 "나는 빵만 구울게!", "나는 고기만 구울게!"라고 역할을 나눠서 일할 수 있어서 아주 큰 햄버거도 쉽고 빠르게 만들 수 있답니다!
