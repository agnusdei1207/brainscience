# 607. 헥사고날 아키텍처 (Hexagonal Architecture) - 포트와 어댑터의 유연한 결합

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 비즈니스 로직(Core)을 중심에 두고 외부 요소(DB, UI, 외부 API 등)와 철저히 분리하여, **'포트(Port)'와 '어댑터(Adapter)'라는 인터페이스를 통해서만 소통**하게 만드는 아키텍처이다.
> 2. **혁신**: 의존성 역전 원칙(DIP)을 극대화하여 비즈니스 로직이 외부 기술 스택에 종속되지 않게 함으로써, DB나 프레임워크가 바뀌어도 핵심 비즈니스 코드는 수정 없이 그대로 유지된다.
> 3. **가치**: 핵심 도메인 로직을 외부 환경 없이도 독립적으로 테스트 가능하게 하며, 기술적 변화에 민첩하게 대응해야 하는 현대의 복잡한 엔터프라이즈 시스템과 MSA 환경에 최적화되어 있다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
헥사고날 아키텍처는 앨리스테어 코번(Alistair Cockburn)이 제안한 방식으로, '포트와 어댑터 아키텍처'라고도 불린다. 시스템을 내부(비즈니스 영역)와 외부(인프라 영역)로 명확히 나누고, 내부는 외부에 대해 전혀 모른 채 오직 규격화된 '포트'만 노출한다. 외부는 이 포트에 맞는 '어댑터'를 끼워 소통한다. 육각형(Hexagon) 모양으로 표현되는 이유는 여러 면에 다양한 어댑터를 끼울 수 있다는 대칭성과 유연성을 강조하기 위함이다.

### 💡 비유
- **가정용 콘셉트 (Hexagon)**: 우리 집 거실의 '벽면 콘센트'는 구멍(Port)이다. 여기에 선풍기를 꽂든, 냉장고를 꽂든, 스마트폰 충전기를 꽂든 전기가 공급되는 핵심 원리는 변하지 않는다.
- **어댑터 (Adapter)**: 가전제품의 플러그가 어댑터다. 한국형 220V 플러그든 미국형 110V 플러그든, 구멍(Port)에 맞는 변환 어댑터만 있으면 벽면 안쪽의 복잡한 전선 구조(Business Logic)를 건드리지 않고도 전기를 쓸 수 있다.

### 등장 배경: 'Layered'의 의존성 전염
전통적인 계층형 아키텍처는 비즈니스 계층이 하위의 DB 계층에 의존하는 경우가 많다. 이로 인해 DB 기술이 바뀌면 비즈니스 로직까지 흔들리는 '의존성 전염'이 발생했다. 이를 해결하고 "비즈니스 로직이 왕이다"라는 주권을 되찾기 위해 탄생했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 헥사고날 아키텍처 구성도 (ASCII Diagram)

```ascii
       [ Outside ] (Infrastructure)         [ Inside ] (Application Core)
┌─────────────────────────────────────────────────────────────────────────────┐
│   ┌───────────────┐                                                         │
│   │ Driver Adapter│ (Inbound)                                               │
│   │ (REST Controller)─────┐                                                 │
│   └───────────────┘       │      ┌─────────────────────────────┐            │
│                           ▼      │      [ Application Core ]   │            │
│                   ┌──────────────┤                             │            │
│                   │ Inbound Port │      ┌──────────────┐       │            │
│                   │ (Interface)  │<─────┤ Domain Logic │       │            │
│                   └──────────────┤      └──────────────┘       │            │
│                           ▲      │                             │            │
│                   ┌──────────────┤                             │            │
│                   │ Outbound Port│      (Dependency Flows IN)  │            │
│                   │ (Interface)  │<────────────────────────────┘            │
│   ┌───────────────┴──────────────┘                                          │
│   │ Driven Adapter│ (Outbound)                                              │
│   │ (JPA / Mail)  │                                                         │
│   └───────────────┘                                                         │
└─────────────────────────────────────────────────────────────────────────────┘

* 핵심 원리: 모든 의존성의 화살표가 외부에서 내부(Core)를 향하게 설계함 (DIP).
```

### 핵심 구성 요소 상세

#### ① 어플리케이션 코어 (Application Core)
순수한 비즈니스 로직과 도메인 모델이 존재하는 곳이다. 외부 기술(Spring, JPA, AWS 등)에 대한 어떤 의존성도 포함해서는 안 된다.

#### ② 포트 (Ports)
코어로 들어오거나(Inbound) 나가는(Outbound) 길목을 규정하는 인터페이스다.
- **Inbound Port**: 서비스 인터페이스. 외부 주도자(Driver)가 코어를 호출하는 통로.
- **Outbound Port**: 리포지토리 인터페이스 등. 코어가 외부(Driven)의 힘을 빌리는 통로.

#### ③ 어댑터 (Adapters)
외부 세계와 포트 사이의 통역사다.
- **Driver Adapter**: 웹 요청을 받아 서비스(Inbound Port)를 호출하는 컨트롤러.
- **Driven Adapter**: 코어의 요청(Outbound Port)을 받아 실제 DB에 저장하거나 메일을 보내는 구현체.

## Ⅲ. 융합 비교 및 다각도 분석

### 계층형 vs 헥사고날 아키텍처 상세 비교

| 비교 항목 | 계층형 (Layered) | **헥사고날 (Hexagonal)** |
|-----------|------------------|--------------------------|
| **의존성 방향** | 상위 -> 하위 (DB 종속적) | **외부 -> 내부 (Core 독립적)** |
| **테스트 용이성** | DB 연동 없이는 어려움 | **Core만 떼어내어 100% Mock 테스트** |
| **변경 영향도** | 하위 변경 시 상위 영향 큼 | **기술 스택 변경 시 Adapter만 교체** |
| **복잡도** | 낮음 (직관적) | **높음 (많은 인터페이스 필요)** |
| **주요 철학** | 작업의 순서 분할 | **관심사의 완벽한 격리** |

### 헥사고날 vs 클린 아키텍처 (Clean Architecture)
두 아키텍처는 "의존성을 안으로 향하게 한다"는 철학을 공유한다. 헥사고날은 **'포트와 어댑터'**라는 구체적인 통신 방식에 집중하고, 클린 아키텍처는 이를 더 확장하여 **'엔티티, 유즈케이스'** 등 더 세밀한 계층 분리와 원칙을 강조한다. 헥사고날은 클린 아키텍처의 실천적 모델 중 하나로 볼 수 있다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 도입 전략)

**상황**: 급성장 중인 이커머스 서비스에서 결제 시스템을 리뉴얼하려 한다. 현재는 카드사 연동 코드와 비즈니스 로직이 섞여 있어 테스트가 불가능하고 카드사를 바꿀 때마다 장애가 난다.
**판단 및 헥사고날 적용 전략**:
1. **Core Isolation**: 결제 승인, 취소 로직을 담은 `PaymentService`를 순수 자바 코드로 코어에 분리한다.
2. **Outbound Port 정의**: 카드사 통신을 위한 `PaymentGatewayPort` 인터페이스를 만든다.
3. **Adapter 구현**: `NHNAdapter`, `TossAdapter`를 각각 만들어 포트를 구현한다.
4. **결과**: 가짜 어댑터(Stub)를 끼워 비즈니스 로직만 초당 수천 번 테스트할 수 있게 되었으며, 새로운 결제 수단 추가 시 기존 로직 수정 없이 어댑터만 하나 더 만들면 되게 됨.

### 안티패턴 및 고려사항
- **Over-abstraction**: 단순한 CRUD 애플리케이션에 헥사고날을 적용하면 인터페이스와 DTO가 과도하게 늘어나 코드 가독성만 떨어진다. 비즈니스 복잡도가 낮은 곳에는 독이다.
- **Leaking Infrastructure**: 포트 인터페이스에 JPA 엔티티나 HTTP 관련 객체를 직접 노출하면 헥사고날의 의미가 사라진다. 반드시 **도메인 객체로 변환**하여 코어로 넘겨야 한다.

## Ⅴ. 미래 전망 및 결론

### 결론: 아키텍처의 지속 가능성
헥사고날 아키텍처는 소프트웨어의 생명을 연장한다. 기술은 2~3년마다 바뀌지만 비즈니스 가치는 10년 이상 지속된다. 이 둘을 격리함으로써 기술의 변화에 휘둘리지 않는 **'지속 가능한 코드'**를 가능케 한다.

### 미래 전망
최근 마이크로서비스(MSA)와 서버리스(Serverless) 환경이 확산되면서 헥사고날의 가치는 더욱 높아지고 있다. 람다(Lambda) 함수 하나를 짤 때도 핵심 로직을 헥사고날 코어로 짜두면, 나중에 이를 컨테이너 환경으로 옮겨도 로직 수정 없이 '어댑터'만 바꾸면 되기 때문이다. 클라우드 네이티브 설계의 가장 강력한 무기가 될 전망이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **헥사고날 아키텍처가 뭔가요?**: 게임기를 만들 때, 게임기 본체(로직)와 조이스틱(외부 장치)을 찰칵하고 끼울 수 있는 '연결 구멍'을 예쁘게 만드는 방법이에요.
2. **왜 좋나요?**: 조이스틱이 고장 나거나 다른 걸로 바꾸고 싶을 때, 게임기 본체를 뜯어볼 필요 없이 구멍에 맞는 새 조이스틱만 사서 끼우면 되거든요.
3. **뭐가 제일 신기해요?**: 게임기 본체만 들고도 "나는 지금 게임 중이야!"라고 상상하며 테스트해 볼 수 있을 정도로 본체가 아주 똑똑하고 독립적이랍니다!
