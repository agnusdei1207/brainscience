# 608. 클린 아키텍처 (Clean Architecture) - 프레임워크로부터의 독립 선언

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 로버트 C. 마틴(Uncle Bob)이 집대성한 설계 철학으로, 시스템의 중심에 '도메인(엔티티)과 유즈케이스'를 두고, **외부 환경(UI, DB, 프레임워크)이 중심을 향해 의존하도록 화살표를 역전시킨 아키텍처**이다.
> 2. **혁신**: 소프트웨어를 '세부 사항(웹, 데이터베이스)'과 '정책(비즈니스 규칙)'으로 완벽히 분리하여, 정책이 세부 사항의 변화에 절대 영향을 받지 않는 궁극의 유연성을 제공한다.
> 3. **가치**: "좋은 아키텍처는 결정을 가능한 한 오랫동안 미룰 수 있게 해준다"는 원칙을 실현하며, 데이터베이스가 무엇인지 UI가 무엇인지 정하지 않은 상태에서도 핵심 로직의 개발과 테스트를 완료할 수 있게 한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
클린 아키텍처는 양파 모양의 동심원으로 표현된다. 중심부에는 비즈니스 핵심 규칙이 있고, 바깥쪽으로 갈수록 프레임워크나 데이터베이스 같은 구체적인 기술이 위치한다. 가장 중요한 유일한 규칙은 **"의존성의 화살표는 반드시 밖에서 안으로만 향해야 한다"**는 것이다.

### 💡 비유
- **나쁜 아키텍처 (장비 종속적)**: 요리사가 특정 상표의 가스레인지와 특정 칼이 없으면 요리를 아예 못하는 상태다. 가스레인지가 단종되면 요리법 자체를 다 바꿔야 한다.
- **클린 아키텍처 (본질 집중)**: 요리사(도메인)는 오직 '맛을 내는 비법(유즈케이스)'만 집중한다. 불을 피우는 게 가스든 인덕션이든(DB), 손님에게 서빙하는 접시가 은쟁반이든 종이접시든(UI) 요리사의 비법은 절대 변하지 않는다.

### 등장 배경: 헥사고날, 어니언 아키텍처의 통합
전통적인 계층형 아키텍처는 DB 주도 설계(DB-Driven Design)로 빠지기 쉬웠다. 이를 극복하기 위해 헥사고날, 어니언(Onion) 등의 아키텍처가 나왔고, 로버트 마틴은 이들의 공통된 철학("관심사의 분리와 의존성 역전")을 하나로 묶어 '클린 아키텍처'라는 이름으로 정립했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 클린 아키텍처의 동심원 구조 (ASCII Diagram)

```ascii
     [ External Interfaces (Web, DB, Devices) ]  <-- 변경이 가장 잦음 (Volatile)
 ┌────────────────────────────────────────────────┐
 │     [ Interface Adapters (Controllers, Gateways, Presenters) ]
 │ ┌──────────────────────────────────────────────┐
 │ │       [ Application Business Rules (Use Cases) ]
 │ │ ┌────────────────────────────────────────────┐
 │ │ │         [ Enterprise Business Rules ]      │
 │ │ │         [        (Entities)         ]      │ <-- 가장 변하지 않음 (Stable)
 │ │ └────────────────────────────────────────────┘
 │ │         ▲ (Dependency Rule: Always points INWARD)
 │ └─────────┼────────────────────────────────────┘
 │           │
 └───────────┼────────────────────────────────────┘
             │
```

### 4계층 핵심 구조 상세

#### ① 엔티티 (Entities)
전사적인 핵심 비즈니스 규칙을 캡슐화한 객체다. 가장 변하지 않는 중심이다. (예: 은행 시스템의 '계좌' 클래스가 가진 '이자 계산' 메서드). 특정 애플리케이션의 동작과 무관하다.

#### ② 유즈케이스 (Use Cases)
애플리케이션에 특화된 비즈니스 규칙이다. 시스템이 수행해야 할 '작업 흐름'을 지휘한다. (예: "계좌 이체를 할 때 잔액을 확인하고, 돈을 빼고, 내역을 저장하라"). 엔티티를 조종하지만, DB나 UI는 모른다.

#### ③ 인터페이스 어댑터 (Interface Adapters)
유즈케이스와 외부 기술 사이의 통역사다. 웹의 HTTP 요청을 유즈케이스가 이해하는 포맷으로 바꾸거나(Controller), 유즈케이스의 결과를 SQL이나 UI 포맷으로 바꾼다(Presenter/Gateway).

#### ④ 프레임워크와 드라이버 (Frameworks & Drivers)
가장 바깥쪽 원으로, 데이터베이스(MySQL), 웹 프레임워크(Spring, Express), UI 등 모든 구체적인 세부 사항이 속한다. 이곳에는 핵심 로직이 절대 들어가선 안 되며, 다른 계층을 호출하는 연결 코드만 존재해야 한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 클린 아키텍처의 의존성 역전(DIP) 적용 방식

유즈케이스(안쪽)가 DB(바깥쪽)에 데이터를 저장해야 할 때, 안에서 밖으로 의존성이 향하면 규칙 위반이다. 어떻게 해결할까?

1. **인터페이스 선언**: 안쪽(Use Case)에 `UserRepository`라는 **인터페이스**를 만든다.
2. **호출**: 안쪽 모듈은 안쪽 인터페이스를 호출한다. (의존성 화살표가 내부에 머묾).
3. **구현**: 바깥쪽(DB Layer)에서 이 인터페이스를 **구현(Implement)**한다. `UserRepositoryImpl`.
4. **결과**: 소스 코드의 의존성 화살표는 밖에서 안을 향하지만, 런타임의 제어 흐름(Control Flow)은 안에서 밖으로 흐르는 '역전'이 발생한다.

### 클린 아키텍처가 주는 4가지 독립성
- **프레임워크 독립성**: 스프링 같은 프레임워크는 도구일 뿐, 시스템의 구조를 강제할 수 없다.
- **UI 독립성**: 웹 UI를 콘솔 UI로 바꾸어도 비즈니스 로직은 수정되지 않는다.
- **데이터베이스 독립성**: 오라클을 몽고DB로 바꾸어도 엔티티와 유즈케이스는 모른다.
- **외부 에이전시 독립성**: 외부 결제 API가 바뀌어도 코어는 안전하다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 도입 전략)

**상황**: 수명이 10년 이상 예상되는 핵심 금융 코어 뱅킹 시스템을 차세대 환경(Java/Spring)으로 새로 구축하려 한다.
**판단 및 아키텍처 전략**:
1. **패키지 분리**: 프로젝트를 아예 `core-domain`, `core-usecase`, `infra-db`, `infra-web`의 독립된 모듈(빌드 단위)로 나눈다.
2. **의존성 차단**: `core-domain`의 `build.gradle`에는 Spring이나 JPA 같은 프레임워크 의존성을 일절 넣지 못하게 차단한다. (오직 순수 Java 코드만 허용).
3. **DTO 매핑**: DB에서 가져온 엔티티(JPA Entity)를 순수 도메인 엔티티로 변환하는 매핑 과정을 반드시 거치게 하여 데이터베이스의 속성이 도메인을 오염시키는 것을 막는다.
4. **결과**: 단위 테스트 커버리지를 95% 이상 쉽게 달성하며, 추후 프레임워크 버전 업그레이드나 DB 교체 시 비즈니스 로직의 안전을 완벽히 보장함.

### 안티패턴 및 고려사항 (Over-engineering)
- **보일러플레이트의 늪**: 클린 아키텍처는 데이터를 밖에서 안으로, 안에서 밖으로 전달할 때마다 DTO 변환 코드를 엄청나게 짜야 한다. (Entity ↔ UseCase DTO ↔ Controller DTO).
- 단순한 게시판이나 빠르게 폐기될 수 있는 프로토타입에 이를 적용하는 것은 '빈대 잡으려다 초가삼간 태우는 격'이다. 비즈니스 복잡도가 높은 핵심 도메인에만 선별적으로 적용하는 지혜가 필요하다.

## Ⅴ. 미래 전망 및 결론

### 결론: 아키텍처의 이데아
클린 아키텍처는 완벽한 소프트웨어를 향한 이데아와 같다. 실무에서 100% 원칙을 지키기는 어렵지만, 이 동심원의 원리를 이해하고 설계하는 것과 모르는 것은 시스템의 수명과 유지보수 비용에서 하늘과 땅 차이를 만든다.

### 미래 전망
앞으로는 코드 생성기(AI 등)가 클린 아키텍처의 최대 단점인 '수많은 매핑 코드(보일러플레이트)' 작성을 대신해 줄 것이다. 개발자는 오직 안쪽 원(엔티티, 유즈케이스)의 핵심 로직만 고민하고, 바깥쪽 원과의 연결은 AI가 자동 생성하는 방식으로 진화하여 클린 아키텍처의 진입 장벽이 대폭 낮아질 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **클린 아키텍처가 뭔가요?**: 양파처럼 여러 겹으로 둘러싸서, 가장 소중한 '알맹이 보물(비즈니스 로직)'을 안전하게 지키는 성벽 만들기 규칙이에요.
2. **어떤 규칙인가요?**: 바깥쪽 성벽을 고치거나 색칠할 때, 절대로 안쪽 알맹이를 건드리지 말고 밖에서만 작업하라는 아주 엄격한 규칙이에요.
3. **뭐가 좋나요?**: 겉모습을 아무리 많이 바꿔도, 알맹이 보물은 영원히 고장 나지 않고 안전하게 살아남을 수 있답니다!
