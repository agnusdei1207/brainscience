# 609. 이벤트 기반 아키텍처 (Event-Driven Architecture, EDA) - 비동기와 느슨한 결합의 정수

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템의 상태 변화(Event)가 발생하면 이를 메시지 형태로 발행(Publish)하고, 관심 있는 다른 시스템들이 이를 비동기적으로 구독(Subscribe)하여 반응하는 아키텍처 패턴이다.
> 2. **혁신**: 서비스를 직접 호출(A가 B를 호출)하는 강한 결합을 끊고, **이벤트 브로커(Kafka, RabbitMQ 등)를 매개로 시스템들을 철저히 분리(Decoupling)**하여 확장성과 복원력을 극대화한다.
> 3. **가치**: 마이크로서비스(MSA) 간의 데이터 동기화와 대규모 실시간 트래픽 처리에 필수적인 설계 방식으로, 특정 서비스 장애가 전체로 전파되는 도미노 현상을 완벽히 차단한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
이벤트 기반 아키텍처(EDA)에서 시스템의 모든 움직임은 '이벤트'로 정의된다. "사용자가 결제를 완료했다(OrderPaidEvent)"라는 사실이 발생하면 시스템은 이 사실을 중앙 우체국에 던져놓고 제 갈 길을 간다. 재고 시스템이나 배송 시스템은 우체국에서 이 사실을 가져와 각자의 일을 처리한다.

### 💡 비유
- **요청-응답 방식 (Request-Response)**: 피자 가게 주인이 주문을 받고(A), 요리사에게 "피자 만들어!"라고 소리친다(B). 요리사가 피자를 다 만들 때까지 주인은 꼼짝 않고 기다려야 한다. 요리사가 쓰러지면 주인도 일을 못 한다 (동기식/강결합).
- **이벤트 기반 방식 (EDA)**: 주인이 주문을 받으면 전광판에 "피자 주문 1건 발생!"이라고 포스트잇을 붙이고 다시 손님을 맞이한다. 요리사는 자기 속도에 맞춰 전광판의 포스트잇을 떼서 피자를 만든다. 요리사가 화장실에 가도 주인은 계속 포스트잇을 붙이면 되니 장사가 멈추지 않는다 (비동기/느슨한 결합).

### 등장 배경: 동기식 HTTP 통신의 한계
MSA 환경에서 수십 개의 서비스가 서로 REST API로 동기식 호출을 하면, 하나의 서비스가 느려졌을 때 전체 API 응답이 지연되는 병목(Bottleneck)이 발생했다. 이를 해결하기 위해 서비스를 서로 모르게 만들고 중간에 '버퍼'를 두는 이벤트 통신 방식이 부상했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### EDA의 핵심 토폴로지 (ASCII Diagram)

```ascii
[ Event Producer ]                      [ Event Broker / Router ]                   [ Event Consumers ]
┌────────────────┐                    ┌───────────────────────────┐                 ┌─────────────────┐
│                │  "OrderCreated"    │    Message Queue / Bus    │  (Subscribe)    │                 │
│ Order Service  ├───────────────────>│      (e.g., Kafka)        ├────────────────>│ Shipping Service│
│                │   (Publish)        │ ┌────────┐ ┌─────────┐    │                 │                 │
└────────────────┘                    │ │ Event1 │ │ Event2  │... │                 └─────────────────┘
                                      │ └────────┘ └─────────┘    │  (Subscribe)    ┌─────────────────┐
┌────────────────┐  "UserSignedUp"    │                           ├────────────────>│                 │
│  User Service  ├───────────────────>│                           │                 │ Loyalty Service │
│                │                    └───────────────────────────┘                 │                 │
└────────────────┘                                                                  └─────────────────┘

* 핵심 원리: Producer는 누가 이벤트를 가져가는지 모르고, 
  Consumer는 누가 이벤트를 만들었는지 신경 쓰지 않음 (Fire and Forget).
```

### EDA를 구성하는 3가지 주요 패턴

#### ① 이벤트 알림 (Event Notification)
이벤트 안에 최소한의 정보(예: "주문 ID 123번 상태 변경됨")만 담는다. Consumer는 이벤트 수신 후 세부 정보가 필요하면 Producer의 API를 다시 찔러서 조회한다. 결합도는 가장 낮으나 추가 네트워크 호출이 발생한다.

#### ② 이벤트 상태 전송 (Event-Carried State Transfer)
이벤트 안에 변경된 데이터의 모든 상태 정보(예: 주문 ID, 상품명, 가격, 고객 주소)를 꽉꽉 채워 보낸다. Consumer는 수신한 데이터로 자신의 로컬 DB를 업데이트(Materialized View)하여 추가 API 호출 없이 독자적으로 작업한다.

#### ③ 이벤트 소싱 (Event Sourcing)
현재 상태(예: 잔고 1만 원)를 DB에 저장하는 것이 아니라, 발생한 모든 이벤트(1만 원 입금, 5천 원 출금, 5천 원 입금)를 원장(Ledger)에 순서대로 모두 저장한다. 상태가 필요할 때 이벤트를 처음부터 재생하여 현재 값을 도출한다.

## Ⅲ. 융합 비교 및 다각도 분석

### Request-Response (REST) vs Event-Driven (EDA)

| 비교 항목 | Request-Response (REST API) | **Event-Driven (EDA)** |
|-----------|-----------------------------|------------------------|
| **결합도 (Coupling)**| 높음 (A가 B의 주소를 알아야 함) | **매우 낮음 (Broker만 알면 됨)** |
| **통신 방식** | 동기 (결과를 기다림) | **비동기 (Fire and Forget)** |
| **장애 전파** | 타겟 서비스 장애 시 실패 반환 | **큐에 쌓아두고 나중에 재시도 (복원력)** |
| **확장성** | 트래픽 증가 시 일대일 확장 필요 | **Consumer만 늘리면 무한 병렬 처리** |
| **복잡도** | 직관적이고 쉬움 | **어려움 (비동기 에러 추적, 순서 보장 등)**|

### 안무(Choreography) vs 오케스트레이션(Orchestration)
- **오케스트레이션**: 중앙의 지휘자 서비스가 "A해, 그다음 B해"라고 순서를 제어한다. (과도하게 중앙 집중화될 위험).
- **코레오그래피 (순수 EDA)**: 중앙 지휘자 없이, 무용수들이 음악(이벤트)에 맞춰 각자 알아서 춤을 춘다. 결합도가 가장 낮지만 전체 흐름(비즈니스 트랜잭션)을 파악하기 어렵다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 도입 전략)

**상황**: 대형 쇼핑몰에서 특정 상품에 트래픽이 몰리는 '선착순 특가 이벤트'를 하면 매번 결제와 재고 차감 API가 병목을 일으켜 전체 사이트가 다운된다.
**판단 및 EDA 전환 전략**:
1. **Fire and Forget**: 사용자가 결제 버튼을 누르면 즉시 화면에 "주문 접수 중"을 띄우고, 실제 작업은 `OrderRequestedEvent`로 만들어 **Kafka**로 보낸다.
2. **비동기 분산 처리**: 재고, 결제, 배송 컨슈머들이 큐에서 이벤트를 꺼내가 각자의 속도대로 처리한다. 트래픽이 몰리면 큐에 데이터가 쌓일 뿐 서버는 죽지 않는다.
3. **최종적 일관성 (Eventual Consistency)**: 당장 재고 차감이 안 되더라도 1~2초 뒤에는 반드시 동기화되도록 시스템을 설계한다.
4. **결과**: 서버 다운을 완벽히 방지하고, 초당 10만 건의 폭주 트래픽을 유실 없이 안정적으로 소화함.

### 안티패턴 및 고려사항
- **이벤트 핀볼 (Event Pinball)**: A 이벤트가 B를 부르고, B가 C를, C가 다시 A를 부르는 등 흐름이 꼬여서 장애 원인을 추적할 수 없게 되는 현상. 이벤트 흐름을 추적할 수 있는 **Correlation ID** 분산 추적(Zipkin 등)이 필수다.
- **멱등성 무시**: 네트워크 오류로 같은 이벤트가 두 번 도착할 수 있다. 한 번 처리된 이벤트를 다시 처리해도 결과가 똑같게(Idempotent) 컨슈머 로직을 방어적으로 짜야 한다.

## Ⅴ. 미래 전망 및 결론

### 결론: 실시간 세계를 위한 필수 설계
EDA는 "모든 것은 즉시 처리되어야 한다"는 전통적인 트랜잭션의 환상을 깨고, "데이터는 흐르고 시스템은 각자의 템포에 맞춰 반응한다"는 현실적인 분산 시스템의 정답을 제시한다.

### 미래 전망
최근 클라우드 플랫폼들은 Amazon EventBridge, Azure Event Grid 같은 **'완전 관리형 이벤트 버스'**를 제공하며 EDA 도입 장벽을 낮추고 있다. 앞으로는 서버리스(Serverless/Functions) 아키텍처와 완벽하게 결합하여, 이벤트가 발생할 때만 코드가 실행되고 사라지는 '비용 제로에 가까운 실시간 반응형 인프라'가 대세가 될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **이벤트 기반 아키텍처가 뭔가요?**: 친구에게 심부름을 시킬 때, 친구가 다 할 때까지 옆에서 서서 기다리지 않고 칠판에 메모지만 딱 붙여놓고 내 할 일을 하러 가는 방법이에요.
2. **왜 그렇게 하나요?**: 내가 너무 바쁜데 친구가 심부름을 늦게 하면 내 일까지 다 멈추잖아요. 메모지만 붙여두면 나는 안 멈추고 계속 놀 수 있거든요.
3. **가장 좋은 점은?**: 심부름할 친구 10명이 메모지를 각자 하나씩 가져가서 동시에 일을 할 수 있어서, 일이 엄청나게 빨리 끝난답니다!
