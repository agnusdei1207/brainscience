# 611. 클린 아키텍처 의존성 규칙 - 제어의 흐름을 지배하는 황금률

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클린 아키텍처(Clean Architecture)의 동심원 구조를 관통하는 단 하나의 절대 법칙으로, **"모든 소스 코드의 의존성은 반드시 바깥쪽(외부 메커니즘)에서 안쪽(핵심 비즈니스 정책)으로만 향해야 한다"**는 원칙이다.
> 2. **혁신**: 안쪽 원(도메인/유즈케이스)의 코드는 바깥쪽 원(데이터베이스, 웹 프레임워크, UI)에 선언된 클래스, 함수, 변수를 절대로 알아서는 안 되며 그 이름을 언급해서도 안 된다.
> 3. **가치**: 가장 빈번하게 변하는 기술적 세부사항(프레임워크, DB)이 바뀌더라도, 기업의 돈을 벌어다 주는 가장 중요한 핵심 비즈니스 로직 코드는 단 한 줄도 수정할 필요가 없는 궁극의 보호막을 제공한다.

*(※ 본 문서는 [608. 클린 아키텍처](./608_clean_architecture.md)의 핵심 철학인 '의존성 규칙'의 메커니즘을 기술적으로 심층 분석합니다.)*

---

## Ⅰ. 개요 (Context & Background)

### 개념
의존성 규칙(Dependency Rule)은 "누가 누구를 아는가(Who knows whom)"에 대한 엄격한 방향 통제다. 클린 아키텍처에서 고수준(High-level)은 비즈니스 로직을 뜻하고, 저수준(Low-level)은 DB나 UI 같은 I/O를 뜻한다. 의존성은 언제나 저수준에서 고수준으로만 향해야 한다.

### 💡 비유
- **규칙을 어긴 경우 (왕이 신하 눈치를 봄)**: 회사 사장님(핵심 비즈니스)이 문서를 결재할 때 "지금 경리부 직원이 엑셀 2013 버전을 쓰니까 그 포맷에 맞춰서 줘야지"라며 직원의 도구(DB/Framework)에 맞춰 자신의 결재 방식을 바꾼다. 직원이 퇴사하면 사장님도 일을 못 한다.
- **의존성 규칙을 지킨 경우 (신하가 왕에게 맞춤)**: 사장님은 그저 '표준 결재판(Interface)'을 책상에 둔다. 직원이 엑셀을 쓰든 한글을 쓰든 알아서 사장님의 결재판 규격에 맞춰 문서를 가져와야 한다. 직원이 바뀌어도 사장님의 업무 방식은 전혀 변하지 않는다.

### 등장 배경: 'DB-Driven Design'의 덫 탈출
전통적인 개발 방식에서는 DB 스키마(테이블)를 먼저 짜고, 그 위에 로직을 얹고, 마지막에 UI를 붙였다. 자연스럽게 로직(고수준)이 DB(저수준)에 의존하는 하향식 종속이 발생했다. 클린 아키텍처는 이를 완전히 뒤집어 "DB는 가장 마지막에 결정해도 되는 플러그인(Plugin)일 뿐이다"라고 선언하며 이 규칙을 세웠다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 제어 흐름(Control Flow)과 의존성 방향(Dependency)의 충돌 (ASCII Diagram)

비즈니스 로직(Use Case)이 데이터를 저장하기 위해 DB를 호출해야 할 때 역설이 발생한다. 제어 흐름은 안에서 밖으로 나가야 하는데, 의존성은 밖에서 안으로 들어와야 하기 때문이다.

```ascii
[ Use Case (Inside) ] ─────────(제어 흐름 원함)─────────▶ [ DB Layer (Outside) ]
  (저수준을 호출하면 안쪽이 바깥쪽을 의존하게 되어 규칙 위반!)

================================================================================

[ 1. 인터페이스를 통한 의존성 역전 (DIP 적용) ]

           (Inside : Core)                 │            (Outside : Infra)
                                           │
┌────────────────┐      ┌────────────────┐ │  (Implements) ┌────────────────┐
│ Use Case (순수)│───> │ <<Interface>>  │<│───────────────│ DB Repository  │
│ - SaveData()   │ Call │ IRepository    │ │               │ (MySQL 코드)   │
└────────────────┘      └────────────────┘ │               └────────────────┘
                                           │
[ ----- 제어의 흐름(Runtime Execution) -----------------------------------> ]
[ <---- 소스 코드 의존성의 방향 (Compile-time Dependency) ----------------- ]

* 핵심 원리: 안쪽에 뚫어놓은 구멍(Interface)을 바깥쪽이 구현하게 함으로써 
  의존성의 방향을 안쪽으로 강제로 꺾어버린다.
```

### 규칙 준수를 위한 3가지 메커니즘

#### ① 의존성 역전 원칙 (DIP: Dependency Inversion Principle)
위 그림처럼, 안쪽 원에 인터페이스를 정의하고, 바깥쪽 원의 클래스가 이 인터페이스를 구현(Implement)하도록 만든다. 런타임에는 안쪽에서 바깥쪽으로 데이터가 흘러가지만, 컴파일 시점의 소스 코드 의존성은 바깥에서 안으로 향하게 된다.

#### ② DTO (Data Transfer Object) 활용
바깥쪽 원의 데이터 구조(예: DB 프레임워크의 ORM Entity 객체나 웹의 JSON 객체)가 안쪽 원으로 침범하면 안 된다. 경계를 넘을 때는 안쪽 원이 정의한 순수한 자료 구조(DTO)로 변환(Mapping)하여 전달해야 한다.

#### ③ 의존성 주입 (DI: Dependency Injection)
인터페이스를 만들었어도, 안쪽 원에서 `new MySQLRepository()`라고 코드를 치는 순간 의존성 규칙이 깨진다. 외부의 조립기(Spring의 IoC 컨테이너나 Main 함수)가 바깥쪽 구현체를 생성하여 안쪽으로 밀어 넣어줘야(Inject) 한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 규칙 준수 여부에 따른 코드 변경 파급력 (Table)

| 시나리오 | 의존성 규칙 위반 (DB 중심 설계) | **의존성 규칙 준수 (클린 아키텍처)** |
|:---|:---|:---|
| **DB를 Oracle에서 MongoDB로 교체** | 로직 곳곳의 SQL 객체 수정으로 대공사 | **DB 레이어의 어댑터만 교체 (Core 수정 0)** |
| **웹(API)을 CLI나 gRPC로 교체** | HTTP 세션 객체가 로직에 박혀 있어 분리 불가 | **UI 프레젠터만 교체 (Core 수정 0)** |
| **순수 비즈니스 로직 단위 테스트**| DB 서버를 띄워야만 테스트 가능 (느림) | **DB 없이 Mocking만으로 0.1초 만에 테스트** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 의존성 통제 전략)

**상황**: 개발팀이 "클린 아키텍처를 도입했다"고 자랑하지만, 꼼꼼히 코드를 리뷰해보니 핵심 비즈니스 로직(Service 계층) 파일 상단에 `import org.springframework.web...` 이나 `import javax.persistence.Table` 같은 코드가 존재한다.
**판단 및 리팩토링 지시**:
1. **규칙 위반 적발**: 안쪽 원(Service, Entity)이 바깥쪽 원(Spring 웹 프레임워크, JPA 데이터베이스 기술)의 라이브러리에 의존하고 있다. 프레임워크가 바뀌면 핵심 로직도 에러가 나는 종속 상태다.
2. **격리 조치 (Isolation)**: `javax.persistence.Entity` 어노테이션이 붙은 클래스는 바깥쪽(DB Layer)으로 치워버리고, 비즈니스 로직에는 어노테이션이 전혀 없는 **순수한 자바 객체(POJO)**만을 사용하게 한다.
3. **빌드 도구 강제 (Enforcement)**: 개발자의 실수를 막기 위해, Gradle이나 Maven 설정에서 Core 모듈이 Infra 모듈을 참조하지 못하도록 컴파일 레벨에서 의존성(Dependencies)을 원천 차단한다. (ArchUnit 같은 툴 활용).
4. **결과**: "프레임워크는 언제든 갈아 끼울 수 있는 플러그인"이라는 완벽한 클린 아키텍처 체제로 회귀.

### 안티패턴 및 고려사항
- **의존성 규칙의 피로도**: 이 규칙을 완벽히 지키려면 모델 변환(Mapper) 코드를 엄청나게 짜야 한다(예: JpaEntity -> DomainEntity -> DTO). 서비스의 중요도가 낮거나 수명이 짧은 프로젝트에서 이 규칙을 맹목적으로 고수하는 것은 개발 리소스 낭비다. '실용주의적 타협'이 항상 필요하다.

## Ⅴ. 미래 전망 및 결론

### 결론: 아키텍처의 영원한 방패
클린 아키텍처의 의존성 규칙은 "중요한 것(로직)과 덜 중요한 것(기술)을 분리하고, 중요한 것이 덜 중요한 것에 휘둘리지 않게 하라"는 지극히 상식적인 철학의 프로그래밍적 구현체다. 이는 코드를 건강하게 유지하는 궁극의 방어막이다.

### 미래 전망
앞으로 프레임워크와 기술 트렌드는 더욱 빠르게 변할 것이다(예: RDBMS -> NoSQL -> Vector DB). 외부 도구가 요동칠수록, 변하지 않는 중심(도메인)을 지키기 위한 '의존성 역전 규칙'의 가치는 더욱 빛을 발할 것이다. 최근의 컴파일러와 린터(Linter) 도구들은 이러한 의존성 규칙 위반을 코딩 중에 실시간으로 찾아내어 경고하는 방향으로 발전하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **의존성 규칙이 뭔가요?**: 왕(비즈니스 로직)과 신하(데이터베이스, 인터넷)가 있을 때, "왕은 절대 신하의 눈치를 보지 말고, 신하가 무조건 왕의 명령에 맞춰서 움직여야 한다"는 아주 엄격한 궁궐의 법칙이에요.
2. **왜 그런 법칙이 있나요?**: 신하들이 바뀔 때마다 왕이 일하는 방식을 바꾸면 나라가 너무 혼란스러워지기 때문이에요.
3. **어떻게 법칙을 지키나요?**: 왕은 방 밖으로 절대 나가지 않고 문틈(인터페이스)으로 쪽지만 전달해요. 바깥의 신하들이 그 쪽지를 보고 자기들끼리 지지고 볶고 일해서 결과를 다시 문틈으로 넣어주는 방식이랍니다!
