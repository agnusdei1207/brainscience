# 612. 헥사고날 포트와 어댑터 (Ports and Adapters) - 외부 세계와 소통하는 표준 규격

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 헥사고날 아키텍처에서 핵심 비즈니스 로직(내부)과 외부 환경(DB, 웹 등)을 연결하는 통로로, **'포트(Port)'는 내부가 정의한 통신 규약(인터페이스)이며 '어댑터(Adapter)'는 그 규약에 맞춰 외부 기술을 구현한 부품**이다.
> 2. **혁신**: 시스템의 입력과 출력을 방향성에 따라 **Inbound(Driving)**와 **Outbound(Driven)**로 명확히 분리하여, 비즈니스 로직이 외부 기술의 변경에 전혀 영향을 받지 않도록 철저히 격리한다.
> 3. **가치**: 이 패턴을 적용하면 데이터베이스나 웹 서버가 아직 완성되지 않은 프로젝트 초기 단계에서도 핵심 도메인 로직을 가짜 어댑터(Mock)에 끼워 100% 완벽하게 테스트할 수 있는 극강의 개발 민첩성을 얻는다.

*(※ 본 문서는 [607. 헥사고날 아키텍처](./607_hexagonal_architecture.md)의 핵심 요소인 포트와 어댑터의 구체적 구조와 구현 패턴을 심화 학습합니다.)*

---

## Ⅰ. 개요 (Context & Background)

### 개념
포트와 어댑터는 헥사고날 아키텍처의 원래 이름(Ports and Adapters Architecture)일 만큼 이 아키텍처의 모든 것이다. 
- **포트(Port)**: 비즈니스 로직이 "나랑 대화하려면 이 모양(인터페이스)에 맞춰서 와라"라고 뚫어놓은 구멍이다.
- **어댑터(Adapter)**: 외부의 다양한 기술(REST, gRPC, MySQL, Redis)을 포트의 모양에 딱 맞게 깎아주는 변환 젠더다.

### 💡 비유
- **노트북(Core)과 모니터(DB)**: 노트북 안에 있는 그래픽 카드(비즈니스 로직)는 영상 신호를 어떻게 그릴지만 계산한다. 노트북 옆면에는 **HDMI 포트(Port)**가 뚫려 있다.
- 내가 삼성 모니터를 쓰든 LG 모니터를 쓰든, 그 모니터 선 끝에 **HDMI 어댑터(Adapter)**만 달려 있다면 노트북은 모니터의 브랜드를 몰라도 영상을 훌륭하게 출력할 수 있다. 모니터를 바꾼다고 노트북의 뚜껑을 열어 회로를 고칠 필요가 전혀 없다.

### 등장 배경: 대칭성(Symmetry)의 발견
전통적 계층형 모델은 위(UI)에서 아래(DB)로 흐르는 일방통행이었다. 앨리스테어 코번은 "사용자(UI)가 앱을 찌르는 것이나, 앱이 데이터베이스(DB)를 찌르는 것이나 결국 외부 시스템과 상호작용한다는 점에서 똑같다"라는 '대칭성'을 발견했다. 이에 상하 개념을 없애고 안과 밖으로 나누어 양쪽에 동일한 '포트/어댑터' 개념을 적용했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 인바운드와 아웃바운드 구조 (ASCII Diagram)

```ascii
[ Driving Adapters (주도자) ]             [ Driven Adapters (수동자) ]
(사용자, 웹 API, 메시지 큐 섭취기)        (DB, 외부 API 결제, 이메일 전송)

  ┌──────────────────┐                    ┌──────────────────┐
  │ REST API Adapter │                    │  JPA Repository  │
  └────────┬─────────┘                    │     Adapter      │
           │(구현/호출)                   └────────▲─────────┘
           ▼                                       │ (구현)
    [ Inbound Port ]                        [ Outbound Port ]
  ┌──────────────────┐                    ┌──────────────────┐
  │ Interface:       │                    │ Interface:       │
  │ CreateUserUseCase│                    │ UserRepository   │
  └────────┬─────────┘                    └──────────────────┘
           │ (구현)                                ▲ (호출)
           ▼                                       │
  ┌────────────────────────────────────────────────────────┐
  │                 [ Application Core ]                   │
  │  - UserService (Implements CreateUserUseCase)          │
  │    * 비즈니스 로직 수행                                │
  │    * UserRepository.save(User) 호출                    │
  └────────────────────────────────────────────────────────┘
```

### 핵심 구성 요소 상세 분석

#### ① Inbound (Driving) Port & Adapter
코어(비즈니스 로직)를 **작동시키는(Driving)** 쪽이다.
- **Inbound Port**: 유즈케이스(Use Case) 인터페이스다. 코어가 "내가 할 수 있는 일은 이것들이다"라고 외부로 메뉴판을 내어놓은 것이다.
- **Inbound Adapter**: 외부에서 들어온 요청(HTTP GET, CLI 명령어)을 Inbound Port의 메뉴판 규격에 맞게 변환하여 호출하는 역할이다. (예: Spring MVC의 Controller).

#### ② Outbound (Driven) Port & Adapter
코어에 의해 **조종당하는(Driven)** 쪽이다.
- **Outbound Port**: 코어가 작업을 끝내기 위해 외부에 "이 일을 해달라"고 요청할 때 쓰는 규약(Interface)이다. (예: `save()`, `findById()`). 코어 내부에 선언된다.
- **Outbound Adapter**: DB나 외부 시스템에 접속하여 Outbound Port가 시킨 일을 실제로 수행하는 구현체(Class)다. (예: JPA를 이용한 Repository 구현체).

## Ⅲ. 융합 비교 및 다각도 분석

### 포트와 어댑터의 런타임 제어 흐름 (Control Flow)

의존성 역전(DIP)이 어떻게 작용하는지 방향을 주의 깊게 봐야 한다.

| 구분 | 소스 코드 의존성 (의존성 화살표) | 런타임 제어 흐름 (실행 순서) | 설명 |
|:---:|:---|:---|:---|
| **Inbound** | Adapter ──> Port (코어 방향) | Adapter ──> Port (코어 방향) | 방향이 일치함. 컨트롤러가 서비스 인터페이스를 앎. |
| **Outbound**| Adapter ──> Port (코어 방향) | Port ──> Adapter (외부 방향) | **방향이 역전됨!** 코어는 인터페이스만 호출하지만, 실제 실행은 외부 어댑터에서 일어남. |

### DTO (Data Transfer Object)의 엄격한 경계 통제
어댑터가 포트를 통과할 때, 웹 프레임워크의 `HttpServletRequest`나 데이터베이스의 `Entity` 객체가 코어 내부로 들어와서는 절대 안 된다. 반드시 포트가 정의한 순수한 도메인/유즈케이스 DTO로 변환하여 넘겨주어야 한다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 테스트 최적화 전략)

**상황**: 대형 프로젝트를 시작했다. 백엔드 로직 개발자는 당장 코딩을 시작하고 싶은데, 인프라 팀에서 오라클 DB 계정을 아직 만들어주지 않았고, 프론트엔드 팀의 화면도 안 나왔다.
**판단 및 헥사고날 개발 전략**:
1. **코어 우선 개발**: DB나 UI 신경 쓰지 않고, 순수 자바 코드로 비즈니스 로직(User 생성, 검증)과 `Outbound Port(UserRepository)` 인터페이스만 먼저 개발한다.
2. **Mock Adapter 활용**: 오라클 DB가 없으므로, 메모리 맵(Map)을 사용하는 아주 단순한 가짜 어댑터(`InMemoryUserRepositoryAdapter`)를 만들어 Outbound Port에 끼워 넣는다.
3. **Inbound 테스트**: 웹 화면이 없으므로 JUnit 테스트 코드(Test Adapter)를 만들어 Inbound Port를 찔러본다.
4. **결과**: 인프라가 0인 상태에서도 핵심 로직의 완벽한 검증이 끝난다. 한 달 뒤 오라클 DB가 세팅되면, `OracleUserRepositoryAdapter`만 새로 짜서 갈아 끼우면 배포 준비가 끝난다.

### 안티패턴 및 고려사항
- **포트 생략(Short-circuiting)**: "Inbound 어댑터(컨트롤러)에서 그냥 코어의 Service 클래스를 바로 호출하면 되지, 귀찮게 Inbound Port 인터페이스를 왜 만드냐?"라는 유혹에 빠지는 것. 작은 프로젝트에선 타협할 수 있으나, 시스템이 커지면 Use Case의 경계가 무너져 유지보수가 급격히 힘들어진다.

## Ⅴ. 미래 전망 및 결론

### 결론: 레고 블록 소프트웨어의 완성
포트와 어댑터 패턴은 소프트웨어를 '교체 가능한 부품'으로 만드는 궁극의 설계법이다. 이 패턴을 마스터하면 어떤 최신 프레임워크나 요란한 기술 트렌드가 쏟아져도 묵묵히 중심(비즈니스 가치)을 지키는 견고한 엔지니어가 될 수 있다.

### 미래 전망
최근 마이크로서비스(MSA) 통신에서 gRPC, Kafka, REST 등 다양한 기술이 섞여 쓰이면서 포트와 어댑터의 중요성이 더욱 커지고 있다. 향후 **코드 제너레이터 AI** 도구들이 비즈니스 코어(Port)만 작성해주면, 이를 몽고DB용, AWS 람다용, 웹소켓용 어댑터 코드로 순식간에 자동 생성해 주는 방식으로 개발의 판도가 바뀔 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **포트와 어댑터가 뭔가요?**: 벽에 뚫려 있는 '110V 모양의 돼지코 구멍(포트)'과, 그 구멍에 꽂기 위해 모양을 바꿔주는 '변환 플러그(어댑터)'예요.
2. **왜 변환 플러그를 쓰나요?**: 우리나라 가전제품이든 미국 가전제품이든, 변환 플러그만 딱 맞게 꽂아주면 벽을 뜯어고치지 않고도 전기를 잘 쓸 수 있거든요.
3. **가장 좋은 점은?**: 핵심 로직(벽 속의 전기)은 절대 건드리지 않으면서, 밖에서 어떤 새로운 기계(데이터베이스, 웹사이트)를 가져와도 쉽게 연결할 수 있는 완벽한 보호막이랍니다!
