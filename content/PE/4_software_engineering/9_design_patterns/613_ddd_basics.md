# 613. 도메인 주도 설계 (DDD) 기본 구성 - 비즈니스 언어로 코드를 번역하다

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 에릭 에반스(Eric Evans)가 창시한 설계 철학으로, 복잡한 소프트웨어를 기술(DB, Framework) 중심이 아닌 **해결해야 할 현실 세계의 비즈니스 문제(Domain)를 중심으로 모델링**하는 방법론이다.
> 2. **혁신**: 개발자와 비즈니스 전문가(현업)가 소통할 때 사용하는 **유비쿼터스 언어(Ubiquitous Language)**를 정의하고, 그 언어 그대로 소스 코드의 클래스명과 메서드명이 되도록 하여 기획과 구현의 간극을 없앤다.
> 3. **가치**: 엔티티(Entity), 값 객체(Value Object), 리포지토리(Repository) 등의 전술적 패턴을 통해, 복잡하게 꼬인 스파게티 코드를 도메인 규칙을 스스로 방어하는 견고한 무결성 객체들로 탈바꿈시킨다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
도메인 주도 설계(Domain-Driven Design, DDD)는 복잡성이 높은 엔터프라이즈 시스템을 성공적으로 만들기 위한 전략적/전술적 지침서다. 기술적인 로직보다 비즈니스 규칙이 훼손되지 않게 객체 안에 단단히 가두는 것을 최우선 목표로 한다.

### 💡 비유
- **일반적인 개발 (번역기 없는 대화)**: 기획자(현업)가 "고객이 주문을 취소하면 적립금이 환수된다"고 말하면, 개발자는 `UPDATE user SET point = point - 100 WHERE status = 'CANCEL'` 이라는 SQL 언어로 머릿속에서 번역하여 짠다. 코드를 보면 기획 의도를 알 수 없다.
- **DDD (유비쿼터스 언어)**: 기획자와 개발자가 단어 사전을 맞춘다. 그리고 소스 코드에 `Order.cancel()`을 부르면 내부적으로 `Customer.deductPoint()`가 호출되게 짠다. 소스 코드를 마치 기획서 읽듯 한글/영어로 자연스럽게 읽어낼 수 있다.

### 등장 배경: '복잡성'의 저주
소프트웨어 실패의 80%는 기술이 부족해서가 아니라, "개발자가 비즈니스를 제대로 이해하지 못해서" 발생한다. 코드가 방대해질수록 데이터가 이리저리 수정되며 비즈니스 룰이 깨지기 일쑤였다. 이를 막기 위해 비즈니스 모델(도메인) 자체가 코드의 뼈대가 되어야 한다는 DDD가 주목받게 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### DDD의 전술적 패턴 (Tactical Patterns) 모델링 (ASCII Diagram)

```ascii
[ Domain Layer ]
      ┌──────────────────────────────────────────────────┐
      │  [ Aggregate (애그리거트: 묶음) ]                  │
      │                                                  │
      │ ┌────────────────┐ 1      N ┌────────────────┐   │
      │ │  Order         │ ───────> │  OrderLine     │   │
      │ │ (Root Entity)  │          │  (Entity)      │   │
      │ ├────────────────┤          ├────────────────┤   │
      │ │ - orderId (ID) │          │ - productId    │   │
      │ │ - totalAmount  │          │ - quantity     │   │
      │ │ - shippingAddr │ ────┐    └────────────────┘   │
      │ └────────────────┘     │                         │
      └───────▲────────────────┼─────────────────────────┘
              │ (불러오기)     │ 1
     ┌────────┴───────┐    ┌───▼────────────┐
     │ OrderRepository│    │ Address        │
     │ (Interface)    │    │ (Value Object) │ <-- 불변 객체
     └────────────────┘    └────────────────┘
```

### DDD의 3대 핵심 빌딩 블록

#### ① 엔티티 (Entity)
고유한 **식별자(Identity)**를 갖는 객체다. 시간이 지나 속성이 바뀌어도 식별자가 같으면 같은 객체로 본다.
- 예: 사람의 얼굴이 변하거나 나이가 먹어도 '주민등록번호'가 같으면 같은 사람이다. (User, Order 등).

#### ② 값 객체 (Value Object, VO)
식별자가 없고, **속성값 그 자체**로 의미를 갖는 객체다. 가장 중요한 특징은 한 번 만들어지면 절대 값이 변하지 않는 **불변성(Immutability)**이다.
- 예: 5만 원짜리 지폐 2장은 일련번호(식별자)가 달라도 나에게는 완벽히 동일한 '5만 원'이다. (Address, Money, Color 등). 값이 바뀌면 객체를 통째로 새로 만들어야 한다.

#### ③ 리포지토리 (Repository)
도메인 객체(엔티티)를 영구적으로 저장하고 불러오는 **도메인 계층의 인터페이스**다. 데이터베이스의 테이블 중심이 아니라, 애그리거트(객체 묶음)를 통째로 넣고 빼는 논리적인 '컬렉션(창고)' 역할을 한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 빈혈 도메인 모델 vs 풍부한 도메인 모델 (DDD의 핵심)

| 비교 항목 | 빈혈 도메인 모델 (Anemic) | **풍부한 도메인 모델 (Rich, DDD 지향)** |
|-----------|---------------------------|-----------------------------------------|
| **객체의 상태** | Getter/Setter만 잔뜩 있음 | **Setter가 없음 (의미 있는 메서드만 존재)** |
| **비즈니스 로직**| 서비스(Service) 클래스에 몰려 있음| **엔티티(Entity) 자신이 직접 처리함** |
| **무결성 유지** | 서비스 로직 개발자의 주의력에 의존| **객체 스스로 상태를 검증하고 방어함** |
| **결과** | 거대한 스파게티 절차적 코드 | **응집도 높은 순수 객체지향 코드** |

### 기술적 임팩트: 'Setter'의 종말
DDD에서는 `order.setStatus("CANCEL")`처럼 외부에서 상태를 마음대로 조작하는 것을 극도로 혐오한다. 대신 `order.cancel()` 이라는 행위 메서드를 만들어, 그 안에서 "이미 배송 중이면 취소 불가"라는 도메인 규칙을 객체 스스로 검사하게 만든다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 도메인 설계 전략)

**상황**: 배달 앱에서 '주문 금액'을 계산하는 로직이 서비스 클래스 곳곳에 퍼져 있어, 할인 쿠폰이나 포인트 적용 로직이 바뀔 때마다 버그가 터진다. `int totalAmount`를 단순히 원시 타입(Primitive)으로 쓰고 있다.
**판단 및 DDD (VO) 적용 전략**:
1. **원시 타입 포장 (Primitive Obsession 타파)**: 단순 `int` 대신 **`Money`라는 값 객체(Value Object)**를 생성한다.
2. **로직 캡슐화**: `Money` 객체 안에 `add()`, `subtract()`, `isGreaterThan()` 같은 연산 메서드를 넣고, "금액은 0보다 작을 수 없다"는 검증 로직을 객체 생성자 안에 단단히 박아둔다.
3. **불변성 보장**: `add()` 연산을 하면 기존 Money 값을 고치는 게 아니라, 더해진 새로운 `Money` 객체를 반환하게 짜서 부작용(Side-effect)을 막는다.
4. **결과**: `Money`와 관련된 복잡한 비즈니스 룰을 도메인 객체 한 곳에 응집시켜, 서비스 로직이 놀랍도록 단순해지고 버그 발생률이 제로에 수렴하게 됨.

### 안티패턴 및 고려사항
- **데이터베이스 주도 설계 (DB-Driven)**: ERD(테이블)를 먼저 그리고 거기에 맞춰 엔티티 클래스를 만드는 것. DDD는 도메인 객체를 먼저 설계하고(메모리 위주), 그 객체의 모양을 나중에 DB에 매핑(JPA/Hibernate)하는 **도메인 주도**로 가야 한다.

## Ⅴ. 미래 전망 및 결론

### 결론: 소프트웨어 장인정신의 결정체
도메인 주도 설계는 단기간에 빨리 제품을 찍어내는 데는 적합하지 않다. 하지만 5년, 10년 유지보수하며 끊임없이 진화해야 하는 복잡한 시스템에서는 코드의 부패를 막아주는 유일하고도 가장 훌륭한 백신이다.

### 미래 전망
최근 마이크로서비스(MSA)가 폭발적으로 성장하면서, "서비스를 도대체 어떤 기준으로 쪼개야 하는가?"에 대한 답을 찾기 위해 DDD의 전략적 패턴인 **바운디드 컨텍스트(Bounded Context)**가 모든 클라우드 네이티브 설계의 표준 방법론으로 완전히 자리 잡았다. DDD 없이는 성공적인 MSA도 없다.

## 👶 어린이를 위한 3줄 비유 설명
1. **도메인 주도 설계(DDD)가 뭔가요?**: 코드를 짤 때 컴퓨터가 좋아하는 복잡한 컴퓨터 언어부터 생각하지 않고, '은행원'이나 '요리사'가 일상생활에서 쓰는 진짜 단어들을 모아 코드의 주인공으로 삼는 방법이에요.
2. **왜 그렇게 하나요?**: 컴퓨터 전문가가 아니어도 코드를 보면 "아, 주문을 취소하면 포인트가 깎이는구나!" 하고 동화책 읽듯이 바로 이해할 수 있게 하려고요.
3. **가장 좋은 점은?**: 이렇게 예쁘게 이름을 짓고 각자의 역할을 명확히 주면, 나중에 시스템이 아무리 복잡해져도 로봇(객체)들이 엉뚱한 실수를 하지 않고 똑똑하게 스스로를 지켜낸답니다!
