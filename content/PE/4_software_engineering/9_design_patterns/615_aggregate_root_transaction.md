# 615. 애그리게이트 (Aggregate) 루트와 트랜잭션 경계 - 객체들의 운명 공동체

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 도메인 주도 설계(DDD)에서 연관된 엔티티(Entity)와 값 객체(VO)들을 데이터 변경의 단위로 묶은 군집을 **'애그리게이트(Aggregate)'**라 하며, 이 군집을 외부에서 통제하는 유일한 출입구가 **'애그리게이트 루트(Root)'**다.
> 2. **혁신**: 데이터베이스의 트랜잭션(Commit/Rollback) 경계와 객체의 일관성 유지 경계를 일치시켜, 외부 객체가 내부 객체를 마음대로 수정하여 데이터 정합성이 깨지는 것을 원천적으로 차단한다.
> 3. **가치**: 동시성 문제(Concurrency)와 복잡한 데이터 잠금(Locking) 오버헤드를 애그리게이트 단위로 최소화하여, 안전하고 확장 가능한 마이크로서비스(MSA)와 고성능 백엔드 설계를 가능하게 한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
애그리게이트는 '하나로 뭉친 덩어리'다. 복잡한 시스템에는 수많은 객체가 거미줄처럼 얽혀 있다. DDD는 이 객체들을 논리적으로 강하게 결합된 무리(Aggregate)로 나누고, 각 무리에 대장(Root Entity)을 하나씩 둔다. 외부에서는 오직 이 대장에게만 명령을 내릴 수 있으며, 대장만이 자신의 부하 객체들을 통제할 수 있다.

### 💡 비유
- **패턴 없는 객체망 (난장판 병원)**: 외부인이 병실에 마음대로 들어가 환자(엔티티)에게 주사를 놓고 약(값 객체)을 바꾼다. 누가 언제 환자의 상태를 바꿨는지 기록도 없고 환자는 위험해진다.
- **애그리게이트 루트 (철저한 보호자 통제)**: 환자와 장기, 혈액 수치는 하나의 '환자 애그리게이트'로 묶인다. 그리고 대장인 '주치의(Aggregate Root)'가 문 앞을 지킨다. 외부인은 무조건 주치의에게만 "약을 처방해 주세요"라고 요청해야 한다. 주치의가 허락하고 직접 약을 투여해야만 환자의 상태가 변한다. 완벽한 통제와 일관성이 유지된다.

### 등장 배경: 'Spaghetti Relations'의 파국
엔티티 클래스끼리 무분별하게 양방향 참조를 걸어두면, '주문' 객체를 수정했는데 연결된 '회원' 객체의 상태가 의도치 않게 바뀌고 DB 트랜잭션이 꼬여 데드락(Deadlock)이 발생했다. 복잡성을 통제하기 위해 "한 번의 트랜잭션에서는 단 하나의 애그리게이트만 수정한다"는 강력한 룰이 필요해졌다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 애그리게이트 경계 및 참조 구조 (ASCII Diagram)

```ascii
      [ Order Aggregate (주문 군집) ]             [ Customer Aggregate ]
  ┌─────────────────────────────────────┐      ┌─────────────────────────┐
  │ ┌─────────────────┐                 │      │  ┌────────────────┐     │
  │ │  Order (ROOT)   │                 │      │  │ Customer (ROOT)│     │
  │ ├─────────────────┤    (ID 참조)    │      │  ├────────────────┤     │
  │ │ - orderId       │ ────────────┐   │      │  │ - customerId   │     │
  │ │ - customerId    │             │   │      │  │ - totalPoints  │     │
  │ │ - totalAmount   │             │   │      │  └────────────────┘     │
  │ └────────┬────────┘             │   │      │         ▲               │
  │          │(내부 객체 참조)      │   │      └─────────┼───────────────┘
  │ ┌────────▼────────┐             │   │                │
  │ │  OrderLine (VO) │             │   │       <절대 다른 애그리게이트의
  │ │ - productId     │             │   │        내부 객체를 직접 참조할 수 없음>
  │ │ - price, qty    │             │   │
  │ └─────────────────┘             │   │
  └─────────────────────────────────┼───┘
                                    ▼ (다른 애그리게이트는 오직 대장의 ID만 참조)
```

### 3대 설계 원칙

#### ① 애그리게이트 루트 (Aggregate Root)
군집 내에서 유일하게 전역적 식별자(Global ID)를 가지며, 외부에서 DB 조회(Repository)가 가능한 유일한 엔티티다. `OrderRepository.findById()`는 가능하지만, 내부 부품인 `OrderLineRepository`는 아예 존재해서는 안 된다.

#### ② 일관성 경계 (Consistency Boundary)
애그리게이트 내의 비즈니스 룰은 언제나 완벽해야 한다. "주문 금액의 총합은 개별 상품 금액의 합과 같아야 한다"는 룰이 있다면, 외부에서 상품을 하나 추가할 때 무조건 Root인 `Order`를 통해 추가해야, `Order`가 금액을 다시 계산하여 무결성을 유지한다.

#### ③ 다른 애그리게이트 참조는 오직 ID로만
'주문'이 '고객' 정보를 알고 싶을 때, 객체 포인터(메모리 참조, 예: `order.getCustomer().addPoint()`)를 쓰면 헬게이트가 열린다. 반드시 `customerId`라는 String이나 Long 값(ID)만 들고 있어야 한다. 이는 나중에 마이크로서비스로 찢어질 때 완벽한 분리를 보장한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 트랜잭션 처리 방식 비교: Jpa Entity vs DDD Aggregate

| 비교 항목 | 무분별한 ORM (JPA) 연관관계 | **DDD 애그리게이트 설계** |
|-----------|-----------------------------|---------------------------|
| **트랜잭션 범위**| 여러 객체망을 타고 끝없이 퍼짐 | **단 하나의 애그리게이트로 제한** |
| **객체 참조** | `@ManyToOne`, `@OneToMany` 남발 | **내부는 객체 참조, 외부는 ID 참조**|
| **Repository 수**| 테이블 개수만큼 무한 생성 | **루트(Root) 엔티티당 1개만 존재** |
| **성능 (Locking)**| 테이블 여러 개에 락이 걸려 느림 | 하나의 문서(군집)만 락을 걸어 빠름|
| **MSA 전환** | 쪼개기 불가능 (외래키 지옥) | **ID 참조이므로 즉시 쪼개기 가능** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 동시성 제어 전략)

**상황**: 선착순 티켓 예매 시스템에서 수천 명이 동시에 한 공연의 좌석을 예매하려 한다. DB 데드락이 빈번하고 남은 좌석 수와 실제 예약된 사람 수가 안 맞는 정합성 에러가 터지고 있다.
**판단 및 애그리게이트 경계 재설계**:
1. **AS-IS 문제점**: '회원', '좌석', '공연' 테이블이 하나의 트랜잭션 안에서 묶여 동시에 업데이트되고 있어 락(Lock) 경합이 극심하다.
2. **애그리게이트 도출**: '공연 예매(TicketBooking)'라는 새로운 애그리게이트 루트를 설계한다. 이 안에는 예매 상태와 선택한 좌석(VO)만 둔다.
3. **Optimistic Locking (낙관적 잠금) 적용**: 애그리게이트 루트(대장)에 `Version` 필드를 둔다. 두 명이 동시에 같은 좌석을 선점하려 하면, 먼저 대장을 업데이트한 사람만 버전을 올리고 승리한다. 늦은 사람은 `OptimisticLockException`을 맞고 튕겨 나간다. 대장만 통제하면 내부 룰은 절대 깨지지 않는다.
4. **결과**: 트랜잭션의 범위가 애그리게이트 하나로 극도로 짧아져 데드락이 사라지고, 초당 처리량(TPS)이 10배 이상 향상됨.

### 안티패턴 및 고려사항
- **거대 애그리게이트 (God Aggregate)**: 통제하기 편하다는 이유로 '사용자(User)' 애그리게이트 밑에 사용자가 쓴 '게시글', '댓글', '결제내역'을 모조리 욱여넣는 것. 이 경우 사용자가 프로필 사진을 바꿀 때 다른 사용자가 그 게시글에 댓글을 달면 DB 락이 충돌(Concurrency Conflict)하는 참사가 벌어진다. **애그리게이트는 최대한 작게(Rule of thumb: 1 엔티티 + 몇 개의 VO)** 유지해야 한다.

## Ⅴ. 미래 전망 및 결론

### 결론: MSA를 위한 완벽한 설계도
애그리게이트 설계는 단순히 코드를 깔끔하게 짜는 기법이 아니다. "이 덩어리(애그리게이트)는 미래에 언제든 다른 서버나 다른 DB로 똑 떨어져 나갈 수 있다"는 것을 보장하는 물리적/논리적 절취선이다.

### 미래 전망
관계형 DB(RDBMS)의 외래키(FK) 중심 설계는 객체의 자율성을 훼손하여 점차 힘을 잃고 있다. 애그리게이트 단위로 데이터를 하나의 JSON 문서로 통째로 저장하고 불러오는 **문서형 NoSQL(MongoDB 등)**이나 **이벤트 소싱(Event Sourcing) 아키텍처**가 애그리게이트 개념과 완벽한 찰떡궁합을 이루며 차세대 백엔드 데이터 모델링의 표준이 되고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **애그리게이트가 뭔가요?**: 자동차, 바퀴, 핸들을 따로따로 굴러다니게 두지 않고, 하나의 투명한 '자동차 상자(군집)' 안에 예쁘게 묶어두는 거예요.
2. **루트(대장)는 뭐 하는 애죠?**: 상자 바깥에 있는 '조종석(루트)'이에요. 바퀴를 돌리고 싶으면 내가 직접 상자 안으로 손을 넣어서 바퀴를 굴리면 안 되고, 꼭 조종석에 앉은 대장한테 "핸들 좀 꺾어줘!"라고 부탁해야만 바퀴가 움직여요.
3. **왜 그렇게 귀찮게 해요?**: 내 마음대로 바퀴만 반대로 돌려버리면 자동차가 고장 나잖아요? 똑똑한 대장에게 맡기면 알아서 안전하게 자동차 전체를 조종해 주기 때문이에요!
