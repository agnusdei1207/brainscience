# 617. 서비스 디스커버리 (Eureka) - 마이크로서비스들의 살아있는 전화번호부

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 수시로 생성되고 사라지는 수십~수백 개의 마이크로서비스(MSA) 인스턴스들의 **IP 주소와 포트 번호를 동적으로 관리하고 찾아주는 중앙 레지스트리(Registry) 시스템**이다.
> 2. **혁신**: 서비스 A가 서비스 B를 호출할 때 고정된 IP를 하드코딩하지 않고, 서비스 이름(예: `order-service`)만 부르면 디스커버리 서버가 현재 살아있는 건강한 IP를 즉시 반환(또는 클라이언트가 직접 캐싱)해 준다.
> 3. **가치**: 클라우드 환경의 오토 스케일링(Auto-scaling)으로 인해 IP 주소가 시시각각 변하는 불안정한 네트워크 환경에서, 서비스 간 통신(East-West Traffic)을 완벽하게 자동화하고 유연성을 보장한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
서비스 디스커버리는 MSA 망 안의 '114 안내센터'다. 서비스들은 태어나자마자 디스커버리 서버(예: Netflix Eureka, Consul, Zookeeper)에 자신의 주소(IP)를 등록(Register)하고, 주기적으로 "나 아직 살아있어"라고 생존 신고(Heartbeat)를 보낸다. 다른 서비스를 찾고 싶은 놈은 이 센터에 물어보면 된다.

### 💡 비유
- **모놀리식 환경 (유선 전화)**: 회사 부서들이 평생 안 바뀌는 내선 번호를 쓴다. 마케팅팀은 무조건 101번이다. 전화번호부(설정 파일)를 한 번만 적어놓으면 고칠 일이 없다.
- **MSA 환경 (외근직 영업사원)**: 직원들이 맨날 자리를 옮기고 카페(컨테이너)에서 일한다. 마케팅팀 직원의 핸드폰 번호(IP)가 매일 바뀐다.
- **서비스 디스커버리 (최신 연락처 동기화 앱)**: 직원들이 아침에 눈을 뜨면 앱에 "오늘 제 번호는 010-XXXX입니다"라고 등록한다(Register). 내가 "마케팅팀 김대리 연결해 줘"라고 앱에 치면, 앱이 오늘 자 최신 번호를 찾아서 바로 전화를 걸어준다.

### 등장 배경: 'Static IP'의 종말
클라우드가 대중화되고 도커(Docker) 컨테이너가 쓰이면서 인프라가 미친 듯이 생겼다 사라지는 '탄력성(Elasticity)'을 얻었다. 이 과정에서 서버의 IP가 계속 바뀌는데, 다른 서버 설정 파일에 IP를 고정(하드코딩)해두면 통신이 죄다 끊어지는 대참사가 벌어졌다. 이 유동적인 IP를 실시간으로 추적할 동적 연락망이 필수적으로 요구되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### Eureka 기반 서비스 디스커버리 아키텍처 (ASCII Diagram)

```ascii
                     [ Eureka Server (Service Registry) ]
                     ┌────────────────────────────────┐
                     │ "order-service" : 10.0.0.1:80  │
                     │ "order-service" : 10.0.0.2:80  │
                     │ "user-service"  : 10.0.0.5:80  │
                     └─────────▲───────────────┬──────┘
  (1) Register & Heartbeat │               │ (3) Fetch Registry (주소록 복사)
       (30초마다 생존 신고) │               │
                  ┌────────┴────────┐ ┌────────▼────────┐
                  │ Order Service   │ │  User Service   │
                  │ (Instance 1)    │ │  (Client)       │
                  └─────────────────┘ └────────┬────────┘
                                               │ (4) API Call (10.0.0.1)
                                               ▼
                                      [ Order Service (Target) ]

* 핵심 원리: 클라이언트는 서버에 한 번 물어본 주소록을 자기 메모리에 캐싱(Client-side Discovery)하여 로드밸런싱까지 수행함.
```

### 3대 핵심 작동 메커니즘

#### ① 서비스 등록 (Service Registration)
마이크로서비스가 부팅(Start-up)될 때, 유레카 클라이언트 라이브러리가 자신의 논리적 이름(App Name)과 물리적 IP/포트를 유레카 서버에 등록한다.

#### ② 하트비트 (Heartbeat)와 자가 치유
등록된 서비스는 기본적으로 30초마다 유레카 서버로 핑(Ping)을 보내 "나 살아있다"고 보고한다. 만약 90초 동안 하트비트가 오지 않으면, 유레카 서버는 그 서비스가 죽은 것으로 간주하고 전화번호부에서 삭제(Eviction)해버린다.

#### ③ 클라이언트 사이드 디스커버리 (Client-side Discovery)
호출하는 쪽(User Service)이 통신할 때마다 유레카 서버에 물어보면 유레카 서버가 병목이 된다. 따라서 클라이언트가 주기적으로 유레카 서버의 전체 주소록을 자기 메모리로 훔쳐 온다(Fetch). 그리고 자기 안에서 스스로 로드밸런싱(Ribbon 등)을 돌려 대상 서버를 직접 찌른다.

## Ⅲ. 융합 비교 및 다각도 분석

### 클라이언트 사이드 vs 서버 사이드 디스커버리

| 비교 항목 | **Client-side (Eureka, Consul)** | Server-side (K8s Service, AWS ELB) |
|-----------|----------------------------------|------------------------------------|
| **로드밸런싱 위치**| 클라이언트(호출자)가 직접 수행 | 로드밸런서 장비(프록시)가 수행 |
| **병목(Bottleneck)**| 없음 (P2P 통신) | 프록시가 트래픽 병목이 될 수 있음 |
| **언어 종속성** | 클라이언트에 특정 언어(Java 등) 라이브러리 탑재 필수 | 언어에 독립적 (네트워크 레벨 처리) |
| **장점** | 자유로운 로드밸런싱 라우팅 규칙 구현 | 코드가 깔끔해지고 적용이 매우 쉬움 |
| **트렌드** | K8s 등장 이전의 대세 (Spring Cloud) | **K8s 등장 이후의 현재 대세 (Service Mesh)** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 아키텍처 의사결정)

**상황**: 레거시 VM 서버 50대에서 돌아가던 Spring Boot 모놀리식 앱을 MSA로 쪼갰다. 아직 쿠버네티스(K8s)를 도입할 인프라적 역량이 안 되어, 일반 EC2 인스턴스 위에서 MSA를 돌려야 한다.
**판단 및 Eureka 도입 전략**:
1. **Eureka Server 군집화**: SPOF(단일 장애점)를 막기 위해 유레카 서버 자체를 3대로 띄워 서로 주소록을 복제(Peer-to-Peer 복제)하게 구성한다.
2. **Client-side 로드밸런싱**: 각 서비스에 `Spring Cloud Netflix Eureka Client`와 클라이언트 로드밸런서(`Spring Cloud LoadBalancer`)를 탑재한다.
3. **서킷 브레이커 연동**: 주소록이 있어도 일시적으로 뻗은 서버를 호출할 수 있으므로, [서킷 브레이커(Resilience4j)](./618_circuit_breaker.md)를 함께 묶어 장애가 난 IP를 즉시 우회하도록 세팅한다.
4. **결과**: 쿠버네티스 없이도 애플리케이션 레벨의 완벽한 자동 스케일링과 서비스 추적 기능을 구현함.

### 안티패턴 및 고려사항
- **좀비 인스턴스 (Self-Preservation Mode)**: 유레카의 치명적 단점. 갑자기 일시적인 네트워크 단절로 전체 서버의 85%가 하트비트를 보내지 못하면, 유레카는 "이건 서버가 죽은 게 아니라 잠시 망이 끊긴 거다!"라고 오판하여 주소록에서 죽은 서버들을 삭제하지 않는 '자기 보호 모드'에 들어간다. 이 상태에서 진짜 죽은 서버로 트래픽이 몰리면 대형 장애가 터진다. 망이 불안정한 곳에서는 이 기능을 강제로 꺼야(Disable) 할 때도 있다.

## Ⅴ. 미래 전망 및 결론

### 결론: MSA의 자율성을 여는 열쇠
서비스 디스커버리는 MSA가 수백 개로 쪼개져도 통신이 미아가 되지 않도록 꽉 잡아주는 필수 인프라다. 넷플릭스 유레카(Eureka)는 이 개념을 전 세계 소프트웨어 엔지니어들에게 각인시킨 일등 공신이다.

### 미래 전망
현재 순수 애플리케이션 레벨의 서비스 디스커버리(Eureka)는 **쿠버네티스(K8s)의 CoreDNS 기반 Server-side Discovery**와 **Service Mesh (Istio)** 기술로 급격히 대체되고 있다. 개발자가 소스 코드에 라이브러리를 박아 넣어야 하는 유레카의 불편함(언어 종속성)을 버리고, 인프라 계층(사이드카 프록시)에서 알아서 주소를 찾아주는 투명한(Transparent) 방향으로 기술의 주도권이 완전히 넘어가는 중이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **서비스 디스커버리가 뭔가요?**: 매일매일 이사 다니는 홍길동, 김철수 친구들의 '오늘의 바뀐 집 주소'를 실시간으로 다 외우고 있는 아주 똑똑한 동네 이장님이에요.
2. **왜 이장님이 필요한가요?**: 내가 홍길동에게 놀러 가고 싶은데 매번 주소가 바뀌니까 찾을 수가 없잖아요. 그럴 때 이장님한테 전화해서 "길동이 오늘 주소 어때요?" 하고 물어보면 바로 알려주거든요.
3. **가장 신기한 점은?**: 친구들이 실수로 다치거나 잠들면 이장님이 딱 알아채고 주소록에서 쫙 지워버려서, 내가 아픈 친구 집에 헛걸음하지 않게 막아준답니다!
