# 620. 이벤트 소싱 (Event Sourcing) - 상태의 덮어쓰기를 거부하는 불변의 역사책

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스에 객체의 '현재 상태(Current State)'만을 저장하는 전통적인 방식을 버리고, 상태를 변경시킨 **'모든 사건(Event)들의 연속된 이력'을 순차적으로 저장하여 데이터의 진실(Source of Truth)을 구성**하는 아키텍처 패턴이다.
> 2. **혁신**: 데이터의 업데이트(`UPDATE`)나 삭제(`DELETE`)가 아예 존재하지 않고 오직 추가(`INSERT` / Append-only)만 발생하므로, 데이터베이스 락(Lock) 경합이 사라지고 극강의 쓰기 성능과 동시성(Concurrency)을 얻게 된다.
> 3. **가치**: 과거의 어떤 시점으로든 타임머신을 타고 상태를 완벽하게 재구성(Replay)할 수 있어 버그 추적과 회계 감사가 완벽해지며, 마이크로서비스(MSA) 간의 확실한 이벤트 전파(Message Broker 연동)를 보장한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
이벤트 소싱에서 '이벤트'는 과거형 시제로 명명된 불변의 사실(예: `OrderCreated`, `ItemAdded`, `ShippingAddressChanged`)을 뜻한다. 사용자의 잔고가 현재 5만 원이라는 것을 저장하지 않고, "10만 원 입금됨, 3만 원 출금됨, 2만 원 출금됨"이라는 이벤트 3개만 저장한다. 현재 잔고를 알고 싶으면 이 3개의 이벤트를 순서대로 더해서(Replay) 5만 원이라는 상태를 도출해 낸다.

### 💡 비유
- **일반 DB 방식 (통장 잔액 덮어쓰기)**: 은행원이 통장 맨 뒷장에 "현재 잔고 5만 원"이라고만 연필로 쓴다. 내일 1만 원을 빼면 지우개로 5만 원을 지우고 4만 원이라고 고쳐 쓴다. 한 달 뒤에 내가 "내 돈이 왜 4만 원밖에 안 남았지?" 물어봐도 은행원은 이유를 모른다.
- **이벤트 소싱 (은행 거래 내역서)**: 지우개가 없다. 통장에 무조건 "언제 얼마 입금, 언제 얼마 출금" 내역(이벤트)만 볼펜으로 계속 줄줄이 적는다. 누군가 내 잔고를 물어보면 맨 윗줄부터 끝까지 덧셈 뺄셈을 쫙 한 번 해서 대답해 준다. 거래 내역이 완벽히 남아있어 돈이 사라진 이유를 100% 증명할 수 있다.

### 등장 배경: 'Update'의 한계와 비즈니스 맥락의 증발
RDBMS에서 `UPDATE`를 하는 순간, 이전의 데이터는 영원히 사라진다. 사용자가 장바구니에 물건을 넣었다가 뺀 사실(맥락)은 마케팅적으로 엄청난 가치가 있지만 일반 DB에서는 최종 결제 결과만 남고 과정은 증발한다. 모든 사용자 행동(도메인 이벤트)을 하나도 버리지 않고 영구 기록하려는 욕구가 이벤트 소싱을 탄생시켰다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 이벤트 소싱 아키텍처 흐름 (ASCII Diagram)

```ascii
[ User Action: "장바구니에 사과 1개 추가" ]
          │
          ▼ 1. Command (명령)
┌──────────────────────────────────────┐
│ [ Application Command Handler ]      │
│  - 기존 이벤트를 Replay하여 현재 상태 확인 │
│  - 비즈니스 룰 검증 (사과 재고 있나?)     │
└─────────────────┬────────────────────┘
                  │ 2. Event Created (ItemAddedEvent)
                  ▼
┌──────────────────────────────────────┐
│ [ Event Store (Append-only DB) ]     │  <-- Single Source of Truth
│ 1: CartCreatedEvent                  │
│ 2: ItemAddedEvent (Banana, 1)        │
│ 3: ItemAddedEvent (Apple, 1)   <-- 새 이벤트 추가 (UPDATE 없음)
└─────────────────┬────────────────────┘
                  │ 3. Publish Event (비동기)
                  ▼
        [ Kafka / RabbitMQ ] ───> [ 다른 마이크로서비스 (결제, 배송) ]
```

### 3대 핵심 메커니즘

#### ① 이벤트 스토어 (Event Store)
이벤트를 물리적으로 저장하는 특수 DB다. 이벤트는 무조건 꼬리에 꼬리를 물고 추가(Append)만 되므로 데이터 파편화가 없고 I/O 속도가 디스크 한계까지 뽑힌다. (EventStoreDB, Apache Kafka, 또는 일반 RDB를 활용하기도 함).

#### ② 리플레이 (Replay) 와 프로젝션 (Projection)
데이터를 읽어야 할 때 매번 10년 치 이벤트를 처음부터 계산하면 너무 느리다. 그래서 '현재 상태'만 쫙 정리해서 보여주는 읽기 전용 뷰(Read Model)를 별도의 DB(예: Redis나 MongoDB)에 몰래 업데이트해 둔다. 이를 **프로젝션**이라고 한다.

#### ③ 스냅샷 (Snapshot)
이벤트가 1만 개 쌓이면 아무리 빨라도 리플레이에 시간이 걸린다. 그래서 1,000번째 이벤트마다 "1000번까지의 결과는 잔고 50만 원"이라는 스냅샷을 찍어둔다. 다음번엔 처음부터가 아니라 스냅샷 이후의 이벤트만 리플레이하여 성능을 높인다.

## Ⅲ. 융합 비교 및 다각도 분석

### 상태 저장(CRUD) vs 이벤트 소싱 비교

| 비교 항목 | 기존 CRUD (State-oriented) | **이벤트 소싱 (Event-oriented)** |
|-----------|----------------------------|----------------------------------|
| **저장 방식** | 현재 상태를 덮어씀 (Update/Delete) | **발생한 사실을 계속 추가함 (Append)** |
| **데이터 진실성**| 마지막 상태만 알 수 있음 | **변경의 모든 과정과 이유(Context) 보존** |
| **쓰기 성능** | 인덱스 업데이트와 Lock 경합으로 느려짐 | **경합 없는 순차 쓰기로 압도적으로 빠름** |
| **조회 성능** | 즉시 조회 가능 (빠름) | **이벤트 리플레이 필요 (Projection 없으면 느림)** |
| **에러 복원력** | 백업본이 없으면 복구 불가능 | **이벤트를 거꾸로 지우거나 다시 재생하여 100% 복구**|

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 CQRS 결합 전략)

**상황**: 암호화폐 거래소의 '지갑(Wallet) 코어 시스템'을 개발한다. 초당 10만 건의 입출금이 터지고, 금융 당국의 감사(Audit)를 위해 단 1원도 틀리지 않는 완벽한 거래 내역 증명이 필요하다.
**판단 및 이벤트 소싱 + CQRS 아키텍처 전략**:
1. **쓰기(Write) 설계**: 모든 입출금 행위를 이벤트 소싱으로 설계하여 **Event Store**에 순차 기록한다. 락(Lock)이 없어 초당 10만 건의 쓰기를 가뿐히 처리한다.
2. **읽기(Read) 분리 (CQRS)**: 사용자가 앱에서 "내 잔고"를 볼 때 리플레이를 하면 느리므로, Event Store에 이벤트가 꽂힐 때마다 메시지 큐(Kafka)를 통해 읽기 전용 DB(Redis)의 잔고 값을 즉시 갱신해 둔다.
3. **버그 복구(Time Travel)**: 어제 배포한 코드에 버그가 있어 1만 명의 잔고가 틀어졌다. 당황하지 않고, 어제 배포 시점 이후의 이벤트를 무효화(보상 이벤트 발행)하고 버그를 고친 후 이벤트를 다시 리플레이하여 잔고를 100% 원상 복구한다.

### 안티패턴 및 고려사항
- **러닝 커브와 디버깅 지옥**: 이벤트 소싱은 개발 패러다임이 완전히 달라서 개발자들이 적응하기 극도로 힘들다. DB를 열어봐도 '현재 상태'가 없으니 직관적 디버깅이 불가능하다. 모든 도메인에 이벤트 소싱을 적용하는 것은 최악의 오버엔지니어링이며, **돈이나 계약이 오가는 핵심(Core) 트랜잭션 도메인에만 제한적으로 사용**해야 한다.

## Ⅴ. 미래 전망 및 결론

### 결론: 데이터 철학의 완벽한 징반합
이벤트 소싱은 단순히 코딩 패턴이 아니다. "세상의 모든 상태는 과거에 일어난 사건들의 누적된 결과일 뿐이다"라는 철학적 명제를 컴퓨터 아키텍처로 증명해 낸 아름다운 설계다.

### 미래 전망
분산 시스템(MSA)이 커지면서 두 서비스 간의 데이터 동기화(Outbox Pattern 등) 문제가 극심해지고 있다. 이벤트 소싱은 애초에 '저장'과 '이벤트 발행'이 하나의 행위로 통합되므로 이 문제를 원천적으로 해결한다. 향후 **CQRS(명령과 조회의 분리)**와 결합된 이벤트 소싱 아키텍처는 고성능/고가용성을 요구하는 모든 클라우드 네이티브 핵심 금융/커머스 시스템의 교과서적 표준으로 굳어질 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **이벤트 소싱이 뭔가요?**: 내 돼지저금통에 지금 '얼마가 있는지' 적지 않고, "어제 백 원 넣음, 오늘 오백 원 넣음, 동생이 백 원 빼감"처럼 내가 한 행동(이벤트)만 빠짐없이 일기장에 계속 적는 거예요.
2. **지금 얼마 있는지 알려면 어떻게 해요?**: 일기장 첫 장부터 지금까지 적힌 돈을 덧셈 뺄셈으로 쭈욱 계산해 보면, 내 저금통에 정확히 얼마가 남아있는지 알 수 있어요.
3. **가장 좋은 점은?**: 엄마가 "너 돈 어디다 썼어?" 물어볼 때 당황하지 않고 일기장을 보여주며 "동생이 훔쳐 갔잖아요!"라고 100% 완벽한 증거를 댈 수 있답니다!
