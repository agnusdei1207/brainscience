# 621. CQRS (Command and Query Responsibility Segregation) - 읽기와 쓰기의 완벽한 이혼

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스의 상태를 변경하는 **'명령(Command, 쓰기/수정)' 작업**과 데이터를 읽어오는 **'조회(Query, 읽기)' 작업**을 완전히 분리하여 서로 다른 모델과 구조로 설계하는 아키텍처 패턴이다.
> 2. **혁신**: 복잡한 비즈니스 로직(도메인 모델)은 쓰기(Command) 모델에만 집중시키고, 조회를 위한 모델은 화면 출력에 딱 맞게(DTO/View) 단순화시켜 성능과 복잡도라는 두 마리 토끼를 모두 잡는다.
> 3. **가치**: 극단적인 트래픽이 몰리는 서비스(예: 초당 10만 번 조회, 10번 결제)에서, 읽기 DB와 쓰기 DB를 물리적으로 쪼개어(Scale-out) 병목 현상을 원천적으로 제거하는 마이크로서비스(MSA)의 필수 무기다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
CQRS는 "상태를 바꾸는 메서드(명령)는 값을 반환하면 안 되고, 값을 반환하는 메서드(조회)는 상태를 바꾸면 안 된다"는 객체지향의 대원칙(CQS: Command Query Separation)을 아키텍처 수준으로 확장한 것이다. 즉, 쓰기 전용 API(와 DB)와 읽기 전용 API(와 DB)를 아예 둘로 찢어버리는 기법이다.

### 💡 비유
- **일반 아키텍처 (혼자 다 하는 은행원)**: 은행원 1명이 돈을 입금(쓰기)하는 아주 복잡하고 조심스러운 작업과, 단순히 잔고를 물어보는(읽기) 가벼운 작업을 혼자 다 처리한다. 잔고 확인하려는 수백 명의 손님 때문에 진짜 입금해야 하는 손님이 1시간을 기다려야 한다.
- **CQRS (창구 완벽 분리)**: '입금 전문 창구(Command)'와 '잔액 조회 전광판(Query)'을 완전히 분리한다. 입금 창구에는 꼼꼼한 전문가 1명만 두고 깐깐하게 작업한다. 반면 전광판은 수백만 명이 동시에 봐도 전혀 문제가 없다. 손님들은 기다림 없이 각자의 볼일을 순식간에 끝낸다.

### 등장 배경: '하나의 모델'이 주는 고통
과거에는 JPA나 ORM을 쓸 때 `User` 엔티티 하나로 데이터도 저장하고, 화면에 그릴 리스트도 뽑아냈다. 그러다 보니 화면에 보여줄 텍스트를 조합하기 위해 순수한 도메인 객체 안에 `getUserDisplayNameForMobile()` 같은 잡다한 화면 전용 코드가 덕지덕지 붙어 모델이 뚱뚱해지고 유지보수가 불가능해졌다(괴물 객체 탄생). 읽는 놈과 쓰는 놈의 목적이 다르니 아예 분리하자는 아이디어가 나왔다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### CQRS의 3가지 진화 단계 (ASCII Diagram)

```ascii
[ 1단계: 코드 분리 (단일 DB) ]
  ┌─ Command Service ──> (쓰기용 엔티티 모델 적용) ──┐
UI┤                                                  ├─> [ 단일 RDBMS ]
  └─ Query Service ────> (단순 SQL/뷰 모델 적용) ────┘

========================================================================
[ 2단계: DB 물리적 분리 (Master-Slave 복제) ]
  ┌─ Command API ────> [ 쓰기 전용 DB (Master) ] ────(DB Replication)────┐
UI│                                                                      ▼
  └─ Query API ──────> [ 읽기 전용 DB (Slave 1) ] [ 읽기 전용 DB (Slave 2) ]

========================================================================
[ 3단계: 궁극의 CQRS (이기종 DB 분리 + 이벤트 동기화) ]
  ┌─ Command API ────> [ 쓰기 DB (관계형/EventStore) ]
UI│                           │ (이벤트 발행: "데이터 변함!")
  │                           ▼
  │                    [ Message Broker (Kafka) ]
  │                           │ (비동기 구독)
  │                           ▼
  └─ Query API <────── [ 읽기 DB (NoSQL / Redis / Elasticsearch) ]
                       (화면에 그리기 딱 좋은 포맷으로 미리 저장해둠)
```

### 핵심 매커니즘 (3단계 기준)

#### ① 독립된 모델 유지
- **명령(Command)**: 데이터 정합성이 생명이므로 무거운 도메인 로직, 상태 검증(Validation), 정규화된 RDBMS를 사용한다.
- **조회(Query)**: 속도가 생명이므로 조인(JOIN)을 하지 않아도 되게끔 데이터를 뭉뚱그려(Denormalization) 캐시 메모리나 Elasticsearch 같은 빠른 저장소에 넣어둔다.

#### ② 비동기 이벤트 동기화 (Event Synchronization)
쓰기 DB에 데이터가 저장되면, 이 사실을 카프카(Kafka) 같은 브로커에 던진다. 읽기 쪽 담당자(Projector)가 이 이벤트를 주워다 읽기 DB의 모양에 맞게 변환하여 쓱 업데이트한다. 이 과정은 비동기(Asynchronous)로 일어난다.

#### ③ 최종적 일관성 (Eventual Consistency) 감수
쓰자마자 바로 읽으면 업데이트가 아직 안 되어 옛날 데이터가 보일 수 있다. CQRS는 "당장 0.1초 동안 데이터가 옛날 거라도, 결국엔 최신으로 맞춰진다(최종적 일관성)"는 철학을 기꺼이 받아들인다.

## Ⅲ. 융합 비교 및 다각도 분석

### 단일 모델(CRUD) vs 궁극의 CQRS 비교

| 비교 항목 | 전통적 CRUD 아키텍처 | **CQRS (3단계 구조)** |
|-----------|----------------------|-----------------------|
| **도메인 모델** | 하나의 뚱뚱한 객체가 모두 처리 | **쓰기 객체와 읽기(DTO) 객체 완벽 분리** |
| **데이터베이스**| 1개의 DB로 조인 연산 남발 | **RDBMS(쓰기)와 NoSQL(읽기) 이기종 사용**|
| **확장성** | 읽기/쓰기가 얽혀 스케일 아웃 어려움 | **조회 트래픽만 폭주하면 읽기 DB만 100배 늘림** |
| **복잡도** | 구조가 직관적이고 쉬움 | **아키텍처가 매우 복잡하고 동기화 버그 잡기 지옥**|
| **데이터 일관성**| 즉각적 일관성 보장 (ACID) | **최종적 일관성 (약간의 딜레이 발생)** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 대규모 검색/추천 시스템 적용)

**상황**: 배달 앱의 메인 화면이다. "현재 내 위치에서 2km 반경, 평점 4 이상, 치킨 카테고리"를 만족하는 매장 리스트를 띄워야 한다. 초당 수만 명이 앱을 켠다. 이 데이터를 RDBMS에서 JOIN으로 쿼리하니 서버가 죽어버렸다.
**판단 및 CQRS 전환 전략**:
1. **Command 분리**: 사장님이 메뉴를 수정하거나 영업시간을 바꾸는 작업은 기존의 정규화된 **MySQL(쓰기 DB)**에서 안전하게 처리한다.
2. **이벤트 브로커 연동**: 사장님이 데이터를 바꿀 때마다 `MenuUpdatedEvent` 등을 **Kafka**로 발행한다.
3. **Query 모델 (Elasticsearch) 구축**: 이벤트를 구독하여 화면에 그리기 가장 완벽한 JSON 문서 형태로 조합한 뒤, 공간 검색과 필터링에 극강인 **Elasticsearch(읽기 DB)**에 저장해 둔다.
4. **결과**: 사용자가 앱을 켜면 MySQL은 건드리지도 않고 Elasticsearch에서 0.01초 만에 화면을 쫙 뿌려준다. 쓰기 트래픽(사장님 앱)과 읽기 트래픽(고객 앱)이 완벽히 물리적으로 격리되어 시스템 생존성이 1000% 향상됨.

### 안티패턴 및 고려사항
- **무조건적 CQRS 도입 (오버엔지니어링)**: 관리자용 백오피스 게시판처럼 사용자가 10명이고 트래픽도 없는 곳에 이기종 DB를 깔고 카프카로 이벤트를 쏘는 짓을 하면 시스템 유지보수 비용에 회사가 망한다. CQRS는 로직이 복잡하거나 트래픽 비대칭이 극단적인 '핵심 도메인'에만 국소적으로 써야 하는 마취제다.

## Ⅴ. 미래 전망 및 결론

### 결론: 조회와 갱신의 이질성 인정
읽기와 쓰기는 완전히 다른 종족이다. 성격이 다른 둘을 한 곳에 가둬두면 반드시 사고가 난다. CQRS는 이 둘의 완벽한 이혼을 통해 각자의 영역에서 최고의 퍼포먼스를 내게 해주는 통찰력 깊은 패턴이다.

### 미래 전망
CQRS는 상태를 아예 저장하지 않고 과거의 기록만 추가하는 **[이벤트 소싱(Event Sourcing)](./620_event_sourcing.md)** 패턴과 영혼의 단짝이다. 이벤트 소싱으로 극강의 쓰기 속도를 내고, CQRS의 읽기 DB(Projection)로 검색 속도를 내는 아키텍처가 결합하여, 향후 초대규모 클라우드 네이티브 금융/커머스 시스템의 궁극적인 정답지(Defacto Standard)로 자리매김할 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **CQRS가 뭔가요?**: 일기를 쓰는 일(쓰기)과 일기를 사람들에게 보여주는 일(읽기)을 완전히 쪼개서, 일기장도 따로 만들고 담당자도 따로 두는 방법이에요.
2. **왜 쪼개나요?**: 글 쓰는 작가는 펜으로 조심조심 쓰느라 바쁜데, 수만 명의 팬이 와서 "일기 좀 보자!"라고 매달리면 작가가 글을 아예 못 쓰게 되거든요.
3. **어떻게 해결하나요?**: 작가는 비밀 방에서 글만 쓰고(명령), 글이 완성되면 밖의 거대한 게시판(조회)에 복사해서 똭! 붙여놔요. 팬들은 게시판만 보면 되니까 작가를 방해할 일이 전혀 없답니다!
