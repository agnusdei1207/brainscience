# 622. 모듈러 모놀리스 (Modular Monolith) - MSA의 지옥에서 탈출한 현실적 타협안

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로서비스 아키텍처(MSA)처럼 시스템을 논리적인 도메인(모듈) 단위로 완벽하게 쪼개면서도, 배포와 실행은 과거의 모놀리식(Monolithic)처럼 **하나의 거대한 애플리케이션 안에서 띄우는 하이브리드 아키텍처**이다.
> 2. **혁신**: MSA의 치명적 단점인 네트워크 통신 지연(Latency), 복잡한 분산 트랜잭션, 지옥 같은 인프라 관리 비용을 0으로 만들면서도, 모놀리식의 단점인 '스파게티 코드'를 도메인 격리로 완벽히 방어한다.
> 3. **가치**: "처음부터 MSA로 시작하지 마라"는 업계의 뼈저린 교훈 속에서 탄생했으며, 프로젝트 초기나 중견 규모의 시스템에서 **높은 개발 생산성과 독립적인 모듈 구조를 동시에 챙길 수 있는 가장 우아하고 실용적인 설계 방식**이다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
모듈러 모놀리스는 하나의 소스 코드 덩어리(Codebase)와 하나의 DB를 사용한다(Monolith). 하지만 그 내부를 들여다보면 주문(Order), 결제(Payment), 배송(Shipping) 등의 모듈이 자바의 `패키지(Package)`나 `접근 제어자(Internal/Private)`를 통해 철저하게 벽이 쳐져 있다(Modular). 모듈 간의 통신은 네트워크 API가 아니라 단순한 메모리 함수 호출(Method Call)로 일어난다.

### 💡 비유
- **거대 모놀리스 (칸막이 없는 텐트)**: 수십 명의 가족이 거대한 텐트 안에서 칸막이 없이 산다. 프라이버시가 없고 서로 물건이 섞여 아수라장이 된다. (스파게티 코드).
- **마이크로서비스 (멀리 떨어진 1인 가구들)**: 텐트에서 나와 각자 다른 도시에 원룸을 얻어 독립했다. 사생활(독립성)은 완벽하지만, 가족끼리 회의 한 번 하려면 기차(네트워크)를 타고 가야 하고 식비(인프라 비용)가 수십 배로 든다.
- **모듈러 모놀리스 (방음벽이 쳐진 거대한 아파트)**: 모두가 '하나의 아파트 건물(단일 배포)' 안에 산다. 하지만 각자의 방은 아주 두꺼운 방음벽(모듈 격리)으로 완벽히 나뉘어 있다. 프라이버시도 보장되고, 할 얘기가 있으면 거실로 잠깐 나오면 되니(메모리 호출) 기차를 탈 필요도 없다.

### 등장 배경: MSA에 데인 사람들의 반성
몇 년 전 "모든 것은 MSA로 쪼개야 한다"는 광풍이 불었다. 작은 스타트업들마저 앱을 30개의 컨테이너로 쪼갰다가, 분산 데이터베이스 정합성 파괴와 인프라 비용 폭발에 시달리며 폐업 위기에 처했다. 결국 "아키텍처 설계가 나빴던 거지, 모놀리식 자체가 나빴던 게 아니다"라는 자성이 일었고, 코드를 쪼개되 물리적 서버는 쪼개지 않는 모듈러 모놀리스가 구원투수로 떠올랐다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 모듈러 모놀리스 아키텍처 (ASCII Diagram)

```ascii
[ Single Application Process (배포 덩어리 1개) ]
┌────────────────────────────────────────────────────────┐
│                        (Memory Method Calls)           │
│  ┌────────────────┐         ┌─> ┌────────────────┐     │
│  │ Module: Order  │ ──(API)─┘   │ Module: Pay    │     │
│  │ [Public API]   │ <──(API)──  │ [Public API]   │     │
│  │ [Internal Biz] │             │ [Internal Biz] │     │
│  │ [Internal DB]  │             │ [Internal DB]  │     │
│  └────────────────┘             └────────────────┘     │
│                                                        │
└────────────────────────────┬───────────────────────────┘
                             │
            [ Single Database (Schema 1개) ]
  (단, 논리적으로 Order 테이블과 Pay 테이블의 강제 JOIN은 금지됨)
```

### 3대 핵심 작동 메커니즘

#### ① 엄격한 모듈 경계 (Strict Boundaries)
자바(Java)를 예로 들면, `Payment` 모듈 안에 있는 세부 비즈니스 로직 클래스는 `public`이 아니라 `package-private(default)`로 선언된다. 오직 외부에 열어둔 `PaymentApi` 인터페이스(Facade)만 `public`이다. 이로 인해 `Order` 개발자는 `Payment`의 내부를 알 수도 없고 직접 호출할 수도 없게 강제된다.

#### ② 데이터베이스의 논리적 격리
물리적으로는 하나의 거대한 MySQL 서버를 쓰지만, 모듈 A가 모듈 B의 테이블에 직접 SQL JOIN을 거는 행위는 린터(Linter)나 코드 리뷰 단계에서 철저히 금지된다. 모듈 B의 데이터가 필요하면 모듈 B의 `Public API` (함수)를 호출해서 받아와야 한다.

#### ③ 인메모리 통신 (In-memory Communication)
모듈 간 통신은 HTTP나 gRPC 같은 네트워크를 타지 않고, 같은 메모리 공간 안에서 함수를 부르거나 인메모리 이벤트 버스(예: Spring ApplicationEventPublisher)를 통해 이루어진다. 실패 확률이 거의 0%고 지연 시간(Latency)은 나노초 수준이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 모놀리스 vs 모듈러 모놀리스 vs MSA

| 비교 항목 | 전통적 모놀리스 (Big Ball of Mud) | **모듈러 모놀리스 (Modular Monolith)** | 마이크로서비스 (MSA) |
|-----------|-----------------------------------|----------------------------------------|----------------------|
| **코드 결합도** | 매우 높음 (스파게티) | **낮음 (모듈 간 격리)** | 매우 낮음 |
| **배포 단위** | 1개 | **1개 (단일 애플리케이션)** | N개 (독립 배포) |
| **성능 (통신)** | 빠름 (함수 호출) | **빠름 (함수/인메모리 호출)** | 느림 (네트워크 호출) |
| **인프라 비용** | 매우 낮음 | **낮음** | 매우 높음 (K8s, API Gateway 등 필수)|
| **트랜잭션** | 쉽고 안전함 (로컬 DB) | **쉽고 안전함 (로컬 DB)** | 지옥 (사가 패턴, 2PC 필요) |
| **확장성(Scale)**| 전체 서버를 복제해야 함 | 전체 서버를 복제해야 함 | 특정 서비스만 무한 확장 가능 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 아키텍처 의사결정 가이드)

**상황**: 시리즈 A 투자를 받은 20명 규모의 스타트업. 사용자가 조금씩 늘고 있어 서비스를 전면 개편하려 한다. CTO가 최신 유행인 쿠버네티스 기반의 30개 MSA로 가자고 강력히 주장한다.
**판단 및 모듈러 모놀리스 방어 논리**:
1. **분산 프리미엄(Distributed Premium) 회피**: 20명의 인력으로 MSA의 분산 트랜잭션, 분산 로깅, 네트워크 실패를 디버깅할 시간적/비용적 여유가 없다. MSA로 가는 순간 개발 속도는 반토막 난다.
2. **모듈러 모놀리스 선택**: 코드를 **도메인 단위(바운디드 컨텍스트)**로 철저히 쪼개는 '모듈러 모놀리스'를 도입한다. ArchUnit 같은 테스트 도구를 빌드 파이프라인에 넣어, 다른 모듈을 함부로 침범하는 코드가 있으면 빌드를 실패시켜 아키텍처를 강제한다.
3. **미래 지향적 설계**: 모듈러 모놀리스로 경계가 완벽히 나뉘어 있기 때문에, 2년 뒤 회사가 대박이 나서 결제(Pay) 쪽에 트래픽이 몰리면, 그때 결제 모듈 코드를 칼로 자르듯 툭 떼어내서(Extract) 진짜 마이크로서비스로 분리(Strangler Pattern)하면 된다.
4. **결과**: 인프라 비용을 최소화하면서도, MSA의 '코드 독립성'이라는 가장 큰 이점을 100% 챙겨 신속하게 제품(MVP)을 시장에 출시함.

### 안티패턴 및 고려사항
- **허울뿐인 모듈화**: 폴더만 `order`, `pay`로 예쁘게 나눠놓고, 실제로는 `order` 안의 코드가 `pay` 안의 JPA Entity 객체를 직접 가져와 마음대로 `save()`를 호출하는 현상. 이럴 거면 그냥 스파게티 코드와 다를 바 없다. 자바 9의 모듈 시스템(JPMS)이나 헥사고날 아키텍처를 이용해 물리적/컴파일 수준의 격리를 구축해야 한다.

## Ⅴ. 미래 전망 및 결론

### 결론: 아키텍처의 황금 중도 (Golden Mean)
마틴 파울러(Martin Fowler)는 "모놀리식 시스템을 제대로 쪼개지 못하는 팀은 마이크로서비스도 결코 성공시킬 수 없다"고 일갈했다. 모듈러 모놀리스는 MSA의 달콤한 이상향으로 가기 전에 조직의 설계 역량을 증명해야 하는 필수적인 '통과 의례'이자 훌륭한 종착지다.

### 미래 전망
앞으로는 **프레임워크가 아키텍처 진화를 자동화하는 시대**가 온다. 구글의 Service Weaver 같은 프레임워크는 개발자가 하나의 모듈러 모놀리스 코드로 작성을 마치면, 배포할 때 프레임워크가 알아서(마치 컴파일러처럼) 코드를 여러 개의 마이크로서비스로 찢어서 클라우드에 올려버리는 마법을 연구 중이다. 즉, 코딩은 모놀리스로 편하게 하고 실행은 MSA로 하는 시대가 열릴 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **모듈러 모놀리스가 뭔가요?**: 커다란 장난감 상자 하나에 모든 걸 다 넣는 건 똑같은데, 그 상자 안에 다시 아주 튼튼한 '칸막이'를 쳐서 자동차, 인형, 공을 절대 섞이지 않게 보관하는 방법이에요.
2. **왜 아예 상자를 여러 개 쓰지 않나요?**: 상자(서버)가 너무 많아지면 관리하기도 힘들고 이사 갈 때 돈도 많이 들거든요. 하나의 큰 상자를 쓰면 옮기기 아주 편해요.
3. **가장 좋은 점은?**: 짐 싸기도 엄청 쉬우면서도 칸막이가 튼튼해서, 눈 감고 손만 넣어도 자동차가 어디 있는지 완벽하게 찾을 수 있답니다!
