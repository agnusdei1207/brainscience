# 345. 배치 처리 (Batch Processing) 파이프라인 - 대규모 데이터의 정기적 정전기

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 실시간 응답이 필요 없는 대용량 데이터를 일정 기간(일, 주, 월) 동안 모았다가 **한꺼번에(Batch) 일괄적으로 처리**하는 데이터 아키텍처 및 공정이다.
> 2. **혁신**: 처리량을 극대화하기 위해 자원을 집중 투입하며, 실패 시 처음부터 다시 실행하거나 특정 지점(Checkpoint)부터 재시작할 수 있는 결함 허용(Fault Tolerance)을 보장한다.
> 3. **가치**: 데이터 웨어하우스(DW) 구축을 위한 ETL(Extract, Transform, Load)의 근간이며, 복잡한 통계 분석 및 머신러닝 모델 학습 등 대규모 연산이 필요한 분야의 핵심 기술이다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
배치 처리(Batch Processing)는 사용자의 개입 없이 미리 정해진 로직에 따라 데이터를 뭉치 단위로 처리하는 방식이다. 데이터의 입력과 결과 출력이 즉각적이지 않으며, 보통 시스템 부하가 적은 심야 시간에 수행되는 경우가 많다.

### 💡 비유
- **실시간 처리 (Stream)**: 손님이 올 때마다 즉석에서 한 그릇씩 요리해 주는 식당이다. 응답은 빠르지만, 손님이 몰리면 요리사가 지친다.
- **배치 처리 (Batch)**: 학교 급식실에서 수천 명분의 밥을 한꺼번에 짓는 것과 같다. 준비 시간이 오래 걸리고 밥이 다 될 때까지 기다려야 하지만, 수천 명을 한 번에 먹이는 데는 이보다 효율적인 방법이 없다.

### 등장 배경: 효율성과 비용의 균형
모든 데이터를 실시간으로 처리하면 좋겠지만, 시스템 자원은 유한하다. 굳이 지금 당장 결과가 안 나와도 되는 통계나 백업 데이터 등을 실시간으로 처리하면 서버 비용이 폭증한다. 배치 처리는 "조금 기다리는 대신, 저렴하고 확실하게 대량의 데이터를 처리하자"는 전략에서 시작되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 배치 처리 파이프라인 아키텍처 (ASCII Diagram)

```ascii
[ Data Sources ]       [ Batch Pipeline ]        [ Storage/Serving ]
┌────────────┐        ┌──────────────────┐       ┌────────────────┐
│ Legacy DB  ├───────>│  1. Extraction   │       │                │
├────────────┤        └────────┬─────────┘       │   Data         │
│ Log Files  ├───────>│  2. Transform    │──────>│   Warehouse    │
├────────────┤        └────────┬─────────┘       │   (BigQuery,   │
│ SaaS APIs  ├───────>│  3. Validation   │       │    Snowflake)  │
└────────────┘        └────────┬─────────┘       └────────────────┘
                      │  4. Loading      │
                      └──────────────────┘
         ▲                     │
         └─────────────────────┘
            Monitoring & Retry Logic
```

### 파이프라인의 4단계 (ETL 프로세스)

#### ① 추출 (Extraction)
다양한 소스(RDBMS, NoSQL, 로그 파일, 외부 API)로부터 원천 데이터를 읽어온다. 이때 원천 시스템에 부하를 주지 않기 위해 CDC(Change Data Capture) 기술을 활용하기도 한다.

#### ② 변환 (Transformation)
수집된 데이터를 목적에 맞게 가공한다. 데이터 정제(Cleaning), 형식 변환, 중복 제거, 여러 소스의 데이터 결합(Join) 등이 이 단계에서 일어난다. 가장 많은 연산 자원이 소모되는 단계다.

#### ③ 검증 (Validation)
가공된 데이터가 비즈니스 규칙에 맞는지, 데이터 품질에 이상이 없는지 확인한다. 만약 심각한 오류가 발견되면 파이프라인을 중단시키고 관리자에게 알람을 보낸다.

#### ④ 적재 (Loading)
최종 데이터를 분석용 DB나 데이터 웨어하우스에 저장한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 배치 처리 vs 스트림 처리 비교

| 비교 항목 | **배치 처리 (Batch)** | 스트림 처리 (Stream) |
|-----------|-----------------------|-----------------------|
| **데이터 범위** | 전체 데이터 셋 (Fixed) | 끝없는 데이터 흐름 (Infinite) |
| **처리 단위** | 큰 묶음 (Hours/Days) | 개별 이벤트 (Real-time) |
| **지연 시간** | 높음 (분~시간 단위) | **매우 낮음 (밀리초~초 단위)** |
| **처리 성능** | **전체 처리량(Throughput) 최적화** | 응답 시간(Latency) 최적화 |
| **기술 스택** | Apache Spark, Spring Batch | Apache Flink, Kafka Streams |

### 배치 처리의 성공 조건 (3대 요소)
1. **Idempotency (멱등성)**: 동일한 데이터를 여러 번 돌려도 항상 같은 결과가 나와야 한다. (재시도 시 데이터 중복 방지).
2. **Scalability (확장성)**: 데이터가 2배로 늘어나면 서버를 2배로 늘려서 처리 시간을 유지할 수 있어야 한다.
3. **Observability (관측 가능성)**: 어느 단계에서 얼마나 시간이 걸리는지, 어디서 실패했는지 한눈에 파악 가능해야 한다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 구축 전략)

**상황**: 이커머스 기업에서 매일 밤 전날의 매출 통계를 집계하여 대시보드에 뿌려줘야 한다. 데이터양은 매일 500GB씩 쌓인다.
**판단 및 배치 구축 전략**:
1. **프레임워크 선정**: 대규모 분산 처리가 필요하므로 **Apache Spark**를 사용한다.
2. **스케줄링**: 작업 간의 의존성 관리(추출 완료 후 변환 시작 등)를 위해 **Apache Airflow**를 도입한다.
3. **증분 로딩 (Incremental Load)**: 매번 500GB 전체를 다시 읽는 대신, 마지막 처리 시점 이후의 데이터만 가져오도록 설계하여 처리 시간을 90% 단축한다.
4. **결과**: 매일 아침 8시 정각에 전 직원이 어제의 성과를 정확한 수치로 확인할 수 있는 데이터 환경 구축.

### 안티패턴 및 고려사항
- **Monolithic Batch**: 하나의 배치 작업에 너무 많은 비즈니스 로직을 집어넣는 것. 실패 시 원인 파악이 어렵고 재시작 비용이 너무 크다. 잘게 쪼개서 단계별로 관리해야 한다.
- **원천 DB 직접 공격**: 배치가 돌아가는 동안 서비스 DB의 CPU를 100% 점유하여 실시간 서비스가 마비되는 경우. 반드시 복제본(Read Replica)이나 파일 형태의 덤프를 활용해야 한다.

## Ⅴ. 미래 전망 및 결론

### 결론: 배치는 죽지 않는다
실시간 기술이 발전하면서 "배치는 구식이다"라는 오해가 있지만, 사실이 아니다. 데이터의 **전체적인 맥락(Context)**을 파악하고 정확한 통계를 내는 데는 배치 처리가 가장 저렴하고 정확한 방법이다.

### 미래 전망
앞으로는 배치와 스트림의 경계가 무너지는 **Kappa Architecture**나 **Unified Batch/Stream Processing**이 가속화될 것이다. 사용자는 로직만 짜면 시스템이 알아서 실시간으로 처리할지, 배치로 모아서 처리할지 결정하는 지능형 파이프라인이 보편화될 전망이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **배치 처리가 뭔가요?**: 매일 저녁 일기를 모아서 한꺼번에 정리하거나, 일주일 동안 모은 빨래를 토요일에 세탁기 돌리는 것과 같아요.
2. **왜 한꺼번에 하나요?**: 양말 한 짝 생길 때마다 세탁기를 돌리면 전기세(비용)도 많이 들고 힘들잖아요. 모아서 하면 훨씬 편하고 효율적이에요.
3. **가장 좋은 점은?**: 우리가 잠든 사이 로봇(컴퓨터)이 모든 복잡한 계산과 정리를 다 끝내놓기 때문에, 아침에 일어나면 깨끗한 결과만 바로 볼 수 있답니다!
