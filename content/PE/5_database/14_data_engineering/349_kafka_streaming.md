# 349. 실시간 데이터 스트리밍 (Kafka) - 데이터의 거대한 실시간 혈맥

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산된 여러 시스템 간에 발생하는 대량의 데이터를 **발행-구독(Publish-Subscribe) 모델** 기반으로 실시간으로 주고받으며 저장하는 고성능 분산 메시지 플랫폼이다.
> 2. **혁신**: 메모리가 아닌 **디스크에 데이터를 순차적으로 기록(Sequential Log)**하여 영속성을 보장하면서도, 수백만 건의 이벤트를 밀리초 단위로 처리하는 압도적 처리량(Throughput)을 달성한다.
> 3. **가치**: 데이터 소스와 목적지를 분리(Decoupling)하여 시스템 복잡도를 낮추고, 실시간 로그 수집, 사기 탐지(FDS), 마이크로서비스 간 통신 등 현대 데이터 중심 아키텍처의 중추 역할을 수행한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
Apache Kafka는 링크드인(LinkedIn)에서 내부 데이터 파이프라인 문제를 해결하기 위해 처음 개발된 오픈소스 분산 이벤트 스트리밍 플랫폼이다. 데이터가 발생하는 'Producer'와 데이터를 사용하는 'Consumer' 사이에서 데이터를 안전하게 보관하고 전달하는 'Buffer'이자 'Hub'이다.

### 💡 비유
- **전통적 방식 (Direct Call)**: 친구 수십 명에게 일일이 전화를 걸어 내 소식을 전하는 것이다. 친구가 전화를 안 받으면 소식을 전할 수 없고, 친구가 늘어날수록 나는 전화를 돌리느라 아무 일도 못 한다.
- **카프카 방식 (게시판/신문)**: 나는 중앙 게시판(Kafka)에 글을 한 번만 올린다. 내 소식이 궁금한 친구들은 언제든 게시판에 와서 읽어간다(Subscribe). 내가 글을 올릴 때 친구들이 자고 있어도 상관없다. 게시판은 글을 지우지 않고 보관해주기 때문이다.

### 등장 배경: 'Spaghetti Architecture'의 종말
회사가 커지면 수많은 서비스와 데이터베이스가 서로 엉망으로 얽히게 된다(N:N 관계). 이를 해결하기 위해 모든 데이터를 한곳으로 모으고 필요한 곳에서 가져가게 만드는 '중앙 집중형 데이터 버스'가 필요해졌고, 카프카가 그 표준이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 카프카 기본 구조도 (ASCII Diagram)

```ascii
[ Producers ]          [ Kafka Cluster (Brokers) ]          [ Consumers ]
┌───────────┐          ┌─────────────────────────┐          ┌───────────┐
│ App Logs  ├──┐       │   Topic: "User_Click"   │       ┌──┤ Analytics │
└───────────┘  │       │ ┌─────────┐ ┌─────────┐ │       │  └───────────┘
               ├──────>│ │ Part. 0 │ │ Part. 1 │ ├───────┤
┌───────────┐  │       │ └─────────┘ └─────────┘ │       │  ┌───────────┐
│ DB Events ├──┘       └────────────┬────────────┘       └──┤ Dashboard │
└───────────┘                       │                       └───────────┘
                                    ▼
                          [ ZooKeeper / KRaft ] (Consensus)

* 핵심 원리: 데이터를 'Topic'으로 나누고, 이를 다시 'Partition'으로 쪼개어 여러 서버에 분산 저장/처리.
```

### 핵심 기술 요소

#### ① 토픽(Topic)과 파티션(Partition)
데이터는 '토픽'이라는 카테고리로 분류된다. 각 토픽은 병렬 처리를 위해 여러 개의 '파티션'으로 나뉜다. 파티션은 카프카 성능의 핵심으로, 파티션이 많을수록 더 많은 컨슈머가 동시에 데이터를 읽을 수 있다.

#### ② 프로듀서(Producer) & 컨슈머(Consumer)
- **Producer**: 데이터를 생성하여 카프카 토픽으로 보낸다.
- **Consumer**: 토픽의 데이터를 읽어간다. 컨슈머는 자신이 어디까지 읽었는지 표시(Offset)해두기 때문에, 서버가 꺼졌다 켜져도 읽던 부분부터 다시 시작할 수 있다.

#### ③ 브로커(Broker) & 복제(Replication)
카프카 서버 하나를 브로커라고 한다. 데이터를 여러 브로커에 복제해두기 때문에, 서버 한두 대가 고장 나도 데이터 유실 없이 안정적으로 운영된다(High Availability).

#### ④ Zero-copy
디스크에서 데이터를 읽어 네트워크로 보낼 때 CPU의 개입을 최소화하는 기술이다. 이 덕분에 카프카는 디스크 기반임에도 불구하고 메모리 기반 시스템에 필적하는 속도를 낸다.

## Ⅲ. 융합 비교 및 다각도 분석

### Kafka vs RabbitMQ (Traditional MQ) 비교

| 비교 항목 | **Apache Kafka** | RabbitMQ |
|-----------|------------------|-----------|
| **처리 방식** | 이벤트 스트리밍 (Pull) | 메시지 큐 (Push) |
| **데이터 보관** | **장기간 보관 가능 (Log)** | 소비 즉시 삭제가 원칙 |
| **처리량** | **압도적 (High Throughput)** | 보통 |
| **순서 보장** | 파티션 내에서 엄격히 보장 | 보장하기 까다로움 |
| **주요 용도** | 빅데이터 수집, 실시간 분석 | 비동기 작업 처리, 앱 간 통신 |

### 카프카의 3대 핵심 가치
1. **Scalability**: 서버만 추가하면 무한대로 성능이 늘어난다.
2. **Durability**: 데이터를 디스크에 저장하므로 시스템 장애에도 안전하다.
3. **Decoupling**: 데이터 생산자와 소비자가 서로를 몰라도 통신이 가능하다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 구축 전략)

**상황**: 초당 10만 건의 결제 이벤트가 발생하는 커머스 시스템에서 실시간으로 사기 탐지(FDS)를 하고, 동시에 데이터 웨어하우스(DW)에 로그를 쌓아야 한다.
**판단 및 카프카 도입 전략**:
1. **토픽 설계**: `payments`라는 토픽을 만들고, 부하 분산을 위해 파티션을 30개 이상으로 넉넉히 잡는다.
2. **Consumer Group 활용**: FDS 팀과 DW 팀에 각각 별도의 'Consumer Group'을 부여한다. 이렇게 하면 두 팀이 서로 간섭 없이 동일한 결제 데이터를 각자의 속도로 읽어갈 수 있다.
3. **멱등성 프로듀서 (Idempotent Producer)**: 결제 데이터는 중복되면 안 되므로, 프로듀서 설정에서 `enable.idempotence=true`를 활성화하여 정확히 한 번(Exactly-once)만 전달되게 보장한다.
4. **결과**: 결제와 동시에 0.1초 만에 이상 징후를 탐지하여 승인을 차단하고, 모든 결제 이력이 누락 없이 DW에 실시간으로 적재됨.

### 안티패턴 및 고려사항
- **파티션 수 줄이기 불가**: 카프카는 파티션을 늘릴 수는 있지만 줄이는 것은 불가능하다. 처음부터 너무 많이 잡기보다 필요에 따라 늘려가는 것이 좋다.
- **큰 메시지 전송**: 카프카는 작은 이벤트 수백만 개를 처리하는 데 특화되어 있다. 고화질 이미지나 영상 같은 큰 파일을 통째로 올리면 전체 성능이 급격히 떨어진다.

## Ⅴ. 미래 전망 및 결론

### 결론: 데이터 중심 아키텍처의 심장
카프카는 이제 단순한 메시지 전달자를 넘어, 기업의 모든 데이터가 흐르는 **'중앙 신경망'**이 되었다. 카프카가 멈추는 것은 기업의 혈류가 멈추는 것과 같다.

### 미래 전망
앞으로는 카프카 자체에서 데이터를 실시간으로 가공하고 분석하는 **'카프카 스트림즈(Kafka Streams)'**와 **'ksqlDB'**의 비중이 커질 것이다. 또한, 클러스터 관리를 위해 반드시 필요했던 주키퍼(ZooKeeper)를 제거한 **'KRaft'** 모드가 대중화되어 운영 편의성이 극대화될 전망이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **카프카가 뭔가요?**: 세상에서 가장 크고 빠른 '디지털 우체국'이에요. 수천만 통의 편지가 동시에 들어와도 절대 잃어버리지 않고 안전하게 보관해 줘요.
2. **왜 쓰나요?**: 편지를 보내는 사람(앱)은 우체국에 맡기기만 하면 되고, 편지를 받는 사람(서버)은 자기가 한가할 때 언제든 우체국에 가서 가져가면 되니까 서로 너무 편해요.
3. **뭐가 제일 신기해요?**: 아주 옛날 편지도 우체국 창고에 잘 보관되어 있어서, 원한다면 작년에 받은 편지부터 다시 읽어보는 것도 가능하답니다!
