# 364. 데이터 리니지 (Data Lineage) - 데이터의 족보와 이동 경로를 추적하는 투명성 지도

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터가 최초에 어디서 생성(Origin)되어 어떤 파이프라인(ETL)과 변환 과정을 거쳐 최종 목적지(대시보드, AI 모델 등)에 도착했는지, 그 **전체 이동 경로와 가공 이력을 시각적으로 추적하고 기록하는 메타데이터 관리 기술**이다.
> 2. **혁신**: 대규모 데이터 레이크나 마이크로서비스 환경에서 특정 데이터에 오류가 발생했을 때, "이 잘못된 값이 도대체 어디서부터 시작되었는가?"를 역추적(Root Cause Analysis)하여 **영향도를 순식간에 파악**할 수 있게 한다.
> 3. **가치**: 기업의 규제 준수(Compliance, 예: 금융 감사, 개인정보 추적)를 완벽히 입증하는 법적 증거가 되며, 'Garbage In, Garbage Out'을 막아 데이터 자체의 **신뢰성(Trust)과 품질(Data Quality)**을 보장하는 데이터 거버넌스의 핵심 척추다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
리니지(Lineage)의 사전적 의미는 혈통, 족보다. 데이터 리니지는 특정 테이블이나 컬럼의 값이 나오기까지의 조상(소스 데이터)이 누구인지, 그리고 내 자손(이 데이터를 참조해서 만들어진 다른 뷰나 차트)은 누구인지를 거미줄 같은 그래프 형태로 보여주는 체계다. 

### 💡 비유
- **리니지가 없을 때 (원산지 불명 요리)**: 식당에서 스테이크를 먹다가 상한 고기를 발견했다. 요리사에게 따졌지만, 요리사도 고기를 시장 상인에게 샀고, 상인은 도매상에게, 도매상은 어느 농장에서 가져왔는지 몰라서 범인을 찾을 수 없다. 다른 손님들의 식중독(에러 전파)을 막을 방법이 없다.
- **데이터 리니지 도입 (이력 추적제)**: 스테이크 접시에 QR코드가 있다. 찍어보면 "이 고기는 1일 전 B 도매상에서 칼질(변환)되었고, 3일 전 A 농장(Origin)에서 출하되었다"는 이력이 쫙 뜬다. 즉시 A 농장을 폐쇄하고, A 농장의 고기가 들어간 다른 모든 식당(영향받는 시스템)에 판매 중지 경고를 내려 피해를 완벽히 막는다.

### 등장 배경: 파이프라인의 복잡성과 블랙박스화
과거에는 DBA 혼자서 데이터를 옮겼기 때문에 경로가 머릿속에 다 있었다. 하지만 수백 명의 데이터 엔지니어가 수천 개의 Airflow 스케줄러로 복잡한 변환(Transformation) 스크립트를 짜는 현대의 빅데이터 환경에서는, 경영진 대시보드의 '매출액' 수치가 틀렸을 때 그 원인을 찾기 위해 코드를 까보고 수일 밤을 새워야 했다. 이 '블랙박스'를 투명하게 열어젖히기 위해 리니지 자동화 도구들이 발전했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 데이터 리니지 수집 및 시각화 프로세스 (ASCII Diagram)

```ascii
[ Data Pipeline Execution ]
 (Kafka) ───> (Spark ETL) ───> (Snowflake DB) ───> (Tableau Dashboard)
    │              │                  │                    │
    └──────┬───────┴────────┬─────────┴─────────┬──────────┘
           │                │                   │
           ▼                ▼                   ▼
    (Logs & API)    (SQL Query Parse)  (Metadata Extraction)
           │                │                   │
┌──────────▼────────────────▼───────────────────▼───────────┐
│  [ Data Lineage Engine (e.g., Apache Atlas, Amundsen) ]   │
│                                                           │
│ 1. Metadata Ingestion: 모든 도구의 로그를 빨아들임           │
│ 2. Query Parsing: "INSERT INTO A SELECT * FROM B"를 분석  │
│ 3. Graph Construction: B(부모) -> A(자식) 연결 관계 생성    │
└───────────────────────────┬───────────────────────────────┘
                            ▼
           [ Lineage Graph UI (시각화 맵) ]
   (DB_1.table_A) ---> (Job_Script_X) ---> (DB_2.table_B)
```

### 리니지 추출의 3대 핵심 방법

#### ① SQL 파싱 (SQL Parsing)
시스템이 실행하는 SQL 문장(예: `CREATE VIEW`, `INSERT SELECT`)을 파서(Parser)가 텍스트 수준에서 분석하여 어떤 테이블이 조합되어 결과 테이블이 만들어졌는지 추론한다. 가장 널리 쓰이는 방법이다.

#### ② 런타임/엔진 수집 (Runtime Collection)
Apache Spark나 하둡 같은 데이터 처리 엔진이 작업을 수행할 때, 프레임워크 자체가 직접 API를 통해 "내가 방금 A 파일 읽어서 B 파일 만들었다"라고 리니지 서버에 리포팅하는 방식이다. 매우 정확하다.

#### ③ 로그 분석 및 수동 매핑 (Manual/Log Analysis)
소스 코드가 없는 외부 API 연동이나 블랙박스 솔루션의 경우, 네트워크 로그를 분석하거나 관리자가 직접 카탈로그 툴에 관계를 손으로 그려 넣는 방식이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 리니지의 2가지 방향성 (추적 vs 영향도)

| 분석 방향 | 영어 명칭 | 질문(Question) | 활용 목적 |
|:---:|:---|:---|:---|
| **역방향 추적 (Upstream)** | Root Cause Analysis | "이 대시보드의 이상한 수치는 **어디서부터 시작된 에러**인가?" | **장애 원인 분석**, 데이터 품질 검증 및 디버깅 |
| **순방향 추적 (Downstream)**| Impact Analysis | "이 컬럼의 데이터 타입을 String에서 Int로 **바꾸면 어디어디가 고장 나는가?**" | 스키마 변경 시 **영향도 사전 검증**, 서비스 중단 방지 |

### 비즈니스 리니지 vs 테크니컬 리니지
- **기술적 리니지**: 컬럼 A가 컬럼 B로 복사되는 물리적인 쿼리 흐름을 보여준다. (엔지니어용).
- **비즈니스 리니지**: "고객의 생년월일"이라는 개념이 "고객 등급 평가"라는 업무로 어떻게 흘러가 영향을 미치는지 논리적으로 보여준다. (기획자/경영진용).

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 규제 대응 전략)

**상황**: 대형 은행이 금융감독원으로부터 "AI 대출 심사 모델이 편향되지 않았음을 증명하기 위해, AI가 학습한 모든 고객 데이터의 출처와 가공 과정을 투명하게 제출하라"는 감사를 받게 되었다.
**판단 및 데이터 리니지 구축 전략**:
1. **메타데이터 플랫폼 도입**: 오픈소스인 **Apache Atlas**나 상용 툴인 **Collibra**를 도입하여 전사 데이터 카탈로그 및 리니지 시스템을 띄운다.
2. **자동화 파이프라인 연동**: ETL 스케줄러인 Airflow에 리니지 플러그인을 달아, 데이터 변환 작업이 돌 때마다 자동으로 이력을 리니지 서버로 쏘게(Push) 만든다.
3. **개인정보 매핑**: "대출 AI 모델"의 노드(자식)를 클릭했을 때, 역방향 그래프를 따라가면 최종적으로 "고객정보_DB"의 "소득_컬럼(부모)"이 나오도록 완벽한 족보를 시각화한다. 도중에 개인정보 비식별화(마스킹) 처리가 잘 되었는지 변환 스크립트(Job) 이력을 첨부한다.
4. **결과**: 투명한 이력 추적 지도를 제출하여 금융 규제(Compliance) 심사를 무사히 통과하고, AI 모델의 신뢰성(Trustworthy AI)을 공식적으로 확보함.

### 안티패턴 및 고려사항
- **사후 수동 기록의 맹신**: 개발자에게 데이터가 만들어질 때마다 위키나 엑셀에 이력을 손으로 적어놓으라고 지시하는 것. 시스템이 복잡해지면 하루 만에 문서가 실제 아키텍처와 어긋나게 되고, 틀린 지도는 아예 지도가 없는 것보다 못한 대참사를 부른다. 리니지는 반드시 쿼리와 파이프라인 엔진에서 **'자동으로(Automated)'** 추출되어야 한다.

## Ⅴ. 미래 전망 및 결론

### 결론: 데이터 자산화의 필수 영수증
물건을 샀을 때 영수증과 품질 보증서가 없으면 그 물건의 가치를 인정받을 수 없다. 데이터 리니지는 데이터가 언제 어디서 어떻게 만들어졌는지 증명하는 '품질 보증서'다. 이것 없이는 데이터 레이크는 그저 쓰레기 더미일 뿐이다.

### 미래 전망
앞으로는 **액티브 리니지(Active Lineage)** 시대가 열린다. 단순히 지도를 '보여주는' 것을 넘어, AI가 리니지 지도를 실시간으로 감시하다가 상류(Upstream) 파이프라인에서 데이터 에러를 감지하면, 하류(Downstream)의 대시보드 업데이트를 스스로 멈춰버리고 담당자에게 알람을 보내는 '스마트 데이터 방어망'으로 진화할 것이다. 이는 데이터 메시(Data Mesh)와 데이터 패브릭(Data Fabric)을 지탱하는 가장 근본적인 신경망이 될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **데이터 리니지가 뭔가요?**: 내가 지금 마시고 있는 오렌지 주스가 '어느 나라 농장의 어떤 나무'에서 열렸고, '어떤 주스 공장'을 거쳐 내 컵에 오게 되었는지 그 긴 여행의 과정을 한눈에 보여주는 지도예요.
2. **왜 지도를 그려야 하나요?**: 주스를 마셨는데 배가 아플 때, 지도가 있으면 "아! A 농장 오렌지가 상했구나!" 하고 금방 원인을 찾아서 다른 사람들이 그 주스를 못 마시게 바로 막을 수 있거든요.
3. **가장 좋은 점은?**: 컴퓨터 세상에 있는 엄청나게 많은 숫자들이 믿을 수 있는 진짜 정보인지, 아니면 불량품인지 족보를 보고 100% 안심하고 쓸 수 있게 해준답니다!
