# 380. 시퀀스 데이터베이스 객체 특징 (Auto Increment vs Sequence) - 유일성을 보장하는 번호 생성기

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 내에서 중복되지 않는 고유한 숫자를 자동으로 생성해 주는 객체 또는 속성으로, 주로 테이블의 **기본키(Primary Key) 생성**을 위해 사용된다.
> 2. **차이**: 테이블의 컬럼에 종속되어 자동으로 증가하는 **Auto Increment** 방식과, 테이블과 독립된 객체로서 여러 테이블이 공유할 수 있는 **Sequence** 방식으로 구분된다.
> 3. **가치**: 애플리케이션 레벨에서 번호를 생성할 때 발생하는 동시성(Concurrency) 문제와 번호 중복을 하드웨어 및 DBMS 엔진 레벨에서 원천적으로 해결하여 데이터 무결성을 보장한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
데이터베이스에 새로운 행(Row)을 추가할 때마다 1번, 2번, 3번... 순서대로 번호를 매겨야 할 때가 많다. 이를 사람이 직접 관리하면 동시에 두 명이 입력할 때 같은 번호가 생길 수 있다. 시퀀스(Sequence)와 오토 인크리먼트(Auto Increment)는 DB가 직접 "다음 번호는 이거야!"라고 관리해 주는 기술이다.

### 💡 비유
- **Auto Increment (은행 번호표 기계 - 창구 고정)**: 각 은행 창구마다 번호표 기계가 하나씩 붙어 있는 것이다. 1번 창구 기계는 무조건 1번 창구용 번호만 뱉는다. 창구가 없어지면 기계도 없어진다.
- **Sequence (중앙 번호표 발행기)**: 은행 입구에 커다란 번호표 기계(독립 객체)가 하나 있다. 이 기계에서 뽑은 번호로 1번 창구도 가고, 2번 창구도 갈 수 있다. 기계는 창구와 상관없이 혼자서 번호를 계속 올린다.

### 등장 배경: 'Primary Key'의 자동화
과거에는 `MAX(ID) + 1` 쿼리를 써서 번호를 땄다. 하지만 사용자가 몰리면 똑같은 번호가 두 개 생기는 'PK 중복 에러'가 발생했다. 이를 막기 위해 DB 엔진 내부에 초고속으로 번호만 생성하고 메모리에 캐싱(Caching)해두는 전용 객체가 필요해졌다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 시퀀스 및 오토 인크리먼트 구조 (ASCII Diagram)

```ascii
[ 1. Auto Increment (MySQL, MSSQL 등) ]
  Table: Users (id INT AUTO_INCREMENT PRIMARY KEY, name VARCHAR)
  - Insert (name='Alice')  ──▶ DB가 컬럼 속성 확인 ──▶ id=1 할당
  - Insert (name='Bob')    ──▶ DB가 컬럼 속성 확인 ──▶ id=2 할당
  * 특징: 특정 테이블/컬럼에 묶여 있음.

------------------------------------------------------------------

[ 2. Sequence Object (Oracle, PostgreSQL 등) ]
  CREATE SEQUENCE my_seq START WITH 1 INCREMENT BY 1;
  - Insert into Table_A (id) values (my_seq.NEXTVAL); ──▶ id=1
  - Insert into Table_B (id) values (my_seq.NEXTVAL); ──▶ id=2
  * 특징: 테이블과 독립적. my_seq 하나로 여러 테이블이 공유 가능.
```

### 3대 핵심 속성 (Parameters)

#### ① INCREMENT BY (증가치)
번호가 한 번에 얼마나 커질지 정한다. 보통 1씩 증가하지만, 보안이나 분산 처리를 위해 2, 5, 10 단위로 늘리기도 한다.

#### ② CACHE (캐시)
매번 디스크에서 번호를 가져오면 느리다. 메모리에 "1번부터 20번까지" 미리 뽑아두고 손님에게 바로 나눠주는 방식이다. 성능은 비약적으로 빨라지지만, DB가 비정상 종료되면 캐시된 번호가 사라져 '번호 건너뜀(Gap)' 현상이 발생할 수 있다.

#### ③ CYCLE (순환)
최대값에 도달했을 때 다시 1번부터 시작할지 여부다. 기본키 용도라면 절대 사용하지 않지만, 대기 순번처럼 로테이션이 필요한 경우 사용한다.

## Ⅲ. 융합 비교 및 다각도 분석

### Auto Increment vs Sequence 상세 비교 (Table)

| 비교 항목 | **Auto Increment (Identity)** | **Sequence Object** |
|:---:|:---|:---|
| **소속성** | 특정 테이블 컬럼에 종속 | **독립적인 데이터베이스 객체** |
| **공유 가능성**| 불가능 (테이블당 1개) | **가능 (여러 테이블에서 호출)** |
| **번호 획득** | Insert 시점에 자동 할당 | **`NEXTVAL` 호출로 명시적 획득** |
| **유연성** | 낮음 (값 수정이 까다로움) | **높음 (다양한 옵션 제공)** |
| **대표 DBMS** | MySQL, MariaDB, SQL Server | **Oracle, PostgreSQL, DB2** |
| **비유** | **내 전용 개인 비서** | **마을 공동 안내소** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 대규모 분산 환경 PK 설계 전략)

**상황**: 글로벌 분산 환경에서 여러 개의 DB 노드가 동시에 번호를 생성해야 한다. 일반 시퀀스를 쓰니 중앙 DB의 병목이 심각하다.
**판단 및 대안 전략**:
1. **중앙 집중형 지양**: 네트워크 지연이 있는 환경에서 단일 시퀀스 객체는 SPOF가 된다.
2. **범위 분할 (Range Allocation)**: A 노드는 1~1억 번, B 노드는 1억~2억 번을 쓰도록 시퀀스 시작값을 물리적으로 격리한다.
3. **UUID/Snowflake 고려**: 만약 노드 간 합의가 불가능할 정도로 규모가 크다면, 시퀀스 대신 **[UUID](../../5_database/1_logic/uuid_concept.md)**나 트위터의 **[Snowflake](../../5_database/1_logic/snowflake_id.md)**(시간+머신ID+시퀀스 조합) 알고리즘 사용을 강력 권고한다.
4. **결과**: 중앙 병목 없는 무한 확장 가능한 유일 ID 체계 구축.

### 안티패턴 및 고려사항
- **빈틈없는 번호(No Gap) 집착**: "왜 3번 다음에 5번이 나오죠? 4번 채워주세요"라는 현업의 요구에 응해 번호 재정렬 로직을 짜는 것은 전형적인 안티패턴이다. 시퀀스는 **'순서'와 '유일성'**을 보장하는 것이지 '연속성'을 보장하는 것이 아니다. 연속성을 강제하면 성능이 극도로 저하된다.

## Ⅴ. 미래 전망 및 결론

### 결론: 데이터 무결성의 첫 단추
시퀀스 객체는 단순한 숫자 생성기가 아니라, 데이터라는 집을 지을 때 가장 먼저 박는 '주춧돌(PK)'과 같다. 하드웨어 리소스를 최소한으로 쓰면서 동시성을 해결하는 DBMS 엔진의 정교함을 보여준다.

### 미래 전망
앞으로는 **'분산형 서버리스 시퀀스'**가 대세가 될 것이다. DB 엔진에 접속하지 않고도 엣지(Edge) 노드들이 수학적으로 절대 겹치지 않는 번호를 스스로 생성하는 방식이다. 또한, **[블록체인 기반의 글로벌 시퀀스]** 연구를 통해, 전 세계 어떤 DB에 데이터를 넣어도 전 지구적으로 유일한 순번을 보장받는 '글로벌 유일 ID 체계'가 인프라 레벨에서 통합될 전망이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **시퀀스가 뭔가요?**: 놀이공원에서 줄을 설 때, "너는 1번, 너는 2번..." 하고 차례대로 번호표를 나눠주는 '친절한 기계'예요.
2. **왜 필요해요?**: 기계가 없으면 사람들이 서로 "내가 먼저 왔어!"라고 싸울 수 있는데, 기계가 번호를 딱 정해주면 싸울 일이 없기 때문이에요.
3. **가장 좋은 점은?**: 기계가 엄청나게 빨라서 수천 명이 동시에 와도 절대 헷갈리지 않고 정확하게 번호를 매겨준답니다!
