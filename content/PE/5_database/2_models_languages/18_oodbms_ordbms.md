# 18. 객체지향 데이터 모델 (OODBMS) / 객체 관계형 데이터 모델 (ORDBMS)

## Ⅰ. 모델의 개요

### 1. 객체지향 데이터베이스 (OODBMS: Object-Oriented DBMS)의 정의
* 객체지향 프로그래밍(OOP)의 개념(클래스, 상속, 다형성, 캡슐화 등)을 데이터베이스 모델에 그대로 적용하여, 데이터와 그 데이터를 조작하는 메서드(Method)를 하나의 객체(Object)로 저장하고 관리하는 데이터베이스 시스템입니다.
* 복잡한 데이터 타입(멀티미디어, CAD/CAM 데이터, 공간 데이터 등)을 효과적으로 처리하기 위해 등장했습니다.

### 2. 객체 관계형 데이터베이스 (ORDBMS: Object-Relational DBMS)의 정의
* 기존의 관계형 데이터베이스(RDBMS)의 견고하고 입증된 기술적 기반 위에 객체지향 데이터 모델의 장점(복잡한 데이터 구조, 사용자 정의 타입 등)을 수용하여 확장한 하이브리드(Hybrid) 형태의 데이터베이스 시스템입니다.
* 상용 데이터베이스 시장의 주류(Oracle, PostgreSQL 등)가 이 모델을 채택하고 있습니다.

---

## Ⅱ. 아키텍처 및 핵심 원리

### 1. 객체지향 모델의 핵심 구성 요소
* **객체(Object)와 식별자(OID):** 데이터베이스 내의 각 객체는 고유한 식별자(Object Identifier)를 가지며, 이는 시스템에 의해 자동으로 할당되고 불변합니다.
* **클래스(Class)와 인스턴스(Instance):** 객체의 구조(속성)와 행위(메서드)를 정의하는 틀이 클래스이며, 이를 실체화한 것이 인스턴스입니다.
* **캡슐화(Encapsulation)와 상속성(Inheritance):** 데이터와 행위를 묶어 외부 접근을 통제하며, 상위 클래스의 특성을 하위 클래스가 물려받아 재사용성과 확장성을 높입니다.

### 2. ORDBMS의 확장된 구조적 특징
* **사용자 정의 타입 (User-Defined Types, UDT):** 기본 데이터 타입 외에 사용자가 복합적인 구조를 가진 새로운 데이터 타입을 정의할 수 있습니다.
* **참조(Reference) 타입과 컬렉션(Collection):** 포인터와 유사한 참조를 통해 다른 테이블의 로우(객체)를 직접 가리킬 수 있으며, 배열(Array), 중첩 테이블(Nested Table) 등의 컬렉션 타입을 지원합니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. RDBMS, OODBMS, ORDBMS 비교

| 비교 항목 | RDBMS | OODBMS | ORDBMS |
| :--- | :--- | :--- | :--- |
| **기본 구조** | 2차원 테이블 (행/열) | 객체(Object), 클래스 | 테이블 + 확장된 객체 기능 |
| **데이터 무결성** | 매우 강력함 | 상대적으로 약함 | 강력함 (RDBMS 기반) |
| **복잡한 데이터 처리** | 어려움 (BLOB/CLOB 등 한계) | 매우 우수함 | 우수함 (사용자 정의 타입) |
| **질의 언어** | 표준 SQL | OQL (Object Query Language) | 확장 SQL (SQL3 등) |
| **시장 점유율** | 높음 (전통적 시스템) | 매우 낮음 (특수 목적 틈새) | 가장 높음 (현대 주류 DBMS) |
| **성능 오버헤드** | 구조가 단순하여 빠름 | 복잡한 쿼리 처리 최적화 부족 | 객체 변환 오버헤드 존재 |

### 2. OODBMS의 시장 실패와 ORDBMS의 성공 요인
OODBMS는 패러다임의 혁신성에도 불구하고, 표준 질의 언어의 부재, 기존 관계형 시스템과의 호환성 부족, 트랜잭션 처리 성능의 한계로 인해 시장에서 주류가 되지 못했습니다. 반면, ORDBMS는 기존 RDBMS의 자산(SQL, 트랜잭션, 옵티마이저 등)을 보호하면서 객체 지향의 유연성을 단계적으로 수용함으로써 기업 환경의 복잡한 요구사항을 완벽히 흡수하였습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 실무 적용 시 고려사항
* **타입 설계의 복잡성 통제:** ORDBMS에서 과도한 사용자 정의 타입 및 중첩 테이블 사용은 질의 최적화기의 경로 탐색을 어렵게 만들고 인덱스 효율을 떨어뜨릴 수 있습니다. 복잡한 구조 설계 시에는 반드시 성능 검증(PoC)이 동반되어야 합니다.
* **임피던스 불일치(Impedance Mismatch) 해결:** 애플리케이션의 객체 지향 모델과 데이터베이스의 관계형 모델 간의 불일치를 해결하기 위해, ORDBMS의 네이티브 객체 기능을 사용할 것인지, 아니면 ORM(Object-Relational Mapping) 프레임워크(예: Hibernate, JPA)를 사용하여 RDBMS로 매핑할 것인지에 대한 아키텍처적 결단이 필요합니다. 실무에서는 보통 강력한 커뮤니티와 도구가 지원되는 ORM + 순수 RDBMS(또는 ORDBMS의 R-엔진) 구조를 더 선호합니다.

### 2. 기술사적 통찰
현대의 상용 DBMS는 사실상 대부분 ORDBMS입니다. 그러나 현장에서는 여전히 전통적인 RDBMS 방식으로 데이터베이스를 설계 및 운영하고 있습니다. 이는 데이터베이스 레벨에서 비즈니스 로직(메서드)을 캡슐화하는 것이, 애플리케이션 티어의 스케일 아웃(Scale-out)을 방해하고 데이터베이스에 종속성을 심화시키기 때문입니다. 따라서 기술사는 ORDBMS가 제공하는 다중 매체 저장 능력, 파티셔닝, 분석 함수 등은 적극 활용하되, 비즈니스 로직의 DB 내재화(캡슐화)는 시스템의 아키텍처 원칙에 따라 신중하게 제한해야 합니다.

## Ⅴ. 기대효과 및 결론

### 1. 도입 기대효과
* **유연한 데이터 모델링:** 공간 데이터(GIS), 시계열 데이터, 멀티미디어 등 비정형 및 반정형 데이터를 효과적으로 수용하고 쿼리할 수 있는 확장성을 제공합니다.
* **생산성 향상:** 복잡한 데이터 구조를 데이터베이스 자체에서 지원함으로써 애플리케이션의 데이터 가공 로직을 간소화할 수 있습니다.

### 2. 결론
객체지향 데이터 모델과 객체 관계형 데이터 모델의 발전 과정은, 혁신적 패러다임(OODBMS)이 점진적이고 실용적인 패러다임(ORDBMS)으로 어떻게 융화되는지를 보여주는 대표적인 사례입니다. 
데이터의 형태가 다변화되는 빅데이터 시대에서, ORDBMS는 NoSQL 데이터베이스와의 경쟁 및 융합을 거치며 JSON 데이터 타입의 네이티브 지원 등 지속적으로 진화하고 있습니다. 향후에도 ORDBMS는 강력한 트랜잭션 처리 능력과 데이터 타입의 확장성을 무기로 엔터프라이즈 시스템의 핵심 저장소 역할을 견고히 유지할 것입니다.
