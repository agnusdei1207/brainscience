# 20. DDL (Data Definition Language) - 데이터 정의 언어

## Ⅰ. DDL의 개요

### 1. DDL(Data Definition Language)의 정의
* 데이터베이스 구조와 데이터가 저장될 논리적 틀(Schema)을 정의, 변경, 삭제하는 언어입니다.
* 관계형 데이터베이스에서 테이블(Table), 뷰(View), 인덱스(Index), 스키마(Schema), 시퀀스(Sequence) 등 데이터베이스 객체(Database Objects)를 생성하고 관리하는 역할을 합니다.

### 2. 주요 명령어 및 특성
* **주요 명령어:** `CREATE`(생성), `ALTER`(변경), `DROP`(삭제), `TRUNCATE`(내용 전제 삭제), `RENAME`(이름 변경)
* **Auto-Commit 특성:** 대부분의 DBMS에서 DDL 명령어는 실행 즉시 트랜잭션이 커밋(Auto-Commit)됩니다. 즉, `ROLLBACK`을 통해 이전 상태로 복구할 수 없는 비가역적(Irreversible) 특성을 가지므로 시스템 운영 중 각별한 주의가 요구됩니다. (일부 DBMS 제외)

## Ⅱ. 아키텍처 및 핵심 원리

### 1. 메타데이터(Metadata)와 데이터 딕셔너리(Data Dictionary)
* DDL의 핵심 원리는 데이터가 아닌 '데이터에 대한 데이터(메타데이터)'를 다룬다는 점입니다.
* DDL 명령어가 성공적으로 실행되면, DBMS 내부의 시스템 카탈로그(System Catalog) 또는 데이터 딕셔너리(Data Dictionary)에 해당 객체의 구조, 제약조건, 권한 등의 메타데이터가 영구적으로 기록되거나 갱신됩니다.

### 2. DDL 명령어 동작 메커니즘 (ASCII 다이어그램)

```text
[DDL Execution and Dictionary Management Flow]

                     [User / DBA]
                          |
                          | (e.g., CREATE TABLE Emp ...)
                          v
+-------------------------------------------------------------+
|                     DBMS DDL Processor                      |
|                                                             |
|  1. Syntax Check (구문 검증)                                |
|  2. Semantic Check (권한 확인, 객체 이름 중복 검사 등)      |
|  3. Lock Acquisition (DDL Lock - 동시성 제어)               |
+-------------------------+-----------------------------------+
                          |
                          | 4. Update Catalog
                          v
+-------------------------------------------------------------+
|                  System Catalog / Data Dictionary           |
|  +----------------+   +----------------+   +-------------+  |
|  | Tables Info    |   | Columns Info   |   | Constraints |  |
|  | (Table Name..) |   | (Data Types..) |   | (PK, FK..)  |  |
|  +----------------+   +----------------+   +-------------+  |
+-------------------------+-----------------------------------+
                          |
                          | 5. Allocate Storage (if needed)
                          v
+-------------------------------------------------------------+
|                  Physical Storage System                    |
|  [Tablespace Allocation] -> [Extent Allocation]             |
+-------------------------------------------------------------+
```

## Ⅲ. 융합 비교 및 다각도 분석

### 1. DELETE, DROP, TRUNCATE의 명확한 비교 분석
데이터나 테이블을 지우는 연산들은 실무와 이론에서 매우 중요하게 다루어지는 비교 주제입니다.

| 비교 항목 | `DELETE` (DML) | `TRUNCATE` (DDL) | `DROP` (DDL) |
| :--- | :--- | :--- | :--- |
| **목적** | 테이블 내의 특정 로우(데이터) 삭제 | 테이블 내의 모든 데이터 삭제 | 테이블 자체(구조 포함) 삭제 |
| **명령어 종류** | DML | DDL | DDL |
| **트랜잭션 여부** | Rollback 가능 (Undo 영역 사용) | Rollback 불가 (Auto-Commit) | Rollback 불가 (Auto-Commit) |
| **실행 속도** | 레코드 단위로 로그를 남겨 매우 느림 | 테이블 단위 공간 반환, 매우 빠름 | 구조 삭제, 매우 빠름 |
| **데이터 공간(HWM)**| 데이터는 지워지나 공간(High Water Mark)은 반환 안 함 | 초기 할당 상태로 공간(HWM) 완전 초기화 | 공간 전체 반환 및 객체 삭제 |
| **제약 조건** | 트리거 작동, FK 참조 시 제약 | 트리거 미작동 | 뷰, 프로시저 등이 무효화(Invalid)됨 |

### 2. ALTER 명령어의 구조적 유연성과 위험성
`ALTER` 명령어는 테이블의 속성(Column)을 추가, 삭제, 수정할 수 있습니다. 하지만 대용량 테이블에 대해 `ALTER TABLE ... ADD COLUMN` (기본값 포함)을 수행할 경우, DBMS 버전에 따라 전체 테이블 데이터를 재구성(Reorg)하거나 대규모 락(Lock)을 유발하여 시스템 장애를 초래할 수 있으므로 동작 방식을 정확히 이해해야 합니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 무중단 서비스에서의 DDL (Online DDL)
* **문제점:** 과거의 DBMS는 DDL 수행 시 테이블 단위의 배타적 잠금(Exclusive Lock)을 요구하여, 스키마를 변경하는 동안 다른 사용자의 읽기/쓰기 작업이 차단되었습니다.
* **해결책 및 적용:** 최신 DBMS는 스키마 변경 중에도 DML을 허용하는 Online DDL(또는 In-Place DDL) 기능을 지원합니다. 실무에서는 스키마 변경 시 반드시 `ALGORITHM=INPLACE, LOCK=NONE` (MySQL 예시)과 같은 옵션을 적용하여 서비스 중단(Downtime)을 제로화하는 아키텍처 운영 역량이 필수적입니다.
* **보완 전략:** DDL을 수행할 수 없는 엄격한 무중단 환경에서는 기존 테이블을 두고 새 구조의 뷰(View)를 만들거나, 마이그레이션 툴(예: pt-online-schema-change)을 활용하는 우회 전략을 병행해야 합니다.

### 2. 기술사적 통찰 (DDL의 형상 관리)
데이터베이스의 스키마 구조는 애플리케이션의 클래스 구조와 직결됩니다. 애플리케이션 코드는 Git 등으로 엄격히 형상 관리가 되지만, DDL 명령어는 DBA의 수동 작업에 의존하는 경우가 많아 버전 불일치로 인한 장애가 빈번합니다.
기술사는 Flyway, Liquibase 같은 '데이터베이스 마이그레이션 자동화 도구'를 CI/CD 파이프라인에 통합하여, 모든 DDL 스크립트를 코드로 관리(Database as Code)하고 이력 추적과 자동 롤백 환경을 구성하는 아키텍처를 주도해야 합니다.

## Ⅴ. 기대효과 및 결론

### 1. 체계적인 DDL 관리의 기대효과
* **데이터 무결성 확립:** 초기 DDL 단계에서 정확한 타입과 무결성 제약조건(Constraints)을 정의함으로써 애플리케이션의 논리적 오류를 DB 레벨에서 원천 차단할 수 있습니다.
* **공간 및 성능 최적화:** 파티셔닝 전략 등이 반영된 고급 DDL을 통해 대용량 데이터의 분산 저장과 검색 성능 극대화를 초기 단계부터 확보할 수 있습니다.

### 2. 결론
DDL은 데이터베이스 생태계의 기반을 다지는 설계도와 같습니다. 견고하게 작성된 DDL은 데이터베이스의 안정성, 무결성, 성능을 보장하는 첫 단추입니다. 
초대용량 데이터 환경과 애자일(Agile) 개발 방법론이 교차하는 현대 IT 환경에서, DDL은 단순히 테이블을 생성하는 명령어를 넘어 자동화된 배포 파이프라인 안에서 엄격하게 버전 통제되어야 하는 핵심 인프라 자산으로 진화하고 있습니다. 기술사는 논리적 모델링의 완벽함뿐만 아니라 무중단 배포를 고려한 물리적 DDL 설계 능력을 갖추어야 합니다.
