# 389. 대용량 트랜잭션의 배칭(Batching) 삽입 최적화 - DB를 숨 쉬게 하는 데이터 대량 수송

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 수천~수만 건의 데이터를 하나씩 처리하지 않고 **하나의 트랜잭션으로 묶어(Batching) 한 번에 디스크에 쓰는 기술**로, 커밋(Commit) 오버헤드와 네트워크 왕복 시간(RTT)을 획기적으로 줄이는 최적화 기법이다.
> 2. **혁신**: 매 건마다 발생하는 로그 기록(WAL)과 인덱스 갱신을 몰아서 처리하며, **Bulk Insert나 PostgreSQL의 COPY 명령어**를 활용해 DB 엔진의 내부 처리 단계를 최소화한다.
> 3. **가치**: 로그 수집, 대규모 정산, 초기 데이터 마이그레이션 시 전체 처리 시간(Throughput)을 10배에서 100배까지 단축하며, 시스템 자원(CPU, I/O) 낭비를 방지한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
배칭(Batching) 삽입은 "몰아서 넣기"다. 10,000개의 행을 넣을 때 `INSERT` 문을 10,000번 날리는 게 아니라, 하나의 문장에 담거나 하나의 트랜잭션 안에서 수행하는 것이다. DB 입장에서는 도장을 10,000번 찍는 대신 딱 한 번만 찍고 끝내는 효율적인 방식이다.

### 💡 비유
- **일반 삽입 (우체국 10,000번 방문)**: 편지 한 통을 보낼 때마다 우체국에 간다. 차 기름값(네트워크 비용)이 엄청나고, 접수원(DB)은 매번 똑같은 확인(트랜잭션)을 하느라 지친다.
- **배칭 삽입 (우편 자루)**: 편지 10,000통을 자루 하나에 담아서 한 번에 우체국에 가져간다. 기름값은 한 번만 들고, 접수원도 자루 전체에 도장 한 번(Commit)만 쾅 찍으면 된다. 훨씬 빠르고 경제적이다.

### 등장 배경: 'Commit Overhead'의 벽
DB는 데이터를 쓸 때마다 "진짜 썼다"고 장부(WAL)에 적고 디스크에 물리적으로 고정해야 한다. 이 과정이 매우 느리다. 초당 수십만 건의 로그가 쏟아지는 빅데이터 환경에서 건별 커밋 방식은 하드웨어 사양과 상관없이 무조건 병목을 일으킨다.

## Ⅱ. 배칭 삽입의 핵심 원리 및 최적화 기법 (Deep Dive)

### 배칭 삽입 프로세스 비교 (ASCII Diagram)

```ascii
[ 1. Row-by-Row (Bad) ]              [ 2. Batching (Good) ]
Application      Database            Application      Database
     │--- INSERT ---▶│ (Lock/Log)         │                │
     │◀--- OK (1) ---│ (Commit)          │--- INSERT(1) --│
     │--- INSERT ---▶│ (Lock/Log)         │--- INSERT(2) --│ (Memory Buffer)
     │◀--- OK (2) ---│ (Commit)          │--- INSERT(n) --│
     │      ...      │                    │--- COMMIT ----▶│ (Flush to Disk)
                                          │◀----- OK ------│ (Once!)

* 최적화 기법 종류:
1. Multi-row Insert: INSERT INTO table VALUES (row1), (row2), ...
2. Transaction Grouping: BEGIN; INSERT...; INSERT...; COMMIT;
3. Loader API: PostgreSQL COPY, Oracle SQL*Loader (파일 직주입)
```

### 3대 핵심 최적화 요소

#### ① 네트워크 지연 제거 (RTT Reduction)
클라이언트와 서버 사이의 통신 횟수를 줄인다. 10,000번의 네트워크 왕복이 1번으로 줄어들면, 네트워크 레이턴시가 10ms일 때만 해도 100초의 시간을 아낄 수 있다.

#### ② 인덱스 갱신 지연 (Index Deferred)
데이터를 넣을 때마다 인덱스 나무(B-Tree)를 재배치하면 매우 느리다. 배칭 시에는 대량의 데이터를 일단 다 집어넣고, 인덱스를 한꺼번에 정렬하여 갱신함으로써 I/O 부하를 획기적으로 낮춘다.

#### ③ WAL (Write Ahead Log) 병합
매건 커밋 시 디스크 헤더가 로그 파일의 끝으로 달려가서 써야 한다(fsync). 배칭은 이 물리적인 디스크 쓰기 횟수를 줄여 SSD의 수명을 보호하고 처리량을 높인다.

## Ⅲ. 융합 비교 및 다각도 분석

### 일반 Insert vs Multi-row vs COPY 명령어 비교 (Table)

| 구분 | 일반 Insert (Single) | Multi-row Insert | **Bulk Copy (COPY/LOAD)** |
|:---:|:---|:---|:---|
| **속도** | 매우 느림 | 빠름 | **가장 빠름 (최고 100배)** |
| **로그 발생** | 매 건 발생 | 트랜잭션당 발생 | **최소화 (Minimal Logging)** |
| **SQL 파싱** | 매번 수행 | 1번 수행 | **건너뜀 (바이너리 직주입)** |
| **제약 조건** | 즉시 체크 | 문장 종료 후 체크 | 데이터 로드 후 한꺼번에 체크 |
| **주 사용처** | 일반 웹 요청 처리 | 수백 건의 리스트 저장 | **수백만 건의 마이그레이션** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 대규모 로그 수집 시스템 설계 전략)

**상황**: 매분 100만 건의 트래픽 로그를 PostgreSQL DB에 쌓아야 한다. 현재 자바 앱에서 `save()`를 호출하면 DB CPU가 100%를 찍으며 지연이 발생한다.
**판단 및 최적화 전략**:
1. **버퍼링 도입**: 앱 메모리에 5,000건 혹은 1초 단위로 로그를 모으는 **큐(Queue)**를 구현한다.
2. **COPY 명령어 활용**: JDBC의 `copyIn` API를 사용하여 텍스트나 바이너리 스트림으로 데이터를 직주입한다. 이는 일반 SQL보다 10배 이상 빠르다.
3. **병렬 배치**: 서버 한 대에서만 쏘지 않고, 로그 소스별로 **병렬 커넥션**을 맺어 멀티코어 성능을 활용한다.
4. **인덱스 임시 해제**: 마이그레이션처럼 엄청난 양이라면, **Index와 Constraint를 일시 해제**(Drop)하고 데이터 로드 후 재생성한다.
5. **결과**: CPU 점유율을 30%로 낮추면서도 초당 10만 건 이상의 데이터 적재 성공.

### 주의사항 및 안티패턴
- **너무 큰 배치 사이즈**: 한 번에 100만 건을 배치로 묶으면, 트랜잭션이 끝날 때까지 DB의 **언두(Undo) 로그**나 메모리가 부족해져서 시스템이 뻗을 수 있다. 보통 1,000~10,000건 사이에서 최적의 사이즈를 벤치마킹해야 한다.
- **배치 실패와 롤백**: 배치 중 단 1건이라도 에러가 나면 10,000건 전체가 롤백된다. 실패한 데이터만 따로 골라내어 재시도하는 **'Dead Letter Queue'** 설계가 필수적이다.

## Ⅴ. 미래 전망 및 결론

### 결론: 효율이 곧 지능이다
대용량 데이터 처리는 화려한 알고리즘보다 'I/O의 경제성'이 본질이다. 배칭 삽입 최적화는 하드웨어의 물리적 한계 안에서 소프트웨어가 낼 수 있는 최대의 효율을 추구하는 과정이다.

### 미래 전망
앞으로는 **'자동 가변 배치(Auto-Batching)'**가 보편화될 것이다. DB 드라이버나 커널 레벨에서 트래픽을 감지하여 최적의 배치 크기를 0.1초마다 조절하는 것이다. 또한, **[LSM-Tree](./377_lsm_tree.md)**와 같이 아예 쓰기에 특화된 엔진이 보편화되면서, 별도의 배치 코드를 짜지 않아도 인프라 레벨에서 대량 유입 데이터를 순차적으로 소화하는 'Write-optimzed Cloud DB' 시대가 가속화될 전망이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **배칭 삽입이 뭔가요?**: 장난감 100개를 상자에 담을 때, 한 개씩 들고 뛰어가서 넣는 게 아니라 커다란 바구니에 다 담아서 한 번에 쏟아붓는 거예요.
2. **왜 좋나요?**: 100번 왔다 갔다 하면 다리가 너무 아프고 시간도 오래 걸리지만, 바구니를 쓰면 한 번만 가면 되니까 힘도 안 들고 아주 빨리 끝나요.
3. **미래에는요?**: 우리가 장난감을 바닥에 내려놓기만 해도, 보이지 않는 로봇 팔이 알아서 상자에 척척 담아주는 똑똑한 정리함이 생길 거랍니다!
