# 275. NoSQL 데이터 모델 4가지 (Key-Value, Document, Column, Graph)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 관계형 DB(RDBMS)의 엄격한 테이블 스키마 한계를 벗어나, 다루고자 하는 데이터의 특성(단순성, 유연성, 대용량, 관계성)에 최적화된 4가지 고유한 저장 구조(데이터 모델)를 제공하는 비관계형 데이터베이스 아키텍처다.
> 2. **가치**: 대규모 분산 환경에서 페타바이트급 데이터의 수평적 확장(Scale-out)을 쉽게 하며, 비정형/반정형 데이터의 빠른 읽기/쓰기를 지원하여 클라우드 네이티브 애플리케이션의 핵심 저장소로 작용한다.
> 3. **융합**: 빅데이터 아키텍처, 캐싱 시스템(Redis), 실시간 분석 시스템과 결합되며, 각 모델은 CAP 정리(Consistency, Availability, Partition Tolerance)의 각기 다른 측면을 트레이드오프하여 설계되었다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
NoSQL(Not Only SQL) 데이터 모델 4가지는 데이터를 테이블 행과 열의 2차원 구조로 강제하는 RDBMS와 달리, 애플리케이션의 접근 패턴에 맞춘 고유한 자료구조로 데이터를 저장하는 방식이다. 크게 **키-값(Key-Value), 문서(Document), 컬럼 패밀리(Column Family / Wide Column), 그래프(Graph)** 4가지 모델로 분류된다.

### 💡 비유
- **Key-Value**: "사물함 보관소" (열쇠 번호를 주면, 안에 뭐가 들었든 상자 통째로 내어줌)
- **Document**: "인사 기록 서류철" (한 사람의 모든 복잡한 이력, 가족관계가 한 장의 유연한 JSON 서류에 다 적혀 있음)
- **Column Family**: "대형 도서관의 십진분류법 책장" (관련된 주제의 책들을 같은 선반에 모아두어, 수백만 권 속에서도 특정 주제를 빠르게 뽑아냄)
- **Graph**: "소셜 네트워크 인물 관계도" (누가 누구와 친구인지, 선으로 이어진 관계 자체를 핵심 데이터로 저장함)

### 등장 배경 및 발전 과정

#### 1. 관계형 DB의 한계 폭발
2000년대 중반, 웹 2.0 시대로 진입하며 구글, 아마존, 페이스북 등은 폭증하는 비정형 데이터(로그, 이미지, 소셜 피드)와 수천만 명의 동시 접속을 RDBMS의 수직적 확장(Scale-up, 비싼 장비 도입)으로는 도저히 감당할 수 없었다. 엄격한 ACID 트랜잭션이 모든 서비스에 필수적인 것은 아니었다.

#### 2. 패러다임 변화 (다양성 모델의 등장)
아마존의 Dynamo 논문(Key-Value 기반 고가용성), 구글의 Bigtable 논문(Column Family 기반 분산 스토리지)이 발표되며, 데이터 특성에 맞게 스키마를 없애고 분산 클러스터 확장에 최적화된 4가지 독자적 NoSQL 모델 생태계가 정립되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 4대 데이터 모델 구성 및 특징 (심층 분석표)

| 데이터 모델 | 데이터 구조 | 핵심 원리 | 대표 제품 | 장점 및 비유 |
|-------------|-------------|-----------|-----------|--------------|
| **Key-Value** | `Key` : `Value(BLOB)` | 가장 단순한 Hash Map 구조. 값은 DB가 해석하지 않는 블록(Opaque) | Redis, Amazon DynamoDB, Riak | 초고속 읽기/쓰기, 캐싱 최적화. "사물함" |
| **Document** | `Key` : `JSON/BSON` | Key-Value와 유사하나, Value가 내부 계층 구조(Tree)를 가진 문서 형식 | MongoDB, CouchDB, Elasticsearch | 스키마 유연성, 복잡한 데이터 한 번에 조회. "서류철" |
| **Column Family** | `Row Key` : `Column Family` | 하나의 Row Key 아래 연관된 컬럼들을 묶음(Family)으로 저장. (희소 데이터 최적) | Apache Cassandra, HBase, ScyllaDB | 압축률 우수, 쓰기 속도 극대화, 분산 노드 확장성. "주제별 선반" |
| **Graph** | `Node` - `Edge` - `Property` | 개체(Node)와 개체 간의 관계(Edge)를 포인터로 직접 연결하여 저장 | Neo4j, Amazon Neptune, TigerGraph | 깊은 관계 탐색(Join 대체) 압도적 성능. "마인드 맵" |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ 1. Key-Value Store ]                     [ 2. Document Store ]
================================================================================
  Hash Table                                JSON Object (Hierarchical)
  ┌────────────┬──────────────────┐         ┌────────────┬─────────────────────┐
  │   Key      │      Value       │         │   Key      │      Document       │
  ├────────────┼──────────────────┤         ├────────────┼─────────────────────┤
  │ session_01 │ "user:bob,age:20"│         │  user_101  │ {                   │
  ├────────────┼──────────────────┤         │            │  "name": "Alice",   │
  │ cart_892   │ [item1, item2]   │         │            │  "age": 30,         │
  └────────────┴──────────────────┘         │            │  "address": {       │
  * DB는 Value 내용물에 관심 없음               │            │    "city": "NY"    │
  * O(1)의 접근 속도                            │            │  }                  │
                                            │            │ }                   │
                                            └────────────┴─────────────────────┘

================================================================================
[ 3. Column Family Store ]
================================================================================
  Row 기반이 아닌, 연관된 Column들의 묶음 단위로 물리적 분산 저장
  
  Row Key: "User123"
  ┌────────────────────────────────────────────────────────────────────────┐
  │ Column Family: [ Profile ]           │ Column Family: [ Activity ]     │
  ├─────────┬─────────┬────────┬─────────┼──────────┬──────────┬───────────┤
  │ Name    │ Email   │ Age    │ (null)  │ Log_In   │ Click    │ Purchase  │
  ├─────────┼─────────┼────────┼─────────┼──────────┼──────────┼───────────┤
  │ "Bob"   │ b@x.com │ 25     │         │ "10:00"  │ "Item_A" │ "None"    │
  └─────────┴─────────┴────────┴─────────┴──────────┴──────────┴───────────┘
  * 각 Column은 (이름, 값, 타임스탬프) 구조를 가짐. 값이 없는 컬럼은 저장 공간을
    차지하지 않음 (Sparse Matrix 지원). 극단적인 쓰기 스케일아웃(Write-heavy)에 최적.

================================================================================
[ 4. Graph Store ]
================================================================================
                    [ Node: Person ]
                     (name: "Alice")
                        /       \
       [ Edge: KNOWS ] /         \ [ Edge: LIKES ]
       (since: 2021)  /           \
                     v             v
            [ Node: Person ]    [ Node: Movie ]
            (name: "Bob")       (title: "Inception")

  * RDBMS라면 여러 테이블을 만들고 수많은 JOIN을 걸어야 할 관계를,
    메모리 상에서 직접 물리적 포인터(Index-free adjacency)로 연결해버림.
    친구의 친구의 친구 탐색 시 RDBMS 대비 수백 배 빠름.
```

### 심층 동작 원리

#### ① Document Store (MongoDB 등)
JSON 형식을 바이너리 포맷(BSON)으로 저장한다. RDBMS가 사용자 정보, 주문 정보, 배송 정보를 정규화하여 각 테이블에 쪼개어 담는다면, Document 모델은 이 모든 것을 하나의 거대한 JSON 객체로 비정규화(Denormalization)하여 저장한다. 읽을 때 JOIN이 필요 없어 성능이 매우 빠르다. 스키마가 없으므로 개발자가 객체의 구조를 마음대로 바꿀 수 있다(Schema on Read).

#### ② Column Family Store (Cassandra 등)
구글 Bigtable의 논문을 기초로 한다. 논리적으로는 표(Table)처럼 보이지만, 물리적으로는 같은 Row에 있더라도 컬럼 패밀리가 다르면 다른 파일(SSTable)이나 노드에 저장될 수 있다. 새로운 컬럼을 제약 없이 동적으로 추가할 수 있으며, 클러스터 노드 간의 마스터 없는 P2P 분산 구조(Gossip Protocol)를 통해 수천 대의 서버로 선형적 확장이 가능하다.

#### ③ Graph Store (Neo4j 등)
'Index-Free Adjacency(인덱스 없는 인접성)'가 핵심이다. 관계형 DB는 관계를 찾을 때마다 인덱스를 스캔하여 외래키(Foreign Key)를 대조해야 하지만, Graph DB는 노드가 생성될 때 주변 노드의 메모리 주소를 직접 엣지에 기록해둔다. 따라서 관계의 깊이(Depth)가 10단계, 20단계로 깊어져도 탐색 속도가 기하급수적으로 느려지지 않고 일정하게 빠르다.

## Ⅲ. 융합 비교 및 다각도 분석

### RDBMS vs NoSQL 4개 모델 활용 매트릭스

| 사용 목적 | 추천 데이터 모델 | 이유 | 대표 사례 |
|-----------|------------------|------|-----------|
| **복잡한 트랜잭션, 정합성** | **RDBMS** | 엄격한 스키마, ACID 트랜잭션 보장 | 금융 시스템, 회계 원장 |
| **세션 처리, 장바구니, 캐싱** | **Key-Value** | 응답 시간 < 1ms 수준의 극강의 속도 | Redis 기반 웹 세션 저장소 |
| **콘텐츠 관리, 카탈로그, CMS** | **Document** | 스키마 제약 없이 유연한 객체 구조 저장 | 이커머스 상품 카탈로그 |
| **시계열 데이터, IoT 센서 로그** | **Column Family** | 페타바이트급 분산 쓰기 최적화 | 넷플릭스 유저 활동 로그 |
| **추천 엔진, 소셜 네트워크, 사기 탐지** | **Graph** | 복잡한 관계망 실시간 탐색 최적화 | 링크드인 인맥, 금융 FDS |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 글로벌 서비스 런칭을 앞두고, 유저의 접속 상태 로그(초당 수십만 건 발생)와 유저 간의 복잡한 친구 추천 및 팔로워 매핑 기능을 구축해야 함. 기존 오라클 RDBMS로는 로그 쓰기 병목과 다중 JOIN 시 타임아웃 발생.
**아키텍처 전략 (Polyglot Persistence 도입)**:
1. **유저 결제/계정 정보**: 데이터 정합성이 생명이므로 기존 **RDBMS(PostgreSQL)** 유지.
2. **접속 상태 및 초당 발생 로그**: 엄청난 쓰기 부하를 감당하고 서버 증설이 쉬운 **Column Family(Cassandra)**에 분산 저장.
3. **친구 추천 및 팔로우 관계망**: 다대다 관계의 깊은 탐색을 위해 **Graph DB(Neo4j)**를 도입하여, 실시간으로 "친구가 아는 친구 중 관심사가 같은 사람"을 밀리초 내에 쿼리함.
**결론**: 하나의 "만능 DB"는 없다. 마이크로서비스 아키텍처(MSA) 관점에서 도메인 특성에 가장 잘 맞는 여러 NoSQL 모델을 혼용하는 폴리글랏 퍼시스턴스(Polyglot Persistence)를 설계해야 한다.

### 주의사항 및 안티패턴
- **Document DB 남용**: 관계(Relation)가 매우 많은 데이터를 비정규화해서 Document DB에 넣으면, 데이터 중복이 극심해지고 추후 데이터 수정 시 수만 개의 문서를 동시에 업데이트해야 하는 '데이터 불일치 늪'에 빠진다.
- **분산 트랜잭션의 착각**: NoSQL은 기본적으로 단일 문서/로우 수준의 원자성(Atomicity)만 보장한다. RDBMS처럼 여러 테이블을 묶어 롤백(Rollback)하는 기능을 기대하고 설계하면 비즈니스 로직에 치명적 결함이 발생한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **확장성**: RDBMS의 Scale-up 한계를 벗어나, 저렴한 x86 서버 수백 대를 병렬 연결(Scale-out)하여 저장 용량과 처리량을 무한대에 가깝게 확장 가능.
- **개발 생산성**: Document DB의 경우 ORM 매핑 없이 프론트엔드/백엔드의 JSON 데이터를 그대로 DB에 밀어 넣을 수 있어 스키마 관리 공수가 50% 이상 절감됨.

### 미래 전망 및 진화 방향
초기에는 4가지 모델이 서로 완전히 분리된 제품(Redis, Mongo, Neo4j)으로 존재했으나, 최근에는 클라우드 벤더들을 중심으로 **Multi-model Database (예: Azure Cosmos DB, ArangoDB, PostgreSQL의 JSONB 지원)** 하나가 이 4가지 데이터 모델을 모두 지원하도록 융합되는 추세이다. 또한, NewSQL이라는 이름으로 NoSQL의 확장성과 RDBMS의 ACID를 결합한 차세대 분산 데이터베이스(Spanner, CockroachDB)가 시장을 재편하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **NoSQL 모델 4가지란?**: 물건의 모양에 따라 다른 종류의 정리함을 쓰는 거예요. 옛날에는 무조건 엑셀표 같은 칸막이 서랍(RDBMS)에만 구겨 넣어야 했어요.
2. **어떤 상자들이 있나요?**: 그냥 통째로 던져넣는 '바구니(Key-Value)', 복잡한 장난감 세트를 묶어놓는 '보관함(Document)', 같은 종류끼리 모으는 '선반(Column)', 점을 선으로 잇는 '거미줄(Graph)'이 있어요.
3. **왜 이렇게 나눴나요?**: 일일이 칸막이에 맞추느라 시간 낭비하지 않고, 물건 특성에 딱 맞는 상자를 써서 엄청나게 많은 장난감을 빛의 속도로 찾기 위해서랍니다!
