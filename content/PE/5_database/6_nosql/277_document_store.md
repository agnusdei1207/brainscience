# 277. 문서 저장소 (Document Store)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터를 행(Row)과 열(Column)의 테이블 구조가 아닌, JSON, BSON, XML 형태의 내부 계층 구조(Tree)를 가진 유연한 '문서(Document)' 단위로 저장하고 조회하는 NoSQL 데이터베이스 모델이다.
> 2. **가치**: 스키마(Schema)를 미리 정의할 필요가 없어 비즈니스 요구사항 변화에 극도로 민첩하게 대응할 수 있으며, 개발자가 사용하는 객체(Object) 형태를 그대로 DB에 저장하여 객체-관계 불일치(Impedance Mismatch)를 해소한다.
> 3. **융합**: 웹/모바일 애플리케이션의 백엔드(Node.js, Spring Boot 등), 콘텐츠 관리 시스템(CMS), 카탈로그 관리, 실시간 로깅 시스템 등 유연성과 고속의 읽기/쓰기가 필요한 모든 영역의 기본 저장소로 광범위하게 활용된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
문서 저장소(Document Store)는 Key-Value 저장소의 진화된 형태이다. Key-Value가 데이터의 내용물(Value)을 전혀 모르는 불투명한 블록으로 취급하는 반면, 문서 저장소는 Value가 JSON(JavaScript Object Notation)과 같은 표준화된 형식을 갖는다고 이해(Aware)한다. 따라서 DB 엔진이 문서 내부의 특정 필드(예: `address.city`)를 조건으로 검색하거나 인덱스를 생성할 수 있다.

### 💡 비유
문서 저장소는 **"병원 환자의 종합 차트 파일"**과 같다.
전통적 관계형 DB(RDBMS)에서는 환자의 정보를 보려면 '기본 인적사항 서랍', '과거 병력 서랍', '처방전 서랍'을 다 열어서 정보를 조립(JOIN)해야 한다. 하지만 문서 저장소는 파일 하나(Document) 안에 그 환자의 모든 인적사항, 과거 병력 배열, 처방 내역 리스트가 트리(Tree) 구조로 다 들어있다. 차트 하나만 딱 꺼내면 환자의 모든 것을 한 번에 알 수 있어 매우 빠르고 직관적이다.

### 등장 배경 및 발전 과정

#### 1. 관계형 DB의 임피던스 불일치 (Impedance Mismatch)
개발자들은 Java, Python, JS 객체(Object)와 배열(Array)을 사용하여 계층적인 데이터를 다루는데, 이를 RDBMS에 저장하려면 ORM(Hibernate 등)을 사용해 데이터를 평면적인 테이블 구조로 해체(정규화)해야 했다. 이 변환 과정은 무겁고, 코드를 복잡하게 만들었다.

#### 2. 애자일 개발과 스키마 변경의 고통
스타트업과 애자일 개발 환경에서는 새로운 기능이 매일 추가된다. RDBMS에서 테이블에 컬럼 하나를 추가(`ALTER TABLE`)하려면 서비스 중단(Downtime)이나 심각한 락(Lock)이 발생했다. 이에 "개발 코드의 JSON 객체를 변환 없이 그대로 던져 넣을 수 있는 데이터베이스"의 필요성이 폭발했고, **MongoDB**를 필두로 한 Document DB가 대성공을 거두게 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (표)

| 요소명 | 상세 역할 | 내부 동작 메커니즘 | RDBMS 대응 개념 | 비유 |
|--------|----------|-------------------|-----------------|------|
| **Document** | 데이터 저장의 최소 단위 | JSON, BSON 포맷의 계층적 데이터 객체 (내부에 배열, 하위 객체 포함 가능) | Row (행) | 환자 1명의 차트 |
| **Collection** | 문서들의 논리적 그룹 | 스키마를 강제하지 않지만, 비슷한 용도의 문서들을 모아두는 바구니 | Table (테이블) | 소아과 차트 보관함 |
| **_id (Object ID)**| 문서의 고유 식별자 | DB가 자동 생성하거나 사용자가 지정하는 고유 키 (B+Tree 기본 인덱스) | Primary Key (기본키) | 차트 일련번호 |
| **BSON (Binary JSON)**| 직렬화 포맷 | 텍스트 JSON을 이진 형태로 압축하여 파싱 속도와 저장 효율 극대화 | Storage Format | 암호화된 압축 서류 |
| **Secondary Index** | 검색 최적화 보조 인덱스 | 문서 내부의 깊은(Deep) 계층에 있는 특정 필드에 대한 B-Tree 인덱싱 지원 | Index | 색인 스티커 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ RDBMS (Normalized) vs Document Store (Denormalized) ]
================================================================================

[ 1. RDBMS: 정규화 구조 (JOIN 필수) ]
  Table: Users              Table: Addresses            Table: Hobbies
  +----+-------+          +------+----+---------+     +------+----+---------+
  | ID | Name  | 1 : N    | U_ID | Seq| City    |     | U_ID | Seq| Hobby   |
  +----+-------+ ───────> +------+----+---------+     +------+----+---------+
  | 1  | Alice |          | 1    | 1  | Seoul   |     | 1    | 1  | Reading |
  +----+-------+          | 1    | 2  | Jeju    |     | 1    | 2  | Gaming  |
                          +------+----+---------+     +------+----+---------+
  * Alice의 전체 정보를 보려면 3개의 테이블을 JOIN 연산해야 함 (I/O 병목 발생)

[ 2. Document Store (MongoDB): 비정규화(Embedded) 구조 ]
  Collection: Users
  ┌─────────────────────────────────────────────────────────────┐
  │ Document _id: ObjectId("5f3a...123")                        │
  │ {                                                           │
  │   "name": "Alice",                                          │
  │   "addresses": [                                            │
  │      { "city": "Seoul", "type": "home" },  <-- 하위 객체 임베딩 │
  │      { "city": "Jeju", "type": "work" }                     │
  │   ],                                                        │
  │   "hobbies": ["Reading", "Gaming"]        <-- 배열 직접 지원  │
  │ }                                                           │
  └─────────────────────────────────────────────────────────────┘
  * 단 한 번의 디스크/메모리 I/O(Read)로 Alice의 모든 정보를 가져옴. (극강의 성능)
  * 새로운 필드(예: "age": 30)를 다른 문서에만 추가해도 에러가 안 남 (Schema-less).
```

### 심층 동작 원리

#### ① 스키마 온 리드 (Schema-on-Read)
RDBMS는 데이터를 쓰기(Write) 전에 구조가 엄격하게 검증된다(Schema-on-Write). 반면 문서 저장소는 어떤 모양의 JSON이든 우선 무조건 저장한다. 데이터의 구조와 유효성 검증은 데이터베이스가 아닌, 데이터를 읽어서 사용하는 애플리케이션(코드 레벨)에서 책임을 진다.

#### ② 임베딩(Embedding) vs 참조(Referencing)
- **임베딩(비정규화)**: 연관된 데이터를 한 문서 안에 몰아넣는 방식. 읽기(Read) 성능이 극대화되지만, 문서의 크기 제한(예: MongoDB 16MB)을 초과할 수 있으며, 데이터 수정 시 중복된 모든 곳을 바꿔야 하는 단점이 있다. (1:1 또는 1:N 관계에 적합)
- **참조(정규화)**: 문서 안에 다른 문서의 `_id`만 적어두는 방식. 데이터 중복은 피할 수 있으나, 읽을 때 애플리케이션 레벨에서 여러 번 쿼리를 날려야(Join 모방) 한다. (N:M 관계에 적합)

#### ③ 인덱싱과 쿼리 엔진
Key-Value와 달리 문서 저장소는 `db.users.find({"addresses.city": "Seoul"})` 처럼 문서 깊숙한 곳의 값으로 검색이 가능하다. 내부적으로는 B-Tree 인덱스를 유지하여 풀 스캔(Full Scan)을 방지하며, 지리공간(Geospatial) 인덱스나 텍스트 검색(Full-text) 인덱스도 지원한다.

## Ⅲ. 융합 비교 및 다각도 분석

### RDBMS vs Document Store 비교

| 비교 지표 | RDBMS | Document Store (MongoDB 등) |
|-----------|-------|-----------------------------|
| **데이터 모델** | 2차원 테이블 (행/열) | 계층적 문서 (JSON/Tree) |
| **스키마 강제성** | 강함 (엄격한 제약조건, 정규화) | 없음 / 유연함 (Dynamic Schema) |
| **JOIN 연산** | 데이터베이스 자체에서 고속 처리 | DB 레벨 지원 한계 (주로 애플리케이션 조인) |
| **트랜잭션(ACID)**| 완벽한 다중 테이블 트랜잭션 보장 | 기본적으로 단일 문서 수준 ACID (최근 다중 문서 지원 중이나 성능 저하) |
| **수평적 확장** | 구조적으로 매우 어려움 (Sharding 복잡) | 아키텍처 자체가 분산 Sharding에 최적화 |
| **개발 생산성** | ORM 세팅 및 테이블 관리 공수 큼 | 객체를 JSON으로 직렬화하여 바로 저장 (매우 높음) |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대규모 이커머스 시스템에서 '상품 카탈로그'를 구축해야 한다. 스마트폰, 의류, 신선식품 등 상품마다 필요한 속성(크기, 색상, 유통기한, 해상도 등)이 수십, 수백 가지로 파편화되어 있다. RDBMS로 설계하려니 수백 개의 컬럼을 만들거나 EAV(Entity-Attribute-Value) 안티패턴을 써야 해서 성능이 극도로 저하됨.
**판단 및 전략**:
1. **Document Store 도입**: 상품의 속성을 미리 규정하지 않고 각 상품의 특성에 맞는 JSON 객체 그대로 **MongoDB**에 저장한다.
   - 스마트폰: `{"screen_size": "6inch", "ram": "8GB"}`
   - 의류: `{"color": "Red", "size": "XL", "material": "Cotton"}`
2. **동적 필터링**: 사용자가 특정 카테고리에서 "8GB RAM"을 검색하면, 해당 필드에 동적으로 생성된 인덱스를 태워 즉각 응답한다.
3. **분할 전략 (CQRS)**: 결제 및 재고 차감 같은 강한 정합성이 필요한 정보는 RDBMS에 두고, 고객에게 보여지는 카탈로그 전시 정보만 Document Store에 배치하여 트래픽 부하를 분산한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **무분별한 데이터 중복 (Data Anomaly)**: RDBMS의 JOIN을 피하려고 작성자 이름과 프로필 사진을 수백만 개의 게시글 문서 안에 임베딩(Embedding)해버리면, 작성자가 프로필 사진을 바꿀 때 수백만 개의 문서를 일일이 업데이트(Write)해야 하는 재앙이 발생한다.
- **거대 문서 생성 (Mega Document)**: 유저의 '로그인 이력'을 유저 문서 내의 배열(Array)로 끝없이 추가(Append)하면, 문서 크기 제한을 초과하고 패딩을 뚫어 디스크 단편화를 유발한다. 무한히 커지는 배열은 반드시 별도의 Collection으로 분리(참조)해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **개발 리드타임 단축**: DB 스키마 설계 및 DDL 승인 절차 생략, 백엔드 로직(JSON)과의 완벽한 일치로 API 개발 속도 **2배 이상 향상**.
- **조회 성능(Read IOPS)**: 하나의 문서를 읽어 객체를 복원하는 데 단 한 번의 디스크/캐시 접근만 필요하므로 복잡한 RDBMS JOIN 대비 **조회 지연 시간 대폭 감소**.

### 미래 전망 및 진화 방향
과거 Document Store는 "스키마가 없어 관리가 안 되고 트랜잭션이 약한 장난감 DB"라는 비판을 받기도 했다. 그러나 MongoDB 4.0 이후 다중 문서 ACID 트랜잭션과 JSON Schema Validation(강제 검증 기능)이 추가되면서 엔터프라이즈급 안정성을 확보했다. 향후에는 RDBMS 진영(PostgreSQL의 JSONB 기능 등)과 서로의 장점을 흡수하며 **멀티-모델(Multi-model) 데이터베이스**의 핵심 엔진으로 융합될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **문서 저장소란?**: 엑셀표처럼 답답한 칸에 맞춰 글씨를 쓰는 게 아니라, 자유로운 하얀 도화지(JSON 문서)에 내가 원하는 모양대로 그림과 글씨를 마음대로 적어서 보관하는 금고예요.
2. **왜 좋나요?**: 장난감 설명서엔 '색깔'을, 로봇 설명서엔 '배터리 종류'를 따로 칸을 만들 필요 없이 자유롭게 적을 수 있어서 새로운 장난감이 나와도 당황하지 않아요.
3. **특징은 뭔가요?**: 관련된 정보(나의 친구 목록, 내가 좋아하는 음식)를 도화지 한 장에 다 그려놓으니까, 나중에 내 정보를 찾을 때 여러 장을 뒤지지 않고 딱 한 장만 뽑아보면 끝난답니다!
