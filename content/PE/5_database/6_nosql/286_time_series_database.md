# 286. 시계열 데이터베이스 (Time Series Database, TSDB)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시간에 따라 지속적으로 생성되는 데이터(타임스탬프와 값의 쌍)를 저장하고 분석하는 데 극도로 최적화된 특수 목적의 데이터베이스로, Insert 연산이 전체의 95% 이상을 차지하는 특징을 가진다.
> 2. **가치**: 관계형 DB(RDBMS)에서는 며칠 만에 디스크와 인덱스가 파탄 날 초당 수백만 건의 IoT 센서 및 서버 모니터링 데이터를 압축하여 가볍게 저장하고, 시간 기반의 범위 검색을 밀리초(ms) 단위로 완수한다.
> 3. **융합**: 클라우드 네이티브의 관측성(Observability) 생태계를 이끄는 Prometheus(프로메테우스) 모니터링 시스템, InfluxDB, 그리고 대시보드 도구인 Grafana와 융합되어 현대 인프라 운영의 핵심 신경망으로 작용한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
시계열 데이터베이스(TSDB)는 데이터를 항상 **시간(Timestamp)**이라는 절대적 축을 기준으로 배열하여 저장하는 데이터베이스 시스템이다. 이 시스템에서 데이터는 수정(Update)되거나 삭제(Delete)되는 일이 거의 없으며, 오직 최신 시간의 데이터가 끝없이 덧붙여지는(Append-only) 구조를 갖는다.

### 💡 비유
TSDB는 **"병원 중환자실의 심전도 심박수 기록지"**와 같다.
심전도 기계는 환자의 심박수(Value)를 매초(Timestamp) 쉼 없이 종이 위로 그려낸다(Insert). 의사는 과거에 그려진 1시간 전의 그래프 선을 지우개로 지우고 수정할(Update) 필요가 전혀 없으며, 그럴 수도 없다. 의사에게 필요한 것은 오직 "최근 10분 동안 심박수 평균이 어떻게 변했는가?"를 한눈에 훑어보는(Range Query) 것이다. TSDB는 이 끊임없이 쏟아지는 종이 기록을 가장 효율적으로 보관하고 검색하는 특수 철제 캐비닛이다.

### 등장 배경 및 발전 과정

#### 1. RDBMS의 시계열 처리 붕괴
과거에는 서버 CPU 온도나 공장 센서 데이터를 MySQL 같은 RDBMS에 넣었다. 그러나 데이터가 1초마다 쌓이면서 테이블 용량이 수십 억 건이 되면, B-Tree 인덱스는 계속해서 분할(Split)을 일으키며 디스크 I/O를 소모했고 쓰기 속도는 바닥을 쳤다. 오래된 데이터를 지우는(Delete) 작업조차 엄청난 락(Lock)과 리소스 고갈을 유발했다.

#### 2. 모니터링과 IoT 폭발 시대의 도래
서버가 수백 대에서 K8s의 수만 개 컨테이너(파드)로 쪼개지고, 수백만 대의 IoT 기기가 1초 단위로 측정값을 쏘기 시작했다. "쓰기 작업이 압도적으로 많고, 시간순 정렬이 보장되어야 하며, 과거 데이터는 자동으로 삭제되는" 새로운 저장소의 필요성이 폭발했다. 이를 해결하기 위해 Hbase/Cassandra 기반의 OpenTSDB를 거쳐, 아예 시계열 전용 엔진을 바닥부터 새로 짠 **InfluxDB**와 **Prometheus**가 대세로 등극했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (표)

| 요소명 | 상세 역할 | 내부 동작 메커니즘 | 관련 도구/기능 | 비유 |
|--------|----------|-------------------|----------------|------|
| **Timestamp (시간축)**| 모든 데이터의 절대적 기준키(Primary Key) | 나노초(ns) 단위까지 정밀하게 시간 순서대로 데이터를 인덱싱함 | x축 좌표 | 일기장의 날짜 |
| **Metric (측정값)** | 실제 저장하려는 수치 데이터 | Float, Integer 형태의 값. 극단적인 압축 알고리즘(Gorilla 등)을 거쳐 저장됨 | y축 좌표 | 심박수, 온도 수치 |
| **Tags / Labels** | 메트릭의 식별자(메타데이터) | "서버=웹1, 지역=서울" 처럼 데이터를 그룹화하고 필터링하는 조건 (역색인 구조 가짐) | Prometheus Labels | 포스트잇 분류 태그 |
| **LSM-Tree / TSM** | 초고속 쓰기 및 압축 엔진 | 데이터를 메모리에 모았다가 디스크에 블록 단위로 순차적(Append)으로 써 내려감 | TSM (Time-Structured Merge Tree) | 메모장 속기 후 제출 |
| **Retention Policy** | 자동 데이터 삭제 (수명 주기) | "30일이 지난 데이터는 디스크에서 자동 파기하라"는 내장 정책 | TTL (Time-to-Live) | 오래된 신문 폐기 |

   [ CPU_Usage, host=Server_A, region=Seoul ] 09:00:02 -> 46.1%

  │             │                                          │
  │             ▼                                          │
  │ < Immutable Disk Blocks (SSTable) >                    │

     "Server_A의 08시~09시 평균 CPU 사용량을 알려줘!"

### 심층 동작 원리

#### ① 컬럼 지향 저장과 타임스탬프 압축 (Delta-of-Delta)
TSDB는 RDBMS와 달리 시계열 데이터의 특성을 이용해 엄청난 압축률을 낸다. 센서 데이터는 `09:00:01`, `09:00:02` 처럼 일정한 시간 간격으로 들어온다. 페이스북의 Gorilla 논문에서 제안된 **Delta-of-Delta 압축**은 시간을 일일이 긴 숫자(64비트)로 저장하지 않고, 이전 시간과의 '차이(Delta)의 차이'를 계산하여 거의 1~2비트만으로 시간축을 저장한다. 이를 통해 디스크 사용량을 RDBMS 대비 **1/10 수준**으로 극단적으로 줄인다.

#### ② 다운샘플링 (Downsampling / Continuous Queries)
데이터가 무한히 쌓이는 것을 막기 위해 TSDB는 자동 집계 기능을 수행한다.
- 1시간 이내의 최신 데이터: 1초 단위의 원본 고해상도(Raw) 데이터 유지.
- 1주일이 지난 데이터: 백그라운드 엔진이 1초 단위 데이터를 읽어 '1시간 평균값' 1개로 압축(Downsampling)하여 다시 저장하고, 원본 3,600개는 날려버린다. 
이를 통해 1년 치 장기 보관 데이터도 디스크 용량을 거의 차지하지 않게 된다.

#### ③ LSM-Tree 기반의 쓰기 최적화
시간순으로 들어오는 데이터의 특성에 맞춰, B-Tree처럼 디스크 여러 곳을 랜덤하게 건드리지 않고, 로그를 남기듯 파일 끝에 계속 이어 붙이는(Append-only) **LSM-Tree (Log-Structured Merge-Tree)** 아키텍처를 변형(TSM)하여 사용한다. 쓰기 속도는 메모리에 쓰는 속도와 거의 동일하다.

## Ⅲ. 융합 비교 및 다각도 분석

### RDBMS vs NoSQL(Document) vs TSDB 비교

| 비교 지표 | RDBMS (PostgreSQL) | Document DB (MongoDB) | TSDB (Prometheus/InfluxDB) |
|-----------|--------------------|-----------------------|----------------------------|
| **주요 연산(Workload)**| Read, Update, Delete 중심 | Read, Insert, Update 혼재 | **99% Insert (Append-only)** |
| **인덱스 구조**| B+Tree (랜덤 I/O 발생) | B-Tree | 시간 중심의 시계열 블록 (순차 I/O) |
| **압축 효율** | 낮음 (압축 시 CPU 오버헤드 큼) | 중간 | **극도로 높음 (시계열 특화 델타 압축)** |
| **오래된 데이터 삭제**| `DELETE` 실행 시 테이블 락 및 I/O 폭발 | TTL Collection 지원하나 I/O 부하 존재 | **Retention 설정으로 파일 단위 통째로 즉각 폐기 (오버헤드 0)** |
| **집계 쿼리 속도** | Group By 연산 무거움 | Aggregation Pipeline 제공 | **초당 수백만 건의 시간 범위 윈도우(Window) 평균을 밀리초 내 계산** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 블록체인 거래소 시스템. 수천 개의 암호화폐에 대한 실시간 호가(1초 단위 틱 데이터)가 쏟아진다. 사용자는 모바일 앱에서 "비트코인의 최근 1년 치 일봉 차트(캔들)"부터 "최근 1시간의 1분봉 차트"까지 자유롭게 줌인/줌아웃하며 조회해야 한다. 기존 MySQL에서는 차트 로딩에 10초가 넘게 걸림.
**판단 및 TSDB 아키텍처 전략**:
1. **InfluxDB 도입**: 쏟아지는 틱 데이터를 모두 InfluxDB에 타임스탬프와 가격, 거래량 구조로 쏟아 붓는다.
2. **다운샘플링(CQ) 자동화**: 백그라운드 태스크(Continuous Query)를 설정하여, 원본 1초 틱 데이터에서 '1분 평균/고가/저가(OHLC)', '1시간 평균', '1일 평균' 데이터를 백그라운드에서 실시간으로 미리 만들어 별도의 최적화된 테이블(Measurement)에 꽂아둔다.
3. **Retention 정책**: 1초 단위 초고해상도 원본 데이터는 7일이 지나면 자동 삭제(Drop)되도록 TTL(Time-to-Live)을 건다. 장기 데이터는 압축된 1일 봉 데이터만 남긴다.
4. **결과**: 차트 앱에서 1년 치 데이터를 불러올 때, 3천만 건의 데이터를 계산하는 대신 미리 압축된 365건의 데이터만 조회하므로 차트 로딩 속도가 0.1초 이내로 단축된다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **높은 카디널리티(High Cardinality)의 저주**: Prometheus나 InfluxDB를 쓸 때 태그(Label)에 유저 ID나 세션 ID처럼 **"무한대로 늘어나는 고유한 값"**을 넣으면 절대 안 된다. `Tag(Label)` 하나가 생길 때마다 TSDB 내부에서는 시계열 인덱스(Series) 파일이 하나씩 새로 생긴다. 천만 명의 유저 ID를 태그에 넣으면 천만 개의 인덱스 파일이 생성되어 메모리가 폭발(OOM)하고 DB가 즉각 사망한다. 태그에는 '지역(서울, 부산)', '서버 상태(200, 500)' 같은 제한된 종류(Enum)의 값만 넣어야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **디스크 절약 (TCO)**: RDBMS로 1TB를 차지하던 로그 데이터가 TSDB의 시계열 압축 알고리즘을 거치면 약 **100GB (1/10 수준)**로 줄어들어 스토리지 비용 대폭 절감.
- **초고속 집계 (Performance)**: 특정 시간 범위에 대한 Aggregation(Sum, Avg, P99 등) 연산 시, RDBMS 대비 수십 배~수백 배 빠른 응답성 확보.

### 미래 전망 및 진화 방향
클라우드 네이티브와 마이크로서비스의 등장으로 "우리 시스템이 지금 정상인가?"를 파악하는 **관측성(Observability)**이 시스템 아키텍처의 심장으로 떠올랐다. 이 심장을 뛰게 하는 피(Metrics)를 저장하는 곳이 바로 TSDB다. 최근에는 로그(Log), 메트릭(Metric), 분산 추적(Trace) 3대 요소를 융합하여 하나의 플랫폼에서 시간축을 넘나들며 원인 분석을 해주는 통합 Observability 백엔드(예: VictoriaMetrics, Grafana Mimir)로 TSDB 기술이 융합 및 거대화되고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **TSDB란?**: 1초마다 변하는 온도계의 온도를 끝없이, 가장 빠르게 받아 적는 '시간 전용 관찰 일기장'이에요.
2. **일반 공책(RDBMS)과 뭐가 달라요?**: 일반 공책은 옛날에 쓴 글씨를 지우개로 지우고 고치는 걸 잘하지만, 이 일기장은 한 번 쓴 건 절대 안 고치고 시간 순서대로 무조건 빨리 쓰기만 하는 데 세계 최고예요.
3. **가장 큰 장점은요?**: "작년 여름 온도의 평균이 어땠지?" 하고 물어보면, 수천 장의 일기를 순식간에 휘리릭 넘겨서 1초 만에 딱 대답해주는 엄청난 계산 능력을 가졌어요!
