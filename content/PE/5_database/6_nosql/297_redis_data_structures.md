# 297. 레디스 (Redis) 자료구조 - String, List, Set, Sorted Set, Hash

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 단순히 키-값(Key-Value)을 저장하는 메모리 저장소를 넘어, 애플리케이션 레벨에서 복잡하게 구현해야 했던 데이터 로직을 **O(1) 또는 O(log N)**의 성능으로 서버 단에서 원자적(Atomic)으로 처리해 주는 '데이터 구조 서버(Data Structure Server)'이다.
> 2. **가치**: 서비스의 성격에 맞는 최적의 자료구조를 선택함으로써, 랭킹 시스템(Sorted Set), 실시간 채팅 큐(List), 중복 없는 방문자 카운팅(Set) 등을 최소한의 CPU 비용으로 구현할 수 있게 해 준다.
> 3. **융합**: 각 자료구조는 내부적으로 메모리 효율을 위해 **ZipList, SkipList, IntSet** 등 특수한 인코딩 기술과 융합되어 있으며, 비즈니스 요구사항에 따라 영속성(RDB/AOF) 옵션과 결합되어 최적의 성능을 발휘한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
Redis(Remote Dictionary Server)는 메모리 기반의 NoSQL 데이터베이스로, 다양한 추상 자료구조를 기본적으로 제공한다. 모든 연산은 메모리 내에서 수행되므로 매우 빠르며, 싱글 스레드(Single-threaded) 기반의 이벤트 루프를 통해 경쟁 상태(Race Condition) 없는 안전한 데이터 처리를 보장한다.

### 💡 비유
Redis 자료구조는 **"맥가이버 칼(Swiss Army Knife)"**과 같다.
- **일반 DB (창고)**: 물건을 넣고 빼는 기능만 있다. 복잡한 가공은 내가 직접 도구를 가져와서 해야 한다.
- **Redis (맥가이버 칼)**: 단순히 칼(String)만 있는 게 아니라, 가위(List), 핀셋(Hash), 드라이버(Sorted Set)가 한 몸에 다 붙어 있다. 내가 굳이 도구를 만들 필요 없이, 용도에 맞는 도구를 펴서(자료구조 선택) 현장에서 즉시 문제를 해결할 수 있다.

### 등장 배경 및 발전 과정

#### 1. "Simple is not always better"
초기 캐시 시스템(Memcached)은 오직 '문자열'만 저장할 수 있었다. 만약 리스트 형태의 데이터를 수정하려면 전체를 가져와서(GET), 코드에서 수정하고, 다시 통째로 저장(SET)해야 했다. 이 과정에서 네트워크 대역폭 낭비와 데이터 유실 위험이 컸다.

#### 2. 살바토레 산필리포(antirez)의 혁신
2009년, Redis의 창시자는 "데이터베이스가 자료구조를 직접 이해하고 부분 수정(Push, Pop, Increment)을 지원해야 한다"는 철학으로 Redis를 세상에 내놓았다. 이 5대 핵심 자료구조는 현대 웹 서비스 아키텍처의 필수 요소가 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 5대 핵심 자료구조 상세 분석 (표)

| 자료구조 | 특징 | 주요 명령어 | 시간 복잡도 | 실제 활용 사례 |
|----------|------|-------------|-------------|----------------|
| **String** | 가장 기본. 최대 512MB 바이너리 저장 가능 | SET, GET, INCR | O(1) | 세션 캐싱, HTML 페이지 캐시, 카운터 |
| **List** | Linked List 구현. 양 끝에서 삽입/삭제 빠름 | LPUSH, RPOP, Lrange | O(1) (양 끝) | **메시지 큐(MQ)**, 최신 뉴스 피드 10개 |
| **Set** | 중복 없는 무순서 집합. 집합 연산(교/합/차) 지원 | SADD, SISMEMBER | O(1) | 좋아요 누른 유저 목록, 태그 시스템 |
| **Sorted Set (Zset)**| Set + Score(가중치). 점수 기준 자동 정렬 유지 | ZADD, ZRANGE | **O(log N)** | **게임 랭킹**, 실시간 급상승 검색어 |
| **Hash** | Field-Value 쌍의 묶음. 객체 저장에 최적 | HSET, HGET | O(1) | 유저 프로필(이름, 나이, 이메일) 정보 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ Redis Memory Layout & Internal Encodings ]
================================================================================

 [ Client ] ───( Command: ZADD myrank 100 "PF" )───▶ [ Redis Server ]
                                                         │
                                                         ▼
 ┌────────────────────────────────────────────────────────────────────────────┐
 │  Key: "myrank"  (Redis Object)                                             │
 │  Type: Sorted Set                                                          │
 │  Encoding: ⭐️ SkipList + Hash Map ⭐️                                        │
 └───────────────────────┬────────────────────────────────────────────────────┘
                         │
         ┌───────────────▼──────────────────────────┐
         │ [ Index Layer: SkipList ]                │
         │  (O(log N) 정렬 검색)                     │
         │  Level 3: [10] -----------------> [90]   │
         │  Level 2: [10] ------> [50] ----> [90]   │
         │  Level 1: [10]->[20]->[50]->[70]->[90]   │
         └───────────────┬──────────────────────────┘
                         │ (Pointer)
         ┌───────────────▼──────────────────────────┐
         │ [ Data Layer: Dict / Hash Table ]        │
         │  (O(1) 데이터 값 직접 접근)                │
         │  "PF"  => Score: 100                     │
         │  "Bob" => Score: 50                      │
         └──────────────────────────────────────────┘
```

### 심층 동작 원리

#### ① Sorted Set의 심장: SkipList (스키프 리스트)
Sorted Set은 데이터가 들어올 때마다 실시간으로 정렬을 유지해야 한다. RDBMS의 B-Tree보다 메모리 친화적이고 구현이 단순한 **SkipList** 자료구조를 쓴다. 여러 층의 인덱스(Express Lane)를 두어, 수천만 명의 랭킹 데이터 사이를 성큼성큼 건너뛰며 순위를 찾아낸다.

#### ② 메모리 극한의 절약: ZipList (압축 리스트)
Redis는 메모리가 비싸다는 것을 잘 안다. 그래서 List나 Hash에 데이터가 몇 개 없을 때는(예: 512개 이하), 포인터 연결 방식인 Linked List를 쓰지 않는다. 대신 **ZipList**라는 '연속된 메모리 덩어리'에 데이터를 다닥다닥 붙여서 저장한다. CPU 연산은 조금 더 들지만, 포인터(8바이트)가 차지하는 엄청난 메모리 낭비를 줄여주는 영리한 최적화다.

## Ⅲ. 융합 비교 및 다각도 분석

### 자료구조 선택 결정 매트릭스

| 고민 중인 문제 | 추천 자료구조 | 이유 |
|----------------|---------------|------|
| "전체 유저 100만 명 중 상위 10명을 실시간으로 보여주고 싶다" | **Sorted Set** | Score 기반 자동 정렬 및 범위 조회 성능이 가장 뛰어남 |
| "A 유저와 B 유저가 공통으로 아는 친구 목록을 뽑고 싶다" | **Set** | `SINTER` (교집합) 명령어로 서버 단에서 즉시 결과 도출 |
| "주문 데이터의 상태값을 필드별로 자주 수정해야 한다" | **Hash** | 객체 전체를 직렬화할 필요 없이 특정 필드만 `HINCRBY` 가능 |
| "작업을 순서대로 쌓아두고 하나씩 꺼내 처리하고 싶다" | **List** | `LPUSH` / `BRPOP` 조합으로 완벽한 FIFO 큐 구현 가능 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: "실시간 동시 접속자 수 100만 명"의 초대형 모바일 게임의 리더보드(랭킹) 시스템을 설계해야 한다. 1초마다 수천 명의 점수가 바뀌고, 유저는 자신의 현재 등수를 즉시 확인하고 싶어 한다.
**판단 및 자료구조 설계 전략**:
1. **Sorted Set (Zset) 채택**: `ZADD`로 점수를 업데이트하고, `ZREVRANK`로 내 등수를 0.001초 만에 가져온다.
2. **Key 설계 (Shard)**: 한 개의 Key에 100만 명을 다 넣으면 특정 Redis 노드에 부하(Hotspot)가 쏠릴 수 있다. 국가별(KR, US 등) 또는 점수 구간별로 Key를 쪼개는 **Shard** 전략을 검토한다.
3. **결과**: RDBMS로 구현했다면 매번 `ORDER BY` 쿼리로 DB가 비명을 질렀겠지만, Redis는 메모리 상에서 이미 정렬된 SkipList를 훓기만 하면 되므로 초당 수만 건의 랭킹 조회를 가뿐히 소화한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **거대 컬렉션 (Big Key) 주의**: 하나의 Set이나 List에 수백만 개의 데이터를 넣고 `SMEMBERS`나 `HGETALL` 같은 명령어를 날리는 것은 **'금기'**다. Redis는 싱글 스레드이므로, 이 거대한 데이터를 읽어오느라 1초 동안 멈춰버리면 그동안 들어오는 다른 모든 요청이 타임아웃되어 전체 서비스가 마비된다. 큰 데이터는 반드시 `SCAN` 계층 명령어로 쪼개서 읽거나, 데이터를 여러 키로 분산해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **연산 속도 가속**: 애플리케이션 단의 정렬/필터링 로직을 Redis 내부 연산으로 오프로딩하여 서버 CPU 부하 **40% 이상 절감**.
- **네트워크 효율**: 필요한 필드만 수정(HSET)하거나 증감(INCR)하여 불필요한 전체 데이터 전송(Payload) 최소화.

### 미래 전망 및 진화 방향
Redis는 최근 **Redis Stack**으로 진화하며 JSON 문서를 직접 다루는 `RedisJSON`, 벡터 검색을 지원하는 `RedisSearch`, 시계열 데이터를 처리하는 `RedisTimeSeries` 등 자료구조의 지평을 무한히 넓히고 있다. 이제 Redis는 단순한 보조 캐시가 아니라, 특정 도메인(AI, IoT, 실시간 분석)에서는 메인 데이터베이스의 자리를 위협하는 강력한 **'멀티 모델 데이터 저장소'**로 거듭나고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **레디스 자료구조란?**: 컴퓨터 안에 있는 아주 똑똑하고 빠른 '정리함'이에요. 그냥 물건을 막 던져넣는 게 아니라, 물건 종류에 따라 전용 보관함이 따로 준비되어 있어요.
2. **어떤 보관함이 있나요?**: 등수를 자동으로 매겨주는 '계단 보관함(Sorted Set)', 중복된 걸 골라내는 '마법 주머니(Set)', 순서대로 줄을 세우는 '기차 칸(List)' 등이 있어요.
3. **왜 좋은가요?**: 내가 직접 정리할 필요 없이, 레디스한테 "이거 등수 좀 매겨줘!"라고 던져주기만 하면 눈 깜짝할 사이에 정리를 끝내고 대답해 주기 때문이랍니다!
