# 298. 몽고DB (MongoDB) 아키텍처 - 레플리카 셋 및 샤드 클러스터

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: JSON과 유사한 유연한 **BSON(Binary JSON) 문서 모델**을 기반으로 하며, 고가용성을 보장하는 레플리카 셋(Replica Set)과 무한 확장을 지원하는 샤드 클러스터(Sharded Cluster)를 통해 대규모 데이터를 처리하는 대표적인 도큐먼트 지향 NoSQL이다.
> 2. **가치**: 스키마리스(Schema-less) 특성 덕분에 요구사항이 시시각각 변하는 애자일 개발 환경에 최적이며, 자동 페일오버(Failover)와 수평적 확장(Horizontal Scaling) 기능을 하드웨어 레벨이 아닌 DB 엔진 레벨에서 완벽하게 제공한다.
> 3. **융합**: 복잡한 트리를 형성하는 임베디드 도큐먼트 구조, 분산 노드 간의 데이터 정합성을 맞추는 롸이트 컨선(Write Concern) 설정, 그리고 지능형 쿼리 라우터인 **mongos** 기술이 융합되어 거대 플랫폼의 핵심 저장소 역할을 수행한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
MongoDB는 데이터를 '행(Row)'이 아닌 '문서(Document)' 단위로 저장한다. 
- **Replica Set**: 데이터 복제본을 여러 서버에 두어 한 대가 죽어도 서비스가 유지되게 하는 고가용성(HA) 구성.
- **Sharded Cluster**: 데이터를 여러 대의 서버(Shard)에 쪼개서 저장하여 성능과 용량을 확장하는 분산 구성.

### 💡 비유
MongoDB 아키텍처는 **"대형 프랜차이즈 도서관 운영 방식"**과 같다.
- **도큐먼트 (책)**: 책 한 권에 제목, 저자, 내용, 댓글까지 다 들어있다. (JOIN이 필요 없음).
- **레플리카 셋 (복사본)**: 똑같은 책을 3권 복사해서 3명의 사서에게 준다. 1번 사서가 아프면 2번 사서가 즉시 책을 빌려준다.
- **샤드 클러스터 (분점)**: 책이 너무 많아지면 'ㄱ~ㄹ'은 강남점, 'ㅁ~ㅅ'은 강북점으로 나눠서 보관한다. 손님은 어디에 책이 있는지 몰라도 된다. 입구에 있는 '안내 로봇(mongos)'에게 물어보면 알아서 해당 지점으로 연결해 준다.

### 등장 배경 및 발전 과정

#### 1. RDBMS의 고정 스키마(Schema) 지옥
전통적인 DB는 테이블 구조를 바꾸려면 서비스를 멈추거나 복잡한 이관 작업을 해야 했다. SNS나 로그 데이터처럼 필드가 제멋대로인 데이터를 담기엔 너무 뻣뻣했다.

#### 2. "Humongous (엄청나게 큰)" 데이터의 시대
MongoDB라는 이름은 'Humongous'에서 왔다. 2009년 10gen(현 MongoDB Inc.)은 데이터가 폭발적으로 늘어나는 클라우드 시대에 맞춰, 개발자가 코딩하는 객체 그대로 DB에 꽂아 넣고, 서버를 늘리면 용량이 자동으로 늘어나는 DB를 목표로 탄생했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 샤드 클러스터의 3대 핵심 구성 요소 (표)

| 요소명 | 상세 역할 | 작동 메커니즘 | 비유 |
|--------|----------|-------------------|------|
| **Shard** | 실제 데이터가 저장되는 구역 | 데이터의 하위 집합을 보유하며, 보통 고가용성을 위해 **Replica Set**으로 구성됨 | 각 구의 분점 도서관 |
| **Config Server** | 클러스터의 메타데이터 관리 | 어떤 데이터(Shard Key 범위)가 어느 Shard에 있는지 지도를 보관함 | 전체 도서 목록 장부 |
| **mongos** | 쿼리 라우터 (인터페이스) | 클라이언트의 요청을 받아 Config Server를 참조하여 적절한 Shard로 전달 | 입구의 안내 데스크 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii

                         ▼
             ┌────────────────────────┐
             │    ⭐️ mongos ⭐️        │ (Query Router)
             │ (Stateless, Load Bal)  │
             └─────┬──────────────┬───┘
                   │              │ (3) 쿼리 결과 병합 후 반환
 (1) "어디에 있지?" │              │

        ┌──────────────────┐    ┌──────────────────────────────────────────────┐
```

### 심층 동작 원리

#### ① 레플리카 셋의 합의(Consensus)와 페일오버
레플리카 셋의 노드들은 서로 **하트비트(Heartbeat)**를 주고받으며 생사 확인을 한다. 만약 리더(Primary)가 죽으면, 남은 팔로워(Secondary)들이 투표를 실시한다. 가장 최신 데이터를 가진 노드가 새로운 리더가 되며, 이 과정은 **보통 10초 이내**에 자동으로 끝난다. (고가용성 보장).

#### ② 샤딩과 밸런싱 (Balancing)
데이터를 쪼개는 기준을 **샤드 키(Shard Key)**라고 한다. 특정 샤드에 데이터가 너무 몰리면(Hotspot), MongoDB 내부의 **Balancer** 프로세스가 백그라운드에서 조용히 데이터를 다른 한가한 샤드로 옮긴다(Chunk Migration). 서비스 중단 없이 데이터 분포를 평평하게 맞추는 지능형 확장 기술이다.

#### ③ 인메모리 스토리지 엔진 (WiredTiger)
MongoDB 3.2부터 기본 탑재된 WiredTiger 엔진은 도큐먼트 레벨의 락(Document-level Locking)을 지원한다. 과거 테이블 전체에 락을 걸던 방식과 달리, 동시에 수만 명이 다른 문서를 수정해도 병목이 거의 발생하지 않는다.

## Ⅲ. 융합 비교 및 다각도 분석

### 관계형 DB(RDBMS) vs MongoDB

| 비교 지표 | RDBMS (MySQL, Oracle) | MongoDB (NoSQL) |
|-----------|------------------------|-----------------|
| **데이터 모델** | 테이블 (행/열) | **문서 (BSON/JSON)** |
| **스키마** | 엄격함 (변경 시 DDL 필요) | **유연함 (동적 스키마)** |
| **확장 방식** | 수직 확장 (Scale-up) 위주 | **수평 확장 (Scale-out) 기본** |
| **조인 (Join)** | 강력함 | **비권장 (Embedded Document로 해결)** |
| **트랜잭션** | 완벽한 ACID | 4.0부터 다중 문서 ACID 지원 (성능 페널티 있음) |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 급성장 중인 배달 앱의 '리뷰 시스템'. 유저가 사진, 별점, 댓글을 남기는데 나중에 기획팀이 "리뷰에 '도움돼요' 버튼과 '사장님 답글' 기능을 추가해달라"고 한다. RDBMS라면 리뷰 테이블, 답글 테이블을 쪼개고 JOIN 하느라 고생했을 것이다. 
**판단 및 MongoDB 설계 전략**:
1. **임베디드 모델(Embedded) 채택**: 리뷰 문서 하나 안에 `replies: [...]` 배열을 넣어 답글을 통째로 저장한다. JOIN 없이 문서 1개만 읽으면 화면이 바로 그려진다 (조회 성능 극대화).
2. **레플리카 셋 구성 (3노드)**: 결제만큼 중요하진 않지만 장애는 없어야 하므로, 최소 3개 노드의 레플리카 셋을 구성하여 데이터 유실을 방지한다.
3. **결과**: 기획 변경이 와도 DB 스키마 변경 없이 코드에서 필드만 추가하면 끝이다. 배달 앱 특유의 잦은 UI 변경에 완벽히 대응하는 유연한 시스템이 완성된다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **무분별한 샤드 키 선택**: 샤드 키를 '날짜'나 'ID'처럼 순차적으로 증가하는 값으로 잡으면, 모든 데이터가 항상 '최신 샤드'로만 쏠린다(Single Shard Write). 이럴 경우 수십 대의 서버를 사놔도 한 대만 일하고 나머지는 노는 비효율이 발생한다. 반드시 **Cardinality가 높고 분포가 고른 필드**를 샤드 키로 골라야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **개발 생산성**: 데이터 매핑 코드(ORM) 오버헤드 감소 및 스키마 변경 자유도로 개발 속도 **30% 이상 향상**.
- **운영 안정성**: 자동 페일오버 기능을 통해 인적 개입 없는 99.99% 이상의 서비스 가용성 유지.

### 미래 전망 및 진화 방향
MongoDB는 이제 단순한 DB를 넘어 **MongoDB Atlas**라는 거대 클라우드 데이터 플랫폼으로 진화했다. 서버리스(Serverless) DB, 전문 검색(Search), 모바일 동기화(Realm), 데이터 시각화(Charts) 기능을 통합 제공한다. 특히 최근에는 인공지능(AI) 열풍에 맞춰 벡터 검색(Vector Search) 기능을 추가하며, LLM(거대 언어 모델)의 지식 창고인 **벡터 데이터베이스** 시장의 강력한 강자로 자리매김하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **도큐먼트 DB란?**: 정보를 보관할 때, 딱딱한 표에 채워 넣는 대신 예쁜 포스트잇(문서)에 써서 봉투에 담아 보관하는 방식이에요.
2. **레플리카 셋은요?**: 내가 쓴 일기장을 동생 두 명한테도 똑같이 베껴 쓰게 해서, 내 일기장을 잃어버려도 동생들 걸 보면 되니까 안심할 수 있는 거예요.
3. **샤딩은 뭐예요?**: 일기장이 1,000권이 넘어서 내 방 책꽂이에 다 안 들어가면, 친구들 방 책꽂이에 조금씩 나눠서 꽂아두는 '나눠서 보관하기' 비법이랍니다!
