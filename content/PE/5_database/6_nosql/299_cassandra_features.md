# 299. 카산드라 (Cassandra) 특징 - 가십 프로토콜 및 튜너블 컨시스턴시

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 구글 Bigtable의 데이터 모델과 아마존 Dynamo의 분산 설계(P2P 링 구조)를 결합하여, 단일 장애점(SPOF)이 없는 완벽한 **마스터리스(Masterless)** 아키텍처를 지향하는 고가용성 분산 NoSQL 데이터베이스이다.
> 2. **가치**: 노드 간 상태 정보를 주고받는 **가십 프로토콜(Gossip Protocol)**을 통해 클러스터 자가 치유를 실현하며, 읽기/쓰기 시 응답 노드 수를 조절하여 '속도'와 '일관성' 사이의 균형을 직접 결정하는 **튜너블 컨시스턴시(Tunable Consistency)** 기능을 제공한다.
> 3. **융합**: 시계열 데이터 처리에 최적화된 LSM-Tree 기반 저장 엔진, 데이터 위치를 결정하는 일관된 해싱(Consistent Hashing), 그리고 장애 상황에서도 쓰기를 수용하는 힌티드 핸드오프(Hinted Handoff) 기술이 융합되어 전 세계 규모의 대형 서비스를 지탱한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
Apache Cassandra는 페이스북에서 Inbox 검색 문제를 해결하기 위해 처음 개발된 컬럼 패밀리(Column-family) 기반의 분산 DB다. 중앙 관리자(Master)가 없는 대칭적 구조로, 모든 노드가 동일한 역할을 수행하여 수천 대 규모의 서버 확장이 가능하다.

### 💡 비유
카산드라 클러스터는 **"왕이 없는 원탁의 기사단"**과 같다.
- **기존 방식 (Master-Slave)**: 왕(Master) 한 명에게 모든 보고가 올라가고, 왕이 죽으면 나라가 멈춘다. (Single Point of Failure).
- **카산드라**: 모든 기사(Node)가 평등하게 원탁에 둘러앉아 있다. 기사들은 옆 사람과 쑥덕거리며(Gossip) 나라 상황을 공유한다. 기사 한두 명이 전쟁터에서 전사해도, 나머지 기사들이 그 업무를 나눠 맡아 나라는 멀쩡하게 돌아간다.

### 등장 배경 및 발전 과정

#### 1. 대규모 쓰기 부하와 가용성 요구
페이스북은 초당 수백만 건의 메시지 알림을 저장해야 했다. 기존 RDBMS는 쓰기 작업 시 락(Lock) 때문에 확장이 불가능했다. 또한 "결제는 아니니까 0.1초 정도 데이터가 안 맞아도 좋으니, 서비스는 절대 죽지 말아야 한다"는 가용성 우선 원칙이 필요했다.

#### 2. 아마존 Dynamo와 구글 Bigtable의 만남
Cassandra는 아마존 Dynamo의 **'링 기반 분산 아키텍처'**와 구글 Bigtable의 **'LSM-Tree 기반 고성능 쓰기 모델'**을 섞어서 2008년 탄생했다. 이후 넷플릭스, 애플 등 초거대 IT 기업의 메인 저장소로 쓰이며 분산 DB의 교과서가 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 카산드라 분산 아키텍처의 3대 요소 (표)

| 요소명 | 상세 역할 | 작동 메커니즘 | 관련 기술 |
|--------|----------|-------------------|-----------|
| **Gossip Protocol** | 노드 간 상태 공유 | 노드들이 1초마다 무작위로 다른 노드와 정보를 교환하여 클러스터 지도를 완성 | P2P 통신, 하트비트 |
| **Consistent Hashing**| 데이터 분산 배치 | 데이터를 고유한 Hash 값(Token)으로 변환하여 링(Ring) 위의 특정 노드에 할당 | Partition Key, Token Ring |
| **Quorum Consistency**| 일관성 수준 조절 | 쓰기/읽기 시 최소 몇 명의 승인이 필요한지 설정하여 정합성 보장 수준을 튜닝 | W+R > N 법칙 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ Cassandra Masterless Ring Architecture & Data Replication ]
================================================================================

           (Token: 0)
          ┌── Node A ──┐
          │ (Replica)  │
 (Token:  │            │ (Token:
   75)  Node D        Node B    25)
          │            │
          │            │
          └── Node C ──┘
           (Token: 50)

 [ Data Write Flow ]
 (1) Client가 아무 노드(Node B)에나 데이터를 던짐. (Node B가 Coordinator 역할 수행)
 (2) Node B: "이 데이터의 Hash 값은 30이군! 주인은 Node C야."
 (3) Node B가 진짜 주인인 Node C와, 복제본을 가질 Node D에게 동시에 패킷 전송.
 (4) 설정된 Quorum(예: ANY or ONE)만큼 응답 오면 클라이언트에게 성공 반환.

 * Gossip: "A야, 나 방금 C가 죽은 걸 확인했어. 너도 알고 있어!" (0.1초 만에 전파)
```

### 심층 동작 원리

#### ① 가십 프로토콜 (Gossip Protocol)
중앙 서버 없이도 수천 대의 상태를 파악하는 비법이다. 각 노드는 자신이 아는 모든 노드의 상태(Seed 정보)를 리스트로 가지고 있다. 매초 마다 무작정 3대의 노드를 골라 내 리스트와 비교하고 업데이트한다. 이 전염병(?) 같은 정보 전파 속도는 매우 빨라서, 수천 대 클러스터에서 한 대가 죽었을 때 1~2초 내에 모든 노드가 이를 인지하게 된다.

#### ② 튜너블 컨시스턴시 (Tunable Consistency)
- **ANY / ONE**: 1명만 대답해도 성공. 속도는 빛처럼 빠르지만 데이터 유실 위험 있음.
- **QUORUM**: 과반수(N/2 + 1)가 응답해야 성공. 속도와 정합성의 가장 훌륭한 타협점.
- **ALL**: 모든 복제본 노드가 응답해야 성공. 정합성은 RDBMS 급이지만 노드 하나만 아파도 시스템이 멈춤.
개발자는 쿼리 한 줄마다 이 수준을 다르게 줄 수 있다. (예: 결제는 QUORUM, 좋아요는 ONE).

#### ③ 힌티드 핸드오프 (Hinted Handoff)
데이터를 받아야 할 노드 C가 잠시 점검 중이라면, 옆 노드 B가 대신 데이터를 받아 '쪽지(Hint)'를 써둔다. 나중에 C가 살아나면 B가 쪽지를 건네주어 데이터를 동기화한다. 장애 상황에서도 쓰기를 거절하지 않는 카산드라의 무서운 가용성 철학이다.

## Ⅲ. 융합 비교 및 다각도 분석

### RDBMS vs MongoDB vs Cassandra

| 비교 지표 | MySQL (Cluster) | MongoDB | **Cassandra** |
|-----------|------------------|---------|---------------|
| **구조** | Master-Slave | Master-Slave (Replica Set) | **P2P (Masterless)** |
| **확장성** | 보통 (수직적) | 우수 (Sharding) | **최상 (Linear Scaling)** |
| **쓰기 성능**| 보통 | 우수 | **극상 (Append-only LSM)** |
| **조인(Join)**| 매우 강력 | 불가능 | **절대 불가능** |
| **데이터 모델**| Table | Document | Column-family |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 글로벌 메신저 서비스의 '메시지 전송 이력' DB를 구축해야 한다. 초당 쓰기 트래픽이 50만 건 이상이며, 전 세계 3개 대륙(미국, 유럽, 아시아) 데이터센터에 데이터가 실시간으로 공유되어야 한다. 한 대륙의 해저 케이블이 끊겨도 서비스는 유지되어야 한다.
**판단 및 카산드라 도입 전략**:
1. **Multi-Region Cluster 구성**: 카산드라는 노드 간 물리적 거리를 인식하는 **NetworkTopologyStrategy**를 제공한다. 대륙별로 복제본을 배치한다.
2. **Local Quorum 설정**: 쓰기 시 "우리 대륙(Local)에서만 과반수 응답 오면 오케이"라고 설정한다. 대륙 간 통신 지연(RTT) 때문에 전 세계 응답을 기다리면 메신저가 너무 느려지기 때문이다.
3. **결과**: 한쪽 대륙이 고립되어도 나머지 대륙은 중단 없이 메시지를 주고받는다. 나중에 망이 복구되면 카산드라가 백그라운드에서 조용히 대륙 간 데이터를 맞춘다(Anti-Entropy Repair).

### 주의사항 및 안티패턴 (Anti-patterns)
- **Secondary Index 남용**: 카산드라는 Partition Key로 데이터를 찾을 때만 빠르다. 일반 필드에 인덱스를 걸고 검색하면 클러스터 전체 노드를 다 뒤져야 하므로(Scattering), 성능이 1/1000로 떨어진다. 카산드라는 **"쿼리에 맞춰서 테이블을 설계(Query-first Design)"** 해야 하며, 검색이 필요하면 Elasticsearch를 연동하는 것이 정석이다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **선형적 확장(Linear Scalability)**: 서버를 2배 늘리면 처리량도 정확히 2배 늘어나는 기적의 경제성 달성.
- **Zero-Downtime**: 노드 업그레이드나 교체 시에도 서비스 중단 없이 운영 가능.

### 미래 전망 및 진화 방향
카산드라는 최근 **Cassandra 4.0/5.0**으로 진화하며 그동안 약점으로 지적받던 복잡한 동기화 지연을 해결하기 위해 Paxos v2 알고리즘을 도입했다. 또한 클라우드 네이티브 환경에 맞춰 **Astra DB(서버리스)** 형태로 제공되며, K8s 위에서 자동으로 운영되는 오퍼레이터 생태계가 완성되었다. 데이터가 너무 커서 RDBMS가 감당 안 되는 모든 영역에서 카산드라는 영원한 '최후의 보루'가 될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **카산드라가 뭔가요?**: 대장(Master)이 따로 없는 사이좋은 로봇 친구들 모임이에요. 친구 한 명이 고장 나서 고치러 가도, 나머지 친구들이 그 일을 대신해주기 때문에 모임은 절대 멈추지 않아요.
2. **가십 프로토콜은요?**: 로봇들이 서로 만나면 "얘들아, 3번 로봇 지금 배터리 충전하러 갔대!"라고 소문을 내서, 1초 만에 모든 로봇이 상황을 알게 되는 신기한 소문내기 방법이에요.
3. **가장 좋은 점은 뭐예요?**: 데이터가 많아지면 로봇 친구들을 더 데려오기만 하면 돼요. 로봇이 많아질수록 일 처리 속도가 똑같이 빨라지는 아주 힘센 모임이랍니다!
