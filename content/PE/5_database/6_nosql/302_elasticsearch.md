# 302. 엘라스틱서치 (Elasticsearch) - 루씬 기반 텍스트 분석 DB

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 아파치 루씬(Apache Lucene) 라이브러리를 기반으로 구축된 분산형 RESTful 검색 및 분석 엔진으로, 데이터를 **역색인(Inverted Index)** 구조로 저장하여 대규모 비정형 데이터에서도 밀리초(ms) 단위의 전문(Full-text) 검색 성능을 제공한다.
> 2. **가치**: 단순 검색을 넘어 로그 분석(ELK 스택), 실시간 지표 모니터링, 그리고 고도의 텍스트 정제(Tokenizer/Analyzer) 기능을 통해 기업의 방대한 데이터를 지식화하는 핵심 플랫폼 역할을 수행한다.
> 3. **융합**: Logstash(수집), Kibana(시각화)와 융합된 **ELK 스택**은 전 세계 표준 옵저버빌리티(Observability) 도구이며, 최근에는 벡터 검색 기술과 결합하여 AI 검색 엔진으로 진화하고 있다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
Elasticsearch는 JSON 문서 형태로 데이터를 저장하며, 수평적 확장이 용이한 분산 검색 엔진이다. 관계형 DB가 데이터의 '저장'과 '무결성'에 집중한다면, Elasticsearch는 데이터의 **'검색'**과 **'유사도 분석'**에 특화되어 있다.

### 💡 비유
Elasticsearch는 **"도서관의 인덱스 카드함과 전문 사서"**와 같다.
- **RDBMS (서가 뒤지기)**: "사과"라는 단어가 들어간 책을 찾으려면 1번 서가부터 100만 번 서가까지 모든 책의 첫 페이지부터 끝까지 다 읽어봐야 한다. (Full Scan).
- **Elasticsearch (인덱스 카드)**: 사서가 미리 "사과"라는 단어가 10번 책, 500번 책, 9000번 책에 있다고 적힌 카드(역색인)를 만들어두었다. 질문을 던지자마자 사서가 카드를 쓱 보고 "여기 여기 여기 있습니다!"라고 0.1초 만에 대답해 준다.

### 등장 배경 및 발전 과정

#### 1. 샤이 배논(Shay Banon)의 Compass와 Lucene
창시자 샤이 배논은 아내를 위한 요리법 검색 앱을 만들려다 루씬의 복잡함에 좌절하고, 이를 쓰기 쉽게 감싼 Elasticsearch를 2010년 발표했다.

#### 2. 빅데이터 로그 분석의 혁명
기업들이 서버 로그를 모으기 시작하면서, 텍스트 형태의 로그에서 장애 징후를 빠르게 찾는 것이 중요해졌다. Elasticsearch는 분산 처리 능력을 앞세워 Splunk 같은 유료 솔루션을 대체하며 오픈소스 검색 시장을 평정했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 엘라스틱서치 핵심 논리 구조 (표)

| 개념 | RDBMS 대응 개념 | 특징 | 비유 |
|------|-----------------|------|------|
| **Index** | Database / Table | 유사한 특성을 가진 문서들의 집합 | 도서관의 한 구역 (예: 과학) |
| **Document** | Row (행) | 검색의 최소 단위, JSON 포맷 | 책 한 권 |
| **Field** | Column (열) | 문서에 포함된 데이터의 키-값 쌍 | 책의 제목, 저자, 출판일 |
| **Mapping** | Schema | 데이터 타입과 분석 방법을 정의 | 도서 분류 규칙 |

### 정교한 구조 다이어그램 (분산 클러스터 구조)

```ascii
================================================================================
[ Elasticsearch Distributed Cluster & Sharding ]
================================================================================

  [ 🌐 Client (REST API / Kibana) ]
           │ (HTTP 9200)
           ▼
  ┌────────────────────────────────────────────────────────────────────────┐
  │  ⭐️ Elasticsearch Cluster ⭐️                                           │
  │                                                                        │
  │  ┌──────────────────┐      ┌──────────────────┐      ┌──────────────────┐  │
  │  │  Node 1 (Master) │      │  Node 2 (Data)   │      │  Node 3 (Data)   │  │
  │  │                  │      │                  │      │                  │  │
  │  │  [Idx A - P0] ⭐️ │      │  [Idx A - P1] ⭐️ │      │  [Idx A - R0] ☁️ │  │
  │  │  [Idx A - R1] ☁️ │      │  [Idx B - P0] ⭐️ │      │  [Idx B - R0] ☁️ │  │
  │  └──────────────────┘      └──────────────────┘      └──────────────────┘  │
  └────────────────────────────────────────────────────────────────────────┘

 * Shard: 데이터를 쪼갠 단위. (P: Primary, R: Replica)
 * 분산 원리: 인덱스 하나를 여러 샤드로 쪼개 노드들에 골고루 배분.
   - Node 2가 죽어도 Node 3에 복제본(R0)이 있어 서비스는 무중단 유지됨.
```

### 심층 동작 원리

#### ① 역색인 (Inverted Index)
텍스트를 단어(Token) 단위로 쪼개고, 각 단어가 어느 문서에 들어있는지를 매핑한 표다. 
- 문서 1: "PF likes database"
- 문서 2: "Redis is a database"
- **역색인**: `database` -> [문서 1, 문서 2], `PF` -> [문서 1], `Redis` -> [문서 2]
검색어 `database`가 들어오면 즉시 [1, 2]를 반환한다.

#### ② 텍스트 분석 (Analysis) 파이프라인
데이터가 들어올 때 다음 과정을 거친다.
1. **Character Filter**: HTML 태그 제거나 특수문자 변환.
2. **Tokenizer**: 문장을 단어 단위로 쪼갬 (공백 기준 등).
3. **Token Filter**: 대문자를 소문자로(Lowercasing), 불용어 제거(the, a), 어근 추출(Stemming).
이 과정을 통해 "Database"를 검색해도 "databases"가 들어간 문서를 찾아낼 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### RDBMS vs Elasticsearch

| 비교 지표 | RDBMS (MySQL 등) | Elasticsearch |
|-----------|-------------------|---------------|
| **검색 방식** | Like '%검색어%' (매우 느림) | **Inverted Index (매우 빠름)** |
| **데이터 형태**| 정규화된 정형 데이터 | **비정형/반정형 JSON** |
| **트랜잭션** | 강력함 (ACID) | 약함 (Near Real-time) |
| **언어/접속** | SQL / JDBC | **Restful API / JSON** |
| **유사도** | 없음 (일치 여부만 판단) | **Score(유사도 점수) 기반 정렬** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대형 쇼핑몰의 상품 검색 기능. 사용자가 "빨간색 원피스"라고 검색했는데, DB의 Like 쿼리가 너무 느려 결과가 3초 뒤에 나온다. 또한 "원피쓰"라고 오타를 치면 결과가 아예 안 나온다.
**판단 및 Elasticsearch 도입 전략**:
1. **역색인 기반 검색**: 상품 명칭과 설명을 Elasticsearch 인덱스로 마이그레이션한다.
2. **N-gram / Fuzzy Search**: 오타 대응을 위해 `Fuzzy Query`를 적용하고, 한국어 형태소 분석기(**Nori**)를 탑재한다.
3. **유사도 랭킹**: 판매량과 클릭수가 높은 상품에 가중치(Boost)를 주어 상단에 노출시킨다.
4. **결과**: 검색 속도가 **0.1초**로 단축되고, 오타를 쳐도 정확한 상품을 추천하여 구매 전환율을 20% 이상 향상시킨다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **주 데이터베이스(Primary DB)로 사용**: Elasticsearch는 트랜잭션과 롤백 기능이 취약하다. 또한 인덱싱 오버헤드 때문에 업데이트가 잦은 원장 데이터 관리에는 부적합하다. 반드시 **RDBMS를 원천(Source)으로 두고, 검색용으로만 복제**해서 쓰는 패턴을 유지해야 한다.
- **샤드(Shard) 배치 실수**: 샤드를 너무 작게 쪼개면 관리 오버헤드로 성능이 떨어지고, 너무 크게 잡으면 노드 복구가 힘들다. 보통 하나의 샤드 크기는 **20GB ~ 50GB** 정도로 유지하는 것이 베스트 프랙티스다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **검색 성능 향상**: 1억 건 이상의 데이터에서 전문 검색 시 RDBMS 대비 **100배 이상** 속도 향상.
- **분석 가시성**: Kibana를 통해 실시간 로그 데이터를 대시보드화하여 장애 대응 시간(MTTR) **50% 단축**.

### 미래 전망 및 진화 방향
Elasticsearch는 이제 **벡터 데이터베이스(Vector DB)**로 진화하고 있다. 텍스트의 의미를 벡터 값으로 변환하여 검색하는 `KNN (K-Nearest Neighbors)` 기능을 강화하며, ChatGPT와 같은 LLM 서비스의 **RAG (검색 증강 생성)**를 지원하는 필수 인프라로 자리 잡고 있다. 단순한 검색 엔진을 넘어 AI 기반의 '지능형 지식 베이스'로 거듭나고 있는 것이다.
