# 314. NoSQL 모델링 전략 - 쿼리 패턴 주도 설계 (Query-driven Modeling)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터의 중복을 최소화하는 RDBMS의 정규화(Normalization) 원칙을 버리고, 오직 **애플리케이션이 데이터를 어떻게 읽을 것인가(Query Pattern)**에 맞춰 데이터 구조를 미리 조립하고 중복을 허용하는 역정규화 기반의 설계 방식이다.
> 2. **가치**: 런타임 시의 무거운 JOIN 연산을 제거하여 분산 환경에서도 밀리초(ms) 단위의 일관된 응답 속도를 보장하며, 수평적 확장(Scale-out) 시 발생하는 네트워크 병목을 원천적으로 방지한다.
> 3. **융합**: 비즈니스 도메인의 경계를 정의하는 DDD(도메인 주도 설계), 대용량 조회를 위한 CQRS 패턴, 그리고 NoSQL의 물리적 샤딩 구조가 융합되어 거대 서비스의 데이터 설계를 완성한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
NoSQL 모델링은 "데이터를 어떻게 저장할까?"가 아니라 **"데이터를 어떻게 꺼내 쓸까?"**에서 시작한다. RDBMS가 '데이터 중심'이라면, NoSQL은 '기능 중심'의 설계다. 동일한 데이터를 여러 형태로 중복 저장(Denormalization)하더라도, 조회 한 번에 모든 데이터를 가져오는 것이 지상 과제다.

### 💡 비유
NoSQL 모델링은 **"뷔페 식당의 식단 구성"**과 같다.
- **RDBMS (정규화)**: 주방에 식재료별로(고기방, 채소방, 소스방) 따로 보관하는 것. 손님이 "비빔밥"을 주문하면 그때서야 각 방을 돌며 재료를 모아 비벼준다. (JOIN 부하).
- **NoSQL (역정규화)**: 손님이 비빔밥을 많이 찾을 걸 알고, 미리 그릇에 모든 재료를 비벼서 담아두는 것. 손님이 오면 그냥 그릇째 건네주면 된다. 재료(데이터)가 중복되어 주방 공간은 많이 차지하지만, 서비스 속도는 압도적으로 빠르다.

### 등장 배경 및 발전 과정

#### 1. 분산 환경의 JOIN 한계
데이터가 100대의 서버에 쪼개져 있을 때, A 테이블과 B 테이블을 JOIN 하려면 네트워크를 타고 데이터가 왔다 갔다 해야 한다. (Network Shuffle). 이는 성능 재앙을 부른다.

#### 2. 저장소 비용 하락 vs 컴퓨팅 비용 상승
과거엔 하드디스크가 비싸서 데이터 중복을 죄악시했다. 하지만 이제는 디스크는 싸고, CPU 연산과 사용자의 기다림(Latency)이 훨씬 비싸졌다. 이에 따라 "디스크를 더 쓰고 속도를 사자"는 NoSQL 철학이 정립되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### RDBMS vs NoSQL 모델링 원칙 비교 (표)

| 비교 항목 | RDBMS 모델링 | NoSQL 모델링 |
|-----------|---------------|--------------|
| **설계 중심** | 데이터 관계 (Schema-first) | **쿼리 패턴 (Query-first)** |
| **중복성** | 중복 배제 (Normalization) | **중복 허용 (Denormalization)** |
| **연산 시점** | 조회 시 연산 (JOIN) | **저장 시 연산 (Pre-aggregation)** |
| **확장성** | 수직 확장 (Scale-up) | **수평 확장 (Scale-out)** |
| **목표** | 데이터 정합성, 공간 절약 | **저지연 조회, 고가용성** |

### 정교한 구조 다이어그램 (역정규화 설계 과정)

```ascii
================================================================================
[ NoSQL Modeling: From Query Pattern to Schema ]
================================================================================

 [ 1. Query Analysis ]
  - "사용자가 상품을 조회할 때, 상품명 + 가격 + 판매자 이름 + 최근 리뷰 3개를 한꺼번에 보고 싶어 한다."

 [ 2. RDBMS Model (Join-heavy) ]
  Order Table ◀── JOIN ──▶ Product Table ◀── JOIN ──▶ Seller Table ◀── JOIN ──▶ Review Table
  * 문제: 조회 한 번에 4개 테이블 JOIN 발생. 분산 환경에서 매우 느림.

 [ 3. NoSQL Model (Embedded Document) ]
  ┌──────────────────────────────────────────────────────────┐
  │  Document: Product_Detail                                │
  │  {                                                       │
  │    "prod_id": 101,                                       │
  │    "name": "Super Coffee",                               │
  │    "price": 5000,                                        │
  │    "seller_info": { "name": "PF", "rank": "Gold" },      │ ◀── 역정규화 (Embedded)
  │    "recent_reviews": [                                   │ ◀── 미리 조립
  │       { "user": "Alice", "score": 5, "comment": "Good" },│
  │       { "user": "Bob",   "score": 4, "comment": "Nice" } │
  │    ]                                                     │
  │  }                                                       │
  └──────────────────────────────────────────────────────────┘
  * 효과: 단 한 번의 Key-Value 조회로 화면에 필요한 모든 데이터 획득 완료!
```

### 심층 동작 원리

#### ① 원자적 집계 (Atomic Aggregates)
함께 사용되는 데이터는 하나의 '애그리거트(Aggregate)'로 묶어서 저장한다. 이렇게 하면 데이터를 읽거나 쓸 때 여러 노드에 걸쳐 락(Lock)을 걸 필요가 없어 분산 트랜잭션의 복잡도가 사라진다.

#### ② 인덱싱 최적화 (Secondary Indexes)
NoSQL은 기본적으로 Partition Key로만 조회가 가능하다. 만약 다른 필드로도 검색이 잦다면, 데이터가 들어올 때 아예 그 필드를 Key로 하는 **인덱스용 테이블(GSI)**을 하나 더 만들어 관리한다. (쓰기 1번에 내부적으로 2번 쓰기 발생).

## Ⅲ. 융합 비교 및 다각도 분석

### 주요 NoSQL 데이터 모델링 기법

| 기법 명칭 | 상세 설명 | 비유 |
|-----------|-----------|------|
| **Embedding** | 연관 데이터를 자식 노드로 포함 (1:N) | 일기장에 사진을 직접 붙여놓기 |
| **Linking** | 데이터의 ID(참조)만 저장 (1:N) | 일기장에 "사진첩 5페이지 참조"라고 적기 |
| **Bucketing** | 시계열 데이터를 시간 단위로 뭉쳐 저장 | 영수증을 날짜별 봉투에 모아 담기 |
| **Computed Field**| 통계값(합계 등)을 실시간 미리 계산 저장 | 계산기 두드리지 않게 미리 총점 적어두기 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 한 글로벌 SNS에서 '팔로워 피드' 기능을 개발 중이다. 유저가 앱을 켜면 내가 팔로우하는 500명의 최신 글을 시간순으로 보여줘야 한다. RDBMS에서 `SELECT ... JOIN`으로 짰더니 유저가 100만 명이 넘는 순간 피드 로딩에 10초가 걸린다.
**판단 및 NoSQL 모델링 전략**:
1. **Fan-out on Write 전략**: 글을 쓸 때(`POST`), 그 글을 읽을 팔로워 500명의 **개인별 피드 박스(Redis List 등)**에 글 ID를 미리 배달(Push)해 둔다.
2. **역정규화**: 피드 박스에는 글 ID만 넣는 게 아니라, 글의 일부 내용과 작성자 프로필 사진 경로까지 미리 담아둔다.
3. **결과**: 유저가 앱을 켤 때(`GET`)는 JOIN이나 복잡한 연산 없이, 오직 **자기 피드 박스만 쓱 읽어오면 끝**이다. 조회 성능이 10초에서 0.01초로 1,000배 빨라진다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **RDBMS식 설계 이식**: NoSQL 테이블을 만들면서 RDBMS처럼 모든 데이터를 쪼개놓고(Normalize), 앱 코드에서 JOIN 로직을 짜는 것은 NoSQL을 쓰는 의미가 없다. 이는 네트워크 왕복만 늘려 성능을 더 악화시킨다.
- **데이터 일관성 무시**: 역정규화로 인해 데이터가 여러 곳에 흩어져 있으므로, 원본 데이터가 바뀌었을 때 모든 중복본을 업데이트하는 **'최종적 일관성(Eventual Consistency)'** 보장 매커니즘(예: Kafka 기반 동기화)이 반드시 설계에 포함되어야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **조회 응답 속도**: 대규모 분산 환경에서 JOIN 배제를 통해 **응답 속도 90% 이상 개선**.
- **확장 한계 돌파**: 데이터 간 물리적 의존성을 제거하여 서버 대수에 비례한 **선형적 성능 확장** 달성.

### 미래 전망 및 진화 방향
NoSQL 모델링은 이제 **'AI 기반 자동 모델링'**으로 나아가고 있다. 개발자가 쿼리 패턴을 분석하지 않아도, DB 엔진이 실제 유입되는 쿼리 로그를 학습하여 "이 데이터는 묶어서 저장하는 게 이득이다"라고 제안하거나 자동으로 테이블 구조를 바꾸는 지능형 DB 시대가 오고 있다. 또한, 관계형과 비관계형의 장점을 합친 **'NewSQL'**과의 융합을 통해, 모델링의 자유도와 데이터 정합성을 동시에 잡으려는 아키텍처적 시도가 주류가 될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **NoSQL 모델링이 뭔가요?**: 내일 학교 갈 준비물을 챙길 때, 가방 안에 교과서랑 공책을 미리 다 넣어두는 거예요. 아침에 급하게 책꽂이(RDBMS)를 뒤지지 않게 말이죠.
2. **왜 중복해서 저장하나요?**: 거실에도 연필을 두고 내 방에도 연필을 두면, 어디서든 바로 숙제를 시작할 수 있어서 훨씬 빠르기 때문이에요.
3. **가장 좋은 점은요?**: "이거 어딨지?"라고 고민하며 집안을 뛰어다닐 필요 없이, 내 앞에 놓인 바구니(Document)만 딱 열면 필요한 게 다 들어있다는 점이랍니다!
