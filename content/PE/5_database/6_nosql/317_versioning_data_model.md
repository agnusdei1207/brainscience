# 317. 버저닝 (Versioning) 데이터 모델 설계 - 시간 여행을 위한 테이블 설계법

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터의 변경 이력(History)을 덮어쓰지 않고, 과거의 상태를 언제든 조회하고 복원할 수 있도록 **모든 변경 사항을 버전(Version)으로 관리하는 데이터베이스 설계 기법**이다.
> 2. **혁신**: 단순한 CRUD(Create, Read, Update, Delete)를 넘어, 변경 시작 시간(Valid_From)과 종료 시간(Valid_To)을 기록하는 **이력 테이블 분리 방식**이나 **SCD(Slowly Changing Dimension)** 기법을 통해 데이터 무결성을 시간에 따라 보장한다.
> 3. **가치**: 금융 거래, 계약서 관리, 위키 문서 등 과거의 특정 시점의 데이터 상태 증명이 필수적인 시스템(Audit & Compliance)에서 반드시 적용되어야 하는 핵심 아키텍처다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
버저닝 데이터 모델은 데이터의 생성부터 삭제까지의 모든 생명주기를 기록한다. 데이터베이스에서 일반적인 `UPDATE`는 이전 데이터를 영원히 파괴하지만, 버저닝 환경에서의 `UPDATE`는 기존 데이터의 상태를 '과거'로 변경하고 새로운 '현재' 데이터를 `INSERT`하는 방식으로 작동한다.

### 💡 비유
- **일반 DB (칠판)**: 칠판에 글씨를 쓴다. 내용이 바뀌면 지우개로 싹 지우고 새로 쓴다. 3일 전에 칠판에 뭐라고 쓰여 있었는지는 아무도 모른다.
- **버저닝 DB (노트 필기)**: 노트에 글을 쓴다. 내용이 바뀌면 두 줄을 긋고(만료 처리), 그 밑에 새로운 내용을 적고 날짜를 적어둔다. 언제든 노트를 뒤적여서 "아, 3일 전에는 이런 내용이었지"라고 확인할 수 있다.

### 등장 배경: 'Compliance'와 'Audit'
기업의 회계 감사나 법적 분쟁이 발생했을 때, "지난달 15일 기준 고객의 요금제 상태"를 증명해야 하는 경우가 많다. 하지만 일반적인 RDBMS 설계에서는 덮어쓰기가 발생하여 과거를 증명할 수 없었다. 이 '데이터 증발'을 막기 위해 시계열적 관점이 결합된 버저닝 설계가 표준으로 자리 잡았다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 버저닝 설계의 3가지 주요 방식 (ASCII Diagram)

```ascii
[ 1. In-Table Versioning (단일 테이블) ]
┌────┬────────┬────────────┬─────────────┬───────────┐
│ ID │ Name   │ Valid_From │ Valid_To    │ Is_Active │
├────┼────────┼────────────┼─────────────┼───────────┤
│ 1  │ Alice  │ 2025-01-01 │ 2026-03-01  │ N         │ <-- 과거 데이터
│ 1  │ Alice_K│ 2026-03-01 │ 9999-12-31  │ Y         │ <-- 현재 데이터
└────┴────────┴────────────┴─────────────┴───────────┘

[ 2. History Table Separation (이력 테이블 분리) ]
 [ Master Table ]               [ History Table ]
┌────┬────────┐               ┌────┬──────┬─────────┬────────────┐
│ ID │ Name   │   (Update)    │ HID│ M_ID │ Old_Name│ Changed_At │
├────┼────────┤ ────────────> ├────┼──────┼─────────┼────────────┤
│ 1  │ Alice_K│               │ 101│ 1    │ Alice   │ 2026-03-01 │
└────┴────────┘               └────┴──────┴─────────┴────────────┘
```

### 핵심 설계 기법

#### ① 이력 테이블 분리 (History Table)
가장 대중적인 방식이다. 현재 상태를 저장하는 마스터 테이블과, 변경 전의 상태들을 누적해서 저장하는 히스토리 테이블을 분리한다.
- **장점**: 현재 데이터를 조회하는 일반적인 애플리케이션의 쿼리 성능(Read)에 영향을 주지 않는다.
- **단점**: 구조가 2배로 늘어나고, JOIN 없이 과거를 조회하기 까다롭다.

#### ② 단일 테이블 시계열 (SCD Type 2)
하나의 테이블에 여러 버전의 데이터를 넣고, `시작일`과 `종료일`, 그리고 `활성화 여부(Flag)` 칼럼을 두어 관리한다. (Slowly Changing Dimension Type 2).
- **장점**: 특정 시점(Point-in-time)의 데이터를 쿼리 한 번으로 완벽하게 재현할 수 있다.
- **단점**: 복합 기본키(PK) 설계가 필수적이며, 데이터가 방대해지면 현재 데이터 조회 속도까지 느려질 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### 버저닝 전략 (SCD Types) 비교 (Table)

| 구분 | 설명 | 특징 | 사용처 |
|:---:|:---|:---|:---|
| **SCD Type 1** | 기존 값을 덮어쓴다. (버저닝 안 함) | 이력 유실 | 단순 오타 수정 |
| **SCD Type 2** | 새로운 행(Row)을 추가하여 버전을 유지. | **가장 완벽한 이력 관리** | 요금제 변경, 직급 변경 |
| **SCD Type 3** | 기존 테이블에 `Old_Value` 컬럼을 추가. | 최근 1번의 변경만 기억 | 주소 변경 이력 (이전 주소) |
| **SCD Type 4** | Type 2와 3의 결합. (Mini-dimension 추가) | 빠르고 정교함 | 복잡한 DW 환경 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 설계 전략)

**상황**: 전자 결재 시스템의 '기안 문서' 테이블을 설계해야 한다. 기안이 반려되고 다시 수정될 때마다 이전 문서의 내용이 모두 보존되어야 한다.
**판단 및 버저닝 적용 전략**:
1. **단일 테이블 (버전 번호 부여)**: 결재 문서는 히스토리와 현재 상태가 동일한 맥락에서 자주 조회되므로, 단일 테이블에 `Document_ID`와 `Version_Num`을 복합 기본키(Composite PK)로 설정한다.
2. **트리거(Trigger) 또는 애플리케이션 로직**: 문서를 수정(Update) 요청이 오면, DB 락(Lock)을 걸고 가장 높은 `Version_Num + 1`로 새로운 행을 `INSERT` 하도록 로직을 통제한다. 기존 데이터는 절대 삭제(`DELETE`)하지 못하게 권한을 제한한다.
3. **효과**: "누가 언제 어떻게 문서를 고쳤는지" 완벽한 감사 추적(Audit Trail)이 가능해짐.

### 안티패턴 및 고려사항
- **의미 없는 데이터까지 버저닝**: '최근 접속 시간', '로그인 실패 횟수'처럼 비즈니스 규칙과 무관하게 빈번히 바뀌는 상태값까지 버저닝하면 DB 용량이 터진다. 버저닝 대상은 '핵심 비즈니스 데이터'로 엄격히 제한해야 한다.

## Ⅴ. 미래 전망 및 결론

### 결론: 데이터는 지워지지 않는다
과거 데이터베이스 공간이 비싸던 시절에는 덮어쓰기(UPDATE)가 미덕이었다. 하지만 현재 스토리지는 저렴해졌고 데이터의 맥락(Context)은 비싸졌다. 버저닝 데이터 모델은 데이터베이스가 단순한 저장소를 넘어 '기억의 저장소'로 진화하는 기초 공사다.

### 미래 전망
최근에는 이러한 버저닝 로직을 애플리케이션 개발자가 직접 짜지 않아도 되는 기술들이 등장하고 있다. 예를 들어 **Event Sourcing** 아키텍처나, AWS QLDB 같은 **원장(Ledger) 전용 데이터베이스**, 그리고 PostgreSQL의 **Temporal Table** 확장이 버저닝 설계를 자동화하고 표준화하는 방향으로 발전할 것이다.
