# 318. 트리 구조 저장을 위한 NoSQL 모델 - 계층형 데이터의 우아한 보관법

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 조직도, 카테고리, 게시판 댓글처럼 '부모-자식' 계층을 가진 **트리(Tree) 구조의 데이터를 관계형이나 NoSQL 데이터베이스에 효율적으로 저장하고 조회하기 위한 모델링 기법**이다.
> 2. **혁신**: 단순한 부모 참조(Adjacency List)의 재귀 호출(Recursive) 성능 저하 문제를 해결하기 위해, **Materialized Path(경로 구체화)**나 **Nested Sets(중첩 집합)** 같은 혁신적인 패턴을 사용하여 O(1) 수준의 검색 속도를 달성한다.
> 3. **가치**: 대규모 메뉴 카테고리나 수천 개의 대댓글이 달리는 시스템에서 트리 탐색 병목 현상을 제거하여 애플리케이션의 응답 속도를 획기적으로 향상시킨다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
데이터베이스는 기본적으로 '평면(Flat)' 구조다. 이 평면 구조에 '입체적인 계층(Tree)'을 구겨 넣는 작업은 생각보다 까다롭다. 트리 모델링은 데이터를 저장할 때 약간의 부가 정보(경로, 좌우 번호 등)를 함께 저장하여, 트리 전체를 순회하지 않고도 자식 노드들을 한 번의 쿼리로 긁어올 수 있게 하는 기법들이다.

### 💡 비유
- **일반적인 부모 참조 방식 (Adjacency List)**: 회사에서 "내 직속 상사는 누구입니다"만 적어두는 방식이다. "사장님 밑에 있는 모든 말단 직원을 다 불러와!"라고 하면, 사장→부장→과장→대리→사원을 일일이 찾아가며 물어봐야 한다 (매우 느림).
- **경로 구체화 (Materialized Path)**: 직원 명찰에 "전자/모바일/마케팅/홍길동"처럼 아예 전체 부서 소속을 길게 다 적어놓는 것이다. "모바일 부서 사람 다 모여!"라고 하면 명찰에 '모바일'이라고 적힌 사람만 한 번에 찾으면 된다 (매우 빠름).

### 등장 배경: Recursive 쿼리의 한계
전통적 RDBMS에서 트리를 찾으려면 `CONNECT BY` 같은 재귀 쿼리(Recursive Query)를 써야 했다. 하지만 이는 트리가 깊어질수록 성능이 기하급수적으로 느려졌고, MongoDB 같은 NoSQL에서는 아예 JOIN이나 재귀 쿼리를 쓰기 어려웠다. 따라서 데이터를 넣을 때 아예 경로를 계산해서 넣는 영리한 패턴들이 고안되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 3대 트리 저장 모델 (ASCII Diagram)

```ascii
[ Tree Structure ]
        (Electronics) 1
         /         \
    (Mobile) 2   (TV) 3
      /
(Phone) 4

[ 1. Materialized Path (경로 구체화) ]
┌────┬──────────────┬────────────────────────┐
│ ID │ Name         │ Path                   │
├────┼──────────────┼────────────────────────┤
│ 1  │ Electronics  │ /1                     │
│ 2  │ Mobile       │ /1/2                   │
│ 4  │ Phone        │ /1/2/4                 │ <-- "LIKE '/1/2/%'" 로 하위 탐색 가능
└────┴──────────────┴────────────────────────┘

[ 2. Nested Sets (중첩 집합) ]
┌────┬──────────────┬─────┬──────┐
│ ID │ Name         │ Lgt │ Rgt  │ (Left, Right)
├────┼──────────────┼─────┼──────┤
│ 1  │ Electronics  │ 1   │ 8    │ <-- 자식들은 이 숫자 범위 안에 존재함
│ 2  │ Mobile       │ 2   │ 5    │ <-- (Phone의 Lgt:3, Rgt:4)
└────┴──────────────┴─────┴──────┘
```

### 핵심 모델링 기법 상세

#### ① Materialized Path (구체화된 경로)
각 노드가 루트(Root)부터 자신까지의 모든 경로를 문자열 형태로 저장한다.
- **장점**: 정규표현식(`LIKE '/1/2/%'`)을 통해 특정 노드의 모든 하위 노드를 한 번의 쿼리로 가져온다. 직관적이고 구현이 가장 쉽다.
- **단점**: 경로 문자열의 길이에 제한이 생길 수 있으며, 중간 노드의 위치가 바뀌면 하위 노드들의 Path 문자열을 전부 업데이트해야 한다.

#### ② Nested Sets (중첩 집합)
트리를 '집합'으로 보고, 부모 집합이 자식 집합을 완전히 포함하도록 왼쪽(Left)과 오른쪽(Right) 번호를 부여한다.
- **장점**: "A노드의 모든 자식을 찾아라"라는 쿼리를 `BETWEEN left AND right` 연산으로 가장 완벽하고 빠르게(Index 활용) 수행한다.
- **단점**: 데이터 추가(Insert) 시 뒤쪽 노드들의 번호를 싹 다 밀어서 다시 매겨야 하므로, 잦은 추가/삭제가 일어나는 환경에서는 최악의 성능을 낸다.

#### ③ Closure Table (클로저 테이블)
노드 테이블과 별개로, '모든 부모-자식 간의 관계(직속이 아니더라도)'를 저장하는 별도의 브릿지 테이블을 둔다.
- **장점**: 계층 이동, 추가, 조회가 모두 빠르고 트리 깊이에 제한이 없다.
- **단점**: 관계 테이블의 데이터 양(Row 수)이 기하급수적으로 커져 공간을 많이 차지한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 트리 저장 모델 비교 분석 (Table)

| 모델명 | 쿼리 난이도 | 하위 노드 검색 (Read) | 노드 이동/추가 (Write) | 최적의 사용처 |
|:---:|:---|:---|:---|:---|
| **Adjacency List** | 쉬움 | 매우 느림 (재귀 필요) | 매우 빠름 | 깊이가 얕은 단순 카테고리 |
| **Materialized Path**| 쉬움 | 빠름 (문자열 검색) | 느림 (경로 업데이트) | 폴더 구조, 게시판 대댓글 |
| **Nested Sets** | 매우 어려움| **가장 빠름 (Between)** | **매우 느림 (번호 재할당)** | 읽기 전용 거대 분류 (쇼핑몰 카테고리) |
| **Closure Table** | 보통 | 빠름 | 빠름 | 깊고 복잡한 조직도, 다중 부모 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 도입 전략)

**상황**: MongoDB 기반의 글로벌 쇼핑몰을 구축 중이다. 상품 카테고리는 최대 10단계까지 깊어질 수 있다(의류 > 여성 > 아우터 > 코트...). 카테고리는 한 번 정해지면 거의 바뀌지 않지만, 사용자 조회 트래픽은 초당 수만 건에 달한다.
**판단 및 모델링 전략**:
1. **패턴 선정**: 조회(Read)가 압도적으로 많고 쓰기(Write)가 거의 없는 구조이므로 **Materialized Path** 패턴을 채택한다. Nested Sets도 좋지만 NoSQL 환경에서는 구현이 너무 복잡하다.
2. **MongoDB 최적화**: Document 구조 안에 배열(Array) 속성으로 `ancestors: [ "의류", "여성", "아우터" ]` 형태를 추가한다.
3. **인덱스 설계**: `ancestors` 배열 필드에 멀티키 인덱스(Multikey Index)를 걸어둔다.
4. **결과**: `db.categories.find({ ancestors: "의류" })` 쿼리 한 번으로 10단계 밑에 있는 수만 개의 하위 카테고리까지 0.01초 만에 불러오게 됨.

### 안티패턴 및 고려사항
- **Nested Sets의 잦은 업데이트**: 게시판 댓글처럼 실시간으로 데이터가 추가되는 곳에 Nested Sets를 쓰면, 댓글 하나 달릴 때마다 수백만 개의 행(Row)에 Lock이 걸리며 데드락(Deadlock) 지옥이 열린다.

## Ⅴ. 미래 전망 및 결론

### 결론: 데이터의 형태가 아키텍처를 결정한다
트리 모델링은 관계형이나 평면형 DB의 한계를 설계적 꼼수(Design Pattern)로 극복한 훌륭한 사례다. 시스템의 Read/Write 비율과 트리의 깊이를 분석하여 최적의 모델을 고르는 것이 백엔드 엔지니어의 핵심 역량이다.

### 미래 전망
최근에는 [그래프 데이터베이스](./351_graph_database.md)(Neo4j 등)가 대중화되면서, 복잡한 트리나 네트워크 구조를 억지로 RDBMS나 문서 DB에 넣기보다는 아예 그래프 DB로 오프로딩(Off-loading)하는 추세다. 향후 멀티모델 데이터베이스가 표준이 되면, 개발자는 별도의 경로 패턴을 구현할 필요 없이 네이티브 트리/그래프 기능을 사용하게 될 것이다.

3. **어떻게 해결하나요?**: 사람 이름 옆에 "나는 김씨 집안, 서울 파, 37대손입니다"라고 주소를 길게 다 적어놓으면(경로 구체화), 족보 책을 일일이 넘기지 않아도 한 번에 찾을 수 있답니다!
