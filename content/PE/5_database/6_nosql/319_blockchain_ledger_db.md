# 319. 원장 데이터베이스 (Ledger DB / Amazon QLDB) - 무결성을 증명하는 불변의 장부

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 블록체인의 **'변조 방지(Immutable)'** 특성과 **'암호학적 검증(Cryptographic Verifiable)'** 능력을 차용하되, 복잡한 분산 합의 과정 없이 중앙 집중형으로 고성능을 내도록 설계된 목적형 데이터베이스다.
> 2. **혁신**: 일반 DB처럼 관리자(DBA)가 데이터를 몰래 수정하는 것을 원천적으로 차단하며, 데이터의 생성부터 변경까지의 모든 이력(History)을 체인 형태로 암호화하여 저장한다.
> 3. **가치**: 금융 거래, 공급망 추적, 의료 기록 등 데이터의 투명성과 신뢰성이 절대적으로 보장되어야 하는 엔터프라이즈 환경에서 퍼블릭 블록체인의 느린 속도를 완벽히 대체한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
원장 데이터베이스(Ledger Database)는 데이터를 기록하는 '장부(Ledger)'의 개념을 시스템화한 것이다. 가장 유명한 서비스인 Amazon QLDB(Quantum Ledger Database)는 데이터의 변경 이력을 순차적인 블록(Block)으로 기록하고, 각 블록을 이전 블록의 해시(Hash)값으로 묶어버린다.

### 💡 비유
- **일반 데이터베이스 (연필로 쓴 장부)**: 회계장부를 연필로 쓴다. 사장님(DBA)이 밤에 몰래 지우개로 100만 원을 10만 원으로 고쳐 써도 아무도 증명할 수 없다.
- **원장 데이터베이스 (도장 찍힌 릴레이 장부)**: 회계장부를 지워지지 않는 잉크로 쓴다. 한 줄을 쓸 때마다 이전 줄의 내용과 합쳐서 '암호 도장(해시)'을 찍는다. 만약 사장님이 한 글자라도 고치면 전체 도장이 전부 깨져버리기 때문에, 변조 시도가 즉시 발각된다.

### 등장 배경: 'Private Blockchain'의 비효율성
기업들은 데이터의 조작을 막기 위해 프라이빗 블록체인(Hyperledger Fabric 등)을 도입했다. 하지만 블록체인은 여러 노드가 합의(Consensus)를 거쳐야 하므로 너무 느리고 구성이 복잡했다. "합의는 필요 없고, 그냥 절대 고칠 수 없는 빠른 중앙 DB가 필요해!"라는 기업들의 요구에 맞춰 QLDB가 탄생했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 원장 DB의 암호학적 구조 (ASCII Diagram)

```ascii
[ Journal (Append-only & Immutable Log) ]
┌────────────────┐    ┌────────────────┐    ┌────────────────┐
│ Block 1        │    │ Block 2        │    │ Block 3        │
│ Data: Insert A │    │ Data: Update A │    │ Data: Delete A │
│ Hash: H(1)     │<───│ Prev Hash: H(1)│<───│ Prev Hash: H(2)│
└────────────────┘    │ Hash: H(2)     │    │ Hash: H(3)     │
                      └────────────────┘    └────────────────┘
                              │
          (Asynchronous Materialization)
                              ▼
        ┌───────────────────────────────────────────┐
        │ [ Current State ]      [ History State ]  │
        │ A: Deleted             A1 -> A2 -> Del    │ (Queryable Views)
        └───────────────────────────────────────────┘

* 핵심 원리: SHA-256 해시 함수를 이용한 체이닝으로 위변조를 원천 봉쇄.
```

### 핵심 구성 요소

#### ① 저널 (Journal)
데이터베이스의 심장이다. 모든 트랜잭션(Insert, Update, Delete)은 가장 먼저 저널에 순차적으로(Append-only) 기록된다. 이 저널은 수정이나 삭제가 불가능(Immutable)하다.

#### ② 해시 체이닝 (Hash Chaining)
각각의 트랜잭션은 이전 트랜잭션의 결과값을 포함하여 SHA-256 해시 함수로 암호화된다. 중간에 누군가 하드디스크를 뜯어서 Block 2를 수정하면, 그 뒤의 모든 해시값이 틀어지게 되어 데이터 훼손이 수학적으로 증명된다.

#### ③ 상태 뷰 (Current & History View)
저널은 순차 기록용이라 검색이 느리다. 그래서 원장 DB는 저널의 내용을 바탕으로 현재 상태만 보여주는 뷰(Current)와 과거 이력을 보여주는 뷰(History)를 별도의 테이블 형태로 자동 생성해 준다. 사용자는 SQL과 유사한 PartiQL 언어로 쉽게 쿼리할 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### RDBMS vs 블록체인 vs 원장 DB 비교

| 비교 항목 | RDBMS (PostgreSQL) | 퍼블릭 블록체인 (Ethereum) | **원장 DB (Amazon QLDB)** |
|-----------|--------------------|----------------------------|---------------------------|
| **신뢰성 (무결성)**| DBA에 의해 조작 가능 | 완벽함 (다수결 합의) | **완벽함 (암호학적 증명)** |
| **처리 성능 (TPS)**| 수십만 ~ 수백만 | 수십 ~ 수백 (매우 느림) | **수만 ~ 수십만 (빠름)** |
| **통제권** | 중앙 집중 | 완전 분산 (탈중앙화) | **중앙 집중 (단일 권한자)** |
| **기록 방식** | Overwrite (덮어쓰기) | Append-only | **Append-only** |
| **적합한 용도** | 일반 비즈니스 로직 | 가상화폐, 스마트 컨트랙트 | **엔터프라이즈 감사, 물류** |

### 기술적 임팩트: 'Trust but Verify'
과거에는 DBA나 클라우드 제공자(AWS)를 맹목적으로 믿어야(Trust) 했다. 원장 DB는 데이터를 외부 기관에 넘기고 "해시값을 검증(Verify)해 보라"고 당당하게 요구할 수 있게 해준다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시스템 설계 전략)

**상황**: 자동차 제조사에서 차량의 부품 조립부터 출고, 수리, 중고차 거래까지의 모든 '차량 이력 추적 시스템'을 만들어 정부에 제출해야 한다. 기록의 위변조가 절대 없어야 한다.
**판단 및 DB 도입 전략**:
1. **DB 선정**: 비싼 프라이빗 블록체인을 구축하는 대신, AWS의 완전 관리형 **Amazon QLDB**를 채택하여 인프라 관리 비용을 0으로 만든다.
2. **이벤트 드리븐 결합**: 차량 상태가 변경될 때마다 QLDB에 기록을 남기고, 동시에 **Kinesis/EventBridge**로 이벤트를 쏴서 다른 마이크로서비스(보증기간 계산 등)가 작동하게 한다.
3. **무결성 검증 제공**: 중고차 구매자에게 차량 이력의 '다이제스트(Digest, 해시 요약본)' 파일을 제공하여, 구매자가 직접 AWS API를 통해 해당 기록이 위조되지 않았음을 검증할 수 있게 포털을 구축한다.
4. **결과**: 투명한 이력 관리로 중고차 가격 방어에 기여하고 법적 감사 규제(Compliance)를 완벽히 통과함.

### 안티패턴 및 고려사항
- **일반 DB의 대체재로 사용**: QLDB는 일반 관계형 DB를 완전히 대체할 수 없다. `UPDATE`나 `DELETE`가 잦은 휘발성 데이터(예: 쇼핑몰 장바구니 세션)를 QLDB에 넣으면 저널 용량이 폭발하여 심각한 비용이 발생한다.
- **분산 신뢰(탈중앙화) 요구 시**: 여러 회사가 서로 믿지 못해서 공동의 장부를 만들어야 하는 컨소시엄 환경이라면, 중앙 집중형인 원장 DB보다는 프라이빗 블록체인(Hyperledger)이 올바른 아키텍처다.

## Ⅴ. 미래 전망 및 결론

### 결론: 실용주의적 블록체인
원장 데이터베이스는 '블록체인의 철학'에서 비효율성(합의 알고리즘, 코인 채굴)을 걷어내고, '엔터프라이즈에 진짜 필요한 기능(변조 방지)'만 엑기스로 뽑아낸 최고의 실용주의 기술이다.

### 미래 전망
앞으로는 모든 클라우드 RDBMS가 기본 기능이나 확장팩 형태로 '원장 모드(Ledger Mode)'를 탑재하게 될 것이다. (실제로 Oracle이나 SQL Server는 블록체인 테이블 기능을 추가하고 있다). 결국 데이터의 무결성 증명은 특별한 DB의 기능이 아니라 모든 데이터베이스가 갖춰야 할 기본 표준 스펙으로 자리 잡을 전망이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **원장 DB가 뭔가요?**: 절대로 지워지지 않는 특수 마법 잉크로만 글씨를 쓸 수 있는 아주 단단한 일기장이에요.
2. **누가 몰래 일기장을 찢으면요?**: 일기장의 모든 페이지가 앞장과 보이지 않는 마법의 실로 묶여 있어서, 한 페이지라도 찢거나 고치면 전체 일기장에 빨간불이 번쩍 들어오면서 엉망이 돼요.
3. **어디에 쓰나요?**: 은행의 돈거래 기록이나, 경찰서의 수사 기록처럼 "절대로 누군가 몰래 고쳐서는 안 되는 아주 중요한 정보"를 보관할 때 쓴답니다!
