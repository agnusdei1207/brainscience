# 320. 다중 모델 데이터베이스 (Multi-model Database) - 데이터 모델의 대통합

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 관계형(Table), 문서형(JSON Document), 그래프(Graph), 키-밸류(Key-Value) 등 형태가 다른 여러 데이터 모델을 **단일 백엔드 데이터베이스 엔진 안에서 동시에 지원**하는 기술이다.
> 2. **혁신**: 데이터를 이동시키거나 동기화할 필요 없이, 서로 다른 형태의 데이터들을 하나의 쿼리 언어와 단일 ACID 트랜잭션 안에서 묶어서(Cross-model Join) 처리할 수 있다.
> 3. **가치**: 폴리글랏 퍼시스턴스(Polyglot Persistence)로 인해 파편화된 데이터 인프라의 복잡성과 비용을 획기적으로 낮추어, 현대 애플리케이션의 유연성을 극대화한다.

*(※ 본 문서는 멀티모델 데이터베이스의 기본 원리에 집중하며, 활용 전략은 [353. 멀티모델 데이터베이스](./353_multimodel_db.md)와 연계됩니다.)*

---

## Ⅰ. 개요 (Context & Background)

### 개념
다중 모델 데이터베이스는 이름 그대로 여러 얼굴을 가진 데이터베이스다. 사용자는 같은 DB 안에서 사용자 프로필을 JSON(문서)으로 저장하고, 그 사용자들 간의 친구 관계를 간선(그래프)으로 연결하며, 세션 상태를 KV 구조로 초고속 캐싱할 수 있다. ArangoDB, Azure Cosmos DB 등이 선구자다.

### 💡 비유
- **싱글 모델 DB**: 양식 전문점(RDBMS), 중식 전문점(문서 DB), 일식 전문점(그래프 DB)이 따로 있다. 친구들과 3가지 요리를 다 먹으려면 식당 3곳을 돌아다니거나 배달을 각각 시켜야 해서 피곤하다.
- **다중 모델 DB (푸드코트/뷔페)**: 거대한 푸드코트다. 주방(Storage Engine)은 하나로 통합되어 효율적으로 재료를 쓰면서도, 창구(API)에서는 피자, 짜장면, 초밥을 한 쟁반에 담아 단 한 번의 결제(트랜잭션)로 제공한다.

### 등장 배경: 'Polyglot'의 역습
마이크로서비스 시대에는 "각 용도에 가장 잘 맞는 DB를 쓰자"는 폴리글랏 퍼시스턴스가 유행했다. 하지만 현실은 참혹했다. 개발팀은 MongoDB, Neo4j, Redis의 각기 다른 문법을 배워야 했고, 인프라 팀은 3가지 DB 클러스터를 관리하느라 야근을 밥 먹듯 했다. 결국 "이걸 하나로 합칠 순 없을까?"라는 엔지니어들의 염원이 다중 모델 DB를 탄생시켰다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 다중 모델 엔진의 코어 아키텍처 (ASCII Diagram)

```ascii
[ Native Multi-model Architecture (e.g., ArangoDB) ]

      [ Application Layer ]
               │
               ▼
┌───────────────────────────────┐
│     Multi-model Query API     │  (SQL, AQL, GraphQL)
│  (Parse, Optimize, Execute)   │
└──────────────┬────────────────┘
               │  <-- 단일 실행 계획 (Single Execution Plan)
┌──────────────▼────────────────┐
│   Core Storage Engine (C++)   │
│                               │
│  [ Document Store (JSON) ]    │  <-- 데이터를 기본적으로 JSON으로 저장
│  [ Edge Index (Graph)    ]    │  <-- JSON 문서의 ID를 연결하는 특수 인덱스
│  [ Key-Value Index       ]    │  <-- 빠른 조회를 위한 KV 매핑
└───────────────────────────────┘
               ▼
        [ Physical Disk ]

* 핵심 원리: 문서를 기본 단위로 하되, 강력한 '그래프 엣지 인덱스'를 
  같은 코어 단에서 처리하여 모델 간의 벽을 허뭄.
```

### 핵심 구현 방식 2가지

#### ① Native Multi-model (진성 다중 모델)
처음부터 여러 모델을 고려하여 설계된 엔진이다. 예를 들어 ArangoDB는 모든 데이터를 Document로 저장하지만, Document 사이에 포인터(Edge)를 물리적으로 직접 연결하는 구조를 내장하여 완벽한 그래프 순회 성능을 낸다. **이질적 데이터 간의 JOIN과 트랜잭션**이 완벽히 지원된다.

#### ② Layered Multi-model (확장형 다중 모델)
기존의 유명한 DB 위에 껍데기를 씌운 형태다. 예를 들어 PostgreSQL은 원래 RDBMS지만, JSONB 데이터 타입을 추가하여 문서 DB처럼 쓰고, Apache AGE 확장을 달아 그래프 DB처럼 쓴다. 안정성은 최고지만 성능 튜닝에 구조적 한계가 존재할 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### 다중 DB 체제 vs 다중 모델 DB 체제

| 비교 기준 | 다중 DB (Polyglot) | **다중 모델 DB (Multi-model)** |
|:---:|:---|:---|
| **아키텍처** | 파편화된 분산 아키텍처 | **단일 통합 엔진 아키텍처** |
| **운영 복잡도**| 매우 높음 (다수 클러스터, 백업) | **낮음 (하나의 클러스터 관리)** |
| **데이터 동기화**| 카프카 등 외부 브로커/ETL 필수 | **불필요 (한 곳에 존재)** |
| **크로스 쿼리** | 거의 불가능 (앱 레벨에서 조인) | **네이티브 언어로 원클릭 조인 가능** |
| **특화 성능** | 특정 워크로드에서 100점 만점 | 모든 워크로드에서 80~90점 수준 달성 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 도입 전략)

**상황**: 사물인터넷(IoT) 기반 스마트 팜 관제 시스템을 만든다. 센서의 시계열 로그(TSDB), 센서들의 네트워크 연결 상태(Graph), 장비 스펙 메타데이터(Document)를 동시에 처리해야 한다.
**판단 및 데이터 모델 통합 전략**:
1. **DB 통합 결정**: 3개의 DB를 띄우면 라이선스 및 유지보수 비용이 폭증하므로, Azure Cosmos DB나 ArangoDB를 도입한다.
2. **모델링 설계**: 장비 스펙은 유연한 JSON Document로 저장하고, 장비 간의 통신 토폴로지는 Edge 속성을 활용해 Graph로 연결한다.
3. **통합 쿼리 활용**: "최근 온도가 100도를 넘은 센서(문서 조건)와 물리적으로 직접 연결된(그래프 조건) 모든 장비들의 목록을 뽑아라"라는 질의를 단일 쿼리로 작성하여 앱의 로직을 대폭 단순화한다.
4. **결과**: 개발 기간 단축 및 데이터 정합성 불일치 문제(Sync Issue)를 아키텍처 레벨에서 근본적으로 제거.

### 안티패턴 및 고려사항
- **성능의 은탄환 오해**: 다중 모델 DB가 편리하긴 하지만, 극단적인 대용량 트래픽(예: 초당 1,000만 건의 Key-Value 캐싱)에서는 전용 솔루션(Redis)을 절대 이길 수 없다. 80%의 범용적인 비즈니스 복잡도를 해결하는 데 특화되어 있음을 인지해야 한다.

## Ⅴ. 미래 전망 및 결론

### 결론: 컨버전스(Convergence)의 시대
스마트폰이 전화기, 카메라, MP3를 하나로 흡수했듯, 데이터베이스 시장도 파편화를 넘어 통합의 시기로 접어들었다. 다중 모델 데이터베이스는 엔지니어들이 데이터 모델의 제약 없이 비즈니스 로직에만 집중하게 해주는 해방구다.

### 미래 전망
앞으로 데이터베이스 시장은 순수한 '단일 모델 NoSQL'은 점점 도태되고, PostgreSQL과 같은 레거시 거인들이 다중 모델 기능을 흡수하거나, Cosmos DB 같은 클라우드 네이티브 플랫폼이 시장을 장악하는 형태로 양분될 것이다. 나아가 Vector(AI 임베딩) 모델까지 흡수하는 **'초거대 다중 모델 엔진'**이 AI 시대의 표준 저장소가 될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **다중 모델 DB가 뭔가요?**: 글씨 쓰는 연필, 그림 그리는 크레파스, 종이 자르는 가위가 하나의 필통 안에 쏙 들어가 있는 '만능 학용품 세트'예요.
2. **왜 이게 필요한가요?**: 예전에는 글씨 쓰다 가위질하려면 다른 방에 가서 가위를 가져와야 했는데, 만능 세트는 그 자리에서 바로 꺼내 쓸 수 있어서 안 귀찮거든요.
3. **가장 좋은 점은?**: 방을 여러 개 어지를 필요 없이 책상 하나만 잘 치우면 되니까 관리가 아주 쉽고 편하답니다!
