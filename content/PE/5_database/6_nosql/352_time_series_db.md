# 352. 시계열 데이터베이스 (TSDB) - 시간의 흐름을 기록하는 지휘자

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터에 반드시 **'시간(Timestamp)'** 정보가 포함되며, 시간이 지남에 따라 변하는 데이터를 초고속으로 저장, 조회, 압축하기 위해 특화된 목적형 데이터베이스다.
> 2. **혁신**: 데이터는 항상 새로운 시간대에 순차적으로 추가(Append-only)된다는 특성을 이용해 삽입 속도를 극대화하고, 과거 데이터는 자동으로 묶어서 압축(Downsampling)하여 저장 공간을 혁신적으로 줄인다.
> 3. **가치**: 서버 모니터링(CPU/메모리 지표), IoT 센서 데이터, 주식/가상화폐 틱 데이터 등 초당 수만~수백만 건씩 쏟아지는 시계열 데이터를 분석하고 시각화(Grafana 등)하는 데 필수적인 인프라다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
시계열 데이터베이스(Time-Series Database, TSDB)는 시간을 기본 축으로 하여 값(Value)의 변화를 기록한다. 대표적인 오픈소스 솔루션으로 InfluxDB, Prometheus, TimescaleDB 등이 있다.

### 💡 비유
- **일반 DB (주소록)**: 사람들의 주소와 전화번호를 적어둔다. 정보가 바뀌면 기존 내용을 지우고(Update) 새 번호를 쓴다.
- **TSDB (심전도 그래프 기록기)**: 환자의 심장 박동을 0.1초마다 계속 종이에 찍어낸다. 한 번 찍힌 심장 박동은 지워지거나 수정되지 않는다. 그저 새로운 점이 끝없이 뒤에 추가(Append)될 뿐이며, 나중에는 1시간 치를 평균 내어 한 줄로 요약(Downsampling)해 보관한다.

### 등장 배경: 'Update'의 종말과 'Append'의 시대
기존의 관계형 DB(RDBMS)에 초당 10만 개의 온도 센서 데이터를 집어넣으면, 복잡한 인덱스(B-Tree) 업데이트 때문에 서버가 멈춰버린다. 또한 시계열 데이터는 한 번 기록되면 수정될 일이 거의 없다. 이러한 시계열 데이터만의 독특한 생명주기(Lifecycle)에 완벽히 맞춘 전용 DB가 필요해졌다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### TSDB 데이터 구조 (ASCII Diagram)

```ascii
[ Time-Series Data Model ]

Metric Name: cpu_usage
Tags (Dimensions): host=serverA, region=us-east  <-- 인덱싱됨 (빠른 검색용)
Fields (Values): value=85.5                      <-- 시간에 따라 변하는 실제 값

[ Timestamp ]          [ Metric + Tags ]        [ Fields ]
2026-03-05 10:00:00  | cpu_usage (serverA)  |  85.5
2026-03-05 10:00:01  | cpu_usage (serverA)  |  87.2
2026-03-05 10:00:02  | cpu_usage (serverA)  |  89.1
       │                      │                   │
    (Time 축)            (Indexed)           (Append-only)

* 핵심 원리: 시간 기반의 파티셔닝(Time Chunk)을 통해 쓰기 속도를 극대화.
```

### 핵심 기술 요소

#### ① Append-Only & Time-partitioning
데이터는 디스크의 끝부분에 순차적으로 기록된다(Sequential Write). 그리고 데이터를 '1시간 단위', '1일 단위'의 청크(Chunk)로 잘라서 저장한다. 이렇게 하면 오래된 데이터를 지울 때(Retention) 그냥 파일 덩어리 하나만 날려버리면 되므로 I/O 부하가 없다.

#### ② 다운샘플링 (Downsampling)과 보존 정책 (Retention)
- 1초 단위의 원본 데이터는 7일만 보관한다.
- 7일이 지나면 1분 단위의 평균값으로 요약해서 1달을 보관한다.
- 1달이 지나면 1시간 단위 평균값으로 요약해 영구 보관한다.
이를 통해 디스크 용량의 폭발적 증가를 막는다.

#### ③ 고압축 알고리즘 (Gorilla Compression 등)
시간 데이터나 온도 데이터는 1초 전 데이터와 값이 거의 비슷하다. TSDB는 차이값(Delta)만 저장하거나 XOR 연산을 통해 데이터를 극도로 압축하여, 디스크 사용량을 일반 DB 대비 1/10 수준으로 줄인다.

## Ⅲ. 융합 비교 및 다각도 분석

### RDBMS vs TSDB 상세 비교

| 비교 항목 | RDBMS (PostgreSQL 등) | **TSDB (InfluxDB 등)** |
|-----------|-----------------------|------------------------|
| **주요 연산** | CRUD (Read/Update 위주) | **Read / Insert (Update 거의 없음)** |
| **인덱스 구조** | B-Tree (범용) | **LSM Tree 변형 (쓰기 최적화)** |
| **데이터 보관** | 영구적 유지 | **시간 경과 시 자동 삭제 (Retention)** |
| **압축 효율** | 보통 | **매우 높음 (Delta-of-Delta 압축)** |
| **질의 특징** | "유저 A의 정보는?" | **"최근 1시간 CPU 평균 사용량 추이는?"**|

### 주요 TSDB 솔루션 포지셔닝
- **Prometheus**: 쿠버네티스(K8s) 생태계의 표준 모니터링 시스템. 주기적으로 데이터를 가져오는(Pull) 방식에 특화.
- **InfluxDB**: 가장 널리 쓰이는 범용 TSDB. 데이터를 밀어넣는(Push) 방식에 강함.
- **TimescaleDB**: PostgreSQL 기반으로 확장한 TSDB. 기존 SQL 지식을 그대로 쓸 수 있어 인기가 높음.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 도입 전략)

**상황**: 전국 10,000대의 전기차 충전기에서 1초마다 전압, 전류, 온도 센서 데이터가 날아온다. 이 데이터를 실시간 대시보드에 띄우고, 1년 치 과거 데이터를 AI로 분석해 고장을 예측해야 한다.
**판단 및 TSDB 적용 전략**:
1. **솔루션 선정**: IoT 데이터 대량 수집(Push)에 유리하고 압축률이 높은 **InfluxDB**를 메인 저장소로 선정한다.
2. **Retention Policy 설정**: 1초 단위 초정밀 데이터는 14일만 유지하고, 한 달 단위는 1분 평균, 1년 단위는 1시간 평균으로 **CQ(Continuous Query)**를 돌려 다운샘플링을 자동화한다.
3. **시각화 연동**: **Grafana**를 InfluxDB에 연동하여 경영진을 위한 실시간 전국 충전소 모니터링 화면을 구성한다.
4. **결과**: 기존 MySQL 대비 쓰기 성능 20배 향상, 스토리지 비용 80% 절감, 시계열 집계(Aggregate) 쿼리 응답 속도 10배 이상 향상.

### 안티패턴 및 고려사항
- **High Cardinality (카디널리티 폭발)**: TSDB는 태그(Tag) 값을 인덱스로 만든다. 만약 태그에 '초 단위로 변하는 임의의 ID'나 '사용자 세션 키' 같은 무한한 값을 넣으면 메모리(인덱스)가 폭발하여 서버가 뻗어버린다. 태그에는 유한한 값(지역, 장비 종류)만 넣어야 한다.
- **Update 남용**: TSDB에서 과거의 특정 데이터를 수정하거나 삭제하는 연산은 아키텍처 구조상 매우 비효율적이고 느리다. 수정이 잦은 비즈니스 로직에는 절대 사용하면 안 된다.

## Ⅴ. 미래 전망 및 결론

### 결론: 기계의 맥박을 기록하는 청진기
세상의 모든 사물이 인터넷에 연결되는 IoT 시대에, 그들이 뿜어내는 끝없는 데이터를 감당할 수 있는 유일한 그릇이 TSDB다. TSDB는 데이터를 시간의 순리대로 받아들이는 가장 자연스러운 데이터베이스다.

### 미래 전망
최근 TSDB는 클라우드 네이티브 아키텍처로 진화하며 컴퓨팅(Query)과 스토리지(S3 등)를 분리하는 **'서버리스 TSDB'** 형태로 발전하고 있다. 또한, 벡터 데이터베이스와 결합하여 단순 값의 변화뿐만 아니라 "이 시간대의 데이터 패턴이 과거 어떤 패턴과 유사한가?"를 실시간으로 찾아내는 **AI 모니터링 엔진**으로 진화할 전망이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **TSDB가 뭔가요?**: 자동차의 속도계나 환자의 심장 박동처럼 '시간에 따라 계속 변하는 숫자들'을 1초도 안 쉬고 아주 빨리 적어두는 특별한 일기장이에요.
2. **일반 일기장이랑 뭐가 다르나요?**: 일반 일기장은 중간에 지우개로 고치기도 하지만, 이 일기장은 절대 지우지 않고 무조건 맨 뒷장에 새로운 숫자만 미친 듯이 빨리 적어나간답니다.
3. **오래된 숫자가 너무 많아지면 어떡해요?**: 옛날 일기는 "이번 주 평균 속도는 이 정도였어"라고 아주 짧게 줄여서(요약해서) 보관하기 때문에 일기장이 터질 일이 없어요!
