# 281. 샤드 키 (Shard Key / Partition Key)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 데이터베이스(Sharding) 환경에서 수십~수백 대의 서버 중 특정 데이터가 저장될 '물리적 위치(노드)'를 결정짓는 기준이 되는 컬럼(속성)이다.
> 2. **가치**: 샤드 키를 어떻게 설계하느냐에 따라 트래픽이 모든 서버에 완벽히 분산될 수도 있고, 특정 서버 하나만 터지는 핫스팟(Hotspot) 재앙이 발생할 수도 있는 아키텍처 성공의 핵심 열쇠다.
> 3. **융합**: 해시(Hash), 레인지(Range), 디렉토리(Directory) 분배 알고리즘과 결합되며, 데이터 모델링 시 애플리케이션의 쿼리 패턴(Query Pattern) 분석과 융합되어 결정되어야 한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
샤드 키(Shard Key, 파티션 키라고도 함)는 거대한 테이블의 레코드를 여러 샤드(Shard) 서버로 분할할 때 사용하는 결정자(Determinant)이다. 데이터베이스 라우터(예: MongoDB의 `mongos`)는 클라이언트로부터 쿼리가 들어올 때 이 샤드 키의 값을 수학적 함수(해시 등)에 넣어, 데이터를 읽고 쓸 정확한 타겟 서버를 1000분의 1초 만에 식별해 낸다.

### 💡 비유
샤드 키는 **"전국 우체국의 우편번호(Zip Code)"**와 같다.
편지(데이터)가 우체국 집중국(DB 라우터)에 엄청나게 쏟아진다. 우체부는 편지의 모든 내용을 읽고 분류하지 않는다. 오직 편지 겉면에 적힌 '우편번호 5자리(Shard Key)'만 보고 "아, 이건 서울 강남구(Shard 1)로, 저건 부산 해운대구(Shard 2)로 보내면 되겠군!" 하고 기계적으로 즉시 분류(라우팅)한다. 만약 우편번호 체계를 잘못 만들면 서울 우체국에만 전국의 편지가 몰려 마비될 것이다.

### 등장 배경 및 발전 과정

#### 1. 샤딩의 필수 조건
단일 서버의 용량 한계(Scale-up)를 극복하기 위해 데이터를 여러 서버에 찢어놓는 샤딩(Scale-out)이 필수적이 되었다. 그러나 무작정 데이터를 던져놓으면 쿼리를 할 때마다 "내 데이터가 어느 서버에 있지?" 하고 100대의 서버를 전부 다 뒤져야 하는 브로드캐스트(Scatter-Gather) 오버헤드가 발생한다.

#### 2. 라우팅 나침반의 필요성
이 오버헤드를 막고 단 1대의 타겟 서버에만 쿼리를 꽂아 넣기(Targeted Query) 위해, 데이터의 소속을 증명하는 강력한 나침반 역할이 필요해졌고, 그것이 바로 모든 테이블 설계 시 1순위로 지정해야 하는 '샤드 키'이다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 좋은 샤드 키의 3대 조건 (카디널리티, 균등성, 쿼리 패턴) (표)

| 평가 지표 | 설명 및 원리 | 나쁜 예 (Anti-Pattern) | 좋은 예 (Best Practice) |
|-----------|--------------|------------------------|-------------------------|
| **1. 높은 카디널리티 (High Cardinality)** | 값의 종류가 얼마나 다양한가. 쪼갤 수 있는 최대 조각의 수를 결정함 | `성별(Gender)`: 남/여 2가지뿐이라 서버를 3대 이상 늘릴 수 없음 | `사용자 ID (User_ID)`: 수천만 개의 고유값을 가져 무한대 분산 가능 |
| **2. 균등한 분배 (Even Distribution)** | 특정 값에 데이터(트래픽)가 쏠리지 않고 고르게 퍼지는가 | `가입 연도`: 2023년 가입자가 10명이고, 2024년 가입자가 10만명이면 핫스팟 발생 | `주문 번호 (Order_ID)의 해시값` |
| **3. 쿼리 격리성 (Query Isolation)** | 자주 쓰는 쿼리가 1대의 샤드에서만 완결(Targeted)되는가 | 조회가 잦은 컬럼을 제외하고 시스템 내부 UUID만 샤드 키로 씀 | `상점 ID (Tenant_ID)`: 특정 상점의 쿼리는 무조건 1대의 서버에서 완결됨 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ Query Routing based on Shard Key ]
================================================================================
 
 [ Client Request ]
  UPDATE orders SET status = 'Shipped' 
  WHERE  order_id = 90210  AND  user_id = 'Alice55';  <-- (user_id가 샤드 키로 지정됨)

           │
           ▼
 ┌────────────────────────────────────────────────────────┐
 │                      DB Router                         │
 │                                                        │
 │  1. 쿼리 파싱: "어라, 샤드 키(user_id) 값이 'Alice55'네?" │
 │  2. 해시 계산: Hash("Alice55") = 0x8A2F                │
 │  3. 라우팅: 0x8A2F 범위는 Shard 2가 담당하고 있음!        │
 └─────────┬───────────────────┬────────────────────┬─────┘
           │                   │ Targeted Query!    │
           │ (무시)             ▼ (적중)             │ (무시)
     ┌─────▼─────┐       ┌─────▼─────┐        ┌─────▼─────┐
     │  Shard 1  │       │  Shard 2  │        │  Shard 3  │
     │(user: A~H)│       │(user: I~P)│        │(user: Q~Z)│
     └───────────┘       └───────────┘        └───────────┘
   * DB Router는 Shard 1, 3을 전혀 괴롭히지 않고, Shard 2에만 쿼리를 전달함.
   * 만약 WHERE 절에 user_id(샤드 키)가 없었다면? 
     -> Router는 데이터가 어딨는지 몰라 Shard 1, 2, 3 모두에게 쿼리를 뿌리는(Broadcast) 대참사 발생.
```

### 심층 동작 원리 (복합 샤드 키 설계)

단일 컬럼만으로는 좋은 샤드 키의 조건을 모두 맞추기 어렵다. 예를 들어 `상점 ID (Tenant_ID)`를 샤드 키로 쓰면 쿼리 격리성은 좋지만, 특정 유명 상점(예: 이마트)에만 트래픽이 몰려 해당 샤드 서버가 터지는 **핫스팟(Hotspot/Jumbo Shard)** 문제가 발생한다.

이를 막기 위해 **복합 샤드 키(Compound Shard Key)**를 사용한다.
- `[ Tenant_ID, Order_Month ]` 또는 `[ Tenant_ID, Hash(Order_ID) ]`
- 앞부분(Prefix)은 논리적으로 데이터를 그룹화하여 읽기 성능을 보장하고, 뒷부분(Suffix)은 트래픽을 해시로 흩뿌려 쓰기 병목을 방지하는 고도의 엔지니어링 전략이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 샤드 키 값의 특성에 따른 물리적 분할 방식 (Strategy)

| 분할 알고리즘 | 샤드 키의 특성 | 장점 | 단점 (치명적 약점) |
|---------------|----------------|------|--------------------|
| **해시(Hash) 분할** | 원본 값과 무관하게 무작위 난수로 변환 (`Hash(ID) % N`) | 무조건 데이터와 쓰기 트래픽이 N대 서버에 **가장 균등하게 1/N 씩 분배**됨 | 범위 검색(`A부터 B까지`)이 불가능하여 무조건 전체 스캔 브로드캐스트 발생 |
| **레인지(Range) 분할**| 값의 오름차순/내림차순 범위 유지 (`A~M`, `N~Z`) | 범위 검색 쿼리가 극도로 빠름. 연관 데이터가 같은 서버에 존재 | 단조 증가하는 키(`날짜`, `Auto-Increment PK`) 사용 시 100% 핫스팟(최신 샤드만 죽어남) 발생 |
| **디렉토리(Directory)**| 룩업 테이블 맵핑 (A=Shard1, B=Shard2) | 라우팅 규칙을 내 마음대로 동적으로 변경 가능 | 룩업 테이블(Config Server) 자체가 단일 장애점(SPOF) 및 병목 지점이 됨 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 초당 만 건의 접속 로그가 쌓이는 글로벌 게임 서비스. 샤딩을 처음 구축할 때 개발자가 아무 생각 없이 테이블의 기본키(PK)인 `Log_Sequence (Auto-Increment 정수)`를 샤드 키로, 레인지(Range) 샤딩을 걸어버림.
**현상 (장애 발생)**: 서버를 10대나 구축했는데, 항상 가장 마지막(최신) 번호가 들어가는 서버 10번의 CPU가 100%를 치고 뻗어버림. 나머지 1~9번 서버는 과거 데이터만 들고 놀고 있음 (Monotonically Increasing Key 패턴의 전형적 참사).
**판단 및 턴어라운드 전략**:
1. **샤드 키 재설계**: 단조 증가하는 값을 Range 샤드 키로 쓰는 것은 자살 행위다. 접속 로그의 쓰기 트래픽을 10대 서버에 흩뿌리기 위해, 로그가 발생한 **`User_ID`를 해시(Hash) 함수**에 통과시키는 방식으로 샤드 키를 전면 수정한다. (해시 샤딩)
2. **트레이드 오프(Trade-off) 수용**: 이렇게 하면 "오늘 들어온 전체 로그 보여줘" 같은 범위 쿼리는 10대 서버 전부에 날려야 해서 조금 느려지지만(Scatter-Gather), 일단 초당 만 건의 쓰기 폭주로 시스템이 다운되는 현상은 완벽히 막아낼 수 있다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **한 번 정한 샤드 키는 영원하다**: MongoDB 등의 NoSQL에서 한 번 샤드 키를 지정하고 데이터가 들어가기 시작하면, 샤드 키를 변경하는 것은 불가능에 가깝다 (데이터 전체를 Dump 뜨고 스키마를 부수고 다시 밀어 넣어야 함). 시스템 오픈 전 쿼리 패턴을 분석하여 신중히 결정해야 한다.
- **Cross-shard Transaction**: 서로 다른 샤드 키를 가진 데이터(예: User A의 지갑, User B의 지갑)를 묶어서 트랜잭션을 걸면, 분산 락(Two-Phase Commit)이 걸려 속도가 수십 배 느려진다. 한 트랜잭션으로 묶일 데이터는 무조건 **같은 샤드 키 값**을 갖도록 설계해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **처리량 분산**: 초당 10만 건의 트래픽을 10대의 서버에 정확히 1만 건씩 분배하여 하드웨어 성능의 **100% 선형적 효율** 달성.
- **디스크 I/O 최적화**: 쿼리에 샤드 키가 포함될 경우(Targeted Query), 나머지 90%의 디스크는 전혀 건드리지 않아 전체 시스템의 I/O 대역폭 낭비 극소화.

### 미래 전망 및 진화 방향
클라우드 네이티브 시대에는 개발자가 샤드 키의 Hash/Range 수식을 일일이 튜닝하던 고통이 사라지고 있다. **Google Cloud Spanner**나 **CockroachDB** 같은 NewSQL 시스템들은 내부적으로 머신러닝을 활용해 트래픽이 몰리는(Hotspot) 데이터 범위를 실시간으로 쪼개고(Split) 여러 노드로 이동(Rebalance)시키는 **동적 오토 샤딩(Dynamic Auto-sharding)** 체계로 진화하여 인프라의 투명성을 극대화하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **샤드 키란?**: 도서관에 수백만 권의 책이 100개의 방에 나뉘어 있을 때, "어느 방으로 가야 할지" 단 1초 만에 알려주는 마법의 팻말이에요.
2. **어떻게 정하나요?**: 책 제목의 첫 글자(가, 나, 다)로 정할 수도 있고, 책을 지은 작가 이름으로 정할 수도 있어요. 이 기준을 잘 정해야 방마다 책이 꽉꽉 골고루 차요.
3. **잘못 정하면 어떻게 되나요?**: 기준을 '새로 나온 책'으로 정해버리면, 모든 아이들이 최신 책이 있는 100번 방으로만 우르르 몰려가서 문이 터져버리겠죠? (핫스팟 문제!)
