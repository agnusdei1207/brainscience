# 282. 해시 샤딩 (Hash Sharding) vs 레인지 샤딩 (Range Sharding)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 데이터베이스에서 데이터를 여러 노드(Shard)에 나누어 담을 때, 샤드 키(Shard Key)를 해시 함수로 변환하여 무작위로 분배할 것인지(해시), 아니면 값의 연속된 순서를 유지하여 분배할 것인지(레인지)를 결정하는 양대 핵심 알고리즘이다.
> 2. **가치**: 해시 샤딩은 트래픽을 모든 서버에 100% 고르게 분산하여 특정 서버가 마비되는 핫스팟(Hotspot) 문제를 원천 차단하지만, 레인지 샤딩은 연관된 데이터가 같은 서버에 모여 있어 범위 검색(A~Z) 시 극강의 속도를 자랑한다.
> 3. **융합**: 어느 한 쪽의 완벽한 승리는 없으므로, 데이터의 증가 패턴(단조 증가 여부)과 애플리케이션의 핵심 쿼리 패턴(단건 조회 vs 범위 조회)을 융합적으로 분석하여 치밀하게 선택해야 하는 트레이드오프(Trade-off)의 정점이다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
데이터베이스 샤딩 아키텍처에서 가장 중요한 결정은 "이 레코드를 도대체 어느 서버(Shard)에 넣을 것인가?"이다. 이를 결정하는 분배 규칙이 파티셔닝 전략이며, **해시 샤딩(Hash Sharding)**은 샤드 키를 해시 함수에 통과시켜 난수화하여 분산시키는 방식이고, **레인지 샤딩(Range Sharding)**은 샤드 키 값의 연속적인 구간(예: 1~1000, 1001~2000)을 나누어 서버를 배정하는 방식이다.

### 💡 비유
- **해시 샤딩**: "카드 섞기 후 딜러가 무작위로 나눠주기". 카드의 숫자가 연속되든 아니든, 딜러가 4명의 플레이어에게 완벽하게 똑같은 장수의 카드를 무작위로 나눠준다. 4명 모두 일거리가 똑같다.
- **레인지 샤딩**: "도서관의 십진분류법 책장". 철학 책은 1번 방, 역사 책은 2번 방, 문학 책은 3번 방에 넣는다. '철학'에 관한 책을 찾고 싶을 땐 1번 방만 가면 다 모여 있어서 무척 빠르다. 하지만 최근 '문학' 책만 미친 듯이 출판된다면 3번 방만 터져나갈 것이다.

### 등장 배경 및 발전 과정

분산 DB 초창기, 사람들은 직관적인 레인지 샤딩을 선호했다. (A~M 사용자는 1번 서버, N~Z는 2번 서버). 그러나 시스템이 운영되다 보니 특정한 성 씨(예: Kim, Lee)나 특정 날짜(오늘 데이터)에 트래픽이 90% 이상 몰리며 서버 하나가 박살 나는 **핫스팟(Hotspot) 문제**가 속출했다. 이를 해결하기 위해 데이터의 의미를 파괴하고 수학적으로 완벽한 균등 분배를 보장하는 해시 샤딩 기법(예: 일관된 해싱)이 대두되며 현재 NoSQL(Cassandra, DynamoDB 등)의 기본 전략으로 자리 잡았다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 해시 vs 레인지 동작 메커니즘 (비교표)

| 비교 지표 | 해시 샤딩 (Hash Sharding) | 레인지 샤딩 (Range Sharding) |
|-----------|--------------------------|------------------------------|
| **분배 기준** | `Hash(Key) % Node_Count` (난수화) | 연속적인 값의 구간 (예: 2023년, 2024년) |
| **데이터 분포** | 완벽에 가까운 균등 분배 (Uniform) | 데이터 쏠림 발생 가능성 매우 높음 (Skewed) |
| **핫스팟(Hotspot)**| 근본적으로 방지됨 | 키 설계에 따라 빈번하게 발생 (예: Auto-Increment) |
| **범위 검색(Range Query)**| **최악** (모든 샤드 서버에 쿼리를 다 날려야 함) | **최상** (특정 구간 데이터를 가진 1대 서버에서 해결) |
| **단건 조회(Point Query)**| 최상 (해시 연산 1번으로 즉시 타겟 서버 탐색) | 우수 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ 1. Hash Sharding: Hotspot 완벽 방어, 범위 검색 실패 ]
================================================================================
 [ Write Data (Logs) ] 
 ID: 101, 102, 103 (단조 증가하는 값)

        (해시 함수 통과: 데이터의 순서가 완전히 파괴됨!)
        Hash(101) = 0x8A -> Shard 1로
        Hash(102) = 0x1F -> Shard 3으로
        Hash(103) = 0x5C -> Shard 2로

  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐
  │   Shard 1   │   │   Shard 2   │   │   Shard 3   │
  │ [ID: 101]   │   │ [ID: 103]   │   │ [ID: 102]   │  <- 데이터가 1개씩 사이좋게 나뉨!
  └─────────────┘   └─────────────┘   └─────────────┘
  * 치명적 단점: "ID 101~103까지 가져와!" (범위 쿼리) -> Router가 Shard 1, 2, 3 모두에게
    질문(Scatter)하고 응답을 모아야(Gather) 해서 엄청난 I/O 낭비 발생.

================================================================================
[ 2. Range Sharding: 범위 검색 최강, Hotspot 재앙 ]
================================================================================
 [ Write Data (Logs) ] 
 ID: 101, 102, 103 (단조 증가하는 값)

        (구간 설정: Shard1(1~50), Shard2(51~100), Shard3(101~150))
        ID가 모두 101~150 구간이므로 전부 Shard 3으로 직행!

  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐
  │   Shard 1   │   │   Shard 2   │   │   Shard 3   │
  │ (과거 데이터) │   │ (어제 데이터) │   │ [ID: 101]   │  <- 트래픽 100% 몰림 (Hotspot!)
  │ (놀고 있음)  │   │ (놀고 있음)  │   │ [ID: 102]   │     이 서버의 CPU 100% 치고 터짐!
  └─────────────┘   └─────────────┘   │ [ID: 103]   │
                                      └─────────────┘
  * 장점: "ID 101~103 가져와!" -> Shard 3에만 물어보면 1초 만에 연속으로 읽어옴.
```

### 심층 동작 원리: 핫스팟과 단조 증가 키의 저주

RDBMS 설계자들은 습관적으로 기본 키(PK)를 `Auto-Increment(자동 증가 정수)`나 `Timestamp(시간)`로 잡는다. 이는 값이 계속 한 방향으로 커지는 단조 증가(Monotonically Increasing) 특성을 가진다. 
만약 이 단조 증가 키를 **레인지 샤딩**의 샤드 키로 지정하면 대재앙이 발생한다. 데이터는 무조건 "가장 최근 구간"을 담당하는 단 1대의 노드(위 그림의 Shard 3)로만 쏟아져 들어온다. 개발자는 서버를 10대로 분산(Scale-out)시켰다고 뿌듯해하지만, 실제 쓰기 트래픽은 최신 노드 1대가 100% 독박을 쓰고 나머지 9대는 과거 데이터를 안고 놀게 된다. 이를 막기 위해서는 무조건 **해시 샤딩**을 쓰거나, 샤드 키를 난수(UUID)로 변경해야 한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 주요 데이터베이스들의 기본 샤딩 전략 채택 현황

| 데이터베이스 | 주력 권장 전략 | 이유 |
|--------------|----------------|------|
| **Cassandra / DynamoDB** | **Hash Sharding** (Consistent Hashing) | 분산 P2P 아키텍처 특성상, 완벽한 부하 분산과 노드 무중단 추가가 지상 과제이기 때문. 범위 검색은 약점. |
| **HBase** | **Range Sharding** (Region 분할) | 하둡 생태계 특성상 특정 시간대/키 범위의 대용량 연속 스캔(Scan) 분석이 주 목적이기 때문. |
| **MongoDB** | **혼용 지원** (개발자 선택) | 컬렉션 특성에 따라 해시를 쓸지 레인지를 쓸지 인덱스 생성 시 선언. |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대규모 이커머스에서 `Order(주문)` 테이블을 샤딩하려 한다. 요구사항은 두 가지다.
(A) 초당 수만 건의 결제 트래픽을 5대의 DB 서버가 골고루 나눠서 버텨야 한다. (쓰기 부하 분산)
(B) 특정 사용자(User_ID)가 "내 주문 내역 1년 치 보여줘" 할 때 엄청나게 빨라야 한다. (연관 데이터 범위 검색)
**판단 (하이브리드/복합 샤드 키 적용)**:
1. 단순 `Order_ID(순차 번호)`로 레인지 샤딩 -> 특정 서버 폭발 (기각)
2. `Order_ID`로 해시 샤딩 -> 부하는 분산되지만, 유저가 내 주문내역을 볼 때 5대 서버를 다 뒤져야 함 (기각)
3. **해법 (Compound Shard Key)**: **`[ User_ID (해시) + Order_Date (레인지) ]`** 로 복합키 설계.
   - 첫 번째 기준인 `User_ID`에 해시를 걸어 전체 유저를 5대 서버에 완벽히 균등하게 1/5씩 찢어 놓는다. (A 달성)
   - 한 번 특정 서버에 배정된 1명의 유저는, 그 안에서 `Order_Date`라는 두 번째 기준으로 시간순으로 정렬(레인지)되어 저장된다. 
   - 결과적으로 유저가 자기 주문 내역을 부르면 라우터는 단 1대의 서버로만 쿼리를 쏘고(Targeted), 그 서버 안에서는 연속된 디스크 공간을 스와이프하여 데이터를 긁어오므로 범위 검색이 극대화된다 (B 달성).

### 주의사항 및 안티패턴 (Anti-patterns)
- **해시 알고리즘 변경의 공포**: 해시 샤딩을 쓸 때 단순 모듈러 연산(`Hash(ID) % 3대`)을 썼다가 노드 한 대를 추가해 4대가 되면, 100%의 데이터가 새롭게 해시(`% 4대`)되어 모든 서버 간에 끔찍한 데이터 대이동(Rebalancing Storm)이 발생한다. 해시 샤딩 시에는 반드시 노드 변동에도 데이터 이동이 최소화되는 **일관된 해싱(Consistent Hashing)** 토폴로지를 적용해야 한다.

## Ⅴ. 기대효과 및 결론

### 아키텍처적 결론 (Trade-off)
분산 환경에서 데이터 균등 분배(Hash)와 범위 검색 속도(Range)를 동시에 100% 만족하는 마법은 존재하지 않는다 (데이터 베이스 물리 법칙의 한계).
- **무조건 쓰기(Write) 폭주를 막아야 한다면 -> 해시 샤딩**
- **과거 데이터의 대규모 집계/분석(Read)이 중요하다면 -> 레인지 샤딩**

### 미래 전망 및 진화 방향
클라우드 네이티브의 NewSQL DB(Spanner, CockroachDB 등) 진영에서는 단순한 정적 해시/레인지를 넘어, 시스템이 내부적으로 **자동 분할(Auto-splitting)**을 수행한다. 평소에는 레인지 샤딩으로 연속성을 유지하다가, 특정 데이터 구간(예: 방금 출시된 아이폰 판매 기록)에 트래픽이 폭주하면, 시스템이 실시간으로 그 뜨거운(Hot) 데이터 범위의 레인지를 잘게 쪼개어 다른 서버로 이주(Live Migration)시키는 자율 주행형 샤딩 시대로 발전하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **해시 샤딩은요**: 선생님이 반 친구들에게 번호표를 뽑게 해서, 1조, 2조, 3조에 아주 공평하게 똑같은 인원수만큼 나눠서 배정하는 거예요. 아무 조나 일이 몰리지 않아 좋아요!
2. **레인지 샤딩은요**: 키가 150~160인 친구들은 1번 방, 160~170은 2번 방으로 묶는 거예요. "160대 친구들 다 모여!"라고 부를 때 방 하나만 가면 되니까 엄청 찾기 쉬워요.
3. **단점은 없나요?**: 해시 방식은 친구들이 뿔뿔이 흩어져서 키가 비슷한 친구를 한 번에 부르기 힘들고요, 레인지 방식은 갑자기 160대 친구만 잔뜩 전학 오면 2번 방이 좁아서 터져버리는 단점이 있답니다.
