# 283. 일관된 해싱 (Consistent Hashing)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 저장소(NoSQL, 캐시 서버)에서 데이터를 여러 노드에 해시 분할할 때, 해시 공간을 거대한 논리적 원형(Ring)으로 배열하여 데이터와 노드를 같은 공간에 매핑하는 알고리즘이다.
> 2. **가치**: 일반 모듈러 연산(`Hash % N`)의 치명적 단점인 '노드 증설/삭제 시 전체 데이터 재배치 폭풍(Rehashing Storm)'을 막고, 오직 인접한 노드의 데이터(전체의 $1/N$)만 이동하게 하여 무중단 스케일 아웃을 보장한다.
> 3. **융합**: 가상 노드(Virtual Node) 개념과 결합하여 데이터의 쏠림(Hotspot)을 방지하며, Amazon DynamoDB, Apache Cassandra, 분산 캐시(Memcached) 아키텍처의 근본적인 데이터 분배 토대로 융합된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
일관된 해싱(Consistent Hashing)은 분산 해시 테이블(DHT)을 구현하는 기술 중 하나다. 출력 범위가 고정된(예: 0 ~ $2^{32}-1$) 해시 함수를 사용하여, 데이터를 저장할 서버(Node)의 주소와 데이터의 키(Key)를 모두 동일한 해시 링(Ring) 위에 배치한다. 데이터는 링 위에서 자신의 위치로부터 시계 방향으로 돌아 가장 먼저 만나는 서버에 저장되는 우아한 수학적 규칙을 따른다.

### 💡 비유
일관된 해싱은 **"원탁의 수건돌리기 게임"**과 같다.
커다란 원탁(해시 링)에 3명의 술래(서버 A, B, C)가 일정한 간격으로 서 있다. 테이블 중앙에서 수백 개의 공(데이터)이 굴러 나온다. 공이 멈춘 자리에서 시계 방향으로 가장 가까이 서 있는 술래가 그 공을 주워서 보관한다. 만약 술래 1명(D)이 원탁 사이에 새로 끼어들었다고 치자. 테이블 위의 모든 공을 처음부터 다시 주울 필요 없이, 새로 끼어든 D의 바로 앞부분에 떨어진 공들만 D가 가져가고 나머지 A, B, C가 가진 공들은 그대로 두면 된다.

### 등장 배경 및 발전 과정

#### 1. 전통적 해시(Modular Hashing)의 대참사
과거에는 캐시 서버를 분산할 때 `Hash(Key) MOD N(서버 대수)` 방식을 썼다. 서버가 3대면 N=3이다. 그런데 트래픽이 폭주하여 서버 1대를 추가(N=4)하는 순간 재앙이 발생했다. 분모가 3에서 4로 바뀌면서 모든 데이터의 나머지(위치)값이 완전히 틀어져 버렸고, 기존 캐시 데이터의 99%가 미스(Cache Miss) 처리되며 원본 DB로 엄청난 트래픽 폭풍(Thundering Herd)이 쏟아져 시스템이 붕괴했다.

#### 2. Karger의 논문과 Dynamo의 상용화
1997년 MIT의 David Karger 등이 분산 웹 캐싱을 위해 'Consistent Hashing' 논문을 발표하며 이 문제를 수학적으로 해결했다. 노드의 수(N)에 의존하지 않는 링 토폴로지를 고안한 것이다. 이후 2007년 아마존이 Dynamo 논문에서 이 개념을 스토리지 복제와 결합하여 대규모 클라우드 NoSQL의 핵심 아키텍처로 상용화하며 현대 분산 시스템의 바이블이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (표)

| 요소명 | 상세 역할 | 내부 동작 메커니즘 | 비유 |
|--------|----------|-------------------|------|
| **Hash Ring (해시 링)** | 전체 데이터 공간 | $0$부터 $2^{32}-1$까지의 값을 가진 논리적인 원형 궤도 | 360도 룰렛 판 |
| **Node Hash (서버 위치)**| 서버를 링에 배치 | 서버의 IP나 MAC 주소를 해시하여 링 위 특정한 점(좌표)에 고정시킴 | 룰렛 판 위의 바구니 |
| **Data Hash (데이터 위치)**| 데이터를 링에 배치 | 저장할 데이터의 Key를 동일한 해시 함수로 돌려 링 위에 점을 찍음 | 룰렛 판에 던져진 구슬 |
| **Routing Rule** | 데이터 소속 결정 | 데이터 위치에서 시계 방향으로 이동하며 가장 처음 만나는 노드에 저장 | 시계 방향 탐색 |
| **Virtual Node (VNode)** | 균등 분배(쏠림 방지) | 물리 서버 1대당 수십~수백 개의 가상 노드 해시값을 링 위에 흩뿌림 | 바구니 여러 개로 쪼개기 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ 1. Consistent Hashing Ring & Data Placement ]
================================================================================
               (Hash: 0)
                   │
           [ Data 1 ]  (Hash: 100)
             /           \     (시계방향 ─▶)
            /             \
 (Hash:     |             |
  800)   [ Node C ]       [ Node A ] (Hash: 250)
            |             |   <-- Data 1은 Node A에 저장됨
            \             /
             \           / 
              [ Node B ] 
              (Hash: 500)
                  │
             [ Data 2 ] (Hash: 600)  <-- Data 2는 시계 방향으로 돌아 Node C에 저장됨

================================================================================
[ 2. The Power of Consistent Hashing (노드 추가 시 데이터 재배치 최소화) ]
================================================================================
 * 만약 Node A와 Node B 사이에 **새로운 [ Node D ] (Hash: 350)**를 추가한다면?
 
 1. 기존 데이터 매핑: (0~250)->A, (251~500)->B, (501~800)->C, (801~999)->A
 2. Node D 추가 후: (0~250)->A, **(251~350)->D**, (351~500)->B ...
 
 * 결과: Node B가 가지고 있던 데이터 중 앞부분(251~350)만 Node D로 이동하면 끝난다!
   Node A와 Node C에 있는 데이터는 1바이트도 건드릴 필요가 없다. (이동 최소화)
```

### 심층 동작 원리

#### ① 데이터 쏠림 현상과 '가상 노드(Virtual Nodes)'의 도입
순수한 일관된 해싱에도 치명적인 약점이 있다. 해시 함수 특성상 노드 A, B, C가 링 위에 예쁘고 균등한 간격(120도)으로 배치되지 않고 우연히 한곳에 몰려 찍힐 수 있다. 이 경우 간격이 넓은 특정 노드 하나가 전체 데이터의 70%를 독박 쓰는 **데이터 편중(Data Skew)**이 발생한다.
이를 해결하기 위해 1대의 물리 서버(Node A)를 링 위에 1번만 찍는 게 아니라, 해시 함수에 소금(Salt)을 쳐서 `Hash(NodeA_1)`, `Hash(NodeA_2)`... 형식으로 100번에서 250번 정도의 가짜 점(Virtual Node, 토큰)을 링 전체에 고르게 흩뿌린다. 결과적으로 데이터는 통계적으로 완벽에 가깝게 N등분 된다.

#### ② 노드 증설과 성능 격차 보정
서버 3대가 있는데, 새로 산 최신 서버를 추가한다고 가정하자. 최신 서버는 기존 서버보다 처리 용량이 2배 크다. 이때 가상 노드 기법을 응용하면, 구형 서버는 가상 노드를 100개씩만 링에 배치하고, 신형 서버는 200개를 배치한다. 그러면 신형 서버의 면적이 넓어져 자연스럽게 전체 트래픽의 더 많은 비중(가중치)을 흡수하게 되는 고도의 유연성을 확보할 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### Modular Hashing vs Consistent Hashing 수학적 비용 비교

| 시나리오 | Modular Hashing `(Hash % N)` | Consistent Hashing |
|----------|------------------------------|--------------------|
| **서버 10대 -> 1대 추가 (11대)** | 분모가 10에서 11로 바뀜. 전체 데이터의 **약 91% (10/11) 가 다른 위치로 이동** | 전체 데이터의 **단 9% (1/11) 만 새로운 노드로 이동** |
| **서버 10대 -> 1대 고장 (9대)** | 분모가 9로 바뀜. 데이터의 **약 89% 이동**. 시스템 연쇄 붕괴 확률 매우 높음 | 고장난 노드의 데이터 10%만 시계 방향의 다음 노드들로 분산 흡수됨 |
| **운영 확장성 (Scale-out)** | 다운타임을 감수하는 대작업 (빅뱅 마이그레이션 필요) | 런타임 중 무중단 점진적 증설/축소 가능 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대규모 사용자 세션(Session) 정보를 5대의 Redis 서버에 파티셔닝하여 저장 중이다. 이벤트가 터져 트래픽이 폭주, 서버를 8대로 긴급 증설(Scale-out)해야 한다. 단순 모듈러(`% 5`) 방식을 사용 중이었다.
**판단 및 발생할 재앙**:
단순 모듈러 상태에서 서버를 8대로 늘리면, 기존 세션 정보의 80% 이상을 캐시 미스(Cache Miss)로 잃어버리고 백엔드 RDBMS로 직접 쿼리가 쏟아져 DB가 죽는다(Cache Stampede).
**아키텍처 전환 전략**:
1. 애플리케이션의 Redis 클라이언트(예: Jedis, Lettuce) 설정이나 중간 프록시(Envoy, Twemproxy)에서 라우팅 로직을 **일관된 해싱(Ketama 알고리즘 등)**으로 전면 교체한다.
2. 서버 증설: 5대에서 8대로 늘리면, 기존 데이터의 5/8는 유지되고 나머지 3/8의 데이터만 빈 공간(새 서버)으로 리밸런싱된다.
3. **가상 노드 증폭**: 추가로, 장애 시 특정 노드에 부하가 집중되는 것을 막기 위해 `vnode` 설정을 최소 160 이상으로 주어 링의 밀도를 촘촘하게 구성한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **가상 노드의 메모리 낭비**: 가상 노드(vnode)를 많이 만들수록 데이터 분배는 완벽에 가까워지지만, 클러스터를 관리하는 마스터나 클라이언트가 이 방대한 라우팅 맵(링 테이블)을 메모리에 들고 있어야 한다. vnode가 수백만 개 단위로 커지면 맵 갱신 및 탐색 오버헤드가 발생하므로 적절한 타협점(보통 노드당 256개 내외)을 찾아야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **데이터 이동 최소화**: 클러스터 노드 증감 시 전체 데이터 수 $K$, 노드 수 $N$일 때 이동하는 데이터의 개수를 수학적으로 $O(K/N)$으로 제한하여 재배치 비용 극소화.
- **가용성 보장**: 단일 노드 장애 시, 그 부하가 하나의 다른 노드로 몰리지 않고 링 위에 흩뿌려진 가상 노드들에 의해 나머지 모든 정상 노드로 N분의 1씩 분산 흡수되어 연쇄 장애(Cascading Failure) 차단.

### 미래 전망 및 진화 방향
일관된 해싱은 Apache Cassandra, Amazon DynamoDB, Riak 같은 리더 없는(Masterless) 분산 데이터베이스의 기본 골격을 완성했다. 최근에는 링 구조를 넘어 데이터를 고정된 파티션 단위(Hash Slot)로 미리 쪼개놓고(예: Redis Cluster의 16384개 슬롯), 노드 간에 슬롯 단위로 덩어리째 이주시키는 **해시 슬롯(Hash Slot)** 방식이 융합되어 운영 편의성과 상태 관리의 직관성을 더욱 높여나가고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **일관된 해싱이란?**: 커다란 원탁 주변에 바구니 3개를 놓고, 구슬을 굴려 멈춘 곳에서 가장 가까운 바구니에 구슬을 담는 마법의 정리법이에요.
2. **왜 옛날 방식(나누기)보다 좋나요?**: 옛날 방식은 바구니를 하나 추가하면 그동안 정리한 모든 구슬을 바닥에 쏟고 처음부터 다시 담아야 해서 너무 힘들었어요.
3. **무엇이 편해졌나요?**: 이 마법은 새 바구니를 원탁 빈자리에 그냥 슬쩍 놓기만 하면, 그 자리 근처에 있던 몇 개의 구슬만 새 바구니로 옮기면 되니까 정리가 눈 깜짝할 새 끝난답니다!
