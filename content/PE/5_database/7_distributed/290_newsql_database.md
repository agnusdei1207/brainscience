# 290. NewSQL 데이터베이스 (RDB의 ACID + NoSQL의 확장성 결합)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 전통적인 관계형 데이터베이스(RDBMS)가 자랑하는 완벽한 데이터 무결성(ACID 트랜잭션, SQL)과, NoSQL이 자랑하는 클라우드 환경에서의 무한한 수평적 확장성(Scale-out)을 하나의 엔진에 융합한 차세대 데이터베이스 패러다임이다.
> 2. **가치**: 마이크로서비스(MSA) 분산 환경에서 골치를 앓던 '분산 트랜잭션'과 '최종적 일관성(Eventual Consistency)으로 인한 데이터 틀어짐' 문제를 해결하여, 개발자가 복잡한 보상 트랜잭션 로직 없이도 맘 편히 SQL을 짤 수 있게 해 준다.
> 3. **융합**: 분산 합의 알고리즘(Raft, Paxos), 원자 시계/GPS 기반의 시각 동기화(TrueTime), 그리고 컴퓨팅과 스토리지의 물리적 분리 아키텍처가 융합된 분산 시스템 공학의 결정체이다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
NewSQL은 구글 스패너(Google Spanner), 칵로치DB(CockroachDB), TiDB, 볼트DB(VoltDB) 등으로 대표되는 분산 관계형 데이터베이스 시스템이다. NoSQL처럼 값싼 상용 서버(Commodity Server) 수십~수백 대를 클러스터로 묶어 무중단 확장을 하면서도, 겉보기에는 완벽한 단일 오라클/MySQL 데이터베이스처럼 동작하며 표준 SQL을 100% 지원한다.

### 💡 비유
NewSQL은 **"전국 지점을 하나로 묶은 초고속 양자 통신 은행 망"**과 같다.
- **기존 RDBMS (단일 은행 금고)**: 장부가 한 곳에 있어 완벽하게 안전하지만, 손님이 수백만 명이 몰리면 금고 문이 터져버린다(Scale-up의 한계).
- **NoSQL (각 지점의 독자적 장부)**: 손님이 1억 명 와도 전국의 지점이 나눠서 처리하니 엄청 빠르다. 하지만 서울에서 입금한 사실이 부산 지점에 전달되는 데 3초가 걸려, 그 3초 사이에 잔고가 틀릴 위험이 있다(최종적 일관성).
- **NewSQL**: 전국 지점이 분산되어 손님을 무한히 받으면서도, 기적의 통신망(분산 합의)을 통해 서울에서 돈을 넣는 즉시 부산 지점 장부에도 완벽하게 동시 기록(ACID 보장)된다.

### 등장 배경 및 발전 과정

#### 1. NoSQL의 눈물과 CAP 정리의 타협
2010년대, 대형 IT 기업들은 폭증하는 트래픽을 막기 위해 RDBMS를 버리고 NoSQL(Cassandra, MongoDB)로 피난을 갔다. 하지만 결제나 금융 시스템 등 돈이 오가는 서비스에서는 NoSQL의 '언젠가는 일치하겠지(Eventual Consistency)'라는 철학을 용납할 수 없었다. 개발자들은 DB가 못해주는 정합성을 맞추기 위해 애플리케이션 코드에 수백 줄의 사가(Saga) 패턴과 예외 처리 로직을 구겨 넣으며 끔찍한 고통을 겪었다.

#### 2. 구글 Spanner 논문의 충격
이 딜레마를 해결하기 위해, 2012년 구글은 데이터센터 간 수평 확장이 가능하면서도 글로벌 단위의 강력한 ACID 트랜잭션을 보장하는 **Spanner(스패너)** 시스템의 논문을 발표했다. 구글은 소프트웨어적 한계를 극복하기 위해 물리적인 원자시계(Atomic Clock)와 GPS를 데이터센터에 박아 넣는 기행을 선보였고, 이것이 NewSQL 생태계 폭발의 신호탄이 되었다.

| **Global Clock** | 분산 트랜잭션 순서 보장 | 노드 간 시간 오차로 인한 덮어쓰기 오류를 막기 위해 H/W(원자시계)나 정교한 S/W(HLC) 논리 시계 사용 | TrueTime, Hybrid Logical Clock | 전 세계 절대 동기화 시계 |

  │   2. 트랜잭션 코디네이터 역할 (어디에 저장되어 있는지 확인)   │
  └────────┬────────────────────────────────┬──────────────┘
           │ (내부 gRPC/RPC 통신)            │
           ▼                                ▼
  ┌──────────────────┐             ┌──────────────────┐
  │ Storage Node A   │             │ Storage Node B   │
  │ (Data Range:1~50)│             │(Data Range:51~99)│
  │                  │             │                  │
  │   [ Raft Leader ] <─(동기화 복제)─> [ Raft Follower] │
  │    (Commit 결정)  │             │    (Ack 응답)     │
  │   [ RocksDB K-V ]│             │   [ RocksDB K-V ]│
  └──────────────────┘             └──────────────────┘

     3대 중 2대(과반수)가 "저장 완료!"라고 응답(Ack)해야만 최종 Commit을 치고 Client에 성공 반환.

### 심층 동작 원리

#### ① 컴퓨팅과 스토리지의 분리 (Decoupling)
전통 DB는 쿼리 연산(CPU)과 데이터 저장(Disk)이 하나의 거대한 프로세스 안에 묶여 있었다. NewSQL은 쿼리를 파싱하고 실행 계획을 세우는 **Compute 레이어**와 실제 데이터를 분산 저장하는 **Storage 레이어**를 물리적으로 분리했다. 연산이 빡세면 Compute 노드만 늘리고, 용량이 부족하면 Storage 노드만 끼워 넣으면 되는 완전한 클라우드 네이티브 아키텍처다.

#### ② Raft 합의 알고리즘에 의한 ACID 보장
기존 NoSQL의 복제(Replication)는 마스터가 먼저 Commit을 때리고 나중에 슬레이브에 비동기로 전달(Asynchronous)하여 유실 위험이 있었다. NewSQL은 **Raft 알고리즘**을 사용하여 트랜잭션 발생 시 3대의 복제본 중 2대(Majority)가 디스크에 쓰는 것을 확인한 후(Synchronous)에야 사용자에게 성공(OK)을 반환한다. 노드 1대가 불타 없어져도 데이터 유실은 수학적으로 0%다.

#### ③ 논리적 클럭과 외부 일관성 (External Consistency)
분산된 서버들은 네트워크 지연 때문에 시스템 시계가 미세하게 다르다. 서울 서버의 12시 00분 01초와 뉴욕 서버의 01초는 완벽히 일치하지 않는다. 이 오차 때문에 A가 돈을 뺀 트랜잭션보다 B가 돈을 넣은 트랜잭션이 과거로 기록되는 심각한 오류가 발생한다. 구글은 위성 GPS와 원자시계(TrueTime)로 이 오차를 1ms 이내로 통제했고, 칵로치DB는 소프트웨어적인 하이브리드 논리 시계(HLC)를 발명하여 분산 환경에서의 트랜잭션 발생 순서(직렬화, Serializable)를 완벽히 꿰맞추었다.

### 3대 데이터베이스 패러다임 비교

| 특성 | 1세대: RDBMS (Oracle, MySQL) | 2세대: NoSQL (MongoDB, Cassandra) | 3세대: NewSQL (Spanner, CockroachDB) |
|------|------------------------------|-----------------------------------|--------------------------------------|
| **데이터 모델** | 정형화된 관계형 테이블 | JSON, Key-Value 등 비정형 | **정형화된 관계형 테이블 (표준 SQL)** |
| **트랜잭션(ACID)**| 완벽한 ACID 보장 (최상) | 단일 행/문서만 제한적 보장 (최종적 일관성) | **글로벌 분산 환경에서 완벽한 ACID 보장** |
| **확장성** | 수직 확장(Scale-up), 쓰기 분산 불가 | 무한 수평 확장(Scale-out) | **무한 수평 확장(Scale-out)** |
| **설계 철학** | "데이터의 정합성이 생명이다" | "정합성을 버리고 속도와 확장을 취하자" | **"정합성과 확장성, 둘 다 포기할 수 없다"** |
| **단점 (오버헤드)**| 트래픽 폭주 시 한계 봉착 | 개발자가 애플리케이션 코드로 예외 처리 필수 | **분산 합의 과정으로 인해 쓰기(Write) 지연 시간이 약간 높음** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 급성장하는 핀테크 유니콘 기업. 기존 MySQL 레플리케이션(Master-Slave) 구조에서 마스터 노드의 CPU가 100%를 치며 결제 쓰기(Write) 지연이 발생함. MySQL을 샤딩(Sharding)하려니 조인(JOIN)을 다 깨야 하고 개발 코드를 수만 줄 뜯어고쳐야 하는 재앙이 예상됨.
**판단 및 턴어라운드 전략 (NewSQL 전환)**:
1. **수동 샤딩(Sharding) 기각**: 애플리케이션단에서 샤딩 키를 계산하고 분산 트랜잭션(Saga)을 짜는 개발 리스크가 너무 크다.
2. **TiDB 또는 CockroachDB 도입**: MySQL/PostgreSQL과 와이어 프로토콜(Wire Protocol)이 100% 호환되는 NewSQL 클러스터를 도입한다.
3. **아키텍처 적용**: 앱 코드(JDBC 연결)를 전혀 수정하지 않고, 단지 DB 접속 엔드포인트를 NewSQL 라우터로 변경한다.
4. **결과**: 트래픽이 2배로 뛰면 그냥 저렴한 클라우드 VM(노드)을 클러스터에 버튼 클릭으로 2대 추가하기만 하면 된다. DB 엔진이 알아서 백그라운드에서 데이터를 조각내어 새 서버로 분산(Rebalancing)시키며 쓰기 부하를 완벽하게 분산한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **"모든 게 다 빠른 마법의 지팡이"라는 착각**: NewSQL은 NoSQL(Redis 등)처럼 초당 수백만 건의 밀리초(ms) 이하 응답을 내는 캐시 메모리가 아니다. 완벽한 ACID 트랜잭션을 위해 3대 이상의 노드 간에 네트워크 합의(Raft)를 거쳐야 하므로, 1건의 Insert를 할 때 기존 단일 MySQL보다 레이턴시(지연 시간)가 오히려 2~3배(수 밀리초) 더 걸릴 수 있다. NewSQL의 강점은 '단건의 속도'가 아니라, **'아무리 트래픽이 쏟아져도 노드만 늘리면 선형적으로 버텨내는 엄청난 처리량(Throughput)과 불변의 정합성'**에 있음을 인지해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **운영 비용(Ops) 제로화**: 수동 파티셔닝, 샤딩, 페일오버(Failover) 스크립트 등 골치 아픈 DBA 수작업이 엔진 자체의 자동화에 의해 100% 소멸.
- **가용성(High Availability)**: 활성-활성(Active-Active) 다중 마스터 구조이므로, 리전(Region) 단위의 화재가 발생해 데이터센터가 날아가도 RTO(복구 시간 목표)가 0초(무중단)에 수렴함.

### 미래 전망 및 진화 방향
초기 NewSQL은 단순한 분산 트랜잭션 처리에 머물렀으나, 최근에는 **HTAP (Hybrid Transactional/Analytical Processing)**으로 진화하고 있다. TiDB와 같은 현대 NewSQL 엔진은 하나의 클러스터 내부에 실시간 결제용 로우(Row) 기반 스토리지와 분석용 컬럼(Column) 기반 스토리지를 동시에 유지하며, 트랜잭션이 들어오는 즉시 분석용 DB로 복제하여 실시간 빅데이터 분석까지 단 하나의 쿼리로 끝내버리는 데이터베이스 최후의 완전체 모델로 나아가고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **기존 은행(RDBMS)과 동네 구멍가게(NoSQL)**: 은행은 돈 계산은 100% 정확한데 손님이 몰리면 엄청 오래 기다려야 했고, 구멍가게는 여러 군데 있어서 빠르지만 가끔 계산 실수가 있었어요.
2. **NewSQL이 뭔가요?**: 슈퍼 천재들이 모여서 만든 '절대 계산이 틀리지 않으면서도 전 세계 어디에나 1초 만에 분점을 뚝딱 만들 수 있는 완벽한 최첨단 은행'이에요.
3. **가장 좋은 점은요?**: 은행원이 아무리 바빠도 손님이 늘어나면 알아서 창구를 100개, 1,000개로 무한히 복사해 내고, 모든 창구의 장부가 마법처럼 0.001초 만에 100% 똑같이 맞춰진답니다!
