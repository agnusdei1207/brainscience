# 292. 칵로치DB (CockroachDB) - 생존성 극대화 분산 SQL 데이터베이스

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: "핵전쟁이 나도 바퀴벌레(Cockroach)는 살아남는다"는 철학 아래 설계된, 클라우드 네이티브 환경에 최적화된 뛰어난 생존력(Survivability)과 강력한 일관성을 자랑하는 오픈소스 기반 **NewSQL 데이터베이스**이다.
> 2. **가치**: 데이터베이스 노드(서버)가 불타 없어지거나 지역 데이터센터가 정전되어도, 시스템이 스스로 중단 없이 살아남은 노드로 트래픽을 재조정하고 완벽한 ACID 트랜잭션을 보장하여 RTO(복구 시간) 0초를 실현한다.
> 3. **융합**: 구글 스패너(Spanner)의 논문을 계승하되, 비싼 원자 시계 하드웨어 없이 일반 서버의 NTP와 소프트웨어적인 **HLC(하이브리드 논리 시계)**를 융합하여 전 세계 어디서든 범용 서버 위에서 글로벌 분산 트랜잭션을 구현해 냈다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
CockroachDB는 PostgreSQL과 100% 호환되는 와이어 프로토콜(Wire Protocol)을 제공하는 분산 관계형 데이터베이스다. 내부적으로는 데이터를 64MB 크기의 'Range(범위)' 단위로 잘게 쪼개어(Key-Value 형태) 클러스터 노드 전역에 분산시키며, Raft 합의 알고리즘을 통해 데이터의 동기식 다중 복제본을 유지하여 멈추지 않는 글로벌 스케일 아웃을 제공한다.

### 💡 비유
칵로치DB는 **"스스로 재생하고 이사 가는 액체 금속 로봇(터미네이터 T-1000)"**과 같다.
일반 RDBMS는 단단한 강철 로봇이다. 팔(서버)이 잘리면 고치기 전까지 움직이지 못한다.
하지만 칵로치DB는 액체 금속이다. 미국 지사의 노드가 지진으로 폭파되어 날아가면, 남은 유럽과 아시아의 노드들이 즉시 뭉쳐서 폭파된 조각(데이터)을 순식간에 재생성해 낸다. 심지어 아시아에 손님(트래픽)이 갑자기 몰리면, 스스로 유럽에 있던 데이터 덩어리를 아시아로 쓱 밀어내어(Geo-partitioning) 손님을 빨리 맞이한다. 이 모든 과정이 관리자의 개입 없이 살아 움직이듯 이루어진다.

### 등장 배경 및 발전 과정

#### 1. 구글 스패너의 한계와 "보편적" 해결책의 부재
구글이 2012년 발표한 Spanner는 완벽한 분산 DB였지만, 구글 데이터센터에 설치된 특수 하드웨어(원자 시계)가 있어야만 돌아가는 구글 내부용/종속적 기술이었다. 일반 기업들은 값싼 AWS 클라우드 VM이나 온프레미스 서버에서 구동할 수 있는 Spanner의 대안이 절실했다.

#### 2. 전직 구글러들의 반란 (바퀴벌레의 탄생)
이러한 시장의 갈증을 해결하기 위해, 구글 출신 엔지니어들이 모여 "하드웨어 제약 없이 어디서나 설치 가능한 오픈소스 스패너를 만들자"며 2015년 Cockroach Labs를 설립했다. 이들은 원자 시계 대신 분산 시스템의 논리적 시계 알고리즘(HLC)을 도입하여 시간 오차 문제를 극복했고, 완벽한 클라우드 네이티브(Kubernetes 친화적) 데이터베이스를 탄생시켰다.

| **SQL Layer** | 쿼리 파싱 및 분산 실행 | 사용자의 SQL을 파싱하여 논리적 실행 계획을 세우고, 관련 데이터가 있는 노드들로 쿼리를 쪼개어 보냄(분산 쿼리) | PostgreSQL 호환 | 통역사 및 작전 참모 |

| **Storage Layer** | 물리적 데이터 기록 | 하드 디스크(SSD)에 Key-Value 쌍으로 데이터를 읽고 쓰는 엔진 | Pebble (기존 RocksDB) | 실제 금고 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ CockroachDB Architecture & Raft Replication ]
================================================================================

 [ Client App ] (그냥 Postgres인 줄 알고 "INSERT INTO users" 쿼리 전송)
       │
       ▼
 ┌─────────────┐ (어떤 노드로 접속하든 상관없음. 모든 노드가 대칭형/Symmetric)
 │   Node 1    │ (Gateway 역할 수행: SQL 파싱 -> Range가 어디 있는지 찾음)
 └──────┬──────┘
        │ (내부 RPC 통신)
        ▼
   [ Range 1 (A~M) 에 대한 Raft 합의 그룹 (Replica 3개) ]
  ┌────────────────┐         ┌────────────────┐         ┌────────────────┐
  │     Node 2     │         │     Node 3     │         │     Node 4     │
  │[Raft Leader ★] │ ──────> │ [Raft Follower]│         │ [Raft Follower]│
  │ (Data 쓰기 주도) │ <─Ack── │                │         │                │
  └────────────────┘         └────────────────┘         └────────────────┘
   * 리더(Node 2)가 데이터를 받으면 Follower들에게 복제본을 쏜다.
   * Node 3(또는 4) 중 1대만 "저장 완료(Ack)" 응답을 주면 과반수(2/3) 합의가 이루어져 
     최종 Commit 처리되고 Client에게 성공을 반환한다.
   * 지진으로 Node 2가 파괴되면? Node 3과 4가 즉각 투표를 열어 Node 3을 새 리더로 뽑고 
     서비스를 무중단 지속한다 (Survive!).
```

### 심층 동작 원리

#### ① 하이브리드 논리 시계 (HLC, Hybrid Logical Clocks)
구글 스패너의 원자 시계 없이 분산 트랜잭션 순서를 맞추는 핵심 비밀이다. HLC는 물리적 시계(NTP 기반 서버 시간)와 논리적 카운터(노드 간 패킷을 주고받을 때마다 증가하는 시퀀스)를 융합한 것이다. 트랜잭션이 발생할 때마다 "물리 시간 + 논리 카운터"를 타임스탬프로 찍어 저장한다. 이를 통해 네트워크 지연이나 NTP 오차가 발생해도 "무엇이 먼저 일어났는가(인과 관계, Causality)"를 수학적으로 완벽하게 추적하고 직렬화(Serializable) 격리 수준을 유지한다.

#### ② 핫스팟(Hotspot) 자동 회피 (Auto-rebalancing)
어떤 데이터 구역(Range)에 트래픽이 폭주하면, 칵로치DB는 개발자가 명령하지 않아도 스스로 그 64MB짜리 Range를 32MB 2개로 쪼개버린다(Split). 그리고 쪼개진 절반을 한가하게 놀고 있는 다른 노드로 슥 옮겨준다(Rebalancing). 이 덕분에 트래픽 폭주로 단일 서버가 터지는 RDBMS 고유의 핫스팟 문제가 아키텍처 레벨에서 자동 치유된다.

#### ③ 지정학적 파티셔닝 (Geo-Partitioning)
EU의 개인정보보호법(GDPR)은 "유럽 시민의 데이터는 물리적으로 유럽 밖으로 나가면 안 된다"고 강제한다. 칵로치DB의 진가가 여기서 발휘된다. 단일 테이블(Users) 내에서, 'Country=EU'인 데이터(Range)는 무조건 유럽 지역 데이터센터 노드에만 저장되도록 핀(Pin) 고정시킬 수 있다. 법적 규제 준수와 현지 유저의 조회 지연 시간(Latency) 최소화를 테이블 스키마 설정 하나로 끝낸다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대형 게임 플랫폼 서비스. 기존 MySQL에서 글로벌 런칭을 준비하며 AWS 리전 3개(미국, 유럽, 아시아)에 DB를 분산 구축해야 한다. 단, 게임 머니 결제가 발생하므로 완벽한 ACID가 필요하며, 한 리전이 통째로 셧다운 되더라도 게임은 멈추면 안 된다(RTO=0).
**판단 및 칵로치DB 도입 전략**:
1. **Multi-Region 클러스터 구성**: AWS 미국, 유럽, 아시아에 각각 3대씩 총 9대의 노드로 구성된 하나의 칵로치DB 클러스터를 구축한다.
2. **Raft Quorum 생존성 확보**: 데이터 복제 계수(Replication Factor)를 5로 설정하여, 데이터를 3개 대륙에 흩뿌린다. 아시아 리전이 통째로 날아가도, 미국과 유럽에 남은 과반수 노드가 즉시 리더를 승계하여 1초의 중단 없이 결제가 진행된다.
3. **Geo-Partitioning 성능 튜닝**: 글로벌 DB의 치명적 약점인 '쓰기 지연(Raft 합의를 위해 태평양을 건너야 함)'을 극복하기 위해, '아시아 유저'의 데이터는 아시아 노드가 리더가 되도록 설정(Leaseholder Preferences)하여 아시아 내에서만 빠르게 쓰기 합의가 끝나게 튜닝한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **Clock Skew (시간 오차) 방치**: HLC를 쓴다고 해도 서버 간의 기초적인 물리적 시간(NTP) 오차가 500ms(0.5초) 이상 벌어지면 칵로치DB는 치명적인 데이터 정합성 오류를 막기 위해 스스로 해당 노드를 **"클러스터에서 강제 퇴출(Kill)"** 시켜버린다. 즉, 칵로치DB 운영의 핵심은 인프라 서버들의 정밀한 NTP(또는 PTP) 시간 동기화 관리에 있다.
- **RDBMS식 외래키(FK) 남용**: NoSQL 기반 위에 지어진 시스템이므로, 거대한 여러 테이블을 외래키 제약조건으로 묶고 복잡한 조인(JOIN)을 걸면 네트워크를 타고 분산 쿼리가 폭주하여 성능이 극도로 저하된다. 관계형의 껍데기를 썼지만 내부 설계는 NoSQL처럼 가볍게 비정규화하는 지혜가 필요하다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **운영 공수(Toil) 증발**: DBA가 밤새워 하던 DB 수동 페일오버(Failover), 샤딩 키 리밸런싱, 백업/복구 작업이 엔진 자체의 자동화 덕분에 **거의 0에 수렴**함.
- **다운타임 제로(Zero)**: 계획된 롤링 업데이트나 물리 디스크 파괴 상황에서도 서비스 가용성 99.999% 달성.

### 미래 전망 및 진화 방향
단일 벤더(Oracle)에 종속되었던 RDBMS 시장은 클라우드 네이티브 시대를 맞아 CockroachDB와 같은 분산 SQL(Distributed SQL) 엔진으로 급격히 재편되고 있다. 특히 쿠버네티스의 철학(Pod는 언제든 죽을 수 있다)과 칵로치DB의 철학(노드는 언제든 죽어도 생존한다)이 100% 일치하기 때문에, 컨테이너 환경에서 구동하는 영속성(Stateful) 데이터베이스의 가장 완벽한 진화 형태이자 미래의 기본(Default) DB 인프라로 자리매김할 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **칵로치DB(바퀴벌레DB)란?**: 이름처럼 밟아도 밟아도 절대 죽지 않고 끈질기게 살아남는 좀비 같은 마법의 금고예요.
2. **왜 바퀴벌레인가요?**: 은행 지점 3곳에 똑같은 장부를 나눠뒀는데, 지점 1곳에 불이 나서 잿더미가 되어도 남은 2곳이 즉시 똑같은 장부를 만들어내서 손님은 불이 난 줄도 모르고 돈을 뽑을 수 있거든요.
3. **가장 신기한 점은요?**: 돈(데이터)이 꽉 차서 금고가 좁아지면, 스스로 금고를 반으로 쪼갠 다음에 옆방 빈 금고로 쓱~ 하고 이사를 가는 완전 자동화 로봇이랍니다!
