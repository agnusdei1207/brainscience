# 295. 메모리 캐싱 (Caching) 기술 적용 - Look-aside, Write-through 패턴

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 디스크 기반의 느린 원본 데이터베이스(RDBMS) 앞단에, 초고속 인메모리(In-Memory) 저장소(Redis, Memcached)를 배치하여 애플리케이션의 데이터 읽기/쓰기 성능의 병목을 해소하는 아키텍처 전략이다.
> 2. **가치**: 캐싱 전략(Read/Write Pattern)을 어떻게 설계하느냐에 따라, 무거운 쿼리로 인한 DB의 붕괴를 막아내는 '방패' 역할을 하거나 데이터 정합성(Consistency)을 깨뜨리는 '시한폭탄'이 될 수도 있다.
> 3. **융합**: 읽기 중심의 서비스에는 **Look-aside (Lazy Loading)** 패턴이, 쓰기가 잦고 정합성이 중요한 시스템에는 **Write-through** 패턴이 주로 사용되며, 현대 마이크로서비스 아키텍처(MSA)에서는 이벤트 소싱 등 비동기 동기화와 결합하여 고도화된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
메모리 캐싱(Memory Caching)은 "가장 자주 사용되고, 연산이 비싸며, 잘 변하지 않는 데이터"를 디스크보다 수만 배 빠른 RAM 공간에 임시로 저장해 두는 기술이다. 이 캐시(Cache)와 원본 DB(Database) 사이에 데이터를 어떻게 읽고 쓸 것인지에 대한 설계 규칙을 **캐싱 패턴(Caching Pattern)**이라 부른다.

### 💡 비유
캐싱 패턴은 **"도서관의 베스트셀러 진열대 운영 방식"**과 같다.
- **원본 DB (지하 서고)**: 책이 수백만 권 있지만, 직원이 내려가서 책을 찾아오려면 10분이 걸린다.
- **캐시 (안내 데스크 앞 작은 진열대)**: 공간은 좁아서 책을 100권밖에 못 두지만, 손님이 1초 만에 바로 집어갈 수 있다.
이때 '손님이 책을 빌릴 때', '새 책이 들어왔을 때' 진열대와 지하 서고 중 어디부터 먼저 갈 것인가를 정하는 규칙이 바로 Look-aside와 Write-through 등의 패턴이다.

### 등장 배경 및 발전 과정

#### 1. RDBMS 디스크 I/O의 근본적 한계
인터넷 초기에는 웹 서버와 DB만으로 충분했다. 그러나 대규모 트래픽 시대(웹 2.0)가 도래하면서, RDBMS의 디스크 탐색 시간(Seek Time)과 무거운 조인(JOIN) 연산은 응답 속도 저하의 주범이 되었다. 

#### 2. Redis와 Memcached의 부상
서버의 RAM 가격이 하락하면서, 자주 읽는 데이터를 메모리에 올려두는 캐싱 전용 스토어들이 대중화되었다. 하지만 단순히 캐시를 도입한다고 끝나는 것이 아니었다. "캐시에는 옛날 데이터가 있고, 원본 DB에는 최신 데이터가 있는 정합성 불일치(Stale Data)" 문제가 폭발했고, 이를 논리적으로 해결하기 위해 표준화된 캐싱 설계 패턴들이 확립되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 주요 캐시 읽기/쓰기 패턴 (표)

| 패턴 분류 | 패턴명 | 핵심 동작 메커니즘 | 장점 | 단점 |
|-----------|--------|--------------------|------|------|
| **읽기 (Read)** | **Look-aside** (Lazy Loading) | 앱이 캐시를 먼저 확인 -> 없으면(Miss) DB에서 읽어와 앱이 캐시에 저장 | 가장 흔한 표준. 캐시가 죽어도 DB로 우회 가능 (가용성) | 첫 조회 시 무조건 Miss 발생(지연), 데이터 최신화 지연 |
| **읽기 (Read)** | **Read-through** | 앱은 무조건 캐시만 읽음. 캐시에 없으면 **캐시 스토어가 직접 DB에서** 읽어옴 | 앱 코드가 단순해짐 (캐시만 바라봄) | 캐시 스토어 자체가 DB와 연동을 지원해야 함 (구축 복잡) |
| **쓰기 (Write)**| **Write-through** | 앱이 캐시에 먼저 데이터를 씀 -> **캐시가 동기적으로 DB에 씀** | 데이터 정합성(최신화) 100% 보장 | 쓰기 할 때마다 2번 저장하므로 쓰기 지연(Latency) 증가 |
| **쓰기 (Write)**| **Write-back** (Write-behind) | 앱이 캐시에만 데이터를 쓰고 즉시 응답 -> **캐시가 모아뒀다 비동기로 DB에 씀** | 극한의 쓰기 속도 방어, DB 부하 최소화 | **캐시 서버가 뻗으면 데이터가 영구 유실됨 (위험)** |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ 1. Look-aside (Read 패턴의 표준) ]
================================================================================
  [ Client ] ──(1) 조회 요청──> [ Application ] ──(2) 먼저 물어봄──> [ Redis Cache ]
                                     │                                (결과: Miss!)
                                     │
                                     ├─(3) DB에서 읽어옴 ──────────> [ Database ]
                                     │                                (무거운 쿼리)
                                     └─(4) 읽어온 결과를 저장 ───────> [ Redis Cache ]
                                           (다음 사람부터는 2번에서 바로 끝남!)

================================================================================
[ 2. Write-through (Write 패턴: 정합성 중시) ]
================================================================================
  [ Client ] ──(1) 저장 요청──> [ Application ] 
                                     │
                                     └─(2) 캐시에 저장 ──────────────> [ Redis Cache ]
                                                                             │
                                              (3) 캐시가 DB에 직접 동기화(저장) │
                                                                             ▼
                                                                     [ Database ]
```

### 심층 동작 원리

#### ① Cache Miss와 TTL (Time-To-Live)
Look-aside 패턴에서 데이터 정합성을 맞추는 가장 현실적인 방법은 **TTL(만료 시간)**이다. DB의 데이터가 바뀌었을 때 캐시를 일일이 업데이트해 주는 코드를 짜는 것은 매우 어렵다. 따라서 캐시에 데이터를 넣을 때 `Expire=60초`를 걸어둔다. 60초 동안은 약간의 과거 데이터(Stale)를 보여주더라도 감수하고, 60초가 지나면 캐시가 날아가면서 자연스럽게 다음 요청 시 DB에서 최신 데이터를 퍼와(Miss) 캐시를 리프레시한다.

#### ② 캐시 워밍 (Cache Warming)
Look-aside의 단점은 내일 아침 9시 이벤트 시작 시, 캐시가 텅텅 비어있어(Cold Cache) 수만 명이 동시에 DB로 쏟아져 들어가는 재앙(Cache Stampede)이 발생한다는 것이다. 이를 막기 위해 이벤트 시작 10분 전에, 백그라운드 스크립트를 돌려 DB의 데이터를 캐시에 미리 꽉꽉 채워 넣는 작업을 **캐시 워밍**이라고 한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 트래픽 성격에 따른 패턴 조합 가이드

| 비즈니스 특성 | 추천 읽기 패턴 | 추천 쓰기 패턴 | 실제 적용 사례 |
|---------------|----------------|----------------|----------------|
| **읽기 극단적 폭주** (Read-heavy) | Look-aside | Write-around (DB에만 바로 쓰기) | 뉴스 기사, 쇼핑몰 상품 상세 페이지, 유튜브 랭킹 |
| **쓰기 극단적 폭주** (Write-heavy)| Look-aside | **Write-back** | 유튜브 좋아요 버튼, 게임 실시간 로그, 로그 수집 |
| **데이터 정합성 최우선** (금융) | Read-through | **Write-through** | 은행 잔고 표기, 장바구니 결제 직전 최종 금액 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 배달의 민족 같은 앱에서 "사용자가 배달 기사의 실시간 위치(GPS)를 보는 화면"을 만든다. 기사의 스마트폰은 1초마다 GPS(쓰기)를 쏘고, 사용자는 1초마다 화면(조회)을 새로고침한다. 
- 만약 RDBMS만 쓰면? 초당 1만 기사의 INSERT와 10만 유저의 SELECT가 DB를 10분 안에 터뜨린다.
**판단 및 Caching 패턴 혼합 전략**:
1. **GPS 쓰기 전략 (Write-back)**: 기사의 GPS 데이터는 DB에 즉시 저장할 필요가 없다. 잃어버려도 큰일 안 난다. **Write-back** 패턴을 써서 무조건 Redis에만 GPS 데이터를 덮어쓴다(Update). 앱은 Redis에만 쓰니까 초당 10만 건도 거뜬하다. 그리고 1시간에 한 번씩 Redis의 마지막 위치만 RDBMS(경로 이력용)에 비동기로 저장(Flush)한다.
2. **GPS 읽기 전략 (Read-through / Look-aside)**: 사용자는 무조건 Redis에 있는 기사 위치만 읽는다. DB까지 내려갈 필요조차 없다.
3. **결과**: RDBMS의 부하는 0에 가깝게 수렴하고, 엄청난 양의 GPS 실시간 트래픽을 오직 인메모리 캐시에서 모두 소화해 내는 고성능 아키텍처가 완성된다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **무분별한 객체 통째 캐싱 (Big Object Caching)**: 사용자 정보(User)를 캐싱할 때, `비밀번호, 가입일, 상세 주소` 등 잘 쓰지도 않는 1MB짜리 객체를 통째로 JSON으로 직렬화해서 캐시에 넣는 경우가 있다. 나중에 '상세 주소' 하나만 바뀌어도 이 1MB짜리 거대 캐시를 날리고(Invalidate) 다시 만들어야 한다. 캐시는 반드시 화면이나 API 응답에 딱 맞게 **가장 잘게 쪼개진 단위(Granular)**로 저장해야 무효화(Invalidation) 관리가 가능해진다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **응답 속도 향상**: 디스크 I/O(수십 밀리초)를 메모리 연산(수십 마이크로초)으로 대체하여 시스템 전체 API 평균 응답 시간을 **90% 이상 단축**.
- **DB 인프라 비용 절감**: 캐시 도입 시 RDBMS의 CPU 부하가 1/10로 줄어들어, 천문학적인 오라클/MySQL 스케일업(Scale-up) 라이선스 비용을 극적으로 방어.

### 미래 전망 및 진화 방향
과거에는 개발자가 앱 코드에 직접 `if (redis.get(key) == null) { db.query() ... }` 같은 캐싱 로직을 수동으로 짜야 했다. 현대에는 Spring의 `@Cacheable` 어노테이션 같은 AOP 기반 추상화가 대세가 되었으며, 더 나아가 DB 자체가 내부적으로 인메모리 캐시 계층을 투명하게 갖추고 알아서 관리해 주는 시스템(예: AWS DynamoDB DAX, TiDB의 TiFlash 인메모리 최적화 등)으로 진화하며, "개발자가 캐시 패턴을 고민할 필요 없는" 방향으로 나아가고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **메모리 캐시가 뭔가요?**: 자주 쓰는 국어사전(데이터)을 무거운 가방(DB)에서 매번 꺼내지 않고, 책상 위(캐시)에 딱 올려두고 1초 만에 찾아보는 지혜로운 방법이에요.
2. **Look-aside 패턴은 뭔가요?**: 모르는 단어가 생겼을 때 먼저 책상 위 사전(캐시)을 찾아보고, 거기에 없으면(Miss) 그때서야 무거운 가방(DB)을 열어서 단어를 찾은 뒤, 다음을 위해 책상 위 사전에 뜻을 적어두는 방식이에요.
3. **Write-through 패턴은 뭔가요?**: 내가 새 단어를 알게 돼서 적어둘 때, 무조건 책상 위 사전과 가방 속 사전에 "동시에 똑같이" 적어두는 거예요. 조금 귀찮지만 헷갈릴 일이 절대 없답니다!
