# 296. 캐시 스탬피드 (Cache Stampede) / Thundering Herd 문제

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인메모리 캐시(Redis 등)에 저장된 매우 인기 있는 데이터의 수명(TTL)이 만료(Expire)되는 정확히 그 순간, 데이터를 조회하려던 수만 개의 동시 트래픽이 캐시 미스(Cache Miss)를 내며 원본 DB로 일제히 쏟아져 들어가 DB를 마비시키는 치명적인 장애 현상이다.
> 2. **가치**: 캐싱은 시스템의 부하를 막는 방패지만, 캐시 스탬피드는 그 방패가 깨지는 순간 방패 뒤에 숨어있던 약한 DB가 무방비로 학살당하는 구조적 약점을 보여준다. 이를 막는 락(Lock) 기법과 만료 설계가 고성능 백엔드의 핵심 역량이다.
> 3. **융합**: Look-aside 캐싱 패턴의 태생적 한계에서 발생하며, 이를 방어하기 위해 분산 락(Distributed Lock), 확률적 조기 갱신(Probabilistic Early Expiration, PER), 백그라운드 리프레시(Background Refresh) 같은 고급 동시성 제어 알고리즘이 융합되어 사용된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
캐시 스탬피드(Cache Stampede) 현상은 초고트래픽 환경에서 특정 캐시 키(Key)가 만료(TTL Expiration)되거나 삭제되었을 때 발생한다. 평소에는 캐시 서버(Redis)가 10만 TPS의 조회를 다 막아주고 있었지만, 캐시가 만료된 0.1초의 틈 사이에 10만 개의 요청이 모두 "어? 캐시에 없네(Cache Miss)? DB에서 가져와야지!" 하고 동시에 백엔드 RDBMS로 동일한 무거운 무거운 쿼리(Heavy Query)를 쏘게 되어, DB의 커넥션 풀이 고갈되고 CPU가 100%를 치며 서버가 뻗어버리는 현상이다.

### 💡 비유
캐시 스탬피드는 **"댐(Dam)의 붕괴와 대홍수"**와 같다.
- 평소에는 거대한 튼튼한 댐(Redis 캐시)이 초당 수만 톤의 거센 강물(트래픽)을 완벽하게 막아주고, 마을(원본 DB)에는 졸졸졸 필요한 물만 흘려보내 평화롭다.
- 그런데 정기 점검(TTL 만료)으로 인해 댐 수문을 1초 동안 닫았다 여는 순간, 막혀있던 수만 톤의 물(동시 요청)이 댐의 빈틈을 타고 마을(DB)로 한 번에 쓰나미처럼 몰려간다. 마을은 순식간에 홍수로 쓸려나가 파괴된다. 

### 등장 배경 및 발전 과정

#### 1. Memcached/Redis 도입에 따른 방심
웹 서비스들이 DB 앞단에 캐시를 도입하면서 속도가 비약적으로 상승했다. 개발자들은 "캐시에 넣고 TTL(만료 시간) 1시간만 주면 DB는 안전하겠지"라고 방심했다. 하지만 '네이버 메인 뉴스'나 '블랙프라이데이 메인 상품'처럼 초당 수만 명이 보는 단일 페이지의 캐시가 정확히 1시간 뒤에 만료되자, 서비스를 오픈하자마자 1시간 주기로 DB가 원인 모르게 죽어버리는 미스터리한 현상이 발생했다.

#### 2. Thundering Herd (우르르 몰려다니는 양 떼) 문제의 재발견
운영체제에서 여러 프로세스가 하나의 자원을 기다리다 동시에 깨어날 때 발생하는 'Thundering Herd' 현상이 분산 아키텍처의 캐시 계층에서 똑같이 재현된 것이다. 이 거대한 양 떼(수만 개의 스레드)가 동시에 DB로 달려가는 것을 "단 한 마리만 대표로 보내도록" 통제하는 알고리즘들이 연구되기 시작했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 스탬피드 발생의 3단계 메커니즘

| 단계 | 현상 | 내부 동작 | 결과 |
|------|------|-----------|------|
| **1. 징조** | TTL 만료 | 인기 키(`main_page_data`)의 TTL이 0이 되어 Redis에서 삭제됨 | 캐시 댐 붕괴 |
| **2. 발동** | 동시 Cache Miss | 10,000명의 유저(스레드)가 동시에 Redis에 `get(main_page_data)` 요청 -> 모두 `NULL` 반환 받음 | 대규모 캐시 미스 |
| **3. 재앙** | Thundering Herd | 10,000개의 스레드가 일제히 각자의 DB 커넥션을 열고 똑같은 `SELECT ... JOIN` 쿼리를 DB에 쏨 | DB 풀 고갈, 타임아웃, 전체 장애 |

### 정교한 구조 다이어그램 (ASCII Art)

```ascii
================================================================================
[ Cache Stampede Disaster vs Mutex Lock Defense ]
================================================================================

 [ 1. 장애 발생 (Cache Stampede) ]
                           (TTL 만료! Key 삭제됨)
  User 1 ──(Cache Miss)──┐      ▼
  User 2 ──(Cache Miss)──┼─> [ Redis ] (없다!)
  ...                    │
  User 99 ─(Cache Miss)──┘
                           "다 같이 DB로 가자!!"
                           (99개의 동일한 무거운 쿼리가 쏟아짐)
                           ────────────> [ Database ] (💥 CPU 100% 폭발!)

 [ 2. 방어 기법 1: Mutex Lock (단 1명만 희생하라) ]
                           (TTL 만료! Key 삭제됨)
  User 1 ──(Cache Miss)──┐
  User 2 ──(Cache Miss)──┼─> [ Redis ] (없다!)
  ...                    │
  User 99 ─(Cache Miss)──┘
      ▼
  (앱 로직: 분산 락 획득 시도)
  User 1: "내가 락(Lock) 잡았다! 나 혼자 DB 다녀올게!" ──> [ Database ] (안전하게 1번만 쿼리)
  User 2~99: (락 획득 실패) "User 1이 캐시에 데이터 채울 때까지 100ms 대기하자..." (Wait)
      ▼
  User 1이 DB 결과를 캐시에 저장하고 락을 풂. 대기하던 User 2~99가 드디어 캐시에서 읽음.
```

### 심층 방어 원리 및 알고리즘

#### ① Mutex Lock (상호 배제 락)
가장 확실한 방어법이다. 캐시 미스가 났을 때 무작정 DB로 가지 못하게 하고, `Redis SETNX` 등을 이용해 분산 락(Lock)을 획득한 오직 **단 1개의 스레드(대표 스레드)**만 DB에 다녀올 권한을 준다. 나머지 스레드는 짧은 시간(예: 50ms) 슬립(Sleep) 후 다시 캐시를 조회해 본다. 대표 스레드가 데이터를 채워놨을 테니 미스가 나지 않는다.

#### ② 확률적 조기 갱신 (Probabilistic Early Expiration, PER)
스탬피드는 캐시가 "완전히 사라진 상태(TTL 0)"에서 발생한다. 그렇다면 완전히 사라지기 전에 갱신하면 어떨까? 
XFetch(XFetch) 알고리즘이라고도 불리며, 남은 TTL 시간이 짧아질수록(예: 5초 남음) 캐시를 요청한 스레드에게 아주 낮은 확률(예: 5%)로 가짜 캐시 미스(Fake Miss)를 유발한다. "너 당첨됐어. 넌 데이터 줄 건데, 백그라운드에서 남몰래 DB 다녀와서 캐시 수명 좀 늘려놔 줘."라는 식이다. TTL이 완전히 0이 되기 전에 누군가가 미리 캐시 수명을 연장시켜 스탬피드를 원천 봉쇄한다.

#### ③ 논리적 만료 (Logical Expiration)
Redis의 실제 TTL은 `무제한(Never Expire)`으로 두어 절대 물리적으로 캐시가 날아가지 않게 만든다. 대신 캐시 데이터 JSON 안에 `{"expireAt": "12:00:00", "data": "..."}` 처럼 **논리적인 만료 시간**을 적어둔다.
스레드가 캐시를 읽었을 때 시간이 12시 1분이라면, 스레드는 일단 "만료된 옛날 데이터"를 클라이언트 화면에 0.01초 만에 띄워주고, 백그라운드 스레드(비동기)를 하나 몰래 띄워서 DB에서 최신 데이터를 퍼와 캐시를 조용히 덮어쓴다. 사용자 경험(Latency)이 절대 느려지지 않는 극강의 방어법이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 캐시 스탬피드 방어 기법 트레이드오프

| 방어 기법 | 메커니즘 | 장점 | 단점 (Trade-off) | 적용 적합성 |
|-----------|----------|------|------------------|-------------|
| **Mutex Lock (분산 락)**| 대표 1명만 DB에 다녀옴 | DB 완벽 보호, 데이터 최신화 즉각 보장 | 락을 기다리는 나머지 9,999명은 응답 지연(Latency Spikes) 발생 | 실시간 정합성이 매우 중요한 금융, 재고 |
| **Logical Expiration** | 기한 지난 데이터를 일단 반환하고 비동기 갱신 | 응답 지연 0%, 사용자 대기 없음 (최고의 UX) | 잠시 동안 만료된 옛날 데이터(Stale Data)가 사용자에게 노출됨 | 좋아요 수, 인기 게시물 목록, 메인 배너 |
| **Cache Warming** | 봇(배치)이 TTL 끝나기 전 무조건 주기적 갱신 | Cache Miss 자체를 아예 발생시키지 않음 | 무거운 배치 아키텍처 필요 (크론잡 등) | 매일 아침 9시 열리는 핫딜 상품, 증권 시가 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: "선착순 1만 명, 100원 치킨 핫딜 이벤트"를 오픈한다. 상품 상세 페이지의 데이터를 Redis에 `TTL 10분`으로 캐싱했다. 이벤트 시작 10분 뒤, TTL이 만료되자마자 접속해 있던 50만 명의 트래픽이 캐시 미스를 내고 동시에 DB로 돌진해, 오라클 DB 커넥션 2,000개가 순식간에 차버리고 DB 서버가 다운되는 대형 사고가 발생했다.
**판단 및 방어 아키텍처 수립 (이중화 방어)**:
1. **1차 방어 (Logical Expiration 도입)**: 상품 이름, 사진 같은 변하지 않는 데이터는 TTL을 무한으로 두고, `expireAt` 필드를 넣는다. 10분이 지나도 캐시가 삭제되지 않으므로 DB가 터지는 대재앙은 사라진다. 누군가 1명이 백그라운드로 갱신해 준다.
2. **2차 방어 (Jitter / TTL 분산)**: 모든 캐시 키의 TTL을 칼같이 '600초'로 주면 안 된다. 상품 A, B, C가 동시에 만료되면 역시 DB에 뭉터기로 부하가 온다. TTL을 부여할 때 `600초 + 랜덤(0~60초)` 즉 **Jitter(난수 흩뿌리기)**를 더해 캐시 만료 시간을 넓게 분산(Smearing)시킨다.
3. **결과**: 이벤트 오픈 시점이나 캐시 만료 시점에도 DB 부하 곡선(CPU 커브)이 쓰나미처럼 치솟지 않고, 완만하고 평탄하게 통제되는 고가용성(High Availability) 아키텍처를 달성한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **무한 재시도 락의 덫 (Deadlock & Spinlock)**: Mutex Lock으로 스탬피드를 방어할 때, 락을 획득하지 못한 대기 스레드들이 10ms마다 "락 풀렸냐?"며 무한 `while` 루프(Spinlock)를 돌리게 짜는 초보적인 실수가 흔하다. 이러면 DB는 살렸을지언정, 웹 서버(WAS)의 스레드 CPU가 100%를 치면서 WAS가 뻗어버린다. 반드시 락 대기 시 백오프(Exponential Backoff, 점점 길게 대기) 로직을 넣거나, Redis의 Pub/Sub을 이용해 락이 풀렸을 때만 우아하게 알림을 받아 깨어나게 설계해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **장애 시간(Downtime) 제거**: 고트래픽 서비스에서 원인 모를 주기적 DB 스파이크(Spike)와 다운타임을 원천 차단하여 가용성(SLA) 99.99% 확보.
- **인프라 효율 극대화**: 스탬피드 시의 피크 부하를 막기 위해 평소에 10배 큰 DB(Over-provisioning)를 유지하던 비용을 적정 수준으로 되돌려 클라우드 비용을 대폭 절감.

### 미래 전망 및 진화 방향
캐시 스탬피드는 개발자가 애플리케이션 레벨에서 코딩으로 방어하기엔 너무 귀찮고 버그가 나기 쉬운 영역이다. 따라서 최근에는 **Redis 6.0의 클라이언트 사이드 캐싱(Client-side Caching)** 기능이나, **Envoy / Nginx** 같은 프록시 사이드카(Sidecar) 계층에서 Request Collapsing(똑같은 요청 1만 개가 들어오면 프록시가 9,999개를 홀드하고 딱 1개만 백엔드로 보내는 기능)을 하드웨어 레벨에서 자동으로 처리해 주는 인프라 네이티브 방어 체계로 진화하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **캐시 스탬피드가 뭔가요?**: 놀이공원에서 인기 놀이기구 표(캐시)가 딱 떨어져서 매표소(Redis) 문이 닫히는 순간, 화가 난 1만 명의 사람들이 환불을 받으려고 한 명뿐인 사장님(DB)에게 미친 듯이 동시에 달려들어 사장님이 기절하는 무서운 현상이에요.
2. **어떻게 막을 수 있나요?**: 사장님 방 앞에 안전요원(락, Lock)을 세우고, "대표로 딱 1명만 들어가서 표를 더 가져와! 나머지는 10초만 얌전히 줄 서서 기다려!"라고 통제하는 거예요.
3. **더 똑똑한 방법은요?**: 표가 10장 남았을 때(아직 떨어지기 전), 한 손님에게 "너 몰래 사장님한테 가서 표 한 뭉치만 더 가져와 줄래?"라고 시켜서(확률적 갱신), 표가 아예 바닥나는 상황 자체를 만들지 않는 마법이랍니다!
