# 301. 데이터 마이그레이션 도구 및 CDC (Change Data Capture)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스의 소스 테이블에서 발생하는 모든 변경 사항(Insert, Update, Delete)을 실시간으로 감지하여, 타겟 시스템(Data Warehouse, NoSQL 등)으로 지연 없이 전송하고 동기화하는 **실시간 데이터 통합 기술**이다.
> 2. **가치**: 운영 DB에 부하를 주는 무거운 일괄 처리(Batch ETL)를 지양하고, 트랜잭션 로그를 몰래 읽어 처리함으로써 시스템 가용성을 유지하면서도 **'실시간 데이터 신선도'**를 확보한다.
> 3. **융합**: Kafka와 같은 이벤트 스트리밍 플랫폼, **Debezium**과 같은 오픈소스 커넥터, 그리고 마이크로서비스(MSA)의 데이터 동기화 패턴인 트랜잭션 아웃박스(Outbox) 기술과 융합되어 거대한 데이터 파이프라인을 형성한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
CDC(Change Data Capture)는 데이터베이스 내의 상태 변화를 식별하고 캡처하는 일련의 소프트웨어 디자인 패턴이다. 단순히 데이터를 '복사'하는 것을 넘어, "언제, 어떤 값이, 어떻게 변했는지"에 대한 **이벤트 스트림**을 생성하는 것이 핵심이다.

### 💡 비유
CDC는 **"CCTV를 보고 받아 적는 서기"**와 같다.
- **기존 방식 (Batch ETL)**: 매일 밤 12시에 장부를 통째로 복사해 가는 것. 장부가 너무 무거워 들고 가기 힘들고, 낮에 일어난 일은 밤이 되어야 알 수 있다.
- **CDC**: 장부 옆에 앉아 CCTV를 보며, 누군가 한 글자라도 쓰거나 지우면 그 즉시 포스트잇에 적어서 옆 부서로 전달하는 것. 장부를 옮길 필요도 없고, 옆 부서는 실시간으로 상황을 파악할 수 있다.

### 등장 배경 및 발전 과정

#### 1. Zero-Downtime 마이그레이션의 요구
수 테라바이트의 데이터를 이관할 때, 서비스를 중단하고 옮기는 것은 불가능하다. 먼저 초기 데이터를 옮겨둔 뒤, 이관 작업 중에 발생하는 변경분만 실시간으로 따라잡는(Catch-up) CDC 기술이 필수 장치가 되었다.

#### 2. MSA와 CQRS의 확산
주문 서비스(RDBMS)와 검색 서비스(Elasticsearch)의 데이터를 맞춰야 할 때, 애플리케이션 코드에 이중으로 저장하는 로직을 넣으면 버그가 생기기 쉽다. DB 로그를 직접 읽어 동기화하는 CDC가 가장 안정적인 대안으로 부상했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### CDC 구현 방식 비교 (표)

| 방식 | 상세 내용 | 장점 | 단점 |
|------|-----------|------|------|
| **Audit Column** | `updated_at` 같은 컬럼을 폴링(Polling)하여 변경분 추출 | 구현이 매우 쉬움 | 삭제된 데이터 감지 불가, DB 부하 발생 |
| **Trigger 기반** | DB 트리거를 생성하여 별도 히스토리 테이블에 기록 | 실시간성 우수, DB 독립적 | 운영 DB 연산 오버헤드 큼, 유지보수 어려움 |
| **Log-based (표준)**| **DB 트랜잭션 로그(Binlog 등)를 직접 파싱** | **운영 DB 부하 최소화**, 모든 변경(삭제 포함) 감지 | DB 엔진별 로그 포맷 분석 기술 필요 |

### 정교한 구조 다이어그램 (Log-based CDC Flow)

```ascii
================================================================================
[ Real-time Data Synchronization with Log-based CDC (Debezium) ]
================================================================================

  [ Source DB (MySQL) ]           [ CDC Platform (Kafka) ]         [ Target Systems ]
 ┌───────────────────┐           ┌──────────────────────┐         ┌────────────────┐
 │  Primary Table    │           │  ⭐️ Kafka Connect ⭐️  │         │ Search Engine  │
 │  (Order Data)     │           │  (Debezium Plugin)   │         │ (Elasticsearch)│
 └─────────┬─────────┘           └──────────▲───────────┘         └────────────────┘
           │                                │                              ▲
           │ (1) Write Event                │ (2) Read Log                 │ (4) Sync
           ▼                                │ (Non-blocking)               │
 ┌───────────────────┐           ┌──────────┴───────────┐         ┌────────┴───────┐
 │ Transaction Log   │──────────▶│  Kafka Topic         │────────▶│ Data Warehouse │
 │ (Binlog / WAL)    │           │  "db.orders.change"  │         │ (Snowflake)    │
 └───────────────────┘           └──────────────────────┘         └────────────────┘

 * 핵심 포인트: Debezium이 MySQL의 Binlog를 한 줄씩 읽어 JSON 이벤트로 변환.
   "주문번호 101번의 상태가 '배송중'으로 바뀜" -> 모든 타겟에 0.1초 만에 전파.
```

### 심층 동작 원리

#### ① 트랜잭션 로그 마이닝 (Log Mining)
DB는 정전 등으로 꺼졌을 때 복구하기 위해 모든 변경 사항을 순차적으로 로그 파일(MySQL: Binlog, PostgreSQL: WAL, Oracle: Redo Log)에 적는다. CDC 도구는 이 파일을 꼬리 물기(Tail) 방식으로 읽는다. DB 엔진은 이미 로그를 쓰고 있으므로, CDC가 이를 읽는 것은 실제 쿼리 실행(SELECT)보다 훨씬 가볍다.

#### ② 스냅샷과 스트리밍의 결합
CDC를 처음 켜면, 과거 데이터는 없다. 따라서 먼저 전체 데이터를 한 번 긁어오는 **'Initial Snapshot'** 과정을 거친 후, 그 시점의 로그 위치(LSN/Binlog Position)부터 실시간 스트리밍으로 전환하는 정교한 핸드오버 로직이 작동한다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 레거시 모놀리스 시스템을 마이크로서비스(MSA)로 전환하고 있다. '사용자' 테이블은 여전히 메인 RDBMS에 있는데, 신규 서비스인 '마케팅 서비스'는 실시간으로 가입자 정보를 받아 쿠폰을 쏴줘야 한다. 메인 DB를 건드리기엔 너무 위험하다.
**판단 및 CDC 도입 전략**:
1. **Debezium on Kafka 구축**: 메인 RDBMS의 사용자 테이블에 Debezium 커넥터를 붙인다.
2. **이벤트 스트리밍**: 사용자 가입(Insert) 정보가 Kafka 토픽으로 실시간 발행되게 한다.
3. **Consumer 개발**: 마케팅 서비스는 Kafka 토픽을 구독(Subscribe)하다가 가입 이벤트가 오면 즉시 쿠폰 발송 로직을 태운다.
4. **결과**: 메인 시스템의 소스 코드를 단 한 줄도 수정하지 않고(Zero-code change), 신규 서비스와 실시간 데이터 연동을 성공시킨다. 메인 DB 부하도 거의 없다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **Schema Evolution 대응 부재**: 운영 DB의 컬럼명 하나를 바꾸면 CDC 파이프라인이 즉시 깨질 수 있다. 타겟 시스템(DW 등)의 스키마가 소스의 변화를 따라가지 못하면 전체 데이터 흐름이 중단된다. 반드시 **Schema Registry**를 사용하여 변경 이력을 관리하고, 변화에 유연한 Avro나 Protobuf 포맷을 사용하는 것이 정석이다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **데이터 신선도(Freshness)**: 데이터 가공 지연 시간을 수 시간(Batch)에서 **수백 밀리초(Real-time)**로 단축.
- **운영 안정성**: 소스 DB에 쿼리를 날리지 않으므로, 데이터 추출로 인한 서비스 장애 확률 **Zero** 달성.

### 미래 전망 및 진화 방향
최근에는 DB 자체적으로 CDC 기능을 내재화하거나, 클라우드 벤더들이 **Zero-ETL**이라는 이름으로 DB와 분석 도구(Redshift 등) 사이의 CDC 연결을 버튼 하나로 자동화하고 있다. 향후에는 AI가 데이터 변경 패턴을 학습하여 이상 징후를 실시간 감지하는 **'AI-powered CDC'**와, 네트워크 대역폭을 아끼기 위해 엣지 단에서 데이터 변경분을 필터링하는 기술이 주류가 될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **CDC가 뭔가요?**: 일기장에 새로운 글을 쓰자마자, 옆에서 보던 요정이 그 내용을 똑같이 베껴서 친구에게 바로 전달해 주는 마법 같은 일이에요.
2. **왜 좋나요?**: 일기장을 통째로 빌려주지 않아도 친구가 내 소식을 실시간으로 알 수 있고, 나도 일기 쓰는 방해를 받지 않아서 좋아요.
3. **가장 좋은 점은요?**: 내가 글씨를 지웠을 때도 요정이 "지웠어!"라고 알려주니까, 친구와 나 사이의 소식이 항상 1초 만에 똑같이 맞춰진답니다!
