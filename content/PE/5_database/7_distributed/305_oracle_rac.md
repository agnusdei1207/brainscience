# 305. 오라클 RAC (Real Application Clusters) - 공유 디스크 기반 클러스터링

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 여러 대의 서버(인스턴스)가 하나의 물리적 데이터베이스(공유 디스크)를 동시에 사용하도록 하여, 서버 한 대가 죽어도 서비스가 중단되지 않게 하는 **공유 디스크(Shared Disk)** 방식의 고가용성 DB 클러스터링 기술이다.
> 2. **가치**: 특정 노드 장애 시 즉시 다른 노드로 업무를 승계(Failover)하여 무중단 서비스를 보장하며, 서버를 추가함에 따라 연산 능력을 확장(Scale-up의 한계 완화)할 수 있는 엔터프라이즈급 안정성을 제공한다.
> 3. **융합**: 고속 인터커넥트망(Private Network), 분산 락 관리(Cache Fusion), 그리고 공유 스토리지(SAN/NAS) 기술이 융합되어 오라클 데이터베이스의 강력한 정합성을 분산 환경에서도 유지한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
오라클 RAC(Real Application Clusters)는 단일 데이터베이스 파일을 여러 대의 인스턴스가 공유하며, 각 인스턴스가 독립적으로 트랜잭션을 처리할 수 있는 아키텍처다. 

### 💡 비유
오라클 RAC는 **"하나의 거대한 도서 대출 장부를 공유하는 여러 명의 사서"**와 같다.
- **일반 DB (사서 1명)**: 사서가 아프면 도서관 문을 닫아야 한다.
- **오라클 RAC**: 사서가 3명이다. 장부는 중앙의 큰 책상(공유 디스크)에 하나만 있고, 사서들은 각자 자기 책상에서 손님을 맞는다. 사서 1명이 쓰러져도 다른 사서 2명이 장부를 보고 계속 책을 빌려줄 수 있다. 다만, 사서들끼리 "내가 지금 이 장부 10페이지 수정 중이야!"라고 수시로 속삭여야(Interconnect) 데이터가 꼬이지 않는다.

### 등장 배경 및 발전 과정

#### 1. 싱글 DB의 한계
과거에는 DB 성능을 높이려면 무조건 비싼 슈퍼컴퓨터를 사야 했다(Scale-up). 하지만 한 대의 서버는 반드시 언젠가 고장 나기 마련이고, 고장 나면 전사 서비스가 멈췄다.

#### 2. 오라클의 혁신 (Oracle Parallel Server -> RAC)
오라클은 9i 버전에서 RAC를 발표하며 "비교적 저렴한 서버 여러 대를 묶어서 슈퍼컴퓨터급 성능과 절대 죽지 않는 안정성을 동시에 잡겠다"는 전략을 취했다. 이는 금융, 공공 등 미션 크리티컬한 시스템의 절대적 표준이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### RAC의 주요 구성 요소 (표)

| 요소명 | 상세 역할 | 특징 | 비유 |
|--------|----------|------|------|
| **DB Instances** | 메모리(SRAM)와 프로세스 집합 | 각 노드에서 실제 쿼리를 수행하는 주체 | 개별 사서 |
| **Shared Storage** | 실제 데이터 파일 저장소 | SAN/NAS를 통해 모든 인스턴스가 동시에 접근 가능 | 공용 장부 |
| **Interconnect** | 노드 간 고속 전용 네트워크 | **Cache Fusion**을 위해 데이터 블록을 주고받는 통로 | 사서들의 귓속말 통로 |
| **Grid Infrastructure**| 클러스터 관리 소프트웨어 | 노드 생사 확인(Heartbeat) 및 자원 관리 수행 | 도서관 관리 시스템 |


### 심층 동작 원리

#### ① 캐시 퓨전 (Cache Fusion) - RAC의 꽃
만약 인스턴스 1이 데이터 블록 A를 수정하려고 하는데, 그 블록이 인스턴스 2의 메모리에 이미 있다면 어떻게 할까? 
- 과거 방식: 인스턴스 2가 디스크에 쓰고, 인스턴스 1이 다시 디스크에서 읽어옴. (매우 느림).
- **Cache Fusion**: 인스턴스 2가 고속 인터커넥트망을 통해 메모리에서 메모리로 직접 블록을 쏴줌. 디스크 I/O 없이 마치 하나의 거대한 메모리를 쓰는 것처럼 동작하게 함.

#### ② 분산 락 관리 (DLM, Distributed Lock Manager)
여러 노드가 동시에 같은 데이터를 고치지 못하도록 순서를 정해주는 기술이다. 특정 노드가 'Resource Master'가 되어 락을 관리하며 데이터 무결성을 100% 보장한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 공유 디스크(Shared Disk) vs 셰어드 낫띵(Shared Nothing)

| 비교 지표 | Shared Disk (RAC) | Shared Nothing (NoSQL/NewSQL) |
|-----------|--------------------|-------------------------------|
| **데이터 소유권**| 모든 노드가 모든 데이터 접근 가능 | 각 노드가 자기 영역의 데이터만 관리 |
| **확장성** | 보통 ( Scale-out에 물리적 한계 있음) | **최상 (무한 수평 확장)** |
| **관리 편의성** | 우수 (기존 SQL/App 수정 불필요) | 낮음 (샤딩 키 설계 필요) |
| **비용** | 고가 (SAN 스토리지 필수) | 저렴 (일반 서버 디스크 활용) |
| **장점** | **강력한 정합성과 즉각적 페일오버** | 대규모 데이터 처리 효율 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 한 은행의 계정계(Core Banking) 시스템. 거래량이 늘어나면서 단일 서버로는 한계에 부딪혔고, 만약 서버가 10분이라도 멈추면 수조 원의 금융 사고가 우려된다.
**판단 및 RAC 구축 전략**:
1. **2-Node RAC 구성**: 일단 서버 2대를 묶어 한 대가 죽어도 나머지 한 대가 100% 업무를 승계하게 한다.
2. **SCAN (Single Client Access Name)**: 클라이언트가 서버 IP를 알 필요 없이 하나의 가상 이름으로 접속하게 하여, 노드 추가/삭제 시에도 앱 수정이 없게 한다.
3. **ASM (Automatic Storage Management)**: 오라클 전용 볼륨 매니저를 써서 디스크 부하를 여러 채널로 분산(Striping)한다.
4. **결과**: 서비스 가동률 99.999%를 달성하고, 서버 부하 분산을 통해 거래 처리 속도를 40% 이상 개선한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **인터커넥트 병목**: 노드 사이의 속삭임(Cache Fusion)이 너무 많아지면 배보다 배꼽이 더 커진다. 특히 인덱스 경합(Index Contention)이 심한 데이터를 여러 노드에서 동시에 수정하면, 데이터 전송하느라 CPU가 다 소모된다. RAC 환경에서는 가급적 **'데이터 지역성'**을 고려하여 업무를 노드별로 분산하는 지혜가 필요하다.

---

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **가용성 향상**: 하드웨어 고장 시에도 서비스 중단 시간 **30초 이내**로 방어.
- **성능 확장**: 서버 추가 시 온라인 상태에서 처리량(Throughput) **1.5배 ~ 1.8배** 증가.

### 미래 전망 및 진화 방향
오라클은 이제 RAC를 클라우드로 옮긴 **'Oracle Autonomous Database'**를 밀고 있다. 사람이 튜닝하지 않아도 AI가 RAC 노드 간 부하를 알아서 조절하고, 패치도 무중단으로 수행한다. 비록 NoSQL의 셰어드 낫띵 방식이 유행이지만, 금융권처럼 **'단 1원의 오차도 허용하지 않는 강력한 ACID'**가 필요한 영역에서 RAC는 여전히 대체 불가능한 제왕의 자리를 지킬 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **오라클 RAC가 뭔가요?**: 공부 잘하는 쌍둥이 형제(서버)들이 '하나의 똑같은 교과서(DB)'를 보면서 같이 시험을 치는 거예요.
2. **왜 그렇게 하나요?**: 형이 갑자기 아파서 보건실에 가도, 동생이 교과서를 보고 형이 하던 문제까지 마저 풀 수 있기 때문이에요.
3. **가장 신기한 점은요?**: 형과 동생이 텔레파시(인터커넥트)를 주고받아서, 형이 교과서에 정답을 적으면 동생 머릿속에도 즉시 그 정답이 떠오르는 신기한 마법이랍니다!
