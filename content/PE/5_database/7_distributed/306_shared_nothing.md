# 306. 셰어드 낫띵 (Shared Nothing) 아키텍처 - 무한 확장의 근간

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 각 노드(서버)가 자기만의 독자적인 CPU, 메모리, 디스크를 가지며 데이터의 특정 부분만 책임지고 관리하는 **독립형 분산 구조**로, 노드 간에 자원을 공유하지 않는 아키텍처이다.
> 2. **가치**: 자원 공유로 인한 경합(Contention)이 없으므로 서버를 추가하는 만큼 성능이 선형적으로 늘어나는 **수평적 확장성(Scale-out)**이 완벽하며, 노드 하나가 죽어도 다른 노드에 물리적 영향을 주지 않는다.
> 3. **융합**: 데이터 샤딩(Sharding), 일관된 해싱(Consistent Hashing), 그리고 무상태(Stateless) 서비스 설계와 융합되어 구글, 페이스북, 넷플릭스 등 초거대 인터넷 서비스의 표준 저장소 구조로 사용된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
셰어드 낫띵(Shared Nothing, SN) 아키텍처는 분산 시스템의 노드들이 하드웨어 자원을 아예 공유하지 않는 방식이다. 각 노드는 전체 데이터 중 자신이 맡은 구역(Shard)에 대해서만 '왕'처럼 군림하며, 다른 노드와 소통할 때는 오직 네트워크 메시지(RPC)를 통해서만 대화한다.

### 💡 비유
셰어드 낫띵은 **"각자 자기 주방과 식재료를 가진 프랜차이즈 식당들"**과 같다.
- **공유 디스크 (RAC)**: 커다란 냉장고 하나를 10명의 요리사가 같이 쓰는 것. 냉장고 문 열 때마다 서로 부딪히고(경합), 냉장고가 고장 나면 장사를 다 접어야 한다.
- **셰어드 낫띵**: 전국의 100개 지점 식당이 각자 자기 냉장고와 가스레인지를 가지고 있는 것. 강남점에 손님이 몰려도 강북점 냉장고는 평화롭다. 강남점이 불이 나도(노드 장애) 다른 지점은 떡볶이를 계속 팔 수 있다. 지점이 늘어날수록 전국의 떡볶이 생산량은 정확히 늘어난다.

### 등장 배경 및 발전 과정

#### 1. 대규모 데이터의 습격 (Big Data)
데이터가 수십 페타바이트가 되면서, 아무리 비싼 슈퍼컴퓨터라도 하나의 디스크 뭉치(SAN)에 연결하는 것은 물리적으로 불가능해졌다.

#### 2. 마이클 스톤브레이커(Michael Stonebraker)의 통찰
DB 업계의 거장 스톤브레이커 교수는 1986년 논문에서 "미래의 고성능 DB는 자원 공유를 포기하고 네트워크로 묶인 독립 노드들의 집합(Shared Nothing)이 될 것"이라고 예언했다. 이 예언은 20년 뒤 NoSQL(Cassandra, MongoDB)과 분산 DB(Teradata, Greenplum)의 폭발적 성장으로 증명되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 주요 분산 아키텍처 비교 (표)

| 비교 지표 | Shared Memory (SMP) | Shared Disk (RAC) | **Shared Nothing (SN)** |
|-----------|----------------------|-------------------|-------------------------|
| **공유 자원** | CPU, Mem, Disk 전체 | **Disk (Storage)** | **없음 (Nothing)** |
| **통신 방식** | 공유 메모리 참조 | 공유 디스크 I/O | **네트워크 패킷 (RPC)** |
| **확장성** | 매우 낮음 | 보통 | **매우 높음 (수천 대 이상)** |
| **장애 격리** | 불가능 (커널 패닉 시 전체 종료) | 부분적 | **완벽함 (노드 간 독립)** |
| **대표 기술** | 단일 고성능 서버 | Oracle RAC | **Cassandra, Hadoop, TiDB** |

### 정교한 구조 다이어그램 (자원 독립성 중심)

```ascii
================================================================================
[ Shared Nothing (SN) Distributed Architecture ]
================================================================================

      [ 💻 Client Requests ] ──▶ [ Hash / Router ]
                                         │
           ┌─────────────────────────────┼─────────────────────────────┐
           ▼                             ▼                             ▼
  ┌──────────────────┐          ┌──────────────────┐          ┌──────────────────┐
  │ 🚀 Node 1        │          │ 🚀 Node 2        │          │ 🚀 Node 3        │
  │ [ CPU ] [ MEM ]  │          │ [ CPU ] [ MEM ]  │          │ [ CPU ] [ MEM ]  │
  │        │         │          │        │         │          │        │         │
  │ [ 🗄️ Disk A ]    │          │ [ 🗄️ Disk B ]    │          │ [ 🗄️ Disk C ]    │
  └──────────────────┘          └──────────────────┘          └──────────────────┘
           ▲                             ▲                             ▲
           │                             │                             │
           └────────── [ 🌐 High-speed Network ] ──────────────────────┘
                        (Inter-node Messaging only)

 * 핵심: Node 1은 Node 2의 메모리나 디스크를 직접 들여다볼 수 없음. 
   데이터가 필요하면 "2번아, 네가 가진 데이터 좀 나한테 보내줘"라고 부탁(Request)해야 함.
```

### 심층 동작 원리

#### ① 데이터 샤딩 (Data Sharding)
데이터를 특정 기준(예: 사용자 ID의 해시값)으로 쪼개어 각 노드에 분산 저장한다. 1번 노드는 'A~L' 성씨 사용자만, 2번 노드는 'M~Z' 성씨 사용자만 관리한다. 쿼리가 들어오면 해당 알파벳을 담당하는 노드만 움직이므로 간섭이 발생하지 않는다.

#### ② 로컬리티 (Locality)의 극대화
데이터 연산은 항상 데이터가 저장된 물리적인 위치(Local)에서 수행된다. 데이터를 네트워크로 끌어오는 게 아니라, 프로그램을 데이터가 있는 노드로 보내서 처리(Hadoop의 철학)하므로 네트워크 트래픽을 비약적으로 줄인다.

## Ⅲ. 융합 비교 및 다각도 분석

### 셰어드 낫띵의 장단점 트레이드오프

| 장점 (Strength) | 단점 (Weakness) |
|-----------------|-----------------|
| **무한 확장성**: 이론적으로 수만 대까지 선형 확장 가능 | **조인(Join)의 고통**: 서로 다른 노드의 데이터를 합치려면 네트워크 비용이 큼 |
| **장애 방화벽**: 한 노드의 하드웨어 고장이 전체로 전이되지 않음 | **데이터 일관성 유지**: 노드 간 상태가 다를 수 있어 합의 알고리즘 필요 |
| **저비용**: 비싼 SAN 장비 없이 일반 범용 서버(PC급)로 구축 가능 | **관리 복잡도**: 수천 대의 노드 상태를 모니터링하기 힘듦 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 스타트업의 결제 알림 서비스가 대박이 나서 초당 100만 건의 로그가 쏟아진다. 기존 고성능 오라클 서버로는 감당이 안 되고, 서버 한 대를 더 좋은 걸로 바꾸려니(Scale-up) 수십억 원이 든다.
**판단 및 셰어드 낫띵 기반 전환 전략**:
1. **NoSQL 도입 (Cassandra)**: 전형적인 셰어드 낫띵 구조인 카산드라를 채택한다.
2. **저가형 서버 클러스터링**: 수억 원짜리 장비 대신, 수백만 원짜리 일반 리눅스 서버 50대를 묶는다.
3. **복제 전략(Replication)**: 셰어드 낫띵은 노드가 자주 죽는 것을 전제로 한다. 데이터를 3개 노드에 복제(N=3)하여 한두 대가 죽어도 서비스가 유지되게 설계한다.
4. **결과**: 기존 대비 1/10 비용으로 10배 높은 처리량을 확보하며, 향후 트래픽이 더 늘어나면 서버를 한 대씩 사서 꽂기만 하면 되는 유연한 인프라를 완성한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **핫스팟(Hotspot) 문제**: 특정 노드에 데이터가 쏠리면(예: 유명 연예인의 게시물), 셰어드 낫띵의 장점이 사라진다. 99개 노드는 노는데 1개 노드만 비명을 지르는 상황이다. 이를 막기 위해 **'Salt'**를 치거나 샤드 키를 정교하게 분산하는 아키텍처 튜닝이 필수적이다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **TCO (총소유비용) 절감**: 고가 장비 배제 및 범용 하드웨어 활용으로 **비용 대비 성능비 5배 이상** 향상.
- **성능 선형성**: 노드 수 증가에 따른 처리량 증가율 **90% 이상** 유지.

### 미래 전망 및 진화 방향
셰어드 낫띵은 이제 서버리스(Serverless)와 클라우드 네이티브의 기본값이 되었다. 최근에는 하드웨어 자원을 아예 논리적으로 쪼개 쓰는 **'컴포저블 인프라(Composable Infrastructure)'**와 결합하여, 셰어드 낫띵의 확장성과 공유 자원의 효율성을 동시에 잡으려는 시도가 이어지고 있다. 또한 AI 학습을 위한 거대 GPU 클러스터 역시 전형적인 셰어드 낫띵 아키텍처로 설계되고 있어, 이 철학은 현대 컴퓨팅의 거부할 수 없는 흐름이 되었다.

## 👶 어린이를 위한 3줄 비유 설명
1. **셰어드 낫띵이 뭔가요?**: 학급에서 조별 과제를 할 때, 도화지 한 장을 30명이 같이 그리는 게 아니라, 학생마다 자기만의 작은 종이와 색연필을 가지고 각자 맡은 부분만 그리는 거예요.
2. **왜 좋나요?**: 도화지 한 장에 몰려 있으면 서로 손이 부딪히고 싸우는데, 각자 종이에 그리면 방해받지 않고 아주 빠르게 완성할 수 있거든요.
3. **가장 좋은 점은요?**: 한 친구가 지우개 똥을 흘려도(서버 고장) 내 종이는 깨끗하니까, 나머지 친구들이 합쳐서 멋진 그림을 완성할 수 있답니다!
