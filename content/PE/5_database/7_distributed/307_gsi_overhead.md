# 307. 글로벌 보조 인덱스 (GSI, Global Secondary Index) 분산 환경 오버헤드

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 데이터베이스에서 파티션 키(Partition Key)가 아닌 다른 일반 속성으로 데이터를 빠르게 찾기 위해, 전체 파티션을 가로질러 별도의 색인 데이터를 생성하고 유지하는 **분산 색인 기술**이다.
> 2. **가치**: 데이터를 쪼개서 저장하는 NoSQL 환경에서도 RDBMS처럼 다양한 조건(예: 사용자 ID가 아닌 이메일 주소로 검색)으로 조회를 가능케 하여 애플리케이션의 유연성을 극대화한다.
> 3. **융합**: 비동기 복제 방식의 **최종적 일관성(Eventual Consistency)**, 쓰기 부하를 분산시키는 인덱스 전파 매커니즘, 그리고 검색 엔진(Elasticsearch)과의 결합 모델과 융합되어 거대 규모의 데이터 탐색을 지원한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
글로벌 보조 인덱스(GSI)는 원본 데이터가 저장된 파티션(Node)의 위치와 상관없이, 인덱스 전용의 별도 테이블을 물리적으로 따로 구성하는 방식이다. 원본 테이블이 `사용자 ID`로 샤딩되어 있어도, `이메일`을 GSI로 설정하면 모든 파티션을 뒤지지 않고도 이메일 주소로 사용자를 단번에 찾을 수 있다.

### 💡 비유
GSI는 **"전국 도서관 통합 검색용 이메일 주소록"**과 같다.
- **원본 데이터 (도서관)**: 책들은 각 지역(파티션) 도서관에 '도서번호' 순으로 꽂혀 있다.
- **로컬 인덱스**: 강남 도서관에 가서 "여기서 'PF'가 쓴 책 있나요?"라고 묻는 것. (내 동네만 뒤짐).
- **GSI (통합 주소록)**: 서울시청 중앙에 '작가 이름'으로 정리된 거대한 카드함을 둔다. 강남에 있든 강북에 있든 작가 이름만 알면 어느 도서관 몇 번 칸에 있는지 바로 알려준다. 다만, 새 책이 들어올 때마다 서울시청까지 가서 카드함을 업데이트해야 하는 번거로움(오버헤드)이 있다.

### 등장 배경 및 발전 과정

#### 1. NoSQL의 태생적 한계 (Scatter-Gather)
분산 DB에서 파티션 키가 아닌 값으로 검색하면, 시스템은 모든 노드에 "너희 그 데이터 있니?"라고 물어봐야 한다. 이를 Scatter-Gather라고 하며, 노드가 100대면 검색 속도도 100배 느려진다.

#### 2. DynamoDB와 Cassandra의 해법
이를 해결하기 위해 데이터가 들어올 때 백그라운드에서 인덱스용 테이블을 하나 더 만들어주는 GSI 기능이 도입되었다. 사용자 편의성은 비약적으로 늘어났지만, 이로 인한 '쓰기 비용'과 '정합성' 문제가 새로운 과제로 떠올랐다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 로컬 보조 인덱스(LSI) vs 글로벌 보조 인덱스(GSI) (표)

| 비교 지표 | Local Secondary Index (LSI) | Global Secondary Index (GSI) |
|-----------|-----------------------------|------------------------------|
| **파티션 범위**| 동일 파티션 내로 제한 | **전체 파티션(Cluster-wide)** |
| **정합성** | 강력한 일관성 (Strong) | **최종적 일관성 (Eventual)** |
| **쓰기 부하** | 낮음 (자기 노드만 수정) | **높음 (별도 노드에 추가 쓰기 발생)** |
| **조회 성능** | 특정 파티션 내 검색 시 유리 | **어떤 조건으로든 즉시 탐색 가능** |
| **비유** | 아파트 입구 게시판 | 시청 중앙 민원실 장부 |

### 정교한 구조 다이어그램 (GSI 전파 과정)

```ascii
================================================================================
[ GSI (Global Secondary Index) Write & Read Path ]
================================================================================

 [ 1. Write Path: Update Original Data ]
  Client ──▶ [ Partition A (Master) ] ── (1) Save Record {ID: 101, Email: "pf@abc.com"}
                    │
                    │ (2) Asynchronous Replication (Change Data Capture)
                    ▼
             [ 🗄️ GSI Table (Partition C) ] ── (3) Create Index {Email: "pf@abc.com", ID: 101}
             * 원본과 인덱스가 물리적으로 다른 서버(C)에 있을 확률이 높음!

 [ 2. Read Path: Query by Email ]
  Client ──▶ [ Query Router ] 
                    │ (4) "Email이 pf@abc.com인 ID가 누구니?"
                    ▼
             [ 🗄️ GSI Table ] ──▶ (5) "ID는 101번이야!" (Index Scan)
                    │
                    └─────────────────▶ [ Partition A ] ──▶ (6) "여기 101번 데이터 전체야!"
                                                            (Final Result)
```

### 심층 동작 원리 및 오버헤드 요인

#### ① 증폭된 쓰기 부하 (Write Amplification)
사용자가 데이터 1건을 저장할 때, 시스템 내부적으로는 인덱스 테이블에도 1건을 더 써야 한다. GSI가 5개라면 쓰기 부하는 6배가 된다. 특히 원본 데이터의 특정 필드가 수정될 때마다 인덱스 테이블의 기존 값을 지우고 새 값을 써야 하므로 디스크 I/O와 네트워크 트래픽이 폭증한다.

#### ② 일관성 랙 (Consistency Lag)
대부분의 GSI는 성능을 위해 비동기(Async)로 동기화된다. 0.1초 전에 회원가입을 했는데, 이메일로 검색하면 아직 인덱스에 안 올라와서 "검색 결과 없음"이 뜨는 현상이 발생할 수 있다.

#### ③ 인덱스 핫스팟 (Hotspot)
인덱스 필드의 값이 특정 값에 쏠려 있다면(예: 성별, 도시 등), 인덱스 테이블의 특정 파티션만 과부하가 걸려 전체 시스템의 병목이 될 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### 분산 환경의 인덱싱 전략 비교

| 전략 | 작동 방식 | 장점 | 단점 |
|------|-----------|------|------|
| **Scatter-Gather** | 모든 노드에 직접 물어봄 | 추가 저장 공간 없음 | 노드 수에 비례해 조회 속도 급락 |
| **GSI (Global Index)**| 별도의 인덱스 테이블 유지 | **초고속 검색 가능** | **쓰기 비용 및 정합성 지연** |
| **Search Engine** | ES 등으로 데이터 이관 | 복잡한 검색(오타 등) 가능 | 시스템 복잡도 및 동기화 비용 극상 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 초당 10만 건의 트래픽이 발생하는 게임 서버의 '유저 업적' 테이블. 유저 ID가 파티션 키인데, '전국 랭킹'을 위해 '점수' 필드에 GSI를 걸었더니 게임 속도가 현저히 느려지고 DB 비용이 3배로 늘어났다.
**판단 및 튜닝 전략**:
1. **GSI 최소화**: 꼭 필요한 검색 조건인지 재검토한다. 점수는 1초마다 바뀌는데 그때마다 GSI를 갱신하는 것은 비효율의 극치다.
2. **Projection 제한**: 인덱스 테이블에 원본 데이터 전체를 다 복사(Include All)하지 말고, 오직 `ID` 값만 포함(Key Only)하도록 설정을 변경한다. (저장 공간 및 쓰기 부하 감소).
3. **Caching 전환**: 실시간성이 중요한 랭킹은 GSI 대신 **Redis Sorted Set**으로 워크로드를 분리한다.
4. **결과**: DB 쓰기 부하를 60% 절감하고 게임의 실시간 반응성을 회복한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **Primary Key처럼 GSI 사용**: GSI는 보조 수단일 뿐이다. 핵심 비즈니스 로직(예: 결제 완료 처리)을 GSI 검색 결과에 의존하면, 비동기 동기화 지연 때문에 결제가 누락되는 대형 사고가 터질 수 있다. 중요한 작업은 반드시 **파티션 키(Strong Consistency)**를 사용해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **조회 응답 시간**: 수천만 건 데이터에서 보조 필드 검색 속도를 $O(N)$에서 **$O(1)$** 수준으로 단축.
- **유연한 데이터 모델링**: 파티션 설계에 얽매이지 않고 비즈니스 요구사항에 맞는 다양한 API 제공 가능.

### 미래 전망 및 진화 방향
최근의 분산 DB(CockroachDB, TiDB 등)는 GSI의 비동기 문제를 해결하기 위해 **분산 트랜잭션** 기술을 적용, 원본과 인덱스를 100% 동기화하면서도 성능을 유지하는 방향으로 발전하고 있다. 또한, 인덱스 데이터를 메모리에 상주시키는 **In-memory GSI** 기술이 대중화되면서, 쓰기 부하는 하드웨어가 흡수하고 읽기는 빛의 속도로 처리하는 '성능과 정합성의 조화'가 완성되어가고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **글로벌 보조 인덱스가 뭔가요?**: 반 친구들 100명의 사물함이 번호순으로 정리되어 있는데, 갑자기 "안경 쓴 친구"만 찾고 싶을 때 모든 사물함을 열어보지 않고도 바로 찾을 수 있는 '특별한 명단'이에요.
2. **왜 힘든가요?**: 친구가 안경을 새로 쓰거나 벗을 때마다 명단을 계속 고쳐야 하는데, 명단이 너무 크면 고치는 데 시간이 오래 걸리고 종이도 많이 들기 때문이에요.
3. **가장 좋은 점은요?**: 고칠 때는 조금 힘들어도, 명단만 딱 보면 수천 명 중에서 내가 원하는 친구를 1초 만에 쏙 찾아낼 수 있어서 검색할 때 정말 편하답니다!
