# 308. 폴리글랏 퍼시스턴스 (Polyglot Persistence) - 목적 기반 DB 혼용

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 하나의 애플리케이션이나 마이크로서비스 아키텍처(MSA) 내에서, 모든 데이터를 하나의 거대 DB(RDBMS)에 몰아넣는 대신 **데이터의 특성과 서비스 요건에 따라 최적화된 여러 종류의 이기종 데이터베이스(NoSQL, RDBMS, Graph 등)를 혼용**하여 사용하는 설계 전략이다.
> 2. **가치**: "모든 문제에 맞는 단 하나의 도구는 없다"는 원칙에 충실하여, 결제 데이터는 RDBMS(정합성), 검색은 Elasticsearch(속도), 친구 추천은 Graph DB(관계)에 맡김으로써 시스템 전체의 성능과 비용 효율을 극대화한다.
> 3. **융합**: 마이크로서비스의 '데이터 독립성' 원칙, 비동기 이벤트 스트리밍(Kafka), 그리고 분산 데이터 동기화 기술(CDC)이 융합되어 거대한 분산 데이터 생태계를 형성한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
폴리글랏(Polyglot)은 '여러 언어를 구사하는'이라는 뜻이다. 폴리글랏 퍼시스턴스는 데이터 저장소 역시 '데이터 언어'에 맞춰 다양하게 구사하자는 의미다. 2006년 닐 포드(Neal Ford)가 처음 제안했으며, 마틴 파울러(Martin Fowler)에 의해 대중화되었다.

### 💡 비유
폴리글랏 퍼시스턴스는 **"용도에 맞는 전용 도구함"**과 같다.
- **과거 (모놀리식 DB)**: 망치 하나로 못도 박고, 나사도 돌리고, 구멍도 뚫으려 하는 것. (비효율적이고 위험함).
- **폴리글랏**: 못을 박을 땐 망치(RDBMS), 나사를 돌릴 땐 드라이버(NoSQL), 구멍을 뚫을 땐 드릴(Search Engine)을 쓰는 것. 각 도구가 제 역할을 할 때 가장 빠르고 튼튼한 집(시스템)이 완성된다.

### 등장 배경 및 발전 과정

#### 1. 관계형 DB(RDBMS)의 만능주의 탈피
과거에는 오라클이나 MySQL이 모든 데이터(이미지 경로, 세션, 로그, 통계)를 다 담았다. 하지만 데이터가 거대해지자 RDBMS의 엄격한 스키마와 조인(Join) 부하가 성능의 발목을 잡았다.

#### 2. NoSQL의 폭발적 성장
2010년대 초반, 특정 분야(Key-Value, Document, Graph)에 특화된 NoSQL들이 쏟아져 나왔다. 개발자들은 "결제는 오라클에 남기되, 장바구니는 Redis에 담으면 훨씬 빠르다"는 것을 깨닫고 DB를 섞어 쓰기 시작했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 데이터 유형별 최적의 DB 선택 (표)

| 데이터 성격 | 요구 사항 | 추천 DB 유형 | 실제 사례 |
|-------------|-----------|---------------|-----------|
| **원장/결제** | 강력한 정합성 (ACID) | **RDBMS** | MySQL, PostgreSQL |
| **세션/캐시** | 초고속 읽기/쓰기 | **Key-Value** | Redis, Memcached |
| **상품/콘텐츠** | 유연한 스키마, 계층 구조 | **Document** | MongoDB, Couchbase |
| **소셜/추천** | 복잡한 관계망 탐색 | **Graph** | Neo4j, JanusGraph |
| **로그/메트릭** | 시계열 데이터 대량 적재 | **Time-series** | InfluxDB, Prometheus |
| **검색/분석** | 텍스트 전문 검색 | **Search Engine** | Elasticsearch, Solr |

### 정교한 구조 다이어그램 (MSA 환경의 폴리글랏 모델)

```ascii
================================================================================
[ Polyglot Persistence Architecture in E-Commerce ]
================================================================================

  [ 📱 Mobile / Web App ]
           │
  ┌────────▼────────────────────────────────────────────────────────┐
  │  🏢 API Gateway / BFF                                           │
  └────────┬───────────────────┬───────────────────┬───────────────┘
           │                   │                   │
  ┌────────▼─────────┐ ┌───────▼─────────┐ ┌───────▼─────────┐
  │ 🛒 Order Service │ │ 🕵️ Search Svc   │ │ 👥 Social Svc   │
  │ (Transactional)  │ │ (Full-text)     │ │ (Relationship)  │
  └────────┬─────────┘ └───────┬─────────┘ └───────┬─────────┘
           │                   │                   │
  ┌────────▼─────────┐ ┌───────▼─────────┐ ┌───────▼─────────┐
  │  ⭐️ PostgreSQL   │ │  ⭐️ Elastic     │ │  ⭐️ Neo4j       │
  │ (Acid / Orders)  │ │ (Inverted Idx)  │ │ (Nodes/Edges)   │
  └──────────────────┘ └─────────────────┘ └─────────────────┘
           ▲                   ▲                   ▲
           └───────── [ ⚡️ Kafka / CDC ] ─────────┘
             (실시간 데이터 동기화 파이프라인)

 * 핵심: 각 서비스는 자기가 제일 잘 다룰 수 있는 DB를 독립적으로 가짐.
```

### 심층 동작 원리

#### ① 데이터 캡슐화 (Data Encapsulation)
폴리글랏 퍼시스턴스의 핵심은 서비스 간에 DB를 공유하지 않는 것이다. 서비스 A가 서비스 B의 DB를 직접 조회(JOIN)하는 순간, 폴리글랏의 유연성은 사라지고 의존성의 늪에 빠진다. 모든 데이터 교환은 **API**나 **이벤트**를 통해서만 이루어져야 한다.

#### ② 최종적 일관성 (Eventual Consistency)
여러 DB에 데이터가 흩어져 있으므로, 한 번에 모든 DB를 업데이트하는 '분산 트랜잭션'은 가급적 피한다. 대신, 메인 DB(RDBMS)에 먼저 쓰고, 변경 이벤트를 Kafka로 쏴서 다른 DB(ES, Redis 등)들이 0.1초 뒤에 스스로를 갱신하게 만드는 방식을 주로 사용한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 단일 DB(Monolith) vs 폴리글랏 퍼시스턴스 비교

| 비교 지표 | Single Database Strategy | Polyglot Persistence Strategy |
|-----------|--------------------------|-------------------------------|
| **복잡도** | 낮음 (하나만 배우면 됨) | **높음 (여러 엔진 운영 지식 필수)** |
| **성능 최적화**| 한계가 명확함 | **최상 (목적별 최적의 엔진 사용)** |
| **정합성** | 강력함 (ACID Join 가능) | **최종적 일관성 (동기화 지연 발생)** |
| **비용** | 라이선스 비용 비쌈 | 인건비 및 관리 비용 상승 가능 |
| **운영** | 쉬움 (백업/복구 단일화) | 어려움 (DB별 백업 전략 상이) |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 한 동영상 스트리밍 플랫폼(Netflix 라이벌)이 성장하고 있다. 모든 데이터를 오라클에 넣었더니, "내가 찜한 목록" 조회와 "실시간 인기 순위" 연산 때문에 메인 결제 DB가 계속 느려지고 있다.
**판단 및 폴리글랏 전환 전략**:
1. **캐시 분리**: 잦은 조회가 발생하는 "사용자 세션"과 "영화 상세 정보"는 **Redis**로 옮겨 DB 부하를 80% 줄인다.
2. **검색 분리**: "영화 제목 검색"은 오라클 Like 검색을 버리고 **Elasticsearch**를 도입한다.
3. **추천 엔진 도입**: "이 영화를 본 사람이 본 다른 영화" 연산은 조인이 너무 무거우므로 **Graph DB(Neo4j)**를 도입하여 관계 쿼리 성능을 100배 높인다.
4. **결과**: 메인 DB(오라클)는 오직 결제와 회원 정보만 담당하게 되어 안전해지고, 서비스 전체 반응 속도는 비약적으로 빨라진다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **과도한 DB 파편화 (Golden Hammer의 역설)**: 개발자가 새로운 기술을 배우고 싶다고 해서, 서비스가 10개인데 DB 종류를 10개(MongoDB, Cassandra, Couchbase...)나 도입하는 것은 운영 지옥으로 가는 지름길이다. 기업 내에서 표준화된 3~4개의 DB 후보군(예: Postgres, Redis, ES)을 정해두고 그 안에서 선택하는 **'선택적 폴리글랏'**이 현명하다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **애플리케이션 성능**: 각 도메인별 최적 DB 사용으로 평균 응답 속도 **50~90% 단축**.
- **확장 편의성**: 트래픽이 몰리는 특정 기능(예: 검색)의 DB만 독립적으로 늘릴 수 있어 리소스 효율 극대화.

### 미래 전망 및 진화 방향
폴리글랏 퍼시스턴스는 이제 **'멀티 모델 데이터베이스(Multi-model DB)'**의 도전을 받고 있다. 최근의 DB들은 하나의 엔진 안에서 관계형, 문서형, 그래프형 데이터를 모두 지원하려 한다(예: AWS Aurora, PostgreSQL의 JSONB 지원 등). 하지만 진정한 고성능과 대규모 확장을 위해서는 여전히 물리적으로 분리된 폴리글랏 아키텍처가 우위에 있으며, 클라우드 네이티브 환경이 고도화될수록 이 경향은 더욱 짙어질 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **폴리글랏 퍼시스턴스가 뭔가요?**: 공부할 때는 연필을 쓰고, 그림 그릴 때는 붓을 쓰고, 종이를 자를 때는 가위를 쓰는 것처럼, 데이터 종류에 따라 가장 잘 어울리는 '전용 도구'를 골라 쓰는 거예요.
2. **왜 그냥 연필 하나만 안 쓰나요?**: 연필로 그림을 그릴 순 있지만 멋진 색칠을 하긴 힘들고, 연필로 종이를 자르려다간 종이가 다 찢어져 버리기 때문이에요.
3. **가장 좋은 점은요?**: 용도에 맞는 도구를 딱딱 골라 쓰니까 숙제도 훨씬 빨리 끝내고, 더 멋진 작품을 만들 수 있게 된답니다!
