# 309. CQRS 아키텍처와 DB 동기화 (Event Sourcing 연동)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템의 상태를 변경하는 **명령(Command)**과 상태를 조회하는 **조회(Query)**의 책임을 분리하고, 각각에 최적화된 이기종 데이터베이스(예: RDBMS ↔ NoSQL)를 사용하여 성능과 확장성을 극대화하는 아키텍처이다.
> 2. **가치**: 쓰기 중심의 정규화된 모델과 읽기 중심의 비정규화된 뷰(View) 모델을 독립적으로 진화시킬 수 있으며, 특히 조회(Read) 성능이 중요한 대규모 트래픽 환경에서 병목 현상을 원천 제거한다.
> 3. **융합**: 모든 상태 변경 이력을 이벤트로 저장하는 **이벤트 소싱(Event Sourcing)**과 결합하여, 이벤트를 재생(Replay)함으로써 읽기 전용 DB를 언제든 최신 상태로 재구축하거나 과거 시점의 데이터를 복원하는 강력한 동기화 체계를 완성한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
CQRS(Command Query Responsibility Segregation)는 전통적인 CRUD 모델의 한계를 극복하기 위해 제안되었다. 
- **Command Side**: 데이터 삽입, 수정, 삭제 수행. 강력한 일관성(ACID)과 비즈니스 로직 보호가 우선.
- **Query Side**: 데이터 조회 수행. 빠른 응답 속도와 사용자 UI에 맞춤화된 데이터 구조가 우선.

### 💡 비유
CQRS와 DB 동기화는 **"대형 서점의 재고 관리와 도서 검색기"**와 같다.
- **명령 (Command)**: 서점 직원이 창고(쓰기 DB)에 새 책을 넣고 장부를 고치는 일. 아주 꼼꼼해야 하고 시간이 걸린다.
- **조회 (Query)**: 손님이 입구의 컴퓨터(읽기 DB)로 책 위치를 찾는 일. 0.1초 만에 결과가 나와야 한다.
- **동기화 (Event Sourcing)**: 직원이 책을 넣을 때마다 "A책 입고!"라고 방송(Event)을 하면, 검색기 시스템이 그 방송을 듣고 자동으로 자기 목록을 업데이트하는 것과 같다.

### 등장 배경 및 발전 과정

#### 1. 복잡한 조회 쿼리의 고통
데이터가 많아질수록 10개 이상의 테이블을 JOIN 하는 조회 쿼리는 DB를 죽게 만든다. "조회용 테이블을 아예 미리 합쳐놓으면 어떨까?"라는 생각에서 시작되었다.

#### 2. MSA로의 전환
서비스가 쪼개지면서 여러 서비스의 데이터를 합쳐서 보여줘야 할 일이 많아졌다. 각 서비스의 DB를 실시간으로 JOIN 할 수 없으니, 데이터를 한곳으로 모아 조회 전용 DB를 만드는 CQRS가 필수가 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### CQRS와 이벤트 소싱의 시너지 (표)

| 구성 요소 | 상세 역할 | 동기화 방식 | 비유 |
|-----------|----------|-------------|------|
| **Command DB** | 유일한 진실의 원천(SSOT). 모든 이벤트의 원본 저장 | **Event Append (쓰기만 함)** | 일기장의 글쓰기 |
| **Event Bus** | 발생한 이벤트를 조회 측으로 나르는 통로 | **Message Streaming (Kafka)** | 우편 배달부 |
| **Projection** | 이벤트를 해석하여 조회용 DB에 맞게 가공 | **Asynchronous Sync (비동기)** | 편지 내용을 요약해 장부에 적기 |
| **Query DB** | 조회에 최적화된 데이터 구조 (JSON 등) | **Materialized View (실체화된 뷰)** | 요약된 장부 보기 |

### 정교한 구조 다이어그램 (Event Sourcing 연동 모델)

```ascii
================================================================================
[ Advanced CQRS Architecture with Event Sourcing ]
================================================================================

 [ Command Side ] (Write)                [ Query Side ] (Read)
 ┌─────────────────────┐                ┌─────────────────────┐
 │ 🛒 Order Service    │                │ 🕵️ Search Service   │
 └──────────┬──────────┘                └──────────▲──────────┘
            │                                      │
 (1) Create Order Event                            │ (4) Fast Query (No Join)
            ▼                                      │
 ┌─────────────────────┐                ┌──────────┴──────────┐
 │ 📦 Event Store (DB) │                │ ⚡️ Elasticsearch    │
 │ (Append-only Log)   │                │ (Read-optimized)    │
 └──────────┬──────────┘                └──────────▲──────────┘
            │                                      │
            │ (2) Publish Event                    │ (3) Project/Update
            └──────────────▶ [ ⚡️ Kafka ] ──────────┘
                             (Message Bus)

 * 핵심: 주문이 생성되면 '결과'를 저장하는게 아니라 '사건(Event)'을 저장함.
   이 이벤트를 Kafka가 낚아채서 검색 엔진(Elasticsearch)에 "조회하기 좋게" 
   모든 데이터를 미리 합쳐서(Flat JSON) 넣어둠.
```

### 심층 동작 원리

#### ① 비동기 동기화와 최종적 일관성
Command DB에 데이터가 저장된 후 Query DB에 반영되기까지 아주 짧은 시차(Lag)가 발생한다. 이를 **'최종적 일관성(Eventual Consistency)'**이라 한다. 사용자가 글을 쓰자마자 새로고침하면 안 보일 수 있지만, 0.1초 뒤에는 반드시 보이게 된다는 약속이다.

#### ② 프로젝션 (Projection)과 리플레이
조회 DB가 고장 나거나, 새로운 형태의 조회 화면이 필요해지면 어떻게 할까? 
- 이벤트 소싱이 연동되어 있다면, 과거의 모든 이벤트를 처음부터 다시 읽어(Replay) 새로운 조회 DB를 0에서부터 완벽하게 100% 재구축할 수 있다. 데이터 유실 걱정이 없는 무적의 아키텍처다.

## Ⅲ. 융합 비교 및 다각도 분석

### 일반 DB 동기화 vs CQRS/ES 동기화

| 비교 지표 | 단순 Read Replica (Master-Slave) | **CQRS + Event Sourcing** |
|-----------|----------------------------------|---------------------------|
| **데이터 모델** | 읽기/쓰기 모델이 동일함 | **읽기/쓰기 모델이 완전히 다름** |
| **저장소 선택** | 동일 제품군 (MySQL ↔ MySQL) | **이기종 혼용 (Postgres ↔ Mongo/ES)** |
| **복구 능력** | 백업 데이터에 의존 | **이벤트 재생을 통한 완벽 복구** |
| **구현 난이도** | 낮음 (설정 위주) | **매우 높음 (코드 복잡도 증가)** |
| **최적 용도** | 단순 부하 분산 | **복잡한 비즈니스 로직, 고성능 조회** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대규모 배달 앱 서비스. '내 주문 내역'을 보려면 주문, 결제, 배달, 리뷰 총 4개 서비스의 데이터를 합쳐야 한다. 각 서비스의 DB를 실시간으로 부르니 속도가 너무 느려 사용자 불만이 폭주한다.
**판단 및 CQRS 도입 전략**:
1. **이벤트 발행**: 주문, 결제 등 각 서비스가 상태가 변할 때마다 Kafka로 이벤트를 쏜다.
2. **Materialized View 구축**: **MongoDB**에 `User_Order_History`라는 조회 전용 컬렉션을 만든다. 
3. **Consumer 개발**: Kafka 이벤트를 듣고 있다가, 4개 서비스의 정보를 하나의 JSON 문서로 합쳐서 MongoDB에 덮어쓴다.
4. **결과**: 사용자가 주문 내역을 볼 때 JOIN 없이 MongoDB에서 문서 1개만 읽으면 끝난다. 조회 속도가 **2초에서 0.05초**로 40배 향상된다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **과도한 엔지니어링**: 단순한 게시판 사이트에 CQRS와 이벤트 소싱을 도입하는 것은 '닭 잡는 데 도끼'를 쓰는 격이다. 코드 복잡도가 수배로 늘어나고 운영 비용이 폭증한다. 반드시 **"조회 성능이 비즈니스의 핵심인가?"**를 자문해야 한다.
- **이벤트 누락**: 메시지 브로커(Kafka) 장애 시 이벤트가 유실되면 읽기 DB와 쓰기 DB가 영원히 달라진다. 반드시 **'Transactional Outbox'** 패턴을 함께 써서 DB 저장과 이벤트 발행이 한 몸으로 움직이게 보장해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **읽기 처리량(TPS)**: 단일 DB 대비 **10배 이상** 증가.
- **비즈니스 유연성**: 기존 기능을 건드리지 않고 새로운 읽기 전용 DB(예: 통계용)를 추가하는 데 소요되는 시간 **70% 단축**.

### 미래 전망 및 진화 방향
CQRS는 이제 클라우드 네이티브의 기본 소양이 되었다. 특히 최근에는 데이터 분석을 위한 **데이터 레이크하우스(Data Lakehouse)**와 결합하여, 운영 데이터(Command)를 실시간으로 분석용 레이크(Query)로 들이붓는 형태가 대세가 되고 있다. 또한, AI가 이벤트를 분석하여 최적의 조회용 인덱스를 스스로 생성하는 **'자율 튜닝 CQRS'** 시대로 진화할 전망이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **CQRS가 뭔가요?**: 요리를 하는 주방(명령)과 음식을 먹는 식탁(조회)을 아예 다른 방으로 분리하는 거예요. 
2. **왜 분리하나요?**: 요리사가 복잡하게 요리하는 곳에 손님이 와서 기웃거리면 위험하고 느려지니까, 요리가 끝나면 완성품만 식탁으로 쓱 옮겨주는 방식이랍니다.
3. **가장 좋은 점은요?**: 주방에서 아무리 바쁘게 칼질을 해도, 식탁에 앉은 손님은 이미 차려진 맛있는 음식을 1초 만에 바로 먹을 수 있어서 아주 편안해요!
