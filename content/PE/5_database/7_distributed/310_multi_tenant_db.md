# 310. 멀티테넌트 (Multi-tenant) 데이터베이스 구조

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 하나의 소프트웨어 인스턴스와 데이터베이스를 통해 **서로 독립된 여러 고객(Tenants)의 데이터를 안전하게 격리하고 동시에 서비스**하는 클라우드(SaaS) 기반의 데이터 아키텍처이다.
> 2. **가치**: 고객별로 인프라를 따로 구축하는 비효율을 없애 운영 비용을 획기적으로 낮추며, 수만 명의 고객 패치를 단 한 번의 업데이트로 끝낼 수 있는 강력한 유지보수 효율성을 제공한다.
> 3. **융합**: 논리적 스키마 분리, 물리적 인스턴스 분리, 그리고 공유 테이블 기반의 Row-level 격리 기술이 융합되어 보안성과 경제성 사이의 최적 균형을 찾는다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
멀티테넌시(Multi-tenancy)는 '다세대 주택'과 같다. 건물(인프라)은 하나지만, 각 방(데이터)은 벽으로 막혀 있어 옆집 사람이 내 방을 들여다볼 수 없다. SaaS(Software as a Service) 기업이 수많은 고객사 데이터를 효율적으로 관리하기 위한 필수 구조다.

### 💡 비유
멀티테넌트 DB 구조는 **"대형 호텔의 운영 방식"**과 같다.
- **Single-tenant (별장)**: 고객마다 집을 한 채씩 지어주는 것. 보안은 최고지만, 청소부(관리자)가 집집마다 돌아다녀야 해서 관리비가 엄청 비싸다.
- **Multi-tenant (호텔)**: 큰 건물 하나에 방이 수백 개 있다. 엘리베이터와 복도(CPU/MEM)는 같이 쓰지만, 자기 방(Data)은 카드키가 있어야 들어간다. 청소부는 건물 하나만 치우면 되니 관리비가 매우 싸다.

### 등장 배경 및 발전 과정

#### 1. 규모의 경제와 SaaS의 부상
Salesforce나 Slack 같은 서비스가 수만 개의 기업 고객을 수용하려면, 고객마다 서버와 DB를 따로 두는 방식으로는 절대 수익을 낼 수 없었다.

#### 2. 클라우드 네이티브의 진화
과거에는 단순히 테이블에 `tenant_id` 컬럼 하나 추가하는 수준이었으나, 이제는 개인정보 보호법(GDPR 등) 강화로 인해 "논리적으로는 합쳐져 있어도 물리적으로는 분리된 것과 같은" 고도의 격리 기술이 요구되고 있다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 멀티테넌시의 3가지 주요 모델 (표)

| 모델명 | 상세 내용 | 격리 수준 | 비용 효율 | 비유 |
|--------|-----------|-----------|-----------|------|
| **Database-level** | 고객마다 별도의 DB 인스턴스 사용 | **최상** (완벽 격리) | 최하 (자원 낭비) | 각자 자기 집 거주 |
| **Schema-level** | 하나의 DB 내에 고객별 스키마 분리 | 우수 (논리적 분리) | 보통 | 한 층씩 나눠 쓰기 |
| **Table-level** | 하나의 테이블을 공유하고 `tenant_id`로 구분 | 낮음 (어플리케이션 책임) | **최상** (자원 집중) | 한 방에서 칸막이 쓰기 |

### 정교한 구조 다이어그램 (하이브리드 멀티테넌시)

```ascii
================================================================================
[ Multi-tenant DB Architecture: Shared vs Isolated ]
================================================================================

  [ 🌐 Tenant A Request ]      [ 🌐 Tenant B Request ]      [ 🌐 Tenant C (VIP) ]
           │                            │                            │
  ┌────────▼────────────────────────────▼────────────────────────────▼────────┐
  │  🏢 Application SaaS Layer (Shared Logic)                                │
  │  - Routing Logic: "Which DB/Schema for this Tenant ID?"                  │
  └────────┬────────────────────────────┬────────────────────────────┬───────┘
           │                            │                            │
  ┌────────▼─────────┐        ┌─────────▼────────┐        ┌──────────▼────────┐
  │ 🗄️ Shared DB 1   │        │ 🗄️ Shared DB 2   │        │ 💎 Dedicated DB  │
  │                  │        │                  │        │ (Tenant C Only)  │
  │ ┌──────────────┐ │        │ ┌──────────────┐ │        └──────────────────┘
  │ │ Schema: T_A  │ │        │ │ Schema: T_B  │ │        * VIP 고객에게는 
  │ └──────────────┘ │        │ └──────────────┘ │          물리적으로 격리된 
  └──────────────────┘        └──────────────────┘          자원(Performance) 보장
```

### 심층 동작 원리

#### ① 데이터 격리 (Data Isolation)
애플리케이션 코드가 DB에 쿼리를 날릴 때, `tenant_id`가 빠진 쿼리는 아예 실행되지 못하게 가로채는(Interceptor) 기술이 핵심이다. 만약 개발자의 실수로 `WHERE tenant_id = 'A'`를 빼먹으면 다른 고객의 정보가 유출되는 참사가 발생하므로, 현대 아키텍처는 **DB View**나 **Row-level Security (RLS)** 기능을 써서 하드웨어 레벨에서 필터링한다.

#### ② 자원 경합과 '시끄러운 이웃' (Noisy Neighbor) 문제
한 호텔 방에서 파티를 열면 옆방이 시끄러운 것과 같다. 한 테넌트가 무거운 통계 쿼리를 날리면 같은 DB를 쓰는 다른 테넌트의 속도가 느려진다. 이를 막기 위해 **쿼리 할당량(Quota)**을 제한하거나, 부하가 심한 테넌트를 자동으로 다른 한가한 DB로 이사시키는 **'Dynamic Tenant Placement'** 기술이 사용된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 테넌트 격리 방식 트레이드오프

| 비교 지표 | 별도 DB 방식 | 공유 스키마 방식 | **공유 테이블 방식** |
|-----------|--------------|------------------|----------------------|
| **최대 테넌트 수**| 수십 개 | 수백 개 | **수만 개 이상** |
| **스키마 변경** | 지옥 (DB마다 실행) | 힘듦 | **쉬움 (한 번만 변경)** |
| **보안 사고 위험**| 거의 없음 | 낮음 | **높음 (코드 버그 시 유출)** |
| **백업/복구** | 고객별로 매우 쉬움 | 보통 | 매우 어려움 (부분 복구 힘듦) |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 새롭게 창업한 협업 툴(SaaS) 스타트업. 초기에는 비용 절감을 위해 '공유 테이블' 방식을 썼으나, 최근 대기업 고객들이 영입되면서 "우리 데이터가 다른 중소기업과 같은 테이블에 섞여 있는 것을 용납할 수 없다"며 물리적 격리를 요구하고 있다.
**판단 및 하이브리드 아키텍처 전환 전략**:
1. **Sharding by Tier**: 고객 등급에 따라 아키텍처를 나눈다.
2. **Standard Tier**: 저렴한 가격의 고객들은 기존처럼 **공유 테이블** 방식을 유지하여 수익성을 높인다.
3. **Enterprise Tier**: 비싼 값을 지불하는 대기업 고객은 **Database-per-tenant** 방식을 적용하여 전용 RDS 인스턴스를 프로비저닝 해준다.
4. **Tenant Router 도입**: 앱 앞단에 `Tenant-Mapping-Table`을 두어, 로그인 정보에 따라 각기 다른 DB 커넥션 풀을 동적으로 선택하게 한다.
5. **결과**: 보안 요건이 까다로운 대기업을 만족시키면서도 전체 운영 비용은 최적화된 상태를 유지한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **과도한 인스턴스 분리**: 고객이 100명인데 DB 서버를 100대 띄우는 것은 관리 오버헤드 때문에 회사가 망하는 지름길이다. 반드시 **관리 자동화(Terraform/Operator)**가 준비된 상태에서만 분리 아키텍처를 고려해야 한다.
- **공통 스키마 업데이트 누락**: 테이블 공유 방식에서 컬럼 하나를 추가했는데, 특정 고객의 커스텀 로직과 충돌하여 에러가 나는 경우가 잦다. 모든 테넌트의 코드를 한꺼번에 검증하는 **회귀 테스트 자동화**가 필수다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **운영 비용**: 서버 당 테넌트 밀집도 향상으로 인프라 비용 **60~80% 절감**.
- **배포 민첩성**: 수천 명의 고객 기능을 단 한 번의 CI/CD로 동시 업그레이드.

### 미래 전망 및 진화 방향
멀티테넌시 기술은 이제 **'서버리스(Serverless) DB'**와 결합하고 있다. **Amazon Aurora Serverless**나 **CockroachDB Serverless**는 데이터가 없을 때는 자원을 거의 안 먹다가, 특정 테넌트의 요청이 올 때만 0.1초 만에 자원을 할당한다. 미래의 SaaS는 물리적 서버의 개념이 완전히 사라지고, 수억 개의 테넌트가 논리적인 가상 공간에서 자원을 숨 쉬듯 나눠 쓰는 **'지능형 유동 아키텍처'**로 진화할 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **멀티테넌트가 뭔가요?**: 아파트 한 동에 여러 가족이 모여 사는 것과 같아요. 엘리베이터는 같이 쓰지만, 우리 집 거실(데이터)은 우리 가족만 볼 수 있는 거예요.
2. **왜 그렇게 하나요?**: 가족마다 아파트를 한 동씩 따로 지으면 땅도 많이 차지하고 돈도 너무 많이 들지만, 같이 살면 관리비를 훨씬 아낄 수 있기 때문이에요.
3. **가장 조심해야 할 점은요?**: 옆집 소음이 우리 집까지 들리거나(시끄러운 이웃), 우리 집 문 잠그는 걸 깜빡해서 옆집 사람이 들어오지 않게(보안 유출) 관리실 아저씨가 아주 잘 감시해야 한답니다!
