# 311. 데이터베이스 퍼징 (Database Fuzzing) 및 취약점 테스트

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 관리 시스템(DBMS)의 안정성과 보안성을 검증하기 위해, 유효하지 않거나 예상치 못한 **무작위 데이터(Fuzz)나 비정상적인 SQL 쿼리를 대량으로 주입**하여 시스템의 크래시, 메모리 누수, 보안 취약점을 찾아내는 동적 테스트 기법이다.
> 2. **가치**: 일반적인 기능 테스트로는 발견하기 어려운 SQL 인젝션 변종, 버퍼 오버플로우, 논리적 정합성 오류를 사전에 탐지하여 데이터 유출 및 서비스 중단 사고를 예방한다.
> 3. **융합**: 변이 기반(Mutation-based) 생성 기술, 화이트박스 코드 커버리지 분석, 그리고 클라우드 네이티브 기반의 분산 테스트 인프라와 융합되어 고신뢰성 DB 엔진 개발의 필수 과정으로 자리 잡았다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
퍼징(Fuzzing)은 "바보 같은 질문을 던져서 똑똑한 척하는 시스템을 당황하게 만드는 것"이다. 데이터베이스 퍼징은 DB 엔진의 쿼리 파서(Parser), 실행 계획기(Optimizer), 스토리지 엔진 등을 대상으로 비정상적인 입력을 쏟아부어 약점을 찾는 행위다.

### 💡 비유
데이터베이스 퍼징은 **"금고의 내구성을 테스트하는 망치질"**과 같다.
- **일반 테스트**: 열쇠를 넣고 돌리면 열리는지 확인하는 것.
- **퍼징**: 금고를 거꾸로 흔들어보고, 열쇠 구멍에 껌을 채워보고, 망치로 아무 데나 두드려보고, 불로 지져보는 것. 예상치 못한 상황에서도 금고(DB)가 폭발하거나 문이 덜컥 열리지(보안 뚫림) 않는지 확인하는 과정이다.

### 등장 배경 및 발전 과정

#### 1. SQL 인젝션의 진화
해커들이 단순히 `' OR 1=1 --` 같은 고전적인 수법이 아니라, 복잡한 중첩 쿼리와 특수 문자를 조합하여 방화벽을 우회하기 시작했다.

#### 2. DB 엔진 자체의 복잡성
현대 DB 엔진은 수백만 줄의 코드로 이루어져 있다. 사람이 모든 예외 케이스를 예측하여 테스트하는 것은 불가능해졌고, 이에 따라 컴퓨터가 자동으로 무작위 쿼리를 생성하여 DB를 공격해보는 '자동화된 퍼징' 기술이 발전했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 데이터베이스 퍼징의 3대 유형 (표)

| 유형 | 방식 상세 | 특징 | 비유 |
|------|-----------|------|------|
| **Generation-based** | SQL 문법 규칙을 학습하여 처음부터 새로운 쿼리 생성 | 문법적으로 유효하지만 논리적으로 괴상한 쿼리 생성 | 소설 쓰기 |
| **Mutation-based** | 정상적인 SQL 샘플을 조금씩 비틀어서 변이 생성 | 기존 보안 정책을 살짝 우회하는 케이스 탐지에 유리 | 오타 내기 |
| **Differential Fuzzing**| 두 개 이상의 DB(예: MySQL vs MariaDB)에 같은 쿼리를 날려 결과 비교 | **논리적 정합성 오류**를 찾는 데 탁월 | 대조 심문 |

### 정교한 구조 다이어그램 (Feedback-driven Fuzzer 아키텍처)

```ascii
================================================================================
[ Feedback-driven Database Fuzzing Loop ]
================================================================================

  ┌──────────────────────────────────────────────────────────┐
  │  🎯 Fuzzing Engine (e.g. AFL, SQLSmith)                  │
  │  - Mutation Logic: bit flip, byte replacement            │
  │  - Syntax Logic: SQL Grammar-aware generation            │
  └─────────────┬────────────────────────────────────────────┘
                │
                │ (1) Inject Fuzzed SQL 💉
                ▼
  ┌──────────────────────────────────────────────────────────┐
  │  🗄️ Target DBMS (Instrumented)                           │
  │  - Query Parser / Optimizer / Executor                   │
  └─────────────┬────────────────────────────────────────────┘
                │
                │ (2) Feedback: Coverage & Crashes 📊
                ▼
  ┌──────────────────────────────────────────────────────────┐
  │  ⚖️ Monitor & Analyzer                                    │
  │  - Did it crash? (Memory leak / Segfault)                │
  │  - New code path reached?                                │
  │  - Save "Interesting" inputs to Corpus                   │
  └─────────────┬────────────────────────────────────────────┘
                │
                └─▶ (3) Update Strategy and Loop Back to (1)
```

### 심층 동작 원리

#### ① 문법 인지형 퍼징 (Grammar-aware Fuzzing)
단순 무작위 바이트 주입은 DB 입구(Parser)에서 바로 컷 당한다. 고도화된 퍼저는 SQL 문법 구조를 이해하여 `SELECT [RANDOM_COL] FROM [RANDOM_TABLE] WHERE [COMPLEX_CONDITION]` 같은 유효한 명령어를 만들어 깊숙한 스토리지 엔진 로직까지 파고든다.

#### ② 코드 커버리지 피드백
퍼징 엔진은 DB 내부의 어떤 코드 라인이 실행되었는지 감시한다. 만약 어떤 무작위 쿼리가 이전에는 한 번도 안 가본 '깊은 골목길 코드'를 건드렸다면, 엔진은 그 쿼리를 '우수한 샘플'로 저장하고 그 주변을 집중적으로 더 공격한다.

#### ③ 교차 검증 (Differential Testing)
동일한 쿼리를 날렸는데 MySQL은 `10`을 리턴하고 PostgreSQL은 `11`을 리턴한다면? 두 DB 중 하나는 연산 로직에 심각한 버그(논리 오류)가 있다는 뜻이다. 이를 통해 소스 코드 크래시가 없더라도 데이터 오염을 찾아낼 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### 일반 테스트 vs 퍼징(Fuzzing)

| 비교 지표 | Unit / Integration Test | **Database Fuzzing** |
|-----------|--------------------------|----------------------|
| **테스트 시나리오**| 사람이 정의한 시나리오 | **컴퓨터가 만드는 무작위 시나리오** |
| **예측 가능성** | 높음 (정해진 결과 확인) | **낮음 (예상치 못한 버그 발견)** |
| **발견 버그** | 비즈니스 로직 오류 | **보안 취약점, 메모리 손상, 정합성 파괴** |
| **수행 시간** | 짧음 | **매우 김 (수일~수주간 지속 실행)** |
| **비유** | 정해진 길로 걷기 | 지뢰 찾기 위해 온 땅 두드리기 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 기업 내부에서 커스텀하게 수정한 오픈소스 DB(예: MySQL 기반 특수 목적 DB)를 운영 환경에 배포하려 한다. 기본 기능은 잘 돌아가지만, 대규모 트래픽이나 복잡한 쿼리에서 데이터 유실이나 보안 구멍이 있을까 봐 불안하다.
**판단 및 퍼징 수행 전략**:
1. **SQLsmith 도입**: 문법 인지형 퍼징 도구인 **SQLsmith**를 사용하여 수백만 개의 복잡한 JOIN/Subquery를 24시간 쏟아붓는다.
2. **ASan (Address Sanitizer) 결합**: DB 빌드 시 ASan 옵션을 켜서, 퍼징 도중 발생하는 미세한 메모리 오염(Use-after-free 등)을 즉시 감지하게 한다.
3. **Differential Test**: 원본 MySQL과 커스텀 DB에 동일 쿼리를 날려 결과값이 1비트라도 틀린지 대조한다.
4. **결과**: 사람이 1년 동안 테스트해도 못 찾았을 '특정 조건에서의 데이터 정합성 버그' 3건을 일주일 만에 발견하여 수정 후 안전하게 배포한다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **운영 DB에서 퍼징 수행**: 퍼징은 시스템을 '망가뜨리기 위해' 하는 작업이다. 절대로 운영 환경이나 실제 데이터가 담긴 곳에서 하면 안 된다. 반드시 **격리된 샌드박스 환경**에서 수행해야 한다.
- **결과 분석 방치**: 퍼징 엔진이 수만 개의 크래시 리포트를 뱉어내도 사람이 분석하지 않으면 소용없다. 중복되는 오류를 그룹화하고 우선순위를 정하는 **'Triage'** 프로세스가 동반되어야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **제로데이(Zero-day) 방어**: 알려지지 않은 보안 취약점을 사전에 **80% 이상 제거**.
- **시스템 가용성**: 극한의 입력 상황에서도 서버가 뻗지 않는 '견고한(Robust)' 엔진 완성.

### 미래 전망 및 진화 방향
데이터베이스 퍼징은 이제 **'AI 강화학습'**과 결합하고 있다. AI가 DB 엔진의 취약 지점을 스스로 학습하여 가장 효과적으로 서버를 죽일 수(?) 있는 쿼리를 골라내는 **'Smart Fuzzing'**이 대두되고 있다. 또한, 클라우드 환경에서 수만 개의 퍼징 인스턴스를 동시에 돌리는 **'Fuzzing as a Service'**가 엔터프라이즈 소프트웨어 품질 보증의 새로운 표준이 될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **데이터베이스 퍼징이 뭔가요?**: 컴퓨터 대장(DB)이 얼마나 튼튼한지 보려고, "말도 안 되는 엉터리 질문"을 수억 번 던져서 골탕을 먹여보는 거예요.
2. **왜 그런 짓을 하나요?**: 가끔 엉터리 질문을 받았을 때 컴퓨터가 당황해서 비밀번호를 알려주거나 아예 고장 날 수 있는데, 그걸 미리 찾아내서 고쳐주기 위해서예요.
3. **가장 좋은 점은요?**: 나중에 진짜 나쁜 악당이 엉뚱한 공격을 해도, 이미 연습을 많이 한 컴퓨터 대장은 눈 하나 깜짝 안 하고 안전하게 우리 데이터를 지켜준답니다!
