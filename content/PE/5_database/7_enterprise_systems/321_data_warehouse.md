# 321. 데이터 웨어하우스 (Data Warehouse, DW) - 기업 의사결정의 등대

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기업 내 여러 시스템(인사, 재무, 영업 등)에 흩어져 있는 데이터들을 추출, 정제, 변환(ETL)하여 한곳에 모아둔 **의사결정 지원 목적의 거대한 중앙 데이터 저장소**이다.
> 2. **혁신**: 일반적인 운영 DB(OLTP)가 데이터의 빠르고 안전한 갱신(Update)에 초점을 맞춘다면, DW는 방대한 양의 과거 데이터를 바탕으로 한 **복잡한 통계 분석과 읽기(Read/OLAP)에 최적화**되어 있다.
> 3. **가치**: "우리 회사의 지난 5년간 지역별, 분기별 매출 추이는 어떠한가?"와 같은 경영진의 전략적 질문에 즉각적이고 정확한 단일 진실(Single Source of Truth)을 제공한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
데이터 웨어하우스(DW)는 1990년대 빌 인몬(Bill Inmon)에 의해 "주제 지향적(Subject-oriented)이고, 통합적(Integrated)이며, 시계열적(Time-variant)이고, 비휘발성(Non-volatile)인 데이터의 집합"으로 정의되었다. 기업의 모든 히스토리 데이터가 예쁘게 정돈된 거대한 도서관이다.

### 💡 비유
- **운영 DB (OLTP)**: 동네 마트의 포스기(계산대)다. 지금 당장 손님이 산 물건과 거스름돈을 빠르고 정확하게 처리하는 것이 목적이다. 너무 바빠서 작년 매출 장부를 찾아볼 여유가 없다.
- **데이터 웨어하우스 (DW)**: 마트 본사의 통계실이다. 전국의 모든 포스기에서 하루 일과가 끝나면 영수증을 모아 정해진 규격의 결산 장부로 옮겨 적어 보관한다. 이곳에서는 "작년 여름에 제일 잘 팔린 아이스크림"을 아주 쉽게 분석할 수 있다.

### 등장 배경: '정보의 섬(Islands of Information)' 문제
과거 기업들은 부서마다 서로 다른 DB를 썼다. 사장님이 "이번 달 매출이 얼마인가?" 물어보면 영업부와 재무부의 숫자가 달랐다. 흩어진 섬들을 하나로 연결하여 모두가 똑같은 기준과 포맷으로 데이터를 바라보기 위해 DW가 탄생했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 데이터 웨어하우스 기본 아키텍처 (ASCII Diagram)

```ascii
[ Data Sources ]       [ ETL Process ]        [ Data Warehouse ]       [ BI & Analytics ]
┌────────────┐        ┌──────────────┐        ┌────────────────┐       ┌──────────────┐
│  ERP (SQL) │        │  Extract (E) │        │                │       │  Dashboards  │
├────────────┤ ─────> │              │ ─────> │  Enterprise    │ ────> │  (Tableau)   │
│  CRM (Web) │        │ Transform(T) │        │  Data          │       ├──────────────┤
├────────────┤ ─────> │  (Cleansing) │ ─────> │  Warehouse     │ ────> │  Data Mining │
│  Log Files │        │              │        │                │       │  (AI/ML)     │
└────────────┘        │   Load (L)   │        └────────────────┘       └──────────────┘
                      └──────────────┘                ▲
                                                      │ (Data Marts)
                                             [ Sales ] [ HR ] [ Marketing ]
```

### 3대 핵심 구성 요소

#### ① ODS (Operational Data Store)
소스 시스템에서 데이터를 뽑아 DW로 넣기 전, 임시로 모아두는 정거장이다. 원본 데이터의 구조를 그대로 유지하며 1차적인 통합만 수행한다.

#### ② ETL (Extract, Transform, Load) 파이프라인
DW의 심장이다. 이기종 시스템에서 데이터를 뽑아(Extract), 회사 표준 포맷으로 날짜나 단위를 맞추고 결측치를 제거한 뒤(Transform), DW에 적재(Load)하는 일련의 과정이다.

#### ③ Data Mart (데이터 마트)
거대한 DW에서 영업부, 마케팅부 등 **특정 부서의 목적에 맞게 필요한 데이터만 따로 떼어내어 만든 소규모 데이터 웨어하우스**다. (할인 매장의 코너와 같다).

## Ⅲ. 융합 비교 및 다각도 분석

### OLTP (운영 DB) vs OLAP (데이터 웨어하우스) 상세 비교

| 비교 항목 | OLTP (Online Transaction Processing) | **OLAP / DW (Online Analytical Processing)** |
|-----------|--------------------------------------|----------------------------------------------|
| **주요 목적** | 일상적인 비즈니스 업무 처리 (주문, 이체) | **경영진의 의사결정 지원 및 통계 분석** |
| **데이터 구조**| 정규화 (중복 제거 최우선) | **반정규화 (조회 속도 최우선, 다차원 구조)** |
| **작업 형태** | 짧고 빠른 CRUD 트랜잭션 다수 | **소수의 복잡하고 거대한 SELECT 쿼리** |
| **데이터 범위**| 현재 상태 위주의 데이터 | **과거 수년~수십 년의 이력 데이터 포함** |
| **설계 철학** | 쓰기(Write) 최적화 | **읽기(Read) 최적화** |

### Inmon vs Kimball 아키텍처 논쟁
- **Bill Inmon (Top-down)**: 엔터프라이즈 전체의 거대한 정규화된 DW를 먼저 짓고, 거기서 데이터를 뽑아 부서별 데이터 마트를 나중에 만드는 방식 (철저한 중앙 통제).
- **Ralph Kimball (Bottom-up)**: 부서별로 데이터 마트(다차원 모델)를 먼저 빠르고 작게 만든 다음, 이것들을 합쳐서 전체 DW를 구성하는 방식 (민첩성 중시, 현대에 더 많이 쓰임).

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 도입 전략)

**상황**: 급성장 중인 유통 스타트업에서 데이터가 쌓이면서 운영 DB에 직접 통계 쿼리를 날렸다가 서비스가 마비되는 일이 발생했다.
**판단 및 DW 구축 전략**:
1. **분리(Isolation)**: 트랜잭션 DB(MySQL)와 분석용 DB를 물리적으로 완벽히 분리한다. 분석 쿼리가 고객의 결제를 방해해선 안 된다.
2. **솔루션 선정**: 초기 구축 비용을 줄이고 확장성을 확보하기 위해 **클라우드 네이티브 DW(Google BigQuery 또는 Snowflake)**를 도입한다.
3. **ETL 자동화**: 매일 새벽 3시에 전날 발생한 데이터를 DW로 옮기는 배치(Batch) 파이프라인을 구축한다.
4. **결과**: 운영 서비스 성능은 100% 보장하면서, 마케팅 팀은 언제든 수천만 건의 데이터를 3초 안에 분석하여 타겟 마케팅을 할 수 있게 됨.

### 안티패턴 및 고려사항
- **가비지 인 가비지 아웃 (GIGO)**: ETL 과정에서 데이터 정제(Cleansing)를 소홀히 하여 쓰레기 데이터가 DW에 들어가면, 경영진은 그 쓰레기를 보고 잘못된 전략을 세우게 된다. 데이터 품질 거버넌스가 기술보다 더 중요하다.
- **실시간 강박**: DW는 기본적으로 '배치(Batch)' 기반이다. 모든 것을 0.1초 단위의 실시간으로 DW에 넣으려 하면 비용이 폭증하고 시스템이 붕괴한다. 실시간이 꼭 필요한 데이터만 스트리밍 아키텍처로 분리해야 한다.

## Ⅴ. 미래 전망 및 결론

### 결론: 비즈니스 인텔리전스의 영원한 고향
데이터 웨어하우스는 30년이 넘은 개념이지만 여전히 데이터 아키텍처의 중심에 있다. 데이터가 돈이 되는 시대에, 흩어진 모래알을 모아 순금으로 제련하는 DW의 역할은 결코 대체될 수 없다.

### 미래 전망
현대의 DW는 클라우드를 만나 진화하고 있다. 스토리지와 컴퓨팅 자원을 완벽히 분리하여 무한대로 확장하는 **Cloud Data Warehouse**가 대세가 되었으며, 최근에는 비정형 데이터를 저장하는 데이터 레이크(Data Lake)의 장점과 융합하여 **데이터 레이크하우스(Lakehouse)**라는 궁극의 플랫폼으로 발전하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **데이터 웨어하우스가 뭔가요?**: 집안 곳곳에 흩어져 있는 장난감, 책, 옷들의 위치와 정보를 한눈에 볼 수 있게 아주 예쁘게 정리해 놓은 '거대한 도서관'이에요.
2. **왜 따로 만드나요?**: 슈퍼마켓 계산대(운영 DB)에서 장사를 막 하고 있는데, 사장님이 "작년에 바나나 몇 개 팔렸어?" 하고 계산기를 뒤적거리면 손님들이 짜증 나고 장사가 멈춰버리기 때문이에요.
3. **가장 좋은 점은?**: 사장님들이 도서관에 와서 "우리가 돈을 어디서 많이 벌었지?"라는 아주 복잡한 질문에 대한 정답을 금방 찾을 수 있답니다!
