# 323. 데이터 마트 (Data Mart) - 부서 맞춤형 데이터 편의점

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 거대한 전사적 데이터 웨어하우스(DW) 환경에서, 특정 부서(영업, 마케팅, 인사 등)나 특정 사용자의 요구 목적에 맞게 **소규모로 가공하여 떼어낸 주제별 데이터 저장소**이다.
> 2. **혁신**: 방대하고 복잡한 엔터프라이즈 DW 전체를 뒤질 필요 없이, 내 부서 업무에 딱 맞는 요약된 데이터만 다차원 모델(Star Schema 등)로 구성하여 제공하므로 **분석 쿼리 성능이 극대화**된다.
> 3. **가치**: 랄프 킴볼(Ralph Kimball)의 상향식(Bottom-up) 아키텍처의 핵심 요소로, 구축 시간이 짧고 비용이 저렴하여 현업 부서의 실질적이고 즉각적인 비즈니스 인텔리전스(BI) 요구를 해결한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
데이터 마트(Data Mart, DM)는 엔터프라이즈 데이터 웨어하우스(EDW)의 축소판이다. EDW가 전사적 관점의 거대한 중앙 도서관이라면, 데이터 마트는 마케팅팀 전용 서가, 재무팀 전용 서가처럼 특정 타겟을 위해 데이터를 미리 분류하고 요약해 놓은 공간이다.

### 💡 비유
- **데이터 웨어하우스 (초대형 도매 센터)**: 코스트코 같은 거대한 창고형 매장이다. 세상 모든 물건(데이터)이 다 있지만, 우유 하나를 사려면 매장 전체를 돌아다니며 엄청난 시간을 소비해야 한다.
- **데이터 마트 (동네 편의점)**: 거대 창고에서 '과자'와 '음료수'만 떼어내 우리 집 앞에 차려놓은 소형 편의점이다. 물건 종류는 적지만, 내가 원하는 과자를 10초 만에 바로 찾아 결제(분석)할 수 있다.

### 등장 배경: 'Too Big to Fail?' No, Too Big to Use!
전사적 DW 구축(Inmon 방식)은 모든 부서의 요구사항을 반영하려다 보니 수년의 시간이 걸리고 수십억 원이 들었다. 현업 부서는 당장 이번 달 마케팅 분석을 해야 하는데 DW가 완성될 때까지 기다릴 수 없었다. 그래서 작고 빠르게, 부서별 목적에 딱 맞는 '데이터 마트'를 먼저 짓는 방식(Kimball 방식)이 대세로 떠올랐다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 데이터 마트의 위치적 아키텍처 (ASCII Diagram)

```ascii
[ Data Sources ]       [ Enterprise DW ]        [ Data Marts ]         [ BI / Users ]
┌────────────┐        ┌────────────────┐       ┌──────────────┐       ┌──────────────┐
│  ERP       │        │                │  ETL  │ Sales Mart   │ ────> │ Sales Team   │
├────────────┤  ETL   │  Centralized   │ ────> ├──────────────┤       ├──────────────┤
│  CRM       │ ─────> │  Data          │       │ Finance Mart │ ────> │ Finance Team │
├────────────┤        │  Warehouse     │ ────> ├──────────────┤       ├──────────────┤
│  Web Logs  │        │  (Normalized)  │       │ Mktg Mart    │ ────> │ Mktg Team    │
└────────────┘        └────────────────┘       └──────────────┘       └──────────────┘
                       <-- Inmon 방식 (Top-down: DW 먼저, DM 나중) -->

[ Data Sources ] ──ETL──> [ Data Marts (Star Schema) ] ──조합──> [ Virtual DW (Kimball) ]
                       <-- Kimball 방식 (Bottom-up: DM 먼저 짓고 묶어서 DW화) -->
```

### 핵심 설계 기법

#### ① 다차원 모델링 (Dimensional Modeling)
데이터 마트는 쿼리 속도와 현업의 이해도를 높이기 위해 철저히 다차원 모델(스타 스키마, 스노우플레이크 스키마)로 설계된다.
- **팩트(Fact) 테이블**: '매출액', '수량' 등 분석의 대상이 되는 측정값.
- **차원(Dimension) 테이블**: '시간', '지역', '제품' 등 데이터를 잘라보는 기준.

#### ② 요약 및 집계 (Aggregation)
DW에는 개별 영수증(상세 데이터)이 있다면, 마트에는 '일별 지점 매출 합계'처럼 미리 계산된 요약 데이터가 주로 담긴다. 이 덕분에 대시보드 로딩 속도가 비약적으로 빠르다.

#### ③ 종속적 마트 vs 독립적 마트
- **종속적 데이터 마트 (Dependent)**: 중앙 DW에서 데이터를 뽑아와서 생성한다. 데이터 정합성이 일치한다. (권장)
- **독립적 데이터 마트 (Independent)**: DW를 거치지 않고 원본 시스템에서 직접 데이터를 뽑아와 만든다. 부서별로 수치가 달라지는 데이터 사일로(Silo)를 유발할 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### 데이터 웨어하우스(DW) vs 데이터 마트(DM) 비교

| 비교 항목 | 엔터프라이즈 DW | **데이터 마트 (Data Mart)** |
|-----------|------------------|-----------------------------|
| **목적** | 전사적 데이터 통합 관리 | **특정 부서/주제의 분석 및 리포팅** |
| **데이터 범위**| 기업 전체 데이터 | **특정 업무 영역 (Sales, HR 등)** |
| **데이터 구조**| 정규화 구조 (3NF 등) | **비정규화 다차원 구조 (스타 스키마)** |
| **크기** | 수십 TB ~ PB (거대함) | **수십 GB ~ 수 TB (비교적 작음)** |
| **구축 시간** | 수개월 ~ 수년 | **수주 ~ 수개월 (빠름)** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 구축 전략)

**상황**: 대형 백화점에서 경영진용 통합 DW를 구축 중이나 일정이 지연되고 있다. 마케팅팀은 당장 다음 주부터 '지역별/연령대별 프로모션 반응률'을 분석할 대시보드가 필요하다고 아우성이다.
**판단 및 데이터 마트 구축 전략**:
1. **Conformed Dimensions (공통 차원) 정의**: 킴볼(Kimball) 방법론을 적용하여, 전사에서 공통으로 쓸 '날짜', '매장', '고객' 차원 테이블을 가장 먼저 정의한다.
2. **독립적 마트 구축 (Agile)**: 마케팅 캠페인 팩트 테이블을 빠르게 설계하고, ODS에서 직접 데이터를 당겨와 마케팅 전용 **독립적 데이터 마트**를 2주 만에 구축한다.
3. **추후 통합**: 차후 다른 부서(영업 등)의 마트가 지어지면, 처음에 정의한 공통 차원(Conformed Dimensions)을 링크고리로 삼아 여러 마트들을 논리적인 하나의 DW(Data Warehouse Bus Architecture)로 묶어낸다.
4. **결과**: 현업의 긴급한 분석 요구를 즉각 해소(Quick Win)하면서도 전사 데이터 정합성을 해치지 않는 아키텍처 진화 달성.

### 안티패턴 및 고려사항
- **데이터 사일로의 귀환**: 부서마다 자기들 입맛대로 독립적 마트를 중구난방으로 만들면, 마케팅팀의 '총매출'과 재무팀의 '총매출' 결과가 달라지는 최악의 참사가 벌어진다. **반드시 중앙 거버넌스 하에 공통 차원(Dimension)을 공유해야 한다.**

---

## Ⅴ. 미래 전망 및 결론

### 결론: 애자일(Agile) 데이터 아키텍처의 표본
데이터 마트는 거대하고 무거운 IT 프로젝트(EDW)를 작고 기민한 비즈니스 단위로 쪼개어 가치를 빠르게 창출하는 실용주의적 접근의 승리다.

### 미래 전망
클라우드 DW(Snowflake, BigQuery) 시대가 되면서 스토리지와 컴퓨팅 자원이 넘쳐나게 되었다. 이로 인해 물리적으로 데이터를 복사해서 마트를 따로 짓는 대신, 거대한 클라우드 DW 내부에서 **가상 데이터 마트 (Virtual Data Mart / 논리적 View)**를 즉석에서 생성하여 권한만 분리해주는 형태로 진화하고 있다. 데이터 복사(ETL) 비용은 줄이고 마트의 편리함은 유지하는 구조다.

## 👶 어린이를 위한 3줄 비유 설명
1. **데이터 마트가 뭔가요?**: 세상 모든 책이 다 있는 무지막지하게 큰 도서관(DW)에서, '공룡 그림책'만 따로 모아서 내 방 책상에 꽂아둔 아주 작은 '나만의 책장'이에요.
2. **왜 책장을 따로 만드나요?**: 공룡 책이 보고 싶을 때마다 매번 거대한 도서관을 뱅뱅 돌며 고생할 필요 없이, 내 방에서 1초 만에 꺼내 볼 수 있거든요.
3. **가장 좋은 점은?**: 요리사는 요리 책장, 과학자는 우주 책장처럼 자기가 제일 필요한 정보만 모아놓아서 일하기가 엄청나게 편해진답니다!
