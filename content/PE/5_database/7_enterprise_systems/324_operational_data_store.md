# 324. ODS (Operational Data Store) - 분석과 운영 사이의 데이터 환승역

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 여러 개의 원천 운영 시스템(OLTP)에서 생성된 데이터를 데이터 웨어하우스(DW)로 보내기 전에, 일시적으로 모아서 **1차적인 통합과 정제를 수행하는 임시 데이터 저장소**이다.
> 2. **혁신**: 운영 DB에 무거운 분석 쿼리를 날려 시스템이 다운되는 것을 막아주는 '버퍼(Buffer)' 역할을 하며, DW처럼 과거 이력이 아닌 **가장 최신의(Current) 통합 데이터**를 제공한다.
> 3. **가치**: DW가 제공하지 못하는 '실시간~준실시간(Near Real-time) 전사 통합 리포팅'을 가능하게 하여, 현업의 일상적인 실무 운영(Operational) 의사결정을 강력하게 지원한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
ODS(Operational Data Store)는 데이터 아키텍처의 중간 기착지다. 각기 다른 시스템(영업, 재무, 인사)에서 쏟아지는 데이터를 1차로 모아 포맷을 맞추고 중복을 제거한다. 여기서 정제된 깨끗한 데이터가 최종적으로 거대한 DW로 흘러 들어가게 된다.

### 💡 비유
- **원천 DB (각 가정의 쓰레기통)**: 데이터가 원초적인 형태로 제각각 쌓여 있다.
- **DW (최종 재활용 공장)**: 데이터를 수십 년간 압축 보관하고 복잡한 통계를 낸다. 처리 시간이 오래 걸린다.
- **ODS (지역 쓰레기 집하장)**: 각 가정에서 나온 데이터를 일단 한곳에 모아 종이는 종이대로, 캔은 캔대로 1차 분류(정제)를 하는 곳이다. 공장(DW)으로 가기 전의 깨끗한 최신 상태이며, 언제든 어떤 쓰레기가 들어왔는지 바로바로 확인할 수 있다.

### 등장 배경: DW의 느린 속도와 OLTP의 부하
과거에는 운영 DB에서 바로 DW로 데이터를 쏘았다. 하지만 이 변환(ETL) 과정은 너무 무거워 야간에만 한 번씩 돌려야 했다(T+1일 지연). 현업에서는 "당장 오늘 오후 2시 기준 전사 통합 매출 현황"을 보고 싶은데 DW는 내일 아침에나 볼 수 있고, 운영 DB는 시스템이 뻗을까 봐 조회를 못 하게 막았다. 이 간극을 메우기 위해 실시간 조회가 가능한 통합 DB인 ODS가 등장했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### ODS가 포함된 엔터프라이즈 데이터 흐름도 (ASCII Diagram)

```ascii
[ Data Sources (OLTP) ]          [ ODS ]                [ Data Warehouse ]
┌────────────┐   (Real-time/  ┌──────────────┐   (Batch   ┌────────────────┐
│  CRM (Web) ├─── Near-time)─>│              │   ETL)     │                │
├────────────┤                │ Operational  ├───────────>│  Enterprise    │
│  ERP (SQL) ├─── CDC/Sync ──>│ Data Store   │            │  Data          │
├────────────┤                │ (Current     │            │  Warehouse     │
│  Billing   ├─── ETL ───────>│  State)      │            │  (Historical)  │
└────────────┘                └──────┬───────┘            └────────┬───────┘
                                     │                             │
                        ┌────────────▼───────────┐      ┌──────────▼───────────┐
                        │ Tactical/Operational BI│      │ Strategic BI         │
                        │ (당일 배송 지연 현황)  │      │ (최근 5년 배송 트렌드)│
                        └────────────────────────┘      └──────────────────────┘
```

### 핵심 특징 및 역할

#### ① Data Cleansing & Integration (정제 및 통합)
A 시스템의 `Cust_NM`과 B 시스템의 `Customer_Name`을 ODS에서 `Customer_Name` 하나로 통일한다. DW로 넘어가기 전 쓰레기 데이터(GIGO)를 걸러내는 정수기 필터 역할을 한다.

#### ② 최신성 유지 (Current Data)
ODS는 데이터를 장기간 보관하지 않는다(보통 1~3개월). 새로운 데이터가 들어오면 기존 데이터를 **덮어쓴다(Update)**. 즉, DW의 비휘발성(Non-volatile) 원칙이 ODS에는 적용되지 않으며, 항상 '가장 최근 상태'를 유지한다.

#### ③ 잦은 업데이트와 실시간 연동
배치(Batch)로 움직이는 DW와 달리, ODS는 OGG(Oracle GoldenGate)나 Kafka 같은 CDC(Change Data Capture) 기술을 이용해 원천 DB의 변경 사항을 거의 실시간(Near Real-time)으로 동기화받는다.

## Ⅲ. 융합 비교 및 다각도 분석

### OLTP vs ODS vs DW 3자 심층 비교

| 비교 항목 | 운영 DB (OLTP) | **ODS (운영 데이터 스토어)** | 데이터 웨어하우스 (DW) |
|-----------|----------------|------------------------------|------------------------|
| **데이터 범위**| 단일 업무 시스템 | **전사 통합 (최신 데이터만)** | 전사 통합 (모든 과거 이력) |
| **적재 주기** | 실시간 (Real-time) | **실시간 ~ 준실시간** | 일/주/월 단위 배치 (Batch) |
| **데이터 수정**| 잦은 덮어쓰기 (C,U,D) | **덮어쓰기 발생 (Sync)** | 쓰기 불가 (Append-only) |
| **분석 주체** | 창구 직원, 시스템 | **실무 관리자 (전술적 분석)** | 경영진, 데이터 과학자 (전략적) |
| **쿼리 성격** | 개별 건 조회 | **단순 집계 및 상태 파악** | 초거대 다차원 집계 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 아키텍처 전략)

**상황**: 대형 통신사에서 고객센터(콜센터) 직원이 고객의 전화를 받았을 때, 고객의 유선 인터넷 상태, 모바일 요금제, 최신 불만 접수 내역을 한 화면에 2초 안에 띄워줘야 한다. 이 데이터들은 각기 다른 3개의 레거시 DB에 쪼개져 있다.
**판단 및 ODS 활용 전략**:
1. **ODS 구축**: 실시간성이 중요하므로 무거운 DW나 하둡(Hadoop)이 아닌 관계형 DB 기반의 ODS를 중앙에 배치한다.
2. **CDC 동기화**: 각 레거시 DB에 CDC 솔루션을 붙여, 고객 정보가 변할 때마다 즉시 ODS로 데이터를 흘려보내(Stream) 병합한다.
3. **API 연동**: 콜센터 직원용 화면(Front-end)은 개별 DB를 찌르지 않고 ODS에만 쿼리를 날려 통합된 최신 고객 프로필(360-degree view)을 가져온다.
4. **결과**: 콜센터 응답 속도 향상으로 고객 만족도를 높이고, 원천 DB에 가해지던 무거운 통합 쿼리 부하를 완벽히 덜어냄.

### 안티패턴 및 고려사항
- **ODS 비대화 (ODS를 DW처럼 쓰기)**: ODS에 데이터를 1년, 5년 치씩 지우지 않고 계속 쌓으면 결국 검색 속도가 느려지고 ODS가 제 기능을 상실한다. 오래된 데이터는 과감히 비우고 DW로 이관해야 한다.
- **직접 운영 시스템으로 역류**: ODS는 데이터를 모아서 보여주는 곳(Read-heavy)이지, 데이터를 새로 생성하는 곳이 아니다. ODS에서 데이터를 수정한 뒤 원천 DB로 밀어넣는(Two-way sync) 아키텍처는 데이터 정합성 파괴의 지름길이다.

## Ⅴ. 미래 전망 및 결론

### 결론: 데이터 파이프라인의 필수 완충지대
ODS는 분석(DW)과 운영(OLTP) 사이에서 톱니바퀴가 부서지지 않게 돌아가도록 돕는 윤활유이자 환승역이다. 실시간 전사 통합 뷰가 필요한 모든 대기업 아키텍처에서 절대 빠질 수 없는 핵심 계층이다.

### 미래 전망
최근 카프카(Kafka)와 Flink 같은 **실시간 스트리밍 기술**이 발달하면서, 물리적인 ODS DB를 거치지 않고 메모리 상에서 실시간으로 데이터를 합치고 정제하여 바로 서비스로 쏘는 '스트리밍 ODS(In-memory ODS)' 형태로 아키텍처가 진화하고 있다. 또한 클라우드 HTAP(Hybrid Transactional/Analytical Processing) 데이터베이스가 운영과 분석을 동시에 처리하게 되면서 ODS와 DW의 경계가 옅어지는 추세다.

## 👶 어린이를 위한 3줄 비유 설명
1. **ODS가 뭔가요?**: 장난감 공장(데이터 웨어하우스)으로 물건을 보내기 전에, 동네 쓰레기를 한곳에 모아서 쓸만한 것만 깨끗하게 씻어놓는 '중간 분류장'이에요.
2. **왜 이걸 만드나요?**: 공장은 너무 커서 한 번 돌아가는데 하루 종일 걸리지만, 중간 분류장에서는 "지금 동네에 어떤 쓰레기가 제일 많지?"를 바로바로 확인할 수 있거든요.
3. **가장 좋은 점은?**: 원본 장부(운영 DB)를 뒤적거리다 전산망이 망가지는 걸 막아주고, 사장님들이 방금 전까지 일어난 최신 소식을 한눈에 깔끔하게 볼 수 있게 해준답니다!
