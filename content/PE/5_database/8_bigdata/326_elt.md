# 326. ELT (Extract, Load, Transform) 프로세스 - 클라우드 기반의 혁신

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 원천 시스템에서 데이터를 추출(Extract)한 뒤, 정제 없이 타겟 데이터 웨어하우스(DW)에 우선 적재(Load)하고, 이후 **DW의 강력한 병렬 연산 자원을 활용하여 데이터를 변환(Transform)**하는 차세대 데이터 통합 프로세스이다.
> 2. **가치**: 중간 변환 서버의 병목을 없애 대용량 데이터를 고속으로 수집하며, 일단 모든 로우 데이터(Raw Data)를 적재해둠으로써 데이터 분석가들이 나중에 언제든지 원하는 형태로 가공할 수 있는 '분석의 유연성'을 제공한다.
> 3. **융합**: 클라우드 기반 데이터 웨어하우스(BigQuery, Snowflake), 데이터 레이크(S3), 그리고 SQL 기반의 변환 도구인 **dbt (data build tool)**와 융합되어 현대적 데이터 스택(MDS)의 핵심이 된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
ELT는 전통적인 ETL에서 'T'와 'L'의 순서를 뒤집은 것이다. 변환(Transform) 작업을 비싼 별도 서버가 아닌, 이미 구축된 초고성능 DW 엔진에게 맡긴다는 발상의 전환이다.

### 💡 비유
ELT는 **"대형 마트 시식 코너의 운영 방식"**과 같다.
- **ETL**: 산지(원천 DB)에서 채소를 다 씻고 썰어서(변환) 진열대에 올리는 것. (진열 속도가 느림).
- **ELT**: 산지에서 흙만 털고 일단 마트(DW)로 몽땅 보낸다. 마트는 공간도 넓고 요리사도 많다. 고객(분석가)이 "볶음밥 해주세요"라고 하면, 그 자리에서 바로 재료를 꺼내 요리(변환)해 준다. 훨씬 신선하고 다양하게 대응할 수 있다.

### 등장 배경 및 발전 과정

#### 1. MPP (Massive Parallel Processing)의 등장
과거 DW는 연산 능력이 약했지만, 현대의 클라우드 DW는 수천 대의 서버를 동시에 돌려 변환 작업을 처리할 수 있게 되었다. "굳이 밖에서 요리해 올 필요가 없다"는 깨달음이 온 것이다.

#### 2. 비정형 데이터의 폭증
로그나 이미지 같은 데이터는 어떻게 가공할지 미리 정하기 힘들다. "일단 다 담아놓고 나중에 생각하자"는 데이터 레이크(Data Lake) 철학이 ELT를 가속화했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### ETL vs ELT 상세 프로세스 비교 (표)

| 단계 | ETL (전통적) | ELT (현대적) |
|------|---------------|---------------|
| **추출 (Extract)** | 정해진 데이터만 선별 | **거의 모든 로우 데이터 추출** |
| **적재 (Load)** | 정제 완료 후 마지막에 수행 | **추출 즉시 타겟에 적재** |
| **변환 (Transform)** | 중간 서버의 메모리/CPU 사용 | **타겟 DW의 MPP 엔진 활용 (SQL 중심)** |
| **데이터 보존** | 정제 후 원본 폐기 | **로우 데이터 영구 보존 가능** |
| **도구** | Informatica, Talend | **Fivetran, Airbyte, dbt** |

### 정교한 구조 다이어그램 (Cloud-Native ELT Architecture)

```ascii
================================================================================
[ Modern ELT Architecture: Leveraging Cloud Power ]
================================================================================

  [ Source Systems ]           [ Cloud DW / Lake ]           [ Analytics Layer ]
 ┌────────────────┐           ┌───────────────────┐         ┌────────────────┐
 │ 🛒 Sales DB    ├──(E)──┐   │ ⭐️ Staging Area ⭐️  │         │ 🎨 BI Tools    │
 └────────────────┘       │   │ (Raw JSON / CSV)  │         │ (Tableau/Looker)│
 ┌────────────────┐       └──▶│                   │         └────────────────┘
 │ 📱 Log Files   ├──(E)──┐   ├───────────────────┤                ▲
 └────────────────┘       │   │ ⚡️ MPP Engine     │                │
 ┌────────────────┐       └──▶│ (BigQuery/Snowflake)│                │
 │ 🌐 SaaS APIs   ├──(E)──┘   │                   │         ┌──────┴─────────┐
 └────────────────┘           │  (Transform via)  │◀────────┤ 🛠️ dbt (SQL)   │
                              │  (SQL/Python)     │         └────────────────┘
                              └───────────────────┘

 * 핵심 포인트: 모든 데이터는 날것 그대로 DW에 먼저 들어옴. 
   dbt 같은 도구가 DW 안에서 SQL을 날려 '분석용 테이블'로 최종 변환함.
```

### 심층 동작 원리

#### ① 스키마 온 리드 (Schema-on-read)
ELT의 핵심 철학이다. 데이터를 넣을 때 스키마를 따지지 않고 일단 넣는다. 나중에 데이터를 읽어서 분석할 때 비로소 스키마를 정의하여 변환한다. 이를 통해 원천 시스템의 변화에 매우 유연하게 대처할 수 있다.

#### ② dbt (data build tool)와 T(Transform)의 혁명
ELT에서 변환은 보통 SQL로 이루어진다. dbt는 SQL에 버전 관리, 테스트, 문서화 기능을 더해 소프트웨어 공학적으로 데이터를 가공하게 해준다. 이제 데이터 분석가(Analytics Engineer)가 직접 데이터 엔지니어링의 핵심인 '변환'을 주도한다.

## Ⅲ. 융합 비교 및 다각도 분석

### ELT 아키텍처의 트레이드오프

| 장점 (Pros) | 단점 (Cons) |
|-------------|-------------|
| **압도적 속도**: 대량 데이터를 병렬로 부어 넣음 | **비용 발생**: DW 내에서 연산을 많이 하므로 클라우드 비용이 튈 수 있음 |
| **유연성**: 나중에 분석 요구가 바뀌어도 원본 데이터가 있어 대응 가능 | **보안 위험**: 정제 안 된 데이터(개인정보 등)가 DW에 그대로 들어갈 수 있음 |
| **단순성**: 복잡한 중간 서버 관리 필요 없음 | **거버넌스**: DW가 쓰레기장(Data Swamp)이 되기 쉬움 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 급성장하는 이커머스 스타트업. 하루에 발생하는 로그가 100GB를 넘어서면서 기존 ETL 배치가 타임아웃으로 매일 실패한다. 분석가들은 "새로운 지표를 보고 싶은데 ETL 코드 고치는 데만 1주일 걸린다"며 아우성이다.
**판단 및 ELT 전환 전략**:
1. **Ingestion Tool 도입**: **Fivetran**이나 **Airbyte**를 써서 로직 없이 모든 데이터를 **Google BigQuery**로 1시간마다 들이붓는다. (Extract & Load).
2. **dbt 기반 변환**: 데이터 분석가들에게 dbt 사용법을 교육하여, 각자 필요한 지표를 SQL로 직접 정의하게 한다. (Transform).
3. **Raw Data 보관**: 모든 로우 데이터를 'Bronze' 레이어에 영구 보존하여 과거 데이터 재분석이 가능하게 한다.
4. **결과**: 데이터 수집 장애가 사라지고, 신규 지표 개발 기간이 **1주일에서 1일**로 단축된다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **민감 정보의 '묻지마 적재'**: ELT라고 해서 개인정보(주민번호, 암호)를 정제 없이 그대로 DW에 넣으면 보안 사고가 난다. 적재 전 단계에서 최소한의 **'익명화/마스킹'** 처리는 반드시 수행해야 한다.
- **비효율적 쿼리**: DW 자원을 믿고 비효율적인 SQL로 변환을 돌리면 수천만 원의 비용 고지서를 받을 수 있다. **'증분 빌드(Incremental model)'** 전략을 써서 바뀐 데이터만 가공하도록 튜닝해야 한다.

---

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **엔지니어링 리소스**: 파이프라인 관리 시간 **50% 이상 절감**.
- **분석 민첩성**: 데이터 요청 후 대시보드 반영까지의 시간(TTI) **70% 이상 단축**.

### 미래 전망 및 진화 방향
ELT는 이제 **'제로 ETL (Zero-ETL)'**로 진화하고 있다. DB와 DW가 한 몸이 되어, DB에 데이터를 넣으면 1초 뒤에 DW에서 바로 분석 가능한 수준이다. 또한, 변환 단계에 머신러닝 모델을 직접 심어 데이터가 들어오는 즉시 예측값을 생성하는 **'AI-embedded ELT'**가 현대 엔터프라이즈 데이터 아키텍처의 최종 목적지가 될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **ELT가 뭔가요?**: 시장에서 사 온 식재료를 집 밖에서 미리 다 손질해서 가져오는 게 아니라, 일단 장바구니에 몽땅 담아와서 우리 집의 아주 큰 부엌(DW)에서 요리하는 거예요.
2. **왜 좋나요?**: 우리 집 부엌은 아주 넓고 요리 도구도 많아서, 밖에서 힘들게 요리해 오는 것보다 집에서 한꺼번에 하는 게 훨씬 빠르고 편하기 때문이에요.
3. **가장 신기한 점은요?**: 재료를 안 쓰고 그냥 냉장고에 넣어뒀다가, 나중에 배가 고플 때 언제든지 내가 먹고 싶은 요리로 척척 변신시킬 수 있다는 점이랍니다!
