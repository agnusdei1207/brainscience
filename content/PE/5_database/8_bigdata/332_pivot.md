# 332. 피벗 (Pivot) - 데이터 시점의 90도 회전

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 다차원 데이터 분석에서 보고서의 **행(Row)과 열(Column)의 기준이 되는 차원을 서로 맞바꾸거나 재배치**하여, 데이터를 바라보는 시각적 구조를 동적으로 변경하는 연산이다.
> 2. **가치**: 나열된 로우 형태의 데이터를 요약된 교차표(Cross-tabulation) 형태로 변환함으로써, 특정 항목 간의 상관관계나 시간 흐름에 따른 변화를 직관적으로 비교·분석할 수 있게 한다.
> 3. **융합**: 관계형 데이터베이스의 `PIVOT` 함수, 엑셀의 피벗 테이블, 그리고 데이터 분석 라이브러리(Pandas 등)의 데이터 재구조화(Reshaping) 기술과 융합되어 사용된다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
피벗(Pivot)은 데이터를 "회전"시키는 것이다. 예를 들어, [날짜, 지역, 매출]로 길게 늘어진 데이터를 [행:지역, 열:날짜] 형태의 바둑판 모양으로 바꾸는 행위다. 데이터의 값 자체는 변하지 않지만, 값이 배치되는 '틀'이 바뀐다.

### 💡 비유
피벗은 **"정면에서 보던 집을 옆면에서 보는 것"**과 같다.
- **정면 뷰 (Normal)**: 집의 현관문과 창문이 보인다. (지역별 매출 확인).
- **피벗 뷰 (Side)**: 90도 돌려서 옆에서 보니 집의 깊이와 뒷마당이 보인다. (시간 흐름에 따른 지역별 추이 확인).
집(데이터)은 그대로지만, 내가 서 있는 위치(차원 배치)를 바꿈으로써 보이지 않던 정보가 드러난다.

### 등장 배경 및 발전 과정

#### 1. 로우 중심 저장의 한계
DB는 데이터를 아래로 길게 쌓는다. 하지만 사람은 보고서를 볼 때 왼쪽에서 오른쪽으로 비교하는 것을 선호한다. 

#### 2. 스프레드시트의 혁명
1990년대 엑셀(Excel)의 '피벗 테이블' 기능이 대히트를 치면서, 일반 비즈니스 사용자들이 수만 건의 데이터를 단 몇 초 만에 요약 표로 만드는 경험을 하게 되었고, 이후 모든 BI 도구의 표준 기능이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 피벗 연산의 전후 구조 비교 (표)

| 구분 | 피벗 전 (Flat/Long Format) | **피벗 후 (Cross-tab/Wide Format)** |
|------|----------------------------|-----------------------------------|
| **데이터 모양** | 세로로 길게 늘어짐 | 가로와 세로가 조화로운 바둑판 모양 |
| **행(Row)** | 개별 트랜잭션 기록 | 기준 차원 (예: 상품명) |
| **열(Column)** | 차원 명칭 (Date, Region, Sales) | 그룹화된 차원 값 (1월, 2월, 3월...) |
| **값(Cell)** | 단일 관측값 | **집계된 수치 (Sum, Avg 등)** |

### 정교한 구조 다이어그램 (데이터 회전 시각화)

```ascii
================================================================================
[ Pivot Operation: Changing Axes of Data ]
================================================================================

 [ Input: Raw Data (Long) ]         [ Pivot Operation ]        [ Output: Pivot Table (Wide) ]
 ┌────────┬────────┬───────┐                                   ┌──────────┬──────┬──────┐
 │ Region │ Month  │ Sales │        (Row: Region)              │ Region/Mo│ Jan  │ Feb  │
 ├────────┼────────┼───────┤        (Col: Month)               ├──────────┼──────┼──────┤
 │ Seoul  │ Jan    │ 100   │ ──▶    (Value: Sum)      ──▶      │ Seoul    │ 100  │ 150  │
 │ Seoul  │ Feb    │ 150   │                                   ├──────────┼──────┼──────┤
 │ Busan  │ Jan    │ 80    │                                   │ Busan    │ 80   │ 90   │
 │ Busan  │ Feb    │ 90    │                                   └──────────┴──────┴──────┘
 └────────┴────────┴───────┘

 * 핵심 포인트: 데이터의 'Month' 컬럼에 있던 값들이 
   피벗 후에는 새로운 '열 헤더'로 변신함.
```

### 심층 동작 원리

#### ① 데이터 재구조화 (Reshaping)
피벗은 내부적으로 **Group By**와 **Conditional Aggregation**의 조합이다.
SQL로 표현하면 다음과 같다.
```sql
SELECT Region,
       SUM(CASE WHEN Month='Jan' THEN Sales END) AS Jan,
       SUM(CASE WHEN Month='Feb' THEN Sales END) AS Feb
FROM Sales_Table
GROUP BY Region;
```
하드웨어는 이 쿼리를 처리하기 위해 전체 데이터를 한 번 훑으며(Scan), 각 지역별로 메모리 버킷을 만들어 값을 채워 넣는다.

#### ② 언피벗 (Unpivot / Melt)
피벗의 반대 연산이다. 가로로 넓은 보고서용 데이터를 다시 DB에 넣기 좋게 세로로 길게 푸는 과정이다. 데이터 클렌징 단계에서 자주 사용된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 피벗 테이블의 강점과 약점

| 장점 (Strength) | 단점 (Weakness) |
|-----------------|-----------------|
| **비교 용이성**: 여러 항목의 수치를 한눈에 비교 가능 | **데이터 고착**: 행과 열이 고정되어 추가 분석 시 다시 돌려야 함 |
| **요약 능력**: 수천만 건의 데이터를 한 페이지로 압축 | **성능 부하**: 대량 데이터 피벗 시 메모리 소모가 극심함 |
| **직관적 UI**: 비전문가도 드래그 앤 드롭으로 분석 가능 | **스키마 유연성 부족**: 새로운 데이터 타입 추가 시 틀이 깨짐 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 대형 마트의 재고 관리 시스템. 매일 1,000개 매장에서 50,000개 품목의 판매량이 쌓인다. 재고 담당자가 "매장별로 어떤 요일에 어떤 물건이 많이 팔리는지" 보고 싶어 하는데, 조회가 너무 느려 화면이 멈춘다.
**판단 및 성능 최적화 전략**:
1. **Pre-aggregation**: 실시간 피벗은 무리다. 매일 밤 '매장 x 요일 x 품목' 단위의 **요약 테이블(Summary Table)**을 미리 만들어둔다.
2. **In-memory Processing**: 피벗 연산은 메모리 내에서 행렬 전환이 일어날 때 가장 빠르다. **Redis**나 **Spark**의 메모리 캐시를 활용하여 피벗 연산을 수행한다.
3. **Lazy Pivot**: 화면에 보이는 부분(Top 100 매장 등)만 먼저 피벗하고, 나머지는 스크롤 시에 처리한다.
4. **결과**: 재고 담당자는 대기 시간 없이 요일별 판매 패턴을 바둑판 모양으로 확인하며 적정 재고량을 즉시 결정할 수 있게 된다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **너무 많은 열(Column) 생성**: 피벗할 차원의 고유값(Cardinality)이 너무 많으면(예: 수만 명의 고객 이름을 열로 피벗), 표가 너무 옆으로 길어져서 아무도 읽을 수 없게 된다. 열에는 보통 **날짜, 성별, 등급** 같이 종류가 적은 차원을 배치하는 것이 정석이다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **가독성 향상**: 나열된 데이터 대비 정보 습득 속도 **5배 이상 향상**.
- **인사이트 도출**: 항목 간 교차 분석을 통해 숨겨진 패턴 발견 확률 증가.

### 미래 전망 및 진화 방향
피벗은 이제 **'자연어 피벗'**으로 진화하고 있다. 사용자가 "지역별로 월별 매출 보여줘"라고 말하면 AI가 알아서 행과 열을 배치하는 방식이다. 또한, **'3D 피벗'** 기술이 도입되어 가상 현실 공간에서 데이터의 축을 손으로 돌려가며 입체적으로 분석하는 시대로 나아가고 있으며, 이는 데이터 분석이 단순한 '표 보기'를 넘어 '체험'이 되는 미래를 예고한다.

1. **피벗이 뭔가요?**: 장난감 블록들을 색깔별로 세로로 쫙 세워놓았다가, 이번엔 모양별로 가로로 눕혀서 다시 정리해보는 거예요.

3. **가장 좋은 점은요?**: 엉망진창으로 흩어져 있던 정보들이 바둑판처럼 예쁘게 정리되어서, "아하! 이게 이거였구나!" 하고 한눈에 정답을 찾을 수 있게 된답니다!
