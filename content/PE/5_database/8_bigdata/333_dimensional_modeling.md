# 333. 다차원 모델링 (Dimensional Modeling) - 비즈니스 언어로 데이터를 그리다

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 웨어하우스와 데이터 마트를 구축할 때, 복잡한 정규화 테이블을 버리고 사용자가 데이터를 직관적으로 이해하고 고속으로 분석할 수 있도록 **사실(Fact)과 차원(Dimension)으로 단순화하여 재설계하는 기법**이다.
> 2. **혁신**: 업무 처리(Insert/Update)를 위한 제3정규형(3NF)의 사슬을 끊고 의도적인 **반정규화(De-normalization)**를 통해 JOIN 연산을 최소화하여, 방대한 과거 데이터의 집계 및 조회(Read) 성능을 극대화한다.
> 3. **가치**: IT 전문가뿐만 아니라 일반 비즈니스 부서(경영, 마케팅) 사용자들도 "시간별, 지역별, 상품별 매출액" 같은 다차원 쿼리를 블록 조립하듯 쉽게 만들 수 있는 BI(비즈니스 인텔리전스) 환경을 제공한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
다차원 모델링은 랄프 킴볼(Ralph Kimball)이 제안한 데이터 설계 방법론이다. 기업의 비즈니스 현상을 측정 가능한 수치인 **'팩트(Fact)'**와, 그 수치를 쪼개어 볼 수 있는 기준인 **'차원(Dimension)'**의 두 가지 요소로 철저하게 분리하여 데이터를 모델링한다.

### 💡 비유
- **일반 관계형 모델 (3NF)**: 백화점 창고에 물건들이 "품목 코드", "제조사 코드", "분류 코드" 박스에 갈기갈기 찢어져 보관되어 있다. 티셔츠 하나를 조립하려면 창고 5군데를 돌아다녀야 한다 (JOIN Hell).
- **다차원 모델**: 엑셀의 피벗 테이블(Pivot Table)과 같다. 한가운데에 '매출액(Fact)'이라는 거대한 숫자가 있고, 위쪽과 왼쪽에는 '날짜(시간 차원)', '매장 위치(공간 차원)', '상품 종류(상품 차원)'라는 탭이 달려 있다. 원하는 탭을 누르기만 하면 데이터가 마법처럼 요약되어 나타난다.

### 등장 배경: RDBMS의 태생적 한계
전통적인 RDBMS는 중복을 없애기 위해 테이블을 잘게 쪼갰다. 그 결과 수천만 건의 판매 이력을 분석하려 할 때마다 10~20개의 테이블을 JOIN해야 했고 시스템은 멈춰버렸다. "조회 속도를 위해서는 차라리 데이터 중복을 허용하자"는 발상의 전환이 다차원 모델링의 시작이다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 사실(Fact)과 차원(Dimension) 구조 (ASCII Diagram)

```ascii
             [ Dimension ]                  [ Dimension ]
           ┌───────────────┐              ┌───────────────┐
           │ Date_Dim      │              │ Product_Dim   │
           │ - Date_Key(PK)│<─┐        ┌─>│ - Prod_Key(PK)│
           │ - Year, Month │  │        │  │ - Name, Cat.  │
           └───────────────┘  │        │  └───────────────┘
                              ▼        ▼
                      ┌──────────────────────┐
                      │    Sales_Fact        │  <--- 측정할 비즈니스 이벤트 (중앙)
                      │ - Date_Key (FK)      │
                      │ - Prod_Key (FK)      │
                      │ - Store_Key (FK)     │
                      │ -------------------- │
                      │ - Quantity (Measure) │
                      │ - Amount (Measure)   │
                      └──────────────────────┘
                              ▲
                              │
           ┌──────────────────┴──┐
           │ Store_Dim           │  [ Dimension ]
           │ - Store_Key (PK)    │
           │ - City, Region      │
           └─────────────────────┘

* 핵심 원리: 팩트 테이블을 중심으로 차원 테이블들이 방사형으로 뻗어나감 (Star Schema).
```

### 2대 핵심 구성 요소

#### ① 팩트 테이블 (Fact Table)
비즈니스에서 발생한 이벤트(측정값)를 기록한다. "철수가 어제 강남점에서 사과를 3개 샀다"에서 **'3개(수량)'**와 **'결제 금액'**이 팩트에 해당한다. 
- 주로 수치형 데이터이며 지속해서 방대하게 쌓인다. (초거대 테이블).
- 각 차원 테이블과 연결하기 위한 외래키(Foreign Key)의 집합소다.

#### ② 차원 테이블 (Dimension Table)
팩트 테이블의 숫자를 분석하는 렌즈(필터) 역할을 한다. "누가, 언제, 어디서, 무엇을"에 해당하는 텍스트형 속성값들을 가진다.
- **시간 차원 (Time)**: '년, 월, 일, 요일, 분기, 휴일 여부' 등 가장 기본적이고 필수적인 차원.
- **지리 차원 (Geography)**: '국가, 도시, 매장 크기' 등.
- 차원 테이블은 팩트 테이블보다 행(Row) 수는 적지만 가로로 길고 뚱뚱한(Denormalized) 경우가 많다.

## Ⅲ. 융합 비교 및 다각도 분석

### 관계형 모델링(ERD) vs 다차원 모델링 비교

| 비교 항목 | 3NF 관계형 모델링 (OLTP) | **다차원 모델링 (OLAP)** |
|-----------|--------------------------|--------------------------|
| **핵심 목표** | 데이터 중복 제거, 갱신 이상 방지 | **직관적인 쿼리 작성, 압도적 조회 속도** |
| **테스트 시각**| 애플리케이션 개발자 관점 | **비즈니스 분석가(현업) 관점** |
| **테이블 수** | 많고 잘게 쪼개짐 | **적고 큼 (반정규화)** |
| **JOIN 연산** | 매우 빈번함 (느림) | **최소화 (팩트-차원 간 1 depth 결합)** |
| **이력 관리** | 보통 최신 상태만 유지 | **SCD (서서히 변하는 차원) 기법으로 과거 이력 유지** |

### SCD (Slowly Changing Dimension) 기법
다차원 모델에서 차원의 속성(예: 고객의 주소, 직원의 부서)이 시간이 지남에 따라 서서히 변할 때 이력을 관리하는 핵심 기술이다.
- **SCD Type 1**: 그냥 새 값으로 덮어쓴다 (과거 이력 포기).
- **SCD Type 2**: 기존 행을 만료(End_Date)시키고, 새로운 식별자(Surrogate Key)를 발급하여 새 행을 추가한다 (가장 완벽한 이력 관리).

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 모델링 전략)

**상황**: 통신사에서 고객들의 데이터 사용량, 통화 시간, 과금 내역을 분석하여 요금제 개편 인사이트를 얻기 위한 데이터 마트를 설계하려 한다.
**판단 및 다차원 모델링 전략**:
1. **비즈니스 프로세스 식별**: '일별 통신 사용량 이력'을 분석 대상으로 정한다.
2. **Grain (입자) 결정**: 팩트 테이블에 쌓일 데이터의 최소 단위를 "1명의 고객이 하루 동안 특정 요금제로 사용한 총량"으로 정의한다. (Grain이 명확해야 중복 집계 오류를 막을 수 있다).
3. **차원 (Dimension) 설계**: '날짜 차원', '고객 차원(연령대, 성별 포함 반정규화)', '요금제 차원'을 도출한다.
4. **팩트 (Fact) 설계**: `Usage_Fact` 테이블에 시간 키, 고객 키, 요금제 키를 넣고 `Data_Usage(MB)`, `Voice_Usage(Min)`, `Billing_Amount`를 측정값으로 둔다.
5. **결과**: "20대 여성이 주말에 무제한 요금제에서 데이터를 얼마나 썼는가?"를 단 한 번의 JOIN으로 수 초 내에 쿼리할 수 있게 됨.

### 안티패턴 및 고려사항
- **차원의 과도한 정규화 (Snowflaking)**: 속도를 올리려고 다차원 모델을 썼으면서, 차원 테이블을 다시 3NF처럼 쪼개어 스노우플레이크(Snowflake) 스키마를 남발하는 것. 쿼리 성능이 다시 하락하므로 특별한 이유가 없다면 **스타 스키마(Star Schema)**를 유지해야 한다.
- **단일 팩트에 여러 Grain 섞기**: 한 팩트 테이블에 '일별 매출'과 '월별 매출' 데이터를 섞어 넣으면 나중에 SUM() 집계 시 데이터가 이중 뻥튀기된다.

## Ⅴ. 미래 전망 및 결론

### 결론: 데이터 분석의 영원한 프레임워크
다차원 모델링은 비즈니스 사용자가 데이터를 "원하는 각도(차원)에서 썰어보고(Dice), 쪼개어보는(Slice)" 다차원 큐브(OLAP Cube)의 근간을 제공한다. 수십 년이 지났지만, 인간이 데이터를 인지하는 가장 자연스러운 방법이라는 점에서 그 지위를 잃지 않고 있다.

### 미래 전망
최근 클라우드 DW 솔루션(BigQuery 등)이 극도로 강력해지면서 복잡한 조인(JOIN) 성능조차 압도해버리는 시대가 왔다. 이에 따라 팩트와 차원을 아예 하나의 거대한 테이블로 몽땅 합쳐버리는 **OBT (One Big Table)** 모델링 방식도 새롭게 부상하고 있다. 하지만 여전히 데이터의 논리적 의미와 거버넌스를 부여하는 측면에서 다차원 모델링의 설계 철학은 필수적이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **다차원 모델링이 뭔가요?**: 엄청 복잡한 레고 블록들을 아무렇게나 쌓지 않고, 가운데에 '제일 중요한 몸통(팩트)'을 두고 주변에 '팔, 다리, 머리(차원)'를 보기 좋게 붙여서 예쁜 로봇을 만드는 방법이에요.
2. **왜 이렇게 만드나요?**: 머리 모양만 다른 걸로 바꿔보고 싶을 때, 로봇 전체를 부수지 않고 머리 부품(차원)만 쏙 빼서 다른 걸로 갈아 끼우며 쉽게 놀 수 있거든요.
3. **가장 좋은 점은?**: 컴퓨터 전문가가 아니어도 "나는 빨간색 팔에 파란색 다리를 붙인 로봇의 점수가 궁금해!"라고 아주 쉽게 명령을 내리고 결과를 금방 볼 수 있답니다!
