# 334. 스타 스키마 (Star Schema) - 분석 속도를 지배하는 별의 중심

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 다차원 모델링을 관계형 데이터베이스(RDBMS)에 물리적으로 구현할 때 가장 널리 쓰이는 구조로, 중앙에 거대한 하나의 **팩트(Fact) 테이블**을 두고 그 주위를 **차원(Dimension) 테이블**들이 별 모양처럼 둘러싼 형태다.
> 2. **혁신**: 차원 테이블의 데이터 중복을 기꺼이 허용하는 **반정규화(De-normalization)**를 통해, 팩트 테이블에서 차원 테이블로 나가는 조인(JOIN) 뎁스를 오직 1단계(1-depth)로 최소화하여 압도적인 쿼리 성능을 낸다.
> 3. **가치**: 쿼리 구조가 매우 직관적이라 BI(비즈니스 인텔리전스) 툴이 SQL을 자동 생성하기 쉬우며, 현업 비즈니스 분석가들이 데이터의 구조를 직관적으로 이해하고 탐색할 수 있게 해준다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
스타 스키마는 데이터 마트나 데이터 웨어하우스 설계의 교과서적인 패턴이다. 중앙의 팩트 테이블에는 측정 가능한 수치(매출액, 판매량 등)와 외래키(FK)들이 빽빽하게 담겨 있고, 외곽의 차원 테이블에는 분석의 기준이 되는 텍스트 속성(이름, 지역, 카테고리 등)들이 중복을 허용한 채로 넓게 퍼져 있다.

### 💡 비유
- **정규화 모델 (도서관 십진분류법)**: 책(데이터)을 찾으려면 인덱스 카드함(테이블1)에서 위치 코드를 찾고, 분류표(테이블2)에서 층수를 확인하고, 서가 지도(테이블3)를 거쳐야 겨우 책을 찾을 수 있다. 책 보관 공간은 최소화되지만 찾기가 너무 힘들다.
- **스타 스키마 (베스트셀러 매대)**: 매장 한가운데 베스트셀러 진열대(팩트 테이블)가 있고, 그 바로 옆에 소설, 만화, 잡지 코너(차원 테이블)가 별 모양으로 착 붙어 있다. 고객은 가운데 서서 한 발짝만 뻗으면 원하는 책을 즉시 집어 들 수 있다.

### 등장 배경: 'Join'의 저주와 읽기 성능
OLTP(운영 시스템)는 데이터 삽입 시 에러(이상 현상)를 막기 위해 철저히 제3정규화(3NF)를 한다. 하지만 3NF 구조에서 "2025년 서울 강남에서 팔린 전자기기 총매출"을 뽑으려면 연도, 지역, 매장, 상품 카테고리, 상품 테이블까지 6~7번의 무거운 JOIN이 발생한다. 이를 단 한 번의 JOIN으로 끝내기 위해 스타 스키마가 고안되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 스타 스키마 구조 다이어그램 (ASCII Diagram)

```ascii
      [ Time Dimension ]              [ Product Dimension ]
   ┌──────────────────────┐        ┌───────────────────────┐
   │ Date_Key (PK)        │        │ Product_Key (PK)      │
   │ Year, Quarter, Month │        │ Category, Brand, Name │ <-- 카테고리가 중복 저장됨
   │ Day_of_Week          │        │ Price, Color          │     (반정규화)
   └──────────┬───────────┘        └───────────┬───────────┘
              │                                │
              │      ┌──────────────────┐      │
              └─────>│ SALES FACT TABLE │<─────┘
                     ├──────────────────┤
                     │ Date_Key (FK)    │  <-- Star의 중심 (Foreign Keys)
                     │ Product_Key (FK) │
                     │ Store_Key (FK)   │
                     │ Customer_Key (FK)│
                     │ ---------------- │
                     │ Quantity         │  <-- 분석 대상 수치 (Measures)
                     │ Total_Amount     │
              ┌─────>└──────────────────┘<─────┐
              │                                │
   ┌──────────┴───────────┐        ┌───────────┴───────────┐
   │ Store_Key (PK)       │        │ Customer_Key (PK)     │
   │ Region, City, Name   │        │ Age_Group, Gender     │
   │ Manager_Name         │        │ Job, Membership_Lv    │
   └──────────────────────┘        └───────────────────────┘
     [ Store Dimension ]             [ Customer Dimension ]
```

### 3대 설계 원칙

#### ① 반정규화된 차원 (Denormalized Dimensions)
제품 차원(Product) 안에 대분류, 중분류, 소분류가 모두 하나의 테이블에 텍스트로 들어간다. 만약 '나이키' 제품이 1,000개라면 '나이키'라는 글자가 1,000번 중복 저장된다. 스토리지 용량은 낭비되지만 조인 횟수가 줄어들어 속도가 극대화된다.

#### ② 대체 키 (Surrogate Keys)
차원 테이블의 기본키(PK)는 원본 시스템의 사번이나 제품 코드(Natural Key)를 쓰지 않고, 시스템이 일련번호(1, 2, 3...)로 자동 부여하는 대체 키를 쓴다. 속도가 빠르고 데이터 변경 이력(SCD) 추적이 쉬워지기 때문이다.

#### ③ 팩트 테이블의 무거운 측정값
모든 외래키가 모이는 팩트 테이블은 수억~수백억 건의 행(Row)을 가지며 데이터베이스 용량의 80% 이상을 차지한다. 따라서 철저히 숫자(키 값과 측정값)로만 구성하여 날렵하게 유지해야 한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 스타 스키마 vs 스노우플레이크 스키마

| 비교 항목 | **스타 스키마 (Star Schema)** | 스노우플레이크 (Snowflake) |
|-----------|-------------------------------|-----------------------------|
| **형태** | 별 모양 (1단계 방사형) | 눈송이 모양 (여러 단계 가지치기) |
| **차원 정규화**| **반정규화 (중복 허용, 단일 테이블)** | 정규화 (중복 제거, 여러 테이블로 쪼갬) |
| **조인(JOIN) 수**| **매우 적음 (보통 1번)** | 많음 (다단계 조인 필요) |
| **조회 속도(Read)**| **매우 빠름 (BI 툴에 최적화)** | 스타 스키마보다 느림 |
| **관리/수정 편의성**| 차원 수정 시 전체 업데이트 필요 | 정규화되어 있어 일부만 수정하면 됨 |
| **스토리지 용량**| 많이 차지함 | 적게 차지함 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 아키텍처 전략)

**상황**: 사내 마케팅 부서가 Tableau나 PowerBI 같은 시각화 대시보드 도구를 도입했다. 사용자가 드래그 앤 드롭으로 리포트를 만드는데 화면 로딩에 30초 이상 걸려 불만이 쇄도하고 있다.
**판단 및 스타 스키마 전환 전략**:
1. **AS-IS 분석**: 현재 데이터 마트가 스노우플레이크나 3NF 구조로 되어 있어 Tableau가 내부적으로 무거운 다단계 JOIN SQL을 생성하고 있음을 파악한다.
2. **반정규화 실행**: 여러 차원(카테고리, 브랜드, 제조사)으로 쪼개져 있던 테이블들을 `Dim_Product`라는 하나의 뚱뚱한 차원 테이블로 강제 병합(Flattening)한다.
3. **스타 스키마 구축**: 팩트 테이블을 중심으로 1-depth 조인만 발생하도록 뷰(View)나 물리 테이블을 재배치한다.
4. **결과**: BI 툴이 가장 단순하고 효율적인 `SELECT ... GROUP BY` 쿼리를 자동 생성하게 되어 대시보드 로딩 속도를 3초 이내로 단축함.

### 안티패턴 및 고려사항
- **Centipede (지네) 스키마**: 차원 테이블이 너무 잘게 쪼개져서 팩트 테이블에 붙은 외래키가 수십 개를 넘어가는 현상. 팩트 테이블이 지네처럼 보인다고 하여 붙은 이름이다. 차원을 적절히 합쳐서 외래키 수를 줄여야 한다.
- **팩트 없는 팩트 테이블 간과**: 때로는 '출석 체크'나 '이벤트 발생'처럼 수치(매출 등)는 없고 발생했다는 사실 자체만 중요한 경우가 있다. 이때도 팩트 테이블을 빈칸으로 두지 않고 '발생 건수=1' 같은 더미 팩트를 두는 것이 분석에 유리하다 (Factless Fact Table).

## Ⅴ. 미래 전망 및 결론

### 결론: 조회 속도를 위한 가장 아름다운 타협
스타 스키마는 "디스크 공간은 싸고, 사용자의 시간은 비싸다"는 현대 컴퓨팅의 철학을 가장 잘 보여주는 설계다. 데이터베이스 정규화 이론의 금기(중복)를 깨면서까지 비즈니스 속도와 직관성을 확보한 위대한 아키텍처다.

### 미래 전망
최근 스토리지의 발전과 컬럼 기반(Columnar) 클라우드 DW 솔루션(BigQuery 등)의 등장으로 조인 자체를 없애버리는 **OBT (One Big Table, 하나의 거대한 엑셀 시트처럼 팩트와 차원을 아예 합쳐버리는 방식)**가 유행하기도 한다. 그러나 거버넌스와 데이터 일관성 유지 측면에서 스타 스키마의 '중앙 팩트 통제' 철학은 여전히 DW 모델링의 표준 척도로 남을 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **스타 스키마가 뭔가요?**: 태양계처럼 가운데에 가장 크고 무거운 '매출 정보(태양)'를 두고, 그 주변을 '시간, 장소, 물건(행성들)'이 동그랗게 둘러싸고 있는 우주 지도 같은 거예요.
2. **왜 모양이 별이나 태양계 같나요?**: 지구에서 다른 행성으로 놀러 갈 때 여기저기 거치지 않고 한 번에 슝~ 날아갈 수 있게 길을 뚫어놓았기 때문이에요.
3. **가장 좋은 점은?**: 사장님이 "강남 매장에서 어제 팔린 사과 개수 찾아줘!" 했을 때, 우주 지도를 쫙 펴고 한가운데서 한 발짝만 움직이면 바로 정답을 찾을 수 있어서 엄청 빠르답니다!
