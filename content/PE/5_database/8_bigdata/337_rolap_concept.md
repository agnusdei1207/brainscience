# 337. ROLAP (Relational OLAP) - 관계형 DB가 품은 다차원 분석의 힘

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 별도의 다차원 큐브(MOLAP)를 만들지 않고, **기존의 관계형 데이터베이스(RDBMS) 기술과 SQL을 그대로 활용하여 다차원 분석(OLAP)을 수행**하는 아키텍처이다.
> 2. **혁신**: 데이터를 큐브 형태로 미리 구워놓을 필요가 없으므로 데이터 용량 확장에 무한대에 가까운 유연성을 가지며, 사용자가 **요약된 데이터부터 가장 밑바닥의 상세 원천 데이터(Drill-through)까지 자유롭게 탐색**할 수 있다.
> 3. **가치**: 대규모 병렬 처리(MPP) 기술 및 클라우드 DW 인프라와 결합하여, MOLAP의 고질적 한계였던 '차원의 저주(용량 폭발)'를 해결한 현대 데이터 웨어하우스의 실질적 표준 모델이다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
ROLAP(관계형 온라인 분석 처리)은 관계형 데이터베이스 엔진 위에 분석 기능을 얹은 것이다. 데이터는 스타 스키마나 스노우플레이크 스키마 형태의 테이블에 일반적인 행과 열(Row & Column) 구조로 저장되며, 분석 요청이 들어오면 실시간으로 `JOIN`과 `GROUP BY` SQL 쿼리가 실행되어 결과를 만들어낸다.

### 💡 비유
- **MOLAP (미리 포장된 세트 메뉴)**: 햄버거, 감자튀김, 콜라 세트가 수만 개 미리 포장되어 창고에 꽉 차 있다. 주문 즉시 내어줄 순 있지만, 감자튀김 대신 치즈스틱으로 구성을 바꾸려면 포장을 다 뜯고 새로 싸야 해서 불가능에 가깝다.
- **ROLAP (주문 즉시 조리하는 뷔페)**: 엄청나게 넓은 주방(대용량 RDB)에 재료가 널려 있다. "버섯과 소고기를 섞은 피자" 등 손님이 어떤 기상천외한 조합(Ad-hoc Query)을 주문하든, 요리사 백 명이 동시에 투입되어 실시간으로 요리해서 내어준다. 아주 약간 기다려야 하지만 어떤 메뉴든 만들 수 있다.

### 등장 배경: 'Data Explosion'
기업의 데이터가 테라바이트(TB) 단위를 넘어가면서, 이를 모두 MOLAP 큐브로 구워내려다 시스템이 터져버렸다(차원의 저주). "조금 느려도 좋으니 데이터 크기 제한 없이 모든 데이터를 분석하고 싶다"는 요구가 커졌고, RDBMS 하드웨어 성능이 비약적으로 발전하면서 ROLAP이 다시 대세로 떠올랐다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### ROLAP 동작 아키텍처 (ASCII Diagram)

```ascii
[ BI Tool / User Dashboard ]
          │ (1. 사용자가 다차원 분석 요청 클릭: "연도별/지역별 매출")
          ▼
┌──────────────────────────────────────────────┐
│ [ ROLAP Engine (Middleware) ]                │
│  - 다차원 요청(MDX)을 복잡한 SQL 쿼리로 번역 │ (2. SQL 변환)
│  - 예: SELECT SUM(amount) FROM fact JOIN dim │
└──────────────────────────────────────────────┘
          │ (3. 쿼리 실행)
          ▼
┌──────────────────────────────────────────────┐
│ [ Relational Database (DW) ]                 │
│  - Star Schema 기반 테이블 물리적 저장       │
│  - 대규모 병렬 처리(MPP) 엔진 가동           │
│  - Columnar Storage 인덱싱 스캔              │
└──────────────────────────────────────────────┘
          │ (4. 실시간 집계 결과 반환)
          ▼
```

### 핵심 작동 메커니즘

#### ① SQL 실시간 변환 엔진
현업 부서는 코딩을 모른다. 화면에서 차원(Dimension) 버튼을 드래그 앤 드롭하면, ROLAP 미들웨어(예: MicroStrategy)가 이를 최적화된 SQL(수십 줄짜리 복잡한 다중 JOIN 구문)로 즉시 번역해 밑단의 RDB로 던진다.

#### ② MPP (Massively Parallel Processing)
ROLAP의 가장 큰 단점인 '느린 쿼리 속도'를 극복하기 위한 하드웨어적 해결책이다. 1대의 강력한 서버 대신, 100대의 노드가 데이터를 쪼개어 동시에 SQL을 처리(Divide and Conquer)하여 결과를 순식간에 취합한다.

#### ③ 인덱스와 파티셔닝 (Indexing & Partitioning)
전체 데이터를 다 읽으면 너무 느리므로, `년도별` 파티션이나 `비트맵(Bitmap) 인덱스`를 떡칠해두어 SQL 쿼리 엔진이 데이터에 접근하는 디스크 I/O를 최소화한다.

## Ⅲ. 융합 비교 및 다각도 분석

### ROLAP vs MOLAP vs HOLAP 비교 (실무 관점)

| 구분 | **ROLAP (관계형)** | MOLAP (다차원 큐브) | HOLAP (하이브리드) |
|:---:|:---|:---|:---|
| **저장 형태** | RDBMS (테이블) | 다차원 Array 큐브 | 테이블(상세) + 큐브(요약) |
| **속도 (성능)** | **쿼리 복잡도에 따라 가변적**| 항상 즉각적(Sub-second) | 보통 빠름 |
| **확장성 (용량)**| **무한에 가까움 (원시 데이터)**| 매우 제한적 (큐브 폭발) | 중간 |
| **Ad-hoc 분석**| **자유로움 (임의의 질문 가능)** | 불가능 (정해진 길로만) | 부분적 가능 |
| **구축/관리** | SQL 지식으로 유지보수 가능 | 전용 큐브 기술/도구 필요 | 관리가 가장 복잡함 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 아키텍처 전략)

**상황**: 대형 이커머스에서 5년 치 고객 행동 로그(수십 테라바이트)를 기반으로, 마케터가 맘대로 조건을 조합해서 타겟 고객을 추출하는 시스템(Ad-hoc Query)을 구축해야 한다.
**판단 및 ROLAP 아키텍처 전략**:
1. **아키텍처 결정**: 조건이 정해져 있지 않고 데이터 양이 방대하므로 MOLAP 큐브는 생성 자체가 불가능하다. 완벽한 **ROLAP** 환경을 선택한다.
2. **DW 엔진 선정**: 일반 MySQL/Oracle로는 감당할 수 없으므로, 대규모 MPP와 컬럼 기반 압축(Columnar)을 지원하는 **Amazon Redshift**나 **Snowflake**를 도입한다.
3. **스키마 설계**: 다차원 분석 쿼리가 쉽게 생성되도록 팩트 테이블과 차원 테이블을 명확히 분리하는 **스타 스키마**로 모델링을 완료한다.
4. **결과**: 마케터가 상상할 수 있는 모든 조합의 질문(Ad-hoc)에 대해, 시스템이 실시간으로 수억 건을 뒤져 10초 이내에 결과를 반환하는 유연한 데이터 분석 환경 완성.

### 안티패턴 및 고려사항
- **정규화의 집착 (3NF 남용)**: ROLAP을 쓴다고 해서 운영 DB처럼 고도로 정규화된 테이블을 놔두면 안 된다. 조인 뎁스가 5단계를 넘어가면 MPP 할아버지가 와도 느리다. ROLAP이라도 반드시 **반정규화(스타 스키마)**를 통해 조인을 단순화해야 한다.
- **Full Table Scan 간과**: ROLAP은 쿼리할 때마다 테이블을 읽는다. 인덱스 없이 잘못된 쿼리를 날리면 DB가 죽어버리거나 엄청난 클라우드 과금 폭탄을 맞을 수 있다. (Partitioning 전략 필수).

## Ⅴ. 미래 전망 및 결론

### 결론: 하드웨어가 만들어낸 승자
초창기 ROLAP은 느리다는 조롱을 받았지만, 클라우드 컴퓨팅과 MPP(병렬 분산 처리) 하드웨어 기술의 눈부신 발전 덕분에 약점을 극복하고 데이터 웨어하우스의 최종 승리자가 되었다.

### 미래 전망
앞으로 ROLAP 솔루션은 단순히 SQL을 처리하는 것을 넘어, 내부에 머신러닝 엔진을 통합하여 "이 매출 하락의 원인이 무엇인가?"를 물어보면 AI가 수많은 차원을 스스로 조인하며 원인을 찾아주는 **'증강 분석(Augmented Analytics)'** 플랫폼으로 진화할 것이다.
