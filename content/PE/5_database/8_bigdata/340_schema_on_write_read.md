# 340. 스키마 온 라이트 (Schema-on-Write) vs 스키마 온 리드 (Schema-on-Read) - 데이터 규격의 딜레마

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터의 구조와 규칙(스키마)을 **'데이터를 저장(Write)할 때 강제할 것인가' 아니면 '데이터를 조회(Read)할 때 해석할 것인가'**를 결정하는 데이터베이스 아키텍처의 근본적인 철학 차이다.
> 2. **스키마 온 라이트 (RDBMS/DW)**: 저장 시점에 엄격한 검사를 거치므로 데이터 무결성과 쿼리 속도가 높지만, 유연성이 떨어지고 구조 변경(Migration) 비용이 크다.
> 3. **스키마 온 리드 (Data Lake/NoSQL)**: 일단 형태 그대로 자유롭게 저장하고, 필요할 때 읽어내면서 의미를 부여하므로 비정형 데이터 처리에 탁월하고 민첩한 실험이 가능하지만, 읽기 속도가 상대적으로 느리다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
**스키마(Schema)**는 데이터베이스의 틀, 즉 뼈대(컬럼 이름, 데이터 타입 등)를 말한다.
- **Schema-on-Write**: 데이터를 창고에 넣기 전에 틀에 맞는지 꼼꼼히 검사한다.
- **Schema-on-Read**: 일단 창고에 다 때려 넣고, 나중에 꺼내 볼 때 "이 모양은 이 틀을 씌워서 봐야겠다"라고 그 순간에 해석한다.

### 💡 비유
- **스키마 온 라이트 (계란판 포장)**: 양계장에서 계란을 포장할 때, 딱 맞는 모양의 계란판(스키마)에만 넣는다. 판에 안 들어가는 타조알이나 메추리알은 저장할 수 없다. 대신 나중에 꺼낼 때 "이 판에는 무조건 계란 10개가 있다"라고 100% 확신할 수 있다.
- **스키마 온 리드 (커다란 바구니)**: 계란, 타조알, 사과, 오렌지까지 커다란 바구니(데이터 레이크)에 막 담아놓는다. 나중에 바구니에서 물건을 꺼낼 때, 사과 깎는 칼(해석 코드)을 가져와 깎아보고 "아, 이건 사과구나" 하고 판단하며 먹는다. 아무 과일이나 담을 수 있어서 편하다.

### 등장 배경: 빅데이터의 역설
과거에는 데이터가 숫자나 짧은 글씨뿐이어서 처음부터 틀(테이블)을 짜는 게 당연했다. 하지만 SNS 글, 사진, 센서 로그 등 제각각인 데이터가 쏟아지면서 처음부터 완벽한 틀을 짜는 것이 불가능해졌다. 저장부터 하고 의미는 나중에 찾는 빅데이터 철학이 대두되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 두 철학의 데이터 처리 파이프라인 (ASCII Diagram)

```ascii
[ 1. Schema-on-Write (전통적 RDBMS, DW) - ETL 방식 ]

Raw Data ───> [ Transform & Validate ] ───> [ Strict Database ]
                  (이 컬럼은 INT형 맞지?        (미리 정의된 Table)
                   이 컬럼은 50자 안 넘지?)         │
                   * 틀리면 에러 내고 버림!         ▼
                                          [ Fast Query (SQL) ]

------------------------------------------------------------------
[ 2. Schema-on-Read (Data Lake, Hadoop) - ELT 방식 ]

Raw Data ───> [ Load to Storage ] ────────> [ Big Storage (S3) ]
             (검사 없이 무조건 저장)           (JSON, CSV, 이미지)
                                                    │
                                                    ▼
   [ Apply Schema & Parse on the fly ] <── [ Query Engine (Spark) ]
   (음, 첫 번째 쉼표까지가 '이름'이겠군)
```

### 핵심 차이점 분석

#### ① 데이터 정제 시점 (ETL vs ELT)
- **온 라이트**: 밖에서 미리 데이터를 가공/정제(Transform)한 뒤에 DB에 적재(Load)한다 (ETL).
- **온 리드**: 가공 없이 원형 그대로 일단 적재(Load)하고, 나중에 분석 도구가 데이터를 읽어내며 런타임에 가공(Transform)한다 (ELT).

#### ② 에러 발생 시점
- **온 라이트**: 저장할 때 튕겨낸다. 깐깐한 경비원. (저장 에러).
- **온 리드**: 꺼내서 분석할 때 포맷이 달라서 에러가 난다. (런타임 에러).

## Ⅲ. 융합 비교 및 다각도 분석

### 5가지 차원의 비교 분석 (Table)

| 비교 항목 | **Schema-on-Write (저장 시 강제)** | **Schema-on-Read (조회 시 해석)** |
|:---:|:---|:---|
| **저장 유연성** | 매우 낮음 (사전 정의 필수) | **매우 높음 (일단 넣고 봄)** |
| **데이터 변경 대응**| ALTER TABLE 등 마이그레이션 고통 | **코드에서 읽는 방식만 바꾸면 됨** |
| **적재(Write) 속도**| 느림 (유효성 검사 병목) | **빠름 (무검사 적재)** |
| **조회(Read) 속도** | **매우 빠름 (정형화된 인덱스 구조)** | 느릴 수 있음 (매번 파싱 및 해석 필요)|
| **대표 플랫폼** | Oracle, PostgreSQL, Snowflake | AWS S3, Hadoop, MongoDB |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 아키텍처 선택 가이드)

**상황**: 병원에서 환자의 '기본 인적 사항(이름, 나이)'과, 웨어러블 시계에서 실시간으로 보내오는 '초당 심박수 센서 로그(JSON 배열)'를 수집하는 시스템을 만든다.
**판단 및 듀얼 트랙 전략**:
1. **환자 인적 사항 (Schema-on-Write)**: 이름이나 연락처는 절대 틀리면 안 되고 조회 규칙이 명확하다. **PostgreSQL** 테이블에 깐깐한 데이터 타입과 제약 조건(Not Null)을 걸어 저장한다.
2. **웨어러블 센서 로그 (Schema-on-Read)**: 센서의 버전이 바뀔 때마다 JSON 항목이 수시로 변한다. 테이블 컬럼을 일일이 바꿀 수 없으므로, 데이터 레이크(S3)에 일단 쌓아둔다. 나중에 연구원들이 분석 코드를 짤 때 JSON의 특정 태그만 파싱해서 읽어낸다.
3. **결과**: 데이터의 성격(명확성 vs 변동성)에 따라 두 가지 철학을 융합하여 시스템의 안정성과 유연성을 동시에 잡음.

### 안티패턴 및 고려사항
- **Schema-on-Read의 게으름**: "나중에 알아서 읽어지겠지"하고 아무 파일이나 호수에 던져두면 결국 해독 불가능한 쓰레기장이 된다(Data Swamp). 반드시 데이터 카탈로그(메타데이터)를 통해 최소한 "이 파일이 대충 어떤 파일인지"는 적어두어야 온 리드가 성립한다.

---

## Ⅴ. 미래 전망 및 결론

### 결론: 데이터의 진화 방향
과거에는 스키마 온 라이트만이 진정한 DB로 여겨졌으나, 폭증하는 빅데이터 환경에서 스키마 온 리드는 필수 불가결한 대세가 되었다. 무엇이 더 우월한지가 아니라, '데이터를 사용할 목적'이 얼마나 명확한가에 따라 선택해야 할 도구다.

### 미래 전망
최근의 기술 발전은 이 둘의 경계를 허물고 있다. **데이터 레이크하우스(Lakehouse)** 환경에서는 S3에 자유롭게 저장(Schema-on-read)하면서도, Delta Lake 같은 계층을 덮어씌워 필요시 엄격한 스키마 강제(Schema Enforcement)를 적용할 수 있게 되었다. 미래의 데이터 플랫폼은 사용자가 버튼 하나로 '유연한 저장'과 '엄격한 관리' 모드를 자유롭게 오고 가는 하이브리드 형태로 굳어질 것이다.

1. **스키마 온 라이트**: 장난감을 상자에 넣을 때 "여기는 미니카 자리, 저기는 인형 자리"라고 딱 정해진 틀에만 조심해서 넣는 방식이에요. 나중에 찾기는 엄청 편해요.
2. **스키마 온 리드**: 일단 아주 큰 장난감 바구니에 로봇이든 레고든 다 쏟아붓고 뚜껑을 닫는 방식이에요. 치우기는 엄청나게 빠르죠!
