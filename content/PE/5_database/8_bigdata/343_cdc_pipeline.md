# 343. 변경 데이터 캡처 (CDC) 데이터 파이프라인 아키텍처

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 운영 데이터베이스(Source)의 트랜잭션 로그를 실시간으로 스트리밍하여, 원본 시스템에 부하를 주지 않으면서 **데이터의 삽입, 수정, 삭제 이벤트를 분석 시스템(Target)으로 지연 없이 동기화**하는 데이터 엔지니어링 기술이다.
> 2. **가치**: 매일 밤 대량의 데이터를 옮기는 무거운 배치(Batch)의 한계를 극복하여 **'실시간 데이터 신선도'**를 확보하며, 여러 마이크로서비스 간의 데이터 일관성을 맞추는 핵심 접착제 역할을 한다.
> 3. **융합**: 메시지 브로커(Apache Kafka), 오픈소스 커넥터(**Debezium**), 그리고 스트림 처리 엔진(Flink, Spark Streaming)과 융합되어 현대적 실시간 데이터 인프라를 형성한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
CDC(Change Data Capture) 파이프라인은 데이터의 '순간 이동 장치'다. 원본 DB에서 한 글자만 바뀌어도 그 변화를 즉시 낚아채서 타겟(Data Warehouse, Search Engine 등)으로 보낸다. 

### 💡 비유
CDC 파이프라인은 **"실시간 뉴스 속보 시스템"**과 같다.
- **기존 방식 (Batch)**: 매일 아침 전날의 사건들을 모아 신문(배치 보고서)을 찍어 배달하는 것. 어제 낮에 일어난 일을 오늘 아침에야 알 수 있다.
- **CDC 방식**: 사건이 터지는 순간 현장 기자가 속보(Event)를 띄우고, TV 뉴스 자막으로 즉시 흘려보내는 것. 시청자(데이터 분석가)는 1초 전의 상황을 바로 알 수 있다.

### 등장 배경 및 발전 과정

#### 1. "DB가 너무 느려요" (Query Load)
전통적인 데이터 수집은 `SELECT *` 쿼리를 날려 데이터를 가져왔다. 운영 중인 DB에 이런 무거운 쿼리를 날리면 실제 서비스가 느려지는 부작용이 컸다.

#### 2. 실시간 비즈니스의 요구
배달 앱이나 주식 앱처럼 1초가 중요한 서비스에서는 데이터 분석 결과가 실시간으로 반영되어야 했고, 이를 위해 DB의 '로그'를 몰래 훔쳐보는(?) CDC 방식이 표준이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### CDC 파이프라인의 3대 계층 (표)

| 계층 (Layer) | 주요 구성 요소 | 상세 역할 | 비유 |
|--------------|----------------|-----------|------|
| **Capture Layer** | **Debezium**, Canal | DB의 Binlog/WAL을 읽어 이벤트 생성 | 속보 기자 |
| **Transport Layer**| **Apache Kafka**, SQS | 이벤트를 유실 없이 고속으로 나름 | 방송 송신탑 |
| **Delivery Layer** | JDBC Sink, S3 Sink | 타겟 시스템에 맞는 포맷으로 적재 | 시청자의 TV 화면 |

### 정교한 구조 다이어그램 (Modern CDC Architecture)

```ascii
================================================================================
[ Modern CDC Pipeline: Debezium + Kafka + Target ]
================================================================================

  [ Source: MySQL ]               [ Kafka Cluster ]               [ Targets ]
 ┌─────────────────┐             ┌───────────────────┐           ┌──────────────┐
 │  Primary Table  │             │  ⭐️ Kafka Topic   │           │ Search Engine│
 │  (Order Table)  │             │  "db.orders"      │──────────▶│ (Elastic)    │
 └────────┬────────┘             └─────────▲─────────┘           └──────────────┘
          │ (1) Write Event                │                              ▲
          ▼                                │ (3) Push Event               │
 ┌─────────────────┐             ┌─────────┴─────────┐           ┌────────┴─────┐
 │ Transaction Log │────────────▶│  CDC Connector    │──────────▶│ BigQuery /   │
 │ (Binlog / WAL)  │  (2) Tail   │  (Debezium)       │           │ Snowflake    │
 └─────────────────┘             └───────────────────┘           └──────────────┘

 * 동작 원리: 
   1. DB는 원래 복구용 로그를 쓴다. 
   2. Debezium이 이 로그의 끝(Tail)을 실시간으로 읽는다. 
   3. "값 A가 B로 바뀜"이라는 이벤트를 Kafka로 쏜다. 
   4. 타겟들이 이를 낚아채서 자기 데이터를 고친다.
```

### 심층 동작 원리

#### ① 비간섭 추출 (Non-intrusive Capture)
운영 DB에 `SELECT` 명령을 내리지 않는다. DB 엔진이 자기 몸을 지키기 위해 자동으로 기록하는 **트랜잭션 로그(Redo Log, Binlog 등)**를 파일 시스템 수준에서 읽어온다. 따라서 운영 서버의 CPU 점유율에 거의 영향을 주지 않는다.

#### ② 원자적 스냅샷과 스트리밍 연동
처음 파이프라인을 구축할 때, 과거 데이터(Snapshot)를 먼저 안전하게 옮긴 뒤, 그 시점부터 발생하는 실시간 로그를 매끄럽게 이어붙이는 '체크포인트' 관리 기술이 핵심이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 일괄 처리(Batch) vs 실시간(CDC) 파이프라인

| 비교 지표 | Batch Pipeline (ETL) | **CDC Pipeline (Real-time)** |
|-----------|----------------------|------------------------------|
| **지연 시간** | 시간 ~ 일 단위 | **밀리초 ~ 초 단위** |
| **운영 부하** | 주기적 피크 발생 | **낮고 일정한 부하** |
| **데이터 신선도**| 낮음 (과거 데이터) | **최상 (지금 이 순간)** |
| **구현 복잡도** | 낮음 | **높음 (메시지 큐 관리 필요)** |
| **주요 용도** | 월말 결산, 장기 통계 | **실시간 추천, 사기 탐지, MSA 동기화** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 레거시 모놀리스 시스템을 마이크로서비스(MSA)로 전환 중이다. 주문 데이터는 여전히 메인 DB에 있는데, 새로 만든 '검색 서비스'와 '쿠폰 서비스'도 실시간 주문 정보를 알아야 한다. 메인 DB를 직접 조회하게 하면 DB가 터질 것 같다.
**판단 및 CDC 구축 전략**:
1. **Kafka-Debezium 스택 도입**: 메인 DB(Oracle/MySQL)에 Debezium 커넥터를 붙인다.
2. **이벤트 드리븐 통신**: 주문 성공 시 발생하는 이벤트를 Kafka 토픽으로 발행한다.
3. **Consumer 분리**: 검색 서비스는 이벤트를 받아 **Elasticsearch**를 갱신하고, 쿠폰 서비스는 이벤트를 받아 즉시 쿠폰을 발송한다.
4. **결과**: 메인 DB의 코드를 한 줄도 고치지 않고(Zero-code change), 신규 서비스들과의 실시간 연동을 완벽히 성공시킨다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **순서 보장 무시**: 네트워크 문제로 '주문 생성' 이벤트보다 '주문 취소' 이벤트가 먼저 도착할 수 있다. 반드시 Kafka의 **Partition Key**를 '주문 ID'로 설정하여, 동일 주문에 대한 이벤트는 항상 동일한 순서로 처리되게 보장해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **데이터 분석 지연(Latency)**: 수 시간에서 **1초 이내**로 단축.
- **운영 안정성**: 쿼리 방식 대비 운영 DB 부하 **80% 이상 절감**.

### 미래 전망 및 진화 방향
CDC 파이프라인은 이제 **'Zero-ETL'**이라는 이름으로 클라우드 서비스에 내재화되고 있다. AWS나 Google Cloud는 클릭 한 번으로 DB와 DW를 CDC로 연결해주는 기능을 강화하고 있다. 향후에는 AI가 데이터 변경 패턴을 실시간 감시하여 보안 위협을 탐지하는 **'Intelligent CDC'**와, 데이터가 이동하는 도중 실시간으로 정합성을 검증하는 자동화 기술이 데이터 엔지니어링의 표준이 될 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **CDC 파이프라인이 뭔가요?**: 친구가 일기장에 글을 쓰자마자, 옆에서 몰래 지켜보던 요정이 그 내용을 똑같이 베껴서 나한테 실시간으로 전달해 주는 마법 통로예요.
2. **왜 좋나요?**: 친구가 일기를 다 쓸 때까지 기다릴 필요 없이, 한 글자 한 글자 쓸 때마다 바로 알 수 있어서 소식이 정말 빠르기 때문이에요.
3. **어떻게 하나요?**: 일기장을 뺏어오는 게 아니라, 일기장이 복사되는 소리(로그)를 듣고 따라 적기 때문에 친구는 내가 훔쳐보는지도 모를 만큼 조용하답니다!
