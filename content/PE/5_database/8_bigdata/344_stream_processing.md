# 344. 스트림 처리 (Stream Processing) DB 기술 - 실시간 이벤트의 자산화

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 정지된 데이터(Data-at-rest)를 조회하는 것이 아니라, **끊임없이 흘러 들어오는 데이터 스트림(Data-in-motion)에 대해 실시간으로 쿼리를 수행**하고 상태를 갱신하는 데이터베이스 및 처리 기술이다.
> 2. **가치**: 데이터가 생성된 지 수 초 이내에 분석 결과(알람, 추천, 사기 탐지 등)를 도출하여 비즈니스 가치를 극대화하며, 무한한 데이터 흐름을 윈도우(Window) 단위로 쪼개어 정교한 통계 연산을 수행한다.
> 3. **융합**: 분산 메시지 브로커(Apache Kafka), 스트림 연산 엔진(Apache Flink), 그리고 인메모리 저장소 기술이 융합되어 '실시간 데이터 기반의 자율 반응 시스템'을 구축한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
스트림 처리 DB는 "멈추지 않는 컨베이어 벨트 위의 검사기"다. 전통적 DB가 도서관에 꽂힌 책을 찾아보는 것이라면, 스트림 DB는 쏟아지는 우편물을 실시간으로 분류하고 통계를 내는 시스템이다. (예: ksqlDB, Apache Flink, Spark Streaming).

### 💡 비유
스트림 처리는 **"흐르는 강물에서 물고기 세기"**와 같다.
- **배치 처리 (전통적)**: 강물을 댐(DB)에 가둔 뒤, 물이 꽉 차면 배를 띄워 물고기가 몇 마리인지 센다. (어제 들어온 물고기 수 확인).
- **스트림 처리**: 강둑에 앉아 지나가는 물고기를 실시간으로 지켜보며 "방금 연어 한 마리 지나갔음!"이라고 수첩에 적는 것. (지금 지나가는 물고기 즉시 확인).

### 등장 배경 및 발전 과정

#### 1. 시간의 가치 (Perishable Insight)
데이터는 시간이 지날수록 가치가 떨어진다. 결제 사기(Fraud)는 발생 후 1시간 뒤에 알면 소용없다. 0.1초 만에 잡아내야 한다.

#### 2. 무한한 데이터 (Unbounded Data)
IoT 센서나 로그 데이터는 끝이 없다. 이를 다 쌓아두고 나중에 분석하려면 저장 비용과 연산 시간이 감당 안 된다. "지나갈 때 분석하고 버리자"는 철학이 스트림 처리를 탄생시켰다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 스트림 처리의 3대 핵심 윈도우(Window) 기법 (표)

무한한 데이터를 연산하기 위해선 '시간의 창'으로 쪼개야 한다.

| 기법 | 작동 방식 | 특징 | 비유 |
|------|-----------|------|------|
| **Tumbling Window** | 고정된 시간 단위로 겹치지 않게 자름 | 1분마다 합계 계산 | 1분 모래시계 |
| **Sliding Window** | 일정 주기마다 윈도우가 미끄러지듯 이동 | 겹치는 구간 발생 (최근 5분간 통계) | 계속 흐르는 CCTV 화면 |
| **Session Window** | 데이터 유입이 끊길 때까지를 하나의 단위로 묶음 | 사용자 활동 주기 분석에 유리 | 손님이 카페에 머문 시간 |

### 정교한 구조 다이어그램 (Streaming Data Pipeline)

```ascii
================================================================================
[ Stream Processing Architecture: Event-driven Intelligence ]
================================================================================

  [ 📥 Event Sources ] (IoT Sensors, App Logs, Clickstream)
           │
           ▼ (1) Ingest: Unbounded Stream
 ┌──────────────────────────────────────────────────────────┐
 │  🚀 Message Broker (Apache Kafka / Pulsar)               │
 │  - Commit Log: Immutable, Ordered Event Storage          │
 └─────────────┬────────────────────────────────────────────┘
               │
               │ (2) Process: Stateful Computation
               ▼
 ┌──────────────────────────────────────────────────────────┐
 │  ⚙️ Stream Processor (Apache Flink / ksqlDB)             │
 │  - Map/Filter/Join/Aggregate                             │
 │  - ⭐️ Managed State: Remember recent 5 min sales ⭐️       │
 └─────────────┬────────────────────────────────────────────┘
               │
               ▼ (3) Sink: Real-time Action
 ┌──────────────────┐    ┌──────────────┐    ┌──────────────────┐
 │ 🔔 Alert Engine  │    │ 📈 Real-time │    │ 🗄️ Serving DB    │
 │ (Stop Fraud!)    │    │ Dashboard    │    │ (Redis / Mongo)  │
 └──────────────────┘    └──────────────┘    └──────────────────┘

 * 핵심 포인트: 프로세서는 데이터를 메모리에 잠시 들고(State) 있으면서 
   새 데이터가 올 때마다 합계를 갱신함.
```

### 심층 동작 원리

#### ① 상태 관리 (State Management)
스트림 DB의 핵심은 '기억력'이다. "방금 들어온 데이터"와 "1분 전 데이터"를 합쳐야 하기 때문에, 연산 엔진 내부의 고속 메모리(RocksDB 등)에 중간 계산 결과를 저장해둔다.

#### ② 워터마크 (Watermark)
네트워크 문제로 데이터가 순서대로 도착하지 않을 수 있다. 워터마크는 "이제 10시 5분 이전 데이터는 더 이상 안 올 거야, 계산 끝내자!"라고 선을 긋는 매커니즘이다. 지연된 데이터(Late data)를 처리하는 지능적인 기준점이 된다.

#### ③ 정확히 한 번 (Exactly-once) 보장
시스템이 다운되어도 중복 계산이나 누락이 없어야 한다. 롤백과 체크포인트를 통해, 마치 장애가 없었던 것처럼 정확한 합계 수치를 유지하는 고도의 분산 정합성 기술이 적용된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 전통적 DB(DBMS) vs 스트림 DB

| 비교 지표 | Traditional DBMS | **Stream Processing DB** |
|-----------|-------------------|--------------------------|
| **데이터** | 정적 (Persistent) | **동적 (Ephemeral / Flow)** |
| **쿼리** | 일시적 (Ad-hoc) | **영구적 (Continuous Query)** |
| **응답** | 요청 시 결과 반환 | **데이터 유입 시 즉시 갱신/전송** |
| **주요 목적** | 과거 데이터 저장 및 조회 | **실시간 상황 인지 및 반응** |
| **비유** | 도서관 | **증권 거래소 전광판** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오)

**상황**: 한 글로벌 신용카드사. 전 세계에서 초당 10만 건의 결제가 일어난다. "동일한 카드로 1분 이내에 서로 다른 3개 국가에서 결제 시도"가 발생하면 즉시 승인을 거절하고 주인에게 알림을 줘야 한다.
**판단 및 스트림 처리 전략**:
1. **Kafka-Flink 스택 구축**: 모든 결제 이벤트를 Kafka로 수집하고 Flink 엔진으로 실시간 처리한다.
2. **Keyed Session Window 적용**: 카드 번호(`card_id`)별로 파티션을 나누고, 1분 단위의 세션 윈도우를 생성한다.
3. **Stateful Logic**: 윈도우 내에서 `country` 필드의 유니크 개수를 센다. 개수가 3개 이상이 되는 즉시 차단 API를 호출한다.
4. **결과**: 결제 시점부터 탐지까지 지연 시간(Latency)을 **100ms 이내**로 단축하여, 피해가 발생하기 전 물리적으로 거래를 차단하는 '완벽한 방어'를 성공시킨다.

### 주의사항 및 안티패턴 (Anti-patterns)
- **과도한 상태 저장 (State Bloat)**: "최근 1년치 모든 기록을 메모리에 들고 분석하겠다"는 것은 불가능하다. 스트림 엔진의 메모리는 한정되어 있으므로, 윈도우 크기를 적절히 조절하고 오래된 데이터는 **TTL(Time To Live)**을 설정해 비워줘야 한다.

---

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과
- **비즈니스 반응 속도**: 배치 처리 대비 인사이트 도출 속도 **1,000배 이상 단축**.
- **인프라 효율**: 필요한 시점에 필요한 데이터만 처리하여 전체 데이터 이동량 최적화.

### 미래 전망 및 진화 방향
스트림 처리는 이제 **'스트리밍 데이터 레이크'**와 결합하고 있다. **Apache Iceberg**와 같은 기술을 통해 실시간 스트림 데이터가 들어오자마자 분석용 테이블로 바로 변신하는 구조다. 또한, AI 모델을 스트림 파이프라인에 직접 심어 "패킷이 지나가는 찰나에 해킹 여부를 AI가 판단"하는 **'In-stream AI'**가 미래 보안과 운영의 핵심이 될 것이다.

1. **스트림 처리가 뭔가요?**: 텔레비전 생중계를 보면서, 골이 들어가는 순간 바로 "골이다!"라고 소리치는 것과 같아요.
2. **왜 좋나요?**: 축구 경기가 다 끝나고 다음 날 신문을 보고 좋아하는 것보다, 골이 들어가는 그 찰나에 같이 기뻐하는 게 훨씬 신나고 빠르기 때문이에요.
3. **가장 좋은 점은요?**: 컴퓨터가 쉬지 않고 눈을 부릅뜨고 지켜보기 때문에, 나쁜 악당이 몰래 들어오려고 할 때 1초도 안 되어서 바로 잡아낼 수 있답니다!
