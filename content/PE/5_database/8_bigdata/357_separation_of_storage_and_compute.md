# 357. 스토리지와 컴퓨팅 분리 아키텍처 (Separation of Storage and Compute) - 클라우드 네이티브의 진정한 해방

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스를 구성하는 두 가지 핵심 자원인 **'데이터를 저장하는 공간(Storage)'과 '쿼리를 연산하는 머리(Compute)'를 물리적/논리적으로 완전히 분리하여 각각 독립적으로 관리**하는 아키텍처 설계다.
> 2. **혁신**: 과거에는 데이터가 늘어나면 필요 없는 CPU까지 같이 사야 했지만(결합 구조), 분리 아키텍처에서는 저장 공간이 필요하면 저렴한 스토리지만 늘리고, 연산이 빡셀 때만 컴퓨팅 자원을 잠깐 늘렸다 줄이는 **개별적 탄력성(Independent Elasticity)**을 달성했다.
> 3. **가치**: Snowflake, BigQuery, 데이터 레이크하우스 등 현대 클라우드 빅데이터 플랫폼이 막대한 확장성과 비용 효율성을 동시에 거머쥘 수 있게 해 준 가장 결정적인 '아키텍처 혁명'이다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
스토리지와 컴퓨팅의 분리는 컴퓨터의 '하드디스크'와 'CPU'를 네트워크 선으로 길게 떼어놓은 것과 같다. 중앙에는 무한히 확장되는 값싼 저장소(AWS S3 등)가 위치하고, 그 주변에 쿼리를 실행하는 연산 서버(Compute Nodes) 군단이 필요할 때만 생성되어 중앙 저장소의 데이터를 읽어다 처리하는 방식이다.

### 💡 비유
- **결합형 아키텍처 (일체형 냉장고 식당)**: 식당 주방(Compute)과 냉장고(Storage)가 한 세트로 붙어 있다. 장사가 잘되어 요리사(CPU)를 더 뽑으려면 요리사만 뽑을 수 없고 냉장고까지 통째로 달린 거대한 주방 세트를 사야 한다. 공간과 돈이 엄청 낭비된다.
- **분리형 아키텍처 (중앙 공유 창고)**: 마을 중앙에 어마어마하게 크고 저렴한 공용 냉장고(S3)가 하나 있다. 요리사가 필요하면 빈 천막(Compute)만 치고 요리사를 고용한다. 요리사들은 중앙 냉장고에서 재료를 가져다 요리한다. 손님이 빠지면 요리사를 즉시 집에 보내고(Scale-in) 천막을 접으면 비용이 나가지 않는다.

### 등장 배경: Shared-Nothing (MPP) 아키텍처의 한계
초기 빅데이터 플랫폼(Hadoop, 초기 Redshift)은 각 서버가 자기 몫의 CPU와 디스크를 함께 가지고 데이터를 쪼개서 처리하는 구조(Shared-Nothing)였다. 데이터 용량이 PB 단위로 커지자, 데이터를 저장하기 위해 굳이 필요도 없는 값비싼 CPU가 달린 서버를 수백 대씩 사서 꽂아야 하는 비효율성(Over-provisioning)이 발생하여 분리형 아키텍처가 등장했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 결합형 vs 분리형 아키텍처 (ASCII Diagram)

```ascii
[ 과거: Coupled Architecture (Shared-Nothing) ]
 ┌─────────┐   ┌─────────┐   ┌─────────┐
 │ Node 1  │   │ Node 2  │   │ Node 3  │
 │ [CPU]   │   │ [CPU]   │   │ [CPU]   │ (연산 능력을 늘리려면 
 │ [DiskA] │   │ [DiskB] │   │ [DiskC] │  반드시 디스크도 늘어남)
 └─────────┘   └─────────┘   └─────────┘
      (데이터가 각 노드에 종속됨)

==================================================================

[ 현대: Decoupled Architecture (Separation of Storage & Compute) ]
                 (Auto-scaling)
  ┌─────────┐   ┌─────────┐   ┌─────────┐
  │ Compute │   │ Compute │   │ Compute │  <-- 1. 컴퓨팅 계층 (Stateless)
  │ Cluster │   │ Cluster │   │ Cluster │      수십 대씩 1초 만에 켜고 끔
  └────┬────┘   └────┬────┘   └────┬────┘
       │             │             │ (High-speed Network)
 ┌─────▼─────────────▼─────────────▼─────┐
 │       Central Cloud Storage (S3)      │  <-- 2. 스토리지 계층 (Stateful)
 │       [Data A] [Data B] [Data C]      │      무한 확장, 쿼리 안 할 땐 보관료만 발생
 └───────────────────────────────────────┘
```

### 3대 핵심 작동 메커니즘

#### ① 상태 비저장 컴퓨팅 (Stateless Compute)
연산을 담당하는 서버에는 영구적인 데이터가 저장되지 않는다. 따라서 쿼리가 폭주할 때 쿠버네티스 컨테이너 띄우듯 서버를 수십 대 추가(Scale-out)해도 데이터 복사나 리밸런싱(Rebalancing)이 필요 없어 확장이 순식간에 일어난다.

#### ② 고속 네트워크와 캐싱 (Network & Caching)
저장소와 CPU가 떨어져 있으면 데이터를 가져오는 네트워크 시간이 오래 걸릴 수 있다. 이를 극복하기 위해 컴퓨팅 노드 내부에 빠른 SSD를 두고, 자주 쓰는 데이터블록을 일시적으로 캐싱(Local Caching)하여 100Gbps 이상의 네트워크 속도로 한계를 돌파한다.

#### ③ 멀티 클러스터 동시 접근 (Multi-cluster Concurrency)
하나의 거대 저장소를 여러 개의 독립된 컴퓨팅 클러스터가 동시에 읽을 수 있다. 영업팀 CPU와 마케팅팀 CPU가 물리적으로 분리되어 있어, 한쪽에서 미친 듯이 무거운 쿼리를 돌려도 다른 쪽 화면은 전혀 느려지지 않는다 (Workload Isolation).

## Ⅲ. 융합 비교 및 다각도 분석

### 아키텍처별 자원 확장 효율성 비교

| 상황 | Shared-Nothing (결합형) | **분리형 (Separation)** |
|:---:|:---|:---|
| **저장 용량만 10배 필요할 때** | 서버를 10대 사야 함 (CPU 낭비 극심) | **S3 용량만 추가 결제 (매우 저렴)** |
| **연산 속도만 10배 필요할 때** | 서버를 10대 사고 데이터를 재분배해야 함 (수 시간~수일 소요) | **Compute 노드만 10대 임대 (1분 내 완료)** |
| **야간에 아무도 안 쓸 때** | 서버를 끌 수 없음 (데이터가 날아가므로) | **Compute 노드 0대 세팅 (연산 비용 0원)** |
| **서버 1대가 고장 났을 때** | 해당 서버의 데이터를 복구할 때까지 장애 | **다른 노드가 즉시 S3에서 데이터 읽어 대체** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 아키텍처 현대화)

**상황**: 온프레미스 Hadoop 클러스터에 10PB의 데이터가 쌓여 있다. 데이터는 계속 늘어나는데 쿼리는 하루에 아침 9시(배치 작업)에만 집중적으로 몰린다. 장비 증설 기안을 올렸더니 CFO가 비용 문제로 반려했다.
**판단 및 분리 아키텍처 전환 전략**:
1. **스토리지 이관**: 10PB의 데이터를 Hadoop HDFS에서 클라우드의 **S3 (Object Storage)**로 모두 옮긴다. S3는 장비 구매 없이 GB당 아주 저렴한 과금으로 무한 보관이 가능하다.
2. **서버리스 연산 도입**: 아침 9시에만 연산이 폭발하므로, **AWS Athena**나 **Snowflake**의 Auto-scaling 컴퓨팅 자원을 설정한다.
3. **비용 최적화 (Cost Arbitrage)**: 9시부터 10시까지 1시간 동안만 컴퓨팅 노드 100대를 풀가동하여 분석을 끝내고, 10시 이후에는 컴퓨팅 노드를 0으로 꺼버린다(Suspend).
4. **결과**: 장비 도입 비용(CAPEX)을 완전히 없애고, 하루 23시간 동안 발생하던 잉여 컴퓨팅 유지비용을 제로화하여 전체 TCO를 70% 삭감함.

### 안티패턴 및 고려사항
- **초고빈도 단건 트랜잭션(OLTP)**: 분리형 아키텍처는 네트워크 레이턴시가 필수적으로 개입된다. 분석(OLAP)처럼 한 번에 1GB를 쫙 읽어오는 작업에는 엄청나게 유리하지만, 주식 거래나 게임 서버처럼 1ms(밀리초) 단위로 데이터를 1건씩 넣고 빼는 작업(OLTP)에는 네트워크 지연 탓에 쥐약이다. 여전히 로컬 디스크가 결합된 아키텍처가 필요하다.

## Ⅴ. 미래 전망 및 결론

### 결론: 데이터 플랫폼의 르네상스
스토리지와 컴퓨팅의 분리는 클라우드가 빅데이터에 내린 최고의 축복이다. 데이터를 "일단 싼 곳에 무한정 쌓아둘 수 있는 자유"와 "필요할 때만 폭발적인 힘을 빌려 쓰는 자유"를 동시에 선사하며 빅데이터 생태계의 판도를 바꿨다.

### 미래 전망
앞으로는 이 구조를 더욱 극단으로 밀어붙인 **'마이크로 컴퓨팅 풀(Micro-compute Pools)'**이 대세가 될 것이다. 쿼리의 복잡도를 AI가 실시간으로 분석하여, "이 쿼리는 CPU 3.2개와 RAM 12GB면 2초 안에 끝난다"고 정확히 계산해 서버리스 펑션(Function) 단위로 아주 미세하게 자원을 할당하고 즉시 회수하는 극한의 비용 최적화 시대가 도래할 것이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **분리 아키텍처가 뭔가요?**: 장난감 창고(스토리지)와 장난감을 조립하는 테이블(컴퓨팅)을 완전히 떼어놓는 방법이에요.
2. **왜 떼어놓나요?**: 예전엔 책상 밑에만 장난감을 보관할 수 있어서 장난감이 많아지면 책상을 계속 사야 했어요. 하지만 떼어놓으면 아주 큰 창고 하나만 지어놓고, 조립할 때만 싼 접이식 책상을 잠시 폈다가 접으면 돼서 돈이 훨씬 안 들거든요.
3. **가장 좋은 점은?**: 내 창고에 있는 장난감을 친구가 같이 놀고 싶다고 할 때, 내 책상을 빌려줄 필요 없이 친구가 자기 접이식 책상을 펴서 놀면 되니까 절대 싸우거나 방해받을 일이 없답니다!
