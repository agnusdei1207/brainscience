# 365. 데이터베이스 암호화 (TDE, Transparent Data Encryption) - 흔적 없이 데이터를 굳히는 투명한 방패

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 응용 프로그램(Application) 코드를 단 한 줄도 수정하지 않고, **데이터베이스 관리 시스템(DBMS) 엔진 내부나 파일 시스템 레벨에서 데이터를 물리적 디스크에 쓸 때 암호화하고, 읽을 때 복호화하는 기술**이다.
> 2. **혁신**: 해커가 하드디스크나 백업 테이프를 통째로 훔쳐 가더라도(휴지 상태 암호화, Data at Rest), 암호화 키가 없으면 데이터가 완벽한 쓰레기 값으로 보여 원천적인 정보 유출을 차단한다.
> 3. **가치**: 가장 적은 도입 비용과 시스템 수정 없이 강력한 컴플라이언스(개인정보보호법, PCI-DSS 등)를 만족시킬 수 있는 엔터프라이즈 환경의 가장 기본적인 데이터 보안 표준이다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
TDE(투명한 데이터 암호화)에서 '투명(Transparent)'하다는 것은 개발자나 사용자 눈에 암호화 과정이 전혀 보이지 않는다는 뜻이다. SQL로 `INSERT`를 날리면 DB 엔진 뒷단에서 알아서 암호화하여 하드디스크에 저장하고, `SELECT`를 날리면 알아서 암호를 풀어서(복호화) 원본을 보여준다.

### 💡 비유
- **API/App 레벨 암호화 (직접 포장하기)**: 직원이 회사 기밀을 창고(DB)에 넣을 때, 직접 비밀번호를 걸고 박스를 테이프로 칭칭 감아서 넣는다. 꺼낼 때도 자기가 직접 테이프를 뜯어야 한다. 귀찮고, 실수로 안 묶고 넣을 수도 있다.
- **TDE (자동 금고 문)**: 직원은 아무 생각 없이 서류를 창고 문턱에 던져놓는다. 그러면 마법의 창고 문(DBMS 엔진)이 닫히는 순간 모든 서류가 스르륵 암호화 금고로 변한다. 직원이 "서류 꺼내줘"라고 하면 문턱을 지날 때 자동으로 원래 종이로 바뀐다. 직원은 창고 안에서 무슨 일이 일어났는지 전혀 모른다(투명성). 하지만 도둑이 벽을 뚫고 창고 안으로 들어오면 꽁꽁 잠긴 금고 덩어리밖에 못 본다.

### 등장 배경: 'Media Theft'와 법적 규제
2000년대 초반, 서버를 통째로 해킹하는 것이 아니라 아예 야간에 전산실 침입해서 '백업 하드디스크'를 물리적으로 훔쳐 가는 사건이 빈번했다. 디스크를 훔치면 데이터가 평문(Plaintext)으로 노출되어 피해가 막심했다. 결국 "하드디스크가 털려도 안전하게 만들라"는 법적 규제가 강화되었고, 기존 소스코드를 고치는 고통 없이 이를 해결해 주는 TDE가 오라클, MS SQL 등을 중심으로 기본 기능이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### TDE 아키텍처 (ASCII Diagram)

```ascii
[ Application Layer ]
   (SELECT, INSERT, UPDATE - 일반적인 평문(Plaintext) SQL 사용)
          │
┌─────────▼────────────────────────────────────────┐
│ [ Database Engine (Memory / Buffer Cache) ]      │
│  - 메모리 상에서는 데이터가 "평문"으로 존재           │
│                                                  │
│      ┌───────────────────────────────────┐       │
│      │ [ TDE Encryption/Decryption Engine]       │  <-- 암복호화 엔진
│      │ (AES-256 알고리즘 수행)                │       │
│      └─────────────────┬─────────────────┘       │
└────────────────────────┼─────────────────────────┘
   (디스크로 내려갈 때 암호화) │ (디스크에서 올라올 때 복호화)
                         ▼
             [ Physical Storage (Disk) ]
      - 암호화된 데이터 파일 (Data files: .mdf, .dbf)
      - 암호화된 백업 파일
      - 암호화된 트랜잭션 로그
```

### 2단계 암호 키(Key) 관리 아키텍처

TDE의 핵심은 '열쇠(Key)'를 어떻게 안전하게 보관하느냐다. 자물쇠와 열쇠를 같은 하드디스크에 두면 도둑이 같이 훔쳐 가기 때문이다.
1. **DEK (Data Encryption Key)**: 실제 데이터를 암호화하는 열쇠다. 데이터 파일 헤더에 숨겨져 있다.
2. **MEK (Master Encryption Key)** 또는 **KEK (Key Encryption Key)**: DEK를 암호화하는 마스터 열쇠다. 이 마스터 열쇠는 DB 외부에 있는 아주 안전한 '지갑(Wallet)'이나 하드웨어 보안 모듈(HSM, KMS)에 별도로 격리 보관한다.

## Ⅲ. 융합 비교 및 다각도 분석

### DB 암호화 방식 3대장 비교 (Application vs DB vs TDE)

| 구분 | Application 암호화 (API 방식) | DB 내부 암호화 (Plug-in 방식) | **TDE (투명한 데이터 암호화)** |
|:---:|:---|:---|:---|
| **암호화 주체** | 애플리케이션 서버 (Web/WAS) | DB 엔진 플러그인 / 트리거 | **DBMS 커널 자체 / OS 파일시스템**|
| **소스 수정 여부**| 대대적인 코드 수정 필요 | 쿼리 및 DB 설정 일부 수정 | **전혀 없음 (Transparent)** |
| **보안 범위** | 네트워크 전송 구간부터 완벽 | 네트워크 구간은 별도 암호화 필요| DB 밖(메모리/네트워크)은 평문 |
| **디스크 해킹 방어**| 완벽히 방어 | 완벽히 방어 | **완벽히 방어 (최대 강점)** |
| **인덱스 검색** | 암호화된 상태라 색인 검색 어려움| 일치 검색만 가능 (LIKE 검색 한계)| **기존 B-Tree 인덱스 100% 정상 작동** |
| **성능 저하** | WAS CPU 부하 | DB CPU 부하 큼 | **OS I/O 부하 약간 (가장 성능 좋음)**|

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 개인정보 보호 전략)

**상황**: 병원 시스템에 1,000만 명의 환자 정보가 있다. 정부에서 "주민등록번호와 진료 기록을 당장 암호화하라"는 지시가 내려왔다. 하지만 시스템 소스코드가 너무 낡아서(스파게티 코드) 개발자가 코드를 건드리면 병원 전체가 멈출 위험이 있다.
**판단 및 TDE 도입 전략**:
1. **방식 선정**: 소스코드를 단 한 줄도 건드리지 않는 **TDE(Transparent Data Encryption)** 방식을 선택한다. (Oracle TDE 또는 MS SQL TDE 적용).
2. **범위 설정**: 전체 DB를 다 암호화하면 디스크 I/O 성능이 10~20% 떨어질 수 있으므로, 환자 정보가 들어있는 특정 '테이블스페이스(Tablespace)'나 '파일 그룹'만 선택적으로 TDE 암호화를 건다.
3. **KMS(키 관리) 분리**: 마스터 암호 키(MEK)는 DB 서버 안에 두지 않고, 물리적으로 완벽히 격리된 AWS KMS나 외부 사내 HSM(Hardware Security Module) 장비에 보관한다.
4. **결과**: 금요일 자정 2시간의 DB 재시작 작업만으로 소스코드 수정 제로(0), 1,000만 명의 데이터를 법적 기준에 맞춰 완벽히 암호화 완료.

### 안티패턴 및 고려사항
- **메모리(RAM) 탈취 공격에 무방비**: TDE는 오직 '하드디스크가 뽑혔을 때(Data at Rest)'만 보호한다. 해커가 DB 서버에 접속해서 `SELECT` 쿼리를 날릴 권한을 얻거나, 서버 메모리(RAM)를 덤프 떠버리면 데이터는 평문(Plaintext)으로 훤히 보인다. 즉, TDE를 했다고 안심하면 안 되며 **DB 접근 제어 방화벽**이 반드시 세트로 들어가야 한다.

## Ⅴ. 미래 전망 및 결론

### 결론: 보안의 가성비 끝판왕
TDE는 가장 적은 고통으로 가장 확실한 규제(Compliance) 방패를 얻는 마법이다. "암호화는 비즈니스 로직(개발자)의 영역이 아니라, 인프라(DBA)의 영역이다"라고 선을 그어버린 기념비적인 기술이다.

### 미래 전망
현대의 모든 클라우드 데이터베이스(AWS RDS, Aurora, DynamoDB 등)는 생성 버튼을 누를 때 **'스토리지 암호화(KMS 연동 TDE)'가 기본적으로 켜진 채(Default ON)** 만들어진다. 즉, TDE는 더 이상 특별한 기술이 아니라 수돗물처럼 당연히 깔려있는 인프라가 되었다. 앞으로의 보안 패러다임은 디스크(TDE)를 넘어, 메모리 연산 중에도 데이터를 암호화하는 **'컨피덴셜 컴퓨팅'**이나 **[동형 암호(FHE)](./362_privacy_preserving_db.md)**의 영역으로 이동하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. **TDE가 뭔가요?**: 일기장에 글을 쓸 때, 평범하게 글씨를 쓰면 일기장 겉표지가 닫히는 순간 일기장이 알아서 외계어 암호로 싹 바꿔버리는 신기한 투명 마법이에요.
2. **왜 투명하다고 하나요?**: 내가 일기장을 펼쳐서 읽을 때는 다시 원래 글씨로 짠! 하고 돌아오기 때문에, 나는 일기가 암호로 바뀌었는지 눈치도 못 채고 아주 편하게 쓸 수 있거든요.
3. **뭐가 좋나요?**: 도둑이 내 일기장을 훔쳐 가서 몰래 열어보면 이상한 외계어만 가득해서 절대 내 비밀을 알 수 없답니다!
