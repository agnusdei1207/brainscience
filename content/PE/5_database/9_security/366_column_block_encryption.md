# 366. 컬럼 레벨 암호화 / 블록 레벨 암호화 - 데이터 보안의 정밀 타격과 광역 방어

## 핵심 인사이트 (3줄 요약)
> 1. **컬럼 레벨 암호화(Column-Level)**: 테이블 전체가 아닌 **주민등록번호, 비밀번호 등 극도로 민감한 특정 '열(Column)'만을 선택하여 암호화**하는 방식으로, 가장 정밀한 접근 통제가 가능하지만 구현이 복잡하고 검색 성능이 떨어질 수 있다.
> 2. **블록 레벨 암호화(Block/File-Level)**: OS나 디스크 수준에서 **데이터베이스 파일이 저장되는 '블록' 전체를 뭉텅이로 암호화(예: TDE)**하는 방식으로, 시스템 성능 저하가 적고 앱 수정이 필요 없지만 DB 계정을 탈취당하면 데이터가 평문으로 노출된다.
> 3. **비교 가치**: 보안의 깊이(정밀도)를 택할 것인가, 운영의 효율성과 성능(광역성)을 택할 것인가에 대한 아키텍처적 결단이며, 최상의 보안을 위해서는 두 방식을 혼합(Hybrid)하여 사용하는 것이 일반적이다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
데이터베이스 암호화의 '단위(Granularity)'에 관한 이야기다. 
- **컬럼 레벨**: 수술실에서 환자의 아픈 부위(컬럼)만 정확히 도려내 치료하는 '외과 수술'과 같다.
- **블록 레벨**: 아픈 부위뿐만 아니라 몸 전체가 있는 병실(디스크 블록) 전체에 소독 가스를 뿌려버리는 '광역 소독'과 같다.

### 💡 비유
- **블록 레벨 암호화 (은행 대형 금고)**: 은행 건물 안에 거대한 무쇠 금고실이 있다. 금고 문을 닫으면 그 안의 서류(데이터)는 통째로 보호된다. 하지만 은행 직원이 금고 문을 열고 들어가면 수십만 장의 고객 명부를 마음대로 볼 수 있다.
- **컬럼 레벨 암호화 (개인용 자물쇠 서류철)**: 금고실 밖 책상에 서류가 널려 있다. 하지만 서류에 적힌 '주민번호' 부분에는 까만색 자물쇠 스티커(암호)가 붙어 있다. 은행 직원이라도 이 스티커를 떼어내는 '개인 전용 열쇠'가 없으면 주민번호는 절대 볼 수 없다. 특정 부위만 완벽히 잠그는 방식이다.

### 등장 배경: '성능'과 '세밀한 통제'의 줄다리기
과거에는 DB 서버 통째로 블록 암호화(TDE)를 하면 끝인 줄 알았다. 하지만 내부자인 DBA나 개발자가 DB에 합법적으로 접속하여 고객의 신용카드 번호를 싹 긁어가는 내부 유출 사고가 터졌다. 이를 막으려면 DBA조차 볼 수 없게 데이터의 가장 작은 단위(컬럼)에 암호를 걸어야 했다. 하지만 컬럼 암호화를 너무 많이 걸면 시스템이 심각하게 느려지는 딜레마가 발생했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 두 방식의 작동 계층 (ASCII Diagram)

```ascii
[ Application Layer ]  <-- 컬럼 레벨 암호화 1 (앱에서 직접 암호화)
         │  
         ▼ 
┌──────────────────────────────────────┐
│ [ Database Management System (DBMS) ]│
│                                      │
│  - SQL Parser / Optimizer            │
│  - Column: [Name:홍길동][SSN: @!#$*] │ <-- 컬럼 레벨 암호화 2 (DB Plug-in)
│          (평문)        (암호문)       │
└────────────────┬─────────────────────┘
                 │
                 ▼ (디스크로 내려갈 때 블록 전체를 암호화)
┌──────────────────────────────────────┐
│ [ File System / OS Layer ]           │
│  - Block 1: [ ^&*QWE!@#ZXCV... ]     │ <-- 블록 레벨 암호화 (TDE, OS 암호화)
│  - Block 2: [ %^&YUIOP!@#...   ]     │ 
└──────────────────────────────────────┘
```

### 핵심 매커니즘 차이

#### ① 컬럼 레벨 암호화 (Column-level Encryption)
- **방식**: `INSERT` 문이 실행될 때 `SSN(주민번호)` 컬럼의 값만 콕 집어서 암호화 함수(`ENCRYPT()`)를 태운다.
- **키(Key) 관리**: 컬럼마다 서로 다른 암호 키를 부여할 수 있다. 인사팀은 급여 컬럼 키를 갖고, 고객센터는 주소 컬럼 키를 갖게 하여 완벽한 **접근 제어 분리**가 가능하다.
- **인덱스 문제**: 값이 찌그러진 암호문으로 바뀌었기 때문에, "주민번호가 90년대생인 사람(`LIKE '90%'`)" 같은 범위 검색이나 인덱스 스캔이 불가능해진다.

#### ② 블록 레벨 암호화 (Block-level Encryption)
- **방식**: [TDE (Transparent Data Encryption)](./365_tde_encryption.md)처럼 데이터베이스가 OS의 파일(블록)에 기록할 때 뭉텅이로 암호화한다.
- **특징**: DB 엔진 메모리(RAM) 상에서는 데이터가 평문으로 존재하므로, DB 내부의 인덱스 검색, 조인, 정렬 기능이 100% 완벽하게 동작한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 컬럼 레벨 vs 블록 레벨 핵심 비교표

| 비교 기준 | **컬럼 레벨 (Column-level)** | **블록 레벨 (Block/TDE)** |
|-----------|------------------------------|---------------------------|
| **방어 대상** | 디스크 도난 + **내부자(DBA) 유출** 방어 | **물리적 디스크 도난** 방어 |
| **암호화 단위** | 특정 컬럼 (SSN, 비밀번호, 카드번호) | DB 파일, 테이블스페이스 전체 |
| **애플리케이션 수정**| 필요함 (플러그인 방식이 아닐 경우) | 전혀 필요 없음 (투명함) |
| **검색(Index) 성능**| 제약 큼 (범위/부분 검색 불가) | 제약 없음 (100% 기능 보장) |
| **CPU 부하** | 컬럼을 암/복호화 할 때마다 폭증 | 낮음 (I/O 시점에만 한 번 수행) |
| **도입 목적** | 강력한 컴플라이언스와 내부 통제 | 기본적이고 광범위한 법적 규제 준수|

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 금융권 하이브리드 전략)

**상황**: 새로운 핀테크 서비스를 론칭한다. 고객의 기본 프로필, 수백만 건의 결제 내역, 그리고 극비 데이터인 '신용카드 번호'를 저장해야 한다.
**판단 및 하이브리드 암호화 전략**:
1. **광역 방어 (블록 레벨 TDE 적용)**: 전체 DB 스토리지에 대해 AWS KMS와 연동된 **블록 레벨 암호화(TDE)**를 무조건 켠다. 이로써 해커가 클라우드 스냅샷을 털어가도 안전하다. (성능 저하 없음, 규제 1차 통과).
2. **정밀 방어 (컬럼 레벨 적용)**: 결제 내역이나 이름은 TDE 방어에 맡긴다. 하지만 **'신용카드 번호' 컬럼 단 하나**만은 애플리케이션 단에서 **컬럼 레벨 암호화(API 방식)**를 한 번 더 적용하여 저장한다.
3. **통제권 분리**: 만약 나쁜 마음을 먹은 DBA가 DB에 몰래 접속(`SELECT *`)하더라도, 카드 번호는 컬럼 암호화 때문에 `f9X!@2#` 같은 쓰레기 값으로 보여 도난을 완벽히 방지한다.
4. **결과**: 인덱스 검색 성능은 99% 보존하면서, 가장 치명적인 핵심 데이터의 내부자 유출 리스크를 0%로 차단하는 최적의 가성비 아키텍처 완성.

### 안티패턴 및 고려사항
- **모든 컬럼의 암호화 (과잉 보호)**: 컬럼 레벨 암호화가 안전하다고 해서 '성별', '나이', '주소' 같은 수십 개의 컬럼에 모두 암호화를 거는 행위. DB CPU가 100%를 치고 쿼리 속도가 수십 배 느려져 서비스가 파탄 난다. 컬럼 암호화는 법적으로 강제된 **고유 식별 정보(PII)**에만 핀셋처럼 써야 한다.

## Ⅴ. 미래 전망 및 결론

### 결론: 두 겹의 갑옷이 만드는 완벽함
컬럼 암호화와 블록 암호화는 어느 하나가 우월한 것이 아니라 보완재다. 성벽을 두껍게 쌓는 것(블록 레벨)과, 왕의 방에 들어가는 열쇠를 따로 만드는 것(컬럼 레벨)을 병행해야 진짜 뚫리지 않는 성이 완성된다.

### 미래 전망
컬럼 레벨 암호화의 가장 큰 적인 '검색(Index) 불가' 문제를 해결하기 위해, 최근 데이터의 길이와 형태를 유지한 채 암호화하는 **FPE (Format Preserving Encryption)** 기술이 대세로 떠오르고 있다. (예: `010-1234-5678` -> `010-8822-1923` 형태로 변환). 나아가 컬럼을 암호화한 상태에서도 크기 비교(>, <)나 검색이 가능하게 해주는 **순서 보존 암호(OPE, Order-Preserving Encryption)** 기술이 결합되어 성능과 보안의 두 마리 토끼를 잡을 전망이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **블록 레벨 암호화**: 집(데이터베이스) 전체를 투명 망토로 씌워서 밖에서 못 보게 하는 거예요. 집안에 있는 사람(DB 관리자)은 모든 물건을 편하게 다 볼 수 있어요.
2. **컬럼 레벨 암호화**: 집 안에 있는 수많은 서랍 중, '비밀 일기장(주민번호)'이 든 서랍 딱 하나에만 튼튼한 자물쇠를 거는 거예요. 같은 집에 사는 사람이라도 열쇠가 없으면 절대 못 봐요.
3. **어떻게 쓰나요?**: 보통은 집 전체에 투명 망토(블록)를 씌워두고, 진짜진짜 중요한 일기장에만 추가로 자물쇠(컬럼)를 달아두는 게 가장 안전하고 똑똑한 방법이랍니다!
