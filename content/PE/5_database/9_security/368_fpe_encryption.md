# 368. FPE (Format Preserving Encryption) - 데이터의 형태를 지키는 영리한 암호

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 암호화된 데이터가 원본 데이터와 **동일한 형식(Format), 길이, 문자 집합을 그대로 유지**하도록 설계된 특수 암호화 기술이다. (예: 16자리 카드번호 암호화 시 결과값도 16자리 숫자).
> 2. **혁신**: 일반 암호화(AES 등)가 데이터를 바이너리나 더 긴 문자열로 바꾸어 DB 스키마 변경을 강제하는 반면, FPE는 **기존 DB 구조나 애플리케이션의 유효성 체크 로직을 단 한 줄도 고치지 않고 적용** 가능하다.
> 3. **가치**: 신용카드 번호(PCI-DSS), 주민등록번호 등 엄격한 형식이 정해진 민감 정보를 암호화할 때 인프라 수정 비용을 최소화하며, 암호화된 상태에서도 레거시 시스템과의 호환성을 완벽히 보장한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
FPE(형태 보존 암호화)는 암호학의 '변신 마술'이다. 숫자를 암호화하면 숫자가 나오고, 한글을 암호화하면 한글이 나온다. NIST SP 800-38G 표준으로 제정된 FF1, FF3-1 알고리즘이 대표적이다.

### 💡 비유
- **일반 암호화 (바구니에 담기)**: 사과(데이터)를 암호화하면 거대한 보따리(바이너리 데이터)가 된다. 원래 사과 상자(DB 컬럼)에 넣으려니 보따리가 너무 커서 안 들어간다. 상자를 새로 짜야 한다.
- **FPE (빨간 사과를 파란 사과로)**: 빨간 사과를 암호화하면 파란 사과가 된다. 색깔(내용)은 바뀌었지만 크기와 모양(형태)은 똑같다. 원래 있던 사과 상자(DB 컬럼)에 아무 문제 없이 쏙 들어간다. 심지어 상자를 관리하는 사람(애플리케이션)은 "어? 사과 색깔이 좀 특이하네"라고만 생각하고 평소대로 상자를 옮길 수 있다.

### 등장 배경: 'Schema Migration'의 지옥
전통적인 AES 암호화로 카드번호 16자리를 암호화하면 최소 32바이트 이상의 Base64 문자열이 된다. 이를 저장하려면 전사 시스템의 DB 컬럼 크기를 늘리고, 카드번호 16자리만 입력받게 설계된 모든 소스코드를 고쳐야 했다. 이 막대한 마이그레이션 비용을 줄이기 위해 "형태는 놔두고 값만 섞자"는 요구가 FPE를 탄생시켰다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### FPE 작동 프로세스 (ASCII Diagram)

```ascii
[ Original Data ] : 1234-5678-9012-3456 (16 digits)
          │
┌─────────▼────────────────────────────────────────┐
│ [ FPE Engine (FF1 / FF3-1 Algorithm) ]           │
│                                                  │
│  1. Domain Definition: {0-9} (숫자 집합 정의)      │
│  2. Tweak: 'User_ID_123' (암호문에 변동성 부여)    │
│  3. Feistel Network: 여러 라운드의 치환/전치 수행   │
└─────────────────────────┬────────────────────────┘
                          │
[ Encrypted Data ] : 9821-0455-7139-2284 (16 digits)

* 결과: 데이터 타입(Numeric), 길이(16자)가 원본과 완벽히 동일함.
```

### 핵심 매커니즘 (Feistel Network 기반)

FPE는 일반 암호화와 달리 특정 '도메인(문자 집합)' 안에서만 순열(Permutation)을 일으킨다.
1. **FF1 알고리즘**: 가변적인 길이의 문자열을 암호화하는 데 최적화되어 있으며, 보안성이 매우 높다.
2. **Tweakable Encryption**: 암호 키(Key) 외에 '트윅(Tweak)'이라는 추가 변수를 쓴다. 똑같은 카드번호라도 사용자 ID에 따라 다른 암호문이 나오게 하여 보안을 강화한다.
3. **사이클 워킹 (Cycle Walking)**: 만약 결과값이 허용된 범위(예: 0~9)를 벗어나면, 범위 안으로 들어올 때까지 내부적으로 다시 암호화를 반복하여 최종적으로 형상을 맞춘다.

## Ⅲ. 융합 비교 및 다각도 분석

### AES (표준 암호) vs FPE (형태 보존 암호) 비교

| 비교 항목 | AES (Advanced Encryption Standard) | **FPE (Format Preserving)** |
|-----------|------------------------------------|-----------------------------|
| **결과 형태** | 고정된 블록 크기의 바이너리/Base64 | **원본과 동일한 타입/길이** |
| **DB 영향** | **컬럼 크기 변경 필수 (D D L 수정)** | **변경 없음 (스키마 유지)** |
| **앱 영향** | 입출력 로직 및 유효성 체크 대폭 수정 | **수정 최소화 (데이터 형식 유지)** |
| **보안 강도** | 매우 높음 (표준) | 높음 (NIST 표준 승인) |
| **연산 속도** | 매우 빠름 (HW 가속 지원) | AES보다 상대적으로 느림 |
| **주 사용처** | 일반 파일, 전문 전송, 비밀번호 | **카드번호, 주민번호, 전화번호** |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 레거시 금융망 보안 전략)

**상황**: 20년 된 메인프레임 기반 카드 결제 시스템에 보안 감사가 떴다. 모든 카드번호를 즉시 암호화해야 한다. 하지만 시스템이 너무 복잡해서 DB 스키마를 고치면(Varchar 크기 확대 등) 연동된 100개의 프로그램이 다 깨질 판이다.
**판단 및 FPE 도입 전략**:
1. **알고리즘 선정**: 스키마 변경이 절대 불가한 상황이므로 **NIST 표준 FF1 기반 FPE**를 채택한다.
2. **Tweak 설계**: 데이터의 유일성을 보장하기 위해 '가맹점 번호'를 트윅(Tweak)으로 사용하여 암호화 시 중복 발생(Collision)을 방지한다.
3. **부분 암호화(Masking 병행)**: 카드번호 16자리 중 앞 6자리(BIN)와 뒤 4자리는 업무상 조회가 필요하므로 남겨두고, 중간 6자리만 FPE로 암호화하여 '부분적 형태 보존'을 실시한다.
4. **결과**: DB 테이블 구조 수정 없이 단 1주일 만에 암호화를 적용하고, 기존 결제 애플리케이션의 90%를 수정 없이 그대로 가동함. (막대한 비용 절감).

### 안티패턴 및 고려사항
- **짧은 데이터의 취약성**: 3자리 숫자(예: CVV 번호)를 FPE로 암호화하면 경우의 수가 1,000개뿐이다. 해커가 모든 경우의 수를 다 대입해보는 '사전 공격(Brute-force)'에 쉽게 뚫릴 수 있다. FPE는 **최소 10자리 이상의 충분한 엔트로피를 가진 데이터**에 사용해야 안전하다.

## Ⅴ. 미래 전망 및 결론

### 결론: 실용주의 보안의 정수
FPE는 암호학이 비즈니스 현장의 고통(호환성)을 이해하고 양보하여 만들어진 기술이다. "보안을 위해 모든 것을 바꿔라"가 아닌 "보안이 시스템에 맞춰주겠다"는 철학의 승리다.

### 미래 전망
앞으로는 토큰화(Tokenization) 기술과 결합하여, 실제 암호화된 값조차 DB에 두지 않고 FPE로 생성된 '가짜 값(Token)'만 유통하는 방식이 대세가 될 것이다. 또한, 클라우드 환경의 **[데이터 마스킹](./367_data_masking.md)** 솔루션들이 내부적으로 FPE를 탑재하여, "암호화된 상태로 개발하고, 운영에서만 복호화하는" 투명한 개발 생태계를 완성할 전망이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **FPE가 뭔가요?**: 내 비밀번호 "1234"를 암호로 바꿀 때, 엉뚱하고 긴 글자로 바꾸는 게 아니라 똑같이 숫자 4자리인 "8729"로 바꿔주는 기술이에요.
2. **왜 그렇게 하나요?**: 비밀번호 칸이 딱 4칸밖에 없어서, 암호가 너무 길어지면 칸에 안 들어가서 기계가 고장 날 수 있거든요.
3. **가장 좋은 점은?**: 기계를 새로 사거나 고치지 않고도, 비밀번호만 싹 바꿔서 도둑을 완벽하게 속일 수 있답니다!
