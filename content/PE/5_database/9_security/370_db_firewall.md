# 370. DB 접근 제어 (DB 방화벽) - 데이터 요새의 최전방 수문장

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 서버 앞단에 위치하여 접속자의 신원, IP, 시간, SQL 문장을 실시간으로 감시하고, **사전에 정의된 정책에 따라 허용되지 않은 접근이나 위험한 SQL 실행을 즉시 차단하는 데이터 전문 보안 솔루션**이다.
> 2. **혁신**: 단순히 네트워크 연결을 막는 일반 방화벽과 달리, **SQL의 내용(Content)을 심층 분석(DPI)**하여 "낮 시간엔 조회 가능하지만, 새벽엔 금지", "한 번에 1,000건 이상의 데이터 삭제 시 차단" 등 정교한 비즈니스 룰 기반 통제를 수행한다.
> 3. **가치**: 관리자(DBA)나 개발자의 실수 및 권한 오남용을 막고, 외부 해커의 SQL 인젝션 공격을 최전방에서 무력화하여 기업의 핵심 자산인 데이터를 물리적/논리적으로 보호한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
DB 접근 제어(Database Access Control) 또는 DB 방화벽은 데이터베이스로 가는 길목을 지키는 '검문소'다. 모든 사용자는 이 검문소를 통과해야만 DB에 명령을 내릴 수 있다. 

### 💡 비유
- **일반 방화벽 (아파트 정문 경비)**: 입주민인지 외부인인지만 확인하고 들여보낸다. 일단 들어온 사람이 집 안에서 물건을 부수든 훔치든 신경 쓰지 않는다.
- **DB 방화벽 (집 안의 보디가드)**: 거실에 서 있는 보디가드다. 가족이라도 "금고 열어!"라고 말하면 "지금은 금고 열 시간이 아닙니다"라고 막아선다. 손님이 "TV를 망치로 때려!"라고 명령하면 그 즉시 입을 막고 밖으로 끌어낸다. 말의 내용(SQL)을 알아듣고 행동을 통제하는 것이다.

### 등장 배경: 'Root' 권한의 위험성
DB 관리자(Root)는 신과 같은 권한을 갖는다. 마음만 먹으면 전체 테이블을 지울 수도 있고, 고객 정보를 빼낼 수도 있다. "관리자는 누가 관리하는가?"라는 질문에 대한 답으로, Root 계정이라도 특정 조건(야간 접속 금지, 대량 삭제 금지 등) 하에서는 명령을 거부할 수 있는 독립적인 감시 체계가 필요해졌다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### DB 방화벽의 구축 방식 (ASCII Diagram)

```ascii
[ 1. Sniffing 방식 (복제 모드) ]
  User ─────> [ Switch (Mirroring) ] ─────> Database
                    │
                    ▼ (패킷 복사본 감시)
              [ DB Firewall ]  <-- "사후 탐지 및 알람 위주"

------------------------------------------------------------------

[ 2. Inline/Proxy 방식 (차단 모드) ]
  User ─────> [ DB Firewall (Gateway) ] ─────> Database
                    │ (실시간 SQL 분석)
                    ▼
              "위험한 SQL 탐지 시" ──▶ [ DROP CONNECTION ]
                                     "강제 연결 종료 (실시간 차단)"

* 핵심 원리: 모든 SQL 패킷을 7계층(Application)까지 뜯어보고 
  화이트리스트/블랙리스트 정책과 대조함.
```

### 3대 핵심 제어 정책 (Policy)

#### ① 접속 제어 (Access Control)
- **대상**: IP 주소, MAC 주소, 접속 계정, 접속 시간, 사용하는 툴(예: Toat, SQL Developer).
- **예시**: "휴일에는 개발팀 IP에서 운영 DB 접속을 전면 차단하라."

#### ② 권한 제어 (Command Control)
- **대상**: SQL 명령어 종류 (`DROP`, `TRUNCATE`, `GRANT` 등).
- **예시**: "일반 사원은 `DELETE` 문을 사용할 수 없다."

#### ③ 데이터 제어 (Data Control)
- **대상**: 조회되는 데이터의 양, 특정 컬럼 접근.
- **예시**: "한 번의 `SELECT`로 10,000건 이상의 고객 정보를 조회하려 하면 세션을 즉시 끊어라."

## Ⅲ. 융합 비교 및 다각도 분석

### DB 접근 제어 vs DB 암호화 vs DB 감사

| 보안 기술 | 주요 역할 | 시점 | 특징 |
|:---:|:---|:---|:---|
| **접근 제어 (방화벽)**| **나쁜 놈 못 들어오게 막기** | **사전 (Pre)** | 실시간 차단, 정책 기반 통제 |
| **DB 암호화** | 데이터를 뺏겨도 못 읽게 하기 | 지속 (Always) | 저장/전송 중인 데이터 보호 |
| **DB 감사 (Logging)**| **누가 들어왔었는지 기록하기** | **사후 (Post)** | 증적 보존, 법적 근거 자료 |

*※ 완벽한 보안을 위해서는 이 3가지를 **'방암감(방화벽, 암호화, 감사)'** 세트로 구축하는 것이 정석이다.*

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 DB 보안 강화 전략)

**상황**: 금융 앱 서비스 중, 최근 유출된 관리자 계정을 이용해 새벽 시간대 해외 IP에서 대량의 고객 명단을 조회하려는 시도가 포착되었다.
**판단 및 DB 방화벽 대응 전략**:
1. **Proxy 모드 전환**: 감시만 하는 Sniffing 모드에서 실시간 차단이 가능한 **Inline/Proxy 모드**로 설정을 변경한다.
2. **지리적 차단 (Geo-IP)**: 한국 이외의 IP 대역은 DB 서버 접근을 원천 차단하는 정책을 즉시 활성화한다.
3. **Threshold(임계치) 설정**: 고객 테이블 조회 시 `LIMIT` 없이 대량으로 긁어가는 행위를 막기 위해, **"1분 내 동일 계정에서 5,000건 이상 조회 시 자동 차단"** 정책을 적용한다.
4. **우회 경로 차단**: 서버 자체에 직접 SSH로 붙어서 날리는 SQL을 막기 위해, 서버 내부에도 **Agent**를 설치하여 로컬 접속까지 완벽히 통제한다.
5. **결과**: 유출된 계정으로 로그인은 성공했더라도, 비정상적인 대량 조회 행위가 발생하자마자 0.1초 만에 세션이 강제 종료되어 데이터 유실을 막음.

### 안티패턴 및 고려사항
- **Proxy 방식의 성능 저하**: 모든 쿼리가 방화벽을 거쳐가므로 병목(Bottleneck)이 발생할 수 있다. 특히 초당 수십만 건의 짧은 쿼리가 발생하는 고성능 서비스에는 Proxy보다는 **Sniffing 방식과 하드웨어 ACL**을 적절히 혼용해야 한다.
- **예외 계정 남발**: 배치가 돌아가는 시스템 계정을 방화벽 정책에서 제외하면, 해커는 가장 먼저 그 예외 계정을 노린다. 시스템 계정이라도 **'수행 가능한 SQL'**을 화이트리스트로 묶어두는 깐깐함이 필요하다.

## Ⅴ. 미래 전망 및 결론

### 결론: 데이터의 최후 보루
DB 접근 제어는 데이터 보안 아키텍처의 꽃이다. 암호화가 데이터라는 '보물' 자체를 잠그는 것이라면, 접근 제어는 그 보물 창고로 가는 '길목'을 지키는 것이다. 

### 미래 전망
앞으로는 **'ID 기반의 제로 트러스트(Zero Trust)'** 가 DB 보안에도 적용될 것이다. 단순히 IP를 보는 것이 아니라, 접속자의 최근 행동 패턴을 분석하여 "이 사람은 평소와 다르게 행동하고 있으니 해커일 확률이 높다"라고 AI가 실시간으로 판단하는 **'사용자 행위 분석(UEBA) 결합형 DB 방화벽'**이 시장의 주류가 될 전망이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **DB 방화벽이 뭔가요?**: 중요한 보물 창고(DB) 문 앞에 서 있는 아주 깐깐한 '로봇 경비원'이에요.
2. **무슨 일을 하나요?**: 창고에 들어오려는 사람의 신분증을 검사하고, "금고를 망치로 부술 거야!" 같은 이상한 소리를 하면 바로 쫓아버려요.
3. **가장 좋은 점은?**: 나쁜 도둑이 몰래 열쇠를 훔쳐 와도, 경비원이 "너 밤에는 원래 안 오잖아!"라고 말하며 문을 절대 안 열어주기 때문에 보물을 안전하게 지킬 수 있답니다!
