# 371. SQL 인젝션 (SQL Injection) 공격 및 방어 수단 - 데이터베이스 침투의 고전이자 치명타

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 사용자의 입력값이 검증 없이 SQL 쿼리문의 일부로 직접 삽입되어, **공격자가 의도한 악의적인 SQL 구문이 데이터베이스 엔진에서 그대로 실행되게 만드는 코드 주입 공격**이다.
> 2. **위험**: 로그인 우회, 데이터 무단 조회(덤프), 데이터 변조 및 삭제, 심지어 OS 명령 실행까지 가능하여 기업의 신뢰도와 비즈니스 연속성에 치명적인 피해를 준다.
> 3. **방어**: 사용자 입력값을 데이터가 아닌 실행 코드로 오인하지 않도록 강제하는 **'Prepared Statement (바인드 변수)' 사용이 가장 완벽한 방어책**이며, 입력값 검증과 최소 권한 원칙을 병행해야 한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
SQL 인젝션은 웹 애플리케이션의 허술한 보안을 틈타 DB를 조종하는 공격이다. 개발자가 예상하지 못한 특수문자(`'`, `--`, `;`)를 입력창에 넣어 쿼리의 논리적 구조를 파괴한다. 20년 넘게 상위권을 차지하는 가장 흔하면서도 강력한 웹 취약점(OWASP Top 10)이다.

### 💡 비유
- **정상적인 상황 (정해진 양식 쓰기)**: 동사무소 직원이 "이름을 쓰세요"라고 종이를 준다. 나는 "홍길동"이라고 적는다. 직원은 내 이름을 장부(DB)에 적는다.
- **SQL 인젝션 (서류 조작)**: 이름 칸에 "홍길동', 그리고 내 벌금 100만 원 기록을 다 삭제해!"라고 적는다. 직원이 아무 생각 없이 이 문장 전체를 시스템에 그대로 입력하면, 시스템은 내 이름을 적은 뒤 **뒷부분의 명령까지 진짜 명령으로 착각해서 실행**해 버린다. 이름 칸에 글자(데이터)가 아닌 명령어(Code)를 섞어 넣은 것이다.

### 등장 배경: 'String Concatenation'의 유혹
과거 개발자들은 쿼리를 짤 때 ` "SELECT * FROM users WHERE id = '" + userInput + "'" ` 처럼 문자열을 단순히 더하는 방식을 썼다. 이 방식은 빠르고 편하지만, `userInput`에 어떤 악마 같은 코드가 들어올지 전혀 대비하지 못한 구조적 결함이었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### SQL 인젝션 공격 메커니즘 (ASCII Diagram)

```ascii
[ User Login Page ]
  ID: ' OR '1'='1
  PW: (아무거나)
          │
          ▼ [ Application Server ]
  String Query = "SELECT * FROM users WHERE user_id = '" + ID + "' AND pw = '" + PW + "';"
          │
          ▼ [ DB에 전달되는 최종 쿼리 ]
  "SELECT * FROM users WHERE user_id = '' OR '1'='1' --' AND pw = '...';"
                                       └─(A)─┘  └─(B)─┘
  (A) 항시 참(True) 조건: ID가 무엇이든 무조건 통과
  (B) 주석 처리(--): 뒷부분의 비밀번호 체크 로직을 통째로 무시
          │
          ▼ [ Database Engine ]
  "음, 조건이 참이네? 로그인 승인!" ──▶ [ Admin 권한 획득 ]
```

### 3대 주요 공격 유형

#### ① 인증 우회 (Authentication Bypass)
위 비유처럼 `' OR '1'='1` 구문을 사용하여 아이디와 비밀번호를 몰라도 관리자 계정으로 로그인하는 수법이다.

#### ② 데이터 탈취 (Union-based / Error-based)
`UNION SELECT` 구문을 이어 붙여서, 원래 쿼리 결과에 다른 테이블(예: 신용카드 정보)의 내용을 합쳐서 화면에 출력하게 만든다. 에러 메시지에 DB 구조를 노출하는 수법도 포함된다.

#### ③ 블라인드 인젝션 (Blind SQLi)
쿼리 결과가 화면에 보이지 않을 때 사용한다. `IF` 문과 `Sleep` 함수를 섞어 "데이터의 첫 글자가 'A'이면 5초간 쉬어라"라고 명령한 뒤, 응답 시간을 보고 데이터를 한 글자씩 알아내는 인내심 강한 공격이다.

## Ⅲ. 융합 비교 및 다각도 분석

### SQL 인젝션 방어 기술 비교

| 방어 기법 | 방식 | 효과성 | 특징 |
|:---:|:---|:---:|:---|
| **Prepared Statement**| **바인드 변수 활용** | **최고 (완벽)** | **입력값을 무조건 '문자열 데이터'로만 취급** |
| **입력값 검증** | 특수문자 필터링 | 보통 | `'`, `--` 등을 막지만 우회 기법이 많음 |
| **ORM 사용** | JPA, MyBatis 등 활용 | 높음 | 내부적으로 Prepared Statement를 사용함 |
| **최소 권한 원칙** | DB 계정 권한 제한 | 보통 | 공격받아도 전체 삭제(Drop) 등은 막을 수 있음 |
| **Stored Procedure** | DB 내 저장 프로시저 | 높음 | 정적으로 컴파일된 쿼리를 사용해 안전함 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시큐어 코딩 가이드)

**상황**: 외주 업체가 개발한 쇼핑몰 소스 코드를 보안 점검한 결과, 검색 기능에서 `like '%" + keyword + "%'` 형태의 동적 쿼리가 다수 발견되었다.
**판단 및 조치 전략**:
1. **즉시 수정**: 문자열 더하기 방식은 100% 취약하므로 즉시 **Prepared Statement** 방식으로 변경하도록 지시한다.
2. **바인드 변수 적용**: 
   - (AS-IS) `stmt.executeQuery("SELECT ... WHERE title like '%" + key + "%'");`
   - (TO-BE) `pstmt = conn.prepareStatement("SELECT ... WHERE title like ?");`
     `pstmt.setString(1, "%" + key + "%");`
3. **WAF(웹 방화벽) 연동**: 코드 수정이 완료될 때까지 임시로 **WAF에서 SQL 관련 특수 패턴을 차단**하는 룰을 적용한다.
4. **결과**: 공격자가 검색창에 아무리 복잡한 SQL 구문을 넣어도, DB는 이를 실행 명령이 아닌 단순한 '검색 단어'로만 인식하여 공격을 원천 무력화함.

### 안티패턴 및 고려사항
- **블랙리스트 필터링의 한계**: `select`, `union` 같은 단어만 막는 것은 안티패턴이다. 공격자는 `sEleCt`, `un/**/ion` 처럼 대소문자를 섞거나 주석을 넣어 필터링을 쉽게 우회한다. 막으려고 애쓰지 말고, **데이터와 명령을 분리하는 구조(DIP)**를 선택하는 것이 정답이다.

## Ⅴ. 미래 전망 및 결론

### 결론: 기본이 최고의 보안이다
SQL 인젝션은 기술이 부족해서 생기는 공격이라기보다, '개발자의 부주의'에서 생기는 공격이다. 라이브러리가 제공하는 기본 가이드(Prepared Statement)만 잘 따라도 99% 막을 수 있는 '인재(人災)'에 가깝다.

### 미래 전망
앞으로는 인공지능(AI) 기반의 **SAST(정적 분석 도구)**가 코드를 짤 때 실시간으로 SQL 인젝션 취약점을 찾아내어 경고를 줄 것이다. 또한, 클라우드 네이티브 환경의 **[DB 접근 제어](./370_db_firewall.md)** 솔루션들이 머신러닝을 통해 평소와 다른 기괴한 패턴의 SQL이 들어오면 자동으로 차단하는 '지능형 탐지'가 보편화되어 보안의 틈새를 메울 전망이다.

## 👶 어린이를 위한 3줄 비유 설명
1. **SQL 인젝션이 뭔가요?**: 도서관 대출증 이름 칸에 "내 연체 기록 다 지워줘!"라고 적어서 사서 선생님을 속이는 장난꾸러기 같은 공격이에요.
2. **왜 속나요?**: 사서 선생님이 이름만 읽어야 하는데, 뒤에 적힌 "기록 지워줘"라는 말까지 진짜 심부름인 줄 착각하고 실행해 버리기 때문이에요.
3. **어떻게 막나요?**: 이름 칸에는 오직 '이름'만 쓸 수 있게 아주 좁은 칸을 만들고, 칸 밖으로 삐져나온 글자는 아예 읽지 않기로 약속(Prepared Statement)하면 된답니다!
