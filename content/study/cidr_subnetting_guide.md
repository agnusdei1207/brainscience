# CIDR(Classless Inter-Domain Routing) 기반 서브넷 설계 가이드

CIDR(Classless Inter-Domain Routing) 기준으로 서브넷을 “잘게 쪼개는” 핵심은 프리픽스(prefix) 길이를 늘리는 것입니다.

---

## 1. 핵심 개념
* **CIDR 표기**: `IP/프리픽스` (예: `192.168.1.0/24`)
* **의미**:
    * **앞 비트**: 네트워크 영역
    * **뒤 비트**: 호스트 영역

👉 **프리픽스를 늘리면**: 네트워크 수 ↑ / 각 서브넷 크기 ↓

---

## 2. 기본 공식
* **호스트 수**: `2^(32 - prefix) - 2` (네트워크 주소와 브로드캐스트 주소 제외)
* **서브넷 수**: `2^(늘어난 비트 수)`

---

## 3. 단계별 서브넷 쪼개기 예시

원본 네트워크: `192.168.1.0/24` (총 256개 IP, 사용 가능 254개)

### 1단계: /25로 쪼개기 (1비트 추가)
→ **2개의 서브넷**
* `192.168.1.0/25` (0 ~ 127)
* `192.168.1.128/25` (128 ~ 255)
* *각 서브넷*: 128 IP (사용 가능 126)

### 2단계: /26로 쪼개기 (2비트 추가)
→ **4개의 서브넷**
* `192.168.1.0/26`
* `192.168.1.64/26`
* `192.168.1.128/26`
* `192.168.1.192/26`
* *각 서브넷*: 64 IP (사용 가능 62)

### 3단계: /27로 쪼개기 (3비트 추가)
→ **8개의 서브넷**
* 각 32 IP (사용 가능 30)

### 4단계: /28로 쪼개기 (4비트 추가)
→ **16개의 서브넷**
* 각 16 IP (사용 가능 14)

---

## 4. 실전! 빠르게 쪼개는 방법 (블록 크기 계산)

**블록 크기 = 256 - 서브넷 마스크의 마지막 옥텟**

| CIDR | 서브넷 마스크 | 블록 크기 |
|:---:|:---|:---:|
| **/25** | 255.255.255.128 | 128 |
| **/26** | 255.255.255.192 | 64 |
| **/27** | 255.255.255.224 | 32 |
| **/28** | 255.255.255.240 | 16 |

👉 **시작 주소는 블록 크기 단위로 증가합니다.** (예: /26인 경우 0, 64, 128, 192)

---

## 5. 인프라 설계 실무 팁

### 1) 낭비 줄이기
* **서버용**: `/28` 또는 `/29` (소규모 클러스터)
* **대규모 네트워크**: `/24`, `/23` 이상

### 2) 클라우드(VPC/AWS) 스타일 설계 예시
* **Public Subnet**: `/26` (외부 노출 서비스)
* **Private Subnet**: `/24` (애플리케이션 서버 등 대규모)
* **DB Subnet**: `/28` (소수 인스턴스)

### 3) 정렬 기준 유지
* 항상 **2의 배수 경계**를 유지해야 합니다.
* 경계를 무시하고 쪼개면 라우팅 테이블이 복잡해지고 관리가 어려워집니다.

---

## 6. 핵심 요약
* `/24` → `/25` → `/26` → `/27` → `/28`
* **프리픽스 +1 = 서브넷 개수 2배**
* **블록 크기**로 시작 주소를 빠르게 계산
* 실무에서는 항상 **"필요한 IP 수"**를 기준으로 적절한 프리픽스를 선택함
