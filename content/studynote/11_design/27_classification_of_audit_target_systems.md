# 27. 감리 대상 시스템의 분류

#### 핵심 인사이트 (3줄 요약)
> 1. **분류 필요성**: 감리 대상 시스템은 구축 방식(시스템 통합 SI, 단위 개발, 유지보수), 서비스 형태(온프레미스, 클라우드, SaaS), 적용 산업(금융, 의료, 공공)에 따라 감리 범위와 방법론이 근본적으로 달라지므로 체계적 분류가 선행되어야 한다.
> 2. **4대 유형**: 감리 대상 시스템은 크게 신규 구축(SI), 유지보수(SM), 데이터 이전(마이그레이션), 클라우드 전환으로 분류되며, 각 유형은 고유한 감리 포인트와 위험 프로파일을 가진다.
> 3. **중요도 기준**: 시스템의 처리 정보의 중요도(기밀성, 무결성, 가용성)에 따라 감리 수준이 차등 적용되며, 이는 NIST의 정보 보호 등급 체계와도 연계된다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

감리 대상 시스템을 분류하는 이유는 모든 정보시스템에 동일한 감리 잣대를 적용하는 것이 불가능하기 때문이다. 수백억 원 규모의 금융 실시간 결제 시스템과 수천만 원 규모의 사내 게시판 시스템을 동일한 깊이로 감리하는 것은 자원 낭비이며, 반대로 중요한 시스템을 얕게 감리하면 정작 위험을 놓치게 된다. 따라서 감리전에는 반드시 대상 시스템의 분류 분석을 수행하여 감리의 규모, 깊이, 방법을 전략적으로 결정해야 한다.

시스템 분류는 크게 세 가지 축으로 수행된다. **첫째, 구축 방식 기준**으로 신규 시스템 구축(SI), 유지보수(SM), 마이그레이션, 클라우드 전환으로 구분한다. **둘째, 서비스 형태 기준**으로 온프레미스(On-premise), 인프라 서비스(IaaS), 플랫폼 서비스(PaaS), 소프트웨어 서비스(SaaS)로 나눈다. **셋째, 산업 분야 기준**으로 공공部门, 금융部门, 의료部门, 제조部門 등으로 분류하며, 각 산업 분야마다 특수한 규제 요건이 존재한다.

다음 다이어그램은 감리 대상 시스템 분류의 3축 체계를 보여준다.

```text
┌─────────────────────────────────────────────────────────────┐
│ [감리 대상 시스템 분류 3축 체계]                              │
│                                                             │
│  축 1: 구축 방식                                             │
│  ├─ SI (시스템 통합 구축) ────── 대규모, 다수 사업자, 복잡  │
│  ├─ SM (유지보수)          ────── 소규모, 기존 시스템 수정    │
│  ├─ 마이그레이션           ────── 데이터 이전, 플랫폼 전환   │
│  └─ 클라우드 전환          ────── 온프레미스 → 클라우드 이동 │
│                                                             │
│  축 2: 서비스 형태                                             │
│  ├─ 온프레미스            ────── 직접 보유 인프라            │
│  ├─ IaaS                 ────── VM, 스토리지, 네트워크 대여 │
│  ├─ PaaS                 ────── 개발 플랫폼 대여             │
│  └─ SaaS                 ────── 완제품 소프트웨어订阅        │
│                                                             │
│  축 3: 산업 분야                                             │
│  ├─ 공공부문              ────── 전자정부법, 예산편성 시스템 │
│  ├─ 금융부문              ────── 금융보안원 기준, 지급결제   │
│  ├─ 의료부문              ────── 개인정보보호, HIPPA 등    │
│  └─ 제조/민간             ────── 지적재산권, 영업비밀 등    │
└─────────────────────────────────────────────────────────────┘
```

이 분류 체계를 기반으로 감리팀은 프로젝트의 특성에 맞는 감리 계획서를 작성한다. 예를 들어, 금융기관의 지급 결제 시스템(공공+SI+IaaS/PaaS 혼합)은 가장 높은 감리 수준이 요구되고, 사내イントラネット維持保全省(SM+온프레미스)은 상대적으로 경감된 감리 수준이 적용될 수 있다.

📢 **섹션 요약 비유**: 감리 대상 시스템 분류는 의사의 진료 과目分類와 같다. 외과 환자와 내과 환자를 구분하듯이, 심장 수술이 필요한 환자(금융 핵심 시스템)와 감기 환자를 같은 수술室里에 넣지 않는다. 적절한 분류를 통해 각 환자에게 필요한 처치를 전략적으로 결정한다.

---

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**2.1 구축 방식별 시스템 분류와 감리 특성**

**신규 시스템 구축 (SI, 시스템 통합)**: SI 사업은 요구정의에서 설계, 구현, 테스트, 배포까지全過程을cover하는 종합 프로젝트이다. 감리에서는 특히 사업 관리 영역의 진척 관리, 응용 시스템 영역의 아키텍처 설계 품질, 데이터베이스 영역의 정규화 적절성에 초점을 맞춘다. SI 사업의 감리 특징은 단계별 감리(3단계 또는 2단계)가 적용되어 각 마일스톤에서 체계적으로 품질이 검증된다는 점이다.

**유지보수 (SM, System Maintenance)**: SM 사업은 기존 운영 시스템의 버그修正, 기능 추가, 환경 변화 대응을主要内容으로 한다. 유지보수 감리에서는 변경管理(Change Management)의 적정성이 핵심이다. 특히紧急修改emergency fix)의 경우 정상 감리 절차를 생략하고 반영되는 경우가 많으므로, 사후Retroactively 검토의 중요성이 높아진다. 변경 요율(change rate), 평균修復時間(MTTR), 그리고 유지보수 후 발생 신규 결함율이 주요 감리 지표가 된다.

**데이터 마이그레이션**: 데이터 이전 사업은 기존 시스템의 데이터를 새 플랫폼으로 이동하는 工作이다. 감리에서는 데이터 무결성(Data Integrity)이 가장 핵심적인 검증 항목이다. 이전 전 데이터 건수와 이전 후 데이터 건수가 일치하는지, 체크섬(Checksum) 검증이 수행되었는지, 이전 과정에서 데이터 손실이나 변조가 발생하지 않았는지를 검증해야 한다.

```text
[데이터 마이그레이션 감리 핵심 체크포인트]
┌─────────────────────────────────────────────────────────┐
│  단계          │  감리 포인트                 │  증거 유형  │
│  ────────────────────────────────────────────────────────  │
│  추출 (Extract)│  원본 데이터 추출 권한 확인    │  로그 추출  │
│  정제 (Transform)│  데이터 정제 규칙 적용 여부  │  정제ルール │
│  적재 (Load)  │  적재 후 건수/체크섬 일치     │  비교 리포트│
│  검증 (Verify) │  샘플 데이터 무결성 테스트     │  샘플 검증  │
│  전환 (Cutover)│  가동 중 데이터 동기화 완료    │  동기화 로그│
└─────────────────────────────────────────────────────────┘
```

**클라우드 전환 (Cloud Migration)**: 온프레미스에서 클라우드 환경으로 시스템을 이전하는 사업이다. 감리에서는 기존에 직접 관리하던 보안, 백업, 장애 복구 체계가 클라우드 환경에서도同等 이상으로 구현되었는지를重点적으로 검증한다. 특히 멀티테넌시(Multi-tenancy) 환경에서의 데이터 격리, SLA(서비스 수준 협약) 충족 여부, 종속성(Lock-in) 위험도评估가 중요하다.

**2.2 서비스 형태별 분류**

클라우드 서비스 형태(IaaS, PaaS, SaaS)에 따라 감리 Responsibility이 달라진다. IaaS에서는 이용자가 OS 이상의 계층에 대한 통제를 수행하므로 보안 설정 감리가 광범위하고, SaaS에서는 제공자가全-stack을管理하므로 이용자는 설정 가능한 항목(액세스 제어, 데이터エクス포트等)에 대한 감리만 수행한다.

**2.3 산업 분야별 분류**

금융 부문 정보시스템은 금융보안원(FSOC)의 보호 조항과 지급결제 시스템等 핵심 금융 인프라에 대한 특수 규제가 적용된다. 공공 부문은 전자정부법 제57조에 따른 의무 감리와 예산편성심사 기준이 적용된다. 의료 부문은 개인정보보호법 및 의료법의 특별 규정에 따라 건강정보의 비의도적 노출防止에 감리가 집중된다.

📢 **섹션 요약 비유**: 시스템 분류는 도서관의 도서 분류와 같다. 소설, 학술서적, 잡지가 각각 다른서가 시스템으로 분류되듯이, 시스템도 그 유형에 따라 적절한 감리 도구를 선택하고 관리 기준을 달리 적용해야 한다. 모든 책을 동일하게 취급하면 찾는 사람이 difficulty 겪고, 모든 시스템을 동일하게 감리하면 자원이 낭비되거나 위험이 놓친다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

감리 대상 시스템의 분류는 단순한 종류 구분 이상의 의미를 가진다. 분류 결과에 따라 감리 방법론, 인력 구성, 일정, 비용이 모두 달라지므로, 프로젝트 시작 시 정확한 분류가 필수적이다. 여기서는 주요 분류 유형 간의 감리 방법론 차이를 비교分析한다.

**SI vs SM 감리의 핵심 차이**: SI 사업은 '창조적 산출물'의 감리에 초점이 맞춰진다. 요구사항이 설계로, 설계가 코드로 전환되는 과정에서 의미의 변화나 누락이 발생하지 않았는지를 추적한다. 반면 SM 사업은 '기존 것의 수정'의 감리에 집중한다. 변경 사항이 기존 시스템에予期せぬ 부작용(side effect)을 일으키지 않았는지를 확인하는 데 감리 자원이 집중된다.

```text
┌──────────────── SI vs SM 감리 핵심 비교 ─────────────────┐
│                                                         │
│  항목           │  SI (신규 구축)   │  SM (유지보수)    │
│  ──────────────────────────────────────────────────────  │
│  감리 초점      │  요구↔설계↔구현    │  변경↔영향↔검증   │
│                 │  의미 전환 적정성   │  역효과 최소화    │
│  핵심 감리 영역 │  아키텍처 설계     │  변경 관리 절차    │
│                 │  산출물 완결성     │  회귀 테스트      │
│  위험 특성      │  설계 결함        │  연쇄 결함       │
│                 │  (치명적, 발견 늦음) │  (점진적, 다수)  │
│  감리 시점      │  단계별 마일스톤   │  변경 요청 단위   │
│  증거 유형      │  설계 문서, UML   │  변경 이력, 테스팅 │
│                 │  산출물 대조       │  결과, 환경 설정  │
└─────────────────────────────────────────────────────────┘
```

**온프레미스 vs 클라우드 감리의 책임 범위**: 온프레미스 환경에서는 HW, 네트워크, OS, 미들웨어, SW的全層에 대한 통제 권한이 이용자에게 있으므로 감리 범위도全層을cover한다. 클라우드 환경에서는 서비스 모델(IaaS/PaaS/SaaS)에 따라 제공자와 이용자 간의 책임 분계점(Shared Responsibility Model)이 설정되며, 감리는 이용자 책임 범위인 '고객 통제 영역(Customer Control Plane)'에 집중한다.

**공공 vs 민간 감리의 법制度差**: 공공 부문 감리는 전자정부법, 국가예산법, 정보시스템감리기준(행정안전부 고시)에 의해 의무적으로 수행되며, 감리 결과가 국고 보조금 사용 내역과 연계되어 법적 구속력이 강하다. 민간 부문 감리는 계약상 합의에 의한、商行為的 성격이 강하며, 감리 범위와 방법은 당사자 간 협의를 통해 결정된다. 다만, 금융, 의료 등特殊 산업에서는 산업별 규제기관의 감리 기준이 추가 적용된다.

이 분류들의交叉点是 감리 전략 수립에서 특히 중요하다. 예를 들어, 공공 부문의 금융 결제 시스템(공공+SI+클라우드/IaaS)은 세 가지 분류 축이 겹치므로 가장 높은 수준의 감리 요건이 적용된다. 반면, 민간中小企業の 사내 공유 드라이브 유지보수(민간+SM+온프레미스)는 상대적으로 낮은 감리 수준으로 운영될 수 있다.

📢 **섹션 요약 비유**: 시스템 분류는 음식점의 메뉴 구性与 같다. 한 식당에서 한식, 중식, 양식, 일식을 모두 만들 수는 없듯이, 시스템도 그 유형에 맞는 요리사(감리 인력)와 조리법(감리 방법론)을 선택해야 한다. 한식 전문 점장에 중식의 맛을 평가해 달라고 하면 적절한 판단이 나오지 않는다.

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무에서 감리 대상 시스템 분류는 단순히 문서로 끝나지 않고, 실제 감리 계획과 인력 배치에 직결되는 전략적 판단이다. 여기서는 분류 결과를 감리 실무에 적용하는 핵심 판단 기준을 다룬다.

**4.1 시스템 중요도 등급 결정**

시스템 중요도 등급은 시스템이 처리하는 정보의 중요도와 서비스 중단 시 영향 범위를 기준으로 결정된다. 금융시스템의 지급결제, 의료시스템의 환자 데이터, 공공시스템의 国政情報 등은 가장 높은 등급(等급 1)에 분류되며, 이等级의 시스템에는 반드시 3단계 감리와 상주 감리가 적용된다.

아래 의사결정 트리는 시스템 중요도 등급을 결정하는 흐름을 보여준다.

```text
[시스템 중요도 등급 결정 흐름]
   │
   ├──▶ 서비스 중단 시 인명/재산 피해 발생?
   │    ├── YES ───▶ [등급 1: 치명적] ──▶ 3단계+상주+심화보안
   │    │
   │    └── NO
   │         │
   │         ├──▶ 수만 명 이상 사용자 영향?
   │         │    ├── YES ───▶ [등급 2: 심각] ──▶ 2단계+설계/종료
   │         │
   │         └──▶ NO
   │              │
   │              ├──▶ 일상 업무 필수 시스템?
   │              │    ├── YES ───▶ [등급 3: 보통] ──▶ 종료 감리
   │              │
   │              └──▶ NO ───▶ [등급 4: 사내] ──▶ 감리 면제/선택
```

**4.2 분류에 따른 감리 일정 계획**

등급 1 시스템은 감리 기간이 길고, 중간 보고 회수를 여러 번设定了. 반면 등급 4 시스템은 종료 시 단순 감리 보고서로 마무리될 수 있다. 이 일정은 프로젝트 전체 일정과 조율되어야 하며, 특히 단계 감리 시점(설계 완료, 개발 완료 등)과 감리 일정이 중첩되지 않도록 마스터 스케줄을管理한다.

**4.3 분류 결과의 comunicação**

감리 분류 결과는 착수 회의(Kick-off Meeting)에서 피감리인에게 명확히 설명되어야 한다. 피감리인이 감리 범위와 수준에 대해 불만을 품는 경우, 분류 근거를 객관적 기준(정보 보호 등급, 예산 규모, 법적 요건 등)에 기반하여 합리적으로 설명해야 한다. 이때 감리 계획서와 위험 평가 보고서가 분류 결정의根拠として提示된다.

분류 결과는 감리보고서에서도重要な meaning를 갖는다. 동일 등급 내 시스템끼리 비교 분석이 가능하므로, 감사 기관이나 규제 기관에서는 등급별 평균 감리 지적 사항发生率을 산출하여将来的な 감리 기준修订에 활용한다.

📢 **섹션 요약 비유**: 시스템 분류는 집의保安等급화와 같다. 백화점 골동품 매장(등급 1)에는 경비원과 금고, 카메라까지 갖춰야 하고, 동네 문구점(등급 4)에는 자물쇠만 있으면 된다. 같은 '가게'라도 취급하는 물건의 가치에 따라保安 수준을 적절히 차등 적용해야 자원이 낭비되지 않는다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

체계적인 시스템 분류는 감리 자원 배분의 효율화를 통해 감리 품질과 비용 효율성을 동시에 달성하는 핵심 기반이다. 분류 체계가 잘 마련되어 있으면, 신규 감리 인력이라도 분류 결과만 확인하면 즉각 적절한 감리 전략을 수립할 수 있다.

분류 체계 도입의 기대효과는 크게 세 가지로 요약된다. **첫째, 감리 품질 향상**으로, 시스템 특성에 맞는 감리 방법론이 적용되므로 중요한 시스템에서 결함을 놓칠 확률이 낮아진다. **둘째, 비용 절감**으로, 상대적으로 덜 중요한 시스템에 과도한 감리 자원이 투입되는 것이 방지된다. **셋째, 의사결정 속도 향상**으로, 시스템 등급이 사전에 설정되어 있으면 감리 범위 논쟁이 줄어들고 프로젝트 진행이滑らかに 된다.

미래에는 AI가 시스템의処理량, 사용자 수,数据类型등의 参数를 분석하여 자동으로 시스템 등급을 추천하는 분류自动化ツールが開発될 전망이다. 또한 블록체인 기반의 분류 결과 관리系统를 통해同一 시스템에 대한 반복적인 분류 평가作业를省略できる可能性がある.

```text
[시스템 분류 기대효과]
┌─────────────────────────────────────────────────────────┐
│  구분          │  도입 전 (분류 없음)  │  도입 후 (분류 체계) │
│  ──────────────────────────────────────────────────────  │
│  감리 자원 배분 │  균일 배분            │  위험도 비례 배분    │
│  감리品質     │  중요한 시스템 결함 놓힘 │  체계적 全coverage   │
│  일정 계획    │  일괄적 일정 적용      │ 等级별 차등화        │
│  분쟁 위험    │  범위 놓고 잦은 분쟁   │  사전 합의로 분쟁预防 │
│ 監査証拠管理  │  均一存档             │ 等级별 차등 archiving │
└─────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 시스템 분류는 옛날 度量衡(길이와 무게의 단위)과 같다. 미터와 센티미터가 없으면 모든 길이를 같은 자로 재야 하듯, 분류 체계 없이는 모든 시스템을同等하게 감리해야 되어 큰 시스템은 부족하고, 작은 시스템에는 과잉투자가 발생한다.標準화된 분류 체계こそ가 자원 배분의公正合理的基础이다.

### 👶 어린이를 위한 3줄 비유 설명
1. **개념**: 우리 학교에는 큰 운동장(핵심 시스템), 보통 체육관(중요 시스템), 작은 자료실(사소한 시스템)이 있는데, 각각 지키는 방법을 다르도록 하고 있는 거예요.
2. **원리**: 운동장에는警備 아저씨가 상시 지키고, 체육관에는 경비원과 카메라, 자료실에는 그냥 잠금장치만 있는 것처럼, 시스템도 크기와 중요도에 따라保安 수준을 다르게 설정하는 거예요.
3. **효과**: 이렇게 하면 경비원을 다 배치하면 비용이 너무 많이 드는데, 중요한 곳에는 꼭 필요한保安人员이 배치되어 있어 위험을 막을 수 있답니다!
