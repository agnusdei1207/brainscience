# 32. SW 수분 확인 및 분석

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: SW 수분(규모 산정)은 정보화 사업의 비용과 기간을 현실적으로 추정하고, 진척 상태를 객관적으로 측정하기 위한 핵심 기반 기술이다. 기능점수(FP)와 코드 라인수(LOC)가 대표적 규모 산정 방법이다.
> 2. **핵심 기능**: 수분 산정은 사업 관리의Baseline设定, 비용 협상의 근거, 진척율 측정 기준, 감리에서의 객관적 증거로 활용된다.
> 3. **실무 핵심**: 실제 감리에서는 사업자가 제시한 수분数据和实际 выполнен工作量 사이의GAP分析이 가장 중요한 검증 포인트이다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

SW 수분(Software Sizing)이란 소프트웨어 개발 프로젝트의 규모를定量化하는 활동이다. 건물 건축에서 면적(㎡)이 핵심 수분 단위인 것처럼, 소프트웨어 개발에서도工作量을客观적으로 측정하는 단위가 필요하다. 수분이 명확해야 비용과 기간을 현실적으로 추정할 수 있고, 발주자와 사업자 간의 비용 협상의 근거가 마련된다.

정보화 사업에서 수분이 필요한 이유는 네 가지 측면에서 설명된다. **첫째, 비용 결정의客観적 근거**로, 수분 없이는报价根拠가 없으므로 사업자 제시 비용의 적정성 여부를 판단할 수 없다. **둘째, 진척율 측정**으로, 프로젝트가 얼마나 진행되었는지를定量적으로 판단하려면已完成工作量을 수분 단위로 측정해야 한다. **셋째, 사업자との契約 단위**로, 기능점수(FP) 기반 정산은 공공 정보화 사업의 표준 계약 방식으로 자리잡았다. **넷째, 감리에서의 객관적 증거**로, 감리인은 수분 데이터를활용하여 사업자의工作报告의 신뢰성을検証한다.

다음 다이어그램은 SW 수분이 프로젝트 전 과정에서 어떻게 활용되는지를 보여준다.

```text
┌─────────────────────────────────────────────────────────────┐
│ [SW 수분의生命周期 활용]                                     │
│                                                             │
│  ① 요구정의 단계                                            │
│     └─ RFP에 요구사항의 功能수 (미확정) 명시               │
│                                                             │
│  ② 제안 단계                                                │
│     └─ 사업자별 功能点수(FP) 기반报价 차이 비교             │
│                                                             │
│  ③ 계약 단계                                                │
│     └─ 확정된 功能点수(FP) × 단가 = 계약 금액              │
│                                                             │
│  ④ 개발 단계                                                │
│     └─ 제공된 功能点수(PV) 대비 완료 기능점수(EV) 측정     │
│                                                             │
│  ⑤ 종료/정산 단계                                           │
│     └─ 최종 实점수(AF) × 단가 = 최종 정산 금액             │
│                          │                                  │
│                          ↓                                  │
│              [발주자-사업자 간 금전적 분쟁防止]             │
└─────────────────────────────────────────────────────────────┘
```

이 중에서도 특히 중요한 것은 종료 단계에서의最終功能점수 검증이다. 계약 단계에서 확정된 수분과 실제 개발 후 완성된 수분 사이에GAP이 발생하면, 그 원가가どちらの責任かを분석하고 적절한 정산처리를 해야 하기 때문이다.

📢 **섹션 요약 비유**: SW 수분은 요리와.recipe의 재료 목록과 같다.番茄 3개,たまご 2개, 소금少许와 같이 재료의 양을 특정해야 비싼 지점인지 알 수 있듯이, 소프트웨어도 기능의 수와 복잡도를定量화해야 비용이 적절한지 판단할 수 있다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**2.1 기능점수 (Function Point, FP)**

기능점수(FP)는 1979년 앨런 얼브레히트(Allen Albrecht)가 개발한 소프트웨어 규모 산정 방법으로, 사용자에게 제공하는 기능의 수와 복잡도를 기준으로 규모를 측정한다. 기능점수는 개발 기술이나 언어에 독립적이므로, 요구사항 정의 단계에서早期 산정이 가능하다는 것이最大 장점이다.

기능점수는 5가지 데이터 기능(DT)과 5가지 트랜잭션 기능(ILF, EIF, EI, EO, EQ)으로 구성된다.

```text
[기능점수 구성 요소]
┌─────────────────────────────────────────────────────────┐
│  데이터 기능 (Data Functions)                             │
│     ├─ ILF (Internal Logical File)                      │
│     │    └─ 내부逻辑적 파일: 사용자가 유지관리하는 데이터   │
│     └─ EIF (External Interface File)                     │
│          └─ 외부接口 파일: 다른 시스템에서 참조만 하는 데이터│
│                                                          │
│  트랜잭션 기능 (Transaction Functions)                    │
│     ├─ EI (External Input)     ─ 입력: 외부からのデータ入力  │
│     ├─ EO (External Output)    ─ 출력: 가공/계산 후 내보냄 │
│     └─ EQ (External Inquiry)  ─ 조회: 가공 없이 단순 검색  │
└─────────────────────────────────────────────────────────┘
```

각 기능 요소는 복잡도에 따라 baixa(3~5 FP), média(6~10 FP), alta(10~15 FP)의 3단계로 분류되며, 복잡도高低는 데이터 항목수(DET)와 레코드 수(RET)에 의해 결정된다.

功能点수 산정의 一般적 순서는 다음과 같다.

```text
[기능점수 산정 절차]
   │
   ├──▶ 1단계: 기능 식별
   │    └─ DT(데이터 기능)와 TT(트랜잭션 기능) 요소 식별
   │
   ├──▶ 2단계: 복잡도 결정
   │    └─ DET(데이터 항목 수), RET(레코드 수) 기준 분류
   │
   ├──▶ 3단계: 미수정 요정(Unadjusted FP) 산출
   │    └─ 각 기능 요소의 수 × 복잡도 등급별 FP 합산
   │
   └──▶ 4단계: 조정된 기능점수(Adjusted FP) 산출
        └─ 미수정 FP × 조정 계수 (VAF: Value Adjustment Factor)
           VAF = 0.65 + 0.01 × Σ(Technical Complexity Factor)
```

调整 계수(VAF)는 시스템 특성(처리 효율성, 분산 처리, 사용 편의성等 14개 항목)에 따라 0.65~1.35 사이의 값을 가진다.

**2.2 코드 라인수 (Lines of Code, LOC)**

코드 라인수(LOC)는 실제 작성된 소스코드의 줄 수를 세서 규모를 측정하는 전통적 방법이다. 기능점수와 달리 개발完成后에测定 가능하므로,プロジェクトの実績管理와 생산성 분석에 주로 활용된다.

LOC 산정 시 기준을명확히 해야 한다. 순수한 문(statement)의 수, 주석 제외 여부, 공백 줄 포함 여부 등을 사전에 정의해야 사업자 간 비교가 가능하다.业界에서는 통상 '실제 코드 라인수(Non-comment, Non-blank)'를 기준으로 한다.

```text
[LOC vs FP 비교]
┌─────────────────────────────────────────────────────────┐
│  항목           │  기능점수 (FP)    │  코드 라인수 (LOC)   │
│  ─────────────────────────────────────────────────────  │
│  측정 시점       │ 요구사항 단계에서 가능│ 개발 완료 후 측정    │
│  기술 독립성     │ 독립적            │ 언어/프레임워크 종속   │
│  복잡도 반영     │ 명시적 (3단계)    │ 암묵적 (경로 복잡도) │
│  주 활용처       │ 비용 협상, 계약   │ 생산성 분석, 예측    │
│  장점          │Early 산정 가능,   │ 정밀한 실제 측정     │
│                │ 언어 독립적        │                      │
│  단점          │ 주관성 개입 가능,  │ 개발 후才知道,       │
│                │ DT/TTr 구분 곤란   │ 언어별 차이大       │
└─────────────────────────────────────────────────────────┘
```

**2.3 과업대비표 (Task Traceability Matrix, TTM)**

과업대비표는 발주자의 요구사항이 사업자의 과업(산출물)으로 얼마나 추적 가능한지를 matrix 형태로 표현한 문서이다. 감리에서는 이 TTM을활용하여 要求漏れや追加의44한 要求가 없는지, 과업이 제대로 수행되었는지를検証한다.

TTM의 핵심 기능은 요구사항과 산출물 간의 双方向 추적성(Bidirectional Traceability) 확보이다. 요구사항에서 산출물로의 정방향 추적(요구사항이 모두 구현되었는가?)과, 산출물에서 요구사항으로의 역방향 추적(산출물이 요구사항에 기반한 것인가?)을 모두 확인해야 한다.

```text
[과업대비표 (TTM) 예시]
┌─────────────────────────────────────────────────────────┐
│ 요구사항ID │ 요구사항 내용    │ 산출물ID │ 산출물명      │ 이행율 │
│ ──────────────────────────────────────────────────────── │
│ REQ-001   │ 회원 가입 기능    │ DEL-001  │ 회원 가입 설계서│  100% │
│ REQ-002   │ 결제 처리 기능    │ DEL-002  │ 결제 처리 설계서│  100% │
│ REQ-002   │ 결제 처리 기능    │ DEL-003  │ 결제 처리 코드   │   80% │←
│ REQ-003   │ 환불 처리 기능    │ (없음)   │ -              │    0% │←
│ REQ-004   │ 배송 조회 기능    │ DEL-004  │ 배송 조회 설계서│  100% │
└─────────────────────────────────────────────────────────┘
     ↑                                              ↑
     └──── 요구 → 산출물 (정방향 추적)            │
     └──── 산출물 → 요구 (역방향 추적) ────┘
```

이 예시에서 REQ-003(환불 처리)은 산출물이 존재하지 않아 이행율 0%이고, REQ-002는 설계서는 100%이지만 코드가 80%이므로部分 이행 상태이다. 이러한 추적 분석을 통해漏과잉을同時に把握할 수 있다.

📢 **섹션 요약 비유**: SW 수분은 building 면적 산출과 같다. 공사가 진행중인 건물에서 시공된 벽면의 면적을 정밀 측정하면 현재 공정 완료율을 알 수 있듯이, 소프트웨어에서도功能점수나 LOC를 측정하여完成度를客观적으로 산출할 수 있다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

SW 수분은 프로젝트 관리, 감리, 비용 협상 등 여러 영역에서 활용되므로, 그 활용 맥락에 따른 분석이 필요하다.

**진척율 측정에서의 수분 활용**: 프로젝트 진척율 측정에서 가장 科学적인 방법은已完成工作量을 수분 단위로 측정하여 계획 대비 比를 산출하는 것이다. 이를 획득 가치 관리(Earned Value Management, EVM)라고 한다.

```text
[획득 가치 관리 (EVM) 핵심 지표]
┌─────────────────────────────────────────────────────────┐
│  PV (Planned Value)    = 계획된工作量 (계획된 수분 × 단가)  │
│  EV (Earned Value)     = 완료工作量 (실제 완료 수분 × 단가)  │
│  AC (Actual Cost)       = 실제 소요 비용                         │
│                                                          │
│  진척율 = EV / PV × 100 (%)                              │
│  비용 효율 = EV / AC (1 이상이면 비용 효율적)               │
│  일정 효율 = EV / (PV × 일정 비율)                          │
└─────────────────────────────────────────────────────────┘
```

예를 들어, 전체 100 FP 사업에서 50 FP를 계획했지만 40 FP만 완료하고 45 FP 비용이 들었다면, 진척율은 40%, 비용 효율은 0.89(=40/45)로 예산을 초과한进度이다.

**감리에서의 수분 검증**: 감리에서 수분 검증은主に 다음 3가지를 확인한다. **첫째, 수분 산정 근거의 적정성**으로, FP 또는 LOC 산정 시 적용한 기준과 복잡도 결정이合理的인지를 확인한다. **둘째, 진척 보고와의 일치성**으로, 사업자가 보고한 진척율과 실제 확인된 수분 완료율이GAP이 있는지를 확인한다. **셋째, 정산 반영의 적정성**으로, 최종 정산 시 적용된 수분과 단가가 계약 조건에 부합하는지를 확인한다.

수분 데이터는 감리에서 핵심 증거로 활용된다. 진척율 数据가 实勢와 다르게 reported 되면, 이는 곧 비용 청구 과다로 이어질 수 있으며, 감리에서 반드시 적발해야 할 중대한 지적 사항에 해당한다.

📢 **섹션 요약 비유**: SW 수분과 진척율 검증은 학교 숙제의 진행 상황 확인과 같다. "나는 오늘 3페이를 다 풀었다"고 말하지만, 실제로는 1페이지만 풀었다면家长(발주자)는 정확한Progress를 알 수 없다. 따라서ページ당 문제 수(FP) 대비 푼 문제 수(EV)를 측정하여 Progress를 객관적으로 확인해야 한다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**4.1 수분 산정 방법론 선택 기준**

FP와 LOC 중 어느 방법을 선택할지는 프로젝트 단계와 목적에 따라 달라진다. 일반적인 선택 기준은 다음과 같다.

```text
[수분 방법론 선택 기준]
   │
   ├──▶ 계약 전 단계 (제안/견적)
   │    └─ 기능점수(FP)가 유일한 선택
   │    └─ 요구사항에서 직접 FP 산정 가능
   │
   ├──▶ 개발 중 진척 관리
   │    └─ FP 기반의 EV (획득 가치) 측정
   │    └─ LOC는 개발 완료 후 확인
   │
   ├──▶ 개발 후実績分析
   │    └─ LOC 기반 생산성 (FP/LOC) 분석
   │    └─ 향후 프로젝트 예측에 활용
   │
   └──▶ 감리 검증
        └─ 계약 FP vs 실제 FP 대조
        └─ LOC 기반 물리적 완료도 검증
```

**4.2 진척 과다报告의 기술사적 판단**

실무에서 사업자가 진척율을 실제보다 높게 보고하는 경우는 자주 발생하는 문제이다. 감리인이 이를 적발하기 위해 사용하는 방법은 다음과 같다.

**코드 라인수 실측**: 실제 개발된 코드를 CVS(형상 관리)에서 추출하여 LOC를 实測한다. 계약 시 제시된 생산성(FP/人月) 대비 실측 생산성이著しく 낮으면, 진척율이 실제보다 높게 보고된 것일 수 있다.

**기능점수 재산정**: 감리인이 독립적으로 FP를 재산정하여 사업자 보고 수분과 대조한다. 차이가 크면 추가 검증이 필요하다.

**샘플 코딩 검증**: 전체 코드가 아닌 샘플을 추출하여, 해당 코드가 설계 대로 구현되었는지를 확인한다. 샘플 코드의 完成度를확인하여 전체 것에 반영한다.

```text
[진척 과다 보고 적발 체크리스트]
   │
   ├──▶ 진척율 (%): 계약/계획 대비 실제 완료율
   │    └─ 차이 > 10% 이상 시 추가 검증 필요
   │
   ├──▶ 생산성 (FP/人月): 실측 생산성 vs 계약 시 예상 생산성
   │    └─ 著しく 낮으면工作量過少报告 우려
   │
   ├──▶ 코드 커밋 이력: 형상 관리 시스템(Git)의 커밋 빈도와 크기
   │    └─ 커밋이集中되지 않고散在하면 진척이不規則的임
   │
   └──▶ 테스트 결과: 단위 테스트 커버리지, 성공률
       └─ 테스트 결과가 없거나 著しく 낮으면 完成渡ailed 가능
```

**4.3 수분 분쟁의 조정 원칙**

발주자와 사업자 간 수분 관련 분쟁 시 감리인의 판단 기준은 다음과 같다.

```text
[수분 분쟁 조정 원칙]
   │
   ├──▶ 계약 시 합의된 산정 방법 우선 적용
   │    └─ 계약서에서 명시한 FP 산정 기준과 복잡도 분류 기준
   │
   ├──▶ 계약이 불명확하면 第三者的 공인基準 적용
   │    └─ IFPUG(International Function Point Users Group) 기준
   │
   ├──▶ 실측이 가장 확실한 증거
   │    └─ 형상 관리 시스템의 실제 코드, DB의 실제 데이터 등
   │
   └──▶ 추가 기능은协商된 단가로 별도 정산
        └─ 계약 범위 밖의追加要求은 기존 단가로 정산
```

📢 **섹션 요약 비유**: SW 수분 분쟁 조정은 수입 QC와 같다.荷主가 "화물이 100톤이다"라고 주장하고 선박 회사가 "80톤이다"라고 하면, 실제 저울에 달어보는 것(실측)이 가장 확실하다. 화물 일부를抽出하여 단위 무게를 측정하고 전체를 外挿하면實重量에 가깝게 알 수 있다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

체계적인 SW 수분 관리의 기대효과는 프로젝트 관리의 투명성 확보, 발주자-사업자 간 분쟁 예방, 그리고 감리에서의 객관적 증거 확보로 나타난다.

| 구분 | 수분 미흡 시 | 수분 체계적 관리 시 |
|:---|:---|:---|
| 비용 협상 | 근거 없이 진행 | FP 기반 공정한 협상 |
| 진척 관리 |主观적 판단 |客观적 수치로Progress 관리 |
| 분쟁 예방 | 빈번한 비용 분쟁 | 수분 데이터로 투명하게 해결 |
| 감리 품질 | 증거 부족 | 객관적 수분 데이터로 검증 |
| 생산성 분석 | 불가능 | FP/LOC 기반 생산성 산출 가능 |

기능점수(FP) 방법은 지속적으로 更新되어 왔다. 초기에는 수동 산정에 의존했지만, 최근에는 자동화된 FP 산정 도구들이 등장하여 더욱 정확한 측정이 가능해지고 있다. 또한 AI 기반 코드 분석 도구를 활용하면, 코드를 分析하여 자동으로 功能 요소(ILF, EI, EO等)를識別하고 복잡도를 산정하는 것이 가능해지고 있다.

📢 **섹션 요약 비유**: SW 수분 관리는 장사에서의 계산대와 같다.商品的 개수와 단가를 정확히 기록해야 금액을 정확히낼 수 있고, 실제 판매량과 기록이 일치하는지 확인해야 창고 관리가 되는 것처럼, 소프트웨어에서도工作量을 정확히 측정하고 管理해야生意가 성립한다.

### 📌 관련 개념 맵 (Knowledge Graph)
*   기능점수 (FP, Function Point) | 소프트웨어 규모 산정의 代表적 方法으로, 기능 수 × 복잡도를 定量化
*   코드 라인수 (LOC, Lines of Code) | 실제 소스코드의 줄 수로 규모를 측정하는 方法으로, 개발 후実績分析에 활용
*   획득 가치 관리 (EVM, Earned Value Management) | PV/EV/AC 지표를 활용한 프로젝트 성과 측정 방법
*   과업대비표 (TTM, Task Traceability Matrix) | 요구사항과 산출물 간 추적성을 matrix 형태로 표현한 문서
*   IFPUG | International Function Point Users Group, FP 산정의 国际標準을 管理하는業界団体

### 👶 어린이를 위한 3줄 비유 설명
1. **개념**: 학원 원장님(발주자)과 과외 선생님(사업자)이 수업료를 정할 때, "수업이 몇 회 진행됐는지"를 세어야 비용을 정확히낼 수 있어요. 레슨 시간 수(FP) × 1회당 가격 = 총 수업료처럼요.
2. **원리**: 그런데 원장님이 "이번 달 10회 했어요"라고 하고, 선생님이 "8회 했어요"라고 하면 둘 사이에打架가 생겨요.그래서 출석부를 확인(형상 관리에서 실측)하여 실제로 몇 회 했는지 정확히 세야 해요.
3. **효과**: 이렇게 정확히 세면, 원장님은 적정한 수업료를 내고, 선생님은 자신의 노력을 정당하게 인정받을 수 있답니다!
