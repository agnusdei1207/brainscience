# 36. 감리 결과 보고서 작성

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 감리 결과 보고서는 감리 활동의 최종成果物으로, 감리 대상 프로젝트의 품질 상태와 개선 권고 사항을 객관적 증거에 기반하여 명확하고 체계적으로 기술하는 전문 문서이다.
> 2. **핵심 원칙**: 좋은 감리 보고서는 명확성(명확한 사실 기술), 객관성(감정 배제, 증거 기반), 완결성(누락 없이 全coverage),适时性(적시에 제공)의 4대 원칙을 충족해야 한다.
> 3. **구조**: 총평, 분야별 감리 결과, 시정 조치 권고 사항의 3층 구조로 구성되며, 각 지적 사항에는evidence와改善方向을 明記해야 한다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

감리 결과 보고서(Audit Report)는 감리 활동의 최종成果물이자, 감리인과 피감리인/발주자 사이의 主要 커뮤니케이션 수단이다. 현장의 감리 수행이 아무리 우수하더라도, 이를 적절한 보고서 형태로 문서화하지 못하면 감리의 가치를伝えることができない。

감리 보고서가 중요한 이유는 세 가지 측면에서 설명된다. **첫째, 의사결정의 근거**로, 발주자는 감리 보고서를 기반으로 사업자 대응 조치 요구, 계약 조정, 또는 사업 승인 여부를 결정한다. **둘째, 법적 증거**로, 분쟁 상황에서 감리 보고서는 전문가意見의 증거로 法廷에서 활용될 수 있다. **셋째, 지식 축적**으로, 감리 보고서는 조직의 교훈과 개선 방향을蓄積하여 향후プロジェクト에 환류할 수 있다.

```text
┌─────────────────────────────────────────────────────────────┐
│ [감리 보고서의 역할]                                        │
│                                                             │
│  [감리팀] ──▶ [감리 보고서] ──▶ [발주자]                  │
│                     │                    │                  │
│                     │                    ├── 사업 승인 판단  │
│                     │                    ├── 조치 요구 결정   │
│                     │                    └── 계약 조정 여부  │
│                     │                                        │
│                     ├─▶ [피감리인]                          │
│                     │                    ├── 시정 조치 파악   │
│                     │                    ├── 개선 방향 설정   │
│                     │                    └── 이행 증거 제출   │
│                     │                                        │
│                     ├─▶ [미래 프로젝트]                      │
│                     │                    └──教訓 학습 (Lessons Learned)│
│                     │                                        │
│                     └─▶ [감리 업계]                        │
│                          └─ 전문성 발전, 기준 고도화        │
└─────────────────────────────────────────────────────────────┘
```

감리 보고서는 감리 활동全体를compress하여 전달하는 역할이므로, 그質が直接 이해관계자들의 후속 행동과 의사결정에 영향을 미친다.

📢 **섹션 요약 비유**: 감리 보고서는医者の診断書と 같다. 의사가 환자를再檢하고 진단을 내렸지만, 그 내용을 진단서라는 문서로 작성하지 않으면患者는 다음 치료를 받을 수 없고, 다른 의사도これまでの経緯을 알 수 없다. 진단서だからこそ、모든 진료 결과가压缩되어 전달되고,后续治疗의基礎가 된다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**2.1 감리 보고서의 기본 구조**

감리 보고서는 一般적으로 다음과 같은 구조로 구성된다.

```text
[감리 보고서 표준 구성]
┌─────────────────────────────────────────────────────────────┐
│  1. 표지                                                    │
│     ├─ 보고서 제목 (감리 대상, 감리 단계)                   │
│     ├─ 보고서 번호,发行일                                  │
│     ├─ 감리 법인명, 감리팀장 서명                           │
│     └─ 발주처, 피감리인 정보                                │
│                                                             │
│  2. 요약 (총평, Executive Summary)                          │
│     ├─ 감리 배경 및 목적                                   │
│     ├─ 감리 범위 및 방법                                    │
│     ├─ 주요 감리 결과 (종합 의견)                          │
│     └─ 종합 의견 (적합/조건부 적합/부적합)                 │
│                                                             │
│  3. 분야별 감리 결과                                        │
│     ├─ 사업 관리 영역 감리 결과                             │
│     ├─ 응용 시스템 영역 감리 결과                          │
│     ├─ 데이터베이스 영역 감리 결과                          │
│     └─ 시스템 아키텍처/보안 영역 감리 결과                  │
│                                                             │
│  4. 시정 조치 권고 사항                                     │
│     ├─ 필수 조치 (Major) 목록 및 상세 내용                  │
│     └─ 권고 사항 (Minor) 목록 및 상세 내용                  │
│                                                             │
│  5. 첨부                                                    │
│     ├─ 감리 체크리스트                                       │
│     ├─ 증거 자료 목록                                        │
│     ├─ 용어 정의                                            │
│     └─ 감리팀简历                                           │
└─────────────────────────────────────────────────────────────┘
```

**2.2 총평 (Executive Summary) 작성 원칙**

총평은 보고서를 읽지 않는 경영진을 위해서도 감리 결과를 한눈에 파악할 수 있도록 작성하는 핵심 섹션이다. 일반적으로 1~2 페이지 내외로 작성하며, 다음 네 가지 항목을 반드시 포함해야 한다.

```text
[총평 필수 포함 항목]
   │
   ├──▶ 감리 배경
   │    └─ 왜 이 감리가 수행되었는지 (사업 개요, 감리 요청 경위)
   │
   ├──▶ 감리 범위 및 방법
   │    └─ 무엇을, 어떻게 감리했는지 요약
   │
   ├──▶ 주요 발견 사항
   │    └─ 핵심 지적 사항 3~5가지를 요약
   │
   └──▶ 종합 의견
        └─ 적합 / 조건부 적합 / 부적합 + 그 근거
```

종합 의견의 판단 기준은 다음과 같다.

```text
[종합 의견 판단 기준]
   │
   ├──▶ 적합 (Pass)
   │    └─ 모든 필수 지적 사항이 없거나,仅有的 것들이軽微
   │
   ├──▶ 조건부 적합 (Pass with Conditions)
   │    └─ 중대한 지적 사항이 있으나, 시정 조치 계획이 제시되고 이행 가능한 경우
   │
   └──▶ 부적합 (Fail)
        └─ 필수 지적 사항이多수 있고, 시스템 오픈이나 사업 완료에 중대한 영향을 미치는 경우
```

**2.3 분야별 감리 결과 작성**

각 감리 영역별로 감리 결과를 작성할 때는 다음 구조를 따른다.

```text
[분야별 감리 결과 기술 구조]
   │
   ├──▶ 감리 개요
   │    ├─ 해당領域의 감리 목적 및 범위
   │    └─ 적용된 감리 기준 (기준 번호 및 내용)
   │
   ├──▶ 감리 방법
   │    └─ 수행한 감리 기법 (문서 검토, 인터뷰, 테스트 등)
   │
   ├──▶ 발견 사항
   │    ├─ 제목: 발견 사항의 핵심을 한 줄 요약
   │    ├─ 발견 상세: 사실관계, 근거, 영향도
   │    └─ 해당 기준: 어떤 감리 기준에 위배되는지
   │
   └──▶ 소결
        └─ 해당 영역에 대한 종합 평가
```

**2.4 지적 사항의 구성 요소**

각 지적 사항은 다음 요소를 모두 포함해야 한다.

```text
[지적 사항 구성 5요소]
   │
   ├──▶ 1. 제목 (Finding Title)
   │    └─ 발견 사항을 한눈에 파악할 수 있도록 명확하게
   │
   ├──▶ 2. 사실관계 (Facts)
   │    └─ 관찰된 사실 (who, what, when, where, how)
   │
   ├──▶ 3. 근거 (Evidence)
   │    └─ 사실을 뒷받침하는 증거 (문서명, 면담 내용, 테스트 결과 등)
   │
   ├──▶ 4. 영향 (Impact)
   │    └─ 해당 문제가 시스템/사업에 미치는 영향
   │
   └──▶ 5. 권고 (Recommendation)
        └─ 구체적이고 실행 가능한 조치 방안
```

📢 **섹션 요약 비유**: 감리 보고서 작성은小说的執筆と似ている。표지(보고서 표지), 목차(총평), 각 챕터(분야별 결과), 그리고 최종 해피엔딩 또는教訓(시정 조치 권고)으로 구성된다. 모든 챕터가 主題와 연결되어 있어야完整한 이야기가 되고,最重要的教訓을 readers(발주자)가 확실히 알아채야 한다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

감리 보고서는 다양한 이해관계자에게 전달되므로, 각 이해관계자의 정보 요구에 따라보고서 활용 방식이 달라진다. 이에 대한 비교 분석이 필요하다.

**발주자용 vs 피감리인용 보고서**: 발주자용 보고서는 경영진의 의사결정에 초점을 맞추어, 종합 의견과 주요 위험 사항을 강조하고, 자세한 기술적 세부 사항은 첨부로 처리한다. 피감리인용 보고서는 지적 사항의 구체적內容과 개선 방향을 명확히 전달하여 조치 수행을용이하게 한다.

**중간 보고서 vs 최종 보고서**: 프로젝트中途에 발행하는 중간 보고서는 발견된 중대한 문제를 즉시 전달하기 위한 것으로, 형식이 간략하고措正要求에 집중한다. 최종 보고서는 전체 감리 결과를 체계적으로 정리한 것으로, 완전한 구조를 갖춘다.

```text
┌─────────────────── 보고서 유형별 비교 ───────────────────┐
│                                                          │
│  유형              │ 목적           │ 특징              │
│  ─────────────────────────────────────────────────────  │
│  중간 보고서        │ 중대 문제 조기 전달 │ 형식 간소,措正要求中心 │
│  최종 보고서        │ 全結果 정리/전달 │ 완전한 구조, 全指摘記載 │
│  실행 보고서        │ 조치 결과 확인   │ 이행 증거 중심 평가  │
│  특별 보고서        │突発 문제 보고   │사전 양식 무시, 即時 보고│
└────────────────────────────────────────────────────────┘
```

**감리 보고서 vs 감사 보고서**: 감리 보고서와 감사(Audit) 보고서는 혼용되지만,厳密には 다르다. 감리 보고서는 개선 권고에 중점을 두지만, 감사 보고서는 적합/부적합 판정에 중점을 둔다. ISMS-P 인증심사에서는 감사(Audit) 보고서를 사용하고, 정보시스템 감리에서는 감리(Audit) 보고서를 사용한다.

감리 보고서의質를評価하는 기준은 네 가지이다. **명확성**은 독자가 누구나 동일한 의미를 이해할 수 있어야 한다는 것이고, **객관성**은 감리인의 主観的 판단이나 편견이 배제되어야 한다는 것이며, **완결성**은 모든重要的 사항이 빠짐없이 포함되어야 한다는 것이고, **适时性**은 의사결정에 의미있는 시기에 제공되어야 한다는 것이다.

📢 **섹션 요약 비유**: 감리 보고서 유형 구분은医院 진단서 종류와 같다. 중간 보고서는"응급실 도착, 출혈 확인, 즉각 수혈 필요"처럼紧急 내용을 먼저 전하고, 최종 진단서는"全검사 결과, 종합 소견, 향후 치료 계획"을全都記録한다. 목적에 맞는 적절한報告形式가 필요하다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**4.1 감리 지적 사항의 문구 선택**

감리 지적 사항의 문구(표현)는 매우 중요하다. 너무 부드럽게 표현하면 피감리인이 심각성을 인식하지 못하고, 너무 강하게 표현하면防卫적 반응을유발하여 오히려 조치 이행을 어렵게 한다.

```text
[지적 사항 문구 전략]
   │
   ├──▶ 사실에 기반한 기술
   │    └─ "A 시스템의 패스워드가 평문으로 저장되어 있다"
   │    而不是 "보안이 엉망이다"
   │
   ├──▶ 영향과 근거를 명시
   │    └─ "이 경우 외부 공격자가 DB에 직접 접근하여 개인정보를 유출할 수 있다"
   │    而不是 "심각한 보안 문제다"
   │
   ├──▶ 구체적인 개선 방향 제시
   │    └─ "SHA-256 이상의 해시 알고리즘 적용을 권고한다"
   │    而不是 "암호화 강도를 높여야 한다"
   │
   └──▶ 균형 잡힌 표현
        └─ 위험한 사실과 함께 긍정적 발견도记载
        └─ "이 부분은 잘 설계되어 있으나, 다음과 같은 개선이 필요하다"
```

**4.2 종합 의견 결정의 기술사적 판단**

종합 의견(적합/조건부 적합/부적합)은 감리인의 전문가적 판단에 의한 것으로, 명확한 기준 없이 주관적으로 결정되어서는 안 된다. 판단 프레임워크는 다음과 같다.

```text
[종합 의견 결정 프레임워크]
   │
   ├──▶ 필수 조치(Major) 지적 사항 수
   │    ├─ 0개: 적합 가능성이 높음
   │    ├─ 1~2개: 조치 가능 여부에 따라 조건부 적합/부적합
   │    └─ 3개 이상: 부적합 가능성이 높음
   │
   ├──▶ 지적 사항의 系统적 영향
   │    ├─ 핵심 기능(결제, 보안 등)에 영향: 加点
   │    └─ 단순 UI/문서 수준 영향: 감点
   │
   ├──▶ 시정 조치의 현실성
   │    ├─ 단기간内 조치 가능: 조건부 적합 가능
   │    └─ 구조적 대규모 개편 필요: 부적합 고려
   │
   └──▶ 발주자 의사
        └─ 사업 특성 및 위험 감내 능력 고려
```

**4.3保密 및アクセス 管理**

감리 보고서에는 보안상 민감한 정보가 포함되므로, appropriate한 접근 관리와秘密保持가 필요하다. 보고서 배부 목록을管理하고, 전자 파일에는アクセス制御를 적용하며, 수기 보고서는登録管理体系로 관리해야 한다.

보고서受众에 따른差别 배포가 필요할 수 있다. 발주자에게는 Full 보고서를 제공하고, 피감리인에게는 지적 사항 중심의 보고서를 제공하며, 이해관계가 없는 third party에게는 요약만 제공하는 등의 차등 접근이 있을 수 있다.

📢 **섹션 요약 비유**: 감리 보고서의 문구 선택은 의사의 환자 전달 방식과 같다. "암울한预感이 든다"보다는 "검사 결과 certain한 이상치가 발견되어 추가 정밀 검사가 필요합니다"라고 표현하는 것이適切이다. 같은 사실이라도 전달 방식에 따라 환자의 대응이 달라지듯이, 감리 보고서도 객관적이면서도 건설적인 표현이 필요하다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

체계적인 감리 보고서의 기대효과는 의사결정의Quality 향상, 지식 축적, 그리고profession의 신뢰 구축으로 나타난다.

| 구분 | 보고서 미흡 시 | 체계적 보고서 작성 시 |
|:---|:---|:---|
| 의사결정 근거 | 불명확, 판단 어려움 | 명확, 신속한 결정 가능 |
| 조치 이행 | 지적 사항 이해 부족 | 명확한 개선 방향 파악 |
| 분쟁 대응 | 증거력弱的 | 객관적 증거로補强 |
| 지식 축적 | 교훈 날림 | 다음 프로젝트 환류 |
| profession 이미지 | 감리 가치 저평가 | 감리 중요성 인식 증대 |

미래에는 AI 기반 감리 보고서Draft 도구가 등장하여, 감리 체크리스트 결과와 증거 데이터를 입력하면 자동으로 보고서Draft를 생성하는 것이 가능해질 것이다. 그러나 AI가生成한 보고서도 감리 전문가의 검토와 보완이 필수적이며, 특히 종합 의견 판단과 권고 사항의 타당성 평가는 여전히 인간 전문가의 몫이다.

📢 **섹션 요약 비유**: 감리 보고서의 미래はAI活用의사处方 자동 生成システムと似ている。검사 결과 데이터를入力하면 AI가처방전Draft를 제안하지만, 의사의 전문적 판단으로 최종 처방을 내리듯이, AI 도구와 인간 전문가의 협업이 최고의報告書を produced 한다.

*   Exit Meeting | 감리 종료 시 감리 결과를 피감리인에게 발표하고 이견을 조율하는 회의

### 👶 어린이를 위한 3줄 비유 설명
1. **개념**: 선생님(감리인)이 시험(日曜?)을 보고 나서 결과지를 쓰는 거예요. 전체적으로 어떤 시험이었는지(총평), 각 과목별로 무엇이 잘 되고 무엇이 안 되었는지(분야별 결과), 어떻게 공부해야 하는지(시정 조치 권고)를文書로 만들어서 엄마(발주자)와 학생(피감리인) 모두에게 주는 거예요.
2. **원리**: 결과지만、"너 멍청이다"这样的话로는 안 되고, 실제로 어떤 문제가 있었는지(근거), 어떻게 하면 좋아지는지(권고)를 명확하게 적어야 해요. 그래야 다음에 어떻게 해야 하는지 알 수 있잖아요.
3. **효과**: 이렇게 적으면 엄마도、项目内容를 잘 이해할 수 있고, 학생도、"그러면 이렇게 해보자"하고 개선할 수 있답니다!
