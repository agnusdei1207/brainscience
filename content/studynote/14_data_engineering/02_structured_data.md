# 02. 정형 데이터 (Structured Data)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 정형 데이터 (Structured Data)는 관계형 데이터베이스(RDBMS)의 테이블과 같이 행(Row)과 열(Column)로 구성된 엄격하고 고정된 스키마(Schema)를 갖춘 데이터이다.
> 2. **가치**: 스키마 강제를 통해 데이터의 무결성(Integrity)과 일관성(Consistency)을 완벽하게 보장하며, 트랜잭션 처리(ACID)와 복잡한 관계 연산(JOIN)에 최적화되어 있다.
> 3. **융합**: 고정된 구조 덕분에 B-Tree 인덱스와 같은 고속 검색 자료구조와 완벽히 결합하며, 기업의 핵심 원장(Core Banking, ERP)을 지탱하는 기반 인프라로 작용한다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)
정형 데이터 (Structured Data)는 사전에 정의된 데이터 모델, 즉 스키마(Schema)에 맞추어 체계적으로 저장되는 데이터 형식이다. 기업 비즈니스의 가장 핵심이 되는 금융 거래 내역, 고객 정보, ERP (Enterprise Resource Planning) 원장 데이터가 이에 해당한다.
초기 데이터베이스 시스템에서는 데이터의 중복 저장을 막고 디스크 공간을 절약하며 이상 현상(Anomaly)을 방지하는 것이 최우선 과제였다. 이를 위해 E.F. Codd가 제안한 관계형 대수(Relational Algebra) 기반의 모델이 도입되었고, 이는 데이터를 2차원 표(Table) 형태로 엄격하게 통제하는 방식으로 발전했다.
따라서 정형 데이터는 단순히 정돈된 데이터를 넘어, 데이터가 입력되는 시점(Schema-on-Write)에 모든 규칙을 검증하여 시스템의 신뢰성을 극한으로 끌어올리는 철학적 기반을 가진다.

[데이터의 구조화 수준에 따른 한계 스펙트럼 (문제 배경도)]
```text
[비정형] 이미지/음성          [반정형] JSON/XML         [정형] RDB 테이블
    │                         │                        │
    ├── 규칙 없음               ├── 태그 기반 자체 구조      ├── 엄격한 행/열, 타입 강제
    ├── 쿼리 불가능/어려움        ├── 유연함, 검색 가능        ├── 완벽한 SQL 질의 가능
    └── [무결성 보장 불가] <==========================> [강력한 무결성 및 제약조건]
```
이 도식은 데이터 형태에 따른 통제 수준과 무결성 보장 관계를 보여준다. 비정형 및 반정형 데이터는 유연성을 제공하지만 비즈니스 핵심 로직의 정합성을 보장할 수 없다. 기업의 결제, 이체, 재고 관리 등 절대 오차가 발생해선 안 되는 영역에서는 반드시 가장 오른쪽의 정형 데이터 구조와 스키마 강제 정책이 필요함을 시사한다.

📢 **섹션 요약 비유**: 정형 데이터는 마치 규격이 1mm의 오차도 없이 깎여진 레고 블록과 같아서, 설계도(스키마)대로 튼튼하고 완벽하게 끼워 맞출 수 있는 뼈대입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)
정형 데이터는 저장소 레벨에서 엄격한 제약조건(Constraint)과 관계성(Relation)을 내부 메커니즘으로 갖는다.

| 구성 요소 | 역할 | 내부 동작 메커니즘 | 실무 의미 |
|:---|:---|:---|:---|
| **스키마 (Schema)** | 데이터 구조와 타입의 청사진 | 테이블의 열 이름, 데이터 타입(INT, VARCHAR) 정의 | 설계 변경(DDL) 시 DBLock 발생 등 큰 비용 수반 |
| **튜플 (Tuple / Row)** | 하나의 완전한 개체 데이터 | 물리적 디스크의 블록 단위로 로우 형태로 직렬화되어 저장 | 1건의 트랜잭션 발생 단위 |
| **속성 (Attribute / Column)** | 개체의 특정 속성 및 도메인 | 특정 데이터 타입과 도메인 제약(Not Null 등) 검증 | 검색 및 정렬의 기준(Index 대상) |
| **기본키 (Primary Key)** | 행을 고유하게 식별하는 식별자 | 내부적으로 B-Tree 인덱스를 자동 생성하여 클러스터링 | 데이터 중복 완전 차단 |
| **외래키 (Foreign Key)** | 다른 테이블과의 참조 관계 | 참조 무결성(Referential Integrity) 제약을 통한 연쇄 삭제 방지 | 복잡한 JOIN 연산의 연결 고리 |

[정형 데이터의 내부 무결성 검증 및 B-Tree 인덱싱 흐름도]
```text
[INSERT / UPDATE 요청] : {id: 1, amount: "A백원"}
        ↓
[1. 스키마 검증 (Schema-on-Write)]
 ├─ 타입 체크 (Type Check)     : "A백원"은 INT가 아님 => (에러/Reject)
 ├─ 도메인 체크 (Constraint)   : NOT NULL, CHECK 조건 검증
 └─ 참조 무결성 (Foreign Key)  : 부모 테이블에 존재하는 ID인가?
        ↓ (성공 시)
[2. 인덱스 업데이트 (B-Tree)]
 ┌─ Root Node ───┐
 │ id < 50       │
 └─┬─────────────┘
   └─> [Leaf Node: 직렬화된 데이터 블록(Page)에 Row 쓰기]
```
이 흐름의 핵심은 데이터가 디스크에 기록되기 전(Schema-on-Write)에 매우 무거운 검증 단계를 반드시 거친다는 점이다. 타입 불일치나 제약조건 위반 시 즉각 에러를 반환(Fail-fast)하여 쓰레기 데이터의 유입을 원천 차단한다. 성공적으로 검증된 데이터는 B-Tree 구조의 리프 노드 페이지에 정렬되어 기록되므로, 이후 읽기(SELECT) 성능이 극대화된다. 실무에서는 이 쓰기 시점의 검증 오버헤드가 대규모 동시 요청 시 락(Lock) 경합과 성능 병목을 유발한다.

📢 **섹션 요약 비유**: 정형 데이터 시스템은 깐깐한 출입국 관리소와 같습니다. 여권(타입)이 없고 비자(외래키)가 없으면 아예 입국(저장)을 불허하여, 나라(DB) 안의 질서를 완벽하게 유지합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

비정형/반정형 데이터 중심의 NoSQL 생태계와 정형 데이터(RDBMS)의 본질적 차이를 이해해야 한다.

| 비교 항목 | 정형 데이터 (RDBMS 테이블) | 반정형 데이터 (Document NoSQL 등) | 판단 포인트 |
|:---|:---|:---|:---|
| **스키마 적용** | Schema-on-Write (쓰기 시 강제) | Schema-on-Read (읽기 시 해석) | 데이터 유입 속도 vs 품질 |
| **관계 표현** | 엄격한 정규화(Normalization) 및 JOIN | 비정규화, 내포(Embedding) 기반 구조 | 연산 복잡도 vs 데이터 중복 |
| **수직/수평 확장** | 수직 확장(Scale-up) 위주, 샤딩 어려움 | 수평 확장(Scale-out) 용이 | 시스템 인프라 증설 비용 |
| **트랜잭션(ACID)**| 완벽한 ACID 트랜잭션 보장 | 제한적 트랜잭션, Base(Eventual Consistency) | 금융결제 여부 |

[정규화 기반 조인(JOIN) vs 반정형 임베딩(Embedding) 구조 비교도]
```text
< 정형 데이터 (정규화, 참조) >        |  < 반정형 데이터 (비정규화, 임베딩) >
[User Table]       [Order Table]  |  [User Document]
id | name          id | u_id | amt|  {
---------          ---------------|    "id": 1, "name": "Kim",
1  | Kim  <====FK= 10 |  1   | 50 |    "orders": [
2  | Lee           11 |  1   | 30 |       {"id": 10, "amount": 50},
                                  |       {"id": 11, "amount": 30}
(조인을 위해 CPU 연산/메모리 부하 발생)  |    ]
                                  |  } (조인 불필요, 디스크 I/O 1회)
```
이 비교도는 두 데이터 모델의 근본적인 트레이드오프를 명확히 보여준다. 왼쪽 방식(정형)은 데이터의 중복이 없어 업데이트 이상(Anomaly)이 원천 방지되지만, 읽을 때 조인 연산 부하가 크다. 오른쪽 방식은 읽기 성능은 좋지만 'Kim'의 이름이 바뀌면 중복 저장된 모든 문서를 찾아 고쳐야 하는 일관성 유지의 치명적 단점이 있다.

📢 **섹션 요약 비유**: 정형 데이터는 각 부품을 따로 보관하다가 필요할 때 조립(JOIN)하는 맞춤형 공구 세트라면, 반정형 데이터는 자주 쓰는 부품을 아예 하나로 붙여놓은 일체형 멀티툴과 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)
실무에서 정형 데이터 모델을 다룰 때는 스키마의 경직성으로 인한 장애와 병목을 피하는 것이 핵심이다.

1. **스키마 변경(Schema Evolution) 장애 안티패턴**: 수십억 건의 데이터가 있는 테이블에 `ALTER TABLE ADD COLUMN`을 수행하면 장시간의 테이블 락(Table Lock)이 발생해 서비스 전체가 마비된다.
   - **판단**: 대규모 시스템에서는 무중단 스키마 변경을 위해 페트로나스(pt-online-schema-change) 같은 툴을 사용하거나, 신규 테이블을 만들고 백그라운드에서 데이터를 마이그레이션하는 섀도우 롤아웃 전략을 취해야 한다.
2. **과도한 정규화에 따른 성능 저하**: "데이터 중복을 없앤다"는 교과서적 정규화 원칙에 매몰되어 5~6개 이상의 테이블을 JOIN하면, 쿼리 플랜 최적화가 실패하고 풀 테이블 스캔이 발생한다.
   - **판단**: 읽기 성능이 중요한 서비스 환경에서는 의도적인 역정규화(De-normalization)를 통해 조회용 집계 테이블이나 뷰(Materialized View)를 생성하여 CPU 조인 부하를 공간 복잡도로 치환해야 한다.
3. **NULL 처리의 모호성**: 정형 데이터에서 스키마에 정의된 속성 값이 비어있을 때 NULL을 허용하면, 애플리케이션 단에서 NPE(Null Pointer Exception) 발생 확률이 높아지고 인덱스 스캔 효율이 떨어진다.
   - **판단**: 실무에서는 가급적 `NOT NULL DEFAULT 'N/A'` 형태로 기본값을 강제하여 옵티마이저의 인덱스 활용성을 높이고 코드 방어력을 확보한다.

[스키마 변경 시 락(Lock) 경합 및 장애 전파 시각화]
```text
[Client A] --(SELECT)--> [Table X (Shared Lock)] => 정상 응답
[DBA]      --(ALTER) ---> [Table X (Exclusive Lock 요청)] => (대기 상태 진입)
[Client B] --(SELECT)--> [Table X] => (DBA 작업 뒤에서 무한 큐잉)
                           ▲
          결과: DB 커넥션 풀 고갈 및 전면 서비스 장애 (장애 전파 지점)
```
이 도식은 스키마가 고정된 정형 데이터베이스에서 구조를 변경할 때 발생하는 치명적인 장애 전파 과정을 보여준다. 단순히 컬럼 하나를 추가하는 행위가 메타데이터 락(MDL)을 유발하고, 이로 인해 후속 읽기 요청들까지 연쇄적으로 블로킹되어 전체 애플리케이션의 커넥션 풀이 고갈된다. 실무에서는 정형 데이터의 경직성을 극복하기 위해 아키텍처 수준의 우회 기법이 필수적이다.

📢 **섹션 요약 비유**: 튼튼한 콘크리트 건물(정형 데이터)은 안전하지만, 창문 하나를 새로 내려면(스키마 변경) 건물 전체의 안전 진단을 받고 한동안 건물 출입을 통제해야 하는 큰 공사가 됩니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)
정형 데이터는 기업의 가장 중요한 자산인 원장(Ledger) 데이터를 지탱하며, 데이터 일관성 보장의 마지노선 역할을 수행한다.

| 관점 | 정형 데이터 도입 가치 | 비고 |
|:---|:---|:---|
| 데이터 신뢰성 | 무결성 제약을 통한 트랜잭션 완벽 보장 | 금융, 결제, ERP 시스템 필수 |
| 표준화 | SQL (Structured Query Language) 국제 표준 | 호환성 및 풍부한 분석 생태계 |

최근 클라우드 및 분산 환경의 발전으로, 정형 데이터의 한계였던 스케일 아웃(Scale-out) 문제를 극복하기 위해 구글 스패너(Spanner)나 코크로치DB(CockroachDB) 같은 NewSQL이 등장하고 있다. 이들은 정형 데이터의 강력한 일관성을 유지하면서도 글로벌 분산 저장을 지원하는 방향으로 진화하고 있다. 데이터 생태계가 아무리 레이크와 비정형 위주로 흘러가도, 'Single Source of Truth (단일 진실 공급원)'로서의 정형 데이터 저장소는 영원히 대체할 수 없는 코어 역할을 수행할 것이다.

📢 **섹션 요약 비유**: 정형 데이터는 화려하지 않지만 모든 비즈니스의 척추와 같습니다. 아무리 살(비정형/AI)이 찌고 몸집이 커져도, 척추가 바르지 않으면 결국 무너지게 됩니다.

### 📌 관련 개념 맵 (Knowledge Graph)
* RDBMS (Relational Database Management System) | 정형 데이터를 관리하고 SQL을 제공하는 엔진
* 정규화 (Normalization) | 정형 데이터의 중복을 제거하고 무결성을 높이는 설계 기법
* B-Tree 인덱스 | 정형 데이터의 빠른 검색을 지원하는 계층형 트리 자료구조
* ACID 속성 | 원자성, 일관성, 고립성, 지속성을 보장하는 트랜잭션 원칙
* 데이터 웨어하우스 (Data Warehouse) | 분산된 정형 데이터를 의사결정을 위해 통합해 놓은 중앙 저장소

1. 정형 데이터는 과목, 점수, 이름이 딱딱 맞춰진 학교 성적표와 같아요.
2. 규칙이 너무 엄격해서 이름 칸에 숫자를 쓰면 아예 저장이 안 되고 튕겨 나가요.
3. 깐깐하지만, 덕분에 절대로 정보가 뒤섞이거나 틀릴 일이 없어서 은행의 돈 계산 같은 중요한 일에 꼭 필요하답니다.
