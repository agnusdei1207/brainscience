# 03. 반정형 데이터 (Semi-structured Data)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 반정형 데이터 (Semi-structured Data)는 고정된 테이블 스키마 대신 데이터 자체 내부에 태그나 키(Key) 형태로 구조(메타데이터)를 포함하는 자가 기술적(Self-describing) 데이터이다.
> 2. **가치**: 스키마의 사전 정의 없이도 데이터의 계층적 표현이 가능해, 비즈니스 요구사항 변화에 즉각 대응하고 이기종 시스템 간의 API 데이터 교환을 유연하게 만든다.
> 3. **융합**: JSON, XML 형태를 기반으로 Document NoSQL(MongoDB 등)과 완벽하게 결합하며, 웹 아키텍처(RESTful, GraphQL) 및 마이크로서비스(MSA) 생태계의 표준 페이로드로 기능한다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)
반정형 데이터 (Semi-structured Data)는 정형 데이터(테이블)와 비정형 데이터(텍스트/이미지)의 중간 지점에 위치하는 데이터 형태이다. 대표적으로 XML (eXtensible Markup Language), JSON (JavaScript Object Notation), YAML 등이 있다.
과거 모놀리식(Monolithic) 환경에서는 RDBMS의 고정 스키마로 충분했으나, 웹의 폭발적 성장과 마이크로서비스 아키텍처(MSA)의 도입으로 수많은 이기종 언어와 플랫폼 간에 데이터를 주고받아야 할 필요성이 생겼다. 이 과정에서 각자의 DB 스키마를 맞추는 것은 불가능했기 때문에, 데이터 덩어리 자체에 '이 값이 무엇을 의미하는지' 태그를 붙여서 보내는 유연한 표준 포맷이 필연적으로 요구되었다.
따라서, 반정형 데이터는 스키마 변경 시 발생하는 RDBMS의 락(Lock) 장애를 우회하고, 애플리케이션 주도의 애자일(Agile)한 개발 속도를 보장하기 위한 현대 웹 아키텍처의 핵심 기반이 된다.

[데이터 교환 패러다임의 변화와 반정형 데이터의 위치 (문제 배경도)]
```text
[과거: 강결합 아키텍처]             [현재: MSA 기반 API 통신 (반정형 데이터)]
Client ──> [DB 직접 접근]          Service A (Java) ──(JSON)──> Service B (Node.js)
            (스키마 종속)              |                           |
            (언어 종속)                └── { "id": 1, "name": "Kim", "skills": ["A","B"] }
                                        (언어 독립적, 구조 자체 포함, 즉시 파싱 가능)
```
이 도식은 마이크로서비스 간 통신에서 반정형 데이터가 왜 필수적인지 보여준다. 서로 다른 데이터베이스와 언어를 사용하는 시스템들이 통신하기 위해서는 공통의 '어휘'가 필요한데, 데이터 스스로 구조를 설명(Self-describing)하는 JSON 포맷이 이 역할을 완벽히 수행한다. 데이터에 구조가 내포되어 있으므로 수신자는 사전 스키마 합의 없이도 즉각적인 파싱과 직렬화/역직렬화가 가능하다.

📢 **섹션 요약 비유**: 반정형 데이터는 내용물마다 "이것은 양말", "이것은 책"이라고 라벨표가 붙어 있는 이삿짐 박스입니다. 미리 박스 규격을 정하지 않아도, 도착한 뒤에 라벨만 보고 쉽게 물건을 정리할 수 있습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)
반정형 데이터는 내부적으로 트리(Tree) 또는 그래프(Graph) 기반의 계층적 논리 구조를 가진다.

| 구성 요소 | 역할 | 구조적 특징 | 실무 의미 |
|:---|:---|:---|:---|
| **키-값 (Key-Value) 쌍** | 데이터의 의미(메타데이터)와 실제 값을 매핑 | `"age" : 25` 형태로 자체 설명적 | 속성을 자유롭게 추가/삭제 가능 |
| **계층(Hierarchy) 내포** | 부모-자식 관계의 복잡한 구조 표현 | 객체 내부에 또 다른 객체나 배열 임베딩 | RDBMS의 JOIN을 없애 읽기 속도 향상 |
| **태그 (Tag) / 마크업** | XML 등에서 요소의 시작과 끝 지정 | `<user>...</user>` 형태 구성 | 데이터 파싱 시 문맥(Context) 식별 |
| **배열 (Array / List)** | 다중 값(Multi-value)의 직관적 표현 | `["Admin", "User"]` 형태 보관 | 테이블 쪼개기(1:N 정규화) 불필요 |

[JSON 기반 반정형 데이터의 내부 트리 파싱 구조도]
```text
[원시 JSON 문자열] : {"id":1, "loc": {"city":"Seoul", "zip":123}}
        ↓ (애플리케이션 역직렬화 / Deserialization)
[메모리 내 논리적 트리(Tree) 변환]
         (Root Object)
          ├── [Key: id]  ── (Value: 1, Type: Number)
          └── [Key: loc] ── (Object)
                              ├── [Key: city] ── (Value: "Seoul", Type: String)
                              └── [Key: zip]  ── (Value: 123, Type: Number)
```
이 구조도는 단순한 문자열로 전송된 반정형 데이터가 메모리 상에서 어떻게 트리 자료구조로 변환되는지 보여준다. 반정형 데이터 파서(Parser)는 괄호 `{ }` 와 쉼표 `,` 등의 문법적 구분자(Delimiter)를 기준으로 노드를 분기하여 계층 구조를 조립한다. 이렇게 파싱된 트리는 애플리케이션 레벨에서 즉시 객체(Object)로 매핑되어 사용된다. 이 과정에서 CPU 자원이 소모되는 오버헤드가 발생하지만, 그 대가로 무한한 구조적 유연성을 얻게 된다.

📢 **섹션 요약 비유**: 반정형 데이터 구조는 마치 폴더 안에 폴더를 계속 만들 수 있는 컴퓨터의 디렉터리 시스템과 같아서, 어떤 복잡한 서류(데이터)라도 자유롭게 끼워 넣을 수 있습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

저장 관점에서 정형(RDBMS)과 반정형(Document NoSQL) 아키텍처의 트레이드오프를 비교한다.

| 비교 항목 | RDBMS (정형 테이블) | MongoDB 등 Document Store (반정형) | 판단 포인트 |
|:---|:---|:---|:---|
| **스키마 관리** | 중앙 DB 엔진이 엄격히 통제 (강제) | 애플리케이션 코드 레벨에서 통제 (위임) | 개발 민첩성 vs 데이터 파편화 |
| **조회(Query) 성능** | 인덱스와 다중 JOIN 최적화 | 임베딩(내포)으로 단건 조회 극강, JOIN 취약 | 읽기 패턴 (단건 vs 관계 탐색) |
| **저장 효율 (공간)** | 중복 배제로 디스크 공간 최적화 | 키/태그가 매 데이터마다 반복되어 공간 낭비 큼 | 스토리지 비용의 수용 여부 |
| **데이터 변경** | DDL(Alter table) 시 시스템 부하 | 아무런 작업 없이 JSON 필드 추가 가능 | 비즈니스 로직의 변동성 |

[RDBMS와 Document NoSQL의 스키마 변경 대응 비교 매트릭스]
```text
[요구사항: User 정보에 "취미(hobby)" 필드를 새롭게 추가하라]

(방식 A: RDBMS 정형 구조)
  ALTER TABLE users ADD COLUMN hobby VARCHAR(50); => (테이블 락 유발 위험, 스키마 마이그레이션 필요)
  
(방식 B: Document NoSQL 반정형 구조)
  기존 데이터: { "id": 1, "name": "Kim" }
  신규 데이터: { "id": 2, "name": "Lee", "hobby": "Golf" } => (DB 수정 작업 0. 그냥 넣으면 들어감)
```
이 비교는 반정형 데이터의 핵심 가치인 '스키마 진화(Schema Evolution)의 용이성'을 보여준다. B 방식에서는 새로운 속성이 추가되어도 과거 데이터를 마이그레이션할 필요가 없다. 애플리케이션 단에서 `hobby` 키가 존재하는지 유무만 분기 처리하면 되므로 서비스 중단 없이 즉각적인 배포가 가능하다. 반면 반정형 데이터는 `name`이라는 키(메타데이터) 글자 자체가 수십억 번 반복 저장되므로 스토리지 낭비가 발생한다는 단점이 있다.

📢 **섹션 요약 비유**: 정형 DB는 붕어빵 틀이어서 모양을 바꾸려면 무거운 무쇠 틀을 새로 파야 하지만, 반정형 DB는 찰흙놀이와 같아서 언제든 손으로 빚어 새로운 모양을 추가할 수 있습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)
반정형 데이터를 실무에 적용할 때는 유연성이 초래하는 데이터 파편화(Fragmentation)와 무결성 붕괴를 경계해야 한다.

1. **무결성 상실 및 쓰레기 데이터 유입 (GIGO)**: 반정형 DB는 어떤 구조든 다 받아주기 때문에, 클라이언트 버그로 인해 오타가 난 키(예: `nname` 대신 `name`)가 들어와도 에러 없이 저장된다.
   - **판단**: DB 엔진이 스키마를 검증하지 않는다면, 책임을 애플리케이션으로 이관해야 한다. 실무에서는 JSON Schema Validator 등을 애플리케이션 앞단에 배치하여 로직 레벨에서 최소한의 필수 속성 제약을 강제(Application-side Validation)해야 단단한 시스템을 유지할 수 있다.
2. **무한 임베딩(Embedding) 안티패턴**: 조인을 피하겠다고 하나의 JSON 문서 안에 수만 건의 자식 배열 데이터를 무제한 밀어 넣으면, 문서 최대 크기 제한(예: 몽고DB 16MB)에 걸리거나 단건 조회 시 엄청난 네트워크 I/O 부하가 발생한다.
   - **판단**: 일대다(1:N) 관계에서 N이 수천 건 이상 지속적으로 증가하는 무한 확장 구조(예: 게시글의 끝없는 댓글)라면 임베딩을 피하고, 정규화처럼 문서 참조(Reference) 방식으로 분리 설계해야 한다.
3. **직렬화/역직렬화 CPU 병목**: 초당 수만 건의 통신이 발생하는 마이크로서비스 환경에서 텍스트 기반의 JSON이나 XML 파싱은 무거운 CPU 연산을 수반한다.
   - **판단**: 내부 백엔드 간의 고속 통신망(RPC)에서는 반정형 텍스트(JSON)를 피하고, 프로토콜 버퍼(Protocol Buffers, gRPC)나 아파치 아브로(Avro) 같은 이진(Binary) 직렬화 포맷을 사용하여 파싱 오버헤드를 극적으로 제거해야 한다.

[반정형 데이터의 유효성 검증 아키텍처 운영 플로우]
```text
[Client API Request: JSON]
       ↓
[API Gateway / BFF]
       ↓
[JSON Schema Validator (애플리케이션 계층)]  <-- 핵심: DB 밖에서 구조 통제
 ├─ 필수 키 누락? ──(Yes)──> HTTP 400 Bad Request 리턴
 └─ 타입 일치?    ──(Yes)──> 통과
       ↓
[Document DB (MongoDB / ElasticSearch)]
 (저장 시점에서는 무조건 성공 보장)
```
이 의사결정 플로우는 반정형 데이터의 스키마-리스(Schemaless) 특성이 가져오는 위험성을 방어하는 표준 아키텍처다. 유연성을 누리면서도 데이터의 품질을 유지하기 위해, 데이터베이스의 제약 조건 엔진이 하던 역할을 애플리케이션 게이트웨이나 서비스 로직 내의 Validator 모듈로 승격시켜 검증 책임을 이동시켰다. 

📢 **섹션 요약 비유**: 반정형 DB는 입구에 경비원(스키마)이 없는 클럽과 같습니다. 누구든 들어올 수 있어 자유롭지만, 난장판이 되는 것을 막으려면 클럽 문 밖(애플리케이션)에서 미리 신분증(JSON Schema)을 철저히 검사해야 합니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)
반정형 데이터는 현대 애플리케이션의 빠른 반복(Iteration) 배포를 가능하게 한 1등 공신이다.

| 관점 | 반정형 데이터의 도입 가치 | 정량/정성적 지표 |
|:---|:---|:---|
| 개발 생산성 | 사전 DDL 작업 생략 및 즉각적인 모델 매핑 | 신규 기능 개발 리드타임 40% 이상 단축 |
| 시스템 확장성 | Scale-out 구조에 적합, MSA 독립적 진화 지원 | 서비스 간 결합도(Coupling) 최소화 |

미래의 반정형 데이터 기술은 RDBMS 진영과의 융합(Convergence)으로 나아가고 있다. PostgreSQL, MySQL 등 전통적 정형 DB들도 내부적으로 JSON 데이터 타입을 네이티브로 지원하며, JSON 내의 특정 키에 GIN(Generalized Inverted Index) 등을 걸어 고속 검색을 지원하는 하이브리드 형태(Multi-model DB)로 발전하고 있다. 즉, 시스템 간 통신과 가변 속성은 반정형 포맷으로, 엄격한 원장 결제는 정형 구조로 한 DB 내에서 병행 처리하는 것이 기술사적 아키텍처의 표준 모델이다.

📢 **섹션 요약 비유**: 반정형 데이터 기술은 과거 딱딱한 뼈대(정형)만 있던 로봇에, 자유자재로 늘어나고 변형되는 인공 근육(JSON)을 달아주어 어디든 유연하게 적응할 수 있게 만든 혁신입니다.

### 📌 관련 개념 맵 (Knowledge Graph)
* JSON (JavaScript Object Notation) | 가벼운 텍스트 기반의 데이터 교환을 위한 반정형 포맷 표준
* Document DB | 반정형 데이터(JSON, BSON 등)를 계층형 문서 구조로 직관적으로 저장하는 NoSQL
* Schema-on-Read | 저장할 때는 원시 포맷으로 두고 읽을 때(조회 시) 구조를 해석하는 데이터 처리 사상
* 직렬화 (Serialization) | 메모리 상의 객체(Object)를 전송 가능한 반정형 바이트/문자열로 변환하는 과정
* 마이크로서비스 (MSA) | 작고 독립적인 서비스들이 API와 반정형 데이터(JSON)로 상호 통신하는 아키텍처

### 👶 어린이를 위한 3줄 비유 설명
1. 반정형 데이터는 장난감 상자마다 "이건 로봇", "저건 자동차"라고 이름표(태그)가 붙어있는 것과 같아요.
2. 상자 모양이나 크기를 미리 정해놓지 않아도, 이름표만 보면 안에 뭐가 들었는지 쉽게 알 수 있죠.
3. 그래서 새로운 종류의 장난감이 생겨도 상자만 하나 더 가져와서 이름표를 붙이면 되니까 아주 편리하답니다!
