# 빌드, 릴리스, 실행 단계의 엄격한 분리 (Build, Release, Run Strict Separation)
#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소스 코드에서부터 최종 구동에 이르는 전체 라이프사이클을 빌드(Build), 릴리스(Release), 실행(Run)이라는 세 가지 상호 배타적인 단계로 엄격하게 단절시켜 관리하는 클라우드 네이티브 아키텍처 원칙이다.
> 2. **가치**: 런타임 환경에서 코드가 수정되는 것을 원천 차단하여 불변성(Immutability)을 확보하며, 문제 발생 시 즉각적이고 예측 가능한 과거 버전으로의 롤백 및 무중단 배포 속도를 극대화한다.
> 3. **융합**: 컨테이너 기술(Docker), CI/CD 파이프라인 자동화 도구(Jenkins, GitHub Actions), 그리고 쿠버네티스(Kubernetes)의 컨테이너 라이프사이클 관리와 융합되어 선언적 인프라 운영의 근간을 이룬다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

과거의 모놀리식(Monolithic) 시스템 운영에서는 라이브 서버에 직접 접속하여 코드를 수정하거나 패치를 적용하는 일이 빈번했다. 이러한 방식은 런타임 환경과 빌드 환경이 혼재되는 결과를 낳았고, 이로 인해 서버 간 설정이 불일치하는 구성 편류(Configuration Drift) 현상을 야기했다. 12 팩터 앱(The Twelve-Factor App) 방법론은 이러한 불안정성을 타파하기 위해 '빌드, 릴리스, 실행' 단계의 엄격한 분리를 주장한다. 빌드 단계는 소스 코드를 실행 가능한 아티팩트로 변환하고, 릴리스 단계는 이 아티팩트와 환경 설정(Config)을 결합하며, 실행 단계는 최종 릴리스를 런타임 환경에서 가동하는 것으로 철저히 구분된다. 이 과정을 통해 런타임에서 발생할 수 있는 부수 효과를 차단하고, 소프트웨어 배포의 멱등성을 보장할 수 있다.

다음 도식은 세 단계가 혼재되었을 때 발생하는 장애 전파의 한계를 보여준다. 코드를 배포하는 중간에 설정이 섞이게 되면, 특정 노드에서만 장애가 발생하는 스파게티 운영 구조가 만들어진다.

```text
이 도식은 기존 레거시 배포 환경에서 코드 빌드와 설정 런타임 주입이 혼재되어 구성 편류(Configuration Drift)가 발생하는 악순환을 보여준다.

[레거시 혼재 배포 환경의 병목]
   ↓
[코드 커밋] => [서버 1: 빌드+실행] (수동 패치 발생) ──+ 
   ↓                                                | => 서버 간 상태 불일치
[환경 변수] => [서버 2: 빌드+실행] (오류 상태)     ──+    (어느 시점으로 롤백할지 추적 불가)
```

이 그림의 핵심은 빌드와 실행이 같은 공간과 시간대에 발생함으로써 롤백의 기준점이 사라졌다는 점이다. 따라서 장애 발생 시 어떤 버전의 아티팩트에 어떤 설정이 결합되었는지 추적하기 어려우며, 전체 복구 시간(MTTR)에 치명적인 악영향을 미친다. 실무에서는 배포 주기 단축을 위해 이러한 결합 상태를 반드시 끊어내야만 자동화된 파이프라인 구성이 가능해진다.

> 📢 **섹션 요약 비유**: 마치 자동차 공장에서 부품 조립(빌드), 고객 맞춤형 옵션 장착(릴리스), 주행(실행) 단계를 완벽히 분리해 주행 중인 차에서 엔진을 뜯어고치는 위험천만한 상황을 원천 차단하는 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

빌드, 릴리스, 실행 단계의 아키텍처는 데이터가 한 방향으로만 흐르는 단방향 파이프라인(Uni-directional Pipeline) 구조를 가진다. 각 단계는 이전 단계의 산출물을 입력으로 받아 변형 없이 사용하며, 역방향으로의 수정은 절대 허용되지 않는다.

| 구성 요소 | 역할 | 내부 동작 메커니즘 | 실무 매핑 도구 | 비유 |
|:---|:---|:---|:---|:---|
| **Build (빌드)** | 코드 아티팩트 생성 | Git 저장소의 특정 커밋을 가져와 종속성을 설치하고 컴파일하여 실행 가능한 바이너리나 컨테이너 이미지로 고정시킴. | CI 런너, Docker Build | 식재료 가공 |
| **Release (릴리스)** | 아티팩트 + 설정 결합 | 빌드된 불변 아티팩트와 특정 배포 환경(Dev, Staging, Prod)의 Config(환경 변수 등)를 고유 ID와 함께 매핑하여 저장. | Helm Release, GitOps | 밀키트 포장 |
| **Run (실행)** | 프로세스 가동 | 특정 릴리스 ID를 기반으로 런타임 환경에 프로세스를 스폰(Spawn)하고 트래픽을 처리함. 코드가 비정상 종료되면 자동 재시작. | K8s Pod, AWS ECS | 조리 및 서빙 |

아래의 계층 구조도는 이 세 단계가 어떻게 단방향으로 결합되어 불변의 릴리스를 생성하는지를 명확히 보여준다.

```text
이 도식은 소스 코드와 설정 정보가 어떻게 독립적으로 관리되다가 릴리스 단계에서 고유한 식별자를 가진 배포 패키지로 결합되는지를 보여준다.

┌────────────── Pipeline Architecture ──────────────┐
│                                                   │
│  [Codebase]      [Config (Env)]                   │
│      │                 │                          │
│      ▼                 │                          │
│  ┌───────┐             │                          │
│  │ Build │             │                          │
│  └───────┘             ▼                          │
│      │ (Artifact)  ┌─────────┐   (Release ID)     │
│      └────────────▶│ Release │──────────┐         │
│                    └─────────┘          │         │
│                                         ▼         │
│                                     ┌───────┐     │
│                                     │  Run  │     │
│                                     └───────┘     │
└───────────────────────────────────────────────────┘
```

이 흐름의 핵심은 빌드 단계와 릴리스 단계가 구조적으로 분리되어, 하나의 빌드 산출물을 가지고 QA, Staging, Production 등 여러 환경의 릴리스를 생성할 수 있다는 점이다. 이러한 배치는 빌드 시간을 크게 단축시키며, 환경 간 격차(Dev/Prod Parity)를 최소화한다. 실무에서는 이러한 구조가 없을 경우 매 환경마다 코드를 새로 빌드해야 하여, 컴파일러 버전에 따른 예기치 않은 오류가 프로덕션에 전파될 위험이 있다.

실제로 쿠버네티스 환경에서는 이러한 원리가 YAML 매니페스트를 통해 다음과 같이 구현된다.
```yaml
# 실행(Run) 단계를 선언하는 릴리스(Release) 결합 코드
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
  annotations:
    release-id: "v1.2.3-prod" # 릴리스 고유 식별자
spec:
  template:
    spec:
      containers:
      - name: app
        image: myrepo/app:v1.2.3 # 빌드(Build) 단계의 불변 아티팩트
        envFrom:
        - configMapRef:
            name: prod-config # 릴리스 환경 설정(Config)
```

> 📢 **섹션 요약 비유**: 마치 인쇄소에서 금형(빌드)을 한 번 파놓고, 잉크 색깔(설정)만 바꿔가며 수만 장의 서로 다른 전단지(실행)를 일관되게 찍어내는 공정과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

빌드/릴리스/실행의 엄격한 분리 방식과 전통적인 혼합형 배포 방식을 구조적으로 비교하면, 왜 현대의 SRE(Site Reliability Engineering) 조직이 이 패턴을 필수적으로 강제하는지 명확해진다.

| 비교 항목 | 전통적 혼합형 (Mutable) | 단계 분리형 (Immutable - 12 Factor) | 트레이드오프 판단 포인트 |
|:---|:---|:---|:---|
| **배포 단위** | 소스 코드 파일의 직접 동기화 (rsync, FTP) | 고유 ID가 부여된 릴리스 컨테이너/바이너리 | 롤백 속도 및 정합성 보장 여부 |
| **환경 의존성** | 서버 내 OS 환경에 강하게 결합됨 | 빌드된 아티팩트 내에 종속성이 완벽히 격리됨 | 확장 시 신규 서버 세팅 소요 시간 |
| **장애 복구(MTTR)**| 원인 파악 및 코드 수동 롤백 (시간 예측 불가) | 이전 릴리스 ID로 컨테이너 즉각 재시작 (초 단위) | 운영 리스크 대처 역량 |
| **스케일 아웃** | 설정이 각기 달라 수평 확장이 몹시 까다로움 | 동일 릴리스를 복제만 하면 되므로 매우 단순함 | 트래픽 스파이크 대응력 |

아래의 비교 흐름도는 두 패턴에서 장애 발생 시 롤백에 소요되는 타이밍과 복구 메커니즘의 차이를 나타낸다.

```text
이 타이밍 그래프는 장애 발생 시 전통적 방식과 분리형 릴리스 방식 간 복구 레이턴시의 극명한 차이를 보여준다.

[전통적 방식 (수동/혼합형 롤백)]
장애 발생 --(10분:원인분석)--> [코드 재수정] --(5분:재빌드/배포)--> 복구 완료 (MTTR: 15분+)

[분리형 방식 (릴리스 ID 기반 롤백)]
장애 발생 --(1분:알람)--> [Release vN-1 재실행] -----------------> 복구 완료 (MTTR: 1~2분)
                              ▲
                       즉각적인 상태 복원
```

이 매트릭스와 도식의 핵심은, 단계 분리형 방식이 평상시의 배포 파이프라인 구축 비용(초기 복잡도)은 높지만 위기 상황에서의 회복 탄력성(Resiliency)을 기하급수적으로 끌어올린다는 점이다. 복구 과정에 코드를 다시 엮는(Re-build) 과정을 생략하고 이미 검증된 '과거의 릴리스 패키지'를 그대로 실행만 하면 되기 때문이다. 따라서 실시간 트래픽 처리가 중요한 B2C 서비스에서는 이 분리 원칙이 필수적인 안전망으로 작용한다.

> 📢 **섹션 요약 비유**: 저장하지 않고 문서 작업을 하다 날려먹고 처음부터 다시 쓰는 것(혼합형)과, 매 분마다 자동 저장된 버전 기록(히스토리) 버튼을 눌러 1초 만에 과거로 되돌아가는 것(분리형)의 차이입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

이 원칙을 실무 파이프라인에 적용할 때는, 어느 환경에서 문제가 터져도 '빌드' 단계로 되돌아가지 않도록 설계하는 것이 관건이다. 개발자들이 가장 흔히 저지르는 안티패턴 중 하나가 스테이징 환경에서 버그를 발견했을 때 스테이징 서버에 접속해 코드를 임의로 핫픽스(Hotfix)하는 행위다. 

**실무 의사결정 시나리오 및 운영 플로우**
1. **취약점 긴급 패치 상황**: 라이브 서버(Run)에서 보안 취약점이 발견된 경우, 라이브 서버를 멈추고 직접 패치하지 않는다. 반드시 로컬(Code)에서 취약점을 패치한 코드를 푸시하고, 새로운 빌드(Build)를 거쳐 새 릴리스(Release) 번호를 따고, 이를 프로덕션에 교체 투입(Run)해야 한다.
2. **설정 오류로 인한 장애**: DB 패스워드나 API 엔드포인트 설정 오류라면 코드를 다시 빌드할 필요가 없다. 아티팩트는 그대로 둔 채 Config만 수정한 새로운 릴리스 객체를 생성하여 실행 단계로 넘기면 파이프라인의 레이턴시를 크게 단축시킬 수 있다.

```text
이 의사결정 트리는 장애 유형에 따라 파이프라인의 어느 지점부터 재시작해야 하는지를 판단하는 실무적 기준을 제시한다.

[장애 감지]
   │
   ├─▶ (비즈니스 로직 / 버그 인가?) ──[YES]──▶ 【 Code / Build 단계부터 재시작 】
   │                                          (신규 이미지 태그 생성)
   │
   └─▶ (환경 변수 / 키값 오류 인가?) ─[YES]──▶ 【 Release 단계만 재시작 】
                                              (기존 이미지 + 신규 Config 결합)
```

이 그림의 핵심은 장애의 속성에 따라 피드백 루프의 길이를 동적으로 조절해야 한다는 점이다. 모든 수정사항을 맹목적으로 빌드 단계부터 태우면 CI/CD 병목이 발생하여 변경 리드 타임(Lead Time)이 길어진다. 반면, 런타임 수정을 허용하면 시스템의 상태 추적성이 완전히 상실된다. 따라서 실무에서는 CI 도구(Jenkins 등)와 CD 도구(ArgoCD 등)를 분리하여, 빌드 산출물을 저장소(Container Registry)에 격리 보관하고 릴리스 단계가 이를 참조하도록 아키텍처를 강제해야 한다.

**도입 체크리스트**
- [ ] 실행 중인 서버(Pod, VM)에 SSH 등으로 접속하여 코드나 파일을 직접 수정할 수 없도록 권한이 차단되어 있는가? (불변성)
- [ ] 릴리스된 결과물은 유일한 타임스탬프나 해시 태그(예: v1.0.3-43a9b)로 추적 가능한가?
- [ ] 이전 릴리스 버전을 실행하는 데 걸리는 롤백 시간이 5분 이내로 자동화되어 있는가?

> 📢 **섹션 요약 비유**: 전투 중 무기가 고장 났을 때, 전방(런타임)에서 드라이버를 꺼내 무기를 고치는 것이 아니라, 후방 기지(빌드)에서 완벽히 수리된 새 무기 패키지를 즉시 보급(릴리스)받아 교체하는 전술적 규율과 같습니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

빌드, 릴리스, 실행 단계의 엄격한 분리는 클라우드 네이티브 생태계에서 서비스 무중단 배포와 스케일 아웃을 가능하게 하는 토대이다. 이 원칙을 준수하면 DORA 메트릭스에서 권장하는 '배포 빈도(Deployment Frequency) 증가'와 '변경 실패율(Change Failure Rate) 감소'라는 상충되는 두 마리 토끼를 동시에 잡을 수 있다.

| 지표 | 도입 전 (레거시) | 도입 후 (12 Factor 분리) | 향상 포인트 |
|:---|:---|:---|:---|
| **가시성** | 어떤 코드가 라이브에 있는지 불명확 | Git 태그와 릴리스 ID의 1:1 매핑 | 감사(Audit) 및 컴플라이언스 대응력 확보 |
| **안정성** | 인간의 수동 개입에 의한 휴먼 에러 | 코드와 설정의 분리로 인한 멱등성 보장 | 인지 부하 감소 및 SRE 토일(Toil) 최소화 |

미래의 데브옵스 환경에서는 GitOps와 쿠버네티스의 선언적 릴리스 관리가 더욱 고도화되어, 빌드와 릴리스 파이프라인조차 AI 기반으로 이상 탐지 및 자동 치유(Auto-remediation)가 이뤄지는 방향으로 진화할 것이다. 결국 이 3단계 분리 원칙은 단순한 파이프라인 관리 기법을 넘어, 인프라를 일관된 코드로 통제하는 불변 인프라(Immutable Infrastructure) 패러다임의 가장 기본적이고 굳건한 표준 규격(Standard)으로 지속될 것이다.

> 📢 **섹션 요약 비유**: 한 번 굳어지면 모양을 바꿀 수 없는 도자기(빌드)처럼, 완벽히 통제된 공정만이 예기치 못한 환경의 변화 속에서도 절대 부서지지 않는 견고한 신뢰(SRE)를 구축하는 지름길이 됩니다.
