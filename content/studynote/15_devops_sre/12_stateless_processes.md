# 무상태 프로세스 (Stateless Processes)
#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 애플리케이션 프로세스는 단일 트랜잭션 처리 과정에서 메모리나 디스크에 어떠한 상태 정보(세션, 파일 등)도 영구적으로 저장하지 않으며, 모든 상태는 외부 백엔드 서비스(DB, Cache)에 위임하는 설계 패러다임이다.
> 2. **가치**: 특정 서버 인스턴스에 종속되지 않으므로 트래픽 급증 시 자유롭게 프로세스를 수평 확장(Scale-out)할 수 있고, 장애 시 언제든 프로세스를 폐기/재생성할 수 있는 탄력성을 확보한다.
> 3. **융합**: RESTful API 통신, JWT(JSON Web Token) 기반 인증, 쿠버네티스의 레플리카셋(ReplicaSet) 오토스케일링과 완벽히 융합되어 분산 클라우드 환경의 동시성 제어를 지원한다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

전통적인 웹 애플리케이션 환경에서는 사용자가 로그인하면 그 세션(Session) 정보를 특정 웹 서버(예: Tomcat)의 메모리에 저장하는 상태유지(Stateful) 방식이 일반적이었다. 이 경우, 로드 밸런서는 동일한 사용자의 요청을 항상 같은 서버로 보내야 하는 스티키 세션(Sticky Session) 제약에 묶이게 된다. 이는 특정 서버에 부하가 몰리거나 서버가 다운되었을 때 사용자의 세션이 날아가는 치명적인 단일 장애점(SPOF) 문제를 야기한다.

이러한 한계를 극복하기 위해 12 팩터 앱(The Twelve-Factor App)은 프로세스가 '무상태(Stateless)'로 실행되어야 함을 강력히 규정한다. 무상태 프로세스는 어떠한 사용자 상태도 프로세스 내부에 기억하지 않는다. 상태 보존이 필요한 데이터는 Redis, Memcached, 혹은 RDBMS와 같은 '상태를 전담하는 외부 백엔드 서비스(Backing Services)'로 아웃소싱해야 한다. 

```text
이 도식은 스티키 세션 기반의 상태 유지(Stateful) 환경이 가지는 구조적 병목과 트래픽 불균형 문제를 시각화한다.

[Stateful 환경의 한계]
               ┌─▶ [Web Server 1 (Session A, B)] (과부하)
[Load Balancer]┼─▶ [Web Server 2 (Session C)]
 (Sticky Rule) └─▶ [Web Server 3 (Empty)] (유휴 자원 낭비)
                      * Server 1 다운 시 사용자 A, B는 강제 로그아웃 됨.
```

이 그림의 핵심은 상태(State)가 프로세스에 결합되어 있을 때 로드 밸런싱의 효율이 급격히 떨어지며 자원 활용의 비대칭성이 발생한다는 점이다. 상태를 쥐고 있는 서버는 함부로 재시작할 수도, 확장할 수도 없다. 클라우드 네이티브 환경에서 이러한 결합은 자동 복구 및 탄력적 확장을 불가능하게 만드는 가장 큰 장애물이다.

> 📢 **섹션 요약 비유**: 마치 지정된 전담 은행원에게만 가야 업무 처리가 가능해서 그 은행원 앞에만 줄이 길게 늘어서는 구형 은행(Stateful)에서, 번호표만 뽑으면 어떤 창구 직원을 만나도 전산망(외부 DB)을 통해 똑같이 빠르고 원활하게 업무를 처리할 수 있는 현대식 은행(Stateless)으로 진화한 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

무상태 프로세스 아키텍처는 비즈니스 로직을 처리하는 '연산 계층'과 데이터를 보관하는 '저장 계층'을 물리적, 논리적으로 완벽히 분리한다. 클라이언트의 모든 요청은 독립적이며, 요청을 처리하는 데 필요한 모든 컨텍스트 정보는 요청 자체(헤더, 페이로드)에 포함되어 있거나, 외부 데이터스토어에서 그때그때 조회된다.

| 구성 요소 | 역할 | 내부 동작 메커니즘 | 관련 기술/프로토콜 | 비유 |
|:---|:---|:---|:---|:---|
| **Stateless App** | 연산 및 로직 수행 | 요청 수신 시 로컬 메모리 참조 없이 독립적 연산만 수행 후 결과 반환 | Spring Boot, Express, Go | 창구 직원 (머리) |
| **Load Balancer** | 트래픽 라우팅 분산 | 세션 제약 없이 라운드 로빈 등 단순 알고리즘으로 유휴 서버에 균등 분배 | AWS ALB, Nginx | 번호표 기계 |
| **Cache Store** | 세션/임시 상태 보관 | 애플리케이션이 필요로 하는 세션 토큰이나 장바구니 상태를 초고속 인메모리로 서빙 | Redis, Memcached | 단기 임시 보관소 |
| **Persistent DB** | 영구 데이터 보관 | 영속성이 필요한 최종 트랜잭션 상태(결제, 유저 정보) 저장 및 무결성 보장 | PostgreSQL, MongoDB | 금고 (장부) |

아래의 흐름도는 무상태 아키텍처에서 사용자 요청이 어떻게 상태의 결합 없이 처리되며, 어떠한 파드(Pod)로 라우팅되어도 동일한 결과를 반환하는지를 나타낸다.

```text
이 도식은 무상태 기반 프로세스들이 어떻게 백엔드 서비스(Redis)에 상태를 위임함으로써 완벽한 병렬 확장성과 장애 격리를 달성하는지를 보여준다.

┌───────────────── 무상태 아키텍처 흐름도 ──────────────────┐
│                                                           │
│ [Client 1] (JWT Token 보유)                               │
│      │                                                    │
│      ▼                                                    │
│ [Load Balancer] ──(Round Robin)──┐                        │
│      │                           │                        │
│      ▼                           ▼                        │
│ ┌─────────┐                 ┌─────────┐                   │
│ │ App #1  │  (무상태)       │ App #2  │  (무상태)         │
│ │(No State)                 │(No State)                   │
│ └─────────┘                 └─────────┘                   │
│      │                           │                        │
│      └───────────▶ ┌────────┐ ◀──┘                        │
│                    │ Redis  │ (세션/상태 저장 전담)       │
│                    └────────┘                             │
└───────────────────────────────────────────────────────────┘
```

이 도식의 핵심 트레이드오프는 상태 위임에 따른 네트워크 I/O 비용이다. App #1과 App #2가 자체 메모리를 쓰지 않는 대신, 사용자 상태를 검증하기 위해 지속적으로 Redis를 호출해야 한다. 따라서 프로세스의 수평 확장은 무한히 가능해졌지만, 반대급부로 중앙 집중된 Redis나 DB 병목 현상이 새로운 과제로 대두된다. 실무에서는 이러한 백엔드 호출 지연을 최소화하기 위해 읽기 복제본(Read Replica) 확장이나 효율적인 데이터 구조화가 필수적으로 요구된다.

> 📢 **섹션 요약 비유**: 어떤 요리사(프로세스)가 요리를 하더라도 레시피와 재료를 자신의 앞치마 주머니에 숨겨두지 않고, 항상 공용 대형 냉장고(Redis)에서 꺼내 쓰도록 강제함으로써 주방에 요리사 100명을 언제든 교체 투입할 수 있게 만든 시스템입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

무상태 구조는 마이크로서비스(MSA)와 클라우드 네이티브 설계에서 가장 강력한 무기지만, 전통적 상태유지 구조와의 차이점과 부작용을 이해해야 올바른 설계가 가능하다.

| 비교 항목 | Stateful (상태 유지) 프로세스 | Stateless (무상태) 프로세스 | 실무적 판단 포인트 |
|:---|:---|:---|:---|
| **스케일 아웃** | 매우 어려움 (데이터 마이그레이션 및 동기화 락 발생) | 매우 쉬움 (단순 복제, 제약 없음) | 트래픽 변동성이 클수록 Stateless 필수 |
| **요청/응답 레이턴시** | 빠름 (로컬 메모리/캐시에서 즉각 조회) | 약간의 지연 발생 (외부 DB/Cache 네트워크 통신 필요) | 극단적 저지연 요구 시 로컬 캐시 융합 고려 |
| **인증 관리** | 서버 측 세션 쿠키 기반 (서버가 관리) | 클라이언트 측 JWT 기반 (서버는 검증만 수행) | 토큰 만료 및 탈취 방어 로직 설계 난이도 |
| **장애 영향도 (Blast Radius)** | 서버 다운 시 해당 서버 접속자의 모든 정보 증발 | 서버 다운 시 다른 서버로 즉시 라우팅 (무손실) | 고가용성(HA) 및 SRE 회복 탄력성 목표 |

특히 보안/인증 패러다임과의 융합 관점에서 볼 때, 무상태 프로세스는 JWT(JSON Web Token) 및 OAuth 2.0 프로토콜과 결합하여 시너지를 폭발시킨다. 서버가 상태를 관리할 필요 없이, 암호학적으로 서명된 JWT 토큰 자체에 권한 정보가 내포되어 있으므로 어떤 서버에 요청이 떨어져도 서명 검증만으로 즉각적인 비즈니스 로직 수행이 가능해진다.

```text
이 매트릭스는 스케일링 전략에 따른 트레이드오프를 보여준다. 상태를 버리는 대신 네트워크 레이턴시 비용을 지불해야 함을 암시한다.

[스케일 아웃 유연성]
 ▲
 │                          ● Stateless (Cloud Native)
 │                      (무한 확장, 네트워크 통신 비용 증가)
 │
 │
 │      ● Stateful (Legacy)
 │ (확장성 제한, 단일 요청 속도 극대화)
 └─────────────────────────────────────────▶ [네트워크 I/O 의존성]
```

이 그래프의 핵심은 무상태 애플리케이션으로 넘어가면 필연적으로 네트워크 통신량이 증가한다는 사실이다. 백엔드 서비스(DB/캐시)의 성능 병목이 전체 애플리케이션의 처리량(TPS)을 결정짓는 구조로 전환된다. 따라서 실무에서는 데이터 파티셔닝, 샤딩, 커넥션 풀(Connection Pool) 관리를 통해 외부 서비스의 지연을 통제해야 한다.

> 📢 **섹션 요약 비유**: 로컬 하드디스크에 파일을 저장해두면(Stateful) 내 컴퓨터에서 열 때는 빠르지만 다른 자리에서는 쓸 수 없고 컴퓨터가 망가지면 자료가 날아가는 반면, 구글 드라이브(Stateless/DB)에 저장하면 어디서든 동기화해서 열 수 있지만 인터넷 연결 속도에 의존해야 하는 원리와 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무 환경에서 '완전한 무상태'를 구축하는 것은 생각보다 많은 규율을 요구한다. 파일 업로드나 로컬 디스크 캐싱과 같이 개발자들이 편의상 로컬 상태를 만드는 안티패턴을 철저히 막아야 한다.

**실무 의사결정 시나리오 및 운영 플로우**
1. **파일 업로드 기능 설계**: 사용자가 업로드한 이미지를 웹 서버의 `/tmp/uploads` 로컬 디렉터리에 저장하면 안 된다. 서버가 종료되거나 다른 서버가 응답할 때 이미지를 찾을 수 없기 때문이다. 반드시 AWS S3나 MinIO 같은 외부 객체 스토리지(Object Storage)로 직접 전송하고 파일 메타데이터만 DB에 남기도록 강제해야 한다.
2. **트래픽 스파이크 대응 (HPA 구성)**: 쿠버네티스의 HPA(Horizontal Pod Autoscaler)를 구성할 때, 애플리케이션 파드가 무상태여야만 CPU 부하 감지 시 파드 수를 1개에서 100개로 수 초 안에 늘리고 안전하게 줄일 수 있다.

```text
이 의사결정 트리는 상태 정보의 성격에 따라 이를 프로세스 내부에 둘지, 어떤 백엔드에 위임할지를 판단하는 가이드를 제공한다.

[상태 데이터 발생]
   │
   ├─▶ (일회성 연산용 / 즉시 휘발 인가?) ─[YES]─▶ 【 프로세스 메모리 사용 (허용) 】
   │
   ├─▶ (사용자 세션 / 인증 정보 인가?) ───[YES]─▶ 【 JWT 토큰 변환 또는 Redis 저장 】
   │
   └─▶ (업로드 파일 / 영속적 자산인가?) ──[YES]─▶ 【 AWS S3 등 외부 스토리지 위임 】
```

이 흐름의 병목 지점은 프로세스가 상태를 너무 잘게 쪼개어 외부 캐시를 과도하게 호출할 때 발생한다(Chatty I/O 현상). 예를 들어 1번의 API 응답을 위해 Redis를 50번 호출한다면, 무상태성은 달성했으나 심각한 성능 저하를 초래한다. 따라서 실무 판단에서는 적절한 응집도(Aggregated Payload)를 통해 I/O 왕복 횟수를 통제하고, 분산 락(Distributed Lock) 등 동시성 제어 메커니즘을 외부 DB 단에서 처리하도록 아키텍처를 방어해야 한다.

**도입 체크리스트**
- [ ] 서버 인스턴스가 예고 없이 즉각 삭제(Kill)되더라도 사용자 데이터 손실이 전혀 없는가?
- [ ] 여러 인스턴스에 트래픽을 분산할 때 스티키 세션(Sticky Session) 설정 없이 라운드 로빈 처리가 가능한가?
- [ ] 세션 검증을 위한 Redis/DB가 단일 장애점(SPOF)이 되지 않도록 고가용성 클러스터로 구성되었는가?

> 📢 **섹션 요약 비유**: 개인 금고(로컬 디스크)를 아예 없애버리고 모든 귀중품은 무조건 중앙의 거대한 메인 은행 금고(AWS S3, DB)에 보관하도록 강제하여, 도둑이 빈 사무실(프로세스)을 털어가도 아무런 타격이 없게 만드는 무소유의 철학과 같습니다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

무상태 프로세스 원칙을 적용하면 클라우드 환경에서 시스템 확장성(Scalability)과 내결함성(Fault Tolerance)이 비약적으로 상승한다. 어떠한 형태의 트래픽 변동에도 오토스케일링을 통해 즉각적인 컴퓨팅 자원 확장이 가능해지며, 장애 발생 시 원인 분석을 위해 고장 난 서버를 살려둘 필요 없이 즉시 폐기하고 새 인스턴스로 교체할 수 있다.

| 지표 | 상태 유지 (Stateful) | 무상태 (Stateless) | 향상 포인트 |
|:---|:---|:---|:---|
| **스케일링 속도** | 분~시간 단위 (동기화) | 초 단위 (단순 복제) | 대규모 트래픽 스파이크 즉각 대응 |
| **운영 인지 부하** | 특정 상태 노드 관리 압박 | 클러스터 관점으로 추상화 | SRE 팀의 오퍼레이션 단순화(Toil 제거) |

미래의 서버리스(Serverless, AWS Lambda 등) 환경은 이러한 무상태성을 극단적으로 추구하는 방향으로 나아가고 있다. 함수(Function)가 1밀리초 단위로 생성되고 사라지는 서버리스 환경에서는 상태 저장이 원천적으로 불가능하기 때문이다. 결론적으로 무상태 프로세스는 현대 분산 컴퓨팅이 도달해야 할 필수적 표준(Standard)이며, 개발자는 모든 코드를 작성할 때 이 코드가 어떤 서버에서 실행되든 일관된 동작을 보장하도록 상태에 대한 미련을 완전히 버려야만 한다.

> 📢 **섹션 요약 비유**: 물이 담긴 무거운 그릇(Stateful)을 들고 뛰는 대신, 호스 파이프(Stateless)만 들고 뛰면서 필요할 때마다 언제든 상수도(DB)에 연결해 물을 뿜어낼 수 있는 극강의 기동성을 확보하는 최종 종착지입니다.
