# 502. 비순차적 실행 (OoO Execution)과 이슈 큐

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로그램에 적힌 명령어의 '순서(In-Order)'를 무시하고, 실행 준비(데이터 의존성 해결)가 먼저 끝난 명령어부터 하드웨어가 런타임에 동적으로 먼저 실행(Out-of-Order)해버리는 파이프라인 극대화 기술이다.
> 2. **가치**: 캐시 미스나 느린 연산(나눗셈 등)으로 인해 발생한 파이프라인 멈춤(Stall) 시간 동안 멍하니 대기하지 않고, 뒤에 있는 명령어를 당겨와 유휴 실행 유닛(ALU)을 100% 가동시킴으로써 IPC(사이클당 명령어 수)를 획기적으로 높인다.
> 3. **융합**: 레지스터 리네이밍(Register Renaming), 슈퍼스칼라(Superscalar), 분기 예측(Branch Prediction)과 결합되어 현대 고성능 x86 및 ARM 프로세서 성능의 중추를 이룬다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 비순차적 실행(OoO, Out-of-Order Execution)은 CPU가 메모리에 적힌 순서대로(순차적, In-Order) 명령어를 인출(Fetch)하고 해독(Decode)하되, 실행(Execute) 단계에서는 데이터가 준비된 순서대로 마구잡이로 연산한 뒤, 최종 결과 저장(Retire/Commit)은 다시 원래 프로그램 순서대로 정렬하여 소프트웨어적 의미(Semantics)를 완벽히 유지하는 아키텍처다.
- **💡 비유**: 패스트푸드점 주방과 같다. 주문이 "1. 수제버거(10분 소요), 2. 콜라(1초 소요), 3. 감자튀김(3분 소요)" 순서로 들어왔다. 순차적 실행(In-Order)은 앞사람의 수제버거가 나올 때까지 콜라와 감자튀김을 절대 만들지 않고 모두가 기다린다. 반면 **비순차적 실행(OoO)**은 주문은 순서대로 받되(In-Order Fetch), 금방 끝나는 콜라와 감튀부터 먼저 튀겨내고(Out-of-Order Execute), 마지막에 쟁반에 담아 손님에게 내어줄 때만 원래 주문 번호 순서대로(In-Order Commit) 내어주어 주방 회전율을 극대화한다.
- **등장 배경 및 발전 과정**:
  1. **In-Order 스톨(Stall)의 낭비**: 파이프라인 스칼라 구조에서는 하나의 명령어가 메모리에서 데이터를 가져오지 못해 멈추면(Cache Miss), 그 뒤에 있는 명령어들도 줄줄이 멈춰 서야(Pipeline Bubble) 했다.
  2. **토마술로 알고리즘 (Tomasulo's Algorithm, 1967)**: IBM 360/91 부동소수점 유닛에서 명령어의 순서를 재배치하여 데이터 의존성을 하드웨어적으로 해결하는 혁신적인 알고리즘이 최초로 구현되었다.
  3. **슈퍼스칼라와의 결합 (1990년대 이후)**: 명령어 폭(Width)이 넓은 슈퍼스칼라 구조가 도입되면서 빈 연산기가 많아졌다. 이 빈 연산기를 채우기 위해 인텔 펜티엄 Pro부터 본격적으로 OoO 엔진이 도입되었고, 현재는 모든 고성능 칩의 기본이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (Tomasulo 기반 OoO 엔진)

| 요소명 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **디스패치 유닛 (Dispatch)** | 디코드된 명령어를 읽어와 의존성을 분석 | 레지스터 리네이밍을 수행 후 예약 스테이션으로 보냄 | 홀 매니저 (주문 접수 및 번호표 부여) |
| **레지스터 리네이밍 (Register Renaming)** | 논리 레지스터(예: EAX)를 물리 레지스터(예: P12)로 매핑 | 데이터 쓰기 후 쓰기(WAW), 읽기 후 쓰기(WAR) 충돌 해소 | 중복된 진동벨 번호 교체 |
| **명령어 윈도우 / 이슈 큐 (Issue Queue)** | 대기 중인 명령어들이 모여 있는 저장소 | 필요한 데이터(Operand)가 모두 도착하면 즉시 발사(Issue) | 요리사들이 각자 재료 대기 중인 주방 |
| **실행 유닛 (Execution Unit)** | 이슈된 명령어를 실제로 연산 | ALU, FPU 등 복수의 하드웨어 연산기 병렬 가동 | 프라이팬, 오븐, 튀김기 |
| **공통 데이터 버스 (CDB, Common Data Bus)** | 실행이 끝난 결과를 모든 대기 중인 명령어에 동시 전파 | 결과를 Broadcast 하여 데이터 의존성(RAW)을 한 번에 풂 | "감튀 나왔습니다!" 주방 방송 |
| **리오더 버퍼 (ROB, Reorder Buffer)** | 무작위로 끝난 결과들을 원래 프로그램 순서로 정렬 보관 | 가장 오래된 명령어(Oldest)부터 순서대로 레지스터에 기록(Commit/Retire) | 완성된 음식을 주문 번호 순으로 쟁반에 담기 |

### In-Order vs Out-of-Order 실행 흐름 비교

순차 실행(In-Order)에서는 앞 명령어의 지연이 전체를 멈추게 하지만, 비순차 실행(OoO)에서는 독립적인 명령어가 먼저 실행되어 파이프라인의 유휴 시간을 없앤다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         In-Order vs Out-of-Order 파이프라인 타이밍 비교             │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [프로그램 코드]                                                 │
  │ I1: R1 = Load(Mem)    ← 메모리에서 가져오느라 오래 걸림(10클럭)       │
  │ I2: R2 = R1 + 1       ← I1의 결과(R1)를 기다려야 함 (의존성 존재)     │
  │ I3: R3 = 5 * 2        ← I1, I2와 전혀 관계없이 독립적임.             │
  │ I4: R4 = R3 + 3       ← I3의 결과만 기다리면 됨.                   │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │ [1] 순차적 실행 (In-Order) - "막히면 뒤에도 다 멈춤"               │
  │                                                             │
  │ I1(Load) : ▓▓▓▓▓▓▓▓▓▓ (10 클럭 소요)                          │
  │ I2(Add)  :           ██ (I1 끝날 때까지 10클럭 Stall 후 실행)      │
  │ I3(Mul)  :             ██ (I2 뒤에 있으므로 무작정 Stall 후 실행)    │
  │ I4(Add)  :               ██                                 │
  │           = 총 16 클럭 소요 (파이프라인 텅 빔)                    │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │ [2] 비순차적 실행 (Out-of-Order) - "가능한 놈부터 먼저 가!"         │
  │                                                             │
  │ I1(Load) : ▓▓▓▓▓▓▓▓▓▓ (여전히 10 클럭 소요)                      │
  │ I2(Add)  :           ██ (I1 기다림 - Data Dependency)        │
  │ I3(Mul)  : ██ (I1과 무관하므로 빈 ALU에서 즉시 실행!)               │
  │ I4(Add)  :   ██ (I3 끝나자마자 빈 ALU에서 즉시 실행!)               │
  │           = 총 12 클럭 소요 (I3, I4가 I1 대기 시간에 미리 끝남)     │
  │                                                             │
  │ * 마무리는 ROB(Reorder Buffer)가 I1, I2, I3, I4 순서대로 저장함.│
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상단의 `I1` 명령어는 메모리 접근(Load)으로 매우 느리다. In-Order 방식에서는 `I3`와 `I4`가 `I1`과 아무런 데이터 연관이 없음에도 불구하고, 단순히 프로그램상 뒤에 적혀 있다는 이유만으로 앞의 `I1`과 `I2`가 끝날 때까지 멍하니(Stall) 기다려야 한다. 반면 하단의 **Out-of-Order(OoO)** 방식은 하드웨어 스케줄러(이슈 큐)가 명령어의 의존성을 런타임에 파악한다. `I1`이 메모리에서 오기를 기다리는 동안, 하드웨어는 뒤에 있는 `I3`를 당겨와 놀고 있는 정수 연산기(ALU)에 밀어 넣는다(OoO Execute). 그 결과 전체 프로그램 수행 사이클이 비약적으로 단축된다.

### 심층 동작 원리: 3대 데이터 해저드와 해결책 (Register Renaming)
비순차적 실행을 가로막는 것은 '데이터 의존성'이다. 이를 해결하는 핵심 마법이 **레지스터 리네이밍(Register Renaming)**이다.
① **RAW (Read After Write, 진성 의존성)**: `A = B; C = A;` 앞의 연산 결과를 뒤가 써야 한다. 물리적으로 절대 순서를 바꿀 수 없다. 뒤 명령어는 앞이 끝날 때까지 예약 스테이션에서 대기해야 한다.
② **WAW (Write After Write, 출력 의존성)**: `R1 = 1; ... ; R1 = 2;` 같은 레지스터에 두 번 덮어쓴다. 순서를 바꾸면 최종값이 1이 되어버린다.
③ **WAR (Write After Read, 반의존성)**: `A = R1; R1 = 2;` 앞이 읽기도 전에 뒤가 2를 덮어쓰면 앞 연산이 망가진다.
* **해결책 (Renaming)**: WAW와 WAR는 진짜 의존성이 아니라 '레지스터 이름(R1)이 같아서 생기는 가짜(False) 충돌'이다. 하드웨어는 보이지 않는 물리 레지스터 수백 개를 가지고, `A = PR1; PR2 = 2;` 처럼 뒤의 R1을 새로운 물리 레지스터(PR2)로 이름을 몰래 바꿔버린다(Renaming). 이렇게 되면 가짜 의존성이 사라져 완벽한 비순차적 실행이 가능해진다.

## Ⅲ. 융합 비교 및 다각도 분석

### 실행 방식 패러다임 비교 (In-Order vs Out-of-Order)

| 비교 항목 | 순차 실행 (In-Order) | 비순차 실행 (Out-of-Order) |
|:---|:---|:---|
| **실행 순서** | 프로그램에 작성된 순서 (1-2-3-4) | 의존성이 풀린 순서 (1-3-4-2 등) |
| **파이프라인 멈춤 (Stall)** | 캐시 미스 1번에 전체 멈춤 (성능 치명적) | 멈춘 명령어 뒤로 **다른 명령어들을 우회(Bypass)** 시켜 실행 |
| **하드웨어 복잡도/전력** | 단순함. 코어 크기가 작고 저전력 | **극도로 복잡함.** 큐, ROB 등 방대한 논리 회로로 전력 소모 큼 |
| **컴파일러 의존도** | 컴파일러가 미리 순서를 잘 섞어줘야 함 | 컴파일러가 못해도 하드웨어가 런타임에 멱살 잡고 캐리함 |
| **주요 사용처** | 과거 프로세서, 초소형 임베디드, IoT 코어 | **현대 데스크탑/서버 (Intel Core, AMD Zen, ARM Cortex-X)** |

### 과목 융합 관점
- **보안 (사이드 채널 공격 - Meltdown)**: 비순차적 실행의 가장 큰 맹점은 "일단 미리 실행해 보고, 조건이 안 맞으면(예: 권한 없음, 분기 실패) 실행 결과를 취소(Flush)한다"는 점이다. 2018년 발견된 **멜트다운(Meltdown)** 공격은, 권한이 없는 OS 커널 메모리를 일반 유저가 읽게 한 뒤 비순차적 실행으로 데이터를 캐시에 남기고, 이후 권한 실패로 명령어가 취소(Rollback)되어도 캐시에 남은 미세한 흔적(사이드 채널)을 통해 커널 비밀번호를 빼내는 충격적인 취약점이었다.
- **아키텍처 (슈퍼스칼라와의 결합)**: 슈퍼스칼라(한 사이클에 ALU 4개를 동시 가동)의 폭이 아무리 넓어도, In-Order 방식이면 의존성 때문에 ALU 4개를 꽉 채울 수가 없다. 따라서 **광폭 슈퍼스칼라는 필연적으로 Out-of-Order 엔진과 짝을 이루어야만** 그 막대한 트랜지스터 투자값을 회수(IPC 극대화)할 수 있다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — ARM big.LITTLE 프로세서의 설계 전략**: 스마트폰 AP(예: 스냅드래곤, 애플 실리콘)는 배터리 소모와 성능의 딜레마를 겪는다.
   - **아키텍처 적용**: 평상시 백그라운드 작업은 전력 소모가 적고 단순한 In-Order 코어(리틀 코어)에 맡겨 배터리 광탈을 막는다. 반면 게임을 켜거나 웹브라우저를 열 때는 막대한 전력을 소모하지만 거대한 명령어 윈도우(Instruction Window)를 가진 무식한 Out-of-Order 코어(빅 코어)를 깨워 연산을 순식간에 끝낸 뒤 다시 잠재우는(Race to Sleep) 전략을 취한다.
2. **시나리오 — 데이터베이스 락(Lock) 경합과 파이프라인 플러시**: 개발자가 멀티스레드 코드에서 락(Lock)을 잘못 설계하여 스레드 간 컨텍스트 스위칭(Context Switch)이 초당 수만 번 발생하는 상황.
   - **성능 하락 원인**: 분기 실패나 컨텍스트 스위칭이 발생하면, 하드웨어는 열심히 비순차적으로 미리 계산해 두었던 명령어 윈도우(수백 개) 안의 중간 결과들을 전부 쓰레기통에 버려야(Flush) 한다. 아무리 뛰어난 OoO 엔진이라도 파이프라인이 텅 비어버리면 IPC가 0에 수렴하여 CPU 사용률은 100%인데 처리량(TPS)은 바닥을 기는 최악의 상황이 된다.
3. **시나리오 — VLIW의 몰락과 OoO의 승리**: 인텔과 HP가 "하드웨어가 동적으로 스케줄링(OoO)하는 건 전력 낭비가 심하니, 컴파일러가 알아서 순서를 다 맞춰주는 VLIW(아이테니엄)를 만들자"고 시도했다.
   - **결과**: 메모리 지연 시간이나 캐시 미스는 컴파일 시점(Static)에 예측할 수가 없었다. 반면 AMD(x86-64)는 기존 코드를 그대로 받되 내부 하드웨어가 런타임(Dynamic)에 괴물처럼 순서를 재배치하는 강력한 OoO 아키텍처를 발전시켜 시장을 제패했다. 

### 도입 체크리스트
- **기술적**: 최신 아키텍처를 평가할 때, 단순히 코어 수나 클럭만 볼 것이 아니라 **ROB(Reorder Buffer)의 크기**를 봐야 한다. (예: 인텔 최신 코어는 수백 개의 명령어를 버퍼에 담아두고 OoO를 수행한다.) 이 윈도우 창이 클수록 더 멀리 있는 명령어를 찾아내어 병렬성을 극대화할 수 있다.
- **운영적**: 코드가 분기문(If-else) 위주의 스파게티 코드라면 분기 예측 실패 시 OoO 엔진의 플러시 페널티가 너무 크다. 가능하면 조건 분기를 없애고 순차적 수학 연산으로 처리하는 데이터 지향 설계(DOD)가 하드웨어 성능을 끌어낸다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | In-Order (순차 실행) | Out-of-Order (비순차 실행) | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (IPC 성능)** | 캐시 미스 시 파이프라인 즉시 마비 | 유휴 연산기에 다른 명령어 선투입 | 캐시 레이턴시 은닉으로 **IPC 2배 이상 향상** |
| **정량 (효율성)** | ALU(연산기) 가동률 50% 미만 | OOO 스케줄링으로 빈틈없이 채워 넣음 | 하드웨어 연산기(ALU, FPU) 활용률 극대화 |
| **정성 (보안 리스크)** | 사이드 채널 공격 불가능 | 투기적 실행(Speculative) 잔재로 인한 틈새 | Meltdown, Spectre 등 마이크로아키텍처 해킹 위험 내포 |

### 미래 전망
- **거대 명령어 윈도우의 물리적 한계**: 비순차적 실행을 위해 명령어들 간의 의존성을 검사(Wakeup & Select)하는 회로의 크기와 전력 소모는 윈도우 크기의 제곱($O(N^2)$)에 비례한다. 무작정 창을 넓힐 수 없으므로, 향후에는 머신러닝 예측 모델을 하드웨어 스케줄러에 탑재하여 의존성 검사를 건너뛰거나 확률적으로 실행하는 차세대 OOO 엔진이 대두될 것이다.
- **VLIW 사상의 부활 (AI 가속기)**: 범용 CPU 시장은 OoO가 제패했지만, 예측 불확실성이 전혀 없고 행렬 곱만 죽어라 반복하는 딥러닝 TPU나 NPU에서는 복잡하고 전력 소모가 큰 OoO 엔진을 떼어버리고 다시 In-Order/VLIW 사상을 접목하는 아키텍처의 회귀 현상이 일어나고 있다.

### 참고 표준
- **Tomasulo's Algorithm**: 레지스터 리네이밍과 예약 스테이션(Reservation Station), 공통 데이터 버스(CDB)를 도입하여 런타임 데이터 의존성을 해결한 가장 위대한 컴퓨터 공학 알고리즘. (모든 OOO의 조상)

## 📌 관련 개념 맵 (Knowledge Graph)
- **[슈퍼스칼라 (Superscalar)](./501_superscalar.md)**: 연산기를 여러 개 배치해 폭을 넓히는 기술. 이 넓은 폭을 꽉꽉 채워주는 일꾼이 바로 Out-of-Order 엔진이다.
- **[분기 예측 (Branch Prediction)](../5_control_unit_pipelining/233_branch_prediction.md)**: OoO 실행 중 If문을 만나면 스케줄링이 멈춘다. 이를 막기 위해 "아마 이쪽 길일 거야"라고 찍고 먼저 실행(투기적 실행, Speculative Execution)하게 해주는 짝꿍 기술.
- **[멜트다운 (Meltdown) / 스펙터 (Spectre)](../14_hardware_security_trends/482_meltdown.md)**: 프로세서가 성능을 위해 미리 실행(OoO)해버리고 나중에 취소하는 과정에서 캐시에 미세한 흔적을 남기는데, 이 흔적을 역추적해 보안 구역의 비밀번호를 훔쳐내는 희대의 하드웨어 취약점.
- **[파이프라이닝 (Pipelining)](../5_control_unit_pipelining/230_pipelining.md)**: 폰 노이만 구조에서 명령어를 쪼개어 연속 실행하는 기술. 이 파이프라인 안에 거대한 병목(Stall)이 생기는 것을 회피하는 것이 OoO의 존재 이유다.

## 👶 어린이를 위한 3줄 비유 설명
1. 학교에서 선생님이 1번, 2번, 3번 순서대로 수학 문제를 풀라고 했어요. (순차적 실행)
2. 그런데 1번 문제가 엄청 어려워서 낑낑대고 있으면, 똑똑한 친구(OoO)는 1번을 붙잡고 멍때리는 대신 쉬운 2번, 3번 문제부터 쓱싹 먼저 풀어버려요. (비순차적 실행)
3. 대신 선생님한테 공책을 검사받을 때(결과 저장)는, 1번을 마저 푼 다음에 원래 순서대로 예쁘게 정리해서 내밀기 때문에 선생님은 순서를 바꾼 줄 모르고 칭찬해 준답니다!
