# 511. QoS (Quality of Service) in Architecture

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 네트워크 장비에 주로 쓰이던 QoS(서비스 품질) 개념을 컴퓨터 내부의 하드웨어 아키텍처(CPU 캐시, 메모리 대역폭, 스토리지 I/O)로 확장하여, 여러 워크로드가 자원을 공유할 때 우선순위에 따라 자원을 격리하고 할당하는 제어 기술이다.
> 2. **가치**: 클라우드(멀티테넌트) 환경에서 덜 중요한 작업(배치 처리 등)이 중요 작업(실시간 웹 서버 등)의 자원을 뺏어 성능을 저하시키는 '시끄러운 이웃(Noisy Neighbor)' 문제를 하드웨어 레벨에서 원천 차단한다.
> 3. **융합**: 인텔의 RDT(Resource Director Technology)와 ARM의 MPAM(Memory Partitioning and Monitoring) 같은 최신 하드웨어 기술과 쿠버네티스(Kubernetes) 같은 컨테이너 오케스트레이션 플랫폼이 강력하게 융합되어 SLA(Service Level Agreement)를 보장한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 컴퓨터 아키텍처에서의 QoS(Quality of Service)는 동일한 물리적 자원(L3 캐시, 메모리 버스, PCIe 대역폭)을 공유하는 여러 스레드나 가상머신(VM) 간에, 중요도에 따라 자원 사용량의 상한(Limit)이나 최소 보장치(Guarantee)를 하드웨어적으로 강제하는 메커니즘이다.
- **💡 비유**: 하나의 거대한 식당(서버)이 있다. 일반 손님(배치 작업)들이 너무 많이 들어와 뷔페 음식(캐시 메모리)을 다 쓸어가 버리면, 비싼 돈을 낸 VIP 손님(실시간 중요 작업)이 먹을 음식이 없어 화를 낸다(Noisy Neighbor 문제). 식당 매니저는 아예 뷔페 코너를 물리적으로 나누고 "VIP 전용 60%, 일반 손님 40%만 가져갈 수 있다"라고 펜스(하드웨어 QoS)를 치는 것이다. 
- **등장 배경 및 발전 과정**:
  1. **코어 수 증가와 공유 자원 병목**: 멀티코어 프로세서가 발전하면서 코어 수는 64개, 128개로 늘어났지만, 이 코어들이 공유하는 L3 캐시(LLC)와 메모리 버스는 한정되어 있었다.
  2. **Noisy Neighbor(시끄러운 이웃)의 공포**: 클라우드 서버에서 한 고객이 메모리 대역폭을 100% 갉아먹는 무식한 작업을 돌리면, 옆에 있는 다른 고객의 서버 응답 속도가 10배 이상 느려지는 치명적인 피해가 발생했다. 소프트웨어(OS 스케줄러)만으로는 이 하드웨어 버스 병목을 막을 수 없었다.
  3. **하드웨어 기반 자원 격리(Isolation) 도래**: 인텔은 2016년부터 RDT(자원 디렉터 기술)를 도입해, 코어별로 L3 캐시의 용량과 메모리 대역폭을 강제로 쪼개어 할당하는 아키텍처 수준의 QoS 시대를 열었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (Intel RDT 기반)

| 요소명 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **RMID (Resource Monitoring ID)** | 각 스레드나 앱에 부여되는 고유 태그 | OS가 스레드를 코어에 할당할 때 MSR 레지스터에 기록 | 식당 손님의 VIP 등급 팔찌 |
| **CMT (Cache Monitoring Technology)** | 특정 RMID가 L3 캐시를 얼마나 점유하고 있는지 추적 | 하드웨어 카운터가 LLC 라인의 점유량을 실시간 집계 | 각 손님이 먹은 접시 수 세기 |
| **CAT (Cache Allocation Technology)** | RMID별로 L3 캐시(Way)의 사용 한도를 물리적으로 강제 제한 | 비트마스크(Capacity Bitmask, CBM)를 적용해 캐시 방 배정 | VIP 전용, 일반 전용 식사 구역 분리 |
| **MBM (Memory Bandwidth Monitoring)** | 특정 RMID가 메모리 버스 대역폭을 얼마나 쓰는지 측정 | 메모리 컨트롤러 앞단에서 바이트(Byte) 트래픽 집계 | 음료수대에서 뽑아간 리터(L) 측정 |
| **MBA (Memory Bandwidth Allocation)** | 중요도가 낮은 스레드의 메모리 요청을 강제로 딜레이 시킴 | 토큰 버킷(Token Bucket)이나 크레딧 기반으로 요청(Request) 속도 지연 | 음료수 자판기 나오는 속도 늦추기 |

### 소프트웨어 스케줄링 vs 하드웨어 QoS(CAT) 비교

OS가 CPU 시간(Time)을 잘 배분해도, 캐시(Space)를 뺏기면 성능이 폭락한다. 이를 하드웨어 CAT가 막아준다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         시끄러운 이웃(Noisy Neighbor) 문제와 CAT 캐시 분할 해결         │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [1] QoS 미적용 (일반 공유 L3 캐시 상태)                        │
  │                                                             │
  │  [Core 0] (VIP 앱)       [Core 1] (배치 처리, Noisy Neighbor) │
  │        ▼                        ▼                           │
  │ ┌─────────────────────────────────────────────────────────┐ │
  │ │  L3 Cache (Shared LLC)                                  │ │
  │ │  [배치][배치][배치][배치][배치][배치][배치][VIP][배치][배치]│ │
  │ └─────────────────────────────────────────────────────────┘ │
  │   * 결과: Core 1의 무식한 작업이 캐시를 다 덮어써서(Thrashing),    │
  │           VIP 앱이 캐시 미스로 인해 10배 느려짐.                 │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │                                                             │
  │ [2] Intel CAT (Cache Allocation Technology) 적용            │
  │                                                             │
  │  [Core 0] (VIP: RMID 1)  [Core 1] (배치: RMID 2)            │
  │        ▼ (CBM: 11110000)        ▼ (CBM: 00001111)           │
  │ ┌─────────────────────────────────────────────────────────┐ │
  │ │  L3 Cache (Shared LLC)                                  │ │
  │ │  [VIP][VIP][VIP][VIP][VIP]  |  [배치][배치][배치]           │ │
  │ │   (VIP 전용 격리 공간 보장)      (배치 전용 제한 공간)          │ │
  │ └─────────────────────────────────────────────────────────┘ │
  │   * 결과: 하드웨어 비트마스크(CBM)로 캐시 웨이(Way)를 물리적으로 분리. │
  │           배치 작업이 아무리 난리를 쳐도 VIP 영역을 침범할 수 없음! │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상단의 [1]번 상황은 클라우드에서 흔히 겪는 최악의 상황이다. 내 가상머신(Core 0)이 웹 서버를 돌리는데, 같은 물리 서버를 쓰는 옆집 가상머신(Core 1)이 머신러닝 빅데이터를 읽기 시작한다. Core 1의 거대한 데이터가 공용 L3 캐시를 싹 다 밀어내고 자기 데이터로 채워버린다(Cache Thrashing). 내 웹 서버는 매번 메모리(DRAM)까지 다녀와야 해서 속도가 엉망이 된다.
하단의 [2]번 **QoS(CAT)** 상황에서는 운영체제가 각 코어에 RMID(ID)와 CBM(할당 마스크)을 씌운다. 예를 들어 L3 캐시가 총 16개의 웨이(Way)를 가졌다면, "RMID 1번(VIP)은 12개를 독점하고, RMID 2번(배치)은 4개만 써라"라고 하드웨어에 명령을 내린다. 이제 배치 작업이 아무리 폭주해도, 하드웨어 컨트롤러가 VIP의 캐시 구역에는 데이터를 쓰지 못하게 철벽 방어를 해준다. 성능 격리(Performance Isolation)가 완벽히 이루어지는 것이다.

---

### 심층 동작 원리: MBA (Memory Bandwidth Allocation)의 지연 제어
캐시뿐만 아니라 메모리 버스 대역폭 자체도 고갈될 수 있다. 하드웨어의 MBA 기술은 중요도가 낮은 스레드(예: 배치 백업 프로세스)가 메모리에 접근하려 할 때, 일부러 파이프라인 사이에 인위적인 **딜레이(Delay)** 값을 삽입한다. 예를 들어 "너는 최대 10GB/s만 써"라고 제한하면, 하드웨어 컨트롤러는 해당 RMID의 메모리 트랜잭션 빈도를 모니터링하다가 한도를 넘어서면 요청을 큐(Queue)에 잠시 가둬버리고(Throttling) 중요한 스레드의 요청부터 먼저 처리(QoS Priority)해 버린다.

## Ⅲ. 융합 비교 및 다각도 분석

### 시스템 계층별 QoS 제어 비교

| 비교 계층 | 소프트웨어 (OS / K8s) | 하드웨어 아키텍처 (Intel RDT / ARM MPAM) |
|:---|:---|:---|
| **제어 대상** | CPU 실행 시간 (Time Slice), 메모리 용량 | **L3 캐시 점유율 (Space), 메모리 버스 대역폭** |
| **구현 방식** | Cgroups (CFS 스케줄러, OOM Killer) | **하드웨어 레지스터 튜닝, 캐시 웨이 마스킹** |
| **제어 주기** | 밀리초(ms) 단위 스케줄링 간격 | **나노초(ns) 단위 하드웨어 컨트롤러 즉시 개입** |
| **격리 효과** | 겉보기엔 나누었으나 하드웨어 병목(LLC)엔 무력함 | **물리적인 성능 격리 (Performance Isolation)** |
| **주요 목적** | 공정성(Fairness) 및 OOM 방지 | **SLA 보장 및 Tail Latency(꼬리 지연) 방어** |

### 과목 융합 관점
- **가상화 및 클라우드 (Kubernetes 융합)**: 쿠버네티스의 `Guaranteed`, `Burstable`, `BestEffort` QoS 클래스는 기본적으로 리눅스의 Cgroups(소프트웨어)에 의존한다. 그러나 최신 클라우드 인프라(Intel Node Feature Discovery 등)는 쿠버네티스가 파드(Pod)를 스케줄링할 때 파드의 중요도에 맞춰 인텔 하드웨어의 CAT(L3 캐시 분할) 설정까지 직접 연동하여 깎아버리는 초정밀 융합 아키텍처를 구성한다.
- **네트워크 장비 (QoS 마킹)**: 하드웨어 QoS는 단순히 CPU에 머물지 않는다. 서버의 PCIe 버스와 네트워크 카드(SmartNIC)는 패킷의 IP 헤더(DSCP/ToS 필드)를 읽어, 데이터센터 스위치망을 통과할 때 패킷이 버려지지 않도록 전 구간(End-to-End)에서 우선순위를 유지하는 네트워크 QoS와 직접 맞물려 돌아간다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — 금융권 초단타 매매(HFT) 서버의 꼬리 지연(Tail Latency) 잡기**: 서버에서 핵심 매매(Trading) 엔진과 로깅(Logging) 에이전트가 함께 돌아감. 로깅 에이전트가 디스크에 데이터를 플러시(Flush)할 때마다 순간적으로 캐시와 메모리 대역폭을 빼앗아, 매매 엔진의 응답 지연 시간(P99 Latency)이 1ms에서 10ms로 튀는 현상 발생.
   - **해결책**: Intel RDT(CAT, MBA)를 적용한다. `pqos` 도구를 사용해 트레이딩 엔진의 스레드(RMID 1)에 L3 캐시의 80%를 전용(Exclusive)으로 할당하고, 로깅 에이전트(RMID 2)에는 L3 캐시를 10%만 열어주며 메모리 대역폭을 20%로 스로틀링(MBA) 건다. 백그라운 작업이 느려지는 대신, 핵심 엔진은 어떠한 외부 요인에도 흔들리지 않는 칼 같은 지연 시간(Deterministic Performance)을 보장받는다.
2. **시나리오 — 클라우드 사업자의 티어(Tier)별 상품 차등화**: AWS나 Azure 같은 하이퍼스케일러가 "프리미엄 인스턴스"와 "저가형 스팟(Spot) 인스턴스"를 같은 물리 서버에서 섞어 팔 때.
   - **아키텍처 전략**: 저가형 VM 고객이 무한 `while` 루프로 캐시를 학대해도 프리미엄 고객에게 피해가 가면 안 된다. 하이퍼바이저는 프리미엄 VM에 높은 캐시 웨이 마스크와 무제한 대역폭을 보장하고, 스팟 VM에는 남는 캐시 부스러기만 던져주어 서로 철저히 격리(Isolation)시킨다. 이는 클라우드 사업자의 수익성(자원 활용률 극대화)을 지키는 핵심 무기다.

### 도입 체크리스트
- **기술적 호환성**: 모든 하드웨어가 이를 지원하는 것은 아니다. Intel 제온의 경우 V4 이후 모델부터 RDT를 지원하며, AMD는 EPYC 7000 시리즈부터, ARM은 v8.4 아키텍처부터 MPAM을 지원한다. OS(리눅스 커널 `resctrl` 서브시스템)에서 이 하드웨어 피처가 올바르게 인식되고 마운트되었는지 점검해야 한다.
- **운영적 파편화 (Fragmentation)**: 캐시 공간을 너무 잘게 쪼개어 여러 앱에 할당하면, 오히려 공용으로 크게 쓸 때(Statistical Multiplexing) 얻을 수 있는 이득을 날려버려 서버 전체의 처리량(Throughput)이 하락할 수 있다. 철저한 사전 벤치마킹을 통해 반드시 보호해야 할 1~2개의 크리티컬 워크로드에만 선택적으로 울타리를 쳐야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 하드웨어 QoS 미적용 (공유 상태) | 하드웨어 QoS (CAT/MBA) 적용 시 | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (꼬리 지연, P99)** | 백그라운드 노이즈 시 응답 시간 5~10배 변동 | 어떠한 상황에서도 1.1배 이내의 일정한 응답 | 마이크로서비스 간의 P99 지연 시간 완벽 보장 |
| **정량 (서버 통합률)** | 노이즈를 피하기 위해 물리 서버 1대에 1개 앱만 띄움 | 간섭이 없으므로 서버 1대에 10개 앱 빽빽히 통합 | 데이터센터 서버 활용률(Utilization) **30% -> 60% 상승** |
| **정성 (SLA 준수)** | 장애 시 원인 분석(네트워크인지 CPU인지) 불가 | 모니터링(CMT)을 통해 캐시 도둑놈을 정확히 색출 | 고객과의 엄격한 SLA 계약 방어 가능 |

### 미래 전망
- **End-to-End 데이터센터 QoS의 완성**: 지금은 CPU 캐시와 로컬 메모리 정도만 통제하지만, 앞으로 PCIe(CXL 기반 확장 메모리), NVMe-oF 스토리지, SmartNIC 네트워크 큐(Queue)까지 서버의 모든 부품에 일관된 하드웨어 QoS 태그(Fabric ID)를 부착하여 전 구간에서 완벽한 격리를 달성하게 될 것이다.
- **AI 스케줄러의 자율 주행화**: 사람이 수동으로 비트마스크를 설정하는 단계를 넘어, 머신러닝 모델이 각 프로세스의 IPC 하락을 실시간으로 감지하고 초당 수백 번씩 캐시 울타리 크기를 늘렸다 줄였다 하는 동적(Dynamic) 자원 할당이 리눅스 커널의 기본 기능이 될 것이다.

### 참고 표준
- **Intel RDT (Resource Director Technology)**: x86 환경에서 하드웨어 자원 모니터링(CMT, MBM)과 할당(CAT, MBA)을 정의하는 사실상의 업계 표준 프레임워크.
- **ARM MPAM (Memory Partitioning and Monitoring)**: ARM 진영에서 캐시와 메모리 대역폭을 파티셔닝하기 위해 도입한 동급의 하드웨어 확장 표준.

## 📌 관련 개념 맵 (Knowledge Graph)
- **[캐시 메모리 계층 구조](../6_memory_hierarchy_cache/245_memory_hierarchy.md)**: QoS가 지키고자 하는 핵심 공유 자원. 이 자원이 오염(Thrashing)되면 CPU 속도가 폭락한다.
- **[서버 가상화와 하이퍼바이저](../13_cloud_architecture/1_virtualization/xx_hypervisor.md)**: 하나의 서버를 쪼개 파는 클라우드 환경에서 시끄러운 이웃(Noisy Neighbor) 문제가 탄생했으며, 이를 막기 위한 무기가 QoS다.
- **[에너지 비례 컴퓨팅 (Energy Proportional Computing)](./474_energy_proportional_computing.md)**: 낮은 부하의 서버들을 고부하 서버 1대로 욱여넣는(통합, Consolidation) 작업을 할 때, 성능 간섭을 막아주는 QoS가 뒷받침되어야만 안전하게 에너지를 아낄 수 있다.
- **[비순차적 실행 (Out-of-Order Execution)](./502_ooo_execution_window.md)**: QoS 실패로 메모리 병목이 길어지면, 아무리 좋은 비순차적 엔진도 결국 큐가 꽉 차서 멈춰(Stall) 버리게 된다.

## 👶 어린이를 위한 3줄 비유 설명
1. 뷔페식당(서버)에 갔는데, 식탐 많은 먹보(시끄러운 이웃 프로그램)가 맛있는 고기(캐시 메모리)를 혼자 다 퍼가서 텅텅 비어버렸어요!
2. 비싼 돈을 낸 다른 손님들이 굶게 되자, 똑똑한 식당 매니저(하드웨어 QoS)가 나섰어요. 고기 코너에 울타리를 치고 "먹보 손님은 저쪽 풀밭 구역만 드세요!"라고 규칙을 정했죠.
3. 덕분에 이제 뷔페는 아주 평화로워졌고, 중요한 손님들은 언제든지 원하는 만큼 고기를 맛있게 먹을 수 있게 되었답니다.
