# 528. SR-IOV (Single Root I/O Virtualization)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 하나의 물리적인 PCIe 장치(랜카드, GPU 등)를 하드웨어 레벨에서 여러 개의 독립적인 '가상 장치(Virtual Function)'로 쪼개어, 각 가상머신(VM)에 직접 할당하는 PCIe 버스 표준 가상화 기술이다.
> 2. **가치**: 하이퍼바이저(vSwitch 등)의 소프트웨어 중개를 완전히 건너뛰고(Kernel Bypass) VM이 하드웨어와 직접 통신하게 만들어, 가상화 환경에서 I/O 병목(지연 시간 및 CPU 점유율)을 제로(0)에 가깝게 소멸시킨다.
> 3. **융합**: 고성능 네트워크가 필수인 통신사 5G NFV(네트워크 기능 가상화)와 퍼블릭 클라우드(AWS ENI 등)의 근간 기술이며, 최근에는 vGPU(가상 GPU) 분할 등 스토리지와 가속기 영역으로 전면 확장되었다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: SR-IOV(Single Root I/O Virtualization)는 단일 PCIe 엔드포인트(Single Root) 장치가 하이퍼바이저 개입 없이 여러 가상머신과 I/O 리소스를 안전하게 공유할 수 있도록 지원하는 하드웨어 규격이다.
- **💡 비유**: 일반 가상화 네트워크가 100명의 세입자(VM)가 아파트 '공용 우편함(하이퍼바이저 vSwitch)'을 통해 경비 아저씨(CPU)가 일일이 우편물을 나눠주는 방식이라면, SR-IOV는 **각 세입자의 집 문 앞에 전용 직통 우체통(가상 랜카드)을 64개 따로 만들어 준 것**이다. 우체부(하드웨어)가 경비실을 거치지 않고 각 집으로 직접 배달하므로 속도가 엄청나게 빠르다.
- **등장 배경 및 발전 과정**:
  1. **소프트웨어 I/O 가상화의 한계**: 초기 클라우드 환경에서는 수십 대의 VM이 하나의 랜카드를 공유하기 위해 하이퍼바이저 내부에 브릿지(Bridge)나 OVS(Open vSwitch) 같은 소프트웨어 스위치를 돌렸다. 10Gbps 망 시대가 오자 이 패킷을 라우팅하느라 호스트 CPU가 터져나가는 I/O Tax 병목이 발생했다.
  2. **PCI Passthrough (VT-d)의 딜레마**: 소프트웨어를 거치기 싫어서 아예 물리적 랜카드(PCIe)를 통째로 1대의 VM에 꽂아주는 기능(Passthrough)이 생겼다. 속도는 베어메탈 급이 되었지만, 물리적 랜카드 1개를 VM 1개밖에 못 쓴다는 치명적인 공유 불가(자원 낭비) 문제가 생겼다.
  3. **SR-IOV 표준 제정 (2007)**: PCI-SIG가 이 문제를 해결했다. 하드웨어(랜카드) 스스로가 "나는 1개의 랜카드가 아니라 64개의 랜카드다!"라고 복제(가상화)하여, 이 가짜 랜카드 64개를 64대의 VM에 각각 Passthrough로 꽂아주는 궁극의 타협안(속도+공유)을 만들어냈다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (PF와 VF)

SR-IOV의 가장 핵심적인 개념은 물리적 기능(PF)과 가상 기능(VF)의 분리다.

| 요소명 | 역할 | 내부 동작 / 특징 | 비유 |
|:---|:---|:---|:---|
| **PF (Physical Function)** | 전체 하드웨어를 통제하는 물리적 원본 | 장치의 설정, VF의 생성 및 파괴 등 전권 행사 (보통 호스트 OS가 쥠) | 아파트 건물주 (관리자) |
| **VF (Virtual Function)** | PF에서 떨어져 나온 가벼운 복제본 | I/O 데이터 전송 기능만 가짐 (설정 변경 권한 없음). VM에 다이렉트로 할당됨 | 건물주가 세입자에게 준 전용 출입증 |
| **IOMMU (Intel VT-d)** | 메모리 접근을 안전하게 격리 | VF가 자신에게 할당된 VM의 메모리 영역만 DMA(직접 접근) 하도록 주소 변환/통제 | 카드키 출입 통제 시스템 |
| **PCIe Switch / 버스** | PF와 수십 개의 VF를 논리적인 PCIe 주소로 분리 | 각 VF는 OS 장치 관리자에서 독립된 PCIe 카드로 인식됨 (예: `eth0`, `eth1`...) | 아파트의 각 호실 주소 |

### Legacy I/O Virtualization vs SR-IOV 아키텍처 비교

SR-IOV는 데이터 평면(Data Plane)에서 하이퍼바이저를 완전히 삭제해 버린다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         일반 소프트웨어 가상 스위치 vs SR-IOV 아키텍처 비교           │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [1] Legacy Software vSwitch (기존 가상화) - "I/O 병목 발생"     │
  │                                                             │
  │     [ VM 1 ]          [ VM 2 ]          [ VM 3 ]            │
  │   (가상 랜카드)      (가상 랜카드)      (가상 랜카드)             │
  │        │                 │                 │                │
  │ ───────┼─────────────────┼─────────────────┼─────────────   │
  │ [Hypervisor]             ▼                                  │
  │                [ Virtual Switch (vSwitch/OVS) ]             │
  │                  (패킷 복사, 라우팅 - CPU 30% 낭비)              │
  │                                  │                          │
  │ ─────────────────────────────────┼───────────────────────── │
  │ [Hardware]                       ▼                          │
  │                        [ 물리적 랜카드 (NIC) ]                  │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │                                                             │
  │ [2] SR-IOV Architecture - "하드웨어 스위칭 & Kernel Bypass" │
  │                                                             │
  │     [ VM 1 ]          [ VM 2 ]          [ VM 3 ]            │
  │   (VF 1 드라이버)    (VF 2 드라이버)    (VF 3 드라이버)           │
  │        │                 │                 │                │
  │ ───────┼─────────────────┼─────────────────┼─────────────   │
  │ [Hypervisor]             │                 │                │
  │     (PF 통제)            ▼                 ▼                │
  │       │          (하이퍼바이저를 완전히 우회! / PCIe Passthrough)  │
  │ ──────┼──────────────────┼─────────────────┼─────────────   │
  │ [Hardware]               │                 │                │
  │       │                  ▼                 ▼                │
  │   ┌───▼──────────────────▼─────────────────▼────────┐       │
  │   │ [ PF ]             [ VF 1 ]  [ VF 2 ]  [ VF 3 ]  │       │
  │   │          [ NIC 내부 하드웨어 스위치 (eSwitch) ]       │       │
  │   └──────────────────────────────────────────────────┘       │
  │     * 장점: VM이 물리 하드웨어(VF)와 DMA로 직접 통신함.           │
  │             CPU 점유율 0%, 베어메탈과 동일한 네이티브 성능 달성!   │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상단의 레거시 방식은 VM들이 보낸 패킷이 하이퍼바이저 안의 소프트웨어 스위치(vSwitch)로 모인다. 여기서 리눅스 커널이 패킷 헤더를 까보고, 맥(MAC) 주소를 확인한 뒤 물리 랜카드로 복사(Copy)해서 보낸다. 10G 이상의 트래픽에서는 커널이 이 짓을 하다가 뻗어버린다.
하단의 **SR-IOV** 방식은 물리 랜카드가 스스로 여러 개의 가상 랜카드(VF 1, VF 2, VF 3)를 생성한다. 하이퍼바이저는 이 VF들을 각각의 VM에 물리적 PCIe 슬롯을 통째로 넘겨주는 방식(Passthrough)으로 꽂아준다. 이제 VM 1번이 패킷을 쏘면, 하이퍼바이저를 거치지 않고 **IOMMU(VT-d)**의 호위를 받으며 랜카드 안의 'VF 1' 하드웨어 큐로 직접 데이터가 꽂힌다(DMA). 하드웨어가 직접 패킷을 분류하고 쏘기 때문에, 가상화 환경임에도 물리 서버(베어메탈)와 거의 100% 똑같은 속도가 나온다.

---

### 심층 동작 원리 (IOMMU의 역할)
SR-IOV가 안전하게 동작하려면 반드시 **IOMMU (Intel VT-d 등)**가 필요하다. IOMMU가 없다면, VF 1을 할당받은 VM 1번이 해킹당해 "메모리 0x0000 번지에 접근해!"라고 DMA 명령을 내릴 때, 이 주소가 하이퍼바이저의 핵심 메모리를 건드려 서버 전체가 붕괴할 수 있다. IOMMU는 VF 1이 요청하는 가상 물리 주소(GPA)를 진짜 호스트 물리 주소(HPA)로 실시간 변환해 주며, "VM 1번은 자기한테 할당된 메모리 영역 외에는 절대 DMA를 쏘지 못하게" 철통같이 방어(Isolation)하는 하드웨어 방화벽 역할을 한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 네트워크 I/O 가상화 기술 심층 비교

| 비교 항목 | Paravirtualization (Virtio / Vhost-net) | DPDK (OVS-DPDK) | SR-IOV |
|:---|:---|:---|:---|
| **동작 방식** | 하이퍼바이저 커널에서 소프트웨어 스위칭 처리 | 유저 스페이스에서 코어를 100% 태워 폴링(Polling) 처리 | **하드웨어(NIC) 자체의 eSwitch로 직접 스위칭** |
| **CPU 점유율** | 매우 높음 (I/O 병목의 주범) | 극도로 높음 (코어 1~2개를 영구 희생해야 함) | **거의 0% (CPU 완전 해방)** |
| **성능 (지연/대역폭)**| 낮음 (높은 Latency) | 매우 높음 | **최고 (베어메탈 성능, Native HW)** |
| **가상머신 라이브 마이그레이션(vMotion)** | 100% 완벽 지원 (소프트웨어 추상화 덕분) | 완벽 지원 | **기본적으로 불가능** (하드웨어 종속성 때문) |
| **도입 환경** | 일반적인 웹/DB 클라우드 환경 | 높은 유연성과 고성능이 동시에 필요한 통신사 NFV | **극단적 성능이 필요한 HPC, AI, HFT 서버** |

### 과목 융합 관점
- **클라우드 컴퓨팅 (라이브 마이그레이션의 딜레마)**: SR-IOV의 가장 치명적인 약점은 하드웨어 종속성이다. VM이 특정 인텔 랜카드의 VF 하드웨어 레지스터를 직접 물고 있기 때문에, 이 VM을 멈추지 않고 통째로 다른 서버(AMD 랜카드가 꽂혀있음)로 실시간 이동(Live Migration)시키는 것이 불가능하다. (클라우드 벤더의 골칫거리). 이를 해결하기 위해 최근 AWS 등은 SR-IOV 하드웨어 인터페이스를 표준화한 Virtio-net 하드웨어 오프로딩(VDUSE) 기술로 이를 우회하고 있다.
- **가속기 아키텍처 (MIG / vGPU)**: SR-IOV는 네트워크 랜카드에서 시작했지만, 이제는 GPU로 넘어갔다. 엔비디아의 **MIG (Multi-Instance GPU)**나 AMD의 **MxGPU** 기술은 3천만 원짜리 H100 GPU 1장을 7개의 독립된 가상 GPU(VF)로 하드웨어 레벨에서 쪼개어, 7명의 학생(VM)에게 간섭(Noisy Neighbor) 없이 완벽히 격리된 환경으로 나눠줄 수 있는 SR-IOV 철학의 궁극적 진화다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — 통신사 5G Core (UPF) 장비의 x86 가상화(NFV)**: 5G 망에서 수백만 사용자의 트래픽을 처리하는 UPF(User Plane Function) 장비를 범용 x86 서버의 KVM 가상머신 위로 올리려 한다.
   - **아키텍처 적용**: 일반 Virtio 가상 네트워크를 쓰면 핑(Ping)이 10ms 이상 튀어 5G의 초저지연(URLLC) 규격을 만족시킬 수 없다. 통신사는 서버에 100G 멜라녹스 랜카드를 꽂고 **SR-IOV를 활성화**하여 VF 10개를 10대의 5G UPF 가상머신에 다이렉트로 꽂아준다. VM 내부에서는 DPDK 프레임워크가 이 VF를 물고 패킷을 처리한다. 결과적으로 가상화 장비임에도 1ms 미만의 지연 시간과 100Gbps 대역폭을 베어메탈처럼 방어해 낸다.
2. **시나리오 — AWS Elastic Network Interface (ENA)**: AWS에서 EC2 인스턴스(VM)를 띄울 때 "최대 100Gbps 네트워크 대역폭 지원"이라고 광고한다. 
   - **비하인드 씬**: AWS는 KVM 하이퍼바이저 소프트웨어로 이 속도를 절대 낼 수 없다는 걸 알고, 커스텀 하드웨어 칩인 'Nitro 카드(SmartNIC)'를 서버에 꽂아 넣었다. 이 카드는 SR-IOV 기술을 이용해 수십 개의 ENA(Elastic Network Adapter)라는 VF를 생성하고, 이를 EC2 인스턴스에 직접 꽂아준다. 고객은 가상머신을 쓰지만 실제로는 하드웨어 랜카드를 독점하는 셈이다.

### 도입 체크리스트
- **호스트 및 게스트 OS 드라이버 종속성**: SR-IOV를 쓰려면 하이퍼바이저뿐만 아니라 **가상머신 내부의 게스트 OS(예: Ubuntu, Windows)에도 해당 물리 랜카드 제조사(Intel, Mellanox)의 전용 VF 드라이버(예: `iavf`)가 깔려 있어야** 한다. 이로 인해 VM 이미지가 특정 하드웨어에 묶이게 되므로, 인프라 팀은 배포 이미지 관리에 각별한 주의를 기울여야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 소프트웨어 가상화 (Virtio) | SR-IOV 도입 시 | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (I/O Throughput)** | 약 10~20 Gbps에서 병목 발생 | 랜카드 물리 대역폭 (100G, 200G) 100% 소화 | 네트워크 처리 한계 돌파 (Line-rate 달성) |
| **정량 (Host CPU 점유율)** | 패킷 처리로 Host 코어 20~30% 점유 | **Host CPU 개입 0%** | 아낀 CPU 자원으로 더 많은 가상머신 판매 가능 |
| **정성 (보안 및 격리)** | 하이퍼바이저 큐를 공유하여 간섭 발생 | PCIe 레벨에서 완전히 분리된 하드웨어 큐 할당 | 시끄러운 이웃(Noisy Neighbor) 문제 원천 차단 |

### 미래 전망
- **SmartNIC / DPU 중심의 SR-IOV (VDUSE, Virtio-net 오프로드)**: 기존 SR-IOV의 치명적 약점인 "VM 라이브 마이그레이션 불가" 문제를 해결하기 위해, 차세대 DPU는 물리적인 VF를 넘겨주면서도 게스트 OS 입장에서는 하드웨어 벤더 종속이 없는 범용 'Virtio' 카드로 인식되게끔 속이는(Hardware Offloaded Virtio) 기술이 대세가 되고 있다. 이로써 하드웨어의 미친 성능과 소프트웨어의 완벽한 유연성이 결합하고 있다.
- **SDI(소프트웨어 정의 인프라)의 종착지**: 데이터센터 서버에 꽂힌 CXL 메모리, NVMe 스토리지, GPU, NIC 등 모든 하드웨어가 SR-IOV를 지원하게 될 것이다. 하이퍼바이저는 더 이상 자원을 에뮬레이션하지 않고, 오직 "이 하드웨어의 VF 조각을 저 VM에 꽂아라"라고 배분만 하는 얇은 제어기(Thin Control Plane)로 역할이 축소될 것이다.

### 참고 표준
- **PCI-SIG SR-IOV Specification**: PCI Special Interest Group에서 제정한 공식 규격. 한 개의 PCIe 물리적 장치가 PCIe 구성 공간(Configuration Space)을 통해 여러 개의 가상 장치(VF)로 자신을 어떻게 복제하고 메모리 영역을 나눌지 정의한 산업 표준.

## 📌 관련 개념 맵 (Knowledge Graph)
- **[가상화 보조 (Intel VT-d)](./527_hardware_assisted_virtualization.md)**: SR-IOV가 안전하게 동작하려면, 각 가상 장치(VF)가 다른 가상머신의 메모리를 침범하지 못하도록 막아주는 하드웨어 메모리 격리 장치(IOMMU, VT-d)가 반드시 짝꿍으로 필요하다.
- **[DPDK (Data Plane Development Kit)](./518_dpdk.md)**: SR-IOV로 가상머신 안에 물리적 랜카드(VF)를 직접 꽂아줬다면, 이제 그 가상머신 내부에서 패킷을 빛의 속도로 퍼 올리기 위해 사용하는 사용자 공간 통신 라이브러리다. 찰떡궁합의 기술.
- **[PCI Express (PCIe)](./508_pci_express.md)**: SR-IOV는 본질적으로 이 PCIe 버스 프로토콜의 확장 기능(Extension)이다.
- **[서버 가상화와 하이퍼바이저](../13_cloud_architecture/1_virtualization/xx_hypervisor.md)**: 하이퍼바이저가 가장 고생하던 '네트워크 스위칭 작업'을 하드웨어에게 완전히 떠넘겨(Offload) 버린 것이 SR-IOV 기술이다.
