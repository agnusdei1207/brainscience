# 537. 오토 벡터라이제이션 (Auto-vectorization)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로그래머가 작성한 일반적인 순차적 반복문(for-loop, 스칼라 연산)을 컴파일러가 알아서 분석하여, CPU의 병렬 처리 명령어인 **SIMD(Single Instruction Multiple Data)** 명령어로 자동 변환해 주는 똑똑한 최적화 기술이다.
> 2. **가치**: 개발자가 복잡하고 하드웨어 종속적인 어셈블리 내장 함수(Intrinsics)를 직접 짜지 않아도, 컴파일러(GCC, LLVM)가 자동으로 하드웨어의 벡터 레지스터(AVX 등)를 최대한 쥐어짜 내어 연산 속도를 2배~8배 이상 폭발적으로 높여준다.
> 3. **융합**: 멀티코어의 스레드 병렬성(Thread-level Parallelism)과 결합하여, 각 코어 내부의 데이터 병렬성(Data-level Parallelism)을 극대화하는 고성능 컴퓨팅(HPC)과 AI 연산의 필수 관문이다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 오토 벡터라이제이션(Auto-vectorization)은 소스 코드를 기계어로 번역하는 과정에서, 컴파일러가 특정 루프(Loop) 내의 데이터들이 서로 독립적이라는 것을 증명해 내고, 한 번에 1개씩 처리하던 스칼라(Scalar) 연산을 한 번에 N개씩 처리하는 벡터(Vector) 연산으로 바꿔치기하는 과정이다.
- **💡 비유**: 김밥 공장에서 사람이 '김 깔고 → 밥 얹고 → 단무지 얹고'를 김밥 1줄마다 순서대로 반복(스칼라 연산)하고 있었다. 이걸 본 똑똑한 공장장(컴파일러)이 기계를 가져와서 '김 8장 깔고 → 밥 8번 얹고 → 단무지 8개 얹고'를 한 번의 기계 조작(SIMD 명령어)으로 동시에 처리하도록 작업 지시서를 싹 바꿔버린 것이다.
- **등장 배경 및 발전 과정**:
  1. **클럭 경쟁의 종말**: 2000년대 중반, CPU 클럭을 높이는 것이 발열(Power Wall) 때문에 막혔다. 성능을 올릴 방법은 '한 사이클에 더 많은 데이터를 처리하는 것'뿐이었다.
  2. **SIMD 명령어의 진화**: 인텔은 MMX(64비트)를 시작으로 SSE(128비트), AVX-256, AVX-512 등 한 번에 수용할 수 있는 데이터 버스의 크기(레지스터)를 무지막지하게 키웠다.
  3. **소프트웨어의 한계**: 하드웨어가 512비트(32비트 정수 16개 동시 처리)를 지원해도, 프로그래머가 옛날 방식으로 `for`문을 짜면 CPU는 여전히 1개씩만 계산하며 놀았다. 결국 프로그래머 대신 **컴파일러가 코드를 알아서 병렬로 바꿔주는 오토 벡터라이제이션**이 필수 최적화 기술로 대두되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (벡터화 파이프라인)

오토 벡터라이제이션은 컴파일러의 Middle-end(LLVM IR) 최적화 단계에서 수행된다.

| 요소명 | 역할 | 특징 | 비유 |
|:---|:---|:---|:---|
| **Scalar Code (원본 코드)** | 프로그래머가 작성한 일반적인 순차 루프 | 한 번에 하나의 요소(element)만 처리 (`a[i] = b[i] + c[i]`) | 1줄씩 싸는 김밥 |
| **의존성 분석기 (Dependency Analyzer)** | 루프 내의 변수들이 꼬여있는지 확인 | `a[i] = a[i-1] + 1` 처럼 이전 결과가 현재에 영향을 주면(Loop-carried dependency) 벡터화를 포기함 | 앞사람이 싸야 뒷사람이 쌀 수 있는지 확인 |
| **Vector Code (벡터화 코드)** | 컴파일러가 생성한 병렬 루프 | 스칼라 변수를 벡터 레지스터 팩으로 묶어버림 (SIMD 명령어) | 8줄 동시 제조 기계 |
| **루프 프롤로그/에필로그 (Prolog/Epilog)** | 딱 맞아떨어지지 않는 나머지 데이터 처리 | 만약 데이터가 10개고 벡터가 8개씩 묶는다면, 남은 2개는 원래대로(스칼라) 처리하는 앞뒤 예외 처리 구간 | 기계에 안 들어가는 자투리 재료 손수 처리 |

### 스칼라 처리(Scalar) vs SIMD 벡터 처리(Vectorization) 구조 비교

컴파일러가 코드를 어떻게 바꿔치기하는지 보여주는 핵심 아키텍처다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │           Scalar 실행 vs Auto-Vectorization (SIMD) 실행         │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [ 원본 C 코드 ]                                             │
  │   for (int i = 0; i < 4; i++) {                             │
  │       C[i] = A[i] + B[i];                                   │
  │   }                                                         │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │                                                             │
  │ [1] 일반 컴파일 (Scalar Execution) - "4번의 반복 연산"           │
  │                                                             │
  │   t=1:  [ A[0] ] + [ B[0] ]  =  [ C[0] ]  (32비트 레지스터 사용)│
  │   t=2:  [ A[1] ] + [ B[1] ]  =  [ C[1] ]                    │
  │   t=3:  [ A[2] ] + [ B[2] ]  =  [ C[2] ]                    │
  │   t=4:  [ A[3] ] + [ B[3] ]  =  [ C[3] ]                    │
  │   * 결과: 총 4 클럭(사이클) 소요                                │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │                                                             │
  │ [2] 오토 벡터라이제이션 (Vector Execution / AVX) - "1번 만에 끝남" │
  │                                                             │
  │   t=1:  [ A[3] | A[2] | A[1] | A[0] ]  (128비트 거대 레지스터) │
  │                           +                                 │
  │         [ B[3] | B[2] | B[1] | B[0] ]                       │
  │                           =                                 │
  │         [ C[3] | C[2] | C[1] | C[0] ]                       │
  │                                                             │
  │   * 결과: 단 1 클럭(사이클) 소요 (연산 속도 4배 향상!)            │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상단의 **스칼라 방식**은 전통적인 코딩 방식이다. 32비트짜리 정수(int) 4개를 더하려면 ALU(산술논리연산장치)를 4번 거쳐야 한다.
하지만 컴파일러에 `-O3` (또는 벡터화 옵션)을 주면, 컴파일러가 코드를 쓱 보고 "어? 이거 배열 순서대로 더하기만 하고 서로 간섭이 없네?"라고 판단한다. 그리고 하단의 **벡터 방식**으로 어셈블리어를 싹 바꿔버린다. CPU 내부의 128비트(또는 256비트) 거대 레지스터에 A배열 4개, B배열 4개를 한 방에 욱여넣고, `VADDPS` 같은 SIMD 덧셈 명령어 딱 하나를 쏘아서 1클럭 만에 C배열 4개를 동시에 토해낸다. 코드 수정 없이 하드웨어의 성능을 4배 공짜로 얻는 셈이다.

---

### 심층 제약 조건: Loop-carried Dependency (루프 의존성)
컴파일러가 세상 모든 루프를 벡터화할 수는 없다. 가장 무서운 적은 '데이터 의존성'이다.
만약 코드가 `A[i] = A[i-1] + B[i]` 라면? 현재 배열 값을 계산하려면 반드시 바로 직전 루프의 결과(`A[i-1]`)를 알아야만 한다. 이런 경우 4개를 한꺼번에 레지스터에 올려서 덧셈을 시도하면 앞의 계산 결과가 안 나와서 쓰레기 값(Garbage)이 나온다. 컴파일러는 이 의존성을 수학적으로 검증하다가 조금이라도 애매하면 (예: C언어의 포인터 앨리어싱(Aliasing) 문제) 시스템 붕괴를 막기 위해 **벡터화를 포기하고 안전한 스칼라 코드로 후퇴(Bail-out)**한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 병렬 처리 아키텍처 수준 비교

병렬 처리는 어느 스케일(Scale)에서 이루어지느냐에 따라 나뉜다.

| 병렬 처리 수준 | 약어 | 주체 | 처리 방식 및 특징 | 벡터화와의 관계 |
|:---|:---|:---|:---|:---|
| **노드 수준** | Cluster | 분산 시스템(K8s) | 여러 대의 물리 서버가 네트워크로 통신하며 데이터 처리 | 최상위 레벨 |
| **스레드 수준** | TLP | OS 스케줄러, OpenMP | 여러 개의 CPU 코어(Core)가 각각 다른 스레드를 잡고 연산 | 코어 간의 병렬 |
| **명령어 수준** | ILP | CPU 하드웨어 (수퍼스칼라) | 파이프라인 안에서 여러 명령어를 동시에 발급(비순차 실행) | 코어 내부 보이지 않는 병렬 |
| **데이터 수준** | **DLP** | **컴파일러 (Auto-vectorization)** | **하나의 명령어로 여러 데이터를 동시에 묶어서 연산 (SIMD)** | **코어 내부 물리적 연산 병렬** |

### 과목 융합 관점
- **소프트웨어 공학 (Restrict 키워드)**: C/C++ 개발자들은 컴파일러가 오토 벡터라이제이션을 포기하는 것을 막기 위해 사투를 벌인다. 함수 인자로 포인터를 넘길 때, `void add(int *restrict A, int *restrict B)` 처럼 `restrict` 키워드를 붙여준다. 이는 컴파일러에게 "A와 B는 절대 같은 메모리를 가리키지 않아! 의존성(Aliasing)이 없으니 제발 나를 믿고 안심하고 벡터화(SIMD)를 해줘!"라고 개발자가 직접 맹세(Hint)하는 문법이다.
- **인공지능 가속 (NPU 및 Tensor Core)**: AI 연산은 수십억 번의 행렬 곱셈이다. 이 행렬 연산이야말로 데이터 간섭이 없는 완벽한 벡터 연산의 대상이다. 1차원 SIMD(AVX)의 성공에 힘입어, NPU와 GPU는 아예 2차원/3차원 데이터를 한 번에 곱하고 더해버리는 **매트릭스 단위 벡터화(Systolic Array)**로 아키텍처를 진화시켰다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — 데이터베이스(DBMS)의 분석 쿼리 (OLAP) 최적화**: 수십억 건의 은행 거래 기록에서 특정 금액 이상의 데이터만 뽑아내어 합산하는 쿼리 `SELECT SUM(amount) FROM tx WHERE amount > 1000`.
   - **아키텍처 적용**: 최신 인메모리 DB(ClickHouse, DuckDB 등)는 데이터를 로우(행) 단위가 아니라 칼럼(열) 단위의 연속된 배열(Array)로 메모리에 적재한다. 연속된 데이터는 캐시 히트율이 높을 뿐만 아니라, C++ 컴파일러가 비교 연산(`>`)과 덧셈 연산(`SUM`)을 완벽하게 **오토 벡터라이제이션**할 수 있다. CPU가 한 번에 8개~16개의 레코드를 필터링하고 더해버리므로 쿼리 속도가 순식간에 10배 이상 빨라진다.
2. **시나리오 — AWS Graviton(ARM) 서버로의 마이그레이션 실패**: 잘 돌아가던 인텔(x86) 서버의 영상 인코딩 프로그램을 AWS의 저렴한 ARM 서버로 컴파일만 해서 옮겼더니 성능이 반토막이 났다.
   - **원인 분석**: 원본 코드가 인텔 전용 벡터 명령어(AVX-256 Intrinsics)를 하드코딩해서 짜놓았기 때문이다. ARM에는 AVX가 없고 NEON이라는 독자적인 벡터 명령어가 있다.
   - **해결책**: 하드웨어 종속적인 어셈블리 내장 함수를 걷어내고, 평범한 C++ `for` 루프로 다시 짠 뒤 최신 GCC/LLVM의 **오토 벡터라이제이션( `-O3 -ftree-vectorize` )**에 맡긴다. 컴파일러가 인텔로 뽑을 땐 알아서 AVX로, ARM으로 뽑을 땐 알아서 NEON으로 최적으로 번역해 주어 아키텍처 독립성을 확보한다.

### 도입 체크리스트
- **구조체 배열 (AoS) vs 배열의 구조체 (SoA)**:
  - `AoS` (Array of Structures): `struct Point { x, y, z }; Point arr[100];`
  - `SoA` (Structure of Arrays): `struct Points { x[100], y[100], z[100]; };`
  오토 벡터라이제이션을 유도하려면 반드시 데이터를 **SoA 형태로 메모리에 연속 배치**해야 한다. AoS 형태는 데이터(x, y, z)가 인터리빙(Interleaving)되어 있어 컴파일러가 연속된 x값들을 한 번에 레지스터로 퍼 올리지 못해 벡터화에 실패한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 스칼라(Scalar) 컴파일 연산 | 오토 벡터라이제이션 (SIMD) 연산 | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (연산 처리량, IPC)** | 1클럭당 1개 데이터 처리 | 1클럭당 4~16개 데이터 동시 처리 | 산술 연산(FPU/ALU) 병목 구간 속도 **최대 4~16배 수직 상승** |
| **정량 (명령어 패치 횟수)** | 100번 루프 시 100번 명령어 읽어옴 | 100번 루프를 12번 만에 퉁침 | I-Cache(명령어 캐시) 미스 감소 및 프론트엔드 파이프라인 부담 경감 |
| **정성 (코드 이식성)** | 하드웨어마다 어셈블리를 수동으로 짬 | 범용 for문 작성 시 컴파일러가 자동 변환 | **소스 코드의 우아함(Readability)과 범용성(Portability) 동시 확보** |

### 미래 전망
- **SVE (Scalable Vector Extension)의 등장**: 기존 SIMD는 하드웨어 레지스터 크기(128비트, 256비트)가 바뀔 때마다 컴파일러를 새로 업데이트해야 했다. ARM이 발표한 차세대 벡터 확장(SVE) 아키텍처는 벡터의 길이(VLA)를 하드웨어가 실행 시간에 동적으로 결정한다. 즉, 컴파일러는 "이 루프를 벡터화하라"고만 번역해 두면, 스마트폰(128비트)이든 슈퍼컴퓨터(2048비트)든 동일한 기계어가 하드웨어 체급에 맞춰 알아서 팽창하여 실행된다.
- **LLVM MLIR 기반의 초거대 벡터화**: 현재의 오토 벡터라이제이션은 중첩된 다중 for문(2중, 3중 루프) 분석에 취약하다. LLVM 프로젝트의 최신 다층 IR(MLIR) 아키텍처는 루프의 구조 자체를 다면체 모델(Polyhedral Model)이라는 수학적 기법으로 뜯어고쳐, 인간이 도저히 상상할 수 없는 기괴한 방식으로 메모리 엑세스를 재배열하여 궁극의 벡터화를 끌어낸다.

### 참고 표준
- **OpenMP SIMD 지시어**: 컴파일러가 의존성 때문에 벡터화를 쫄아서(Bail-out) 안 해줄 때, 개발자가 코드 위에 `#pragma omp simd`라는 표준 지시어를 달아서 "내 목숨을 걸 테니 무조건 이걸 SIMD 명령어로 변환해!"라고 강제하는 C/C++ 표준 규격.

- **[LLVM IR 변환](./536_llvm_ir.md)**: 오토 벡터라이제이션은 특정 CPU(x86)에 종속되기 전에, 이 추상적인 LLVM IR 중간 단계(Middle-end)에서 코드를 분석하고 병렬 구조로 뜯어고치는 핵심 최적화 과정(Pass)이다.
