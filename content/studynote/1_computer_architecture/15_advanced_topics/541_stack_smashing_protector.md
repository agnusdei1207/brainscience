# 541. 스택 스매싱 프로텍터 (SSP / Canary)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: C/C++ 프로그램에서 발생하는 고전적인 버퍼 오버플로우(Buffer Overflow) 공격을 방어하기 위해, 컴파일러가 함수의 지역 변수와 반환 주소(Return Address) 사이에 **랜덤한 더미 값(Canary, 카나리아)**을 끼워 넣는 소프트웨어적 보안 기법이다.
> 2. **가치**: 해커가 반환 주소를 덮어쓰려면 필연적으로 그 앞에 있는 카나리아 값을 먼저 덮어써야 한다는 점을 이용하여, 함수가 종료될 때 카나리아 값의 변조 여부만 확인하면 99% 이상의 스택 공격을 매우 적은 오버헤드로 차단할 수 있다.
> 3. **융합**: GCC의 `-fstack-protector` 플래그로 대표되며, 현대 운영체제의 ASLR(주소 무작위화), DEP/NX(데이터 실행 방지)와 함께 가장 기본적인 3대 메모리 보안 완화(Mitigation) 기술로 모든 상용 소프트웨어에 기본 적용되어 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 스택 스매싱 프로텍터(Stack Smashing Protector, SSP)는 스택 기반의 버퍼 오버플로우 공격(Stack Smashing)을 탐지하고 프로그램의 실행 흐름 탈취를 막기 위한 컴파일러 단의 보안 기술이다. 가장 유명한 구현체는 스택 카나리아(Stack Canary)다.
- **💡 비유**: 옛날 광부들이 유독가스를 탐지하기 위해 탄광에 일찍 죽는 새인 '카나리아'를 새장에 넣어 들어간 것에서 유래했다. 광부(반환 주소)가 죽기 전에 유독가스(오버플로우)가 퍼지면 카나리아가 먼저 죽어서 경고를 해주는 것처럼, 반환 주소가 오염되기 전에 스택 카나리아 값이 먼저 깨져서 해킹을 탐지하는 원리다.
- **등장 배경 및 발전 과정**:
  1. **Aleph One의 논문 (1996)**: "Smashing The Stack For Fun And Profit"이라는 전설적인 논문이 발표되면서, 전 세계의 해커들이 `strcpy()`, `gets()` 같은 취약한 C 함수를 이용해 서버의 스택 메모리를 박살 내고 루트(Root) 권한을 탈취하기 시작했다.
  2. **StackGuard의 등장 (1997)**: 이를 막기 위해 컴파일러를 수정하여, 함수 시작 시 스택에 무작위 값을 넣고 끝날 때 검사하는 아이디어가 제안되었다.
  3. **현대 컴파일러의 표준화**: 이후 GCC, Clang, MSVC 등 모든 메이저 컴파일러가 이 기능을 기본으로 탑재(`-fstack-protector-strong`)하게 되었으며, 현재는 이 기능 없이 컴파일된 C/C++ 프로그램은 상용 배포가 금지될 정도의 절대 표준이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (SSP 작동 요소)

| 요소명 | 역할 | 특징 | 비유 |
|:---|:---|:---|:---|
| **Return Address (반환 주소)** | 함수 종료 후 CPU가 돌아갈 원래 코드의 위치 | 해커의 최종 타겟. 여기가 악성 코드 주소로 덮어씌워지면 해킹 완료 | 광부의 생명줄 (탈출구 지도) |
| **Stack Buffer (지역 변수 배열)** | 프로그램이 입력을 받는 공간 | 이 공간이 넘쳐흘러(Overflow) 위쪽 메모리를 침범하는 것이 공격의 원리 | 유독 가스가 새어 나오는 구멍 |
| **Canary Value (카나리아 값)** | 버퍼와 반환 주소 사이에 삽입되는 비밀 번호 | 프로그램이 켜질 때마다(또는 스레드마다) OS가 랜덤하게 생성함 (보통 32/64bit) | 유독 가스 탐지용 카나리아 새 |
| **Master Canary (마스터 카나리아)** | 스레드 전역 저장소(TLS)에 숨겨둔 진짜 원본 값 | 함수가 끝날 때, 스택의 카나리아가 이 원본과 똑같은지 대조함 | 본부에 남겨둔 새의 건강 기록부 |
| **__stack_chk_fail()** | 변조를 확인했을 때 호출되는 에러 핸들러 | 대조 결과가 다르면 즉시 프로그램을 `SIGABRT`로 비정상 종료시킴 | 광산 붕괴 경보 및 강제 폐쇄 |

### 일반 스택 구조 vs 카나리아(Canary) 삽입 스택 구조 비교

해커가 어떻게 공격을 실패하게 되는지 스택 메모리 맵(Memory Map)으로 확인한다. (스택은 위에서 아래로, 버퍼는 아래에서 위로 자란다고 가정)

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         일반 스택 버퍼 오버플로우 vs 스택 스매싱 프로텍터(Canary) 방어     │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [1] 일반 스택 (보안 없음) - "해커의 놀이터"                      │
  │                                                             │
  │  고주소 ┬─────────────────────────────────┬                   │
  │    ▲   │ Return Address (원래 주소)      │ ◀── [해커가 악성 주소로 덮어씀!]│
  │    │   ├─────────────────────────────────┤                   │
  │  덮어씀 │ Saved EBP (이전 함수 프레임 포인터)│                   │
  │    │   ├─────────────────────────────────┤                   │
  │    │   │ Char Buffer[16] (사용자 입력 칸) │ ◀── "A"를 30글자 밀어 넣음  │
  │  저주소 ┴─────────────────────────────────┴                   │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │                                                             │
  │ [2] SSP 적용 스택 (Canary 삽입) - "카나리아의 죽음"               │
  │                                                             │
  │  고주소 ┬─────────────────────────────────┬                   │
  │        │ Return Address (원래 주소)      │                   │
  │        ├─────────────────────────────────┤                   │
  │    ▲   │ Saved EBP (이전 함수 프레임 포인터)│                   │
  │    │   ├─────────────────────────────────┤                   │
  │  덮어씀 │ [ Canary (카나리아 무작위 값!) ] │ ◀── "A"에 의해 무조건 깨짐! │
  │    │   ├─────────────────────────────────┤                   │
  │    │   │ Char Buffer[16] (사용자 입력 칸) │ ◀── "A"를 30글자 밀어 넣음  │
  │  저주소 ┴─────────────────────────────────┴                   │
  │                                                             │
  │   * 방어 로직: 함수 종료(RET) 직전, CPU가 Canary 값이 원래 원본과 같은지 │
  │              검사한다. 해커가 Return Address를 조작하려면 반드시 버퍼를 │
  │              넘쳐야 하므로, 중간에 낀 Canary 값이 "A"로 덮어씌워져      │
  │              다르다는 것을 발견하고 즉시 프로그램을 죽여버린다!         │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 버퍼 오버플로우의 핵심은 `strcpy` 같은 함수가 버퍼 크기를 검사하지 않고 입력을 계속 메모리의 '위쪽'으로 써 내려간다는 것이다. 보안이 없는 시스템에서는 사용자 입력(예: 30개의 'A')이 할당된 버퍼 크기(16)를 넘쳐서 바로 위에 있는 중요한 반환 주소(Return Address)를 덮어버린다.
SSP가 활성화되면 컴파일러가 함수 도입부(Prologue)에 어셈블리 코드를 몰래 한 줄 추가하여, 마스터 카나리아 값을 가져와 버퍼와 EBP 사이에 끼워 넣는다. 해커가 버퍼를 흘러 넘치게 해서 반환 주소까지 도달하려면, **무조건 물리적으로 그사이에 위치한 카나리아 값을 덮어쓰고(파괴하고) 지나가야만 한다.** 함수가 끝날 때(Epilogue) 추가된 검사 코드가 이 카나리아 값의 변조를 확인하면, 해킹 시도로 간주하고 즉각 실행을 차단한다.

### 심층 동작 원리: 널 카나리아 (Null Canary)와 마스터 카나리아 은닉
초창기 카나리아는 무작위 숫자만 썼는데, C언어 문자열 함수(`strcpy`, `puts`)의 특성을 이용해 해커가 널 바이트(`\x00`)를 지워버리고 카나리아 값을 읽어가는(Leak) 공격을 시도했다.
이를 막기 위해 최신 OS(리눅스)는 카나리아 값의 첫 바이트를 무조건 **널 바이트(`\x00`)**나 **EOF(`\xff`), 개행문자(`\x0a`)**로 설정한다. (Terminator Canary). 이렇게 하면 해커가 `strcpy`로 카나리아를 덮어쓰려 하다가 널 바이트를 만나는 순간 복사가 멈춰버려 공격이 차단된다. 또한 마스터 카나리아 값 원본은 스레드 로컬 스토리지(TLS)인 `FS` 또는 `GS` 세그먼트 레지스터(일반 포인터로는 접근 불가능한 커널 영역)에 꽁꽁 숨겨두어 해커가 원본을 알아낼 수 없게 만든다.

## Ⅲ. 융합 비교 및 다각도 분석

### 컴파일러 최적화 및 보호 플래그 비교 (GCC 기준)

SSP는 오버헤드가 있기 때문에, 컴파일러는 '어느 함수에 카나리아를 넣을지' 3단계로 타협한다.

| GCC 컴파일 플래그 | 적용 대상 함수 | 보안성 | 성능 오버헤드 |
|:---|:---|:---|:---|
| **`-fno-stack-protector`** | 아무 곳에도 카나리아를 넣지 않음 | **최악 (0%)** | 없음 (가장 빠름) |
| **`-fstack-protector`** | `char` 배열이 8바이트 이상인 함수만 넣음 | 낮음 (배열 외의 덮어쓰기는 무방비) | 낮음 |
| **`-fstack-protector-strong`**| 배열 크기 상관없이, 로컬 배열을 쓰거나 포인터를 인자로 받는 모든 함수에 넣음 | **높음 (현재 업계 표준/Default)** | 중간 (약 1~2% 성능 하락) |
| **`-fstack-protector-all`**| 프로그램 내의 100% 모든 함수에 무조건 넣음 | 극강 (가장 안전함) | **높음 (불필요한 함수까지 검사해 느려짐)** |

### 과목 융합 관점
- **시스템 해킹 (카나리아 우회 기법)**: 방패가 나오면 창도 발전한다. 해커들은 카나리아를 뚫기 위해 1) 포맷 스트링 버그(Format String Bug, `%x` 연산자)를 이용해 메모리를 읽어 카나리아 값을 유출(Leak)한 뒤 똑같은 값으로 덮어쓰거나, 2) 스레드를 강제로 크래시 내고 자식 스레드를 무한 생성하며 카나리아의 1바이트씩 브루트포싱(Brute-forcing)으로 찍어 맞추는 우회 기법을 개발했다. (이를 막기 위해 ASLR이 세트로 쓰인다).
- **하드웨어 아키텍처 (Intel CET와의 관계)**: 소프트웨어인 카나리아가 우회당하기 시작하자, 결국 인텔이 하드웨어 섀도 스택(Shadow Stack, [참조: 540번])을 만들게 된 근본 원인이 되었다. 즉, 스택 스매싱 프로텍터(소프트웨어)의 한계가 하드웨어 CFI(Control-Flow Integrity)의 발전을 강제한 것이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — 오픈소스 기반 C/C++ 서버 데몬(Nginx, Redis) 컴파일**: 리눅스 배포판(Ubuntu, Debian) 환경에서 패키지를 직접 빌드하여 배포하려고 한다.
   - **아키텍처/보안 적용**: 현대 리눅스는 빌드 툴체인(GCC)에 기본적으로 `-fstack-protector-strong`이 걸려 있다. 만약 퍼포먼스를 1%라도 올리겠다고 `Makefile`에서 `-fno-stack-protector`를 선언해버리면, 단순한 1바이트 덮어쓰기 버그(Off-by-one Error) 하나만으로 전체 서버가 루트 쉘(Root Shell)을 탈취당하는 치명적인 재앙을 초래한다. 보안 담당자는 릴리즈된 바이너리에 대해 `checksec` 유틸리티를 돌려 "Canary found" 플래그가 켜져 있는지 반드시 감사(Audit)해야 한다.
2. **시나리오 — 펌웨어 용량이 부족한 초소형 IoT 센서 (RTOS 환경)**: 메모리가 16KB밖에 안 되는 센서 칩에서 작동하는 C 코드를 컴파일해야 함.
   - **엔지니어링 판단**: SSP를 활성화하면 모든 함수 호출과 반환 시 EAX 레지스터를 쓰고 비교(CMP)하는 어셈블리 명령어가 4~5줄씩 추가된다. 함수가 1,000개라면 이 오버헤드만으로 펌웨어 용량을 초과할 수 있다. 외부 네트워크 연결이 전혀 없는 폐쇄망 센서이고 입력값이 엄격하게 제한된 상태라면, 예외적으로 SSP를 끄고(`-fno-stack-protector`) 코드 사이즈(`-Os`)를 줄이는 트레이드오프 결정을 내릴 수 있다.

### 도입 체크리스트
- **멀티스레딩 환경의 카나리아 동기화**: 리눅스에서 메인 스레드와 `pthread`로 생성된 자식 스레드들은 각각 독립적인 스택을 가진다. OS 커널은 스레드 생성 시 `TLS(Thread Local Storage)` 블록을 안전하게 초기화하여 스레드별로 고유한 카나리아 값을 가지도록(또는 마스터 카나리아를 안전하게 상속하도록) 보장해야 한다. 커널의 스레드 구현이 취약하면 하나의 스레드에서 유출된 카나리아로 다른 스레드 스택을 해킹할 수 있다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | SSP(카나리아) 미적용 | SSP(카나리아) 적용 (`-fstack-protector-strong`) | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (버퍼 오버플로우 방어율)**| 취약점 존재 시 100% 실행 흐름 탈취 가능 | 일반적인 선형 버퍼 오버플로우 **99% 차단** | 스크립트 키디(Script Kiddie) 및 범용 익스플로잇 무력화 |
| **정량 (성능 오버헤드)** | 0% | 약 1~3% (명령어 추가로 인한 지연) | 극히 적은 성능 하락으로 막대한 보안 무결성 확보 |
| **정성 (침해 사고 대응)** | 백도어 설치 및 데이터 유출 | 프로그램 강제 종료 (`*** stack smashing detected ***`) | 해커가 시스템을 탈취하는 대신 서비스 거부(DoS)로 끝남 |

### 미래 전망
- **하드웨어 보호로의 점진적 교체**: SSP는 20년 넘게 C/C++ 생태계를 지켜온 훌륭한 파수꾼이지만, 결국 소프트웨어(메모리)에 값을 쓴다는 태생적 한계로 인해 지속적인 유출(Leak) 및 우회 공격(Master Canary Overwrite 등)에 시달리고 있다. 향후 인텔 CET(Shadow Stack)와 ARM PAC 기술이 모든 칩셋에 보급되면, 소프트웨어 카나리아는 역사적 소명을 다하고 하드웨어 보호 회로로 완벽히 대체될 것이다.
- **안전한 언어(Memory-Safe Language)로의 전환**: 애초에 카나리아가 필요한 이유는 C/C++가 배열의 경계(Bound)를 검사하지 않기 때문이다. Rust, Go, Java 같은 모던 언어들은 컴파일러 단에서 메모리 쓰기 범위를 엄격하게 검사하므로 버퍼 오버플로우 자체가 불가능하다. 미국 정부(NSA)의 "C/C++ 사용 중단" 권고에 따라 보안 취약점의 근본 원인 자체가 언어 교체로 사라지고 있다.

### 참고 표준
- **GCC / Clang Compiler Security Flags**: 전 세계 모든 C/C++ 컴파일러 표준 커맨드 라인 옵션. 보안을 위해 `ASLR(-fPIE)`, `NX(-z noexecstack)`, `RELRO(-z relro, -z now)`와 함께 **`Canary(-fstack-protector-strong)`**는 필수 4대 보안 플래그로 지정되어 있다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **[버퍼 오버플로우 하드웨어 방어 (Intel CET)](./540_hardware_buffer_overflow_defense.md)**: 소프트웨어 기반의 '스택 스매싱 프로텍터(Canary)'가 뚫리는 것을 막기 위해 아예 CPU 하드웨어에 보이지 않는 두 번째 스택(Shadow Stack)을 만든 차세대 보안 기술.

## 👶 어린이를 위한 3줄 비유 설명
1. 나쁜 도둑(해커)이 우리 집(메모리 스택)에 몰래 들어와서 문 앞에 놓인 '비밀 지도(반환 주소)'를 가짜 지도로 바꿔치기하려고 해요.
2. 그래서 엄마(컴파일러)가 지도 바로 앞에 **'아무도 모르는 비밀번호가 적힌 유리구슬(카나리아)'**을 딱 붙여 놨어요.
3. 도둑이 지도를 훔치려면 무조건 이 유리구슬을 깨뜨려야만 해요. 집에 돌아와서 유리구슬이 깨진 걸 발견하면, "도둑이 들었구나!"하고 바로 집을 닫아버려서(프로그램 종료) 도둑의 계획을 망쳐버리는 작전이랍니다!
