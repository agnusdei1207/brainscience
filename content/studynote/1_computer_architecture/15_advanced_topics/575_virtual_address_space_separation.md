# 가상 주소 공간 분리

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 가상 주소 공간 분리는 각 프로세스가 독립적인 0부터 시작하는 가상 메모리 주소를 갖게 함으로써, 타 프로세스나 커널의 메모리 영역에 무단으로 접근하거나 간섭하지 못하도록 격리(Isolation)하는 기술이다.
> 2. **가치**: 특정 프로세스의 버그나 악성 코드가 시스템 전체의 붕괴로 이어지는 것을 방지하고, 다중 사용자 환경에서 데이터 보안 및 시스템 안정성을 보장하는 현대 운영체제의 근간이다.
> 3. **융합**: 하드웨어의 MMU(Memory Management Unit), 페이지 테이블(Page Table), 그리고 커널의 보호 모드(Protection Mode)와 결합하여 물리 메모리의 추상화와 보호를 동시에 실현한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 가상 주소 공간 분리는 프로세스에게 "나 혼자 메모리 전체를 사용하고 있다"는 환상을 심어주는 기술이다. 실제 물리 메모리는 여러 프로세스가 조각나서 나누어 쓰지만, 각 프로세스는 자신만의 독립적인 논리 주소 공간을 가지며, OS는 페이지 테이블 매핑을 통해 각 프로세스의 가상 주소가 서로 다른 물리 주소로 연결되도록 관리한다.

- **💡 비유**: 주소 공간 분리는 "아파트의 개별 호수"와 같습니다. 아파트 전체(물리 메모리)는 하나지만, 각 세대(프로세스)는 자기 집 안에서는 1번 방, 2번 방(가상 주소)을 자유롭게 씁니다. 옆집 사람이 우리 집 안방으로 바로 들어올 수 없도록 벽(격리)이 쳐져 있고, 오직 복도(공유 메모리/커널 인터페이스)를 통해서만 소통할 수 있는 것과 같습니다.

- **등장 배경**:
  1. **초기 시스템의 취약성**: 모든 프로그램이 동일한 물리 주소를 직접 사용하던 시절에는, 한 프로그램의 오류가 다른 프로그램의 데이터를 덮어씌워 시스템 전체가 멈추는 일이 빈번했다.
  2. **보안 요구 사항의 증대**: 멀티태스킹 환경에서 사용자 간의 사생활과 데이터 보호가 중요해지면서, 프로세스 간의 엄격한 경계선이 필요해졌다.
  3. **메모리 할당의 복잡성 해결**: 프로그램이 실행될 때마다 비어있는 물리 주소를 찾아 코드를 재배치(Relocation)해야 하는 번거로움을 가상 주소를 통해 해결했다.

- **📢 섹션 요약 비유**: 서로 다른 꿈을 꾸는 사람들이 같은 침대(물리 메모리)에 누워 있어도, 각자의 꿈속 공간(가상 주소)은 절대 섞이지 않는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 및 격리 메커니즘

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **가상 주소 (VA)** | 프로세스 내부 논리 주소 | 0부터 Max까지의 연속된 가상 공간 제공 | 32/64-bit Address | 개인의 집 주소 |
| **물리 주소 (PA)** | 실제 RAM 하드웨어 주소 | 데이터가 실제로 저장되는 물리적 위치 | DRAM Addressing | 아파트의 실제 땅 번지 |
| **MMU** | 실시간 주소 변환 장치 | 가상 주소를 물리 주소로 초고속 변환 | TLB (Translation Lookaside Buffer) | 주소 변환 우체국 |
| **페이지 테이블** | 매핑 정보 저장소 | VA와 PA의 연결 고리 및 권한(RWX) 관리 | Multi-level Page Table | 주소록 |
| **커널/사용자 분리** | 권한 수준에 따른 격리 | 상위 주소는 커널, 하위 주소는 사용자 할당 | Privilege Level (Ring 0-3) | 관리자 전용 구역 |

### 프로세스 간 격리 및 주소 변환 구조

서로 다른 두 프로세스가 동일한 가상 주소를 사용하더라도 실제로는 다른 물리 공간에 매핑되는 구조를 시각화한다.

```text
 ┌──────────────────────────────────────────────────────────────────┐
 │               가상 주소 공간 분리 및 매핑 메커니즘                  │
 ├──────────────────────────────────────────────────────────────────┤
 │                                                                  │
 │   [ Process A (VA) ]       [ Page Table A ]       [ Physical RAM ]│
 │   ┌──────────────┐         ┌────────────┐         ┌──────────┐   │
 │   │ Addr: 0x1000 ├────────▶│ 0x1000-▶0x5000│ ────▶ │  0x5000  │   │
 │   └──────────────┘         └────────────┘         ├──────────┤   │
 │                                                   │  0x6000  │   │
 │   [ Process B (VA) ]       [ Page Table B ]       ├──────────┤   │
 │   ┌──────────────┐         ┌────────────┐         │  0x7000  │   │
 │   │ Addr: 0x1000 ├────────▶│ 0x1000-▶0x8000│ ────▶ ├──────────┤   │
 │   └──────────────┘         └────────────┘         │  0x8000  │   │
 │                                                   └──────────┘   │
 │                                                                  │
 │  * 각 프로세스는 자신만의 페이지 테이블을 소유함                      │
 │  * 하드웨어(CR3 레지스터 등)는 컨텍스트 스위칭 시 테이블을 교체함       │
 │  * Process A는 절대로 Process B의 0x8000 영역을 볼 수 없음           │
 │                                                                  │
 └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 위 도식에서 프로세스 A와 B는 모두 가상 주소 `0x1000`에 데이터를 저장하려고 한다. 하지만 각 프로세스는 자신만의 독립적인 페이지 테이블을 가지고 있다. CPU가 프로세스 A를 실행할 때는 A의 테이블을 참조하여 `0x1000`을 물리 주소 `0x5000`으로 변환한다. 반면 프로세스 B를 실행할 때는 B의 테이블을 통해 `0x8000`으로 변환한다. 따라서 두 프로세스는 논리적으로는 같은 주소를 쓰지만 물리적으로는 완전히 격리되어 서로의 데이터를 침범할 수 없다. 또한, 페이지 테이블에는 읽기 전용(R), 실행 금지(NX) 등의 비트가 포함되어 있어, 데이터 영역에서 코드를 실행하거나 커널 영역을 읽으려 할 때 CPU 차원에서 예외(Segmentation Fault)를 발생시켜 보호한다.

---

### 심층 동작 원리: 사용자 공간과 커널 공간의 분리

1. **User Space (사용자 영역)**:
   - 일반 응용 프로그램이 실행되는 낮은 권한의 영역.
   - 타 프로세스나 커널 메모리에 직접 접근이 금지됨.

2. **Kernel Space (커널 영역)**:
   - OS 핵심 코드가 상주하는 높은 권한의 영역.
   - 모든 프로세스의 가상 주소 공간 상단(예: 32비트 Linux의 경우 0xC0000000 이상)에 매핑되지만, 사용자 모드에서는 접근할 수 없도록 페이지 권한 비트로 잠겨 있음.
   - 시스템 콜(System Call)을 통해서만 안전하게 진입 가능함.

3. **ASLR (Address Space Layout Randomization)**:
   - 보안 강화를 위해 스택, 힙, 라이브러리 주소를 매번 실행할 때마다 랜덤하게 배치하여 공격자가 특정 주소를 예측하지 못하게 함.

- **📢 섹션 요약 비유**: 은행 창구(사용자 영역)에서 직원이 돈(데이터)을 만질 수는 있지만, 금고 안(커널 영역)으로 직접 들어갈 수는 없고 오직 은행원(시스템 콜)을 통해서만 일을 처리할 수 있는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 가상 주소 분리 vs 물리 주소 직접 접근 (Real Mode)

| 비교 항목 | 가상 주소 공간 분리 (Protected Mode) | 물리 주소 직접 접근 (Real Mode) |
|:---|:---|:---|
| **안정성** | 한 프로세스 고장이 시스템에 영향 없음 | 한 프로세스 고장이 전체 다운 유발 |
| **보안성** | 프로세스 간 완벽 격리 | 타 프로그램 데이터 무단 열람 가능 |
| **메모리 크기** | 물리 RAM보다 큰 주소 사용 가능 | 실제 물리 RAM 크기로 제한됨 |
| **구현 난이도** | 하드웨어(MMU) 및 OS 지원 필수 | 하드웨어 지원 불필요 (단순함) |
| **비유** | 개별 호실이 있는 호텔 | 칸막이 없는 대형 강당 |

### 비교 2: 프로세스 간 격리 vs 스레드 간 공유

| 구분 | 프로세스 (Process) | 스레드 (Thread) |
|:---|:---|:---|
| **주소 공간** | 각자 독립된 가상 주소 공간 소유 | 같은 프로세스 내의 주소 공간 공유 |
| **격리 수준** | 매우 높음 (H/W 기반) | 낮음 (S/W적으로 주의 필요) |
| **통신 비용** | 높음 (IPC 필요) | 낮음 (전역 변수 등 직접 접근) |
| **오류 전파** | 차단됨 | 한 스레드의 잘못된 포인터가 전체 파괴 가능 |

---

### 과목 융합 관점

- **운영체제**: 컨텍스트 스위칭(Context Switching) 시 페이지 테이블 베이스 레지스터를 교체하는 작업이 주소 공간 분리를 유지하는 핵심 과정이다.
- **보안**: 멜트다운(Meltdown)이나 스펙터(Spectre)와 같은 CPU 보안 취약점은 이 가상 주소 공간 분리의 허점을 이용해 커널 메모리를 훔쳐보는 공격이다.

- **📢 섹션 요약 비유**: 프로세스 격리가 "개인별 금고"라면, 스레드 공유는 "가족들이 함께 쓰는 서랍장"과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 웹 브라우저의 멀티 프로세스 구조**: 크롬(Chrome) 브라우저는 각 탭을 별도의 프로세스로 분리한다. 한 탭에서 악성 스크립트가 돌아가거나 페이지가 깨져도 가상 주소 공간이 분리되어 있기 때문에, 다른 탭의 비번을 훔쳐보거나 브라우저 전체가 종료되는 것을 방지한다.

2. **시나리오 — 커널 패닉 방지**: 일반 앱 개발자가 실수로 `NULL` 포인터를 참조하거나 잘못된 메모리 주소(0xDEADBEEF 등)에 쓰기를 시도하면, MMU가 페이지 테이블을 확인하고 해당 프로세스만 `Segmentation Fault`로 종료시킨다. 덕분에 OS 커널은 안전하게 유지된다.

3. **시나리오 — 공유 메모리(Shared Memory) 활용**: 두 프로세스가 데이터를 아주 빠르게 주고받아야 할 때, OS는 각자의 가상 주소 공간의 일부를 동일한 물리 주소로 매핑해준다. 격리 원칙을 지키면서도 필요할 때만 구멍을 뚫어주는 기술이다.

### 도입 체크리스트
- **TLB 히트율**: 주소 변환은 매 메모리 접근마다 일어나므로 TLB 성능이 중요하다. 잦은 컨텍스트 스위칭은 TLB 플러시를 유발해 성능을 깎아먹는다.
- **NX 비트 설정**: 데이터 영역(Stack, Heap)은 실행 권한을 제거하여 버퍼 오버플로우 공격을 통한 코드 실행을 원천 차단해야 한다.
- **커널 영역 보호**: KPTI(Kernel Page Table Isolation) 설정을 통해 사용자 모드 실행 시 커널 페이지 테이블을 완전히 분리하여 보안을 강화했는가?

### 안티패턴
- **단일 주소 공간 OS**: 일부 구형 임베디드 OS는 성능을 위해 격리를 생략하기도 하는데, 이는 보안이 중요한 현대 IoT 환경에서는 매우 위험한 설계다.
- **과도한 공유 메모리 사용**: 격리의 이점을 포기하고 너무 많은 영역을 공유하면, 동기화 문제와 데이터 오염 위험이 기하급수적으로 증가한다.

- **📢 섹션 요약 비유**: 보안을 위해 높은 성벽(주소 분리)을 쌓는 것도 좋지만, 소통을 위한 출입문(공유 메모리)을 너무 많이 만들면 성벽의 의미가 퇴색됩니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 내용 | 개선 지표 (예시) |
|:---|:---|:---|
| **정량적** | 시스템 업타임(Uptime) 향상 | 앱 오류에 의한 시스템 다운 99% 감소 |
| **정량적** | 메모리 이용 효율성 증대 | 실제 RAM보다 큰 가상 주소 공간 활용 |
| **정성적** | 개발 편의성 증대 | 주소 재배치 고민 없이 0번지부터 코딩 가능 |
| **정성적** | 다중 사용자 보안 강화 | 사용자 간 데이터 침범 완벽 차단 |

### 미래 전망
- **Enclave 기반 격리 (SGX/TrustZone)**: 단순히 주소 공간만 나누는 게 아니라, 커널조차 볼 수 없는 하드웨어 암호화 격리 영역(Enclave) 기술이 대중화될 것이다.
- **가상화의 심화**: 컨테이너(Docker)와 가상 머신(VM) 기술을 통해 주소 공간 분리를 넘어 운영체제 환경 자체를 겹겹이 격리하는 방향으로 발전 중이다.

### 참고 표준
- **IEEE 754**: 주소 계산에 쓰이는 수치 연산 표준
- **Intel Manual Vol 3A**: 시스템 프로그래밍 및 메모리 관리 아키텍처 정의

- **📢 섹션 요약 비유**: 결국 가상 주소 공간 분리는 "나만의 자유로운 공간"과 "공동체 전체의 안전"을 동시에 달성하기 위한 디지털 세계의 사회 계약과 같습니다.

- **[KPTI (Kernel Page Table Isolation)](./578_kpti.md)**: 보안을 위해 커널과 사용자의 주소 공간을 더 엄격히 나누는 기술.
- **[세그멘테이션 (Segmentation)](./xxx_segmentation.md)**: 가상 주소 공간을 의미 단위로 나누는 또 다른 격리 방식.

## 👶 어린이를 위한 3줄 비유 설명
1. 컴퓨터 안에는 여러 프로그램(게임, 인터넷 등)이 살고 있는데, 서로의 집(메모리)에 함부로 들어가지 못하게 울타리를 쳐주는 게 주소 공간 분리예요.
2. 각 프로그램은 자기 집이 "이 세상에서 가장 넓다"고 생각하며 0번 방부터 마음대로 방 번호를 붙여서 써요.
3. 하지만 실제로는 컴퓨터 아저씨가 뒷마당에 작은 텐트들을 여러 개 쳐주고, 겉으로만 크고 화려한 집처럼 보이게 마법을 부려놓은 거랍니다!
