# 분기 목표 주입 (Branch Target Injection)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분기 목표 주입 (BTI, Branch Target Injection)은 공격자가 CPU의 분기 타겟 버퍼(BTB)를 인위적으로 오염시켜, 프로세서가 보안상 민감한 '가젯(Gadget)' 코드를 투기적으로 실행(Speculative Execution)하도록 유도하는 사이드 채널 공격 기법이다.
> 2. **가치**: 스펙터 변종 2(Spectre Variant 2)로 알려진 이 공격은 현대 CPU의 성능 최적화 기능인 분기 예측을 역이용하여, 커널 메모리나 타 프로세서의 비밀 정보를 비정상적인 실행 경로를 통해 유출시킨다.
> 3. **융합**: 하드웨어적인 IBPB(Indirect Branch Predictor Barrier) 및 소프트웨어적인 Retpoline(Return Trampoline) 기술과 결합하여, 성능과 보안 사이의 트레이드오프를 관리하는 현대 시스템 보안의 핵심 과제다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 분기 목표 주입(BTI)은 CPU가 간접 분기(Indirect Branch)의 목적지를 예측할 때 사용하는 BTB(Branch Target Buffer)의 동작 원리를 악용한다. 공격자는 자신의 주소 공간에서 특정 분기 명령어를 반복 실행하여 BTB를 특정 주소로 훈련시킨 뒤, 피해자(커널 등)가 동일한 분기 예측기를 사용할 때 공격자가 의도한 주소로 투기적 실행을 하게 만든다.

- **💡 비유**: BTI는 "내비게이션의 목적지 조작"과 같습니다. 범인이 내비게이션(BTB)에 "집으로 가는 길"이라는 버튼을 누르면 "절벽(공격자 가젯)"으로 가도록 미리 학습시켜 놓습니다. 집주인(피해자)이 아무 의심 없이 버튼을 누르면, 차(CPU)는 실제 경로를 확인하기 전에 일단 절벽 방향으로 급발진(투기적 실행)하게 되고, 그 과정에서 집안의 비밀번호(데이터)가 노출되는 것과 같습니다.

- **등장 배경**:
  1. **투기적 실행의 부작용**: CPU 성능을 높이기 위해 결과를 미리 예측해서 실행하는 기능이 보안 경계를 넘나들 수 있음이 밝혀졌다.
  2. **BTB의 공유 구조**: 많은 CPU 아키텍처에서 여러 프로세스나 커널이 동일한 하드웨어 분기 예측기를 공유하여 사용한다는 점이 취약점으로 작용했다.
  3. **가젯(Gadget)의 존재**: 메모리 어딘가에 존재하는, 데이터를 읽어 사이드 채널로 흘릴 수 있는 짧은 코드 조각(Gadget)들이 공격의 도구가 되었다.

- **📢 섹션 요약 비유**: CPU의 "미리 짐작하기"라는 장점을 "잘못된 길로 인도하기"라는 약점으로 바꿔치기한 지능형 공격입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 및 공격 단계

| 요소명 | 역할 | 공격 시 악용 방법 | 비유 |
|:---|:---|:---|:---|
| **BTB (Branch Target Buffer)** | 간접 분기 주소 저장/예측 | 공격자가 원하는 가젯 주소로 강제 훈련(Poisoning) | 조작된 이정표 표지판 |
| **간접 분기 명령어** | 레지스터 기반 점프 (`jmp rax`) | 실제 주소 확인 전 BTB 예측에 의존해 점프 | 목적지가 적힌 쪽지 |
| **가젯 (Gadget)** | 부수 효과를 일으키는 코드 | 투기적 실행 중 비밀 데이터를 캐시에 남김 | 함정이 설치된 비밀의 방 |
| **투기적 실행 엔진** | 예측된 경로 선행 수행 | 잘못된 경로임에도 불구하고 일단 명령어를 수행함 | 급발진하는 엔진 |
| **캐시 사이드 채널** | 유출된 데이터 측정 | 투기적 실행의 흔적(Cache Hit/Miss)을 분석 | 바닥에 남겨진 발자국 |

### BTI 공격 메커니즘: BTB 오염과 가젯 실행

공격자가 어떻게 하드웨어를 속여 비정상적인 경로로 유도하는지 시각화한다.

```text
 ┌──────────────────────────────────────────────────────────────────┐
 │               분기 목표 주입 (BTI) 공격 및 유출 과정                │
 ├──────────────────────────────────────────────────────────────────┤
 │                                                                  │
 │    [ Attacker Space ]                [ Kernel / Victim Space ]    │
 │                                                                  │
 │  1. BTB Poisoning ──┐               4. Indirect Branch Execution │
 │     (Train BTB with │                  (e.g., call [rax])        │
 │      Gadget Addr)   │                         │                  │
 │            │        ▼                         ▼                  │
 │            │    ┌───────┐           5. Speculative Jump          │
 │            └──▶ │  BTB  │ ◀───────── (Based on Poisoned BTB)     │
 │                 └───────┘                     │                  │
 │                     │                         ▼                  │
 │                     │               6. Speculative Execution     │
 │                     └─────────────▶    of Secret-Leaking         │
 │                                        Gadget (Access Cache)     │
 │                                               │                  │
 │  2. Side-channel Measurement ◀────────────────┘                  │
 │     (Flush+Reload etc.)                                          │
 │                                                                  │
 └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 먼저 공격자는 자신의 프로세스에서 피해자의 간접 분기 주소와 유사한 가상 주소에서 반복적인 점프를 수행하여, CPU 내부의 BTB가 "이 위치의 분기는 항상 저 가젯 주소로 간다"고 믿게 만든다(1). 이후 피해자(커널 등)가 실행될 때 간접 분기 명령어(`call [rax]`)를 만나면, CPU는 실제 `rax` 값을 메모리에서 읽어오기 전(수백 사이클 소요)에 BTB의 오염된 정보를 바탕으로 가젯 주소로 즉시 뛰어버린다(5). 비록 나중에 잘못된 경로임을 깨닫고 실행 결과를 폐기(Rollback)하더라도, 가젯이 실행되는 동안 비밀 데이터를 읽어 캐시에 로드한 흔적(사이드 채널)은 지워지지 않는다. 공격자는 이 캐시 응답 시간 차이를 측정하여 비밀 데이터를 복원한다(2).

---

### 심층 동작 원리: 훈련과 유출의 핵심

1. **BTB 알리아싱 (Aliasing)**:
   - 많은 CPU가 주소 전체가 아닌 하위 일부 비트만으로 BTB 인덱스를 관리한다.
   - 공격자는 이 허점을 이용해 자신의 주소와 피해자의 주소가 동일한 BTB 엔트리를 공유하게 만든다.

2. **가젯 (Gadget)의 역할**:
   - 가젯은 보통 `secret = *kernel_addr; dummy = cache_probe[secret * 4096];` 와 같은 형태를 띤다.
   - 비밀 데이터의 값에 따라 접근하는 캐시 라인이 달라지도록 설계된다.

3. **투기적 윈도우 (Speculative Window)**:
   - 실제 분기 타겟이 메모리로부터 로드되어 오기 전까지의 짧은 시간(약 100~200 사이클) 동안 CPU는 최대 수십 개의 명령어를 투기적으로 처리할 수 있으며, 이 시간이 공격에 충분히 활용된다.

- **📢 섹션 요약 비유**: 이정표(BTB)를 살짝 고쳐놓아, 목적지로 가는 손님(CPU)이 엉뚱한 길(가젯)로 들어섰다가 다시 돌아 나오게 만든 뒤, 그 손님이 흘린 물건(캐시 흔적)을 줍는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: BTI (Variant 2) vsBounds Check Bypass (Variant 1)

| 비교 항목 | BTI (Spectre Variant 2) | Bounds Check Bypass (Variant 1) |
|:---|:---|:---|
| **악용 대상** | 간접 분기 예측기 (BTB) | 조건부 분기 예측기 (PHT) |
| **공격 위치** | 프로세스 간 / 권한 레벨 간 가능 | 주로 동일 프로세스 내 |
| **핵심 기법** | BTB 훈련 (Training) | 경계 검사 우회 (Array Bounds) |
| **주요 방어** | Retpoline, IBPB | LFENCE, Array Index Masking |
| **난이도** | 높음 (정밀한 BTB 제어 필요) | 보통 (조건문 제어) |

### 비교 2: 하드웨어 방어 vs 소프트웨어 방어

| 구분 | 하드웨어 방어 (IBPB / IBRS) | 소프트웨어 방어 (Retpoline) |
|:---|:---|:---|
| **메커니즘** | 예측기 상태를 강제로 비우거나 격리 | 간접 분기를 리턴(Return) 구조로 대체 |
| **성능 오버헤드** | 높음 (컨텍스트 스위칭 시 부하) | 상대적으로 낮음 (명령어 치환) |
| **호환성** | 마이크로코드 업데이트 필요 | 컴파일러 업데이트 및 재컴파일 필요 |
| **안전성** | 근본적인 하드웨어 격리 | 특정 아키텍처(Skylake+)에서 불완전성 존재 |

---

### 과목 융합 관점

- **운영체제**: 커널 진입 시나 프로세스 전환 시 `IBPB` 명령어를 실행하여 이전 사용자가 남긴 BTB의 '독'을 해독해야 한다.
- **컴파일러**: 모든 간접 점프와 호출을 `ret` 명령어를 이용한 안전한 트램펄린(Retpoline) 구조로 변환하여 빌드해야 한다.

- **📢 섹션 요약 비유**: 하드웨어 방어가 "교대할 때마다 안내판을 닦는 것"이라면, 소프트웨어 방어는 "안내판을 아예 안 보고 지도만 보고 가는 법을 배우는 것"과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 클라우드 멀티테넌시(Multi-tenancy) 위협**: 동일한 물리 서버에서 실행되는 두 가상 머신(VM) 중 하나가 공격자일 때, 공격자 VM이 BTB를 오염시켜 옆 동네 VM의 암호화 키를 훔쳐보려 한다. 클라우드 사업자는 이를 막기 위해 하이퍼바이저 수준에서 `IBPB`를 활성화하여 VM 간 분기 예측 이력을 격리해야 한다.

2. **시나리오 — 브라우저 내 샌드박스 우회**: 자바스크립트 엔진이 최적화를 위해 간접 분기를 많이 사용하는데, 이를 이용해 브라우저 프로세스의 메모리를 읽으려는 공격. 브라우저 제조사는 JIT 컴파일러가 생성하는 코드에 `LFENCE`를 삽입하거나 Retpoline을 적용하여 방어한다.

3. **시나리오 — 레거시 장비의 패치 딜레마**: 아주 오래된 CPU는 BTI 방어를 위한 마이크로코드 업데이트를 지원하지 않는다. 이 경우 보안을 포기하거나, 성능이 30~50% 떨어지더라도 모든 간접 분기를 소프트웨어적으로 에뮬레이션하는 극단적인 조치를 취해야 한다.

### 도입 체크리스트
- **CPU 지원 여부**: `CPUID`를 통해 `spec_ctrl`, `ibrs`, `ibpb` 지원 여부를 확인해야 한다.
- **성능 하락 측정**: BTI 방어 패치 적용 전후의 시스템 처리량(Throughput) 변화를 측정하여, 비즈니스 영향도를 평가해야 한다. (평균 5~15% 하락 가능)
- **컴파일러 버전**: `gcc`나 `clang`이 Retpoline 기능을 지원하는 최신 버전인지 확인해야 한다.

### 안티패턴
- **선택적 패치**: "커널만 패치하고 유저 앱은 괜찮겠지"라는 생각은 위험하다. 유저 모드에서도 민감한 데이터를 다루는 앱(예: 암호화 지갑)은 독자적인 방어가 필요하다.
- **마이크로코드 업데이트 누락**: 소프트웨어 패치만으로는 하드웨어의 근본적인 예측기 공유 문제를 완벽히 해결할 수 없으므로 BIOS/펌웨어 업데이트가 병행되어야 한다.

- **📢 섹션 요약 비유**: 앞문만 잠그고 뒷문(사이드 채널)을 열어두면, 도둑은 결국 뒷문으로 들어와서 모든 것을 가져갑니다.

### 정량/정성 기대효과

| 구분 | 기대효과 내용 | 개선 지표 (예시) |
|:---|:---|:---|
| **정량적** | 익스플로잇 난이도 증가 | BTI 기반 공격 성공률 99.9% 감소 |

| **정성적** | 하드웨어 신뢰성 회복 | CPU 설계 결함에 대한 소프트웨어적 보완 완료 |
| **정성적** | 보안 컴플라이언스 준수 | 금융/국방 정보보호 가이드라인 만족 |

### 미래 전망
- **하드웨어 격리 강화 (Enhanced IBRS)**: 차세대 CPU는 성능 저하 없이 하드웨어적으로 분기 예측 이력을 완벽히 격리하는 'Enhanced IBRS'를 탑재하여 출시되고 있다.
- **성능 위주 최적화의 재검토**: "빠른 것이 최고"였던 시대에서 "안전하게 빠른 것"이 최고의 가치가 되는 포스트 스펙터(Post-Spectre) 아키텍처 시대가 열리고 있다.
