# Retpoline (Return Trampoline)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: Retpoline (Return Trampoline)은 간접 분기(Indirect Branch)를 리턴 명령어(`ret`) 구조로 변환하여, CPU의 투기적 실행(Speculative Execution) 엔진이 오염된 분기 예측 정보 대신 안전한 무한 루프를 타게 만드는 소프트웨어적 방어 기술이다.
> 2. **가치**: 스펙터 변종 2(BTI) 공격에 대해 마이크로코드 업데이트 없이 컴파일러 수준에서 대응할 수 있는 가장 효율적인 해결책이며, 하드웨어 장벽(IBPB) 대비 성능 저하를 최소화하면서 보안 격리를 실현한다.
> 3. **융합**: 구글(Google)에 의해 고안되어 GCC, Clang 등 주요 컴파일러와 리눅스 커널에 표준으로 채택되었으며, 현대 프로세서의 RSB(Return Stack Buffer) 매커니즘을 역이용하는 창의적 아키텍처 우회 기법이다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: Retpoline은 "Return"과 "Trampoline"의 합성어로, 간접 호출(`call [rax]`)이나 간접 점프(`jmp [rax]`)를 직접 수행하지 않는다. 대신, 스택에 가짜 리턴 주소를 넣고 `ret` 명령어를 실행하는 특수한 코드 시퀀스로 대체한다. 이를 통해 CPU 내부의 간접 분기 예측기(BTB)가 개입할 여지를 없애고, 투기적 실행 엔진을 의도적으로 설계된 안전한 '대기 구역'에 가둔다.

- **💡 비유**: Retpoline은 "가짜 이정표와 막다른 골목"과 같습니다. 도둑(공격자)이 이정표를 조작해 위험한 길로 유도하려고 할 때, 집주인(CPU)은 이정표를 아예 보지 않습니다. 대신 일단 안전한 막다른 골목(무한 루프)으로 들어가서 기다립니다. 잠시 후 진짜 지도를 확인하고 목적지가 확실해지면, 그때서야 순간이동(실제 실행)하여 올바른 길로 가는 방식입니다.

- **등장 배경**:
  1. **스펙터 변종 2의 파급력**: 하드웨어가 분기 예측기를 여러 프로세스 간에 공유함으로써 발생하는 근본적인 설계 결함을 해결해야 했다.
  2. **하드웨어 패치의 부재/지연**: 하드웨어 제조사가 제공하는 IBPB 등의 패치가 나오기 전이나, 성능 저하가 너무 심한 환경을 위한 대안이 절실했다.
  3. **RSB의 특성 활용**: CPU는 `ret` 명령어를 처리할 때 BTB 대신 별도의 전용 버퍼인 RSB(Return Stack Buffer)를 사용한다는 점에 착안했다.

- **📢 섹션 요약 비유**: 적의 함정(오염된 예측기)을 피하기 위해, 스스로 안전한 감옥(무한 루프)에 들어갔다가 정답이 나오면 문을 열고 나가는 "전략적 후퇴"와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 및 코드 시퀀스

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **Trampoline Call** | 실행 흐름 전환 | 실제 목적지 대신 `ret` 명령어가 있는 곳으로 호출 | `call load_label` | 함정 피하기 입구 |
| **Capture Loop** | 투기적 실행 포획 | 잘못된 예측 시 CPU가 무한 루프를 돌게 함 | `pause; jmp capture_loop` | 막다른 골목 |
| **Return Stack Buffer** | 리턴 주소 예측 | `ret` 실행 시 BTB 대신 참조되는 안전한 하드웨어 버퍼 | RSB (Hardware Unit) | 전용 비상구 |
| **Stack Manipulation** | 주소 위장 | 스택의 리턴 주소를 실제 실행할 목적지로 덮어씀 | `mov [rsp], rax` | 신분 세탁 |
| **Ret Instruction** | 최종 분기 수행 | 조작된 스택 정보를 바탕으로 실제 코드 실행 | `ret` opcode | 진짜 출구로 나가기 |

### Retpoline의 동작 메커니즘과 투기적 실행 제어

간접 분기가 어떻게 안전한 무한 루프로 유도되고 실제 실행이 이루어지는지 시각화한다.

```text
 ┌──────────────────────────────────────────────────────────────────┐
 │                  Retpoline 동작 단계 및 흐름 제어                   │
 ├──────────────────────────────────────────────────────────────────┤
 │                                                                  │
 │  1. Indirect Branch (jmp *rax) ──▶ 2. Retpoline Trampoline       │
 │                                                                  │
 │  [ Trampoline Code ]                                             │
 │  ┌──────────────────────────────────┐                            │
 │  │ (A) call load_label ─────────────┼──────────────┐             │
 │  │ (B) capture_loop:                │              │             │
 │  │     pause; jmp capture_loop      │ ◀─── [ 3. Speculative ]    │
 │  │                                  │      [ Execution Path ]    │
 │  │ (C) load_label:                  │                            │
 │  │     mov [rsp], rax (Target Addr) │ ◀────────────┘             │
 │  │     ret ─────────────────────────┼──────▶ 4. Final Execution  │
 │  └──────────────────────────────────┘           (Correct Addr)   │
 │                                                                  │
 │  * (3단계): CPU는 (A)에서 push된 주소로 돌아가려 함 (RSB 활용)       │
 │  * 하지만 (C)에서 스택을 변조했으므로, 투기적 엔진은 (B)로 진입함      │
 │  * (4단계): 실제 실행은 변조된 스택의 rax 주소로 점프하여 완료됨       │
 │                                                                  │
 └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Retpoline의 핵심은 `call`과 `ret`의 짝을 맞추어 CPU 내부의 RSB를 속이는 것이다. (A)에서 `call`을 수행하면, 하드웨어는 자동으로 다음 명령어 주소(B)를 RSB에 저장한다. 이후 (C)로 이동하여 스택에 저장된 리턴 주소(B)를 우리가 진짜 가고 싶은 목적지(`rax`)로 살짝 바꾼다. CPU의 투기적 실행 엔진은 `ret` 명령을 만나자마자 하드웨어 버퍼(RSB)에 적힌 대로 (B) 지점으로 선행 질주하여 무한 루프에 갇힌다. 반면, 실제 실행 유닛은 스택에 정성껏 적어놓은 `rax` 주소를 읽어와서 진짜 목적지로 안전하게 이동한다. 이 과정을 통해 BTB에 주입된 독(Poison)은 한 번도 참조되지 않으며, 공격자의 가젯은 절대로 투기적으로 실행될 수 없다.

---

### 심층 동작 원리: 왜 RSB를 사용하는가?

1. **예측기 분리**:
   - BTB는 간접 분기의 목적지를 동적으로 학습하므로 외부 오염이 쉽다.
   - RSB는 함수 호출 시 주소를 쌓고(Push) 리턴 시 빼는(Pop) LIFO 구조이므로, 프로그램의 실제 호출 스택과 일치하여 훨씬 안전하다.

2. **투기적 실행의 포획 (Speculation Capture)**:
   - `ret` 명령어 직후에 배치된 `pause; jmp` 루프는 투기적 실행 엔진의 자원을 소모시키며, 실제 실행 결과가 확정되어 파이프라인이 리타이어(Retire)될 때까지 시간을 번다.

3. **성능 오버헤드 최적화**:
   - IBPB는 예측기를 통째로 비워버려 한동안 모든 분기에서 딜레이가 발생하지만, Retpoline은 간접 분기 당 수 사이클의 고정 비용만 발생시키므로 I/O가 잦은 시스템에서 훨씬 유리하다.

- **📢 섹션 요약 비유**: 앞문(BTB)에 도둑이 매복해 있다는 것을 알고, 뒷문(RSB)으로 나가는 척하면서 실제로는 개구멍(스택 조작)으로 빠져나가는 기발한 탈출 작전입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: Retpoline vs IBPB (하드웨어 장벽)

| 비교 항목 | Retpoline (Software) | IBPB (Hardware) |
|:---|:---|:---|
| **수행 주체** | 컴파일러 (Code level) | OS 커널 + CPU Microcode |
| **성능 특징** | 간접 분기 시 약간의 고정 지연 | 컨텍스트 스위칭 시 대량 지연 |
| **호환성** | 소스 코드 재컴파일 필수 | OS 업데이트만으로 가능 |
| **안전성** | Skylake 이후 CPU에서 취약점 가능성 | 모든 아키텍처에서 가장 확실함 |
| **비유** | 개별 보초 세우기 | 성문 통째로 닫고 검사하기 |

### 비교 2: Retpoline 유형별 차이

| 유형 | 적용 방식 | 특징 |
|:---|:---|:---|
| **Generic Retpoline** | 표준적인 `call/ret` 조합 | 대부분의 아키텍처에서 범용적으로 사용 |
| **LFENCE Retpoline** | 간접 분기 전 `lfence` 삽입 | 투기적 실행 자체를 멈춤, 성능 저하 심함 |
| **VMExit Retpoline** | 하이퍼바이저 탈출 시 적용 | 클라우드 가상화 환경 보안의 핵심 |
| **Thunk** | 공유 라이브러리 형태의 코드 | 바이너리 크기 증가를 최소화함 |

---

### 과목 융합 관점

- **컴파일러**: `-mindirect-branch=thunk-extern` 플래그를 통해 Retpoline 코드를 생성하며, 이는 목적 파일의 크기를 약간 증가시킨다.
- **운영체제**: 리눅스 커널은 부팅 시 CPU 성능을 측정하여 Retpoline을 쓸지 IBPB를 쓸지 동적으로 결정하는 'Alternatives' 매커니즘을 사용한다.

- **📢 섹션 요약 비유**: Retpoline은 "똑똑한 프로그래머가 하드웨어의 약점을 이용해 하드웨어의 결함을 고친" 역설적인 기술의 정수입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 리눅스 서버 커널 업데이트**: 최신 보안 패치를 적용한 후 서버 성능이 10% 하락했다는 보고가 들어왔다. 시스템 엔지니어는 `dmesg | grep Spectre`를 통해 Retpoline 적용 여부를 확인하고, 성능 복구가 절실한 경우(그리고 하드웨어가 안전한 경우) `spectre_v2=off`로 튜닝하거나 최신 아키텍처의 CPU로 교체할 계획을 세운다.

2. **시나리오 — 웹 브라우저 JIT 가속**: 크롬 V8 엔진이 자바스크립트를 기계어로 바꿀 때 간접 호출이 빈번히 발생한다. 이때 Retpoline을 적용하면 브라우저 내 탭 간의 정보 유출을 막을 수 있지만 가베지 컬렉션 성능이 떨어질 수 있다. 이를 해결하기 위해 'V8 Retpoline Thunks'라는 최적화된 공유 코드를 사용한다.

3. **시나리오 — 임베디드 실시간 시스템**: RTOS 환경에서 Retpoline의 지연 시간이 인터럽트 응답성을 해치는 상황. 아키텍트는 하드웨어적으로 격리된 'Enclave' 영역을 사용하거나, 간접 호출을 직접 호출로 바꾸는 코드 리팩토링을 통해 Retpoline 의존도를 낮춘다.

### 도입 체크리스트
- **컴파일러 지원**: GCC 7.3+, Clang 6.0+ 이상 버전이 준비되었는가?
- **CPU 아키텍처 확인**: Intel Skylake 이후 아키텍처(RSB 언더플로우 발생 시 BTB 참조)에서는 Retpoline만으로 부족할 수 있으므로 추가 패치(RSB Filling)가 적용되었는지 확인한다.
- **바이너리 성능 테스트**: 데이터베이스 등 간접 분기가 잦은 워크로드에서 실제 처리량(TPS) 하락 폭이 허용 범위 이내인가?

### 안티패턴
- **부분적 적용**: 일부 라이브러리만 Retpoline으로 빌드하고 나머지는 그대로 두면, 패치 안 된 부분의 간접 분기를 통해 여전히 공격이 가능하다. 전체 시스템의 'Full Retpoline' 빌드가 권장된다.
- **구형 커널에서의 강제 활성화**: 하드웨어 지원이 없는 너무 오래된 시스템에서 Retpoline을 억지로 돌리면 예상치 못한 시스템 크래시나 무한 루프 오류가 발생할 수 있다.

- **📢 섹션 요약 비유**: 미로의 모든 출구를 막지 않고 일부만 막는 것은(부분 패치) 도둑에게 "여기로 오세요"라고 안내하는 것과 다름없습니다.

### 정량/정성 기대효과

| **정성적** | 시스템 안정성 신뢰 | 클라우드 환경에서 테넌트 간 데이터 격리 신뢰도 향상 |

### 미래 전망
- **하드웨어 네이티브 방어 (eIBRS)**: 최신 CPU는 하드웨어 레벨에서 Retpoline의 도움 없이도 간접 분기를 격리하는 기능을 탑재하여, Retpoline은 점차 레거시 시스템을 위한 기술로 남게 될 것이다.
- **자동화된 코드 변환**: AI 컴파일러가 간접 분기 패턴을 분석하여, 보안이 필요한 지점에만 Retpoline을 선택적으로 적용하고 나머지는 고속 실행하는 '지능형 보안 컴파일' 기술이 발전할 것이다.

- **Google Project Zero Blog**: "Reading privileged memory with a side-channel"
- **Intel® Software Guard Extensions (SGX)**: 하드웨어 격리 영역 내에서의 Retpoline 활용 가이드

1. 컴퓨터가 길을 찾을 때 가짜 표지판에 속지 않게 하려고, 일단 안전한 "대기실"로 가게 만드는 마법이 Retpoline이에요.
2. 컴퓨터는 잠시 대기실에서 쉬면서 진짜 지도를 확인하고, 정답이 확실할 때만 텔레포트를 해서 목적지로 슝 날아간답니다.
3. 덕분에 나쁜 사람들이 이정표를 아무리 망쳐놔도 컴퓨터는 절대 엉뚱한 길로 빠지지 않게 되었어요!
