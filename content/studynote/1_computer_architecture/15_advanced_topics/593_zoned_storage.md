# 존 스토리지 (Zoned Storage)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 존 스토리지 (Zoned Storage)는 저장 장치의 전체 공간을 독립적인 '존 (Zone)' 단위로 분할하고, 각 존 내부에서는 반드시 순차적 쓰기 (Sequential Write)만 허용하는 새로운 스토리지 아키텍처 모델이다.
> 2. **가치**: SSD 내부의 가비지 컬렉션 (GC)을 원천적으로 제거하여 쓰기 증폭 (WAF)을 1.0으로 낮추고, SMR (Shingled Magnetic Recording) HDD의 용량 한계를 극복하여 대규모 클라우드 데이터센터의 비용 효율성을 극대화한다.
> 3. **융합**: NVMe ZNS (Zoned Namespaces) 표준과 리눅스 커널의 존 블록 장치 (ZBD) 서브시스템을 통해, 파일 시스템 (F2FS 등) 및 데이터베이스 (RocksDB 등)와 긴밀하게 협력하는 '호스트 주도형 스토리지' 생태계를 형성한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 존 스토리지 (Zoned Storage)는 데이터의 임의 쓰기 (Random Write)를 허용하지 않는 대신, 하드웨어가 관리해야 했던 복잡한 매핑 계층 (FTL)을 획기적으로 단순화한 방식이다. 호스트는 각 존의 '쓰기 포인터 (Write Pointer)'를 준수하며 데이터를 순차적으로 기록해야 하며, 데이터를 수정하려면 존 전체를 초기화 (Reset)해야 한다.

- **💡 비유**: 존 스토리지는 "연필로 쓴 뒤 지우개로 지울 수 없는 공책"과 같다. 공책의 각 페이지(존)에는 앞에서부터 차례대로 글을 써야 하며, 중간 내용을 고치고 싶다면 그 페이지 전체를 찢어버리고(Reset) 처음부터 다시 써야 하는 규칙을 가진 특수 공책이다.

- **등장 배경**:
  1. **NAND 플래시의 물리적 한계**: 낸드 플래시는 덮어쓰기가 불가능하고 블록 단위로 지워야 하므로, 무작위 쓰기가 많아지면 내부 데이터 이동 (GC)으로 인해 성능과 수명이 급격히 저하된다.
  2. **HDD의 고밀도화 (SMR)**: 하드디스크의 기록 밀도를 높이기 위해 트랙을 겹쳐 쓰는 SMR 방식이 도입되었는데, 이 역시 중간 트랙 수정 시 주변 데이터를 파괴하므로 순차 쓰기가 강제되었다.
  3. **데이터센터의 대형화**: 수십만 개의 드라이브를 관리하는 클라우드 사업자들은 개별 드라이브의 성능 편차 (Jitter)를 줄이기 위해 스토리지 동작의 완벽한 통제권을 원하게 되었다.

- **📢 섹션 요약 비유**: 자유분방한 무작위 쓰기(Random Write)라는 무질서를 순차 쓰기(Sequential Write)라는 엄격한 질서로 재편하여, 하드웨어의 물리적 잠재력을 100% 끌어내는 스토리지 혁신입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **Zone** | 데이터 저장의 기본 단위 | 수백 MB ~ 수 GB 단위의 고정된 크기 섹션 | ZNS / SMR | 공책의 한 페이지 |
| **Write Pointer (WP)** | 다음 쓰기 위치 지시 | 현재 존에서 데이터가 어디까지 써졌는지 가리키는 커서 | Sequential Access | 공책의 커서 |
| **Zone Capacity** | 실제 사용 가능 용량 | 존의 물리적 크기 중 데이터 기록이 가능한 유효 공간 | LBA Mapping | 페이지 내 여백 |
| **Zone State Machine** | 존의 상태 관리 | Empty, Open, Full, Read-only 등 상태 전이 제어 | NVMe State Transition | 페이지 사용 상태 |
| **Zone Reset** | 존 전체 데이터 삭제 | 쓰기 포인터를 0으로 되돌리고 데이터를 일괄 삭제 | Erase Operation | 페이지 찢기 |

### 존 스토리지의 쓰기 메커니즘과 상태 전이

존 스토리지는 각 존별로 독립적인 상태를 가지며, 반드시 쓰기 포인터 (WP)가 가리키는 지점에만 데이터를 쓸 수 있다.

```text
 ┌──────────────────────────────────────────────────────────────────┐
 │                존 스토리지 (ZNS) 동작 및 상태 전이                   │
 ├──────────────────────────────────────────────────────────────────┤
 │                                                                  │
 │  [ Zone Structure ]                                              │
 │  ┌─────────────────────────────────┐                             │
 │  │ Data (Written)  │ WP ──▶ Free  │                             │
 │  └─────────────────┴───────────────┘                             │
 │                                                                  │
 │  [ Zone States ]                                                 │
 │      (Reset)             (Write)              (Finish)           │
 │  ┌───────────┐       ┌───────────┐       ┌───────────┐           │
 │  │   Empty   │ ──▶   │   Open    │ ──▶   │   Full    │           │
 │  └───────────┘       └───────────┘       └───────────┘           │
 │        ▲                                       │                 │
 │        └───────────────────────────────────────┘                 │
 │                      (Reset Zone)                                │
 │                                                                  │
 │  * 규칙 1: 반드시 WP 위치에만 써야 함 (Random Write 불가)           │
 │  * 규칙 2: 수정이 필요하면 Reset 후 다시 써야 함                    │
 │                                                                  │
 └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 그림과 같이 존(Zone) 내부는 이미 써진 데이터와 비어 있는 공간으로 나뉜다. 쓰기 포인터 (Write Pointer)는 항상 비어 있는 공간의 시작점을 가리키며, 쓰기 명령이 수행될 때마다 오른쪽으로 이동한다. 만약 호스트가 WP가 아닌 엉뚱한 위치에 쓰려고 하면 하드웨어는 즉시 에러를 반환한다. 존이 가득 차면 'Full' 상태가 되며, 이 상태에서는 읽기만 가능하다. 데이터를 수정하거나 공간을 재사용하려면 'Reset Zone' 명령을 내려 존을 비워야 (Empty) 한다. 이 단순한 규칙 덕분에 SSD 내부는 복잡한 L2P (Logical to Physical) 매핑 테이블을 관리할 필요가 없어지고, DRAM 사용량과 컨트롤러 부하가 획기적으로 줄어든다.

---

### 심층 동작 원리: ZNS (Zoned Namespaces)

1. **L2P 매핑 간소화**: 일반 SSD는 4KB 단위의 촘촘한 매핑 테이블을 관리하지만, ZNS는 존 단위의 거친 (Coarse-grained) 매핑만 관리한다. 이로 인해 SSD 내부 DRAM 용량을 1/10 이하로 줄일 수 있다.
2. **쓰기 증폭 (WAF) 최적화**: 일반 SSD는 GC 과정에서 데이터를 옮겨 쓰느라 실제 수치보다 더 많은 쓰기를 발생시킨다. 존 스토리지는 호스트가 데이터를 존 단위로 모아서 순차적으로 쓰기 때문에, 내부 데이터 이동이 사라져 WAF가 이론적 최솟값인 1.0에 도달한다.
3. **오버 프로비저닝 (OP) 제거**: 일반 SSD는 GC를 위해 전체 용량의 7~28%를 보이지 않는 여유 공간 (OP)으로 남겨둬야 한다. 존 스토리지는 GC가 필요 없으므로 이 공간을 사용자에게 모두 돌려주어 가용 용량을 20% 이상 늘릴 수 있다.

- **📢 섹션 요약 비유**: 무질서하게 흩어진 퍼즐 조각들을 맞추는 수고(GC)를 덜기 위해, 애초에 그림을 순서대로 그려나가는(Sequential Write) 방식을 택한 것입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 일반 SSD vs 존 SSD (ZNS)

| 비교 항목 | 일반 SSD (Conventional) | 존 SSD (ZNS) |
|:---|:---|:---|
| **쓰기 방식** | 임의 쓰기 (Random Write) 가능 | **순차 쓰기 (Sequential) 강제** |
| **내부 관리 (GC)** | 하드웨어가 자율적으로 수행 | **수행 안 함 (호스트가 관리)** |
| **쓰기 증폭 (WAF)** | 높음 (2~5 이상 가능) | **낮음 (1.0 근접)** |
| **가용 용량** | OP 공간 때문에 상대적으로 적음 | **OP 공간 불필요로 더 큼 (+20%)** |
| **성능 일관성** | GC 발생 시 급락 (Jitter) | **항상 일정함 (Predictable)** |
| **비유** | 화이트 보드 (어디든 쓰고 지움) | 잉크 펜 (순서대로 쓰고 페이지 찢음) |

### 비교 2: 존 스토리지 (ZBD) vs 오픈 채널 SSD

| 특성 | 존 스토리지 (Zoned Storage) | 오픈 채널 SSD (Open-Channel) |
|:---|:---|:---|
| **표준화** | NVMe ZNS / SMR 표준 준수 | 비표준 (특정 벤더 종속적) |
| **추상화 수준** | 존(Zone) 단위의 추상화 제공 | 물리적 낸드 구조 전체 노출 |
| **호스트 부담** | 중간 (순차 쓰기만 신경 쓰면 됨) | 매우 높음 (배드 블록 관리까지 직접) |
| **이식성** | 표준 드라이버로 손쉽게 적용 가능 | 전용 소프트웨어 스택 구축 필수 |

### 과목 융합 관점

- **운영체제 (OS)**: 리눅스 커널 5.9 이상에서는 ZNS를 공식 지원하며, `f2fs`나 `btrfs`와 같이 원래부터 로그 구조로 동작하던 파일 시스템은 존 스토리지와 결합했을 때 최고의 궁합을 보여준다.
- **데이터베이스 (DB)**: LSM-Tree 구조를 사용하는 RocksDB는 데이터를 SStable 파일로 순차 기록하므로, 이를 존 스토리지의 존과 1:1 매핑하면 소프트웨어와 하드웨어 간의 데이터 배치 최적화가 완벽히 이루어진다.

- **📢 섹션 요약 비유**: 하드웨어의 "물리적 제약"을 소프트웨어의 "논리적 설계"로 수용함으로써, 양쪽의 오버헤드를 동시에 제거한 아키텍처적 승리입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 클라우드 스토리지 서비스 원가 절감**: 테라바이트당 단가가 중요한 환경에서 SMR HDD를 도입하려 하나, 임의 쓰기 성능 저하가 우려된다. 아키텍트는 존 스토리지 기술을 적용하여 호스트에서 데이터를 순차적으로 정렬해 내려보냄으로써, 성능 저하 없이 HDD 용량을 20% 늘리고 비용을 절감한다.

2. **시나리오 — 고성능 NoSQL 데이터베이스 가속**: RocksDB를 사용하는 서비스에서 SSD의 테일 레이턴시가 문제가 된다. 기술사는 NVMe ZNS SSD를 도입하고 `ZenFS` (ZNS 전용 파일 시스템)를 적용하여 가비지 컬렉션을 제거함으로써, 읽기/쓰기 응답 시간을 90% 안정화한다.

3. **시나리오 — 대규모 영상 아카이빙 시스템**: 수천 대의 카메라 영상이 24시간 기록되는 시스템. 영상 데이터는 본질적으로 순차적이므로, 기술사는 존 스토리지를 도입하여 쓰기 증폭을 최소화하고 SSD의 기대 수명을 3년에서 10년으로 연장한다.

### 도입 체크리스트
- **소프트웨어 준비도**: 애플리케이션이나 파일 시스템이 '순차 쓰기 전용' 인터페이스를 지원하는가?
- **존 크기 (Zone Size)**: 워크로드의 평균 파일 크기가 존 크기와 적절히 매칭되는가?
- **가비지 컬렉션의 호스트 이전**: SSD 내부에서 하던 GC를 호스트 CPU가 수행할 때 발생하는 추가적인 CPU 부하를 감당할 수 있는가?

### 안티패턴
- **기존 레거시 파일 시스템 그대로 사용**: `ext4`나 `NTFS`와 같이 임의 수정을 전제로 설계된 파일 시스템을 존 스토리지에 그대로 올리면, 쓰기 명령마다 에러가 발생하거나 극심한 성능 저하가 일어난다.
- **소용량 데이터 무작위 수정 워크로드**: 짧은 메시지를 수시로 고쳐 써야 하는 DB 워크로드를 존 스토리지에 담으려 하면, 잦은 'Zone Reset'으로 인해 오히려 효율이 떨어진다.

- **📢 섹션 요약 비유**: 순차적인 기록이 미덕인 역사책(로그 데이터)에는 존 스토리지가 정답이지만, 수시로 지우고 고쳐 써야 하는 연습장(랜덤 수정 데이터)에는 맞지 않는 옷입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 내용 | 개선 지표 (예시) |
|:---|:---|:---|
| **정량적** | 쓰기 수명 (Durability) 향상 | WAF 감소로 인해 수명 최대 3~5배 연장 |
| **정량적** | 실질 가용 용량 증대 | OP 공간 환원으로 가용 용량 15~25% 증가 |
| **정성적** | 성능 결정성 (Determinism) | GC에 의한 급격한 지연 시간 (Tail Latency) 제거 |
| **정성적** | 하드웨어 비용 절감 | 저렴한 SMR HDD나 QLC 낸드를 고성능으로 활용 가능 |

### 미래 전망
- **엔드투엔드 존 스토리지**: 애플리케이션부터 SSD 내부 낸드 페이지까지 모든 데이터 흐름이 존 단위로 통일되는 '심리스 존 (Seamless Zone)' 아키텍처가 확산될 것이다.
- **오브젝트 스토리지와의 결합**: 파일이 아닌 오브젝트(Key-Value) 자체가 존으로 직접 매핑되는 스마트 존 스토리지 기술이 클라우드 네이티브 스토리지의 표준이 될 전망이다.

### 참고 표준
- **NVMe 2.0 Zoned Namespaces (ZNS)**: SSD용 존 스토리지 표준
- **ZBC/ZAC (Zoned Block/Ata Commands)**: HDD용 존 스토리지 표준 (SMR 특화)

- **📢 섹션 요약 비유**: 결국 존 스토리지는 "편리함(임의 쓰기)"을 버리는 대신 "완벽한 통제와 효율(순차 쓰기)"을 얻은, 데이터 폭증 시대의 가장 현실적이고 강력한 스토리지 대안입니다.

- **[F2FS (Flash Friendly File System)](./xxx_f2fs.md)**: 존 스토리지와 최고의 궁합을 자랑하는 로그 구조 파일 시스템.
- **[LSM-Tree (Log-Structured Merge-Tree)](./xxx_lsm_tree.md)**: 존 스토리지 아키텍처에 가장 적합한 데이터베이스 인덱스 구조.

## 👶 어린이를 위한 3줄 비유 설명
1. 일반 SSD는 아무 데나 낙서하고 지울 수 있는 **화이트보드** 같은데, 존 스토리지는 앞에서부터 차례대로만 써야 하는 **공책** 같아요.
2. 중간에 내용을 고치려면 그 페이지 전체를 찢어버려야 하지만, 대신 공책이 훨씬 두껍고(용량) 아무리 많이 써도 종이가 닳지 않아요(수명).
3. 규칙을 잘 지켜서 차곡차곡 기록하면, 컴퓨터가 훨씬 더 많은 정보를 헷갈리지 않고 아주 빠르게 저장할 수 있게 된답니다!
