# 진법 변환 (Base Conversion)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 진법 변환(Base Conversion)은 특정 기수(Radix, 예: 10진수)로 표현된 숫자의 본질적 가치(Value)를 유지한 채, 다른 기수(예: 2진수, 16진수)의 표기 체계로 재구성하는 수학적/알고리즘적 과정이다.
> 2. **가치**: 인간의 입력(10진수)을 기계의 언어(2진수)로 번역하고, 기계의 상태(2진수 덤프)를 엔지니어가 분석하기 쉬운 형태(16진수)로 추상화하는 필수 불가결한 인터페이스 역할을 한다.
> 3. **융합**: 컴파일러의 렉서(Lexer) 최적화, 네트워크 프로토콜의 엔디안 변환, 암호학의 큰 수(BigInt) 연산 등에서 오버헤드를 줄이기 위한 핵심 최적화 대상이 된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 진법 변환은 하나의 진법으로 쓰인 수를 다른 진법의 수로 바꾸는 작업이다. 가장 대표적으로 10진수(인간) ↔ 2진수(하드웨어) 간의 변환이 있으며, 2진수의 가독성 문제를 해결하기 위해 2진수 ↔ 8/16진수 변환도 빈번하게 사용된다.

- **💡 비유**: 진법 변환은 **'환전(Currency Exchange)'**과 같다. 10만 원(10진수)을 달러(2진수)로 바꾸면 숫자의 모양(표기법)은 약 75달러로 달라지지만, 그 돈으로 살 수 있는 햄버거의 개수(본질적 가치)는 똑같다. 마찬가지로 진법이 바뀌어도 데이터가 담고 있는 정보의 크기는 변하지 않는다.

- **등장 배경 및 발전 과정**:
  1. **수기 계산의 시대**: 초기 컴퓨터 과학자들은 천공 카드에 구멍을 뚫기 위해 10진수 연산 결과를 수작업으로 2진수로 변환해야 했다. (나눗셈-나머지 기법의 활용)
  2. **하드웨어 디코더의 등장**: BCD(Binary Coded Decimal) 회로와 7세그먼트 디스플레이 디코더가 등장하며, 10진수 ↔ 2진수 변환을 하드웨어 로직 게이트가 직접 수행하기 시작했다.
  3. **소프트웨어 런타임 최적화**: 오늘날에는 Java, Python 등 고수준 언어의 런타임(Runtime)이나 표준 라이브러리(`Integer.parseInt`, `itoa` 등)가 고도로 최적화된 비트 시프트 연산을 통해 눈 깜짝할 사이에 진법 변환을 처리한다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 주요 변환 알고리즘

| 변환 유형 | 핵심 원리 및 알고리즘 | 시간 복잡도 | 비유 |
|:---|:---|:---|:---|
| **10진수 → R진수** | 타겟 기수 $R$로 연속해서 나누고(정수부), 곱하기(소수부). 나머지를 역순 취합. | $O(\log_R N)$ | 바구니에 나누어 담기 |
| **R진수 → 10진수** | 각 자리의 숫자와 $R^n$ (자리 가중치)을 곱하여 모두 더함 (다항식 전개). | $O(D)$ ($D$: 자릿수) | 바구니 안의 내용물 모두 합치기 |
| **2진수 ↔ 16진수** | **4비트(1니블)씩 그룹화**하여 16진수 1자리로 1:1 직역. 나눗셈 불필요. | $O(D)$ (비트 마스킹) | 4명씩 짝지어 조장 뽑기 |
| **2진수 ↔ 8진수** | **3비트씩 그룹화**하여 8진수 1자리로 1:1 직역. | $O(D)$ (비트 마스킹) | 3명씩 짝지어 조장 뽑기 |

### 정수부와 소수부의 10진수 → 2진수 변환 원리

10진수를 2진수로 변환할 때 정수부는 '나눗셈(Division)'을, 소수부는 '곱셈(Multiplication)'을 사용한다. 이 과정에서 소수부가 0으로 떨어지지 않고 무한 반복되면 부동소수점 오차가 발생한다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │        [ 10진수 13.625 ]  →  [ 2진수 변환 과정 ]             │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  1. 정수부 (13) 변환: 2로 나누며 나머지(Remainder) 취합            │
  │     13 / 2 = 6  ... 나머지 1  (LSB)                           │
  │      6 / 2 = 3  ... 나머지 0                                  │
  │      3 / 2 = 1  ... 나머지 1                                  │
  │      1 / 2 = 0  ... 나머지 1  (MSB)                           │
  │     ──────────────────────────────                           │
  │     ▶ 아래에서 위로 읽음: 1101 (2진수)                           │
  │                                                              │
  │  2. 소수부 (0.625) 변환: 2를 곱하며 정수부(Carry) 취합              │
  │      0.625 × 2 = 1.25  ... 정수부 1  (최상위 소수)               │
  │      0.25  × 2 = 0.50  ... 정수부 0                            │
  │      0.50  × 2 = 1.00  ... 정수부 1  (떨어짐, 종료)               │
  │     ──────────────────────────────                           │
  │     ▶ 위에서 아래로 읽음: .101 (2진수)                           │
  │                                                              │
  │  3. 최종 결합: 13.625 (10진수) = 1101.101 (2진수)                │
  └──────────────────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────────────────┐
  │          [ 2진수 ↔ 16진수 ] 초고속 1:1 매핑 변환                │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │   2진수:    1011  0101  1111  0000                           │
  │             │     │     │     │                              │
  │             ▼     ▼     ▼     ▼                              │
  │   16진수:     B     5     F     0     (단순 치환, 연산 없음)       │
  │                                                              │
  │  * 2진수와 16진수/8진수 간의 변환은 승수 배수 관계(16=2^4)이므로,    │
  │    CPU(ALU)의 복잡한 나눗셈 회로를 거치지 않고 비트 결합만으로 처리됨. │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상단 다이어그램은 인간의 숫자(13.625)가 컴퓨터의 숫자(1101.101)로 바뀌는 고전적인 알고리즘을 보여준다. 정수부는 계속 반으로 쪼개면서 남는 홀수/짝수 여부(나머지)를 기록하고, 소수부는 계속 2배로 부풀리면서 1을 넘치게(Carry) 만드는 방식으로 작동한다. 만약 소수부가 `0.1`이라면 2를 곱해도 영원히 0으로 떨어지지 않아 `0.000110011...`로 순환하게 되며, 이것이 IT 시스템에서 돈을 계산할 때 부동소수점을 쓰면 안 되는 이유를 수학적으로 증명한다. 하단 다이어그램은 2진수와 16진수의 아름다운 상호 변환을 보여준다. 16진수 1자리는 2진수 4자리와 수학적으로 완전한 동치이므로 연산이 아닌 '단순 매핑(룩업 테이블)'으로 변환이 끝난다.

## Ⅲ. 융합 비교 및 다각도 분석

### 심층 비교: 10진수 기반 vs 2진수 기반 진법의 변환 오버헤드

| 비교 항목 | 10진수 ↔ 2진수 (이종 변환) | 16진수 ↔ 2진수 (동종 진법군 변환) |
|:---|:---|:---|
| **CPU 연산 비용** | 매우 높음 (무거운 나눗셈/나머지 연산 수반) | 극히 낮음 (단순 비트 시프트 및 마스킹) |
| **하드웨어 지원** | 느리고 복잡한 곱셈기/나눗셈기 의존 | 배선 연결(Wiring) 수준에서 즉시 처리 가능 |
| **정밀도 손실** | 소수점 변환 시 순환소수 발생(오차 가능성) | 완벽한 1:1 치환 (오차 제로) |
| **실무적 해결책** | I/O 경계에서만 변환, 내부 연산은 2진수로 통일 | 시스템 내부 덤프, 로그 남길 때 적극 활용 |

### 과목 융합 관점

- **컴파일러 구조 (Software Engineering)**: 소스 코드에 `int a = 255;` (10진수)와 `int b = 0xFF;` (16진수)를 작성했을 때, 컴파일러의 렉서(Lexer)는 이를 파싱하여 어셈블리어(기계어)로 번역할 때 두 값을 완벽히 동일한 2진수 비트 패턴(`11111111`)으로 변환하여 심볼 테이블에 저장한다. 즉, 진법은 프로그래머의 가독성을 위한 것일 뿐, 컴파일 후의 바이너리에는 진법의 개념이 사라진다.
- **알고리즘 및 자료구조 (Algorithm)**: 큰 수(Big Integer)의 연산이나 암호학(RSA, ECC)에서 수백~수천 비트의 숫자를 다룰 때, 이를 10진수 문자열로 변환하여 출력하는 `toString()` 함수 자체가 $O(N^2)$의 무거운 알고리즘 부하를 발생시킨다. 따라서 디버깅 시에는 10진수 변환 대신 16진수 변환($O(N)$)을 사용하는 것이 압도적으로 빠르다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 대용량 데이터 덤프 로깅 최적화**: 초당 10만 건의 트랜잭션을 처리하는 시스템에서, 디버깅을 위해 메모리 상의 바이너리 페이로드를 로그로 남겨야 한다.
   - **문제**: 페이로드를 10진수 배열 `[255, 10, 128...]` 형태로 변환하여 로깅하자, CPU의 나눗셈 연산 부하가 치솟고 I/O 스레드가 지연(Blocking)되었다.
   - **해결책**: 10진수 변환을 전면 제거하고 16진수 헥스 덤프(Hex Dump, 예: `FF 0A 80...`) 형태로 변환 로직을 수정한다. 16진수 변환은 배열 크기만큼의 비트 시프트(`>> 4`)와 비트 마스킹(`& 0xF`), 그리고 미리 정의된 16바이트 룩업 테이블(`"0123456789ABCDEF"`) 접근만으로 끝나므로 CPU 부하를 거의 0에 가깝게 줄일 수 있다.

2. **시나리오 — 센서 데이터 소수점 전송 (IoT)**: 온도 센서가 측정한 값 `23.45`도를 서버로 전송해야 한다.
   - **문제**: 이 값을 32비트 Float(IEEE 754) 2진수 바이트 스트림으로 변환하여 보내면, 서버 측에서 언어(Python, JS 등)나 파서에 따라 `23.449998` 등으로 해석되는 부동소수점 변환 오차가 발생한다.
   - **해결책**: 진법 변환 오차를 원천 차단하기 위해 센서(Client) 측에서 10진수 `23.45`에 100을 곱해 `2345`라는 **정수(Integer)**로 만든 뒤 전송한다. 서버는 이 정수 2진수를 받아 다시 10진수 시스템에서 100으로 나누어 화면에 표시한다. (Fixed Point 방식 채택)

### 도입 체크리스트
- **기술적**: 클라이언트와 서버 간 데이터 통신 시, JSON 등 10진수 문자열 기반 포맷을 쓸 것인가, Protobuf/FlatBuffers 등 2진수 직렬화 포맷을 쓸 것인가? (CPU 변환 비용 vs 가독성 트레이드오프)
- **운영적**: 시스템 코어 덤프(Core Dump)를 분석하는 엔지니어들이 16진수(Hex) 오프셋 표기법 및 리틀 엔디안 방식의 읽기에 충분히 숙련되어 있는가?

### 안티패턴
- **반복적인 진법 파싱(Parsing)**: 반복문 내부에서 `Integer.parseInt(str, 10)` 이나 `toString()`을 수만 번 호출하는 행위. 진법 변환은 문자열을 스캔하고, 매 글자마다 기수를 곱하고 더하는 무거운 수학 연산이다. 성능이 중요한 핫 패스(Hot Path)에서는 문자열 기반의 10진수 숫자 처리를 배제하고 원시 2진수(바이트/인트) 상태로 데이터를 다뤄야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 최적화 사례 | 기대효과 |
|:---|:---|:---|
| **정량** | 반복 루프 내 10진수 출력(로깅)을 16진수 출력으로 변경 | CPU 로그 포매팅 소요 시간 **70% 이상 단축** |
| **정량** | 실수 전송 시 Fixed-Point(정수 변환) 기법 도입 | 진법 변환에 의한 소수점 오차 **0% (완전 무결성)** |
| **정성** | 바이너리 직렬화(Protobuf 등)로 전환 | 파싱(10진수 변환) 오버헤드 제거로 시스템 처리량 증대 |

### 미래 전망
앞으로 하드웨어가 아무리 발전해도 인간은 10진수를 버리지 못할 것이고, 디지털 기계는 2진수를 벗어날 수 없다. 따라서 진법 변환은 영원한 숙제다. 최근에는 SIMD(단일 명령어 다중 데이터) 벡터 명령어를 활용하여 수십 바이트의 10진수 문자열을 단 1~2사이클 만에 2진수 정수로 일괄 변환하는 초고속 파싱 알고리즘(예: Daniel Lemire의 `simdjson`)이 등장하며 소프트웨어 파싱 병목을 하드웨어 가속으로 타개하고 있다.

### 참고 표준
- **POSIX `strtol`, `strtod`**: C 표준 라이브러리에서 문자열을 특정 기수(Radix)의 정수/실수로 안전하고 일관되게 변환하도록 규정한 함수 표준.
- **IEEE 754-2008 (Decimal Floating Point)**: 10진수 ↔ 2진수 변환 오차를 소프트웨어 없이 하드웨어 레벨에서 없애기 위해 제정된 십진 부동소수점 규격.

## 👶 어린이를 위한 3줄 비유 설명
1. 진법 변환은 한국 돈을 미국 달러로 바꾸는 **환전소**와 같아요. 돈의 모양(숫자)은 바뀌지만 돈의 진짜 가치는 그대로랍니다.
2. 사람들은 손가락이 10개라 '10진수'라는 돈을 쓰고, 컴퓨터는 스위치가 2개라 '2진수'라는 돈을 써요. 서로 대화하려면 항상 이 환전소를 거쳐야 하죠.
3. 소수점이 있는 돈을 환전할 때는 가끔 1원짜리 동전이 딱 맞아떨어지지 않아서 계산이 아주 조금 틀려지는(부동소수점 오차) 재미있는 마법이 일어나기도 해요!
