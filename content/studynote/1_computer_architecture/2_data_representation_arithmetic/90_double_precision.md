# 배정밀도 (Double Precision, FP64)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 배정밀도(Double Precision, FP64)는 IEEE 754 표준의 64비트 부동소수점 포맷으로, 1비트 부호·11비트 지수·52비트 가수로 구성되어 약 15~16자리 십진수 정밀도와 10³⁰⁸ 범위를 제공한다.
> 2. **가치**: 과학·공학·금융 시뮬레이션에서 요구되는 "오차 누적 허용 불가" 요건을 충족시키는 최소 정밀도로, 천문학(천체 궤도), 기상 예측, 구조 해석, 몬테카를로 시뮬레이션 등의 표준이다.
> 3. **융합**: FP64는 CPU(고성능 코어)의 기본 연산 단위이며, GPU에서는 FP32 대비 성능이 1/2~1/64로 떨어지지만, HPC(High-Performance Computing)와 AI 훈련의 Master Weight 저장에는 필수적이다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 배정밀도(Double Precision)는 64비트(8바이트)를 사용하여 실수를 표현하는 IEEE 754 표준 포맷이다. C/C++의 `double`, Java/C#의 `double`, Python의 `float`(기본 64비트), Fortran의 `DOUBLE PRECISION`이 모두 이에 해당한다. 내부 구조는 부호(S) 1비트, 지수(E) 11비트, 가수(M) 52비트로 구성되며, 암시적 선행 1을 포함하면 실제 정밀도는 53비트(약 15.9 십진 자릿수)다.

- **💡 비유**: 배정밀도는 **'전문가용 망원경'**과 같다. 일반 망원경(FP32)으로도 달을 볼 수 있지만, 전문가용 망원경은 목성의 위성들이 정확히 어디에 있는지, 얼마나 떨어져 있는지를 16자리 숫자만큼 정밀하게 측정할 수 있다. 단, 망원경이 무겁고 비싸서(메모리 2배, 연산량 2배 이상) 일반인용이 아니다.

- **등장 배경 및 발전 과정**:
  1. **과학 계산의 요구 (1960~70년대)**: 핵물리, 우주 궤도 계산, 기상 예측에서 7자리 정밀도(FP32)는 부족했다. 오차가 수천 번의 연산 후 누적되면 결과가 완전히 무의미해졌다.
  2. **IEEE 754-1985의 표준화**: 64비트 포맷을 "double"로 명명하고, 53비트 유효숫자(약 16자리)와 11비트 지수(±308 범위)를 표준화했다. 이후 모든 주요 언어와 하드웨어가 이를 채택했다.
  3. **현대 HPC와의 관계**: TOP500 슈퍼컴퓨터 성능은 FP64 FLOPS로 순위가 매겨진다. 2024년 Frontier는 1.2 ExaFLOPS(FP64)를 달성했다. AI 가속기는 FP16/FP8에 최적화되지만, 과학 계산용 GPU(NVIDIA H100, AMD MI300)는 FP64 성능을 30~50 TFLOPS까지 끌어올린다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 비트 레이아웃과 값 계산

FP64의 64비트는 3개의 필드로 나뉘며, 값은 수식 $(-1)^S \times (1 + M) \times 2^{E-1023}$으로 계산된다(정규화 수 기준).

```text
  ┌──────────────────────────────────────────────────────────────────────────────┐
  │                 FP64 (배정밀도) 비트 레이아웃와 값 해석                         │
  ├──────────────────────────────────────────────────────────────────────────────┤
  │                                                                              │
  │   63     62             52    51                                       0     │
  │   ┌───┬──────────────────┬────────────────────────────────────────────────┐ │
  │   │ S │   Exponent (E)   │                Mantissa (M)                    │ │
  │   └───┴──────────────────┴────────────────────────────────────────────────┘ │
  │    1비트      11비트                          52비트                        │
  │                                                                              │
  │   ─────────────────────────────────────────────────────────────────────    │
  │   값 계산 (정규화 수: 0 < E < 2047):                                          │
  │                                                                              │
  │   Value = (-1)^S × (1.M) × 2^(E-1023)                                        │
  │                   ↑                                                          │
  │             암시적 선행 1 (Implicit Leading 1)                                │
  │             52비트 M 앞에 1이 붙어 53비트 유효숫자가 됨                         │
  │                                                                              │
  │   ─────────────────────────────────────────────────────────────────────    │
  │   FP64 주요 경계값:                                                           │
  │   ┌─────────────────────────────────────────────────────────────────────┐  │
  │   │ 최소 양의 정규화 수: E=1, M=0  → 2^-1022 ≈ 2.23 × 10^-308           │  │
  │   │ 최대 정규화 수:     E=2046, M=全1 → ≈ 1.80 × 10^308               │  │
  │   │ 최소 서브노멀:     E=0, M=1   → 2^-1074 ≈ 4.94 × 10^-324          │  │
  │   │ 정밀도(ulp):       2^-52 ≈ 2.22 × 10^-16 (약 16자리)               │  │
  │   │ 2^53 = 9,007,199,254,740,992 (정수를 정확히 표현 가능한 최댓값)      │  │
  │   └─────────────────────────────────────────────────────────────────────┘  │
  │                                                                              │
  │   ─────────────────────────────────────────────────────────────────────    │
  │   FP64로 10진수 12.375를 표현하는 과정:                                        │
  │                                                                              │
  │   ① 10진수 → 2진수:  12.375 = 1100.011₂                                      │
  │   ② 정규화:  1100.011₂ = 1.100011₂ × 2³                                      │
  │   ③ 각 필드 결정:                                                             │
  │      S = 0 (양수)                                                             │
  │      E = 3 + 1023 = 1026 = 10000000010₂                                      │
  │      M = 1000110000000000000000000000000000000000000000000000₂               │
  │   ④ 최종 64비트:                                                              │
  │      0 10000000010 1000110000000000000000000000000000000000000000000000      │
  │      └─┬──┘ └─────┬─────┘ └──────────────────────┬──────────────────────┘   │
  │        S        E=1026                      M (52비트)                       │
  │                                                                              │
  └──────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** FP64는 FP32에 비해 지수가 3비트, 가수가 29비트 늘어났다. 편향값 1023은 지수를 -1022~+1023 범위로 옮긴다. 53비트 유효숫자(약 log₁₀(2⁵³) ≈ 15.95자리)는 십진수 16자리까지 정확히 표현 가능함을 의미한다. 특히 2⁵³ = 9,007,199,254,740,992 이하의 정수는 FP64에서 오차 없이 정확히 표현된다. 이는 53비트 정수형(INT53)과 동등한 정밀도다.

### 정수와의 관계: 안전한 정수 범위

FP64의 중요한 특성은 2⁵³(약 9×10¹⁵) 이하의 정수를 손실 없이 정확히 표현할 수 있다는 점이다. JavaScript의 `Number` 타입이 FP64이므로, 이 범위를 초과하면 정수로서의 정확성이 깨진다.

```text
  ┌──────────────────────────────────────────────────────────────────────────────┐
  │         FP64의 정수 표현 능력과 안전한 정수 범위 (Safe Integer)                │
  ├──────────────────────────────────────────────────────────────────────────────┤
  │                                                                              │
  │   FP64 유효숫자: 53비트 (암시적 1 + 명시적 52)                                 │
  │                                                                              │
  │   정수 표현 가능 범위: -2^53 ~ 2^53 (포함)                                    │
  │   ┌───────────────────────────────────────────────────────────────────────┐ │
  │   │  2^53 = 9,007,199,254,740,992                                         │ │
  │   │  2^53 + 1 = 9,007,199,254,740,993 ← FP64에서 2^53과 동일하게 표현됨!    │ │
  │   │  2^53 + 2 = 9,007,199,254,740,994 ← 구분 가능 (ULP = 2)               │ │
  │   └───────────────────────────────────────────────────────────────────────┘ │
  │                                                                              │
  │   JavaScript 예시:                                                           │
  │   ┌───────────────────────────────────────────────────────────────────────┐ │
  │   │  9007199254740992 === 9007199254740993  // true (같은 FP64 값!)        │ │
  │   │  9007199254740992 + 1 === 9007199254740992  // true                   │ │
  │   │  9007199254740992 + 2 === 9007199254740994  // true                   │ │
  │   │  Number.MAX_SAFE_INTEGER === 2^53 - 1  // 9007199254740991            │ │
  │   └───────────────────────────────────────────────────────────────────────┘ │
  │                                                                              │
  │   실수선 (2^53 주변):                                                        │
  │                                                                              │
  │   2^53          2^53+2        2^53+4        2^53+6                          │
  │   │              │             │             │                               │
  │   └── ULP = 2 ──┘                                             │              │
  │   (2^53+1은 표현 불가, 2^53+2로 반올림됨)                       │              │
  │                                                                              │
  │   ⚠ 교훈: 10^15(천조) 이상의 ID, 금액 등은 BigInt 또는 문자열로 처리해야 함   │
  │                                                                              │
  └──────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 2⁵³ 이상에서는 ULP가 2 이상이 되어, 연속된 홀수/짝수 정수를 구분할 수 없다. JavaScript는 모든 숫자를 FP64로 처리하므로, `Number.MAX_SAFE_INTEGER`(2⁵³-1)를 넘는 정수는 `BigInt`로 처리해야 한다. 금융 시스템에서 10¹⁵ 원(천 조 원) 이상의 금액을 JavaScript `Number`로 처리하면 오류가 발생할 수 있다. 이는 JSON 직렬화에서도 문제가 되며, 64비트 정수를 받는 API는 문자열로 전송해야 안전하다.

### FP64 연산의 하드웨어 성능 비교

FP64 연산은 FP32에 비해 하드웨어 자원을 2배 이상 소모하며, GPU에서는 FP32 대비 성능이 크게 떨어진다.

| 하드웨어 | FP32 성능 | FP64 성능 | FP64/FP32 비율 |
|:---|:---|:---|:---:|
| **Intel Xeon (SSE/AVX)** | 기준 | 기준 | 1:1 (거의 동일) |
| **AMD EPYC (AVX-512)** | 기준 | 기준 | 1:1 (거의 동일) |
| **NVIDIA RTX 4090** | 82.6 TFLOPS | 1.29 TFLOPS | 1:64 |
| **NVIDIA A100 (Data Center)** | 19.5 TFLOPS | 9.7 TFLOPS | 1:2 |
| **NVIDIA H100 (HPC)** | 67 TFLOPS | 34 TFLOPS | 1:2 |
| **AMD MI300X** | 163 TFLOPS | 81 TFLOPS | 1:2 |

```text
  ┌──────────────────────────────────────────────────────────────────────────────┐
  │         GPU 아키텍처별 FP64 성능 차이의 원인: FP64 유닛 수                     │
  ├──────────────────────────────────────────────────────────────────────────────┤
  │                                                                              │
  │   Consumer GPU (RTX 4090): FP64 유닛이 FP32 유닛의 1/64                       │
  │   ┌───────────────────────────────────────────────────────────────────────┐ │
  │   │  [CUDA Core]  [CUDA Core]  [CUDA Core]  ... (10,752개)               │ │
  │   │       │            │            │                                     │ │
  │   │    FP32만      FP32만       FP32만       ← 대부분 FP32 전용           │ │
  │   │                                                                       │ │
  │   │  [특수 유닛] × 소수만 FP64 지원 → FP64 성능 = FP32의 1/64            │ │
  │   └───────────────────────────────────────────────────────────────────────┘ │
  │                                                                              │
  │   Data Center GPU (A100/H100): FP64 유닛이 FP32 유닛의 1/2                   │
  │   ┌───────────────────────────────────────────────────────────────────────┐ │
  │   │  [FP64 Unit]  [FP64 Unit]  [FP64 Unit]  ...                          │ │
  │   │       │            │            │                                     │ │
  │   │    FP32/64     FP32/64      FP32/64      ← 하이브리드 설계             │ │
  │   │                                                                       │ │
  │   │  FP64 성능 = FP32의 1/2 (HPC용으로 설계됨)                            │ │
  │   └───────────────────────────────────────────────────────────────────────┘ │
  │                                                                              │
  │   ⚠ 이유: FP64 연산기는 FP32 대비 게이트 수가 약 4배, 전력 소모가 높음       │
  │      Consumer GPU는 게임/ML용으로 FP32를 우선, HPC GPU는 FP64를 우선         │
  │                                                                              │
  └──────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** GPU 아키텍처에서 FP64 성능이 떨어지는 이유는 FP64 연산기를 위한 실리콘 면적과 전력 소모가 크기 때문이다. Consumer GPU(게이밍용)는 FP64가 거의 필요 없으므로(게임, 비디오, ML 추론은 FP32/FP16으로 충분), NVIDIA는 FP64 유닛을 최소화하여 코어 수를 늘리고 전력을 아낀다. 반면 Data Center GPU(A100, H100)는 기상 예측, 유체 역학, 분자 동역학 등 FP64가 필수적인 HPC 워크로드를 위해 FP64 유닛을 FP32의 1/2까지 탑재한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: FP64 vs FP32 (배정밀도 vs 단정밀도)

| 비교 항목 | FP64 (배정밀도) | FP32 (단정밀도) |
|:---|:---|:---|
| **비트 수** | 64 (8바이트) | 32 (4바이트) |
| **유효숫자** | 53비트 (~16자리) | 24비트 (~7자리) |
| **지수 범위** | 10⁻³⁰⁸ ~ 10³⁰⁸ | 10⁻³⁸ ~ 10³⁸ |
| **메모리 대역폭** | 2배 소모 | 기준 |
| **안전한 정수 범위** | 2⁵³ (9×10¹⁵) | 2²⁴ (1.6×10⁷) |
| **CPU 성능** | FP32와 유사 | 기준 |
| **GPU 성능** | FP32의 1/2 ~ 1/64 | 기준 |
| **주 용도** | 과학 계산, 금융, 천문학 | 게임, ML, 오디오 |

### 비교 2: FP64 vs 64비트 정수(INT64)

| 비교 항목 | FP64 (부동소수점) | INT64 (정수) |
|:---|:---|:---|
| **표현 범위** | ±1.8×10³⁰⁸ | ±9.2×10¹⁸ |
| **정확한 정수 표현** | 2⁵³까지만 | 2⁶³-1까지 |
| **소수점 표현** | 가능 | 불가능 (별도 스케일링 필요) |
| **연산 오차** | 반올림 오차 존재 | 오차 없음 (정확한 결과) |
| **나눗셈** | 부동소수점 결과 | 정수 몫(버림) |

### 과목 융합 관점

- **컴퓨터구조(컴구)**: x86-64 CPU는 AVX-512를 통해 512비트 벡터 레지스터(ZMM)에 FP64 8개를 동시에 연산한다. 이론적 최대 처리량은 클럭당 2 FMA × 8 FP64 = 16 FLOP/cycle이다. 3GHz CPU 코어 하나당 48 GFLOPS(FP64)다.

- **운영체제(OS)**: 64비트 프로세스에서 `double`은 8바이트 정렬(alignment)이 권장된다. 잘못된 정렬은 성능 저하(일부 아키텍처에서는 크래시)를 유발한다. `malloc`은 보통 16바이트 정렬을 보장한다.

- **네트워크(NW)**: FP64를 네트워크로 전송할 때는 바이트 오더(Big/Little Endian)에 주의해야 한다. x86은 Little Endian이지만, 네트워크 표준은 Big Endian이다. `htond`/`ntohd`(double용)는 표준이 아니므로 직접 구현해야 한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 기상 예측 모델의 정밀도 요건**: 전지구 기상 모델(ECMWF, GFS)은 대기의 상태를 3차원 격자로 나누어 계산한다. 격자 수가 10⁹ 개, 시간 단계가 10⁴ 개이면 연산 횟수는 10¹³ 회 이상이다. FP32의 오차(10⁻⁷)가 10¹³ 번 누적되면 최종 오차는 10⁶(백만) 수준에 도달하여 예측이 무의미해진다. 기술사는 FP64를 기본으로 사용하고, I/O와 메모리를 절약하기 위해 결과 저장만 FP32로 수행하는 전략을 취해야 한다.

2. **시나리오 — JavaScript에서의 큰 정수 ID 처리**: 소셜 미디어 API가 사용자 ID로 64비트 정수(예: 9,223,372,036,854,775,807)를 반환한다. JavaScript `Number`로 파싱하면 2⁵³ 이상의 비트가 손실되어, ID 검색이 실패한다. 기술사는 API 응답에서 ID를 문자열로 받도록 수정하거나, `BigInt`를 사용하여 정확히 파싱해야 한다.

3. **시나리오 — GPU 선택에서의 FP64 요건**: 분자 동역학 시뮬레이션(LAMMPS, GROMACS)을 GPU 가속으로 수행해야 한다. 게이밍 GPU(RTX 4090)는 저렴하지만 FP64 성능이 1.29 TFLOPS에 불과하고, Data Center GPU(H100)는 비싸지만 34 TFLOPS를 제공한다. 기술사는 워크로드의 FP64 의존도를 분석한 후, FP64가 90% 이상이면 H100을, 혼합 정밀도가 가능하면 RTX 4090 클러스터를 선택한다.

### 도입 체크리스트
- **기술적**: 문제의 조건 수(Condition Number)가 10¹⁰ 이상인가? 그렇다면 FP64로도 부족할 수 있으며, 4배정밀도(FP128) 또는 임의 정밀도 라이브러리(GNU MPFR)를 고려해야 한다.
- **운영·보안적**: 금융 계산에서 FP64를 사용할 때, 오차 허용 범위가 규제 문서에 명시되어 있는가? 10⁻¹⁵ 오차도 수천 번의 연산 후 누적되면 의미 있을 수 있다.

### 안티패턴
- **게이밍 GPU로 FP64 과부하**: RTX 4090으로 기상 시뮬레이션을 수행하면, FP64 성능 부족으로 CPU보다 느려질 수 있다. HPC 워크로드는 Data Center GPU를 사용해야 한다.
- **FP64로 화폐 계산**: FP64도 0.1, 0.01을 정확히 표현할 수 없다. 화폐는 여전히 고정소수점 또는 정수 연산이 필요하다.

### 정량/정성 기대효과

| 구분 | FP32 사용 시 | FP64 사용 시 | 효과 |
|:---|:---|:---|:---|
| **정량** | 7자리 정밀도 | 16자리 정밀도 | 오차 10⁹ 배 감소 |
| **정량** | 오차 누적 심각 | 오차 누적 허용 가능 | 시뮬레이션 신뢰성 확보 |
| **정성** | 과학 계산 부적합 | 과학 계산 표준 | 논문/특허 결과 재현 가능 |

### 미래 전망
- **혼합 정밀도 AI 훈련**: 2025년 현재, 딥러닝 훈련은 FP16/BF16으로 수행하지만, "Master Weights"는 FP64가 아닌 FP32로 저장하는 것이 일반적이다. FP64는 과학 ML(Science ML, PINN)에서만 요구된다.
- **FP64 Tensor Core**: NVIDIA Hopper(H100) 아키텍처는 FP64에도 Tensor Core를 적용하여, 종래의 2배 성능(34 TFLOPS)을 달성했다. 과학 AI(Science for AI) 분야의 성장이 배경이다.

## 👶 어린이를 위한 3줄 비유 설명
1. 배정밀도(FP64)는 **'전문가용 슈퍼 망원경'**이에요. 일반 망원경(FP32)보다 2배 무겁지만, 별의 위치를 16자리 숫자까지 정확히 측정할 수 있어요!
2. 과학자들은 이 망원경으로 날씨를 예측하거나, 로켓의 궤도를 계산해요. 조금만 틀려도 로켓이 화성 대신 엉뚱한 곳으로 갈 수 있거든요.
3. 일반 게임이나 스마트폰 앱은 이렇게 정밀한 계산이 필요 없어서, 가볍고 빠른 일반 망원경(FP32)을 써요. 적재적소에 도구를 선택하는 게 중요하답니다!
