# 162. 범용 레지스터 (GPR, General Purpose Register)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 특정 용도에 제한 없이 다양한 데이터 저장 가능한 레지스터
> 2. **가치**: 프로그래밍 유연성, 컴파일러 최적화, 파이프라인 효율
> 3. **융합**: 레지스터 할당, 함수 호출 규약(ABI), 컨텍스트 스위칭

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

범용 레지스터(GPR, General Purpose Register)는 **특정 용도에 제한되지 않고 다양한 목적으로 사용할 수 있는 CPU 내부 레지스터**다. 데이터 저장, 주소 계산, 산술 연산 등 모든 용도로 활용 가능하다.

### 💡 비유: 다용도 서랍

GPR은 **다용도 서랍**과 같다. 서랍마다 "양말", "속옷"처럼 용도가 정해진 게 아니라 필요한 것을 넣을 수 있다. 오늘은 양말을, 내일은 장갑을 넣을 수 있듯이 GPR도 필요한 데이터를 자유롭게 저장한다.

### GPR 특징

```
┌─────────────────────────────────────────────────────────────────────┐
│                GPR 특징                                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【GPR의 특성】                                                      │
│  ──────────────                                                     │
│                                                                     │
│  • 범용성: 모든 데이터 타입 저장 가능                               │
│  • 동일 접근: 모든 GPR은 동일한 속도로 접근                          │
│  • 프로그래머 관리: 컴파일러/프로그래머가 용도 결정                   │
│  • 개수: ISA에 따라 8~128개 (RISC-V: 32개, ARM64: 31개)             │
│                                                                     │
│  【GPR vs 특수 목적 레지스터(SPR)】                                  │
│  ────────────────────────────────────                               │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  구분           GPR                 SPR                        │ │
│  │  ─────────────  ───────────────────  ────────────────────     │ │
│  │  용도           모든 용도             특정 용도                  │ │
│  │  개수           다수 (16~128)         소수 (3~10)               │ │
│  │  명칭           R0~R31, X0~X30       PC, SP, LR, FP            │ │
│  │  하드웨어       동일 구조             특수 기능                  │ │
│  │  프로그래머      자유 사용             제한적 사용                │ │
│  │  예시           데이터 저장           PC: 명령어 주소            │ │
│  │                연산 결과              SP: 스택 포인터             │ │
│  │                주소 계산             LR: 복귀 주소               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【ISA별 GPR 수】                                                    │
│  ──────────────────────                                             │
│                                                                     │
│  ISA          GPR 개수    명명 규칙         비고                    │
│  ──────────   ─────────  ───────────────  ──────────               │
│  x86 (32비트)   8        EAX~EDI          제한적                    │
│  x86-64         16       RAX~R15          확장됨                    │
│  ARM32          16       R0~R15           R13(SP),R14(LR),R15(PC)  │
│  ARM64          31       X0~X30           X31(Zero), SP 별도       │
│  RISC-V (32)    32       x0~x31           x0=0                      │
│  MIPS32         32       $0~$31           $0=0, $31=ra             │
│  SPARC          32       %g,%o,%l,%i      레지스터 윈도우           │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  GPR이 많을수록:                                                 │ │
│  │  ────────────────                                                │ │
│  │  • 메모리 접근 감소 → 성능 향상                                   │ │
│  │  • 더 많은 변수 레지스터 저장                                    │ │
│  │  • 컴파일러 최적화 용이                                          │ │
│  │  • but 컨텍스트 스위칭 비용 증가                                 │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

### 레지스터 윈도우 (Register Window)

```
┌─────────────────────────────────────────────────────────────────────┐
│                레지스터 윈도우 (SPARC 예시)                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【개념】                                                            │
│  ────────                                                            │
│  • 함수 호출마다 다른 레지스터 세트 사용                            │
│  • 일부 레지스터는 윈도우 간 공유                                   │
│  • 컨텍스트 스위칭 비용 감소                                        │
│                                                                     │
│  【SPARC 레지스터 윈도우 구조】                                     │
│  ────────────────────────────────                                   │
│                                                                     │
│  전역(g)     출력(o)    지역(l)    입력(i)                        │
│  ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐                       │
│  │ %g0  │   │ %o0  │   │ %l0  │   │ %i0  │ ← 이전 윈도우의 %o     │
│  │ %g1  │   │ %o1  │   │ %l1  │   │ %i1  │                       │
│  │ ...  │   │ ...  │   │ ...  │   │ ...  │                       │
│  │ %g7  │   │ %o7  │   │ %l7  │   │ %i7  │                       │
│  └──────┘   └──────┘   └──────┘   └──────┘                       │
│     ↑          │                   ↑         │                     │
│     │          └───────────────────┘         │                     │
│     │                공유                    │                     │
│   모든 윈도우에서 동일                      │                     │
│                                              ▼                     │
│                                        다음 윈도우로 전달          │
│                                                                     │
│  【윈도우 오버플로우/언더플로우】                                    │
│  ────────────────────────────────────                               │
│                                                                     │
│  • 오버플로우: 윈도우 모두 사용 → 메모리에 저장                      │
│  • 언더플로우: 윈도우 부족 → 메모리에서 복원                        │
│  • 트랩 발생 → OS가 처리                                            │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  장단점:                                                         │ │
│  │  ────────                                                         │ │
│  │  장점:                                                           │ │
│  │  • 함수 호출 시 레지스터 저장/복원 불필요                        │ │
│  │  • 빠른 함수 호출/반환                                           │ │
│  │                                                               │ │
│  │  단점:                                                           │ │
│  │  • 더 많은 레지스터 필요                                         │ │
│  │  • 윈도우 오버플로우 시 오버헤드                                  │ │
│  │  • 현대 CPU에서는 덜 사용                                        │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### GPR vs 누산기 vs 스택

| 구분 | GPR | 누산기 | 스택 |
|:---|:---|:---|:---|
| **레지스터 수** | 16~128 | 1~2 | 0 |
| **유연성** | 높음 | 낮음 | 낮음 |
| **명령어 길이** | 김 | 짧음 | 짧음 |
| **파이프라인** | 용이 | 어려움 | 어려움 |
| **현대 사용** | 주류 | 임베디드 | 일부 |

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

**시나리오 1: 컴파일러 최적화**
- 문제: 레지스터 부족으로 성능 저하
- 분석: 레지스터 활용률 확인
- 해결: 레지스터 할당 알고리즘 개선
- 결과: 20% 성능 향상

**시나리오 2: 함수 호출 빈도 높음**
- 문제: 호출 오버헤드 큼
- 분석: Caller/Callee-saved 레지스터 사용
- 해결: 자주 쓰는 변수를 Callee-saved로
- 결과: 호출 비용 30% 감소

**시나리오 3: 실시간 시스템**
- 문제: 컨텍스트 스위칭 지연 변동
- 분석: GPR 저장/복원 시간
- 해결: 레지스터 수 줄인 MCU 사용
- 결과: 일정한 응답 시간

**시나리오 4: 임베디드 저전력**
- 문제: 전력 소모 큼
- 분석: 레지스터 파일 전력
- 해결: GPR 수가 적은 아키텍처
- 결과: 15% 전력 절감

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 요소 | GPR 많을 때 | GPR 적을 때 |
|:---|:---|:---|
| **메모리 접근** | 적음 | 많음 |
| **성능** | 높음 | 낮음 |
| **전력** | 높음 | 낮음 |
| **컨텍스트 비용** | 높음 | 낮음 |

### 핵심 요약

```
• GPR: 용도 제한 없는 범용 레지스터
• 개수: ISA별 8~128개
• 역할: 데이터 저장, 연산, 주소 계산
• ABI: 호출 규약으로 용도 정의
• 최적화: 레지스터 할당이 핵심
```
