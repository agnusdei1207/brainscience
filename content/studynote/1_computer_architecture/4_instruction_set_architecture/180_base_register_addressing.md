> **핵심 인사이트**
> 1. 베이스 레지스터 주소 지정(Base Register Addressing)은 지정된 **베이스 레지스터에 저장된 '기준 주소(Base Address)'에 명령어의 오퍼랜드 필드에 있는 '변위(Displacement)' 값을 더하여 유효 주소(EA)를 산출**하는 방식이다.
> 2. 프로그램이 메모리의 어느 위치에 적재되더라도 코드 내부의 주소를 수정할 필요 없이 베이스 레지스터 값만 바꾸면 되는 **재배치 가능 코드(Relocatable Code)**를 구현하는 핵심 기술이다.
> 3. 멀티프로그래밍 환경에서 운영체제가 각 프로세스에게 독립된 메모리 영역(세그먼트)을 할당하고 보호하는 **메모리 세그멘테이션(Memory Segmentation)** 기법의 하드웨어적 근간이 된다.

---

## Ⅰ. 베이스 레지스터 주소 지정의 원리

이 방식은 변위 주소 지정(Displacement Addressing)의 대표적인 파생 형태입니다. 
수식으로 표현하면 $EA = (Base Register) + Displacement$ 입니다.

- **Base Register (기준점):** 보통 운영체제(OS)가 프로그램(프로세스)을 메모리에 적재(Load)할 때, 그 프로그램이 시작되는 메모리의 첫 번째 물리적 주소를 이 레지스터에 기록해 둡니다.
- **Displacement (상대 거리):** 컴파일러가 만들어낸 기계어 명령어 내부에 고정된 상수로 기록되며, 프로그램의 시작점으로부터 대상 데이터까지 떨어진 거리(Offset)를 의미합니다.

```ascii
[베이스 레지스터 주소 지정 방식의 구조]

+------------------+-------------------------+
| Opcode (예: LOAD)| Base Reg | Disp(예: 50) |
+------------------+----------+--------------+
                       |              |
    [ Base Register ]  |              |
    | (OS가 1000 할당)  |              |
    +------------------+              |
                |                     |
                v                     v
            [ + ] (ALU에서 1000 + 50 연산 수행)
                  \         /
                   \       /
                    v     v
                [ EA = 1050 ] ---> [ 메인 메모리 참조 ]
```

📢 **섹션 요약 비유:**
아파트 단지(메모리)에서 "101동 입구(베이스 주소)에서부터 50걸음(변위) 떨어진 벤치"를 찾아가는 것과 같습니다.

## Ⅱ. 프로세스 재배치 (Process Relocation)와 OS의 역할

초창기 컴퓨터(단일 프로그래밍)에서는 프로그램이 항상 메모리의 0번지부터 적재된다고 가정하고 직접 주소 지정(Direct Addressing)으로 코딩을 했습니다. 하지만 여러 프로그램을 동시에 실행하는 멀티프로그래밍 시대가 오면서 문제가 생겼습니다.

1. **절대 주소의 한계:**
   - A 프로그램이 0번지에 적재되어 실행 중일 때, B 프로그램을 실행하려면 B는 0번지가 아닌 빈 공간(예: 5000번지)에 들어가야 합니다. 만약 B 프로그램 내부에 "100번지 데이터를 읽어라"라는 코드가 하드코딩되어 있다면, 엉뚱하게 A 프로그램의 영역을 침범하게 됩니다.

2. **베이스 레지스터의 마법:**
   - B 프로그램은 메모리의 0번지부터 시작한다고 가정한 채로 명령어(변위 100)를 만듭니다(논리 주소).
   - OS가 B 프로그램을 5000번지에 적재할 때, B 프로세스의 베이스 레지스터 값만 5000으로 설정해 줍니다.
   - CPU가 명령어를 실행할 때 하드웨어가 자동으로 `5000 + 100 = 5100`으로 물리 주소를 변환(Mapping)합니다.

📢 **섹션 요약 비유:**
캠핑장에서 텐트를 어디에 치든(메모리 적재), 텐트 입구(베이스 레지스터)를 기준으로 "텐트 입구에서 3걸음 앞 화로대"라는 설계도(코드)는 항상 똑같이 적용될 수 있습니다.

## Ⅲ. 인덱스 주소 지정과의 차이점 (Base vs Index)

베이스 레지스터 방식과 인덱스 주소 지정 방식은 구조($Reg + Disp$)가 똑같습니다. 하지만 **어느 쪽이 변하고 어느 쪽이 고정되어 있는지**에 따라 용도가 완전히 다릅니다.

| 비교 항목 | 베이스 레지스터 주소 지정 | 인덱스 주소 지정 (Indexed) |
| :--- | :--- | :--- |
| **기준 레지스터 값** | 프로그램 실행 내내 **고정 (OS가 세팅)** | 루프를 돌 때마다 **1씩 증가 (변동)** |
| **오퍼랜드 (변위) 값** | 데이터나 함수의 위치를 가리키는 **상수 (다양함)** | 배열의 시작 주소로 **고정** |
| **주요 활용 목적** | 메모리 재배치(Relocation), 구조체 멤버 접근 | 배열(Array) 데이터의 순차 탐색 |
| **주체** | 주로 **운영체제(OS)**와 하드웨어 | 주로 **프로그래머**와 컴파일러 |

📢 **섹션 요약 비유:**
베이스 방식은 엘리베이터(베이스)가 고정된 층에 서 있고 내가 층 안에서 걸어 다니는(변위) 것이라면, 인덱스 방식은 나는 가만히 서 있고(변위 고정) 에스컬레이터 계단(인덱스)이 계속 올라가며 뷰를 바꿔주는 것입니다.

---

## Ⅳ. 메모리 보호 (Memory Protection)와 세그멘테이션

베이스 레지스터는 단순히 주소를 더해주는 것을 넘어, **다른 프로그램의 메모리를 훔쳐보지 못하게 막는 보안의 핵심**입니다.

- **한계 레지스터 (Limit Register)와의 결합:**
  베이스 레지스터 주소 지정 구조를 갖춘 CPU는 짝꿍으로 '한계 레지스터'를 둡니다. 이 레지스터에는 해당 프로그램이 사용할 수 있는 최대 크기(예: 2000바이트)가 적혀 있습니다.
- **하드웨어 인터럽트 (Trap):**
  CPU가 계산한 변위(Offset) 값이 한계 레지스터에 적힌 크기(2000)를 초과하면, 즉시 '세그멘테이션 결함(Segmentation Fault)' 트랩(인터럽트)을 발생시켜 해당 프로그램을 강제 종료(Kill)시킵니다.

📢 **섹션 요약 비유:**
베이스 레지스터가 "여기서부터 놀아라"라고 시작 선을 그어준다면, 한계 레지스터는 "저 선 밖으로 나가면 퇴장!"이라고 끝 선을 그어주는 엄격한 심판과 같습니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[주소 지정 방식 (Addressing Modes)]`: 베이스 레지스터를 활용한 변위 주소 계산
- `[Relocatable Code (재배치 가능 코드)]`: 물리적 메모리 위치에 얽매이지 않는 독립적인 프로그램
- `[Memory Management Unit (MMU)]`: 베이스 및 한계 레지스터를 포함하여 주소 변환과 메모리 보호를 수행하는 하드웨어
- `[Segmentation (세그멘테이션)]`: 베이스 레지스터의 개념을 확장하여 프로세스 메모리를 논리적 단위로 관리하는 기법
- `[Segmentation Fault (Segfault)]`: 프로그램이 허용된 변위 범위를 초과하여 메모리에 접근하려 할 때 발생하는 치명적 오류

## 👶 어린이를 위한 3줄 비유 설명
1. 베이스 레지스터 주소 지정은 "우리 반 교실 앞문"을 기준점으로 정해놓고 물건을 찾는 방식이에요.
2. 반이 1층에서 3층으로 이사(프로그램 적재)를 가도, "앞문에서 3걸음 뒤"라는 보물지도(코드)는 새로 그릴 필요가 없어요.
3. 기준점(앞문)만 3층 교실로 딱 바꿔주면 컴퓨터가 알아서 똑똑하게 물건 위치를 찾아낸답니다!
