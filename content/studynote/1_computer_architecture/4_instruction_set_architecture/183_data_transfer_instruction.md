> **핵심 인사이트**
> 1. 데이터 전송 명령어(Data Transfer Instruction)는 레지스터와 레지스터, 혹은 레지스터와 메모리 간에 **데이터를 '복사(Copy)'하여 이동시키는 가장 빈번하게 사용되는 명령어 그룹**이다.
> 2. 실제 데이터의 연산이나 수정은 일어나지 않으며, 데이터를 연산하기 위해 위치를 준비하거나 연산 결과를 저장하는 **'사전/사후 작업'의 핵심**을 담당한다.
> 3. RISC 아키텍처에서는 메모리에 접근할 수 있는 유일한 통로인 `Load`와 `Store` 명령어로 엄격하게 제한되어, 파이프라인의 단순성과 속도 향상의 기반이 된다.

---

## Ⅰ. 데이터 전송 명령어의 본질

'데이터 전송'이라는 이름이 붙어있지만, 실제 동작은 데이터를 잘라내어 옮기는 것(Move)이 아니라 복사해서 덮어쓰는 것(Copy & Overwrite)입니다. 즉, 원본 데이터는 그대로 유지되고 목적지의 데이터만 새로운 값으로 변경됩니다.

- **역할:** 산술 논리 연산 장치(ALU)가 연산을 수행하기 위해서는 모든 데이터가 반드시 레지스터(Register)에 있어야 합니다. 데이터 전송 명령어는 이 ALU에게 재료를 먹여주고, 요리가 끝난 결과물을 다시 창고(메모리)에 보관하는 셔틀 역할을 합니다.

```ascii
[데이터 전송 명령어의 흐름]

+--------------------+     (Load)    +---------------------+
|    Main Memory     |  ---------->  |    Register File    |
| [1000: Data A]     |               | [R1: Data A]        |
| [1004: Data B]     |  <----------  | [R2: Result C]      |
+--------------------+     (Store)   +---------------------+
```

📢 **섹션 요약 비유:**
요리사(ALU)가 직접 창고(메모리)에 가지 않도록, 보조 주방장(데이터 전송 명령어)이 창고에서 재료를 도마(레지스터)로 가져오고, 다 만든 요리를 다시 창고에 가져다 놓는 심부름꾼 역할입니다.

---

## Ⅱ. 주요 명령어 유형과 특징

아키텍처에 따라 명령어의 니모닉(Mnemonic, 어셈블리어 기호)은 다르지만, 크게 세 가지 범주로 나뉩니다.

### 1. Load (메모리 $\rightarrow$ 레지스터)
- **개념:** 메인 메모리에 있는 특정 주소의 데이터를 읽어서 CPU 내부의 레지스터로 복사합니다.
- **예시:** `LDR R1, [R2]` (ARM), `LW $t0, 0($t1)` (MIPS)
- **특징:** 캐시 미스(Cache Miss) 발생 시 엄청난 지연(Stall)을 유발할 수 있어, 컴파일러는 가급적 Load 명령어를 분산시키거나 일찍 실행하도록 스케줄링합니다.

### 2. Store (레지스터 $\rightarrow$ 메모리)
- **개념:** CPU 레지스터에 있는 연산 결과 값을 메인 메모리의 특정 주소로 내보내어 복사(저장)합니다.
- **예시:** `STR R1, [R2]` (ARM), `SW $t0, 0($t1)` (MIPS)

### 3. Move (레지스터 $\rightarrow$ 레지스터)
- **개념:** CPU 내부의 레지스터 간에 데이터를 복사합니다. 메모리 참조가 없으므로 가장 빠릅니다.
- **예시:** `MOV R1, R2` (x86, ARM)
- **트릭:** 일부 순수 RISC 구조에서는 Move 명령어가 따로 없고, 0을 가지고 있는 0번 레지스터(Zero Register)와 더하는 연산(`ADD R1, R2, $0`)으로 Move를 대신 구현하기도 합니다.

📢 **섹션 요약 비유:**
택배 수령(Load), 택배 발송(Store), 그리고 내 양손 안에서 물건 바꿔 들기(Move)의 세 가지 행동으로 데이터를 옮깁니다.

## Ⅲ. CISC vs RISC의 데이터 전송 철학 비교

데이터 전송 명령어를 대하는 철학은 컴퓨터 아키텍처를 양분하는 CISC와 RISC의 가장 결정적인 차이점입니다.

| 비교 항목 | CISC (예: x86) | RISC (예: ARM, RISC-V) |
| :--- | :--- | :--- |
| **메모리 접근 권한** | 거의 **모든 명령어**가 메모리에 직접 접근 가능 | 오직 **Load와 Store 명령어**만이 메모리 접근 가능 |
| **명령어 복잡도** | 메모리에서 메모리로 직접 복사하는 등 매우 복잡함 | 레지스터 단위로 엄격히 통제되어 단순함 |
| **파이프라이닝** | 명령어 길이가 다르고 실행 사이클이 불규칙해 까다로움 | 명령어 길이가 같고 메모리 접근이 격리되어 파이프라이닝 극대화 |

- **CISC 예시:** `ADD [1000], [2000]` (메모리 간 덧셈 허용)
- **RISC 예시:** 위 연산을 하려면 `LOAD R1, [1000]`, `LOAD R2, [2000]`, `ADD R3, R1, R2`, `STORE R3, [1000]` 의 4개 명령어로 반드시 쪼개야 합니다.

📢 **섹션 요약 비유:**
CISC는 만능 요리사라서 창고에 있는 재료를 가져오면서 동시에 칼질까지 다 해버리지만, RISC는 분업이 철저한 공장이라서 배달부는 배달(Load/Store)만 하고 요리사는 칼질(ALU)만 하도록 규칙을 정했습니다.

## Ⅳ. 스택 조작과 데이터 전송 (Push & Pop)

메모리 중에서도 **스택(Stack)** 영역을 조작하는 `PUSH`와 `POP` 역시 본질적으로는 특수한 형태의 데이터 전송 명령어입니다.

1. **PUSH (레지스터 $\rightarrow$ 스택 메모리):**
   - 현재 데이터를 보존하기 위해 레지스터의 값을 메모리(스택 탑)에 저장하고, 스택 포인터(SP)의 주소를 자동으로 1 증가/감소시킵니다.
   - 논리적 의미: `STORE` 명령어 + SP `자동 증감`
2. **POP (스택 메모리 $\rightarrow$ 레지스터):**
   - 스택 탑에 있는 데이터를 레지스터로 복사해 오고, SP의 주소를 반대로 돌려놓습니다.
   - 논리적 의미: `LOAD` 명령어 + SP `자동 증감`

📢 **섹션 요약 비유:**
일반 Load/Store가 빈 공간 아무 데나 자유롭게 물건을 넣고 빼는 것이라면, Push/Pop은 길쭉한 프링글스 통(스택)에 순서대로 과자를 쌓고(Store) 맨 위에서부터 다시 꺼내 먹는(Load) 규칙이 추가된 전송 방식입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Load/Store Architecture]`: 데이터 전송 명령어를 통해서만 메모리에 접근하게 하는 RISC의 설계 철학
- `[Memory Hierarchy]`: Load 명령어가 데이터를 가져오기 위해 거쳐야 하는 캐시-메인메모리의 계층 구조
- `[Register File]`: 데이터 전송 명령어의 목적지이자 출발점이 되는 가장 빠른 저장 공간
- `[Stack Pointer (SP)]`: Push/Pop 명령어 수행 시 자동 갱신되는 포인터 레지스터
- `[Data Hazard]`: Load 명령어로 데이터를 가져오기 전에 다음 명령어가 그 데이터를 쓰려고 할 때 발생하는 파이프라인 충돌
