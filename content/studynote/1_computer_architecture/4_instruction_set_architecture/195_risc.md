> **핵심 인사이트**
> 1. RISC(Reduced Instruction Set Computer)는 **'단순한 명령어를 아주 빠르게, 1사이클에 실행하자'**는 철학을 바탕으로 명령어의 종류와 길이를 최소화하고 고정시킨 컴퓨터 아키텍처이다.
> 2. 오직 Load/Store 명령어만이 메모리에 접근할 수 있고 나머지 연산은 전부 레지스터 내부에서만 이루어지는 **Load/Store 아키텍처**를 채택하여 파이프라이닝 효율을 극대화했다.
> 3. 하드웨어 제어 유닛이 단순해져 전력 소모가 적고 칩 면적을 아낄 수 있어, 현대의 **모바일(ARM) 및 임베디드, 그리고 최신 데이터센터(Graviton 등) 시장을 완전히 장악**했다.

---

## Ⅰ. RISC의 탄생 배경과 철학 (The 80/20 Rule)

과거 메모리가 극도로 비싸고 컴파일러 기술이 부족했던 시절에는 복잡한 명령어를 가진 CISC가 주류였습니다. 하지만 1980년대 컴퓨터 과학자들이 프로그램들의 실제 실행 통계를 분석하면서 놀라운 사실을 발견합니다.

- **80/20 법칙 (Amdahl's Insight):** 전체 프로그램 명령어의 20%에 불과한 아주 단순한 명령어들(LOAD, STORE, ADD 등)이 전체 실행 시간의 80% 이상을 차지하고 있었습니다.
- **RISC의 철학:** "어쩌다 한 번 쓰는 복잡한 명령어를 위해 하드웨어를 무겁게 만들지 말자! 그런 복잡한 일은 단순한 명령어 여러 개를 조합해서 소프트웨어(컴파일러)가 알아서 하게 만들고, 하드웨어는 가장 자주 쓰이는 **단순한 명령어 20%를 무조건 1클럭 사이클 만에 찍어내듯 실행**하는 데 올인하자!"

📢 **섹션 요약 비유:**
수백 가지 요리 도구가 꽉 찬 복잡한 주방(CISC) 대신, 오직 '가장 잘 드는 식칼'과 '도마' 딱 두 개만 놓고(RISC) 셰프의 손놀림(컴파일러)을 극한으로 연마하여 요리 속도를 미친 듯이 끌어올린 식당입니다.

## Ⅱ. RISC 아키텍처의 4대 핵심 특징

1. **고정 길이 명령어 (Fixed-Length Instruction):**
   - 모든 명령어가 예외 없이 32비트(또는 64비트)로 크기가 똑같습니다. 따라서 CPU가 명령어를 읽어 올 때(Fetch) 명령어의 끝이 어딘지 계산할 필요 없이 무조건 4바이트씩 뚝뚝 끊어 읽으면 됩니다.
2. **Load/Store 아키텍처:**
   - 메모리 접근은 무조건 `LOAD`와 `STORE` 두 명령어로만 가능합니다. 덧셈(`ADD`) 같은 연산 명령어는 절대 메모리에 손을 댈 수 없고 오직 레지스터 값만 다룹니다. (메모리 병목 철저히 격리)
3. **풍부한 범용 레지스터 (Large Register File):**
   - 연산을 레지스터에서만 해야 하므로, CISC보다 훨씬 많은 수(보통 32개 이상)의 범용 레지스터(GPR)를 하드웨어에 탑재합니다.
4. **하드와이어드 제어 (Hardwired Control):**
   - 명령어가 너무 단순하기 때문에, 이를 해독하기 위한 복잡한 마이크로프로그램(Micro-code ROM) 없이 순수 논리 회로만으로 제어 유닛을 만들어 속도를 극대화합니다.

📢 **섹션 요약 비유:**
모든 택배 상자의 크기가 똑같고(고정 길이), 배달부(Load/Store)와 공장 조립 직원(ALU)의 역할이 완벽히 분리되어 있으며, 공장 안에 부품 보관함(레지스터)이 아주 많은 최첨단 컨베이어 벨트 시스템입니다.

## Ⅲ. 파이프라이닝 (Pipelining)과의 환상적인 궁합

RISC가 세상을 지배하게 된 결정적 이유는 명령어 파이프라이닝과의 궁합이 완벽했기 때문입니다.

- 파이프라인(IF-ID-EX-MEM-WB)이 막힘없이 돌아가려면 **각 단계(Stage)가 걸리는 시간이 똑같아야** 합니다.
- CISC는 어떤 명령어는 1사이클, 어떤 건 10사이클이 걸려서 파이프라인이 엉망진창이 됩니다.
- 하지만 RISC는 모든 명령어의 길이가 같고(IF 단계 일정), 메모리 접근은 정해진 명령어만 하며(MEM 단계 격리), 연산은 무조건 레지스터에서만 하므로(EX 단계 1사이클), **모든 명령어가 마치 공장 톱니바퀴처럼 1사이클마다 딱딱 맞춰서 쏟아져 나옵니다(CPI $\approx$ 1).**

📢 **섹션 요약 비유:**
자동차 조립 공장에서 문짝 달기, 바퀴 달기, 색칠하기 등 모든 작업이 정확히 1분씩만 걸리게 쪼개 놓아서, 1분마다 무조건 새 자동차가 한 대씩 완성되어 나오는 완벽한 시스템입니다.

## Ⅳ. RISC의 단점과 컴파일러의 부담

물론 RISC에도 치명적인 약점이 있었습니다. 하드웨어가 덜어낸 짐을 소프트웨어가 고스란히 짊어져야 했습니다.

1. **코드 밀도 저하 (Code Size 증가):**
   - CISC에서는 1줄이면 끝날 복잡한 작업(예: 메모리에서 메모리로 데이터 더하기)을 RISC에서는 4줄(`LOAD`, `LOAD`, `ADD`, `STORE`)로 늘려 써야 합니다.
   - 이로 인해 프로그램의 용량이 커지고, 명령어 캐시(L1 I-Cache) 미스가 자주 발생할 위험이 커집니다.
2. **컴파일러(Compiler) 최적화 필수:**
   - 텅 빈 레지스터를 효율적으로 분배하고, 파이프라인이 멈추지 않도록 명령어의 순서를 교묘하게 섞어주는(Instruction Scheduling) 컴파일러의 똑똑한 지능이 없으면 RISC는 제 성능을 낼 수 없습니다.

📢 **섹션 요약 비유:**
간단한 도구밖에 없기 때문에 화려한 요리를 만들려면 레시피(코드)가 수십 장으로 늘어나고, 이 레시피를 기가 막히게 잘 짜는 일류 셰프(컴파일러)가 없으면 밥이 나올 수 없는 구조입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. RISC는 복잡한 스위치가 다닥다닥 붙은 기계 대신, 가장 자주 쓰는 스위치 몇 개만 남기고 다 없애버린 심플한 기계예요.
2. 버튼이 몇 개 없으니까 기계가 무겁지도 않고 배터리(전력)도 아주 조금만 먹어서 스마트폰에 딱 맞아요.
3. 대신 복잡한 일을 시키려면, 간단한 버튼 여러 개를 다다다닥 엄청나게 빠르게 눌러서 해결해야 한답니다!
