# 로드/스토어 아키텍처 (Load/Store Architecture)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 로드/스토어 (Load/Store) 아키텍처는 메모리에 접근하는 명령어(Load, Store)와 산술/논리 연산을 수행하는 명령어를 **엄격히 분리**하여, 연산 명령어는 오직 CPU 내부 레지스터만을 대상으로 동작하게 설계된 구조이다.
> 2. **가치**: 명령어 실행 시간을 일정하게 유지함으로써 **명령어 파이프라이닝 (Pipelining)**의 효율을 극대화하며, 메모리 지연(Latency)이 연산 유닛에 직접적인 영향을 주지 않도록 격리한다.
> 3. **융합**: RISC(Reduced Instruction Set Computer) 설계의 핵심 기둥이며, 현대의 ARM, RISC-V 등 대부분의 고성능 프로세서가 복잡한 메모리 직접 연산을 배제하고 이 구조를 채택하고 있다.

---

## Ⅰ. 로드/스토어 아키텍처의 정의 및 특징

### 1. 정의
- 산술 연산(ADD, SUB 등)의 피연산자가 반드시 레지스터여야만 하는 구조이다. 메모리에 있는 데이터를 더하려면 반드시 먼저 레지스터로 읽어와야(Load) 한다.

### 2. 주요 특징: '관심사의 분리'
- **연산 명령어**: 오직 레지스터 파일(Register File)하고만 소통한다. 실행 시간이 매우 짧고 일정하다.
- **전송 명령어**: 메모리와 레지스터 사이의 데이터 이동만 전담한다. (Load: Memory→Reg, Store: Reg→Memory)
- **단순한 제어 유닛**: 명령어마다 메모리 접근 여부를 고민할 필요가 없어 하드웨어 로직이 단순해진다.

### 3. 💡 비유: '요리사와 재료 관리자'
- 로드/스토어 아키텍처는 **'요리사(연산 유닛)는 조리대(레지스터) 위에 있는 재료만 가지고 요리하고, 재료 관리자(전송 유닛)가 창고(메모리)에서 재료를 가져다주는 분업 시스템'**과 같다.
- 요리사가 직접 창고를 왔다 갔다 하지 않아도 되므로, 요리사는 쉬지 않고 요리(연산)에만 집중할 수 있어 주방의 전체 효율(Throughput)이 올라간다.

---

## Ⅱ. 동작 메커니즘 비교 (Deep Dive)

수식 `C = A + B`를 처리하는 과정의 차이다.

### 1. Register-Memory 구조 (CISC)
- `ADD C, A, B` ──▶ 단 한 줄로 끝난다. 하지만 내부적으로는 [메모리 읽기 x2] -> [덧셈] -> [메모리 쓰기]의 복잡한 단계가 숨어 있다.

### 2. Load/Store 구조 (RISC)
1. **LOAD R1, A**: 메모리 A의 값을 R1로 가져옴.
2. **LOAD R2, B**: 메모리 B의 값을 R2로 가져옴.
3. **ADD R3, R1, R2**: 레지스터 R1, R2를 더해 R3에 저장. (메모리 접근 없음)
4. **STORE R3, C**: R3의 결과값을 메모리 C에 저장.

### 3. ASCII 구조 다이어그램
```text
    [ Load/Store Logic ]
    ┌────────────────────────────────────────────────────────┐
    │  Memory Access Instructions (LOAD, STORE only)         │
    └───────────┬───────────────────────────────────▲────────┘
                │ (Data)                            │ (Data)
                ▼                                   │
    ┌───────────────────────────────────────────────┴────────┐
    │               Register File (GPRs)                     │
    └───────────┬───────────────────┬───────────────▲────────┘
                │ (Operand 1)       │ (Operand 2)   │ (Result)
                ▼                   ▼               │
    ┌───────────────────────────────────────────────┴────────┐
    │           ALU (Arithmetic Logic Unit)                  │ ◀── 메모리와 격리됨
    └────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 복잡한 바깥세상(메모리)과 차단된 깨끗한 실험실(연산기) 환경을 조성하여, 연산의 순수성과 속도를 보존하는 아키텍처입니다.

---

## Ⅲ. 로드/스토어 구조의 장점과 기술적 도전

| 구분 | 장점 (Pros) | 기술적 도전 (Cons) |
|:---|:---|:---|
| **성능** | 파이프라인 단계별 소요 시간의 균일화 | **코드 팽창 (Code Bloat)**: 명령어 수가 늘어남 |
| **설계** | 하드웨어 복잡도 감소 및 전력 효율 향상 | 많은 수의 **범용 레지스터 (GPR)** 확보 필수 |
| **확장성** | 비순차 실행(OoO) 및 스케줄링 최적화 유리 | 컴파일러의 정교한 레지스터 할당 능력 요구 |

- **📢 섹션 요약 비유**: 문장이 길어지는 단점(명령어 증가)은 있지만, 발음이 꼬이지 않고 일정하게 내뱉을 수 있어 전체적인 대화 속도(성능)가 빨라지는 것과 같습니다.

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: 고성능 CPU의 파이프라인 설계
- **상황**: 5단계 파이프라인(IF-ID-EX-MEM-WB)을 가진 프로세서를 설계함.
- **판단**: **Load/Store 구조**를 채택한다. 모든 산술 연산은 EX(실행) 단계에서 끝나고, 메모리 접근은 오직 LOAD/STORE 명령어만 MEM 단계에서 수행한다. 이를 통해 연산 명령어는 MEM 단계를 그냥 지나치게(Bypass) 설계하여 하드웨어 자원 충돌(Hazard)을 최소화한다.

### 2. 기술사적 인사이트: 메모리 의존성 (Memory Dependency)
- 기술사라면 Load/Store 아키텍처가 **메모리 벽 (Memory Wall)** 문제를 어떻게 관리하는지 논해야 한다. 연산과 메모리 접근이 분리되어 있으므로, 똑똑한 스케줄러는 LOAD 명령어를 실제 연산보다 훨씬 앞서 실행(Prefetching)하거나, 데이터를 기다리는 동안 다른 연산을 수행하는 **지연 숨기기 (Latency Hiding)** 전략을 취할 수 있음을 강조해야 한다.

- **📢 섹션 요약 비조**: 요리(연산)가 끊기지 않도록, 재료(데이터)를 미리미리 꺼내놓는 스마트한 주방 관리 기법이 로드/스토어 아키텍처의 정수입니다.

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **클럭 주기 단축**: 하드웨어 경로 단순화를 통한 고주파수 달성 가능.
- **컴파일러 최적화 용이**: 명확한 데이터 흐름 분석을 통한 코드 재배치 지원.

### 2. 미래 전망
앞으로는 메모리 지연 시간이 더욱 극심해짐에 따라, 로드/스토어 구조는 하드웨어가 자동으로 데이터를 가져오는 **자율적 데이터 인출** 기술과 결합할 것이다. 또한, 인공지능 전용 프로세서(NPU)에서는 수만 개의 데이터를 한 번에 로드하고 레지스터에서 한꺼번에 연산하는 극단적인 로드/스토어 방식이 지배할 것이다. 로드/스토어 아키텍처는 속도가 지배하는 현대 컴퓨팅의 가장 강력한 질서이다.

- **📢 섹션 요약 비유**: 복잡한 절차를 걷어내고 '가져오기'와 '하기'를 명확히 나눈, 분업화된 현대 지식 산업의 공정 시스템과 같습니다.
