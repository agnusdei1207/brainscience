# 보호 도메인 (Protection Domain)과 최소 권한의 원칙

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스가 실행 중에 접근할 수 있는 자원(객체)과 해당 자원에 대해 행사할 수 있는 권한(Access Right)의 집합을 **보호 도메인**이라 하며, 이를 업무 수행에 필요한 **최소한의 수준으로만 제한**하는 것이 최소 권한의 원칙이다.
> 2. **가치**: 특정 프로세스가 해킹당하거나 오작동하더라도, 그 영향 범위를 해당 도메인 내로 한정시켜 시스템 전체의 피해(Blast Radius)를 최소화한다.
> 3. **융합**: 운영체제의 사용자 모드/커널 모드 분리, 가상화의 샌드박싱, 그리고 현대의 클라우드 IAM 정책 수립의 철학적 및 기술적 근간이 된다.

---

### Ⅰ. 보호 도메인 (Protection Domain)의 개념

- **구성 요소**: `Domain = { <object, {rights-set}>, ... }`
- **정적/동적 전이**:
  - **정적 도메인**: 프로세스 생명주기 동안 권한이 고정됨.
  - **동적 도메인 (Domain Switching)**: 실행 단계에 따라 필요한 권한이 달라질 때 도메인을 교체함 (예: 일반 실행 $\to$ 시스템 콜 호출 시 권한 상승).

---

### Ⅱ. 최소 권한의 원칙 (Principle of Least Privilege)

- **정의**: 프로그램, 사용자, 시스템이 작업을 완료하는 데 필요한 **가장 낮은 수준의 권한**만 할당받아야 한다는 보안 원칙.
- **💡 비유**: **인턴 사원에게 회사 전체 마스터키를 주지 않는 것**과 같다. 인턴은 자신이 일하는 사무실 문만 열 수 있는 열쇠만 가져야 하며, 그래야 실수로 서버실에 들어가 장비를 망가뜨리는 일을 막을 수 있다.

---

### Ⅲ. 보호 도메인 구현 아키텍처 (ASCII)

프로세스가 실행 문맥에 따라 도메인을 전환하며 권한을 제어하는 모습이다.

```ascii
    [ Subject: Process P1 ]
           |
           | (Execution Context A)
           v
    +-----------------------+      +-----------------------+
    |  Domain D1 (Normal)   |      |  Domain D2 (Privileged)|
    |-----------------------|      |-----------------------|
    | File A: Read          |      | File A: Write         |
    | Printer: Use          |      | Network: All          |
    +-----------|-----------+      +-----------^-----------+
                |                              |
                +---- (Domain Switching) ------+
                (via System Call / SetUID)
```

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 구현 기술: Capability vs ACL
- **Capability**: 프로세스가 권한 증표(Token)를 직접 들고 다니는 방식. (주체 중심)
- **ACL**: 자원에 허가 명단을 붙여놓는 방식. (객체 중심)
- **기술사적 결단**: 현대 시스템은 관리가 용이한 **ACL**을 기본으로 하되, 고성능이나 분산 환경이 필요한 경우 **Capability(OAuth Token 등)** 방식을 혼용하여 상호보완적인 보안 체계를 구축한다.

#### 2. 기술사적 인사이트: 권한 상승 공격 (Privilege Escalation)
- 공격자는 취약한 프로세스의 보호 도메인을 탈취한 후, 더 높은 권한의 도메인(Root)으로 전이하려 시도한다. 이를 막기 위해 **Sandboxing** 기술을 사용하여 도메인 간의 물리적 장벽을 강화해야 한다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **침해 사고 피해 감소**: 해킹 발생 시 공격자의 활동 범위를 원천 봉쇄.
- **시스템 가용성 향상**: 앱의 버그가 커널이나 다른 앱의 자원을 건드리는 사고 예방.

#### 2. 미래 전망
최근의 **제로 트러스트 (Zero Trust)** 아키텍처는 최소 권한의 원칙을 실시간으로 구현한 것이다. 과거처럼 한 번 로그인하면 모든 권한을 주는 것이 아니라, 매 요청마다 "지금 이 작업을 할 권한이 있는가"를 검증하는 **Just-in-Time (JIT) 권한 부여** 방식으로 발전하고 있으며, 이는 운영체제 보호 도메인 이론의 궁극적인 실천 모델이다.

### 👶 어린이를 위한 3줄 비유 설명
1. **보호 도메인**은 우리 집의 '내 방'과 같은 구역이에요. 내 방에서는 장난감을 맘대로 할 수 있지만, 동생 방 물건은 만질 수 없죠.
2. **최소 권한의 원칙**은 "딱 필요한 만큼만 허락받기"예요. 숙제할 때는 공책만 빌리고, 요리할 때만 주방 도구를 빌리는 규칙이죠.
3. 이렇게 정해진 곳에서만 활동하니까, 실수로 남의 소중한 장난감을 고장 내는 일을 미리 막을 수 있답니다!
