# 리눅스 컨테이너 네임스페이스(Namespace) 격리 메커니즘

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스 실행 시 시스템 자원(Process ID, Network, Mount 등)을 **논리적으로 분리된 전용 뷰(View)**로 제공하여, 동일한 커널을 공유하면서도 독립된 운영체제처럼 작동하게 하는 리눅스 커널 기술.
> 2. **가치**: 하드웨어 전체를 가상화하는 가상 머신(VM)보다 훨씬 가볍고 빠르며, 프로세스 간 간섭을 차단하여 **애플리케이션의 이식성과 보안성**을 극대화한다.
> 3. **융합**: 도커(Docker)나 쿠버네티스(Kubernetes)의 핵심 엔진이며, 자원 사용량을 제한하는 **Cgroups**와 결합하여 현대 '컨테이너 가상화'의 기술적 근간을 이룬다.

---

### Ⅰ. 네임스페이스 (Namespace)의 주요 유형

리눅스 커널은 7가지 이상의 독립적인 네임스페이스를 제공한다.

| 유형 | 격리 대상 | 역할 설명 |
|:---|:---|:---|
| **PID** | 프로세스 ID | 컨테이너 내부 프로세스에게 독립된 PID 부여 (컨테이너 내 PID 1번 생성). |
| **NET** | 네트워크 | 독립적인 네트워크 인터페이스, IP 주소, 라우팅 테이블 할당. |
| **MNT** | 마운트 | 컨테이너마다 독립적인 파일 시스템 트리 소유 (chroot의 확장). |
| **UTS** | 호스트네임 | 호스트명(Hostname)과 도메인명을 독립적으로 설정. |
| **IPC** | 프로세스 간 통신 | 메시지 큐, 세마포어 등 IPC 자원의 격리 및 보호. |
| **USER** | 사용자/그룹 | 호스트의 일반 사용자를 컨테이너 내에서 root 권한으로 매핑 가능 (보안). |

---

### Ⅱ. 컨테이너 격리 아키텍처 (ASCII)

하나의 커널 위에서 네임스페이스가 프로세스들을 어떻게 가두는지 보여준다.

```ascii
    [ User Application 1 ]      [ User Application 2 ]
    (PID: 1, IP: 10.0.0.1)      (PID: 1, IP: 10.0.0.2)
    +--------------------+      +--------------------+
    | [ Namespace A ]    |      | [ Namespace B ]    |
    +--------------------+      +--------------------+
              |                          |
    ========================================================
    [ Shared Linux Kernel ]
    - Process Management (Real PIDs: 1234, 5678)
    - Network Stack (Physical NIC)
    - File System (Root FS)
    ========================================================
    [ Physical Hardware ]
```

**[격리 동작 원리]**
- **논리적 분리**: 프로세스 1234는 자기 세상을 보며 "내 PID는 1이야"라고 믿는다.
- **물리적 공유**: 하지만 커널 입장에서는 프로세스 1234와 5678은 그저 커널이 관리하는 수많은 태스크(Task) 중 하나일 뿐이다.

---

### Ⅲ. 네임스페이스 생성 시스템 콜

운영체제는 새로운 격리 환경을 만들기 위해 아래 API를 제공한다.

1. **`clone()`**: 새로운 프로세스를 생성하면서 부모와 다른 네임스페이스를 할당.
2. **`unshare()`**: 현재 프로세스의 일부 자원을 새로운 네임스페이스로 분리.
3. **`setns()`**: 이미 존재하는 다른 네임스페이스로 프로세스를 진입시킴.

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 컨테이너 vs 가상 머신(VM) 선택
- **기술사적 결단**: 
  - 실행 속도와 밀도(Density)가 중요하다면 **컨테이너(Namespace)**를 선택한다.
  - 커널 수준의 완전한 격리와 서로 다른 OS 실행이 필요하다면 **가상 머신(Hypervisor)**을 선택한다.

#### 2. 기술사적 인사이트: 컨테이너 탈출 (Container Escape)
- 네임스페이스는 논리적인 벽일 뿐이므로, 커널의 취약점을 이용해 벽을 뚫고 호스트 시스템을 장악하는 공격에 취약할 수 있다. 이를 방어하기 위해 **AppArmor**나 **SELinux**와 같은 보안 모듈을 병행 사용하여 다층 방어를 구축해야 한다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **자원 효율성 극대화**: VM 대비 10~100배 이상의 많은 서비스를 동일 장비에서 실행.
- **배포 혁신**: 환경 종속성 제거를 통한 '한 번 빌드, 어디서나 실행(Build Once, Run Anywhere)'.

#### 2. 미래 전망
최근에는 네임스페이스 격리 기술을 더 미세하게 쪼갠 **마이크로 VM**이나, 웹 어셈블리(WASM)를 이용한 **런타임 격리** 기술이 컨테이너의 강력한 경쟁자로 떠오르고 있다. 하지만 리눅스 커널의 네임스페이스는 이미 수많은 인프라의 표준이 되었으며, 앞으로도 클라우드 네이티브 아키텍처의 가장 견고한 바닥 기술로 남을 것이다.
