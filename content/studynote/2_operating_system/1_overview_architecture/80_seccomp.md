# 시스템 호출 차단 (Seccomp)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Seccomp (secure computing mode)는 리눅스 커널이 제공하는 보안 메커니즘으로, 특정 프로세스가 호출할 수 있는 **시스템 호출(System Call)의 종류를 제한**하여 커널로의 진입로를 좁히는 샌드박스 기술이다.
> 2. **가치**: 응용 프로그램에 취약점이 있어 해커에게 장악되더라도, 위험한 시스템 호출(예: `execve`, `reboot`, `mount`)을 사전에 차단함으로써 커널을 보호하고 공격자의 권한 상승 시도를 무력화한다.
> 3. **융합**: BPF(Berkeley Packet Filter)와 결합한 **Seccomp-BPF**를 통해 시스템 콜의 인자값까지 검사하는 정교한 필터링을 수행하며, 현대 컨테이너 보안(Docker, Kubernetes)의 필수 방어 계층이 된다.

---

## Ⅰ. Seccomp의 개요

### 1. 정의
- 프로세스가 스스로의 권한을 낮추어 정해진 시스템 호출 외에는 아무것도 할 수 없게 만드는 '자발적 격리' 기술이다.

### 2. 등장 배경: '커널은 너무 넓고 위험하다'
- 리눅스 커널은 수백 개의 시스템 콜을 제공한다. 하지만 일반적인 앱(예: 웹 브라우저 렌더러)은 그중 일부만 사용한다. 나머지 불필요한 시스템 콜은 해커가 커널 취약점을 공격하는 통로(공격 표면)가 되는데, 이를 원천 봉쇄하기 위해 탄생했다.

### 3. 💡 비유: '은행 창구의 칸막이와 특정 서류 제한'
- 시스템 콜이 은행 금고(커널)로 가는 '창구'라면, Seccomp은 **'손님이 쓸 수 있는 신청서 종류를 딱 한두 개로 제한하는 것'**과 같다.
- 손님이 은행원(커널)을 협박해도 쓸 수 있는 서류가 '잔액 조회'뿐이라면, 금고를 열거나(execve) 은행 전체를 폐쇄(reboot)하는 일은 불가능해진다.

## Ⅱ. Seccomp의 동작 및 필터링 (Deep Dive)

### 1. Seccomp 모드 분류
- **Strict Mode (Mode 1)**: `read`, `write`, `exit`, `sigreturn` 4가지 시스템 콜만 허용한다. 매우 안전하지만 실용성이 낮다.
- **Filter Mode (Mode 2 / Seccomp-BPF)**: BPF 필터를 사용하여 허용할 시스템 콜 목록(Allow-list)과 인자 조건을 자유롭게 설정한다.

### 2. Seccomp-BPF 동작 과정
1. **필터 작성**: 허용/거부할 시스템 콜 번호와 조건을 담은 BPF 바이트코드를 작성한다.
2. **커널 로드**: `prctl()` 시스템 콜을 통해 작성된 필터를 커널에 주입한다.
3. **상속**: 한 번 설정된 필터는 `fork()`나 `exec()`를 통해 생성된 자식 프로세스에게도 강제 적용된다.
4. **위반 조치**: 프로세스가 금지된 호출을 시도하면 커널은 즉시 해당 프로세스를 종료(`SIGSYS`)하거나 에러를 반환한다.

### 3. ASCII 구조 다이어그램
```text
    [ User Process ]
           │
           │  ( syscall: reboot() 시도! )
           ▼
    ┌──────────────────────────────────┐
    │  Seccomp Filter (BPF Engine)     │
    ├──────────────────────────────────┤
    │  if (syscall == reboot) {        │
    │      Action: KILL_PROCESS;       │ ◀── 커널 진입 전 차단
    │  }                               │
    └──────────────────┬───────────────┘
                       │ (Blocked!)
                       ▼
    [   OS Kernel Internal Logic   ]
```

- **📢 섹션 요약 비유**: 건물 출입구(시스템 콜)에 경비원(Seccomp 필터)을 세워두고, 출입 명단(Allow-list)에 없는 사람은 절대 들여보내지 않는 철저한 검문소 시스템과 같습니다.

## Ⅲ. 실무 적용 및 컨테이너 보안 (Strategy)

### 1. 실무 시나리오: 도커(Docker) 기본 보안 프로필
- **상황**: 컨테이너 안에서 해커가 커널 모듈을 조작하려 함.
- **판단**: 도커는 기본적으로 약 300여 개의 시스템 콜 중 40여 개를 차단하는 **기본 Seccomp 프로필**을 적용한다. 해커가 컨테이너 내에서 `root` 권한을 얻더라도, 중요 시스템 콜이 Seccomp에 의해 막혀 있으므로 호스트 OS를 무너뜨릴 수 없다.

### 2. 기술사적 인사이트: 최소 권한의 원칙 (Least Privilege)
- Seccomp은 '최소 권한의 원칙'을 가장 강력하게 구현하는 도구이다. 기술사라면 앱의 프로파일링을 통해 실제로 사용하는 시스템 콜만 추출하여 **Custom Seccomp Profile**을 제작하고 배포하는 프로세스를 제안할 수 있어야 한다. 이는 방화벽에서 불필요한 포트를 닫는 것만큼이나 현대 시스템 보안에서 중요한 작업이다.

- **📢 섹션 요약 비유**: 칼을 쓰는 요리사에게는 칼(필요한 호출)만 주고, 망치나 드릴(위험한 호출)은 뺏어서 주방(샌드박스)의 안전을 지키는 지혜입니다.

## Ⅳ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **공격 표면 축소**: 커널 취약점 공격의 90% 이상을 입구에서 차단 가능.
- **심층 방어 (Defense in Depth)**: 파일 권한이나 사용자 격리가 뚫려도 마지막 방어선 역할 수행.

### 2. 미래 전망
최근에는 머신러닝을 사용하여 앱이 사용하는 시스템 콜을 자동으로 학습하고 최적화된 Seccomp 필터를 생성하는 기술이 연구되고 있다. 운영체제는 이제 모든 앱에게 동일한 커널 문을 열어주는 것이 아니라, 앱의 신원과 용도에 따라 **'개별화된 시스템 호출 관문'**을 제공하는 방향으로 진화할 것이다.

- **📢 섹션 요약 비유**: 모두에게 열린 광장이 아니라, 각자에게 꼭 필요한 통로만 허용하는 '스마트 게이트'가 운영체제의 미래 보안 표준입니다.
