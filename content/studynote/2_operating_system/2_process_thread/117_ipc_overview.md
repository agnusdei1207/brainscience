# 프로세스 간 통신 (IPC, Inter-Process Communication) 개요

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 독립된 주소 공간을 가진 프로세스들이 데이터를 주고받기 위해 운영체제가 제공하는 메커니즘입니다.
> 2. **가치**: 프로세스 간 격리(Isolation)라는 보안 원칙을 지키면서도, 필요에 따라 효율적으로 정보를 공유하고 협업할 수 있는 통로를 제공합니다.
> 3. **융합**: 공유 메모리(Shared Memory)와 메시지 전달(Message Passing)이라는 두 가지 핵심 패러다임을 기반으로, 분산 시스템 및 MSA 아키텍처의 근간이 됩니다.

---

## Ⅰ. 개요 (Context & Background)

**프로세스 간 통신 (IPC, Inter-Process Communication)** 은 운영체제 내에서 실행 중인 서로 다른 프로세스들이 데이터를 교환하고 실행을 동기화하기 위한 수단입니다.

### 💡 비유: 다른 나라에 사는 사람과 소통하기
각 프로세스는 가상 메모리라는 '국경'으로 둘러싸인 독립된 **나라**와 같습니다. IPC는 이 나라들 사이에 놓인 **국제전화(메시지 전달)** 나 **공동 무역 지대(공유 메모리)** 와 같습니다. 국경을 넘는 것은 금지되어 있지만, 정해진 통로를 통해서는 물자와 정보를 교환할 수 있는 것과 같습니다.

### 등장 배경
1. **프로세스 격리**: 현대 OS는 안정성을 위해 프로세스 간 메모리 접근을 원천 차단합니다.
2. **협업의 필요성**: 하지만 멀티태스킹 환경에서 프로세스들은 서로 결과를 주고받아야만 큰 기능을 완성할 수 있습니다.
3. **분업화**: 계산, 출력, 네트워크 등 역할을 나눈 프로세스들을 연결할 '접착제'가 필요해졌습니다.

**📢 섹션 요약 비유**: 철저한 보안이 유지되는 개별 연구실들 사이에 자료를 주고받기 위해 설치한 공식적인 '문서 전달함'이나 '공동 회의실'과 같은 체계입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### IPC의 두 가지 핵심 모델

| 비교 항목 | 공유 메모리 (Shared Memory) | 메시지 전달 (Message Passing) |
|:---|:---|:---|
| **기본 원리** | 프로세스 간 공유되는 메모리 영역 구축 | 커널을 통해 메시지를 송수신(Send/Receive) |
| **통신 속도** | **매우 빠름** (직접 접근) | 상대적 느림 (커널 개입, 복사 발생) |
| **동기화 책임** | 사용자 프로세스 (Semaphore/Mutex 필요) | 운영체제 커널이 직접 관리 |
| **데이터 크기** | 대용량 데이터 전송에 유리 | 작은 크기의 제어 정보 전달에 유리 |
| **커널 개입** | 영역 설정 시에만 개입 | 매 통신마다 개입 |

### IPC 아키텍처 다이어그램

**공유 메모리와 메시지 전달 방식의 구조적 차이**입니다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                      IPC (Inter-Process Communication) 모델               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【모델 A: 공유 메모리 (Shared Memory)】    【모델 B: 메시지 전달 (Message Passing)】 │
│  ───────────────────────────────────    ─────────────────────────────────── │
│  ┌───────────┐       ┌───────────┐      ┌───────────┐       ┌───────────┐  │
│  │ 프로세스 A │       │ 프로세스 B │      │ 프로세스 A │       │ 프로세스 B │  │
│  └─────┬─────┘       └─────┬─────┘      └─────┬─────┘       └─────┬─────┘  │
│        │     ┌───────┐     │                  │   ┌───────────┐   │        │
│        └────▶│ 공유  │◀────┘                  └──▶│  메시지   │◀──┘        │
│              │ 메모리│                            │    큐     │            │
│              └───────┘                            └─────┬─────┘            │
│  ───────────────────────────────────      ──────────────┼───────────────────  │
│        운영체제 (OS Kernel)                     운영체제 (OS Kernel)           │
│                                           (커널이 복사 및 큐 관리 수행)        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
1. **공유 메모리**: 프로세스 A와 B의 주소 공간 일부가 동일한 물리 메모리를 가리키게 합니다. 한쪽이 쓰면 다른 쪽이 즉시 읽을 수 있어 오버헤드가 거의 없지만, 동시에 쓸 때 발생하는 경합(Race Condition)을 막기 위한 복잡한 동기화 로직이 필요합니다.
2. **메시지 전달**: 프로세스 A가 커널에 메시지를 '복사'해주면, 커널이 이를 큐에 담았다가 프로세스 B에게 다시 '복사'해줍니다. 속도는 느리지만 커널이 중간에서 안전하게 관리해주므로 충돌 위험이 적고 구현이 단순합니다.

### 주요 IPC 기법 리스트 (Preview)
- **Pipe / Named Pipe**: 부모-자식 간 또는 독립 프로세스 간 단방향 통신
- **Message Queue**: 메시지 단위의 비동기 통신
- **Shared Memory**: 메모리 공유를 통한 고속 통신
- **Semaphore**: 프로세스 간 동기화 수단
- **Socket**: 로컬 및 네트워크를 통한 프로세스 간 통신

**📢 섹션 요약 비유**: 복잡하지만 직접 물건을 주고받는 '직거래(공유 메모리)'와, 안전하게 우체국을 거쳐 택배를 보내는 '우편 거래(메시지 전달)'의 차이와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### IPC 선택 가이드라인 (Decision Matrix)

| 요구 사항 | 추천 방식 | 이유 |
|:---|:---|:---|
| **최고 성능 (Latency)** | Shared Memory | 메모리 복사 및 컨텍스트 스위칭 최소화 |
| **네트워크 확장성** | Socket / RPC | 서로 다른 호스트 간 통신 가능 |
| **간단한 상태 통지** | Signal | 비동기적인 즉각적 알림 가능 |
| **대용량 스트림 데이터** | Pipe | 순차적인 흐름 제어(Flow Control) 내장 |
| **안정성 및 동기화** | Message Queue | 커널이 동기화를 보장하여 구현 오류 감소 |

### 과목 융합 관점
- **컴퓨터 구조 (CA)**: 공유 메모리 방식은 CPU 캐시 일관성(Cache Coherency) 문제와 직결됩니다. 여러 코어에서 공유 메모리에 접근할 때 하드웨어적인 도움(Snooping 등)이 필요합니다.
- **데이터베이스 (DB)**: DB 엔진 내부적으로 쿼리 프로세스와 저장소 프로세스가 통신할 때 고속 IPC(Shared Memory)를 사용하여 성능을 극대화합니다.

**📢 섹션 요약 비유**: 상황에 따라 퀵서비스, 등기 우편, 공동 창고를 적절히 선택해야 비즈니스의 효율성과 안전성을 모두 잡을 수 있는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오: 안드로이드의 바인더 (Binder)
안드로이드 OS는 모든 서비스(카메라, GPS 등)가 별도 프로세스로 동작합니다.
- **판단**: 안드로이드는 성능을 위해 '공유 메모리'를 기반으로 하면서도, 보안과 편리함을 위해 커널 드라이버가 메시지 전달처럼 관리해주는 **Binder IPC**를 독자적으로 구축하여 사용합니다.

### 도입 체크리스트
1. **데이터 양**: 한 번에 오가는 데이터가 수 MB 이상인가? (그렇다면 Shared Memory 고려)
2. **프로세스 관계**: 부모-자식 관계인가, 전혀 무관한 프로세스인가? (Pipe vs Named Pipe/MQ)
3. **플랫폼**: 로컬 장비 내 통신인가, 향후 서버 확장 가능성이 있는가? (LPC vs Socket)

### 안티패턴
- **IPC 남용**: 모든 기능을 프로세스로 쪼개어 IPC로 연결하면, 통신 오버헤드가 실제 연산 시간보다 길어지는 배보다 배꼽이 더 큰 상황이 발생할 수 있습니다.

**📢 섹션 요약 비유**: 너무 잦은 회의(IPC)는 정작 실제 업무(Computation)를 할 시간을 뺏어 생산성을 떨어뜨리는 것과 같습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 기대효과
- **시스템 안정성**: 격리된 구조 덕분에 한 프로세스의 오류가 전체 시스템으로 확산되는 것을 방지합니다.
- **성능 최적화**: 적절한 IPC 선택을 통해 멀티코어 및 분산 환경의 성능을 극대화할 수 있습니다.

### 결론
IPC는 단순히 데이터를 주고받는 기술을 넘어, 현대 소프트웨어 아키텍처의 **"혈관"** 과 같습니다. 모듈화된 프로세스들이 유기적으로 살아 움직이게 만드는 핵심 기반이며, 최근 클라우드 네이티브 환경의 gRPC, 메시지 브로커 등으로 그 개념이 무한히 확장되고 있습니다. 엔지니어는 각 기법의 트레이드오프를 명확히 이해하고 최적의 통신 경로를 설계해야 합니다.

**📢 섹션 요약 비유**: 개별적으로 완벽한 부품들을 하나로 연결하여 거대한 기계가 작동하게 만드는 정교한 '연결 링크' 기술과 같습니다.
