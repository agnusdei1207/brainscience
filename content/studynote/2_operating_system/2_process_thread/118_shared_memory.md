# 공유 메모리 (Shared Memory) 방식

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 두 개 이상의 프로세스가 자신의 주소 공간 일부를 공유하여, 데이터를 복사 없이 직접 읽고 쓰는 가장 빠른 IPC 방식입니다.
> 2. **가치**: 커널의 개입을 최소화하여 오버헤드를 줄이고 대용량 데이터 전송에서 압도적인 성능을 제공합니다.
> 3. **융합**: 고속 연산이 필요한 DB 엔진, 그래픽 처리, 실시간 시스템의 데이터 교환 등에 필수적으로 활용됩니다.

---

## Ⅰ. 개요 (Context & Background)

**공유 메모리 (Shared Memory)** 는 운영체제가 여러 프로세스에게 동일한 물리 메모리 영역에 접근할 수 있는 권한을 부여하는 IPC 메커니즘입니다.

### 💡 비유: 공동 작업용 화이트보드
두 사람이 각자 자기 방(독립된 주소 공간)에서 일하다가, 복도에 있는 **공동 화이트보드(공유 메모리)** 에 정보를 적어두는 것과 같습니다. 쪽지를 써서 우체국(커널)을 통해 전달할 필요 없이, 그냥 보드에 적으면 상대방이 즉시 볼 수 있습니다. 단, 두 사람이 동시에 같은 곳에 적으려 하면 글자가 뭉개질 수 있으므로 "지금 내가 적을게"라는 신호(동기화)가 필요합니다.

### 등장 배경
1. **성능 극대화**: 메시지 전달 방식은 데이터를 커널로 복사하고 다시 사용자 공간으로 복사하는 과정에서 성능 저하가 발생합니다.
2. **대용량 데이터**: 수십 MB 이상의 데이터를 실시간으로 주고받아야 할 때, 복사 방식은 CPU와 메모리 대역폭에 큰 부담을 줍니다.
3. **직접 접근**: 프로세스 간 포인터를 이용한 직접적인 데이터 구조 조작이 필요한 경우가 발생했습니다.

**📢 섹션 요약 비유**: 물건을 배달(메시지)하지 않고, 아예 한 창고(메모리)를 같이 쓰기로 하여 물류 비용과 시간을 제로(Zero)로 만든 혁신입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 및 단계

| 요소명 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **물리 메모리 영역** | 실제 데이터가 저장되는 공간 | RAM의 특정 블록을 공유 영역으로 지정 | 공동 창고 |
| **페이지 테이블 (Page Table)** | 가상-물리 주소 매핑 | 서로 다른 프로세스의 가상 주소가 동일한 물리 주소를 가리키게 함 | 각자의 내비게이션에 찍힌 동일 목적지 |
| **세그먼트 ID / 키** | 공유 영역 식별자 | 특정 공유 메모리 영역을 찾기 위한 고유 번호 | 창고 열쇠 번호 |
| **Attach / Detach** | 영역 연결 및 해제 | 프로세스의 주소 공간에 공유 영역을 붙이거나 떼는 작업 | 창고 출입 등록 |
| **동기화 객체 (Semaphore)** | 접근 제어 | 동시에 쓰기 방지 및 순서 보장 | "작업 중" 표지판 |

### 공유 메모리 아키텍처 다이어그램

**두 프로세스가 물리 메모리를 공유하는 구조**입니다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                      공유 메모리 (Shared Memory) 아키텍처                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   【프로세스 A (가상 주소 공간)】            【프로세스 B (가상 주소 공간)】   │
│   ┌──────────────────────────┐            ┌──────────────────────────┐  │
│   │ [Stack]                  │            │ [Stack]                  │  │
│   ├──────────────────────────┤            ├──────────────────────────┤  │
│   │ [공유 메모리 영역 (V1)]  │──┐      ┌──│ [공유 메모리 영역 (V2)]  │  │
│   ├──────────────────────────┤  │      │  ├──────────────────────────┤  │
│   │ [Heap / Data / Text]     │  │      │  │ [Heap / Data / Text]     │  │
│   └──────────────────────────┘  │      │  └──────────────────────────┘  │
│                                 │      │                                │
│   ──────────────────────────────┼──────┼──────────────────────────────  │
│                                 ▼      ▼                                │
│                      ┌──────────────────────────┐                       │
│                      │    실제 물리 메모리 (RAM)   │                       │
│                      │  ┌────────────────────┐  │                       │
│                      │  │   공유 세그먼트    │  │                       │
│                      │  │ (Shared Segment)    │  │                       │
│                      │  └────────────────────┘  │                       │
│                      └──────────────────────────┘                       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
1. **매핑 (Mapping)**: 프로세스 A의 가상 주소 `V1`과 프로세스 B의 가상 주소 `V2`는 서로 다를 수 있지만, 운영체제의 페이지 테이블 설정을 통해 실제로는 **동일한 물리 메모리 영역**을 가리킵니다.
2. **복사 제거 (Zero-copy)**: 프로세스 A가 `V1` 영역에 값을 쓰면 커널을 거치지 않고 물리 메모리에 즉시 기록되며, 프로세스 B는 자신의 주소 `V2`를 통해 그 값을 즉각 읽을 수 있습니다.

### 심층 동작 원리 (System V 기준)
1. **생성 (`shmget`)**: 커널에 공유 메모리 세그먼트 생성을 요청하고 ID를 받습니다.
2. **연결 (`shmat`)**: 프로세스의 주소 공간에 해당 세그먼트를 붙입니다. (포인터 획득)
3. **사용**: 일반 포인터 변수처럼 읽고 씁니다. (가장 중요한 단계: 이때 **동기화 필수**)
4. **해제 (`shmdt`)**: 사용이 끝나면 주소 공간에서 떼어냅니다.
5. **제거 (`shmctl`)**: 더 이상 필요 없으면 커널에서 세그먼트를 완전히 삭제합니다.

**📢 섹션 요약 비유**: 서로 다른 언어를 쓰는 사람들이 같은 그림(물리 메모리)을 보고 소통하되, 서로 붓을 뺏기지 않게 순서(동기화)를 지키는 화가들과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### IPC 성능 및 특징 비교

| 비교 항목 | 공유 메모리 (Shared Memory) | 파이프 / 메시지 큐 (Message) |
|:---|:---|:---|
| **통신 속도** | **최상 (Memory Speed)** | 보통 (Context Switch & Copy) |
| **데이터 복사 횟수** | 0회 (Zero-copy) | 2회 (User→Kernel→User) |
| **커널 관여도** | 낮음 (최초 설정 시에만) | 높음 (모든 전송 시 개입) |
| **동기화** | **사용자 책임 (매우 중요)** | 커널이 보장 (Blocking 등) |
| **적정 데이터** | 대용량 데이터, 빈번한 교환 | 작은 제어 데이터, 스트림 |

### 과목 융합 관점
- **컴퓨터 구조 (CA)**: 여러 코어에서 공유 메모리에 쓰기를 할 때, 각 코어의 L1/L2 캐시 값이 달라지는 **캐시 일관성 (Cache Coherency)** 문제가 발생합니다. 하드웨어의 MSI/MESI 프로토콜과 소프트웨어의 동기화 기법이 조화를 이루어야 합니다.
- **데이터베이스 (DB)**: PostgreSQL은 'Shared Buffers'라는 거대한 공유 메모리 영역을 사용하여 디스크에서 읽어온 데이터를 모든 쿼리 프로세스가 공유하게 함으로써 디스크 I/O를 획기적으로 줄입니다.

**📢 섹션 요약 비유**: 고속도로 통행료를 내지 않는 전용 차선(공유 메모리)을 뚫었지만, 사고가 나지 않게 차선 제어(동기화)를 운전자가 직접 해야 하는 상황과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오: 고성능 로그 수집기
수만 개의 프로세스가 발생하는 로그를 실시간으로 수집해야 하는 상황입니다.
- **판단**: 메시지 큐는 커널 복사 비용 때문에 병목이 생길 수 있습니다. 거대한 공유 메모리 링 버퍼(Ring Buffer)를 만들고, 각 프로세스가 여기에 쓰고 수집기(Collector)가 읽어가는 방식을 선택합니다.
- **의사결정**: 이때 쓰기 경합을 줄이기 위해 **Lock-free 알고리즘**이나 **CAS (Compare-And-Swap)** 연산을 활용하여 성능을 극대화합니다.

### 도입 체크리스트
1. **데이터 정합성**: 동기화 로직(Mutex/Semaphore)이 완벽하게 설계되었는가?
2. **메모리 누수**: 프로세스가 비정상 종료되어도 공유 메모리가 커널에 남지 않도록 정리 로직이 있는가?
3. **보안**: 공유 메모리에 민감한 정보가 있을 때, 권한 설정(`shmctl`)이 적절한가?

### 안티패턴
- **동기화 없는 접근**: 공유 메모리 최대의 재앙은 동기화 누락입니다. 데이터 오염은 즉각적으로 나타나지 않고 간헐적으로 발생하여 디버깅이 불가능에 가깝습니다.

**📢 섹션 요약 비유**: 브레이크 없는 스포츠카(공유 메모리)를 모는 것과 같아서, 속도는 최고지만 운전자의 실력(동기화 설계)이 없으면 대형 사고로 이어집니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 기대효과
- **처리량(Throughput) 극대화**: 복사 오버헤드 제거로 시스템 전체의 데이터 처리 능력이 수 배 이상 향상됩니다.
- **지연 시간(Latency) 최소화**: 마이크로초(μs) 단위의 응답이 필요한 실시간 시스템 구현이 가능해집니다.

### 결론
공유 메모리는 IPC 기법 중 **"가장 강력하지만 가장 위험한 양날의 검"** 입니다. 운영체제의 격리 원칙을 성능을 위해 일부 양보하는 기법인 만큼, 개발자의 정교한 동기화 능력이 요구됩니다. 클라우드와 고집적 멀티코어 시대로 갈수록 데이터 이동 비용을 줄이기 위한 공유 메모리 활용 기술(예: DPDK, NVMe-over-Fabrics)은 더욱 고도화될 것입니다.

**📢 섹션 요약 비유**: 복잡한 절차를 생략한 'VIP 전용 통로'를 만들어 최고의 효율을 내되, 그 통로의 안전은 이용자가 직접 책임지는 프로들의 방식입니다.
