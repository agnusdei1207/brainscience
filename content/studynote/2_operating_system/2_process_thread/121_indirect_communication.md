# 간접 통신 (Indirect Communication)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 메시지를 프로세스 간에 직접 주고받지 않고, '사서함(Mailbox)'이나 '포트(Port)'라는 매개체를 통해 교환하는 IPC 방식입니다.
> 2. **가치**: 송수신자가 서로의 존재를 몰라도 통신이 가능하여 결합도(Coupling)를 낮추고, 다대다(N:M) 통신을 유연하게 지원합니다.
> 3. **융합**: 현대 분산 시스템의 메시지 브로커(Kafka, RabbitMQ)나 이메일 시스템, 클라이언트-서버 구조의 근간이 됩니다.

---

## Ⅰ. 개요 (Context & Background)

**간접 통신 (Indirect Communication)** 은 메시지 전달 시스템에서 공유되는 **사서함 (Mailbox)** 또는 **포트 (Port)** 를 사용하여 프로세스들이 소통하는 기법입니다.

### 💡 비유: 아파트 무인 택배함
철수가 영희에게 선물을 줄 때, 직접 영희의 집을 찾아가는 대신 **아파트 무인 택배함(사서함)** 에 넣어두는 것과 같습니다. 영희는 나중에 시간이 될 때 택배함 번호만 알고 가서 선물을 찾으면 됩니다. 철수와 영희는 서로 만날 필요도 없고, 심지어 서로의 호수를 몰라도 택배함 번호만 공유하면 소통이 가능합니다.

### 등장 배경
1. **유연성 확보**: 프로세스가 종료되거나 ID가 바뀌어도 사서함 이름만 유지되면 통신이 중단되지 않습니다.
2. **다대다 통신 필요성**: 하나의 사서함에 여러 송신자가 넣고, 여러 수신자가 가져가는 유연한 구조가 필요해졌습니다.
3. **비동기성 강화**: 송신자가 메시지를 넣는 시점과 수신자가 가져가는 시점을 완전히 분리(Decoupling)할 수 있습니다.

**📢 섹션 요약 비유**: 서로의 얼굴을 모르는 익명의 기부자와 수혜자가 '사랑의 빵집(사서함)'을 통해 빵을 나누는 것과 같은 따뜻하고 유연한 연결 방식입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 주요 특징

| 항목 | 상세 내용 | 비유 |
|:---|:---|:---|
| **매개체** | 사서함(Mailbox) 또는 포트(Port) 객체 사용 | 공용 우편함 |
| **링크 설정** | 두 프로세스가 공통의 사서함을 공유할 때만 형성됨 | 같은 단톡방 입장 |
| **연결 관계** | 하나의 링크에 여러 프로세스가 연결될 수 있음 (N:M) | 게시판 (누구나 쓰고 읽음) |
| **소유권** | 사서함은 프로세스가 소유하거나 운영체제가 소유함 | 개인 금고 vs 은행 대여 금고 |
| **동작 방식** | `send(mailbox, message)`, `receive(mailbox, message)` | 사서함에 넣기 / 빼기 |

### 간접 통신 아키텍처 다이어그램

**사서함을 경유하는 비동기적 메시지 교환 구조**입니다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                      간접 통신 (Indirect Communication) 구조              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   【송신 프로세스들】             【매개체: 사서함】             【수신 프로세스들】  │
│   ┌─────────────┐             ┌─────────────┐             ┌─────────────┐  │
│   │ 프로세스 A  │───┐         │             │         ┌──▶│ 프로세스 X  │  │
│   └─────────────┘   │  send   │   사서함    │ receive │   └─────────────┘  │
│   ┌─────────────┐   ├───▶(Mailbox/Port)├───▶┌─────────────┐  │
│   │ 프로세스 B  │───┘         │  [Msg 3]    │         └──▶│ 프로세스 Y  │  │
│   └─────────────┘             │  [Msg 2]    │             └─────────────┘  │
│                               │  [Msg 1]    │                              │
│                               └─────────────┘                              │
│                                                                     │
│   ──────────────────────────────────────────────────────────────────  │
│   【운영체제 커널 (Kernel) 의 역할】                                      │
│   1. 새로운 사서함 생성 (Create)                                        │
│   2. 사서함을 통한 메시지 송수신 관리                                    │
│   3. 사서함 삭제 (Delete)                                               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
1. **사서함 중심**: 통신의 중심은 프로세스가 아니라 '사서함'입니다. 프로세스 A, B는 사서함 이름만 알면 메시지를 던질 수 있습니다.
2. **다중 수신 이슈**: 만약 사서함에 메시지가 하나 있는데 프로세스 X, Y가 동시에 `receive`를 호출한다면? 운영체제는 ① 한 명에게만 전달, ② 라운드 로빈, ③ 선착순 등의 알고리즘으로 이를 해결해야 합니다.
3. **영속성**: 송신 프로세스가 메시지를 넣고 바로 종료되어도, 메시지는 사서함에 남아 수신자를 기다릴 수 있습니다.

**📢 섹션 요약 비유**: 주소지로 직접 배달하는 것이 아니라 '포스트잇'을 붙여두는 벽을 공유하여, 누가 언제 보든 정보를 전달받게 하는 게시판 방식입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 직접 통신 vs 간접 통신 (정밀 비교)

| 비교 항목 | 직접 통신 (Direct) | 간접 통신 (Indirect) |
|:---|:---|:---|
| **결합도** | **강한 결합 (Tightly Coupled)** | **느슨한 결합 (Loosely Coupled)** |
| **프로세스 식별** | 상대방의 ID가 필수 | 사서함 ID만 알면 됨 |
| **유연성** | 낮음 (ID 변경 시 수정 불가피) | 높음 (ID 몰라도 됨, 확장 용이) |
| **링크 특성** | 1:1 전용 통로 | N:M 공용 통로 |
| **주요 사례** | 부모-자식 제어, 시그널 | 메시지 큐, 서버 포트, 소켓 |

### 과목 융합 관점
- **네트워크 (Network)**: 포트(Port) 번호 시스템이 전형적인 간접 통신입니다. IP 주소(컴퓨터) 내에서 특정 서비스(포트)를 찾아가는 방식입니다.
- **클라우드 (Cloud)**: MSA 환경에서 서비스 간 통신을 담당하는 **메시지 브로커** (RabbitMQ, SQS 등)가 간접 통신의 철학을 엔터프라이즈 급으로 확장한 것입니다.

**📢 섹션 요약 비유**: 전화(직접)는 양방향이 실시간으로 연결되어야 하지만, 이메일(간접)은 상대방이 부재중이어도 소통이 가능한 혁신적인 유연성을 제공합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오: 다중 작업 처리 (Worker Pool)
사용자의 요청(메시지)이 폭주할 때 여러 명의 일꾼(Worker) 프로세스가 이를 나눠서 처리해야 합니다.
- **적용**: 요청들을 하나의 **공용 사서함(Queue)** 에 쌓아두고, 워커 프로세스들이 각자 여유가 될 때마다 가져가게 합니다.
- **판단**: 직접 통신으로는 구현이 불가능에 가깝습니다. 간접 통신을 통해 로드 밸런싱(Load Balancing)과 확장성(Scalability)을 동시에 확보할 수 있습니다.

### 도입 체크리스트
1. **확장성**: 향후 수신 프로세스의 숫자를 늘릴 계획이 있는가?
2. **비동기성**: 송신자가 수신자의 즉각적인 처리를 기다리지 않아도 되는가?
3. **관리 주체**: 사서함의 수명 주기를 누가 관리할 것인가? (프로세스 종료 시 사서함 자동 삭제 여부)

**📢 섹션 요약 비유**: 고객이 상담사 개인 번호로 전화하는 것이 아니라, 대표 번호(사서함)로 전화하여 비어 있는 상담사가 전화를 받는 효율적인 콜센터 시스템과 같습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 기대효과
- **모듈화 증대**: 프로세스들이 서로의 세부 정보를 몰라도 협력할 수 있어 소프트웨어 구조가 깔끔해집니다.
- **결함 격리**: 수신 프로세스가 잠시 죽어도 사서함에 메시지가 쌓여 있으므로 데이터 유실을 방지할 수 있습니다.

### 결론
간접 통신은 현대 소프트웨어 공학의 핵심 가치인 **"느슨한 결합 (Loose Coupling)"** 을 IPC 레벨에서 실현한 기술입니다. 직접 통신의 경직성을 해결하고 복잡한 다대다 통신을 우아하게 처리함으로써, 오늘날의 거대한 분산 시스템과 클라우드 네이티브 아키텍처를 가능케 한 일등 공신입니다.

**📢 섹션 요약 비유**: 서로 직접 연결된 전선 대신, 공용 허브를 통해 전원을 공급받는 것처럼 시스템의 구성 요소를 언제든 갈아 끼울 수 있는 유연한 세상을 만든 기술입니다.
