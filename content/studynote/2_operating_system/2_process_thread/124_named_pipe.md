# 지명 파이프 (Named Pipe / FIFO)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 파이프에 고유한 파일 이름을 부여하여, 부모-자식 관계가 아닌 전혀 독립적인 프로세스 간에도 통신할 수 있게 한 IPC 기법입니다.
> 2. **가치**: 파일 시스템의 경로를 식별자로 사용함으로써 범용성을 확보하고, 데이터가 들어온 순서대로 처리되는 FIFO(First-In, First-Out) 특성을 보장합니다.
> 3. **융합**: 로컬 서버-클라이언트 통신, 로그 수집 데몬, 복잡한 워크플로우를 가진 프로세스 체인 구축에 활용됩니다.

---

## Ⅰ. 개요 (Context & Background)

**지명 파이프 (Named Pipe)** 또는 **FIFO**는 익명 파이프의 제약(부모-자식 관계 필수)을 극복하기 위해 등장한 통신 메커니즘입니다.

### 💡 비유: 공용 게시판 우체통
익명 파이프가 우리 집 안에서만 쓰는 '가족 전용 인터폰'이라면, 지명 파이프는 마을 광장에 설치된 **'공용 우체통'** 과 같습니다. 누구든지 우체통의 이름(파일 경로)만 알면 편지를 넣을 수 있고, 열쇠를 가진 사람(수신자)은 그 편지를 가져갈 수 있습니다. 보내는 사람과 받는 사람은 서로 가족일 필요도, 아는 사이일 필요도 없습니다.

### 등장 배경
1. **익명 파이프의 한계**: `pipe()`로 생성된 파이프는 파일 디스크립터를 상속받을 수 있는 프로세스들끼리만 쓸 수 있었습니다.
2. **독립 프로세스 간 통신**: 서버 프로세스와 클라이언트 프로세스처럼 전혀 별개로 실행된 프로그램 간에 가볍게 통신할 수 있는 수단이 필요했습니다.
3. **파일 인터페이스 활용**: 별도의 API 대신 익숙한 파일 읽기/쓰기(`open`, `read`, `write`) 인터페이스를 그대로 쓰고 싶어 했습니다.

**📢 섹션 요약 비유**: 혈연관계(부모-자식)가 아니더라도, 공통의 주소(파일명)만 공유하면 누구와도 거래할 수 있는 '시장'과 같은 통신 기법입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 주요 특징

| 항목 | 상세 내용 | 비유 |
|:---|:---|:---|
| **식별자** | 파일 시스템 상의 경로 (예: `/tmp/my_fifo`) | 우체통의 주소 |
| **생성 방법** | `mkfifo()` 시스템 콜 또는 `mkfifo` 명령어 | 우체통 설치 |
| **방향성** | 기본적으로 단방향 (시스템에 따라 양방향 가능) | 일방통행로 |
| **영속성** | 프로세스가 종료되어도 파일은 남음 (데이터는 휘발성) | 빈 우체통은 계속 그 자리에 있음 |
| **동기화** | 읽기/쓰기 시 상대방이 없으면 Blocking 됨 | 상대방이 문을 열 때까지 기다림 |

### 지명 파이프 (FIFO) 동작 구조

**독립된 두 프로세스가 파일 경로를 통해 만나는 과정**입니다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                       지명 파이프 (Named Pipe) 구조                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【송신 프로세스 (Writer)】              【수신 프로세스 (Reader)】         │
│  ┌──────────────────┐                  ┌──────────────────┐         │
│  │ fd = open("pipe")│                  │ fd = open("pipe")│         │
│  │ write(fd, data)  │───┐          ┌──▶│ read(fd, buffer) │         │
│  └──────────────────┘   │ (Write)  │   └──────────────────┘         │
│                         ▼          │                                │
│  ───────────────────────┼──────────┼──────────────────────────────  │
│  【파일 시스템 (VFS)】     │          │                                │
│       ┌─────────────────┼──────────┼─────────────────┐              │
│       │ /tmp/my_fifo    │          │                 │              │
│       │ (Special File)  │          │                 │              │
│       └────────┬────────┘          │                 │              │
│                ▼                   │                 │              │
│  【커널 (Kernel)】      ┌───────────┴───────────┐     │              │
│       │                │  파이프 커널 버퍼      │     │              │
│       │                │  (FIFO Queue)         │     │              │
│       │                └───────────────────────┘     │              │
│       └──────────────────────────────────────────────┘              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
1. **특수 파일**: `/tmp/my_fifo`는 디스크에 실제 데이터가 저장되는 일반 파일이 아니라, 커널 버퍼를 가리키는 **특수 파일(Inode)** 입니다.
2. **이름 기반 접근**: 프로세스 A와 B는 서로의 존재를 모르더라도, 사전에 약속된 파일 경로인 `my_fifo`를 `open` 함으로써 연결됩니다.
3. **데이터 흐름**: 데이터는 디스크를 거치지 않고 메모리(커널 버퍼) 상에서 직접 이동하므로 파일 I/O보다 훨씬 빠릅니다.
4. **동기화 포인트**: 쓰기 프로세스가 `open` 하려는데 읽기 프로세스가 아직 `open` 하지 않았다면, 쓰기 프로세스는 그 지점에서 멈춰 기다립니다. (반대도 동일)

**📢 섹션 요약 비유**: 두 사람이 같은 사물함(FIFO 파일)을 열려고 할 때, 한 사람이 먼저 와서 기다리다가 다른 사람이 오면 그제야 물건을 주고받는 '직거래 장터'와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 익명 파이프 vs 지명 파이프 (FIFO)

| 비교 항목 | 익명 파이프 (Anonymous) | 지명 파이프 (Named) |
|:---|:---|:---|
| **파일 이름** | 없음 (fd만 존재) | **있음 (파일 시스템 경로)** |
| **프로세스 관계** | 부모-자식 등 혈연관계 | **아무 관계 없는 프로세스** |
| **생성 시점** | 실행 중 `pipe()` 호출 | 실행 전 또는 중 `mkfifo()` 호출 |
| **종료 후 흔적** | 즉시 소멸 | 파일 이름은 남음 (삭제 필요) |
| **통신 방향** | 단방향 | 단방향 (양방향은 소켓 권장) |

### 과목 융합 관점
- **보안 (Security)**: 지명 파이프는 파일 시스템 상에 존재하므로, 파일 권한(Owner, Group, Permission)을 통해 통신에 접근할 수 있는 프로세스를 제어할 수 있습니다.
- **데이터베이스 (DB)**: 오라클이나 MySQL 같은 DB에서 대량의 덤프 데이터를 처리할 때, 중간 파일을 만들지 않고 지명 파이프를 통해 즉시 다른 프로그램(압축기 등)으로 전달하여 디스크 공간을 절약합니다.

**📢 섹션 요약 비유**: 익명 파이프가 '가족 간의 비밀 대화'라면, 지명 파이프는 '공식적인 창구를 통한 민원 접수'와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오: 마이크로서비스 간의 로그 스트리밍
웹 서버 프로세스가 남기는 로그를 별도의 분석 프로세스가 실시간으로 읽어 가야 합니다.
- **적용**: `/var/run/log.pipe`라는 FIFO를 생성하고 웹 서버는 여기에 로그를 씁니다. 분석기는 이 파일을 읽어 처리합니다.
- **판단**: 네트워크 소켓을 쓰기에는 같은 장비 내라 오버헤드가 아깝고, 일반 파일을 쓰기에는 디스크 I/O와 용량 관리가 부담스러울 때 최적의 선택입니다.

### 도입 체크리스트
1. **휘발성**: 시스템 재부팅 시 데이터가 사라져도 괜찮은가? (데이터는 메모리에만 존재)
2. **동기화**: 한쪽이 읽지 않으면 다른 쪽이 멈추는 Blocking 현상이 서비스에 치명적이지 않은가?
3. **정리**: 사용이 끝난 FIFO 파일을 `unlink`로 지워주는 로직이 포함되었는가?

### 안티패턴
- **N:1 통신 시 주의**: 여러 프로세스가 하나의 FIFO에 동시에 쓰면 메시지가 섞일 수 있습니다. (Atomicity 보장 크기 이하로만 써야 함)
- **일반 파일로 착각**: FIFO를 일반 텍스트 파일처럼 에디터로 열면, 누군가 데이터를 넣어줄 때까지 에디터가 멈춰버리는 현상이 발생합니다.

**📢 섹션 요약 비유**: 편리한 공유 함체이지만, 사용 후 뒷정리(파일 삭제)를 하지 않으면 시스템에 지저분한 흔적이 남는 것과 같습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 기대효과
- **모듈 결합도 완화**: 프로그램들이 서로의 실행 여부나 상세 정보를 몰라도 경로명 하나로 협업이 가능해집니다.
- **자원 절약**: 디스크 쓰기 없이 메모리 내에서 스트리밍 처리가 가능하여 시스템 부하를 줄입니다.

### 결론
지명 파이프는 **"파일이라는 보편적 추상화를 IPC로 승화시킨 도구"** 입니다. 익명 파이프의 단순함에 파일 시스템의 범용성을 더함으로써, 로컬 시스템 내의 다양한 프로세스들을 유기적으로 엮어주는 훌륭한 접착제 역할을 합니다. 현대적인 네트워크 소켓에 비해 화려함은 덜하지만, 그 견고함과 효율성은 여전히 대체 불가능한 가치를 지닙니다.

**📢 섹션 요약 비유**: 화려한 최신 메신저는 아니지만, 정해진 장소에서 소리 없이 강력하게 소통을 돕는 '비밀 사서함'과 같은 존재입니다.
