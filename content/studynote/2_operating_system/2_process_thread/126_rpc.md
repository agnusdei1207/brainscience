# RPC (Remote Procedure Call)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 네트워크를 통해 원격 호스트에 있는 함수(Procedure)를 로컬 함수처럼 호출할 수 있게 해주는 통신 추상화 기법입니다.
> 2. **가치**: 복잡한 네트워크 소켓 통신(소켓 생성, 주소 지정, 데이터 직렬화 등)을 숨겨 개발자가 비즈니스 로직에만 집중하게 합니다.
> 3. **융합**: 분산 시스템, 마이크로서비스 아키텍처(MSA)의 gRPC, 클라우드 네이티브 서비스 간 통신의 핵심 기반입니다.

---

## Ⅰ. 개요 (Context & Background)

**RPC (Remote Procedure Call)** 는 별도의 원격 제어나 네트워크 프로그래밍 없이 다른 주소 공간에 있는 서브루틴을 실행할 수 있게 하는 기술입니다.

### 💡 비유: 해외 지사에 있는 전문가에게 전화하기
로컬 함수 호출이 내 사무실 옆 책상에 있는 동료에게 일을 시키는 것이라면, RPC는 **해외 지사에 있는 전문가**에게 국제전화를 걸어 일을 부탁하는 것과 같습니다. 나는 수화기를 들고 요청만 하면, 중간에 통신사(RPC 런타임)가 복잡한 위성 통신과 번역을 거쳐 전문가에게 전달하고 결과까지 다시 나에게 알려줍니다. 나는 그 과정이 얼마나 복잡한지 신경 쓸 필요 없이 마치 옆자리 동료와 대화하듯 결과를 얻습니다.

### 등장 배경
1. **분산 컴퓨팅의 복잡성**: 여러 대의 컴퓨터가 협업해야 하는 상황에서 매번 소켓을 열고 닫고 데이터를 쪼개는 작업이 너무 번거로웠습니다.
2. **함수 호출 패러다임 유지**: 프로그래머들에게 가장 익숙한 '함수 호출' 방식을 네트워크 통신에도 그대로 적용하고 싶어 했습니다.
3. **투명성 (Transparency)**: 로컬 호출과 원격 호출의 문법적 차이를 없애어 코드의 가독성과 유지보수성을 높이려는 시도입니다.

**📢 섹션 요약 비유**: 복잡한 물류 배송 과정(소켓/패킷)을 '해외 직구 대행 서비스(RPC)'를 통해 클릭 한 번(함수 호출)으로 단순화한 혁신입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 상세 동작 | 비유 |
|:---|:---|:---|:---|
| **클라이언트 스터브 (Client Stub)** | 호출 대행자 | 매개변수를 메시지로 변환(Marshalling)하여 네트워크로 송신 | 주문 대행 업체 |
| **RPC 런타임 (Runtime)** | 통신 관리자 | 메시지 전송, 재전송, 타임아웃 처리 등을 담당 | 국제 특송 물류망 |
| **서버 스터브 (Server Stub)** | 요청 해석자 | 수신 메시지를 역변환(Unmarshalling)하여 실제 함수 호출 | 현지 가이드/번역가 |
| **실제 함수 (Remote Proc)** | 실제 로직 수행 | 서버에서 정의된 비즈니스 로직 실행 | 현지 전문가 |
| **네임 서비스 (Registry)** | 위치 정보 제공 | 어느 서버에 어떤 함수가 있는지 알려주는 주소록 | 114 안내 서비스 |

### RPC 동작 메커니즘 (Step-by-Step)

**클라이언트가 원격 함수를 호출하고 결과를 받기까지의 전 과정**입니다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                       RPC (Remote Procedure Call) 구조                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│    【클라이언트 호스트 (Client)】            【서버 호스트 (Server)】         │
│    ┌──────────────────┐                  ┌──────────────────┐       │
│    │  1. 함수 호출     │                  │  4. 함수 실행     │       │
│    │  result = add(a,b)│                  │  int add(a,b) {..}│       │
│    └────────┬─────────┘                  └────────▲─────────┘       │
│             │                                     │                 │
│    ┌────────▼─────────┐                  ┌────────┴─────────┐       │
│    │  2. 클라이언트 스터브│                  │  3. 서버 스터브   │       │
│    │  (Marshalling)   │                  │ (Unmarshalling)  │       │
│    └────────┬─────────┘                  └────────▲─────────┘       │
│             │                                     │                 │
│    ─────────┼─────────────────────────────────────┼─────────        │
│             └───────────▶ 런타임/네트워크 ──────────┘                 │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
1. **마샬링 (Marshalling)**: 클라이언트 스터브가 함수 이름과 인자(a, b)를 네트워크 전송이 가능한 형태(XDR 등)로 포장합니다.
2. **네트워크 전송**: RPC 런타임이 포장된 데이터를 서버로 보냅니다. (이때 클라이언트는 보통 결과가 올 때까지 대기합니다.)
3. **언마샬링 (Unmarshalling)**: 서버 스터브가 패킷을 뜯어 원래의 데이터로 복원하고 서버 내부의 진짜 함수를 호출합니다.
4. **리턴**: 계산된 결과값이 다시 역순(서버 스터브 마샬링 → 전송 → 클라이언트 스터브 언마샬링)을 거쳐 클라이언트에게 전달됩니다.

**📢 섹션 요약 비유**: 내가 주문서를 쓰면(Call), 비서(Stub)가 외국어로 번역해 부치고, 현지 비서가 받아 다시 모국어로 번역해 사장님께 보고하는 유기적인 협업 시스템입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### RPC vs 전통적 소켓(Socket) 통신

| 비교 항목 | 전통적 소켓 통신 | RPC (Remote Procedure Call) |
|:---|:---|:---|
| **추상화 수준** | 낮음 (Low-level, 직접 제어) | **높음 (High-level, 함수 호출형)** |
| **데이터 처리** | 개발자가 직접 직렬화(Serialize) | 스터브가 자동 처리(Marshalling) |
| **결합도** | 낮음 (메시지 포맷만 맞으면 됨) | 상대적 높음 (인터페이스 정의 필요) |
| **생산성** | 낮음 (통신 코드 비중이 큼) | **매우 높음 (비즈니스 로직에 집중)** |
| **유연성** | 높음 (다양한 스트림 처리 가능) | 제한적 (함수 호출 형태에 고착) |

### 과목 융합 관점
- **네트워크 (Network)**: RPC는 전송 계층(TCP/UDP) 위에서 동작하며, 최근에는 HTTP/2 위에서 동작하는 gRPC가 웹과 클라우드의 표준이 되었습니다.
- **데이터베이스 (DB)**: 데이터베이스의 저장 프로시저(Stored Procedure)를 외부 앱에서 호출할 때 내부적으로 RPC 메커니즘이 사용됩니다.

**📢 섹션 요약 비유**: 수동 변속기 자동차(소켓)로 엔진 회전수를 직접 맞추며 운전하다가, 자동 변속기(RPC)로 바뀌어 운전이 훨씬 편해진 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오: 마이크로서비스(MSA) 간 통신
결제 서비스가 사용자 정보 서비스에 잔액 확인을 요청해야 합니다.
- **적용**: `UserService.getBalance(userId)`와 같은 인터페이스를 RPC로 정의하여 호출합니다.
- **판단**: 각 서비스가 다른 언어(Java, Go, Python 등)로 짜여 있더라도, RPC(특히 IDL을 지원하는 gRPC 등)를 통하면 언어 장벽을 넘어 손쉽게 협업할 수 있습니다.

### 도입 체크리스트
1. **언어 중립성**: 여러 언어가 섞인 환경인가? (IDL 기반 RPC 필요)
2. **성능 요구**: 텍스트 기반(JSON/REST)의 오버헤드가 부담스러운가? (바이너리 기반 RPC인 gRPC/Thrift 권장)
3. **장애 전파**: 원격 서버가 죽었을 때 로컬 프로세스가 무한 대기하지 않도록 '서킷 브레이커'나 '타임아웃' 전략이 있는가?

### 안티패턴
- **로컬 함수인 줄 알고 남용**: 원격 호출은 로컬보다 수백 수천 배 느립니다. 이를 인지하지 못하고 반복문 안에서 RPC를 날리면 전체 시스템 성능이 처참하게 무너집니다. (Network Latency 간과)

**📢 섹션 요약 비유**: 해외 지사와 협력할 때는 시차와 통신 지연(Latency)이 있음을 항상 인지하고, 한 번에 모아서 일을 시키는(Batching) 요령이 필요합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 기대효과
- **개발 생산성 향상**: 네트워크 프로그래밍의 복잡성을 제거하여 시장 출시 속도(Time-to-Market)를 높입니다.
- **시스템 확장성**: 함수 호출 하나로 다른 서버의 자원을 내 것처럼 쓸 수 있어 분산 확장이 매우 쉬워집니다.

### 결론
RPC는 **"분산 컴퓨팅의 가장 인간적인 인터페이스"** 입니다. 기술의 발전과 함께 CORBA, SOAP, REST를 거쳐 오늘날의 gRPC에 이르기까지 형태는 변해왔지만, "원격지의 기능을 내 것처럼 편하게 쓰겠다"는 RPC의 핵심 철학은 현대 소프트웨어 아키텍처의 절대적인 표준으로 자리 잡았습니다.

**📢 섹션 요약 비유**: '어디서나 똑같이 일할 수 있는 원격 데스크톱'처럼, 서비스가 어디에 있든 내 컴퓨터 안에 있는 것처럼 부려먹는 마법 같은 기술입니다.
