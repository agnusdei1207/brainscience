# 마샬링 (Marshalling) / 언마샬링 (Unmarshalling)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 메모리 내의 객체나 데이터 구조를 네트워크 전송이나 저장이 가능한 바이트 스트림 형태로 변환하고(Marshalling), 이를 다시 원래 상태로 복원하는(Unmarshalling) 프로세스입니다.
> 2. **가치**: 서로 다른 메모리 레이아웃이나 언어 환경을 가진 시스템 간에 복잡한 데이터 구조를 손실 없이 교환할 수 있게 합니다.
> 3. **융합**: RPC, REST API, 분산 데이터베이스, 영속성 프레임워크(ORM) 등 현대 데이터 통신의 필수 관문입니다.

---

## Ⅰ. 개요 (Context & Background)

**마샬링 (Marshalling)** 은 데이터의 의미를 유지하면서 전송 가능한 포맷으로 '포장'하는 행위이며, **언마샬링 (Unmarshalling)** 은 그 포장을 풀어 원래의 데이터 구조로 '복원'하는 행위입니다.

### 💡 비유: 가구 분해와 재조립
이사를 갈 때 커다란 침대(객체/데이터 구조)를 통째로 트럭(네트워크)에 실을 수 없습니다. 침대를 나사 하나하나 분해하여 상자에 담는 과정이 **마샬링**입니다. 목적지에 도착해서 상자를 열고 다시 나사를 조여 침대 모양으로 만드는 과정이 **언마샬링**입니다. 이때 조립 설명서(스키마/프로토콜)가 정확해야만 원래와 똑같은 침대가 완성됩니다.

### 등장 배경
1. **메모리 격리**: 서로 다른 주소 공간을 가진 프로세스나 컴퓨터 간에는 포인터(메모리 주소)를 직접 전달할 수 없습니다.
2. **이기종 호스트**: CPU 아키텍처나 운영체제마다 데이터를 저장하는 방식(Endianness 등)이 달라 표준화된 전송 포맷이 필요했습니다.
3. **복잡한 데이터 구조**: 단순한 정수뿐만 아니라 리스트, 트리, 객체와 같은 복잡한 관계를 유지하며 전송해야 했습니다.

**📢 섹션 요약 비유**: 복잡한 실타래(메모리 구조)를 한 줄의 실(바이트 스트림)로 풀어내어 바늘구멍(네트워크)을 통과시키는 기술입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 마샬링 vs 직렬화 (Serialization)

흔히 혼용되지만, 기술적으로는 약간의 뉘앙스 차이가 있습니다.

| 용어 | 주요 초점 | 상세 |
|:---|:---|:---|
| **직렬화 (Serialization)** | 데이터의 **상태 저장** | 객체를 바이트로 변환하여 파일 저장이나 전송을 준비함 |
| **마샬링 (Marshalling)** | 데이터의 **원격 전송** | 직렬화에 더해, 원격 호출에 필요한 코드 정보(Stub)나 메타데이터까지 포함함 |

### 마샬링 / 언마샬링 프로세스

**클라이언트와 서버 간의 데이터 변환 흐름**입니다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                    마샬링 (Marshalling) & 언마샬링 흐름                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【송신 측 (Sender)】                         【수신 측 (Receiver)】         │
│  ┌──────────────────┐                       ┌──────────────────┐    │
│  │ 복합 객체 (Object) │                       │ 복합 객체 (Object) │    │
│  │ { id: 1, name: A } │                       │ { id: 1, name: A } │    │
│  └────────┬─────────┘                       └────────▲─────────┘    │
│           │ [Marshalling]                            │ [Unmarshalling] │
│           ▼                                          │                 │
│  ┌────────────────────────┐                 ┌────────┴──────────────┐  │
│  │   바이트 스트림 (Byte)  │───(Network)───▶│   바이트 스트림 (Byte)   │  │
│  │ [01 00 00 00 41 00...] │                 │ [01 00 00 00 41 00...] │  │
│  └────────────────────────┘                 └────────────────────────┘  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
1. **객체 분석**: 송신 측 마샬러(Marshaller)가 메모리 상의 객체를 훑으며 필드 값들을 추출합니다.
2. **평탄화 (Flattening)**: 계층적인 객체 구조를 1차원의 바이트 배열로 길게 늘어뜨립니다. 이때 포인터 참조는 실제 값으로 치환되거나 고유 ID로 대체됩니다.
3. **인코딩**: 약속된 표준(XDR, JSON, Protobuf 등)에 맞춰 바이트 순서를 맞추고 타입 정보를 추가합니다.
4. **역순 복원**: 수신 측은 들어온 바이트를 읽어 메모리를 할당하고 필드 값을 채워 넣어 원래와 동일한 객체를 생성합니다.

### 주요 고려 사항
- **Deep Copy vs Shallow Copy**: 참조된 다른 객체들까지 모두 마샬링할 것인가의 결정.
- **순환 참조 (Circular Reference)**: A가 B를 참조하고 B가 A를 참조할 때 무한 루프에 빠지지 않도록 처리하는 기술.
- **스키마 버전 관리**: 송수신 측의 데이터 구조 버전이 다를 때의 호환성 보장.

**📢 섹션 요약 비유**: 입체적인 레고 성(객체)을 부수어 상자에 담은 뒤(마샬링), 설명서(포맷)를 보고 다시 똑같은 성을 쌓는(언마샬링) 정교한 과정입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 데이터 포맷별 특성

| 포맷 | 마샬링 속도 | 데이터 크기 | 가독성 | 특징 |
|:---|:---|:---|:---|:---|
| **JSON** | 보통 | 큼 | **최상** | 텍스트 기반, 유연함, 웹 표준 |
| **XML** | 느림 | 매우 큼 | 높음 | 엄격한 구조, 오버헤드 큼 |
| **Protobuf** | **매우 빠름** | **매우 작음** | 낮음 | 바이너리 기반, 스키마 필수 |
| **Java Serial** | 보통 | 보통 | 낮음 | 자바 전용, 보안 취약점 주의 |

### 과목 융합 관점
- **운영체제 (OS)**: RPC 통신 시 스터브(Stub) 내에서 마샬링이 수행되어 커널로 넘겨집니다.
- **보안 (Security)**: 언마샬링 과정에서 악의적으로 조작된 바이트 스트림이 들어올 경우, 원치 않는 코드가 실행되는 **역직렬화 취약점 (Insecure Deserialization)** 이 발생할 수 있어 철저한 검증이 필요합니다.

**📢 섹션 요약 비유**: 어떤 포장지(JSON/Protobuf)를 쓰느냐에 따라 배송료(대역폭)와 하차 속도(파싱 성능)가 결정됩니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오: 모바일 게임 서버 통신
실시간 대전 게임에서 수 밀리초(ms) 단위의 반응 속도가 중요합니다.
- **적용**: JSON 대신 **Google Protocol Buffers (Protobuf)** 를 사용한 마샬링을 선택합니다.
- **판단**: JSON은 텍스트 파싱 오버헤드가 크고 데이터량이 많아 모바일 데이터 소모가 큽니다. 바이너리 기반의 마샬링은 패킷 크기를 최소화하고 CPU 부하를 획기적으로 줄여줍니다.

### 도입 체크리스트
1. **타입 안전성**: 송수신 데이터 형식이 엄격하게 지켜져야 하는가?
2. **성능 vs 가독성**: 디버깅 편의성이 중요한가(JSON), 절대적인 성능이 중요한가(Binary)?
3. **보안**: 신뢰할 수 없는 외부로부터 오는 데이터를 언마샬링하는가? (화이트리스트 기반 검증 필수)

### 안티패턴
- **뚱뚱한 객체 전송 (Fat Object)**: 필요 없는 필드까지 모두 포함된 거대 객체를 통째로 마샬링하면 네트워크 비용이 낭비됩니다. 전송 전용 객체인 **DTO (Data Transfer Object)** 를 사용하여 슬림하게 마샬링해야 합니다.

**📢 섹션 요약 비유**: 이삿짐을 쌀 때 안 쓰는 물건까지 다 싸는 것보다, 꼭 필요한 것만 추려(DTO) 효율적으로 짐을 꾸리는 것이 지혜입니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 기대효과
- **시스템 간 연동성 극대화**: 서로 다른 언어와 환경을 가진 시스템들이 자유롭게 데이터를 주고받을 수 있습니다.
- **추상화 제공**: 개발자는 내부 변환 과정을 몰라도 함수 호출만으로 데이터를 전송할 수 있습니다.

### 결론
마샬링과 언마샬링은 분산 시스템이라는 거대한 기계가 돌아가게 만드는 **"윤활유이자 관절"** 입니다. 데이터가 흐르는 모든 곳에는 변환이 존재하며, 이 과정의 효율성이 곧 전체 시스템의 성능을 좌우합니다. 앞으로는 데이터 크기가 비약적으로 커지는 AI 및 빅데이터 시대에 맞춰, 더욱 빠르고 안전한 차세대 직렬화 기술이 지속적으로 등장할 것입니다.

**📢 섹션 요약 비유**: 세상 모든 정보가 '디지털'이라는 공통 규격으로 포장되어 흐르는 한, 마샬링 기술은 소통의 가장 근본적인 문법으로 남을 것입니다.

2. **원리**: 친구는 택배로 받은 레고 조각들을 다시 설명서대로 조립해서 원래 성으로 만드는 데, 이걸 '언마샬링'이라고 부른답니다.
