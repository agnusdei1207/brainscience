## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 이벤트 루프 (Event Loop)는 단일 스레드 환경에서 비차단 I/O (Non-blocking I/O) 작업을 오케스트레이션하기 위해, 커널이나 워커 스레드로부터 완료된 작업을 감지하고 콜백을 실행하는 무한 루프 제어 구조이다.
> 2. **가치**: 스레드 생성 오버헤드와 컨텍스트 스위칭 비용을 최소화하면서도 수만 개의 동시 접속을 효율적으로 처리할 수 있는 높은 확장성(Scalability)을 제공한다.
> 3. **융합**: 리눅스의 `epoll`, macOS의 `kqueue` 등 운영체제 수준의 I/O 멀티플렉싱 기술과 결합하여, 현대적인 서버 사이드 자바스크립트(Node.js) 런타임의 동시성 모델을 형성한다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: **이벤트 루프 (Event Loop)**는 프로그램의 실행 흐름을 유지하며, 발생한 이벤트(I/O 완료, 타이머 만료 등)를 큐(Queue)에서 하나씩 꺼내어 대응하는 핸들러(콜백 함수)에게 전달하는 메커니즘이다.

- **💡 비유**: 이벤트 루프는 식당의 **"유능한 1인 점원"**과 같다. 점원은 손님의 주문을 받고 요리사(커널/워커)에게 전달한 뒤, 요리가 나올 때까지 멍하니 기다리지 않고 다음 손님의 주문을 받는다. 요리가 완성 벨(이벤트)이 울리면 그때 비로소 음식을 서빙(콜백 실행)한다.

- **등장 배경**:
  1. **C10K 문제**: 스레드당 클라이언트 하나를 할당하는 전통적 방식은 메모리 부족과 컨텍스트 스위칭 부하로 인해 동시 접속자 1만 명(C10K)을 넘기 힘들었다.
  2. **자바스크립트의 단일 스레드 특성**: 웹 브라우저 엔진에서 탄생한 자바스크립트는 구조적으로 단일 스레드에서 동작하므로, 멈춤(Blocking) 없는 실행을 위한 비동기 모델이 필수적이었다.
  3. **I/O 멀티플렉싱의 발전**: OS 수준에서 수만 개의 소켓 상태를 한꺼번에 감시할 수 있는 기술이 성숙하면서, 이를 활용한 중앙 집중식 루프 모델이 가능해졌다.

- **📢 섹션 요약 비유**: 모든 요청을 혼자 관리하지만, 시간이 걸리는 일은 외주(커널)를 주고 결과만 챙기는 스마트한 관리 사무소와 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### Node.js 이벤트 루프의 6단계 (Phases) (표)

| 단계명 (Phase) | 역할 | 처리 내용 | 비유 |
|:---|:---|:---|:---|:---|
| **Timers** | 타이머 콜백 처리 | `setTimeout`, `setInterval` 만료 확인 | 알람 시계 확인 |
| **Pending Callbacks** | 지연된 I/O 콜백 | 이전 루프에서 미처 처리 못한 TCP 오류 등 | 밀린 업무 처리 |
| **Idle, Prepare** | 내부 관리 | Node.js 내부 실행을 위한 준비 단계 | 업무 시작 전 정돈 |
| **Poll** | 새로운 I/O 이벤트 감지 | 새로운 소켓 연결, 파일 읽기 완료 대기 및 실행 | 메일함 확인 |
| **Check** | 즉시 실행 콜백 | `setImmediate`로 등록된 콜백 실행 | 퇴근 전 마지막 체크 |
| **Close Callbacks** | 종료 이벤트 처리 | `socket.on('close', ...)` 등의 정리 작업 | 뒷정리 및 문 잠그기 |

#### libuv 기반의 비동기 I/O 아키텍처

Node.js의 이벤트 루프는 실제로는 C++ 라이브러리인 **libuv** 위에서 동작하며, OS 커널과 워커 스레드 풀을 유기적으로 활용한다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │                 Node.js / libuv 이벤트 루프 구조                 │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │   [V8 Engine] (JS Thread)       [libuv Event Loop]           │
  │   ┌──────────────┐              ┌────────────────────────┐   │
  │   │ Call Stack   │──(Request)──▶│  1. Timers             │   │
  │   │              │              │  2. Pending I/O        │   │
  │   │ [ sync code ]│◀──(Callback)─┤  3. Poll (I/O Wait)    │   │
  │   └──────────────┘              │  4. Check / Close      │   │
  │                                 └──────────▲─────────────┘   │
  │                                            │                 │
  │   [OS Kernel] (Async I/O)       [Worker Threads] (Sync I/O)  │
  │   ┌──────────────┐              ┌────────────────────────┐   │
  │   │ epoll / kqueue│◀────────────▶│ Thread Pool (Default 4) │   │
  │   │ (Network I/O)│              │ (File I/O, Crypto)     │   │
  │   └──────────────┘              └────────────────────────┘   │
  │                                                              │
  │   1. JS: 비동기 함수 호출 (예: fs.readFile)                    │
  │   2. libuv: 커널 혹은 워커 스레드에게 작업 위임                 │
  │   3. Loop: 루프를 돌며 작업 완료 여부를 지속적으로 체크           │
  │   4. JS: 완료된 작업의 콜백을 V8 스택으로 올려 실행              │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 자바스크립트 코드 자체는 단일 스레드(V8 엔진)에서 실행되지만, 시간이 오래 걸리는 작업은 **libuv**가 가로채어 운영체제 커널(네트워크 I/O)이나 자신의 **워커 스레드 풀**(파일 시스템, 암호화 등)로 넘긴다. 이벤트 루프는 이 작업들의 상태를 주기적으로 확인하는 감시자 역할을 한다. 작업이 완료되면 이벤트 루프는 해당 콜백을 큐에 담고, 자바스크립트 메인 스레드가 비어있을 때 하나씩 실행시킨다. 이 덕분에 Node.js는 단일 스레드임에도 불구하고 실제로는 멀티스레드와 비동기 커널의 힘을 빌려 엄청난 동시성을 확보한다.

#### 심층 동작 원리: microtask 큐 (nextTick & Promise)

이벤트 루프의 각 단계 사이사이에는 **microtask queue**가 존재한다. `process.nextTick()`과 `Promise`의 콜백은 현재 단계가 끝나자마자, 다음 루프 단계로 넘어가기 전에 우선적으로 실행된다. 이는 지연 시간을 최소화하기 위한 설계다.

- **📢 섹션 요약 비유**: 큰 업무(Phase)들 사이사이에 급한 쪽지(microtask)를 확인하여 즉시 처리함으로써 응답성을 극대화하는 기민한 비서와 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석

#### 비교: 이벤트 루프 vs 멀티스레드 (Worker Per Connection)

| 비교 항목 | 이벤트 루프 (Node.js) | 멀티스레드 (Traditional Java) |
|:---|:---|:---|
| **동시성 모델** | 단일 스레드 + 비동기 I/O | 스레드당 연결 하나 (Thread per Conn) |
| **메모리 오버헤드** | 낮음 (스레드 하나만 유지) | 높음 (스레드당 수 MB 할당) |
| **컨텍스트 스위칭** | 거의 없음 | 매우 빈번함 (성능 하락 요인) |
| **적합한 상황** | I/O 바운드 (채팅, API 서버) | CPU 바운드 (이미지 처리, 복잡 연산) |
| **개발 난이도** | 비동기 흐름 제어 필요 | 동기화(Lock/Mutex) 주의 필요 |

#### 과목 융합 관점: 운영체제와 네트워크

- **운영체제 (OS)**: 이벤트 루프의 'Poll' 단계는 리눅스의 `epoll_wait` 시스템 콜을 기반으로 한다. 이는 수천 개의 소켓 중 데이터가 도착한 소켓만 핀셋으로 집어내듯 알려주므로 루프 효율이 매우 높다.
- **네트워크 (Networking)**: TCP 3-way Handshake 과정에서 발생하는 대기 시간을 이벤트 루프는 '유휴 시간'으로 활용하여 다른 요청을 처리한다.

- **📢 섹션 요약 비유**: 수천 대의 전화기를 한 명이 지키며 벨이 울리는 전화만 받는 것(이벤트 루프)과, 전화기마다 담당자를 한 명씩 두는 것(멀티스레드)의 차이와 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단

#### 실무 시나리오: 이벤트 루프 차단 (Event Loop Blocking) 사고
실시간 대전 게임 서버를 Node.js로 개발하던 중, 특정 사용자의 랭킹 정렬 알고리즘(CPU 집중 작업)이 1초간 실행되었다. 이 동안 이벤트 루프의 'Poll' 단계로 넘어가지 못해 다른 모든 사용자의 네트워크 패킷 처리가 멈추는 '프리징' 현상이 발생했다. 해결책은 해당 연산을 `Worker Threads`로 분리하거나, `setImmediate`를 사용하여 연산을 여러 루프에 걸쳐 쪼개서 실행하는 것이다.

#### 도입 체크리스트
- **비차단 확인**: 사용하는 라이브러리가 내부적으로 동기(Sync) 함수를 사용하지 않는가? (예: `fs.readFileSync` 금지)
- **모니터링**: `perf_hooks`를 사용하여 이벤트 루프 지연 시간(Lag)을 실시간 측정하고 있는가?
- **부하 분산**: CPU 연산이 필요한가? (그렇다면 Node.js보다는 Go나 Rust를 고려하거나 워커 스레드 사용)

#### 안티패턴
- **Don't block the event loop**: 거대한 JSON 파싱이나 무한 루프 코드는 메인 스레드를 점유하여 서버를 마비시킨다.
- **Callback Hell**: 중첩된 콜백은 가독성을 해치고 에러 처리를 누락시킨다. (Async/Await 필수 사용)

- **📢 섹션 요약 비유**: 점원이 한 손님의 까다로운 요구사항을 들어주느라 다른 손님들을 줄 서게 만들면(Blocking) 식당 전체의 평판이 떨어지는 것과 같습니다.

### Ⅴ. 기대효과 및 결론

#### 정량/정성 기대효과

| 구분 | 도입 전 (Multi-thread) | 도입 후 (Event Loop) | 개선 효과 |
|:---|:---|:---|:---|
| **동시 접속 수** | 서버당 5,000개 한계 | 서버당 50,000개 이상 | **수용량 10배 향상** |
| **메모리 점유** | 스레드 오버헤드로 높음 | 단일 스레드로 매우 낮음 | 하드웨어 비용 절감 |
| **응답성** | 컨텍스트 스위칭 지연 발생 | 즉각적인 이벤트 반응 | 실시간 서비스 품질 개선 |

#### 미래 전망
이벤트 루프 모델은 Node.js의 성공 이후 Python(asyncio), PHP(Swoole), Java(Netty) 등 거의 모든 언어로 확산되었다. 최근에는 Rust의 `Tokio`와 같이 타입 안전성과 성능을 모두 잡은 이벤트 루틴들이 등장하고 있으며, WASM(WebAssembly) 환경에서도 비동기 제어의 핵심 모델로 활약할 전망이다.

- **참고 표준**: CommonJS, ECMAScript (Event Loop semantics)

- **📢 섹션 요약 비유**: "멈추지 않는 흐름"을 만들어내는 이벤트 루프는 현대 네트워크 아키텍처를 지탱하는 가장 효율적인 심장 박동과 같습니다.
