## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스레드 풀 (Thread Pool)은 일정한 개수의 스레드 (Thread)를 미리 생성해 두고, 작업 큐 (Task Queue)에 들어오는 요청을 할당받아 처리하는 동시성 (Concurrency) 제어 기법이다.
> 2. **가치**: 스레드의 생성 및 소멸에 따른 막대한 운영체제 (OS, Operating System) 오버헤드를 제거하고, 과도한 컨텍스트 스위칭 (Context Switching)으로 인한 자원 고갈 (OOM, Out of Memory)을 방지하여 서버의 안정적 처리량을 보장한다.
> 3. **융합**: 자바 (Java)의 ExecutorService, 웹 서버 (Web Server)의 다중 요청 처리 구조, 데이터베이스 커넥션 풀 (DB Connection Pool) 등 시스템 아키텍처 전반의 확장성 (Scalability)을 결정하는 핵심 설계 패턴이다.

### Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 스레드 풀 (Thread Pool)은 프로그램 실행 초기나 필요 시점에 일정 수의 워커 스레드 (Worker Thread)를 풀 (Pool) 형태의 컨테이너에 미리 보관해 두고, 클라이언트의 작업 요청이 발생하면 대기 중인 스레드에 작업을 매핑하여 실행하는 동시성 관리 디자인 패턴이다.
- **필요성**: 모든 요청마다 스레드를 새로 생성하는 'Create-on-Demand' 방식은 짧은 주기의 작업을 처리할 때 작업 처리 시간보다 스레드 생성 및 컨텍스트 스위칭에 소요되는 커널 수준의 시스템 시스템 호출 (System Call) 비용이 더 커지는 배보다 배꼽이 더 큰 현상을 유발한다. 또한 예측할 수 없는 트래픽 폭증 시 무제한적인 스레드 생성은 메모리 부족 현상을 초래하여 시스템 전체를 다운시킨다.
- **등장 배경**: 과거 단순한 단일 스레드 (Single-thread) 모델에서 다중 스레드 (Multi-thread) 모델로 진화하면서 서버의 처리량은 비약적으로 증가했다. 그러나 OS 수준의 하드웨어 스레드 관리는 무거운 커널 (Kernel) 객체를 다루는 일이므로, 이를 애플리케이션 레벨에서 재사용(Reuse)하려는 패러다임이 필수적으로 대두되었다.

이 다이어그램은 요청마다 스레드를 생성하는 기존 방식이 왜 고부하 환경에서 치명적인 한계에 부딪히는지, 그리고 그로 인해 어떤 시스템적 병목이 유발되는지를 보여준다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │        Create-on-Demand 방식의 한계 (스레드 폭발 문제)        │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │   [Client]                                  [Server]        │
  │   Req 1 ────────> [Create Thread 1] ──> 처리 ──> [Destroy]    │
  │   Req 2 ────────> [Create Thread 2] ──> 처리 ──> [Destroy]    │
  │     ...                                                     │
  │   Req 10000 ────> [Create Thread 10000]                     │
  │                        │                                    │
  │                        ▼                                    │
  │             ⚠ Memory Exhaustion (OOM) 발생                  │
  │             ⚠ 막대한 Context Switching 오버헤드             │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 흐름의 핵심은 스레드의 생명 주기 관리 비용이 작업 처리 비용을 압도하는 '오버헤드 역전' 현상이다. 리눅스 (Linux) 환경에서 스레드 하나를 생성하면 커널 공간의 자료구조와 사용자 공간의 스택 (Stack, 일반적으로 1~2MB) 메모리가 할당되어야 한다. 만약 10,000개의 요청이 동시에 들어오면 순식간에 10GB 이상의 메모리가 소비되며 OOM (Out of Memory) 킬러에 의해 프로세스가 강제 종료될 수 있다. 설령 메모리가 버틴다 하더라도 물리 코어 수를 훌쩍 넘는 스레드들 간의 컨텍스트 스위칭이 끊임없이 발생하여 CPU (Central Processing Unit) 캐시 미스 (Cache Miss)가 폭증하고 실제 비즈니스 로직 연산에 쓰이는 시간은 급감하게 된다. 실무에서는 이러한 방식을 '스레드 폭발 (Thread Explosion)' 안티패턴으로 규정하고 엄격히 금지하고 있다.

- **📢 섹션 요약 비유**: 마치 택시 회사에서 손님이 부를 때마다 공장에서 택시를 새로 조립해서 보내고(Create-on-Demand), 운행이 끝나면 택시를 바로 폐차하는 비효율적인 방식 대신, 일정 대수의 택시를 승강장에 미리 대기시켜 놓고(Thread Pool) 교대로 손님을 태우는 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술/프로토콜 | 비유 |
|:---|:---|:---|:---|:---|
| **작업 큐 (Task Queue)** | 대기 중인 작업을 순서대로 보관 | 스레드 안전 (Thread-safe)한 블로킹 큐로 구현 | BlockingQueue, FIFO | 택시 승강장의 손님 대기줄 |
| **워커 스레드 (Worker Thread)** | 큐에서 작업을 꺼내어 실제 실행 | 무한 루프를 돌며 큐의 작업을 Polling | `while(true) { queue.take().run() }` | 교대로 운행하는 택시 기사 |
| **스레드 풀 관리자 (Manager)** | 스레드 생성, 소멸 및 상태 모니터링 | Core 사이즈 유지, 유휴 스레드 회수 정책 적용 | ThreadPoolExecutor (Java) | 택시 배차 관리소장 |
| **거절 정책 (Rejection Policy)** | 큐가 가득 찼을 때 신규 요청 처리 방안 | 예외 발생, 호출자 실행, 버리기 등 선택 | RejectedExecutionHandler | 대기줄 초과 시 입장 통제 안내원 |
| **작업 (Task)** | 처리되어야 할 비즈니스 로직 단위 | Runnable 또는 Callable 인터페이스 구현 | 람다(Lambda), 클로저(Closure) | 손님의 목적지 이동 요청 |

#### 스레드 풀 내부 동작 구조

스레드 풀의 생산자-소비자 (Producer-Consumer) 패턴 기반 아키텍처를 시각화하면, 외부 요청과 실제 처리 스레드가 작업 큐를 매개로 어떻게 완벽히 비동기적으로 분리(Decoupling)되는지 파악할 수 있다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │                 스레드 풀 (Thread Pool) 내부 아키텍처             │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │  [Producer]            [Task Queue]          [Thread Pool]    │
  │                                                               │
  │  Req 1 ──┐                                   ┌──────────┐     │
  │  Req 2 ──┼──> [ Task 1 | Task 2 | ... ] ──>  │ Worker 1 │     │
  │  Req 3 ──┘           (Blocking Queue)        ├──────────┤     │
  │   ...          ▲                             │ Worker 2 │     │
  │  Req N ──┐     │                             ├──────────┤     │
  │          └──> [ Reject Policy (큐 가득참)]     │ Worker K │     │
  │                                              └──────────┘     │
  │                                                               │
  │   1. 작업 제출(Submit) → 2. 큐 대기 → 3. 워커 스레드가 작업 획득 │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 구조도의 핵심은 스레드 풀이 거대한 버퍼 (Buffer) 역할을 하는 '작업 큐'를 통해 예측 불가한 트래픽 스파이크(Spike)의 충격을 안전하게 흡수한다는 점이다. 생산자 (Producer)인 클라이언트 스레드는 작업의 완료를 기다리지 않고 큐에 넣기만 한 뒤 즉시 반환되므로 높은 외부 응답성을 지속적으로 유지할 수 있다. 한편 풀 내부의 워커 스레드는 OS의 스케줄링에 따라 큐에서 작업을 능동적으로 꺼내어 (take) 처리하며, 큐가 비어있을 때는 조건 변수 (Condition Variable)를 통해 대기 (Wait) 상태에 들어가 무의미한 CPU 사이클 낭비(Busy-waiting)를 멈춘다. 만약 물리적 한계로 큐의 용량(Capacity)마저 초과하는 과도한 요청이 유입되면, 스레드 풀 관리자는 거절 정책(Rejection Policy)을 발동하여 서버가 완전히 다운되는 것을 막는 백프레셔 (Backpressure)를 행사한다. 이는 장애 격리를 위한 핵심 방어 메커니즘이다.

#### 심층 동작 원리: 작업 제출부터 완료까지
① **작업 제출**: 클라이언트가 `submit()` 메서드를 호출하여 Task 인터페이스(Runnable 등)를 풀에 전달한다.
② **스레드 할당 판단**: 현재 활성 워커 스레드 수가 설정된 코어 스레드 풀 크기(Core Pool Size)보다 작으면, 즉시 운영체제에 새 커널 스레드를 요청해 생성한 뒤 작업을 위임한다.
③ **큐잉 (Queuing)**: 활성 스레드가 코어 크기에 도달하여 모두 작업 중이라면, 새로 들어온 작업은 임시로 안전한 `BlockingQueue`에 대기 상태로 저장된다.
④ **최대 확장 (Max Pool Expansion)**: 큐마저 가득 찼지만, 전체 활성 스레드가 최대 스레드 크기(Maximum Pool Size)에 도달하지 않았다면, 추가 스레드를 한시적으로 생성하여 밀린 큐의 병목을 강제로 뚫어낸다.
⑤ **거절 (Rejection)**: 최대 스레드 수까지 모두 동작 중이고 큐도 가득 찬 한계 상황에서는 사전에 정의된 거절 정책 (예: `AbortPolicy` 예외 발생)을 발동하여 악의적이거나 비정상적인 폭주 요청을 거부한다.

- **📢 섹션 요약 비유**: 이는 마치 놀이공원에서 한정된 수의 롤러코스터(워커 스레드)를 운영하며, 손님들을 대기줄(작업 큐)에 세워 순서대로 태우고, 대기줄마저 꽉 찰 정도로 인파가 몰리면 공원 안전을 위해 입장객의 출입을 일시 제한(거절 정책)하는 시스템과 동일하게 동작합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 비교 1: 스레드 풀 크기 설정 전략 (Fixed vs Cached)

스레드 풀을 설계할 때 가장 중요한 것은 워크로드의 특성에 맞는 스레드 관리 전략을 선택하는 것이다.

| 비교 항목 | 고정 스레드 풀 (Fixed Thread Pool) | 캐시 스레드 풀 (Cached Thread Pool) | 판단 포인트 |
|:---|:---|:---|:---|
| **스레드 수 제한** | 고정됨 (Core = Max) | 무제한 (Integer.MAX_VALUE) | 시스템의 트래픽 예측 가능성 |
| **작업 큐 특성** | 무한 또는 유한 큐 (LinkedBlockingQueue) | 큐잉이 없는 동기화 큐 (SynchronousQueue) | 대용량 메모리 가용성 여부 |
| **유휴 스레드 관리** | 종료되지 않고 상시 대기 유지 | 일정 시간 (예: 60초) 유휴 시 자동 반환 및 종료 | 유휴 자원의 지속 유지 비용 |
| **적합한 워크로드** | CPU-bound, 장기적이고 안정적인 부하 처리 | I/O-bound, 짧고 변동성이 큰 단기 트래픽 | 스레드 생성 비용 대비 실제 처리 시간의 비율 |

#### 비교 2: 동시성 처리에 따른 성능 분석 (스레드 수 vs 처리량)

스레드 수를 무작정 늘린다고 시스템 성능이 선형적으로 향상되지 않는다. 스레드 개수와 전체 처리량(Throughput) 간의 비선형적 상관관계를 그래프로 이해해야 최적화가 가능하다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │            스레드 풀 크기 설정에 따른 성능(Throughput) 변화       │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ Throughput                                                  │
  │     │                       Ideal Peak                      │
  │     │                      .─────────.                      │
  │     │                    .´           `.                    │
  │     │                  .´               ` .                 │
  │     │                 /                    ` .              │
  │     │                / ⚠ Context Switching    ` .           │
  │     │               /    Overhead 증가로 성능 저하          │
  │     │   Under-     /                                        │
  │     │  utilization/                                         │
  │     └────────────────────────────────────────────────────   │
  │                   적정 스레드 수               스레드 수 (N)│
  │                                                             │
  │  💡 적정 N = CPU 코어 수 * ( 1 + 대기시간(Wait) / 서비스시간(Compute) ) │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 성능 그래프가 시사하는 핵심은 스레드 수가 특정 물리적 임계점 (Ideal Peak)을 넘어서는 순간 오히려 시스템의 전체 처리량이 급감하는 역효과가 발생한다는 점이다. 스레드가 너무 적을 때(Under-utilization)는 물리 CPU 코어가 놀고 있어 가용한 자원 대비 처리량이 떨어지지만, 스레드가 기하급수적으로 많아지면 CPU는 실제 유효한 비즈니스 연산보다 스레드 간 문맥 교환 (Context Switching) 자체에 대부분의 클럭 사이클을 낭비하는 치명적인 스래싱 (Thrashing) 상태에 빠진다. 특히 컨텍스트 스위칭마다 CPU의 L1/L2 캐시 라인이 끊임없이 무효화(Invalidation)되면서 메모리 접근 지연이 폭증한다. 실무에서는 이러한 현상을 방지하기 위해 리틀의 법칙 (Little's Law)이나 공식(적정 N)을 활용하여, I/O 대기 시간이 긴 작업일수록 스레드 풀 크기를 유연하게 크게 가져가고, 순수 수학적 CPU 연산 중심일 경우 논리 코어 수에 맞추는 보수적인 방식으로 튜닝을 수행한다.

#### 과목 융합 관점
- **네트워크 (Network)**: 비동기 논블로킹 (Asynchronous Non-blocking) 서버 아키텍처 (예: Nginx, Netty)는 거대한 스레드 풀 대신 단일 또는 소수의 스레드를 사용하는 이벤트 루프 (Event Loop)를 결합하여 수만 개의 커넥션을 처리하는 C10K 문제를 해결하는 또 다른 동시성 패러다임을 보여준다.
- **데이터베이스 (DB, Database)**: 엔터프라이즈 환경의 DB 커넥션 풀 (Connection Pool, 예: HikariCP) 역시 스레드 풀과 완벽히 동일한 철학(무거운 자원 객체의 선제적 생성 및 재사용을 통한 지연시간 최소화)을 물리적 네트워크 세션 객체 관리에 적용한 강력한 평행 이론을 보여준다.

- **📢 섹션 요약 비유**: 두통약을 먹을 때 의사의 처방에 맞춰 적정량을 먹으면(Ideal Peak) 병이 낫지만 한꺼번에 너무 많이 과다 복용하면(스레드 초과) 오히려 독이 되어 몸이 망가지는 것처럼, 적절한 워커 스레드 크기 설정은 시스템의 건강을 결정짓는 처방전과 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 실무 시나리오 및 판단

1. **시나리오 — I/O-bound API 게이트웨이의 타임아웃 병목 현상**: 마이크로서비스 (Microservices) 아키텍처 환경에서 외부 레거시 API를 호출하는 서비스의 응답 지연으로 인해 스레드 풀의 작업 큐가 가득 차고 서버 전체가 마비되는 상황이다. 이 경우 순수 연산과 달리 CPU는 쉬고 있으므로, 아키텍트는 CPU 코어 수의 수십 배에 달하는 넉넉한 스레드 풀을 할당해야 한다. 단, 큐의 크기를 무한정이 아닌 제한된 크기(Bounded Queue)로 강제하고 네트워크 타임아웃(Timeout)을 엄격히 설정하여 백엔드의 장애가 프론트엔드까지 도미노처럼 전파(Cascading Failure)되는 현상을 반드시 차단해야 한다.
2. **시나리오 — 데드락 (Thread Starvation Deadlock)의 함정**: 스레드 풀 내의 A 워커 스레드가 다른 B 워커 스레드가 처리해야 할 하위 작업의 완료를 `Future.get()` 등으로 대기(Wait)하는 구조를 짠 경우를 가정하자. 풀 내부의 모든 스레드가 큐에 있는 다른 작업의 결과를 기다리며 블로킹 상태에 빠지면, 작업을 꺼내줄 스레드가 한 명도 남아있지 않아 시스템 전체가 멈추는 교착 상태가 발생한다. 이러한 안티패턴을 피하기 위해 동일한 풀 내에서는 상호 의존적인 콜백을 스케줄링하지 말고, 필요 시 계층적 작업을 분리된 서로 다른 격리된 스레드 풀(Bulkhead Pattern)에 배정하여 해결해야 한다.

실제 시스템 설계 시 워크로드 특성에 따라 최적의 스레드 풀과 작업 큐의 조합을 도출하기 위한 논리적 의사결정 흐름을 구성하면 다음과 같다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │               스레드 풀 및 큐 유형 선택 의사결정 플로우               │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │   [워크로드 특성 분석 시작]                                         │
  │           │                                                       │
  │           ▼                                                       │
  │     1. 암호화/압축 등 순수 연산(CPU-Bound) 작업인가?                   │
  │       ├─ 예 ─────▶ [Fixed Pool (풀 크기 = CPU 논리 코어 수)]        │
  │       │                                                           │
  │       └─ 아니오 (네트워크 통신, DB 쿼리 등 I/O-Bound 포함)          │
  │           │                                                       │
  │           ▼                                                       │
  │     2. 트래픽 유입 변동성이 극심하고 단위 작업 시간이 매우 짧은가?        │
  │       ├─ 예 ─────▶ [Cached Pool + Synchronous Queue]             │
  │       │             (유휴 스레드 즉각 반환, 큐잉 대기 없이 직접 위임)    │
  │       │                                                           │
  │       └─ 아니오 ──▶ [Custom Pool (Core/Max 분리) + Bounded Queue] │
  │                           │                                       │
  │                           ▼                                       │
  │                     3. 큐 용량마저 가득 찼을 때의 거절 정책은?        │
  │                       ├─ 중요 비즈니스: CallerRunsPolicy (속도 지연)│
  │                       ├─ 일반 트래픽  : AbortPolicy (즉각적인 빠른 실패)│
  │                       └─ 로깅/분석통계: DiscardPolicy (조용히 무시)  │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 의사결정 트리는 스레드 풀 최적화 튜닝이 절대 단일 공식으로 이루어지지 않음을 명확히 보여준다. CPU-Bound 작업에 불필요하게 많은 스레드를 주면 OS 스케줄링 스위칭 비용만 기하급수적으로 늘어나므로 물리적 코어 한계치에 딱 맞추는 것이 정석적인 판단이다. 반면 I/O 대기 시간이 긴 작업은 DB를 기다리는 동안 CPU가 충분히 쉴 시간이 있으므로 넉넉한 수의 스레드를 부여하여 동시성을 극대화해야 한다. 특히 운영 환경에서 각별히 주의할 기술적 판단은 `Unbounded Queue`(크기 제한이 없는 큐)의 기본값 사용을 절대적으로 피해야 한다는 것이다. 단일 노드의 작업 처리 속도보다 클라이언트 유입 속도가 빠른 구간이 수십 초간 지속되면 결국 힙 메모리 고갈 (OOM)로 서버가 흔적도 없이 다운되기 때문에, `Bounded Queue`와 상황에 최적화된 `Reject Policy`를 설계 초기에 결합하여 시스템의 Graceful Degradation(안전한 점진적 성능 저하)을 유도하는 것이 시니어 아키텍트의 방어적 설계(Defensive Design) 철학이다.

#### 도입 체크리스트
- **기술적**: 서비스가 구동되는 서버의 물리적 코어 수와 예상되는 I/O 블로킹 대기 시간 비율을 정확히 측정하여 적정 스레드 수를 공식 기반으로 도출하고, 부하 테스트(Load Test) 도구를 통해 예상 응답 시간을 엄격히 검증했는가?
- **운영·보안적**: 프레임워크가 제공하는 기본 스레드 풀 설정(종종 무한 작업 큐 사용)을 그대로 사용하여 악의적인 DDoS 공격이나 블랙프라이데이 같은 트래픽 스파이크 시 메모리 고갈(OOM) 연쇄 장애 취약점이 방치되어 있지는 않은가?

- **📢 섹션 요약 비유**: 고속도로(CPU)의 차선 개수는 물리적으로 정해져 있는데 무작정 차(스레드)를 쏟아내면 전체가 꽉 막혀 거대한 교통지옥(스래싱)이 되듯, 톨게이트 진입로에 신호등(거절 정책과 큐잉 제한)을 꼼꼼하게 설치하여 통행량을 치밀하게 통제하는 것이 진정한 트래픽 공학(시스템 운영 기술)입니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량/정성 기대효과

| 구분 | 도입 전 (Create-on-Demand) | 도입 후 (Thread Pool) | 기대 효과 및 개선 수준 |
|:---|:---|:---|:---|
| **정량** | 초당 1,000건 요청 초과 시 즉각 OOM 발생 | 한계치 도달 시 거절 정책(Backpressure) 발동 | 서비스 가용성(Availability) **99.99% 보장**, 메모리 폭증 원천 차단 |
| **정량** | 시스템 호출 오버헤드로 응답 지연 변동폭 극심 | 스레드 재사용으로 지연시간 편차 (Jitter) 대폭 감소 | P99 꼬리 응답 레이턴시 (Tail Latency) 평균 **40% 이상 개선** |
| **정성** | 동시성 자원 관리의 책임을 OS 커널에 방치 | 애플리케이션 레벨의 확고한 자원 통제권 획득| 트래픽 과부하 공격에 대한 예측 가능하고 강건한 서버 아키텍처 확립 |

#### 미래 전망
- **가상 스레드 (Virtual Thread)와 경량 동시성의 부상**: 기존 모델의 한계를 극복하기 위해 Java 21의 가상 스레드 (Project Loom)나 Go 언어의 고루틴 (Goroutine) 등 운영체제 커널 개입 없이 사용자 공간(User Space)에서 수백만 개의 스레드를 JVM이나 런타임이 직접 관리하는 M:N 스케줄링 모델이 대두되고 있다. 이로 인해 향후 I/O-bound 작업에서는 전통적인 스레드 풀의 복잡한 튜닝 없이도 스레드당 하나의 작업을 매핑하는 직관적인 코딩으로 극단적인 동시성을 달성하는 패러다임 전환이 일어나고 있다.
- **반응형 프로그래밍 (Reactive Programming)**: 이벤트 루프 기반의 비동기 스트림 데이터 처리 (예: Spring WebFlux) 아키텍처가 확산되며, 범용적인 거대 스레드 풀 하나에 의존하기보다 도메인 역할별로 완벽히 격리된 소규모 워커 풀을 조합하는 세밀한 마이크로 아키텍처 최적화가 백엔드 설계의 필수 역량이 될 것이다.

#### 참고 표준
- **POSIX Threads (pthreads)**: IEEE POSIX 1003.1c 기반의 유닉스/리눅스 커널 스레드 생성 및 관리 인터페이스 표준
- **Java Concurrency Utilities**: JSR-166 표준 명세에 기반한 `java.util.concurrent` 패키지의 Executor 프레임워크 규격

- **📢 섹션 요약 비유**: 기존 스레드 풀이 엔진의 출력을 바퀴로 부드럽게 전달하기 위해 운전자가 직접 단수를 조절하는 튼튼한 수동 기어 박스였다면, 미래의 가상 스레드 기술은 어떤 복잡한 지형에서도 알아서 최적의 동시성 힘을 분배해 내는 지능형 무단 자동 변속기(CVT)로의 눈부신 진화라 할 수 있습니다.

### 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **동시성 (Concurrency) vs 병렬성 (Parallelism)** | 스레드 풀은 주로 하나의 코어에서 촘촘한 컨텍스트 스위칭 분할을 통해 동시성을 달성하며, 다중 코어를 활용하여 물리적 병렬성을 동시에 끌어올린다. |
| **컨텍스트 스위칭 (Context Switching)** | 스레드 풀 크기를 워크로드 대비 잘못 계산하여 초과 할당할 경우, 레지스터 상태 교체 오버헤드로 인해 시스템 전체의 CPU 캐시 효율이 급락하는 원흉이 된다. |
| **데드락 (Deadlock)** | 큐에 대기 중인 여러 작업이 동일한 스레드 풀 내의 다른 작업 완료를 무한정 기다릴 때 스레드 기아(Starvation) 상태를 유발하여 시스템 전체가 멈추는 핵심 원인이 된다. |
| **코루틴 (Coroutine) & 가상 스레드** | 무거운 커널 개입 없이 순수 사용자 공간에서 스케줄링을 수행하여, 전통적인 OS 하드웨어 스레드 풀의 무거운 생성 비용 및 컨텍스트 스위칭 한계를 획기적으로 돌파하는 미래의 대체재다. |
| **리틀의 법칙 (Little's Law)** | 동시 대기 접속자 수 = 초당 시스템 처리량 × 평균 응답시간이라는 대기행열 공식으로, 스레드 풀의 적정 큐 크기와 워커 수를 수학적으로 가장 정확히 모델링하는 기반 이론이다. |
