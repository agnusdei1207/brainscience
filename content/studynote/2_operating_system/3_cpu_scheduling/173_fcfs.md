# 173. FCFS (First-Come, First-Served)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 준비 큐에 도착한 순서대로 CPU를 할당하는 가장 단순한 비선점형 스케줄링 알고리즘
> 2. **가치**: 구현이 매우 단순하며 오버헤드가 적으나, 긴 프로세스가 앞에 있을 경우 평균 대기 시간이 급증하는 호위 효과(Convoy Effect) 발생
> 3. **융합**: 일괄 처리 시스템(Batch System), 디스크 스케줄링(FIFO), 메시지 큐 등 순차적 처리가 보장되어야 하는 영역의 기본 모델

---

### Ⅰ. 개요 (Context & Background)

#### 개념 정의
**FCFS (First-Come, First-Served)**는 프로세스가 준비 큐(Ready Queue)에 도착한 순서(Arrival Time)에 따라 CPU를 할당하는 비선점형(Non-preemptive) 스케줄링 알고리즘이다. 먼저 요청한 프로세스가 CPU를 다 쓸 때까지 점유하며, 구현상 FIFO(First-In First-Out) 큐를 사용한다.

#### 💡 비유: 은행 번호표 시스템
FCFS는 **은행 창구의 번호표 시스템**과 같다. 업무의 복잡도나 소요 시간과 상관없이 번호표를 빨리 뽑은 사람이 먼저 업무를 보고, 앞 사람의 업무가 끝날 때까지 뒷사람은 무조건 기다려야 한다.

#### 등장 배경
1. **단순성(Simplicity)**: 초기 전산 시스템에서 스케줄링 알고리즘을 복잡하게 설계할 자원이 부족했을 때 가장 직관적으로 도입됨.
2. **공정성(Fairness)**: "먼저 온 사람이 먼저 대접받는다"는 인간 사회의 일반적인 공정성 원칙을 시스템에 적용.
3. **일괄 처리(Batch Processing)**: 사용자와의 상호작용이 적고 순차적인 작업 처리가 중요한 환경에서 유용함.

#### 📢 섹션 요약 비유
"마치 고속도로 톨게이트에서 차선 구분 없이 한 줄로 서서 앞 차가 통과할 때까지 기다리는 것과 같습니다."

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 구성 요소 (표)
| 요소명 | 역할 | 내부 동작 | 비유 |
|:---:|:---|:---|:---|
| **FIFO Queue** | 프로세스 대기열 관리 | 도착 순서대로 프로세스 제어 블록(PCB)을 연결 리스트로 관리 | 대기 줄 |
| **Ready Queue** | 실행 대기 상태 | CPU 할당을 기다리는 프로세스들의 집합 | 창구 대기석 |
| **Scheduler** | 할당 결정 | 큐의 머리(Head)에 있는 프로세스를 선택하여 CPU 할당 | 번호표 호출기 |
| **Dispatcher** | 문맥 전환 | 선택된 프로세스에게 CPU 제어권을 넘김 | 상담원 안내 |
| **Non-preemption** | 점유 보장 | 한 번 할당받으면 종료 또는 I/O 발생 전까지 반납하지 않음 | 상담 완료 전 퇴실 불가 |

#### FCFS 동작 구조 및 호위 효과(Convoy Effect)

FCFS의 가장 큰 특징인 도착 순서에 따른 실행과, 그로 인한 성능 저하 현상인 호위 효과를 시각화한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                 FCFS Scheduling & Convoy Effect                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [Scenario A: Short process first]                                  │
│  P1(3ms), P2(3ms), P3(24ms)                                         │
│  ┌──────┬──────┬─────────────────────────────────────────────────┐  │
│  │  P1  │  P2  │                      P3                         │  │
│  └──────┴──────┴─────────────────────────────────────────────────┘  │
│  0      3      6                                                30  │
│  (Avg Waiting Time: (0 + 3 + 6) / 3 = 3ms)                          │
│                                                                     │
│  [Scenario B: Long process first (Convoy Effect)]                   │
│  P3(24ms), P1(3ms), P2(3ms)                                         │
│  ┌─────────────────────────────────────────────────┬──────┬──────┐  │
│  │                      P3                         │  P1  │  P2  │  │
│  └─────────────────────────────────────────────────┴──────┴──────┘  │
│  0                                                24     27     30  │
│  (Avg Waiting Time: (0 + 24 + 27) / 3 = 17ms)                       │
│                                                                     │
│  [Analysis]                                                         │
│  - Scenario B에서 짧은 작업(P1, P2)들이 긴 작업(P3) 뒤에서            │
│    오랫동안 대기하는 현상이 '호위 효과'임.                            │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**다이어그램 해설**:
위 ASCII 다이어그램은 프로세스의 도착 순서에 따른 평균 대기 시간(AWT)의 극적인 변화를 보여준다. Scenario A에서는 짧은 프로세스가 먼저 처리되어 시스템의 전반적인 반응성이 좋으나, Scenario B에서는 실행 시간이 긴 P3가 먼저 도착함에 따라 뒤에 있는 P1, P2가 불필요하게 오래 대기하게 된다. 이처럼 CPU 집약적인 하나의 긴 프로세스가 CPU를 점유하여 많은 I/O 집약적 프로세스들이 큐에서 기다리는 현상을 **호위 효과(Convoy Effect)**라고 하며, 이는 CPU 및 장치 이용률을 저하시키는 주요 원인이 된다.

#### 심층 동작 원리
1. **Enqueue**: 프로세스가 준비 상태가 되면 해당 PCB를 FIFO 큐의 Tail에 추가한다.
2. **Select**: CPU가 유휴 상태(Idle)가 되면 스케줄러는 큐의 Head에서 프로세스를 꺼낸다.
3. **Execute**: 디스패처가 선택된 프로세스에게 CPU를 할당하고, 프로세스는 실행을 시작한다.
4. **Completion**: 프로세스가 작업을 마치거나 스스로 I/O를 요청하여 CPU를 반납할 때까지 계속 점유한다.
5. **Next**: 프로세스가 종료되면 큐의 다음 프로세스에게 기회가 넘어간다.

#### 📢 섹션 요약 비유
"마치 대형 마트 계산대에서 앞 손님이 카트 가득 물건을 계산하는 동안, 껌 한 통을 든 뒷 손님이 하염없이 기다리는 것과 같습니다."

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 스케줄링 알고리즘 심층 비교
| 비교 항목 | FCFS | SJF (Shortest Job First) | RR (Round Robin) |
|:---:|:---|:---|:---|
| **선점 여부** | 비선점형 | 비선점형 (SRT는 선점) | 선점형 |
| **우선순위** | 도착 순서 | 실행 시간 (짧을수록) | 없음 (순환) |
| **평균 대기 시간** | 높음 (호위 효과 발생 시) | 최적 (Minimum) | 중간 |
| **응답 시간** | 길어질 수 있음 | 짧음 (짧은 작업 위주) | 일정 (Time Quantum 의존) |
| **주요 단점** | 호위 효과, 긴 대기 시간 | 실행 시간 예측 필요, 기아 현상 | 타임 슬라이스 오버헤드 |

#### 과목 융합 관점: 디스크 I/O 및 네트워크
- **디스크 스케줄링**: FCFS(FIFO) 방식은 디스크 헤드의 이동 거리를 고려하지 않아 탐색 시간(Seek Time) 최적화가 불가능하지만, 가장 공평한 처리를 보장한다.
- **네트워크 큐잉**: 라우터의 버퍼 관리에서 FIFO는 가장 기본적인 정책으로, 트래픽 폭주 시 큐가 가득 차면 새로 도착한 패킷을 버리는 Tail Drop 현상과 연관된다.

#### 📢 섹션 요약 비유
"FCFS는 정교한 전략보다는 '순서'라는 절대적 규칙을 따르는, 가장 원시적이지만 정직한 질서 유지 방법입니다."

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 실무 시나리오 및 의사결정
1. **일괄 처리 배치 시스템**: 사용자 대화가 필요 없고 작업의 순차적 완료가 중요한 금융권 정산 배치 등에서는 오버헤드가 적은 FCFS가 효율적일 수 있다.
2. **프로세스 간 통신 (IPC) 메시지 큐**: 메시지가 발생한 순서대로 처리되어야 하는 인과 관계가 중요한 통신 모델에서는 FCFS가 필수적이다.
3. **대화형 시스템 (Anti-pattern)**: 웹 서버나 GUI 환경에서 FCFS를 사용하면 특정 무거운 요청 때문에 전체 서비스가 먹통이 되는 치명적 결함이 발생하므로 RR이나 MLFQ 도입이 필수적이다.

#### 도입 체크리스트
- [ ] 작업의 실행 시간 편차가 적은가?
- [ ] 처리 순서의 공정성이 최우선인가?
- [ ] 스케줄링 오버헤드를 극도로 제한해야 하는 환경인가?
- [ ] 응답 시간(Response Time)의 가변성을 수용할 수 있는가?

#### 📢 섹션 요약 비유
"FCFS는 화려한 기술보다는 기본에 충실해야 하는 '공장 컨베이어 벨트' 같은 환경에 가장 적합한 선택입니다."

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량/정성 기대효과 (표)
| 구분 | 도입 전 (복잡한 알고리즘) | 도입 후 (FCFS) |
|:---:|:---|:---|
| **오버헤드** | 스케줄링 결정 시 CPU 소모 큼 | 극소화 (단순 포인터 조작) |
| **예측 가능성** | 알고리즘에 따라 가변적 | 도착 순서에 따라 100% 예측 가능 |
| **구현 난이도** | 자료구조 및 동기화 복잡 | 매우 낮음 (FIFO 큐 활용) |

#### 미래 전망
현대의 일반적인 OS 스케줄러에서는 메인 알고리즘으로 사용되지 않지만, **분산 시스템의 메시지 브로커(Kafka 등)**나 **실시간 데이터 스트리밍** 분야에서 '순서 보장'을 위한 핵심 메커니즘으로 그 가치가 유지되고 있다. 또한, 양자 컴퓨팅이나 뉴로모픽 칩 등 새로운 아키텍처의 초기 단계에서도 가장 먼저 구현되는 기본 모델이 될 것이다.

#### 📢 섹션 요약 비유
"가장 오래된 지도(Map)가 길의 기본 골격을 보여주듯, FCFS는 모든 현대적 스케줄링의 근간이 되는 '영원한 클래식'입니다."
