# 179. 시간 할당량 (Time Quantum / Time Slice)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시분할 시스템(Time-sharing System)에서 한 프로세스가 CPU를 연속적으로 점유할 수 있도록 허용된 최대 시간 단위
> 2. **가치**: 너무 크면 FCFS와 같아져 응답성이 저하되고, 너무 작으면 문맥 교환 오버헤드가 급증하므로 시스템 특성에 맞는 최적의 임계값 설정이 성능의 핵심
> 3. **융합**: 현대 OS의 RR 스케줄링, 가상화 환경의 vCPU 할당, 클라우드 멀티테넌시(Multi-tenancy) 자원 분배의 기본 정책

---

### Ⅰ. 개요 (Context & Background)

#### 개념 정의
**시간 할당량(Time Quantum / Time Slice)**은 라운드 로빈(Round Robin) 스케줄링 등 선점형 알고리즘에서 각 프로세스에게 부여되는 고정된 CPU 실행 시간이다. 프로세스가 이 시간을 모두 소모하면 타이머 인터럽트가 발생하여 강제로 CPU를 반납하고 준비 큐의 끝으로 이동한다.

#### 💡 비유: 체스 경기의 제한 시간
시간 할당량은 **체스나 바둑 경기의 '초읽기'**와 같다. 각 선수는 정해진 시간 동안만 수를 둘 수 있으며, 시간이 다 되면 다음 사람에게 차례를 넘겨야 한다. 이 시간이 너무 길면 상대방이 지루해하고, 너무 짧으면 제대로 된 수를 두기 어려운 것과 같은 원리다.

#### 등장 배경
1. **대화형 시스템의 요구**: 여러 사용자가 동시에 컴퓨터를 사용하는 환경에서 특정 프로세스의 독점을 방지하고 공평한 응답 시간을 보장하기 위해 도입.
2. **시분할(Time-sharing)**: CPU의 압도적인 속도를 이용하여 마치 여러 작업이 동시에 실행되는 것과 같은 환상(Illusion)을 제공하기 위한 장치.
3. **공정성 확보**: 실행 시간이 긴 작업 때문에 짧은 작업이 무한정 대기하는 현상을 방지.

#### 📢 섹션 요약 비유
"마치 놀이공원 기구의 '1회 이용 시간'을 정해두어, 대기 줄에 있는 모든 사람이 골고루 기구를 탈 수 있게 하는 것과 같습니다."

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 시간 할당량 크기에 따른 트레이드오프 (Trade-off)
시간 할당량($q$)의 결정은 시스템 전체 성능을 좌우하는 가장 민감한 튜닝 포인트다.

| 크기 ($q$) | 특징 | 장점 | 단점 | 비유 |
|:---:|:---|:---|:---|:---|
| **$q \to \infty$** (매우 큼) | FCFS와 동일해짐 | 문맥 교환 오버헤드 최소화 | 응답 시간(Response Time) 폭증, 호위 효과 발생 | 무제한 뷔페 |
| **$q \to 0$** (매우 작음) | 프로세서 공유(Sharing) | 모든 프로세스가 동시 실행되는 느낌 | 문맥 교환 오버헤드로 인해 CPU 자원 낭비 | 한 입씩만 주는 식사 |
| **적정 수준** | 보통 10~100ms | 응답성과 처리량의 균형 | 프로세스 특성에 따라 최적값이 다름 | 정해진 시간의 쿼터제 |

#### 시간 할당량과 문맥 교환(Context Switch)의 관계

시간 할당량이 작아질수록 발생하는 문맥 교환의 빈도와 실제 유효 작업 시간의 비중을 시각화한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│             Time Quantum vs. Context Switch Overhead                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [Scenario A: Large Quantum (q=10)]                                 │
│  ┌───────────────────────────┬───┬───────────────────────────┐      │
│  │       Process A (10)      │ C │       Process B (10)      │      │
│  └───────────────────────────┴───┴───────────────────────────┘      │
│  <---------- 10 ---------->  <1>  <---------- 10 ---------->      │
│  (Overhead: 1/11 ≈ 9%)                                              │
│                                                                     │
│  [Scenario B: Small Quantum (q=1)]                                  │
│  ┌───┬───┬───┬───┬───┬───┬───┬───┐                                  │
│  │ A │ C │ B │ C │ A │ C │ B │ C │ ...                              │
│  └───┴───┴───┴───┴───┴───┴───┴───┘                                  │
│  < 1 > < 1 > < 1 > < 1 >                                            │
│  (Overhead: 1/2 = 50%)                                              │
│                                                                     │
│  * C: Context Switch Time                                           │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**다이어그램 해설**:
위 ASCII 다이어그램은 시간 할당량이 시스템 효율성에 미치는 직접적인 영향을 보여준다. Scenario A에서는 할당량이 커서 한 번의 문맥 교환으로 많은 일을 처리하므로 오버헤드가 낮다. 반면, Scenario B처럼 할당량이 너무 작으면 실제 프로세스가 일하는 시간($q=1$)과 문맥을 교환하는 시간($C=1$)이 비슷해져 CPU 자원의 절반이 낭비된다. 따라서 실무에서는 **문맥 교환 시간이 시간 할당량의 10%를 넘지 않도록** 설정하는 것이 권장된다.

#### 심층 동작 메커니즘
1. **Timer Setting**: 프로세스가 CPU를 할당받는 순간, 하드웨어 타이머에 시간 할당량만큼의 인터럽트 발생 시간을 설정한다.
2. **Execution**: 프로세스는 사용자 모드에서 코드를 실행한다.
3. **Interrupt**: 타이머가 0이 되면 하드웨어는 CPU에 인터럽트 신호를 보낸다.
4. **Preemption**: OS 커널은 실행 중인 프로세스의 상태를 저장(Save Context)하고 준비 큐의 Tail로 보낸다.
5. **Dispatch**: 다음 프로세스를 선택하여 CPU를 넘겨준다(Restore Context).

#### 📢 섹션 요약 비유
"할당량 결정은 '한 사람당 말할 수 있는 시간'을 정하는 토론회 규칙과 같습니다. 너무 짧으면 말을 끝낼 수 없고, 너무 길면 독점하게 됩니다."

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 시스템 유형별 최적 시간 할당량
| 시스템 유형 | 권장 할당량 | 주요 목표 | 이유 |
|:---:|:---|:---|:---|
| **실시간 (RTOS)** | 매우 작음 (<1ms) | 확정적 지연시간 보장 | 빠른 반응이 생명임 |
| **데스크탑/GUI** | 중간 (10-100ms) | 사용자 체감 응답성 | 0.1초 이내 반응 시 '즉시'라고 느낌 |
| **서버/배치** | 큼 (100ms+) | 처리량(Throughput) 극대화 | 문맥 교환 손실 최소화 |
| **클라우드/가상화** | 가변적 | 공정한 자원 분배 | 테넌트 간의 간섭 방지 |

#### 과목 융합 관점: 가상화 및 클라우드
- **Hypervisor 스케줄링**: 가상 머신(VM)의 vCPU가 물리 CPU를 점유하는 시간도 일종의 타임 슬라이스로 관리되며, 이를 'VCPU Time Slice'라고 한다.
- **Docker/k8s CPU Limit**: 컨테이너의 CPU 할당량(`cpu-period` 및 `cpu-quota`)은 시간 할당량 개념을 확장하여 특정 주기 동안 사용할 수 있는 총 마이크로초(μs)를 제한한다.

#### 📢 섹션 요약 비유
"시간 할당량은 시스템의 성격에 따라 길이를 조절하는 '맞춤형 바지'와 같습니다. 활동적이면 짧게, 정적이면 넉넉하게 맞춥니다."

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 실무 시나리오 및 의사결정
1. **고부하 웹 서버 (Optimization)**: 트래픽이 많을 때 문맥 교환으로 인한 오버헤드가 크다면 시간 할당량을 약간 늘려 CPU 처리 효율을 높이는 결정을 내릴 수 있다.
2. **게임/멀티미디어 (Responsiveness)**: 프레임 드랍이나 오디오 끊김이 발생한다면 시간 할당량을 줄여 더 자주 스케줄링 기회를 주어 지연(Jitter)을 방지한다.
3. **리눅스 CFS (Evolution)**: 현대 리눅스는 고정된 시간 할당량 대신, 프로세스 수에 따라 동적으로 변하는 `Target Latency`와 가중치 기반의 가변 타임 슬라이스를 사용한다.

#### 도입 체크리스트
- [ ] 문맥 교환 소요 시간이 전체 할당량의 몇 %인가?
- [ ] 대부분의 프로세스가 할당량 종료 전 I/O로 CPU를 반납하는가? (80-20 규칙 적용 여부)
- [ ] 사용자가 화면 랙(Lag)을 느끼는 임계치는 얼마인가?
- [ ] CPU 캐시 지역성(Locality)이 파괴될 정도로 할당량이 작은가?

#### 📢 섹션 요약 비유
"실무에서의 할당량 튜닝은 '정답'을 찾는 것이 아니라, 현재 시스템의 '입맛'에 가장 잘 맞는 소금의 양을 조절하는 과정입니다."

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량/정성 기대효과 (표)
| 구분 | 부적절한 설정 (너무 작음) | 적정 설정 도입 후 |
|:---:|:---|:---|
| **CPU 이용률** | 낮음 (오버헤드 과다) | 높음 (유효 작업 비중 증가) |
| **응답 시간** | 빠르나 작업 진행 더딤 | 사용자 만족도 최적화 |
| **시스템 안정성** | 불안정 (인터럽트 폭주) | 안정적 (예측 가능한 스케줄링) |

#### 미래 전망
멀티코어와 NUMA 구조가 일반화되면서, 단순한 시간 할당량을 넘어 **캐시 적중률과 코어 간 이동 비용을 고려한 지능형 타임 슬라이싱**으로 발전하고 있다. 또한 AI 가속기(NPU) 스케줄링에서는 연산의 원자성(Atomicity)을 고려하여 할당량 대신 '태스크 단위'의 실행권 보장 방식이 결합되는 추세다.

#### 📢 섹션 요약 비유
"시간 할당량은 디지털 문명을 지탱하는 '공정한 시간의 배급제'이며, 미래에는 더욱 정교한 인공지능이 그 배급량을 조절할 것입니다."

### 👶 어린이를 위한 3줄 비유 설명
**개념**: 시간 할당량은 게임기를 친구들과 "5분씩만 하기"로 약속한 시간이랑 똑같아요.
**원리**: 5분이 지나면 내가 아무리 재미있게 하고 있어도 다음 친구에게 게임기를 넘겨줘야 해요.
**효과**: 5분이 너무 길면 친구들이 심심해하고, 너무 짧으면 게임을 시작하자마자 끝내야 하니까 딱 좋은 시간을 정하는 게 중요해요.
