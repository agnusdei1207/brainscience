# 181. 기아 상태 (Starvation) 및 대책

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 특정 프로세스가 우선순위가 높은 다른 프로세스들에 밀려 CPU를 무한정 할당받지 못하고 대기 상태에 머무는 현상 (Indefinite Blocking)
> 2. **가치**: 시스템의 공정성(Fairness)을 파괴하고 특정 작업의 완결성을 저해하므로, 에이징(Aging)과 같은 동적 우선순위 조정 기전이 필수적
> 3. **융합**: 교착 상태(Deadlock)와의 비교 분석, 실시간 시스템의 우선순위 역전(Priority Inversion) 문제, 분산 시스템의 자원 경합과 연계

---

### Ⅰ. 개요 (Context & Background)

#### 개념 정의
**기아 상태(Starvation)**는 스케줄링 알고리즘의 편향성으로 인해 우선순위가 낮은 프로세스가 자원(CPU)을 영원히 할당받지 못할 가능성이 있는 상태를 말한다. 이는 시스템이 멈춘 것은 아니지만(Liveness는 유지), 특정 프로세스 입장에서는 무한 대기(Indefinite Blocking)에 빠진 것과 같다.

#### 💡 비유: 맛집의 무한 대기
기아 상태는 **예약 없이 맛집에 온 손님**과 같다. 예약 손님(우선순위 높은 프로세스)이 계속해서 들어오는 한, 예약하지 않은 손님은 자리가 날 때까지 영원히 기다려야 한다. 식당(CPU)은 계속 돌아가고 있지만, 이 손님은 밥을 먹지 못해 굶주리게(Starvation) 된다.

#### 등장 배경
1. **우선순위 스케줄링의 부작용**: 정적 우선순위를 부여할 경우, 시스템 부하가 높을 때 저우선순위 작업이 소외됨.
2. **SJF/SRT의 효율성 추구**: 시스템 전체의 평균 대기 시간을 줄이려는 시도가 긴 작업(Long Job)의 희생을 강요함.
3. **자원 부족(Resource Scarcity)**: 수요에 비해 공급이 절대적으로 부족한 멀티태스킹 환경의 숙명적 문제.

#### 📢 섹션 요약 비유
"마치 화려한 주연 배우들의 출연이 계속 이어지는 바람에, 엑스트라 배우의 대사가 영원히 뒤로 밀리는 촬영장 상황과 같습니다."

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 기아 상태의 주요 원인 및 사례
| 원인 | 메커니즘 | 발생 사례 |
|:---:|:---|:---|
| **정적 우선순위** | 고정된 우선순위 체계에서 높은 우선순위 태스크가 지속 유입 | 실시간 인터럽트 처리 프로세스 |
| **SJF (Shortest Job First)** | 짧은 프로세스가 계속 도착하여 긴 프로세스를 뒤로 밀어냄 | CPU 집약적인 계산 작업 |
| **자원 독점** | 특정 프로세스가 자원을 점유한 채 놓지 않거나 상호 배제 실패 | 임계 구역(Critical Section) 진입 실패 |
| **네트워크 폭주** | 특정 경로의 트래픽이 과도하여 소수 패킷이 드랍됨 | TCP/IP 패킷 전송 지연 |

#### 기아 상태 발생 구조 및 에이징(Aging) 해결책

준비 큐에서 시간이 지남에 따라 프로세스의 우선순위가 어떻게 변하는지(Aging)를 시각화한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                 Starvation & Aging Mechanism                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [Problem: Starvation]                                              │
│  High Priority (Inflow)  ===> [ Ready Queue ]                       │
│  Low Priority (P_low)    ---> [ Ready Queue ] (Stuck forever!)      │
│                                                                     │
│  [Solution: Aging]                                                  │
│  Priority Index                                                     │
│    ^                                                                │
│    │  (Higher)                                                      │
│    │     |      /                                                   │
│    │     |     /  <-- Aging Factor (e.g., +1 priority per 10min)    │
│    │     |    /                                                     │
│    │     |   /                                                      │
│    │     |  /                                                       │
│    │     | /                                                        │
│    │  (Lower)                                                       │
│    └────────────────────────────────────────────> Wait Time         │
│                                                                     │
│  * Result: P_low eventually gains High Priority and executes.       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**다이어그램 해설**:
위 ASCII 다이어그램은 기아 상태의 발생과 이를 해결하는 핵심 기법인 **에이징(Aging)**을 보여준다. 왼쪽의 기아 상태 구조에서는 우선순위가 높은 프로세스가 계속 유입되어 준비 큐의 앞자리를 차지함으로써 `P_low`가 실행될 기회를 잃는다. 오른쪽의 해결책은 시간이 지남에 따라 대기 중인 프로세스의 우선순위를 점진적으로 높여주는 방식이다. 이를 통해 아무리 낮은 우선순위의 프로세스라도 언젠가는 시스템 내에서 가장 높은 우선순위를 갖게 되어 반드시 실행됨을 보장(Guarantee)한다.

#### 기아 상태 해결을 위한 3대 대책
1. **에이징 (Aging)**: 대기 시간에 비례하여 우선순위를 동적으로 상향 조정 (가장 보편적).
2. **라운드 로빈 (Round Robin)**: 모든 프로세스에게 공평한 시간 할당량을 주어 순환 실행함으로써 원천 차단.
3. **다단계 피드백 큐 (MLFQ)**: 낮은 큐에 머무는 프로세스를 일정 시간 후 상위 큐로 승격(Boost)시키는 정책 사용.

#### 📢 섹션 요약 비유
"에이징은 '오래 기다린 분에게 우선권을 드립니다'라는 줄 서기 예절을 알고리즘으로 구현한 것입니다."

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 기아 상태 vs 교착 상태 (Starvation vs Deadlock)
| 비교 항목 | 기아 상태 (Starvation) | 교착 상태 (Deadlock) |
|:---:|:---|:---|
| **상태 성격** | 프로세스가 대기 중 (실행 가능) | 프로세스가 차단됨 (실행 불가능) |
| **시스템 전체** | 다른 작업은 수행됨 (Liveness OK) | 전체 또는 일부가 멈춤 (Progress NO) |
| **해결 방법** | 우선순위 조정, 자원 할당 정책 | 상호 배제 부정, 예방/회피/발견 |
| **치명도** | 특정 작업 지연 (품질 문제) | 시스템 전체 마비 (가용성 문제) |

#### 과목 융합 관점: 우선순위 역전 (Priority Inversion)
- **공유 자원 경합**: 낮은 우선순위 프로세스($P_L$)가 자원을 점유한 상태에서 중간 우선순위 프로세스($P_M$)가 실행되어, 높은 우선순위 프로세스($P_H$)가 실행되지 못하는 현상.
- **해결책**: 우선순위 상속(Priority Inheritance) 또는 우선순위 천장(Priority Ceiling) 프로토콜을 사용하여 기아 상태와 유사한 지연 문제를 해결한다.

#### 📢 섹션 요약 비유
"교착 상태가 '모든 차가 엉킨 사거리'라면, 기아 상태는 '직진 신호만 계속되어 좌회전 차가 영원히 못 가는 상황'입니다."

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 실무 시나리오 및 의사결정
1. **실시간 제어 시스템 (Critical)**: 중요도가 낮은 로그 기록 작업이 기아 상태에 빠져 디스크 버퍼 오버플로우를 일으키지 않도록 최소한의 실행 주기를 보장해야 한다.
2. **클라우드 멀티테넌시 (Fairness)**: 유료 가입자의 우선순위가 높더라도, 무료 사용자의 요청이 1분 이상 대기하지 않도록 슬라이딩 윈도우 기반의 우선순위 조정을 적용한다.
3. **데이터베이스 락 (Locking)**: 공유 락(Shared Lock)이 계속 발생하여 베타 락(Exclusive Lock)을 원하는 쓰기 작업이 굶지 않도록 락 승격(Promotion) 정책을 검토한다.

#### 도입 체크리스트
- [ ] 시스템 부하 피크 시 대기 시간의 표준 편차가 너무 큰가?
- [ ] 특정 프로세스의 대기 시간이 무한대로 발산할 가능성이 있는가?
- [ ] 에이징 적용 시 우선순위 역전 오버헤드가 발생하는가?
- [ ] 공정성(Fairness) 보장이 응답성(Throughput) 손실을 정당화하는가?

#### 📢 섹션 요약 비유
"기아 상태 방지는 효율성이라는 명분 아래 소외되는 태스크가 없도록 보살피는 '시스템적 복지'와 같습니다."

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량/정성 기대효과 (표)
| 구분 | 대책 도입 전 | 대책 도입 후 (Aging 등) |
|:---:|:---|:---|
| **최대 대기 시간** | 예측 불가 (Unbounded) | 상한 존재 (Bounded) |
| **사용자 만족도** | 특정 작업 먹통 불만 | 모든 작업의 점진적 진행 |
| **시스템 신뢰성** | 특정 상황에서 오류 유발 가능 | 결정론적 실행 보장 |

#### 미래 전망
AI 기반 스케줄링에서는 고정된 에이징 수식 대신, **강화 학습(Reinforcement Learning)**을 통해 시스템의 전체 처리량과 개별 프로세스의 기아 상태를 동시에 최적화하는 동적 정책이 도입되고 있다. 또한, 엣지 컴퓨팅 환경에서는 에너지 효율과 기아 상태 방지의 상충 관계를 해결하기 위한 전력 효율적 스케줄링(Energy-Aware Scheduling)이 중요해질 것이다.

#### 📢 섹션 요약 비유
"기아 상태 대책은 기술의 차가운 효율성 위에 인간의 '공정'이라는 따뜻한 가치를 입히는 필수 과정입니다."

### 👶 어린이를 위한 3줄 비유 설명
**개념**: 기아 상태는 친구들이 계속 새치기를 해서 내가 영원히 미끄럼틀을 못 타는 상태예요.
**원리**: 힘이 센 친구(우선순위 높은 프로세스)들만 계속 놀다 보니, 힘이 약한 친구는 구석에서 기다리기만 하는 거죠.
**효과**: "오래 기다린 친구부터 타자!"라고 규칙을 바꾸면(에이징), 누구나 기다리면 반드시 미끄럼틀을 탈 수 있게 돼요.
