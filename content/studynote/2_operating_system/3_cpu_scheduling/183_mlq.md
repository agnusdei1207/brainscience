# 183. 다단계 큐 (MLQ, Multi-Level Queue)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 성격이 다른 프로세스들을 여러 개의 독립적인 큐로 분류하고, 각 큐에 서로 다른 스케줄링 알고리즘과 고정된 우선순위를 부여하는 기법
> 2. **가치**: 작업의 특성(대화형 vs 배치)에 따른 맞춤형 처리가 가능하여 시스템의 응답성과 효율성을 동시에 고려할 수 있음
> 3. **융합**: 고정 우선순위 스케줄링의 변형으로, 현대 OS의 실시간(Real-time) 태스크와 일반 태스크 분리 관리의 기초가 됨

---

### Ⅰ. 개요 (Context & Background)

#### 개념 정의
**다단계 큐(MLQ, Multi-Level Queue)**는 준비 큐를 여러 개의 개별 큐로 분할하고, 각 프로세스를 영구적으로 특정 큐에 할당하는 스케줄링 방식이다. 각 큐는 고유의 스케줄링 알고리즘(예: 상위 큐는 RR, 하위 큐는 FCFS)을 가질 수 있으며, 큐 사이에는 엄격한 우선순위가 존재한다.

#### 💡 비유: 공항의 체크인 카운터
MLQ는 **공항의 클래스별 체크인 카운터**와 같다. 퍼스트 클래스(최상위 큐), 비즈니스(중간 큐), 이코노미(최하위 큐) 카운터가 별도로 존재하며, 승객은 자신의 티켓 등급에 따라 정해진 줄에만 서야 한다. 상위 클래스 승객이 한 명이라도 있으면 하위 클래스는 대기해야 하는 구조다.

#### 등장 배경
1. **프로세스 특성 분리**: 빠른 응답이 필요한 전면 작업(Foreground)과 처리량이 중요한 후면 작업(Background)을 동일하게 취급하는 것의 불합리함 개선.
2. **스케줄링 오버헤드 감소**: 한 큐 내에서 모든 프로세스를 관리하는 것보다 성격별로 나누어 단순한 알고리즘을 적용하는 것이 효율적임.
3. **사용자 경험 최적화**: 대화형 프로세스에게 우선권을 주어 시스템이 기민하게 동작하는 것처럼 보이게 함.

#### 📢 섹션 요약 비유
"마치 도로를 버스 전용 차로, 승용차 차로, 화물차 차로로 나누어 각 차량의 특성에 맞게 소통시키는 것과 같습니다."

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### MLQ의 구조 및 우선순위 체계
MLQ는 큐 간의 스케줄링과 큐 내부의 스케줄링이라는 이중 구조를 가진다.

| 큐 유형 | 우선순위 | 스케줄링 알고리즘 | 대상 프로세스 | 비유 |
|:---:|:---:|:---|:---|:---|
| **System Queue** | 최고 | FCFS / Fixed | 커널 태스크, 하드웨어 제어 | 관제탑 명령 |
| **Interactive Queue** | 높음 | RR (Short Quantum) | GUI, 에디터, 사용자 입력 | 매표소 문의 |
| **Batch Queue** | 낮음 | FCFS | 데이터 정산, 컴파일, 백업 | 화물 하역 |
| **Idle Queue** | 최저 | FCFS | 화면 보호기, 시스템 청소 | 청소부 작업 |

#### 다단계 큐 아키텍처 다이어그램

상위 큐가 비어 있어야만 하위 큐가 실행되는 MLQ의 엄격한 계층 구조를 시각화한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                 Multi-Level Queue Architecture                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [High Priority]                                                    │
│  Queue 1: System Tasks  (Fixed Priority)                            │
│    [ P1 ] -> [ P2 ] -> [ P3 ]                                       │
│          |                                                          │
│          v (If Q1 is empty)                                         │
│                                                                     │
│  Queue 2: Interactive   (Round Robin)                               │
│    [ P4 ] -> [ P5 ]                                                 │
│          |                                                          │
│          v (If Q1, Q2 are empty)                                    │
│                                                                     │
│  Queue 3: Batch Jobs    (FCFS)                                      │
│    [ P6 ] -> [ P7 ] -> [ P8 ] -> [ P9 ]                             │
│                                                                     │
│  [Low Priority]                                                     │
│                                                                     │
│  * Problem: Queue 3 may suffer from Starvation if Q1/Q2 are busy.   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**다이어그램 해설**:
위 ASCII 다이어그램은 MLQ의 수직적 계층 구조를 보여준다. 프로세스는 생성 시점에 자신의 성격에 맞는 큐에 영구적으로 배치된다. 스케줄러는 항상 가장 높은 우선순위의 큐(Queue 1)부터 검사하며, 상위 큐가 하나라도 비어 있지 않으면 하위 큐(Queue 2, 3)는 절대 CPU를 할당받을 수 없다. 이 방식은 구조가 단순하고 명확하지만, 상위 큐의 작업이 지속적으로 유입될 경우 하위 큐의 프로세스들이 CPU를 전혀 쓰지 못하는 **기아 상태(Starvation)**가 발생하기 쉬운 치명적 약점을 가지고 있다.

#### 핵심 동작 원리
1. **Fixed Assignment**: 프로세스는 한 번 들어간 큐에서 다른 큐로 이동할 수 없다 (No Movement).
2. **Strict Priority Scheduling**: 상위 큐가 하위 큐를 항상 선점한다. 하위 큐 작업 중 상위 큐에 작업이 들어오면 즉시 문맥 교환이 일어난다.
3. **Time Slicing between Queues**: 기아 방지를 위해 큐 간에 실행 시간을 배분(예: 상위 80%, 하위 20%)하기도 하나, 이는 순수 MLQ보다는 변형된 형태다.

#### 📢 섹션 요약 비유
"MLQ는 '철저한 계급 사회'와 같습니다. 상위 계급이 식사를 마치기 전까지 하위 계급은 식탁 근처에도 갈 수 없습니다."

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### MLQ vs MLFQ (Multi-Level Feedback Queue)
| 비교 항목 | MLQ (Multi-Level Queue) | MLFQ (Multi-Level Feedback Queue) |
|:---:|:---|:---|
| **이동 가능성** | 이동 불가 (고정) | 큐 간 이동 가능 (동적) |
| **판단 기준** | 프로세스의 태생적 성격 | 프로세스의 실제 CPU 사용 패턴 |
| **유연성** | 낮음 (설정 기반) | 높음 (적응형) |
| **기아 상태** | 발생 가능성 매우 높음 | 에이징/부스팅으로 해결 가능 |
| **복잡도** | 단순함 | 복잡함 |

#### 과목 융합 관점: 가상화 및 클라우드
- **VM Priority**: 하이퍼바이저 수준에서 VM의 중요도에 따라 큐를 나누어 물리 자원을 할당하는 방식이 MLQ와 유사하다.
- **네트워크 QoS**: 패킷의 TOS(Type of Service) 필드에 따라 음성(VoIP), 영상, 일반 데이터를 각각 다른 큐에 담아 처리하는 DiffServ 모델이 네트워킹에서의 MLQ 구현 사례다.

#### 📢 섹션 요약 비유
"MLQ가 '전공별 분반'이라면, MLFQ는 성적에 따라 수시로 반이 바뀌는 '수준별 이동 수업'입니다."

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 실무 시나리오 및 의사결정
1. **임베디드 제어 시스템 (Predictability)**: 작업의 성격이 명확하고 예측 가능성이 최우선인 실시간 제어기에서는 동적 이동 오버헤드가 없는 MLQ가 선호될 수 있다.
2. **서버 자원 격리 (Isolation)**: 특정 사용자 그룹의 작업이 다른 그룹에 영향을 주지 않도록 물리적 큐를 분리하여 운영할 때 MLQ 개념을 활용한다.
3. **Anti-pattern**: 일반 목적의 PC 운영체제에서 MLQ를 그대로 사용하면, 백그라운드 업데이트 작업이 영원히 끝나지 않아 시스템 무결성에 문제가 생길 수 있다.

#### 도입 체크리스트
- [ ] 프로세스의 성격을 실행 전에 명확히 구분할 수 있는가?
- [ ] 상위 큐의 워크로드가 전체 CPU 자원을 독점하지 않는다는 보장이 있는가?
- [ ] 기아 상태 발생 시 시스템에 치명적인 영향을 주는가?
- [ ] 단순한 구조를 통해 스케줄링 오버헤드를 극소화해야 하는가?

#### 📢 섹션 요약 비유
"MLQ 도입은 '질서'와 '공정' 사이의 아슬아슬한 줄타기와 같습니다. 질서는 잡히지만 공정은 깨지기 쉽습니다."

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량/정성 기대효과 (표)
| 구분 | 도입 전 (단일 RR) | 도입 후 (MLQ) |
|:---:|:---|:---|
| **시스템 반응성** | 보통 | 대화형 작업에 대해 매우 빠름 |
| **스케줄링 오버헤드** | 낮음 | 매우 낮음 (큐 선택만 하면 됨) |
| **자원 효율성** | 모든 작업 균등 소모 | 중요 작업에 자원 집중 |

#### 미래 전망
순수 MLQ는 기아 상태 문제로 인해 현대 범용 OS에서는 거의 사라졌으나, **컨테이너 오케스트라션(Kubernetes)**의 'Priority Class'나 **클라우드 자원 예약 시스템**에서 작업 등급별로 실행 샌드박스를 나누는 방식으로 그 철학이 계승되고 있다. 향후에는 하드웨어 수준에서 큐를 지원하는 가속기 구조에서 다시 조명받을 것이다.

#### 📢 섹션 요약 비유
"MLQ는 비록 융통성은 부족하지만, 복잡한 세상을 단순한 질서로 정리하려 했던 스케줄링의 '강직한 골격'입니다."
