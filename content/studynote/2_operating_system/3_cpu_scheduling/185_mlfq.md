# 185. 다단계 피드백 큐 (MLFQ)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스의 CPU 점유 패턴을 동적으로 관찰하여, CPU 집약적 작업은 우선순위를 낮추고 I/O 집약적 작업은 높게 유지하는 적응형(Adaptive) 스케줄링 기법
> 2. **가치**: 짧은 작업과 대화형 작업에 최상의 응답성을 제공하면서도, 에이징(Aging)과 부스팅(Boosting)을 통해 기아 상태(Starvation)를 원천 차단하는 현대 OS 스케줄러의 표준 모델
> 3. **융합**: 시분할(Time-sharing) 시스템의 공정성, SJF의 효율성, 그리고 우선순위 스케줄링의 유연성이 결합된 알고리즘의 결정체

---

### Ⅰ. 개요 (Context & Background)

#### 개념 정의
**다단계 피드백 큐(MLFQ, Multi-Level Feedback Queue)**는 프로세스가 여러 개의 큐 사이를 동적으로 이동할 수 있게 설계된 스케줄링 알고리즘이다. 프로세스의 과거 실행 이력(History)을 바탕으로 미래의 행동을 예측하여 적절한 우선순위 큐에 배치함으로써 시스템 전체의 응답성과 처리량을 최적화한다.

#### 💡 비유: 수습 기간이 있는 회사
MLFQ는 **신입 사원의 역량에 따라 부서가 바뀌는 회사**와 같다. 모든 신입은 처음엔 기획실(최상위 큐)에서 일을 시작하지만, 업무 처리가 늦어지면(시간 할당량 초과) 현장직(하위 큐)으로 이동한다. 반대로 현장에서 성실히 일하며 대기 시간이 길어지면 다시 관리직으로 승진(Boosting)할 기회를 얻는다.

#### 등장 배경
1. **MLQ의 한계 극복**: 프로세스가 한 큐에 고정되어 발생하는 유연성 부족과 하위 큐의 기아 상태 문제를 해결하고자 함.
2. **미래 예측의 어려움**: SJF처럼 실행 시간을 미리 알 수 없으므로, 실행 중인 행태(Behavior)를 보고 실시간으로 우선순위를 결정할 필요성 대두.
3. **응답 시간과 처리량의 조화**: 대화형 사용자에게는 짧은 응답 시간을, 연산 위주 작업에는 안정적인 CPU 점유를 보장하려는 목적.

#### 📢 섹션 요약 비유
"마치 성적에 따라 반을 옮기는 이동 수업처럼, 프로세스의 '태도'에 따라 대우를 달리하는 가장 스마트한 관리 방식입니다."

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### MLFQ의 5대 규칙 (Corbato's Rules)
MLFQ의 동작을 결정짓는 핵심 메커니즘은 다음과 같은 규칙으로 요약된다.
1. **우선순위 실행**: 우선순위(A) > 우선순위(B)이면, A를 실행한다.
2. **공통 실행**: 우선순위(A) = 우선순위(B)이면, RR(라운드 로빈) 방식으로 실행한다.
3. **최초 진입**: 작업이 시스템에 들어오면 가장 높은 우선순위 큐에 배치한다.
4. **강등 정책**: 할당된 시간 슬라이스를 모두 소모하면 우선순위를 한 단계 낮춘다. (I/O로 조기 반납 시 유지)
5. **승급 정책 (Priority Boost)**: 일정 시간이 지나면 모든 작업을 최상위 큐로 이동시킨다 (기아 방지).

#### 다단계 피드백 큐 동작 프로세스

작업이 시간 소모량에 따라 아래 단계로 강등되는 과정과 주기적 부스팅을 시각화한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│              Multi-Level Feedback Queue (MLFQ) Mechanism            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [New Job Arrival]                                                  │
│          |                                                          │
│  ┌───────v───────┐                                                  │
│  │    Queue 0    │ (High Priority, Small Quantum q=8ms)             │
│  └───────┬───────┘                                                  │
│          | (Timeout)                                                │
│          v                                                          │
│  ┌───────┴───────┐                                                  │
│  │    Queue 1    │ (Medium Priority, Medium Quantum q=16ms)         │
│  └───────┬───────┘                                                  │
│          | (Timeout)                                                │
│          v                                                          │
│  ┌───────┴───────┐                                                  │
│  │    Queue 2    │ (Low Priority, FCFS or Large Quantum)            │
│  └───────────────┘                                                  │
│                                                                     │
│  [Priority Boost!]                                                  │
│  * Every T ms: All jobs in Q1, Q2  ====> Move to Q0                 │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**다이어그램 해설**:
위 ASCII 다이어그램은 MLFQ의 핵심인 '동적 강등'과 '주기적 승급'을 보여준다. 모든 새로운 작업은 우선순위가 가장 높은 `Queue 0`에서 시작하여 짧은 시간 할당량을 부여받는다. 만약 이 시간 내에 작업을 끝내지 못하면 CPU 집약적 작업으로 판단되어 하위 큐로 내려가며, 더 긴 시간 할당량을 받게 된다. 이 과정은 짧은 작업(Short Job)에게는 빠른 응답을, 긴 작업(Long Job)에게는 문맥 교환 횟수를 줄인 효율적 처리를 제공한다. 또한, 맨 아래의 `Priority Boost`는 하위 큐에서 굶고 있는 프로세스들을 주기적으로 구제하여 시스템의 공정성을 유지한다.

#### 핵심 파라미터 (Design Issue)
- **큐의 개수**: 너무 많으면 오버헤드가 크고, 적으면 특성 분리가 어려움 (보통 3~5개).
- **큐별 스케줄링 알고리즘**: 상위는 RR, 최하위는 FCFS 또는 대형 RR 사용.
- **강등/승급 기준**: 시간 할당량의 크기와 부스팅 주기($T$)의 결정.

#### 📢 섹션 요약 비유
"MLFQ는 '한 번의 실수로 영원히 낙오되지 않게' 주기적으로 패자부활전(Boosting)을 열어주는 자비로운 시스템입니다."

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 스케줄링 기법 간 성능 비교
| 비교 항목 | FCFS | Round Robin | MLFQ |
|:---:|:---|:---|:---|
| **응답 시간** | 최악 | 우수 | 최적 (I/O 위주에 유리) |
| **처리량** | 우수 (오버헤드 적음) | 보통 | 우수 (CPU 위주 작업 배려) |
| **공정성** | 낮음 (호위 효과) | 높음 | 높음 (부스팅 기전) |
| **구현 복잡도** | 매우 낮음 | 낮음 | 높음 (파라미터 튜닝 필요) |

#### 과목 융합 관점: 학습 및 예측
- **강화 학습**: MLFQ의 우선순위 조정 로직은 "과거의 보상(CPU 사용량)을 바탕으로 미래의 정책을 결정"한다는 점에서 강화 학습의 기본 아이디어와 닿아 있다.
- **리눅스 O(1) 스케줄러**: 과거 리눅스 커널(2.6)은 MLFQ의 개념을 활용하여 140개의 우선순위 큐를 통해 상호작용성을 극대화했다.

#### 📢 섹션 요약 비유
"MLFQ가 '성실한 모범생(I/O)'에게 장학금을 주고 '벼락치기생(CPU)'에게 긴 자습 시간을 주는 합리적인 교실 모델입니다."

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 실무 시나리오 및 의사결정
1. **대화형 데스크탑 OS (Windows/macOS)**: 사용자의 마우스 클릭이나 키보드 입력에 즉각 반응해야 하므로 MLFQ를 기반으로 전면 윈도우에 우선순위 부스트를 적용한다.
2. **가상화 환경의 CPU 스케줄링**: 여러 게스트 OS가 혼재된 환경에서 특정 VM의 독점을 방지하기 위해 MLFQ 스타일의 쿼터 배분을 사용한다.
3. **Anti-pattern (Game the Scheduler)**: 프로세스가 시간 할당량 종료 직전에 가짜 I/O를 요청하여 최상위 큐에 계속 머무는 '스케줄러 속이기' 공격을 방지하기 위해, 누적 사용 시간을 체크하는 기법을 병행해야 한다.

#### 도입 체크리스트
- [ ] 워크로드의 CPU/IO 바운드 비율이 극명하게 갈리는가?
- [ ] 프로세스의 실행 특성을 사전에 정의하기 어려운가?
- [ ] 기아 상태 방지를 위한 부스팅 오버헤드를 감당할 수 있는가?
- [ ] 큐 간 이동 시 발생하는 문맥 전환 비용이 수용 가능한가?

#### 📢 섹션 요약 비유
"실무에서 MLFQ는 '이론적으로 완벽한' 알고리즘이 아니라, 수많은 시행착오를 통해 다듬어진 '가장 현실적인' 해결책입니다."

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량/정성 기대효과 (표)
| 구분 | 도입 전 (단일 큐 RR) | 도입 후 (MLFQ) |
|:---:|:---|:---|
| **평균 응답 시간** | 100ms | 20ms (짧은 작업 기준) |
| **CPU 이용률** | 85% | 95% (효율적 타임 슬라이싱) |
| **사용자 체감 성능** | 가끔 랙 발생 | 부드러운 상호작용 유지 |

#### 미래 전망
현대 리눅스의 CFS(Completely Fair Scheduler)는 MLFQ의 복잡한 큐 관리 대신 '가상 런타임'과 '레드-블랙 트리'를 통해 더 우아하게 유사한 효과를 내고 있다. 그러나 **특수 목적 RTOS**나 **분산 배치 프레임워크**에서는 여전히 명시적인 다단계 피드백 구조가 자원 할당의 핵심으로 사용되고 있다.

#### 📢 섹션 요약 비유
"비록 모습은 변할지언정, 프로세스의 행동에 따라 미래를 결정한다는 MLFQ의 '피드백 정신'은 모든 미래 기술의 핵심이 될 것입니다."
