# 188. 프로세스 스케줄링 vs 스레드 스케줄링

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스케줄링의 단위를 실행의 큰 틀인 프로세스(Process)로 볼 것인지, 아니면 그 내부의 경량 실행 단위인 스레드(Thread)로 볼 것인지에 대한 아키텍처적 차이
> 2. **가치**: 스레드 수준 스케줄링은 문맥 교환 오버헤드를 줄이고 병렬성을 극대화하며, 사용자 수준(ULT)과 커널 수준(KLT) 스레드에 따라 제어 주체가 달라짐
> 3. **융합**: 멀티코어 프로세서의 자원 할당, LWP(Light Weight Process) 매핑, 현대 OS의 TCB(Thread Control Block) 중심 관리 체계와 연관

---

### Ⅰ. 개요 (Context & Background)

#### 개념 정의
**프로세스 스케줄링 vs 스레드 스케줄링**은 운영체제가 CPU 시간을 할당하는 기본 단위(Schedulable Entity)에 관한 구분이다. 과거 단일 스레드 환경에서는 프로세스가 곧 스케줄링의 단위였으나, 현대의 멀티스레드 환경에서는 프로세스 내의 개별 스레드가 스케줄링의 주체가 된다.

#### 💡 비유: 회사의 프로젝트 관리
**프로세스 스케줄링**은 회사 전체 프로젝트(프로세스)에 기간을 배분하는 것과 같고, **스레드 스케줄링**은 그 프로젝트 안에서 실제 일을 하는 팀원(스레드)들에게 시간을 나눠주는 것과 같다. 프로젝트 단위로만 시간을 주면 팀원 간의 협업이 효율적으로 관리되지 않지만, 팀원 개개인을 관리하면 더 세밀한 작업이 가능하다.

#### 등장 배경
1. **자원 공유의 효율성**: 같은 프로세스 내 스레드들은 메모리(Code, Data, Heap)를 공유하므로, 프로세스 간 전환보다 스레드 간 전환이 훨씬 빠름.
2. **동시성(Concurrency) 증대**: 하나의 프로세스가 입출력(I/O)으로 차단되어도 다른 스레드가 실행을 계속할 수 있게 하기 위함.
3. **멀티코어 활용**: 여러 개의 스레드를 각기 다른 코어에 배치하여 실질적인 병렬 처리를 구현함.

#### 📢 섹션 요약 비유
"프로세스 스케줄링이 '가족 단위'의 식당 예약이라면, 스레드 스케줄링은 '개인별'로 선호하는 메뉴를 주문받는 맞춤 서비스와 같습니다."

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 스케줄링 엔티티 비교 (PCB vs TCB)
스케줄링의 주체에 따라 운영체제가 관리하는 정보의 깊이가 달라진다.

| 비교 항목 | 프로세스 스케줄링 | 스레드 스케줄링 |
|:---:|:---|:---|
| **관리 단위** | PCB (Process Control Block) | TCB (Thread Control Block) |
| **자원 격리** | 완전 격리 (독립된 주소 공간) | 일부 공유 (Stack/Register만 독립) |
| **문맥 교환** | 무거움 (주소 공간 교체 포함) | 가벼움 (레지스터/스택만 교체) |
| **주체** | 운영체제 커널 | 커널 또는 스레드 라이브러리 |

#### 사용자 수준 스레드(ULT) vs 커널 수준 스레드(KLT) 스케줄링

스레드를 관리하는 주체에 따라 스케줄링의 범위가 달라지는 구조를 시각화한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│             Process vs Thread Scheduling Architecture               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [User-Level Threads (ULT)]       [Kernel-Level Threads (KLT)]      │
│  ┌───────────────────────┐         ┌─────────────────────────┐      │
│  │    User Space         │         │      User Space         │      │
│  │ [T1] [T2] [T3]        │         │  [T1]     [T2]    [T3]  │      │
│  │   \   |   /           │         │   |        |       |    │      │
│  │  Library Scheduler    │         │   |        |       |    │      │
│  └───────|───────────────┘         └───|────────|───────|────┘      │
│          v (1:1 or M:1)                v        v       v           │
│  ┌───────────────────────┐         ┌─────────────────────────┐      │
│  │    Kernel Space       │         │      Kernel Space       │      │
│  │    [ Process ]        │         │ [TCB1]  [TCB2]  [TCB3]  │      │
│  │    [   PCB   ]        │         │      [  Kernel  ]       │      │
│  │  Kernel Scheduler     │         │       Scheduler         │      │
│  └───────────────────────┘         └─────────────────────────┘      │
│                                                                     │
│  * ULT: OS sees only one Process. Thread blocking blocks whole Proc.│
│  * KLT: OS sees each Thread. One blocks, others can run.            │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**다이어그램 해설**:
위 ASCII 다이어그램은 스레드 스케줄링의 두 가지 핵심 모델을 보여준다. 왼쪽의 **ULT(User-Level Thread)**는 커널이 프로세스의 존재만 알 뿐 내부 스레드는 모르는 상태로, 스레드 라이브러리가 스케줄링을 담당한다. 이 방식은 빠르지만 하나의 스레드가 시스템 콜을 호출하면 프로세스 전체가 멈춘다. 오른쪽의 **KLT(Kernel-Level Thread)**는 커널이 각 스레드를 직접 인식하고 관리한다. 스케줄러가 TCB 단위로 CPU를 할당하므로, 멀티코어를 온전히 활용할 수 있고 병렬성이 높지만 커널 개입으로 인한 오버헤드가 ULT보다 크다.

#### 심층 동작 원리
1. **PCS (Process-Contention Scope)**: 사용자 수준 스레드 스케줄링에서 발생하며, 프로세스 내에서 스레드들이 CPU를 차지하기 위해 경쟁하는 범위다.
2. **SCS (System-Contention Scope)**: 커널 수준 스레드 스케줄링에서 발생하며, 시스템 내의 모든 스레드가 CPU를 차지하기 위해 전체적으로 경쟁하는 범위다.
3. **LWP (Light Weight Process)**: ULT와 KLT 사이의 중간 단계로, 사용자 스레드를 커널 스레드에 매핑해주는 '가상 프로세서' 역할을 수행한다.

#### 📢 섹션 요약 비유
"ULT가 '집안일은 우리가 알아서 할게요'라고 부모님(커널)께 말하는 것이라면, KLT는 '공부방 청소까지 부모님이 직접 관리해주시는 것'과 같습니다."

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 기술적 정밀 비교 분석
| 항목 | 프로세스 기반 (Old) | 스레드 기반 (Modern) |
|:---:|:---|:---|
| **공유 메모리** | IPC (Shared Mem, Pipe) 필요 | 동일 주소 공간 직접 참조 |
| **확장성** | 낮음 (프로세스 생성 비쌈) | 높음 (스레드 생성 저렴) |
| **안정성** | 높음 (한 프로세스 죽어도 전파 안 됨) | 낮음 (한 스레드 오류가 프로세스 전체 영향) |
| **멀티코어 활용** | 프로세스 단위 분산 | 스레드 단위 정밀 분산 |

#### 과목 융합 관점: 하드웨어 및 데이터베이스
- **하이퍼스레딩 (SMT)**: 하나의 물리 코어에서 두 개의 스레드 스케줄링 엔티티를 동시에 실행할 수 있게 하는 하드웨어적 지원.
- **DBMS 스케줄링**: 데이터베이스 내의 워커(Worker) 스레드들은 락(Lock) 경합과 트랜잭션의 ACID 특성을 고려하여 커널 스케줄링 외에 별도의 사용자 모드 스케줄링을 수행하기도 한다.

#### 📢 섹션 요약 비유
"프로세스 스케줄링이 '거대한 화물선'을 움직이는 일이라면, 스레드 스케줄링은 그 배에 탄 '수많은 보트'들을 조종하는 일과 같습니다."

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 실무 시나리오 및 의사결정
1. **웹 서버 아키텍처 (Apache vs Nginx)**: Apache는 프로세스/스레드 혼합 모델을 사용하고, Nginx는 비동기 이벤트 기반(Single Process/Multiple Worker) 모델을 사용하여 스케줄링 오버헤드를 줄인다.
2. **실시간 데이터 처리 (Low Latency)**: 문맥 교환 지연을 최소화하기 위해 중요한 스레드를 특정 CPU 코어에 고정(CPU Affinity)시키는 전략을 사용한다.
3. **Java 스레드 모델**: 초기 Green Thread(ULT)에서 현대의 Native Thread(KLT)로 진화했으며, 최근에는 Project Loom을 통해 경량 스레드(Virtual Thread)를 도입하여 다시 ULT의 장점을 결합하고 있다.

#### 도입 체크리스트
- [ ] 작업 간의 데이터 공유가 빈번한가? (그렇다면 스레드)
- [ ] 작업 중 하나가 죽어도 시스템이 견고해야 하는가? (그렇다면 프로세스)
- [ ] 문맥 교환 비용이 전체 응답 시간에 치명적인가?
- [ ] 커널이 지원하는 최대 스레드 수의 제한에 걸리는가?

#### 📢 섹션 요약 비유
"실무에서의 선택은 '속도(스레드)'와 '안전(프로세스)' 중 어디에 더 가중치를 둘 것인가의 결정입니다."

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량/정성 기대효과 (표)
| 구분 | 프로세스 스케줄링 중심 | 스레드 스케줄링 중심 (KLT) |
|:---:|:---|:---|
| **문맥 교환 오버헤드** | 높음 (10~100µs) | 낮음 (1~10µs) |
| **병렬 처리 성능** | 낮음 (프로세스 단위) | 높음 (코어별 스레드 배분) |
| **시스템 유연성** | 낮음 | 매우 높음 (동적 생성/삭제 용이) |

#### 미래 전망
향후에는 OS 커널의 개입을 더욱 줄이는 **사용자 수준 스케줄링(User-level Scheduling)** 기술이 발전할 것이다. 구글의 `ghOSt`나 리눅스의 `eBPF`를 활용한 커스터마이징 스케줄러를 통해, 특정 애플리케이션에 최적화된 스레드 스케줄링 정책을 런타임에 주입하는 형태가 표준이 될 전망이다.

#### 📢 섹션 요약 비유
"미래의 스케줄링은 모든 개인(스레드)의 특성을 실시간으로 파악하여 가장 적합한 자리(코어)를 찾아주는 '인공지능 매칭 서비스'가 될 것입니다."

### 👶 어린이를 위한 3줄 비유 설명
**개념**: 프로세스 스케줄링은 "가족끼리 노는 시간"을 정하는 거고, 스레드 스케줄링은 "가족 안에서 누가 먼저 장난감을 가지고 놀지" 정하는 거예요.
**원리**: 가족 전체가 움직이는 건 힘들지만(프로세스), 가족 안에서 순서를 바꾸는 건 아주 빠르고 쉬워요(스레드).
**효과**: 스레드 스케줄링을 잘하면 여러 명이 동시에 다른 장난감을 가지고 놀 수 있어서 훨씬 재미있게 놀 수 있어요.
