# 198. NUMA-인식 스케줄링 (NUMA-Aware Scheduling)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: CPU가 자신과 물리적으로 가까운 로컬 메모리에 우선적으로 접근하도록 프로세스와 메모리를 배치하는 하이브리드 자원 관리 기법
> 2. **가치**: 원격 메모리 접근(Remote Access)으로 인한 지연(Latency)을 최소화하여 멀티프로세서 시스템의 확장성과 성능을 극대화함
> 3. **융합**: 고성능 컴퓨팅(HPC), 대규모 데이터베이스 서버, 가상화 하이퍼바이저의 CPU 피닝(Pinning) 및 메모리 할당 정책의 핵심

---

### Ⅰ. 개요 (Context & Background)

#### 개념 정의
**NUMA-인식 스케줄링(NUMA-Aware Scheduling)**은 비균일 메모리 접근(NUMA, Non-Uniform Memory Access) 아키텍처를 가진 시스템에서, 프로세스가 실행되는 CPU 코어와 해당 프로세스가 사용하는 메모리 영역을 동일한 NUMA 노드(Node)에 유지하려는 스케줄링 전략이다.

#### 💡 비유: 동네 도서관 이용
NUMA-인식 스케줄링은 **내가 사는 동네의 도서관(Local Memory)에서 책을 빌려보는 것**과 같다. 옆 동네 도서관(Remote Memory)에도 책은 있지만, 빌리러 가는 시간(Latency)이 훨씬 오래 걸린다. 스케줄러는 똑똑한 사서처럼 내가 가급적 우리 동네 도서관 근처에서 공부(CPU 실행)할 수 있게 자리를 잡아준다.

#### 등장 배경
1. **SMP의 확장성 한계**: 모든 CPU가 하나의 공유 버스로 메모리에 접근하는 UMA(Uniform Memory Access) 방식은 CPU 수가 늘어날수록 버스 경합으로 인해 성능이 급격히 저하됨.
2. **메모리 계층의 복잡화**: CPU 다이(Die) 내에 메모리 컨트롤러가 통합되면서, 물리적 거리에 따른 접근 속도 차이가 발생함.
3. **대규모 서버 환경**: 수십 개 이상의 코어와 테라바이트급 메모리를 갖춘 최신 서버에서 자원 배치의 효율성이 시스템 전체 성능의 수십 %를 좌우하게 됨.

#### 📢 섹션 요약 비유
"마치 각 주방(CPU) 옆에 전용 식자재 창고(Memory)를 두어, 요리사가 재료를 가지러 멀리 가지 않게 하는 동선 최적화 전략입니다."

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### NUMA 구조 및 접근 비용 (NUMA Distance)
NUMA 시스템에서는 노드 간의 '거리' 개념이 성능의 핵심 지표가 된다.

| 구성 요소 | 역할 | 비유 |
|:---:|:---|:---|
| **NUMA Node** | CPU 코어 그룹 + 로컬 메모리 + I/O 버스의 결합 단위 | 독립된 사무실 |
| **Interconnect** | 노드 간 데이터를 주고받는 고속 통로 (QPI, UPI, Infinity Fabric) | 사무실 간 복도 |
| **Local Access** | 자신이 속한 노드의 메모리에 접근 (낮은 지연, 높은 대역폭) | 내 책상 서랍 열기 |
| **Remote Access** | 다른 노드의 메모리에 접근 (높은 지연, 낮은 대역폭) | 옆 사무실 서랍 열기 |
| **NUMA Distance** | 로컬 대비 원격 접근 비용의 비율 (보통 1.5x ~ 3x 이상) | 이동 거리 비중 |

#### NUMA 노드 아키텍처 및 스케줄링 배치

프로세스가 로컬 노드에 배치되었을 때와 원격 노드에 걸쳐졌을 때의 차이를 시각화한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                 NUMA Architecture & Scheduling                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│      [ NUMA Node 0 ]                     [ NUMA Node 1 ]            │
│  ┌───────────────────────┐           ┌───────────────────────┐      │
│  │   CPU 0-7 (Cores)     │           │   CPU 8-15 (Cores)    │      │
│  │       [ P_a ]         │           │       [ P_b ]         │      │
│  └──────────|────────────┘           └──────────|────────────┘      │
│             | (Local)                           | (Local)           │
│      ┌──────v──────┐                     ┌──────v──────┐            │
│      │   Memory 0  │ <--- Interconnect ->│   Memory 1  │            │
│      │ [Data of P_a] │        (UPI)      │ [Data of P_b] │            │
│      └─────────────┘                     └─────────────┘            │
│             ^                                   |                   │
│             └───────────────────────────────────┘                   │
│                       (Remote Access: BAD!)                         │
│                                                                     │
│  * Goal: Keep P_a on Node 0 and its data in Memory 0.               │
│  * Anti-pattern: P_a runs on Node 1 but its data is in Memory 0.    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**다이어그램 해설**:
위 ASCII 다이어그램은 두 개의 NUMA 노드로 구성된 시스템을 보여준다. 각 노드는 자체 CPU와 메모리를 가지며, 노드 간은 고속 인터커넥트(Intel UPI, AMD Infinity Fabric 등)로 연결된다. 최적의 성능을 위해서는 프로세스 `P_a`가 `Node 0`의 CPU에서 실행되면서 동시에 `Memory 0`에 데이터를 저장해야 한다. 만약 부하 분산(Load Balancing)을 위해 스케줄러가 `P_a`를 `Node 1`로 옮겨버리면, `P_a`는 모든 메모리 접근 시 인터커넥트를 거쳐 `Memory 0`으로 가야 하므로 성능이 급격히 저하된다. 이를 방지하는 것이 **NUMA-인식 스케줄링**의 핵심이다.

#### 심층 동작 원리 (Linux 사례)
1. **First Touch Policy**: 메모리를 처음 사용하는 CPU가 속한 노드에 메모리를 우선 할당한다.
2. **Periodic Scanning**: 커널은 주기적으로 각 프로세스의 메모리 접근 패턴을 감시(Sample)한다.
3. **Task/Memory Migration**: 프로세스가 주로 접근하는 메모리가 다른 노드에 있다면, 프로세스를 해당 노드로 옮기거나(Task Migration) 메모리 페이지를 현재 노드로 복사(Page Migration)한다.
4. **Affinity Scoring**: 각 노드에 대한 프로세스의 '친화도 점수'를 계산하여 스케줄링 우선순위에 반영한다.

#### 📢 섹션 요약 비유
"NUMA 스케줄링은 요리사(CPU)가 자기 주방의 냉장고(Memory)를 쓰게 하고, 만약 다른 주방으로 옮겨가면 재료 박스도 같이 옮겨주는 세심한 관리입니다."

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### NUMA vs UMA 스케줄링 비교
| 비교 항목 | UMA (Uniform Memory Access) | NUMA (Non-Uniform Memory Access) |
|:---:|:---|:---|
| **물리 구조** | 모든 CPU가 하나의 공유 버스 사용 | CPU별 독립 메모리 컨트롤러 + 연결망 |
| **스케줄링 목표** | 단순 부하 분산 (Load Balancing) | 부하 분산 + 데이터 지역성(Locality) 보장 |
| **데이터 이동** | 고려 대상 아님 | 노드 간 이동 비용(Cost) 발생 |
| **적합 환경** | 개인용 PC, 소규모 서버 | 엔터프라이즈 서버, 데이터센터, HPC |

#### 과목 융합 관점: 가상화 및 데이터베이스
- **vNUMA (Virtual NUMA)**: 가상 머신(VM) 내부에서도 물리적 NUMA 구조를 인식할 수 있게 하여, 게스트 OS 내의 스케줄러가 최적의 배치를 할 수 있도록 지원한다.
- **In-Memory DB (Redis, SAP HANA)**: 메모리 접근 속도가 성능의 절대적 요소이므로, NUMA 노드 단위로 인스턴스를 격리하거나 실행 스레드를 고정(Pinning)하여 사용한다.

#### 📢 섹션 요약 비유
"UMA가 '모두가 하나의 큰 식탁에서 먹는 것'이라면, NUMA는 '개인별 작은 식탁들이 연결된 뷔페'와 같습니다."

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 실무 시나리오 및 의사결정
1. **성능 저하 진단 (Troubleshooting)**: 서버의 CPU 점유율은 낮은데 서비스 지연시간이 높다면, `numastat` 명령어를 통해 원격 메모리 접근(Remote Hit) 비율을 확인하고 NUMA 바인딩 정책을 검토한다.
2. **Docker/k8s 자원 할당**: 컨테이너 실행 시 `--cpuset-mems` 옵션을 사용하여 특정 NUMA 노드의 메모리만 사용하도록 강제함으로써 이웃 테넌트에 의한 성능 간섭(Noisy Neighbor)을 방지한다.
3. **BIOS 설정 (Node Interleaving)**: NUMA 인식이 불가능한 구형 OS를 사용할 경우 BIOS에서 NUMA를 끄고(Interleaving) 성능 손해를 감수하며 호환성을 택하는 결정을 내릴 수 있다.

#### 도입 체크리스트
- [ ] 시스템에 2개 이상의 물리 CPU 소켓이 장착되어 있는가?
- [ ] 애플리케이션이 메모리 집약적(Memory-Bound)인가?
- [ ] 프로세스 마이그레이션 오버헤드가 지역성 이득보다 큰가?
- [ ] BIOS 및 OS에서 NUMA 지원 기능이 활성화되어 있는가?

#### 📢 섹션 요약 비유
"실무에서의 NUMA 최적화는 '가장 빠른 길(Local)'을 찾아주는 것을 넘어, '길이 막히면 돌아가는 비용(Migration)'까지 계산하는 정교한 전략입니다."

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량/정성 기대효과 (표)
| 구분 | 도입 전 (UMA 모드) | 도입 후 (NUMA 최적화) |
|:---:|:---|:---|
| **메모리 지연 시간** | 전체 평균치 수렴 | 로컬 접근 시 30~50% 감소 |
| **처리량 (Throughput)** | 병목 발생으로 정체 | 코어 수 비례 선형 증가 |
| **시스템 확장성** | 노드 추가 시 효율 저하 | 대규모 노드 확장 가능 |

#### 미래 전망
향후에는 CXL(Compute Express Link) 기술을 통해 CPU, GPU, 메모리가 거대한 풀(Pool)로 관리되는 **Composable Infrastructure**로 진화할 것이다. 이때 NUMA-인식 스케줄링은 단순 노드 단위를 넘어, '자원 간 거리'와 '대역폭'을 동적으로 계산하여 최적의 실행 위치를 찾아주는 **자원 오케스트레이션**의 형태로 고도화될 전망이다.

#### 📢 섹션 요약 비유
"NUMA 스케줄링은 거대해지는 디지털 제국을 효율적으로 통치하기 위한 '지방 자치 제도'와 같은 필수적인 지혜입니다."
