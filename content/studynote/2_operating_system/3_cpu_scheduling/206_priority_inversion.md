# 206. 우선순위 역전 (Priority Inversion)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 낮은 우선순위가 높은 우선순위의 실행을 방해
> 2. **가치**: 실시간 시스템에서 데드라인 위반 원인
> 3. **융합**: 우선순위 상속, RT Mutex, 동기화와 연관

---

## Ⅰ. 개요

### 개념 정의

우선순위 역전(Priority Inversion)은 **높은 우선순위 태스크가 낮은 우선순위 태스크에 의해 간접적으로 블록되는 현상**이다. 공유 자원 보호를 위한 동기화로 발생한다.

### 💡 비유: VIP 대기
우선순위 역전은 **VIP가 일반 손님 때문에 기다리는 상황**과 같다. VIP(높은 우선순위)가 화장실(자원)을 쓰려는데, 일반 손님(낮은 우선순위)이 들어갔고, 중간 직원이 일반 손님을 계속 방해한다.

### 우선순위 역전 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                우선순위 역전 구조                                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【발생 시나리오】                                                        │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  시간 ───────────────────────────────────────────────────►    │ │
│  │                                                             │ │
│  │  Low (L):    ┌──락 획득──┐              ┌─락 해제─┐          │ │
│  │              │          │              │         │          │ │
│  │  ────────────┤          ├──────────────┤         ├──────────│ │
│  │                                                             │ │
│  │  Medium (M):        ┌────실행중────┐                          │ │
│  │               ──────┤              ├────────────────────────│ │
│  │                      ↑ M이 L 선점    ↑                         │ │
│  │                                                             │ │
│  │  High (H):                ↑ 락 대기                           │ │
│  │               ────────────┤ 블록됨        ├──실행─┤            │ │
│  │                           ↑             ↑                      │ │
│  │                           │             │                      │ │
│  │                           └─────────────┘                      │ │
│  │                           H이 L을 기다림                         │ │
│  │                           (M 때문에 L 실행 못 함)                │ │
│  │                                                             │ │
│  │  결과: H(높음) > M(중간) > L(낮음)인데                         │ │
│  │        M 때문에 H가 L을 기다림 = 우선순위 역전                    │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【문제점】                                                               │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  • 높은 우선순위 태스크의 응답 시간 증가                            │ │
│  │  • 데드라인 위반 가능                                              │ │
│  │  • 시스템 응답성 저하                                              │ │
│  │  • 예측 불가능한 동작                                               │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【실제 사례: Mars Pathfinder (1997)】                                     │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  • 화성 탐사선에서 우선순위 역전 발생                               │ │
│  │  • 고우선순위 태스크가 지연되어 시스템 리셋                         │ │
│  │  • 우선순위 상속으로 해결                                          │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 상세 분석

### 상세 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                우선순위 역전 상세                                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【해결 방법 1: 우선순위 상속 (Priority Inheritance)】                     │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  원리:                                                           │ │
│  │  • 락 보유 중인 L이 락 대기 중인 H의 우선순위로 승격                │ │
│  │  • L이 빨리 완료 → 락 해제                                        │ │
│  │  • H이 락 획득 → 실행                                             │ │
│  │                                                             │ │
│  │  장점:                                                           │ │
│  │  • 애플리케이션 수정 불필요                                        │ │
│  │  • 커널/라이브러리 레벨에서 처리                                   │ │
│  │                                                             │ │
│  │  단점:                                                           │ │
│  │  • 체인 역전 가능 (여러 태스크 관련 시)                            │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【해결 방법 2: 우선순위 천장 (Priority Ceiling)】                          │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  원리:                                                           │ │
│  │  • 각 락에 "천장 우선순위" 지정                                    │ │
│  │  • 락 획득 시 태스크가 천장 우선순위로 승격                          │ │
│  │  • 락 해제 시 원래 우선순위로 복귀                                  │ │
│  │                                                             │ │
│  │  장점:                                                           │ │
│  │  • 데드락 방지                                                    │ │
│  │  • 블로킹 시간 최소화                                              │ │
│  │                                                             │ │
│  │  단점:                                                           │ │
│  │  • 천장 우선순위 사전 지정 필요                                    │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【상속 vs 천장 비교】                                                    │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  구분              상속                천장                     │ │
│  │  ────                    ────                    ────            │ │
│  │  승격 시점          락 대기 발생 시        락 획득 시               │ │
│  │  승격 대상          대기 중인 태스크      사전 지정값               │ │
│  │  구현 난이도        낮음                  중간                     │ │
│  │  데드락 방지        아니오                예                       │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실무 적용

### 구현 방법
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【Linux RT Mutex (우선순위 상속)】                                       │
│  ──────────────────                                                  │
│  #include <pthread.h>                                                │
│                                                                     │
│  // 우선순위 상속 mutex 초기화                                          │
│  pthread_mutex_t mutex;                                              │
│  pthread_mutexattr_t attr;                                           │
│                                                                     │
│  pthread_mutexattr_init(&attr);                                      │
│  pthread_mutexattr_setprotocol(&attr, PTHREAD_PRIO_INHERIT);         │
│  pthread_mutex_init(&mutex, &attr);                                  │
│                                                                     │
│  // 실시간 우선순위 설정                                                │
│  struct sched_param param;                                           │
│  param.sched_priority = 80;                                          │
│  pthread_setschedparam(pthread_self(), SCHED_FIFO, &param);          │
│                                                                     │
│  // mutex 사용                                                        │
│  pthread_mutex_lock(&mutex);                                         │
│  // 임계 영역                                                          │
│  pthread_mutex_unlock(&mutex);                                       │
│                                                                     │
│  【우선순위 천장 구현】                                                  │
│  ──────────────────                                                  │
│  #include <pthread.h>                                                │
│  #include <sched.h>                                                  │
│                                                                     │
│  // 우선순위 천장 mutex 초기화                                          │
│  pthread_mutexattr_setprotocol(&attr, PTHREAD_PRIO_PROTECT);         │
│  pthread_mutexattr_setprioceiling(&attr, 90);  // 천장 = 90           │
│  pthread_mutex_init(&mutex, &attr);                                  │
│                                                                     │
│  【현재 우선순위 확인】                                                  │
│  ──────────────────                                                  │
│  int get_current_priority() {                                        │
│      struct sched_param param;                                       │
│      int policy;                                                     │
│      pthread_getschedparam(pthread_self(), &policy, &param);         │
│      return param.sched_priority;                                    │
│  }                                                                   │
│                                                                     │
│  【문제 진단】                                                            │
│  ──────────────────                                                  │
│  # 우선순위 역전 의심 시                                                 │
│  $ trace-cmd record -e sched:sched_switch -e sched:sched_wakeup \   │
│      ./realtime_app                                                  │
│  $ trace-cmd report                                                  │
│                                                                     │
│  # 락 대기 시간 분석                                                    │
│  $ perf lock record ./realtime_app                                   │
│  $ perf lock report                                                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 기대효과 및 결론

### 핵심 요약

```
• 개념: 높은 우선순위가 낮은 것에 의해 블록됨
• 원인: 중간 우선순위가 낮은 것 선점
• 문제: 응답 시간 증가, 데드라인 위반
• 해결 1: 우선순위 상속 (PI)
• 해결 2: 우선순위 천장 (PC)
• 구현: RT Mutex (PTHREAD_PRIO_INHERIT)
```
