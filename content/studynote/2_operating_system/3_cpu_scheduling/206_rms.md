# 206. 비율 주기 스케줄링 (Rate Monotonic Scheduling, RMS)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 주기가 짧은 태스크(빈도가 높은 태스크)에 높은 우선순위를 부여하는 정적(Static) 우선순위 기반 선점형 실시간 스케줄링 알고리즘
> 2. **가치**: 구현이 단순하고 결정론적(Deterministic)이며, CPU 이용률이 약 69%($n \ln 2$) 이하일 경우 모든 태스크의 데드라인 준수를 수학적으로 보장함
> 3. **융합**: 임베디드 시스템, 항공우주 제어, 자동차 ECU 통신 등 하드 실시간(Hard Real-time) 환경의 가장 신뢰받는 표준 모델

---

### Ⅰ. 개요 (Context & Background)

#### 개념 정의
**비율 주기 스케줄링(RMS, Rate Monotonic Scheduling)**은 주기적인 태스크들을 처리하기 위한 실시간 스케줄링 기법으로, 각 태스크의 수행 주기(Period)의 역수인 **비율(Rate)**에 비례하여 우선순위를 할당한다. 즉, 주기가 짧을수록 비율이 높으므로 더 높은 우선순위를 갖게 된다.

#### 💡 비유: 심장 박동과 호흡 제어
RMS는 우리 몸의 **자율 신경계**와 같다. 심장 박동(주기 매우 짧음)은 호흡(주기 중간)이나 소화(주기 김)보다 더 자주, 그리고 즉각적으로 처리되어야 생명이 유지된다. 주기가 짧은 필수 기능에 최우선권을 주는 것이 RMS의 철학이다.

#### 등장 배경
1. **실시간성의 보장**: 단순한 RR이나 우선순위 방식으로는 태스크의 데드라인(Deadline) 준수를 엄격히 증명하기 어려움.
2. **정적 스케줄링의 안정성**: 실행 중에 우선순위가 변하는 동적 방식(EDF 등)보다 예측 가능성이 높고 오버헤드가 적은 방식이 필요함.
3. **Liu & Layland의 증명 (1973)**: 정적 우선순위 방식 중 RMS가 최적(Optimal)임을 수학적으로 입증함.

#### 📢 섹션 요약 비유
"마치 알람 시계 여러 개가 울릴 때, 가장 자주 울리는 시계부터 먼저 끄는 단순하고 명확한 규칙입니다."

---

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### RMS의 핵심 규칙 및 조건
RMS가 성공적으로 동작하기 위한 전제 조건과 성능 한계는 다음과 같다.

| 항목 | 상세 내용 | 비유 |
|:---:|:---|:---|
| **우선순위 할당** | $P_i \propto 1/T_i$ (주기 $T$가 작을수록 우선순위 $P$ 높음) | 빈도수 비례 배분 |
| **데드라인 조건** | 각 태스크의 데드라인은 해당 주기와 동일함 ($D_i = T_i$) | 제시간에 끝내기 |
| **선점 가능성** | 높은 우선순위 태스크가 도착하면 현재 태스크를 즉시 선점함 | 긴급 상황 우선 |
| **이용률 상한** | $U = \sum (C_i / T_i) \leq n(2^{1/n} - 1)$ | 안전 여유분 확보 |

#### RMS 스케줄링 타임라인 시각화

두 개의 태스크(T1: 주기 짧음, T2: 주기 김)가 경쟁할 때의 선점 과정을 시각화한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                 RMS Scheduling Timeline Example                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [Task Set]                                                         │
│  - T1: Execution(C1)=2, Period(T1)=5  (High Priority)               │
│  - T2: Execution(C2)=4, Period(T2)=10 (Low Priority)                │
│                                                                     │
│  Timeline:                                                          │
│  0     2     4     5     7     9    10                              │
│  ┌─────┬─────┬─────┬─────┬─────┬─────┐                              │
│  │ T1  │ T2  │(T2) │ T1  │ T2  │(T2) │ ...                            │
│  └─────┴─────┴─────┴─────┴─────┴─────┘                              │
│  ^           ^     ^     ^     ^     ^                              │
│  Start      Preempt|     |     |     |                              │
│             T1 ends|     T1 starts   T2 ends                        │
│                                                                     │
│  * T2는 T1이 실행되지 않는 사이사이에 나누어서 실행됨.              │
│  * T1이 다시 돌아오면 T2는 즉시 중단(Preempted)됨.                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**다이어그램 해설**:
위 ASCII 다이어그램은 RMS의 전형적인 동작을 보여준다. 주기가 5인 T1은 주기가 10인 T2보다 높은 우선순위를 갖는다. 시점 0에서 두 태스크가 동시에 도착하면 T1이 먼저 실행된다. T1이 끝난 후 T2가 실행되다가, 시점 5에서 다시 T1의 주기가 돌아오면 T2는 실행을 멈추고 CPU를 T1에게 양보(선점)한다. T1의 두 번째 실행이 끝난 후에야 T2는 남은 작업을 마무리한다. 이러한 엄격한 규칙을 통해 주기가 짧은 태스크의 실시간성을 최우선으로 보장한다.

#### 이용률 경계값 (Utilization Bound)
RMS의 가장 중요한 특징은 태스크 수($n$)에 따른 CPU 이용률 상한선이다.
- $n=1$: 100%
- $n=2$: 82.8%
- $n \to \infty$: **약 69.3% ($ln 2$)**
- **결론**: 전체 CPU 이용률이 69% 이하이면 어떤 주기의 태스크 세트라도 데드라인 준수가 보장된다. (69% ~ 100% 사이는 보장할 수 없으나 실행 가능할 수도 있음)

#### 📢 섹션 요약 비유
"RMS는 '안전율을 고려한 설계'와 같습니다. CPU의 30% 정도를 비워둠으로써 예기치 못한 지연에도 시스템이 무너지지 않게 합니다."

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### RMS vs EDF (Earliest Deadline First)
| 비교 항목 | RMS (Rate Monotonic) | EDF (Earliest Deadline First) |
|:---:|:---|:---|
| **우선순위 결정** | 정적 (주기 기반 고정) | 동적 (남은 데드라인 기반 변화) |
| **이용률 상한** | 약 69% (보수적) | 100% (공격적) |
| **오버헤드** | 매우 낮음 | 높음 (매 시점 데드라인 계산) |
| **안정성** | 과부하 시에도 상위 태스크 보호 | 과부하 시 도미노 붕괴 위험 |
| **최적성** | 정적 방식 중 최적 | 모든 방식 중 최적 |

#### 과목 융합 관점: 통신 프로토콜 및 신호 처리
- **CAN Bus 스케줄링**: 자동차 내부 통신망인 CAN(Controller Area Network) 프로토콜은 메시지 ID의 우선순위를 RMS 원리에 따라 할당하여 실시간 제어 데이터의 전송을 보장한다.
- **디지털 신호 처리 (DSP)**: 샘플링 주기가 짧은 센서 데이터 처리에 RMS를 적용하여 데이터 유실 없는 필터링과 제어를 수행한다.

#### 📢 섹션 요약 비유
"RMS가 '정해진 시간표대로 움직이는 기차'라면, EDF는 '가장 급한 환자부터 태우는 앰뷸런스'와 같습니다."

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 실무 시나리오 및 의사결정
1. **항공기 비행 제어 (Hard RT)**: 자세 제어(High Rate)와 연료 모니터링(Low Rate) 태스크가 섞여 있을 때, 예측 가능성이 가장 높은 RMS를 사용하여 안전 인증(DO-178C 등)을 획득한다.
2. **멀티미디어 스트리밍 (Soft RT)**: CPU 이용률을 100% 가까이 활용해야 하는 경우에는 RMS 대신 EDF를 검토하지만, 시스템 오동작 시 파급력을 고려하여 RMS를 기반으로 하되 중요도가 낮은 태스크를 탈락시키는 전략을 병행한다.
3. **태스크 세트 재설계**: 만약 이용률이 70%를 넘어 데드라인 미스가 발생한다면, 주기를 조정하거나(Period Tuning) 하드웨어 성능을 올리는 대신 알고리즘의 결정론적 특성을 유지하는 방향으로 의사결정한다.

#### 도입 체크리스트
- [ ] 태스크가 주기적(Periodic)으로 발생하는가?
- [ ] 모든 태스크의 실행 시간(Worst Case Execution Time)이 추정 가능한가?
- [ ] CPU 이용률 합계가 69% 임계치 근처인가?
- [ ] 데드라인 미스가 발생했을 때 시스템 전체가 파손되는가? (Hard vs Soft)

#### 📢 섹션 요약 비유
"실무에서 RMS를 선택한다는 것은 '최고의 효율'보다 '절대적인 안전'을 선택한다는 선언과 같습니다."

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 정량/정성 기대효과 (표)
| 구분 | 도입 전 (일반 우선순위) | 도입 후 (RMS) |
|:---:|:---|:---|
| **데드라인 준수 보장** | 운에 맡김 | 수학적 증명 가능 |
| **예측 가능성** | 낮음 (지터 발생) | 높음 (결정론적 지연) |
| **시스템 복구력** | 과부하 시 전체 마비 | 하위 태스크만 영향 (우아한 성능 저하) |

#### 미래 전망
자율주행 자동차와 스마트 팩토리의 확산으로 실시간 성능은 더욱 중요해지고 있다. 현대의 **멀티코어 RMS**는 코어 간 태스크 분배 문제(Bin-packing)와 결합되어 진화하고 있으며, 가상화 환경에서의 실시간성 보장을 위한 **Partitioned RMS** 기법 등이 표준으로 자리 잡고 있다.

#### 📢 섹션 요약 비유
"RMS는 시대를 불변하는 실시간 시스템의 '헌법'이며, 가장 단순한 규칙이 가장 강력한 안전을 만든다는 진리를 보여줍니다."
