# 210. 리눅스 스케줄링 - CFS (Completely Fair Scheduler)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: CFS (Completely Fair Scheduler)는 실행 가능한 모든 프로세스에 CPU 시간을 공평하게 배분하기 위해 `vruntime (Virtual Runtime)`과 Red-Black Tree 자료구조를 사용하는 리눅스 커널 2.6.23 이후의 표준 스케줄러다.
> 2. **가치**: 기존 O(1) 스케줄러의 복잡한 상호작용성 (Interactivity) 휴리스틱을 제거하고, 수학적으로 정교한 비례 지분 배분을 통해 데스크탑 응답성과 서버 처리량 (Throughput)의 균형을 혁신적으로 달성했다.
> 3. **융합**: 가상화 환경의 스틸 타임 (Steal Time) 관리, Cgroups (Control Groups)를 통한 자원 격리, 그리고 최신 EEVDF (Earliest Eligible Virtual Deadline First) 알고리즘으로의 진화적 토대를 제공한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: **CFS (Completely Fair Scheduler)**는 "이상적인 멀티태스킹 CPU (Ideal Multi-tasking CPU)"를 소프트웨어로 구현한 스케줄러다. 이상적인 CPU라면 $N$개의 프로세스가 있을 때 각 프로세스에게 $1/N$의 속도를 제공해야 하지만, 실제 하드웨어는 한 번에 하나의 프로세스만 실행할 수 있다. CFS는 각 프로세스가 소비한 실행 시간을 가상 실행 시간인 `vruntime (Virtual Runtime)`으로 관리하여, 이 값이 가장 적은 프로세스를 우선적으로 선택함으로써 모든 프로세스가 공평한 CPU 점유를 누리게 한다.

- **💡 비유**: CFS는 **"모두에게 공평한 뷔페 식당"**과 같다. 식당 주인 (스케줄러)은 손님들 (프로세스)이 지금까지 얼마나 먹었는지 (vruntime)를 기록한다. 만약 어떤 손님이 너무 적게 먹었다면, 그 손님을 가장 먼저 식사 자리 (CPU)로 안내한다. VIP 손님 (높은 우선순위)은 일반 손님보다 시간이 천천히 흐르는 시계를 가지고 있어, 같은 시간을 먹어도 장부에는 적게 기록되어 더 자주 먹을 수 있게 된다.

- **등장 배경**:
  1. **O(1) 스케줄러의 한계**: 이전의 O(1) 스케줄러는 대화형 프로세스를 판별하기 위해 복잡한 수식과 휴리스틱 (Heuristics)을 사용했으나, 이는 특정 상황에서 예측 불가능한 지연을 초래했다.
  2. **공정성 (Fairness)의 부재**: 프로세스 간의 CPU 점유율이 수학적으로 정확히 배분되지 않아, nice 값에 따른 시간 할당이 상대적이지 않고 절대적으로 고정되는 문제가 있었다.
  3. **데스크탑 사용자 경험 개선**: 서버 성능뿐 아니라 마우스 움직임, 오디오 재생 등 실시간성에 민감한 데스크탑 환경의 응답성을 높이기 위한 설계가 요구되었다.

- **📢 섹션 요약 비유**: "복잡한 규칙으로 우선순위를 매기던 방식에서, '누가 덜 혜택을 받았는가'라는 단 하나의 공정한 잣대로 전환한 패러다임의 변화입니다."

|:---|:---|:---|:---|:---|
| **vruntime (Virtual Runtime)** | 프로세스의 CPU 점유 이력 관리 | 실제 실행 시간에 가중치 (Weight)를 곱해 계산 | 정밀 시간 타임스탬프 | 먹은 양을 기록한 장부 |

| **Load Weight** | nice 값에 따른 가중치 배분 | nice 0을 기준으로 1.25배씩 차이가 나도록 설계 | 우선순위-가중치 매핑 테이블 | VIP 등급별 할인율 |

### Red-Black Tree 기반의 스케줄링 메커니즘

CFS는 실행 가능한 (Runnable) 프로세스들을 `vruntime` 순서대로 Red-Black Tree에 저장한다. 트리의 가장 왼쪽 노드 (Leftmost Node)가 항상 `vruntime`이 가장 작은 프로세스이며, 스케줄러는 이 노드를 선택하여 CPU를 할당한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────┐
 │                CFS Red-Black Tree & Task Selection                      │
 ├─────────────────────────────────────────────────────────────────────────┤
 │                                                                         │
 │      [vruntime=105] (Root)                                              │
 │           /         \                                                   │
 │    [vruntime=90]   [vruntime=120]                                       │
 │       /      \           \                                              │
 │ [vruntime=85] [vruntime=95] [vruntime=130]                              │
 │      ↑                                                                  │
 │   Leftmost Node (가장 공평하지 못한 대우를 받은 프로세스)                    │
 │                                                                         │
 │  1. pick_next_task: 트리의 최좌측 노드 선택 (P_85)                         │
 │  2. Task 실행: 실제 시간 t만큼 실행 → vruntime 갱신                       │
 │     vruntime += (delta_exec * NICE_0_LOAD / weight)                     │
 │  3. Re-insert: 갱신된 vruntime으로 트리에 재삽입 (예: 85 -> 110)            │
 │  4. Re-balance: 트리가 스스로 균형을 맞춤 (O(log N))                      │
 │                                                                         │
 └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 위 ASCII 구조는 CFS의 핵심 자료구조인 Red-Black Tree를 보여준다. 각 노드는 `sched_entity`를 나타내며, `vruntime` 값이 작을수록 왼쪽에 배치된다. 스케줄러는 항상 **최좌측 노드 (Leftmost Node)**를 캐싱해 두었다가 즉시 꺼내어 실행하므로 선택 과정은 $O(1)$에 가깝지만, 실행 후 다시 삽입할 때는 트리의 높이인 $O(\log N)$ 시간이 소요된다. 이는 수만 개의 프로세스가 있어도 안정적인 성능을 보장하는 비결이다. 특히 `vruntime` 계산 시 분모에 `weight`가 들어가므로, 우선순위가 높은 (weight가 큰) 프로세스는 `vruntime` 증가폭이 작아져 트리 왼쪽 영역에 더 오래 머물게 된다.

---

### 심층 동작 원리: vruntime과 가중치 계산

CFS의 핵심은 실제 실행 시간 (Physical Runtime)을 가상 실행 시간 (Virtual Runtime)으로 변환하는 수식에 있다. nice 값 1 단계 차이는 CPU 점유율 약 10% 차이 (가중치 1.25배)로 설계되어 있다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │              vruntime 업데이트 공식 및 예시                     │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │  Δvruntime = Δactual_time × (NICE_0_LOAD / task_weight)       │
  │                                                               │
  │  [케이스 연구]                                                 │
  │  - 프로세스 A (nice  0): weight = 1024                         │
  │  - 프로세스 B (nice 10): weight = 110                          │
  │                                                               │
  │  만약 두 프로세스가 실제 10ms씩 똑같이 실행되었다면:             │
  │                                                               │
  │  1. 프로세스 A: 10ms × (1024 / 1024) = 10ms 증가               │
  │  2. 프로세스 B: 10ms × (1024 / 110)  ≈ 93ms 증가               │
  │                                                               │
  │  결과: B는 A보다 vruntime이 훨씬 빨리 늘어나 트리 오른쪽으로       │
  │  급격히 밀려나며, 결과적으로 CPU를 적게 할당받게 됨.              │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 수식은 CFS가 어떻게 '공정성'과 '우선순위'를 동시에 달성하는지 보여준다. `NICE_0_LOAD`는 기준값인 1024이다. nice 값이 큰 (우선순위가 낮은) 프로세스는 분모인 `task_weight`가 작아져서, 같은 시간을 실행해도 `vruntime`이 매우 크게 증가한다. 반대로 nice가 낮은 (VIP) 프로세스는 `vruntime`이 느리게 증가하여 트리 왼쪽 (우선순위 구역)에 더 자주 출몰하게 된다. 이 메커니즘 덕분에 CFS는 별도의 복잡한 규칙 없이도 수학적 비례에 따라 CPU 지분을 정확히 나누어 줄 수 있다.

① 타임 인터럽트 발생 → ② 현재 실행 중인 태스크의 `vruntime` 갱신 → ③ 트리 내의 다른 노드들과 `vruntime` 비교 → ④ 만약 최좌측 노드와 현재 노드의 차이가 `sched_wakeup_granularity_ns`보다 크면 선점 (Preemption) 발생 → ⑤ 문맥 교환 (Context Switch).

- **📢 섹션 요약 비유**: "체중계의 눈금을 조절하여, 귀한 손님은 무거운 음식을 먹어도 가볍게 기록되게 함으로써 더 많이 먹을 기회를 주는 정교한 저울질과 같습니다."

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: O(1) 스케줄러 vs CFS

| 비교 항목 | O(1) 스케줄러 (Linux 2.6.0) | CFS (Linux 2.6.23+) |
|:---|:---|:---|
| **자료구조** | Priority Array (Bitmap + List) | Red-Black Tree (Self-balancing) |
| **시간 복잡도** | $O(1)$ (상수 시간) | $O(\log N)$ (로그 시간) |
| **우선순위 정책** | 정적 타임 슬라이스 + 휴리스틱 보너스 | vruntime 기반 비례 배분 (Fair Share) |
| **상호작용성 판단** | Sleep/Run 비율 측정 (복잡함) | 자동으로 vruntime에 반영됨 (단순함) |
| **공정성 수준** | 낮음 (낮은 nice에서 기아 위험) | 매우 높음 (수학적 보장) |

O(1) 스케줄러는 속도($O(1)$)에 집착하여 설계되었으나, 프로세스가 많아질수록 휴리스틱이 꼬이면서 대화형 응답성이 깨지는 문제가 있었다. CFS는 비록 로그 시간($O(\log N)$)을 소요하지만, 현대 CPU의 성능 향상으로 인해 이 비용은 미미해졌으며, 대신 얻는 공정성과 예측 가능성의 가치가 훨씬 크다.

### 과목 융합 관점

- **가상화 (Cloud)**: 하이퍼바이저가 실제 CPU를 뺏어가는 스틸 타임 (Steal Time)이 발생하면, CFS는 이를 인지하여 프로세스의 `vruntime`을 조정함으로써 가상 머신 내부의 시계 왜곡을 방지한다.
- **컨테이너 (Docker/K8s)**: Cgroups (Control Groups)의 `cpu.shares` 값은 CFS의 `weight`에 직접 매핑된다. 이를 통해 컨테이너 간의 자원 격리를 물리적 시간 단위가 아닌 상대적 가중치 단위로 정교하게 관리한다.

- **📢 섹션 요약 비유**: "단순히 빨리 결정하는 것보다, 조금 더 고민하더라도 뒤처지는 사람 없이 모두가 만족하는 배분 방식을 택한 성숙한 사회의 모습과 같습니다."

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 웹 서버 응답 지연 최적화**: 다량의 요청을 처리하는 Nginx 서버에서 특정 워커 (Worker)가 CPU를 독점하여 응답 시간이 튀는 현상 발생. 아키텍트는 `kernel.sched_latency_ns`와 `kernel.sched_min_granularity_ns`를 조정하여 타임 슬라이스를 더 잘게 쪼개 응답성 (Responsiveness)을 높이거나, 처리량 (Throughput)을 위해 길게 설정하는 의사결정을 내려야 한다.

2. **시나리오 — 오디오/비디오 인코딩 워크로드**: CPU 집약적인 인코딩 작업이 GUI 환경의 마우스 움직임을 방해하는 상황. nice 값을 19로 설정하여 인코딩 프로세스의 `vruntime` 증가율을 최대화함으로써, 마우스 인터럽트가 발생했을 때 GUI 프로세스가 즉시 트리 왼쪽으로 올라와 CPU를 선점할 수 있도록 유도한다.

### 도입 체크리스트 (커널 파라미터 튜닝)
- **sched_latency_ns**: 모든 실행 가능한 태스크가 한 번씩은 실행되어야 하는 목표 주기 (기본값 6ms~24ms).
- **sched_min_granularity_ns**: 한 프로세스가 최소한 보장받는 실행 시간. 너무 작으면 문맥 교환 오버헤드가 커진다.
- **sched_wakeup_granularity_ns**: 선점이 일어나기 위한 `vruntime` 차이의 최소 임계값.

### 안티패턴
- **과도한 nice -20 남발**: 모든 중요 프로세스를 -20으로 설정하면 결국 CFS는 이들 간에 다시 $1/N$ 공평 배분을 시작하므로 우선순위 효과가 사라진다. 상대적 차이가 핵심이다.
- **실시간 (Real-time) 태스크와의 혼동**: CFS는 '공정'한 스케줄러이지 '데드라인'을 보장하는 스케줄러가 아니다. 엄격한 시간 보장이 필요하다면 `SCHED_FIFO`나 `SCHED_DEADLINE` 클래스를 사용해야 한다.

- **📢 섹션 요약 비유**: "정교한 악기 (CFS)를 제대로 연주하기 위해서는, 연주 홀의 특성 (워크로드)에 맞춰 현의 긴장도 (커널 파라미터)를 세밀하게 조율하는 과정이 필수적입니다."

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 (O(1)) | 도입 후 (CFS) | 개선 효과 |
|:---|:---|:---|:---|
| **응답성** | 휴리스틱 실패 시 먹통 현상 | vruntime 기반 즉시 선점 | 지연 (Latency) 안정화 |
| **공정성** | nice 값에 따른 배분 불명확 | 수학적 비례 배분 준수 | 자원 예측 가능성 100% |
| **복잡도** | 수천 줄의 휴리스틱 코드 | 단순한 트리 및 수식 구조 | 커널 유지보수성 향상 |

### 미래 전망
- **EEVDF (Earliest Eligible Virtual Deadline First)**: 최근 리눅스 커널 (6.6+)은 CFS를 넘어서 EEVDF로의 전환을 시작했다. 이는 공정성뿐만 아니라 지연 시간 (Latency) 자체를 명시적인 데드라인으로 관리하여, 대화형 작업의 응답성을 한 차원 더 높이려는 시도다.
- **Heterogeneous Architecture (big.LITTLE)**: 서로 성능이 다른 CPU 코어들이 섞인 환경에서, CFS는 각 코어의 성능 계수 (Capacity)를 가중치에 반영하여 진정한 공정성을 실현하는 방향으로 진화하고 있다.

### 참고 표준
- **POSIX.1b**: 실시간 확장 표준 (CFS는 이 중 비실시간 영역의 표준 구현체 역할)
- **LSB (Linux Standard Base)**: 리눅스 커널의 동작 일관성 표준

- **📢 섹션 요약 비유**: "모두에게 공평한 기회를 주던 시대를 지나, 이제는 개개인의 사정 (데드라인)까지 고려하는 맞춤형 복지 (EEVDF)의 시대로 나아가고 있습니다."

## 👶 어린이를 위한 3줄 비유 설명
1. CFS는 장난감을 가지고 노는 **공평한 규칙**이에요. 친구들이 장난감을 얼마나 오래 가지고 놀았는지 매일 장부에 기록해요.
2. 장부에서 가장 적게 논 친구가 누구인지 찾아서 그 친구에게 장난감을 먼저 줘요. 아주 조금만 논 친구는 **순서가 빨리 돌아온답니다.**
3. 힘이 세거나 특별한 친구 (높은 우선순위)는 시계가 천천히 가서, 남들보다 더 오래 놀아도 장부에는 조금만 논 것처럼 적혀서 더 자주 놀 수 있어요!
