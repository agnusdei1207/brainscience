# 220. 윈도우 스케줄링 아키텍처 (Windows Scheduling Architecture)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: Windows 스케줄링은 우선순위 기반의 선점형 (Preemptive), 다단계 피드백 큐 (MLFQ, Multi-level Feedback Queue) 아키텍처를 사용하여 스레드 단위로 CPU 자원을 할당한다.
> 2. **가치**: 사용자 인터페이스 (UI) 응답성을 극대화하기 위해 포그라운드 프로세스 우선권 부여, I/O 완료 시 동적 우선순위 부스트 (Priority Boost), 퀀텀 (Quantum) 연장 등 실무 최적화 기법을 제공한다.
> 3. **융합**: 멀티코어 환경의 프로세서 선호도 (Processor Affinity), NUMA (Non-Uniform Memory Access) 최적화, 그리고 클라우드 가상화 환경의 하이퍼바이저 스케줄링과 연계된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: **Windows 스케줄링 아키텍처**는 시스템의 모든 실행 가능한 스레드 중 가장 높은 우선순위를 가진 스레드를 다음에 실행할 대상으로 선택하는 **우선순위 기반 선점형 스케줄링 (Priority-driven Preemptive Scheduling)** 모델이다. Windows 커널은 프로세스가 아닌 스레드 (Thread)를 스케줄링의 최소 단위로 취급하며, 실시간 (0-31) 우선순위 체계를 통해 시스템 안정성과 응답성을 동시에 확보한다.

- **💡 비유**: Windows 스케줄링은 **"응급실의 트리아지 (Triage) 시스템"**과 같다. 환자 (스레드)들은 도착한 순서대로 진료받는 것이 아니라, 위중도 (우선순위)에 따라 순서가 결정된다. 만약 더 위급한 환자가 들어오면 현재 진료 중인 환자를 중단시키고 (선점) 위급 환자를 먼저 처치한다. 또한, 오랫동안 기다린 환자나 갑자기 상태가 변한 환자에게는 임시로 우선순위를 높여주는 (Boost) 유연함도 갖추고 있다.

- **등장 배경**:
  1. **데스크탑 OS의 요구사항**: 초기 Windows는 워크스테이션 환경에서 사용자의 키보드/마우스 입력에 즉각적으로 반응해야 하는 강력한 대화형 (Interactive) 성능이 필요했다.
  2. **NT 아키텍처의 확장성**: Windows NT 설계 당시부터 서버와 클라이언트 모두를 수용할 수 있도록 32단계의 정교한 우선순위 체계와 대칭형 다중 처리 (SMP, Symmetric Multiprocessing) 지원이 고려되었다.
  3. **멀티미디어 환경 대응**: 비디오 재생이나 게임 등 일정한 CPU 점유가 필요한 워크로드를 위해 멀티미디어 클래스 스케줄러 (MMCSS) 등 특화된 메커니즘이 발전해 왔다.

- **📢 섹션 요약 비유**: "철저한 계급 사회 (우선순위)를 기본으로 하되, 상황에 따라 계급 상승 (부스트)의 기회를 주어 사회 전체의 활력 (응답성)을 유지하는 시스템입니다."

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **Dispatcher (디스패처)** | 실제 문맥 교환 수행 | 스레드 상태 전이 및 레지스터 복원 | Context Switch | 무대 감독 |
| **Priority Levels** | 스레드 계급 체계 (0-31) | 0(Idle), 1-15(Dynamic), 16-31(Real-time) 구분 | MLFQ | 군대 계급 |
| **Quantum (퀀텀)** | 스레드당 할당된 실행 시간 | 시스템 설정에 따라 2~12 퀀텀 단위 부여 | Time Slice | 배정된 마이크 시간 |
| **Scheduler Database** | 실행 대기 스레드 관리 | 우선순위별 대기 큐 (Ready Queue) 관리 | Bitmap Search | 대기자 명단 |
| **Priority Boost** | 임시 우선순위 상향 | I/O 완료, 세마포어 획득 시 우선순위 가산 | Dynamic Priority | 보너스 점수 |

### 다단계 피드백 큐 (MLFQ) 구조 및 상태 전이

Windows 스케줄러는 32개의 우선순위 레벨을 가지며, 각 레벨마다 별도의 대기 큐 (Ready Queue)를 유지한다. 높은 우선순위 큐가 비어있지 않으면 낮은 우선순위 스레드는 절대 실행되지 않는 엄격한 구조를 가진다.

```text
 ┌─────────────────────────────────────────────────────────────────────────┐
 │               Windows Multi-level Feedback Queue Structure              │
 ├─────────────────────────────────────────────────────────────────────────┤
 │                                                                         │
 │  [Priority 31] ──▶ [Thread A] ──▶ [Thread B]  (Real-time Zone)          │
 │  [Priority 30] ──▶ [ ]                                                  │
 │     ...                                                                 │
 │  [Priority 15] ──▶ [Thread C]                 (Foreground Boosted)      │
 │  [Priority 14] ──▶ [Thread D] ──▶ [Thread E]                            │
 │     ...                                                                 │
 │  [Priority  8] ──▶ [Thread F]                 (Normal/Base Priority)    │
 │     ...                                                                 │
 │  [Priority  1] ──▶ [Thread G]                                           │
 │  [Priority  0] ──▶ [Idle Thread]              (System Zero Page)        │
 │                                                                         │
 │  Selection Logic:                                                       │
 │  1. Find highest bit set in 32-bit Priority Bitmap (O(1)).              │
 │  2. Pick the head of the corresponding Ready Queue.                     │
 │  3. If Quantum expires, move to tail of the same (or lower) queue.      │
 │  4. If I/O completes, boost priority and move to higher queue.           │
 │                                                                         │
 └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Windows 스케줄러는 32비트 비트맵을 사용하여 실행 가능한 스레드가 있는 가장 높은 우선순위를 $O(1)$ 시간에 찾아낸다. 16-31번은 실시간 클래스로, 시스템에 의해 우선순위가 절대 변하지 않는다. 1-15번은 동적 클래스로, 사용자의 입력이나 I/O 상태에 따라 스케줄러가 수시로 우선순위를 조정한다. 예를 들어 포그라운드 윈도우를 가진 프로세스의 스레드는 우선순위가 부스트되어 더 높은 번호의 큐로 이동하며, 더 긴 퀀텀 (Quantum)을 할당받아 부드러운 사용자 경험을 제공한다.

### 심층 동작 원리: 퀀텀 (Quantum)과 부스트 (Boost)

Windows의 퀀텀은 단순히 시간이 아니라 '클럭 틱 (Clock Tick)'의 배수로 계산된다. 기본적으로 클라이언트 OS는 짧은 퀀텀 (약 20ms)을, 서버 OS는 긴 퀀텀 (약 120ms)을 사용하여 각각 응답성과 처리량에 최적화된다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │              Windows Priority Boost 메커니즘                   │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │  1. I/O Completion:                                           │
  │     - 키보드/마우스 입력 완료: +6 Boost                        │
  │     - 디스크 I/O 완료: +1 Boost                                │
  │     - 사운드 장치 완료: +8 Boost                               │
  │                                                               │
  │  2. Waiting for Resources:                                    │
  │     - 이벤트/세마포어 대기 해제: +1~+2 Boost                    │
  │                                                               │
  │  3. GUI/Foreground Boost:                                     │
  │     - 현재 활성화된 창의 프로세스: Quantum 3배 연장              │
  │                                                               │
  │  4. Starvation Avoidance (Balance Set Manager):               │
  │     - 4초간 실행 못한 스레드: Priority 15로 강제 부스트 +        │
  │       Quantum 2배 할당 (기아 방지)                              │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Windows 스케줄링의 묘미는 이 '동적 부스트'에 있다. 스레드가 I/O를 기다리다 완료되면, 스케줄러는 이 스레드가 대화형 작업일 가능성이 높다고 판단하여 우선순위를 즉시 높여준다. 이를 통해 I/O 바운드 스레드가 CPU를 빠르게 점유할 수 있게 한다. 또한, '밸런스 셋 매니저 (Balance Set Manager)'라는 감시 프로세스가 있어, 낮은 우선순위 때문에 4초 이상 굶고 있는 스레드를 발견하면 일시적으로 레벨 15까지 끌어올려 실행 기회를 보장함으로써 기아 (Starvation) 현상을 원천 방지한다.

① 스레드가 준비 (Ready) 상태 진입 → ② 비트맵 검색으로 최상위 우선순위 식별 → ③ 디스패처가 문맥 교환 실행 → ④ 퀀텀 소모 또는 자발적 대기 (Wait) 발생 → ⑤ 우선순위 재계산 및 큐 재배치.

- **📢 섹션 요약 비유**: "성적이 낮은 학생이라도 오랫동안 발표를 못 하면 선생님이 특별히 지목 (기아 방지 부스트)해주고, 질문에 대답을 잘하면 사탕 (I/O 부스트)을 주는 교실 운영 방식과 같습니다."

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: Linux CFS vs Windows Scheduling

| 비교 항목 | Linux CFS (Completely Fair) | Windows Scheduling (MLFQ) |
|:---|:---|:---|
| **기본 철학** | 공정성 (Fairness) - vruntime 기반 | 우선순위 (Priority) - 계급 기반 |
| **스케줄링 단위** | 프로세스 및 스레드 (태스크) | 스레드 (Thread) |
| **자료구조** | Red-Black Tree ($O(\log N)$) | Priority Queues ($O(1)$ bitmap search) |
| **퀀텀 관리** | 가변적 (태스크 수에 따라 변함) | 고정적 (OS 설정 및 부스트에 따라 결정) |
| **기아 방지** | vruntime 누적으로 자동 해결 | Balance Set Manager가 명시적으로 부스트 |
| **주 타겟** | 서버, 임베디드, 다양한 워크로드 | 데스크탑 UI, 게임, 엔터프라이즈 환경 |

Linux CFS는 모든 태스크를 공평하게 대우하려는 '민주주의적' 모델인 반면, Windows는 철저히 우선순위에 따라 차등 대우하는 '능력주의적' 모델에 가깝다. Windows 모델은 복잡한 트리 계산 없이 비트맵 검색만으로 다음에 실행할 스레드를 결정하므로 결정 속도가 매우 빠르고, 사용자 인터페이스 응답성에 유리하다.

### 과목 융합 관점

- **컴퓨터 아키텍처 (NUMA)**: Windows는 NUMA 노드 구조를 인식하여, 스레드가 생성된 노드의 로컬 메모리에 접근할 수 있도록 해당 노드의 CPU에 우선 배정하는 '이상적 프로세서 (Ideal Processor)' 전략을 사용한다.
- **가상화 (Hyper-V)**: 가상 머신 (VM) 내에서 동작하는 Windows 스케줄러는 하이퍼바이저와 통신하여, 가상 CPU (vCPU)가 실제 물리 CPU에 스케줄링되는 타이밍을 최적화한다.

- **📢 섹션 요약 비유**: "모두가 똑같은 양의 빵을 나누어 갖는 방식 (Linux)과, 가장 배고프거나 중요한 사람에게 빵을 먼저 주는 방식 (Windows)의 차이입니다."

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — SQL Server 성능 최적화**: 대규모 DB 서버에서 쿼리 처리 스레드들이 퀀텀을 너무 자주 소진하여 문맥 교환 오버헤드가 발생하는 상황. 시스템 설정에서 '프로세서 리소스 할당'을 '프로그램'에서 '백그라운드 서비스'로 변경하여, 퀀텀을 길고 고정적으로 할당받게 함으로써 처리량 (Throughput)을 극대화한다.

2. **시나리오 — 고성능 게임 프레임 드랍 방지**: 백그라운드 업데이트 프로세스가 CPU를 간헐적으로 점유하여 게임 프레임이 튀는 현상. Windows의 **멀티미디어 클래스 스케줄러 (MMCSS)** 서비스를 활용하여 게임 스레드의 우선순위를 실시간 영역에 가깝게 격상시키고, CPU 자원의 80% 이상을 보장받도록 설계한다.

### 도입 체크리스트
- **Priority Class 선택**: `ABOVE_NORMAL_PRIORITY_CLASS` 등을 통해 중요한 워크로드의 우선순위를 명시적으로 지정했는가?
- **Processor Affinity**: 특정 코어에 부하를 고정해야 하는 경우 (캐시 최적화 등) 어피니티 설정을 고려했는가?
- **I/O Priority**: 중요도가 낮은 백그라운드 복사 작업 등에 대해 `IDLE_PRIORITY_CLASS`와 함께 낮은 I/O 우선순위를 적용했는가?

### 안티패턴
- **실시간 클래스 (16-31) 남발**: 우선순위 31로 루프를 도는 스레드를 만들면, 마우스 움직임조차 차단되어 시스템이 완전히 얼어버리는 (Freeze) 현상이 발생한다. 실시간 영역은 매우 짧은 작업에만 사용해야 한다.
- **스레드 과다 생성**: Windows는 스레드 기반 스케줄링이므로, 수천 개의 스레드를 생성하면 스케줄러 데이터베이스 탐색 및 문맥 교환 비용이 실제 연산 비용을 압도하게 된다. (I/O Completion Port 활용 권장)

- **📢 섹션 요약 비유**: "강력한 엔진 (우선순위 31)은 필요할 때만 잠시 써야지, 계속 켜두면 자동차가 과열되어 멈춰버리는 것과 같습니다."

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 효과 | 비고 |
|:---|:---|:---|
| **응답성 (Responsiveness)** | 포그라운드 부스트를 통해 UI 지연 시간 최소화 | 데스크탑 만족도 직결 |
| **처리량 (Throughput)** | 서버 모드 퀀텀 최적화로 컨텍스트 스위치 비용 절감 | 서버급 워크로드 지원 |
| **안정성 (Reliability)** | 기아 방지 메커니즘으로 낮은 순위 작업 생존 보장 | 시스템 데드락 방지 |
| **확장성 (Scalability)** | NUMA 및 멀티코어 최적화로 성능 선형 증가 | 최신 하드웨어 활용 |

### 미래 전망
- **Hardware-guided Scheduling**: Intel의 Thread Director 기술과 연동하여, Windows 스케줄러가 스레드의 특성 (Performance vs Efficient)을 하드웨어로부터 전달받아 가장 적합한 코어 (P-core 또는 E-core)에 배치하는 지능형 스케줄링으로 진화하고 있다.
- **Energy-aware Scheduling**: 모바일 장치와 노트북 환경을 위해, 성능을 희생하지 않으면서도 전력 소비를 최소화하는 방향으로 퀀텀 관리와 부스트 로직이 더욱 세밀화되고 있다.

### 참고 표준
- **Windows API (Win32)**: `SetThreadPriority`, `SetPriorityClass` 등 표준 인터페이스 제공
- **WHEA (Windows Hardware Error Architecture)**: 하드웨어 오류와 스케줄링 연계 표준

- **📢 섹션 요약 비유**: "하드웨어와 운영체제가 서로 정보를 주고받으며 가장 효율적인 길을 찾는 '자율주행형 스케줄링'의 시대로 진입하고 있습니다."

## 👶 어린이를 위한 3줄 비유 설명
1. 윈도우 스케줄러는 줄을 서 있는 친구들 중에서 **가장 급한 친구 (높은 우선순위)**를 먼저 놀이터에 보내주는 선생님 같아요.
2. 장난감을 가지고 놀다가 다 쓴 친구는 다시 줄의 끝으로 가지만, **공부를 열심히 한 친구나 오래 기다린 친구**는 선생님이 특별히 줄의 앞쪽으로 옮겨준답니다.
3. 하지만 너무 급하다고 가짜로 울면 (우선순위 남발) 놀이터가 엉망이 될 수 있어서, 선생님은 항상 공정하게 번호표 (비트맵)를 확인하며 관리해요!
