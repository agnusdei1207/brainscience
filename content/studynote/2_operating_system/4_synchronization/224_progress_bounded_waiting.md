# 진행 (Progress) 및 유한 대기 (Bounded Waiting)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 진행 (Progress)과 유한 대기 (Bounded Waiting)는 상호 배제 (Mutual Exclusion)와 함께 임계 영역 문제 (Critical Section Problem)를 해결하기 위해 반드시 충족해야 하는 필수 설계 조건이다.
> 2. **가치**: 진행은 시스템 전체의 교착 상태 (Deadlock) 방지를 보장하며, 유한 대기는 특정 프로세스가 무한히 굶주리는 기아 상태 (Starvation)를 방지하여 시스템의 공정성 (Fairness)을 확보한다.
> 3. **융합**: 스케줄링 알고리즘의 공정성, 실시간 시스템의 결정론적 (Deterministic) 동작, 그리고 분산 자원 할당 메커니즘의 안정성을 평가하는 핵심 지표로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 임계 영역 문제를 해결하기 위한 세 가지 조건 중 '상호 배제'가 정확성 (Safety)을 담당한다면, **진행 (Progress)**과 **유한 대기 (Bounded Waiting)**는 활성 (Liveness)을 담당한다.
  - **진행 (Progress)**: 임계 영역에 진입한 프로세스가 없고 진입하려는 프로세스가 있다면, 잔류 영역에 있지 않은 프로세스들만 진입 결정을 내릴 수 있으며 이 결정은 무한정 미뤄져서는 안 된다.
  - **유한 대기 (Bounded Waiting)**: 한 프로세스가 임계 영역 진입을 요청한 후 허가될 때까지, 다른 프로세스들이 임계 영역에 진입하는 횟수에 제한이 있어야 한다.

- **💡 비유**: 
  - **진행**은 "빈 식당에 손님이 왔을 때, 종업원들이 자기들끼리 의논만 하느라 손님을 안 들여보내는 일이 없어야 함"과 같다.
  - **유한 대기**는 "내 뒤에 온 손님들이 계속 먼저 들어가는 새치기가 반복되어 내가 영원히 못 들어가는 상황이 없도록 번호표를 운영함"과 같다.

- **등장 배경**: 초기 동기화 알고리즘 (예: Dekker 알고리즘 초기 버전)은 상호 배제는 완벽했으나, 두 프로세스가 서로 양보만 하다가 아무도 못 들어가는 '교착 상태'나, 한 프로세스가 자원을 독점하여 다른 프로세스가 굶어 죽는 '기아 상태'를 해결하지 못했다. 이를 체계적으로 정의한 것이 이 3대 조건이다.

- **📢 섹션 요약 비유**: 사고를 막는 것(상호 배제)만큼이나, 교통이 멈추지 않게 하고(진행), 모든 차선이 공평하게 지나가게 하는 것(유한 대기)이 도로 통제의 완성입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 임계 영역 문제 해결의 3대 필수 요건

| 요건명 | 정의 | 위반 시 발생하는 문제 | 해결 매커니즘 | 비유 |
|:---|:---|:---|:---|:---|
| **상호 배제 (Mutual Exclusion)** | 한 번에 하나만 진입 | 데이터 오염 (Race Condition) | Lock, Semaphore, Monitor | 1인용 화장실 |
| **진행 (Progress)** | 진입 대기 시 결정 지연 불가 | 시스템 정지 (Deadlock/Livelock) | 결정권자의 명확한 선정 | 비어있으면 즉시 입장 |
| **유한 대기 (Bounded Waiting)** | 대기 시간/횟수의 상한 보장 | 기아 상태 (Starvation) | 대기 큐 (FIFO), 순번 부여 | 대기 번호표 |

### 진행(Progress)과 유한 대기(Bounded Waiting)의 상관관계

진행은 '시스템 전체'의 진행을 보장하는 것이고, 유한 대기는 '개별 프로세스'의 권리를 보장하는 것이다. 유한 대기가 만족되면 보통 진행도 만족되지만, 진행이 만족된다고 해서 유한 대기가 보장되는 것은 아니다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│             임계 영역 문제 해결 조건의 논리적 포함 관계                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│      [ 활성 (Liveness) 보장 영역 ]                                    │
│  ┌───────────────────────────────────────────────────────────┐      │
│  │                                                           │      │
│  │  ┌─────────────────────────────────────────────────────┐  │      │
│  │  │                                                     │  │      │
│  │  │    [ 유한 대기 Bounded Waiting ]                     │  │      │
│  │  │    - 개별 프로세스의 기아 방지                          │  │      │
│  │  │    - 큐(Queue)를 통한 공정한 순서 보장                  │  │      │
│  │  │                                                     │  │      │
│  │  └──────────────┬──────────────────────────────────────┘  │      │
│  │                 │ (포함 관계 아님, 상호 보완적)               │      │
│  │                 ▼                                         │      │
│  │  ┌─────────────────────────────────────────────────────┐  │      │
│  │  │                                                     │  │      │
│  │  │    [ 진행 Progress ]                                │  │      │
│  │  │    - 시스템 전체의 멈춤 방지 (Deadlock Free)           │  │      │
│  │  │    - 결정은 즉시 내려져야 함                           │  │      │
│  │  │                                                     │  │      │
│  │  └─────────────────────────────────────────────────────┘  │      │
│  │                                                           │      │
│  └───────────────────────────────────────────────────────────┘      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상호 배제(Safety)가 지켜지는 전제 하에, 진행(Progress)은 '임계 영역이 비어있다면 누군가는 반드시 들어간다'는 최소한의 동작성을 의미한다. 반면 유한 대기(Bounded Waiting)는 더 높은 수준의 품질로, '나보다 늦게 온 프로세스가 나를 앞질러 들어가는 횟수가 제한된다'는 공정성을 의미한다. 현대적인 스핀락(Spinlock)은 진행은 만족하지만, 운이 나쁜 스레드가 계속 락을 뺏기는 경우 유한 대기를 만족하지 못할 수도 있다.

### 원리 상세: 기아 상태 (Starvation) 유발 케이스

유한 대기가 보장되지 않는 전형적인 사례는 우선순위 기반 스케줄링에서 발생한다. 높은 우선순위 프로세스가 계속해서 진입을 시도하면 낮은 우선순위 프로세스는 영원히 임계 영역에 들어가지 못한다.

1. **상황**: 프로세스 P0가 임계 영역 진입 대기 중.
2. **사건**: 고우선순위 P1, P2, P3...가 계속 도착하여 P0를 제치고 진입.
3. **결과**: P0는 진입 요청을 했음에도 불구하고 유한한 시간 내에 허가를 받지 못함 (Bounded Waiting 위반).
4. **해결 전략**: 에이징 (Aging) 기법을 통해 대기 시간이 길어질수록 우선순위를 높이거나, FIFO 큐를 사용하여 진입 순서를 강제한다.

- **📢 섹션 요약 비유**: 식당이 계속 영업 중(진행)이어도, 단골 손님만 먼저 들여보내서 내가 3시간째 대기 중(유한 대기 위반)이라면 그 식당의 운영 시스템은 잘못된 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 진행(Progress) vs 유한 대기(Bounded Waiting) 심층 비교

| 비교 항목 | 진행 (Progress) | 유한 대기 (Bounded Waiting) |
|:---|:---|:---|
| **핵심 목표** | 교착 상태 (Deadlock) 방지 | 기아 상태 (Starvation) 방지 |
| **보장 범위** | 시스템 전체 (Global) | 개별 프로세스 (Individual) |
| **실패 시 현상** | 전체 프로세스 응답 없음 | 특정 프로세스만 응답 없음 |
| **주요 원인** | 상호 양보, 잘못된 조건문 | 무한 경쟁, 우선순위 편향 |
| **해결 방법** | 결정권의 단일화, 타임아웃 | 대기 큐, 에이징, 순번제 |

### 과목 융합 관점

1. **알고리즘 (Algorithm)**: 뱅커 알고리즘 (Banker's Algorithm)은 자원 할당 시 '안정 상태'를 확인하여 진행(Progress)을 보장하는 전략이다.
2. **네트워크 (Network)**: CSMA/CD 방식에서 충돌 후 재전송 시 사용하는 'Binary Exponential Backoff'는 충돌 확률을 낮춰 진행을 돕지만, 운이 없으면 계속 충돌하는 노드가 발생할 수 있어 유한 대기를 완벽히 보장하지는 않는다. 반면 토큰 패싱 (Token Passing)은 유한 대기를 확실히 보장한다.
3. **실시간 시스템 (RTOS)**: 실시간 시스템에서는 모든 대기 시간이 예측 가능 (Deterministic)해야 하므로 유한 대기가 상호 배제만큼이나 중요하다. Worst Case Execution Time (WCET) 계산의 근간이 된다.

- **📢 섹션 요약 비유**: 전체 매출이 오르는 것(진행)도 중요하지만, 소외되는 가맹점 없이 고루 혜택이 돌아가는 것(유한 대기)이 지속 가능한 생태계(OS)를 만듭니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오 및 설계 전략

1. **시나리오: 웹 서버의 커넥션 풀 (Connection Pool) 관리**: 수만 명의 사용자 요청 중 특정 요청이 자원 획득 경쟁에서 계속 밀려 타임아웃이 발생하는 상황.
   - **판단**: 상호 배제는 지켜지고 있으나 유한 대기가 보장되지 않음.
   - **해결**: Fair Lock (Java의 `ReentrantLock(true)`)을 사용하여 대기 큐에 들어온 순서대로 자원을 할당하도록 수정한다. 단, 이 경우 처리량 (Throughput)은 다소 저하될 수 있다.

2. **시나리오: 멀티코어 환경의 스핀락 (Spinlock) 사용**: 락 획득을 위해 CPU를 점유하며 대기하는 스핀락이 성능은 좋지만 특정 코어가 계속 락을 뺏기는 현상.
   - **판단**: 진행은 빠르지만 유한 대기 보장이 약함.
   - **해결**: 티켓 락 (Ticket Lock) 기법을 도입하여 번호표를 발급하고, 자기 순서가 올 때까지 스핀하게 함으로써 유한 대기를 보장한다.

### 기술사적 판단: 성능과 공정성의 트레이드오프
- **공정성 중시 (Fairness)**: 유한 대기를 엄격히 보장하기 위해 FIFO 큐를 관리하면 문맥 교환과 관리 오버헤드가 증가한다.
- **성능 중시 (Performance)**: 진행 속도를 극대화하기 위해 락을 해제하자마자 가장 먼저 도착한 프로세스가 아닌, 현재 CPU 캐시에 데이터가 있는 '운 좋은' 프로세스에게 락을 주면 처리량은 늘어나지만 공정성은 깨진다.
- **결론**: 일반 어플리케이션은 성능 위주, 금융/의료/실시간 제어 시스템은 유한 대기 보장 위주로 설계해야 한다.

- **📢 섹션 요약 비유**: 고속도로 하이패스(성능)는 빠르지만 일반 차로(공정성)가 없으면 안 되듯, 상황에 맞는 '균형 잡힌 대기 전략'이 필요합니다.

## Ⅴ. 기대효과 및 결론

### 도입 및 설계 기대효과

| 구분 | 조건 미충족 시 (Risk) | 조건 충족 시 (Benefit) |
|:---|:---|:---|
| **진행 (Progress)** | 시스템 먹통, 전체 서비스 중단 | 지속적인 서비스 가용성 확보 |
| **유한 대기 (Bounded)** | 특정 사용자 불만, 무한 로딩 | 모든 사용자에게 균등한 응답 시간 보장 |
| **신뢰성** | 예측 불가능한 시스템 장애 | SLA (Service Level Agreement) 준수 용이 |

### 미래 전망
클라우드 네이티브 환경의 마이크로서비스 아키텍처 (MSA)에서는 분산 락 (Distributed Lock)의 진행과 유한 대기가 더욱 중요해지고 있다. 네트워크 파티션 (Partition) 발생 시에도 'Liveness'를 어떻게 유지할 것인가가 연구의 핵심이며, 이를 위해 Raft나 Paxos 같은 합의 알고리즘이 진행과 유한 대기를 보장하는 현대적 도구로 자리 잡고 있다.

- **📢 섹션 요약 비유**: 도시가 멈추지 않고(진행), 모든 시민이 제시간에 출근할 수 있는(유한 대기) 대중교통망을 설계하는 것이 운영체제의 궁극적 목표입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **진행**은 놀이터에 아무도 안 놀고 있으면, 기다리던 친구가 바로 들어가서 놀 수 있어야 한다는 규칙이에요.
2. **유한 대기**는 내가 줄을 섰는데, 내 뒤에 온 친구들이 계속 먼저 들어가는 '새치기'를 당하지 않게 보장해 주는 거예요.
3. 이 두 규칙이 잘 지켜져야 싸움도 안 나고, 모든 친구가 신나게 놀 수 있는 행복한 놀이터(컴퓨터)가 된답니다!
