# Compare-and-Swap (CAS)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: Compare-and-Swap (CAS)은 메모리 위치의 현재 값이 예상하는 값 (Expected Value)과 일치할 때만 새로운 값 (New Value)으로 업데이트하는 하드웨어 지원 원자적 명령어다.
> 2. **가치**: 락 (Lock) 없이도 데이터 일관성을 유지할 수 있게 하여, 문맥 교환 오버헤드가 없는 고성능 락-프리 (Lock-free) 및 웨이트-프리 (Wait-free) 알고리즘 구현의 핵심 동력이다.
> 3. **융합**: 현대 멀티코어 프로세서의 필수 명령어로, JVM (Java Virtual Machine)의 `Atomic` 패키지, 데이터베이스의 낙관적 동시성 제어 (OCC), 그리고 고가용성 분산 시스템의 상태 일관성 보장에 활용된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: **Compare-and-Swap (CAS)**은 "비교하고 바꾼다"는 의미로, 세 개의 인자(메모리 주소 V, 예상하는 기존 값 A, 새로 설정할 값 B)를 받아 `V의 값 == A`이면 `V를 B로` 바꾸고 성공을 반환하며, 그렇지 않으면 아무 작업도 하지 않고 실패를 반환하는 원자적 연산이다.

- **💡 비유**: CAS는 "정답을 아는 사람만 수정 가능한 게시판"과 같다. 게시판에 `10`이라고 써있을 때, 내가 "기존 값이 `10`이면 `11`로 바꿔줘"라고 요청하는 식이다. 만약 그 사이에 다른 사람이 먼저 `11`로 바꿨다면 내 요청은 거절당한다. "기존 값이 `10`이 아니기 때문"이다.

- **등장 배경**: 뮤텍스나 세마포어 같은 전통적인 비관적 락 (Pessimistic Lock)은 스레드를 중단시키고 깨우는 과정에서 막대한 문맥 교환 오버헤드가 발생했다. 특히 하이퍼스레딩과 멀티코어가 보편화되면서, 락을 얻기 위해 대기하는 비용보다 "일단 시도해보고 안 되면 다시 하기"라는 낙관적 (Optimistic) 접근이 성능상 유리해짐에 따라 CAS가 동기화의 표준으로 자리 잡았다.

- **📢 섹션 요약 비유**: 상태를 미리 확인하고 일치할 때만 변화를 허용하는 '지능형 검문소'와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### CAS의 5대 구성 요소 및 파라미터

| 요소명 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **메모리 위치 (V)** | 업데이트 대상 주소 | 공유 자원이 저장된 물리적/논리적 주소 | 수정할 장부 페이지 |
| **기대값 (A)** | 일관성 체크 기준 | 프로세스가 마지막으로 읽었던 데이터 상태 | 내가 알고 있는 옛날 정답 |
| **신규값 (B)** | 업데이트할 목표 데이터 | 조건 충족 시 새로 반영될 데이터 | 내가 제안하는 새로운 정답 |
| **원자성 보장 (Bus Lock)** | 불가분 연산 보장 | 연산 중 타 프로세서의 해당 메모리 접근 차단 | 수정 중 장부 독점 |
| **실패 처리 (Retry)** | 실패 시 재시도 로직 | 루프를 돌며 최신 값을 다시 읽고 CAS 재시도 | 거절당하면 다시 읽고 도전 |

### CAS 논리 구조 및 구현 매커니즘

CAS 명령어의 논리적 동작은 CPU 수준에서 원자적으로 처리된다. 아래는 이를 이용한 안전한 변수 업데이트 (예: 원자적 증가)의 흐름이다.

```c
// 하드웨어가 보장하는 원자적 논리 (Pseudo-code)
bool CAS(int *addr, int expected, int new_val) {
    if (*addr == expected) {
        *addr = new_val;
        return true;  // 성공
    }
    return false;     // 실패
}

// CAS를 이용한 원자적 증가 (Atomic Increment) 구현
void AtomicIncrement(int *addr) {
    int old_val;
    do {
        old_val = *addr; // 1. 현재 값을 읽음
    } while (!CAS(addr, old_val, old_val + 1)); // 2. 내가 읽은 값과 같으면 +1 반영, 다르면 재시도
}
```

```text
┌─────────────────────────────────────────────────────────────────────┐
│                    CAS (Compare-and-Swap) 동작 과정                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│    Memory V = 10                                                    │
│                                                                     │
│    [Thread 1]                      [Thread 2]                       │
│    1. Read V (V=10)                1. Read V (V=10)                 │
│    2. Calculate (10+1=11)          2. Calculate (10+1=11)           │
│                                    3. CAS(V, 10, 11)                │
│                                       - V가 10인가? Yes!             │
│                                       - V를 11로 변경                │
│                                       - [Success]                   │
│    3. CAS(V, 10, 11)                                                │
│       - V가 10인가? No! (현재 11)                                     │
│       - 작업 취소                                                   │
│       - [Fail]                                                      │
│    4. Retry!                                                        │
│       - Read V (V=11)                                               │
│       - Calculate (11+1=12)                                         │
│       - CAS(V, 11, 12) -> [Success]                                 │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 두 스레드가 동시에 값을 1 증가시키려 한다. 스레드 2가 아주 미세하게 빨라 먼저 CAS에 성공하여 메모리 값을 11로 바꿨다. 스레드 1은 자신이 읽었던 '10'이 현재 메모리 값과 다르다는 것을 CAS 단계에서 발견하고 실패한다. 스레드 1은 중단되지 않고 즉시 다시 값을 읽어(11) 재시도함으로써 최종적으로 12를 만든다. 이 과정에서 어떤 스레드도 대기 상태(Sleep)에 빠지지 않는다.

### 심층 동작 원리: ABA 문제 (The ABA Problem)

CAS의 가장 큰 함정은 **ABA 문제**다. 스레드 1이 값 A를 읽고 작업을 준비하는 동안, 스레드 2가 값을 A → B → A로 바꿨을 때 발생한다.

1. **상황**: 스레드 1은 메모리 값이 여전히 A이므로 변경되지 않았다고 착각하고 CAS를 성공시킨다.
2. **위험**: 값은 같아도 내부 상태(예: 연결 리스트의 노드 주소 재사용)가 바뀌었을 경우 데이터 구조가 붕괴될 수 있다.
3. **해결**: **버전 관리 (Versioning)** 기법을 사용한다. 값만 비교하는 것이 아니라, 값이 바뀔 때마다 증가하는 '버전 태그'를 함께 비교한다. (A, v1 → B, v2 → A, v3).

- **📢 섹션 요약 비유**: 여자친구가 잠시 자리를 비운 사이 (A), 다른 사람이 그 자리에 앉았다가 (B), 다시 여자친구가 돌아왔을 때 (A), 남자친구는 그 사이에 무슨 일이 있었는지 모른 채 그대로 대화를 이어가는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비관적 락 (Pessimistic) vs 낙관적 락 (CAS 기반)

| 비교 항목 | 비관적 락 (Mutex/Semaphore) | 낙관적 락 (CAS) |
|:---|:---|:---|
| **기본 철학** | "충돌이 날 것이니 미리 잠근다" | "충돌이 드무니 일단 시도하고 확인한다" |
| **자원 관리** | OS 수준의 스케줄링 관리 | CPU 수준의 원자적 연산 |
| **오버헤드** | 문맥 교환 (Context Switch) 발생 | CPU 루프 (Retry) 발생 |
| **경합 시 성능** | 경합이 심할 때 유리 (대기 상태) | 경합이 적을 때 매우 유리 (초고속) |
| **주 사용처** | 긴 작업 시간, 높은 경합 환경 | 짧은 작업 시간, 낮은 경합 환경 |

### 과목 융합 관점

1. **프로그래밍 언어 (Java)**: `java.util.concurrent.atomic` 패키지의 `AtomicInteger`, `AtomicReference` 등은 내부적으로 Unsafe 클래스의 `compareAndSwap` 명령어를 사용하여 구현되어 있다.
2. **데이터베이스 (DB)**: MVCC (Multi-Version Concurrency Control) 환경에서 레코드 업데이트 시, 버전 번호를 조건절에 넣어 업데이트하는 방식(`WHERE version = last_read_version`)이 논리적 수준의 CAS다.
3. **분산 시스템**: 분산 락 라이브러리 (Redis, Zookeeper)에서도 'Set-if-not-exists'나 'Compare-and-Set' 작업을 통해 노드 간 상태 일관성을 유지한다.

- **📢 섹션 요약 비유**: 비관적 락이 "모든 손님에게 소지품 검사를 하는 철저한 보안"이라면, CAS는 "나갈 때 물건이 없어졌는지 확인하고 재결제하는 자율 계산대"와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오 및 해결 전략

1. **시나리오: 초고성능 주식 거래 엔진의 주문 카운터**: 수만 개의 주문이 동시에 들어오는 상황에서 Mutex를 쓰면 락 경합으로 레이턴시가 튄다.
   - **해결**: CAS 기반의 락-프리 큐와 원자적 카운터를 사용하여 스레드 차단 없이 주문을 처리한다.

2. **시나리오: 공유 메모리 상의 연결 리스트 (Linked List) 관리**: 여러 스레드가 동시에 노드를 추가/삭제할 때 포인터 업데이트 오염 발생.
   - **해결**: 노드의 `next` 포인터를 업데이트할 때 CAS를 사용한다. 이때 ABA 문제를 막기 위해 포인터와 카운터를 결합한 `AtomicStampedReference` 형식을 사용한다.

### 기술사적 판단: CAS의 성능 한계 (Livelock & Contention)
- **락 경합 (Contention)**: 너무 많은 스레드가 하나의 변수에 CAS를 시도하면, 대부분이 실패하고 재시도 루프를 돌게 되어 CPU 사용률만 치솟고 실제 처리량(Throughput)은 떨어지는 현상이 발생한다.
- **해결 (Backoff)**: 실패 시 즉시 재시도하지 않고 아주 짧은 시간 대기 (Exponential Backoff)하거나, 변수를 여러 개로 쪼개어 경합을 분산시키는 `LongAdder` (Java 8+) 방식이 권장된다.

- **📢 섹션 요약 비유**: 너무 많은 사람이 좁은 출입구 하나로 동시에 밀려들면(Contention), 서로 밀치느라 아무도 못 나가는 상황이 되므로 출입구를 여러 개로 늘리거나(LongAdder) 순서를 정해야 합니다.

## Ⅴ. 기대효과 및 결론

### 도입 기대효과

| 구분 | 도입 전 (비관적 락) | 도입 후 (CAS 기반 낙관적 락) |
|:---|:---|:---|
| **응답성 (Latency)** | 스케줄링 대기로 인한 비결정적 지연 | 락-프리 구현으로 수 마이크로초 단위 응답 |
| **처리량 (Throughput)** | 문맥 교환 비용으로 인한 성능 한계 | CPU 코어 수에 비례하는 성능 확장성 |
| **안정성** | 데드락 (Deadlock) 위험 상존 | 데드락으로부터 자유로움 (Livelock 주의) |

### 미래 전망
하드웨어적으로는 CAS를 넘어 여러 메모리 위치를 한꺼번에 원자적으로 업데이트하는 **Transactional Memory (HTM)** 기술이 인텔 (TSX) 등을 통해 도입되고 있다. 소프트웨어적으로는 Rust와 같이 컴파일 타임에 동기화 문제를 제거하는 언어가 부상하고 있지만, 런타임에서의 초고속 동기화가 필요한 영역에서 CAS는 영원한 '골든 스탠다드'로 남을 것이다.

- **📢 섹션 요약 비유**: 개별 자물쇠(CAS)를 넘어 건물 전체의 보안 시나리오(HTM)를 자동으로 관리하는 시대가 오고 있지만, 가장 핵심적인 금고는 여전히 정밀한 비밀번호 확인(CAS)이 지키고 있습니다.
