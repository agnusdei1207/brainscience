# Java 동기화 - synchronized, wait/notify

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: Java 동기화는 모든 객체가 내장하고 있는 고유 락 (Intrinsic Lock)과 모니터 (Monitor) 메커니즘을 활용하여, 멀티스레드 환경에서 임계 영역의 상호 배제와 스레드 간 협력을 보장하는 언어 차원의 솔루션이다.
> 2. **가치**: `synchronized` 키워드로 복잡한 락 관리 로직을 캡슐화하고, `wait()`와 `notify()`를 통해 스레드 간 상태 전이를 효율적으로 제어함으로써 자원 낭비를 방지하고 실행 순서를 결정론적으로 관리한다.
> 3. **융합**: JVM (Java Virtual Machine)의 메모리 모델 (JMM), 스레드 스케줄러, 그리고 하드웨어의 원자적 연산과 긴밀히 결합하여 엔터프라이즈급 병렬 어플리케이션의 안정적 기반을 제공한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: Java는 언어 설계 초기부터 멀티스레딩을 고려하여, 모니터 (Monitor) 이론을 바탕으로 한 동기화 도구를 제공한다. 
  - **`synchronized`**: 특정 코드 블록이나 메서드를 단 하나의 스레드만 실행할 수 있도록 잠그는 키워드다.
  - **`wait()` / `notify()`**: 스레드가 특정 조건을 기다리며 잠들거나, 조건이 만족되었을 때 다른 스레드를 깨우는 통신 수단이다.

- **💡 비유**: Java 동기화는 **"모든 방에 자동으로 달려있는 도어락과 인터폰"**과 같다. `synchronized`는 방에 들어가면서 문을 잠그는 것이고, `wait()`은 재료가 올 때까지 방 안의 침대에서 자는 것이며, `notify()`는 재료를 가져온 사람이 인터폰으로 자는 사람을 깨워주는 것과 같다.

- **등장 배경**: C/C++ 등에서 개발자가 직접 뮤텍스와 세마포어를 관리하며 겪었던 '해제 누락'이나 '데드락' 문제를 해결하기 위해, JVM이 락의 획득과 해제를 자동으로 관리하고 객체 자체에 동기화 기능을 통합시키는 '객체 지향적 동기화' 패러다임이 필요하게 되었다.

- **📢 섹션 요약 비유**: 각 객체가 스스로를 보호하는 방어막(Lock)과 소통 창구(Wait/Notify)를 갖춘 지능형 자원 관리 시스템과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### Java 동기화의 5대 구성 요소

| 요소명 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **고유 락 (Intrinsic Lock)** | 객체별 할당된 상호 배제 도구 | 객체 헤더의 Mark Word를 이용해 락 상태 기록 | 방마다 달린 도어락 |
| **모니터 (Monitor)** | 동기화 전체 프로세스 관리자 | Entry Set과 Wait Set을 관리하는 논리적 단위 | 방 관리 비서 |
| **Entry Set** | 락 획득 대기열 (Blocked) | `synchronized` 진입을 기다리는 스레드 집합 | 복도의 대기 줄 |
| **Wait Set** | 조건 만족 대기열 (Waiting) | `wait()` 호출 후 신호를 기다리는 스레드 집합 | 방 안의 침대 |
| **메모리 가시성 (Visibility)** | 수정 내용의 즉시 공유 보장 | 락 해제 전 캐시 내용을 메인 메모리에 반영 | 작업 일지 공유 |

### synchronized와 wait/notify의 제어 흐름

Java 스레드는 모니터의 제어 하에 세 가지 상태를 오가며 동기화된다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                 Java Monitor 기반 스레드 상태 전이                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│        [ 밖 : Entry Set ]           [ 안 : Owner ]                  │
│       ┌─────────────────┐        ┌─────────────────┐                │
│       │  T2, T3, T4...  │ ──acquire──▶     T1      │                │
│       │  (Blocked)      │        │     (Running)   │                │
│       └─────────────────┘        └────────┬────────┘                │
│                ▲                          │                         │
│                │                       wait()                       │
│             release                       │                         │
│                │                          ▼                         │
│       ┌─────────────────┐        ┌─────────────────┐                │
│       │       T5        │ ◀─notify── [ 안 : Wait Set ]               │
│       │  (Waiting)      │        │      (Sleeping)  │                │
│       └─────────────────┘        └─────────────────┘                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 스레드가 `synchronized` 영역에 진입하려고 하면 'Entry Set'에 머물며 락을 기다린다. 락을 얻은 'Owner' 스레드(T1)가 실행 도중 `wait()`를 호출하면, T1은 락을 반납하고 'Wait Set'으로 이동하여 휴면 상태가 된다. 이후 다른 스레드가 들어와서 작업을 마치고 `notify()`를 호출하면, 'Wait Set'에 있던 T5가 깨어나 다시 'Entry Set'으로 이동하여 락 획득 경쟁에 참여한다.

### 심층 동작 원리: 고유 락의 진화 (Lock Inflation)

JVM은 성능을 위해 락의 상태를 4단계로 최적화한다.
1. **무설정 (Unlocked)**: 락이 걸리지 않은 상태.
2. **편향 락 (Biased Locking)**: 한 스레드가 락을 독점하는 경우, CAS 연산 없이 해당 스레드에게 락을 '편향'시킨다.
3. **경량 락 (Lightweight Locking)**: 여러 스레드가 경합하지만 짧은 시간 내 해결될 때, 스핀락 (Spinlock)을 사용한다.
4. **중량 락 (Heavyweight Locking)**: 경합이 심해지면 OS의 뮤텍스를 사용하여 스레드를 Block 시킨다.

- **📢 섹션 요약 비유**: 처음에는 믿고 열어두다가(편향), 조금 의심되면 가볍게 잠그고(경량), 상황이 험악해지면 철갑을 두른 금고(중량)로 변하는 가변형 보안 시스템과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### synchronized vs ReentrantLock (Java 5+)

| 비교 항목 | synchronized (Implicit) | ReentrantLock (Explicit) |
|:---|:---|:---|
| **구현 방식** | 키워드 기반 (자동 해제) | API 클래스 (명시적 해제) |
| **유연성** | 낮음 (블록 형태 고정) | 높음 (타임아웃, 인터럽트 가능) |
| **공정성 (Fairness)** | 지원 안 함 (선착순 아님) | 선택 가능 (Fairness 옵션) |
| **조건 변수** | 1개 (wait/notify) | 여러 개 (Condition 객체) |
| **성능** | Java 6 이후 자동 최적화로 우수 | 경합이 극심할 때 더 안정적 |

### 과목 융합 관점

1. **메모리 모델 (JMM)**: Java 동기화는 'Happens-Before' 관계를 형성한다. 즉, 스레드 A가 락을 풀기 전의 모든 메모리 변경 사항은, 동일한 락을 얻은 스레드 B에게 100% 가시성이 보장된다.
2. **컴퓨터 구조**: `synchronized`는 CPU의 메모리 배리어 (Memory Barrier) 명령어를 유발하여 캐시 일관성을 강제로 맞춘다.
3. **디자인 패턴**: 싱글톤 패턴의 'Double-Checked Locking' 구현 시 `synchronized`와 `volatile`의 조합이 필수적으로 등장한다.

- **📢 섹션 요약 비유**: `synchronized`가 "자동 변속기 차량"이라면, `ReentrantLock`은 "수동 변속기 레이싱 카"와 같은 용도 차이를 보입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오 및 해결 전략

1. **시나리오: `wait()` 호출 시의 조건 체크 실수**: 신호를 받고 깨어났는데 정작 조건이 아직 만족되지 않아 오류가 발생하는 상황.
   - **해결**: 반드시 **`while` 루프 안에서 `wait()`를 호출**해야 한다. (Spurious Wakeup 방지)
   ```java
   synchronized (lock) {
       while (!condition) { // if가 아닌 while 사용 필수
           lock.wait();
       }
       // 작업 수행
   }
   ```

2. **시나리오: `notify()`와 `notifyAll()`의 선택**: 대기 중인 스레드 중 어떤 것을 깨울지 결정해야 하는 상황.
   - **판단**: `notify()`는 딱 하나의 스레드만 깨우므로 효율적이지만, 엉뚱한 스레드가 깨어나서 다시 잠들면 시스템이 멈출 수 있다.
   - **해결**: 확실하지 않다면 모든 스레드를 깨우는 **`notifyAll()`**을 사용하는 것이 안전하다. (단, Thundering Herd 현상은 감수해야 함)

### 기술사적 판단: 데드락 예방 및 진단
- **예방**: 항상 락을 획득하는 순서를 모든 스레드에서 동일하게 유지해야 한다.
- **진단**: 실무에서 데드락이 의심되면 `jstack`이나 `VisualVM`을 통해 스레드 덤프를 분석한다. Java의 모니터 락은 덤프 상에 `waiting on <0x...>` 형태로 명확히 나타나므로 추적이 용이하다.

- **📢 섹션 요약 비유**: 잠에서 깼다고 바로 일하는 게 아니라 시계를 다시 확인(while)하고, 누구를 깨울지 모를 땐 다 깨우는(notifyAll) 게 안전한 운영의 지혜입니다.

## Ⅴ. 기대효과 및 결론

### 도입 기대효과

| 구분 | 도입 전 (Raw Sync) | 도입 후 (Java synchronized) |
|:---|:---|:---|
| **안전성** | 락 해제 누락으로 인한 데드락 빈번 | 스코프 종료 시 자동 해제로 안전함 |
| **생산성** | OS 종속적인 동기화 코드 작성 | WORA(Write Once, Run Anywhere) 보장 |
| **가시성** | 스레드 간 변수 값 불일치 발생 | 메모리 배리어 자동 적용으로 일관성 확보 |

### 미래 전망
최근의 Java(Project Loom)는 가상 스레드 (Virtual Thread)를 도입하며 동기화 패러다임을 또 한 번 바꾸고 있다. 기존의 중량 락이 운영체제 스레드를 차단하던 방식에서, 가상 스레드를 사용하여 차단 시 CPU를 다른 가상 스레드에게 양보하는 초고효율 방식으로 진화하고 있다. 이에 따라 `synchronized`보다는 `ReentrantLock`의 사용이 권장되는 추세다.

- **📢 섹션 요약 비유**: 집안의 도어락(Lock) 기술이 지문 인식과 원격 제어(Virtual Threads)로 진화하며 점점 더 보이지 않는 곳에서 편리하게 작동하고 있습니다.

### 👶 어린이를 위한 3줄 비유 설명
1. `synchronized`는 방에 들어갈 때 **"나만 쓸게!"** 하고 문을 잠그는 거예요. 다른 친구들은 복도에서 줄을 서서 기다려야 해요.
2. `wait()`는 방 안에서 간식이 올 때까지 **"낮잠 잘게!"** 하고 침대에 눕는 거예요. 그럼 복도에 있던 친구가 들어와서 놀 수 있어요.
3. 간식을 가져온 친구가 `notify()`라고 외치면, 자고 있던 친구가 번쩍 깨어나서 다시 방을 사용할 수 있게 된답니다!
