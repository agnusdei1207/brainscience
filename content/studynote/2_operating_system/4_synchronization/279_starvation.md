# 279. 기아 상태 (Starvation)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스레드가 무기한 리소스 획득 실패
> 2. **가치**: 공정성 위반의 극단적 사례
> 3. **융합**: 공정성, 락 큐, 우선순위와 연관

---

## Ⅰ. 개요

### 개념 정의

기아 상태(Starvation)는 **특정 스레드가 계속해서 리소스 획득에 실패하여 무기한 대기하는 상태**다. 불공정한 스케줄링이나 락 정책에서 발생한다.

### 💡 비유: 놀이터 미끄럼틀
기아 상태는 **놀이터 미끄럼틀**과 같다. 계속 새로운 아이들이 먼저 타면, 끝까지 기다리는 아이는 영영 못 탄다.

### 기아 상태 구조

│                기아 상태 구조                                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │

│  │  T3:            [획득] ─ [해제] ─ [획득] ...                  │ │
│  │                                                             │ │

│  │                                                             │ │
│  │  2. 우선순위 역전:                                                 │ │
│  │     • 낮은 우선순위가 계속 밀림                                      │ │

│  │                                                             │ │
│  │  4. 잘못된 설계:                                                   │ │
│  │     • 균형 잡히지 않은 락 사용                                       │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │

│  【증상】                                                              │

## Ⅱ. 상세 분석

### 상세 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                기아 상태 상세                                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【기아 vs 교착상태】                                                   │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  교착상태 (Deadlock):                                              │ │
│  │  • 모든 관련 스레드가 멈춤                                           │ │
│  │  • 순환 대기                                                       │ │
│  │  • 감지 쉬움                                                       │ │
│  │                                                             │ │
│  │  기아 상태 (Starvation):                                           │ │
│  │  • 일부 스레드만 영향                                                │ │
│  │  • 무한 대기 (단일 스레드 가능)                                       │ │
│  │  • 감지 어려움                                                      │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【Reader-Writer 기아】                                               │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  Writer 기아 (Reader 선호 정책):                                   │ │
│  │  • Reader가 계속 들어오면 Writer 영원히 대기                         │ │
│  │                                                             │ │
│  │  Reader 기아 (Writer 선호 정책):                                   │ │
│  │  • Writer가 계속 들어오면 Reader 영원히 대기                         │ │
│  │                                                             │ │
│  │  해결: 공정 RW 락                                                   │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【해결 방법】                                                         │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. 공정 락 사용:                                                   │ │
│  │     • FIFO 락 큐                                                   │ │
│  │     • 티켓 락, MCS 락                                              │ │
│  │                                                             │ │
│  │  2. 대기 횟수/시간 제한:                                             │ │
│  │     • 일정 횟수 후 우선권 부여                                        │ │
│  │     • aging 기법                                                   │ │
│  │                                                             │ │
│  │  3. 타임아웃:                                                       │ │
│  │     • 무한 대기 방지                                                │ │
│  │     • tryLock 사용                                                 │ │
│  │                                                             │ │
│  │  4. 우선순위 상속/상한:                                              │ │
│  │     • 실시간 시스템                                                  │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실무 적용

### 구현 예시
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【Java 공정 락으로 기아 방지】                                         │
│  ──────────────────                                                  │
│  import java.util.concurrent.locks.*;                               │
│                                                                     │
│  // 불공정 (기아 가능)                                                 │
│  ReentrantLock unfairLock = new ReentrantLock();  // 기본            │
│                                                                     │
│  // 공정 (기아 방지)                                                   │
│  ReentrantLock fairLock = new ReentrantLock(true);                  │
│                                                                     │
│  // 대기 상태 모니터링                                                  │
│  if (fairLock.hasQueuedThreads()) {                                 │
│      int queueLength = fairLock.getQueueLength();                   │
│      if (queueLength > THRESHOLD) {                                 │
│          log.warn("Long wait queue: " + queueLength);               │
│      }                                                              │
│  }                                                                  │
│                                                                     │
│  【Java ReentrantReadWriteLock 기아 방지】                             │
│  ──────────────────                                                  │
│  // 공정 모드로 생성                                                   │
│  ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock(true);  │
│                                                                     │
│  // 공정 모드: FIFO 순서 보장                                          │
│  // Writer 기아 방지                                                  │
│                                                                     │
│  【Aging 기법 구현】                                                   │
│  ──────────────────                                                  │
│  class AgingLock {                                                  │
│      std::mutex mtx;                                                │
│      std::condition_variable cv;                                    │
│      std::queue<std::pair<std::thread::id, int>> waiters;           │
│      bool locked = false;                                           │
│      const int AGING_THRESHOLD = 10;                                │
│                                                                     │
│      void lock() {                                                  │
│          std::unique_lock<std::mutex> lk(mtx);                      │
│          int wait_count = 0;                                        │
│          waiters.push({std::this_thread::get_id(), 0});            │
│                                                                     │
│          cv.wait(lk, [this, &wait_count] {                          │
│              auto& front = waiters.front();                         │
│              if (front.first == std::this_thread::get_id()) {       │
│                  return !locked;                                    │
│              }                                                      │
│              // Aging: 대기 횟수 증가                                  │
│              if (++front.second > AGING_THRESHOLD) {                │
│                  return !locked;  // 우선권 부여                       │ │
│              }                                                      │
│              return false;                                          │
│          });                                                        │
│                                                                     │
│          waiters.pop();                                             │
│          locked = true;                                             │
│      }                                                              │
│  };                                                                 │
│                                                                     │
│  【Linux CFS (Completely Fair Scheduler)】                            │
│  ──────────────────                                                  │
│  // 기아 방지를 위한 완전 공정 스케줄러                                   │ │
│  // 모든 태스크에 공평한 CPU 시간 분배                                    │
│                                                                     │
│  // nice 값으로 우선순위 조정                                           │
│  nice(-20 ~ +19)                                                    │
│                                                                     │
│  【기아 감지 스크립트】                                                  │
│  ──────────────────                                                  │
│  #!/bin/bash                                                        │
│  # 특정 스레드의 대기 시간 모니터링                                       │
│  while true; do                                                     │
│      cat /proc/<pid>/status | grep -E "(voluntary|nonvoluntary)"    │
│      sleep 1                                                        │
│  done                                                               │
│                                                                     │
│  【성능 측정】                                                          │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  락 타입           최대 대기 시간    기아 발생률    처리량           │ │
│  │  ────────           ────────────    ──────────    ──────         │ │
│  │  불공정 mutex       무제한           높음          높음            │ │
│  │  공정 mutex         제한됨           없음          중간            │ │
│  │  티켓 락            O(n)            없음          중간            │ │
│  │  MCS 락            O(1)            없음          높음            │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【모범 사례】                                                          │
│  ──────────────────                                                  │
│  • 기아 가능성: 불공정 락에서 모니터링                                    │
│  • 실시간: 공정 락 또는 우선순위 프로토콜                                  │
│  • 타임아웃: 무한 대기 방지                                              │
│  • 모니터링: 대기 큐 길이, 시간 추적                                      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 기대효과 및 결론

### 핵심 요약

```
• 개념: 스레드가 무기한 리소스 획득 실패
• 원인: 불공정, 우선순위, 높은 부하
• 증상: 완료 안 됨, 타임아웃, 편차
• 해결: 공정 락, aging, 타임아웃
• 도구: fairLock, MCS 락, CFS
• 감지: 큐 길이, 대기 시간 모니터링
```
