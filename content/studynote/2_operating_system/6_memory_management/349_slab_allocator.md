# 슬랩 할당자 (Slab Allocator)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 슬랩 할당자 (Slab Allocator)는 자주 할당·해제되는 커널 객체(task_struct, inode, socket 등)를 미리 초기화된 상태로 캐싱하여, 반복적인 객체 생성·파괴 비용과 단편화를 동시에 제거하는 커널 메모리 관리 기법이다.
> 2. **가치**: 커널 객체는 생성 시 초기화에 큰 비용이 드는데, 슬랩은 한 번 초기화된 객체를 재사용 풀에 유지함으로써 초기화 오버헤드를 대부분 제거하고 메모리 효율을 극대화한다.
> 3. **융합**: Linux의 슬랩(SLAB), 슬럽(SLUB, 현대 기본), 슬록(SLOB, 임베디드) 세 가지 구현이 버디 할당자 위에서 동작하며, `/proc/slabinfo` 또는 `slabtop` 명령으로 커널 슬랩 캐시 상태를 실시간 분석할 수 있다.

---

## Ⅰ. 개요 및 필요성

커널은 초당 수천~수만 번 task_struct, file, inode 같은 구조체를 할당·해제한다. 단순 `kmalloc`으로 매번 새 메모리를 할당하면:
1. 버디 시스템 검색 오버헤드 (O(log N))
2. 객체 초기화 비용 (뮤텍스 초기화, 리스트 초기화 등)
3. 크기 부정합으로 인한 내부 단편화

슬랩 할당자는 이 세 문제를 한방에 해결한다.

**💡 비유**: 커피숍에서 매번 새 잔을 만들지 않고 씻어서 재사용하는 것 — 새 잔 제조(초기화) 비용 없이 레디 상태의 잔(객체)을 즉시 제공한다.

```text
┌──────────────────────────────────────────────────────────────┐
│         슬랩 할당자 구조 (논리적 계층)                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  캐시 (Cache): 동일 타입 객체들의 집합                      │
│  예: "task_struct cache", "inode cache"                      │
│                                                              │
│  슬랩 (Slab): 1개 이상의 연속 페이지 + 객체 배열           │
│  ┌──────────────────────────────────┐                        │
│  │  [obj1][obj2][obj3]... [빈 슬롯] │  ← 1 슬랩 (1~수 페이지)│
│  └──────────────────────────────────┘                        │
│                                                              │
│  슬랩 상태:                                                  │
│  ● full   : 슬랩의 모든 객체가 할당됨                       │
│  ● partial: 일부 할당, 일부 가용 (우선 사용)               │
│  ● empty  : 모두 가용 (free 가능)                           │
│                                                              │
│  할당 흐름:                                                  │
│  요청 → partial 슬랩에서 객체 반환 (O(1))                  │
│  없으면 → empty에서 새 partial 생성                         │
│  없으면 → 버디 할당자에서 새 슬랩 할당                     │
│                                                              │
│  해제 흐름:                                                  │
│  객체 반환 → 객체 정리 (lock) → partial 슬랩에 반납         │
│  (소멸자 호출 없이 캐시 유지 — 다음 할당 시 즉시 재사용)   │
└──────────────────────────────────────────────────────────────┘
```

**📢 섹션 요약 비유**: 슬랩은 식당 조리대 — 자주 쓰는 접시(객체)를 매번 새로 만들지 않고 깨끗이 닦아서 대기시키는 재사용 시스템입니다.

## Ⅱ. 아키텍처 및 핵심 원리

### SLUB — 현대 Linux 기본 구현

Linux 2.6.22부터 SLUB이 SLAB을 대체하는 기본 구현이 되었다. SLUB의 핵심 개선:
- **슬랩 메타데이터를 객체 자체에 내장**: 별도 메타데이터 메모리 오버헤드 제거
- **Per-CPU 캐시**: 각 CPU가 로컬 캐시를 가져 Lock 경합 최소화 (멀티코어 최적화)
- **구조 단순화**: SLAB보다 코드 50% 감소, 디버깅 용이

```text
┌──────────────────────────────────────────────────────────────┐
│         SLUB Per-CPU 캐시 구조 (멀티코어 최적화)            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  CPU 0           CPU 1           CPU 2                       │
│  [로컬 슬랩]     [로컬 슬랩]     [로컬 슬랩]               │
│    │ 배분           │ 배분           │ 배분                  │
│    └───────────────┴───────────────┘                         │
│                         │                                    │
│                   [공유 슬랩 풀]                             │
│                         │                                    │
│                   버디 할당자                                │
│                                                              │
│  → CPU 0은 CPU 1의 슬랩에 직접 경쟁하지 않음               │
│  → 로컬 슬랩 소진 시에만 공유 풀 접근 (Lock 필요)          │
└──────────────────────────────────────────────────────────────┘
```

### 슬랩 진단 명령어

```bash
# 슬랩 캐시 전체 현황 (크기별 사용량)
sudo slabtop

# 상세 슬랩 정보
cat /proc/slabinfo | head -20

# 특정 캐시 확인
grep "task_struct" /proc/slabinfo
```

**📢 섹션 요약 비유**: SLUB의 Per-CPU 캐시는 마트 계산대별 소규모 재고 — 각 계산대(CPU)가 자기 재고에서 먼저 처리하고, 떨어지면 창고(공유 풀)에서 가져와요.

## Ⅲ. 융합 비교 및 다각도 분석

| 구현 | 특징 | 적합 환경 |
|:---|:---|:---|
| **SLAB** | 원조, 복잡, 강력한 디버깅 | 레거시 시스템 |
| **SLUB** | 단순, Per-CPU, 현대 기본 | 일반 서버·데스크탑 |
| **SLOB** | 초경량, 단일 리스트 | 임베디드·IoT |

**📢 섹션 요약 비유**: SLAB은 정교한 대형 식당, SLUB은 효율적인 프랜차이즈 식당, SLOB은 빠른 포장마차 — 환경에 맞는 선택이 중요합니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단

**실무 시나리오**:
1. **메모리 누수 진단**: `slabtop` 에서 특정 캐시가 계속 증가하면 해당 객체의 메모리 누수 의심.
2. **고성능 서버 튜닝**: `task_struct` 등의 슬랩 캐시 크기는 동시 프로세스 수에 비례. 메모리 부족 시 `vm.min_free_kbytes` 조정과 함께 슬랩 캐시 회수 정책 검토.

**안티패턴**:
- **슬랩 모니터링 생략**: 메모리 부족 진단 시 free 명령의 used만 보고 슬랩 캐시 사용량 놓침. `free`의 buff/cache에 슬랩도 포함됨.

**📢 섹션 요약 비유**: 슬랩 모니터링 생략은 냉장고 용량만 보고 냉동실 상태를 모르는 것 — 냉동실(슬랩)이 꽉 차면 냉장 공간도 부족해집니다.

## Ⅴ. 기대효과 및 결론

| 구분 | 단순 kmalloc | 슬랩 할당자 |
|:---|:---|:---|
| 할당 속도 | O(log N) | O(1) 거의 |
| 초기화 비용 | 매번 발생 | 재사용으로 제거 |
| 단편화 | 내부 단편화 | 최소 (객체 맞춤) |
| 다중 CPU | Lock 경합 많음 | Per-CPU 캐시로 최소화 |

슬랩은 커널의 동적 메모리 관리 효율을 수십 배 향상시키는 핵심 기법이다. SLUB이 표준화된 현대 Linux에서 버디 할당자와의 2계층 구조로 커널 메모리 관리의 완성도를 달성한다.

## 👶 어린이를 위한 3줄 비유 설명
1. 슬랩is 재활용 접시 시스템 — 커널이 자주 쓰는 그릇(객체)을 씻어서 즉시 재사용하니까 매번 새 그릇 만들 필요가 없어요!
2. 각 CPU가 자기 접시 바구니를 갖고 있어서 싸우지 않아도 돼요 — 그게 Per-CPU 캐시예요.
3. 슬랩 덕분에 커널이 초당 수천 번 객체를 만들고 지워도 속도가 엄청 빨라요!
