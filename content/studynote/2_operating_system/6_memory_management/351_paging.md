# 페이징 (Paging)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 페이징 (Paging)은 물리 메모리를 고정 크기의 프레임 (Frame)으로, 논리 메모리를 같은 크기의 페이지 (Page)로 분할하여 비연속적 물리 메모리에 프로세스를 적재하는 메모리 관리 기법이다.
> 2. **가치**: 외부 단편화 (External Fragmentation)를 원천 제거하고, 각 프로세스에 독립적 가상 주소 공간을 제공하여 프로세스 간 안전한 격리와 가상 메모리의 기반이 된다.
> 3. **융합**: 페이지 테이블, TLB (Translation Lookaside Buffer), 다단계 페이지 테이블, 역 페이지 테이블이 결합하여 현대 OS의 메모리 관리 핵심 구조를 이루며, 요구 페이징 (Demand Paging)과 COW (Copy-on-Write)의 필수 전제다.

---

## Ⅰ. 개요 및 필요성

연속 메모리 할당에서는 프로세스를 연속된 물리 메모리 공간에 올려야 하므로 외부 단편화가 필연적으로 발생한다. 예를 들어 100MB 가용 공간이 있더라도 50MB + 30MB + 20MB 세 조각으로 분산되면 60MB 프로세스가 적재될 수 없다.

페이징은 이 문제를 근본적으로 해결한다. 물리 메모리와 논리 메모리를 동일한 크기(주로 4KB)의 단위로 쪼개어, 어느 프레임에나 어느 페이지든 올릴 수 있게 한다.

**💡 비유**: 책의 내용을 포스트잇(페이지)에 분산 기록하고, 어떤 벽(물리 메모리)의 어느 빈 곳에나 붙이는 것. 페이지 테이블은 "3번 포스트잇은 남쪽 벽 7번 위치에 있어요" 라는 안내 지도.

```text
┌──────────────────────────────────────────────────────────────┐
│               페이징 주소 변환 구조                         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  논리 주소 (32비트 예시, 4KB 페이지):                       │
│  [페이지 번호 p | 오프셋 d ]                                 │
│  [ 20비트       | 12비트  ]                                  │
│                                                              │
│  변환 과정:                                                  │
│  p ──▶ 페이지 테이블[p] ──▶ 프레임 번호 f                  │
│                                                              │
│  물리 주소 = f × 4KB + d                                    │
│                                                              │
│  예시:                                                       │
│  논리 0x00004200 = Page 4, Offset 0x200                     │
│  페이지 테이블[4] = 프레임 7                                 │
│  물리 주소 = 7 × 0x1000 + 0x200 = 0x7200                   │
│                                                              │
│  [논리 메모리]       [페이지 테이블]    [물리 메모리]       │
│  Page 0 ─────────▶  Frame 3 ─────────▶  Frame 3            │
│  Page 1 ─────────▶  Frame 7 ─────────▶  Frame 7            │
│  Page 2 ─────────▶  Frame 1 ─────────▶  Frame 1            │
└──────────────────────────────────────────────────────────────┘
```

**📢 섹션 요약 비유**: 페이징은 퍼즐 조각 보관 — 퍼즐을 완성하려면 조각이 연속으로 필요 없고, 각 조각이 어디 있는지 목록(페이지 테이블)만 알면 됩니다.

## Ⅱ. 아키텍처 및 핵심 원리

### 페이지 테이블 엔트리 (PTE) 구조

```text
┌──────────────────────────────────────────────────────────────┐
│         페이지 테이블 엔트리 (PTE) 비트 구조               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [프레임 번호] [P] [M] [A] [U/S] [R/W] [PWT] [PCD] [G]     │
│                                                              │
│  P   (Present)     : 1=해당 페이지가 물리 메모리에 적재됨  │
│                      0=미적재 → 접근 시 Page Fault 발생     │
│  M   (Modified)    : 1=페이지가 수정됨 (Dirty Bit)          │
│                      교체 시 디스크 기록 필요 여부           │
│  A   (Accessed)    : 참조 여부 (교체 알고리즘 Clock에서 사용)│
│  U/S (User/Super)  : 0=커널, 1=사용자 접근 허용             │
│  R/W (Read/Write)  : 0=읽기 전용, 1=읽기+쓰기 허용         │
│  PWT (Cache Write-Through): 캐시 정책                        │
│                                                              │
│  → 이 비트들이 메모리 보호·공유·교체 정책의 하드웨어 기반!  │
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** PTE의 P (Present) 비트가 0이면 CPU가 Page Fault 인터럽트를 발생시켜 OS로 제어권이 넘어간다. OS는 디스크에서 해당 페이지를 읽어 빈 프레임에 올리고 PTE를 갱신한 뒤 실패한 명령을 재실행한다 — 이것이 요구 페이징의 핵심 동작이다.

### TLB (Translation Lookaside Buffer) 가속

모든 메모리 접근(데이터·명령어)마다 페이지 테이블을 조회하면 메모리 접근이 2배 느려진다. TLB는 최근 사용한 페이지-프레임 매핑을 캐싱하는 하드웨어 연관 메모리다.

```text
┌──────────────────────────────────────────────────────────────┐
│         TLB 포함 실질 메모리 접근 시간 (EAT)               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  α = TLB 적중률  (보통 99% 이상)                            │
│  t = TLB 접근 시간  (약 1ns)                                │
│  m = DRAM 접근 시간 (약 100ns)                              │
│                                                              │
│  EAT = α × (t + m) + (1-α) × (t + 2m)                     │
│       = t + m + (1-α) × m                                   │
│                                                              │
│  TLB 적중률 99%, t=1ns, m=100ns:                            │
│  EAT = 1 + 100 + 0.01 × 100 = 102ns                        │
│  (TLB 없이 페이지 테이블: 200ns, 약 2배 느림)              │
│                                                              │
│  결론: 99% 적중률 TLB → EAT를 200ns → 102ns으로 절반!     │
└──────────────────────────────────────────────────────────────┘
```

**📢 섹션 요약 비유**: TLB는 자주 가는 장소의 주소를 외워두는 것 — "마트는 골목 3번이지"를 외우면 매번 지도(페이지 테이블)를 볼 필요가 없어요.

## Ⅲ. 융합 비교 및 다각도 분석

### 단일 레벨 vs 다단계 페이지 테이블

32비트 주소 공간(4GB), 4KB 페이지 시 페이지 테이블 크기:
- 페이지 수: 4GB / 4KB = 1,048,576 개
- 각 엔트리 4바이트: 4MB의 페이지 테이블 (프로세스마다!)

64비트 주소 공간이면 이 크기가 조조겠지만, **다단계 페이지 테이블**이 해결한다.

| 방식 | 장점 | 단점 |
|:---|:---|:---|
| 1단계 | 변환 1회 | 테이블 크기 큼 (4MB/프로세스) |
| 2단계 (x86) | 희소 주소공간에 효율 | 변환 2회 (TLB 미스 시 느림) |
| 4단계 (x86-64) | 48비트 주소 지원 | 변환 4회 (TLB의 중요성 ↑↑) |
| 역 페이지 테이블 | 전체 시스템에 단 1개 | 검색에 해시/탐색 필요 |

**📢 섹션 요약 비유**: 다단계 페이지 테이블은 주소록을 지역→동→건물→호실로 나누는 것 — 필요한 페이지만 테이블에 만들어서 공간을 절약합니다.

## Ⅳ. 실무 적용 및 기술사적 판단

**실무 시나리오**:
1. **공유 페이지 (Shared Pages)**: 여러 프로세스가 같은 libc.so 코드 페이지를 공유. 각 프로세스의 페이지 테이블이 같은 물리 프레임을 가리킴 → 메모리 절약.
2. **COW (Copy-on-Write)**: fork() 시 부모·자식이 같은 프레임 공유(읽기 전용). 쓰기 시도 → Page Fault → OS가 별도 프레임 복사 → 수정 허용. 메모리 복사 최소화.

**안티패턴**:
- **페이지 크기 과대**: 4MB Huge Page는 TLB 엔트리 효율이 높지만 내부 단편화 증가. 소용량 데이터에는 낭비.
- **TLB 플러시 과도**: 컨텍스트 스위치 시 TLB 전체 플러시는 비용 큼 → ASID (Address Space Identifier) 사용으로 플러시 최소화.

**📢 섹션 요약 비유**: COW는 가족이 사진 앨범을 공유하다가 한 명이 낙서하려 할 때만 그 사람 몫의 복사본을 만들어주는 방식 — 낙서를 안 하면 한 앨범 공유로 비용 절약!

## Ⅴ. 기대효과 및 결론

| 구분 | 연속 메모리 할당 | 페이징 |
|:---|:---|:---|
| 외부 단편화 | 발생 | 없음 |
| 내부 단편화 | 없음 | 페이지 내 마지막 조각 |
| 주소 변환 | 단순 (base+offset) | 페이지 테이블 (TLB 가속) |
| 프로세스 격리 | 경계 레지스터 | 페이지 테이블 분리 |
| 가상 메모리 지원 | 불가 | 가능 (요구 페이징) |

페이징은 현대 OS 메모리 관리의 핵심이다. 외부 단편화 제거, 주소 공간 격리, 가상 메모리 지원을 동시에 달성하는 유일한 기법으로, TLB의 하드웨어 가속과 함께 거의 모든 현대 CPU·OS에서 채택한다.

## 👶 어린이를 위한 3줄 비유 설명
1. 페이지는 책 페이지처럼 고정 크기 조각 — 도서관(메모리) 어느 서가(프레임)에나 꽂을 수 있어요.
2. 페이지 테이블은 책 목차 — "2장은 15번 서가에 있어요"처럼 논리 번호(페이지)와 물리 위치(프레임)를 연결해줘요.
3. TLB는 자주 찾는 서가 주소를 외워두는 것 — 매번 목차를 뒤지지 않아도 되니 훨씬 빠르게 책을 꺼낼 수 있어요!
