# 분산 공유 메모리 (Distributed Shared Memory, DSM)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 공유 메모리 (DSM)는 물리적으로 분리된 여러 컴퓨터(노드)의 로컬 메모리를 네트워크로 엮어, **개발자에게는 마치 1대의 거대한 단일 공유 메모리(Shared Memory)가 있는 것처럼 속이는(추상화하는) 가상화 기술**이다.
> 2. **가치**: 메시지 패싱(Message Passing)을 위해 `send()`, `receive()` 코드를 짜야 하는 분산 프로그래밍의 지옥 같은 난이도를 해결하고, 개발자가 **일반적인 멀티스레드 코딩(변수 읽기/쓰기) 방식 그대로 클러스터 환경에서 프로그래밍할 수 있게 해 준다.**
> 3. **융합**: 하지만 네트워크 지연(Latency)과 데이터 일관성(Consistency) 유지 오버헤드라는 치명적 단점 때문에 OS 커널 레벨의 순수 DSM은 사장되었고, 현대에는 **Redis 같은 인메모리 데이터 그리드(IMDG)**나 아파치 스파크(Spark)의 RDD 형태로 애플리케이션 레벨에서 융합되어 부활했다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 여러 대의 컴퓨터가 각자의 RAM을 가지고 있지만, 운영체제나 미들웨어가 이 RAM들을 논리적으로 하나의 거대한 주소 공간(Address Space)으로 매핑하여, A 컴퓨터가 B 컴퓨터의 RAM에 있는 변수를 로컬 변수처럼 읽고 쓸 수 있게 해주는 구조다.
- **필요성**: 분산 시스템은 기본적으로 "메시지 패싱" 모델이다. 내가 1번 컴퓨터에 있고 2번 컴퓨터의 데이터를 바꾸려면, 네트워크 소켓을 열고 포맷을 맞춰 데이터를 쏜 다음 ACK를 기다려야 한다. 이 코딩은 너무 복잡하고 고통스러웠다. 개발자들은 "그냥 `int shared_var = 1;` 해놓고 100대의 컴퓨터가 같이 쓰게 해 주면 안 돼?"라고 외쳤고, 이 로망을 실현하기 위해 탄생했다.
- **💡 비유**: 각자 자기 집에 자기 냉장고를 가진 100명의 사람이 요리를 할 때, 남의 집 재료가 필요하면 전화를 걸어 오토바이로 배달(메시지 패싱)시키는 게 아니라, **'어떤 마법의 문을 열면 100대의 냉장고가 하나로 연결된 거대한 공용 냉장고(DSM)가 나타나, 그냥 손만 뻗으면 남의 집 파를 꺼내 쓸 수 있는 환상'**과 같다.
- **등장 배경**: 1980년대 후반 IVY, Munin 등의 분산 OS 프로젝트에서, 멍청한 SMP(대칭형 다중 처리) 프로그램을 분산 환경으로 코드 수정 1줄 없이 포팅(Porting)하기 위한 궁극의 단일 시스템 이미지(SSI) 달성 목표로 연구되었다.

```text
  [메시지 패싱 (Message Passing) vs 분산 공유 메모리 (DSM) 의 프로그래밍 모델 차이]

  [ 1. 메시지 패싱 (현재 분산 환경의 표준) ]
  - 노드 A: `send(Node_B, "변수 X를 10으로 바꿔줘")`
  - 노드 B: 메시지 수신 ─▶ 패킷 파싱 ─▶ `X = 10;` ─▶ `reply(Node_A, "완료")`
  ▶ 코드가 복잡하고, 타임아웃/에러 처리를 다 해줘야 함. (짜증 폭발)

  [ 2. 분산 공유 메모리 (DSM) 의 마법 ]
  - 노드 A: `X = 10;` (그냥 냅다 일반 C언어 변수 쓰듯이 대입함)
  - DSM 미들웨어: (A가 X를 건드린 걸 Page Fault로 낚아챔) ─▶ 
                 (몰래 B에게 네트워크로 변경값 전송) ─▶ (B의 메모리 X값 업데이트)
  ▶ 개발자는 네트워크의 존재를 1%도 모른 채, 그냥 동네 PC에서 코딩하듯 짬.
```
**[다이어그램 해설]** DSM의 핵심 철학은 **"네트워크 통신의 은닉"**이다. 개발자에게는 완벽한 환상(Illusion)을 주지만, 밑바닥의 OS는 그 환상을 유지하기 위해 A가 변수에 접근할 때마다 뒤에서 미친 듯이 패킷을 쏘고 받으며 똥꼬쇼를 해야 한다. 이 똥꼬쇼의 비용(오버헤드)이 DSM의 운명을 결정지었다.

- **📢 섹션 요약 비유**: 해외 주식을 살 때, 예전엔 미국 증권사에 전화를 걸어 달러로 환전하고 영어로 주문(메시지 패싱)을 넣어야 했습니다. 지금은 토스증권(DSM) 앱에서 원화로 "매수" 버튼 하나만 누르면, 토스가 뒤에서 몰래 환전하고 미국에 주문을 넣는 복잡한 과정을 투명하게 다 처리해 줍니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### DSM의 데이터 이동 원리: 가상 메모리(페이징)의 역이용

DSM은 도대체 남의 컴퓨터에 있는 변수를 어떻게 내 것처럼 쓰게 만들까? 운영체제의 **페이지 폴트(Page Fault)** 매커니즘을 해킹(역이용)했다.

1. **글로벌 주소 매핑**: 10대의 컴퓨터가 각자 1GB의 램을 가졌다면, OS는 10GB짜리 거대한 가상 주소 테이블을 모든 노드에 똑같이 만들어 뿌린다.
2. **함정(Invalid) 설치**: 노드 A의 램에는 실제 0~1GB 데이터만 있다. 노드 A가 5GB 번지(노드 C에 있는 데이터)를 읽으려 하면, 페이지 테이블에는 `Invalid(권한 없음)`로 설정되어 있어 **페이지 폴트(Page Fault)**가 빵 터진다.
3. **네트워크 Page Fetch**: 일반 OS라면 디스크(Swap)로 가겠지만, DSM OS는 디스크 대신 **네트워크 랜카드**를 타고 노드 C에게 "야! 그 페이지 통째로 복사해서 나한테 보내!"라고 요청한다.
4. **로컬 매핑 및 재실행**: 노드 C가 보내준 페이지를 노드 A의 빈 램에 꽂아주고, `Valid`로 바꾼 뒤 멈췄던 코드를 다시 실행시킨다.

### DSM의 치명적 독약: 스래싱 (Thrashing)
이 마법은 완벽해 보였지만, 치명적인 물리적 한계에 부딪혔다. 네트워크 지연(Network Latency)이다.
- 디스크에서 페이지를 가져오는 것도 10ms가 걸려 죽을 맛인데, 이더넷(Ethernet) 네트워크를 타고 남의 컴퓨터 램에서 페이지를 뜯어오는 데는 라우터 스위칭을 거쳐 수십~수백 ms가 걸린다.
- **거짓 공유(False Sharing)의 스케일업**: 만약 노드 A와 노드 B가 같은 페이지(4KB) 안에 있는 서로 다른 변수를 1초에 1,000번씩 수정(Write)한다면? 페이지 하나가 랜선을 타고 A와 B 사이를 초당 1,000번씩 탁구공처럼 핑퐁 치며 돌아다닌다. 네트워크 대역폭이 완전히 터져버리며 전체 클러스터가 0.1 TPS로 굳어버리는 우주급 스래싱이 터진다.

- **📢 섹션 요약 비유**: 옆 부서(다른 노드) 서류를 내 책상에 가져다 놓고 펜으로 한 글자 썼습니다. 근데 옆 부서 직원이 "나도 그 서류 다른 줄에 글자 하나 써야 해!" 하고 퀵서비스(네트워크)로 서류를 뺏어갑니다. 서로 한 글자 쓸 때마다 퀵서비스로 서류를 주고받으면 배달비(네트워크 지연) 때문에 회사가 파산합니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 일관성 모델 (Consistency Models)의 타협

네트워크 스래싱을 막으려면 결국 "어차피 핑퐁 칠 거, 내가 변수 1로 바꾼 걸 B한테 당장 안 알려주고 좀 늦게 알려주면 안 될까?"라는 꼼수가 필요해졌다. 이것이 바로 분산 시스템을 지배하는 **일관성(Consistency)의 완화**다.

| 일관성 모델 | 룰 (Rule) | 장점 | 단점 (문제점) |
|:---|:---|:---|:---|
| **엄격한 일관성 (Strict / Sequential)** | A가 쓰면, **빛의 속도(0초)로** B도 즉시 바뀐 값을 봐야 함 | 로컬 PC처럼 완벽하게 코딩 가능 | 네트워크 환경에서 물리적으로 100% 불가능 (아인슈타인 상대성 이론 위배) |
| **릴리즈 일관성 (Release)** | 내가 변수를 막 바꿀 땐 B한테 안 알려주다가, **내가 락을 풀 때(Release)만 동기화**해서 B한테 변경분을 쫙 쏨 | 네트워크 트래픽 90% 감소. 스래싱 극복! | 코딩할 때 무조건 락(Lock)을 걸고 풀어야만 통신이 된다는 제약 발생 |
| **결과적 일관성 (Eventual)** | 언젠가는(Eventual) B도 바뀐 값을 알게 되겠지~ 하고 던짐 | 성능 최강 (No Lock, No Sync) | B가 한동안 옛날(과거) 쓰레기 데이터를 읽고 사고 칠 수 있음 |

90년대 학자들은 완벽한 Strict Consistency를 DSM에 구현하려다 네트워크가 터져서 망했다. 결국 실무 진영에서는 완벽함을 버리고 Release나 Eventual 일관성을 채택함으로써 성능과 분산 공유의 타협점을 찾아냈다. (이것이 현대 NoSQL 데이터베이스의 근간 철학이다).

- **📢 섹션 요약 비유**: Strict 일관성은 내가 일기장에 글을 쓰자마자 전 세계 모든 사람의 일기장이 동시에 타이핑되는 마법입니다(불가능). Release 일관성은 내가 일기를 1시간 동안 막 휘갈겨 쓴 뒤, "발행(Release)!" 버튼을 누를 때만 복사본이 전 세계로 날아가는 현실적이고 효율적인 출판 시스템입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오
1. **OS 커널 레벨 DSM의 장례식**: 
   - 2026년 현재, 리눅스 커널이나 윈도우 안에 "네트워크 너머의 램을 내 램처럼 매핑해 주는" DSM 기능은 아예 삭제되었거나 존재하지 않는다. 
   - **이유**: OS 커널이 4KB 단위의 페이지 폴트를 잡아서 이더넷으로 쏘는 행위는 너무나 무식하고 컨트롤이 불가능한 블랙박스이기 때문이다. 개발자가 `int a = 1;` 쳤는데 핑이 튀어서 1초간 멈추는 OS는 아무도 쓰고 싶어 하지 않았다.
2. **어플리케이션 레벨의 IMDG (In-Memory Data Grid) 부활**: 커널은 버렸지만, Java와 C++ 백엔드 아키텍트들은 이 아름다운 철학을 애플리케이션 힙(Heap) 메모리 단으로 끌어올렸다.
   - **Hazelcast, Apache Ignite (Redis Cluster)**: 이 툴들은 여러 대의 자바 서버 힙(Heap)을 네트워크로 묶어 거대한 1개의 `Map` 이나 `List` 객체처럼 쓰게 해 준다. 
   - **아키텍트의 설계**: 서버 A에서 `sharedMap.put("user", "Alice")`를 하면, Hazelcast 미들웨어가 알아서 백그라운드에서 직렬화(Serialization)를 거쳐 서버 B와 C의 메모리에 복제해 둔다. 커널의 무식한 4KB 페이지 단위가 아니라, 의미 있는 '객체(Object) 단위'로 통신하기 때문에 네트워크 낭비가 없고 거짓 공유(False Sharing)도 발생하지 않는 현대판 DSM의 완벽한 승리다.

```text
  ┌────────────────────────────────────────────────────────┐
  │     거대 데이터 공유 아키텍처 설계: RPC vs IMDG(현대판 DSM) │
  ├────────────────────────────────────────────────────────┤
  │                                                        │
  │   [요구사항: 100대의 게임 서버가 유저의 실시간 위치를 0.1초 내로 공유]│
  │                                                        │
  │   [ ❌ 아키텍처 1: 순수 메시지 패싱 (RPC/gRPC) ]          │
  │     - 구조: 유저가 움직이면 99대 서버에 브로드캐스트 RPC를 쏨. │
  │     - 결과: 100대 * 99대 = N^2 네트워크 폭발. 통신 로직 짜다 개발자 과로사.│
  │                                                        │
  │   [ ✅ 아키텍트의 정답: In-Memory Data Grid (IMDG) ]        │
  │     - 구조: 100대의 램을 묶어 거대한 가상 Redis/Hazelcast 맵 생성.│
  │     - 개발: `GlobalMap.put(myUser, newPosition);` 1줄 작성 끝!│
  │     - 효과: 개발자는 단일 서버에서 코딩하듯 편하게 짜고, 데이터 복제와 │
  │             락(Distributed Lock)은 프레임워크가 뒤에서 완벽히 처리함.│
  └────────────────────────────────────────────────────────┘
```
**[다이어그램 해설]** "분산 시스템을 분산 시스템처럼 보이지 않게 짜는 것"이 최고 티어 아키텍트의 목표다. 코딩할 때마다 "이게 B 서버로 가야 하니까 소켓 열고 JSON 파싱하고..."라는 잡무(Boilerplate)를 쳐내지 못하면 비즈니스 로직에 집중할 수 없다. DSM의 철학은 죽지 않고 IMDG라는 이름으로 백엔드 메모리 아키텍처의 필수 인프라가 되었다.

- **📢 섹션 요약 비유**: 옛날 팩스(RPC) 시대에는 문서를 보내려면 직접 팩스기에 종이를 넣고 번호를 눌러야 했습니다. 지금은 구글 독스(IMDG, 현대판 DSM) 시대입니다. 내가 내 노트북에서 타이핑만 하면, 저 멀리 미국에 있는 동료의 화면에도 0.1초 만에 똑같이 글자가 나타납니다. 우리는 네트워크를 의식하지 않고 그냥 문서를 편집할 뿐입니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 기대효과
분산 공유 메모리(또는 IMDG) 사상을 적용하면, 코드를 멀티코어 단일 서버 환경에서 분산 클러스터 환경으로 마이그레이션 할 때 코드 수정(Refactoring) 비용을 1/10 수준으로 줄일 수 있으며, 스케일 아웃(Scale-out) 시 선형적인 메모리 확장을 획득할 수 있다.

### 결론 및 미래 전망
DSM(Distributed Shared Memory)은 "원격(Remote)을 로컬(Local)처럼 속이겠다"는 컴퓨터 공학의 야심 찬 도전이었다. 물리학적 한계(네트워크 지연) 탓에 OS 커널 바닥에서는 쫓겨났지만, 데이터 그리드(IMDG), 분산 캐시(Memcached), 블록체인(분산 원장) 등 상위 레이어에서 화려하게 꽃을 피웠다.
미래에는 하드웨어의 미친 발전이 이 판도를 다시 뒤엎고 있다. **CXL (Compute Express Link)**과 **RDMA (Remote Direct Memory Access)** 기술이 대중화되면서, CPU가 네트워크 OS 스택(TCP/IP)을 거치지 않고 랜선을 통해 **원격 서버의 RAM에 직접 나노초 단위로 읽기/쓰기를 박아버리는 하드웨어적 DSM 시대**가 열리고 있다. 30년 전 학자들이 꿈꾸던 진정한 의미의 '빛의 속도 분산 메모리'가 실리콘 칩 레벨에서 완성되어 가고 있는 것이다.

- **📢 섹션 요약 비유**: DSM은 1990년대에 "자동차(데이터)가 하늘을 날게 하겠다"는 허무맹랑한 만화책이었습니다. 무거워서 못 날았죠. 하지만 지금 우리는 드론과 UAM(CXL, RDMA 기술)이라는 미친 엔진을 달고 실제로 하늘을 나는 자동차를 타고 출퇴근하는 시대를 눈앞에 두고 있습니다. 상상은 언제나 현실이 됩니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **단일 시스템 이미지 (SSI)** | DSM이 구현하고자 했던 궁극의 이상향. 100대의 메모리를 1대처럼 속이는 마법. |
| **메시지 패싱 (Message Passing)** | DSM의 영원한 라이벌. 내가 남의 메모리를 건드릴 수 없으니 쪽지를 던지는 번거롭지만 확실한 방식이다. |
| **페이지 폴트 (Page Fault)** | OS 레벨의 DSM이 남의 메모리를 뜯어오기 위해 치사하게 역이용한 인터럽트 덫(Trap)이다. |
| **캐시 일관성 (Cache Coherence)** | DSM 환경에서 100대가 같은 데이터를 들고 있을 때, 한 놈이 수정하면 나머지를 다 버리게 만드는 통신 오버헤드의 주범이다. |
| **RDMA (Remote Direct Memory)** | 소프트웨어 DSM의 느린 속도(TCP/IP)를 박살 내기 위해, 랜카드가 직접 남의 램을 찔러버리는 미래 하드웨어 기술이다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 10명의 친구가 각자 자기 스케치북(RAM)에 그림을 그리고 있어요. 친구의 스케치북에 그림을 그려주려면 걸어가서 빌려와야(메시지 패싱) 해서 너무 귀찮았어요.
2. 그래서 **분산 공유 메모리(DSM)**라는 거대한 '마법의 투명 스케치북' 하나를 10명의 책상 위에 홀로그램처럼 띄워줬어요.
3. 이제 친구들은 걸어갈 필요 없이 각자 자기 자리에 앉아서 허공(투명 스케치북)에 쓱쓱 색칠하면, 10명 모두의 진짜 스케치북에 그림이 똑같이 그려지는 엄청난 마술이 일어난답니다!
