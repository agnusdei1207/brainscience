# 쓰기 시 복사 (COW, Copy-on-Write)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 쓰기 시 복사 (COW, Copy-on-Write)는 fork() 시 부모·자식 프로세스가 동일한 물리 페이지를 읽기 전용으로 공유하다가, 어느 한쪽이 실제로 **쓰기를 시도할 때만** OS가 새 물리 페이지를 복사·할당하여 독립 공간을 만드는 Lazy Copy 최적화 기법이다.
> 2. **가치**: fork() 직후 exec()를 호출하는 Unix 전통 패턴에서 COW 없이 부모 메모리 전체를 복사하면 수백MB가 낭비된다. COW로 실제 페이지 복사는 수십~수백 페이지로 최소화되어 fork() 성능이 극적으로 향상된다.
> 3. **융합**: COW는 fork/exec 뿐 아니라 Redis BGSAVE (스냅샷), 파일 시스템 스냅샷 (Btrfs, ZFS), 컨테이너 이미지 레이어 (OverlayFS), Kubernetes Pod 복제, 함수형 언어의 영속 자료구조에 이르기까지 소프트웨어 전반의 핵심 최적화 패턴이다.

---

## Ⅰ. 개요 및 필요성

Unix에서 `fork()`는 부모 프로세스의 정확한 복사본을 생성한다. 만약 부모가 500MB 메모리를 사용 중이라면, 단순 복사 방식은 fork()마다 500MB를 초기화 + 복사해야 한다.

하지만 fork() 직후 대부분은 `exec()`로 완전히 다른 프로그램을 적재하므로, 복사된 500MB 중 극히 일부만 실제로 사용된다 → 완전 복사는 명백한 낭비.

**💡 비유**: 요리책을 복사하는데, 실제로 사용할 레시피는 3쪽뿐이라면? COW는 책을 공유하고, 3쪽을 수정할 때만 그 3쪽만 복사해서 쓰는 방식이다.

```text
┌──────────────────────────────────────────────────────────────┐
│         COW fork() 동작 흐름                                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [fork() 직후]                                              │
│  부모 페이지 테이블 ──▶ [물리 프레임 A: 코드]              │
│  자식 페이지 테이블 ──▶ [물리 프레임 A: 코드] (공유!)      │
│                          ↑ 두 PTE 모두 Read-Only 설정       │
│                                                              │
│  [자식이 프레임 A에 쓰기 시도]                              │
│  MMU: Read-Only 위반 → Protection Fault! (Minor Fault)      │
│                                                              │
│  [OS COW 핸들러]:                                           │
│  1. 새 물리 프레임 B 할당                                   │
│  2. 프레임 A 내용을 B로 복사                                │
│  3. 자식 PTE: 프레임 B, Read-Write로 변경                   │
│  4. 부모 PTE: 아직 프레임 A, 이제 Read-Write 허용          │
│  5. 자식 쓰기 재실행                                        │
│                                                              │
│  결과: 자식은 독립 B, 부모는 원본 A — 서로 독립!           │
└──────────────────────────────────────────────────────────────┘
```

**📢 섹션 요약 비유**: COW fork()는 게임 세이브 파일 복사 — 게임을 분기할 때 파일을 복사하는 대신 공유하다가, 실제로 진행(쓰기)할 때만 그 부분만 복사합니다. 아직 분기된 파트로 안 갔다면 복사할 필요도 없어요.

## Ⅱ. 아키텍처 및 핵심 원리

### vfork() — COW조차 없는 초경량 fork

```bash
vfork():
  - 자식이 exec()를 호출할 때까지 부모의 주소 공간을 직접 공유
  - 자식이 실행되는 동안 부모는 정지됨
  - 메모리 복사 0, 페이지 테이블 복사도 없음
  - 위험: 자식이 실수로 부모 스택을 수정하면 부모 파괴!
  - 현재: exec()가 목적일 때만 사용권장
```

### Redis BGSAVE와 COW

```text
┌──────────────────────────────────────────────────────────────┐
│         Redis BGSAVE — COW 기반 스냅샷                     │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [메인 프로세스: 계속 클라이언트 요청 처리]                 │
│  [자식 (fork)]:                                             │
│  ● DB 메모리를 읽기 전용으로 공유                          │
│  ● 디스크에 RDB 파일로 직렬화·저장                         │
│                                                              │
│  메인 프로세스가 키를 수정하면:                             │
│  → COW로 해당 페이지만 복사됨                              │
│  → 자식은 fork 시점의 스냅샷(원본 데이터)을 읽음           │
│  → 스냅샷 일관성 완벽 보장!                                 │
│                                                              │
│  단점: 쓰기 부하가 높을 때 COW 페이지 복사 급증            │
│       → RSS(Resident Set Size) 최대 2배까지 증가 가능      │
└──────────────────────────────────────────────────────────────┘
```

**📢 섹션 요약 비유**: Redis BGSAVE는 도서관 책 스캔 — 스캔하는 동안 책을 빌려줄 나눠야 하니까, 바뀐 부분만 복사본을 만들어 스캔팀에게 주는 거예요.

## Ⅲ. 융합 비교 및 다각도 분석

### COW의 응용 범위

| 응용 | COW 대상 | 이점 |
|:---|:---|:---|
| fork/exec | 프로세스 메모리 페이지 | fork 속도 99% 향상 |
| Redis BGSAVE | DRAM의 키-값 데이터 | 무중단 스냅샷 |
| Btrfs/ZFS 스냅샷 | 파일시스템 데이터 블록 | 瞬间 스냅샷 생성 |
| Docker OverlayFS | 이미지 레이어 | 컨테이너 수백개 동시 운영 |
| 함수형 언어 (Clojure PersistentMap) | 불변 자료구조 버전 | 여러 버전 효율적 공유 |

**📢 섹션 요약 비유**: COW는 만능 레시피 — 프로세스, DB, 파일시스템, 컨테이너 모두 "공유하다가 수정 시 복사" 원칙 하나로 거대한 효율 향상을 달성합니다.

## Ⅳ. 실무 적용 및 기술사적 판단

**실무 시나리오**:
1. **Kubernetes Pod 스케일 아웃**: 동일 이미지로 여러 Pod 생성 시 OverlayFS COW 레이어로 이미지 데이터 공유 → 수십 Pod도 디스크 N배 불필요.
2. **Ruby on Rails 멀티 Worker (Unicorn)**: 마스터 프로세스가 앱 코드 적재 후 fork() → COW로 수십 Worker가 코드 페이지 공유 → 메모리 절감.

**안티패턴**:
- **Redis 대용량 쓰기 중 BGSAVE**: 초당 수백만 키가 수정되는 Redis에서 BGSAVE 시 COW 페이지 복사 폭증 → RSS 2배 증가 → OOM 위험. BGSAVE 스케줄 신중히 설계.

**📢 섹션 요약 비유**: 대용량 Redis BGSAVE는 요리사(메인)가 바쁘게 요리하는 중에 주방 전체 사진 찍기 — 사진 찍는 동안 바뀐 부분은 계속 복사해야 해서 공간(메모리)이 많이 필요해요.

## Ⅴ. 기대효과 및 결론

| 구분 | 단순 복사 fork() | COW fork() |
|:---|:---|:---|
| fork() 시간 | O(메모리 크기) | O(1) 가까움 |
| 즉시 메모리 사용 | 2배 | 거의 동일 |
| exec() 전 실제 복사 | 전체 | 없음 |
| exec() 후 실제 복사 | 불필요 (폐기) | 없음 |

COW는 메모리 복사 비용을 실제 쓰기 발생 순간으로 지연하는 Lazy Evaluation의 핵심 응용이다. fork/exec, DB 스냅샷, 컨테이너, 함수형 자료구조까지 COW는 현대 소프트웨어 스택 전반에 걸쳐 효율과 일관성을 동시에 제공한다.

## 👶 어린이를 위한 3줄 비유 설명
1. COW는 게임 세이브파일 공유 — 같은 파일을 여러 명이 보다가, 실제로 진행하는 사람만 그 부분을 복사해서 따로 저장해요.
2. fork()는 "아직 아무것도 안 바뀌면 공유해도 돼요!" — 서로 다른 부분이 생길 때만 분리됩니다.
3. Redis도 이 방법으로 저장하는 동안 계속 서비스할 수 있어요 — 바뀐 데이터만 살짝 복사하면서 스냅샷을 찍는 거예요!
