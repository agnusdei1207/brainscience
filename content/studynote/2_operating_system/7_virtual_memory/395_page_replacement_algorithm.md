# 페이지 교체 알고리즘 (Page Replacement Algorithm)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 페이지 교체 알고리즘 (Page Replacement Algorithm)은 물리 메모리의 모든 프레임이 사용 중일 때 Page Fault가 발생하면, 어떤 기존 페이지를 스왑 아웃(디스크로 내보낼)해야 Page Fault 총 횟수를 최소화할 수 있는지를 결정하는 정책이다.
> 2. **가치**: 최적 교체 (OPT)는 미래 참조를 알 수 없어 구현 불가지만 이론적 하한을 제공하고, LRU (Least Recently Used)는 시간 지역성을 반영하여 실무적으로 OPT에 근접한 성능을 보이지만 정확한 구현에 하드웨어 지원이 필요하다.
> 3. **융합**: 실제 OS에서는 하드웨어 접근 비트 (Accessed/Referenced Bit)와 변경 비트 (Dirty/Modified Bit)를 활용한 Clock 알고리즘 (NRU 변형)이 LRU 근사치로 사용되며, Linux의 Active/Inactive 리스트 기반 2차 기회 알고리즘이 현대 표준이다.

---

## Ⅰ. 개요 및 필요성

요구 페이징에서 물리 메모리가 꽉 찼을 때 새 페이지를 올리려면 기존 페이지 중 하나를 내보내야 한다. 이때 어떤 페이지를 선택하느냐에 따라 이후의 Page Fault 횟수가 크게 달라진다.

**💡 비유**: 책상(물리 메모리)이 꽉 찼을 때 새 책(페이지)을 올리려면 어떤 책을 서가(디스크)에 돌려보낼지 결정하는 것 — 가장 오래 안 볼 책을 돌려보내야 다음에 또 꺼내야 하는 횟수가 줄어든다.

```text
┌──────────────────────────────────────────────────────────────┐
│         주요 페이지 교체 알고리즘 비교                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  OPT (Optimal):                                             │
│  ● 미래에 가장 오래 사용 안 될 페이지 교체                 │
│  ● Page Fault 최소 (이론적 하한)                           │
│  ● 미래를 알 수 없으므로 구현 불가 — 비교 기준용          │
│                                                              │
│  FIFO (First-In, First-Out):                               │
│  ● 가장 오래된(먼저 들어온) 페이지 교체                    │
│  ● 구현 단순 (큐)                                           │
│  ● 벨라디 역설: 프레임 수↑ → Page Fault↑ 가능 (이상현상) │
│                                                              │
│  LRU (Least Recently Used):                                 │
│  ● 가장 오랫동안 참조 안 된 페이지 교체                    │
│  ● 시간 지역성 반영, OPT에 근접                            │
│  ● 정확한 구현: 타임스탬프/스택 → 오버헤드 큼             │
│                                                              │
│  LFU (Least Frequently Used):                               │
│  ● 참조 횟수 가장 적은 페이지 교체                         │
│  ● 과거 빈도 반영하나 최근성 무시 → 성능 열위             │
│                                                              │
│  Clock (NRU 근사):                                          │
│  ● 참조 비트(Reference Bit) 활용                            │
│  ● 시계 바늘 방식으로 R=0인 페이지 교체                    │
│  ● 하드웨어 지원 필요 없는 LRU 근사                       │
│  ● Linux, 대부분 OS의 실제 구현 기반                       │
└──────────────────────────────────────────────────────────────┘
```

**📢 섹션 요약 비유**: 책상 정리 방법 — OPT는 미래 독심술로 언제 쓸지 알고 가장 나중에 쓸 책부터 치우는 것. FIFO는 가장 오래된 책 먼저. LRU는 가장 오래 안 읽은 책 먼저. Clock은 최근에 읽었는지 표시를 보면서 결정.

## Ⅱ. 아키텍처 및 핵심 원리

### Clock 알고리즘 (2차 기회, Second Chance)

```text
┌──────────────────────────────────────────────────────────────┐
│         Clock 알고리즘 동작 — 시계 바늘 방식                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  원형 버퍼 (프레임들):                                      │
│  [A:R=1] → [B:R=0] → [C:R=1] → [D:R=0] → [A...]          │
│              ↑                               ↑              │
│           시계 바늘                       다시 A           │
│                                                              │
│  Page Fault 발생 시 교체 대상 탐색:                         │
│  ① 현재 바늘 위치의 R 비트 확인                           │
│  ② R=1이면: R=0으로 초기화 + 바늘 다음으로 이동 (기회 1회)│
│  ③ R=0이면: 이 페이지 교체! (최근에 참조 안 됨)           │
│                                                              │
│  Enhanced Clock (변경 비트 추가):                           │
│  교체 우선순위:                                              │
│  1순위: R=0, M=0 (미참조, 미수정) — 이상적 희생 페이지    │
│  2순위: R=0, M=1 (미참조, 수정됨) — I/O 후 교체           │
│  3순위: R=1, M=0 (참조됨, 미수정)                          │
│  4순위: R=1, M=1 (참조됨, 수정됨) — 최후                  │
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 시계 바늘이 한 바퀴 돌 때 모든 R=1 비트가 0으로 초기화된다. 따라서 최근에 전혀 참조되지 않은 페이지(R=0 상태 유지)가 교체 대상이 된다. 이것이 LRU의 근사치인 이유다.

### 변경 비트 (Dirty Bit)

페이지 교체 시 Modified=1(수정된 페이지)이면 디스크에 Write-Back 필요 → I/O 2회 발생. Modified=0이면 그냥 버리면 됨(디스크에 최신본 있으므로).

**📢 섹션 요약 비유**: Clock은 시계 검사 — 시계 바늘이 돌면서 최근에 쓴 책에는 표시를 지우고, 표시 없는 책을 발견하면 서가로 돌려보냅니다.

## Ⅲ. 융합 비교 및 다각도 분석

### Linux의 페이지 교체 — Active/Inactive 리스트

Linux는 단순 Clock 대신 **2단계 LRU 근사**를 사용한다:
- **Active 리스트**: 최근 2번 이상 참조된 핫 페이지
- **Inactive 리스트**: 1번 이하 참조된 콜드 페이지

교체 시 Inactive 리스트 꼬리에서 먼저 제거. Active→Inactive 순서는 참조 비트 기반으로 주기적 검사.

**📢 섹션 요약 비유**: Active/Inactive 리스트는 책상 2단 카트 — 자주 쓰는 책은 위 칸(Active), 가끔 쓰는 책은 아래 칸(Inactive). 책상이 꽉 차면 아래 칸에서 먼저 서가로 보냅니다.

## Ⅳ. 실무 적용 및 기술사적 판단

**실무 시나리오**:
1. **DB 버퍼 풀 교체 정책**: MySQL InnoDB는 자체 LRU 리스트 관리. 전체 테이블 스캔 시 버퍼 풀 오염 방지를 위해 Young/Old 서브리스트 분리 (5/8-3/8 비율).
2. **Redis LRU**: Redis는 정확한 LRU 대신 랜덤 샘플 N개 중 LRU를 선택하는 근사 LRU (Approximated LRU) 사용. 성능과 메모리 효율 균형.

**안티패턴**:
- **전체 테이블 스캔 후 버퍼 풀 오염**: 대용량 분석 쿼리가 InnoDB 버퍼 풀 전체를 대체 → 이후 OLTP 쿼리의 캐시 히트율 급락. OLAP 쿼리는 별도 인스턴스 또는 전용 버퍼 풀 사용.

**📢 섹션 요약 비유**: 전체 테이블 스캔의 버퍼 풀 오염은 도서관 서가를 한 번에 전부 교체하는 것 — 자주 보던 책(캐시된 데이터)이 모두 사라져서 처음부터 다시 찾아야 해요.

## Ⅴ. 기대효과 및 결론

| 알고리즘 | Page Fault | 구현 복잡도 | 실제 사용 |
|:---|:---|:---|:---|
| OPT | 최소 (이론) | 불가능 | 비교 기준 |
| FIFO | 높음 + 벨라디 역설 | 단순 | 되도록 사용 X |
| LRU | 낮음 | 높음 (타임스탬프) | HW 지원 시 |
| **Clock** | LRU 근사 | 중간 | **현대 OS 표준** |

페이지 교체 알고리즘은 실질 메모리 성능을 결정하는 핵심 정책이다. 이론적 최적(OPT)에 가장 근접하면서 구현 가능한 Clock/Active-Inactive LRU 근사가 현대 OS의 표준으로 자리 잡았다.

## 👶 어린이를 위한 3줄 비유 설명
1. 페이지 교체는 책상이 꽉 찼을 때 어떤 책을 서랍에 넣을지 결정하는 것 — 오래 안 읽은 책을 먼저 치워야 효율적이에요.
2. LRU는 가장 오래 안 읽은 책 먼저, Clock은 최근에 폈는지 표시를 보면서 결정해요.
3. OPT는 미래를 알아야 쓸 수 있는 방법이라 불가능해요 — 그래서 실제론 LRU나 Clock을 씁니다!
