## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인덱스 할당(Indexed Allocation)은 파일마다 '인덱스 블록(Index Block)'이라는 전용 색인 공간을 두어, 모든 데이터 블록의 포인터를 한곳에 모아 관리하는 기법이다.
> 2. **가치**: 연속 할당의 외부 단편화(External Fragmentation) 문제와 연결 할당의 임의 접근(Random Access) 취약성을 동시에 해결하며 현대 파일 시스템의 핵심 기반이 된다.
> 3. **융합**: 이 방식은 UNIX/Linux의 inode 구조로 발전하였으며, 다단계 인덱스(Multi-level Index)를 통해 매우 거대한 파일까지 효율적으로 지원한다.

---

## Ⅰ. 인덱스 할당의 개념 (Definition & Core Concept)

- **정의**: **인덱스 할당(Indexed Allocation)**은 각 파일이 자신의 데이터 블록 번호들을 순서대로 저장하고 있는 **인덱스 블록(Index Block)**을 소유하는 방식이다.
- **메커니즘**: 디렉터리 엔트리는 해당 파일의 데이터가 아닌, '인덱스 블록'의 주소만을 가진다. 사용자가 i번째 블록을 요청하면, 인덱스 블록의 i번째 항목을 읽어 즉시 물리 주소를 찾아낸다.
- **해결 과제**: 연결 할당은 파일의 중간을 읽으려면 앞부분을 다 훑어야 했지만, 인덱스 할당은 중간 포인터가 한곳에 모여 있어 즉각적인 이동이 가능하다.

📢 **섹션 요약 비유**: 인덱스 할당은 책 맨 뒤의 "찾아보기(Index)"와 같다. 단어가 등장하는 모든 페이지 번호가 한 페이지에 모여 있어, 책을 처음부터 넘기지 않고도 원하는 페이지로 바로 갈 수 있는 것과 같다.

---

## Ⅱ. 인덱스 할당의 구조 및 동작 (Structure & Operation)

인덱스 블록은 해당 파일의 데이터 블록들에 대한 포인터 배열(Array of Pointers)이다.

### 1. 접근 프로세스
1. 디렉터리에서 파일의 **인덱스 블록 주소**를 획득한다.
2. 해당 인덱스 블록을 메모리로 읽어 들인다.
3. 논리적 블록 번호(Logical Block Number)를 인덱스로 삼아 실제 물리 블록 번호를 추출한다.
4. 해당 물리 블록으로 직접 이동(Direct Access)한다.

### 2. 인덱스 할당 개념도 (ASCII Diagram)

```text
 [ Directory ]               [ Index Block (Block #19) ]
 ┌───────────────┐           ┌───────┬─────────────────┐
 │ Name: video.mp4            │ Index │ Disk Block Port │
 │ Index: 19     │──────────▶├───────┼─────────────────┤
 └───────────────┘           │   0   │       24        │──┐
                             │   1   │       77        │──┼──┐
                             │   2   │       10        │──┼──┼──┐
                             │   3   │       -1 (Empty)│  │  │  │
                             └───────┴─────────────────┘  │  │  │
                                                          │  │  │
                                [ Physical Disk Blocks ]  │  │  │
       ┌──────────────────────────────────────────────────┘  │  │
       │        ┌───────────────────────────────────────────┘  │
       │        │        ┌─────────────────────────────────────┘
       ▼        ▼        ▼
    ┌─────┐  ┌─────┐  ┌─────┐
    │ B24 │  │ B77 │  │ B10 │
    └─────┘  └─────┘  └─────┘
    (Data 0) (Data 1) (Data 2)
```

📢 **섹션 요약 비유**: 인덱스 할당은 "아파트 관리실의 열쇠함"과 같다. 관리실(인덱스 블록)에 가면 모든 호수(데이터 블록)의 열쇠가 한곳에 모여 있어, 아파트 전체를 돌아다니지 않고도 원하는 집 문을 바로 열 수 있다.

## Ⅲ. 인덱스 할당의 장단점 분석 (Advantages & Disadvantages)

- **장점 (Pros)**:
  - **효율적인 임의 접근**: 중간 블록을 거칠 필요 없이 인덱스 블록만 보면 즉시 원하는 위치로 점프할 수 있다.
  - **외부 단편화 제거**: 데이터 블록들이 디스크 어디에 흩어져 있어도 인덱스에서 순서대로 관리하므로 연속된 공간을 찾을 필요가 없다.
- **단점 (Cons)**:
  - **공간 오버헤드 (Pointer Waste)**: 아주 작은 파일(예: 1바이트)이라도 인덱스 블록 하나(보통 4KB)를 통째로 차지해야 하므로 공간 낭비가 발생한다.
  - **인덱스 블록 크기 제한**: 인덱스 블록 하나에 저장할 수 있는 포인터 개수가 한정되어 있어, 거대한 파일을 지원하기 위한 추가적인 확장 기법이 필요하다.

📢 **섹션 요약 비유**: 인덱스 할당은 "다이어리 인덱스 스티커"와 같다. 원하는 페이지를 빨리 찾게 해주지만, 너무 얇은 다이어리에 큰 스티커를 붙이면 배보다 배꼽이 더 커지는(공간 낭비) 것과 같다.

---

## Ⅳ. 인덱스 할당 vs 다른 방식 비교 (Comparison)

- **vs 연속 할당**: 연속 할당은 빠르지만 외부 단편화가 심하다. 인덱스 할당은 약간의 오버헤드를 감수하고 유연성을 얻는다.
- **vs 연결 할당**: 연결 할당은 임의 접근이 불가능 수준이다. 인덱스 할당은 포인터를 모아둠으로써 임의 접근 성능을 비약적으로 향상시켰다.
- **결론**: 현대 OS는 대부분 인덱스 할당의 변형(inode 등)을 사용하며, 파일 크기에 따라 유동적으로 인덱스 구조를 확장한다.

📢 **섹션 요약 비유**: 연속 할당이 "기차"라면, 연결 할당은 "술래잡기(앞사람만 보고 따라가기)", 인덱스 할당은 "좌석 배치도"와 같다.

---

## Ⅴ. 대형 파일 지원을 위한 확장 전략 (Extension Strategies)

- **Linked Scheme**: 여러 개의 인덱스 블록을 연결 리스트로 잇는다.
- **Multi-level Index**: 인덱스 블록이 또 다른 인덱스 블록을 가리키게 하여 계층 구조를 만든다. (예: 2단계, 3단계 인덱스)
- **Combined Scheme**: UNIX에서 사용하는 방식으로, 직접 포인터와 간접 포인터를 섞어서 작은 파일은 빠르게, 큰 파일은 깊게 관리한다.

📢 **섹션 요약 비유**: 확장 전략은 "서류함의 진화"와 같다. 서류 봉투(단일 인덱스)가 꽉 차면, 봉투들을 모은 폴더(다단계 인덱스)를 만들고, 폴더가 많아지면 캐비닛을 만드는 식으로 용량을 늘리는 것이다.
