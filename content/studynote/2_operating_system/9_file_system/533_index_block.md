## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인덱스 블록(Index Block)은 파일 데이터 자체가 아닌, 해당 파일이 사용하는 실제 데이터 블록들의 주소(Pointer)들을 저장하는 전용 디스크 블록이다.
> 2. **가치**: 흩어진 데이터 블록들의 주소를 한곳에 중앙 집중화함으로써, 임의 접근(Random Access)을 위한 색인(Index) 역할을 수행하며 파일 시스템의 효율성을 보장한다.
> 3. **융합**: 블록의 크기에 따라 담을 수 있는 포인터 수가 결정되므로, 파일 시스템의 최대 파일 크기 제한을 결정짓는 핵심 설계 요소가 된다.

---

## Ⅰ. 인덱스 블록의 정의와 역할 (Definition & Role)

- **정의**: **인덱스 블록(Index Block)**은 인덱스 할당 방식에서 파일 시스템이 생성하는 특수 목적 블록이다. 데이터 블록들의 물리적 위치를 기록한 주소록과 같다.
- **핵심 역할**:
  - **주소 보관**: 논리적 순서에 따른 데이터 블록의 물리 주소를 배열 형태로 유지한다.
  - **임의 접근 지원**: 파일의 특정 오프셋(Offset)이 어느 블록에 있는지 즉시 계산할 수 있게 한다.
  - **데이터 관리**: 파일이 커지거나 작아질 때, 인덱스 블록 내의 포인터 배열을 수정하여 유연하게 대처한다.

📢 **섹션 요약 비유**: 인덱스 블록은 아파트 단지의 "우편함 모음"과 같다. 각 집(데이터 블록)으로 직접 가지 않아도, 관리동의 우편함(인덱스 블록)만 보면 어느 호수가 어디에 있는지 정보를 다 얻을 수 있는 것과 같다.

---

## Ⅱ. 인덱스 블록의 내부 구조 (Internal Structure)

인덱스 블록은 고정된 크기(예: 4KB)를 가지며, 내부에는 디스크 주소를 나타내는 정수형 포인터들이 채워져 있다.

### 1. 물리적 구성 요소
- **포인터 배열 (Pointer Array)**: `Block[0]`, `Block[1]`, ..., `Block[N]` 형태의 연속적인 주소 저장 공간.
- **유효 비트/값**: 해당 포인터가 유효한 블록을 가리키는지, 아니면 비어있는지(Null/0) 표시.

### 2. 인덱스 블록 내부 구조도 (ASCII Diagram)

```text
 [ Index Block (4KB) ]
 ┌───────────────────────────────────────────┐
 │ Entry #0 : Disk Block Address 5021        │──▶ [Data 0]
 ├───────────────────────────────────────────┤
 │ Entry #1 : Disk Block Address 1205        │──▶ [Data 1]
 ├───────────────────────────────────────────┤
 │ Entry #2 : Disk Block Address 9982        │──▶ [Data 2]
 ├───────────────────────────────────────────┤
 │ Entry #3 : 0 (Empty/Unallocated)          │
 ├───────────────────────────────────────────┤
 │ ... (Total 1024 Entries if 4-byte ptr)    │
 ├───────────────────────────────────────────┤
 │ Entry #1023 : 0                           │
 └───────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 인덱스 블록은 "리모컨의 숫자 버튼"과 같다. 1번을 누르면 1번 채널, 5번을 누르면 5번 채널로 바로 가듯, 인덱스 블록의 순서대로 데이터 블록에 즉시 연결되는 구조다.

---

## Ⅲ. 인덱스 블록의 크기와 파일 용량 (Size & Capacity)

인덱스 블록의 크기는 파일 시스템의 성능과 최대 파일 크기에 직접적인 영향을 미친다.

- **포인터 밀도**: 블록 크기가 4KB이고 포인터가 4바이트(32비트)라면, 하나의 인덱스 블록은 1,024개의 포인터를 가질 수 있다.
- **용량 계산**: 만약 블록 하나가 4KB라면, 단일 인덱스 블록이 지원하는 최대 파일 크기는 `1,024 * 4KB = 4MB`가 된다.
- **트레이드오프 (Trade-off)**:
  - **큰 인덱스 블록**: 대용량 파일 관리에 유리하지만, 작은 파일에서는 공간 낭비(Internal Fragmentation)가 심해진다.
  - **작은 인덱스 블록**: 공간 낭비는 적지만, 대용량 파일을 위해 다단계 구조(Multi-level)를 복잡하게 구축해야 한다.

📢 **섹션 요약 비유**: 인덱스 블록의 크기는 "명함첩의 크기"와 같다. 명함첩이 너무 크면 명함 한 장 넣기에 아깝고(낭비), 너무 작으면 많은 사람의 연락처를 한꺼번에 담을 수 없다(용량 제한).

---

## Ⅳ. 인덱스 블록의 관리 기법 (Management Techniques)

- **할당 시점**: 파일이 생성되거나 데이터가 처음 쓰일 때 OS는 빈 블록 하나를 할당하여 인덱스 블록으로 지정한다.
- **해제 시점**: 파일이 삭제되면 데이터 블록들뿐만 아니라 인덱스 블록 자체도 빈 공간 리스트(Free List)로 반환되어야 한다.
- **보호**: 인덱스 블록이 깨지면 파일 전체의 위치 정보를 잃으므로, 저널링(Journaling)이나 체크섬(Checksum)을 통해 무결성을 보호한다.

📢 **섹션 요약 비유**: 인덱스 블록 관리는 "도서관 대출증 관리"와 같다. 책을 다 읽고 반납할 때(파일 삭제) 대출증(인덱스 블록)도 같이 정리해야 도서관 시스템이 꼬이지 않는다.

---

## Ⅴ. 다중 인덱스 블록으로의 확장 (Extension to Multi-blocks)

단일 인덱스 블록의 한계를 극복하기 위해 다음과 같은 구조를 사용한다.

1. **연결 인덱스 (Linked Index)**: 인덱스 블록의 마지막 항목이 다음 인덱스 블록의 주소를 가리킨다.
2. **다단계 인덱스 (Multi-level Index)**: 최상위 인덱스 블록이 실제 데이터가 아닌 '다른 인덱스 블록'의 주소를 가리키는 계층 구조(Tree structure)를 형성한다.

📢 **섹션 요약 비유**: 확장 구조는 "백과사전"과 같다. 한 권으로 부족하면 전집(연결)을 만들거나, 가나다 순서가 적힌 총목록집(다단계)을 따로 만드는 것과 같다.
