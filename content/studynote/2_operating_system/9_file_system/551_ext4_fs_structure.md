## 핵심 인사이트 (3줄 요약)
> 1. **본질**: EXT4(Fourth Extended File System)는 리눅스의 표준 파일 시스템으로, 기존 EXT3의 한계를 극복하기 위해 'Extent' 기반 할당 방식을 도입한 64비트 파일 시스템이다.
> 2. **가치**: 지연 할당(Delayed Allocation)과 다중 블록 할당(MBAlloc)을 통해 디스크 단편화를 최소화하고 I/O 성능을 극대화하며, 최대 1EB의 볼륨과 16TB의 단일 파일을 지원한다.
> 3. **융합**: 저널 체크섬(Journal Checksumming) 기술을 통해 데이터 무결성을 강화하고 하위 호환성을 유지하여 리눅스 서버 및 안드로이드 생태계의 핵심 인프라로 자리잡았다.

---

## Ⅰ. EXT4의 정의와 등장 배경 (Definition & Context)

- **정의**: EXT4는 리눅스 커널 2.6.28부터 정식 채택된 고성능 저널링 파일 시스템(Journaling File System)이다.
- **등장 배경**: EXT3의 16TB 볼륨 제한, 간접 블록 매핑(Indirect Block Mapping)으로 인한 대용량 파일 처리 효율 저하를 해결하기 위해 설계되었다.
- **주요 목표**: 대용량 스토리지 지원, 성능 향상, 신뢰성(Reliability) 확보 및 하위 버전(EXT2/3)과의 호환성 유지.

📢 **섹션 요약 비유**: EXT4는 "구식 창고(EXT3)를 헐지 않고 최신식 자동화 시스템으로 업그레이드한 물류 센터"와 같다.

---

## Ⅱ. EXT4의 핵심 아키텍처: Extent 및 레이아웃 (Architecture)

EXT4는 블록 단위 매핑 대신 연속된 블록 덩어리인 'Extent'를 사용하여 메타데이터를 획기적으로 줄였다.

### 1. Extent 기반 블록 관리
- **기존 (Indirect Mapping)**: 파일의 모든 블록 번호를 일일이 기록.
- **EXT4 (Extents)**: "시작 블록 번호 + 연속된 블록 개수" 형식으로 기록. 하나의 Extent는 최대 128MB까지 표현 가능.

### 2. 디스크 레이아웃 (ASCII Diagram)

```text
[ EXT4 Layout ]
┌────────────────┬───────────────┬────────────────┬───────────────┐
│ Boot Block     │ Block Group 0 │ Block Group 1  │ Block Group N │
└────────────────┴───────────────┴────────────────┴───────────────┘
       │
       ▼ [ Block Group Detail ]
┌──────────────┬─────────────┬─────────────┬─────────────┬─────────────┬─────────────┐
│ Super Block  │ GDT (Desc)  │ Block Bitmap│ Inode Bitmap│ Inode Table │ Data Blocks │
└──────────────┴─────────────┴─────────────┴─────────────┴─────────────┴─────────────┘
```

📢 **섹션 요약 비유**: Extent는 "책 페이지 번호를 1, 2, 3... 따로 적지 않고 '1번부터 10번까지'라고 묶어서 표시하는 것"과 같아 목록이 훨씬 간결해진다.

## Ⅲ. 성능 최적화 기술 (Performance Optimization)

- **지연 할당 (Delayed Allocation)**: 데이터를 즉시 디스크에 쓰지 않고 버퍼에 대기시킨 후, 마지막 순간에 연속된 공간에 한꺼번에 할당하여 단편화를 방지한다.
- **다중 블록 할당 (Multi-Block Allocation, MBAlloc)**: 여러 개의 블록 할당 요청을 한 번에 처리하여 CPU 사용량을 줄이고 디스크 헤드 이동을 최적화한다.
- **빠른 fsck (Fast File System Check)**: 사용되지 않는 inode 테이블 섹션을 건너뛰어 시스템 점검 시간을 획기적으로 단축한다.

📢 **섹션 요약 비유**: 지연 할당은 "손님이 올 때마다 음식을 나르는 게 아니라, 쟁반에 가득 찰 때까지 기다렸다가 한꺼번에 서빙하는 것"과 같다.

---

## Ⅳ. 데이터 무결성 및 저널링 (Data Integrity)

- **저널 체크섬 (Journal Checksumming)**: 저널 데이터에 체크섬을 추가하여 저널 자체가 손상되었는지 감지한다. 이는 EXT3보다 훨씬 안전한 복구를 보장한다.
- **온라인 단편화 제거 (Online Defragmentation)**: 시스템이 마운트된 상태에서도 `e4defrag`를 통해 디스크 성능을 최적화할 수 있다.
- **영속적 사전 할당 (Persistent Pre-allocation)**: 파일 생성 시 미리 물리적 공간을 확보하여 스트리밍이나 DB 작업 시 성능 저하를 막는다.

📢 **섹션 요약 비유**: 저널 체크섬은 "배달 장부에 도장을 찍어 장부 내용이 조작되지 않았음을 증명하는 안전장치"이다.

---

## Ⅴ. 기술적 사양 및 한계 (Specifications & Limitations)

- **최대 파일 크기**: 16 TiB (4KB 블록 기준).
- **최대 볼륨 크기**: 1 EiB (Exabyte).
- **서브 디렉토리 제한**: 64,000개 (EXT3의 32,000개에서 확장).
- **한계점**: CoW(Copy-on-Write) 기능이 없어 스냅샷 생성이 어렵고, 파일 시스템 수준의 압축이나 중복 제거 기능이 기본적으로 포함되어 있지 않다.

📢 **섹션 요약 비유**: EXT4는 "가장 튼튼하고 검증된 트럭"이지만, "자율 주행이나 비행 기능(CoW/스냅샷)"은 없는 정통파 도구이다.
