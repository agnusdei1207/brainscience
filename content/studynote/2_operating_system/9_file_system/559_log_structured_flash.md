## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 로그 구조 쓰기(Log-Structured Writing)는 데이터를 수정할 때 기존 위치를 덮어쓰지 않고, 새로운 빈 공간에 순차적으로 기록하는 아키텍처이다.
> 2. **가치**: NAND 플래시의 'Erase-before-Write(선 삭제 후 쓰기)' 제약을 극복하며, 느린 임의 쓰기(Random Write)를 빠른 순차 쓰기(Sequential Write)로 변환하여 입출력 성능을 극대화한다.
> 3. **융합**: 가상 주소와 물리 주소를 연결하는 매핑 테이블(NAT/FTL)을 통해 데이터의 동적 배치를 지원하며, SSD와 모바일 스토리지 수명 연장의 핵심 메커니즘으로 작동한다.

---

## Ⅰ. 로그 구조 쓰기(LFS)의 정의와 필요성 (Definition)

- **정의**: LFS(Log-structured File System)는 파일 시스템의 모든 변경 사항을 로그(Log)의 형태로 디스크 끝부분에 덧붙여 나가는 방식이다.
- **배경**: 하드디스크 시절에는 탐색 시간(Seek Time)을 줄이기 위해 제안되었으나, 현대에는 플래시 메모리의 물리적 한계를 극복하기 위한 필수 기술로 재조명되었다.
- **플래시 특성 반영**: 플래시 메모리는 페이지(Page) 단위로 쓰지만 삭제는 블록(Block) 단위로 수행하므로, 제자리 덮어쓰기가 불가능하다는 점을 해결한다.

📢 **섹션 요약 비유**: 로그 구조 쓰기는 "공책의 중간 내용을 지우고 다시 쓰는 대신, 맨 뒷장에 새 내용을 이어 쓰는 것"과 같다.

---

## Ⅱ. 플래시 메모리 물리 특성과 LFS (Flash Properties)

플래시 메모리는 다음과 같은 세 가지 주요 제약 사항을 가진다.

### 1. 주요 물리 제약
- **선 삭제 후 쓰기**: 쓰기 전 해당 블록을 반드시 비워야 함.
- **쓰기 횟수 제한 (P/E Cycle)**: 특정 횟수 이상 쓰면 수명이 다함.
- **단위 불일치**: Read/Write는 Page 단위, Erase는 Block 단위.

### 2. 매핑 및 쓰기 메커니즘 (ASCII Diagram)

```text
[ Data Update Process ]
  1. Old Version (Page 1)  ----▶  2. Mark Invalid (Invalid Page)
  3. New Version (Page 50) ----▶  Stored in a new empty page

  [ Block Layout ]
  ┌──────────┬──────────┬──────────┬──────────┐
  │ P0 (Old) │ P1 (Inv) │ P2 (Inv) │ P3 (New) │
  └──────────┴──────────┴──────────┴──────────┘
       ▲                     │          ▲
       └─────────────────────┘          │
         (NAT/FTL Mapping Table Pointing to P3)
```

📢 **섹션 요약 비유**: 플래시 메모리는 "수정테이프가 없는 종이"와 같다. 틀리면 지우지 못하고 빈 종이에 새로 적어야 한다.

---

## Ⅲ. Out-of-place Update와 NAT (Mapping Table)

- **Out-of-place Update**: 기존 데이터를 그대로 두고 새로운 장소에 업데이트를 수행하는 방식.
- **NAT (Node Address Table)**: 파일 시스템 내 노드 번호를 실제 디스크 물리 주소로 변환하는 지도 역할을 한다.
- **장점**: 임의 쓰기 작업을 거대한 순차 로그로 묶어 처리하므로 쓰기 지연 시간을 획기적으로 단축한다.

📢 **섹션 요약 비유**: NAT는 "이사를 자주 가는 친구의 새 주소를 계속 알려주는 주소록"과 같다.

---

## Ⅳ. LFS 방식의 핵심 과제 (Challenges)

- **가비지 컬렉션 (Garbage Collection)**: 새로운 데이터가 계속 쌓이면 유효하지 않은 데이터(Invalid Page)가 블록을 차지하게 된다. 이를 정리하여 빈 공간을 확보하는 과정이 필수적이다.
- **쓰기 증폭 (Write Amplification)**: 데이터를 하나 옮길 때 주변 데이터까지 같이 옮겨야 하므로 실제 요청보다 더 많은 쓰기가 발생하는 현상을 관리해야 한다.
- **일관성 관리**: 데이터가 이곳저곳에 흩어져 있으므로, 시스템 충돌 시 최근 쓰기 지점을 찾는 체크포인트 기술이 필요하다.

📢 **섹션 요약 비유**: LFS의 숙제는 "뒷장에 계속 쓰다 보니 다 써버린 공책을 다시 정리해서 빈 페이지를 만드는 일"이다.

---

## Ⅴ. 현대 저장 장치에서의 위상 (Current Standing)

- **SSD/NVMe**: 컨트롤러 내부의 FTL(Flash Translation Layer)이 로그 구조 방식을 하드웨어 수준에서 구현한다.
- **F2FS**: 운영체제 레벨에서 로그 구조를 직접 구현하여 소프트웨어-하드웨어 간의 시너지를 극대화한다.
- **결론**: 로그 구조 쓰기는 플래시 메모리 시대의 성능과 안정성을 지탱하는 가장 근본적인 아키텍처이다.

📢 **섹션 요약 비유**: 로그 구조 쓰기는 "플래시 메모리라는 최신형 비행기를 날게 하는 엔진 원리"와 같다.

> "이 방식은 **'뒷장에 이어서 쓰는 일기장'**이야. 일기를 쓰다가 틀렸을 때 예전에는 지우개로 빡빡 지워야 했지만(HDD), 이제는 그냥 '아까 그건 틀렸어!'라고 표시만 하고 뒷장에 다시 적는 거란다. 덕분에 지우개 가루(성능 저하)도 안 생기고 아주 빠르게 일기를 끝낼 수 있어. 대신 일기장이 꽉 차면 예전에 틀렸던 부분들을 싹 모아서 버리는 청소 시간(GC)이 필요하단다!"
