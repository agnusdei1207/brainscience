# 서비스 기능 체이닝 (Service Function Chaining)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서비스 기능 체이닝 (Service Function Chaining)은 네트워크 트래픽이 방화벽, 로드 밸런서, DPI, WAN 최적화 장치 등의 여러 서비스 기능(VNF/PNF)을 순서대로 거치게 하는 메커니즘으로, 서비스 체인의 동적 구성을 통해 네트워크 서비스 배포의 유연성을 향상시킨다.
> 2. **가치**: 기존의 물리적 케이블 연결 기반的服务链에서 벗어나, SDN/NFV 기술을 활용하여 소프트웨어적으로 서비스 체인을 동적으로 구성함으로써, 새로운 서비스 추가와 변경 시 물리적 재연결 불필요, 서비스 출시 시간 단축, 그리고 네트워크 자원 활용 효율성 향상을実現한다.
> 3. **융합**: NSH (Network Service Header), OpenFlow, VNF, SD-WAN과 결합되어, 통신사업자의 코어 네트워크,Enterprise의 분산 네트워크, 그리고 5G 네트워크 슬라이싱에서 핵심적인 역할을 수행한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

서비스 기능 체이닝 (Service Function Chaining)은 네트워크 트래픽이 순서대로 거치게 될 서비스 기능(VNF 또는 PNF)의集合을 지정하는 것이다. 예를 들어, 기업의 인터넷 트래픽이 방화벽(vFirewall) → 로드 밸런서(vLB) → DPI(vDPI) → 인터넷 게이트웨이(vRouter)를 순서대로 통과하는 경우, 이것이 하나의 서비스 기능 체인이 된다. 전통적인 방식에서는 이러한 서비스 체인을 물리적 케이블로 연결했지만, SDN/NFV 환경에서는 소프트웨어적으로 서비스 체인을 동적으로 구성한다.

### 왜 서비스 기능 체이닝이 중요한가

전통적인 네트워크 서비스 제공에서 새로운 서비스를 추가하거나 서비스 체인을 변경하려면, 물리적 케이블을 다시 연결해야 했다. 이는 서비스 프로비저닝에 수일이 걸리는 주요 원인 중 하나였다. 또한 트래픽 부하에 따라 서비스 체인의 특정 기능만을 스케일링하거나, 장애 발생 시 대체 경로로 우회하는 것이 물리적 연결 기반으로는 어렵다. 서비스 기능 체이닝은 이러한 문제를 해결한다.

전통적 서비스 체인 vs SDN/NFV 기반 서비스 기능 체이닝의 차이를 비교하면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │      전통적 서비스 체인 vs SDN/NFV 기반 서비스 기능 체이닝 비교              │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [전통적 서비스 체인 (물리적 연결)]                                   │
  │                                                                     │
  │  Server A (FW) ─── 케이블 ─── Server B (LB) ─── 케이블 ─── Server C  │
  │       │                                                              │
  │       │                                                              │
  │       ├── 문제: 새 서비스 추가 시 케이블 재연결 필요!                     │
  │       ├── 문제: 특정 기능 스케일링 시拓扑変更が必要！                    │
  │       └── 문제: 장애 시 우회 경로 설정 어려움                            │
  │                                                                     │
  │  [SDN/NFV 기반 서비스 기능 체이닝]                                     │
  │                                                                     │
  │       ┌───────────────────────────────────────────────────────┐      │
  │       │  Classifier ──▶ vFirewall ──▶ vLB ──▶ vDPI ──▶ Router│      │
  │       │       │                                                      │      │
  │       │       └── NSH / VLAN / Policy-based Routing ──────────┘      │
  │       │                                                              │
  │       └── 동적 경로 설정: SDN Controller가 서비스 체인을 동적으로 변경   │
  │                                                                     │
  │  장점: 새 서비스 추가 시 물리적 재연결 불필요                             │
  │  장점: 서비스 체인의 특정 기능만 스케일링 가능                             │
  │  장점: 장애 시 우회 경로 자동 설정                                      │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 전통적 서비스 체인에서는 각 서비스 기능(방화벽, 로드 밸런서, DPI 등)이 물리적 서버에 설치되고, 이들을 케이블로 순서대로 연결해야 했다. 새로운 서비스를 추가하려면 해당 서버를 네트워크에 케이블로 연결해야 했고, 이는 서비스 프로비저닝 시간의 주요 Bottleneck이었다. SDN/NFV 기반 환경에서는 Classifier가 트래픽을 인식하여 서비스 체인을 분류하고, SDN Controller가 서비스 체인의 경로를 소프트웨어적으로 설정한다. NSH (Network Service Header)나 VLAN, Policy-based Routing 등의 메커니즘을 활용하여 트래픽이 순서대로 각 서비스 기능을 거치도록制御한다. 새로운 서비스를 추가하거나 체인을 변경할 때 물리적 케이블 연결 변경 없이 SDN Controller에서 경로 설정을 변경하면 된다. 또한 특정 서비스 기능에 병목이 발생하면 SDN Controller가 자동으로 우회 경로를 설정하거나, 해당 기능을 Scale-out하여 부하를分散한다.

### 등장 배경

1. **ETSIIsg for NFV의 표준화**: ETSI NFV가 서비스 기능 체이닝을 NFV 아키텍처의 핵심 기능으로 표준화했다.
2. **IETF SFC (Service Function Chaining)工作组**: IETF에서 서비스 기능 체이닝을 위한 표준( RFC 7665 등)을 개발했다.
3. **SDN의 확산**: SDN Controller가 서비스 체인의 동적 제어를 가능하게 하여, 서비스 기능 체이닝의実用性が向上した。

### 💡 비유

서비스 기능 체이닝은，好像是一家テーマパークのアトラクションルートのように、Guest(트래픽)가特定のルートに・サービスを進み、各 서비스(アトラクション)를 순서대로 타고 eventually目的地에 도착하는 것과 같다. 新しいサービスを追加したい場合は、ルートを変更するだけで、実際の道(물리적 연결)을作り直す必要がない。

### 📢 섹션 요약 비유

 서비스 기능 체이닝は，好像是一家レストランのコラー工程のように、食材(트래픽)이特定の順序で各 COOK(서비스 기능)을 거치면서，最终的に美しいコース料理(완성된 서비스)를提供する工程と似ている。

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **Classifier** | 트래픽을 식별하고 서비스 체인에 분류 | 패킷의 L2-L4 정보를 기반으로 분류, NSH 메타데이터附加 | NSH, VLAN, Policy-based |受付係 |
| **Service Function (SF)** | 실제 네트워크 기능 (방화벽, DPI 등) | 패킷을 처리하고 다음 SF로 전달 | VNF, PNF | 料理人 |
| **Service Function Forwarder (SFF)** | SF 간 트래픽 전달 | SFC encapsulation/de-capsulation | NSH, VXLAN-gpe | 料理中介 |
| **SF Chaining Mechanism** | 서비스 체인의 메커니즘 | NSH, VLAN, Steering, etc. | NSH (RFC 8300) | 経路表 |
| **SDN Controller** | 서비스 체인의 동적 제어 | 경로 계산, SFF/SF에 지시 | ODL, ONOS, Ryu | 中央調理師 |

### NSH 기반 서비스 기능 체이닝 동작

NSH (Network Service Header)를 활용한 서비스 기능 체이닝의 동작 과정을 살펴보면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │                    NSH 기반 서비스 기능 체이닝 동작                          │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [NSH 헤더 구조]                                                    │
  │                                                                     │
  │  ┌─────────────────────────────────────────────────────────────┐     │
  │  │  NSH (40바이트)                                              │     │
  │  │  ┌──────────┬──────────┬────────────┬───────────────────┐│     │
  │  │  │ Base Hdr │  Nxt Hdr │  Service  │  Context Header   ││     │
  │  │  │  (4B)    │  (1B)    │  Path ID  │   (N x 4B)       ││     │
  │  │  │          │          │  (2B)     │                  ││     │
  │  │  └──────────┴──────────┴────────────┴───────────────────┘│     │
  │  │       │           │          │               │               │     │
  │  │       │           │          │               │               │     │
  │  │       │           │          │               └── 경로 추적/정책 │     │
  │  │       │           │          └── 서비스 체인 식별자                  │     │
  │  │       │           └── 프로토콜 유형 (IPv4, IPv6 등)                 │     │
  │  │       └── 버전, OTTL, Flags 등                                    │     │
  │  └─────────────────────────────────────────────────────────────┘     │
  │                                                                     │
  │  [NSH 기반 서비스 기능 체이닝 동작]                                    │
  │                                                                     │
  │  Classifier ──▶ SFF1 ──▶ vFirewall ──▶ SFF2 ──▶ vLB ──▶ SFF3 ──▶ vDPI ──▶ SFF4 ──▶ Router ──▶ 인터넷  │
  │      │                                                              │
  │      │ 원본 패킷                                                       │
  │      │                                                              │
  │      ▼                                                              │
  │  ┌─────────────────────────────────────────────────────────────┐     │
  │  │  원본 패킷에 NSH 헤더附加                                    │     │
  │  │  - Service Path ID: 이 패킷이 따라야 할 체인 ID                │     │
  │  │  - SPI (Service Path Index): 체인 내 위치                       │     │
  │  │  - TTL: 남은 홉 수                                          │     │
  │  └─────────────────────────────────────────────────────────────┘     │
  │      │                                                              │
  │      │ NSH가附加된 패킷                                             │
  │      ▼                                                              │
  │  SFF1이 수신 ──▶ Service Path ID로 다음 SF 결정 ──▶ vFirewall으로 전달 │
  │      │                                                              │
  │      │                                                              │
  │      ▼                                                              │
  │  vFirewall가 처리 ──▶ NSH 확인 ──▶ 처리 완료 ──▶ SFF2로 전달          │
  │      │                                                              │
  │      │ NSH가유지된 채로 다음 SF로                                       │
  │      ▼                                                              │
  │  SFF2가 수신 ──▶ Service Path ID로 다음 SF 결정 ──▶ vLB로 전달      │
  │      │                                                              │
  │      ...                                                           │
  │                                                                     │
  │  [동적 서비스 체인 변경]                                              │
  │                                                                     │
  │  상황: vDPI에 장애 발생                                              │
  │                                                                     │
  │  SDN Controller가 이를 감지 ──▶ 서비스 체인 경로 재계산              │
  │      │                                                              │
  │      │ 변경된 Service Path ID를 해당 Classifier에 알림               │
  │      ▼                                                              │
  │  Classifier ──▶ SFF1 ──▶ vFirewall ──▶ SFF2 ──▶ vLB(skip vDPI)───▶ SFF4 ──▶ Router ──▶ 인터넷  │
  │                                  │                                      │
  │                                  └── vDPI 우회 (장애 발생 시)           │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** NSH는 패킷에 추가되는 40바이트 헤더로, Base Header는 버전, TTL (Time to Live), Flags 등을 포함하고, Next Header는 원본 패킷의 프로토콜 유형을 지정하며, Service Path ID는 이 패킷이 따라야 할 서비스 체인을 식별하고, Context Header는 서비스 체인에 관한 추가 정보(경로 추적, 보안 정책 등)를 포함한다. Classifier는 트래픽을 인식하여 적절한 Service Path ID를 NSH 헤더에 부여한다. SFF (Service Function Forwarder)는 NSH의 Service Path ID를 기반으로 다음 서비스 기능(SF)으로 패킷을 전달한다. 각 SF는 NSH 헤더를 확인하고 처리한 후, 다음 SFF로 전달한다. 장애 발생 시 SDN Controller가 이를 감지하고 서비스 체인 경로를 재계산하여, Classifier에게 변경된 Service Path ID를 알려준다. 이를 통해 패킷이 자동으로 우회 경로를 따른다. NSH의 핵심 가치는 서비스 체인의 모든 구성 요소(Classifier, SFF, SF)가 NSH 헤더만 해석하면 되므로, 각厂商 장비 간의 상호운용성이 보장된다는 점이다.

### 📢 섹션 요약 비유

 NSH 기반 서비스 기능 체이닝は，好像是一家レストラングループの料理を運ぶカートのように、各料理(트래픽)에経路情報(NSH)가添付され、を受け取った厨房ロボット(서비스 기능)이自分の担当部分を調理した後、次のロボットに传递する。その場で経路を変更해도、ロボットの役割(서비스 기능)自体は変わらない。

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: NSH vs VLAN 기반 서비스 체이닝

| 비교 항목 | NSH 기반 | VLAN/VXLAN 기반 |
|:---|:---|:---|
| **메타데이터 전달** | NSH 헤더에 다양한 정보 전달 | VLAN ID/VXLAN VNI에 제한적 정보만 |
| **서비스 체인 길이** | 이론상 무제한 | 스위치 테이블 크기 제한 |
| **동적 변경** | 매우 유연 (Controller가 경로만 변경) | 새로운 VLAN/VXLAN 설정 필요 |
| **厂商 지원** | 증가 중 | 기존 장비에서도 가능 (설정 복잡) |
| **표준화** | IETF RFC 8300 | - |

### 비교 2: Service Function Chaining vs MLS (Multi-layer Switching)

| 비교 항목 | Service Function Chaining | MLS |
|:---|:---|:---|
| ** 초점** | 서비스 기능 (방화벽, DPI 등) 간 연결 | 레이어 2/3 스위칭 최적화 |
| ** 구현** | NSH, SDN, VNF | MPLS, LISP |
| ** 주 용도** | 서비스 체인, NFV | 데이터센터 스위칭 |

### 과목 융합 관점

- **SDN**: SDN Controller가 서비스 체인을 동적으로 제어한다. Classifier와 SFF에게 Service Path를 알리고, 장애 발생 시 자동으로 우회 경로를 설정한다.
- **NFV (VNF)**: VNF가 서비스 기능을 제공한다. 서비스 체인은 여러 VNF를 순서대로 연결하여 트래픽을 처리한다. CNF (Container-based VNF)로의 발전으로 더욱 빠른 스케일링과 배포가 가능해지고 있다.

### 📢 섹션 요약 비유

 NSH와 VLAN 기반 서비스 체이닝의 차이는，好像是一家レールの之路子券(NSH)에全程のルート情報が記載されているか、(VLAN)では各駅でのみ次の駅 정보만을 확인하는 것との差에 비유할 수 있다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — Enterprise 분산 보안 서비스**: 본사와 지사 간의 트래픽이 본사의 방화벽, IDS, 로드 밸런서를 순서대로 통과해야 하는 상황. 아키텍트는 NSH 기반 서비스 기능 체이닝을 활용하여, 지사 트래픽이 본사의 보안 VNF 체인을 거치도록 설정했다. 새로운 보안 정책 추가 시에도 물리적 연결 변경 없이 SDN Controller에서 서비스 체인만 업데이트하면 된다.

2. **시나리오 — 통신사업자의 Carrier Grade NAT (CGN) 서비스**: 통신사업자가用户提供 NAT 서비스를 제공하면서, 동시에 보안 기능을 적용해야 하는 상황. 아키텍트는 서비스 기능 체이닝을 활용하여, 트래픽이 먼저 DPI (악성 트래픽 탐지)를 통과하고, 그 다음 CGN (NAT), 마지막으로 방화벽을 통과하도록 설정했다.

### 도입 체크리스트

- **기술적**: 네트워크 장비가 NSH를 지원하는가? 서비스 체인의 성능 요구사항(처리량, 지연)을 충족하는가?
- **운영·보안적**: 서비스 체인의 모니터링 체계가 수립되어 있는가? 장애 감지와 자동 우회가 가능한가?

### 안티패턴

- **서비스 체인의 긴 지연**: 너무 많은 서비스 기능을串联하면 각 기능에서의 지연이 누적되어 최종 서비스 응답 시간이 증가한다. 필요한 기능만 엄격히 선별해야 한다.
- **단일 장애점**: 특정 서비스 기능이 병목이 되면 전체 서비스 체인의 처리량이 저하된다. 해당功能的 자동 스케일링 체계를 마련해야 한다.

 서비스 체인의 긴 지연은，好像是一家餐厅에서过多の料理人を通る过程中、各料理人の處理 시간이 누적되어、お最终的な料理が客に届くのが遅くなることに似ている.

## Ⅴ. 기대효과 및 결론

서비스 기능 체이닝은 SDN/NFV 환경에서 네트워크 서비스의 유연한 제공을 위한 핵심 기술이다. NSH 등의 표준화된 메커니즘을 활용하여, 물리적 연결 변경 없이 서비스 체인을 동적으로 구성하고, 장애 시 자동 우회하며, 필요한 기능만 선별하여 처리량을 최적화할 수 있다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **NSH (Network Service Header)** | 패킷에 서비스 체인 정보를 부가하는 헤더로, RFC 8300으로 표준화되어 있다. |
| **Classifier** | 트래픽을 식별하여 서비스 체인에 분류하는 기능으로, NSH의 Service Path ID를 부여한다. |
| **SFF (Service Function Forwarder)** | NSH를 기반으로 다음 서비스 기능으로 패킷을 전달하는 전달자다. |
| **VNF** | 소프트웨어로 구현된 네트워크 기능으로, 서비스 체인의 구성 요소로 동작한다. |
| **SDN Controller** | 서비스 체인의 경로를 동적으로 계산하고, Classifier와 SFF에 경로를 지시한다. |

## 👶 어린이를 위한 3줄 비유 설명

1. 서비스 기능 체이닝은 마치 **레스토랑의 서빙 경로**와 같아요. 손님(트래픽)이 앞菜(방화벽), 메인 요리(로드밸런서), 후식(DPI)을 순서대로 먹으면서 Eventually 행복한 식사를 하는 것처럼, 트래픽도 각 네트워크 서비스(요리)를 순서대로 거치면서処理받는어요.
2. 만약 어떤 요리사(특정 VNF)에 문제가 생기면,厨師長(SDN Controller)이 다른 요리사에게 대신 요리하게 하고, 서빙 경로를 자동으로 변경해줘요.
3. 새로운 메뉴(새로운 VNF)를 추가해도 中央관리자가 경로표를 수정하면 되니까, 실제 주방 구성(물리적 연결)을 다시할 필요가 없어요!
