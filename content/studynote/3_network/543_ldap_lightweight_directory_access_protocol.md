# 543. LDAP (Lightweight Directory Access Protocol) - X.500 기반 중앙관리

> **핵심 인사이트**: 회사에 이메일, 그룹웨어, 사내 메신저, VPN 시스템이 수십 개다. 신입사원이 올 때마다 이 시스템들에 일일이 계정을 파주는 건 비효율의 극치다. LDAP은 전사의 모든 직원/조직도 정보를 거대한 '디렉터리 전화번호부' 하나에 담아두고, 모든 사내 시스템이 이 전화번호부를 보고 로그인(SSO)을 처리하게 해주는 프로토콜이다.

## Ⅰ. LDAP의 개념
분산되어 있는 사용자 정보, 조직도, 권한, 이메일 주소 등 **디렉터리 서비스(Directory Service)**를 네트워크를 통해 빠르게 조회하고 수정하기 위한 경량화된 응용 계층 표준 프로토콜입니다. 
- **포트**: TCP/UDP 389 (평문), **LDAPS 636** (SSL/TLS 암호화)

## Ⅱ. 구조적 특징 (DIT: Directory Information Tree)
LDAP은 정보를 일반적인 관계형 데이터베이스(RDBMS의 테이블 형태)가 아닌, 나무 뿌리 형태의 **계층적 구조(Tree)**로 저장합니다. 이 구조는 쓰기보다 **'검색과 읽기'**에 압도적으로 빠른 성능을 발휘합니다.
- **DN (Distinguished Name)**: 디렉터리 내에서 특정 엔트리를 고유하게 식별하는 전체 절대 경로입니다.
  - *예시*: `cn=홍길동, ou=개발팀, dc=company, dc=com` (company.com 도메인의 개발팀 소속 홍길동)

## Ⅲ. X.500 프로토콜과의 관계
- 과거 ISO에서 만든 강력하지만 매우 무겁고 복잡한 OSI 기반의 **X.500 디렉터리 접근 프로토콜(DAP)**이 있었습니다.
- LDAP은 이 무거운 X.500 규격을 현대 인터넷(TCP/IP) 환경에 맞게 필수 기능만 남겨 **가볍게(Lightweight)** 개조한 것입니다. 
- 오늘날 전 세계 기업 인프라의 뼈대인 **Microsoft Active Directory (AD)** 역시 그 기반 아키텍처와 통신 규약으로 LDAP을 사용합니다.

## Ⅳ. 실무 활용 (SSO: Single Sign-On의 기초)
사내에 구축된 Jira, Confluence, Gitlab, 사내 VPN 등에 사용자가 로그인을 시도할 때, 각 시스템은 자체 DB를 확인하지 않습니다. 사용자가 입력한 ID/PW를 중앙의 LDAP 서버(AD 서버)로 전송하여 "우리 직원이 맞고, 비밀번호가 일치하는지" 단 한 번에 인증을 위임합니다. 직원이 퇴사하면 LDAP 서버에서 계정 하나만 삭제(Disable)해도 모든 사내 시스템 접근이 즉시 차단됩니다.

> 📢 **섹션 요약 비유**: LDAP은 국가 공인 '전국민 전자 주민등록부'입니다. 병원, 은행, 관공서(사내 시스템들)가 자체적으로 고객 정보를 모으지 않고, 이 거대한 중앙 주민등록부(LDAP 서버)에 "이름이 홍길동이고 개발팀 소속인 사람 맞습니까?"라고 빠르게 쿼리(조회)해서 신분증(로그인)을 한 번에 통과시켜주는 구조입니다.
