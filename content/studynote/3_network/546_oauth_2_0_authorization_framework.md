# 546. OAuth 2.0 - 인가(Authorization) 위임 프레임워크

> **핵심 인사이트**: 옛날에는 새로운 일정 관리 앱을 쓸 때 구글 아이디와 비밀번호를 그 앱에 직접 입력해야 했다. 비밀번호를 남에게 주는 건 너무 위험하다. OAuth 2.0은 내 비밀번호를 주지 않고도, "이 앱이 내 구글 캘린더만 읽을 수 있도록 허락한다"는 '제한적인 위임장(Access Token)'만 발급해 주는 혁명적인 보안 프레임워크다.

## Ⅰ. OAuth 2.0의 개념
사용자가 타사 애플리케이션(Third-party App)에게 자신의 ID/비밀번호를 제공하지 않고도, 특정 서비스(구글, 페이스북, 카카오 등)에 저장된 자신의 데이터(API)에 접근할 수 있는 권한을 안전하게 위임(Delegation)하기 위한 개방형 표준 **인가(Authorization) 프레임워크**입니다. (RFC 6749)

## Ⅱ. 핵심 구성 요소 4가지
1. **Resource Owner (자원 소유자)**: 데이터의 진짜 주인인 일반 사용자입니다.
2. **Client (클라이언트)**: 사용자의 데이터(자원)에 접근하고 싶어 하는 타사 애플리케이션입니다. (예: 내가 새로 다운받은 여행 일정 관리 웹/앱)
3. **Authorization Server (인가 서버)**: 사용자의 신원을 인증하고, 클라이언트에게 접근 권한을 증명하는 **'Access Token(접근 토큰)'**을 최종적으로 발급해 주는 서버입니다. (예: 구글 로그인 서버)
4. **Resource Server (자원 서버)**: 사용자의 데이터(사진, 연락처, 캘린더)가 실제로 저장되어 있는 구글, 카카오 등의 API 서버입니다.

## Ⅲ. 통신 흐름 (Authorization Code Grant 방식 기준)
가장 널리 쓰이고 보안이 강력한 '인가 코드 부여' 방식의 흐름입니다.
1. 클라이언트 앱이 사용자를 구글의 **인가 서버 로그인 페이지로 리다이렉트(이동)** 시킵니다.
2. 사용자가 구글 화면에서 로그인한 후 "이 앱이 내 캘린더를 보는 것에 동의합니다" 버튼을 누릅니다.
3. 구글 인가 서버는 사용자를 다시 클라이언트 앱으로 돌려보내면서 **'임시 인가 코드(Authorization Code)'**를 줍니다.
4. 클라이언트 앱의 백엔드 서버는 브라우저를 통하지 않고, 서버 대 서버로 이 코드를 구글에 보내 **진짜 'Access Token'**으로 교환합니다. (보안 극대화)
5. 이후 클라이언트 앱은 구글 자원 서버에 캘린더 데이터를 요청할 때마다 이 Access Token을 HTTP 헤더에 담아 제시합니다.

## Ⅳ. 주의점: OAuth 2.0은 '인증(Authentication)'이 아니다!
OAuth 2.0의 목적은 철저하게 API 접근 권한을 주는 **인가(Authorization)**입니다. 발급받은 Access Token 안에는 사용자가 누구인지(이름, 이메일) 명시적인 신원 정보가 없습니다. 이를 로그인(인증) 용도로 무리하게 개조해서 쓰다가 보안 사고가 많아져 탄생한 것이 바로 뒤에 나올 **OpenID Connect(OIDC)**입니다.

> 📢 **섹션 요약 비유**: 호텔에 체크인할 때 내 지갑(ID/PW)을 벨보이(Client 앱)에게 통째로 맡기면 매우 위험합니다. 대신 호텔 프론트(Authorization Server)에서 내 신분을 확인한 후, '내 방 문만 열 수 있고 옆방이나 냉장고는 열 수 없는' 제한적인 마스터키(Access Token)를 별도로 발급하여 벨보이에게 짐을 넣어달라고 건네주는 완벽한 보안 시스템이 OAuth 2.0입니다.
