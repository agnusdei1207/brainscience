# 547. SAML 2.0 (Security Assertion Markup Language) - 기업간 SSO

> **핵심 인사이트**: A 회사 직원이 회사 돈으로 B 회사의 클라우드 서비스를 쓸 때마다 새로 회원가입을 해야 한다면 관리가 불가능하다. SAML은 A 회사가 "이 사람은 우리 직원 맞고, 부장급입니다"라는 서명된 'XML 보증서(Assertion)'를 B 회사에 보내, 가입 없이도 즉시 통과시켜주는 B2B(기업 간) 통합 로그인(SSO)의 표준 언어다.

## Ⅰ. SAML의 개념
서로 다른 보안 도메인(다른 회사, 다른 기관) 시스템 간에 인증(Authentication) 및 인가(Authorization) 데이터를 안전하게 교환하기 위한 **XML 기반의 개방형 표준 데이터 포맷**입니다. 주로 기업 환경(B2B, B2E)의 SSO(Single Sign-On, 통합 인증)를 구현하는 핵심 기술로 사용됩니다.

## Ⅱ. 핵심 구성 요소
1. **Principal (주체)**: 웹 브라우저를 통해 인증받고자 하는 사람(직원)입니다.
2. **IdP (Identity Provider, 신원 제공자)**: 직원의 신원을 인증(사내 ID/PW 확인)하고, "이 사람은 우리 직원 맞다"는 보증서(SAML Assertion)를 콱 찍어 발행하는 회사의 중앙 서버입니다. (예: Microsoft Active Directory, Okta)
3. **SP (Service Provider, 서비스 제공자)**: 직원이 실제로 이용하려는 외부의 서비스 서버입니다. IdP가 발행한 보증서를 믿고 로그인을 허락해줍니다. (예: Salesforce, Workday, Zoom 등)

## Ⅲ. 동작 방식 (SP-Initiated Flow 예시)
1. 직원이 브라우저에서 외부 서비스인 Zoom(SP)에 접속을 시도합니다.
2. Zoom은 직원의 브라우저를 회사의 로그인 페이지(IdP)로 튕겨냅니다(리다이렉트). 이를 SAML Request라 합니다.
3. 직원이 회사 사내 시스템 ID와 비밀번호로 IdP에 로그인합니다.
4. IdP는 직원의 직급, 이메일 등의 정보와 굳건한 전자서명이 담긴 **SAML Assertion (XML 형태의 보증서)**을 생성하여, 다시 사용자의 웹 브라우저를 거쳐 Zoom(SP)으로 전달합니다.
5. Zoom은 XML 보증서의 전자서명을 검증하고 진짜임이 확인되면 로그인을 허용합니다.

## Ⅳ. OAuth 2.0 / OIDC와의 차이점
- **데이터 포맷**: SAML은 무겁고 복잡한 **XML**을 사용하고, OAuth/OIDC는 가벼운 **JSON (JWT)**을 사용합니다.
- **주 사용처**: SAML은 대기업 사내망과 강력한 외부 엔터프라이즈 SaaS 솔루션 간의 묵직한 웹 브라우저 기반 **B2B SSO**에 주로 쓰입니다. 반면 OAuth/OIDC는 모바일 앱, REST API 호출 등 현대적인 웹/모바일 생태계(B2C)에서 주로 쓰입니다.

> 📢 **섹션 요약 비유**: 공항 출입국 심사(SP)를 받을 때 심사관이 내 뒷조사를 처음부터 직접 하지 않습니다. 내 조국(IdP) 정부가 나의 신분을 철저히 확인한 뒤 발급해준 '여권(SAML Assertion, XML 보증서)'을 보여주면, 심사관은 그 여권의 위조 방지 홀로그램(전자서명)만 스캔해서 확인하고 곧바로 입국을 허가하는 시스템입니다.
