# 548. OpenID Connect - OAuth 2.0 기반 사용자 식별 프로토콜 (OIDC)

> **핵심 인사이트**: OAuth 2.0은 "이 사람이 내 캘린더를 볼 수 있는가(권한 인가)"를 위한 것이지, "이 사람이 누구인가(신원 인증)"를 알려주진 않는다. 그래서 앱들이 꼼수로 권한 위임장을 신분증처럼 쓰다가 보안 사고가 터졌다. 이를 막고자 OAuth 2.0 위에 '진짜 신분증(ID Token)' 기능을 정식으로 얹어 규격화한 것이 OIDC다.

## Ⅰ. OpenID Connect (OIDC)의 개념
**OAuth 2.0 프레임워크 위에서 동작하는 사용자 신원 인증(Authentication) 레이어**입니다. 타사 클라이언트 앱이 구글/카카오 등의 인가 서버를 통해 로그인한 사용자의 신원 정보(이름, 이메일, 프로필 사진 등)를 안전하고 표준화된 방법으로 검증하고 받아올 수 있게 해줍니다.

## Ⅱ. 등장 배경 (OAuth 2.0의 오용 해결)
- OAuth 2.0이 발급하는 `Access Token`은 호텔 방을 여는 권한 열쇠일 뿐, 그 열쇠를 가진 사람의 '이름표'가 아닙니다.
- 과거 수많은 앱 개발자들이 구글 로그인(OAuth) 구현 시, Access Token을 성공적으로 받아온 사실 그 자체를 '사용자가 정상적으로 로그인(인증)한 것'으로 간주해버리는 보안 취약점(Pseudo-Authentication)을 낳았습니다.
- 이를 해결하기 위해 OIDC는 Access Token과 함께, 사용자 신원 인증 결과물인 **`ID Token` (디지털 신분증)**을 명시적으로 함께 발급하도록 아키텍처를 확장했습니다.

## Ⅲ. ID Token과 JWT (JSON Web Token)
- OIDC에서 사용자의 신원을 보증하기 위해 발급하는 `ID Token`은 반드시 **JWT(JSON Web Token)** 포맷으로 만들어집니다.
- 토큰 페이로드(Payload) 내부에는 다음과 같은 클레임(Claim) 정보가 명확히 담겨 있습니다.
  - `iss`: 토큰을 발행한 곳 (예: https://accounts.google.com)
  - `sub`: 사용자의 고유 식별 번호 (예: 123456789)
  - `aud`: 이 토큰을 받아서 사용할 대상 앱의 ID
  - `exp`: 토큰의 만료 시간
- 이 JWT 안에는 구글(인가 서버)의 전자서명이 찍혀 있기 때문에, 클라이언트 앱은 매번 서버에 "이 사람 진짜 맞냐?"고 물어볼 필요 없이, 자체적으로 서명만 풀어서 사용자가 누구인지 즉시 확신할 수 있습니다.

> 📢 **섹션 요약 비유**: 친구 대신 호텔 카운터에 가서 심부름을 할 때, OAuth 2.0은 방에 들어갈 수 있는 '방 열쇠(Access Token)'만 덜렁 받는 것입니다. 열쇠만으로는 내 친구의 이름이 뭔지, 나이가 몇 살인지 전혀 알 수 없습니다. OIDC는 열쇠와 함께 친구의 얼굴 사진, 이름, 발급처가 명확히 인쇄된 '플라스틱 주민등록증(ID Token, JWT 형태)'을 세트로 발급받아, 문도 열고 신분도 명확히 확인하는 완벽한 체계입니다.
