# 199. 정보 은닉 (Information Hiding) - 내부 구현 상세를 숨김

> **핵심 인사이트**: 리모컨의 뚜껑을 열고 기판 회로를 직접 만지다간 감전되거나 기계가 망가진다. 정보 은닉은 외부인이 내부 변수에 함부로 손대지 못하게 꽁꽁 감추고, 오직 제조사가 허락한 '버튼(공개 메서드)'을 통해서만 기계를 조작하도록 강제하는 철통 방어막이다.

## Ⅰ. 정보 은닉 (Information Hiding)의 개념
소프트웨어 설계에서 데이비드 파나스(David Parnas)가 주창한 원칙으로, **모듈(또는 클래스)의 내부 데이터(자료구조)와 복잡한 알고리즘 구현 상세를 외부 모듈이 볼 수 없도록 완벽하게 숨기고(Hide)**, 오직 미리 정의된 **공개된 인터페이스(Public Interface)** 를 통해서만 접근을 허용하는 설계 원칙입니다.

## Ⅱ. 정보 은닉이 필요한 이유 (문제점과 해결책)

- **문제 (은닉 실패 시)**: 은행 계좌 모듈의 `잔액(balance)` 변수가 밖으로 훤히 노출되어 있다면, 쇼핑몰 모듈 개발자가 실수로 `balance = 0`이라는 코드를 직접 쳐서 고객의 전 재산을 날려버릴 수 있습니다. (사이드 이펙트)
- **해결 (은닉 성공 시)**: `잔액` 변수를 접근 불가(Private) 상태로 감추고, 외부에는 오직 `출금(withdraw)`이라는 함수만 열어줍니다. 출금 함수 내부에는 "잔액보다 큰 돈을 출금하려 하면 에러 반환"이라는 방어 로직이 있으므로, 데이터가 비정상적으로 훼손되는 것을 완벽하게 막을 수 있습니다.

## Ⅲ. 정보 은닉의 주요 장점

| 구분 | 설명 |
|:---|:---|
| **독립성 향상 (Low Coupling)**| 모듈 간의 결합도가 매우 낮아집니다. 외부 모듈은 내부에 뭐가 들었는지 모르고 오직 인터페이스만 바라보므로, A 모듈 내부가 싹 뜯어고쳐져도 B 모듈은 아무런 영향을 받지 않습니다. |
| **유지보수 용이** | 버그가 발생해도 다른 모듈이 이 내부 데이터를 몰래 건드렸을 확률이 0%이므로, 무조건 자기 자신 모듈 내부의 코드만 디버깅하면 됩니다. |
| **데이터 무결성 보장** | 정해진 비즈니스 로직(검증 로직)을 타지 않고 데이터가 강제로 변경되는 것을 원천 차단합니다. |

## Ⅳ. 캡슐화(Encapsulation)와의 관계
- **정보 은닉**은 "내부 구조를 밖에서 모르게 숨기자!"라는 **'목표이자 설계 철학'** 입니다.
- **캡슐화**는 연관된 데이터와 함수를 하나의 캡슐(Class)로 묶어 정보 은닉을 코드 레벨에서 실현 가능하게 해주는 **'구현 기법(도구)'** 입니다. 객체지향 언어에서 `private`, `protected` 같은 접근 제어자(Access Modifier)를 사용하여 이를 달성합니다.

> 📢 **섹션 요약 비유**: 식당에서 손님(외부 모듈)이 주방(내부 모듈)에 불쑥 들어와서 자기 마음대로 국물에 소금을 쳐버리면 요리를 망칩니다(은닉 실패). 정보 은닉은 주방 문을 굳게 닫아걸고(Private), 손님은 홀 서빙 직원(Public Interface)에게 "안 짜게 해주세요"라고 요청만 할 수 있게 하여 주방장의 요리 레시피(무결성)를 완벽히 보호하는 시스템입니다.
