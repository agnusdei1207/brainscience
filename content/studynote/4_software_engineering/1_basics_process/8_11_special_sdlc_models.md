# 8-11. 현대적/특수 개발 모델 (Prototyping, RAD, Cleanroom)

> **핵심 인사이트**: 사용자는 자신이 무엇을 원하는지 스스로도 모를 때가 많다. 프로토타입은 '일단 보여줌'으로써 오해를 줄이고, RAD는 '도구의 힘'으로 시간을 단축하며, 클린룸은 '수학적 검증'으로 버그가 없는 완벽한 소프트웨어를 꿈꾼다.

---

## Ⅰ. 프로토타입 모델 (Prototype Model)
본격적인 개발에 앞서 시스템의 핵심 기능을 **시제품(Prototype)**으로 만들어 보는 방식입니다.

* **가치**: 불투명한 요구사항을 시각화하여 고객과 개발자 사이의 **오해(Communication Gap)를 해소**합니다.
* **프로토타입의 운명**:
  - **Throw-away**: 확인 후 버림.
  - **Evolutionary**: 계속 보완하여 최종 제품으로 만듦.

---

## Ⅱ. RAD (Rapid Application Development) 모델
CASE(Computer-Aided Software Engineering) 도구와 재사용 기술을 활용하여 **60~90일 이내에 초고속으로 개발**하는 모델입니다.

* **특징**:
  - **JAD (Joint Application Development)**: 고객과 개발자가 한자리에 모여 집중적으로 설계.
  - **Time-boxing**: 정해진 시간 내에 무조건 끝내는 것을 목표로 함.
* **단점**: 숙련된 팀원과 강력한 자동화 도구가 필수적입니다.

---

## Ⅲ. 클린룸 소프트웨어 공학 (Cleanroom SE)
반도체 공정처럼 결함이 아예 발생하지 않도록 **수학적 검증(Formal Method)**을 사용하는 결벽증적(?) 모델입니다.

* **원리**: 
  - **정형 명세**: 수학적 기호로 요구사항 정의.
  - **박스 구조**: 기능의 블랙박스, 스테이트박스, 클리어박스 정의.
  - **통계적 테스트**: 버그를 찾는 게 아니라 제품의 신뢰도를 통계적으로 증명함.

---

## Ⅳ. 개념 맵 및 요약

```ascii
[상황별 모델 선택 가이드]

1. 요구사항이 안개 속이다 ───> [ Prototype ]
2. 당장 다음 달에 오픈해야 한다 ───> [ RAD ]
3. 단 하나의 버그도 용납 안 된다 ───> [ Cleanroom ]
4. 대규모이고 위험 부담이 크다 ───> [ Spiral ]
```

📢 **섹션 요약 비유**: **프로토타입**은 집을 짓기 전 '모델하우스'를 보여주는 것이고, **RAD**는 조립식 자재와 중장비를 동원해 순식간에 뚝딱 짓는 '모듈러 건축'입니다. **클린룸**은 먼지 한 톨 안 들어가는 실험실에서 정밀 부품을 조립하듯, 수학적 계산만으로 완벽한 집을 설계하는 '결벽증 건축가'의 작업 방식과 같습니다.
