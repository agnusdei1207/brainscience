# 447. 스트레스 테스트 (Stress Test)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스트레스 테스트(Stress Testing)는 시스템에 임계점을 초과하는 과부하를 가하여 시스템의 붕괴 지점과 복구 반응을 검증하는 성능 테스트 유형이다.
> 2. **가치**: 시스템이 극한 상황에서도 완전히 먹통이 되는 것이 아니라 적절히 성능을 저하시키거나 복구하는지를 사전에 파악함으로써 프로덕션 장애를 예방한다.
> 3. **융합**: 카오스 엔지니어링, 서킷 브레이커 패턴, 자동 확장(Auto Scaling) 등 탄력적 복원력 아키텍처의 유효성을 실증하는 핵심 수단이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 스트레스 테스트는 시스템에 임계점 이상의 부하를 지속적으로 또는 일시적으로 가하여 시스템이 어떤 점에서 어떻게 붕괴하는지를 관찰하고 복구 절차를 검증하는 테스트이다. 부하 테스트가 설계 용량 범위 내의 성능을 검증한다면, 스트레스 테스트는 설계를 초과했을 때의 시스템 행동을 주요 관심사로 한다.

- **필요성**: 모든 시스템은 물리적 자원의 한계를 가진다. 예상치 못한 트래픽 급증, 분산 서비스 공격,促销活动로 인한 순간 폭주 상황에서는 시스템이 과부하 상태에 놓인다. 이때 시스템이 완전히 먹통이 되는가, 아니면 적절히 서비스를 degrade하면서 핵심 기능은 유지하는가를 사전에 파악하는 것이 비즈니스 연속성 관점에서 필수적이다.

- **💡 비유**: 스트레스 테스트는 건물 구조물의 내진 테스트와 같다. 건물이 설계된 층수만큼 사람들이 들어오면 평소와 같이 운영된다. 그러나 화재报警로 모든 입주자가 동시에 대피하면 엘리베이터가 멈추고 계단에는 범람이 발생한다. 이때 건물이 완전히 붕괴하는지, 아니면 대피 경로만 제한되고 구조적 안전은 유지되는지를 시험하는 것이 스트레스 테스트다.

- **등장 배경 및 발전 과정**:
  1. **1990년대 말**: 인터넷 상용화로 예상치 못한 트래픽 증가에 대한 시스템 내성 검증 필요성 대두
  2. **2000년대 초**: 금융 시스템의 극한 거래량 처리 능력 검증 요구, 스트레스 테스트 제도화
  3. **2010년대**: 클라우드 환경에서 탄력적 확장의 한계점 파악을 위한 스트레스 테스트 보편화
  4. **현재**: 카오스 엔지니어링의 선구적 형태로 분산 시스템의 복원력 검증에 핵심적 역할

- **📢 섹션 요약 비유**: 스트레스 테스트는 자동차의 안전气囊 테스트와 같다. 평소 운행에서는气囊이 작동하지 않지만, 충돌 시에는 적절히 터져야 탑승자를 보호한다. 소프트웨어에서도 정상 상황에서는 스트레스 테스트가 불필요하지만, 극한 부하 시 시스템이 완전히 먹통이 되는 것이 아니라 적절히 대응하는지가 핵심 검증 대상이다.

### 스트레스 테스트의 동작 원리

```text
[스트레스 테스트 부하 패턴과 시스템 반응]

부하 │  정상 부하 구간        임계점          과부하 구간
(VU) │         │                    │                │
5000 │─────────┼────────────────────┼────────────────┼───>>> 붕괴
4000 │         │                    │                │
3000 │         │    ████████████████│                │
     │         │    응답시간 증가 시작 │                │
2000 │         │                    │                │
1000 │██████████│                    │                │
     └─────────┴────────────────────┴────────────────┴────────→ 시간
           0            30분           60분           90분

시스템 반응 구간:
① [정상 구간] 응답 시간 안정, 에러율 0%
② [임계점 근처] 응답 시간 서서히 증가, 에러율 1~5%
③ [임계점 초과] 응답 시간 급등, 에러율 20~50%
④ [붕괴 구간] 시스템 응답 없음 또는 완전 불능
⑤ [복구 구간] 부하 제거 후 정상 복구 여부 확인
```

**[다이어그램 해설]** 이 그래프는 스트레스 테스트의 핵심인 부하 증가에 따른 시스템 반응의 5단계를 보여준다. ① 정상 구간에서는 시스템이 설계된 용량 내에서 안정적으로 동작한다. ② 임계점 근처에서는 일부 요청의 응답 시간이 증가하기 시작하지만 시스템은 여전히 서비스 가능하다. ③ 임계점을 초과하면 응답 시간이 급격히 증가하고 에러율이 상승한다. ④에서는 시스템이 완전히 붕괴하거나 대량 에러를 발생시킨다. ⑤에서는 부하를 제거한 후 시스템이 자동으로 복구되는지를 확인한다. 실무에서는 ② 구간에서 경고를 설정하여 사전 경고를 받을 수 있도록 하고, ③ 구간에서 자동 확장 또는 서킷 브레이커가 동작하는지를 검증한다.

### 스트레스 테스트 유형

```text
[스트레스 테스트 유형 분류]

1. 순차적 스트레스 테스트 (Sequential Stress Test)
   - VU를 단계적으로 증가시켜 임계점 발견
   - 예: 100 → 200 → 300 → ... → 5000 VU, 각 단계 10분 유지

2. 갑작스러운 스트레스 테스트 (Sudden/Spike Stress Test)
   - 0에서 최고 부하로 즉각 도달하여 시스템 반응 측정
   - 예: 0 → 5000 VU 즉시 도달, 1시간 유지 후 급감

3. 장시간 스트레스 테스트 (Long-duration Stress Test)
   - 임계점 부하를 장시간 유지하여 내구성 검증
   - 메모리 누수, 세션 만료, 리소스 고갈发现问题

4. 폭발적 스트레스 테스트 (Explosive Stress Test)
   - 특정 자원만 집중적으로 소모하는 테스트
   - 예: SELECT 쿼리 집중 주입으로 DB만 과부하

5. 회복력 테스트 (Resilience Stress Test)
   - 시스템 붕괴 후 복구 절차 및 복구 시간 측정
   - 자동 확장, failover, DB 복구 시간 포함
```

**[다이어그램 해설]** 5가지 유형은 각각 다른 목적과 시나리오를 가진다. 순차적 테스트는 임계점을 발견하는 데 적합하며, 갑작스러운 테스트는 분산 서비스 공격이나 바이럴 이벤트에 대한 시스템 반응을 파악하는 데 적합하다. 장시간 테스트는 메모리 누수나 리소스 고갈처럼 시간 경과와 함께 나타나는 결함을 발견한다. 폭발적 테스트는 특정 구성 요소의 한계를 파악하는 데 유용하며, 회복력 테스트는 RTO와 RPO를 검증하는 데 필수적이다.

- **📢 섹션 요약 비유**: 스트레스 테스트는 항공기의 극한 시험 비행과 같다. 일상 운항에서는 경험하지 않는 극한 고도, 극한 속도, 극한 기온 조건에서 기체가 어떻게 반응하는지를 시험하여万一의 상황에서도 조종사가 안전하게 착륙할 수 있는지를 검증한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 분석: 부하 테스트 vs 스트레스 테스트 vs 스파이크 테스트

| 비교 항목 | 부하 테스트 | 스트레스 테스트 | 스파이크 테스트 |
|:---|:---|:---|:---|
| **목적** | 설계 용량 내 성능 검증 | 임계점 초과 시 시스템 붕괴/복구 검증 | 급격한 부하 변화에 대한 반응 검증 |
| **부하 수준** | 정상 또는 최대 예상 부하 | 임계점 이상 지속적 과부하 | 0에서 최대 또는 최소에서 최대의 순간 전이 |
| **지속 시간** | 중간 | 길음 | 짧음 |
| **측정 대상** | 응답 시간, 처리량, 자원 사용률 | 붕괴 지점, 복구 시간, 에러 유형 | 반응 속도, 자동 확장, circuit breaker |

```text
[성능 테스트 유형별 시스템 반응 비교]

부하 수준
   │
최대─┤                                          ● Stress
     │                                     ●
     │                              ●        (붕괴 구간)
     │                       ●
     │     ● Load (정상 구간)
     │  ●
     │● Spike
     │ (순간 급증/급감)
     │
     └──────────────────────────────────────────────────────→ 시간
```

**[다이어그램 해설]** 세 가지 테스트 유형은 각각 다른 관점에서 시스템의 성능을 검증한다. 부하 테스트는 정상 범위 내에서 일정한 부하를 유지하고, 스트레스 테스트는 임계점을 넘어서는 부하를 지속하며, 스파이크 테스트는 순간적인 급증/급감을 보인다. 각 테스트는 서로 다른 시스템 특성을 검증하므로 종합적인 성능 평가를 위해서는 세 유형을 모두 수행하는 것이 권장된다.

### 과목 융합 관점

- **운영체제**: 프로세스 스케줄링, 컨텍스트 스위칭, 메모리 관리 하위 시스템의 한계를 검증하는 데 활용된다. 특히 메모리 누수는 장시간 스트레스 테스트에서만 발견될 수 있다.
- **데이터베이스**: 동시 연결 수 제한, 잠금 경합, 쿼리 최적화의 한계를 파악한다. 특히 교착 상태는 고부하 상황에서만 재현될 수 있다.

- **📢 섹션 요약 비유**: 세 가지 성능 테스트는 각각 다른 방향으로 시스템의 내성을 시험한다. 부하 테스트는 정상 운영 가능한 최대 인원을 찾고, 스트레스 테스트는 그 이상에서 무슨 일이 일어나는지 확인하며, 스파이크 테스트는 갑자기 밀려드는 인파에 대한 긴급 대처 능력을 검증한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 전자상거래 플랫폼 블랙프라이데이 스트레스 테스트**: 연간 최대 트래픽인 블랙프라이데이 대비하여 평소의 10배 부하를 가정하고 스트레스 테스트를 수행한다. 목표는 5시간 연속 최고 부하를 가하여 시스템 붕괴의 임계점을 파악하고, 자동 확장 및 CDN 분산이 효과적인지 검증하는 것이다. 만약 시스템이 붕괴하면 애플리케이션 서버인지 DB인지を確認し，提前制定扩容计划다.

2. **시나리오 — 금융 거래 시스템의 극한 거래량 테스트**: 오전 9시부터 오후 3시 30분까지의 증시 거래 시간 동안 거래량이 집중되는데, 시스템 장애 시 초래되는 경제적 손실이 막대하므로 엄격한 스트레스 테스트가 필수적이다. 특히 결제 게이트웨이의 TPS 한계, DB 잠금 경합으로 인한 거래 지연, 네트워크 병목 구간을 파악해야 한다.

3. **시나리오 — 클라우드 기반 SaaS의 멀티 테넌시 스트레스 테스트**: 여러 테넌트가 동시에 자원을 사용하는 환경에서 한 테넌트의 극단적 사용이 다른 테넌트에게 미치는 영향을 검증한다. 특히 한 테넌트의 CPU 스로틀링이 다른 테넌트의 성능에 영향을 미치는지, 리소스 격리가 적절히 작동하는지 확인한다.

### 도입 체크리스트

- **기술적**: 스트레스 테스트 목표치(임계점)를 사전에 정의했는가? 테스트 환경이 프로덕션 환경과 동등한 규모와 구성을 갖추었는가? 시스템 모니터링 및 경고 설정이 완료되었는가?
- **운영·보안적**: 테스트 중 생성되는 대량 로그를 저장할 공간이 확보되었는가? 테스트 데이터에 민감 정보가 포함되지 않도록 했는가?

### 안티패턴

- **임계점 미정의 테스트**: 사전에 시스템이 붕괴하는 점을 정의하지 않으면 테스트 결과를 객관적으로 해석할 수 없다. 반드시 사전에 성공/실패 기준을 정의해야 한다.
- **단일 장애점 미검증**: 스트레스 테스트 중 발견한 병목이 단일 장애점인 경우 복구 절차가 무효할 수 있다. 항상 병목 구간의 다중화 여부를 함께 검증해야 한다.

- **📢 섹션 요약 비유**: 스트레스 테스트의 가장 큰 안티패턴은 얼마나 많이 부하를 가할 수 있는가에만 집중하고, 시스템이 어떻게 붕괴하는가와 복구할 수 있는가를 검증하지 않는 것이다. 마치 building의 내진 테스트에서 building이 무너지는 것만 관찰하고 대피 경로는 확인하지 않는 것과 같다.
