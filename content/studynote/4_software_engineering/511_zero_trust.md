# 511. 제로 트러스트 (Zero Trust) 보안 모델

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 제로 트러스트 (Zero Trust)는 "결코 신뢰하지 말고, 항상 검증하라 (Never Trust, Always Verify)"는 철학을 바탕으로, **네트워크 내부와 외부의 구분 없이 모든 접근 요청을 잠재적 위협으로 간주**하고 매번 신원을 확인하는 현대적 보안 패러다임이다.
> 2. **가치**: 경계 보안(Perimeter Security)의 붕괴와 내부자 위협에 대응하기 위해 탄생하였으며, 신원(Identity), 기기(Device), 환경(Context)을 종합 분석하여 최소 권한만을 부여함으로써 공격자의 횡적 이동(Lateral Movement)을 원천 차단한다.
> 3. **융합**: 클라우드 네이티브 아키텍처, SDP (Software Defined Perimeter), IAM 등과 결합되어 구현되며, 단순한 기술 도입이 아닌 보안 거버넌스 전반의 근본적 체질 개선을 요구한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 제로 트러스트는 전통적인 '성곽과 해자(Castle-and-Moat)' 방식의 경계 보안 모델을 부정한다. 예전에는 성문(방화벽)만 통과하면 내부에서는 자유롭게 다닐 수 있었지만, 제로 트러스트 환경에서는 성 안의 모든 방(자원)에 들어갈 때마다 매번 신분을 증명하고 허가를 받아야 한다.

- **💡 비유**: 제로 트러스트는 **"모든 직원이 마스터키를 반납하고, 각자 업무에 꼭 필요한 방의 디지털 도어락 비밀번호를 그때그때 승인받아 사용하는 스마트 오피스"**와 같습니다. 신분증이 있다고 해서 모든 층을 다닐 수 있는 권한이 자동으로 주어지지 않는 것과 같습니다.

- **등장 배경**:
  1. **네트워크 경계의 소멸**: 클라우드 도입과 원격 근무 확산으로 인해 "사내망은 안전하고 외부망은 위험하다"는 전제가 더 이상 유효하지 않게 되었다.
  2. **지능형 지속 위협 (APT)**: 일단 내부망에 침투한 해커가 보안이 허술한 내부 서버들을 마음대로 돌아다니며(횡적 이동) 정보를 빼내는 사고가 빈번해졌다.
  3. **내부자 위협 증가**: 정당한 권한을 가진 직원의 계정이 탈취되거나, 악의적인 내부자에 의한 데이터 유출 사고가 기업 보안의 최대 리스크가 되었다.

- **📢 섹션 요약 비유**: 우리 집 대문을 열어준 손님이라도, 안방이나 금고가 있는 방에는 함부로 들어오지 못하게 매번 확인하고 감시하는 철저한 주인정신 보안 모델입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 제로 트러스트 논리 아키텍처 (NIST SP 800-207)

접근 요청이 자원에 도달하기까지의 통제 구조를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      제로 트러스트 액세스 제어 논리 아키텍처                      │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │  [주체 (Subject)] ──────(접근 요청)──────▶ [제어 평면 (Control Plane)]          │
 │      사용자, 기기                                  │                        │
 │                                                    ▼                        │
 │                                          ┌──────────────────────┐           │
 │                                          │  정책 엔진 (Policy   │           │
 │                                          │  Decision Point)     │           │
 │                                          └──────────┬───────────┘           │
 │                                                    │                        │
 │                                                    ▼                        │
 │  [데이터 평면 (Data Plane)] ◀─────────── [정책 관리 (Policy   │           │
 │                                          │  Enforcement Point)  │           │
 │      ┌──────────────┐                    └──────────┬───────────┘           │
 │      │  신뢰 에이전트 │                                 │                        │
 │      └──────┬───────┘                                 │                        │
 │             │                                         │                        │
 │             ▼                                         ▼                        │
 │      ┌──────────────┐                    ┌─────────────────────────┐        │
 │      │  리소스 서버  │ ◀─────────────────── │   신뢰 영역 (Implicit   │        │
 │      │  (App, DB)   │                    │   Trust Zone 없음)      │        │
 │      └──────────────┘                    └─────────────────────────┘        │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 제로 트러스트 아키텍처는 크게 **제어 평면**과 **데이터 평면**으로 나뉜다. 모든 접근 요청은 먼저 제어 평면의 **PDP (Policy Decision Point)**를 거쳐야 한다. 여기서 사용자의 신원뿐만 아니라, 사용하는 기기의 보안 상태(백신 구동 여부 등), 접속 시간, 위치 등 다양한 맥락(Context)을 분석하여 승인 여부를 결정한다. 승인이 나면 **PEP (Policy Enforcement Point)**가 실제 데이터가 오가는 통로를 일시적으로 열어준다. 핵심은 자원 주위에 어떠한 암묵적 신뢰 영역(Implicit Trust Zone)도 두지 않고, 모든 자원을 개별적으로 격리하여 보호하는 데 있다.

### 제로 트러스트 5대 핵심 기술 요소

| 요소 | 설명 | 역할 |
|:---|:---|:---|
| **IAM** | 계정 및 권한 관리 | 사용자가 누구인지, 어떤 권한을 가졌는지 정의 |
| **SDP** | 소프트웨어 정의 경계 | 자원을 외부에서 보이지 않게 숨기고(Black Cloud) 승인된 사용자에게만 연결 |
| **Micro-segmentation** | 마이크로 세그먼테이션 | 네트워크를 잘게 쪼개어 특정 서버 침투 시 옆 서버로 확산 차단 |
| **MFA** | 다중 요소 인증 | 비밀번호 외에 추가 인증 수단을 강제하여 신뢰도 향상 |
| **Contextual Analysis** | 맥락 분석 | 접속 환경의 이상 징후(예: 갑자기 해외 IP 접속)를 실시간 판단 |

- **📢 섹션 요약 비유**: 통행증(IAM)만 검사하는 게 아니라, 차 상태는 괜찮은지(기기 보안), 짐칸에 위험물은 없는지(데이터 스캔), 왜 지금 이 시간에 오는지(맥락)를 종합 검문하는 '지능형 초소' 시스템과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 전통적 보안(경계 보안) vs 제로 트러스트 보안 비교

| 비교 항목 | 전통적 경계 보안 모델 | 제로 트러스트 보안 모델 |
|:---|:---|:---|
| **기본 전제** | 내부망은 안전하다 (Trust but Verify) | **아무도 믿지 않는다 (Never Trust)** |
| **보안 중심** | **네트워크 경계** (IP, 포트) | **신원(Identity)** 및 자산(Data) |
| **권한 범위** | 광범위한 접근 권한 (VLAN 단위) | **최소 권한** (자원/기능 단위) |
| **침투 시 영향** | 횡적 이동을 통한 전체 장악 가능 | 특정 자원에 국한된 피해 (격리) |
| **비유** | 튼튼한 성벽과 해자 | 모든 방마다 설치된 개별 금고 |

---

### 기술 융합: SASE (Secure Access Service Edge)

제로 트러스트는 현대 클라우드 네트워크 보안 표준인 SASE의 근간이 된다.
- **통합**: 네트워크 기술(SD-WAN)과 보안 기술(CASB, ZTNA, SWG)을 하나로 통합하여, 사용자가 어디에 있든 일관된 제로 트러스트 정책을 적용한다.
- **가치**: 하드웨어 장비 없이도 클라우드 상에서 지능형 보안 제어 평면을 운영할 수 있게 한다.

- **📢 섹션 요약 비유**: 전국에 흩어져 있는 지점마다 보안 요원을 둘 수 없으니, 하늘 위에 떠 있는 위성 감시 센터(SASE)가 모든 직원의 움직임을 실시간으로 통제하는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 재택근무 환경의 '내부망 직접 접속' 보안 강화

- **문제 상황**: VPN을 통해 사내망에 접속한 직원의 PC가 악성코드에 감염되어, 해당 계정을 통해 사내 핵심 DB 서버까지 해킹당하는 사고 발생.
- **의사결정 (제로 트러스트 적용)**:
  1. **VPN 폐기 및 SDP 도입**: IP 기반의 VPN 대신, 사용자 신원과 기기 상태를 먼저 검증한 후 특정 앱에만 터널을 맺어주는 **SDP (Software Defined Perimeter)** 체계로 전환한다.
  2. **지속적 인증 (Continuous Auth)**: 한 번 로그인으로 끝내는 게 아니라, 세션 유지 시간 동안 사용자의 행동 패턴을 계속 감시하여 위험 감지 시 즉시 세션을 차단한다.
  3. **마이크로 세그먼테이션**: 각 서버 간의 통신을 기본 차단(Default Deny)하고, 오직 허용된 서비스 간의 API 호출만 가능하도록 네트워크를 미세하게 분리한다.
  4. **가시성 확보**: 제어 평면에서 발생하는 모든 로그를 수집하여, "누가, 어떤 기기로, 어떤 데이터에 접근했는지" 완벽한 추적성(Traceability)을 확보한다.

### 도입 체크리스트
- **Device Health Check**: 탈옥된 기기나 보안 패치가 안 된 기기의 접근을 차단하는 로직이 있는가?
- **Just-in-Time Access**: 권한이 영구적으로 부여된 것이 아니라, 필요한 시간 동안만 일시적으로 활성화되는가?
- **Automation**: 수만 건의 접근 요청을 사람이 판단할 수 없으므로, 정책 엔진이 자동화되어 작동하는가?

### 안티패턴
- **기술 만능주의**: 솔루션(SDP 등)만 도입하면 제로 트러스트가 완성된다고 생각하는 행위. 조직의 프로세스와 데이터 분류 체계가 선행되어야 한다.
- **불완전한 가시성**: 일부 클라우드 자원은 감시하고, 일부 레거시 서버는 예외로 두는 행위. 해커는 반드시 그 예외 지점을 노린다.

- **📢 섹션 요약 비유**: 앞문만 최신식 지문 인식기로 바꾸고 뒷문(레거시 시스템)은 빗장만 걸어두는 것은 제로 트러스트가 아닙니다. 모든 문을 똑같은 수준으로 엄격하게 관리해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Risk)** | 데이터 유출 사고 80% 감소 | 횡적 이동 차단 및 최소 권한 적용으로 대규모 유출 사고 원천 봉쇄 |
| **정량적 (Ops)** | 보안 관리 효율성 향상 | 물리적 네트워크 경계 설정의 복잡성을 클라우드 기반 논리 제어로 대체 |
| **정성적 (Resilience)** | 사이버 복원력 강화 | 공격을 완벽히 막지 못하더라도 피해를 최소화하고 신속히 복구 가능한 체질 확보 |
| **정성적 (Trust)** | 비즈니스 유연성 증대 | 장소와 기기에 구애받지 않는 안전한 하이브리드 근무 환경 제공 |

### 미래 전망
- **AI-Driven PDP**: 정책 엔진이 고도화되어 사람이 정한 룰이 아닌, AI가 실시간 위협 인텔리전스를 바탕으로 위험 점수를 매기고 접근을 통제하는 '자율 보안' 시대로 진화할 것이다.
- **Passwordless Convergence**: 제로 트러스트의 핵심 식별 수단으로 생체 인식과 FIDO2 기술이 완전히 결합되어, 비밀번호 없는 가장 안전한 인증 체계가 완성될 것이다.

### 참고 표준
- **NIST SP 800-207**: Zero Trust Architecture (가장 권위 있는 기준서)
- **CISA Zero Trust Maturity Model**: 제로 트러스트 도입 성숙도 모델
- **ISO/IEC 27001**: 정보보호 경영시스템 (제로 트러스트 관점 반영 중)

- **📢 섹션 요약 비유**: 과거에는 성벽이 무너지면 끝이었지만, 이제는 성 안의 모든 방이 스스로를 지키는 요새가 되어 거대한 '디지털 안전망'을 형성하는 시대로 변모하였습니다.
