# 514. IAM (Identity and Access Management)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: IAM (식별 및 접근 관리)은 조직 내의 사용자와 기기를 식별하고, 각 주체에게 필요한 적절한 자원에 대한 **접근 권한을 생명주기(Lifecycle) 전체에 걸쳐 중앙 집중적으로 관리**하는 정보보안 체계다.
> 2. **가치**: "누가, 무엇에, 어떻게 접근하는가"를 투명하게 제어하여 계정 도용 및 권한 오남용 리스크를 최소화하며, 인사 시스템과 연동된 자동화된 계정 관리(Provisioning)를 통해 운영 효율성과 컴플라이언스 대응력을 극대화한다.
> 3. **융합**: 제로 트러스트(Zero Trust) 보안 아키텍처의 핵심 엔진이며, OAuth 2.0, OIDC, SAML 등 표준 프로토콜과 결합되어 클라우드 및 멀티 도메인 환경의 통합 인증(SSO)과 정밀한 권한 통제(ABAC/RBAC)를 실현한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: IAM은 단순한 '로그인 시스템'이 아니다. 사용자의 생성(Join)부터 부서 이동(Move), 퇴사(Leave)에 이르는 전 과정을 관리하는 **계정 관리 (Identity Management)**와, 승인된 사용자가 어떤 행동을 할 수 있는지 통제하는 **접근 관리 (Access Management)**가 결합된 포괄적 개념이다.

- **💡 비유**: IAM은 **"회사의 통합 인사 관리 및 스마트 출입 통제 시스템"**과 같습니다. 신입 사원에게 사원증을 발급하고(계정 생성), 직급에 맞는 방의 열쇠를 나눠주며(권한 부여), 부서가 바뀌면 예전 열쇠를 뺏고 새 열쇠를 주는(권한 갱신) 모든 과정을 관리하는 총무팀과 경비실의 역할을 합친 것입니다.

- **등장 배경**:
  1. **시스템의 파편화**: 기업 내 도입된 수많은 SaaS, 온프레미스 서버마다 계정을 따로 관리하면서 발생하는 보안 사각지대와 관리 비효율을 해결해야 했다.
  2. **클라우드 전환**: 고정된 IP가 아닌 '신원(Identity)' 자체가 새로운 보안 경계(New Perimeter)가 되면서, 중앙화된 신원 관리의 중요성이 수직 상승했다.
  3. **규제 강화**: 감사(Audit) 시 "누가 이 데이터를 조회했는가"를 증명하기 위해 모든 계정 활동의 이력 관리가 필수화되었다.

- **📢 섹션 요약 비유**: 수백 개의 서로 다른 열쇠 꾸러미를 들고 다니던 비효율을 끝내고, 단 하나의 지문 인식(SSO)으로 모든 문을 열되 각 문마다 정밀한 권한을 부여하는 '지능형 마스터 키' 체계로 진화한 것입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### IAM의 4대 핵심 기능 (Functional Components)

| 기능 | 설명 | 주요 기술/프로세스 | 비유 |
|:---|:---|:---|:---|
| **Authentication** | 신원 확인 | MFA, Biometrics, SSO | 본인 확인 (지문, 얼굴) |
| **Authorization** | 권한 제어 | RBAC, ABAC, Policy-as-Code | 출입 가능한 구역 결정 |
| **User Management** | 계정 생명주기 관리 | 프로비저닝, 계정 비활성화 | 사원증 발급 및 회수 |
| **Directory Services** | 신원 정보 저장소 | LDAP, Active Directory (AD) | 전 직원 주소록 및 명부 |

### IAM 운영 아키텍처 (Provisioning & Access Control)

인사 시스템과 연동되어 권한이 자동으로 관리되는 흐름을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      IAM 통합 계정 및 접근 권한 관리 흐름도                      │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. 인사 시스템 (HR)] ──────(Event: 채용/이동)─────▶ [2. IAM 엔진]              │
 │                                                    │ (Central Repository)   │
 │                                                    ▼                        │
 │                                          ┌──────────────────────┐           │
 │                                          │  자동 프로비저닝      │           │
 │                                          │  (Provisioning)      │           │
 │                                          └──────────┬───────────┘           │
 │                                                    │                        │
 │           ┌────────────────────────────────────────┼────────────────────────┐
 │           ▼                                        ▼                        ▼
 │    [3. 온프레미스 서버]                      [4. 클라우드 (SaaS)]           [5. DB/Apps]
 │    - 계정 자동 생성                          - SSO 연동 (SAML)             - 정밀 권한 부여
 │    - 최소 권한 할당                          - 라이선스 관리                - 접근 감사 로그
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ⚠ 접근 시점 (Runtime):                                                 │  │
 │  │ Subject ──▶ [인증 (MFA)] ──▶ [인가 (Policy)] ──▶ Resource Access        │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** IAM 아키텍처의 핵심은 **자동화된 거버넌스**다. 1) 인사 시스템에서 인사 이동이 발생하면, 2) IAM 엔진이 이를 감지하여 3, 4, 5) 연동된 모든 타겟 시스템의 계정과 권한을 실시간으로 동기화한다. 이를 **프로비저닝 (Provisioning)**이라 한다. 사용자가 실제 자원에 접근할 때는 매번 중앙의 정책에 따라 **MFA(다중 인증)**를 거치고, 자신이 속한 그룹이나 속성에 기반한 **인가(Authorization)**를 통과해야 한다. 모든 결과는 중앙 로그 저장소로 수집되어 감사(Audit) 자료로 활용된다.

---

### 심층 동작 원리: RBAC vs ABAC (권한 관리 모델)

IAM에서 권한을 정의하는 두 가지 핵심 방식이다.
1. **RBAC (Role-Based Access Control)**: "직무(Role)"를 기준으로 권한을 묶어서 부여. (예: '회계팀' 역할은 '회계 시스템' 접근 가능). 관리가 쉽지만 세밀한 제어가 어렵다.
2. **ABAC (Attribute-Based Access Control)**: "속성(Attribute)"들의 조합으로 판단. (예: '회계팀'이면서 '평일 낮 시간'에 '사내망' 접속 시에만 '급여 명부' 접근 가능). 매우 정밀한 제어가 가능하나 정책 설계가 복잡하다.

- **📢 섹션 요약 비유**: 유니폼(Role)만 보고 들여보내 주는 것이 RBAC라면, 신분증의 소속뿐 아니라 오늘 컨디션과 가방 내용물(Attribute)까지 다 확인하고 들여보내는 것이 ABAC입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### IAM (Identity...) vs PAM (Privileged Access Management) 비교

조직 내에서 보호 대상이 다른 두 체계를 구분한다.

| 비교 항목 | IAM (일반 계정 관리) | PAM (특권 계정 관리) |
|:---|:---|:---|
| **대상 사용자** | 전 임직원, 고객, 파트너 | **IT 관리자, DB 관리자, 루트 계정** |
| **핵심 목적** | 업무 수행을 위한 보편적 접근 | **고권한 계정의 남용 및 사고 방지** |
| **주요 기능** | SSO, 프로비저닝, 셀프 서비스 | 세션 녹화, 패스워드 금고(Vault), 일회용 권한 |
| **비유** | 일반 사원용 출입증 | 은행 금고 열쇠 관리 및 감시 |

---

### 기술 융합: IAM과 제로 트러스트 (Zero Trust)

IAM은 제로 트러스트를 지탱하는 가장 중요한 '신뢰의 근거'가 된다.
- **Identity as the New Perimeter**: IP 주소는 수시로 변하지만 '사용자의 신원'은 변하지 않는 보안의 닻(Anchor) 역할을 한다.
- **Adaptive Authentication**: IAM이 사용자의 평소 패턴(접속 위치, 시간 등)을 학습하여, 위험도가 낮으면 간편 인증을, 높으면 강력한 MFA를 요구하는 지능형 인증을 구현한다.

- **📢 섹션 요약 비유**: 튼튼한 성벽(방화벽)은 사라졌지만, 모든 사람의 가슴에 달린 '스마트 이름표(IAM)'가 실시간으로 변하며 그 사람의 안전 등급을 말해주는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 클라우드 전환에 따른 통합 IAM 구축 로드맵

- **문제 상황**: 기존 온프레미스의 Active Directory(AD)와 신규 도입한 AWS, MS 365, Slack 등의 계정이 분리되어 퇴사자 계정 정리가 안 되고 있음.
- **의사결정 (IAM 고도화 전략)**:
  1. **SSO 통합**: 모든 클라우드 서비스를 **OIDC/SAML 2.0** 기반으로 사내 IAM 서버와 연동하여 'ID 하나로 모든 서비스 접속' 환경을 구축한다.
  2. **Automated De-provisioning**: 인사 시스템에서 퇴사 처리되는 즉시 IAM이 모든 연동 서비스의 계정을 5분 내로 정지하도록 자동화 파이프라인을 구축한다.
  3. **권한 정기 검토 (IGA)**: 6개월마다 팀장들이 팀원의 권한 적절성을 승인하는 '권한 재검토(Certification)' 프로세스를 시스템화하여 권한 파편화를 방지한다.
  4. **최소 권한 가드레일**: 모든 서비스 계정에 대해 '필요할 때만 승인 후 권한 상향(PIM/PAM)' 방식을 적용하여 상시 고권한 노출을 차단한다.

### 도입 체크리스트
- **SSO Latency**: 모든 인증이 중앙 IAM을 거치게 될 때 발생하는 병목 현상과 지연 시간이 업무에 지장을 주지 않는가?
- **MFA Adoption**: 전 직원이 MFA를 불편함 없이 사용하도록 푸시 알림, FIDO 등 다양한 수단을 제공하는가?
- **Orphan Accounts**: 어떤 시스템에도 연결되지 않은 '유령 계정'이 DB에 남아 있지는 않은가?

### 안티패턴
- **수동 프로비저닝**: 관리자가 엑셀을 보고 수십 개의 사이트에 들어가 일일이 계정을 만드는 행위. 오타와 누락이 필연적이며 보안 사고의 주범이다.
- **공용 계정 사용**: "팀 공용 ID"를 만들어 비밀번호를 공유하는 행위. 사고 발생 시 책임 추적성(Accountability)이 완전히 사라진다.

- **📢 섹션 요약 비유**: 지문 인식기에 테이프를 붙여서 아무나 들어오게 하거나(비번 공유), 사원 명부(IAM)를 최신화하지 않아 그만둔 직원이 계속 회사에 드나들게 방치하는 것과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 계정 탈취 피해 99% 차단 | 강력한 MFA 및 실시간 신원 검증을 통해 부적절한 접근 차단 |
| **정량적 (Cost)** | 관리 운영 비용 60% 절감 | 수동 계정 관리 및 비밀번호 초기화 요청 등 단순 반복 업무 자동화 |
| **정성적 (Standard)** | 컴플라이언스 준거성 확보 | ISMS-P, SOC2 등 외부 감사가 요구하는 계정 관리 통제 항목 완벽 이행 |
| **정성적 (UX)** | 임직원 업무 편의성 향상 | 단 한 번의 로그인으로 모든 업무 시스템에 즉시 접근 가능 |

### 미래 전망
- **Decentralized Identity (DID)**: 기업이 중앙 서버에 개인정보를 보관하는 대신, 사용자가 자신의 모바일 기기에 신원 정보를 담고 필요한 것만 증명하는 '자기 주권형 IAM'이 도입될 것이다.
- **AI-Managed Governance**: 수백만 개의 권한 관계를 AI가 분석하여, 오랫동안 안 쓰는 권한을 자동으로 회수하거나 가장 적절한 권한을 추천해주는 '지능형 권한 관리'가 표준이 될 것이다.

### 참고 표준
- **NIST SP 800-63**: Digital Identity Guidelines
- **ISO/IEC 24760**: A framework for identity management
- **SCIM (System for Cross-domain Identity Management)**: 클라우드 간 계정 정보 교환 표준 프로토콜

- **📢 섹션 요약 비유**: 예전에는 문지기가 얼굴을 외워서 문을 열어줬다면, 이제는 전 세계의 데이터를 연결한 거대한 인공지능 두뇌(IAM)가 모든 사람의 정당성과 의도를 파악하여 안전한 디지털 세상을 만드는 수호자가 되었습니다.
