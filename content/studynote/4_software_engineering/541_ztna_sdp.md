# 541. 제로 트러스트 네트워크 액세스 (ZTNA)와 SDP

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 제로 트러스트 네트워크 액세스 (ZTNA)는 사용자의 신원, 기기 상태, 위치 등 다양한 맥락을 기반으로 **특정 애플리케이션이나 서비스에 대해서만 최소 권한 기반의 논리적 통로를 형성**하는 보안 접속 기술이다.
> 2. **관계**: ZTNA는 가트너(Gartner)가 정의한 **서비스/솔루션 관점의 용어**이며, SDP (Software Defined Perimeter)는 ZTNA를 구현하기 위해 클라우드 보안 연합(CSA)이 정립한 **기술적 아키텍처 프레임워크**로서 실질적인 근간을 이룬다.
> 3. **융합**: 기존 VPN의 '선 연결 후 인증' 방식을 대체하여 내부망 침투 시 공격자의 횡적 이동을 차단하며, SASE (Secure Access Service Edge)의 핵심 보안 구성 요소로 통합되어 전 세계 어디서든 안전한 원격 업무 환경을 제공한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **ZTNA**: "아무도 믿지 않는다"는 제로 트러스트 원칙을 네트워크 접속에 적용한 것이다. 사용자가 네트워크 망 전체에 들어오는 것이 아니라, 오직 허가된 '특정 앱'에만 세션 단위로 연결된다.
  - **SDP**: ZTNA의 핵심 구현 기술이다. 자산을 외부에서 보이지 않게 숨기고(Black Cloud), 컨트롤러가 신원을 확인한 후에만 게이트웨이를 통해 통로를 열어주는 구체적인 기술 명세다.

- **💡 비유**: 
  - **ZTNA**: 건물의 모든 방을 잠가두고, 방문객이 올 때마다 **"어떤 방에 왜 가는지"** 확인한 뒤 그 방으로만 연결되는 **"비밀 이동 통로"**를 일시적으로 만들어주는 정책과 같습니다.
  - **SDP**: 그 비밀 통로를 만들기 위해 **"투명 망토(Black Cloud)"**를 씌우고, **"암호 확인 문지기(Controller)"**를 세우는 구체적인 기술 장치들입니다.

- **등장 배경**:
  1. **VPN 리스크**: 한번 로그인하면 사내망 전체(VLAN)에 접근할 수 있는 VPN의 특성이 랜섬웨어 확산의 주원인이 되었다.
  2. **클라우드와 원격 근무**: 직원들이 사무실 밖에서 클라우드 앱을 직접 사용하는 시대가 되면서, 본사로 트래픽을 모으는 방식이 비효율적이 되었다.
  3. **가시성 확보 요구**: 누가 어떤 앱을 얼마나 썼는지 정밀하게 기록하고 통제해야 하는 보안 규제가 강화되었다.

- **📢 섹션 요약 비유**: 성문(VPN)을 열어주던 보안에서, 보이지 않는 개별 방(App)으로 안내하는 '지능형 순간이동 통로(ZTNA/SDP)' 보안으로 패러다임이 전환된 것입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### ZTNA의 2가지 아키텍처 모델

사용자와 자원 사이를 누가 중계하느냐에 따라 나뉜다.

| 모델 | 설명 | 특징 | 비유 |
|:---|:---|:---|:---|
| **Endpoint-Initiated (SDP 기반)** | 사용자 기기에 에이전트를 설치하여 직접 검증 요청 | 강력한 기기 제어 가능, 보안성 최상 | 개인용 신분증 리더기 지참 |
| **Service-Initiated (Proxy 기반)** | 에이전트 없이 브라우저/클라우드 커넥터를 통해 접속 | 배포 용이(Agentless), BYOD에 적합 | 방문 시마다 안내 데스크 경유 |

---

### ZTNA/SDP 통합 동작 아키텍처 (신원 중심 접속)

신원 확인부터 동적 터널 형성까지의 과정을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      ZTNA (SDP 기반) 논리 및 물리 아키텍처                        │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. 주체 (Subject)] ──────(2. 신원/맥락 검증)──────▶ [3. ZTNA 컨트롤러]           │
 │  사용자 + 기기상태                                    (Policy Decision)      │
 │                                                       │                     │
 │                                                       ▼                     │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ 4. 동적 정책 하달 (Dynamic Entitlement)                                 │  │
 │  └──────────────────┬─────────────────────────────────────┬──────────────┘  │
 │                    │                                     │                  │
 │                    ▼                                     ▼                  │
 │ [5. SDP 게이트웨이 A] ◀─────(Secure Tunnel)─────▶ [애플리케이션 A]            │
 │ (Publicly Hidden)                                (ERP, Mail 등)             │
 │                                                                             │
 │ [6. SDP 게이트웨이 B] ◀─────(Disconnected)─────▶ [애플리케이션 B]            │
 │                                                  (권한 없는 자원)            │
 │                                                                             │
 │  ※ 핵심: 사용자는 게이트웨이 A만 볼 수 있고, 게이트웨이 B는 존재조차 모름           │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ZTNA 시스템의 작동 원리는 **'가시성 분리'**에 있다. 1) 사용자가 접속을 시도하면 **컨트롤러**가 신원과 맥락을 검증한다. 2) 승인되면 사용자의 기기와 자원 앞단의 **SDP 게이트웨이** 사이에만 일회용 암호화 터널을 형성한다. 이때 3) 사용자는 자신이 권한을 가진 애플리케이션 A만 네트워크상에서 볼 수 있으며, 권한이 없는 B는 IP 스캔을 해도 응답하지 않는 **블랙 클라우드** 상태가 된다. 이는 공격자가 서버 하나를 뚫었을 때 다른 서버로 옮겨가는 '횡적 이동'을 아키텍처 수준에서 물리적으로 방어하는 구조다.

---

### 심층 동작 원리: 맥락 기반 접근 제어 (Contextual Access)

단순 ID/PW가 아닌 실시간 상황 데이터를 분석한다.
1. **Identity**: 누구인가? (MFA 연동)
2. **Device Health**: 탈옥 여부, 백신 구동, 최신 보안 패치 여부.
3. **Network/Location**: 신뢰할 수 없는 국가 IP인가? 사내 Wi-Fi인가?
4. **Behavior**: 평소 접속 시간인가? 갑자기 대량의 데이터를 요청하는가?
5. **가치**: "정당한 사용자"라도 "정상적인 상황"이 아니면 접근을 거부한다.

- **📢 섹션 요약 비유**: 신분증이 진짜라도, 밤늦은 시간에 술에 취해(기기 위험) 흉기(이상 행위)를 들고 있다면 문을 열어주지 않는 똑똑한 문지기와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### VPN vs ZTNA (SDP) 비교 상세

| 비교 항목 | VPN (전통적 방식) | ZTNA / SDP (현대적 방식) |
|:---|:---|:---|
| **신뢰 모델** | 연결되면 신뢰 (Connect then Auth) | **검증되어야 연결 (Auth then Connect)** |
| **보안 경계** | 네트워크 세그먼트 (L3/L4) | **개별 애플리케이션 (L7)** |
| **가시성** | 모든 자원 노출 (횡적 이동 위험) | **자원 은폐 (Black Cloud)** |
| **사용자 경험** | 속도 저하, 잦은 연결 끊김 | **심리스한 연결 (SASE 통합)** |
| **비유** | 아파트 단지 마스터키 부여 | 각 가구별 현관 비밀번호 부여 |

---

### 기술 융합: ZTNA와 마이크로 세그먼테이션 (Micro-segmentation)

ZTNA는 네트워크의 '점'을 지키고, 마이크로 세그먼테이션은 '면'을 지킨다.
- **ZTNA**: 외부에서 내부로 들어오는 접속을 서비스 단위로 쪼개어 제어.
- **Micro-seg**: 내부 서버와 서버 사이(East-West)의 통신을 촘촘히 쪼개어 제어.
- **결과**: 외부 유입부터 내부 이동까지 완벽한 제로 트러스트 그물을 형성한다.

- **📢 섹션 요약 비유**: 외부 손님에게 특정 방만 보여주는 것(ZTNA)과, 집 안 식구들끼리도 서로의 방에 함부로 들어가지 못하게 칸막이를 치는 것(마이크로 세그먼테이션)의 시너지입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 재택근무용 VPN을 ZTNA로 전환하는 과정

- **문제 상황**: VPN 사용자 폭주로 본사 게이트웨이 부하가 심각하고, 일부 직원의 VPN 계정 탈취로 내부 서버들이 랜섬웨어에 감염되는 사례 발생.
- **의사결정 (ZTNA 전환 로드맵)**:
  1. **에이전트 배포**: 전 직원 노트북에 ZTNA 에이전트를 설치하여 기기 무결성 체크(Posturing)를 활성화한다.
  2. **Application 리스트업**: 전사 모든 내부 자원을 '인사', '재무', '개발' 등 서비스 단위로 정의하고 도메인을 부여한다.
  3. **정책 수립**: "개발팀은 오직 Github와 개발 서버에만, 인사팀은 ERP에만" 접근하도록 신원 기반 정책을 컨트롤러에 입력한다.
  4. **VPN 폐쇄**: 단계적으로 서비스들을 ZTNA 게이트웨이 뒤로 숨기고, 최종적으로 외부 노출된 VPN 포트(1194 등)를 닫아 '블랙 클라우드'를 완성한다.

### 도입 체크리스트
- **User Friction**: 에이전트 설치가 사용자 업무를 방해하거나 기기 성능을 저하시키지 않는가?
- **Cloud Gateway**: 전 세계에 분산된 사용자를 위해 클라우드 기반의 분산 게이트웨이(SASE PoP)를 지원하는가?
- **Legacy Support**: 최신 웹 앱뿐만 아니라 RDP, SSH, 구형 클라이언트-서버 앱도 ZTNA로 수용 가능한가?

### 안티패턴
- **IP 기반 정책 고수**: ZTNA를 도입하면서도 여전히 IP 대역으로 권한을 주는 행위. 제로 트러스트의 '신원 중심' 가치를 퇴색시킨다.
- **상시 연결 허용**: 한번 인증 후 세션을 무기한 유지하는 행위. 반드시 유효 기간을 두고 지속적으로 상태를 재검증해야 한다.

- **📢 섹션 요약 비유**: 새 옷(ZTNA)을 샀는데 안감은 예전 헌 옷(IP 기반 정책)을 그대로 덧대어 입는 어색함을 피해야 한다는 뜻입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 횡적 이동 공격 100% 차단 | 자원 간 연결이 사전에 차단되어 서버 점령 후 확산 불가능 |
| **정량적 (Performance)** | 네트워크 성능 2배 향상 | 본사 우회 없는 최단 거리 접속(SASE 결합 시)으로 응답 속도 개선 |
| **정성적 (Standard)** | 현대적 보안 체계 확립 | 가트너 및 NIST가 권고하는 차세대 보안 표준 완벽 준수 |
| **정성적 (Agility)** | 비즈니스 유연성 증대 | 사무실, 집, 카페 등 어디서나 보안 걱정 없이 동일한 업무 가능 |

### 미래 전망
- **Identity Orchestration**: 수많은 클라우드 IdP를 하나로 묶어, 사용자가 회사를 옮겨도 경력과 신뢰도가 유지되는 '범용 신원 망'과 ZTNA가 결합될 것이다.
- **AI-Autonomous Access**: 사용자의 위험 징후를 AI가 실시간 포착하여, 사람이 판단하기 전 0.1초 만에 ZTNA 통로를 자동으로 폐쇄하는 지능형 방어가 실현될 것이다.

### 참고 표준
- **Gartner Market Guide for ZTNA**: 기술 정의 및 트렌드
- **CSA SDP Specification 2.0**: 기술적 구현 아키텍처의 표준
- **NIST SP 800-207**: 제로 트러스트 아키텍처 가이드라인

- **📢 섹션 요약 비유**: 성벽을 높이 쌓던 '수비의 시대'가 가고, 보이지 않는 곳에서 신원과 맥락으로 안전을 조립하는 '지능형 접속의 시대'가 도래했습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **ZTNA**는 우리 집 모든 방에 비밀번호를 걸어두고, **"숙제할 거면 공부방만, 잠잘 거면 침대방만" 딱 필요한 곳만 열어주는 규칙**이에요.
2. **SDP**는 그 방들을 투명하게 만들어서, **허락받지 않은 사람은 방이 어디 있는지 찾지도 못하게 숨기는 마법**과 같아요.
3. 이 두 친구 덕분에 나쁜 도둑이 우리 집 거실에 몰래 들어와도, 다른 방으로 절대 숨어들 수 없게 아주 안전해진답니다!
