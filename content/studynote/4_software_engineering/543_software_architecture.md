# 543. SW 아키텍처 (Software Architecture) 개요 및 품질 속성

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 아키텍처는 시스템의 전체적인 구조를 정의하는 **설계의 근간(Skeleton)**으로, 시스템 구성 요소 간의 관계와 상호작용 방식, 그리고 이를 제약하는 핵심 설계 원칙의 집합이다.
> 2. **가치**: 기능적 요구사항을 넘어 가용성, 성능, 보안성 등 **비기능적 요구사항(품질 속성)**을 만족시키기 위한 최상위 의사결정이며, 개발 과정의 이해관계자 간 의사소통 도구이자 시스템 변경에 따른 리스크를 관리하는 지도가 된다.
> 3. **융합**: 비즈니스 목표와 기술적 제약을 잇는 가교 역할을 수행하며, 현대 아키텍처는 유연한 변경을 위한 MSA(Microservices)와 품질 검증을 위한 ATAM(Architecture Trade-off Analysis Method) 등 정밀 분석 기법과 결합된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 소프트웨어 아키텍처는 시스템의 "뼈대"다. 개별 기능을 어떻게 코딩할 것인가가 아니라, 시스템을 어떤 컴포넌트로 나눌지, 이들이 어떻게 통신할지, 그리고 시스템이 멈추지 않으려면(가용성) 어떤 구조를 가져야 할지를 결정하는 최상위 설계도다.

- **💡 비유**: 소프트웨어 아키텍처는 **"건축 설계도"**와 같습니다. 방의 크기나 벽지 색깔(구현 상세)을 정하기 전, 건물의 용도에 맞춰 아파트인지 공장인지 결정하고, 기둥의 위치(핵심 컴포넌트)와 수도/전기 배관(데이터 흐름)을 배치하여 건물이 무너지지 않고 제 기능을 하게 만드는 것입니다.

- **등장 배경**:
  1. **시스템 규모의 거대화**: 소프트웨어가 복잡해지면서 전체 구조에 대한 조망 없이는 개발과 유지보수가 불가능해졌다.
  2. **품질의 중요성 증대**: 단순히 돌아가는 것을 넘어 "얼마나 빨리 응답하는가", "얼마나 쉽게 기능을 추가할 수 있는가"와 같은 품질 속성이 비즈니스 성패를 좌우하게 되었다.
  3. **재사용성 및 일관성 확보**: 검증된 아키텍처 패턴을 사용함으로써 개발 비용을 줄이고 시스템의 일관성을 유지하려는 요구가 커졌다.

- **📢 섹션 요약 비유**: 여행을 떠나기 전 목적지만 정하는 것이 아니라, 어떤 이동 수단을 이용하고 어떤 경로를 거칠지 전체 계획을 짜는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 소프트웨어 품질 속성 (Quality Attributes)

아키텍처가 지향하는 핵심 목표인 비기능적 요구사항들이다.

| 구분 | 품질 속성 명칭 | 상세 정의 | 비유 |
|:---|:---|:---|:---|
| **시스템 측면** | **가용성 (Availability)** | 장애 없이 정상적으로 운영되는 능력 (MTBF/MTTR) | 24시간 꺼지지 않는 전등 |
| **시스템 측면** | **성능 (Performance)** | 요청에 대한 응답 시간 및 처리량 (Latency/TPS) | 슈퍼카의 제로백 |
| **시스템 측면** | **보안성 (Security)** | 인가되지 않은 접근을 막고 정보를 보호함 | 튼튼한 금고 자물쇠 |
| **비즈니스 측면** | **변경 용이성 (Modifiability)** | 새로운 기능을 추가하거나 수정할 때 드는 비용 | 레고 블록 갈아 끼우기 |
| **비즈니스 측면** | **시험 용이성 (Testability)** | 소프트웨어의 결함을 쉽고 정확하게 찾아낼 수 있는 정도 | 자동차 정밀 검사 장치 |

---

### 소프트웨어 아키텍처 구성 요소 및 관계도

아키텍처가 시스템을 바라보는 관점을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      소프트웨어 아키텍처의 다각도적 구성 요소                      │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [비즈니스 목표] ──▶ [품질 속성 시나리오] ──▶ [아키텍처 설계 의사결정]             │
 │                                                                             │
 │  ┌──────────────────┐          ┌──────────────────┐          ┌────────────┐ │
 │  │  컴포넌트 (Node)  │          │  커넥터 (Link)    │          │ 제약 사항   │ │
 │  │ (처리 및 데이터)  │ ◀──────▶ │ (통신 및 제어)    │ ◀──────▶ │ (규칙/가이드)│ │
 │  └──────────────────┘          └──────────────────┘          └────────────┘ │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 주요 아키텍처 뷰 (View) - 4+1 View                                    │  │
 │  │ - 논리 뷰: 기능적 요구사항 (Class, Object)                                │  │
 │  │ - 프로세스 뷰: 비기능적 요구사항 (Thread, Perf)                           │  │
 │  │ - 구현 뷰: 개발 환경 중심 (Module, Lib)                                   │  │
 │  │ - 배포 뷰: 하드웨어 배치 (Network, Node)                                  │  │
 │  │ - 유스케이스 뷰: 사용자 관점 (Scenario)                                   │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 소프트웨어 아키텍처는 **컴포넌트**와 **커넥터**, 그리고 이들을 묶는 **제약 사항**으로 이루어진다. 시스템 설계 시에는 하나의 설계도만 보는 것이 아니라, 필립 크루첸의 **4+1 View** 모델처럼 논리, 프로세스, 구현, 배포라는 4가지 다른 관점(View)에서 시스템을 입체적으로 바라봐야 한다. 예를 들어, 성능을 중요시한다면 '프로세스 뷰'에서 쓰레드 할당을 고민해야 하고, 유지보수를 중시한다면 '논리 뷰'와 '구현 뷰'의 모듈화를 챙겨야 한다. 모든 설계의 중심에는 실제 사용자가 사용하는 시나리오인 '유스케이스 뷰'가 위치한다.

---

### 심층 동작 원리: 품질 속성 시나리오 (Quality Attribute Scenario)

품질 속성을 모호한 형용사(예: "빠른")가 아닌 정량적인 목표로 정의하는 방법이다.
1. **자극원 (Source)**: 누가? (예: 사용자)
2. **자극 (Stimulus)**: 무엇을? (예: 조회 버튼 클릭)
3. **환경 (Environment)**: 어떤 상황에서? (예: 평시 부하 상황)
4. **대상 (Artifact)**: 시스템의 어디를? (예: 결제 모듈)
5. **응답 (Response)**: 결과는? (예: 데이터를 조회함)
6. **응답 측정 (Measure)**: 수치는? (예: 1초 이내 응답)

- **📢 섹션 요약 비유**: "튼튼한 차"라고 말하는 대신 "시속 100km로 벽에 충돌해도 운전자가 다치지 않는 차"라고 구체적인 테스트 조건과 결과를 정하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 아키텍처 패턴별 품질 속성 영향도 분석

어떤 아키텍처를 선택하느냐에 따라 강화되는 품질 속성이 다르다. (Trade-off 발생)

| 아키텍처 패턴 | 강화되는 품질 속성 | 약화되는 품질 속성 | 적합한 시스템 |
|:---|:---|:---|:---|
| **Layered (계층)** | 유지보수성, 가독성 | 성능 (단계별 오버헤드) | 표준 엔터프라이즈 앱 |
| **Microservices (MSA)** | 확장성, 유연성 | 복잡성, 가용성 관리 어려움 | 대규모 클라우드 서비스 |
| **Event-driven (이벤트)** | 성능(비동기), 결합도 저하 | 시험 용이성, 추적성 | 실시간 트래픽 처리 |
| **Microkernel (플러그인)** | 확장성, 사용자 정의성 | 통합 및 배포 복잡도 | 브라우저, IDE |

---

### 기술 융합: 아키텍처와 데브옵스 (DevOps)

현대의 아키텍처는 운영 자동화와 떼려야 뗄 수 없는 관계다.
- **Observability**: 아키텍처 설계 시 로깅과 트레이싱 구조를 미리 고려해야 운영 단계의 '가시성'이라는 품질 속성을 확보할 수 있다.
- **Resilience**: 서킷 브레이커(Circuit Breaker) 같은 아키텍처 패턴은 시스템의 '회복력'을 높여 무중단 배포를 가능케 한다.

- **📢 섹션 요약 비유**: 자동차를 만들 때 고치기 쉽게 엔진룸을 설계(아키텍처)해야, 정비소(데브옵스)에서 빠르게 수리할 수 있는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 트래픽 급증이 예상되는 티켓 예매 시스템 설계

- **문제 상황**: 평소 트래픽의 100배가 몰리는 오픈 직후 5분간 시스템이 다운되지 않고 결제까지 원활히 이루어져야 함.
- **의사결정 (아키텍처적 판단)**:
  1. **가용성 우선순위**: 시스템 다운 방지를 위해 **Scale-out**이 용이한 무상태(Stateless) 아키텍처를 채택한다.
  2. **성능 최적화**: DB 부하를 줄이기 위해 조회 전용 **Redis Cache** 계층을 추가하고, 쓰기 작업은 **Kafka 메시지 큐**를 통해 비동기로 처리한다. (Event-driven 전환)
  3. **Trade-off 수용**: 비동기 처리로 인해 사용자가 "결제 완료"를 확인하기까지 몇 초의 시차가 생길 수 있는 '최종 일관성(Eventual Consistency)' 모델을 수용한다. (데이터 무결성 실시간성 포기)
  4. **검증**: 아키텍처 결정 사항이 실제 부하를 견딜 수 있는지 **Load Test 시나리오**를 통해 정량적으로 검증한다.

### 도입 체크리스트
- **Trade-off Analysis**: 특정 품질 속성(예: 보안)을 높이기 위해 다른 속성(예: 편의성/성능)을 얼마나 희생했는지 인지하고 기록했는가?
- **Stakeholder Alignment**: 비즈니스 팀이 요구하는 성능 목표와 기술팀이 구현 가능한 아키텍처 수준이 합치되는가?
- **Simplicity**: 시스템이 해결하려는 문제보다 아키텍처가 더 복잡해지지는 않았는가? (Over-engineering 경계)

### 안티패턴
- **Big Ball of Mud**: 아무런 규칙 없이 모든 기능이 엉켜 있는 구조. 초기 개발은 빠르지만 유지보수가 불가능해진다.
- **Architecture by Accident**: 명확한 설계 결정 없이 코딩부터 시작해서 어쩌다 보니 만들어진 구조. 예상치 못한 장애에 속수무책이다.

- **📢 섹션 요약 비유**: "일단 집부터 짓고 화장실 배관은 나중에 생각하자"고 하는 것이 얼마나 어리석은 일인지, 기초 설계(아키텍처)의 중요성을 늘 명심해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Performance)** | 응답 시간 및 처리량 목표 달성 | 아키텍처 설계를 통한 시스템 성능 지표의 획기적 개선 |
| **정량적 (Cost)** | 유지보수 비용 50% 절감 | 모듈화 및 계층화를 통해 코드 수정 시 영향 범위 최소화 |
| **정성적 (Stability)** | 시스템 신뢰도 향상 | 예외 상황 처리에 대한 구조적 설계로 장애 발생률 감소 |
| **정성적 (Standard)** | 개발 생산성 제고 | 공통 아키텍처 표준 제공으로 개발자 간 협업 효율성 증대 |

### 미래 전망
- **Self-Adaptive Architecture**: 환경 변화에 따라 아키텍처가 스스로 컴포넌트를 재배치하거나 자원을 증설하는 자율형 구조가 실현될 것이다.
- **AI-Augmented Architecting**: AI가 비즈니스 요구사항을 분석하여 최적의 아키텍처 패턴을 추천하고 설계 결함을 미리 찾아주는 지능형 설계 도구가 도입될 것이다.

### 참고 표준
- **IEEE 1471 / ISO/IEC 42010**: 소프트웨어 집약 시스템의 아키텍처 기술 표준
- **SEI Quality Attribute Workshops**: 품질 속성 도출 방법론
- **Software Architecture Patterns (Mark Richards)**: 실무형 아키텍처 패턴 가이드

- **📢 섹션 요약 비유**: 과거의 아키텍처가 고정된 성벽이었다면, 미래의 아키텍처는 환경에 따라 모습을 바꾸며 진화하는 '살아있는 유기체'가 될 것입니다.
