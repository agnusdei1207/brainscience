# 544. 아키텍처 설계 원칙: SoC, 응집도와 결합도, 추상화

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 아키텍처 설계 원칙은 소프트웨어의 복잡성을 관리하고 유지보수성을 극대화하기 위한 공학적 지침으로, **관심사 분리(SoC), 높은 응집도(Cohesion), 낮은 결합도(Coupling), 적절한 추상화(Abstraction)**를 지향한다.
> 2. **가치**: 모듈 간의 의존성을 최소화하여 특정 부분의 변경이 전체 시스템으로 전이되는 '파급 효과'를 차단하며, 코드의 재사용성을 높여 장기적인 개발 생산성과 시스템 안정성을 보장한다.
> 3. **융합**: 객체지향 설계 원칙(SOLID) 및 클린 아키텍처(Clean Architecture)의 핵심 기반이 되며, 현대의 마이크로서비스(MSA)와 서버리스 설계에서도 서비스 경계를 확정 짓는 결정적 잣대로 작용한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 좋은 아키텍처는 "단순함"과 "유연함"을 동시에 가져야 한다. 이를 위해 시스템을 독립적인 조각(모듈)으로 나누고, 각 조각이 하나의 일만 잘하게 하며(응집도), 조각들끼리는 최소한으로만 얽히게(결합도) 설계하는 것이 핵심 원칙이다.

- **💡 비유**: 아키텍처 설계 원칙은 **"조립식 가구(IKEA)의 설계 방식"**과 같습니다. 1) 다리는 다리끼리, 상판은 상판끼리 따로 만들고(SoC), 2) 나사 하나만 풀면 다리를 쉽게 바꿀 수 있게 만들며(낮은 결합도), 3) 다리 부품은 오직 '지탱하는 일'에만 집중하도록(높은 응집도) 설계하는 것입니다.

- **등장 배경**:
  1. **소프트웨어 위기**: 하드웨어 발전에 비해 소프트웨어 개발 속도와 품질이 따라가지 못하고, 수정할 때마다 새로운 버그가 생기는 현상을 해결해야 했다.
  2. **유지보수 비용의 폭증**: 개발비보다 운영 및 수정 비용이 수배 더 드는 현실에서, "고치기 쉬운 소프트웨어"가 경쟁력이 되었다.
  3. **대규모 협업의 필요성**: 수백 명의 개발자가 서로의 코드를 간섭하지 않고 병렬로 작업할 수 있는 '독립적 구조'가 절실해졌다.

- **📢 섹션 요약 비유**: 뒤엉킨 이어폰 줄(나쁜 설계)을 하나하나 정성껏 풀어서, 필요할 때마다 쏙쏙 뽑아 쓸 수 있는 정리함(좋은 설계)으로 만드는 과정입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 4대 핵심 설계 원칙 분석

| 원칙 명칭 | 상세 정의 | 지향점 | 비유 |
|:---|:---|:---:|:---|
| **관심사 분리 (SoC)** | 각 모듈은 고유한 하나의 기능적 책임만 가져야 함 | 복잡도 감소 | 주방과 화장실의 분리 |
| **응집도 (Cohesion)** | 모듈 내부 요소들이 얼마나 밀접하게 관련되어 있는가 | **높을수록 좋음 (High)** | 한 팀의 단결력 |
| **결합도 (Coupling)** | 모듈과 모듈 사이의 의존성/관계의 정도 | **낮을수록 좋음 (Low)** | 모르는 사람과의 거리 |
| **추상화 (Abstraction)** | 복잡한 내부 구현은 숨기고 핵심 인터페이스만 노출 | 유연성/단순성 | 자동차 핸들과 엔진의 관계 |

---

### 응집도와 결합도의 아키텍처적 관계도

시스템의 안정성은 모듈 간의 '거리'와 모듈 내부의 '밀도'에 의해 결정된다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      좋은 설계의 표본: High Cohesion & Low Coupling               │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │  [나쁜 설계: Low Cohesion / High Coupling]    [좋은 설계: High Cohesion / Low Coupling] │
 │                                                                             │
 │      ┌──────────────┐      ┌──────────────┐      ┌──────────────┐      ┌──────────────┐  │
 │      │   Module A   │◀────▶│   Module B   │      │   Module A   │──────▶│   Module B   │  │
 │      │ (여러기능 혼재)│      │ (A와 강하게 결합)│      │  (단일 책임)  │ 인터페이스 │  (단일 책임)  │  │
 │      └──────────────┘      └──────────────┘      └──────────────┘      └──────────────┘  │
 │             ▲                     ▲                     │                     │          │
 │             └─────────(복잡한 의존)─────┘                     └──────(느슨한 연결)──────┘          │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 설계 지침:                                                            │  │
 │  │ 1. 캡슐화(Encapsulation)를 통해 내부 변경을 숨겨 결합도를 낮춤               │  │
 │  │ 2. 단일 책임 원칙(SRP)을 준수하여 응집도를 높임                            │  │
 │  │ 3. 인터페이스 지향 설계로 구체적인 구현체와의 의존성을 제거함 (추상화)         │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 왼쪽의 나쁜 설계는 모듈들이 서로의 속살(내부 로직)을 너무 많이 알고 있고, 한 모듈이 이것저것 다 하는 잡동사니 주머니와 같다. 이 경우 A를 고치면 B가 깨지는 악순환이 반복된다. 반면 오른쪽의 좋은 설계는 각 모듈이 명확한 한 가지 역할만 수행하며(**높은 응집도**), 모듈 간에는 정해진 약속인 '인터페이스'를 통해서만 최소한으로 소통한다(**낮은 결합도**). 이렇게 설계하면 Module B의 내부가 완전히 바뀌어도 인터페이스만 유지된다면 Module A는 아무런 영향을 받지 않는다.

---

### 심층 동작 원리: 추상화 (Abstraction)의 마법

복잡함을 다루는 인간의 인지적 한계를 극복하는 기술이다.
1. **제어 추상화**: "가속 페달을 밟으면 차가 나간다"는 사실만 알면 되지, 엔진 실린더의 폭발 행정을 알 필요가 없게 만드는 것.
2. **데이터 추상화**: 데이터의 저장 방식(파일, DB, 메모리)은 숨기고 "데이터를 저장해줘"라는 명령만 노출하는 것.
3. **가치**: 상세 구현에 의존하지 않게 함으로써 시스템의 부품 교체(예: Oracle DB ──▶ MySQL)를 가능케 한다.

- **📢 섹션 요약 비유**: 텔레비전 리모컨 버튼(추상화)은 누구나 누를 수 있지만, 그 안의 복잡한 회로판(구현 상세)을 다룰 수 있는 사람은 드문 것과 같습니다. 리모컨 덕분에 우리는 전자를 몰라도 TV를 볼 수 있습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 응집도(Cohesion)의 7단계 (우수 ──▶ 취약)

| 등급 | 응집도 유형 | 특징 | 비유 |
|:---|:---|:---|:---|
| **Best** | **기능적 (Functional)** | 단일한 목적의 기능을 완벽히 수행 | 망치 (못 박기 전용) |
| **Good** | **순차적 (Sequential)** | 한 활동의 출력이 다음 활동의 입력으로 쓰임 | 조립 라인 |
| **Fair** | **통신적 (Communicational)** | 동일한 입출력 데이터를 사용하는 기능을 모음 | 고객 정보 조회 및 수정 |
| **Worst** | **우연적 (Coincidental)** | 아무 연관 없는 기능들을 그냥 모아둠 | 잡동사니 서랍 |

---

### 결합도(Coupling)의 6단계 (우수 ──▶ 취약)

| 등급 | 결합도 유형 | 특징 | 비유 |
|:---|:---|:---|:---|
| **Best** | **자료 (Data)** | 파라미터를 통해서만 데이터 교환 (가장 이상적) | 깔끔한 서류 전달 |
| **Good** | **스탬프 (Stamp)** | 자료 구조 전체를 전달하나 일부만 사용 | 가방 통째로 빌려주기 |
| **Fair** | **제어 (Control)** | 무엇을 할지 지시하는 플래그를 전달 | 상사가 부하에게 명령 |
| **Worst** | **내용 (Content)** | 다른 모듈의 내부 데이터/로직을 직접 참조 | 남의 일기장 훔쳐보기 |

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: '스파게티 코드' 쇼핑몰 결제 로직 리팩토링

- **문제 상황**: 주문 로직 안에 결제 처리, 이메일 발송, 재고 차감, 포인트 적립 코드가 모두 뒤섞여 있어, 이메일 서버 주소만 바꿔도 결제 기능이 멈추는 현상 발생.
- **의사결정 (설계 원칙 적용)**:
  1. **관심사 분리**: 결제 서비스, 알림 서비스, 재고 서비스로 클래스를 완전히 분리한다.
  2. **추상화 도입**: 이메일 발송 기능을 직접 호출하지 않고 `NotificationStrategy` 인터페이스를 통해 호출하도록 하여, 나중에 카카오톡 알림으로 바꿔도 주문 로직이 변하지 않게 한다.
  3. **결합도 낮추기 (이벤트 활용)**: 주문 완료 시점에 직접 다른 서비스를 호출하는 대신 'OrderCompleted' 이벤트를 발행하고, 재고와 포인트 모듈이 이 이벤트를 구독(Subscribe)하게 하여 의존성을 끊는다.
  4. **응집도 확인**: 결제 서비스 내부에는 오직 'PG사 연동'과 '결제 결과 처리' 로직만 남겨 기능적 응집도를 최상으로 올린다.

### 도입 체크리스트
- **Ripple Effect**: 이 모듈을 수정했을 때 다른 모듈 몇 개가 영향을 받는가? (결합도 측정)
- **Code Size**: 하나의 함수나 클래스가 너무 길지는 않은가? (응집도 저하 징후)
- **Dependency Graph**: 순환 의존성(A ──▶ B ──▶ A)이 존재하지 않는가?

### 안티패턴
- **God Object (만능 객체)**: 시스템의 거의 모든 일을 혼자 처리하는 거대 클래스. 응집도는 낮고 결합도는 최악이다.
- **Leaky Abstraction (새는 추상화)**: 내부 구현의 복잡성이 인터페이스 밖으로 흘러나와, 사용자가 내부를 알아야만 쓸 수 있는 가짜 추상화.

- **📢 섹션 요약 비유**: 화장실 변기가 막혔는데 주방 싱크대까지 물이 역류한다면(높은 결합도), 그 집의 배관 설계는 처음부터 다시 해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Complexity)** | 코드 복잡도 40% 감소 | 모듈화 및 관심사 분리를 통해 인지 부하 감소 |
| **정량적 (Agility)** | 기능 추가 속도 2배 향상 | 독립적 모듈 구조로 영향 범위 파악 및 테스트 시간 단축 |
| **정성적 (Stability)** | 시스템 안정성 강화 | 특정 모듈의 오류가 전체 시스템으로 전이되는 현상 차단 |
| **정성적 (Reusable)** | 부품 재사용성 확보 | 잘 설계된 응집도 높은 모듈은 다른 프로젝트에서도 즉시 활용 가능 |

### 미래 전망
- **Micro-Architecture Analysis**: AI가 소스 코드의 의존성 그래프를 분석하여, 결합도가 높은 지점을 자동으로 찾아내고 리팩토링 경로를 제시해줄 것이다.
- **Architectural Fitness Functions**: 아키텍처 원칙 준수 여부를 테스트 코드로 검증하는 기법이 보편화되어, 설계가 무너지는 것을 실시간으로 방어할 것이다.

### 참고 표준
- **Clean Architecture (Robert C. Martin)**: 설계 원칙의 현대적 집대성
- **Design Patterns (GoF)**: 원칙을 구체적 상황에 적용한 해결책
- **ISO/IEC 25010**: 소프트웨어 품질 평가 표준 (유지보수성 등)

- **📢 섹션 요약 비유**: 수만 개의 부품이 얽힌 기계가 시계처럼 정교하게 돌아가는 비결은, 각 부품이 자기 일만 열심히 하고(응집도) 서로 방해하지 않게(결합도) 설계된 아키텍처의 힘입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 레고 로봇을 만들 때 **머리, 팔, 다리를 따로따로 만들어서 끼우는 것**과 같아요. (관심사 분리)
2. 팔을 고치고 싶을 때 다리까지 다 뜯어낼 필요 없이, **팔만 쏙 빼서 고칠 수 있게 만드는 게 똑똑한 설계**랍니다. (낮은 결합도)
3. 이렇게 조각조각 잘 나누어 놓으면, 나중에 로봇을 자동차로 변신시키는 것도 훨씬 쉬워져요!
