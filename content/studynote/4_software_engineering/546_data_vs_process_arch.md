# 546. 데이터 중심 vs 프로세스 중심 아키텍처

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 중심 아키텍처는 **데이터의 구조와 저장 방식**을 설계의 최우선 순위에 두는 정적인 접근 방식이며, 프로세스 중심 아키텍처는 시스템이 수행하는 **업무 흐름과 제어 로직**을 중심으로 설계를 풀어나가는 동적인 접근 방식이다.
> 2. **가치**: 데이터 중심은 일관된 정보 관리와 고성능 트랜잭션 처리에 유리하며, 프로세스 중심은 비즈니스 로직의 잦은 변경과 복잡한 업무 절차를 유연하게 수용하는 데 강점이 있다.
> 3. **융합**: 현대 아키텍처는 두 방식의 장점을 결합한 도메인 주도 설계(DDD)나 마이크로서비스(MSA)를 통해, 데이터의 무결성과 프로세스의 민첩성을 동시에 확보하는 방향으로 진화하고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **데이터 중심 (Data-Centric)**: "어떤 데이터를 저장할 것인가?"를 먼저 고민한다. 데이터베이스 스키마를 가장 먼저 설계하고 애플리케이션은 이를 조회/수정하는 도구로 본다.
  - **프로세스 중심 (Process-Centric)**: "어떤 기능을 수행할 것인가?"를 먼저 고민한다. 업무의 흐름(Workflow)을 먼저 정의하고 데이터는 그 과정에서 발생하는 부수적인 산물로 취급한다.

- **💡 비유**: 
  - **데이터 중심**: 도서관을 지을 때 **"책장(DB 스키마)을 먼저 튼튼하게 배치"**하고 책을 채워 넣는 방식입니다. 책장 구조가 완벽하면 책을 찾기(조회) 매우 쉽습니다.
  - **프로세스 중심**: 카페를 지을 때 **"손님이 주문하고 커피를 받는 동선(업무 흐름)"**을 먼저 설계하는 방식입니다. 동선이 잘 짜여 있으면 손님이 많아져도 주문을 빠르게 처리할 수 있습니다.

- **등장 배경**:
  1. **정보 시스템의 초기 모델**: 과거 메인프레임 시절에는 데이터 저장 비용이 비쌌기 때문에 효율적인 데이터 구조화(데이터 중심)가 아키텍처의 전부였다.
  2. **비즈니스 환경의 급변**: 시장의 요구사항이 수시로 변하면서, 고정된 데이터 구조보다는 유연하게 변하는 비즈니스 로직(프로세스 중심)을 관리하는 능력이 중요해졌다.
  3. **객체지향 및 도메인 모델링**: 데이터와 행위를 하나로 묶으려는 시도가 늘어나면서 두 관점을 통합하려는 노력이 지속되고 있다.

- **📢 섹션 요약 비유**: 단단한 금고(데이터 중심)를 만들 것인지, 아니면 빠르게 흐르는 강물(프로세스 중심)을 만들 것인지, 시스템의 목적에 따라 설계의 무게추를 결정하는 전략적 선택입니다.

---

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 데이터 중심 아키텍처 (Data-Centric Architecture)

데이터베이스가 모든 컴포넌트의 중심(Hub)이 되는 구조다.

| 특징 | 상세 내용 | 비유 |
|:---|:---|:---|
| **중앙 집중형** | 공통된 DB를 여러 애플리케이션이 공유함 | 마을 공통 우물 |
| **강력한 정합성** | DB 제약 조건(PK, FK)을 통해 데이터 무결성 보장 | 엄격한 장부 기록 |
| **수동적 앱** | 애플리케이션은 DB의 형태에 맞춰 로직을 작성함 | 틀에 부은 붕어빵 |

---

### 2. 프로세스 중심 아키텍처 (Process-Centric Architecture)

업무의 흐름과 컴포넌트 간의 메시지 교환이 중심이 되는 구조다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      프로세스 중심 아키텍처의 서비스 흐름 (Event-driven)          │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [사용자 요청] ──▶ [주문 프로세스] ──▶ [결제 프로세스] ──▶ [배송 프로세스] ──▶ [완료] │
 │      │                 │                 │                 │                │
 │      │ (Msg)           ▼                 ▼                 ▼                │
 │      └───────────▶ [개별 데이터]      [개별 데이터]      [개별 데이터]            │
 │                    (Local DB)        (Local DB)        (Local DB)           │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 특징:                                                                 │  │
 │  │ 1. 흐름의 유연성: 결제 단계 뒤에 '적립' 단계를 쉽게 끼워 넣을 수 있음           │  │
 │  │ 2. 독립적 진화: 각 프로세스는 자신만의 최적화된 데이터를 가짐                 │  │
 │  │ 3. 메시지 지향: 프로세스 간 소통은 데이터 공유가 아닌 이벤트로 수행됨          │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 프로세스 중심 아키텍처에서는 전체를 관통하는 거대한 공통 DB가 없다. 대신 **비즈니스 워크플로우**를 따라가며 각 단계(프로세스)가 자신의 임무를 수행한다. 각 프로세스는 자신에게 꼭 필요한 **지역 데이터(Local Data)**만을 소유하며, 다른 단계와의 소통은 메시지나 이벤트를 통해 느슨하게 연결된다. 이는 시스템의 특정 단계를 수정하거나 교체할 때 다른 단계에 미치는 영향이 적어 **민첩성(Agility)**을 극대화하는 아키텍처다.

---

### 심층 동작 원리: 결합 방식의 차이 (Coupling)

1. **데이터 공유 결합 (Common Coupling)**: 데이터 중심 방식에서 흔히 발생. 여러 모듈이 동일한 전역 데이터나 DB 테이블을 공유하므로, 테이블 구조 하나만 바꿔도 모든 모듈을 고쳐야 함. (결합도 높음)
2. **메시지 전달 (Message Passing)**: 프로세스 중심 방식. 모듈은 서로의 데이터를 알 필요 없이 "이 일을 해줘"라는 메시지만 던짐. (결합도 낮음)

- **📢 섹션 요약 비유**: 한 장의 도화지에 여러 명이 그림을 같이 그리는 것(데이터 공유)과, 각자 종이에 그린 뒤 테이프로 이어 붙이는 것(프로세스 중심)의 차이와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 데이터 중심 vs 프로세스 중심 비교 매트릭스

| 비교 항목 | 데이터 중심 아키텍처 | 프로세스 중심 아키텍처 |
|:---|:---|:---|
| **핵심 설계 목표** | **정보의 무결성과 통합** | **업무 흐름의 자동화와 유연성** |
| **주요 기술** | RDBMS, SQL, 정규화 | SOA, MSA, BPM, Event-bus |
| **변경 영향도** | 데이터 구조 변경 시 파급력 큼 | 프로세스 변경 시 파급력 작음 |
| **성능 특징** | 대량 데이터 조인 및 집계 강점 | 비동기 처리를 통한 동시성 확보 강점 |
| **적합한 시스템** | 통계 시스템, 전통적 ERP | 마케팅 이벤트, 이커머스, 워크플로우 |

---

### 기술 융합: 현대의 하이브리드 접근법 (DDD)

최근에는 두 방식을 이분법적으로 나누지 않고 융합한다.
- **도메인 주도 설계 (DDD)**: 프로세스(행위)와 데이터(속성)를 하나의 **애그리거트 (Aggregate)**로 묶어 설계한다.
- **가치**: 데이터의 정합성을 지키면서도 업무 로직의 독립성을 확보하여 MSA로 가는 가장 견고한 설계도를 제공한다.

- **📢 섹션 요약 비유**: 서랍장(데이터)도 잘 짜야 하고 동선(프로세스)도 좋아야 훌륭한 주방이 되듯, 현대 소프트웨어는 두 관점의 조화가 필수입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 은행의 계정계 vs 채널계 시스템 설계

- **문제 상황**: 은행이 신규 앱 서비스를 런칭하면서 기존의 견고한 중앙 DB 구조를 유지할지, 아니면 빠른 기능 추가를 위해 분산 구조를 택할지 고민 중.
- **의사결정 (아키텍처 믹스 전략)**:
  1. **Core Banking (데이터 중심)**: 돈의 출납과 이자는 1원이라도 틀리면 안 되므로, 강력한 트랜잭션과 데이터 정합성을 보장하는 **데이터 중심의 중앙 DB 아키텍처**를 유지한다.
  2. **Channel API (프로세스 중심)**: 이벤트 알림, 상품 추천, 사용자 프로필 관리 등은 수시로 UI와 로직이 변하므로, **프로세스 중심의 MSA 아키텍처**를 채택하여 민첩하게 대응한다.
  3. **데이터 동기화 (CQRS)**: 핵심 시스템(Write)의 데이터가 변하면 메시지 큐를 통해 채널계(Read)로 데이터를 전파하여, 두 세계를 안전하게 연결한다.

### 도입 체크리스트
- **Data Integrity vs Flow Agility**: 우리 시스템에서 1순위 가치는 '데이터의 완벽성'인가, '업무의 빠른 변화'인가?
- **Shared Data Risk**: 여러 팀이 공유 DB를 쓸 때, 한 팀의 스키마 변경이 다른 팀의 장애를 유발하고 있지는 않은가?
- **Process Orchestration**: 복잡한 프로세스 중심 설계를 할 때, 전체 흐름을 제어하는 오케스트레이터(Orchestrator)가 병목이 되지는 않는가?

### 안티패턴
- **DB를 IPC 수단으로 사용**: 프로세스 간 통신을 메시지가 아닌 DB 테이블 상태 값(`status = 'Y'`)을 바꿔가며 수행하는 행위. 데이터 중심의 폐해인 '강한 결합'을 초래한다.
- **상태 없는 무분별한 분산**: 프로세스를 너무 쪼개어, 정작 중요한 데이터의 최종 상태를 확인하기 위해 수십 개의 서비스를 조인해야 하는 상황.

- **📢 섹션 요약 비유**: 모든 소식을 마을 게시판(공유 DB)에만 적으라고 강요하거나, 모든 편지를 직접 전달(메시지)만 하라고 강요하는 극단적인 정책은 모두 소통의 장애를 부릅니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Integrity)** | 데이터 오류 발생 0% | 데이터 중심 설계 시 정규화와 제약 조건을 통한 정합성 극대화 |
| **정량적 (Speed)** | 신규 기능 출시 속도 3배 | 프로세스 중심 설계 시 모듈 간 간섭 없는 독립 배포 실현 |
| **정성적 (Standard)** | 비즈니스 로직 명확화 | 프로세스 중심 설계를 통해 실제 업무 흐름을 코드와 일치시킴 |
| **정성적 (Asset)** | 정보 자산의 체계적 축적 | 데이터 중심 설계를 통해 전사 정보의 표준화된 자산화 가능 |

### 미래 전망
- **Data Mesh**: 데이터 중심의 중앙 집중화를 탈피하여, 데이터 자체를 각 도메인이 책임지고 관리하는 '프로세스 중심의 데이터 아키텍처'가 확산될 것이다.
- **Serverless Orchestration**: 서버 인프라는 완전히 사라지고, 오직 비즈니스 흐름(Step Functions 등)만 설계하면 최적의 데이터 저장소가 자동으로 할당되는 'No-infra' 아키텍처가 실현될 것이다.

### 참고 표준
- **Information Engineering (James Martin)**: 데이터 중심 아키텍처의 고전적 이론
- **SOA (Service Oriented Architecture)**: 프로세스 중심 아키텍처의 대표적 모델
- **DAMA-DMBOK**: 데이터 관리 지식 체계 가이드

- **📢 섹션 요약 비유**: 집을 지을 때 튼튼한 금고(데이터)를 먼저 놓을지, 편리한 복도(프로세스)를 먼저 뚫을지는 집주인의 목적에 달렸습니다. 진정한 건축가(아키텍트)는 두 가지의 균형을 맞추는 사람입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **데이터 중심**은 보물 상자를 먼저 아주 튼튼하게 만들고, 그 안에 보물을 예쁘게 차곡차곡 쌓아두는 방식이에요.
2. **프로세스 중심**은 보물을 옮기는 "미끄럼틀 통로"를 먼저 만들고, 보물이 잘 굴러가게 만드는 방식이랍니다.
3. 보물을 잘 보관하고 싶을 때는 상자를 먼저 만들고, 보물을 빨리 보내고 싶을 때는 통로를 먼저 만드는 게 좋겠죠?
