# 571. 서브루틴 (Subroutine)과 코루틴 (Coroutine)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서브루틴은 한 번 호출되면 끝날 때까지 실행권을 독점하는 **계층적(Hierarchical) 실행 구조**인 반면, 코루틴은 실행 도중 멈추고 나중에 멈춘 지점부터 다시 시작할 수 있는 **비선점형 협력적(Cooperative) 멀티태스킹 구조**다.
> 2. **가치**: 코루틴은 쓰레드(Thread) 교체 시 발생하는 값비싼 컨텍스트 스위칭 오버헤드 없이도 수만 개의 작업을 동시에 처리할 수 있는 **경량 동시성(Concurrency)**을 제공하여 고성능 비동기 프로그래밍의 핵심 도구로 쓰인다.
> 3. **융합**: 파이썬(async/await), 코틀린(Coroutine), Go(Goroutine) 등 현대 언어의 비동기 처리 핵심 기술이며, 네트워크 I/O 바운드 작업과 대규모 트래픽 처리가 중요한 마이크로서비스 아키텍처의 필수 요소다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **서브루틴 (Subroutine)**: 우리가 흔히 아는 함수(Function). 호출되면 진입(Entry)하고, 마지막 줄까지 실행한 뒤 종료(Exit)하며 제어권을 돌려준다.
  - **코루틴 (Coroutine)**: '협동(Co)' + '루틴(Routine)'. 실행 중에 스스로 멈추고(Suspend/Yield) 제어권을 양보했다가, 나중에 중단된 시점의 상태 그대로 다시 실행(Resume)할 수 있다.

- **💡 비유**: 
  - **서브루틴**: **"단거리 육상 경기"**와 같습니다. 출발 신호와 함께 뛰기 시작하면, 결승선을 통과할 때까지는 멈추지 않고 오직 앞만 보고 달려야 합니다.
  - **코루틴**: **"온라인 게임의 체크포인트"**와 같습니다. 던전을 돌다가 잠시 저장을 하고 로그아웃(Suspend)해도, 나중에 다시 접속(Resume)하면 그 자리, 그 아이템 그대로 다시 모험을 시작할 수 있는 것과 같습니다.

- **등장 배경**:
  1. **쓰레드 관리의 한계**: 동시성을 위해 쓰레드를 수만 개 만들면 메모리 사용량이 폭증하고, CPU가 쓰레드를 갈아 끼우는 비용(Context Switch) 때문에 배보다 배꼽이 더 커지는 현상이 발생했다.
  2. **비동기 프로그래밍의 복잡성**: 콜백(Callback) 지옥이나 복잡한 상태 머신 없이, 마치 동기 코드처럼 읽기 쉬운 비동기 코드를 짜고 싶은 요구가 커졌다.
  3. **I/O 대기 시간 활용**: 데이터베이스나 네트워크 응답을 기다리는 동안 CPU가 놀지 않고 다른 코루틴을 실행하게 함으로써 자원 효율을 극대화하고자 했다.

- **📢 섹션 요약 비유**: 일을 끝내야만 다음 일을 할 수 있는 '완벽주의자(서브루틴)'에서, 여러 일을 조금씩 진척시키며 조율하는 '멀티태스커(코루틴)'로 프로그래밍의 일하는 방식이 바뀐 것입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 실행 흐름 비교: 계층적 vs 협력적

| 구분 | 서브루틴 (Subroutine) | 코루틴 (Coroutine) |
|:---|:---|:---|
| **진입/탈출** | 한 개의 진입점과 종료점 | **여러 개의 진입/중단/재개 지점** |
| **제어권** | 호출자가 피호출자를 완전히 통제 | **루틴끼리 서로 제어권을 주고받음** |
| **상태 보존** | 종료 시 로컬 변수와 상태 소멸 | **중단 시 스택/변수 상태를 힙에 보존** |
| **메커니즘** | Stack 기반 (LIFO) | **Heap 기반 (State Machine)** |

---

### 2. 코루틴의 비선점형 멀티태스킹 아키텍처

쓰레드와 코루틴이 어떻게 매핑되어 돌아가는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      코루틴 (Coroutine) 실행 및 중단 메커니즘                     │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [OS Thread (Worker)]                                                        │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │  현재 실행 중인 코루틴: Coroutine A                                       │  │
 │  │  (비즈니스 로직 수행...)                                                  │  │
 │  │  1. I/O 요청 발생 (DB 조회 등)                                            │  │
 │  │  2. suspend 키워드 작동! ──▶ [A의 상태(Stack)를 Heap에 저장]               │  │
 │  │  3. 쓰레드 제어권을 스케줄러에 반납                                         │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │          ▲                                     │                            │
 │          │ (Resume)                            │ (Suspend)                  │
 │          │                                     ▼                            │
 │  ┌──────────────────────────────────┐  ┌──────────────────────────────────┐ │
 │  │      스케줄러 (Dispatcher)        │  │      코루틴 대기실 (Heap)          │ │
 │  │  - "놀고 있는 쓰레드 찾기"         │  │  [A (IDLE)]  [B (READY)]         │ │
 │  │  - "B를 실행시켜라!"               │  │  [C (WAIT)]                      │ │
 │  └──────────────────────────────────┘  └──────────────────────────────────┘ │
 │                                                                             │
 │  ※ 핵심: 쓰레드는 멈추지 않고 계속 일함 (Non-blocking)                         │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 코루틴 아키텍처의 핵심은 **쓰레드를 블로킹(Blocking)하지 않는다**는 점이다. **Coroutine A**가 실행 중 네트워크 응답을 기다려야 한다면, 쓰레드 자체를 멈추게 하는 대신 자기 자신을 **Suspend(중단)** 상태로 바꾸고 힙 메모리에 자신의 데이터를 저장한다. 그러면 자유로워진 **OS Thread**는 즉시 대기 중인 **Coroutine B**를 집어 들어 실행한다. 나중에 네트워크 응답이 오면, A는 다시 실행 가능한 상태가 되어 비어있는 쓰레드에 의해 **Resume(재개)** 된다. 이 과정은 커널이 개입하지 않는 유저 레벨(User-mode)에서 일어나므로 속도가 매우 빠르다.

---

### 심층 동작 원리: 상태 머신 (State Machine) 변환

컴파일러가 코루틴 코드를 어떻게 바꾸는지에 대한 원리다.
1. **Continuation**: 중단 시점의 정보와 남은 코드를 담은 객체.
2. **Labeling**: 코루틴 내부의 중단 지점들마다 라벨(`case 0, 1, 2...`)을 붙인다.
3. **Switch-Case**: 재개될 때 라벨 정보를 확인하여, 마지막으로 멈췄던 라벨 위치로 즉시 점프하여 코드를 이어 나간다.

- **📢 섹션 요약 비유**: 긴 소설을 읽다가 책갈피(Continuation)를 끼워두고 덮은 뒤, 나중에 다시 펼쳤을 때 그 페이지부터 읽기 시작하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 쓰레드 (Thread) vs 코루틴 (Coroutine) 상세 비교

| 비교 항목 | 쓰레드 (Thread) | 코루틴 (Coroutine) |
|:---|:---|:---|
| **관리 주체** | OS 커널 (Kernel-level) | **프로그래밍 언어/런타임 (User-level)** |
| **메모리 점유** | 큼 (보통 1MB 이상 스택 할당) | **매우 작음 (수 KB 단위)** |
| **컨텍스트 스위칭** | 비쌈 (커널 모드 전환 필요) | **저렴 (단순한 객체 상태 변경)** |
| **동작 방식** | 선점형 (OS가 강제 중단) | **비선점형 (코드가 스스로 양보)** |
| **수용량** | 수백~수천 개가 한계 | **수십만~수백만 개 가능** |

---

### 기술 융합: Structured Concurrency

코루틴의 생명주기를 안전하게 관리하기 위한 설계 원칙이다.
- **부모-자식 관계**: 부모 코루틴은 모든 자식 코루틴이 끝날 때까지 종료되지 않는다.
- **에러 전파**: 자식에서 에러가 나면 부모에게 즉시 알려 전체 작업을 취소하거나 안전하게 처리한다.
- **가치**: 비동기 작업이 "좀비"가 되어 메모리를 갉아먹는 현상을 방지한다.

- **📢 섹션 요약 비유**: 아이들이 소풍을 갔을 때, 선생님(부모)이 아이들(자식)의 인원수를 다 확인하고 안전하게 버스에 태워야 소풍이 끝나는 책임 관리제와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 동시 접속자가 급증하는 '채팅 서버' 아키텍처 설계

- **문제 상황**: 쓰레드 기반 서버(Thread-per-connection)로 채팅 서버를 만들었더니, 접속자가 1만 명을 넘어서는 순간 쓰레드 생성 한계와 컨텍스트 스위칭 부하로 서버가 뻗어버림.
- **의사결정 (코루틴 도입 전략)**:
  1. **언어 선정**: 코루틴을 네이티브하게 지원하는 **Go (Goroutine)**나 **Kotlin (Coroutine)**을 채택한다.
  2. **I/O 비동기화**: 소켓 통신부를 블로킹 방식이 아닌 **코루틴 서스펜드** 방식으로 변경한다.
  3. **자원 최적화**: 1만 개의 연결마다 쓰레드를 만드는 대신, 실제 CPU 코어 수와 유사한 소수의 워커 쓰레드만 두고 그 위에서 1만 개의 코루틴이 교대로 돌아가게 한다.
  4. **결과**: 메모리 사용량을 1/100로 줄이면서도, 더 많은 접속자를 더 빠른 응답 속도로 수용하는 고가용성 서버를 완성한다.

### 도입 체크리스트
- **Blocking Call**: 코루틴 내부에서 전통적인 '블로킹 I/O' 라이브러리를 쓰고 있지는 않은가? (전체 쓰레드가 멈추는 대참사 발생)
- **Error Handling**: 비동기 흐름 속에서 예외 발생 시 적절하게 Catch하여 전파하고 있는가?
- **Dispatcher Selection**: CPU 연산 작업(Default)과 I/O 작업(IO)에 맞는 적절한 스케줄러를 선택했는가?

### 안티패턴
- **GlobalScope 남용**: 수명 주기를 관리하지 않는 전역 코루틴을 남발하여 메모리 릭을 유발하는 행위.
- **CPU Bound 작업에 코루틴 사용**: 코루틴은 대기(Wait)가 많은 작업에 유리하다. 쉬지 않고 연산만 하는 코드에 코루틴을 쓰면 오히려 오버헤드만 늘어난다.

- **📢 섹션 요약 비유**: 낚시찌가 가라앉기를 기다리는 100명의 낚시꾼(코루틴)은 한 명의 감독관(쓰레드)이 관리할 수 있지만, 100명이 동시에 삽질(CPU 연산)을 해야 한다면 결국 100개의 삽과 100명의 일꾼(쓰레드)이 필요하다는 뜻입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Performance)** | 처리량(Throughput) 5배 향상 | 자원 대기 시간의 효율적 활용으로 서버 가용 자원 극대화 |
| **정량적 (Cost)** | 인프라 비용 절감 | 적은 메모리와 CPU로 더 많은 트래픽을 감당하여 서버 증설 비용 억제 |
| **정성적 (Readability)** | 코드 가독성 향상 | 비동기 로직을 '순차적'인 동기 코드 스타일로 작성 가능 |
| **정성적 (Stability)** | 시스템 안정성 확보 | 구조적 동시성을 통한 리소스 누수 및 경쟁 상태(Race Condition) 예방 |

### 미래 전망
- **Virtual Threads (Project Loom)**: 자바 언어 차원에서 코루틴과 유사한 '가상 쓰레드'를 도입하여, 기존 블로킹 코드를 수정 없이 코루틴처럼 실행하는 혁신이 일어날 것이다.
- **Hardware-level Coroutine**: CPU 아키텍처가 코루틴의 상태 저장과 복원을 하드웨어 수준에서 가속화하여 동시성 성능이 한 단계 더 도약할 것이다.

### 참고 표준
- **Python PEP 492**: Coroutines with async and await syntax
- **Kotlin Coroutines Guide**: 실무 동시성 관리의 표준 가이드
- **C++20 Coroutines**: 고성능 언어의 코루틴 표준

- **📢 섹션 요약 비유**: 쉼표 없이 달려야 했던 과거의 함수에서, 스스로 숨을 고르며(Suspend) 동료와 협력(Cooperative)하는 현대의 코루틴으로의 진화는 소프트웨어가 인간의 사고방식을 닮아가는 과정입니다.
