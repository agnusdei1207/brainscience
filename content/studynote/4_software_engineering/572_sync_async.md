# 572. 동기(Sync) vs 비동기(Async) 및 블로킹(Blocking)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 동기/비동기는 **작업 완료 여부를 누가 확인하느냐(제어의 시점)**에 대한 문제이며, 블로킹/논블로킹은 **제어권(Control)을 누가 가지고 있느냐(실행의 자유)**에 대한 시스템적 상태를 의미한다.
> 2. **가치**: 대규모 트래픽 환경에서 **비동기 논블로킹(Async Non-blocking)** 모델을 적용하면, I/O 대기 시간 동안 CPU가 다른 작업을 수행할 수 있어 하드웨어 자원 효율성과 시스템 처리량(Throughput)을 극대화할 수 있다.
> 3. **융합**: 이벤트 루프(Node.js), 리액티브 스택(WebFlux), 그리고 코루틴 기술의 근간이 되는 이론이며, 현대 분산 시스템 아키텍처에서 서비스 간 통신 성능을 결정짓는 핵심 설계 요소다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **동기 (Synchronous)**: 요청한 작업의 결과가 나올 때까지 기다리거나, 결과를 직접 챙기는 방식. (시간의 일치)
  - **비동기 (Asynchronous)**: 요청한 후 결과는 나중에 통보(Callback)받기로 하고 바로 다른 일을 하는 방식. (시간의 불일치)
  - **블로킹 (Blocking)**: 호출된 함수가 자신의 할 일을 마칠 때까지 제어권을 돌려주지 않아 호출자가 멈춰있는 상태.
  - **논블로킹 (Non-blocking)**: 호출된 함수가 할 일이 남았어도 제어권을 즉시 돌려주어 호출자가 계속 일할 수 있는 상태.

- **💡 비유**: 
  - **동기/블로킹**: 식당에서 음식을 주문하고, **나올 때까지 주방만 쳐다보며 서 있는 것.**
  - **비동기/논블로킹**: 식당에서 진동벨(Callback)을 받고, **자리에 앉아 책을 읽으며 기다리는 것.**

- **등장 배경**:
  1. **C10K 문제**: 동시에 1만 개의 연결을 처리해야 하는 요구사항이 생기면서, 쓰레드를 늘리는 방식(동기/블로킹)의 한계가 드러났다.
  2. **I/O 병목 현상**: CPU 속도에 비해 네트워크나 디스크(I/O)는 수백 배 느리므로, 이 대기 시간을 효율적으로 활용할 방법이 필요해졌다.
  3. **사용자 경험(UX) 향상**: 화면이 멈추지 않고 데이터를 로딩하는 부드러운 웹/앱 인터페이스를 위해 비동기 기술이 필수화되었다.

- **📢 섹션 요약 비유**: 일을 시키고 옆에서 감시할 것인지(Sync), 아니면 결과가 나오면 전화 달라고 하고 내 할 일을 할 것인지(Async)에 대한 효율적인 업무 분담론입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 4가지 조합의 매트릭스 분석

| 구분 | Blocking (제어권 점유) | Non-blocking (제어권 반환) |
|:---|:---|:---|
| **Sync (결과 확인)** | **1. Synchronous Blocking**<br>가장 일반적. 결과 나올 때까지 멈춤. | **2. Synchronous Non-blocking**<br>결과 안 나왔어도 제어권은 받음. 주기적으로 "다 됐니?" 물어봄(Polling). |
| **Async (결과 위임)** | **3. Asynchronous Blocking**<br>실무에선 비효율적. (잘못된 구현 혹은 의도적 대기) | **4. Asynchronous Non-blocking**<br>**최고 효율.** 시켜놓고 내 일 하다가 완료 알림 받음. |

---

### 2. 동기/블로킹 vs 비동기/논블로킹 실행 흐름

두 방식의 처리 효율 차이를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      동기/블로킹 vs 비동기/논블로킹 흐름 비교                     │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. Sync Blocking]                          [2. Async Non-blocking]         │
 │                                                                             │
 │  App (Thread)      OS/DB                     App (Thread)      OS/DB        │
 │      │               │                           │               │          │
 │      │ ───(요청)──▶  │                           │ ───(요청)──▶  │          │
 │      │               │                           │ ◀──(제어권)── ┘          │
 │      │ [대기/정지]    │                           │               │          │
 │      │               │                           │ [다른 작업 수행] │          │
 │      │               │                           │               │          │
 │      │ ◀──(결과)───  │                           │ ◀──(Callback) ┘          │
 │      │               │                           │               │          │
 │      ▼               ▼                           ▼               ▼          │
 │                                                                             │
 │  ※ 특징: Sync는 CPU가 놀고 있음                ※ 특징: Async는 CPU가 풀가동 됨      │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** **Sync Blocking** 방식은 요청을 보낸 후 응답이 올 때까지 앱 쓰레드가 아무것도 못 하고 멈춰 있다. 이는 자원의 낭비다. 반면 **Async Non-blocking** 방식은 요청을 보내자마자 제어권을 돌려받아 다른 업무를 처리한다. 나중에 데이터가 준비되면 OS나 DB가 이벤트를 던져주고, 그때서야 앱은 하던 일을 멈추고 결과를 처리한다. 이 방식은 적은 수의 쓰레드로 수만 개의 요청을 처리할 수 있게 하는 핵심 원리다.

---

### 심층 동작 원리: 커널 I/O 모델 (Select, Poll, Epoll)

비동기 논블로킹을 가능하게 하는 OS 레벨의 기술이다.
1. **Select/Poll**: 수천 개의 소켓을 루프를 돌며 "데이터 왔니?"라고 물어봄. (비효율적)
2. **Epoll (Linux) / Kqueue (BSD)**: 데이터가 들어온 소켓만 OS가 리스트로 알려줌. (O(1) 성능, 고성능 서버의 핵심)
3. **가치**: 수만 명의 손님이 있는 카페에서 벨이 울리는 손님만 찾아가 서빙하는 지능형 대기 시스템과 같다.

- **📢 섹션 요약 비유**: 낚시터에서 100개의 낚싯대를 펴놓고 하나하나 들어보며 고기가 잡혔나 확인하는 것(Poll)과, 물고기가 잡히면 소리가 나는 스마트 방울(Epoll)을 달아두는 것의 차이입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 실무적 관점의 비동기 구현 기술 비교

| 기술 명칭 | 주요 특징 | 장점 | 단점 |
|:---|:---|:---|:---|
| **Callback** | 함수 실행 후 호출할 함수를 인자로 전달 | 직관적 구현 | **Callback Hell (가독성 최악)** |
| **Promise/Future** | 미래에 완료될 작업의 대리자 객체 | 연쇄 호출(Chaining) 가능 | 에러 처리가 다소 복잡함 |
| **async/await** | 비동기를 동기 코드처럼 작성 | **가독성 최고**, 에러 처리 쉬움 | 코루틴 개념 이해 필요 |
| **Reactive** | 데이터 스트림과 전파에 집중 | 대량 스트림 처리에 최적화 | 러닝 커브가 매우 높음 |

---

### 기술 융합: Node.js와 Event Loop

Node.js는 단일 쓰레드로 어떻게 수많은 요청을 처리하는가?
- **원리**: 메인 쓰레드는 비동기 논블로킹 방식으로 명령만 내리고, 실제 무거운 I/O 작업은 내부의 Libuv 쓰레드 풀이나 OS 커널에 맡긴다. 작업이 끝나면 **이벤트 루프**가 콜백을 메인 쓰레드에 전달한다.
- **교훈**: "단순한 일은 빨리 처리하고, 오래 걸리는 일은 남에게 맡긴다"는 분업의 미학이다.

- **📢 섹션 요약 비유**: 주방장(쓰레드)은 한 명이지만, 재료 손질과 설거지는 자동 기계(OS 커널)에게 맡기고 주방장은 쉴 새 없이 요리 지시만 내리는 초고속 주방과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: API 게이트웨이의 성능 최적화

- **문제 상황**: 여러 마이크로서비스로부터 데이터를 모아 응답하는 API 게이트웨이가 동기/블록킹 방식으로 동작하여, 특정 서비스 하나만 느려져도 전체 게이트웨이 쓰레드가 고갈되어 먹통이 됨.
- **의사결정 (Async Non-blocking 전환 전략)**:
  1. **프레임워크 교체**: 블록킹 기반의 Spring MVC 대신, 논블록킹 기반의 **Spring WebFlux**나 **Netty** 엔진을 채택한다.
  2. **Client 비동기화**: 내부 서비스 호출 시 `RestTemplate`(Blocking) 대신 `WebClient`(Non-blocking)를 사용한다.
  3. **병렬 호출**: 서비스 A, B, C를 순차적으로 부르지 않고, 비동기로 동시에 호출한 뒤 결과를 합쳐서(Zip) 반환한다.
  4. **결과**: 쓰레드 수를 늘리지 않고도 동시 처리량이 5배 이상 향상되며, 특정 서비스 지연이 전체 시스템으로 전이되는 것을 차단한다.

### 도입 체크리스트
- **Library Compatibility**: 우리가 쓰는 DB 드라이버가 비동기(Reactive Driver)를 지원하는가? (블록킹 드라이버를 쓰면 비동기 프레임워크가 무용지물임)
- **Debugging**: 비동기 호출은 스택 트레이스(Stack Trace) 추적이 어렵다. 분산 트래킹(Zipkin 등) 환경이 갖춰졌는가?
- **Context Switching**: 비동기가 항상 빠른 것은 아니다. 연산이 아주 짧은 작업은 오히려 비동기 오버헤드가 더 클 수 있다.

### 안티패턴
- **Async-over-Sync**: 내부적으로 블록킹 함수를 호출하면서 겉만 `async`로 감싸는 행위. 실제로는 쓰레드가 멈추므로 비동기의 이점이 전혀 없다.
- **무분별한 비동기**: 순서가 중요한 로직까지 비동기로 처리하여 데이터 정합성이 깨지거나 코드가 난해해지는 경우.

- **📢 섹션 요약 비유**: 컵라면에 물을 붓고 3분을 기다려야 하는데(Blocking), 그동안 아무것도 안 하고 컵라면만 쳐다보고 있으면 시간을 낭비하는 것과 같습니다. 그 시간에 젓가락을 챙기는 것이 비동기적 사고입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Scaling)** | 동시 접속자 처리량 10배 증가 | 적은 쓰레드로 수만 개의 I/O 연결을 효율적으로 핸들링 |
| **정량적 (Latency)** | 시스템 지연 시간 단축 | 병렬 호출 및 I/O 대기 시간 제거로 전체 응답 속도 개선 |
| **정성적 (Resilience)** | 장애 격리 능력 향상 | 특정 작업의 병목이 전체 쓰레드 고갈로 이어지는 연쇄 장애 방지 |
| **정성적 (Standard)** | 현대적 아키텍처 대응 | MSA, 서버리스 등 클라우드 환경에 최적화된 통신 모델 확보 |

### 미래 전망
- **Unified Asynchronous Frameworks**: 언어별로 파편화된 비동기 기술들이 표준 인터페이스(예: Reactive Streams)를 중심으로 통합되어 이기종 시스템 간 연동이 쉬워질 것이다.
- **Zero-copy I/O**: 데이터 전송 시 메모리 복사를 최소화하는 비동기 기술이 하드웨어와 결합하여 네트워크 성능이 극대화될 것이다.

### 참고 표준
- **Reactive Streams**: JVM 기반 비동기 스트림 처리 표준
- **POSIX AIO / Epoll**: OS 레벨의 비동기 I/O 표준
- **Servlet 3.1+**: 자바 진영의 비동기 서블릿 표준

- **📢 섹션 요약 비유**: "기다림"을 "여유"로 바꾸는 마법, 동기와 비동기의 정확한 이해는 멈추지 않는 소프트웨어를 만드는 기술사적 통찰의 시작입니다.
