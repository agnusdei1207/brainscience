# 574. 자원 할당 전략: 스레드 풀 및 커넥션 풀

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스레드 풀(Thread Pool)과 커넥션 풀(Connection Pool)은 생성 및 소멸 비용이 큰 시스템 자원을 **미리 일정량 생성하여 저장소(Pool)에 두고 필요할 때마다 재사용**하는 고성능 자원 관리 기법이다.
> 2. **가치**: 자원 생성에 따른 지연 시간(Latency)을 제거하고, 동시에 실행되는 작업의 수를 제한하여 갑작스러운 트래픽 폭주 시에도 시스템이 **자원 고갈로 다운되지 않도록 보호하는 완충 장치** 역할을 한다.
> 3. **융합**: 웹 서버(Tomcat), 데이터베이스 라이브러리(HikariCP), 그리고 비동기 프로그래밍 엔진의 핵심 구성 요소이며, 시스템 사양에 맞춘 최적의 풀 크기 산정(Tuning)이 전체 성능의 성패를 좌우한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **Thread Pool**: 프로그램 실행 단위인 스레드를 미리 만들어 관리.
  - **Connection Pool (DBCP)**: 데이터베이스와 연결된 통로를 미리 열어 관리.
  - **공통 원리**: "만드는 데 시간이 걸린다면, 미리 만들어두고 돌려쓰자"는 재사용성 원리다.

- **💡 비유**: 자원 할당 전략은 **"콜택시 회사의 운영 방식"**과 같습니다. 손님이 부를 때마다 공장에서 차를 새로 만들어 보내는(New Instance) 것이 아니라, 주차장(Pool)에 일정 대수의 택시를 대기시켜 놓았다가 손님이 부르면 즉시 출동시키고, 일이 끝나면 다시 주차장으로 복귀시키는 것과 같습니다.

- **등장 배경**:
  1. **오버헤드의 한계**: TCP 핸드쉐이크나 OS 커널의 스레드 생성은 매우 무거운 작업이다. 요청마다 이를 반복하면 실제 업무 처리보다 준비하는 데 시간이 더 걸리게 된다.
  2. **시스템 안정성**: 제한 없는 자원 생성은 메모리 부족(OOM)이나 CPU 과부하를 초래한다. '풀'이라는 울타리를 통해 최대 사용량을 통제해야 했다.
  3. **응답 속도 개선**: 이미 준비된 자원을 즉시 할당함으로써 사용자가 체감하는 응답 속도를 획기적으로 낮추고자 했다.

- **📢 섹션 요약 비유**: 손님이 올 때마다 밥을 새로 짓는(Fresh Create) 것이 아니라, 미리 가마솥에 밥을 해두고(Pooling) 공기에 담아 바로 내놓는 맛집 주방의 효율성과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 자원 풀의 주요 구성 요소 및 동작 방식

| 요소 명칭 | 역할 및 정의 | 비유 |
|:---|:---|:---|
| **Core Size** | 평상시에 항상 유지하는 최소 자원 수 | 상시 대기 중인 택시 수 |
| **Max Size** | 최대 부하 시 늘릴 수 있는 자원의 한계치 | 예비 차량을 포함한 최대 대수 |
| **Wait Queue** | 가용 자원이 없을 때 요청이 대기하는 장소 | 택시 승강장의 대기 줄 |
| **Idle Timeout** | 사용되지 않는 유휴 자원을 정리하는 시간 | 손님 없으면 차고지로 들어가는 시간 |

---

### 2. Thread/Connection Pool 동작 아키텍처

자원이 요청되고 반납되는 전 과정을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      자원 할당 및 재사용 프로세스 (Pooling Logic)                │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Client Request] ──▶ [Resource Pool Manager] ──▶ [Active Worker] ──▶ [Job]   │
 │      │                       │ (Borrow)               │              │      │
 │      │                       ▼                        │              │      │
 │      │             ┌──────────────────┐               │              │      │
 │      │             │  Available Pool  │ ◀──────(Return)──────────────┘      │
 │      │             │  [R1][R2][R3]... │               │                     │
 │      │             └────────┬─────────┘               │                     │
 │      │                      │                         │                     │
 │      └──────(If Empty)───▶ [Wait Queue] ──────────────┘                     │
 │                            [Req 4][Req 5]...                                │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 주요 상태 판단:                                                        │  │
 │  │ 1. 가용 자원 있음: 즉시 할당 (Borrow)                                     │  │
 │  │ 2. 가용 자원 없음: 큐에서 대기 (Wait)                                     │  │
 │  │ 3. 큐도 가득 참: 요청 거부 (Rejected / Fail-fast)                         │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 클라이언트 요청이 들어오면 **Pool Manager**는 먼저 **Available Pool**에 여유 자원이 있는지 확인한다. 있다면 즉시 할당하여 업무를 수행하게 한다. 만약 자원이 모두 사용 중(**Active**)이라면, 요청은 **Wait Queue**에서 대기한다. 만약 대기 줄까지 가득 차면 시스템은 더 이상의 부하를 막기 위해 요청을 거절한다. 업무를 마친 자원은 파기되지 않고 다시 Pool로 돌아와(Return) 다음 대기자를 위해 대기한다. 이 구조는 자원 생성 시간을 0으로 만들고 시스템의 최대 부하를 물리적으로 제한한다.

---

### 심층 동작 원리: 커넥션 풀의 'Validation Query'

오랫동안 쓰지 않은 커넥션이 끊어지는 것을 방지하는 기술이다.
1. **상황**: DB는 보안이나 자원 절약을 위해 일정 시간 통신이 없는 연결을 강제로 끊는다.
2. **해결**: 풀 매니저가 주기적으로 `SELECT 1` 같은 가벼운 쿼리를 던져서 연결이 살아있는지 체크한다.
3. **가치**: 사용자가 요청했을 때 "끊어진 연결"을 주는 사고를 예방하여 서비스 신뢰성을 높인다.

- **📢 섹션 요약 비유**: 주차장에 세워둔 택시가 방전되지 않게, 주기적으로 시동을 걸어보고 엔진 상태를 점검하는 정비 활동과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 스레드 풀 vs 커넥션 풀의 차이점

| 비교 항목 | 스레드 풀 (Thread Pool) | 커넥션 풀 (Connection Pool) |
|:---|:---|:---|
| **관리 대상** | CPU 실행 단위 (Software) | **네트워크 연결 (Physical Path)** |
| **병목 지점** | CPU 연산, 메모리 | **DB 서버의 최대 동시 접속 수** |
| **제한 원칙** | CPU 코어 수에 비례하여 설정 | DB 서버 사양 및 라이선스에 비례 |
| **주요 도구** | Java ExecutorService | **HikariCP**, DBCP2 |

---

### 기술 융합: 리액티브 아키텍처와 자원 풀

비동기 논블로킹 환경에서는 풀의 성격이 바뀐다.
- **적은 수의 스레드**: 수천 개의 스레드 풀 대신, CPU 코어 수와 일치하는 아주 작은 스레드 풀(Event Loop)만 유지한다.
- **가치**: 컨테이너 간 컨텍스트 스위칭 비용을 아예 없애 성능을 극대화한다.

- **📢 섹션 요약 비유**: 수백 명의 일꾼이 각자 삽질하는 방식에서, 8명의 슈퍼 로봇(CPU 코어 수)이 수만 개의 작업을 지능적으로 나눠 처리하는 고효율 시스템으로 변하는 것입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: DB 커넥션 고갈로 인한 웹 서버 다운 사고 해결

- **문제 상황**: 갑작스러운 이벤트 트래픽으로 `Connection is not available` 에러가 발생하며 웹 사이트 접속이 불가능해짐.
- **의사결정 (튜닝 전략)**:
  1. **현상 분석**: 현재 `Max Active = 100`, `Wait Timeout = 30s`로 설정되어 있음. 요청이 몰리자 100개가 다 차고 대기 줄만 길어지다 타임아웃 발생.
  2. **임시 조치**: `Max Active`를 무작정 늘리는 대신(DB 부하 위험), 사용자에게 빠른 실패를 알려주기 위해 `Wait Timeout`을 3초로 줄인다. (Fail-fast)
  3. **근본 조치**: **HikariCP**를 도입하고, DB 서버의 `max_connections` 설정과 웹 서버의 풀 크기를 1:1에 가깝게 최적화한다.
  4. **아키텍처 확장**: DB 쿼리 성능 자체가 문제라면 **Read-only Replica**를 추가하고 조회용 커넥션 풀을 분리하여 부하를 분산한다.

### 도입 체크리스트
- **Deadlock Check**: 서비스 A가 서비스 B의 자원을 기다리며 서로 풀을 점유하는 '교착 상태' 가능성이 없는가?
- **Handover Time**: 자원을 풀에서 빌려오고 반납하는 데 드는 자체 오버헤드가 너무 크지는 않은가? (가장 빠른 HikariCP 권장)
- **Leaked Connection**: 코딩 실수로 `close()`를 호출하지 않아 풀로 돌아오지 못하는 '자원 누수'를 감시하고 있는가?

### 안티패턴
- **과도한 풀 크기**: "크면 좋겠지"라는 생각으로 수천 개의 커넥션을 여는 행위. DB 서버의 메모리와 CPU만 소모하고 실제 성능은 오히려 떨어진다. (Context Switch 부하)
- **공용 풀 사용**: 성격이 다른(배치 작업 vs 실시간 조회) 작업들이 하나의 풀을 공유하여 서로 간섭하는 경우.

- **📢 섹션 요약 비유**: 주차장이 너무 넓으면(과도한 풀) 차를 찾는 데 시간이 더 걸리고, 주차장 관리가 안 되면(자원 누수) 정작 필요할 때 차를 못 쓰게 된다는 점을 기억해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Performance)** | 응답 지연 시간 50% 이상 단축 | 자원 생성 단계의 네트워크/시스템 오버헤드 제거 |
| **정량적 (Stability)** | 시스템 가용성 보장 | 자원 사용 한도 설정을 통해 임계치 이상의 부하로부터 시스템 보호 |
| **정성적 (Resource)** | 예측 가능한 자원 관리 | 시스템이 사용하는 전체 메모리와 연결 수를 사전에 파악 및 제어 가능 |
| **정성적 (Reliability)** | 연결 안정성 향상 | 유휴 연결 관리 및 검증을 통해 신뢰할 수 있는 통로 제공 |

### 미래 전망
- **Self-Tuning Pools**: 인공지능이 실시간 트래픽을 분석하여, 풀의 크기와 타임아웃 설정을 실시간으로 자동 변경하는 자율형 자원 관리 기술이 실용화될 것이다.
- **Serverless Resource Pooling**: 클라우드 인프라 자체가 거대한 자원 풀이 되어, 개발자는 풀 크기 설정 없이도 '무제한 확장'되는 자원을 투명하게 사용하게 될 것이다.

### 참고 표준

- **Java Concurrency in Practice**: 스레드 풀 설계의 바이블
- **JDBC / DBCP Standard**: 자바 데이터베이스 연결 표준

- **📢 섹션 요약 비유**: "준비된 자가 승리한다." 자원 할당 전략은 예상치 못한 트래픽의 파도 속에서도 우리 서비스를 안전하게 띄워주는 가장 튼튼한 부표(Buoy)입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **자원 풀**은 장난감을 가지고 놀 때마다 새로 사는 게 아니라, **"커다란 장난감 상자"에 미리 넣어두고 꺼내 쓰는 것**과 같아요.
2. 장난감을 다 가지고 놀면 다시 상자에 넣어둬야 다른 친구가 빌려 쓸 수 있답니다.
3. 이렇게 상자를 잘 관리하면, 장난감을 사러 가는 시간도 아끼고 방이 엉망진창이 되는 것도 막을 수 있어요!
