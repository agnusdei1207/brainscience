# 585. 행동 주도 개발 (BDD): Given-When-Then

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 행동 주도 개발 (Behavior-Driven Development, BDD)은 TDD에서 파생된 소프트웨어 개발 방법론으로, 기술적 관점이 아닌 **사용자의 행위(Behavior)**를 중심으로 테스트 시나리오를 작성하고 이를 통해 개발을 이끌어 나가는 협업 중심의 설계 기법이다.
> 2. **가치**: 비즈니스 기획자, 개발자, QA가 모두 이해할 수 있는 자연어 형태의 **Given-When-Then** 문법을 사용하여 의사소통의 장벽을 허물며, 테스트 자체가 요구사항 명세서이자 사용자 가이드가 되는 '살아있는 문서화'를 실현한다.
> 3. **융합**: ATDD(인수 테스트 주도 개발)의 구체적인 구현 수단으로 쓰이며, Cucumber, JBehave와 같은 도구를 통해 비즈니스 시나리오를 자동화된 실행 코드로 연결하여 '올바른 기능을 올바르게' 만드는 아키텍처를 완성한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 다니엘 테레호스트 노스(Dan North)가 제안한 BDD는 "TDD를 할 때 무엇을 테스트해야 할지 모르겠다"는 질문에서 시작되었다. 핵심은 "테스트"라는 단어 대신 "행동"이라는 단어를 사용하는 것이다. 시스템이 어떻게 구현되었는지가 아니라, 사용자가 무엇을 했을 때 어떤 결과가 나타나야 하는지에 집중한다.

- **💡 비유**: BDD는 **"영화의 연극 대본 리허설"**과 같습니다. 배우(개발자)가 연기를 하기 전, 작가(기획자)와 감독(QA)이 모여 "A가 들어오면(Given), B가 화를 내고(When), C가 도망간다(Then)"라고 장면을 미리 시뮬레이션하며 대본을 맞추는 것과 같습니다. 대본이 명확하면 연기가 틀릴 리가 없습니다.

- **등장 배경**:
  1. **의사소통의 단절 (Domain Gap)**: 기획자가 말한 요구사항과 개발자가 이해한 코드가 서로 다른 '동상이몽' 현상을 기술적 용어가 아닌 비즈니스 용어로 해결하고자 했다.
  2. **TDD의 진입 장벽**: 단위 테스트 위주의 TDD는 비즈니스 가치를 직관적으로 보여주기 힘들었다. 이를 보완하여 전체 시스템의 흐름을 검증하는 방법론이 필요해졌다.
  3. **명세서의 사후 관리**: 한 번 쓰고 버려지는 기획서 대신, 항상 실제 코드와 일치하며 실행 가능한 '살아있는 명세서'를 갖고자 했다.

- **📢 섹션 요약 비유**: 복잡한 설명서 대신 "사용자 이야기(User Story)"를 한 편의 동화책처럼 읽어주어, 모두가 똑같은 결말(비즈니스 가치)을 꿈꾸게 만드는 마법 같은 협업 도구입니다.

---

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. BDD의 표준 문법: Gherkin (Given-When-Then)

누구나 읽을 수 있는 정형화된 시나리오 작성 양식이다.

| 키워드 | 역할 | 상세 내용 | 비유 |
|:---|:---|:---|:---|
| **Given** | **상황 설정** | 시스템의 초기 상태나 전제 조건을 정의 | 무대에 배우가 서 있는 상태 |
| **When** | **사건 발생** | 사용자가 수행하는 핵심 행위나 이벤트 | 갑자기 불이 꺼지는 사건 |
| **Then** | **결과 확인** | 행위에 따른 시스템의 예상 결과나 변화 | 모두가 소리를 지르는 결과 |

---

### 2. BDD 자동화 아키텍처 (Bridge Between Business & Code)

자연어 시나리오가 어떻게 실제 테스트 코드로 변환되어 작동하는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      BDD (Behavior-Driven Development) 실행 구조                  │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. Feature File (Gherkin)] ─────────────────▶ [2. Step Definition]           │
 │ (자연어 명세서)                                  (매핑 코드 - Java/Python)     │
 │                                                                             │
 │  Feature: 회원 가입                             ┌────────────────────────┐  │
 │    Scenario: 성공적인 가입                       │ @Given("가입 화면이면")  │  │
 │      Given 가입 화면에 진입함                     │ void step1() { ... }   │  │
 │      When 아이디 "hong" 입력함 ──────(Binding)──▶ │                        │  │
 │      Then 가입 성공 메시지 보임                    │ @When("아이디 {string}") │  │
 │                                                 │ void step2(String id)  │  │
 │                                                 └───────────┬────────────┘  │
 │                                                             │               │
 │ [4. 결과 리포트] ◀──────────────────────── [3. Application Code]             │
 │ (비즈니스 언어로 통과 여부 출력)                   (실제 돌아가는 로직)            │
 │                                                                             │
 │  ※ 핵심: 기획자가 쓴 문장이 곧 테스트 코드를 실행하는 스위치가 됨                   │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** BDD 아키텍처의 핵심은 **Step Definition (단계 정의)** 레이어다. 기획자가 작성한 **Feature File**의 자연어 문장들을 정규 표현식을 통해 실제 테스트 함수와 연결(Binding)한다. 이렇게 하면 기획자가 시나리오의 문구만 수정해도, 개발자는 그 문구에 연결된 테스트 코드를 실행하며 비즈니스 로직의 변경을 즉시 검증할 수 있다. 이는 기술(Code)과 비즈니스(Feature) 사이의 거대한 다리 역할을 수행한다.

---

### 심층 동작 원리: 유비쿼터스 언어 (Ubiquitous Language)

DDD와 BDD를 잇는 핵심 철학이다.
1. **정의**: 기획자, 개발자, 사용자가 모두 공통으로 사용하는 용어의 집합.
2. **원리**: 코드 내의 메서드 이름이나 테스트 시나리오의 단어를 실제 비즈니스 도메인 용어와 100% 일치시킨다.
3. **가치**: "이 변수명이 그 뜻인가요?"라고 묻는 시간을 없애고, 코드 자체가 비즈니스 그 자체가 되게 한다.

- **📢 섹션 요약 비유**: 한국인과 미국인이 대화할 때 통역기를 쓰는 게 아니라, 아예 '요리'라는 공통의 관심사를 위해 특별히 만든 '요리 공용어'를 사용하여 오해를 없애는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### TDD vs BDD 비교 (관점의 차이)

| 비교 항목 | TDD (Test Driven) | BDD (Behavior Driven) |
|:---|:---|:---|
| **대상** | **단위 기능** (클래스, 메서드) | **사용자 행위** (서비스 흐름) |
| **언어** | 프로그래밍 언어 (JUnit 등) | **비즈니스 언어 (Gherkin)** |
| **가독성** | 개발자만 읽을 수 있음 | **누구나 읽을 수 있음** |
| **목표** | 버그 없는 깨끗한 코드 | **비즈니스 목적에 맞는 코드** |
| **비유** | 벽돌이 튼튼한지 검사 | 집의 동선이 편리한지 리허설 |

---

### 기술 융합: BDD와 인수 테스트 자동화

BDD는 시스템의 최종 승인 단계인 '인수 테스트'와 완벽히 융합된다.
- **E2E Testing**: BDD 시나리오를 Selenium이나 Playwright 같은 도구와 연결하여, 실제 브라우저를 띄워 사용자의 클릭 행위를 자동 검증한다.
- **가치**: "로그인 시나리오 10개가 모두 통과했으니, 이 기능은 배포해도 좋습니다"라고 기획자에게 당당히 말할 수 있는 근거가 된다.

- **📢 섹션 요약 비유**: 수만 개의 부품 검사 성적표(TDD)를 보여주는 것보다, 실제 자동차가 멋지게 달리는 모습(BDD)을 한 번 보여주는 것이 손님(사용자)을 안심시키는 데 훨씬 효과적입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: '장바구니 무료 배송' 로직의 BDD 도입

- **문제 상황**: "5만 원 이상 구매 시 무료 배송"이라는 규칙이 있는데, 쿠폰 사용 시 5만 원 미만이 되면 어떻게 할지에 대해 기획자와 개발자의 이해가 달라 배포 후 버그 발생.
- **의사결정 (BDD 워크숍 및 시나리오 작성)**:
  1. **Three Amigos**: 기획자, 개발자, QA가 모여 칠판 앞에 선다.
  2. **Scenario 작성**:
     - **Given**: 장바구니에 6만 원어치 상품이 담겨 있음
     - **When**: 2만 원 할인 쿠폰을 적용함
     - **Then**: 최종 결제 금액은 4만 원이 되고, 배송비 3천 원이 추가되어야 함
  3. **합의**: 이 시나리오를 보고 기획자가 "맞습니다"라고 하면 이를 **Feature 파일**로 저장한다.
  4. **자동화**: 개발자는 이 시나리오를 통과시키는 코드를 짜고, 매일 자동으로 이 테스트가 도는지 확인한다.

### 도입 체크리스트
- **Readability**: 작성된 시나리오를 개발을 모르는 어머니가 읽어도 무슨 뜻인지 이해할 수 있는가?
- **Granularity**: 시나리오 하나가 너무 길어서 읽기 힘들지는 않은가? (Background 키워드 활용)
- **Maintenance**: UI가 바뀔 때마다 시나리오가 깨져서 고치는 데 시간이 더 걸리고 있지는 않은가? (추상화 수준 조절)

### 안티패턴
- **Technical Gherkin**: `When "login_btn"을 클릭하면` 식의 코드를 시나리오에 넣는 행위. (기획자가 읽지 못하므로 BDD의 의미 상실)
- **QA 전용 도구로 전락**: 개발자는 안 보고 오직 QA 팀만 시나리오를 짜고 테스트하는 상황. (협업의 가치가 사라짐)

- **📢 섹션 요약 비유**: 리허설 대본(BDD)에 "무대 조명 전압을 220V로 맞춘다"라고 적지 마세요. 대본에는 "방 안이 갑자기 환해진다"라고 적어야 배우와 스태프 모두가 이해할 수 있습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Rework)** | 재개발 비용 40% 감소 | 요구사항 오해로 인한 소스 전체 수정 사태 조기 방지 |
| **정량적 (Coverage)** | 비즈니스 케이스 100% 검증 | 모든 기획 내용이 테스트 시나리오로 치환되어 누락 없는 검증 가능 |
| **정성적 (Communication)** | 팀 내 소통 비용 획기적 절감 | "기획서 5페이지 참고하세요" 대신 "시나리오 3번 보세요"로 소통 |
| **정성적 (Document)** | 영구적인 시스템 명세 확보 | 최신 기능이 반영된 문서를 별도의 노력 없이 항상 보유 |

### 미래 전망
- **Natural Language Execution**: AI가 기획 회의 녹취록을 듣고 즉석에서 BDD 시나리오를 생성하고, 이를 코드로 변환하는 '완전 자동화 BDD'가 실현될 것이다.
- **Visual BDD**: 텍스트를 넘어 화면 와이어프레임과 시나리오가 실시간 연결되어 시각적으로 동작을 정의하는 도구가 주류가 될 것이다.

### 참고 표준
- **Cucumber / SpecFlow**: BDD의 실무 표준 도구 명세
- **Gherkin Language Guide**: Given-When-Then 작성 표준
- **The Cucumber Book (Matt Wynne)**: BDD의 정석 가이드

- **📢 섹션 요약 비유**: BDD는 소프트웨어라는 복잡한 기계에 '인간의 언어'라는 숨결을 불어넣는 작업입니다. 기계와 사람이 똑같은 꿈을 꿀 때, 비로소 위대한 제품이 탄생합니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **BDD**는 장난감 성을 만들기 전에 친구들과 **"누가 공주님을 구하러 갈지" 이야기를 먼저 지어보는 것**과 같아요.
2. "공주님이 갇히면(Given), 왕자님이 달려가서(When), 성문을 연다(Then)"라고 미리 약속을 정해두는 거예요.
3. 이렇게 이야기를 먼저 만들면, 나중에 장난감을 가지고 놀 때 친구들과 싸우지 않고 사이좋게 재미있는 연극을 할 수 있답니다!
