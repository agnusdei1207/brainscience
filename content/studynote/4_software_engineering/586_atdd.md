# 586. 인수 테스트 주도 개발 (ATDD)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인수 테스트 주도 개발 (Acceptance Test-Driven Development, ATDD)은 개발 시작 전 비즈니스 고객, 개발자, 테스터가 협력하여 **합의된 인수 조건(Acceptance Criteria)을 실행 가능한 테스트 케이스로 정의**하고 이를 통과시키며 기능을 구현하는 방법론이다.
> 2. **가치**: "무엇을 만들어야 하는가"에 대한 명확한 공통 이해를 형성하여 요구사항 오해로 인한 재개발 비용을 혁신적으로 줄이며, 모든 사용자 스토리가 비즈니스 가치에 부합하게 구현되었음을 실시간으로 증명한다.
> 3. **융합**: BDD(Behavior-Driven Development)와 기술적으로 밀접하게 연관되어 있으며, '살아있는 문서화'를 통해 요구사항-설계-구현-테스트 사이의 추적성(Traceability)을 자동화된 방식으로 확보한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: ATDD는 "고객의 언어를 테스트로" 바꾸는 과정이다. 단순히 코드가 버그 없이 돌아가는지를 보는 것이 아니라, "고객이 원했던 바로 그 기능이 맞는가?"를 확인하는 테스트를 가장 먼저 만들고 개발을 진행하는 방식이다.

- **💡 비유**: ATDD는 **"집을 짓기 전, 집주인과 함께 작성한 꼼꼼한 체크리스트"**와 같습니다. "거실 창문은 남향이어야 함", "수도꼭지에서는 뜨거운 물이 3초 안에 나와야 함"과 같은 구체적인 합격 기준(인수 테스트)을 미리 정해두고, 집을 짓는 중간중간 이 리스트를 체크하며 완성하는 것입니다.

- **등장 배경**:
  1. **요구사항의 모호성**: 기획서의 텍스트는 사람마다 해석이 달라, 개발이 끝난 뒤에야 "이게 아니었는데"라는 말을 듣는 경우가 많았다.
  2. **피드백 지연**: 개발 완료 후 며칠 뒤에 수행되는 인수 테스트는 수정 범위를 너무 넓게 만들었다.
  3. **팀 간의 장벽**: 기획, 개발, 테스트 팀이 각자 자기 일만 하던 사일로(Silo) 문화를 타파하고 '하나의 목표'를 향해 정렬할 필요가 생겼다.

- **📢 섹션 요약 비유**: 수험생이 공부를 시작하기 전, 기출문제(인수 테스트)를 먼저 훑어보고 "어떤 능력을 평가받을지" 정확히 파악한 뒤 집중적으로 공부하는 전략적인 학습법과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. ATDD의 4단계 프로세스: Discuss-Distill-Develop-Demo

협업과 자동화를 순환시키는 아키텍처다.

| 단계 | 수행 활동 | 핵심 목표 | 참여자 |
|:---|:---|:---|:---|
| **1. 논의 (Discuss)** | 사용자 스토리 분석 및 시나리오 도출 | 요구사항 구체화 및 예외 케이스 발굴 | 고객, 개발자, 테스터 |
| **2. 정제 (Distill)** | 시나리오를 실행 가능한 테스트로 변환 | 기계가 이해할 수 있는 정형화된 규격 마련 | 개발자, 테스터 |
| **3. 개발 (Develop)** | 테스트 통과를 위한 실제 코드 구현 | TDD와 병행하여 기능 완성 | 개발자 |
| **4. 시연 (Demo)** | 통과된 테스트를 고객에게 확인받음 | 비즈니스 가치 최종 승인 | 전체 팀원 |

---

### 2. ATDD 연동 및 피드백 아키텍처

비즈니스 요건이 어떻게 기술적 테스트로 변환되어 코드를 이끄는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      ATDD (Acceptance Test-Driven) 협업 모델                     │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [비즈니스 요건] ──────▶ [1. 인수 조건 합의] ──────▶ [2. 자동화 테스트 작성]       │
 │ (User Story)           (Criteria / Given-When-Then)   (Cucumber, FitNesse)  │
 │                                                                  │          │
 │           ┌──────────────────────────────────────────────────────┘          │
 │           ▼                                                                 │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │  [3. 개발 및 내부 테스트 (TDD Cycle)]                                     │  │
 │  │  - Red: 인수 테스트 실패 확인                                             │  │
 │  │  - Green: 로직 구현 후 인수 테스트 통과                                    │  │
 │  │  - Refactor: 코드 품질 개선                                               │  │
 │  └──────────────────┬────────────────────────────────────────────────────┘  │
 │                    │                                                        │
 │                    ▼ [4. 시연 및 배포]                                       │
 │           ┌────────────────────────┐                                        │
 │           │  비즈니스 가치 달성! (OK) │ ◀─── 모든 인수 테스트 Pass 시             │
 │           └────────────────────────┘                                        │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ATDD는 **'삼총사(Three Amigos: 고객, 개발자, 테스터)'**의 협업으로 시작된다. 1) 고객의 모호한 사용자 스토리를 2) 구체적인 **Given-When-Then** 시나리오로 정제하고, 이를 자동화 툴에 입력한다. 3) 개발자는 이 큰 단위의 인수 테스트를 통과시키는 것을 최종 목표로 삼고, 그 내부의 세부 로직은 **TDD**로 촘촘히 메워 나간다. 4) 마지막으로 모든 인수 테스트가 '초록불'이 되면, 개발자는 구구절절 설명할 필요 없이 테스트 결과를 보여줌으로써 고객의 최종 승인을 즉시 얻어낼 수 있다.

---

### 심층 동작 원리: 실행 가능한 명세 (Executable Specification)

ATDD가 지향하는 문서의 미래다.
1. **기존 문서**: 종이에 적힌 글. 코드가 바뀌면 거짓말이 됨.
2. **실행 가능한 명세**: 테스트 코드 그 자체가 문서. 코드가 동작하지 않으면 문서(테스트)가 실패하므로 항상 최신성을 유지함.
3. **가치**: "우리 시스템이 어떻게 동작하나요?"라는 질문에 최신 코드가 반영된 테스트 결과 리포트를 보여주는 것만으로 정답을 줄 수 있다.

- **📢 섹션 요약 비유**: 그림으로만 그려진 가짜 내비게이션 지도(기존 기획서) 대신, 실제 길 안내와 연동되어 길이 막히면 실시간으로 색깔이 변하는 스마트 내비게이션(실행 가능한 명세)을 갖게 된 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### TDD vs ATDD 비교 (계층의 차이)

| 비교 항목 | TDD (단위 테스트 주도) | ATDD (인수 테스트 주도) |
|:---|:---|:---|
| **시야 (Granularity)** | 미시적 (클래스 내부 로직) | **거시적 (사용자 시나리오)** |
| **성격** | 기술적 품질 (Technical) | **비즈니스적 품질 (Functional)** |
| **언어** | 개발자 언어 (Java, Python) | **비즈니스 언어 (자연어)** |
| **성공 기준** | "코드가 에러 없이 돌아감" | **"사용자가 원하는 결과가 나옴"** |
| **비유** | 부품이 튼튼한가? | 자동차가 목적지까지 잘 가는가? |

---

### 기술 융합: ATDD와 DDD (도메인 주도 설계)

ATDD는 DDD의 철학을 완성시킨다.
- **Ubiquitous Language**: 인수 테스트 시나리오를 작성할 때 사용하는 비즈니스 용어를 소스 코드와 일치시킨다.
- **Bounded Context**: 각 도메인 경계별로 인수 테스트 스위트를 구성하여 서비스 간의 간섭 없는 독립적 검증을 가능케 한다.

- **📢 섹션 요약 비유**: 각 나라의 요리사들이 모여(MSA) 파티를 할 때, 각자 만들 요리의 이름과 맛(인수 조건)을 통일된 언어로 미리 정해두어 뷔페 전체의 조화를 맞추는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: '포인트 충전 및 결제' 서비스의 ATDD 도입

- **문제 상황**: 결제 취소 시 포인트가 복구되지 않거나, 유효기간이 지난 포인트가 충전되는 등 비즈니스 예외 상황에 대한 처리가 누락되어 사고 발생.
- **의사결정 (ATDD 워크숍 적용)**:
  1. **Discuss**: 비즈니스 담당자와 "유효기간 1분 남은 포인트로 결제 중 1분이 지나 취소하면?" 같은 엣지 케이스 시나리오를 10여 개 도출한다.
  2. **Distill**: 도출된 시나리오를 **FitNesse**나 **Cucumber**에 표 형식의 테스트 데이터로 입력한다.
  3. **Develop**: 개발자는 이 10개의 실패하는 케이스를 해결하기 위한 로직을 작성한다.
  4. **Demo**: 스프린트 리뷰 때, 10개의 테스트가 모두 'Pass'된 대시보드를 보여주며 "비즈니스 요건이 완벽히 충족되었습니다"라고 보고한다.

### 도입 체크리스트
- **Amigos Participation**: 기획자(PO)가 시나리오 검토 과정에 적극적으로 참여하고 있는가?
- **Domain Language**: 테스트 시나리오에 개발 용어(id, null, db 등)가 아닌 비즈니스 용어(회원번호, 미입력, 저장 등)를 썼는가?
- **Automation Speed**: 인수 테스트가 너무 무거워(E2E 위주) 피드백이 며칠씩 걸리지는 않는가? (API 레벨의 인수 테스트 권장)

### 안티패턴
- **Post-Development Testing**: 개발을 다 하고 나서야 인수 테스트를 짜는 행위. ATDD의 핵심인 '주도(Driven)' 효과가 사라진다.
- **Developer-Only ATDD**: 개발자끼리만 시나리오를 짜는 것. 고객의 의도를 놓칠 확률이 100%다.

- **📢 섹션 요약 비유**: "시험지(테스트)를 먼저 보고 공부(개발)하라"고 했더니, 공부를 다 끝낸 뒤에 시험 문제를 직접 만들어서(사후 테스트) 자기가 다 맞았다고 자축하는 것과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Quality)** | 요구사항 누락 결함 0% 지향 | 구현 전 시나리오 합의를 통해 누락된 로직 사전 식별 |
| **정량적 (Speed)** | 전체 개발 기간 20% 단축 | 재작업(Rework) 발생 빈도를 획기적으로 낮추어 순수 개발 시간 확보 |
| **정성적 (Standard)** | 협업과 신뢰의 문화 | "말한 대로 만들었음"을 증명하여 비즈니스 부서와 개발 부서 간 신뢰 형성 |
| **정성적 (Documentation)** | 최신 사양서 자동 확보 | 별도의 명세서 관리 노력 없이 테스트 코드가 그 역할을 수행 |

### 미래 전망
- **AI-Mediated ATDD**: AI가 기획 회의 내용을 실시간 요약하여 인수 테스트 초안을 만들고, 개발자가 이를 승인하면 즉시 테스트 환경이 구축되는 기술이 보편화될 것이다.
- **Behavioral Analytics Linkage**: 운영 환경의 실제 사용자 행동 로그를 분석하여, 누락된 인수 테스트 시나리오를 AI가 역으로 제안하는 폐쇄형 피드백 루프가 실현될 것이다.

### 참고 표준
- **Specification by Example (Gojko Adzic)**: ATDD의 실전 가이드북
- **ISTQB Expert Level - Test Management**: 비즈니스 가치 테스팅 표준
- **Agile Extension to the BABOK Guide**: 비즈니스 분석 관점의 ATDD

- **📢 섹션 요약 비유**: "내가 만들고 싶은 것"이 아니라 "남이 원하는 것"을 만드는 고난도의 정교함, ATDD는 프로그래머를 단순 기술자가 아닌 '비즈니스 해결사'로 격상시키는 가장 강력한 도구입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **ATDD**는 레고 블록을 사기 전, 엄마랑 **"이 레고가 진짜로 하늘을 날 수 있는지, 불이 번쩍거리는지" 미리 체크리스트를 적어두는 것**과 같아요.
2. 물건을 다 만든 다음에 엄마한테 리스트를 보여주면서 "여기 봐요, 약속한 대로 다 되죠?"라고 확인받는 거예요.
3. 이렇게 미리 약속을 정하면, 나중에 엄마가 "내가 원한 건 이게 아니야"라고 말씀하시는 일을 미리 막을 수 있답니다!
