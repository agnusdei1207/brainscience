# 588. 분산 추적 (Distributed Tracing)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 추적 (Distributed Tracing)은 마이크로서비스(MSA) 환경에서 하나의 사용자 요청이 여러 서비스를 거쳐 처리되는 **전 과정을 시각화하고 각 구간의 성능 지연을 분석**하는 옵저버빌리티 기술이다.
> 2. **가치**: 요청마다 고유한 **Trace ID**를 부여하고 서비스 간 통신 단위인 **Span**을 기록함으로써, 복잡한 호출 체인 속에서 장애 발생 지점과 성능 병목을 단번에 식별하여 평균 복구 시간(MTTR)을 획기적으로 단축한다.
> 3. **융합**: Zipkin, Jaeger와 같은 오픈소스 도구와 OpenTelemetry 표준을 통해 구현되며, 서비스 메시(Service Mesh) 및 API 게이트웨이와 결합되어 시스템 전 구간의 투명한 가시성을 제공한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 분산 추적은 "요청의 일생"을 기록하는 것이다. 사용자가 클릭한 한 번의 요청이 주문 서버, 결제 서버, 재고 서버를 거칠 때, 각 서버가 그 요청을 처리하는 데 얼마나 시간을 썼는지 꼬리에 꼬리를 물고 추적한다.

- **💡 비유**: 분산 추적은 **"택배 배송 추적 시스템"**과 같습니다. 물건(요청)이 발송되어 허브 터미널(주문 서버), 지점(결제 서버), 배송 기사(재고 서버)를 거치는 단계마다 바코드를 찍어(Span 기록), 지금 어디에 있는지와 어디서 배송이 늦어지는지 실시간으로 확인하는 것과 같습니다.

- **등장 배경**:
  1. **MSA의 디버깅 한계**: 서비스가 쪼개지면서 "주문이 실패했다"는 결과는 알지만, 그 원인이 주문 서버 자체인지, 아니면 호출한 결제 서버의 네트워크 지연인지 파악하기가 매우 어려워졌다.
  2. **성능 튜닝의 필요성**: 수십 개의 마이크로서비스가 얽힌 환경에서 전체 응답 속도를 늦추는 단 하나의 '느린 구간'을 찾아내야 했다.
  3. **분산 로그의 파편화**: 서버마다 로그가 따로 저장되어 있어, 하나의 요청에 대한 전체 맥락(Context)을 연결하는 데 엄청난 시간이 소요되었다.

- **📢 섹션 요약 비유**: 흩어진 실타래(파편화된 로그)를 하나의 긴 실(Trace ID)로 꿰어, 요청이 지나간 모든 자리를 선명하게 드러내는 디지털 실타래 추적 기술입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 분산 추적의 핵심 용어 (Data Model)

| 용어 명칭 | 상세 정의 | 비유 |
|:---|:---|:---|
| **Trace (트레이스)** | 하나의 요청이 완료될 때까지의 전체 경로 | 택배의 전체 배송 경로 |
| **Span (스팬)** | 특정 서비스 내부의 작업 단위 (단일 API 호출 등) | 택배가 특정 터미널에 머문 시간 |
| **Trace ID** | 전체 트래이스를 식별하는 유일한 고유 번호 | 운송장 번호 |
| **Span ID** | 개별 스팬을 식별하는 고유 번호 | 터미널 작업 고유 번호 |
| **Context Propagation** | 서비스 간 통신 시 ID 정보를 헤더에 실어 나르는 행위 | 다음 기사에게 운송장 넘겨주기 |

---

### 2. 분산 추적 동작 아키텍처

서비스 간에 ID가 어떻게 전파되고 수집되는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      분산 추적 (Distributed Tracing) 동작 흐름                    │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Client] ──▶ [Service A (Order)] ──▶ [Service B (Pay)] ──▶ [Service C (DB)]   │
 │      │              │                     │                     │           │
 │  1. 요청 발생 ────▶ 2. Trace ID 생성       4. Trace ID 전달 ─────▶ 6. Trace ID 전달 │
 │                    3. Span 1 시작/종료     5. Span 2 시작/종료     7. Span 3...  │
 │                           │                     │                     │     │
 │                           └───────────┬─────────┴──────────┬──────────┘     │
 │                                       ▼                    ▼                │
 │                             ┌────────────────────────────────┐              │
 │                             │    Tracing Collector (Jaeger)  │              │
 │                             │  - 데이터 취합 및 타임라인 생성   │              │
 │                             └────────────────────────────────┘              │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 시각화 결과 (Timeline):                                               │  │
 │  │ [Span 1 (Order)] ■■■■■■■■■■■■■■■■■■■■■■ (100ms)                         │  │
 │  │   [Span 2 (Pay)]       ■■■■■■■■■■■ (50ms)                               │  │
 │  │     [Span 3 (DB)]         ■■■■ (10ms)                                    │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 분산 추적의 핵심 기술은 **Context Propagation (문맥 전파)**이다. 1) 첫 번째 서비스(Service A)가 요청을 받으면 **Trace ID**를 생성한다. 2) 다른 서비스를 호출할 때 HTTP 헤더(예: `X-B3-TraceId`)에 이 ID를 담아 보낸다. 3) 각 서비스는 자신이 처리한 시간 정보를 **Span**이라는 단위로 기록하여 백그라운드에서 **Collector**에게 전송한다. 4) Collector는 동일한 Trace ID를 가진 Span들을 시간 순서대로 재조립하여 우리에게 익숙한 **간트 차트(Gantt Chart)** 형태의 성능 리포트를 제공한다.

---

### 심층 동작 원리: 샘플링 (Sampling) 및 성능 오버헤드

모든 요청을 추적하면 시스템이 느려질 수 있다.
1. **Probabilistic Sampling**: 1,000건 중 1건만 추적하도록 확률적으로 설정 (보통 0.1% ~ 1%).
2. **Rate Limiting Sampling**: 초당 최대 n건만 추적.
3. **Adaptive Sampling**: 시스템 부하가 낮을 땐 많이, 높을 땐 적게 자동으로 조절.
4. **가치**: 실제 서비스 성능에 미치는 영향을 최소화(보통 1~2ms 이내)하면서도 유의미한 통계치를 확보한다.

- **📢 섹션 요약 비유**: 모든 택배 상자를 엑스레이로 검사하는 게 아니라, 무작위로 몇 개만 골라 정밀 검사함으로써 물류 속도를 유지하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 주요 분산 추적 도구 비교: Zipkin vs Jaeger

| 비교 항목 | Zipkin (Twitter 개발) | Jaeger (Uber 개발) |
|:---|:---|:---|
| **아키텍처** | 단순함, Java 기반 | **복잡하고 강력함, Go 기반** |
| **데이터 모델** | B3 Propagation 표준 | **OpenTelemetry 기반** |
| **주요 특징** | 검증된 안정성, 쉬운 설치 | **클라우드 네이티브 최적화**, CNCF 졸업 |
| **UI/UX** | 클래식하고 직관적 | **상세한 통계 및 적응형 샘플링 제공** |
| **비유** | 튼튼한 트럭 | 최첨단 자율주행 트럭 |

---

### 기술 융합: OpenTelemetry (OTEL)와 분산 추적

과거에는 도구마다 데이터 형식이 달랐으나, 이제는 OTEL로 통합된다.
- **Vendor Agnostic**: 코드에는 OTEL SDK만 적용하고, 실제 데이터는 Zipkin으로 보낼지 Jaeger로 보낼지 설정만 바꾸면 된다.
- **가치**: 도구에 종속되지 않는 유연한 가시성 인프라를 구축할 수 있게 해준다.

- **📢 섹션 요약 비유**: 모든 가전제품(보안 도구)이 제각각인 플러그를 쓰다가, 이제는 USB-C 타입(OpenTelemetry)으로 표준화되어 어디든 꽂을 수 있게 된 혁신입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: '결제 실패' 원인 규명을 위한 분산 추적 활용

- **문제 상황**: 고객이 결제 시 "오류가 발생했습니다"라는 메시지를 봤는데, 결제 서버 로그에는 아무런 에러가 찍혀 있지 않음.
- **의사결정 (분산 추적 수사 전략)**:
  1. **Trace ID 검색**: 고객의 주문 번호나 ID를 통해 해당 시점의 **Trace ID**를 추출한다.
  2. **전체 경로 분석**: Jaeger UI에서 해당 ID를 조회하여 전체 호출 그래프를 본다.
  3. **지점 식별**: 결제 서버는 정상(200 OK)이었으나, 결제 서버가 호출한 '카드사 연동 모듈'에서 5초 지연(Latency) 후 타임아웃이 발생했음을 차트에서 확인한다.
  4. **결과**: 결제 서버의 잘못이 아니라, 외부 연동망의 지연이 원인임을 1분 만에 증명하고 해당 구간에 서킷 브레이커를 적용한다.

### 도입 체크리스트
- **Propagation Logic**: 비동기 쓰레드나 메시지 큐(Kafka)를 거칠 때도 Trace ID가 끊기지 않고 전달되는가?
- **Storage Strategy**: 매일 쏟아지는 방대한 추적 데이터를 어디에, 얼마나 오래 보관할 것인가? (ElasticSearch, Cassandra 등 활용)
- **Library Support**: 우리가 쓰는 프레임워크(Spring Boot 등)가 추적 라이브러리와 자동 연동(Auto-instrumentation)되는가?

### 안티패턴
- **전수 추적 (100% Sampling)**: 트래픽이 많은 서비스에서 모든 요청을 추적하려다 모니터링 서버가 먼저 다운되는 경우.
- **추적 ID 누락**: 서비스 간 호출 시 헤더를 누락하여 "끊어진 트레이스"가 양산되는 상황. (가치 상실)

- **📢 섹션 요약 비유**: 끊어진 철길(누락된 ID)로는 기차가 갈 수 없듯, 분산 추적은 모든 구간의 연결 고리를 잇는 정성이 가장 중요합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (MTTR)** | 장애 분석 시간 90% 단축 | 수십 대 서버 로그를 뒤지던 노력을 단일 화면 클릭으로 대체 |
| **정량적 (Latency)** | 성능 최적화 포인트 식별 | 가장 느린 Span을 즉시 시각화하여 튜닝 우선순위 결정 |
| **정성적 (Standard)** | 개발-운영 언어 통일 | "A 서비스가 느려요" 대신 "Span ID xxx에서 500ms 지연 발생"으로 소통 |
| **정성적 (Visibility)** | 복잡한 서비스 의존성 파악 | 시스템이 어떻게 얽혀있는지 실시간으로 그려지는 지도를 확보 |

### 미래 전망
- **AI-Guided Root Cause**: 추적 데이터를 학습한 AI가 장애 발생 시 "원인은 85% 확률로 DB 락(Lock)입니다"라고 분석 리포트를 자동 생성할 것이다.
- **Auto-Instrumentation Expansion**: 소스 코드 수정 없이 바이트코드 조작만으로 모든 통신을 추적하는 기술이 모바일과 웹 브라우저 단까지 완벽히 확장될 것이다.

### 참고 표준
- **W3C Trace Context Specification**: 분산 추적 전파의 국제 표준
- **OpenTelemetry Specification**: 관측성 데이터의 통합 표준
- **Dapper (Google)**: 현대 분산 추적의 시초가 된 논문

- **📢 섹션 요약 비유**: 어두운 밤하늘(복잡한 MSA)에 별자리(분산 추적)를 그려 넣음으로써, 우리가 어디에 있고 어디로 가야 할지 길을 잃지 않게 해주는 시스템의 북극성과 같습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **분산 추적**은 친구들과 릴레이 달리기를 할 때, **누가 바톤을 전달하는 데 얼마나 시간을 썼는지 꼼꼼하게 기록하는 시계**와 같아요.
2. 바톤(Trace ID)에 이름표를 붙여서 넘겨주면, 나중에 달리기 경주가 끝나고 **누가 가장 늦게 뛰었는지 금방 알 수 있답니다.**
3. 이 시계를 보면, 우리가 왜 졌는지(장애 발생) 금방 찾아내서 다음에는 더 빨리 뛸 수 있게 도와준답니다!
