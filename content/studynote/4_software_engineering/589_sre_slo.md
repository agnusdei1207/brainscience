# 589. 사이트 신뢰성 공학 (SRE)과 SLI, SLO, SLA

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SRE (Site Reliability Engineering)는 구글이 제안한 시스템 운영의 공학적 접근 방식으로, **소프트웨어 엔지니어가 운영 문제를 해결**하도록 하여 시스템의 안정성과 자동화를 극대화하는 직무이자 문화다.
> 2. **가치**: 정성적인 "행복"이나 "안정" 대신, **SLI(지표), SLO(목표), SLA(약속)**라는 정량적 수치를 사용하여 개발과 운영 간의 목표를 일치시키고, 데이터에 기반한 합리적인 의사결정(에러 예산 활용)을 가능케 한다.
> 3. **융합**: DevOps 철학을 실질적인 '구현체'로 구체화한 것이며, 현대의 클라우드 네이티브 환경에서 서비스의 가용성(Availability)과 혁신 속도 사이의 균형을 맞추는 핵심 거버넌스로 작동한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **SRE**: "운영을 소프트웨어 문제로 다루는 것"이다. 반복적인 수동 작업(Toil)을 코드로 자동화하고, 시스템의 신뢰성을 정밀하게 관리한다.
  - **SLI / SLO / SLA**: 서비스를 얼마나 잘 제공할지 정의하는 3대 핵심 정량 지표 체계다.

- **💡 비유**: SRE는 **"병원의 자동화된 정밀 중환자실 시스템"**과 같습니다. 의사(SRE)가 환자를 직접 돌보는 대신, 인공지능 로봇과 센서(자동화 도구)를 설계하여 환자의 상태를 24시간 감시하고 위급 상황 시 자동으로 처치하게 만드는 것입니다. 이때 **SLI**는 환자의 체온, **SLO**는 정상 체온 범위, **SLA**는 보호자와의 치료 보장 계약과 같습니다.

- **등장 배경**:
  1. **개발과 운영의 갈등**: 개발팀은 '빠른 변화'를 원하고 운영팀은 '안정성'을 원하면서 발생하는 충돌을 해결할 합리적인 중재 기준이 필요했다.
  2. **수동 작업의 한계**: 시스템 규모가 커지면서 사람이 일일이 명령어를 입력하는 방식으로는 운영 효율을 낼 수 없게 되었다. (Toil 제거 요구)
  3. **가용성의 비즈니스 가치**: 0.1%의 가용성 하락이 수십억 원의 손실로 이어지는 대규모 온라인 서비스 시대에 정교한 신뢰성 관리가 필수화되었다.

- **📢 섹션 요약 비유**: 감(Feeling)에 의존하던 운영에서 숫자(Metric)로 증명하는 공학으로, 주먹구구식 수리에서 자동화된 정비소로의 거대한 전환입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 신뢰성 지표 체계 (SLI / SLO / SLA)

세 지표는 긴밀하게 연결되어 시스템의 '성적표'를 만든다.

| 용어 | 풀네임 | 정의 | 예시 | 비유 |
|:---|:---|:---|:---|:---|
| **SLI** | Service Level **Indicator** | 서비스 수준 **척도** (현재 값) | 가용성 99.95%, 응답 속도 100ms | 현재 체온 |
| **SLO** | Service Level **Objective** | 서비스 수준 **목표** (타겟 값) | **가용성 99.9% 이상 유지** | 정상 체온 (36.5도) |
| **SLA** | Service Level **Agreement** | 서비스 수준 **합의** (대외 약속) | **99.5% 미달 시 이용료 환불** | 병원-보호자 간 계약 |

---

### 2. SRE 운영 및 의사결정 아키텍처

데이터가 어떻게 개발 속도와 안정성을 조절하는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      SRE 기반의 에러 예산(Error Budget) 작동 원리                 │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │  [서비스 가용성 목표: 99.9% (SLO)]                                             │
 │           │                                                                 │
 │           ▼                                                                 │
 │  ┌──────────────────────────────────┐                                       │
 │  │  에러 예산 (Error Budget) = 0.1%  │  ◀─── "우리가 허용할 수 있는 불완전함"      │
 │  └──────┬───────────┬───────────────┘                                       │
 │         │           │                                                       │
 │         ▼           ▼                                                       │
 │  [예산 남음 (Stable)]   [예산 소진 (Risky)]                                     │
 │         │           │                                                       │
 │  - 신규 기능 적극 배포   - **배포 전면 중단 (Freeze)**                             │
 │  - 공격적인 기술 도입   - 안정성 강화 및 자동화 집중                              │
 │  - 리스크 수용 가능     - 사후 분석 (Post-mortem) 수행                           │
 │                                                                             │
 │  ※ 핵심: "100% 신뢰성은 목표가 아니다. 신뢰성은 비용과 혁신의 타협점이다."           │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SRE 아키텍처의 심장은 **에러 예산 (Error Budget)**이다. 가용성 목표가 99.9%라면, 나머지 0.1%는 서비스가 중단되어도 괜찮은 '예산'으로 간주한다. 1) **예산이 넉넉할 때** 개발팀은 자유롭게 신규 기능을 출시한다. 2) 하지만 잦은 에러로 **예산이 소진**되면, 운영(SRE)팀은 모든 배포를 막고 시스템 안정화 작업에만 매달리게 한다. 이 수치 기반의 통제는 개발과 운영 사이의 주관적 감정싸움을 없애고, 비즈니스 가치에 최적화된 시스템 속도를 제공한다.

---

### 심층 동작 원리: Toil (단조로운 수동 작업)의 제거

SRE는 운영자의 시간을 '창의적 업무'에 쓰게 한다.
1. **Toil의 정의**: 전술적이고 반복적이며 자동화 가능한 수동 작업. (예: 서버 리부팅, 수동 권한 부여)
2. **50% 원칙**: SRE는 근무 시간의 최소 50%를 코딩(자동화 시스템 개발)에 써야 하며, 나머지 50%만 실제 운영 업무를 본다.
3. **가치**: 사람이 늘어난다고 운영 인력이 비례해서 늘어나는 선형적 증가(Linear Scale)를 거부하고, 기술로 복잡도를 압도한다.

- **📢 섹션 요약 비유**: 매일 아침 마당을 직접 쓰는 것(Toil) 대신, 마당 쓰는 로봇(자동화 도구)을 고치는 데 시간을 쓰는 것이 진정한 SRE의 자세입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### DevOps vs SRE 비교 (철학과 구현)

| 비교 항목 | DevOps | SRE (Site Reliability Engineering) |
|:---|:---|:---|
| **성격** | 추상적 철학, 문화 | **구체적 직무, 방법론 (실제 구현체)** |
| **핵심 문구** | 벽을 허물라 (Break the silos) | **운영을 소프트웨어 문제로 다루라** |
| **장애 대응** | 공동 책임 | **포스트모텀 (비난 없는 사후 분석)** |
| **관계** | 인터페이스 (정의) | **클래스 (구현)** |
| **비유** | "건강한 생활을 하자" | "매일 1시간 달리기, 닭가슴살 200g 섭취" |

---

### 기술 융합: Observability와 SRE

SRE는 관측성 데이터 없이는 숨을 쉴 수 없다.
- **SLI 도출**: 로그, 메트릭, 추적 데이터를 가공하여 "진짜 사용자가 느끼는 성능"인 SLI를 뽑아낸다.
- **가치**: 단순 CPU 점유율이 아닌, "성공한 요청 중 99%가 200ms 이내에 처리되었는가"와 같은 실질적 지표로 시스템을 관리한다.

- **📢 섹션 요약 비유**: 주방 온도(인프라 지표)가 중요한 게 아니라, 손님 앞에 나간 요리의 맛과 서빙 시간(서비스 지표)이 진짜 성적표가 되어야 한다는 뜻입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 신규 쇼핑몰 런칭을 위한 SRE 거버넌스 수립

- **문제 상황**: 개발팀은 오픈 직후 수시 배포를 원하고, 인프라팀은 장애가 날까 봐 모든 배포를 승인제로 하겠다고 맞서며 갈등 중.
- **의사결정 (SRE 전략 도입)**:
  1. **SLO 합의**: 메인 페이지 조회 성공률 99.9%, 결제 API 99.99%를 목표(SLO)로 정한다.
  2. **에러 예산 수립**: 월간 약 43분의 다운타임(0.1%)을 '혁신을 위한 예산'으로 할당한다.
  3. **배포 정책**: 에러 예산이 남아 있는 한 개발팀은 자유 배포권을 가지며, 예산 소진 시 배포가 자동 차단되는 파이프라인(CI/CD)을 구축한다.
  4. **Post-mortem 도입**: 장애 발생 시 누구의 잘못인지 따지지 않고, "시스템의 어떤 구멍이 이 인재를 가능케 했나"를 분석하여 자동화된 방어 코드를 추가한다.

### 도입 체크리스트
- **Metric Selection**: 우리가 측정하는 SLI가 사용자의 행복과 직접 연관이 있는가? (단순 리소스 지표 지양)
- **Toil Management**: 운영팀이 여전히 수동 엑셀 작업을 하느라 코딩할 시간이 없지는 않은가?
- **Automation Trust**: 자동 복구 스크립트(Self-healing)가 오히려 더 큰 장애를 유발하지 않도록 안전장치가 있는가?

### 안티패턴
- **100% 가용성 목표**: 신의 영역에 도전하느라 엄청난 비용을 쓰고 배포 속도는 거북이가 되는 행위.
- **비난 섞인 사후 분석**: "누가 코드를 그렇게 짰어?"라고 묻는 순간, 팀원들은 실수를 숨기게 되고 시스템은 영원히 고쳐지지 않는다. (Blameless 문화 필수)

- **📢 섹션 요약 비유**: 사고가 났을 때 범인을 잡아 감옥에 보내는(비난) 대신, 다시는 사고가 안 나도록 도로를 고치고 신호등을 다는(공학) 것이 진정한 전문가의 길입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Uptime)** | 고가용성 유지 | 데이터 기반의 정밀한 리스크 관리를 통해 목표 가용성 상시 충족 |
| **정량적 (Efficiency)** | 운영 생산성 3배 향상 | Toil 제거 및 자동화를 통해 인력 추가 없이 대규모 인프라 운영 |
| **정성적 (Alignment)** | 개발-운영 갈등 해소 | 에러 예산이라는 공통의 언어로 상호 이해와 협력 관계 형성 |
| **정성적 (Resilience)** | 조직의 학습 능력 강화 | 비난 없는 사후 분석을 통해 실패를 자산으로 만드는 문화 정착 |

### 미래 전망
- **Self-Governing Systems**: 시스템이 스스로 SLO 달성 여부를 예측하고, 에러 예산이 부족해질 것 같으면 부하 분산이나 배포 주기를 자율적으로 조절할 것이다.
- **SRE for AI**: 인공지능 모델의 신뢰성(환각 방지, 응답 지연 등)을 관리하는 'AI Reliability Engineering' 분야가 새롭게 부상할 것이다.

### 참고 표준
- **Site Reliability Engineering (Google)**: SRE의 탄생과 실천 기록
- **The Site Reliability Workbook**: 구체적인 SLI/SLO 수립 가이드
- **ITIL 4**: 서비스 운영 관리 표준 (SRE 관점 수용 중)

- **📢 섹션 요약 비유**: 과거의 운영이 '기다림과 수리'였다면, 미래의 운영은 '설계와 자동화'입니다. SRE는 불확실한 세상에서 확실한 품질을 만들어내는 가장 정교한 공학적 수단입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **SRE**는 우리 집 모든 전자기기를 **"스스로 고치는 똑똑한 로봇 친구"**로 만드는 선생님과 같아요.
2. **SLI와 SLO**는 "전등이 깜빡거리면 안 돼!"라는 **약속 점수**이고, **SLA**는 그 약속을 못 지키면 선물을 주는 **계약서**랍니다.
3. 이렇게 점수를 매기고 로봇을 잘 관리하면, 우리 집은 언제나 환하고 고장 없는 멋진 집이 된답니다!
