# 595. 데이터 아키텍처 (Data Architecture) 및 설계 원칙

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 아키텍처는 기업의 비즈니스 목적을 달성하기 위해 **데이터의 수집, 저장, 통합, 관리, 배포 방식을 정의**하는 청사진으로, 데이터의 표준과 구조 및 관리 체계를 포괄하는 상위 개념이다.
> 2. **가치**: 파편화된 데이터 사일로(Silo)를 제거하고 데이터의 무결성과 일관성을 확보하여, 신뢰할 수 있는 정보를 의사결정권자에게 적시에 제공함으로써 **데이터 중심 경영(Data-driven)**의 기술적 토대를 마련한다.
> 3. **융합**: 전사 아키텍처(EA)의 핵심 구성 요소이며, 최근에는 데이터 레이크(Data Lake), 클라우드 기반 데이터 플랫폼, 그리고 인공지능 학습을 위한 데이터 공급망 보안과 긴밀하게 결합되고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 데이터 아키텍처는 시스템이 다루는 "정보의 뼈대"다. 어떤 데이터를 보관할지, 데이터들끼리 어떤 관계를 맺는지, 그리고 이 데이터가 전사적으로 어떻게 흐르는지를 설계하는 것이다. 데이터 표준, 데이터 구조, 데이터 관리 체계라는 3대 요소를 조율한다.

- **💡 비유**: 데이터 아키텍처는 **"거대 도시의 수도 및 물류 인프라 계획"**과 같습니다. 개별 건물을 짓는 것(앱 개발)보다 중요한 것은, 도시 전체에 깨끗한 물(데이터)이 끊기지 않고 원활하게 흐르도록 정수장(DB)과 파이프라인(ETL)을 미리 설계하는 것입니다.

- **등장 배경**:
  1. **정보의 복잡성 증대**: 시스템이 거대해지면서 동일한 데이터(예: 고객 주소)가 여러 곳에서 서로 다르게 관리되는 '데이터 불일치' 문제가 심각해졌다.
  2. **빅데이터 시대 도래**: 정형 데이터뿐만 아니라 로그, 영상 등 비정형 데이터가 폭증하면서 이를 체계적으로 수용할 새로운 저장 아키텍처가 필요해졌다.
  3. **데이터 주권 및 보안**: 개인정보보호법 등 규제 준수를 위해 데이터의 생성부터 파기까지 전 과정을 통제할 수 있는 거버넌스 설계가 필수화되었다.

- **📢 섹션 요약 비유**: 뒤죽박죽 섞인 서류 더미에서 정보를 찾는 수고를 덜기 위해, 전용 서류함과 분류 규칙을 정하는 '전사적 지식 정리' 기술입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 데이터 아키텍처의 3대 구성 요소

| 요소 명칭 | 상세 정의 | 비유 |
|:---|:---|:---|
| **데이터 표준** | 용어, 도메인, 코드, 명명 규칙 등 전사 공통 가이드 | 같은 단어, 같은 의미 쓰기 |
| **데이터 구조** | 주제 영역, 엔티티, 관계도 (ERD) | 정보들 사이의 관계 지도 |
| **데이터 관리** | 거버넌스, 품질 관리, 보안, 백업 정책 | 정보를 지키고 가꾸는 법 |

---

### 2. 데이터 아키텍처 프레임워크 및 계층 구조

비즈니스 요구가 물리적 저장소로 구체화되는 과정을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      데이터 아키텍처 (DA) 설계 레이어 및 흐름                     │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. 비즈니스 요건] ──────▶ [2. 개념 모델링] ──────▶ [3. 논리 모델링] ──────▶ [4. 물리 모델링] │
 │ (비즈니스 요구사항)         (주제 영역 도출)         (상세 속성/정규화)       (DB 튜닝/성능)  │
 │                                                                             │
 │           ┌───────────────────────────────────────────────────────┐         │
 │           ▼                                                       ▼         │
 │    [전사 데이터 표준]                                      [데이터 거버넌스]     │
 │    - 단어 사전                                             - 데이터 오너십      │
 │    - 도메인 정의                                           - 품질 지표 관리     │
 │    - 코드 표준                                             - 보안/접근 제어     │
 │                                                                             │
 │  ※ 핵심: "표준이 없으면 모델링은 모래성이다." (Standard First)                   │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 데이터 아키텍처는 **표준** 위에서 세워진다. 1) 비즈니스 목적에 따라 **개념 모델**을 잡고, 2) **논리 모델** 단계에서 정규화를 거쳐 중복을 제거한다. 3) 마지막으로 특정 DBMS(MySQL, Oracle 등)에 최적화된 **물리 모델**로 구현된다. 이 전 과정은 중앙의 **데이터 거버넌스** 팀에 의해 감시되며, 모든 용어는 전사 공통 **데이터 표준**을 따라야 한다. 그래야만 나중에 시스템 A의 'CustomerID'와 시스템 B의 'Cust_No'가 같은 데이터임을 즉시 파악할 수 있다.

---

### 심층 동작 원리: 데이터 미니멀리즘 (Data Minimization)

데이터 아키텍처 설계 시 지켜야 할 최신 보안 및 효율 원칙이다.
1. **수집 최소화**: 서비스 목적에 꼭 필요한 데이터만 수집하도록 아키텍처를 유도한다.
2. **저장 최소화**: 중복을 제거하고 보관 주기가 지난 데이터는 자동으로 파기한다.
3. **가치**: 저장 비용을 줄일 뿐만 아니라, 유출 사고 발생 시의 피해 규모(Blast Radius)를 아키텍처 수준에서 미리 제한한다.

- **📢 섹션 요약 비유**: 냉장고에 먹지도 않을 음식을 가득 채워두면 전기료만 나가고 상한 음식(품질 저하)이 생기기 쉬우니, 딱 필요한 식재료만 신선하게 관리하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 데이터 아키텍처 vs 소프트웨어 아키텍처

| 비교 항목 | 데이터 아키텍처 (DA) | 소프트웨어 아키텍처 (SA) |
|:---|:---|:---|
| **중심 대상** | **정보(Information)**의 구조 | **기능(Function)** 및 제어의 구조 |
| **지향점** | 데이터 통합, 일관성, 공유 | 가용성, 성능, 유지보수성 |
| **수명 주기** | 상대적으로 길고 보수적 | 기술 트렌드에 따라 수시 변화 |
| **비유** | 서가의 분류 체계 | 도서관 대출 서비스 로직 |

---

### 기술 융합: 데이터 패브릭 (Data Fabric)

현대 클라우드 환경에서 DA가 진화한 모습이다.
- **개념**: 흩어져 있는 다양한 소스(On-prem, Multi-cloud)의 데이터를 지능형 자동화 기술로 연결하여 하나의 거대한 가상 저장소처럼 보이게 하는 기술.
- **가치**: 데이터가 어디에 있든 사용자는 위치를 알 필요 없이 표준화된 방식으로 데이터를 꺼내 쓸 수 있다.

- **📢 섹션 요약 비유**: 전국에 흩어진 수천 개의 도서관 책을 내 스마트폰 앱 하나로 빌려볼 수 있게 해주는 '디지털 통합 도서 망'과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 전사 통합 고객 뷰 (Single Customer View) 구축

- **문제 상황**: 마케팅팀은 '고객 A'의 정보를 원하지만, 쇼핑몰 DB와 고객센터 DB의 주소 정보가 달라 정확한 발송이 안 되고 있음. (데이터 파편화)
- **의사결정 (데이터 아키텍처 설계)**:
  1. **표준화**: '주소' 필드의 포맷과 코드(우편번호 체계 등)를 전사 공통 표준으로 확정한다.
  2. **MDM (Master Data Management) 도입**: 고객 정보의 기준이 되는 마스터 DB를 구축하고, 각 시스템의 데이터는 이곳을 참조하도록 동기화한다.
  3. **데이터 혈통(Lineage) 관리**: 데이터가 어디서 생성되어 어떻게 변환되었는지 전 과정을 추적 가능하게 모델링한다.
  4. **보안 계층화**: 데이터 중요도에 따라 개인정보는 별도의 보안 구역(Secure Zone)에 격리하고, 분석용 데이터는 가명화하여 데이터 레이크로 보낸다.

### 도입 체크리스트
- **Semantic Consistency**: 서로 다른 시스템 간에 동일한 데이터를 다른 이름으로 부르고 있지는 않은가?
- **Data Quality Index**: 누락값, 오타, 중복 데이터 비중을 정기적으로 측정하고 관리하는가?
- **Cloud Agnostic**: 특정 클라우드 벤더의 전용 DB 서비스에 너무 종속되어 나중에 이전하기 힘들지는 않은가?

### 안티패턴
- **데이터 사일로 방치**: 부서 간의 이기주의로 데이터를 공유하지 않고 각자 DB를 고집하는 경우. (전사적 통찰력 상실)
- **과도한 정규화**: 물리 모델링 단계에서도 이론적인 정규화만 고집하여 조회 성능이 극도로 떨어지는 상황. (역정규화 고려 필수)

- **📢 섹션 요약 비유**: 집을 지을 때 각 방의 벽지를 다르게 바르는 것(개별 기능)은 괜찮지만, 전선이나 수도관(데이터 표준) 규격이 제각각이면 집 전체가 엉망이 된다는 점을 명심해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Performance)** | 데이터 검색 속도 3배 향상 | 체계적인 인덱싱과 비식별화된 데이터 레이크 구축을 통한 분석 속도 개선 |
| **정량적 (Cost)** | 데이터 중복 저장 비용 30% 감소 | 전사 데이터 통합 및 중복 제거를 통한 스토리지 비용 절감 |
| **정성적 (Standard)** | 비즈니스 커뮤니케이션 명확화 | 전사 공통 용어 사전을 통해 부서 간 데이터 해석 오해 방지 |
| **정성적 (Trust)** | 데이터 신뢰도 확보 | 고품질 데이터 공급을 통해 AI 모델의 정확도 및 경영진 의사결정 신뢰도 향상 |

### 미래 전망
- **Active Metadata**: 메타데이터 자체가 AI와 결합하여 스스로 데이터의 이상을 감지하고 고치는 '자율 주행형 데이터 아키텍처'가 실현될 것이다.
- **Data Privacy Vaults**: 보안이 극도로 강화된 데이터 저장소 아키텍처가 기본이 되어, 애플리케이션은 데이터를 직접 만지지 않고 토큰만으로 업무를 수행하는 시대가 올 것이다.

### 참고 표준
- **DAMA-DMBOK**: 데이터 관리 지식 체계 (글로벌 표준)
- **ISO/IEC 11179**: 메타데이터 등록 및 관리 표준
- **Zackman Framework**: 전사 아키텍처 기술 프레임워크

- **📢 섹션 요약 비유**: "데이터는 새로운 시대의 원유다." 하지만 정제되지 않은 원유는 쓸모가 없습니다. 데이터 아키텍처는 이 원유를 황금과 같은 가치로 바꿔주는 '디지털 정유 공장'의 설계도입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **데이터 아키텍처**는 전교생의 일기장을 **"누가 썼는지, 어떤 내용인지" 한눈에 보기 좋게 정리하는 마법의 도서관 계획표**와 같아요.
2. 일기장을 아무렇게나 쌓아두면 나중에 찾기 힘들겠죠? 그래서 **이름순, 날짜순으로 딱딱 자리를 정해주는 규칙**을 만드는 거예요.
3. 이 규칙을 잘 지키면, 나중에 우리 반에서 누가 제일 책을 많이 읽었는지 아주 금방 찾아낼 수 있답니다!
