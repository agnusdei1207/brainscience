# 596. 데이터 모델링 단계: 개념, 논리, 물리 모델링

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 모델링은 복잡한 현실 세계의 비즈니스 요구사항을 추상화하여 **컴퓨터가 처리할 수 있는 데이터 구조로 변환**하는 핵심 설계 과정으로, 개념-논리-물리의 3단계를 거치며 구체화된다.
> 2. **가치**: 데이터의 중복을 제거하고 정합성을 확보하여 '단일 진실 공급원(SSOT)'을 구축하며, 명확한 ERD(Entity Relationship Diagram)를 통해 비즈니스 이해관계자와 개발자 간의 의사소통 오해를 원천 차단한다.
> 3. **융합**: 소프트웨어 공학의 요구사항 분석과 데이터베이스 기술이 결합되는 지점이며, 현대 아키텍처에서는 마이크로서비스(MSA)의 서비스 경계(Bounded Context)를 획정하는 도구로도 활용된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 데이터 모델링은 추상적인 업무 규칙을 '데이터'라는 틀에 담는 작업이다. 집을 짓기 전 "어떤 방이 필요하고(개념), 가구는 어떻게 배치하며(논리), 벽돌은 무엇을 쓸지(물리)" 결정하는 것과 같다.

- **💡 비유**: 데이터 모델링은 **"도시 계획 및 지도 제작"**과 같습니다. 1) 어디가 상업 지구이고 주거 지구인지 큰 그림을 그리고(개념), 2) 각 구역의 지번과 도로명을 정확히 부여하며(논리), 3) 실제 아스팔트를 깔고 가로등을 설치하는(물리) 과정과 같습니다.

- **등장 배경**:
  1. **정보의 파편화 방지**: 체계적인 모델링 없이 DB를 만들면 동일한 데이터가 중복 저장되어 정합성이 깨지는 '업데이트 이상' 현상이 발생한다.
  2. **비즈니스 로직의 데이터 투영**: 업무 프로세스가 복잡해지면서, 데이터를 어떻게 구조화하느냐가 시스템 성능과 유연성의 80%를 결정하게 되었다.
  3. **데이터 주권 및 표준화**: 전사적으로 공통된 데이터 정의를 사용하여 시스템 간 연동성을 높이려는 요구가 커졌다.

- **📢 섹션 요약 비유**: 생각 속에만 있는 '맛있는 음식'을 실제 '레시피와 식재료 목록'으로 문서화하여, 누가 요리해도 똑같은 맛이 나게 만드는 설계 공정입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 데이터 모델링 3단계 상세 비교

추상화 수준에 따라 하향식(Top-down)으로 진행된다.

| 단계 | 명칭 | 주요 활동 | 산출물 | 비유 |
|:---|:---|:---|:---|:---|
| **1단계** | **개념 모델링** | 주제 영역 도출, 핵심 엔티티 식별 | 개별 ERD (High-level) | 밑그림 그리기 |
| **2단계** | **논리 모델링** | **정규화**, 상세 속성 정의, M:N 해소 | 전사 ERD (Detailed) | 상세 설계도 작성 |
| **3단계** | **물리 모델링** | 저장 구조 결정, 인덱싱, **역정규화** | Table 명세서, DDL | 실제 건축 자재 선택 |

---

### 2. 데이터 모델링 상세 아키텍처 및 흐름

현실 세계가 어떻게 0과 1의 데이터 테이블로 변하는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      데이터 모델링의 단계별 구체화 프로세스 아키텍처               │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [현실 세계 (Reality)] ──▶ [개념 (Conceptual)] ──▶ [논리 (Logical)] ──▶ [물리 (Physical)] │
 │                                                                             │
 │  ┌────────────┐          ┌──────────────┐      ┌──────────────┐      ┌────────────┐ │
 │  │ "고객이     │          │  Entity:     │      │ Table:       │      │ Index:     │ │
 │  │  상품을     │ ──▶      │  고객, 상품,  │ ──▶  │ CUSTOMER,    │ ──▶  │ PK_CUST_ID │ │
 │  │  주문한다"  │          │  주문         │      │ PRODUCT,     │      │ Partition  │ │
 │  └────────────┘          └──────┬───────┘      │ ORDER        │      └────────────┘ │
 │                                 │              └──────┬───────┘             ▲       │
 │                                 ▼                     ▼                     │       │
 │                          [핵심 엔티티 도출]      [정규화 수행]          [성능 최적화]    │
 │                                                                                     │
 │  ※ 핵심: 논리 모델링은 DBMS 독립적이며, 물리 모델링은 DBMS 종속적임                   │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 모델링의 꽃은 **논리 모델링**이다. 이 단계에서는 특정 DB 제품(Oracle, MySQL 등)을 신경 쓰지 않고, 비즈니스 규칙에만 집중하여 데이터의 중복을 제거하는 **정규화(Normalization)**를 수행한다. 반면 **물리 모델링**은 실제 성능을 고려하는 단계다. 조회 성능이 너무 느리다면 정규화된 테이블을 다시 합치는 **역정규화**를 하거나, 특정 데이터를 물리적으로 나누는 **파티셔닝(Partitioning)** 전략을 수립한다.

---

### 심층 동작 원리: 엔티티(Entity) 식별의 3대 요건

무엇을 데이터로 저장할지 결정하는 기준이다.
1. **유일성**: 각 인스턴스를 식별할 수 있는 고유한 키(PK)가 존재해야 함.
2. **복수성**: 최소 2개 이상의 인스턴스가 존재해야 함. (단일 값은 속성임)
3. **업무 연관성**: 실제 비즈니스 프로세스에서 관리할 필요가 있는 정보여야 함.

- **📢 섹션 요약 비유**: 보물 지도에 바위(엔티티)는 표시하지만, 굴러다니는 모래알(사소한 데이터)까지 다 그리지는 않는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 논리 모델링 vs 물리 모델링 상세 비교

| 비교 항목 | 논리 모델링 (Logical) | 물리 모델링 (Physical) |
|:---|:---|:---|
| **사용 언어** | 비즈니스 용어 (한글 등) | **DB Object 명칭 (English)** |
| **목표** | 중복 제거, 데이터 무결성 | **조회 성능, 가용성, 효율성** |
| **DBMS 의존성** | 없음 (전 기종 공통) | **있음 (특정 DBMS 특성 반영)** |
| **핵심 기법** | 정규화 (1~3차) | 인덱싱, 파티셔닝, 반정규화 |
| **비유** | 상세 가구 배치도 | 실제 못과 망치질 |

---

### 기술 융합: DDD와 데이터 모델링

현대적인 개발 방법론에서는 데이터 모델링이 도메인 설계와 융합된다.
- **Aggregate**: 논리적으로 함께 변해야 하는 엔티티들의 묶음.
- **가치**: 데이터를 쪼개는 기준을 단순 수치가 아닌 '비즈니스 트랜잭션의 경계'로 설정하여 MSA 전환을 용이하게 한다.

- **📢 섹션 요약 비유**: 서랍장을 칸칸이 나누는 것(전통적 모델링)보다, 아예 용도별로 독립된 방(MSA/DDD)을 따로 만드는 것이 더 효율적이라는 현대 건축의 지혜입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 트래픽이 폭주하는 '쿠폰 발급 시스템' 모델링

- **문제 상황**: 이벤트 시 초당 수만 건의 쿠폰이 발급되는데, '사용자-쿠폰-이벤트' 테이블이 너무 정교하게 정규화되어 있어 조회 시 조인(Join) 부하로 시스템이 느려짐.
- **의사결정 (단계별 모델링 전략)**:
  1. **논리 단계**: 3차 정규화까지 완벽히 수행하여 데이터 중복을 없애고 정합성을 확보한다.
  2. **물리 단계 (반정규화)**: 성능 저하를 막기 위해 '쿠폰' 테이블에 '이벤트명'과 '사용자 닉네임' 속성을 중복해서 추가(Redundancy)한다. (조인 제거)
  3. **물리 단계 (파티셔닝)**: 쿠폰 번호의 범위에 따라 물리적인 저장소를 나누는 **Range Partitioning**을 적용하여 I/O 부하를 분산한다.
  4. **결과**: 데이터 일관성은 약간의 애플리케이션 로직으로 보완하되, 사용자 응답 속도를 10배 이상 향상시킨다.

### 도입 체크리스트
- **Normalization**: 무결성을 위해 최소 3차 정규화까지는 검토했는가?
- **Relationship Integrity**: 모든 엔티티 간의 관계가 명확하며, 부모-자식 관계가 올바르게 설정되었는가? (FK 설정 여부)
- **Naming Standard**: 전사 데이터 표준에 맞는 명명 규칙을 따르고 있는가?

### 안티패턴
- **논리 모델 건너뛰기**: 요구사항 듣자마자 바로 `CREATE TABLE`부터 치는 행위. 나중에 업무 규칙이 조금만 바뀌어도 DB 전체를 갈아엎어야 한다.
- **전지전능한 엔티티**: 하나의 테이블에 100개가 넘는 컬럼을 때려 넣는 행위. (응집도 파괴)

- **📢 섹션 요약 비유**: "일단 만들고 보자"는 무모함이 나중에 "어디에 뭐가 있는지 모르는" 쓰레기 집(나쁜 데이터 구조)을 만듭니다. 설계의 3단계를 지키는 것이 진정한 프로의 자세입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Quality)** | 데이터 오류 90% 감소 | 정규화를 통해 데이터 중복 및 수정 이상(Anomaly) 원천 제거 |
| **정량적 (Performance)** | 조회 성능 최적화 | 물리 모델링 단계의 인덱스 설계 및 파티셔닝으로 속도 개선 |
| **정성적 (Standard)** | 비즈니스 가독성 확보 | ERD를 통해 복잡한 업무 규칙을 한눈에 파악 가능한 문서 자산화 |
| **정성적 (Stability)** | 시스템 유연성 증대 | 논리 모델이 튼튼하면 하부 DB 기술이 바뀌어도 비즈니스 핵심 유지 |

### 미래 전망
- **NoSQL modeling**: 정형 데이터뿐만 아니라 문서형, 그래프형 데이터를 위한 새로운 차원의 논리 모델링 기법이 표준화될 것이다.
- **AI-Automated Modeling**: 비즈니스 기획서를 AI가 읽고 최적의 3차 정규화 ERD와 DDL 문을 1초 만에 생성해주는 지능형 모델링 시대가 올 것이다.

### 참고 표준
- **ANSI/SPARC 3-level Architecture**: 데이터 독립성을 위한 기본 원리
- **IDEF1X**: 정보 모델링을 위한 표준 표기법
- **DAMA Dictionary of Data Management**: 데이터 모델링 전문 용어 표준

- **📢 섹션 요약 비유**: 데이터 모델링은 소프트웨어라는 집의 '영혼'을 설계하는 일입니다. 겉모양(UI)은 화려할 수 있어도, 뼈대(Data Model)가 부실하면 그 집은 결코 오래 버틸 수 없습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **개념 모델링**은 우리 집 보물 지도를 그릴 때 **"보석함과 열쇠"가 필요하다고 크게 그리는 것**과 같아요.
2. **논리 모델링**은 그 보석함 안에 **칸막이를 만들어서 다이아몬드와 루비를 따로 담는 규칙**을 정하는 거랍니다.
3. **물리 모델링**은 그 보석함을 **나무로 만들지, 쇠로 만들지 진짜 재료를 고르는 것**과 같아요!
