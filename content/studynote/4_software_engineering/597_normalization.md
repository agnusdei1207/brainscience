# 597. 정규화 (Normalization) 및 역정규화 (Denormalization)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 정규화는 데이터의 중복을 제거하고 관계를 분해하여 **데이터의 일관성과 무결성을 보장**하는 논리적 설계 기법이며, 역정규화는 시스템의 성능 향상을 위해 의도적으로 **중복을 허용하거나 테이블을 병합**하는 물리적 최적화 기법이다.
> 2. **가치**: 정규화를 통해 '이상 현상(Anomaly)'을 방지하여 데이터의 정확성을 지키고, 역정규화를 통해 조인(Join) 부하를 줄여 대규모 트래픽 환경에서의 응답 속도를 확보하는 등 **안정성과 성능 사이의 균형**을 조절한다.
> 3. **융합**: 데이터베이스 설계뿐만 아니라 마이크로서비스(MSA)의 서비스 간 데이터 공유 전략과 결합되며, 최근에는 NoSQL의 '스키마리스' 특성과 융합되어 다양한 형태의 데이터 아키텍처로 구현되고 있다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **정규화 (Normalization)**: 중복 데이터를 쪼개서 여러 테이블에 나누어 담는 것이다. (무결성 중심)
  - **역정규화 (Denormalization)**: 쪼개진 테이블을 다시 합치거나 특정 컬럼을 중복해서 넣는 것이다. (성능 중심)

- **💡 비유**: 
  - **정규화**: 집 안의 물건들을 **"용도별로 수납함에 딱딱 나눠 담는 것"**과 같습니다. 필기구는 필기구함에, 약은 약 상자에 넣으면 찾기 쉽고 관리가 편합니다.
  - **역정규화**: 자주 쓰는 물건을 매번 수납함에서 꺼내기 귀찮아서 **"거실 테이블 위에 미리 꺼내두는 것"**과 같습니다. 중복되거나 지저분해 보일 수 있지만, 필요할 때 즉시 쓸 수 있어 빠릅니다.

- **등장 배경**:
  1. **데이터 무결성의 위기**: 정규화되지 않은 데이터는 한 곳만 수정했을 때 다른 곳의 데이터와 불일치가 생기는 '수정 이상' 문제를 일으켰다.
  2. **성능 병목 현상**: 정규화가 너무 잘 되어 있으면 정보를 하나 보려고 해도 수십 개의 테이블을 합쳐야(Join) 했고, 이는 대용량 시스템에서 성능 저하의 주범이 되었다.
  3. **비즈니스 환경의 변화**: 데이터 저장 비용보다 '사용자의 대기 시간'이 더 비싼 유효 가치가 됨에 따라, 의도적인 중복(역정규화)이 전략적 선택이 되었다.

- **📢 섹션 요약 비유**: 옷장을 완벽하게 정리(정규화)할 것인가, 아니면 내일 입을 옷을 의자에 걸어둘(역정규화) 것인가에 대한 실용적인 선택의 문제입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 정규화의 단계별 특징 (1NF ~ BCNF)

함수적 종속성을 제거하여 무결성을 높여가는 과정이다.

| 단계 | 규칙 (Rule) | 제거 대상 | 비유 |
|:---|:---|:---|:---|
| **제1정규형 (1NF)** | 원자값 (Atomic Value) 구성 | 중복된 그룹, 다중 값 | 한 칸에 물건 하나만 넣기 |
| **제2정규형 (2NF)** | **부분 함수적 종속** 제거 | 기본키 일부에만 종속된 속성 | 주소록에서 이름과 성별 분리 |
| **제3정규형 (3NF)** | **이행 함수적 종속** 제거 | 속성 간의 종속 관계 (A->B, B->C) | 친구의 친구 정보 따로 보관 |
| **보이스-코드 (BCNF)** | 결정자이면서 후보키가 아닌 것 제거 | 모든 결정자는 후보키여야 함 | 반장 선거 규칙 정비 |

---

### 2. 정규화와 역정규화의 트레이드 오프 아키텍처

두 기법이 시스템의 서로 다른 지표에 미치는 영향을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      정규화 vs 역정규화 (Stability vs Performance)               │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │  [정규화 방향 ──▶]                                     [◀── 역정규화 방향]     │
 │  (Normalization)                                     (Denormalization)      │
 │                                                                             │
 │  ┌──────────────────┐                                 ┌──────────────────┐  │
 │  │   Stability      │                                 │   Performance    │  │
 │  │  (Data Integrity)│                                 │   (Query Speed)  │  │
 │  └────────┬─────────┘                                 └────────┬─────────┘  │
 │           │                                                    │            │
 │           ▼                                                    ▼            │
 │  - 데이터 중복 최소화                                   - 테이블 조인 최소화        │
 │  - 저장 공간 절약                                      - 조회 쿼리 단순화          │
 │  - **입력/수정/삭제** 성능 우수                         - **대량 조회** 성능 우수     │
 │  - 이상 현상 방지                                      - 데이터 정합성 리스크 존재   │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 실무 원칙: "논리 설계는 정규화로, 물리 설계는 역정규화로"                 │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처 설계자는 두 극단 사이의 **최적점**을 찾아야 한다. **정규화**를 하면 데이터가 쪼개져서 저장 공간이 줄어들고 수정(Update) 시 오차가 없지만, 조인 부하로 조회(Select) 속도가 느려진다. 반면 **역정규화**를 하면 데이터를 미리 합쳐놓거나 중복시켜서 조회 속도는 비약적으로 빨라지지만, 데이터가 수정될 때 여러 곳을 동시에 고쳐야 하므로 일관성이 깨질 위험이 크다. 결론적으로 쓰기 중심 시스템은 정규화를, 읽기 중심 시스템(통계, 포털 등)은 적절한 역정규화를 지향한다.

---

### 심층 동작 원리: 역정규화의 3대 기술

성능을 위해 의도적으로 중복을 만드는 기법들이다.
1. **테이블 병합/분할**: 1:1 관계 테이블을 합치거나, 너무 큰 테이블을 수직/수평으로 쪼갬.
2. **중복 컬럼 추가**: 조인을 피하기 위해 부모 테이블의 값을 자식 테이블에 복사해둠. (예: 주문 테이블에 사용자 닉네임 저장)
3. **계산된 컬럼 추가**: 매번 `Sum`이나 `Avg`를 계산하지 않도록 합계 값을 별도 컬럼에 저장.

- **📢 섹션 요약 비유**: 계산기를 매번 두드리는 게 귀찮아서, 자주 나오는 계산 결과(9단 곱셈 등)를 벽에 붙여놓고 바로 보는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 정규화 단계별 이상 현상 (Anomaly) 방지 효과

| 이상 현상 | 설명 | 정규화 효과 |
|:---|:---|:---|
| **삽입 이상** | 데이터 삽입 시 원치 않는 정보까지 강제로 넣어야 함 | 엔티티 분리를 통해 독립적 삽입 가능 |
| **삭제 이상** | 한 정보를 지울 때 연관된 다른 중요한 정보까지 지워짐 | 관계 분리를 통해 개별 정보 보존 |
| **갱신 이상** | 중복된 정보 중 일부만 수정되어 불일치 발생 | 단일 저장소(SSOT) 구축으로 일관성 유지 |

---

### 기술 융합: MSA와 역정규화의 필연성

마이크로서비스 아키텍처는 구조적으로 역정규화를 강제한다.
- **Service Independence**: 주문 서비스가 사용자 서비스의 DB를 조회할 수 없으므로, 주문 데이터 내에 사용자의 '이름'과 '주소' 정보를 **역정규화하여 중복 저장**하는 것이 MSA의 표준 패턴이다.
- **가치**: 서비스 간의 실시간 API 호출을 줄여 네트워크 지연을 막고 독립성을 높인다.

- **📢 섹션 요약 비유**: 매번 옆 동네 이장님께 주민 명부를 물어보는 대신, 우리 동네 명부에도 옆 동네 이사 온 사람 이름을 적어두는 '자급자족 보안' 방식입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 대규모 트래픽을 견디는 '게시판 시스템' 최적화

- **문제 상황**: 게시글 목록을 조회할 때 작성자 이름과 댓글 수를 실시간으로 조인해서 가져오니, 게시글이 100만 건을 넘어서며 첫 페이지 로딩에 5초가 걸림.
- **의사결정 (정규화/역정규화 전략)**:
  1. **논리 설계**: 사용자, 게시글, 댓글 테이블은 철저히 3차 정규화하여 데이터 무결성을 지킨다.
  2. **물리적 역정규화**: '게시글' 테이블에 `author_name`과 `comment_count` 컬럼을 추가한다. (조인 2개 제거)
  3. **정합성 보장**: 댓글이 달릴 때마다 애플리케이션 레벨에서 '게시글' 테이블의 `comment_count`를 +1 해주는 로직을 추가한다. (갱신 이상 리스크 감수)
  4. **캐시 활용**: 역정규화된 데이터를 Redis에 담아 DB 접근 자체를 90% 이상 줄인다.
  5. **결과**: 데이터 일관성은 약간의 딜레이(Eventual Consistency)가 있을 수 있으나, 조회 속도는 0.1초 이내로 단축하여 사용자 이탈을 막는다.

### 도입 체크리스트
- **Read/Write Ratio**: 우리 시스템은 조회(Read)가 많은가, 수정(Write)이 많은가? (조회가 많을수록 역정규화 유리)
- **Data Integrity Risk**: 데이터가 한순간이라도 틀리면 큰일 나는 금융권 핵심 데이터인가? (그렇다면 역정규화 지양)
- **Indexing Strategy**: 역정규화 전, 적절한 인덱스 설계만으로도 성능 해결이 가능한지 먼저 검토했는가?

### 안티패턴
- **무계획적 역정규화**: 정규화도 안 된 상태에서 무작정 테이블을 합치는 행위. 나중에 데이터가 꼬였을 때 수습이 불가능하다. (반드시 정규화 후 역정규화 순서를 지킬 것)
- **수동 정합성 관리 포기**: 역정규화로 중복된 데이터를 만들어놓고, 한쪽만 고치고 나머지는 방치하는 경우.

- **📢 섹션 요약 비유**: 요리 시간을 단축하겠다고 재료를 미리 다 썰어놓는(역정규화) 것은 좋지만, 재료가 상했는지(데이터 노후화) 수시로 체크하지 않으면 결국 배탈이 난다는 점을 잊지 마세요.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Performance)** | 응답 시간 10배 이상 향상 | 복잡한 조인을 제거하고 단일 테이블 스캔으로 조회 성능 극대화 |
| **정량적 (Consistency)** | 데이터 무결성 100% 확보 | 정규화를 통해 중복에 의한 데이터 불일치 원천 차단 |
| **정성적 (Reliability)** | 시스템 신뢰도 상승 | 이상 현상이 없는 깨끗한 데이터 환경 구축 |
| **정성적 (Scalability)** | 확장성 있는 데이터 설계 | 데이터 구조의 유연함을 통해 미래 비즈니스 요건 변화에 능동적 대응 |

### 미래 전망
- **Polyglot Persistence**: 쓰기(Write)는 정규화된 RDBMS에, 읽기(Read)는 역정규화된 NoSQL에 수행하는 **CQRS** 패턴이 보편화될 것이다.
- **AI-Driven Normalization**: AI가 쿼리 패턴을 분석하여 성능이 떨어지는 지점을 찾아내고, 최적의 역정규화 시나리오를 추천 및 자동 적용하는 지능형 DB 튜닝이 실현될 것이다.

### 참고 표준
- **E.F. Codd의 관계형 모델**: 정규화의 학술적 근거
- **ANSI/SPARC 3단계 구조**: 논리/물리 모델의 분리 표준
- **DAMA-DMBOK**: 전사 데이터 관리 지침

- **📢 섹션 요약 비유**: 정규화는 '데이터의 도덕'이고 역정규화는 '데이터의 실리'입니다. 도덕을 바탕으로 실리를 추구할 때, 가장 아름답고 효율적인 아키텍처가 완성됩니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **정규화**는 장난감을 종류별로 **"색깔별, 크기별로 작은 상자에 나눠 담는 것"**과 같아요. (정리정돈!)
2. **역정규화**는 내일 학교에 가져갈 준비물을 미리 **"책상 위에 다 꺼내놓는 것"**과 같답니다. (빨리 챙기기!)
3. 정리를 잘하면 물건을 잃어버리지 않고(정확성), 미리 꺼내두면 학교에 늦지 않게 갈 수 있어요(빠름)!
