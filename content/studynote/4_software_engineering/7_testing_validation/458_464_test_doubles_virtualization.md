# 458-464. 테스트 더블과 서비스 가상화

> **핵심 인사이트**: 테스트의 적은 '의존성'이다. 결제 기능을 테스트하고 싶은데 은행 서버가 점검 중이라 테스트를 못 하는 상황은 피해야 한다. 테스트 더블은 진짜 객체 대신 '가짜 대역 배우'를 세워 의존성을 끊고, 오직 내가 검증하려는 로직에만 집중하게 해 주는 고마운 장치다.

---

## Ⅰ. 테스트 더블 (Test Double) 5종
xUnit 테스트 패턴에서 유래한, 진짜 객체 대신 사용하는 대역들의 총칭입니다.

1. **Dummy (더미)**: 인스턴스는 필요하지만 실제로 사용되지는 않는 객체. (매개변수 채우기용)
2. **Stub (스텁)**: 호출 시 미리 준비된 답변(하드코딩된 값)만 반환하는 객체. (상태 검증용)
3. **Spy (스파이)**: 스텁의 기능을 수행하면서, 호출 횟수나 인자 등 **호출 정보**를 기록하는 객체.
4. **Mock (목)**: 예상되는 호출 명세를 미리 프로그래밍하고, 그대로 실행되었는지 확인하는 객체. (**행위 검증** 중심)
5. **Fake (페이크)**: 실제 로직이 구현되어 있지만, 운영 환경에는 적합하지 않은 축소판. (예: 인메모리 DB)

---

## Ⅱ. 행위 검증 vs 상태 검증
* **상태 검증 (State Verification)**: 메서드 실행 후 객체의 '데이터 상태'가 변했는지 확인. (Stub 사용)
* **행위 검증 (Behavior Verification)**: 특정 메서드가 '몇 번 호출됐는지', '순서는 맞는지' 등 상호작용을 확인. (Mock 사용)

---

## Ⅲ. 서비스 가상화 (Service Virtualization)
* **개념**: MSA 환경에서 아직 개발되지 않았거나 접근이 불가능한 **외부 API 서버**를 가상의 스텁 서버로 모사하여 테스트를 진행하는 기술입니다.
* **장점**: 병렬 개발 가능, 테스트 환경 구축 비용 절감.

---

## Ⅳ. 개념 맵 및 요약

```ascii
[테스트 대역의 수준]

  (단순) [ Dummy ] ──> [ Stub ] ──> [ Spy ] ──> [ Mock ] (정교)
           │                                      │
           └─ 상태 검증 ────────────── 행위 검증 ───┘
```

📢 **섹션 요약 비유**: **테스트 더블**은 영화 촬영장의 '대역 배우'입니다. **더미**는 뒤에서 배경으로 서 있는 엑스트라, **스텁**은 "네"라는 말만 반복하는 인형입니다. **스파이**는 촬영 중 몇 번이나 넘어졌는지 체크하는 기록원 겸 대역이고, **목**은 감독의 지시대로 정확히 움직였는지 감시받는 주연급 대역입니다. **서비스 가상화**는 야외 촬영 대신 스튜디오에 '가짜 세트장'을 만들어 날씨 상관없이 촬영하는 것과 같습니다.
