# 61. OAuth/OIDC

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: OAuth( Open Authorization)는 제3자 애플리케이션이 사용자의 자원에 접근할 수 있도록 권한을 부여하는 개방형 표준 프로토콜이며, OIDC( OpenID Connect)는 OAuth 2.0 위에 신원 계층을 추가한 개방형 표준이다.
> 2. **가치**: OAuth/OIDC는 사용자가 자신의 자격 증명을 공유하지 않고도 제3자 애플리케이션에 접근 권한을 부여할 수 있게 하여, 편리성과 보안을 동시에 제공한다.
> 3. **융합**: OAuth/OIDC는 모던 인증의 핵심基石로서, SSO, 마이크로서비스, API 게이트웨이,零信任( Zero Trust) 보안과 결합되어 활용된다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

OAuth의 개념은 2006년에 시작되어, 2012년 OAuth 2.0( RFC 6749)이正式的 발표되었다. OAuth는"Open Authorization"의 약자로, 사용자가 비밀번호를 공유하지 않으면서도 다른 애플리케이션이 사용자의 자원에 접근할 수 있도록 권한을 부여하는 프로토콜이다. 이전에는 사용자가 한 서비스의 자격 증명을 다른 서비스에 공유해야 하는 상황이 많았는데, 이는 보안상 큰 위험이였다. OAuth는 이러한 문제를 근본적으로 해결하였다.

OAuth가 필요한 근본적인 이유는 API Economy의 발전과 사용자 편의성 향상 때문이다. 현재 웹과 모바일 애플리케이션은 다양한 서비스(소셜 로그인, 결제, 지도, 메일 등)를 통합하여 풍부한 기능을 제공한다. 이러한 통합에서 각 서비스는 사용자의 자원에 접근해야 하는데, 이때 OAuth는 사용자의 비밀번호를 공유하지 않고도 접근 권한을 부여할 수 있게 한다. 또한 OAuth는 사용자에게 접근 권한을 언제든 취소할 수 있는 control을 제공하여, 보안을 향상시킨다.

OAuth의 발전 과정을 보면, OAuth 1.0(2007)은最初的 버전으로, 서명 기반의 복잡한 메커니즘을 사용하였다. OAuth 2.0(2012)은보다 간단하고灵活한 프로토콜로, 접근 토큰 기반의 권한 부여를 도입하였다.然而 OAuth 2.0은 인증(Authentication) 기능이 없어, 이를 보완하기 위해 OIDC( OpenID Connect)가 2014년에 등장하였다. OIDC는 OAuth 2.0 위에 ID 토큰과 사용자 정보 endpoint를 추가하여, 인증 기능을 제공하면서도 OAuth 2.0의 장점을 활용할 수 있게 하였다.

OAuth 2.0의 주요 권한 부여 유형( Grant Types)을 보면, 첫째 Authorization Code Grant는 웹 애플리케이션 서버 사이드에서 사용되는 표준 흐름이다. 둘째 Implicit Grant는 브라우저 기반 애플리케이션( SPA 등)에서 사용되는 간소화된 흐름이다. 셋째 Resource Owner Password Credentials Grant는 신뢰할 수 있는 애플리케이션에서 직접 자격 증명을 교환하는 방식이다. 넷째 Client Credentials Grant는 서버 간 통신에서 사용되는 방식이다. 다섯째 Device Flow는 스마트 TV, IoT 기기 등 제한된 입력 장치에서 사용된다.

💡 OAuth/OIDC를호텔 벨بو이가 비유하면, 호텔 벨보이는 고객(사용자)에게 객실 접근 권한을 부여하는 역할을 한다. 고객이 프런트 데스크에서 인증을 받으면, 벨보이( OAuth)에게 객실 열쇠를 받을 수 있다. 벨보이는 객실 열쇠를 가지고 있지만, 고객의 동의 없이는 객실에 접근할 수 없다. 또한 고객은 벨보이의 접근 권한을 언제든 박탈할 수 있다. OAuth도 마찬가지로, 사용자가 인증을 하면 접근 토큰을 발급받아 다른 서비스(제3자 애플리케이션)가 사용자의 자원에 접근할 수 있게 한다. 그러나 제3자 애플리케이션은 사용자의 자격 증명을 알지 못하며, 접근 권한은 언제든 취소할 수 있다.

📢 OAuth/OIDC는현대 인터넷 보안의 핵심 프로토콜이다. 그러나 OAuth/OIDC도 Implementation이 부적절하면 보안漏洞이 발생할 수 있다. 따라서 개발자들은 OAuth/OIDC의 사양을 깊이 이해하고, 보안 Best Practice를 따라야 한다. 특히 CSRF 공격 방지를 위한 state 매개변수 사용, HTTPS 사용, 토큰的安全管理, 그리고 적절한 scope 설정 등이 중요하다. 또한 OAuth/OIDC의脆弱性에 대한 지속적인 학습과 업데이트도 필요하다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

OAuth/OIDC의 기술 아키텍처는 네 계층으로 구성된다. 第一層은 자원所有者( Resource Owner) 계층으로, 사용자가 자원의 소유자이며, 제3자 애플리케이션에 접근 권한을 부여하는 주체이다. 第二層은 자원 서버( Resource Server) 계층으로, 사용자의 자원을 호스팅하고, 접근 토큰을 검증하여 자원에 대한 접근을 허용하는 서버이다. 第三層은 권한 서버( Authorization Server) 계층으로, 사용자를 인증하고, 접근 토큰과 ID 토큰을 발급하는 서버이다. 第四層은 클라이언트( Client) 계층으로, 사용자의 자원에 접근하려는 제3자 애플리케이션이다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    OAuth/OIDC 기술 아키텍처                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│                      ┌─────────────────┐                         │
│                      │  자원所有者     │                         │

│  ┌─────────────────────────────────────────────────────────┐    │

│  │  ┌──────────┐  ┌──────────┐  ┌──────────────────┐     │    │
│  │  │ 사용자   │  │ 토큰    │  │ ID 토큰 발급  │     │    │

└─────────────────────────────────────────────────────────────────┘
```

OAuth/OIDC의核心 원리 중 하나는 Scope 기반 권한 관리이다. OAuth에서 scope은 클라이언트가 접근할 수 있는 자원의 범위를定義한다. 예를 들어"email" scope는 사용자의 이메일 주소에만 접근 가능하고,"calendar" scope는 일정에만 접근 가능하다. 사용자는 권한 부여 시哪些 scope를 허용할지 선택할 수 있다. OIDC에서도 similar하게"openid", "profile", "email" 등의 scope가 정의되어 있다. Scope 기반 권한 관리를 통해 최소 권한 원칙을 적용하고, 클라이언트가 필요한最小限の 권한만 부여받을 수 있게 한다.

OAuth/OIDC의 또 다른核心 원리는토큰 기반 인증이다. OAuth 2.0에서 클라이언트는 자원 서버에 접근할 때Bearer 토큰( Bearer Token)을 사용한다.Bearer 토큰을持有하는 누구든 해당 토큰으로 보호된 자원에 접근할 수 있다. 따라서Bearer 토큰의安全管理이非常重要하다. OIDC에서는 여기에 더하여 ID 토큰을 발급하는데, ID 토큰은 JWT( JSON Web Token) 형식으로, 사용자의 인증 정보와Claims를 포함한다. ID 토큰은 클라이언트가 사용자의 신원을 확인하는 데 사용된다.

OAuth/OIDC의 핵심 원리 중 세 번째는권한 부여 흐름의 분리이다. OAuth 2.0은 권한 부여( Authorization)와 자원 접근( Resource Access)을 분리한다. 사용자는 권한 서버에서 인증을 받아 접근 토큰을 받고, 이 토큰으로 자원 서버의 자원에 접근한다. 이러한 분리는 보안을 향상시키고, 확장성을 높이며, 다양한 시나리오에 대한 유연성을 제공한다. 권한 서버와 자원 서버가 분리되어 있어, 각각을독립적으로 확장하고管理할 수 있다.

📢 OAuth/OIDC의 아키텍처는영화관 티켓 교환 시스템과 같다. 영화관(자원 서버)에서 영화를观看하려면 매표소(권한 서버)에서 티켓(접근 토큰)을 발급받아야 한다. 영화관 티켓에는 좌석 정보(접근 권한)가記載되어 있다. 사용자는 매표소에서 인증을 받고 티켓을 발급받으며, 영화관에서는 티켓의 유효성을 확인하고 영화를 제공한다. 만약 사용자가 친구에게让自己的 티켓을 공유하면(Scope 공유), 친구도 동일한 영화관을 방문할 수 있다. OAuth도 마찬가지로, 사용자가 권한 서버에서 인증을 받고 접근 토큰을 발급받아, 자원 서버의 자원에 접근한다.

## Ⅲ. 구현 및 실무 응용 (Implementation & Practice)

OAuth/OIDC 실무 구현은 다음과 같은 단계로 이루어진다. 第一 단계는 Use Case 분석으로, 어떤 유형의 클라이언트(웹, 모바일, SPA, 서버 간 등)가 어떤 자원(API, 데이터 등)에 접근해야 하는지를分析한다. 第二 단계는 권한 서버 선택으로, 자체 개발 또는 외부 서비스( Auth0, Okta, Firebase, Microsoft Entra ID 등)를 선택한다. 第三 단계는 클라이언트 등록으로, 권한 서버에 클라이언트를 등록하고, client_id, client_secret, redirect_uri, allowed scope 등을 설정한다. 第四 단계는 권한 부여 흐름 구현으로, 선택한 grant type에 따라 권한 부여 요청, 토큰 요청, 토큰 저장 등의 로직을 구현한다. 第五 단계는 API 보호 구현으로, 자원 서버에서 접근 토큰 검증을 구현한다.第六 단계는 테스트 및 보안 검토로, 다양한 시나리오에서 올바르게 작동하는지를 테스트하고, 보안을 검토한다.

OAuth/OIDC의대표적 활용 사례를 보면 다음과 같다. 첫째, 소셜 로그인이다. 웹사이트나 앱에서"Google로 로그인", "Facebook으로 로그인" 등의 버튼을 클릭하면, OAuth/OIDC를 통해 사용자의 인증이 이루어진다. 사용자는 자신의 Google 또는 Facebook 자격 증명을 웹사이트에 공유하지 않고도 로그인할 수 있다. 둘째, SaaS 서비스 연동이다. 기업에서 사용하는 여러 SaaS 서비스( Salesforce, Slack, Microsoft 365 등)를 OIDC/ OAuth를 통해 연동하여, SSO(Single Sign-On)를実現한다. 셋째, API Economy이다. 다양한 서비스가 API를对外开放할 때, OAuth를 통해 API 접근 권한을 控制한다. 예를 들어, 택배 서비스 API, 지도 API, 결제 API 등에 OAuth로 접근 권한을 부여한다.

OAuth/OIDC 구현 시 주의해야 할 실무적 과제도 있다. 첫째, client_secret 관리이다. 클라이언트 비밀번호( client_secret)가 유출되면, 공격자가 사용자를 가장할 수 있다. Therefore client_secret의安全한 관리(환경 변수 사용, 정기적 로테이션 등)가 필요하다.둘째, Redirect URI 검증이다. 권한 서버는 redirect_uri를厳格히検証해야 하며, 그렇지 않으면 authorization code를가로채는 공격( PKCE를 통해 완화 가능)에 취약해진다.셋째, Scope의 과도한 부여이다. 필요한 scope보다 더 많은 scope를 부여하면,万一 클라이언트가 Compromised될 때 피해가 커진다.따라서 최소 권한 원칙을遵守하여 필요한最小限の scope만 부여해야 한다.넷째, 토큰 만료 관리이다. Access token과 refresh token의 만료 시간을 적절히 설정하고, 토큰 취소 메커니즘을 구현해야 한다.

OAuth/OIDC의 보안 Best Practice를 보면, 第一은 PKCE( Proof Key for Code Exchange) 사용이다. Public 클라이언트(SPA, 모바일 앱)에서는 PKCE를 반드시 사용하여, authorization code 가로채기 공격을 방지해야 한다. 第二はHTTPS 사용이다. 모든 OAuth/OIDC 통신은 HTTPS를 사용해야 하며, HTTP를 사용하면 통신이傍受될 위험이 있다. 第三はstate 매개변수 사용이다. CSRF 공격 방지를 위해, 권한 부여 요청에 state 매개변수를 포함하고, 콜백에서検証해야 한다. 第四は토큰 저장소의安全한管理이다. Access token과 refresh token을的安全한場所に保存し、불필요하게 노출되지 않도록 해야 한다.

📢 OAuth/OIDC 실무 적용은음식점 예약 시스템과 같다. 음식점(자원 서버)에 가기 위해 고객(사용자)은 플랫폼(권한 서버)에서 예약을 해야 한다. 플랫폼에서 인증을 하면 예약 확인서(접근 토큰)를发放받는다. 음식점에 도착하면 예약 확인서를 제시하고(토큰 검증), 자리를 안내받는다(자원 접근). 만약 친구가 자신의 예약 확인서를入手하면, 해당 음식점을 방문할 수 있기에( 토큰 도용), 예약 확인서를 안전하게保管해야 한다. OAuth도 마찬가지로, 접근 토큰을的安全な管理가非常重要であり、万一 유출되면 적절한 대응(토큰 취소, 재발급)이 필요하다.

## Ⅳ. 품질 관리 및 테스트 (Quality & Testing)

OAuth/OIDC의品質評價는 보안 강도, 상호운용성, 확장성, 그리고 사용자 경험의 네 차원으로 구분된다. 보안 강도에서는 프로토콜 구현의堅牢성, 토큰 보안 수준, 그리고 공격 방어 능력 등이 평가된다. 상호운용성에서는 다양한 클라이언트와 서버 간의 interoperatibility, 표준 준수의 정도 등이 평가된다. 확장성에서는 대량 트래픽 처리 능력, 다수의 클라이언트/자원 서버 관리 능력 등이 평가된다. 사용자 경험에서는 권한 부여 페이지의 사용성, 오류 메시지의 명확성, 그리고 동의 프로세스의 편의성 등이 평가된다.

OAuth/OIDCのテスト方法論으로는단위 테스트, 통합 테스트, 보안 테스트, 그리고互操作性テストが活用된다. 단위 테스트에서는 각 모듈(토큰 생성, 검증, 스코프 처리 등)의 기능을 테스트한다. 통합 테스트에서는 권한 서버, 클라이언트, 자원 서버 간의 흐름을 통합적으로テスト한다. 보안 테스트에서는 다양한 공격 시나리오(토큰 가로채기, CSRF, 리다이렉션 공격 등)를模拟하여 보안을 테스트한다.互操作性テストでは여러 구현체 간의 interoperatibility를テスト한다.

OAuth/OIDC品質管理에서特别注意해야 할 영역은 토큰 보안과 권한 서버의 보안이다. Access token과 refresh token은 매우 중요한 credential이므로, 그 관리가 가장 중요하다. Token은 전송 시 암호화되어야 하고( HTTPS), 저장 시도 암호화되어야 하며, 액세스는 엄격히 통제되어야 한다. 또한 권한 서버는 모든 인증의 핵심이므로, 권한 서버 자체의 보안이 매우 중요하다. 권한 서버에 대한 엄격한 접근 통제, 모니터링, 그리고 정기적 보안 검토가 필요하다.

📢 OAuth/OIDC品質 평가는은행의رقية 보안과 같다. 은행에서는 예약 확인을 위해 다단계 인증, 통신 구간 암호화, 이상 거래 탐지 등 여러保安措置를 적용한다. OAuth/OIDC도 마찬가지로,authorization flow, token 관리,scope 검증 등 여러_security layers를 적용하여 보안을保証한다. 은행의رقية 보안이突破되면 고객의 돈이 도난당하듯이, OAuth/OIDC 보안이突破되면 使用자 정보가 유출될 수 있다. 따라서 OAuth/OIDC도 은행 수준의严格的セキュリティ管理가 필요하다.

## Ⅴ. 최신 트렌드 및 결론 (Trends & Conclusion)

OAuth/OIDC의 최신 트렌드는 크게 세 가지 방향으로 전개되고 있다. 첫째, Passwordless 인증과의 결합이다. OAuth/OIDC가 전통적 비밀번호 기반 인증뿐 아니라, FIDO2/WebAuthn, Passkey 등 비밀번호 없는 인증 방식과 결합되고 있다. 둘째, Continuous Access Evaluation( CAE)의 발전이다. OAuth 토큰이 발급된 후에도, 권한 서버가 클라이언트에게 토큰 무효화나 권한 변경을 실시간으로 알리는 메커니즘이 발전하고 있다. 셋째, GNAP( Grant Negotiation and Authorization Protocol)의 등장이다. OAuth 2.0의 limitations을 극복하기 위한新一代 프로토콜로서, GNAP가 IETF에서 표준화 작업이 진행되고 있다.

OAuth/OIDC의 미래를展望하면, 다음과 같은 발전이 예상된다. Passwordless 인증이 보편화되면, OAuth/OIDC도 이에适应的进化할 것이다. 또한 AI 기반 이상 접근 탐지와自适应 인증이 결합되어, 더욱 지능적인 인증 체계가 될 것이다. 그리고 블록체인 기반 분산 신원( DID, Decentralized Identity)과 OAuth/OIDC의 결합도 주목할 만하다. 향후에도 OAuth/OIDC는 인터넷 인증의 핵심 프로토콜로 자리매김할 것이다.

OAuth/OIDC를 도입하려는 기업에 대한 제언을 한다. 第一は표준 준수의 철저이다. OAuth 2.0, OIDC 사양을 정확히 준수하고, Security Best Practice를 따라야 한다. 第二はMFA 적용이다. 중요한 자원에 접근할 때는 OAuth/OIDC뿐 아니라 MFA를 추가하여 보안을 강화해야 한다. 第三は지속적인 모니터링이다. OAuth/OIDC 관련 로그를 지속적으로 모니터링하고, 이상 징후를 즉각 탐지하여 대응해야 한다.第四는기술 발전에 대한 대응이다. 새로운 공격 기법과 프로토콜 발전에 맞춰, OAuth/OIDC 구현을 지속적으로更新해야 한다.

📢 OAuth/OIDC는현대 인터넷의 인증/권한 표준이다. 인터넷 서비스들이 서로 연결되고,API Economy가 확장됨에 따라, OAuth/OIDC의 중요성은 더욱 커질 것이다. 그러나 OAuth/OIDC도万能자는 아니며, Implementation의 정확성과安全管理가 매우 중요하다. 개발자들은 OAuth/OIDC의原理를 깊이 이해하고, 보안 Best Practice를 철저히 준수하여,安全하고 상호운용 가능한 시스템을構築해야 한다. OAuth/OIDC의 올바른 활용은, 인터넷 사용자에게 편리하고 안전한 디지털 경험을 제공하는基礎이 된다.

## 참고
- OAuth: Open Authorization (열린 인증)
- OIDC: OpenID Connect (오픈아이디 커넥트)
- RFC: Request for Comments (인터넷 표준 문서)
- SPA: Single Page Application (싱글 페이지 애플리케이션)
- JWT: JSON Web Token (제이슨 웹 토큰)
- Bearer Token: حضور 토큰 (토큰 보유자)
- PKCE: Proof Key for Code Exchange (코드 교환용 검증 키)
- CSRF: Cross-Site Request Forgery (크로스사이트 요청 위조)
- HTTPS: Hypertext Transfer Protocol Secure (보안超文本 전송 프로토콜)
- SSO: Single Sign-On (단일 로그인)
- API: Application Programming Interface (애플리케이션 프로그래밍 인터페이스)
- MFA: Multi-Factor Authentication (다단계 인증)
- FIDO2: Fast Identity Online 2 (빠른 온라인 인증 2)
- GNAP: Grant Negotiation and Authorization Protocol (권한 협상 및 부여 프로토콜)
- DID: Decentralized Identity (탈중앙화 신원)
- IETF: Internet Engineering Task Force (인터넷 工程タスク force)
