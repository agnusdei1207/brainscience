### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 거대한 모놀리식 시스템을 마이크로서비스로 전환할 때 발생하는 데이터 결합 문제를 해결하기 위해, 서비스별 독립 DB(Database per Service)를 구축하고 비즈니스 특성에 최적화된 다양한 저장소(Polyglot Persistence)를 활용하는 전략입니다.
> 2. **가치**: 노후화된 레거시 시스템을 일시에 교체하는 위험(Big-bang) 대신, 점진적으로 기능을 가로채어 전환하는 스트랭글러 피그(Strangler Fig) 패턴과 도메인 오염을 방지하는 부패 방지 계층(ACL)을 통해 비즈니스 연속성을 보장하며 성공적인 클라우드 전환을 실현합니다.
> 3. **융합**: 도메인 주도 설계(DDD)의 바운디드 컨텍스트(Bounded Context) 개념을 데이터 계층까지 확장하며, 데이터 동기화를 위한 CDC(Change Data Capture) 및 API 게이트웨이 라우팅 기술과 긴밀하게 결합됩니다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **Database per Service**: 각 마이크로서비스가 자신의 전용 데이터베이스를 소유하며, 타 서비스는 오직 API를 통해서만 데이터에 접근할 수 있게 하는 데이터 격리 원칙입니다.
  - **폴리글랏 퍼시스턴스 (Polyglot Persistence)**: 하나의 시스템에서 RDBMS뿐만 아니라 NoSQL, 그래프 DB, 캐시 등 데이터의 특성과 목적에 맞는 최적의 저장소를 혼합하여 사용하는 방식입니다.

- **💡 비유**: MSA 데이터 전략은 **"개인 금고 운영"**과 같습니다. 예전에는 마을 공용 창고(Shared DB)에 모든 물건을 다 쑤셔 넣었지만, 이제는 집집마다 개인 금고(Database per Service)를 두고, 현금은 지갑에, 귀금속은 금고에, 책은 책꽂이(Polyglot)에 나누어 보관하는 것과 같습니다.

- **등장 배경 및 문제의식**:
  1. **데이터 강결합의 저주**: 공유 데이터베이스 구조에서는 테이블 스키마 하나만 바꿔도 연관된 모든 서비스가 영향을 받습니다. 이는 독립적인 배포와 확장을 핵심으로 하는 MSA의 근간을 흔듭니다.
  2. **레거시 전환의 공포**: 수십 년간 운영된 거대 시스템(Monolithic)을 한 번에 바꾸는 것은 자살 행위에 가깝습니다. 시스템을 멈추지 않고 '살아있는 상태'에서 심장을 하나씩 교체하는 정교한 마이그레이션 기법이 필요해졌습니다.

모놀리식의 공유 DB 구조와 MSA의 독립 DB 구조의 차이를 시각화하면 다음과 같습니다.

```text
┌───────────────────────────────────────────────────────────┐
│           Shared DB vs Database per Service 비교           │
├───────────────────────────────────────────────────────────┤
│                                                           │
│ [ Monolithic (Shared DB) ]       [ Microservices (DB per Svc) ]│
│  ┌───────┐  ┌───────┐            ┌───────┐      ┌───────┐     │
│  │ Svc A │  │ Svc B │            │ Svc A │      │ Svc B │     │
│  └───────┘  └───────┘            └───────┘      └───────┘     │
│      │          │                    │              │         │
│      ▼          ▼                    ▼              ▼         │
│  ┌───────────────────┐          ┌─────────┐    ┌─────────┐    │
│  │    One Giant DB   │          │ DB for A│    │ DB for B│    │
│  │ (Tables Joined)   │          └─────────┘    └─────────┘    │
│  └───────────────────┘             (Isolated & Independent)   │
│   ⚠ Schema Change Risk           ✅ Loose Coupling             │
└───────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 공유 DB 구조에서는 물리적/논리적 격리가 없어 서비스 간의 간섭이 심하고 특정 서비스의 부하가 전체 DB 성능을 떨어뜨립니다. 반면 Database per Service는 각 서비스가 독자적인 스키마와 저장소 타입을 가집니다. 이는 장애 격리(Fault Isolation)와 서비스 팀의 자율성을 극대화하지만, 서비스 간 데이터 조인이나 트랜잭션 관리가 복잡해지는 트레이드오프가 발생합니다.

- **📢 섹션 요약 비유**: "모든 직원이 하나의 커다란 통장을 같이 쓰다가(Shared DB), 부서별로 각자의 예산 주머니를 차고 필요할 때만 결재(API)를 받는 체계로 바뀐 것입니다."

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 레거시 전환의 정석: 스트랭글러 피그 (Strangler Fig) 패턴
기존 시스템 주위에 새로운 서비스를 구축하고, 기능을 하나씩 이관하면서 서서히 기존 시스템을 질식(Strangle)시켜 대체하는 전략입니다.

```text
 (Phase 1: Co-existence)           (Phase 2: Transition)            (Phase 3: Complete)
   ┌───────────────┐                 ┌───────────────┐                ┌───────────────┐
   │ [Proxy / GW]  │                 │ [Proxy / GW]  │                │ [Proxy / GW]  │
   └───────┬───────┘                 └────┬─────┬────┘                └───────┬───────┘
           │                              │     │                             │
    ┌──────▼──────┐               ┌──────▼──┐ ┌─▼───────┐             ┌──────▼──────┐
    │  Legacy App │               │ Legacy  │ │ New MSA │             │   New MSA   │
    └─────────────┘               └─────────┘ └─────────┘             └─────────────┘
                                    (Old API)   (New API)
```

**[다이어그램 해설]**
1. **Facade 배치**: 레거시 전면에 API 게이트웨이나 프록시를 배치합니다.
2. **신규 기능 구현**: 새로운 기능이나 분리할 기능을 MSA로 구현합니다.
3. **라우팅 전환**: 프록시가 특정 요청을 신규 MSA로 보냅니다. 외부 사용자는 내부가 바뀌는지 알 수 없습니다.
4. **점진적 질식**: 모든 기능이 이관되면 레거시 시스템을 종료합니다.

---

#### 2. 모델 오염 방지: 부패 방지 계층 (Anti-Corruption Layer, ACL)
새로운 시스템이 구형 시스템의 낡은 데이터 모델이나 복잡한 용어에 오염되지 않도록 중간에서 변환해 주는 어댑터 계층입니다.

```text
 [ New System Domain ] <──(ACL/Adapter)──> [ Legacy System Domain ]
   (Clean Model)             (Translate)         (Messy Model)
```

- **기능**: 레거시의 복잡한 응답을 신규 시스템이 이해하기 쉬운 깔끔한 DTO로 변환합니다.
- **가치**: 신규 시스템 개발자가 레거시의 고약한 비즈니스 로직을 공부할 필요 없이 깨끗한 인터페이스로 작업할 수 있게 보호막 역할을 합니다.

---

#### 3. 폴리글랏 퍼시스턴스 (Polyglot Persistence) 구성 예시
데이터의 특성에 따라 최적의 엔진을 선택합니다.

| 데이터 성격 | 최적 저장소 타입 | 대표 기술 | 비유 |
|:---:|:---|:---|:---|
| **복잡한 관계, 트랜잭션** | RDBMS | PostgreSQL, MySQL | 꼼꼼한 가계부 |
| **대용량 로그, 비정형** | Document NoSQL | MongoDB, DynamoDB | 자유로운 메모장 |
| **빠른 읽기, 캐시** | Key-Value Store | Redis, Memcached | 포스트잇 |
| **네트워크, 추천, 인맥** | Graph DB | Neo4j | 거미줄 지도 |
| **전문 검색, 분석** | Search Engine | Elasticsearch | 색인 잘 된 사전 |

**📢 섹션 요약 비유**: "낡은 건물을 한 번에 허물지 않고, 살면서 방 하나씩 리모델링하는 것(Strangler)과, 거실에는 소파를 주방에는 식탁을 놓듯 장소에 맞는 가구(Polyglot)를 배치하는 것입니다."

### Ⅲ. 융합 비교 및 다각도 분석

#### 1. 마이그레이션 전략 비교: 빅뱅 vs 점진적 전환

| 비교 항목 | 빅뱅 (Big-bang) 전환 | 스트랭글러 피그 (점진적) |
|:---:|:---|:---|
| **위험도** | 매우 높음 (실패 시 전면 장애) | 낮음 (부분적 전환 및 빠른 롤백) |
| **비용** | 단기 집중 투입 | 장기적 소요 (병행 운영 비용 발생) |
| **시스템 복잡도** | 단순 (신규 시스템만 운영) | 높음 (신/구 시스템 간 데이터 동기화 필요) |
| **적합한 상황** | 시스템 규모가 작거나 모델이 완전히 바뀔 때 | 대규모 엔터프라이즈 레거시 전환 |

#### 2. 과목 융합 관점: 데이터 무결성 및 인프라
- **데이터 엔지니어링 (CDC)**: 점진적 전환 시 레거시 DB와 MSA DB 간의 데이터 동기화가 핵심입니다. **Debezium** 같은 CDC 도구를 활용해 실시간으로 데이터를 복제해야 합니다.
- **클라우드 아키텍처**: 독립 DB 구조는 클라우드의 **Managed DB (RDS, Aurora)** 서비스를 활용해 운영 공수를 줄이는 전략과 시너지가 큽니다.
- **보안**: 폴리글랏 환경에서는 각 DB마다 보안 설정과 접근 제어 정책이 달라지므로, 통합 자격 증명 관리(IAM)가 더욱 중요해집니다.

**📢 섹션 요약 비유**: "수술 중에 환자의 심장을 멈추고 이식하느냐(빅뱅), 인공심폐기를 돌리며 부분적으로 혈관을 연결하느냐(점진적)의 차이입니다."

### Ⅳ. 실무 적용 및 기술사적 판단

#### 1. 실무 시나리오: 분산된 데이터 조인 해결 전략
Database per Service 도입 후, '주문' 정보와 '사용자' 정보를 한 화면에 보여줘야 하는데 DB가 달라 조인이 불가능한 상황입니다.

- **전략 A (API Composition)**: API 게이트웨이나 호출 서비스가 두 API를 각각 호출하여 메모리에서 합칩니다. (단순하지만 네트워크 오버헤드 발생)
- **전략 B (CQRS/Materialized View)**: 주문 시 사용자 정보를 포함한 '조회 전용 테이블'을 별도의 Read DB에 미리 만들어 둡니다. (성능은 최고이나 데이터 일관성 지연 감수)
- **전략 C (Data Replication)**: 주문 DB에 최소한의 사용자 정보(이름 등)를 복제하여 보관합니다.

#### 2. 도입 체크리스트
- **격리 수준**: 서비스 A가 서비스 B의 DB에 직접 SQL을 날리는 'Backdoor' 접근이 완전히 차단되었는가?
- **변환 비용**: ACL 계층 도입으로 인한 지연(Latency)이 비즈니스 허용 범위 내인가?
- **운영 복잡도**: 관리해야 할 DB 인스턴스가 늘어남에 따른 백업, 모니터링 자동화가 준비되었는가?

#### 3. 안티패턴: "Shared DB를 이름만 MSA로 부르기"
서비스는 쪼갰는데 DB는 여전히 하나를 공유하며 테이블 조인을 남발한다면, 그것은 '분산된 모놀리식(Distributed Monolithic)'일 뿐입니다. 장애가 전파되고 독립 배포가 불가능한 최악의 결과가 나옵니다.

**📢 섹션 요약 비유**: "독립 선언을 했지만 여전히 부모님 카드를 쓰고 있다면(Shared DB), 진정한 독립(MSA)이 아닌 것과 같습니다."

### Ⅴ. 기대효과 및 결론

#### 1. 도입의 정량/정성 기대효과

| 구분 | 기대 효과 | 상세 설명 |
|:---:|:---|:---|
| **민첩성 향상** | 배포 주기 50% 이상 단축 | DB 의존성 제거로 서비스별 독립적 배포 가능 |
| **유연성 확보** | 최적의 기술 선택 | 데이터 특성에 맞는 최신 기술(NoSQL 등) 즉시 도입 |
| **안정성 강화** | 장애 격리(Bulkhead) | 특정 DB 장애가 전체 시스템 중단으로 번지지 않음 |
| **전환 성공률** | 레거시 마이그레이션 안전 보장 | 실패 시 프록시 라우팅만 되돌리면 즉시 원복 가능 |

#### 2. 결론: "데이터 독립이 진정한 자유를 만든다"
MSA의 본질은 코드가 아니라 데이터의 분리에 있습니다. Database per Service와 폴리글랏 퍼시스턴스는 단순한 유행이 아니라, 거대해진 엔터프라이즈 시스템이 복잡도에 짓눌리지 않고 계속 진화하기 위한 생존 전략입니다. 기술사는 레거시의 무거움을 인정하되, 스트랭글러 피그와 부패 방지 계층이라는 정교한 도구를 사용하여, 구시대의 유산을 미래의 혁신으로 연결하는 '아키텍처의 가교' 역할을 수행해야 합니다.

**📢 섹션 요약 비유**: "하나의 거대한 바위(Monolith)를 깎아 조각상을 만드는 예술에서, 수많은 레고 블록(MSA)을 조합하여 언제든 모양을 바꿀 수 있는 창의적 건축으로의 진화입니다."

### 📌 관련 개념 맵 (Knowledge Graph)
- **Database per Service**: 서비스 간 데이터 결합을 끊는 핵심 원칙
- **Strangler Fig Pattern**: 점진적 레거시 전환의 표준 패턴
- **Anti-Corruption Layer**: 모델 오염 방지를 위한 변환 어댑터
- **Change Data Capture (CDC)**: 실시간 데이터 동기화의 핵심 기술
- **Polyglot Persistence**: 데이터 목적에 따른 저장소 다변화 전략

### 👶 어린이를 위한 3줄 비유 설명
1. **데이터 전략이란**: 전교생이 하나의 커다란 필통을 같이 쓰는 게 아니라, 친구마다 자기 필통을 따로 가지고 자기 물건을 챙기는 거예요.
2. **레거시 전환이란**: 낡은 자전거를 한 번에 버리는 게 아니라, 바퀴부터 하나씩 새것으로 바꾸면서 계속 타고 다니는 멋진 방법이에요.
3. **좋은 점**: 내 필통(DB)은 내 마음대로 꾸밀 수 있고, 바퀴 하나를 바꿔도 자전거를 멈추지 않고 계속 씽씽 달릴 수 있답니다.
