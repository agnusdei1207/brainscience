# 무상태성 (Statelessness) - REST의 핵심

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: REST 아키텍처의 핵심 제약 조건인 무상태성(Statelessness)은 서버가 클라이언트의 세션 상태를 저장하지 않고, 각 요청이 모든 필요한 문맥(Context)을 자체적으로 포함하여 독립적으로 처리되도록 설계하는 원칙
> 2. **가치**: 서버 측 상태 관리 오버헤드 제거로 수평 확장이 극도로 단순해지며(요청을 어떤 서버 인스턴스로든 라우팅 가능), 장애 복구가 용이하고(서버 장애 시 다른 인스턴스가 즉시 대체), 로드 밸런싱 효율이 극대화됨
> 3. **융합**: MSA(Microservices Architecture)의 기본 전제 조건으로, 컨테이너 오케스트레이션(Kubernetes, Docker Swarm)의 무상태 파드(Pod) 설계와 자연스럽게 정렬되며, 서버리스(Serverless) 아키텍처의 근간이 됨

## Ⅰ. 개요 (Context & Background)

### 개념 정의

무상태성(Statelessness)은 REST 아키텍처의 핵심 제약 조건 중 하나로, 서버가 클라이언트의 상태(State)나 세션(Session) 정보를 저장하지 않는다는 원칙이다. 각 HTTP 요청은 클라이언트-서버 간의 이전 상호작용 문맥에 의존하지 않고, 요청 자체에 모든 필요한 정보를 포함하여 독립적으로 처리되어야 한다.

이 원칙의 핵심은 "서버 측 상태 유지(Stateful Server)를 클라이언트 측 상태 유지(Stateless Client)로 대체"하는 것이다. 서버는 세션 저장소, 메모리 내 캐시, 연결 상태와 같은 상태 정보를 유지하지 않으며, 각 요청을 처음 보는 것처럼 처리한다.

**기술적 정의**

- **Stateless Server**: 서버가 클라이언트 요청 간의 상태를 저장하지 않음
- **Self-contained Request**: 각 요청이 모든 필요한 문맥을 포함
- **Independent Processing**: 요청 처리 순서에 의존하지 않음
- **Scalable Horizontal**: 요청을 어떤 서버 인스턴스로든 라우팅 가능

### 등장 배경

무상태성 원칙은 웹(World Wide Web)의 근본적 설계 철학에서 유래한다. Tim Berners-Lee가 웋을 설계할 때, "각 HTTP 요청이 독립적이어야 한다"는 원칙을 정립했으며, 이는 웹이 전 세계 규모로 확장 가능한 핵심 이유가 되었다.

Roy Fielding이 2000년 박사학위 논문에서 REST 아키텍처를 정립하면서, 무상태성을 5가지 핵심 제약 조건 중 하나로 명시했다. Fielding은 무상태성이 다음과 같은 이점을 제공한다고 주장했다:

1. **가시성(Visibility)**: 각 요청이 독립적이므로 모니터링과 디버깅이 용이
2. **신뢰성(Reliability)**: 서버 장애 시 다른 인스턴스가 즉시 대체 가능
3. **확장성(Scalability)**: 요청을 어떤 서버로든 라우팅 가능하여 수평 확장 단순화

2000년대 중반까지는 세션 기반의 Stateful Web Application(JSP Session, ASP.NET Session)이 주류였으나, 클라우드 컴퓨팅과 MSA의 등장과 함께 무상태성이 현대적 엔터프라이즈 아키텍처의 필수 조건으로 재조명되었다.

### 기술적 필요성

```text
┌─────────────────────────────────────────────────────────────────────────────┐

│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │   │
│  │  클라이언트 A                                                        │   │
│  │  ┌───────────────────────────────────────────────────────────────┐  │   │

│  │  │   (서버 2: 토큰 검증 → 요청 처리, 상태 저장 안 함)            │  │   │
```

이 Stateful vs Stateless 비교의 핵심은 무상태성이 수평 확장을 극도로 단순화한다는 점이다. Stateful 서버에서는 동일한 클라이언트 요청을 항상 같은 서버 인스턴스로 라우팅하는 Sticky Session이 필요하지만, Stateless 서버에서는 요청을 어떤 인스턴스로든 라우팅 가능하다. 실무 데이터에 따르면, 무상태 아키텍처는 로드 밸런싱 효율을 40~60% 향상시키며, 서버 장애 복구 시간을 90% 이상 단축한다.

📢 **섹션 요약 비유**: 무상태성은 자판기와 같습니다. Stateful 서버는 점원이 있는 카운터 서비스로, 같은 점원에게 계속 서비스를 받아야 하며 점원이 자리를 비우면 서비스가 중단됩니다. 반면 Stateless 서버는 자판기처럼 어떤 기계(서버 인스턴스)를 이용해도 동일한 서비스를 받을 수 있으며, 한 기계가 고장 나도 다른 기계를 이용하면 됩니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 무상태성 핵심 원칙

Roy Fielding이 정의한 REST 무상태성의 핵심 원칙을 분석한다.

| 원칙 | 설명 | 구현 방법 | 위반 사례 |
|:-----|:-----|:---------|:---------|
| **독립적 요청** | 각 요청이 이전 요청에 의존하지 않음 | 요청에 모든 문맥 포함 | 세션 ID로 이전 상태 조회 |
| **서버 측 상태 부재** | 서버가 클라이언트 상태를 저장하지 않음 | 상태를 클라이언트나 저장소에 위임 | 서버 메모리에 세션 저장 |
| **자기 충족 요청** | 요청이 모든 필요한 정보를 포함 | Authorization 헤더, 쿼리 파라미터 | 상태 서버에 상태 조회 |
| **순서 독립적** | 요청 처리 순서에 의존하지 않음 | 각 요청이 독립적 처리 | 요청 순서 검증 로직 |

### 무상태성 구현 패턴

**① JWT (JSON Web Token) 기반 인증**

무상태성 구현의 가장 대표적인 패턴은 JWT 기반 인증이다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    JWT-Based Stateless Authentication                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [로그인 요청]                                                              │
│  POST /auth/login                                                           │
│  {username: "userA", password: "***"}                                       │
│       │                                                                     │
│       ▼                                                                     │
│  서버: 자격 증명                                                           │
│       │                                                                     │
│       ▼                                                                     │
│  JWT 토큰 발급 (상태 저장 없음)                                             │
│  {                                                                        │
│    "sub": "userA",                                                         │
│    "role": "admin",                                                        │
│    "exp": 1234567890,                                                      │
│    "iat": 1234560000                                                       │
│  } → 서명(서버 비밀키)                                                      │
│       │                                                                     │
│       ▼                                                                     │
│  클라이언트에게 토큰 반환                                                  │
│       │                                                                     │
│       ▼                                                                     │
│  이후 요청: Authorization 헤더에 토큰 포함                                 │
│  GET /api/orders                                                            │
│  Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...          │
│       │                                                                     │
│       ▼                                                                     │
│  서버: 토큰 서명 검증 → 클레임 추출 → 요청 처리                            │
│  (서버는 상태를 저장하지 않고, 토큰 자체가 모든 정보 포함)                   │
│                                                                             │
│  [특징]                                                                    │
│  • 서버는 상태를 저장하지 않음                                              │
│  • 토큰이 모든 사용자 정보 포함 (self-contained)                            │
│  • 어떤 서버 인스턴스든 토큰 검증 가능                                      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**② 쿠키 기반 상태 저장(위배 패턴)**

전통적인 세션 기반 인증은 무상태성 원칙을 위배한다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                 Session-Based Authentication (Violates Statelessness)       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [로그인 요청]                                                              │
│  POST /auth/login                                                           │
│  {username: "userA", password: "***"}                                       │
│       │                                                                     │
│       ▼                                                                     │
│  서버: 자격 증명 → 세션 생성 → 상태 저장소에 저장                          │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  세션 저장소 (Redis, Database)                                       │   │
│  │  {                                                                  │   │
│  │    "session_id_abc": {                                              │   │
│  │      "user": "userA",                                               │   │
│  │      "role": "admin",                                               │   │
│  │      "cart": [...],                                                 │   │
│  │      "login_time": "2026-03-18T10:00:00Z"                          │   │
│  │    }                                                                │   │
│  │  }                                                                  │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│       │                                                                     │
│       ▼                                                                     │
│  클라이언트에게 쿠키 발급                                                    │
│  Set-Cookie: session_id=abc; HttpOnly; Secure                               │
│       │                                                                     │
│       ▼                                                                     │
│  이후 요청: 쿠키에 세션 ID 포함                                            │
│  GET /api/orders                                                            │
│  Cookie: session_id=abc                                                     │
│       │                                                                     │
│       ▼                                                                     │
│  서버: 세션 저장소에서 session_id=abc 조회 → 상태 확인 → 요청 처리          │
│  (각 요청마다 상태 저장소 조회 필요)                                         │
│                                                                             │
│  [무상태성 위배 요소]                                                        │
│  • 서버가 클라이언트 상태를 저장                                            │
│  • 요청이 상태 저장소 의존                                                 │
│  • 상태 저장소 장애 시 서비스 중단                                         │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

```text
│  │ [상태 저장 ❌]│ │ [상태 저장 ❌]│ │ [상태 저장 ❌]│ │ [상태 저장 ❌]│ │    │
│  │              │ │              │ │              │ │              │    │
│  └──────────────┘ └──────────────┘ └──────────────┘ └──────────────┘    │
│          │              │              │              │                   │
│          ▼              ▼              ▼              ▼                   │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                      Backend Services                               │   │
│  │  • Database (공유 데이터베이스)                                     │   │

│  └─────────────────────────────────────────────────────────────────────┘   │

│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

이 무상태 아키텍처 도식의 핵심은 로드 밸런서가 세션을 고려하지 않고 요청을 어떤 서버 인스턴스로든 라우팅할 수 있다는 점이다. 각 서버 인스턴스는 상태를 저장하지 않으므로, 요청이 서버 1에서 처리되다가 서버 2로 전송되더라도 문제가 없다. 따라서 서버 장애 시 다른 인스턴스가 즉시 대체 가능하며, 수평 확장은 서버 인스턴스를 추가하는 것만으로 완료된다.

### HTTP 계층에서의 무상태성 구현

HTTP 프로토콜 자체가 무상태성을 지원하는 여러 메커니즘을 제공한다.

**① Authorization 헤더**

```http
GET /api/orders HTTP/1.1
Host: api.example.com
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

**② 쿼리 파라미터**

```http
GET /api/orders?page=2&limit=20&sort=date HTTP/1.1
Host: api.example.com
```

**③ HTTP 캐시 제어 헤더**

```http
GET /api/products/123 HTTP/1.1
Host: api.example.com
Cache-Control: max-age=3600
ETag: "abc123"
```

요청이 ETag와 함께 수행되면, 서버는 리소스가 변경되지 않았을 때 304 Not Modified를 반환하여 대역폭을 절약한다.

📢 **섹션 요약 비유**: 무상태성은 택배 시스템에서 운송장 번호로만 물건을 추적하는 것과 같습니다. Stateful 서버는 택배 기사가 "이전에 OO동에 배달한 적이 있어?"라고 기억해야 하지만, Stateless 서버는 운송장 번호(토큰)만 있으면 어떤 택배 기사(서버 인스턴스)든 배달 가능합니다. 기사가 교체되어도 운송장 번호만 있으면 문제없습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 무상태성 vs 상태유지성 비교

| 비교 항목 | Stateless (무상태) | Stateful (상태유지) | 차이점 |
|:---------|:-------------------|:-------------------|:------|
| **서버 상태 저장** | 없음 | 있음(세션, 메모리) | 복잡도 차이 |
| **수평 확장 용이성** | 극도로 용이 | 어려움(세션 복제 필요) | 10배 이상 차이 |
| **로드 밸런싱** | 자유로움 | Sticky Session 필요 | 라우팅 효율 40% 차이 |
| **장애 복구** | 즉시 가능 | 세션 소실 위험 | 가용성 99.9% vs 99.5% |
| **요청 처리 속도** | 빠름(상태 조회 없음) | 느림(상태 저장소 조회) | 레이턴시 20~30% 차이 |
| **클라이언트 복잡도** | 높음(토큰 관리) | 낮음(쿠키 자동) | 개발 부하 차이 |

### 과목 융합: 무상태성과 연관 기술 스택

**② Serverless와의 시너지**

AWS Lambda, Google Cloud Functions, Azure Functions와 같은 서버리스 플랫폼은 무상태성을 필수 조건으로 한다. 각 함수 호출은 독립적인 컨테이너/프로세스에서 실행되며, 이전 호출의 상태를 유지하지 않는다.

**③ CDN과의 시너지**

무상태성은 CDN 에지 캐싱을 극대화한다. 서버가 상태를 저장하지 않으므로, GET 요청은 어떤 CDN 에지 노드에서든 동일한 응답을 반환할 수 있다.

### 무상태성 도입 체크리스트

**서버 측**
- [ ] 세션 저장소(Redis, Database)를 사용하지 않는가
- [ ] 서버 메모리에 클라이언트 상태를 저장하지 않는가
- [ ] 각 요청이 독립적으로 처리 가능한가
- [ ] Sticky Session을 사용하지 않는가

**클라이언트 측**
- [ ] 상태를 클라이언트(Local Storage, Session Storage)에 저장하는가
- [ ] JWT 토큰 또는 API Key를 사용하는가
- [ ] 장바구니, UI 상태를 클라이언트에 저장하는가

**인프라 측**
- [ ] 로드 밸런서가 어떤 서버로든 요청을 라우팅 가능한가
- [ ] 서버 장애 시 다른 인스턴스가 즉시 대체 가능한가
- [ ] 오토스케일링이 상태 저장소와 독립적으로 동작하는가

📢 **섹션 요약 비유**: 무상태성은 카드 결제 시스템과 같습니다. Stateful 서버는 가게 회원카드로 결제하는 것과 같아서, 항상 같은 가게에서만 카드를 사용해야 하며 가게 문이 닫으면 결제가 불가능합니다. 반면 Stateless 서버는 신용카드(JWT)로 결제하는 것과 같아서, 어떤 가게(서버 인스턴스)에서든 결제가 가능하며 가게가 문을 닫아도 다른 가게를 이용하면 됩니다.

```javascript
// ✅ 좋은 예: 명시적 상태 전달
app.post('/api/count', (req, res) => {
  const {currentCount} = req.body;  // 클라이언트가 상태 제공
  const newCount = currentCount + 1;
  res.json({count: newCount});
});
```

📢 **섹션 요약 비유**: 무상태성은 휴대폰과 문자 메시지와 같습니다. Stateful 서버는 전화 통화(상태 유지 연결)와 같아서, 통화가 끊기면 대화 내용이 소실되고 상대방이 부재 중이면 계속할 수 없습니다. 반면 Stateless 서버는 문자 메시지(독립적 메시지)와 같아서, 상대방이 부재 중이도 메시지를 보낼 수 있으며 나중에 확인이 가능합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과

무상태성 도입 시 기대되는 주요 효과를 정량적으로 분석한다.

| 평가 항목 | Stateful | Stateless | 개선 효과 |
|:--------|:---------|:----------|:--------:|
| **수평 확장 복잡도** | 높음(세션 복제) | 낮음(인스턴스 추가) | -80% |
| **장애 복구 시간** | 분 단위 | 초 단위 | -95% |
| **로드 밸런싱 효율** | 제한적(Sticky) | 최적(자유 라우팅) | +40% |
| **서버 요청 처리 속도** | 느림(상태 조회) | 빠름(검증만) | +25% |
| **가용성(SLA)** | 99.5% | 99.9%+ | +0.4% |
| **인프라 비용** | 높음(상태 저장소) | 낮음(무상태) | -30% |

### 기술적 효과

**① 확장성 향상**

무상태 서버는 로드 밸런서가 요청을 어떤 인스턴스로든 자유롭게 라우팅할 수 있으므로, 수평 확장이 극도로 단순해진다. Kubernetes 환경에서는 HPA(Horizontal Pod Autoscaler)가 트래픽 증가에 따라 자동으로 Pod 수를 조정 가능하다.

**② 가용성 향상**

서버 인스턴스 장애 시 다른 인스턴스가 즉시 대체 가능하므로, 가용성이 99.9% 이상으로 향상된다. Stateful 서버에서는 상태 저장소 장애 시 전체 서비스 중단 위험이 있다.

**③ 운용 효율 개선**

롤링 업데이트, 카나리 배포, 블루/그린 배포와 같은 현대적 배포 전략이 무상태성에서 자연스럽게 구현된다.

### 향후 전망

무상태성은 2025-2030년에도 클라우드 네이티브, 서버리스, 엣지 컴퓨팅 환경에서 더욱 중요해질 것이다. 특히:
- **서버리스 플랫폼**: AWS Lambda, Google Cloud Functions, Azure Functions는 무상태성을 필수 조건으로 요구
- **엣지 컴퓨팅**: Cloudflare Workers, AWS Lambda@Edge는 엣지 로케이션에서 무상태 코드 실행
- **WebAssembly**: 브라우저와 엣지에서 실행되는 Wasm 모듈은 본질적으로 무상태

에서 무상태성이 표준이 될 것이다.

### 참고 표준

- **RFC 9110 - HTTP Semantics**: HTTP 무상태성 정의
- **RFC 7519 - JSON Web Token (JWT)**: 무상태 인증 표준
- **RFC 6749 - OAuth 2.0**: 권한 부여 프레임워크
- **RFC 6265 - HTTP State Management Mechanism (Cookies)**: 쿠키 기반 상태 관리(무상태성 위배)

📢 **섹션 요약 비유**: 무상태성은 전자 기기와 배터리의 분리와 같습니다. Stateful 서버는 전화기처럼 전선이 연결되어야 작동하지만, Stateless 서버는 휴대폰처럼 배터리가 없어도 충전기(서버 인스턴스)에만 연결하면 작동합니다. 어떤 충전기를 사용해도 상관없으므로, 배터리 수명과 충전기 위치에 구애받지 않습니다.

## 👶 어린이를 위한 3줄 비유 설명

**① 학교 수업 비유**: 무상태성은 대학 강의와 같습니다. Stateful 서버는 초등학교 수업으로, 항상 같은 담임 교사가 같은 반을 가르치고 교사가 결석하면 대체하기 어렵습니다. 반면 Stateless 서버는 대학 강의로, 어떤 교수가 강의하더라도 강의 내용은 동일하며 한 교수가 결석해도 다른 교수가 대체 강의가 가능합니다.

**② 식당 주문 비유**: 무상태성은 식당에서 주문하는 것과 같습니다. Stateful 서버는 단골 손님을 위한 주문 기록을 기억하는 식당으로, 항상 같은 점원이 주문을 받아야 기억할 수 있습니다. 반면 Stateless 서버는 주문서에 모든 정보를 적어서 제출하는 것과 같아서, 어떤 점원에게 주문하더라도 동일한 음식을 받을 수 있습니다.

**③ 라디오 방송 비유**: 무상태성은 라디오 방송과 같습니다. Stateful 서버는 TV 시리즈와 같아서 이전 에피소드를 봐야 내용을 이해할 수 있습니다. 반면 Stateless 서버는 라디오 방송처럼 언제 튀어도 들어도 내용을 이해할 수 있으며, 이전 방송을 들었는지와 상관없이 현재 방송만으로 정보를 전달받을 수 있습니다.
