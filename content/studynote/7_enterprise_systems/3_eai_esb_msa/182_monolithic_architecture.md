# 모놀리식 아키텍처 (Monolithic Architecture)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프론트엔드(UI), 비즈니스 로직, 데이터 접근 계층이 단일 코드베이스와 단일 배포 단위로 강결합된 전통적 애플리케이션 아키텍처로, 모든 기능이 하나의 프로세스로 실행되며 단일 데이터베이스를 공유
> 2. **가치**: 초기 개발 속도가 빠르고 테스트가 간단하며, 모놀리식 경계(모듈) 내에서는 강한 형식 안전성과 IDE 지원을 받을 수 있으나, 코드베이스 크기가 증가함에 따라 빌드 시간, 배포 복잡도, 확장성 제약이 심각해짐
> 3. **융합**: MSA(Microservices Architecture) 등장 배경이 되는 레거시 아키텍처로, 소규모 팀/프로덕트에서는 여전히 유효한 선택이며, 잘 설계된 모놀리스(Modular Monolith)는 MSA 이행 전 중간 단계로 적합

## Ⅰ. 개요 (Context & Background)

### 개념 정의

모놀리식 아키텍처(Monolithic Architecture)는 애플리케이션의 모든 기능과 계층이 단일 코드베이스에 통합되어, 하나의 실행 단위(단일 WAR 파일, 단일 Executable JAR, 단일 Docker 컨테이너)로 배포되는 전통적 소프트웨어 아키텍처 패턴이다.

이 아키텍처의 핵심 특징은 **강한 결합(Tight Coupling)**과 **단일 배포 단위(Single Deployment Unit)**이다. 프론트엔드(UI 렌더링), 비즈니스 로직(서비스 계층), 데이터 접근(DAO/Repository)이 모두 하나의 프로세스 공간에서 실행되며, 일반적으로 단일 데이터베이스 스키마를 공유한다.

**구조적 특징**

- **단일 코드베이스**: 모든 모듈/기능이 하나의 VCS(Version Control System) 저장소에 존재
- **단일 빌드 아티팩트**: 하나의 배포 가능 파일(WAR, JAR, Docker Image)
- **공유 메모리 공간**: 모든 컴포넌트가 동일한 프로세스 메모리 공유
- **단일 데이터베이스**: 일반적으로 단일 RDBMS 인스턴스에 모든 테이블 존재
- **프로세스 내 통신**: 컴포넌트 간 통신이 메서드 호출(Function Call)로 수행

### 등장 배경

모놀리식 아키텍처는 소프트웨어 개발의 초기부터 자연스럽게 등장한 "기본값" 아키텍처 패턴이다.

**① 초기 소프트웨어 개발의 단순성**

1990년대~2000년대 초반까지 소프트웨어는 주로 단일 서버에서 실행되었으며, 애플리케이션 규모도 작았다. 따라서 모든 기능을 하나의 프로젝트로 개발하는 것이 자연스러운 접근이었다.

**② 계층형 아키텍처(Layered Architecture)의 등장**

2000년대 중반 J2EE(Java 2 Platform, Enterprise Edition)의 등장과 함께 3계층 아키텍처(프레젠테이션 계층, 비즈니스 계층, 데이터 접근 계층)가 표준화되었다. 하지만 이 계층들은 논리적 분리일 뿐, 물리적으로는 단일 애플리케이션에 통합되었다.

**③ 전사적 애플리케이션(Enterprise Application)의 복잡도 증가**

2000년대 후반부터 엔터프라이즈 애플리케이션의 규모와 복잡도가 급격히 증가하면서, 모놀리식 아키텍처의 한계가 드러나기 시작했다. 수백만 명의 사용자를 처리하는 대규모 시스템에서는 단일 모놀리스로는 확장성 한계가 명확했다.

📢 **섹션 요약 비유**: 모놀리식 아키텍처는 주택 단독주택과 같습니다. 소가족이 거주할 때는 편리하고 유지보수가 간단하지만, 대가족(수십 명)이 거주하기 시작하면 주방이 혼잡해지고, 욕실 사용에 대기가 발생하며, 누군가 아프면 전체 가족이 영향을 받습니다. 이때 아파트(MSA)나 빌라(MSA)처럼 독립된 공간으로 분리하는 것을 고려하게 됩니다.

### 모놀리식 장점

모놀리식 아키텍처는 특정 상황에서 여전히 유효한 장점을 갖는다.

**① 초기 개발 속도**

- 단일 프로젝트로 시작하므로 설정 오버헤드가 적음
- 모든 기능에 즉시 접근 가능하여 빠른 프로토타이핑 가능
- 마이크로서비스 간 통신 복잡도가 없음

**② 테스트 간결성**

- 통합 테스트가 단일 프로세스에서 실행되므로 간단
- 끝에서 끝(End-to-End) 테스트 설정이 용이
- 디버깅이 프로세스 내에서 모든 코드를 추적 가능

**③ 배포 단순성**

- 단일 WAR/JAR/Docker 이미지로 배포
- 환경 설정이 단일 파일에 집중
- 롤백이 단순(이전 버전으로 교체만)

**④ IDE 강력 지원**

- 단일 프로젝트에서 모든 코드에 즉시 접근
- 리팩토링 도구가 전체 코드베이스에서 작동
- 코드 네비게이션이 용이

### 모놀리식 단점

모놀리식 아키텍처의 핵심 단점은 규모가 증가함에 따라 기하급수적으로 악화된다.

**① 확장성 제약**


**② 빌드/배포 복잡도**

- 코드베이스 크기가 증가하면 빌드 시간이 기하급수적으로 증가
- 전체 애플리케이션을 재배포해야 하므로 배포 시간이 김
- 한 모듈의 버그가 전체 시스템 배포를 차단

**③ 기술 부채**

- 특정 프레임워크/라이브러리 버전에 모든 모듈이 의존
- 부분적 업그레이드가 불가능하며, 전체를 동시에 업그레이드해야 함
- 새로운 기술 도입이 전체 애플리케이션에 영향

**④ 장애 영향**

- 한 모듈의 메모리 누수가 전체 애플리케이션 장애로 이어짐
- 비즈니스 로직 버그가 전체 서비스 중단 유발

📢 **섹션 요약 비유**: 모놀리식 아키텍처는 대형 쇼핑몰 건물과 같습니다. 소규모일 때는 모든 부서(패션, 전자제품, 식품)가 한 건물에 있어서 고객 이동이 편하지만, 규모가 커지면 특정 층(전자제품)에만 혼잡이 발생해도 전체 건물을 확장해야 하며, 화재가 발생하면 모든 부서가 영향을 받습니다. 이때 전문점 매장(MSA)으로 분리하는 것을 고려하게 됩니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 모놀리식 vs MSA 비교

| 비교 항목 | 모놀리식 | MSA (Microservices) | 차이점 |
|:---------|:--------|:-------------------|:------|
| **서비스 수** | 단일 | 다중(10~1000+) | 독립적 배포 단위 |
| **데이터베이스** | 단일 공유 | 서비스별 독립 | DB per Service |
| **통신 방식** | 메서드 호출 | HTTP/RPC 메시지 | 네트워크 오버헤드 |
| **확장성** | 전체 확장 | 서비스별 확장 | 비용 효율 차이 |
| **배포** | 전체 재배포 | 서비스별 배포 | 배포 빈도 |
| **기술 자유도** | 통합 기술 스택 | 서비스별 다양성 | Polyglot 가능 |
| **장애 격리** | 전체 영향 | 서비스로 한정 | 가용성 차이 |
| **개발 초기 속도** | 빠름 | 느림(인프라 설정) | 시장 출시 속도 |

**② 모듈형 모놀리스 규칙**

1. **명시적 모듈 경계**: 각 모듈은 명확한 패키지/네임스페이스 경계를 가짐
2. **공개 인터페이스**: 모듈 간 통신은 공개 API를 통해서만 수행
3. **직접 참조 금지**: 모듈 A가 모듈 B의 내부 클래스를 직접 호출 불가
4. **의존성 방향**: 도메인 계층 의존성 방향 준수 (상위 계층이 하위 계층 의존)

### 과목 융합: 모놀리식과 연관 기술 스택

**① 클라우드 네이티브 전환**

모놀리식 애플리케이션을 클라우드 네이티브로 전환하는 첫 단계는 컨테이너화이다.

```text
[전통적 배포]
Physical Server → OS → JVM → Application
           ↓
    수주~수주 소요

[컨테이너화]
Docker Container → Application
           ↓
    분~초 단위 배포

[쿠버네티스]
Kubernetes Pods → Auto-scaling + Self-healing
           ↓
    수평 확장 자동화
```

**② 프로그레시브 전략**

모놀리식에서 MSA로 점진적 이동하는 실무 전략이다.

1. **모듈형 모놀리스로 리팩토링** (1~3개월)
2. **새로운 기능을 별도 서비스로 개발** (3~6개월)
3. **핵심 모듈을 MSA로 분리** (6~12개월)
4. **단계적 마이그레이션** (1~2년)

📢 **섹션 요약 비유**: 모놀리식 아키텍처는 한 건물에 모든 부서가 있는 회사와 같습니다. 소규모 스타트업에서는 효율적이지만, 대기업이 되면 각 부서를 별도 건물로 분리(MSA)하는 것을 고려하게 됩니다. 모듈형 모놀리스는 각 부서에 명확한 방(Partition)을 설치해두는 중간 단계로, 향후 분리를 용이하게 합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오별 모놀리식 선택

| 시나리오 | 모놀리식 적합성 | 판단 근거 |
|:--------|:---------------:|:---------|
| **스타트업 MVP** | ✅ 매우 적합 | 빠른 시장 출시, 불확실성 높음 |
| **소규모 팀(3~10명)** | ✅ 적합 | 운영 복잡도 최소화 |
| **단순 CRUD 앱** | ✅ 적합 | 복잡한 도메인 로직 없음 |
| **대규모 트래픽(100만+ DAU)** | ❌ 부적합 | 확장성 제약 심각 |
| **다양한 기술 스택 요구** | ❌ 부적합 | 기술 부채 심각 |
| **긴급 장애 복구 필요** | ❌ 부적합 | 장애 격리 어려움 |

### 모놀리식 개발 체크리스트

**구조적 분리**
- [ ] 명확한 계층 분리(프레젠테이션, 비즈니스, 데이터 접근)

- [ ] 공개 인터페이스 정의

- [ ] 상태 저장소(Redis, Database)를 애플리케이션 외부에 두기
- [ ] 세션 불필요한 API 설계(Stateless)
- [ ] 캐시 전략 수립

**배포 자동화**
- [ ] CI/CD 파이프라인 구축
- [ ] 컨테이너화(Docker)

**안티패턴 2: 데이터베이스 직접 접근**

```java
// ❌ 나쁜 예: 컨트롤러가 직접 DAO 접근
@RestController
public class UserController {
    @Autowired
    private UserRepository userRepository;

    // 모든 계층을 건너뛰어 데이터베이스에 직접 접근
    @GetMapping("/users/{id}")
    public User getUser(@PathVariable Long id) {
        return userRepository.findById(id).orElse(null);
    }
}

// ✅ 좋은 예: 서비스 계층 통해 접근
@RestController
public class UserController {
    @Autowired
    private UserService userService;  // 서비스 계층 경유

    @GetMapping("/users/{id}")
    public UserDTO getUser(@PathVariable Long id) {
        return userService.getUser(id);
    }
}
```

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과

모놀리식 아키텍처와 MSA의 비용/편익을 정량적으로 분석한다.

| 항목 | 모놀리식 | MSA | 차이점 |
|:-----|:--------|:---|:-----:|
| **초기 개발 속도** | 빠름 | 느림(인프라 설정) | 2~3배 차이 |
| **빌드 시간** | 10~30분(소규모), 1~2시간(대규모) | 5~10분(서비스별) | 대규모에서 10배 차이 |
| **배포 빈도** | 1주~1달 | 하루~1주 | 10~30배 차이 |
| **확장 비용** | 전체 확장(비효율) | 서비스별(효율적) | 40~60% 비용 절감 |
| **운영 복잡도** | 단일 서버 | 다중 서비스 + 오케스트레이션 | 3~5배 차이 |
| **장애 격리** | 전체 영향 | 서비스로 한정 | 가용성 99.5% → 99.9%+ |

### 기술적 효과

**① 빠른 시장 출시(Time-to-Market)**

모놀리식 아키텍처는 초기 스타트업이나 MVP(Minimum Viable Product) 개발에 가장 적합하다. 복잡한 분산 시스템 인프라 없이 빠르게 프로토타입하고 시장 피드백을 수집할 수 있다.

**② 간단한 운영**

단일 서버만 모니터링하고, 단일 로그를 분석하며, 단일 배포만 관리하므로 운영 오버헤드가 낮다. 소규모 팀에서는 중요한 고려 사항이다.

**③ 강력한 도구 지원**

단일 IDE에서 전체 코드베이스에 접근하고, 강력한 리팩토링 도구를 활용할 수 있어 개발자 생산성이 높다.

### 향후 전망

모놀리식 아키텍처는 특정 상황에서 여전히 유효한 선택이다.

**① 적합한 상황**

- 스타트업 및 초기 단계
- 소규모 팀(10명 이하)
- 단순한 도메인 로직
- 빠른 시장 출시 필요

**② 부적합한 상황**

- 대규모 트래픽(일일 100만+ DAU)
- 복잡한 도메인 로직
- 다양한 기술 스택 요구
- 독립적 확장 필요

**③ 하이브리드 접근**

많은 기업이 모듈형 모놀리스로 시작하여, 점진적으로 MSA로 이행하는 하이브리드 접근을 채택한다. 새로운 기능은 별도 마이크로서비스로 개발하며, 기존 모놀리스는 개선을 유지한다.

### 참고 표준

- **Clean Architecture**: 계층 분리 원칙
- **SOLID Principles**: 모듈 간 결합도 최소화

📢 **섹션 요약 비유**: 모놀리식 아키텍처는 단독주택과 같습니다. 1~2인 가족에게는 공간 효율이 좋고 유지보수가 간단하지만, 10인 이상 대가족이 되면 방(모듈)별로 분리해야 합니다. 모듈형 모놀리스는 욕실, 주방, 침실을 명확히 구분한 설계로, 향후 증축/분리를 용이하게 한 중간 단계라고 할 수 있습니다.
