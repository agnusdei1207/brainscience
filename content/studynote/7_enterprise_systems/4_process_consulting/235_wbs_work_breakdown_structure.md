# 범위 관리 WBS (Work Breakdown Structure) 100% Rule 규칙 통제

> **약어 (Abbreviation)**: WBS (Work Breakdown Structure)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: WBS (작업 분할 구조도)는 프로젝트가 만들어내야 할 최종 결과물(Deliverables)을 달성하기 위해 필요한 모든 작업을 나뭇가지(Tree)처럼 세분화하여, **'가장 작고 관리 가능한 단위(Work Package)'로 쪼개는 범위 관리의 핵심 도구**다.
> 2. **가치**: 전체 프로젝트의 **100% Rule (백퍼센트 규칙)**을 강제함으로써, 하위 작업들의 합이 상위 작업을 정확히 100% 만족시키게 하여 작업의 **누락(Missing)이나 불필요한 과업(Gold Plating)을 원천 차단**하고 일정과 예산 산정의 절대적 기준점을 제공한다.
> 3. **융합**: 작성된 WBS의 최하단 요소인 워크 패키지(Work Package)는 **간트 차트(Gantt Chart)**의 일정으로 매핑되고, **EVM (Earned Value Management, 획득 가치 관리)**을 통해 프로젝트 진척도와 비용을 계산하는 수학적 뼈대 역할을 한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: WBS는 "어떻게(How) 할 것인가"를 적는 일기장이 아니라, "무엇(What)을 만들어야 하는가"를 적는 명세서다. 집을 지을 때 '기초 공사', '골조 공사', '인테리어'로 나누고, 인테리어를 다시 '도배', '바닥', '조명'으로 끝까지 쪼개어 내려가는 작업이다.
- **💡 비유**: 복잡한 레고 성을 조립할 때 설명서 맨 앞에 있는 '전체 부품 구성표'와 같습니다. 이 표에 있는 부품(Work Package)을 빠짐없이 100% 다 조립하면 거대한 성(Project)이 무조건 완성되는 원리입니다.
- **등장 배경**:
  - **기존 한계**: 과거 프로젝트 매니저(PM)들은 "집 짓기 프로젝트: 1년 소요, 10억 예산"이라는 뭉뚱그린 계획만 가지고 일을 시작했다. 중간에 지붕 공사를 빼먹거나, 예산이 어디서 구멍 났는지 추적할 수 없는 치명적인 실패(Overrun)가 반복되었다.
  - **혁신적 패러다임**: 1960년대 미국 국방부(DoD)와 NASA가 아폴로 계획 같은 초거대 프로젝트를 통제하기 위해, 전체 과업을 아주 작은 단위로 분해하여 각 단위마다 담당자와 예산을 할당하는 WBS 기법을 발명했다.
  - **현재의 비즈니스 요구**: 현대 SI 프로젝트에서는 WBS가 없으면 고객과 대금 결제(검수) 자체를 할 수 없다. "WBS 상의 2.1번 모듈(로그인 기능) 개발이 완료되었으므로 중도금 20%를 지급한다"는 식의 명확한 상거래 계약의 기준으로 쓰인다.

> 다음은 단순히 일정을 나열한 To-Do 리스트와 계층적 WBS의 구조적 차이를 보여주는 비교 도식이다.

```text
┌─────────────────────────────────────────────────────────────┐
│ [Simple To-Do List vs Hierarchical WBS]                     │
├──────────────────────────────┬──────────────────────────────┤
│ 1. To-Do List (비구조적)     │ 2. WBS (계층적 / 100% Rule)  │
│                              │                              │
│ [ ] 서버 사기                │ 1.0 쇼핑몰 구축 프로젝트     │
│ [ ] 로그인 화면 만들기       │  ├── 1.1 인프라 세팅 (20%)   │
│ [ ] 결제 시스템 붙이기       │  │    ├── 1.1.1 서버 구매  │
│ [ ] 데이터베이스 설계하기    │  │    └── 1.1.2 DB 설계    │
│                              │  └── 1.2 소프트웨어 개발(80%)│
│ * 문제: 작업 간의 연관성이   │       ├── 1.2.1 로그인 화면│
│ 없고, 전체 100%를 달성했는지 │       └── 1.2.2 결제 시스템│
│ 증명할 수 없음 (누락 발생)   │ * 강점: 하위 합이 상위 100%  │
└──────────────────────────────┴──────────────────────────────┘
```
- **해설**: To-Do 리스트는 그냥 생각나는 대로 적은 것이다. WBS는 철저한 **MECE (Mutually Exclusive, Collectively Exhaustive)** 원칙에 따라 겹치거나 빠지는 작업 없이 전체를 100%로 쪼개는 수학적 트리 구조다.

> **📢 섹션 요약 비유**: 수만 조각의 직소 퍼즐을 맞출 때, 박스 안에 조각들을 그냥 섞어두는 것(To-Do List)이 아니라, 하늘색 조각, 땅 조각, 집 조각으로 봉투에 나누어 담고 그 봉투들을 합치면 무조건 100% 완성되는 그림(WBS)을 짜는 과정입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. WBS의 계층 구조와 용어 (Hierarchy Levels)

WBS는 보통 3~6레벨까지 내려가며, 최하위 단위를 '워크 패키지'라고 부른다.

| 계층 레벨 | 명칭 (Terminology) | 특징 및 역할 | 비유 |
|:---|:---|:---|:---|
| **Level 1** | **프로젝트 (Project)** | 최종적으로 달성해야 할 유일한 전체 결과물 (100%) | 요리 전체 코스 (디너 세트) |
| **Level 2** | **통제 계정 (Control Account)** | 관리 목적으로 묶은 큰 덩어리. 이 단위로 예산과 성과(EVM)를 주로 측정함 | 애피타이저, 메인 요리, 디저트 파트 |
| **Level 3** | **계획 패키지 (Planning Package)** | 아직 세부 내역은 모르지만 나중에 쪼개기 위해 남겨둔 임시 덩어리 | 메인 요리 중 '스테이크' 부분 |
| **Level 4** | **워크 패키지 (Work Package)** | WBS의 **최하위 요소**. 담당자 1명에게 할당 가능하고, 일정(시간)과 비용 산정이 가능한 최소 단위 | 소고기 굽기, 소스 만들기 |

### 2. 100% Rule (백퍼센트 규칙)의 절대성

WBS의 생명은 '100% 규칙'이다. 이는 두 가지 의미를 갖는다.
1. **상향식 100%**: 1.1.1 작업과 1.1.2 작업을 합치면 상위 1.1 작업이 완벽하게 100% 완성되어야 한다. (99%도 안 되고 101%도 안 된다).
2. **범위의 엄격성**: WBS에 포함되지 않은 작업은 **"이 프로젝트의 범위가 아니다(Out of Scope)"**라는 뜻이다. 누군가 WBS에 없는 일을 하면 그것은 예산 낭비(Gold Plating)다.

> 아래 다이어그램은 100% 룰이 하위에서 상위로 어떻게 합산되는지를 보여주는 논리 도식이다.

```text
[WBS 100% Rule Mechanism]

                  [ 1.0 프로젝트 전체 (100%) ]
                                │
             +------------------+------------------+
             │                                     │
    [ 1.1 설계 (30%) ]                    [ 1.2 구현 (70%) ]
             │                                     │
      +------+------+                       +------+------+
      │             │                       │             │
[1.1.1 UI (10%)] [1.1.2 DB (20%)]     [1.2.1 프론트(30%)] [1.2.2 백엔드(40%)]

* 검증 1: 1.1.1 (10%) + 1.1.2 (20%) = 1.1 설계 (30%) -> 완벽한 100% Rule 만족!
* 검증 2: 설계(30%) + 구현(70%) = 프로젝트 전체(100%) -> 누락 없음!
```

### 3. 워크 패키지(Work Package)를 쪼개는 기준 (8/80 Rule)
"그럼 얼마나 작게 쪼개야 하는가?"라는 딜레마가 생긴다. 너무 크게 쪼개면 관리가 안 되고, 너무 작게(예: "개발자가 커피 마시기") 쪼개면 엑셀 파일이 1만 줄이 넘어 마비된다.
- **8/80 법칙 (8-80 Rule)**: 하나의 워크 패키지는 최소 8시간(하루) 이상, 최대 80시간(2주) 이하로 쪼개야 관리가 가능하다는 PMI의 권장 룰이다.

> **📢 섹션 요약 비유**: 케이크를 자를 때 한 판 전체(Level 1)를 절반(Level 2)으로, 다시 8조각(Work Package)으로 자릅니다. 8조각을 합치면 무조건 한 판(100% Rule)이 되어야 하며, 케이크 조각이 입에 들어가기 적당한 크기(8/80 Rule)여야 먹기(관리하기) 편합니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. WBS vs 로직 트리(Logic Tree) vs 간트 차트(Gantt Chart)

| 비교 항목 | 로직 트리 (Logic Tree) | WBS (작업 분할 구조도) | 간트 차트 (Gantt Chart) |
|:---|:---|:---|:---|
| **핵심 목적** | 문제의 **원인(Why)과 해결책(How)** 도출 | 프로젝트의 **결과물(What)**과 범위 정의 | 작업의 **순서(When)**와 일정 추적 |
| **시간의 흐름** | 없음 | **없음** (무엇을 만들지만 정의함) | **있음** (선후행 관계, 타임라인 필수) |
| **담당자 할당** | 보통 없음 | 워크 패키지 단위로 리소스(사람/비용) 할당 | 각 일정 바(Bar)에 담당자 지정 |
| **관계** | 기획 단계의 생각 정리 도구 | **프로젝트 계획의 뼈대 (가장 먼저 만듦)** | WBS를 바탕으로 간트 차트를 그림 |

### 2. 애자일(Agile) 환경에서의 WBS (제품 백로그)
전통적 폭포수(Waterfall) 모델에서는 WBS를 프로젝트 초기에 100% 완벽하게 짜놓고 시작한다(Big Design Up Front). 하지만 애자일 환경에서는 "초기에는 무엇을 만들지 100% 알 수 없다"고 가정하므로 WBS라는 단어 대신 **제품 백로그(Product Backlog)**라는 개념으로 대체 및 융합된다.

> 다음은 폭포수의 WBS 계층과 애자일의 백로그 계층을 1:1로 비교한 매핑 도식이다.

```text
[Waterfall WBS vs Agile Backlog Hierarchy]

[ Waterfall (WBS) ]                   [ Agile (Scrum) ]
 1.0 프로젝트 (Project)      <=======>  제품 비전 (Product Vision)
      |                                      |
 1.1 통제 계정 (Control Acc.)<=======>  에픽 (Epic) : "거대한 결제 기능"
      |                                      |
 1.1.1 워크 패키지 (Work Pkg)<=======>  사용자 스토리 (User Story) : "카카오페이 결제"
      |                                      |
 (WBS 사전에 명시 안함)      <=======>  태스크 (Task) : "API 연동 코드 작성" (개발자 자율)

* 차이점: WBS는 PM이 초기에 100% 확정하여 통제하지만, 애자일 백로그는 
          시간이 지나면서 밑단(Task)이 구체화되는 점진적 구체화(Rolling Wave)를 따름.
```

> **📢 섹션 요약 비유**: 폭포수 WBS가 건물 짓기 전 완성된 '설계 도면'이라면, 애자일 백로그는 게임을 만들 때 큰 괴물(에픽)을 잡는다는 목표만 두고, 싸우면서 작은 퀘스트(스토리)를 계속 업데이트하는 '살아 숨 쉬는 퀘스트 창'입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오와 의사결정
- **상황**: 50억 원 규모의 사내 포털 개편 프로젝트 중반. 현업 부서장이 "임원용 모바일 뷰 페이지가 왜 없냐! 당장 추가해라!"고 PM에게 화를 낸다. 개발팀은 "그건 계약(범위)에 없었다"며 반발한다.
- **판단**: 요구사항이 자꾸 늘어나는 **Scope Creep(범위 팽창)** 위기다. 이때 PM은 WBS와 그 상세 설명서인 **WBS 사전(WBS Dictionary)**을 꺼내든다.
- **결과**: "본부장님, 프로젝트 착수 시 승인하신 WBS의 '2.3 포털 화면 개발' 패키지 상세 정의서(Dictionary)를 보시면, 모바일 뷰는 개발 범위(Out of Scope)에서 제외되어 있습니다. 100% Rule에 따라 예산이 할당되지 않은 작업입니다. 추가를 원하시면 CCB(변경 통제 위원회)를 열고 예산 5천만 원을 증액해주셔야 합니다."라고 방어하여 개발팀의 야근을 막아낸다.

### 2. 도입 시 핵심 체크리스트 (Technical & Operational)
1. **WBS 사전 (WBS Dictionary)의 동반 작성**: WBS 엑셀표에 "1.1.1 DB 설계"라고만 적어두면 개발자는 "테이블 10개만 그리면 되나?"라고 오해한다. WBS 사전(설명서)을 통해 "테이블 100개, 정규화 3차까지, 산출물은 ERD.pdf"라고 명확한 **인수 기준(Acceptance Criteria)**을 적어두었는가?
2. **산출물(Deliverables) 중심의 분할**: WBS를 "요구분석 -> 설계 -> 개발" 같은 시간/절차(Phase) 중심으로만 쪼개면 100% 검증이 힘들다. "결제 모듈 -> 장바구니 모듈 -> 로그인 모듈"처럼 눈에 보이는 결과물(명사형) 중심으로 쪼개는 것이 훨씬 강력한 통제력을 갖는다.
3. **롤링 웨이브 기획 (Rolling Wave Planning)**: 3년짜리 프로젝트인데, 3년 뒤에 할 디자인 작업을 지금 워크 패키지(8/80 Rule) 단위로 쪼개는 것은 바보짓이다. 가까운 3개월은 워크 패키지로 잘게 쪼개고, 2년 뒤의 일은 통제 계정(큰 덩어리)으로 크게 남겨두는 점진적 구체화 기법을 적용했는가?

### 3. 치명적 안티패턴 (Anti-Pattern)
- **금도금 (Gold Plating) 허용**: 개발자나 기획자가 고객이 요구하지도 않았고 WBS에도 없는 '스스로 멋지다고 생각하는 숨겨진 기능'을 몰래 개발하여 소스코드에 끼워 넣는 행위.

> 아래는 100% Rule을 위반한 '금도금(Gold Plating)'이 불러오는 안티패턴 도식이다.

```text
[Anti-Pattern: Gold Plating (Breaking the 100% Rule)]

(WBS 100% 확정) 
 1.1 로그인 기능 (ID/PW 방식) - 10일 할당

(개발자 김대리의 과도한 열정)
 "요즘 힙한 회사들은 지문 인식(생체 인증) 쓰던데? 내가 주말에 몰래 추가해서 
  깜짝 놀라게 해줘야지! (WBS에 없는 101% 작업 진행)"
       |
       v (오픈 당일)
(결과) 지문 인식 모듈과 기존 DB가 충돌하여 전체 로그인 서버 마비.
       고객: "우린 지문 인식 원한 적도 없는데 왜 넣어서 서버 터트려요?!" 
       => 소송 발생.
```
- **해설**: 프로젝트 관리에 있어서 "고객의 기대를 초과하는 서비스를 공짜로 주는 것(Gold Plating)"은 선행이 아니라 **엄격한 규정 위반(범위 이탈)**이다. 100% Rule은 99%도 안 되지만 101%도 절대 허용하지 않는 무자비한 규칙이다. 모든 것은 WBS 내에서 통제되어야 한다.

> **📢 섹션 요약 비유**: 택배(WBS)로 '김치찌개 1인분'을 시켰는데, 배달원이 자기가 낚시해서 잡은 '생선회(금도금)'를 서비스라며 몰래 같이 넣어주면, 찌개에서 비린내가 나서 음식 전체를 망칩니다. WBS는 시킨 것만, 딱 100%만 배달하게 만드는 엄격한 계약서입니다.

| 기대효과 구분 | 도입 전 (To-Do List 의존) | 도입 후 (WBS 100% Rule 적용) | ROI 및 변화 지표 |
|:---|:---|:---|:---|
| **범위 팽창 방어** | 현업의 무한 추가 요청 수용으로 야근 | WBS 사전을 방패로 변경 요청(CR) 과금 | 프로젝트 예산 준수율(Budget Adherence) 90% 이상 |

### 2. 미래 전망 및 아키텍처 진화
- **AI 기반 WBS 자동 생성 (GenAI for PM)**: 예전에는 PM이 며칠 밤을 새우며 엑셀로 WBS를 그렸다. 이제는 ChatGPT나 Jira AI 플러그인에 "MSA 기반 쇼핑몰 구축 프로젝트의 4단계 WBS를 표 형태로 그려줘"라고 프롬프트를 치면, 전 세계 Best Practice를 긁어와 누락 없는 WBS 초안을 10초 만에 생성해 주는 시대로 진화했다.
- **애자일과 폭포수의 하이브리드 통제**: 기업들은 여전히 예산 핑계로 폭포수(WBS)를 요구하지만, 개발팀은 애자일(Backlog)로 일하길 원한다. 이에 최신 ITSM 도구들은 상위 레벨(Level 1~2)은 WBS로 예산을 통제하고, 최하위 워크 패키지(Level 4)는 스크럼 보드의 User Story로 자동 연동시켜 두 세계를 융합하는 워터스크럼폴(Water-Scrum-Fall) 아키텍처를 채택하고 있다.

> **📢 섹션 요약 비유**: WBS는 건물을 지을 때 바닥에 박는 '눈금자'이자 '설계도'입니다. 이 눈금자가 1cm라도 틀어지면 10층 높이에서는 건물이 무너집니다. 100% Rule이라는 무자비한 철칙이 프로젝트라는 불안정한 건물을 쓰러지지 않게 지탱하는 중력의 역할을 합니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- EVM (Earned Value Management) | WBS의 워크 패키지마다 예산을 매달아 두고, "지금까지 쓴 돈(AC)"과 "실제 완성된 작업의 가치(EV)"를 수학적으로 비교하여 프로젝트가 망해가는지 조기 경보를 울리는 기법
- 제품 백로그 (Product Backlog) | 애자일 프레임워크에서 WBS를 대신하여, 고객에게 가치를 주는 기능들을 우선순위대로 나열해 둔 유동적인 요구사항 목록
- 금도금 (Gold Plating) | 고객이 요구하지 않은 추가 기능이나 노력을 제공하여 100% 범위를 임의로 초과하는 행위로, 치명적인 비용 낭비와 리스크를 초래하는 안티패턴
- 프로젝트 헌장 (Project Charter) | WBS를 만들기 전에 작성되는 문서로, WBS의 뿌리(Level 1)가 되는 프로젝트의 거시적 목적과 예산 통제권을 부여하는 스폰서 서명 문서
- 범위 크립 (Scope Creep) | 100% Rule이 깨지고 통제위원회(CCB)의 공식 승인 없이 프로젝트 범위가 야금야금 끝없이 늘어나는 재앙적 현상

### 👶 어린이를 위한 3줄 비유 설명
1. **개념**: 학교에서 조별 과제로 '커다란 로봇 만들기'를 할 때, 머리, 팔, 다리로 나누고 다시 손가락, 발가락 부품까지 쪼개서 친구들에게 나눠주는 계획표예요.
2. **원리**: 가장 중요한 건 '100% 규칙'이에요. 친구들이 맡은 부품(손가락, 발가락)을 다 합치면 로봇 하나가 완벽하게 100% 만들어져야지, 팔이 하나 비거나 쓸데없이 꼬리가 하나 더 생기면 안 되거든요.
3. **효과**: 이 표(WBS)를 벽에 붙여두면 "철수야, 넌 오른팔 만들기로 했지?"라고 책임질 사람을 딱 정할 수 있어서, 아무도 농땡이 피우지 않고 완벽한 로봇을 기한 내에 만들 수 있답니다!
