# 981. ESP (Encapsulating Security Payload) - IPSec 기밀성 무결성 데이터 암호화 터널 수송 모드 통신망 완벽한 보호 구조 캡슐화 원리

> **핵심 인사이트**: 980번 문서의 AH는 해커가 내용물을 조작하는 건(무결성) 막았지만, 유리 금고라서 해커가 내용물을 눈으로 훔쳐보는 것(도청)은 막지 못하는 반쪽짜리였다. "야! 군사 기밀인데 밖에서 훤히 다 보이게 투명 유리로 보내면 어떡해! 유리 금고 갖다 버려! 내용을 0.1초 만에 시커먼 먹물(AES 암호화)로 칠해서 아무도 못 읽게 만들고, 그 위에 조작 방지(무결성) 자물쇠까지 완벽하게 이중으로 채워버려!" 도청과 조작을 한방에 막아내는 100% 흑연 강철 금고, IPSec의 진짜 주인공 ESP다.

## Ⅰ. ESP (Encapsulating Security Payload)의 개념
- **개념**: 980번 AH의 치명적 한계(도청 가능성)를 극복하기 위해, IP 패킷의 데이터(Payload) 전체를 강력한 대칭키 암호화 알고리즘(AES, 3DES 등)으로 시커멓게 칠해버려서 **'기밀성(Confidentiality)'을 제공하고, 덤으로 무결성(인증) 기능까지 함께 제공하는 IPSec의 핵심 완벽 방어 프로토콜**입니다.

## Ⅱ. ESP의 2대 극강 방어력 🌟

### 1. 완벽한 기밀성 (Confidentiality) - 도청 불가
- 편지 내용물(TCP 헤더 + 웹 데이터 등)을 완전히 암호화합니다.
- 중간에 해커가 광케이블을 따서 스니핑(도청)을 해도, 안에는 암호 해독 키를 모르면 1만 년이 걸려도 풀 수 없는 외계어 쓰레기 텍스트만 잔뜩 들어 있습니다. (VPN의 존재 이유)

### 2. NAT 횡단의 구원자 (AH와의 결정적 차이) 🌟 핵심 🌟
- 앞서 980번 문서에서 AH는 IP 주소까지 같이 도장을 찍어버려서, 중간 공유기(NAT)가 IP를 바꿀 때 에러가 터져 멸망했다고 배웠습니다.
- **ESP의 영리함**: ESP는 도장(무결성 해시)을 찍을 때, **'맨 앞의 바깥쪽 IP 주소 헤더'는 일부러 도장 범위에서 빼고 쿨하게 무시합니다.** 
- **결과**: 중간에 공유기(NAT)가 겉면 IP 주소를 맘대로 뜯어고쳐도, 도착지에서 도장을 확인(무결성 검사)할 때 안쪽 내용물만 멀쩡하면 패킷을 통과시켜 줍니다. 즉, **ESP는 공유기(NAT) 환경을 100% 무사 통과할 수 있습니다.** 실무에서 VPN을 구성할 때 AH를 버리고 100% ESP만 쓰는 절대적 이유입니다.

## Ⅲ. ESP의 캡슐화 구조 (터널 모드 기준) 🌟
VPN 장비에서 가장 흔히 쓰이는 'ESP + 터널 모드'의 구조입니다. (시험 단골)

1. `[원래 IP 헤더] + [진짜 데이터]` (쌩얼 패킷이 VPN 장비에 들어옴)
2. VPN 장비가 쌩얼 전체를 까맣게 암호화합니다.
3. 그 앞에 **`[ESP 헤더]`**를 붙이고, 맨 뒤에 **`[ESP 트레일러]`**와 무결성 도장인 **`[ESP Auth]`**를 붙여 햄버거처럼 완벽히 쌉니다.
4. 맨 앞에 새로운 목적지 주소인 **`[새로운 IP 헤더 (New IP)]`**를 붙여서 쏩니다.
- **최종 구조**: `[New IP] + [ESP 헤더] + {암호화됨: [원래 IP] + [진짜 데이터] + [ESP 트레일러]} + [ESP Auth]`

> 📢 **섹션 요약 비유**: AH가 '투명한 플라스틱 상자에 씰(도장)을 붙인 것'이라면, **ESP(보안 페이로드 캡슐화)**는 내용물 자체를 시커먼 먹물로 칠한 뒤 **'불투명한 티타늄 강철 상자'**에 넣고 용접해버린 것입니다. 우체국 도둑(해커)은 내용물을 훔쳐보기는커녕 이게 무슨 물건인지 짐작조차 할 수 없습니다(도청 완벽 차단). 게다가 강철 상자 바깥쪽에는 택배 기사(NAT 공유기)가 맘대로 주소지 스티커를 떼었다 붙였다 할 수 있도록 주소표 칸을 살짝 비워두었습니다. 이 똑똑한 설계 덕분에, ESP는 목적지까지 가는 길에 주소가 10번 바뀌어도(NAT 횡단) 에러 한 번 안 터지고 완벽한 비밀 유지 상태로 무사히 배달되는 글로벌 VPN 시장의 유일무이한 표준 포장법이 되었습니다.
