# 11. 페이지 폴트 (Page Fault)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스가 실행 중 가상 주소를 참조했는데, 해당 **페이지가 실제 물리 메모리(RAM)에 올라와 있지 않고 보조기억장치(Disk)에 잠들어 있을 때 발생하는 하드웨어 예외(Trap)** 입니다.
> 2. **가치 (필요할 때만 로딩)**: 프로그램 전체를 램에 올리지 않는 **'요구 페이징(Demand Paging)'** 의 핵심 메커니즘으로, 램 공간을 아끼면서도 거대한 프로그램을 실행할 수 있게 해주는 성능과 자원의 타협점입니다.
> 3. **융합**: 운영체제의 **페이지 교체 알고리즘**과 융합되어 작동하며, 잦은 페이지 폴트는 시스템이 연산보다 데이터 전송에 시간을 다 쓰는 **스래싱(Thrashing)** 현상을 유발하는 경계 대상입니다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: "데이터가 집에 없어요!" CPU가 MMU를 통해 주소를 변환하려는데, 페이지 테이블의 유효 비트(Valid Bit)가 0인 경우 발생합니다.
- **💡 비유**: **'도서관의 희망도서 신청'** 과 같습니다. 
  - 내가 읽고 싶은 책(페이지)이 서가(RAM)에 꽂혀 있지 않습니다. 
  - 나는 사서(OS)에게 "이 책 빌려주세요!"라고 요청(Page Fault)합니다. 
  - 사서는 먼 창고(Disk)에서 책을 가져와 서가에 꽂아주고, 나는 비로소 책을 읽을 수 있게 됩니다.
- **등장 배경**:
  1. 메모리 절약: 프로그램의 모든 코드가 동시에 쓰이지는 않습니다. 
  2. 게으른 적재: "쓰기 직전에만 가져오자"는 철학이 페이지 폴트라는 '지연된 처리' 방식을 낳았습니다.

---

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

페이지 폴트 발생 시 시스템이 겪는 6단계 복구 시나리오를 해부합니다.

#### 1. 처리 프로세스 (The Recovery Loop)
1. **Trapping**: CPU가 유효 비트 0을 발견하고 운영체제에 비명(Trap)을 지릅니다.
2. **Back-up**: 현재 수행 중인 레지스터 상태를 스택에 대피시킵니다.
3. **Disk I/O**: OS가 디스크에서 해당 페이지를 찾습니다.
4. **Allocation**: 빈 프레임을 찾아 페이지를 RAM에 적재합니다. (빈 곳이 없으면 기존 놈 하나를 버립니다).
5. **Update**: 페이지 테이블의 유효 비트를 1로 바꾸고 프레임 번호를 적습니다.
6. **Restart**: 멈췄던 명령어를 처음부터 다시 실행합니다.

#### 2. 유효 비트 (Valid-Invalid Bit)
- 페이지 테이블의 한 열을 차지합니다. 
- **1**: 메모리에 있음. (Hit).
- **0**: 디스크에 있음. (Fault).

#### 3. 페이지 폴트의 비용 (The Penalty)
- 메모리 접근: ~100ns.
- 디스크 접근: ~10ms. (약 10만 배 차이).
- **기술사적 판단**: "페이지 폴트가 단 1%만 발생해도 전체 시스템 속도는 기어가는 수준으로 떡락한다."

---

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 캐시 미스 vs 페이지 폴트 결착망
| 항목 | **캐시 미스 (Cache Miss)** | **페이지 폴트 (Page Fault)** |
|---|---|---|
| **관리 주체** | **하드웨어** | **운영체제 (S/W)** |
| **원인** | L1~L3 캐시에 없음 | **RAM(주기억장치)에 없음** |
| **대기 시간** | 수백 클럭 (ns) | **수백만 클럭 (ms)** |
| **비유** | 편의점에 물건 없음 | **공장에 물건 없음** |

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 시스템 아키텍트의 '페이지 폴트' 억제 전략
- **시나리오**: 실시간 영상 처리 서버가 가끔씩 1초 정도 멈추는 프리징 현상이 발생합니다. 
- **아키텍트 분석**: "로그 확인 결과, 가끔씩 발생하는 **페이지 폴트**가 디스크 I/O를 기다리느라 프로세스를 통째로 멈추고 있다. 실시간성(Determinism)이 파괴되었다."
- **전략적 조치 (Memory Locking)**: 
  - 리눅스의 `mlock()` 시스템 콜을 사용하여, 핵심 알고리즘 코드와 데이터를 **물리 메모리에 강제로 고정(Pinning)** 합니다. 
  - 운영체제가 절대로 이 페이지를 디스크로 내쫓지 못하게 함으로써, 페이지 폴트 확률을 0%로 압살하고 무결점 실시간 응답성을 사수합니다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

- **결론**: 페이지 폴트는 가상 메모리라는 거대한 마법을 부리기 위해 지불해야 하는 **'숙명적인 통행료'** 입니다. 이 통행료를 최소화하기 위해 인류는 수십 년간 페이지 교체 알고리즘을 깎아왔습니다. 앞으로의 아키텍처는 초고속 비휘발성 메모리(NVDIMM)를 통해 디스크와 램의 경계를 없애거나, 인공지능이 다음에 필요한 페이지를 미리 짐작하여 페이지 폴트 자체를 소멸시키는 **'제로 폴트 아키텍처'** 를 향해 나아갈 것입니다.
